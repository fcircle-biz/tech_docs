仕上げ

削除制約やエラーハンドリングを追加し、システムを完成させます。注文履歴がある顧客・商品の削除制約を実装し、全機能の動作確認を行います。

このステップで実装すること

  • 削除制約の実装（注文履歴がある顧客・商品は削除不可）
  • エラーハンドリングの改善（ユーザーへのメッセージ表示）
  • UI/UXの最終調整（ローディング表示、エラー表示）
  • 全機能の動作確認とテスト

実装のヒント

削除制約の実装ヒント

  • • 顧客削除: customers.pyで注文履歴の有無をチェック（ordersテーブルでcustomer_id検索）
  • • 商品削除: products.pyで注文明細の有無をチェック（order_itemsテーブルでproduct_id検索）
  • • エラー返却: HTTPException（status_code=400）で「注文履歴があるため削除できません」
  • • 参考: database.mdの「削除時の制約」セクションを参照

エラーハンドリングのヒント

  • • バックエンド: HTTPExceptionで適切なステータスコードとメッセージを返す
  • • フロントエンド: try-catchでエラーをキャッチし、ユーザーにメッセージ表示
  • • ユーザー体験: エラー時は赤いアラート、成功時は緑のトーストを表示
  • • 参考: README.mdの「エラーメッセージ例」を確認

UI/UX改善のヒント

  • • ローディング表示: データ取得中はスピナーを表示（useState("loading")）
  • • 削除確認: window.confirmまたはモーダルで削除確認を追加
  • • 空状態: データがない場合の表示を改善（「まだ登録されていません」等）
  • • レスポンシブ: モバイル表示を確認し、必要に応じて調整

動作確認のポイント

  • • 削除制約: 注文履歴がある顧客・商品を削除しようとした際にエラーが表示される
  • • 全機能: 顧客・商品・注文・ダッシュボードすべての操作を一通りテスト
  • • エラーパス: 不正な入力、存在しないID、バリデーションエラーを確認
  • • チェックリスト: ページ下部の完了チェックリストを活用
