注文管理機能

顧客と商品を組み合わせて注文を作成・管理する機能を実装します。複数テーブルのリレーションとステータス管理を学ぶ、この課題の核心部分です。

このステップで実装すること

  • Order/OrderItemモデルの作成（注文ヘッダーと明細）
  • 注文API（作成、一覧取得、詳細取得、ステータス更新、キャンセル）
  • 注文一覧画面（ステータス別フィルタ）
  • 注文作成画面（顧客選択、商品追加、数量変更、合計計算）
  • 注文詳細画面（明細表示、ステータス更新ボタン）

参照する設計資料

このステップでは、以下の設計資料を参照して実装を進めてください：

システム仕様書 - 注文管理機能、注文API、注文ステータス API設計とステータス遷移ルールを確認

実装のヒント

バックエンド実装のヒント

  • • モデル定義: database.md のordersテーブル、order_itemsテーブル定義を参考にSQLModelクラスを作成
  • • 外部キー: customer_id、product_id に外部キー制約を設定（削除時の挙動も考慮）
  • • 注文作成API: リクエストから顧客ID・商品ID・数量を受け取り、①顧客の存在確認 ②商品の存在・販売状態確認 ③単価取得→小計計算→合計計算 ④注文ヘッダー作成 ⑤注文明細作成
  • • JOINの使用: 一覧・詳細取得で顧客名・商品名を表示するため、JOINが必要（database.md参照）
  • • ステータス更新: README.md の「注文ステータス」セクションを参考に、遷移ルールを実装

フロントエンド実装のヒント

  • • 一覧画面: useEffectでAPI呼び出し、テーブル表示、ステータスでフィルタ（ボタンまたはタブ）
  • • 注文作成画面: ①顧客選択（ドロップダウン）②商品追加（商品一覧から選択→リストに追加）③数量変更（+/-ボタン）④合計計算（リアルタイム更新）
  • • 動的フォーム: 明細行を配列（useState）で管理し、追加・削除ボタンで配列を操作
  • • 注文詳細画面: 明細をテーブル表示、ステータスを色分け（受付:黄、調理中:青、完了:緑、キャンセル:灰）
  • • 参考: 顧客管理・商品管理で作った一覧・フォームを拡張する形で実装

リレーションとJOINのヒント

  • • 1対多の関係: 1つの注文（Order）に複数の明細（OrderItem）が紐づく
  • • 外部キー: OrderItem.order_id → Order.id、OrderItem.product_id → Product.id
  • • JOIN: 一覧で顧客名を表示する場合、OrderとCustomerをJOIN（database.md参照）
  • • 注文作成時: トランザクションを使って、注文ヘッダーと明細を一括で作成

よくあるミス

  • • 注文作成時に顧客ID・商品IDの存在確認を忘れる（存在しないIDでエラー）
  • • 商品の販売状態（is_available）を確認せず、販売停止中の商品を注文できてしまう
  • • 合計金額の計算が間違っている（単価×数量の合計）
  • • ステータス更新時に遷移ルールをチェックせず、不正な遷移を許可してしまう
  • • フロントエンドで明細行の配列管理がうまくできず、商品追加・削除が動作しない
