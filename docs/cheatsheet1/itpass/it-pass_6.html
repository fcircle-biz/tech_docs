6-1 システム開発

システム開発とは、要件定義から始まり、設計、プログラミング、テストを経て、最終的に運用に至るまでの一連のプロセスです。

システム開発のライフサイクル

企画から廃棄までの全過程をシステムライフサイクルと呼びます。

1
要件定義: システムに求められる機能や性能を明確化
2
設計: 外部設計と内部設計に分けて進める
3
プログラミング: 設計書に基づいたコード作成
4
テスト: 単体・結合・システムテストを実施
5
運用・保守: 稼働後の管理と改善
📐

6-2 システム設計

システム設計は外部設計と内部設計の2段階で行われます。

設計の階層構造

設計プロセスは段階的に詳細化されていきます。

1
システム方式設計: システム全体のアーキテクチャを決定
2
ソフトウェア方式設計: ソフトウェアの構造設計
3
ソフトウェア詳細設計: 各モジュールの詳細化
基本設計（外部設計） 詳細設計（内部設計）
ユーザーから見た機能や画面 開発者視点のプログラム構造
入出力の定義、画面遷移 データ構造、アルゴリズム
DB設計（ER図） DB詳細設計
非機能要件の検討 モジュール間インターフェース
ユーザー体験の設計

UI(ユーザーインターフェース)、UX(ユーザー体験)、CX(カスタマー体験)の違いと重要性

UI: 操作画面のデザイン
UX: 使いやすさ全般
CX: 顧客体験の総合
アクセシビリティ: 誰もが使える
優れたUI/UXが顧客満足度と事業成功につながる

設計工程では、UMLDFDE-R図ワイヤーフレームプロトタイプなどの設計手法が用いられます。

6-3 プログラミング

プログラミングは、設計書に基づいてコンピュータが理解できるプログラミング言語で記述する工程です。

主要なプログラミング言語

様々な目的やプラットフォームに応じたプログラミング言語が存在します。

言語 特徴と主な用途
Java クロスプラットフォーム、エンタープライズ系システム開発
C システムプログラミング、ハードウェア制御、高速処理
C++ オブジェクト指向のC、ゲーム開発、システムソフトウェア
Python シンプルな文法、データ分析、AI、Webバックエンド
JavaScript Web開発、ブラウザ上の動的処理、Node.jsでサーバーも
オブジェクト指向プログラミング

現代の主流となっている開発手法で、データと処理をオブジェクトとしてまとめて扱います。

クラス: オブジェクトの設計図
カプセル化: 情報隠蔽
継承: 特性の引き継ぎ
ポリモーフィズム: 多態性
API（アプリケーションプログラミングインターフェース）

ソフトウェア間の連携を可能にする仕組みで、機能の共有と再利用を実現します。

コーディング規約を設けることで、チーム開発での一貫性と保守性を高めることができます。

マークアップ言語と スタイルシート

文書の構造や見た目を定義する言語で、プログラミング言語とは異なる役割を担います。

HTML: Webページの構造
XML: 汎用データ構造
CSS: スタイル定義
XHTML: XMLベースHTML
Web API（アプリケーションプログラミングインターフェース）

WebアプリケーションがHTTPプロトコルを通じて他のシステムとデータ連携する仕組みです。インターネットを介して異なるシステム間での機能連携を可能にし、情報の取得・更新・削除などの操作を実現します。サービス間連携やマッシュアップサービスの基盤技術として重要です。

マークアップ言語とプログラミング言語を組み合わせてWebアプリケーションを構築

6-4 アルゴリズムとデータ構造

アルゴリズムは問題解決の手順、データ構造はデータの効率的な格納方法です。

アルゴリズム 特徴
線形探索 順番に探索する単純な方法
二分探索 ソート済データの高速検索
バブルソート 隣接要素の比較と交換
配列（Array）

同じデータ型の要素を連続したメモリ領域に格納するデータ構造です。インデックス（添え字）を使って各要素に直接アクセスできます。

1
インデックス参照: O(1)の時間で任意の要素に直接アクセス可能
2
メモリ効率: 連続したメモリ領域を使用するため効率的
3
多次元配列: 行列などの表現に使用する二次元以上の配列
4
制約: サイズが固定で、挿入・削除操作が非効率的
配列の要素数がn個のとき、i番目の要素のメモリアドレス = 先頭アドレス + i × 要素サイズ
リスト: 可変長構造
スタック: LIFO構造
キュー: FIFO構造
木構造: 階層表現
グラフ: 関連性表現
ハッシュ表: キー値検索

6-5 システムのテスト

テストはシステムの品質を確保するための重要な工程で、段階的なテストが行われます。

V字モデル

ソフトウェア開発のライフサイクルを表現したモデルで、開発工程とテスト工程の対応関係を示しています。

  • • 要件定義 に対応する 運用テスト
  • • システム設計 に対応する システムテスト
  • • 詳細設計 に対応する 結合テスト
  • • プログラミング に対応する 単体テスト

各工程で作成した成果物に対応するテストを行うことで、品質を段階的に確保します。名前の由来は、これらの対応関係をつなぐと「V」の字を描くことから来ています。

1
単体テスト: 個々のモジュールの動作確認
2
結合テスト: モジュール間の連携確認
3
システムテスト: システム全体の機能検証
4
運用テスト: 実運用環境での最終確認
テスト技法

ブラックボックステスト（機能テスト）とホワイトボックステスト（構造テスト）の両方を活用します。

効率的なテストのためにテスト計画書やテストケースを事前に準備することが重要です。

6-6 開発モデル

システム開発の進め方には様々な開発モデルがあります。

ウォーターフォールモデル: 順次進行
プロトタイプモデル: 試作品検証
スパイラルモデル: 反復的開発
アジャイル開発: 短期反復
リバースエンジニアリング

完成したシステムやソフトウェアを分析・解析して、その設計や仕様を理解・再構築する手法です。既存システムの保守や機能拡張、互換性確保などに活用されます。

エクストリームプログラミング（XP）

アジャイル開発手法の一種で、高品質なソフトウェアを効率的に開発するための実践的アプローチです。

1
テスト駆動開発（TDD）: テストを先に書き、それをパスするコードを実装
2
ペアプログラミング: 2人で1台のコンピュータを使用して共同開発
3
リファクタリング: 外部動作を変えずに内部構造を改善
4
継続的インテグレーション: 頻繁なコード統合とテスト実行
DevOps

Dev(開発)とOps(運用)を統合したアプローチで、開発チームと運用チームの連携を強化し、ソフトウェアデリバリーのスピードと質を向上させます。

継続的インテグレーション: 自動ビルド/テスト
継続的デリバリー: 自動リリース準備
継続的デプロイメント: 自動本番適用
インフラのコード化: 設定の自動化
モニタリング: 常時監視と分析
フィードバック: 迅速な問題対応

プロジェクトの特性に合わせた開発モデルの選択が成功の鍵です。
