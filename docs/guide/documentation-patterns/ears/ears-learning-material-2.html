第2章: 要求記述の基本原則

良い要求の特性とSMART原則

この章で学ぶこと
  • 良い要求が満たすべき基本的な特性
  • SMART原則の理解と実際の適用方法
  • 曖昧さを排除するための具体的技法
  • 検証可能な要求を作成する方法
  • EARSの5つの構文パターンの概要
  • 要求品質評価の観点

2.1 良い要求が満たすべき特性

優れた要求は、以下の6つの基本特性を満たしていなければなりません。 これらの特性は、要求工学の基本原則として広く受け入れられています。

graph TD A["良い要求の特性"] --> B["明確性
(Clarity)"] A --> C["完全性
(Completeness)"] A --> D["一貫性
(Consistency)"] A --> E["検証可能性
(Verifiability)"] A --> F["実現可能性
(Feasibility)"] A --> G["追跡可能性
(Traceability)"] B -.-> B1["曖昧でない"] C -.-> C1["必要な情報が全て含まれる"] D -.-> D1["他の要求と矛盾しない"] E -.-> E1["テストで確認可能"] F -.-> F1["技術的・経済的に実現可能"] G -.-> G1["設計・実装との関連が明確"]

それぞれの特性について詳しく見ていきましょう：

1. 明確性（Clarity）

要求は一つの解釈のみが可能でなければなりません。 読み手によって異なる理解をされてはいけません。

  • 具体的で測定可能な表現を使用する
  • 専門用語は定義を明記する
  • 代名詞や曖昧な修飾語を避ける
2. 完全性（Completeness）

要求を満たすために必要な情報が全て含まれていなければなりません。

  • 前提条件と例外条件を明記する
  • 入力・出力の仕様を完全に記述する
  • パフォーマンス要件を具体的に定義する

2.2 SMART原則：測定可能な要求作成の指針

SMART原則は、もともと目標設定の分野で開発された概念ですが、 要求記述においても極めて有効な指針となります。EARSもこの原則に基づいて設計されています。

classDiagram class SMART { +S : Specific (具体的) +M : Measurable (測定可能) +A : Achievable (達成可能) +R : Relevant (関連性) +T : Time-bounded (時間制約) } class SpecificDetails { +明確な主語 +具体的な動詞 +定量的な基準 } class MeasurableDetails { +数値による基準 +客観的な評価方法 +テスト可能な条件 } SMART --> SpecificDetails SMART --> MeasurableDetails
悪い要求の例とSMART原則による改善

悪い例：

「システムは高速でなければならない」

SMART原則適用による改善：

【S-Specific】どのシステムの何が？
→ 「検索システムの応答時間が」

【M-Measurable】どう測定する？
→ 「平均1.5秒以内で」

【A-Achievable】実現可能？
→ 「現在のハードウェア構成で技術的に可能」

【R-Relevant】なぜ必要？
→ 「ユーザビリティ向上のため」

【T-Time-bounded】いつまでに？
→ 「通常の営業時間内（9:00-17:00）において」

改善後：「通常営業時間（9:00-17:00）において、検索システムは
ユーザーの検索要求に対して平均1.5秒以内で結果を返す」

2.3 曖昧さを排除するための実践的技法

自然言語には本質的に曖昧性が含まれています。EARSでは、以下の技法を使用して 体系的に曖昧さを排除します：

1. 避けるべき曖昧な表現と改善例
曖昧な表現 問題点 改善例
「迅速に」「高速に」 主観的で測定困難 「3秒以内に」
「適切に」「正しく」 基準が不明確 「仕様書3.2項の基準に従って」
「必要に応じて」 判断基準が不明 「CPU使用率が80%を超えた場合」
「ユーザー」 対象者が不特定 「管理者権限を持つユーザー」
2. 明確化のための5W1H技法

要求を明確にするために、以下の観点で情報を整理します：

  • Who（誰が）：アクターやユーザーの種類
  • What（何を）：システムの機能や動作
  • When（いつ）：トリガーや条件
  • Where（どこで）：システムの範囲や場所
  • Why（なぜ）：要求の目的や理由
  • How（どのように）：品質特性や制約

2.4 EARSの5つの構文パターン概要

EARSは、要求の性質に応じて5つの標準化されたパターンを提供します。 各パターンは特定の状況や要求タイプに最適化されています。

graph TD A["EARS構文パターン"] --> B["Ubiquitous
普遍的要求"] A --> C["Event-driven
イベント駆動要求"] A --> D["Unwanted Behavior
望ましくない動作対応"] A --> E["State-driven
状態依存要求"] A --> F["Optional Feature
オプション機能要求"] B --> B1["The システム名 shall 要求内容"] C --> C1["When 条件, the システム名 shall 要求内容"] D --> D1["If 望ましくない条件, then システム名 shall 要求内容"] E --> E1["While 状態, the システム名 shall 要求内容"] F --> F1["Where 機能が含まれる, the システム名 shall 要求内容"]
パターン選択の基準
  • Ubiquitous：システムが常に満たすべき基本的な要求
  • Event-driven：特定のイベントや操作がトリガーとなる要求
  • Unwanted Behavior：エラーや異常状態への対処要求
  • State-driven：システムの特定の状態でのみ有効な要求
  • Optional Feature：条件付きで実装される機能の要求

2.5 検証可能な要求の作成方法

要求は必ず客観的に検証可能でなければなりません。 検証できない要求は、実装が正しいかどうか判断することができません。

検証可能性の3つの観点
  1. 測定基準の明確化：何をどのように測定するかを具体的に定義
  2. 合格条件の設定：どの値以上/以下であれば要求を満たすかを明記
  3. 測定方法の指定：どのようなテスト手順で検証するかを明示
検証困難な要求の例
悪い例：「システムは使いやすくなければならない」
→ 「使いやすい」は主観的で測定困難

良い例：「新規ユーザーは、10分以内のチュートリアルで
基本操作の80%を習得できる」
→ 時間とタスク成功率で客観的に測定可能
実習 2-1: SMART原則による要求改善

以下の問題のある要求文を、SMART原則を適用して改善してください。

課題

次の要求文を分析し、SMART原則に従って改善版を作成してください：

  1. 「システムは安全でなければならない」
  2. 「データベースへのアクセスは効率的でなければならない」
  3. 「エラーメッセージは分かりやすく表示される」
改善手順

各要求について以下の手順で分析・改善してください：

  1. 現状分析：どの点が曖昧で不明確か？
  2. 情報補完：5W1Hの観点で不足情報を特定
  3. SMART適用：具体的で測定可能な表現に変換
  4. 検証方法：どのようにテストするかを検討
改善例（参考）

「システムは安全でなければならない」の改善プロセス：

【分析】「安全」が曖昧、対象・基準・測定方法が不明

【改善】「認証システムは、3回連続でパスワード入力に失敗した
ユーザーアカウントを自動的に30分間ロックアウトする」

【検証方法】意図的に3回失敗させ、4回目のアクセスが
30分間拒否されることを確認
実習 2-2: EARSパターンの分類

以下のシステム要求について、どのEARSパターンが最適かを判断してください。

課題シナリオ

オンライン banking システムの要求を分析してください：

  1. 「ユーザーがログインボタンをクリックした時、システムは認証画面を表示する」
  2. 「システムは全ての取引履歴を暗号化して保存する」
  3. 「ユーザーのセッションがアクティブな間、システムは自動ログアウト機能を無効にする」
  4. 「不正なアクセスが検出された場合、システムは管理者に警告メールを送信する」
  5. 「SMS認証機能が有効な場合、システムは二段階認証を実行する」
分類基準
  • トリガーや条件の有無を確認
  • 要求が適用される状況を分析
  • システムの基本機能か条件付き機能かを判断
理解度確認クイズ
  1. SMART原則の「M」が表すのはどれですか？
    1. Manageable（管理可能）
    2. Measurable（測定可能）
    3. Modifiable（変更可能）
    4. Minimal（最小限）
  2. 良い要求の特性として含まれないのはどれですか？
    1. 明確性（曖昧でない）
    2. 詳細性（実装方法を含む）
    3. 検証可能性（テスト可能）
    4. 一貫性（矛盾がない）
  3. 「When ユーザーがログインボタンをクリックした時、システムは認証画面を表示する」はどのEARSパターンですか？
    1. Ubiquitous
    2. Event-driven
    3. State-driven
    4. Optional Feature
  4. 要求記述で避けるべき曖昧な表現はどれですか？
    1. 「3秒以内に」
    2. 「管理者ユーザーが」
    3. 「適切に処理する」
    4. 「データベースに保存する」
  5. 検証可能な要求を作成するために最も重要なのはどれですか？
    1. 実装の詳細を記述する
    2. 測定基準を明確に定義する
    3. 専門用語を多用する
    4. できるだけ簡潔に記述する

2.6 章のまとめ

重要なポイント
  • 良い要求の6特性：明確性、完全性、一貫性、検証可能性、実現可能性、追跡可能性
  • SMART原則：具体的で測定可能な要求作成の指針
  • 曖昧さの排除：5W1H技法と明確な表現の使用
  • EARSパターン：5つの構文パターンで要求を体系化
  • 検証可能性：客観的な測定基準と合格条件の設定
  • 実践的アプローチ：理論を実際の要求改善に適用

この章で学んだ基本原則は、次章以降で学習する各EARSパターンの基盤となります。 次の章では、最も基本的なパターンである「Ubiquitous要求パターン」について詳しく学習します。
