第1章: EARSの概要と背景

EARS（Easy Approach to Requirements Syntax）の基礎

この章で学ぶこと

  • 要求工学におけるEARSの位置づけ

1.1 要求工学とは何か

要求工学（Requirements Engineering）は、ソフトウェアシステムやハードウェアシステムの開発において、 「何を作るべきか」を明確に定義し、管理する学問分野です。これは単純に「機能リスト」を作ることではありません。

要求工学の本質は、以下の4つのプロセスから構成されています：

graph TD A["要求引出
(Requirements Elicitation)"] --> B["要求分析
(Requirements Analysis)"] B --> C["要求仕様化
(Requirements Specification)"] C --> D["要求検証
(Requirements Validation)"] D --> A A -.-> A1["ステークホルダーとの対話"] B -.-> B1["優先度付けと整理"] C -.-> C1["文書化と記述"] D -.-> D1["正確性の確認"]

このプロセスの中で、「要求仕様化」の段階で重要になるのが、どのように要求を記述するかという問題です。 ここで従来の自然言語による記述方法には多くの課題がありました。

1.2 従来の要求記述方法の問題点

従来、要求は自然言語（日本語や英語）で記述されることが一般的でした。しかし、この方法には以下のような深刻な問題がありました：

従来の要求記述の典型的な問題例

問題のある記述例：

「システムは迅速にユーザーからの要求に応答する必要がある」

この記述の問題点：

  • 曖昧性：「迅速に」とはどの程度の速さなのか？
  • 主語不明確：「システム」とはどの部分を指すのか？
  • 条件不明：どのような状況での要求なのか？
  • 検証不可能：この要求が満たされているかどうか判断できない

これらの問題を分類すると、以下の6つの主要な課題に整理できます：

graph LR A[従来の要求記述の問題] --> B[曖昧性
Ambiguity] A --> C[不完全性
Incompleteness] A --> D[不一貫性
Inconsistency] A --> E[検証困難
Unverifiability] A --> F[保守困難
Poor Maintainability] A --> G[理解困難
Poor Readability] B -.-> B1[解釈が複数存在] C -.-> C1[必要な情報が不足] D -.-> D1[矛盾する記述] E -.-> E1[テスト設計困難] F -.-> F1[変更影響把握困難] G -.-> G1[関係者の理解に差]

1.3 EARSの誕生と発展の歴史

EARS（Easy Approach to Requirements Syntax）は、2009年にイギリスのロールス・ロイス社とヨーク大学の研究者によって開発された要求記述手法です。 航空機エンジンという極めて高い信頼性が求められる分野での実践から生まれました。

EARSが開発された背景には、以下のような産業界の切実なニーズがありました：

EARS開発の動機
  • 安全性重視：航空機システムでは曖昧な要求が人命に関わる
  • 規制対応：厳格な認証プロセスに対応できる明確な要求記述が必要
  • 国際協力：多国籍チームでの一貫した理解を確保
  • コスト削減：要求の誤解による手戻りコストを最小化
  • 品質向上：体系的なレビューと検証を可能にする

EARSの特徴は、「簡単で実用的」であることです。複雑な記法や専門的なツールを必要とせず、 誰でも学習し適用できる構造化された自然言語アプローチを採用しています。

1.4 EARSによる問題解決アプローチ

EARSは、5つの標準化されたパターン（テンプレート）を使用することで、従来の要求記述の問題を体系的に解決します：

classDiagram class EARS { +Ubiquitous Pattern +Event-driven Pattern +Unwanted Behavior Pattern +State-driven Pattern +Optional Feature Pattern } class Benefits { +明確性の向上 +一貫性の確保 +検証可能性の実現 +レビューの効率化 +トレーサビリティの改善 } EARS --> Benefits : "解決する"

例えば、先ほどの問題のある記述をEARSを使って改善すると以下のようになります：

EARS適用による改善例

改善前（問題のある記述）：

「システムは迅速にユーザーからの要求に応答する必要がある」

改善後（EARS適用）：

「When ユーザーがログイン要求を送信した時、the 認証システム shall 
3秒以内に認証結果をユーザーに返す」

改善点：

  • 具体的な条件（ログイン要求）を明記
  • システムの範囲（認証システム）を特定
  • 測定可能な基準（3秒以内）を設定
  • 構造化されたパターンで一貫性を確保
実習 1-1: 問題のある要求文の識別

以下の要求文について、どのような問題があるかを考えてみましょう。

課題

次の要求文の問題点を分析し、なぜこれらが「良くない要求」なのかを説明してください：

  1. 「システムは使いやすくなければならない」
  2. 「データベースは高速でなければならない」
  3. 「エラーが発生した場合、適切に処理する」
  4. 「重要な情報は安全に保管される」
分析の観点
  • 曖昧性：複数の解釈が可能か？
  • 測定可能性：客観的に評価できるか？
  • 完全性：必要な情報が全て含まれているか？
  • 検証可能性：テストで確認できるか？
考察のヒント

それぞれの文について、「誰が」「いつ」「何を」「どのように」「どの程度」という観点から 不足している情報を特定してみてください。

理解度確認クイズ
  1. EARSが開発された主要な背景として最も適切なのはどれですか？
    1. プログラミング言語の統一のため
    2. 自然言語による要求記述の曖昧性を解決するため
    3. 開発コストを削減するため
    4. 新しい開発手法を確立するため
  2. 従来の要求記述の問題点として含まれないのはどれですか？
    1. 曖昧性（複数の解釈が可能）
    2. 検証困難（テストしにくい）
    3. 複雑性（理解が困難）
    4. 実装困難（技術的に不可能）
  3. EARSの特徴として最も適切な説明はどれですか？
    1. プログラミング言語のような厳密な構文を持つ
    2. 5つの標準化されたパターンを使用する構造化自然言語
    3. 図表を中心とした視覚的な記述方法
    4. 数学的記号を用いた形式的仕様記述法
  4. 良い要求の特性として含まれないのはどれですか？
    1. 明確性（曖昧でない）
    2. 検証可能性（テストできる）
    3. 実装詳細性（技術的詳細を含む）
    4. 一貫性（矛盾がない）

1.5 章のまとめ

重要なポイント
  • 要求工学の重要性：「何を作るか」を明確にすることはシステム開発の基盤
  • 従来手法の限界：自然言語による記述は曖昧性や不一貫性の問題を抱える
  • EARSの位置づけ：実用的で学習しやすい構造化要求記述手法
  • 問題解決アプローチ：5つの標準パターンで体系的に品質を向上
  • 適用価値：安全性、品質、効率性の向上に直接貢献

次の章では、EARSの基盤となる「要求記述の基本原則」について学習し、 良い要求が満たすべき特性（SMART原則）について詳しく理解していきます。
