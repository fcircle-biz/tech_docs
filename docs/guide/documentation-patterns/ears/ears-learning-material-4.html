第4章: Event-driven要求パターン

イベント駆動型要求の記述方法

この章で学ぶこと
  • Event-driven（イベント駆動）要求パターンの基本概念
  • 「When [trigger], the [system] shall [requirement]」構文の使用方法
  • トリガーイベントの識別と明確化の技法
  • ユーザーアクションや外部イベントへの応答要求記述
  • 時間的制約を含む要求の表現方法
  • 実システムでの適用パターン分析

4.1 Event-driven要求パターンの基本概念

Event-driven要求パターンは、特定のイベントやトリガーが発生した時にシステムが実行すべき動作を記述するパターンです。 多くのシステム要求は「何かが起きた時」の反応を定義する必要があり、このパターンは最も頻繁に使用されるEARSパターンの一つです。

sequenceDiagram participant User as ユーザー participant Event as トリガーイベント participant System as システム participant Response as システム応答 User->>Event: アクション実行 Event->>System: イベント発生通知 System->>Response: 要求された動作実行 Response->>User: 結果返却 Note over Event,System: When句で定義 Note over System,Response: shall句で定義
Event-drivenパターンの基本構文
When [trigger event/condition], the [system name] shall [system response]

【構成要素】
- When : 条件・イベントを示すキーワード
- [trigger event/condition] : 発生するイベントや条件
- the [system name] : 対象システム名
- shall : 義務を示す助動詞
- [system response] : システムの応答・動作

4.2 トリガーイベントの種類と分類

Event-drivenパターンで記述するトリガーは、大きく5つのカテゴリに分類できます：

graph TD A["トリガーイベントの種類"] --> B["ユーザーアクション
User Actions"] A --> C["システムイベント
System Events"] A --> D["外部イベント
External Events"] A --> E["時間イベント
Temporal Events"] A --> F["状態変化イベント
State Changes"] B --> B1["クリック・入力・選択"] C --> C1["データ処理完了・エラー発生"] D --> D1["外部システム連携・通信"] E --> E1["時刻到達・期限切れ"] F --> F1["閾値超過・状態遷移"]
1. ユーザーアクション駆動

ユーザーの直接的な操作がトリガーとなる要求：

When ユーザーがログインボタンをクリックした時、
the 認証システム shall ログイン画面を表示する

When 管理者が新規ユーザー作成ボタンを押した時、
the ユーザー管理システム shall ユーザー登録フォームを表示する
2. システムイベント駆動

システム内部の処理完了やエラーがトリガーとなる要求：

When データベース接続が失敗した時、
the アプリケーション shall エラーログを記録し代替接続を試行する

When バックアップ処理が完了した時、
the システム shall 管理者に完了通知メールを送信する

4.3 明確で具体的なトリガー記述の技法

Event-drivenパターンの効果は、トリガー条件の明確性に依存します。曖昧なトリガー記述は、実装時の解釈の相違を生む原因となります。

曖昧なトリガー記述の例と改善
曖昧なトリガー問題点改善されたトリガー
When エラーが発生した時 どのエラー？どの段階で？ When データベース接続がタイムアウトエラー（30秒）で失敗した時
When ユーザーが入力した時 何を？どこに？いつの時点で？ When ユーザーが検索フィールドにキーワードを入力し Enterキーを押した時
When システムが過負荷の時 過負荷の基準は？ When CPU使用率が90%を5分間継続して超えた時
When ファイルが更新された時 どのファイル？どんな更新？ When 設定ファイル（config.xml）が外部から変更された時

4.4 時間制約を含む要求の記述

Event-drivenパターンでは、「いつまでに応答するか」という時間的制約も重要な要素です。リアルタイム性が求められるシステムでは特に重要です。

時間制約を含むEvent-driven要求の例
【応答時間制約】
When ユーザーが商品検索ボタンをクリックした時、
the 検索システム shall 3秒以内に検索結果を表示する

【処理期限制約】  
When 日次バッチ処理が午前2:00に開始された時、
the データ処理システム shall 午前6:00までに全処理を完了する

【リアルタイム制約】
When 火災センサーが煙を検知した時、
the 防火システム shall 200ミリ秒以内にアラームを作動させる
実習 4-1: トリガー条件の明確化

以下の曖昧なトリガー条件を、明確で具体的な表現に改善してください。

改善課題
  1. 「When ユーザーが操作した時」
  2. 「When システムに問題が発生した時」
  3. 「When データが変更された時」
  4. 「When 時間になった時」
改善の観点
  • 主体の特定：誰が、何が？
  • 行為の具体化：どのような操作・変更？
  • 対象の明確化：何に対して？
  • 条件の定量化：測定可能な基準
実習 4-2: ECサイトシステムのEvent-driven要求作成

オンラインショッピングサイトについて、Event-driven要求を作成してください。

システム機能
  • 商品閲覧・検索
  • ショッピングカート機能
  • 注文・決済処理
  • ユーザー登録・ログイン
作成すべき要求

以下の各シナリオについて、Event-driven要求を1つずつ作成してください：

  1. 商品をカートに追加する操作
  2. 決済処理中のエラー処理
  3. 在庫不足商品の注文処理
  4. ログインセッション期限切れ
理解度確認クイズ
  1. Event-drivenパターンの基本構文で正しいのはどれですか？
    1. If [condition], the [system] shall [response]
    2. When [trigger], the [system] shall [response]
    3. While [state], the [system] shall [response]
    4. Where [feature], the [system] shall [response]
  2. 最も明確なトリガー記述はどれですか？
    1. When エラーが発生した時
    2. When ユーザーが操作した時
    3. When ユーザーがログインボタンをクリックした時
    4. When システムが動作した時
  3. 時間制約を含むEvent-driven要求として適切でないのはどれですか？
    1. When ユーザーが検索した時、システムは3秒以内に結果を返す
    2. When バックアップが開始された時、システムは迅速に処理する
    3. When アラームが鳴った時、システムは1秒以内に通知する
    4. When 処理が完了した時、システムは即座にログを記録する

4.5 章のまとめ

重要なポイント
  • 基本構文：「When [trigger], the [system] shall [response]」の正確な使用
  • トリガーの具体化：曖昧性を排除した明確な条件記述
  • イベント分類：ユーザーアクション、システムイベント、外部イベント等の理解
  • 時間制約：応答時間や処理期限の明確な定義
  • 実用性：実際のシステム開発での適用方法

Event-drivenパターンは、システムの動的な振る舞いを記述する重要なパターンです。 次の章では、システムの異常状態や望ましくない動作への対処を記述する「Unwanted Behavior要求パターン」について学習します。
