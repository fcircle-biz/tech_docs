8.1 EARSパターンの組み合わせ戦略

実際のシステム開発では、単一のEARSパターンだけでなく、複数のパターンを適切に組み合わせて システム全体の複雑な要求を表現する必要があります。各パターンの特性を理解し、 効果的に組み合わせることが重要です。

graph TD A["システム要求全体"] --> B["基盤要求
(Ubiquitous)"] A --> C["機能要求
(Event-driven)"] A --> D["信頼性要求
(Unwanted Behavior)"] A --> E["状態管理要求
(State-driven)"] A --> F["条件付き要求
(Optional Feature)"] B --> B1["システムの基本特性"] C --> C1["ユーザーインタラクション"] D --> D1["例外・エラー処理"] E --> E1["動的な状態制御"] F --> F1["設定・構成依存機能"] style A fill:#e1f5fe,stroke:#0277bd,stroke-width:3px
パターン組み合わせの基本原則

1. 階層的構造での組み合わせ

【基盤レベル】Ubiquitous: システム全体の基本特性
【機能レベル】Event-driven: 通常機能の動作定義
【制御レベル】State-driven: 状態による動作制御
【保護レベル】Unwanted Behavior: 異常時の保護動作
【拡張レベル】Optional Feature: 条件付き追加機能

2. 相互補完的な組み合わせ

【通常処理】Event-driven: When ログイン成功時...
【状態管理】State-driven: While ログイン状態の間...
【異常処理】Unwanted Behavior: If 認証失敗の場合...
【機能拡張】Optional Feature: Where 二段階認証が有効な場合...

8.2 実プロジェクトでのEARS適用ワークフロー

EARSを実際のプロジェクトで効果的に活用するためには、体系的なワークフローが必要です。 以下は実践的なEARS適用手順です：

graph TD A["要求収集
Requirements Gathering"] --> B["要求分析
Requirements Analysis"] B --> C["パターン選択
Pattern Selection"] C --> D["EARS記述
EARS Documentation"] D --> E["レビュー・検証
Review & Validation"] E --> F["要求仕様書完成
Final Specification"] B --> B1["ステークホルダー要求整理"] C --> C1["各要求に適したパターン判断"] D --> D1["構造化された要求記述"] E --> E1["一貫性・完全性チェック"] F --> F1["実装可能な仕様書"]
ワークフロー各段階での実践ポイント

段階1: 要求収集

  • ステークホルダーインタビュー
  • 既存システム分析
  • 業務フロー調査
  • 制約条件の把握

段階2: 要求分析

  • 機能要求と非機能要求の分離
  • 優先度付けと依存関係整理
  • トリガー・条件・状態の識別
  • 例外・エラーシナリオの洗い出し

段階3: パターン選択

  • 各要求の性質に応じたパターン判断
  • 複合要求の分解と整理
  • パターン間の整合性確認
  • 要求の優先度とパターンの適合性評価

8.3 総合事例研究：病院情報システム

複雑な実システムでのEARS適用例として、病院情報システム（HIS: Hospital Information System）の要求定義を EARSパターンを使って体系的に記述してみましょう。

プロジェクト概要

システム名: 総合病院情報管理システム

主要機能:

  • 患者情報管理（電子カルテ）
  • 診療予約・スケジュール管理
  • 医薬品在庫管理
  • 診療費計算・会計処理
  • 医療スタッフ管理
  • セキュリティ・アクセス制御

特別要件:

  • 24時間365日運用
  • 医療法・個人情報保護法準拠
  • 高可用性・データ保護
  • 多職種スタッフの同時アクセス
病院情報システムのEARS要求例

【Ubiquitous - 基盤要求】

The 病院情報システム shall 患者の個人情報をAES-256暗号化で保護する

The データベースシステム shall 99.9%以上の可用性を維持する

The システム shall 全ての医療データアクセスを監査ログに記録する

【Event-driven - 機能要求】

When 医師が患者IDを入力して検索ボタンをクリックした時、
the 電子カルテシステム shall 3秒以内に患者の診療履歴を表示する

When 薬剤師が処方せんの調剤完了ボタンを押した時、
the 薬剤管理システム shall 在庫数量を自動更新し患者への通知を送信する

【Unwanted Behavior - 例外処理】

If 不正な認証情報で3回連続ログイン失敗が発生した場合、
then the セキュリティシステム shall アカウントを30分間ロックし管理者に警告する

If システム障害でメインサーバーが応答しない場合、
then the システム shall 自動的にバックアップサーバーに切り替える

【State-driven - 状態管理】

While 患者が診察中状態の間、
the 予約システム shall 該当患者の新規予約受付を一時停止する

While システムがメンテナンス中の間、
the 緊急用システム shall 最小限の患者情報アクセス機能を提供する

【Optional Feature - 条件付き機能】

Where 放射線科システム連携オプションが有効な場合、
the 電子カルテシステム shall 医用画像の直接参照機能を提供する

Where 多言語サポートが有効化されている場合、
the 患者向け画面 shall 日本語・英語・中国語での表示機能を提供する

8.4 要求品質評価とレビュー手法

EARS要求の品質を確保するためには、体系的なレビューと評価が必要です。 以下のチェックリストを使用して要求品質を評価しましょう：

EARS要求品質チェックリスト

1. 構文チェック

  • ✓ 適切なEARSパターンを使用しているか
  • ✓ 構文が正確に記述されているか
  • ✓ システム名が明確に特定されているか
  • ✓ "shall"が適切に使用されているか

2. 内容チェック

  • ✓ 要求が具体的で測定可能か
  • ✓ 曖昧な表現がないか
  • ✓ 前提条件が明確か
  • ✓ 期待する結果が明示されているか

3. 一貫性チェック

  • ✓ 他の要求と矛盾していないか
  • ✓ 用語の使用が統一されているか
  • ✓ システム名の表記が一貫しているか
  • ✓ パターン選択が適切か
実習 8-1: オンライン学習システムの総合要求設計

大学向けオンライン学習システムについて、全EARSパターンを使用した包括的な要求仕様を作成してください。

システム要件
  • 基本機能: 講義配信、課題提出、成績管理、ディスカッション
  • ユーザー種別: 学生、教員、管理者
  • 技術要件: 同時1000人アクセス、24時間稼働
  • オプション: 録画機能、AI採点、多言語対応
作成課題

各EARSパターンで5つずつ、合計25の要求を作成してください：

  1. Ubiquitous (5要求): システムの基本特性・制約
  2. Event-driven (5要求): ユーザーアクションに対する応答
  3. Unwanted Behavior (5要求): エラー・例外処理
  4. State-driven (5要求): 状態依存の動作
  5. Optional Feature (5要求): 条件付き・拡張機能
品質基準
  • 各要求は測定可能で検証可能であること
  • システム名を適切に特定すること
  • 要求間に矛盾がないこと
  • 実装可能で現実的であること
実習 8-2: 既存要求文書のEARS変換

以下の従来形式で書かれた要求文をEARS形式に変換してください。

変換対象の要求文
  1. 「システムは高いセキュリティを維持すること」
  2. 「ユーザーが商品を購入する際、在庫確認を行い、在庫不足の場合はエラーメッセージを表示する」
  3. 「管理者モードでは、全てのユーザーデータにアクセス可能とする」
  4. 「ログ出力機能が有効な場合、全ての操作を記録する」
  5. 「データベース接続エラーが発生した場合、適切に処理すること」
変換手順
  1. 要求分析: 各要求の性質とパターンを判断
  2. 情報補完: 不足している情報を特定・補完
  3. EARS記述: 適切なパターンで構造化
  4. 品質確認: チェックリストで検証
総合理解度確認テスト
  1. EARSパターンの組み合わせで最も適切な階層構造はどれですか？
    1. Event-driven → Ubiquitous → State-driven
    2. Ubiquitous → Event-driven → Unwanted Behavior
    3. Optional Feature → State-driven → Event-driven
    4. State-driven → Optional Feature → Ubiquitous
  2. 実プロジェクトでのEARS適用順序として最も適切なのはどれですか？
    1. EARS記述 → 要求収集 → パターン選択
    2. 要求収集 → EARS記述 → 要求分析
    3. 要求収集 → 要求分析 → パターン選択 → EARS記述
    4. パターン選択 → 要求収集 → EARS記述
  3. EARS要求の品質評価で最も重要でない項目はどれですか？
    1. 構文の正確性
    2. 測定可能性
    3. 文章の長さ
    4. 要求間の一貫性
  4. 複雑なシステム要求を記述する際の最良のアプローチはどれですか？
    1. 最も複雑なパターン1つで全てを記述
    2. 複数のパターンを適切に組み合わせて記述
    3. 最もシンプルなパターンのみ使用
    4. 従来の自然言語記述と併用
  5. EARSを組織に導入する際の最初のステップとして最も適切なのはどれですか？
    1. 全要求文書をEARS形式で書き直す
    2. チームメンバーへのEARS教育・訓練
    3. 新しい要求管理ツールを導入する
    4. 顧客にEARSの使用を説明する

8.5 EARSの継続的な活用と発展

EARSを習得したあとは、継続的な活用と改善が重要です。以下の点を意識して、 チームや組織でのEARS活用を発展させていきましょう：

継続活用のための実践ガイドライン

1. チーム内での標準化

  • EARSテンプレートとガイドラインの整備
  • 用語集・システム名規則の統一
  • レビュープロセスの確立
  • 品質基準の設定と評価方法

2. ツールとの連携

  • 要求管理ツールでのEARS記述
  • トレーサビリティ管理の強化
  • 自動化可能な品質チェック
  • 変更管理との統合

3. 継続改善

  • プロジェクト振り返りでのEARS効果測定
  • 要求品質向上の定量的評価
  • チームメンバーのスキル向上支援
  • ベストプラクティスの蓄積・共有

8.6 学習の総まとめと次のステップ

EARS学習の達成成果

この学習コースを通じて、以下のスキルを身につけることができました：

  • EARS基礎理解: 要求工学の課題とEARSによる解決アプローチ
  • 5パターン習得: 各EARSパターンの適切な使い分けと記述技法
  • 品質向上技法: SMART原則に基づく明確で検証可能な要求記述
  • 実践的適用: 実際のシステム開発での効果的なEARS活用方法
  • 総合的設計: 複合パターンによる包括的な要求仕様書作成
推奨される次のステップ

初級から中級への発展:

  • 実際のプロジェクトでのEARS適用実践
  • 要求管理ツールとの連携スキル習得
  • ステークホルダーとのEARS要求レビュー経験

中級から上級への発展:

  • 大規模システムでの要求アーキテクチャ設計
  • EARSとアジャイル開発手法の統合
  • 組織でのEARS導入・推進リーダーシップ

関連分野の学習:

  • SysML（Systems Modeling Language）によるシステムモデリング
  • 形式的仕様記述法（Formal Specification）
  • 要求ベーステスト設計（Requirements-Based Testing）

EARSは単なる記述技法ではなく、システム開発の品質と効率を根本的に改善するアプローチです。 継続的な実践を通じて、より優れた要求エンジニアとしてスキルを発展させていきましょう。
