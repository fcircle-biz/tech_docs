クラウドコンピューティングとは何か

この章で学ぶこと

  • クラウドとオンプレミスの違いと、それぞれのメリット・デメリットを説明できる
  • クラウドの5つの基本特性（NIST定義）を把握する
  • サービスモデル（IaaS、PaaS、SaaS）の違いと具体例を理解する
  • デプロイメントモデル（パブリック、プライベート、ハイブリッド、マルチクラウド）を区別できる

1.1 クラウドコンピューティングの定義と歴史

クラウドコンピューティングとは

クラウドコンピューティング（Cloud Computing）とは、インターネット経由でコンピューティングリソース（サーバー、ストレージ、データベース、ネットワーク、ソフトウェアなど）を必要なときに必要なだけ利用できるサービス形態のことです。

従来のIT環境では、企業が自社でサーバーやネットワーク機器を購入し、物理的なデータセンターに設置・運用する「オンプレミス」形態が主流でした。しかし、クラウドでは、これらのリソースをクラウドプロバイダーが提供し、ユーザーはインターネット経由で必要に応じてアクセスします。

「クラウド」という名称の由来

「クラウド」という名称は、ネットワーク図でインターネットを雲のアイコンで表現していたことに由来します。ユーザーからは、実際のサーバーがどこにあるのか、どのような構成になっているのかは見えず、「雲の中」にあるように見えることから、この名称が定着しました。

クラウドコンピューティングの歴史

クラウドコンピューティングの発展は、以下のような段階を経て進化してきました：

  • 1960年代: 大型コンピュータのタイムシェアリングシステムで、複数ユーザーがリソースを共有する概念が生まれる
  • 1990年代: ASP（Application Service Provider）やデータセンターサービスが登場
  • 2000年代初頭: 仮想化技術の発展により、物理サーバー上で複数の仮想サーバーを動かすことが可能に
  • 2006年: Amazon Web Services（AWS）がEC2（Elastic Compute Cloud）を提供開始。現代的なクラウドコンピューティングの始まり
  • 2008年: Google App Engine、Microsoft Azureが登場
  • 2010年代: クラウドサービスが企業に広く普及し、様々なクラウドサービスが登場
  • 現在: マルチクラウド、ハイブリッドクラウド、クラウドネイティブアプリケーションが主流に

1.2 クラウドの5つの基本特性（NIST定義）

米国国立標準技術研究所（NIST）は、クラウドコンピューティングを定義する5つの基本特性を提示しています。これらの特性を理解することで、クラウドサービスの本質を把握できます。

特性 説明 具体例
1. オンデマンド・セルフサービス
（On-demand self-service）		 ユーザーが必要なときに、人手を介さずに自動的にリソースを利用できる AWSコンソールから数クリックでサーバーを起動できる
2. 広範なネットワークアクセス
（Broad network access）		 インターネット経由で、様々なデバイスからアクセス可能 PC、スマートフォン、タブレットからクラウドサービスを利用できる
3. リソースプーリング
（Resource pooling）		 プロバイダーのリソースがプールされ、複数の顧客で共有される（マルチテナント） 物理サーバーが複数の顧客の仮想サーバーをホストする
4. 迅速な拡張性
（Rapid elasticity）		 需要に応じて迅速かつ柔軟にリソースを増減できる アクセス増加時に自動的にサーバー台数を増やす（Auto Scaling）
5. 測定可能なサービス
（Measured service）		 リソース使用量が測定され、利用した分だけ課金される 使用したCPU時間、ストレージ容量、通信量に応じて課金される
なぜこの5つの特性が重要なのか

これらの特性により、従来のオンプレミス環境では実現が難しかった「柔軟性」「俊敏性」「コスト効率」が実現できるようになりました。例えば、事業が急成長してアクセスが10倍になっても、オンプレミスでは新しいサーバーの購入・設置に数週間から数ヶ月かかりますが、クラウドでは数分でリソースを10倍に拡張できます。

1.3 クラウドサービスモデル：IaaS、PaaS、SaaS

クラウドサービスは、提供されるサービスのレベルに応じて3つの主要なモデルに分類されます。これらのモデルは、ユーザーが管理する範囲とクラウドプロバイダーが管理する範囲の違いを表します。

IaaS（Infrastructure as a Service）- サービスとしてのインフラ

IaaSは、仮想サーバー、ストレージ、ネットワークなどの基本的なITインフラをサービスとして提供します。ユーザーはOS以上のレイヤーを自由に管理・設定できます。

  • 提供されるもの: 仮想サーバー、ストレージ、ネットワーク、ロードバランサーなど
  • ユーザーが管理するもの: OS、ミドルウェア、アプリケーション、データ
  • 具体例:
    • Amazon EC2（AWS）
    • Azure Virtual Machines（Microsoft Azure）
    • Google Compute Engine（GCP）
  • 適している用途: 既存システムのクラウド移行、カスタマイズ性が高い環境が必要な場合

PaaS（Platform as a Service）- サービスとしてのプラットフォーム

PaaSは、アプリケーション実行環境（プラットフォーム）をサービスとして提供します。OS、ミドルウェア、ランタイムなどはプロバイダーが管理し、ユーザーはアプリケーションコードとデータの管理に集中できます。

  • 提供されるもの: アプリケーション実行環境、開発ツール、データベース、OSなど
  • ユーザーが管理するもの: アプリケーションコード、データ
  • 具体例:
    • AWS Elastic Beanstalk（AWS）
    • Azure App Service（Microsoft Azure）
    • Google App Engine（GCP）
    • Heroku
  • 適している用途: アプリケーション開発に集中したい場合、インフラ管理を最小化したい場合

SaaS（Software as a Service）- サービスとしてのソフトウェア

SaaSは、アプリケーション全体をサービスとして提供します。ユーザーはWebブラウザやアプリから利用するだけで、インフラやソフトウェアの管理は不要です。

  • 提供されるもの: 完全なアプリケーション、データ管理機能
  • ユーザーが管理するもの: 設定とユーザーデータのみ
  • 具体例:
    • Gmail、Google Workspace（Google）
    • Microsoft 365（Microsoft）
    • Salesforce（CRM）
    • Slack（コミュニケーション）
    • Dropbox（ファイル共有）
  • 適している用途: すぐに使えるアプリケーションが必要な場合、IT管理を完全に委託したい場合
どのサービスモデルを選ぶべきか

選択は「どこまで自分で管理したいか」「どれだけカスタマイズが必要か」によります：

  • 高いカスタマイズ性が必要、既存システムの移行 → IaaS
  • アプリ開発に集中、インフラ管理は最小限 → PaaS
  • すぐに使える既製のアプリケーション → SaaS

1.4 クラウドデプロイメントモデル

クラウドは、その配置形態によって4つのデプロイメントモデルに分類されます。これらは「誰がクラウドを利用するか」「どこに配置されるか」という観点での分類です。

1. パブリッククラウド（Public Cloud）

一般の企業や個人が利用できる、クラウドプロバイダーが提供する公開されたクラウド環境です。最も一般的なクラウド形態で、AWS、Azure、GCPなどがこれに該当します。

  • 特徴:
    • 複数の顧客がリソースを共有（マルチテナント）
    • 初期投資が不要で、従量課金制
    • 高い拡張性と可用性
    • プロバイダーが全てのインフラを管理
  • メリット: コスト効率が高い、運用負荷が低い、最新技術を利用できる
  • デメリット: セキュリティ・コンプライアンスの懸念、カスタマイズ制限
  • 適している用途: Webアプリケーション、開発・テスト環境、スタートアップ

2. プライベートクラウド（Private Cloud）

特定の組織専用に構築・運用されるクラウド環境です。自社データセンターに構築する場合と、プロバイダーが専用環境を提供する場合があります。

  • 特徴:
    • 単一の組織が専有（シングルテナント）
    • 高いセキュリティとカスタマイズ性
    • オンプレミスまたは専用クラウド環境
    • 既存システムとの統合が容易
  • メリット: セキュリティ・コンプライアンス要件に対応、完全なコントロール
  • デメリット: 初期投資・運用コストが高い、拡張性に制限
  • 適している用途: 金融機関、医療機関、機密性の高いデータを扱う企業

3. ハイブリッドクラウド（Hybrid Cloud）

パブリッククラウドとプライベートクラウド（またはオンプレミス）を組み合わせて利用する形態です。データやワークロードを適切な環境に配置できます。

  • 特徴:
    • パブリックとプライベートが連携
    • データやアプリケーションの配置を柔軟に選択
    • セキュリティとコストのバランスを最適化
  • メリット: 柔軟性が高い、段階的なクラウド移行が可能、コスト最適化
  • デメリット: 管理が複雑、両環境の統合に技術的課題
  • 適している用途: 機密データはプライベート、開発環境はパブリックで運用
  • 具体例: 顧客データはオンプレミスで管理、Webサーバーはパブリッククラウドで運用

4. マルチクラウド（Multi Cloud）

複数のパブリッククラウドプロバイダーのサービスを組み合わせて利用する形態です。例えば、AWSとAzureの両方を使用するケースなど。

  • 特徴:
    • 複数のクラウドプロバイダーを併用
    • 各プロバイダーの強みを活用
    • ベンダーロックインを回避
  • メリット: ベンダー依存リスクの低減、各社の最適なサービスを選択可能
  • デメリット: 管理が複雑、各クラウドの専門知識が必要、コスト管理が難しい
  • 適している用途: グローバル展開、特定サービスの使い分け
  • 具体例: メインシステムはAWS、AI機能はGCP、社内システムはAzure

1.5 オンプレミスとクラウドの比較

従来のオンプレミス環境とクラウド環境の違いを理解することは、クラウド導入判断において重要です。

比較項目 オンプレミス クラウド
初期投資 高額（ハードウェア購入、設置、ライセンス） 低額〜ゼロ（従量課金のみ）
導入スピード 遅い（数週間〜数ヶ月） 速い（数分〜数時間）
拡張性 制限あり（物理的な追加が必要） 高い（瞬時にスケール可能）
運用負荷 高い（自社で全て管理） 低い（プロバイダーが基盤を管理）
セキュリティ 完全にコントロール可能 プロバイダーに依存（ただし高水準）
カスタマイズ性 非常に高い サービスによって制限あり
コストモデル 資本支出（CapEx）- 最初に大きな投資 運用支出（OpEx）- 使った分だけ支払い
災害対策 自社で構築が必要（コスト高） 容易に実装可能（マルチリージョン）
技術の最新性 自社で更新が必要 常に最新技術が利用可能
注意：クラウドが常に最適とは限らない

クラウドには多くのメリットがありますが、すべてのケースで最適というわけではありません。以下のような場合はオンプレミスやハイブリッドが適している可能性があります：

  • 法規制により、データを特定の国・地域に保管する必要がある場合
  • 常に高負荷で稼働するシステムで、長期的にはオンプレミスのほうが安価な場合
  • 既存のレガシーシステムとの統合が複雑な場合
  • ネットワーク依存を避けたい重要システム

1.6 主要クラウドプロバイダーの概要

現在、世界には多数のクラウドプロバイダーが存在しますが、特に「3大クラウドプロバイダー」と呼ばれるAWS、Azure、GCPが市場の大部分を占めています。

Amazon Web Services（AWS）

  • 提供企業: Amazon
  • 市場シェア: 約32%（2024年時点、世界最大）
  • 特徴:
    • 最も歴史が古く、サービス数が最も多い（200以上）
    • 最も広範なグローバル展開（リージョン・データセンター数が最多）
    • 豊富なドキュメントとコミュニティ
    • スタートアップから大企業まで幅広い採用実績
  • 代表的なサービス: EC2（仮想サーバー）、S3（ストレージ）、RDS（データベース）、Lambda（サーバーレス）

Microsoft Azure

  • 提供企業: Microsoft
  • 市場シェア: 約23%（2024年時点、第2位）
  • 特徴:
    • Microsoftエコシステム（Windows Server、Active Directory、Microsoft 365）との親和性が高い
    • エンタープライズ企業に強い
    • ハイブリッドクラウドソリューションが充実
    • .NET開発に最適
  • 代表的なサービス: Virtual Machines、Blob Storage、Azure SQL Database、Azure Functions

Google Cloud Platform（GCP）

  • 提供企業: Google
  • 市場シェア: 約11%（2024年時点、第3位）
  • 特徴:
    • データ分析・機械学習分野に強み（BigQuery、Vertex AI）
    • Kubernetesの開発元であり、コンテナ技術に強い
    • Googleの検索・YouTube等で使われる技術を提供
    • 高速なネットワークインフラ
  • 代表的なサービス: Compute Engine、Cloud Storage、Cloud SQL、Cloud Functions

その他のクラウドプロバイダー

  • Alibaba Cloud: 中国最大のクラウドプロバイダー、アジア市場に強い
  • Oracle Cloud: Oracleデータベースとの統合に強み
  • IBM Cloud: エンタープライズ向け、ハイブリッドクラウドソリューション
  • 国内クラウド: さくらのクラウド、ニフクラ、GMOクラウド等

1.7 クラウドネイティブの概念

クラウドネイティブ（Cloud Native）とは、クラウドの特性を最大限活用するように設計・構築されたアプリケーションやアーキテクチャの考え方です。単にアプリケーションをクラウド上で動かすだけでなく、クラウドの利点（拡張性、回復性、柔軟性）を活かした設計を指します。

クラウドネイティブの主要な要素

  • マイクロサービスアーキテクチャ: アプリケーションを小さな独立したサービスに分割
  • コンテナ化: Docker等でアプリケーションを軽量なコンテナにパッケージング
  • 動的オーケストレーション: Kubernetes等でコンテナを自動管理
  • 継続的インテグレーション/デリバリー（CI/CD）: 自動化されたビルド・テスト・デプロイ
  • DevOps文化: 開発と運用の統合、自動化
flowchart TD A[クラウドネイティブアプリケーション] --> B[マイクロサービス] A --> C[コンテナ化] A --> D[DevOps] A --> E[継続的デリバリー] A --> F[動的オーケストレーション] B --> G[スケーラビリティ] C --> G D --> H[俊敏性] E --> H F --> I[自動化]

従来のアプリケーション vs クラウドネイティブアプリケーション

観点 従来型（モノリシック） クラウドネイティブ
アーキテクチャ 単一の大きなアプリケーション 小さな独立したマイクロサービス
デプロイ 物理/仮想サーバーに直接デプロイ コンテナでパッケージングしてデプロイ
スケーリング 垂直スケーリング（サーバーのスペックアップ） 水平スケーリング（サーバー台数を増やす）
障害対応 アプリ全体が停止 一部サービスのみ影響、自動復旧
リリースサイクル 数ヶ月〜年単位 日次〜週次（継続的デリバリー）
実習 1-1: クラウドサービスの分類演習

以下のサービスをIaaS、PaaS、SaaSに分類してください。また、どのデプロイメントモデル（パブリック、プライベート、ハイブリッド、マルチ）が適しているか考えてみましょう。

分類するサービス
  1. Amazon EC2（仮想サーバー）
  2. Salesforce（CRMソフトウェア）
  3. Google App Engine（アプリケーション実行環境）
  4. Microsoft 365（Officeスイート）
  5. Azure Virtual Machines
  6. Heroku（アプリケーションプラットフォーム）
解答例
  • IaaS: Amazon EC2、Azure Virtual Machines → ユーザーがOS以上を管理
  • PaaS: Google App Engine、Heroku → アプリケーション実行環境が提供される
  • SaaS: Salesforce、Microsoft 365 → 完全なアプリケーションとして提供
考察ポイント

これらのサービスは基本的にパブリッククラウドとして提供されていますが、企業によっては：

  • 機密データを扱う場合はプライベートクラウドで運用
  • 既存オンプレミスシステムと連携する場合はハイブリッドクラウド
  • 複数のプロバイダーを使い分ける場合はマルチクラウド

というように、用途に応じてデプロイメントモデルを選択します。

理解度確認クイズ

  2. IaaS、PaaS、SaaSの違いを、「ユーザーが管理する範囲」の観点から説明してください。
  3. パブリッククラウドとプライベートクラウドの主な違いは何ですか？それぞれのメリット・デメリットを挙げてください。
  4. ハイブリッドクラウドとマルチクラウドの違いを説明してください。
  5. オンプレミス環境と比較した場合のクラウドの主なメリットを3つ挙げてください。
  6. クラウドネイティブアプリケーションの特徴を3つ挙げてください。
  7. AWS、Azure、GCPの主な特徴をそれぞれ1つずつ挙げてください。
  8. 「リソースプーリング」とは何を意味しますか？具体例を挙げて説明してください。
解答のポイント
クリックして解答例を表示
  1. 5つの基本特性:
    • オンデマンド・セルフサービス: 必要時に自動でリソース取得
    • 広範なネットワークアクセス: 様々なデバイスからアクセス可能
    • リソースプーリング: 複数顧客でリソースを共有
    • 迅速な拡張性: 需要に応じて素早くスケール
    • 測定可能なサービス: 使用量を測定し従量課金
  2. サービスモデルの違い:
    • IaaS: OS以上をユーザーが管理（例：EC2）
    • PaaS: アプリケーションとデータのみユーザーが管理（例：App Engine）
    • SaaS: ユーザーは設定のみ、全てプロバイダーが管理（例：Gmail）
  3. パブリック vs プライベート:
    • パブリック: 複数顧客で共有、低コスト、拡張性高い ← → セキュリティ懸念
    • プライベート: 専有、高セキュリティ、カスタマイズ可 ← → 高コスト、拡張性制限

1.8 本章のまとめ

重要ポイント
  • クラウドコンピューティングは、インターネット経由でITリソースを必要な分だけ利用できるサービス形態
  • NIST定義の5つの基本特性（オンデマンド、広範アクセス、リソースプーリング、拡張性、測定可能性）がクラウドの本質
  • サービスモデル（IaaS、PaaS、SaaS）は、管理範囲の違いで分類される
  • デプロイメントモデル（パブリック、プライベート、ハイブリッド、マルチ）は、配置形態の違いで分類される
  • AWS、Azure、GCPが3大クラウドプロバイダーで、それぞれに特徴がある
  • クラウドネイティブは、クラウドの特性を最大限活用する設計思想
  • オンプレミスとクラウドは一長一短があり、用途に応じた選択が重要
