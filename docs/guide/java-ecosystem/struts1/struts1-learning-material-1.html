🏛️ Webアプリケーション開発の歴史的背景

まず、なぜStruts 1.xというフレームワークが生まれたのか、その歴史的な背景から理解しましょう。

1990年代後半〜2000年代初頭のWeb開発の課題

1990年代後半、インターネットの普及とともにWebアプリケーションへの需要が急激に高まりました。この時代のWeb開発には以下のような深刻な問題がありました：

  • スパゲッティコード問題：JSPの中にJavaコードを直接記述し、HTMLとロジックが混在した保守困難なコードが蔓延
  • 責任の境界が不明確：デザイナーとプログラマーが同じファイルを編集する必要があり、作業効率が悪い
  • 再利用性の低さ：類似した処理を何度も書く必要があり、開発効率が悪い
  • テストの困難さ：ビジネスロジックがJSPに埋め込まれ、単体テストが困難

1.1 フレームワークとは何か？基本概念の理解

🔍 フレームワークの本質的理解

フレームワークとは、「アプリケーション開発のための骨組み」です。家を建てる際の基礎工事や骨組みのようなものと考えてください。

フレームワークが提供する価値：
  1. 標準化：開発者が共通のルールとパターンに従って開発できる
  2. 効率化：よくある処理は既に実装されているため、ビジネスロジックに集中できる
  3. 品質向上：実証済みの設計パターンを使用することで、バグを減らし、保守しやすいコードを書ける
  4. 分業の促進：役割分担が明確になり、チーム開発が効率的になる

重要な概念：フレームワークは「制約と引き換えに自由を得る」仕組みです。一定のルールに従うことで、開発の複雑さから解放されます。

1.2 Apache Struts 1.xとは？その誕生と特徴

Apache Struts 1.xは、2000年に Craig R. McClanahan によって開発された、Javaベースのオープンソース Webアプリケーションフレームワークです。

🎯 Struts 1.x の設計思想と目的

Struts 1.xは、以下の明確な目的を持って設計されました：

  1. 関心の分離（Separation of Concerns）
    • プレゼンテーション層（表示）
    • ビジネスロジック層（処理）
    • データアクセス層（データ）
    • これらを明確に分離し、それぞれの責任を独立させる
  2. MVCアーキテクチャパターンの実装
    • Model-View-Controller パターンを Java Web開発に適用
    • 各層の責任を明確化し、保守性を向上
  3. 設定による制御（Configuration over Convention）
    • XMLファイルによる宣言的な設定
    • コードを変更することなく動作を制御
⚠️ 重要な注意事項

Struts 1.x は2013年にEOL（End of Life）に達しており、新規開発には推奨されません。しかし、以下の理由で学習価値があります：

  • レガシーシステムの保守・移行作業で必要となることがある
  • MVCアーキテクチャの理解を深めるための教材として優秀
  • 現代的なフレームワークの基礎となった概念を学べる
  • Webアプリケーション開発の歴史的変遷を理解できる

1.3 MVCアーキテクチャの深い理解

MVCアーキテクチャは、「アプリケーションの責任を3つの層に分離する設計パターン」です。Strutsの理解には、このMVCの本質的理解が不可欠です。

🔄 MVCの各層の役割と責任
Model（モデル）- データとビジネスロジック
  • 役割：アプリケーションの核となるデータとビジネスロジックを管理
  • 責任：データの整合性、業務ルールの実装、データの永続化
  • 具体例：顧客情報の管理、注文処理ロジック、在庫計算など
  • 重要な点：ViewやControllerに依存しない、独立した存在
View（ビュー）- 表示とユーザーインターフェース
  • 役割：ユーザーに情報を表示し、入力を受け取る
  • 責任：HTMLの生成、フォームの表示、エラーメッセージの表示
  • 具体例：商品一覧画面、ログインフォーム、確認画面など
  • 重要な点：ビジネスロジックを含まない、純粋な表示層
Controller（コントローラー）- 処理の制御と調整
  • 役割：ユーザーからの要求を受け取り、適切な処理を実行
  • 責任：リクエストの解析、Modelの呼び出し、Viewの選択
  • 具体例：ログイン処理、商品検索処理、注文確定処理など
  • 重要な点：ModelとViewの仲介役、処理の流れを制御

📈 MVC採用のメリットと価値
1. 保守性の向上

例：商品の価格計算ロジックを変更する場合

  • 従来方式：複数のJSPファイルを修正し、HTML部分も影響を受ける可能性
  • MVC方式：Modelの価格計算部分のみを修正すれば済む
2. 分業の促進
  • デザイナー：ViewのHTMLとCSS部分に集中
  • プログラマー：ControllerとModelのロジック部分に集中
  • データベース担当：Modelのデータアクセス部分に集中
3. テスタビリティの向上
  • ビジネスロジックが独立しているため、単体テストが容易
  • 各層を個別にテストできるため、品質向上が期待できる
4. 再利用性の向上
  • 同じModelを複数のViewで使用可能
  • 同じViewを複数のControllerで使用可能

🔧 Struts 1.x の主要コンポーネント
1. ActionServlet（フロントコントローラー）
  • 役割：すべてのHTTPリクエストの入り口となる中央制御装置
  • 機能：リクエストを解析し、適切なActionクラスに処理を委譲
  • 比喩：会社の受付のような役割。来訪者を適切な部署に案内する
2. Actionクラス（コントローラー）
  • 役割：具体的なビジネス処理を実行するクラス
  • 機能：フォームデータの検証、ビジネスロジックの呼び出し、結果の返却
  • 比喩：各部署の担当者のような役割。専門的な処理を実行する
3. ActionForm（データ転送オブジェクト）
  • 役割：HTMLフォームのデータを Java オブジェクトとして管理
  • 機能：データの受け渡し、基本的な入力検証
  • 比喩：申請書のような役割。情報を構造化して保持する
4. JSP（ビュー）
  • 役割：ユーザーに表示するHTMLを動的に生成
  • 機能：データの表示、フォームの生成、エラーメッセージの表示
  • 比喩：店舗の店頭のような役割。商品（データ）を魅力的に展示する

1.6 Struts 1.x の処理フロー理解

Struts 1.xでWebリクエストがどのように処理されるのか、その流れを詳しく見てみましょう。

sequenceDiagram participant U as 👤 ユーザー participant B as 🌐 ブラウザ participant AS as ⚙️ ActionServlet participant A as 🎯 Action participant AF as 📋 ActionForm participant BL as 💼 Business Logic participant JSP as 📄 JSP U->>B: フォーム送信 B->>AS: HTTPリクエスト AS->>AF: フォームデータ設定 AS->>A: execute()メソッド呼び出し A->>AF: データ取得・検証 A->>BL: ビジネスロジック実行 BL-->>A: 処理結果 A-->>AS: ActionForward返却 AS->>JSP: 結果画面表示 JSP-->>B: HTML応答 B-->>U: 画面表示
🔄 処理フロー詳細解説
ステップ1：リクエスト受信

ユーザーがブラウザでフォームを送信すると、ActionServletが最初にリクエストを受け取ります。これは「フロントコントローラーパターン」と呼ばれる設計です。

ステップ2：ルーティング決定

ActionServletは、struts-config.xmlの設定を参照して、どのActionクラスが処理を担当するかを決定します。

ステップ3：データバインディング

HTMLフォームから送信されたデータを、ActionFormオブジェクトに自動的に設定します。これにより、型安全なデータアクセスが可能になります。

ステップ4：ビジネスロジック実行

Actionクラスがビジネスロジックを実行し、必要に応じてデータベースアクセスや他のシステムとの連携を行います。

ステップ5：結果画面選択

処理の結果に応じて、表示するJSPページを選択し、ActionForwardオブジェクトとして返却します。

ステップ6：レスポンス生成

選択されたJSPページが実行され、最終的なHTMLがブラウザに送信されます。

1.8 現代における Struts 1.x の位置づけ

📊 Struts 1.x の現状と学習意義
現状の課題：
  • 開発終了：2013年にEOLとなり、セキュリティ更新が停止
  • 古い設計思想：現代的な開発手法（注釈ベース設定、RESTful API等）に対応していない
  • 冗長な設定：XMLベースの設定が複雑で、開発効率が悪い
それでも学習する価値：
  1. レガシーシステムの理解：多くの企業システムで稼働中
  2. MVCアーキテクチャの理解：Webアプリケーション設計の基礎概念
  3. フレームワーク進化の理解：現代フレームワークとの比較による学習
  4. 設計思想の習得：関心の分離、設定による制御等の重要概念
実習 1-1: MVCアーキテクチャの理解

この実習では、理論的な理解を深めるため、コードを書く前にMVCの概念を整理します。

課題：オンラインショッピングサイトのMVC設計

「商品購入」機能について、MVC各層の役割を具体的に設計してください。

手順
  1. View（表示層）の責任を定義
    • どのような画面が必要か？
    • それぞれの画面で表示すべき情報は何か？
    • ユーザーからの入力項目は何か？
  2. Controller（制御層）の責任を定義
    • どのような処理が必要か？
    • どのような判定・分岐が必要か？
    • エラー処理はどう行うか？
  3. Model（データ層）の責任を定義
    • どのようなデータが必要か？
    • どのようなビジネスルールがあるか？
    • データの整合性をどう保つか？
期待される結果

商品購入機能について、各層の役割と責任が明確に分離された設計ができること。この設計が次章以降のStruts実装の基礎となります。
