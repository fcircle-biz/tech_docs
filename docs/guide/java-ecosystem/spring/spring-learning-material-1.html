1.1 Spring Frameworkとは何か

Spring Framework（スプリング フレームワーク）は、Java企業アプリケーション開発を効率化し、コードの品質を向上させるための包括的なフレームワークです。現在では世界中の多くの企業でJavaアプリケーション開発の標準として採用されています。

なぜSpring Frameworkが生まれたのか

Javaでのエンタープライズアプリケーション開発は、従来非常に複雑で時間のかかる作業でした。例えば、データベースとの接続設定、トランザクション管理、セキュリティの実装、各クラス間の依存関係の管理など、アプリケーションの本質的なビジネスロジック以外の部分に多くの時間を費やす必要がありました。

Spring Frameworkが解決する主な問題
  • 複雑な設定ファイルの簡素化：XMLベースの複雑な設定から、アノテーションベースのシンプルな設定へ
  • 依存性注入（DI）による疎結合化：クラス間の依存関係を自動的に管理し、保守性を向上
  • ボイラープレートコードの削減：繰り返し書かれる定型コードを大幅に削減
  • 統一的な例外処理：データベースアクセスやWebリクエストの例外を統一的に処理

1.2 Javaの基本概念の復習

Spring Frameworkを理解するためには、Javaの基本的な概念をしっかりと理解している必要があります。特に重要なのは、オブジェクト指向プログラミングの概念です。

オブジェクト指向プログラミングの3大原則

1. カプセル化（Encapsulation）

カプセル化とは、データ（フィールド）とそのデータを操作するメソッドを一つのクラスにまとめ、外部から直接アクセスできないように保護することです。これにより、データの整合性を保ち、予期しない変更を防ぐことができます。

public class User {
    // private修飾子でフィールドを保護
    private String name;
    private int age;
    
    // publicメソッドを通じてアクセスを制御
    public String getName() {
        return name;
    }
    
    public void setName(String name) {
        // 値の検証を行うことができる
        if (name != null && !name.isEmpty()) {
            this.name = name;
        }
    }
    
    public int getAge() {
        return age;
    }
    
    public void setAge(int age) {
        // 年齢の妥当性をチェック
        if (age >= 0 && age <= 150) {
            this.age = age;
        }
    }
}
2. 継承（Inheritance）

継承とは、既存のクラスの特性（フィールドやメソッド）を引き継いで、新しいクラスを作成することです。これにより、コードの再利用性が向上し、共通の機能を効率的に管理できます。

// 基底クラス（親クラス）
public class Animal {
    protected String name;
    
    public void eat() {
        System.out.println(name + "が食事をしています");
    }
}

// 派生クラス（子クラス）
public class Dog extends Animal {
    public Dog(String name) {
        this.name = name;
    }
    
    // 独自のメソッドを追加
    public void bark() {
        System.out.println(name + "が吠えています");
    }
    
    // 親クラスのメソッドをオーバーライド
    @Override
    public void eat() {
        System.out.println(name + "がドッグフードを食べています");
    }
}
3. ポリモーフィズム（Polymorphism）

ポリモーフィズムとは、同じインターフェースを持つ異なるクラスのオブジェクトを、統一的に扱うことができる仕組みです。これにより、柔軟で拡張性の高いプログラムを作成できます。

// インターフェース
public interface Drawable {
    void draw();
}

// 具体的な実装クラス
public class Circle implements Drawable {
    @Override
    public void draw() {
        System.out.println("円を描きます");
    }
}

public class Rectangle implements Drawable {
    @Override
    public void draw() {
        System.out.println("四角形を描きます");
    }
}

// ポリモーフィズムの使用例
public class DrawingApp {
    public void drawShape(Drawable shape) {
        // 実際のオブジェクトの種類に関係なく、統一的に処理
        shape.draw();
    }
    
    public static void main(String[] args) {
        DrawingApp app = new DrawingApp();
        
        // 異なる実装でも同じように扱える
        app.drawShape(new Circle());
        app.drawShape(new Rectangle());
    }
}

1.3 HTTPプロトコルの基礎知識

Webアプリケーションを開発する上で、HTTPプロトコルの理解は不可欠です。HTTPは、WebブラウザとWebサーバーがデータを交換するための通信規約（プロトコル）です。

主要なHTTPメソッド

HTTPメソッドは、サーバーに対して「何をしたいか」を伝える方法です。Webアプリケーション開発では、以下のメソッドが頻繁に使用されます。

RESTful APIでよく使われるHTTPメソッド
  • GET：データの取得（読み取り専用、安全な操作）
  • POST：新しいデータの作成、フォーム送信
  • PUT：データの完全な更新・置換
  • PATCH：データの部分的な更新
  • DELETE：データの削除

HTTPステータスコード

HTTPステータスコードは、サーバーがリクエストをどのように処理したかを示す3桁の数字です。

成功系（2xx）
  • 200 OK：リクエスト成功
  • 201 Created：リソース作成成功
  • 204 No Content：処理成功、返すデータなし
エラー系（4xx, 5xx）
  • 400 Bad Request：リクエストが不正
  • 404 Not Found：リソースが見つからない
  • 500 Internal Server Error：サーバー内部エラー

1.4 MVCアーキテクチャパターン

MVC（Model-View-Controller）は、アプリケーションを3つの役割に分離する設計パターンです。この分離により、各部分を独立して開発・保守でき、チーム開発が効率的に行えます。

各層の役割と責任

1. Model（モデル層）

アプリケーションのデータとビジネスロジックを担当します。データベースとのやり取り、データの検証、計算処理などを行います。

Modelの責任
  • データの格納と取得
  • ビジネスルールの実装
  • データの整合性保証
  • 外部システムとの連携
2. View（ビュー層）

ユーザーに情報を表示する部分です。HTML、CSS、JavaScriptを使ってユーザーインターフェースを構築します。データの表示のみを担当し、ビジネスロジックは含めません。

Viewの責任
  • データの視覚的表現
  • ユーザーインターフェースの提供
  • フォームやボタンなどの入力要素
  • レスポンシブデザインの実装
3. Controller（コントローラー層）

ユーザーからの入力を受け取り、ModelとViewの間を調整する役割を担います。リクエストを解釈し、適切なModelを呼び出し、結果をViewに渡します。

Controllerの責任
  • HTTPリクエストの処理
  • 入力データの検証
  • 適切なModelメソッドの呼び出し
  • Viewテンプレートの選択
  • エラーハンドリング
実習 1-1: オブジェクト指向の概念確認

以下のシンプルなJavaクラスを通して、オブジェクト指向の概念を確認しましょう。

課題

図書館システムの一部として、「Book（本）」クラスを設計してみましょう。

  1. Book クラスに以下のフィールドを定義してください：
    • title（タイトル）
    • author（著者）
    • price（価格）
  2. カプセル化の原則に従って、適切なアクセス修飾子を使用してください
  3. getter/setterメソッドを実装してください
  4. コンストラクタを追加してください
解答例
public class Book {
    // カプセル化：フィールドをprivateで保護
    private String title;
    private String author;
    private int price;
    
    // コンストラクタ
    public Book(String title, String author, int price) {
        this.title = title;
        this.author = author;
        this.price = price;
    }
    
    // getter/setterメソッド
    public String getTitle() {
        return title;
    }
    
    public void setTitle(String title) {
        // 値の検証を追加できる
        if (title != null && !title.trim().isEmpty()) {
            this.title = title;
        }
    }
    
    public String getAuthor() {
        return author;
    }
    
    public void setAuthor(String author) {
        if (author != null && !author.trim().isEmpty()) {
            this.author = author;
        }
    }
    
    public int getPrice() {
        return price;
    }
    
    public void setPrice(int price) {
        // 価格は負の値にならないように制御
        if (price >= 0) {
            this.price = price;
        }
    }
    
    // 本の情報を表示するメソッド
    public void displayInfo() {
        System.out.println("タイトル: " + title);
        System.out.println("著者: " + author);
        System.out.println("価格: " + price + "円");
    }
}
実習 1-2: MVCパターンの理解

簡単なWebアプリケーションの例を通して、MVCパターンの役割分担を理解しましょう。

シナリオ

ユーザー情報を管理するWebアプリケーションにおいて、「ユーザー一覧を表示する」機能を実装する場合を考えてみましょう。

各層での処理の流れ
  1. ユーザーの操作：ブラウザで「/users」にアクセス
  2. Controller：リクエストを受信し、ユーザー一覧データの取得をModelに依頼
  3. Model：データベースからユーザー情報を取得し、Controllerに返却
  4. Controller：取得したデータをViewに渡す
  5. View：HTMLを生成してユーザーに表示
sequenceDiagram participant U as ユーザー participant V as View participant C as Controller participant M as Model participant DB as データベース U->>C: GET /users C->>M: getUserList() M->>DB: SELECT * FROM users DB->>M: ユーザーデータ M->>C: List<User> C->>V: users.html (データ付き) V->>U: HTML表示
考えてみましょう

この例で、各層が他の層の詳細を知らなくても動作する理由は何でしょうか？これがMVCパターンの大きなメリットの一つです。

理解度確認クイズ
  1. Spring Frameworkが解決する主な問題を3つ挙げてください。
    解答を見る

    1. 複雑な設定ファイルの簡素化
    2. 依存性注入による疎結合化
    3. ボイラープレートコードの削減

  2. オブジェクト指向プログラミングの3大原則は何ですか？
    解答を見る

    1. カプセル化（Encapsulation）
    2. 継承（Inheritance）
    3. ポリモーフィズム（Polymorphism）

  3. HTTPメソッドのGETとPOSTの違いは何ですか？
    解答を見る

    GET: データの取得に使用、安全で冪等性がある、URLにパラメータが表示される
    POST: データの作成や更新に使用、副作用があり、リクエストボディにデータを含む

  4. MVCパターンにおいて、ビジネスロジックはどの層で実装すべきでしょうか？その理由も説明してください。
    解答を見る

    Model層で実装すべきです。理由：ビジネスロジックはアプリケーションの核心的な処理であり、表示方法（View）や制御方法（Controller）に依存すべきではないため。また、テストしやすく、再利用しやすくなります。

  5. カプセル化を実装する際に、フィールドをprivateにして、publicのgetter/setterメソッドを提供する利点は何ですか？
    解答を見る

    1. データの妥当性チェックができる
    2. 内部実装を変更してもクライアントコードに影響しない
    3. アクセスログを取得できる
    4. 読み取り専用フィールドを作れる（setterを提供しない）

1.5 まとめ

この章では、Spring Frameworkを学習するための基礎知識を確認しました。

重要なポイント
  • Spring Frameworkは、Java開発の複雑さを解決し、開発効率を向上させるフレームワーク
  • オブジェクト指向の3原則（カプセル化、継承、ポリモーフィズム）は、Spring理解の基礎
  • HTTPプロトコルは、Webアプリケーションの通信基盤
  • MVCパターンは、アプリケーションを整理された構造で開発するための設計パターン
  • これらの基礎知識は、次章以降のSpring Boot学習で具体的に活用される
