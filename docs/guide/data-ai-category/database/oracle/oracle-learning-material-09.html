トランザクション管理とバックアップ基礎

トランザクション制御（COMMIT、ROLLBACK）とデータのエクスポート・インポートによる基本的なバックアップ方法を学習します。

この章で学ぶこと

  • トランザクションの概念とACID特性を理解する
  • COMMIT、ROLLBACK、SAVEPOINTの使い方を習得する
  • データのエクスポート・インポート方法を学ぶ
  • 基本的なバックアップ戦略を理解する

9.1 トランザクションとは

トランザクション（Transaction）とは、データベースにおける「一連の処理のまとまり」を指します。複数のSQL文を1つの論理単位として扱い、すべて成功するか、すべて失敗するかのどちらかに確定させます。

例えば、銀行の送金処理では「A口座から引き落とし」と「B口座へ入金」の2つの処理がありますが、片方だけ成功してもう一方が失敗するとお金が消えたり二重に増えたりしてしまいます。トランザクションを使うことで、両方成功した場合のみ確定させることができます。

変更を確定] D -->|いいえ| F[ROLLBACK
すべて取り消し] E --> G([トランザクション完了]) F --> G

自動コミットモード

SQL DeveloperなどのGUIツールでは、デフォルトで自動コミットが有効になっている場合があります。自動コミットが有効な場合、各SQL文が実行されるたびに自動的にCOMMITされます。手動でトランザクションを制御する場合は、自動コミットを無効にする必要があります。

9.2 ACID特性

信頼性の高いトランザクション処理を実現するために、データベースはACID特性を満たす必要があります。ACIDは4つの重要な性質の頭文字を取ったものです。

Atomicity
（原子性）

トランザクション内のすべての処理が完了するか、すべて実行されないかのどちらかになります。「All or Nothing」の原則です。

Consistency
（一貫性）

トランザクションの前後で、データベースの整合性制約（主キー、外部キー、チェック制約など）が保たれます。

Isolation
（独立性）

複数のトランザクションが同時に実行されても、互いに影響を与えません。それぞれのトランザクションは独立して実行されているように見えます。

Durability
（永続性）

一度コミットされたトランザクションの結果は、システム障害が発生しても失われません。ディスクに永続的に保存されます。

重要ポイント

Oracle DatabaseはACID特性を完全にサポートしており、ミッションクリティカルな業務システムで広く採用されている理由の一つです。これらの特性により、データの信頼性と整合性が保証されます。

9.3 COMMIT、ROLLBACK、SAVEPOINT

COMMIT（確定）

COMMITは、現在のトランザクション内で実行されたすべての変更をデータベースに確定させます。COMMIT後は変更を取り消すことができません。

COMMIT例 
-- 社員テーブルに新しいレコードを挿入
INSERT INTO employees (employee_id, name, salary)
VALUES (101, '山田太郎', 350000);

-- 変更を確定
COMMIT;
-- この時点で変更がデータベースに永続的に保存されます

ROLLBACK（取り消し）

ROLLBACKは、現在のトランザクション内で実行されたすべての変更を取り消します。最後のCOMMIT（またはトランザクション開始時点）の状態に戻ります。

ROLLBACK例 
-- 複数の変更を実行
UPDATE employees SET salary = salary * 1.1 WHERE department_id = 10;
DELETE FROM employees WHERE employee_id = 999;

-- 間違いに気づいたので取り消し
ROLLBACK;
-- すべての変更が取り消され、元の状態に戻ります

SAVEPOINT（保存点）

SAVEPOINTは、トランザクション内に「中間地点」を設定します。エラーが発生した場合、トランザクション全体を取り消すのではなく、特定の保存点まで戻ることができます。

SAVEPOINT例 
-- 処理1: 部門の更新
UPDATE departments SET budget = budget * 1.1 WHERE department_id = 10;

-- 保存点を設定
SAVEPOINT update_dept;

-- 処理2: 社員の更新
UPDATE employees SET salary = salary * 1.1 WHERE department_id = 10;

-- 処理2でエラーが発生した場合、保存点まで戻る
ROLLBACK TO SAVEPOINT update_dept;

-- 処理1の変更は残り、処理2の変更だけが取り消されます
-- 最終的に確定
COMMIT;
flowchart TD A([トランザクション開始]) --> B[SQL文実行1] B --> C[SAVEPOINT作成] C --> D[SQL文実行2] D --> E[SQL文実行3] E --> F{処理結果} F -->|すべて成功| G[COMMIT] F -->|一部失敗| H{復旧可能?} H -->|はい| I[ROLLBACK TO SAVEPOINT] H -->|いいえ| J[ROLLBACK] I --> K[再試行] K --> G G --> L([トランザクション完了]) J --> L

ヒント

複雑な処理を行う場合は、適切な箇所にSAVEPOINTを設定しておくと、エラー発生時に最初からやり直す必要がなくなります。ただし、SAVEPOINTを多用しすぎると管理が複雑になるため、適度に使うことが重要です。

9.4 実習: トランザクション制御

実習 9-1: トランザクション制御の基本

COMMIT、ROLLBACK、SAVEPOINTの動作を実際に試してみましょう。

手順

  1. 1 テスト用テーブルを作成します
  2. 2 ROLLBACKの動作を確認します
  3. 3 SAVEPOINTの動作を確認します

手順1: テーブル作成とデータ投入

テーブル作成 
-- テスト用テーブルを作成
CREATE TABLE test_transaction (
    id NUMBER PRIMARY KEY,
    name VARCHAR2(50),
    amount NUMBER
);

-- 初期データを投入
INSERT INTO test_transaction VALUES (1, '商品A', 1000);
INSERT INTO test_transaction VALUES (2, '商品B', 2000);
COMMIT;

手順2: ROLLBACKの確認

ROLLBACK動作確認 
-- データを更新
UPDATE test_transaction SET amount = amount * 2;

-- 現在のデータを確認
SELECT * FROM test_transaction;
-- 結果: 商品Aは2000、商品Bは4000になっている

-- 取り消し
ROLLBACK;

-- 再度確認
SELECT * FROM test_transaction;
-- 結果: 元に戻り、商品Aは1000、商品Bは2000

手順3: SAVEPOINTの確認

SAVEPOINT動作確認 
-- 商品Aの金額を更新
UPDATE test_transaction SET amount = 1500 WHERE id = 1;

-- 保存点を設定
SAVEPOINT after_update_a;

-- 商品Bの金額を更新
UPDATE test_transaction SET amount = 2500 WHERE id = 2;

-- 現在のデータを確認
SELECT * FROM test_transaction;
-- 結果: 商品Aは1500、商品Bは2500

-- 保存点まで戻る
ROLLBACK TO SAVEPOINT after_update_a;

-- 再度確認
SELECT * FROM test_transaction;
-- 結果: 商品Aは1500（残る）、商品Bは2000（戻る）

-- 最終的に確定
COMMIT;

9.5 データのエクスポート

データのエクスポートは、データベース内のデータを外部ファイルに保存する作業です。バックアップや他の環境へのデータ移行に使用します。

SQL Developerでのエクスポート

SQL Developerを使用すると、GUIで簡単にデータをエクスポートできます。

SQL Developerでのエクスポート手順

  1. 1
    テーブルを選択

    左側のツリーからエクスポートしたいテーブルを右クリック

  2. 2
    「エクスポート」を選択

    メニューから「エクスポート」→「ローダー」または「INSERT文」を選択

  3. 3
    形式とファイル名を指定

    CSV、SQL INSERT文、Excel形式などを選択し、保存先を指定

  4. 4
    エクスポート実行

    「次へ」→「完了」でエクスポートを実行

SQLでのエクスポート（スプール機能）

SQL*Plusやスクリプトでデータをエクスポートする場合は、SPOOL機能を使用します。

スプールでのエクスポート 
-- スプール開始（ファイルへの出力を開始）
SPOOL C:\backup\employees_data.txt

-- データを取得
SELECT employee_id, name, salary
FROM employees;

-- スプール終了
SPOOL OFF;

注意

エクスポートするデータに個人情報や機密情報が含まれる場合は、ファイルの保存場所や取り扱いに十分注意してください。また、大量のデータをエクスポートする場合は、ディスク容量を事前に確認しましょう。

9.6 データのインポート

エクスポートしたデータを別の環境やテーブルに取り込む作業がインポートです。バックアップからの復元やデータ移行に使用します。

SQL Developerでのインポート

SQL Developerでのインポート手順

  1. 1
    テーブルを選択

    左側のツリーからインポート先のテーブルを右クリック

  2. 2
    「データのインポート」を選択

    メニューから「データのインポート」を選択

  3. 3
    ファイルを選択

    インポート元のCSVファイルやExcelファイルを選択

  4. 4
    列マッピングの確認

    ファイルの列とテーブルの列が正しくマッピングされているか確認

  5. 5
    インポート実行

    「次へ」→「完了」でインポートを実行

SQL文でのインポート

エクスポート時にINSERT文形式で保存した場合は、そのSQL文を実行するだけでインポートできます。

SQL文でのインポート 
-- エクスポートしたINSERT文を実行
@C:\backup\employees_insert.sql

-- または直接INSERT文を記述
INSERT INTO employees VALUES (101, '山田太郎', 350000);
INSERT INTO employees VALUES (102, '佐藤花子', 380000);
INSERT INTO employees VALUES (103, '鈴木一郎', 320000);

-- 確定
COMMIT;

ヒント

大量のデータをインポートする場合は、一時的に制約やトリガーを無効にすることでパフォーマンスを向上できます。インポート完了後は、必ず制約を再度有効にしてデータの整合性を確認しましょう。

9.7 基本的なバックアップ戦略

データの損失を防ぐためには、定期的なバックアップが不可欠です。ここでは、初心者でも実践できる基本的なバックアップ戦略を紹介します。

バックアップの種類

種類 特徴 用途
論理バックアップ データをSQL文やCSV形式で保存 テーブル単位のバックアップ、他環境への移行
物理バックアップ データファイルをそのままコピー データベース全体のバックアップ、高速復元
フルバックアップ すべてのデータを保存 週次バックアップ、基準点の作成
増分バックアップ 前回のバックアップ以降の変更分のみ保存 日次バックアップ、ストレージ節約

推奨バックアップスケジュール

週次

  • フルバックアップ実施
  • 日曜深夜など業務時間外に実行

日次

  • 増分バックアップまたはエクスポート
  • 重要テーブルの論理バックアップ

リアルタイム

  • アーカイブログモード有効化
  • トランザクションログの保存

+ 日次増分] B -->|高| E[週次フル
+ 日次増分
+ アーカイブログ] C --> F[外部ストレージ保存] D --> F E --> F F --> G[定期的な復元テスト]

重要ポイント

バックアップを取るだけでなく、定期的に復元テストを行うことが重要です。いざという時にバックアップから復元できないと意味がありません。また、バックアップファイルは必ず別の場所（外部ディスク、クラウドストレージなど）に保存しましょう。

9.8 トラブルシューティング

よくあるトラブルと解決方法

ROLLBACKしたのに変更が残っている

原因: 別のセッションで実行された変更、または自動コミットが有効になっている

解決方法: 自分のセッションで実行した変更のみがROLLBACKの対象です。SQL Developerの自動コミット設定を確認し、必要に応じて無効にしてください。

エクスポートしたファイルが文字化けする

原因: 文字コードの不一致

解決方法: エクスポート時に文字コードをUTF-8に指定してください。Excel等で開く場合は、BOM付きUTF-8を選択するとより互換性が高くなります。

インポート時に主キー重複エラーが発生する

原因: インポート先のテーブルに既に同じ主キーのデータが存在する

解決方法: インポート前にテーブルを空にする（TRUNCATE TABLE）か、重複するデータを削除してからインポートしてください。または、UPDATEとINSERTを組み合わせたMERGE文を使用することも検討できます。

トランザクションがロックされて待機状態になる

原因: 他のセッションが同じレコードをロックしている

解決方法: ロックしているセッションがCOMMITまたはROLLBACKするまで待つか、DBAに依頼してロックセッションを確認・解除してもらいます。開発環境では、長時間トランザクションを開いたままにしないよう注意しましょう。

理解度確認クイズ

  1. Q1. ACID特性の4つの要素を説明してください。

    回答を表示

    回答:
    1. Atomicity（原子性）: トランザクション内のすべての処理が完了するか、すべて実行されないかのどちらかになる。
    2. Consistency（一貫性）: トランザクションの前後で、データベースの整合性制約が保たれる。
    3. Isolation（独立性）: 複数のトランザクションが同時に実行されても、互いに影響を与えない。
    4. Durability（永続性）: 一度コミットされたトランザクションの結果は、システム障害が発生しても失われない。

  2. Q2. SAVEPOINTを使用する利点は何ですか？

    回答を表示

    回答: SAVEPOINTを使用すると、トランザクション全体をロールバックするのではなく、特定の保存点までロールバックできます。複雑な処理の途中でエラーが発生した場合、すべてを最初からやり直す必要がなくなります。これにより、エラーハンドリングが柔軟になり、処理効率が向上します。

  3. Q3. バックアップ戦略で重要なポイントを3つ挙げてください。

    回答を表示

    回答:
    1. 定期的なバックアップスケジュールを設定し、確実に実行する（週次フルバックアップ、日次増分バックアップなど）。
    2. バックアップファイルは必ず別の場所（外部ディスク、クラウドストレージなど）に保存し、災害時のリスクを分散する。
    3. 定期的に復元テストを実施し、バックアップから確実に復元できることを確認する。
