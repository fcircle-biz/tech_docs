パフォーマンス基礎とインデックス

インデックスの概念と作成方法、実行計画の確認方法など、パフォーマンス改善の基礎知識を習得します。

この章で学ぶこと

  • データベースパフォーマンスの基本概念を理解する
  • インデックスの役割と種類を学ぶ
  • インデックスの作成・削除方法を習得する
  • 実行計画の確認方法とパフォーマンスチューニングの基礎を理解する

10.1 データベースパフォーマンスとは

データベースパフォーマンスとは、「SQLクエリがどれくらい速く実行されるか」「同時に何人のユーザーがアクセスできるか」といった、データベースの処理速度や効率性を指します。

データ量が増えるにつれて、何も対策をしていないとクエリの実行時間が極端に長くなることがあります。ユーザーが数秒待たされるだけでも、システムの使い勝手が大きく損なわれます。

パフォーマンス問題の例

  • • 顧客検索に30秒以上かかる
  • • 月次レポート作成に数時間かかる
  • • 複数ユーザーが同時アクセスするとシステムが重くなる
  • • データ件数が増えるとクエリが極端に遅くなる

パフォーマンス改善の主な手法

  • インデックスの作成: 検索速度を大幅に向上させる（本章で詳しく学習）
  • SQLクエリの最適化: 無駄な処理を削減し、効率的なクエリを書く
  • テーブル設計の見直し: 正規化・非正規化のバランスを調整
  • 統計情報の更新: Oracleが最適な実行計画を選択できるようにする

10.2 インデックスとは

インデックス（索引）とは、データベースの検索速度を高速化するための仕組みです。書籍の「索引」と同じように、目的のデータを素早く見つけられるようにします。

インデックスの仕組み

インデックスは、特定のカラムの値とそのデータが格納されている場所（ROWID）を紐付けたデータ構造です。Oracleでは主にB-Tree（バランス木）構造を使用します。

ROWIDとは

ROWIDは、テーブル内の各行に一意に割り当てられる物理的な位置情報です。インデックスはこのROWIDを使って、検索条件に一致するデータに直接アクセスします。

インデックスのメリット・デメリット

メリット

  • SELECT文の検索が高速化
  • WHERE句、JOIN、ORDER BY句のパフォーマンス向上
  • 大量データでも高速にアクセス可能

デメリット

  • INSERT、UPDATE、DELETEの速度が低下
  • ディスク容量を追加で消費
  • 管理コストが増加

重要ポイント

インデックスは「検索を速くするが、更新を遅くする」という特性があります。頻繁に検索されるカラムには有効ですが、頻繁に更新されるカラムには慎重に適用する必要があります。

10.3 インデックスの種類

Oracleでは主に以下のインデックスが使用されます。

種類 説明 用途
B-Treeインデックス 最も一般的なインデックス 等価検索、範囲検索
一意インデックス 重複値を許さないインデックス 主キー、ユニークキー
複合インデックス 複数カラムを組み合わせたインデックス 複数条件での検索
Bitmapインデックス カーディナリティが低いカラム向け 性別、都道府県など
ファンクションインデックス 関数の結果に対するインデックス UPPER()、LOWER()など

カーディナリティとは

カーディナリティとは、カラムに含まれる一意な値の数を指します。例えば、社員IDは高カーディナリティ（ほぼすべて異なる値）、性別は低カーディナリティ（男・女の2値）です。

10.4 インデックスの作成

インデックスはCREATE INDEX文で作成します。基本的な構文と実例を見ていきましょう。

基本構文

SQL 
-- 単一カラムのインデックス作成
CREATE INDEX インデックス名 ON テーブル名(カラム名);

-- 複合インデックスの作成
CREATE INDEX インデックス名 ON テーブル名(カラム1, カラム2);

-- 一意インデックスの作成
CREATE UNIQUE INDEX インデックス名 ON テーブル名(カラム名);

実習: インデックスを作成してみよう

実習: EMPLOYEESテーブルにインデックスを作成

EMPLOYEESテーブルのLAST_NAMEカラムにインデックスを作成します。

ステップ1: テーブル確認

SQL 
-- EMPLOYEESテーブルの構造確認
DESC EMPLOYEES;

ステップ2: インデックス作成

SQL 
-- LAST_NAMEカラムにインデックスを作成
CREATE INDEX idx_emp_lastname
ON EMPLOYEES(LAST_NAME);

ステップ3: インデックスの確認

SQL 
-- ユーザーのインデックス一覧を確認
SELECT INDEX_NAME, TABLE_NAME, COLUMN_NAME
FROM USER_IND_COLUMNS
WHERE TABLE_NAME = 'EMPLOYEES';

複合インデックスの例

SQL 
-- DEPARTMENT_IDとJOB_IDの複合インデックス
CREATE INDEX idx_emp_dept_job
ON EMPLOYEES(DEPARTMENT_ID, JOB_ID);

-- このインデックスが有効なクエリ例
SELECT *
FROM EMPLOYEES
WHERE DEPARTMENT_ID = 50
  AND JOB_ID = 'ST_CLERK';

ヒント: 複合インデックスの順序

複合インデックスは、先頭のカラムから順番に評価されます。WHERE句で「DEPARTMENT_ID = 50」のみで検索する場合は有効ですが、「JOB_ID = 'ST_CLERK'」のみで検索する場合は、このインデックスは使われません。

10.5 インデックスの削除

不要になったインデックスはDROP INDEX文で削除します。

SQL 
-- インデックスの削除
DROP INDEX idx_emp_lastname;

-- 削除確認
SELECT INDEX_NAME, TABLE_NAME
FROM USER_INDEXES
WHERE INDEX_NAME = 'IDX_EMP_LASTNAME';

注意

主キーやユニークキー制約によって自動作成されたインデックスは、制約を削除しない限り削除できません。インデックスを削除する前に、本当に不要か確認しましょう。

10.6 実行計画の確認

実行計画とは、Oracleが「どのようにSQLを実行するか」を示した計画書です。インデックスが使われているか、テーブル全件検索になっていないかを確認できます。

EXPLAIN PLAN文の使い方

SQL 
-- 実行計画を取得
EXPLAIN PLAN FOR
SELECT * FROM EMPLOYEES
WHERE LAST_NAME = 'King';

-- 実行計画を表示
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY());

実行計画の見方

Plan hash value: 1445457117

-------------------------------------------------------------------------
| Id | Operation         | Name        | Rows | Bytes | Cost (%CPU)|
-------------------------------------------------------------------------
|  0 | SELECT STATEMENT  |             |    1 |    69 |     3   (0)|
|* 1 | TABLE ACCESS FULL | EMPLOYEES   |    1 |    69 |     3   (0)|
-------------------------------------------------------------------------

Predicate Information (identified by operation id):
---------------------------------------------------
   1 - filter("LAST_NAME"='King')

この例では「TABLE ACCESS FULL」となっており、テーブル全件検索（フルスキャン）が行われています。 

Plan hash value: 2077747057

-------------------------------------------------------------------------------------
| Id | Operation                   | Name             | Rows | Bytes | Cost (%CPU)|
-------------------------------------------------------------------------------------
|  0 | SELECT STATEMENT            |                  |    1 |    69 |     2   (0)|
|  1 | TABLE ACCESS BY INDEX ROWID | EMPLOYEES        |    1 |    69 |     2   (0)|
|* 2 | INDEX RANGE SCAN            | IDX_EMP_LASTNAME |    1 |       |     1   (0)|
-------------------------------------------------------------------------------------

Predicate Information (identified by operation id):
---------------------------------------------------
   2 - access("LAST_NAME"='King')

こちらは「INDEX RANGE SCAN」となっており、インデックスが使われていることがわかります。

実行計画の主な用語

  • TABLE ACCESS FULL: テーブル全件検索（遅い）
  • INDEX RANGE SCAN: インデックスを使った範囲検索（速い）
  • TABLE ACCESS BY INDEX ROWID: インデックス経由でテーブルアクセス
  • Cost: 実行コスト（小さいほど効率的）

10.7 インデックスを作成すべきカラム

すべてのカラムにインデックスを作成すれば良いというわけではありません。以下のガイドラインを参考にしてください。

作成すべき

  • WHERE句で頻繁に検索条件に使われるカラム
  • JOINの結合キーとして使われるカラム
  • ORDER BY句で並び替えに使われるカラム
  • カーディナリティが高い（一意な値が多い）カラム

避けるべき

  • 頻繁にINSERT/UPDATE/DELETEされるカラム
  • 小さなテーブル（数百件程度）
  • カーディナリティが低い（値の種類が少ない）カラム
  • ほとんど検索されないカラム

10.8 パフォーマンスチューニングの基本

インデックス以外にも、SQLクエリのパフォーマンスを改善する方法があります。

効率的なSQLの書き方

悪い例 

-- SELECT *は不要なカラムも取得
SELECT *
FROM EMPLOYEES;

良い例 

-- 必要なカラムだけ指定
SELECT EMPLOYEE_ID, LAST_NAME
FROM EMPLOYEES;

悪い例 

-- 関数を使うとインデックス無効
SELECT * FROM EMPLOYEES
WHERE UPPER(LAST_NAME) = 'KING';

良い例 

-- カラムに直接条件を指定
SELECT * FROM EMPLOYEES
WHERE LAST_NAME = 'King';

統計情報の更新

Oracleは統計情報を元に最適な実行計画を選択します。大量のデータ更新後は統計情報を更新しましょう。

SQL 
-- テーブルの統計情報を更新
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(
    OWNNAME => 'HR',
    TABNAME => 'EMPLOYEES'
  );
END;
/

10.9 実習: パフォーマンス改善を体験

実習: インデックス作成前後の実行計画比較

ステップ1: インデックスなしの実行計画

SQL 
-- 実行計画を取得
EXPLAIN PLAN FOR
SELECT * FROM EMPLOYEES
WHERE EMAIL = 'SKING';

-- 実行計画を表示
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY());

「TABLE ACCESS FULL」になっていることを確認

ステップ2: インデックスを作成

SQL 
-- EMAILカラムにインデックス作成
CREATE INDEX idx_emp_email
ON EMPLOYEES(EMAIL);

ステップ3: インデックスありの実行計画

SQL 
-- 再度実行計画を取得
EXPLAIN PLAN FOR
SELECT * FROM EMPLOYEES
WHERE EMAIL = 'SKING';

-- 実行計画を表示
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY());

「INDEX RANGE SCAN」に変わっていることを確認

動作確認

  • インデックス作成前は「TABLE ACCESS FULL」が表示される
  • インデックス作成後は「INDEX RANGE SCAN」が表示される
  • Costの値がインデックス作成後の方が小さくなる

トラブルシューティング

問題: インデックスを作成しても実行計画が変わらない

原因: テーブルが小さすぎる、または統計情報が古い可能性があります。
解決策: 統計情報を更新してみてください。DBMS_STATS.GATHER_TABLE_STATSを実行します。

問題: ORA-01408: このリストに対してこのような索引は作成されていません

原因: 存在しないインデックスを削除しようとしています。
解決策: USER_INDEXESビューで存在するインデックス名を確認してください。

理解度確認クイズ

  1. Q1. インデックスの主なメリットとデメリットは何ですか？

    回答を表示

    回答: メリットは、SELECT文の検索速度が大幅に向上することです。特にWHERE句、JOIN、ORDER BY句のパフォーマンスが改善されます。デメリットは、INSERT、UPDATE、DELETEの速度が低下すること、ディスク容量を追加で消費すること、管理コストが増加することです。

  2. Q2. 実行計画で「TABLE ACCESS FULL」と「INDEX RANGE SCAN」の違いは何ですか？

    回答を表示

    回答: 「TABLE ACCESS FULL」はテーブル全件検索（フルスキャン）を意味し、テーブルのすべての行を読み取ります。一方、「INDEX RANGE SCAN」はインデックスを使った範囲検索を意味し、必要なデータだけに効率的にアクセスします。一般的に「INDEX RANGE SCAN」の方が高速です。

  3. Q3. インデックスを作成すべきカラムの条件を3つ挙げてください。

    回答を表示

    回答: ①WHERE句で頻繁に検索条件に使われるカラム、②JOINの結合キーとして使われるカラム、③ORDER BY句で並び替えに使われるカラム、などが挙げられます。また、カーディナリティが高い（一意な値が多い）カラムがインデックスに適しています。
