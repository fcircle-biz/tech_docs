まずは、システムで管理するデータとテーブル構造を設計しましょう。

テーブル設計

以下の4つのテーブルを作成します。

テーブル作成DDL 
-- 1. 会員テーブル
CREATE TABLE lib_members (
    member_id NUMBER PRIMARY KEY,
    member_name VARCHAR2(100) NOT NULL,
    email VARCHAR2(100),
    phone VARCHAR2(20),
    join_date DATE DEFAULT SYSDATE,
    status VARCHAR2(20) DEFAULT 'ACTIVE'
);

-- 2. 図書テーブル
CREATE TABLE lib_books (
    book_id NUMBER PRIMARY KEY,
    title VARCHAR2(200) NOT NULL,
    author VARCHAR2(100),
    isbn VARCHAR2(20),
    total_copies NUMBER DEFAULT 1,
    available_copies NUMBER DEFAULT 1
);

-- 3. 貸出記録テーブル
CREATE TABLE lib_loans (
    loan_id NUMBER PRIMARY KEY,
    member_id NUMBER NOT NULL,
    book_id NUMBER NOT NULL,
    loan_date DATE DEFAULT SYSDATE,
    due_date DATE,
    return_date DATE,
    status VARCHAR2(20) DEFAULT 'BORROWED'
);

-- 4. 延滞管理テーブル
CREATE TABLE lib_overdue (
    overdue_id NUMBER PRIMARY KEY,
    loan_id NUMBER NOT NULL,
    overdue_days NUMBER,
    penalty_fee NUMBER,
    paid_status VARCHAR2(20) DEFAULT 'UNPAID'
);

-- シーケンス作成
CREATE SEQUENCE seq_member START WITH 1;
CREATE SEQUENCE seq_book START WITH 1;
CREATE SEQUENCE seq_loan START WITH 1;
CREATE SEQUENCE seq_overdue START WITH 1;

テーブル設計のポイント

  • lib_members: 会員情報（名前、連絡先、登録日、ステータス）
  • lib_books: 図書情報（タイトル、著者、在庫数）
  • lib_loans: 貸出記録（貸出日、返却期限、返却日、ステータス）
  • lib_overdue: 延滞情報（延滞日数、延滞金、支払いステータス）

12.2 会員登録処理プロシージャの作成

新規会員を登録するプロシージャを作成します。
メールアドレスの重複チェック、例外処理、登録完了メッセージの表示を実装します。

会員登録プロシージャ 
CREATE OR REPLACE PROCEDURE register_member (
    p_name IN VARCHAR2,
    p_email IN VARCHAR2,
    p_phone IN VARCHAR2
) AS
    v_count NUMBER;
    v_member_id NUMBER;
BEGIN
    -- メールアドレス重複チェック
    SELECT COUNT(*) INTO v_count
    FROM lib_members
    WHERE email = p_email;

    IF v_count > 0 THEN
        RAISE_APPLICATION_ERROR(-20001,
            'エラー: このメールアドレスは既に登録されています');
    END IF;

    -- 新規会員ID取得
    v_member_id := seq_member.NEXTVAL;

    -- 会員登録
    INSERT INTO lib_members (member_id, member_name, email, phone)
    VALUES (v_member_id, p_name, p_email, p_phone);

    COMMIT;

    DBMS_OUTPUT.PUT_LINE('会員登録が完了しました');
    DBMS_OUTPUT.PUT_LINE('会員ID: ' || v_member_id);
    DBMS_OUTPUT.PUT_LINE('お名前: ' || p_name);

EXCEPTION
    WHEN OTHERS THEN
        ROLLBACK;
        DBMS_OUTPUT.PUT_LINE('エラー: ' || SQLERRM);
END;
/

12.3 貸出処理プロシージャの作成

図書の貸出処理を実装します。在庫チェック、貸出記録の作成、在庫数の自動更新を行います。

貸出処理プロシージャ 
CREATE OR REPLACE PROCEDURE borrow_book (
    p_member_id IN NUMBER,
    p_book_id IN NUMBER,
    p_loan_days IN NUMBER DEFAULT 14
) AS
    v_available NUMBER;
    v_loan_id NUMBER;
    v_due_date DATE;
BEGIN
    -- 在庫確認
    SELECT available_copies INTO v_available
    FROM lib_books
    WHERE book_id = p_book_id;

    IF v_available <= 0 THEN
        RAISE_APPLICATION_ERROR(-20002,
            '申し訳ございません。現在貸出可能な在庫がありません');
    END IF;

    -- 返却期限計算（貸出日数後）
    v_due_date := SYSDATE + p_loan_days;

    -- 貸出ID取得
    v_loan_id := seq_loan.NEXTVAL;

    -- 貸出記録作成
    INSERT INTO lib_loans (loan_id, member_id, book_id, due_date)
    VALUES (v_loan_id, p_member_id, p_book_id, v_due_date);

    -- 在庫数減少
    UPDATE lib_books
    SET available_copies = available_copies - 1
    WHERE book_id = p_book_id;

    COMMIT;

    DBMS_OUTPUT.PUT_LINE('貸出処理が完了しました');
    DBMS_OUTPUT.PUT_LINE('貸出ID: ' || v_loan_id);
    DBMS_OUTPUT.PUT_LINE('返却期限: ' || TO_CHAR(v_due_date, 'YYYY/MM/DD'));

EXCEPTION
    WHEN NO_DATA_FOUND THEN
        DBMS_OUTPUT.PUT_LINE('エラー: 図書が見つかりません');
    WHEN OTHERS THEN
        ROLLBACK;
        DBMS_OUTPUT.PUT_LINE('エラー: ' || SQLERRM);
END;
/

処理のポイント

  • 在庫チェックで貸出可能かを確認
  • 返却期限を自動計算（デフォルト14日）
  • 貸出記録作成と同時に在庫数を減少
  • エラー時はROLLBACKで全処理を取り消し

12.4 返却処理と延滞チェック

図書の返却処理と延滞チェックを実装します。延滞している場合は自動的に延滞管理テーブルに記録します。

返却処理プロシージャ 
CREATE OR REPLACE PROCEDURE return_book (
    p_loan_id IN NUMBER
) AS
    v_book_id NUMBER;
    v_due_date DATE;
    v_overdue_days NUMBER;
    v_penalty_fee NUMBER;
    v_overdue_id NUMBER;
BEGIN
    -- 貸出情報取得
    SELECT book_id, due_date
    INTO v_book_id, v_due_date
    FROM lib_loans
    WHERE loan_id = p_loan_id
      AND status = 'BORROWED';

    -- 返却日記録
    UPDATE lib_loans
    SET return_date = SYSDATE,
        status = 'RETURNED'
    WHERE loan_id = p_loan_id;

    -- 在庫数増加
    UPDATE lib_books
    SET available_copies = available_copies + 1
    WHERE book_id = v_book_id;

    -- 延滞チェック
    v_overdue_days := TRUNC(SYSDATE - v_due_date);

    IF v_overdue_days > 0 THEN
        -- 延滞金計算（1日100円）
        v_penalty_fee := v_overdue_days * 100;
        v_overdue_id := seq_overdue.NEXTVAL;

        -- 延滞記録作成
        INSERT INTO lib_overdue (
            overdue_id, loan_id, overdue_days, penalty_fee
        )
        VALUES (
            v_overdue_id, p_loan_id, v_overdue_days, v_penalty_fee
        );

        DBMS_OUTPUT.PUT_LINE('返却完了（延滞あり）');
        DBMS_OUTPUT.PUT_LINE('延滞日数: ' || v_overdue_days || '日');
        DBMS_OUTPUT.PUT_LINE('延滞金: ' || v_penalty_fee || '円');
    ELSE
        DBMS_OUTPUT.PUT_LINE('返却完了（期限内）');
    END IF;

    COMMIT;

EXCEPTION
    WHEN NO_DATA_FOUND THEN
        DBMS_OUTPUT.PUT_LINE('エラー: 貸出記録が見つかりません');
    WHEN OTHERS THEN
        ROLLBACK;
        DBMS_OUTPUT.PUT_LINE('エラー: ' || SQLERRM);
END;
/

12.5 図書館管理パッケージの作成

これまで作成したプロシージャをパッケージにまとめて、管理しやすくします。

パッケージ仕様部 
CREATE OR REPLACE PACKAGE library_pkg AS
    -- 会員登録
    PROCEDURE register_member (
        p_name IN VARCHAR2,
        p_email IN VARCHAR2,
        p_phone IN VARCHAR2
    );

    -- 貸出処理
    PROCEDURE borrow_book (
        p_member_id IN NUMBER,
        p_book_id IN NUMBER,
        p_loan_days IN NUMBER DEFAULT 14
    );

    -- 返却処理
    PROCEDURE return_book (
        p_loan_id IN NUMBER
    );

    -- 延滞一覧表示
    PROCEDURE show_overdue_list;

    -- 会員の貸出履歴表示
    PROCEDURE show_member_history (
        p_member_id IN NUMBER
    );
END library_pkg;
/
パッケージ本体（一部抜粋） 
CREATE OR REPLACE PACKAGE BODY library_pkg AS

    -- 延滞一覧表示
    PROCEDURE show_overdue_list AS
        CURSOR c_overdue IS
            SELECT o.overdue_id, m.member_name, b.title,
                   o.overdue_days, o.penalty_fee
            FROM lib_overdue o
            JOIN lib_loans l ON o.loan_id = l.loan_id
            JOIN lib_members m ON l.member_id = m.member_id
            JOIN lib_books b ON l.book_id = b.book_id
            WHERE o.paid_status = 'UNPAID';
    BEGIN
        DBMS_OUTPUT.PUT_LINE('=== 延滞一覧 ===');
        FOR rec IN c_overdue LOOP
            DBMS_OUTPUT.PUT_LINE(
                '会員: ' || rec.member_name ||
                ' / 図書: ' || rec.title ||
                ' / 延滞: ' || rec.overdue_days || '日' ||
                ' / 延滞金: ' || rec.penalty_fee || '円'
            );
        END LOOP;
    END show_overdue_list;

    -- 会員の貸出履歴表示
    PROCEDURE show_member_history (
        p_member_id IN NUMBER
    ) AS
        CURSOR c_history IS
            SELECT b.title, l.loan_date, l.return_date, l.status
            FROM lib_loans l
            JOIN lib_books b ON l.book_id = b.book_id
            WHERE l.member_id = p_member_id
            ORDER BY l.loan_date DESC;
    BEGIN
        DBMS_OUTPUT.PUT_LINE('=== 貸出履歴 ===');
        FOR rec IN c_history LOOP
            DBMS_OUTPUT.PUT_LINE(
                '図書: ' || rec.title ||
                ' / 貸出日: ' || TO_CHAR(rec.loan_date, 'YYYY/MM/DD') ||
                ' / ステータス: ' || rec.status
            );
        END LOOP;
    END show_member_history;

END library_pkg;
/

12.6 自動処理トリガーの追加

図書が追加されたときに自動でログを記録するトリガーを作成します。

図書追加ログトリガー 
-- ログテーブル作成
CREATE TABLE lib_book_log (
    log_id NUMBER PRIMARY KEY,
    book_id NUMBER,
    action VARCHAR2(20),
    log_date DATE DEFAULT SYSDATE
);

CREATE SEQUENCE seq_log START WITH 1;

-- トリガー作成
CREATE OR REPLACE TRIGGER trg_book_insert
AFTER INSERT ON lib_books
FOR EACH ROW
BEGIN
    INSERT INTO lib_book_log (log_id, book_id, action)
    VALUES (seq_log.NEXTVAL, :NEW.book_id, 'INSERT');
END;
/

理解度確認クイズ

  1. Q1. 貸出処理で在庫チェックが必要な理由は何ですか？

    回答を表示

    在庫がない図書を貸し出してしまうと、データの整合性が崩れてしまうためです。
    在庫数をチェックして、貸出可能な場合のみ処理を実行することで、正確なデータ管理が可能になります。

  2. Q2. 返却処理で延滞チェックをする意味は何ですか？

    回答を表示

    返却期限を過ぎている場合、延滞金を計算して記録する必要があるためです。
    延滞チェックにより、適切な延滞金を自動計算し、延滞管理テーブルに記録することで、会計処理を正確に行えます。

  3. Q3. パッケージにまとめるメリットは何ですか？

    回答を表示

    関連する機能をグループ化することで、管理しやすくなり、コードの再利用性が高まります。
    また、パッケージ名を指定することで、どの機能を呼び出しているかが明確になり、メンテナンス性が向上します。

よくあるエラーと対処法

エラー: ORA-00001 (一意制約違反)

原因: 同じIDのデータを重複して挿入しようとしている
対処: シーケンスを使ってユニークなIDを生成する

エラー: NO_DATA_FOUND

原因: SELECT INTOで該当データが見つからない
対処: EXCEPTION句でNO_DATA_FOUNDを処理する

在庫数がマイナスになる

原因: 在庫チェックが不十分
対処: 貸出前に必ずavailable_copies > 0をチェックする
