データベースの変更を自動化 - トリガーで賢い仕組み作り

データの追加・更新・削除時に自動で動くトリガーの作成方法を学習します。在庫の自動更新、履歴の自動記録など、データベースを賢くする仕組みを実装します。

この章で学ぶこと

  • トリガーとは何か、いつ使うのかを理解する
  • トリガーの種類（BEFORE/AFTER、INSERT/UPDATE/DELETE）を知る
  • 在庫自動更新や履歴自動記録のトリガーを作成できる
  • トリガーの動作確認とトラブルシューティングができる

10.1 トリガーとは？ - データベースに仕掛ける「自動実行の罠」

トリガー（TRIGGER）は、テーブルに対してINSERT（追加）、UPDATE（更新）、DELETE（削除）が行われたときに、自動的に実行されるPL/SQLプログラムです。
トリガーという英語は「引き金」という意味で、データの変更が「引き金」となってプログラムが発動します。

flowchart LR A[テーブルにINSERT] -->|引き金| B{{トリガー発動}} B --> C[自動でPL/SQL実行] C --> D[在庫更新や履歴記録] style A fill:#dbeafe,stroke:#3b82f6,stroke-width:2px style B fill:#fef3c7,stroke:#f59e0b,stroke-width:2px style C fill:#fecaca,stroke:#ef4444,stroke-width:2px style D fill:#d1fae5,stroke:#10b981,stroke-width:2px

トリガーの身近な例

ECサイトで商品を購入すると、注文テーブルに1件追加されます。
このとき、トリガーが自動で在庫テーブルの在庫数を減らしてくれます。
手作業で「在庫を減らす処理」を書かなくても、データベースが自動でやってくれるのです。

トリガーとプロシージャの違い

項目 プロシージャ トリガー
実行方法 手動で呼び出す（EXECUTE） データ変更で自動実行
引数 引数を渡せる 引数なし（:NEW、:OLDで値取得）
用途 手動処理の部品化 データ整合性の自動維持

10.2 トリガーの種類 - 「いつ」「何を」実行するか

① タイミング（BEFORE / AFTER）

トリガーは、データ変更の前（BEFORE）または後（AFTER）に実行できます。

BEFOREトリガー

データ変更の前に実行

例: 値の検証、デフォルト値の設定

AFTERトリガー

データ変更の後に実行

例: 履歴記録、在庫更新

② イベント（INSERT / UPDATE / DELETE）

トリガーは、どの操作で発動するかを指定できます。

イベント 発動タイミング 用途例
INSERT 新規データ追加時 作成日時の自動設定
UPDATE データ更新時 更新日時の自動設定
DELETE データ削除時 削除履歴の記録

③ 行レベル（FOR EACH ROW）

FOR EACH ROWを指定すると、変更された各行ごとにトリガーが実行されます。
例えば、10件のデータを更新すると、トリガーが10回実行されます。

:NEWと:OLD

FOR EACH ROWトリガーでは、以下の特殊変数が使えます。
:NEW → 変更後の値
:OLD → 変更前の値

10.3 トリガーの基本構文 - CREATE TRIGGER文

トリガーの基本構文 
CREATE OR REPLACE TRIGGER トリガー名
BEFORE/AFTER INSERT/UPDATE/DELETE ON テーブル名
FOR EACH ROW  -- 行ごとに実行する場合
BEGIN
    -- トリガーで実行する処理
    -- :NEWや:OLDを使って値を参照
END;
/

構文の解説

① トリガー名

分かりやすい名前をつけます。
例: trg_update_stock（在庫更新トリガー）

② タイミングとイベント

BEFORE INSERT（追加前）、AFTER UPDATE（更新後）など
複数イベントも可能: AFTER INSERT OR UPDATE

③ FOR EACH ROW

行ごとにトリガーを実行する場合に指定します。
省略すると、テーブル全体で1回だけ実行されます（文レベルトリガー）。

10.4 実習: 在庫自動更新トリガーを作ろう

注文が追加されたら、自動で商品の在庫を減らすトリガーを作成します。

flowchart LR A[注文テーブルにINSERT] -->|トリガー発動| B[在庫テーブルを更新] B --> C[在庫数 = 在庫数 - 注文数] style A fill:#dbeafe,stroke:#3b82f6,stroke-width:2px style B fill:#fecaca,stroke:#ef4444,stroke-width:2px style C fill:#d1fae5,stroke:#10b981,stroke-width:2px

実習 10-1: 在庫自動更新トリガー

注文テーブルにデータが追加されたら、自動で在庫を減らすトリガーを作成します。

手順

  1. 1 テーブルを作成します（既にある場合はスキップ）
  2. 2 トリガーを作成します
  3. 3 動作確認をします

ステップ1: テーブル作成

テーブル作成 
-- 商品テーブル
CREATE TABLE products (
    product_id NUMBER PRIMARY KEY,
    product_name VARCHAR2(100),
    stock NUMBER  -- 在庫数
);

-- 注文テーブル
CREATE TABLE orders (
    order_id NUMBER PRIMARY KEY,
    product_id NUMBER,
    quantity NUMBER  -- 注文数
);

-- サンプルデータ挿入
INSERT INTO products VALUES (1, 'ノートPC', 100);
INSERT INTO products VALUES (2, 'マウス', 200);
COMMIT;

ステップ2: トリガー作成

在庫更新トリガー 
CREATE OR REPLACE TRIGGER trg_update_stock
AFTER INSERT ON orders  -- 注文テーブルにINSERTされた後
FOR EACH ROW  -- 各行ごとに実行
BEGIN
    -- 商品の在庫を減らす
    UPDATE products
    SET stock = stock - :NEW.quantity  -- :NEWは追加されたデータ
    WHERE product_id = :NEW.product_id;

    DBMS_OUTPUT.PUT_LINE('在庫を更新: 商品ID=' || :NEW.product_id ||
                         ', 減少数=' || :NEW.quantity);
END;
/

:NEWの使い方

:NEW.quantityは、今まさにINSERTされたordersテーブルのquantity列の値です。
注文数が10なら、:NEW.quantityは10になります。

ステップ3: 動作確認

動作確認 
-- 出力を有効化
SET SERVEROUTPUT ON;

-- 注文前の在庫確認
SELECT product_id, product_name, stock FROM products;

-- 注文を追加（トリガーが自動実行される）
INSERT INTO orders VALUES (101, 1, 10);  -- 商品ID=1を10個注文
COMMIT;

-- 注文後の在庫確認
SELECT product_id, product_name, stock FROM products;

期待される結果

ノートPCの在庫が100から90に減っています。

PRODUCT_ID  PRODUCT_NAME  STOCK
----------  ------------  -----
1           ノートPC       90    ← 100から減った！
2           マウス         200

10.5 実習: 履歴自動記録トリガーを作ろう

社員の給与が更新されたら、自動で履歴テーブルに記録するトリガーを作成します。

実習 10-2: 履歴自動記録トリガー

社員テーブルの給与が更新されたら、変更前の値を自動で履歴テーブルに記録します。

ステップ1: テーブル作成

テーブル作成 
-- 社員テーブル
CREATE TABLE employees (
    emp_id NUMBER PRIMARY KEY,
    emp_name VARCHAR2(100),
    salary NUMBER
);

-- 給与履歴テーブル
CREATE TABLE salary_history (
    history_id NUMBER PRIMARY KEY,
    emp_id NUMBER,
    old_salary NUMBER,  -- 変更前の給与
    new_salary NUMBER,  -- 変更後の給与
    changed_date DATE
);

-- シーケンス作成（履歴IDの自動採番用）
CREATE SEQUENCE seq_history START WITH 1;

-- サンプルデータ挿入
INSERT INTO employees VALUES (1, '田中太郎', 300000);
INSERT INTO employees VALUES (2, '佐藤花子', 350000);
COMMIT;

ステップ2: トリガー作成

履歴記録トリガー 
CREATE OR REPLACE TRIGGER trg_log_salary
AFTER UPDATE OF salary ON employees  -- 給与列が更新された後
FOR EACH ROW
BEGIN
    -- 履歴テーブルに記録
    INSERT INTO salary_history
    VALUES (seq_history.NEXTVAL,     -- 履歴ID（自動採番）
            :NEW.emp_id,             -- 社員ID
            :OLD.salary,             -- 変更前の給与
            :NEW.salary,             -- 変更後の給与
            SYSDATE);                -- 変更日時

    DBMS_OUTPUT.PUT_LINE('給与変更を記録: ' || :NEW.emp_name ||
                         ' (' || :OLD.salary || '円 → ' || :NEW.salary || '円)');
END;
/

:OLDと:NEWの使い分け

:OLD.salary → 更新前の給与（例: 300000円）
:NEW.salary → 更新後の給与（例: 350000円）
この2つを比較することで、「何がどう変わったか」を記録できます。

ステップ3: 動作確認

動作確認 
-- 出力を有効化
SET SERVEROUTPUT ON;

-- 給与を更新（トリガーが自動実行される）
UPDATE employees SET salary = 350000 WHERE emp_id = 1;
COMMIT;

-- 履歴を確認
SELECT * FROM salary_history;

期待される結果

履歴テーブルに給与変更の記録が自動で追加されています。

HISTORY_ID  EMP_ID  OLD_SALARY  NEW_SALARY  CHANGED_DATE
----------  ------  ----------  ----------  --------------
1           1       300000      350000      2025-12-08

10.6 トリガーの管理 - 確認・無効化・削除

① トリガーの一覧を確認

トリガー一覧 
-- 自分が作成したトリガーの一覧
SELECT trigger_name, table_name, status
FROM user_triggers
ORDER BY trigger_name;

② トリガーを無効化/有効化

無効化・有効化 
-- トリガーを無効化（一時的に動作を止める）
ALTER TRIGGER trg_update_stock DISABLE;

-- トリガーを有効化
ALTER TRIGGER trg_update_stock ENABLE;

③ トリガーを削除

トリガー削除 
-- トリガーを完全に削除
DROP TRIGGER trg_update_stock;

トリガーのテスト時は無効化を活用

トリガーが意図しない動作をする場合、まずDISABLEで無効化してから修正しましょう。
いきなり削除すると、トリガーのコードが失われてしまいます。

10.7 トリガーの注意点とベストプラクティス

① ミューテーション表エラーに注意

トリガー内で、トリガーが作動したテーブル自体をSELECTすると、ミューテーション表エラーが発生します。

悪い例（エラーになる）

CREATE TRIGGER trg_bad_example
AFTER INSERT ON orders
FOR EACH ROW
DECLARE
    v_count NUMBER;
BEGIN
    -- エラー！ ordersテーブルを参照している
    SELECT COUNT(*) INTO v_count FROM orders;
END;

② トリガーは最小限にする

トリガーが多すぎると、データの変更がどのような影響を及ぼすか分かりにくくなります。
本当に自動化が必要な処理だけをトリガーにしましょう。

③ 無限ループに注意

トリガーAがテーブルXを更新し、それがトリガーBを発動させ、トリガーBがテーブルXを更新すると、無限ループになります。

④ デバッグが難しい

トリガーは自動で実行されるため、エラーが発生したときに原因を特定しにくいです。
DBMS_OUTPUT.PUT_LINEでログを出力しながら動作確認しましょう。

トラブルシューティング

トリガーが実行されない

原因: トリガーが無効化されている

解決方法: ALTER TRIGGER トリガー名 ENABLE;で有効化してください。

ORA-04091: 表が変化しています（ミューテーション表エラー）

原因: トリガー内で、トリガーが発動したテーブル自体をSELECTしている

解決方法: :NEWや:OLDを使うか、文レベルトリガー（FOR EACH ROWを削除）に変更してください。

トリガーのエラーメッセージが分かりにくい

原因: トリガーは自動実行されるため、エラー箇所が特定しにくい

解決方法: DBMS_OUTPUT.PUT_LINEでログを出力し、どこまで処理が進んだか確認してください。

理解度確認クイズ

  1. Q1. トリガーとプロシージャの最大の違いは何ですか？

    回答を表示

    プロシージャは手動で呼び出しますが、トリガーはデータの追加・更新・削除時に自動で実行されます。
    トリガーは「引き金」となるイベント（INSERT、UPDATE、DELETE）で発動します。

  2. Q2. :NEWと:OLDの違いは何ですか？

    回答を表示

    :NEWは変更後の値、:OLDは変更前の値を表します。
    例えば、給与を30万円から35万円に更新すると、:OLD.salaryは30万円、:NEW.salaryは35万円になります。

  3. Q3. ミューテーション表エラーはどんなときに発生しますか？

    回答を表示

    トリガー内で、トリガーが発動したテーブル自体をSELECTしたときに発生します。
    例: ordersテーブルのトリガー内でSELECT * FROM ordersを実行するとエラーになります。

第10章のまとめ

  • トリガーは、データ変更時に自動で実行されるPL/SQLプログラム
  • BEFORE/AFTERで実行タイミング、INSERT/UPDATE/DELETEでイベントを指定
  • FOR EACH ROWで行ごとにトリガーを実行できる
  • :NEWは変更後の値、:OLDは変更前の値
  • 在庫自動更新、履歴自動記録など、データ整合性の維持に活用
  • ミューテーション表エラー、無限ループに注意し、最小限の使用を心がける
