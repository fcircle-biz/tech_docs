トランザクションとロック - データの整合性を守る

トランザクション（BEGIN、COMMIT、ROLLBACK）の基本概念と実践的な使い方を学習します。ACID特性、分離レベル、ロックの仕組みを理解し、データの整合性を保つ方法を習得します。

この章で学ぶこと

  • トランザクションの基本概念（BEGIN、COMMIT、ROLLBACK）を理解する
  • ACID特性（原子性、一貫性、独立性、永続性）を把握する
  • トランザクション分離レベルの違いと使い分けを知る
  • ロックの仕組みと同時実行制御について理解する

8.1 トランザクションとは

トランザクションとは、データベースに対する一連の操作（複数のSQLコマンド)をひとまとまりとして扱う仕組みです。 「全ての操作が成功するか、全てが失敗するか」のどちらかだけが保証されます。

例えば、銀行の送金処理を考えてみましょう。 「A口座から1万円引き出す」と「B口座に1万円入金する」という2つの操作があります。 もし「A口座から引き出し」だけ成功して「B口座への入金」が失敗したら、1万円が消えてしまいます。 トランザクションを使えば、両方成功するか、両方失敗するかのどちらかになるため、データの整合性が保たれます。

トランザクションがないと？

トランザクションを使わない場合、各SQL文は実行された直後にデータベースに反映されます（自動コミット）。 複数の操作をまとめて管理できないため、途中でエラーが発生すると中途半端な状態になり、データの不整合が発生します。

8.2 トランザクションの基本操作

PostgreSQLでトランザクションを使うには、以下の3つのコマンドを覚えましょう。

BEGIN

トランザクションの開始を宣言します。この後のSQL文はトランザクション内で実行されます。

COMMIT

トランザクション内の全ての変更を確定し、データベースに反映します。

ROLLBACK

トランザクション内の全ての変更を取り消し、トランザクション開始前の状態に戻します。

基本的な使い方の例

SQL 
-- トランザクション開始
BEGIN;

-- A口座から1万円引き出し
UPDATE accounts SET balance = balance - 10000 WHERE account_id = 'A';

-- B口座に1万円入金
UPDATE accounts SET balance = balance + 10000 WHERE account_id = 'B';

-- 全ての操作が成功したら確定
COMMIT;

エラー時のロールバック例

SQL 
-- トランザクション開始
BEGIN;

-- 商品の在庫を減らす
UPDATE products SET stock = stock - 1 WHERE product_id = 101;

-- 何らかの理由でエラーが発生した場合
-- （例: 在庫が足りない、など）

-- トランザクションを取り消す
ROLLBACK;

自動ロールバック

トランザクション内でSQL実行エラーが発生した場合、PostgreSQLは自動的にトランザクションをエラー状態にします。 その後はCOMMITできなくなり、ROLLBACKする必要があります。

8.3 ACID特性 - トランザクションの4つの性質

トランザクションが保証すべき4つの性質を「ACID特性」と呼びます。 PostgreSQLはこのACID特性を完全にサポートしているため、信頼性の高いデータ管理が可能です。

A

Atomicity（原子性）

トランザクション内の全ての操作が成功するか、全てが失敗するかのどちらかであることを保証します。 「部分的に成功」という状態は発生しません。

例: 送金処理で「引き出し」だけ成功して「入金」が失敗することはない

C

Consistency（一貫性）

トランザクション実行前後で、データベースの整合性が保たれることを保証します。 制約（PRIMARY KEY、FOREIGN KEYなど）に違反する状態にはなりません。

例: 外部キー制約により、存在しない顧客IDを注文テーブルに登録できない

I

Isolation（独立性・分離性）

複数のトランザクションが同時に実行されても、互いに影響を及ぼさないことを保証します。 あるトランザクションの途中経過は、他のトランザクションからは見えません。

例: ユーザーAが商品価格を更新中に、ユーザーBは古い価格か新しい価格のどちらかを見る（中途半端な値は見えない）

D

Durability（永続性）

COMMITされたトランザクションの結果は、システム障害が発生しても永続的に保存されることを保証します。 一度確定したデータは消えません。

例: 注文をCOMMITした直後に停電が発生しても、注文データは失われない

8.4 トランザクション分離レベル

複数のトランザクションが同時に実行される際、どの程度「独立性（Isolation）」を保つかを決めるのが「分離レベル」です。 PostgreSQLでは4つの分離レベルが定義されていますが、実際には3つがサポートされています。

分離レベル ダーティリード ノンリピータブルリード ファントムリード
READ UNCOMMITTED
（PostgreSQLは未サポート）		 発生する 発生する 発生する
READ COMMITTED
（デフォルト）		 防止 発生する 発生する
REPEATABLE READ 防止 防止 発生する
SERIALIZABLE 防止 防止 防止

分離レベルの設定方法

SQL 
-- トランザクション開始時に分離レベルを設定
BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- ここにSQL文を記述
SELECT * FROM products WHERE product_id = 1;

COMMIT;

補足: 問題の種類

  • ダーティリード: 他のトランザクションの未確定の変更が見えてしまう
  • ノンリピータブルリード: 同じクエリを2回実行すると異なる結果が返る（他が更新した）
  • ファントムリード: 同じクエリを2回実行すると行数が変わる（他が追加/削除した）

8.5 ロックの仕組み

複数のユーザーが同時にデータベースを操作する際、データの不整合を防ぐために「ロック」が使われます。 ロックとは、あるトランザクションがデータを使用中は、他のトランザクションがそのデータを変更できないようにする仕組みです。

主なロックの種類

行レベルロック

特定の行だけをロックします。同じテーブルの他の行は別のトランザクションが操作できます。

テーブルレベルロック

テーブル全体をロックします。大規模な操作（テーブル構造変更など）で使用されます。

共有ロック（Shared Lock）

読み取り専用のロック。複数のトランザクションが同時に読み取れますが、更新はできません。

排他ロック（Exclusive Lock）

更新用のロック。ロックしたトランザクションだけが読み書きでき、他は待機します。

明示的なロックの取得

SQL 
BEGIN;

-- 行レベルの排他ロックを取得（更新前に確保）
SELECT * FROM products WHERE product_id = 1 FOR UPDATE;

-- この行は他のトランザクションからロックされる
UPDATE products SET stock = stock - 1 WHERE product_id = 1;

COMMIT;

デッドロックに注意

2つのトランザクションが互いにロックを待ち合う状態を「デッドロック」と呼びます。 PostgreSQLはデッドロックを自動検出し、一方のトランザクションをロールバックして解決します。 デッドロックを防ぐには、複数テーブルを更新する際は常に同じ順序でロックを取得するなどの工夫が必要です。

実習: トランザクションを試してみよう

ステップ1: テスト用テーブルを作成

SQL 
-- 口座テーブルを作成
CREATE TABLE accounts (
    account_id VARCHAR(10) PRIMARY KEY,
    account_name VARCHAR(100) NOT NULL,
    balance INTEGER NOT NULL CHECK (balance >= 0)
);

-- テストデータを挿入
INSERT INTO accounts VALUES ('A', '太郎の口座', 50000);
INSERT INTO accounts VALUES ('B', '花子の口座', 30000);

ステップ2: トランザクションで送金処理

SQL 
-- トランザクション開始
BEGIN;

-- 現在の残高を確認
SELECT * FROM accounts;

-- A口座から10000円引き出し
UPDATE accounts SET balance = balance - 10000 WHERE account_id = 'A';

-- B口座に10000円入金
UPDATE accounts SET balance = balance + 10000 WHERE account_id = 'B';

-- 結果を確認
SELECT * FROM accounts;

-- 確定
COMMIT;

ステップ3: ロールバックを試す

SQL 
BEGIN;

-- A口座から5000円引き出し
UPDATE accounts SET balance = balance - 5000 WHERE account_id = 'A';

-- 残高を確認（まだコミット前）
SELECT * FROM accounts;

-- やっぱり取り消す
ROLLBACK;

-- 残高を確認（元に戻っている）
SELECT * FROM accounts;

理解度確認クイズ

  1. Q1. ACID特性の「A」が表す「Atomicity（原子性）」とはどのような性質ですか？

    回答を表示

    答え: トランザクション内の全ての操作が成功するか、全てが失敗するかのどちらかであることを保証する性質です。
    「部分的に成功」という中途半端な状態は発生しません。例えば、送金処理で「引き出し」だけ成功して「入金」が失敗することはなく、両方成功するか両方失敗するかのどちらかになります。

  2. Q2. PostgreSQLのデフォルトのトランザクション分離レベルは何ですか？

    回答を表示

    答え: READ COMMITTED（リードコミッティド）
    このレベルでは、他のトランザクションがCOMMITした後のデータだけが読み取れます。ダーティリード（未確定データの読み取り）は防止されますが、ノンリピータブルリードやファントムリードは発生する可能性があります。

  3. Q3. 行レベルの排他ロックを取得するSQLコマンドを書いてください。

    回答を表示

    答え:
    SELECT * FROM テーブル名 WHERE 条件 FOR UPDATE;

    例: SELECT * FROM products WHERE product_id = 1 FOR UPDATE;
    このコマンドにより、指定した行に排他ロックがかかり、他のトランザクションはこの行を更新できなくなります。

第8章のまとめ

  • トランザクションは複数の操作をひとまとまりとして扱い、全て成功か全て失敗かを保証する仕組み
  • BEGIN、COMMIT、ROLLBACKの3つのコマンドでトランザクションを制御する
  • ACID特性（原子性、一貫性、独立性、永続性）によりデータの整合性が保たれる
  • 分離レベルにより、同時実行時の独立性の程度を調整できる（デフォルトはREAD COMMITTED）
  • ロック機能により複数ユーザーの同時操作でもデータの不整合を防ぐ
