これまで学んだ知識を活用して、実際のWebアプリケーション（ECサイトやブログシステム）を想定したデータベースを設計・構築します。テーブル設計、インデックス設計、JSON活用、ビュー作成までを通して実践します。

12.1 課題の概要 - ECサイトのデータベースを設計しよう

この章では、シンプルなECサイト（オンラインショップ）のデータベースを設計・構築します。 基本的な機能に絞ることで、実務でよく使われるパターンを理解しやすくします。

設計する機能

  • ユーザー管理（会員登録・ログイン）
  • 商品管理（商品カタログ、在庫管理）
  • カート機能（買い物かご）
  • 注文管理（注文履歴、注文詳細）
  • レビュー機能（商品レビュー・評価）

12.2 データベースとテーブルの作成

実際にデータベースを作成し、テーブルを構築していきましょう。 制約やインデックスも適切に設定します。

Step 1: データベースの作成

まず、ECサイト用のデータベースを作成します。 

-- データベース作成
CREATE DATABASE ec_shop
    ENCODING = 'UTF8'
    LC_COLLATE = 'ja_JP.UTF-8'
    LC_CTYPE = 'ja_JP.UTF-8'
    TEMPLATE = template0;

Step 2: ユーザーテーブルの作成

会員情報を管理するテーブルを作成します。UUID型を主キーとして使用します。 

-- ユーザーテーブル
CREATE TABLE users (
    user_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    name VARCHAR(100) NOT NULL,
    address JSONB, -- 住所情報をJSON形式で格納
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- メールアドレスでの検索を高速化
CREATE INDEX idx_users_email ON users(email);

設計のポイント

UUID型を使用することで、セキュリティとスケーラビリティが向上します。 住所情報はJSONB型で格納することで、柔軟にデータを管理できます（複数配送先など）。

Step 3: カテゴリテーブルの作成

-- カテゴリテーブル
CREATE TABLE categories (
    category_id SERIAL PRIMARY KEY,
    category_name VARCHAR(100) UNIQUE NOT NULL,
    description TEXT
);

Step 4: 商品テーブルの作成

-- 商品テーブル
CREATE TABLE products (
    product_id SERIAL PRIMARY KEY,
    category_id INT REFERENCES categories(category_id),
    product_name VARCHAR(200) NOT NULL,
    description TEXT,
    price DECIMAL(10, 2) NOT NULL CHECK (price >= 0),
    stock_quantity INT NOT NULL DEFAULT 0 CHECK (stock_quantity >= 0),
    specifications JSONB, -- 商品仕様（サイズ、色など）
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- カテゴリIDでの検索用インデックス
CREATE INDEX idx_products_category ON products(category_id);
-- 商品名の部分一致検索用（GINインデックス）
CREATE INDEX idx_products_name ON products USING gin(to_tsvector('japanese', product_name));

Step 5: カートアイテムテーブルの作成

-- カートアイテムテーブル
CREATE TABLE cart_items (
    cart_item_id SERIAL PRIMARY KEY,
    user_id UUID REFERENCES users(user_id) ON DELETE CASCADE,
    product_id INT REFERENCES products(product_id) ON DELETE CASCADE,
    quantity INT NOT NULL CHECK (quantity > 0),
    added_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    -- 同一ユーザー・同一商品は1件のみ
    UNIQUE(user_id, product_id)
);

-- ユーザーごとのカート検索用
CREATE INDEX idx_cart_user ON cart_items(user_id);

Step 6: 注文テーブルと注文明細テーブルの作成

-- 注文テーブル
CREATE TABLE orders (
    order_id SERIAL PRIMARY KEY,
    user_id UUID REFERENCES users(user_id),
    total_amount DECIMAL(10, 2) NOT NULL,
    status VARCHAR(20) DEFAULT 'pending'
        CHECK (status IN ('pending', 'processing', 'shipped', 'delivered', 'cancelled')),
    shipping_address JSONB NOT NULL,
    ordered_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 注文明細テーブル
CREATE TABLE order_items (
    order_item_id SERIAL PRIMARY KEY,
    order_id INT REFERENCES orders(order_id) ON DELETE CASCADE,
    product_id INT REFERENCES products(product_id),
    quantity INT NOT NULL CHECK (quantity > 0),
    unit_price DECIMAL(10, 2) NOT NULL
);

-- インデックス作成
CREATE INDEX idx_orders_user ON orders(user_id);
CREATE INDEX idx_orders_status ON orders(status);
CREATE INDEX idx_order_items_order ON order_items(order_id);

Step 7: レビューテーブルの作成

-- レビューテーブル
CREATE TABLE reviews (
    review_id SERIAL PRIMARY KEY,
    product_id INT REFERENCES products(product_id) ON DELETE CASCADE,
    user_id UUID REFERENCES users(user_id) ON DELETE CASCADE,
    rating INT NOT NULL CHECK (rating BETWEEN 1 AND 5),
    comment TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    -- 1ユーザーが同じ商品に複数レビュー投稿不可
    UNIQUE(product_id, user_id)
);

-- レビュー検索用インデックス
CREATE INDEX idx_reviews_product ON reviews(product_id);
CREATE INDEX idx_reviews_user ON reviews(user_id);

12.4 便利なビューの作成

頻繁に使用するクエリをビューとして作成し、運用を効率化します。

商品一覧ビュー（カテゴリ名付き）

-- 商品一覧ビュー
CREATE VIEW product_catalog AS
SELECT
    p.product_id,
    p.product_name,
    c.category_name,
    p.price,
    p.stock_quantity,
    COALESCE(AVG(r.rating), 0) AS avg_rating,
    COUNT(r.review_id) AS review_count
FROM products p
JOIN categories c ON p.category_id = c.category_id
LEFT JOIN reviews r ON p.product_id = r.product_id
GROUP BY p.product_id, p.product_name, c.category_name, p.price, p.stock_quantity;

ユーザー注文履歴ビュー

-- ユーザー注文履歴ビュー
CREATE VIEW user_order_history AS
SELECT
    o.order_id,
    o.user_id,
    u.name AS user_name,
    o.total_amount,
    o.status,
    o.ordered_at,
    COUNT(oi.order_item_id) AS item_count
FROM orders o
JOIN users u ON o.user_id = u.user_id
JOIN order_items oi ON o.order_id = oi.order_id
GROUP BY o.order_id, o.user_id, u.name, o.total_amount, o.status, o.ordered_at;

売上集計マテリアライズドビュー

集計が重い売上データはマテリアライズドビューで事前計算します。 

-- 売上集計マテリアライズドビュー
CREATE MATERIALIZED VIEW sales_summary AS
SELECT
    p.product_id,
    p.product_name,
    c.category_name,
    SUM(oi.quantity) AS total_sold,
    SUM(oi.quantity * oi.unit_price) AS total_revenue
FROM order_items oi
JOIN products p ON oi.product_id = p.product_id
JOIN categories c ON p.category_id = c.category_id
JOIN orders o ON oi.order_id = o.order_id
WHERE o.status IN ('shipped', 'delivered')
GROUP BY p.product_id, p.product_name, c.category_name
ORDER BY total_revenue DESC;

-- マテリアライズドビューのリフレッシュ
-- REFRESH MATERIALIZED VIEW sales_summary;

12.5 実践的なクエリ例

実際のアプリケーションで使用する典型的なクエリを見ていきましょう。

JSONデータを活用した商品検索

-- 特定の色の商品を検索
SELECT product_name, price, specifications
FROM products
WHERE specifications @> '{"color": ["black"]}';

ユーザーの配送先住所を取得

-- ユーザーの住所情報を取得
SELECT
    name,
    address->>'prefecture' AS prefecture,
    address->>'city' AS city,
    address->>'address1' AS address1
FROM users
WHERE email = 'user1@example.com';

注文処理（トランザクション使用）

-- 注文処理のトランザクション例
BEGIN;

-- 在庫を確認して減算
UPDATE products
SET stock_quantity = stock_quantity - 2
WHERE product_id = 1 AND stock_quantity >= 2;

-- 注文レコード作成
INSERT INTO orders (user_id, total_amount, status, shipping_address)
VALUES (
    (SELECT user_id FROM users WHERE email = 'user1@example.com'),
    179600,
    'pending',
    (SELECT address FROM users WHERE email = 'user1@example.com')
) RETURNING order_id;

-- 注文明細作成
INSERT INTO order_items (order_id, product_id, quantity, unit_price)
VALUES (currval('orders_order_id_seq'), 1, 2, 89800);

COMMIT;

人気商品ランキング（ウィンドウ関数）

-- 売上上位商品をカテゴリ別にランキング
SELECT
    category_name,
    product_name,
    total_revenue,
    RANK() OVER (PARTITION BY category_name ORDER BY total_revenue DESC) AS rank
FROM sales_summary;

12.6 パフォーマンス最適化のポイント

設計したデータベースを本番運用する際の最適化ポイントを確認しましょう。

インデックスの活用

WHERE句やJOIN条件で頻繁に使用されるカラムにはインデックスを作成します。 ただし、書き込みパフォーマンスが低下するため、必要最低限に留めることが重要です。

EXPLAIN ANALYZEの活用

遅いクエリを見つけたら、EXPLAIN ANALYZEで実行計画を確認します。 シーケンシャルスキャンが発生している場合は、インデックス追加を検討しましょう。

マテリアライズドビューの活用

集計処理が重い場合は、マテリアライズドビューで事前計算します。 定期的にREFRESHすることで、データを最新に保ちます。

VACUUMとANALYZE

定期的にVACUUM（不要領域の回収）とANALYZE（統計情報の更新）を実行します。 通常は自動実行されますが、大量データ投入後は手動実行を検討しましょう。 

-- クエリの実行計画を確認
EXPLAIN ANALYZE
SELECT * FROM product_catalog WHERE avg_rating >= 4.0;

-- VACUUM実行
VACUUM ANALYZE products;

理解度確認クイズ

  1. Q1. ユーザーIDにUUID型を使うメリットを2つ挙げてください。

    回答を表示

    答え:
    1. セキュリティ向上 - 連番と異なり、IDから総ユーザー数などが推測できない
    2. スケーラビリティ - 分散システムで一意なIDを生成できる（複数サーバーで同時生成しても重複しない）

  2. Q2. 住所や商品仕様をJSONB型で保存するメリットは何ですか?

    回答を表示

    答え:
    柔軟なデータ構造を持てる点です。住所なら複数配送先、商品仕様なら色やサイズのバリエーションなど、 アプリケーションの要件変更に応じてテーブル構造を変えずに対応できます。 また、JSONB型は効率的に検索・インデックス化が可能です。

  3. Q3. マテリアライズドビューと通常のビューの違いを説明してください。

    回答を表示

    答え:
    通常のビューはクエリの定義を保存するだけで、参照時に毎回実行されます。 マテリアライズドビューは実行結果をディスクに保存するため、参照が高速です。 ただし、データが古くなるため、定期的にREFRESHする必要があります。
