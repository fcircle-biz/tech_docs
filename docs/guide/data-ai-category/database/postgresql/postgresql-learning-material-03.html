PostgreSQLのデータ型 - 豊富なデータ型を使いこなそう

PostgreSQLが誇る豊富なデータ型を学習します。基本型（整数、文字列、日付時刻）に加え、配列型、JSON/JSONB型、UUID型、範囲型など、PostgreSQLならではのデータ型の特徴と使いどころを紹介します。

この章で学ぶこと

  • PostgreSQLの基本的なデータ型（数値型、文字列型、日付時刻型）を理解する
  • 配列型の宣言と使い方を習得する
  • JSON/JSONB型の違いと使い分けを学ぶ
  • UUID型、範囲型などPostgreSQL独自のデータ型を知る

3.1 基本的なデータ型

PostgreSQLは標準SQLに準拠した基本的なデータ型を提供しています。 まずは数値型、文字列型、日付時刻型など、よく使われるデータ型を理解しましょう。

数値型

整数や小数を扱うためのデータ型です。データの範囲や精度に応じて使い分けます。

データ型 説明 範囲 用途
SMALLINT 2バイト整数 -32768 〜 32767 小さな整数値
INTEGER 4バイト整数 約-21億 〜 21億 一般的な整数（最も使用）
BIGINT 8バイト整数 約-922京 〜 922京 大きな整数値（IDなど）
DECIMAL(p,s) 任意精度の小数 可変 金額など正確な計算が必要
REAL 4バイト浮動小数点 約6桁精度 科学計算（精度低い）
DOUBLE PRECISION 8バイト浮動小数点 約15桁精度 科学計算（精度高い）

データ型の選び方

金額計算など正確な小数点計算が必要な場合はDECIMALを使用します。 浮動小数点（REAL、DOUBLE PRECISION）は丸め誤差が発生するため、金額計算には不向きです。 例: DECIMAL(10, 2)は合計10桁、小数点以下2桁を表します。

文字列型

テキストデータを扱うためのデータ型です。PostgreSQLでは主に3種類が使われます。

データ型 説明 特徴
VARCHAR(n) 可変長文字列（最大n文字） nを超える文字列はエラー
CHAR(n) 固定長文字列（n文字） n未満は空白埋め（あまり使わない）
TEXT 可変長文字列（無制限） 長いテキストに最適（推奨）
文字列型の使用例 
-- 文字列型のテーブル例
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    username VARCHAR(50),  -- 最大50文字のユーザー名
    email VARCHAR(255),    -- 最大255文字のメールアドレス
    bio TEXT               -- 無制限の自己紹介文
);

日付時刻型

日付や時刻を扱うためのデータ型です。タイムゾーン対応の有無で使い分けます。

データ型 説明 例
DATE 日付のみ '2025-01-15'
TIME 時刻のみ '14:30:00'
TIMESTAMP 日付+時刻（タイムゾーンなし） '2025-01-15 14:30:00'
TIMESTAMPTZ 日付+時刻（タイムゾーンあり） '2025-01-15 14:30:00+09'

TIMESTAMPTZを推奨

グローバルに展開するWebアプリケーションでは、タイムゾーン付きのTIMESTAMPTZを使うことを推奨します。 PostgreSQLが自動的にクライアントのタイムゾーンに変換してくれるため、世界中のユーザーに正確な時刻を表示できます。

3.2 配列型 - 複数の値を1つのカラムに格納

PostgreSQLの特徴的な機能の一つが配列型です。 1つのカラムに複数の値を配列として格納でき、複雑なデータ構造をシンプルに表現できます。

配列型の宣言

配列型のテーブル作成 
-- 配列型を使ったテーブル作成
CREATE TABLE articles (
    id SERIAL PRIMARY KEY,
    title VARCHAR(200),
    tags TEXT[],           -- 文字列の配列
    view_counts INTEGER[]  -- 整数の配列
);

配列データの挿入

配列データの挿入 
-- 配列データを挿入（ARRAY構文）
INSERT INTO articles (title, tags, view_counts)
VALUES ('PostgreSQL入門', ARRAY['database', 'postgresql', 'sql'], ARRAY[100, 150, 200]);

-- 配列データを挿入（波括弧構文）
INSERT INTO articles (title, tags)
VALUES ('JSON活用術', '{"json", "web", "api"}');

配列データの検索

配列を使った検索 
-- 特定の要素を含む記事を検索
SELECT * FROM articles WHERE 'postgresql' = ANY(tags);

-- 配列の要素数を取得
SELECT title, array_length(tags, 1) AS tag_count FROM articles;

-- 配列の最初の要素を取得（添字は1から始まる）
SELECT title, tags[1] AS first_tag FROM articles;

配列のインデックスは1から

多くのプログラミング言語では配列のインデックスが0から始まりますが、PostgreSQLの配列は1から始まります。 tags[1]で最初の要素を取得できます。

配列型の使いどころ

適している場面

• タグやラベルの管理
• 複数の電話番号やメールアドレス
• 時系列の測定値

適していない場面

• 頻繁に検索・集計する必要がある
• 個別の要素に複雑な属性がある
• 正規化が必要なデータ

3.3 JSON/JSONB型 - 柔軟なデータ構造を扱う

PostgreSQLはJSON型とJSONB型の2種類のJSON型をサポートしています。 これらを使うことで、柔軟なスキーマレスデータを格納・検索できます。

JSONとJSONBの違い

項目 JSON JSONB（推奨）
保存形式 テキストのまま バイナリ形式に変換
書き込み速度 速い やや遅い（変換処理が必要）
読み込み速度 やや遅い 速い
インデックス 作成不可 GINインデックス対応
キーの順序 保持される 保持されない
推奨用途 順序が重要な場合 検索・集計を行う場合

JSONBを推奨

ほとんどの場合、JSONB型を使うことを推奨します。 インデックスが作成でき、検索が高速で、多くの便利な演算子が使えます。

JSONB型の基本操作

JSONB型のテーブル作成とデータ挿入 
-- JSONB型を使ったテーブル作成
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    name VARCHAR(200),
    attributes JSONB  -- 商品の属性をJSONBで格納
);

-- JSONBデータの挿入
INSERT INTO products (name, attributes) VALUES
('ノートPC', '{"cpu": "Core i7", "ram": "16GB", "ssd": "512GB"}'),
('スマートフォン', '{"os": "Android", "screen": "6.5inch", "camera": "48MP"}');
JSONBデータの検索 
-- 特定のキーの値を取得（-> 演算子はJSON型を返す）
SELECT name, attributes->'cpu' AS cpu FROM products;

-- 特定のキーの値をテキストとして取得（->> 演算子はテキストを返す）
SELECT name, attributes->>'ram' AS ram FROM products;

-- 特定の属性を持つ商品を検索
SELECT * FROM products WHERE attributes ? 'cpu';

-- 特定の値を持つ商品を検索
SELECT * FROM products WHERE attributes->>'os' = 'Android';

JSON型の使いどころ

適している場面

  • • 商品の属性（サイズ、色、仕様など）
  • • ユーザー設定やメタデータ
  • • ログやイベントデータ
  • • スキーマが頻繁に変わるデータ

適していない場面

  • • データ構造が明確で固定されている
  • • リレーションシップが重要
  • • データ整合性の制約が必要
  • • 複雑な集計やJOINが頻繁

3.4 PostgreSQL独自のデータ型

PostgreSQLには他のRDBMSにはないユニークなデータ型があります。 これらを活用することで、より効率的で表現力豊かなデータモデリングができます。

UUID型 - 一意な識別子

UUID（Universally Unique Identifier）は、世界中で一意な識別子を生成できるデータ型です。 分散システムで主キーとして使われることが多いです。

UUID型の使用例 
-- UUID拡張を有効化
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- UUID型を使ったテーブル作成
CREATE TABLE orders (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    customer_id UUID,
    order_date TIMESTAMPTZ DEFAULT NOW()
);

-- UUIDの挿入（自動生成される）
INSERT INTO orders (customer_id)
VALUES ('a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11');

UUIDのメリット・デメリット

メリット: 分散システムで衝突なく一意なIDを生成できる、予測不可能なID

デメリット: 16バイトでINTEGER（4バイト）より大きい、インデックスサイズが増える

範囲型 - 期間や範囲を表現

範囲型（Range Types）は、値の範囲を1つのカラムで表現できるデータ型です。 予約システムや在庫管理など、期間を扱う場面で便利です。

データ型 説明 例
INT4RANGE 整数の範囲 '[1,10)'
DATERANGE 日付の範囲 '[2025-01-01,2025-12-31]'
TSTZRANGE タイムスタンプの範囲 '[2025-01-01 00:00,2025-01-02 00:00)'
範囲型の使用例（会議室予約システム） 
-- 範囲型を使った予約テーブル
CREATE TABLE reservations (
    id SERIAL PRIMARY KEY,
    room_name VARCHAR(100),
    reserved_period TSTZRANGE  -- 予約期間
);

-- 予約データの挿入
INSERT INTO reservations (room_name, reserved_period) VALUES
('会議室A', '[2025-01-15 10:00, 2025-01-15 12:00)'),
('会議室B', '[2025-01-15 14:00, 2025-01-15 16:00)');

-- 重複する予約を検索（時間が重なっているか）
SELECT * FROM reservations
WHERE reserved_period && '[2025-01-15 11:00, 2025-01-15 13:00)';

重複チェックが簡単

範囲型を使うと、&&演算子で期間の重複チェックが簡単にできます。 従来のstart_date/end_dateの2カラムで管理するより、クエリがシンプルで効率的です。

その他の便利なデータ型

BOOLEAN型

TRUE / FALSE / NULLの3値を扱います。 フラグやステータスの管理に便利です。

INET型 / CIDR型

IPアドレスやネットワークアドレスを扱う専用のデータ型です。 ネットワーク範囲の検索やマスク計算が簡単にできます。

MONEY型

通貨を扱う専用のデータ型です。ただし、多通貨対応が難しいため、 実務ではDECIMAL型と通貨コードカラムを使う方が一般的です。

実習 3-1: 様々なデータ型を試してみよう

PostgreSQL独自のデータ型を実際に使ってみましょう。 商品カタログを管理するテーブルを作成し、配列型とJSONB型を活用します。

手順

  1. 1 pgAdminを開き、practice_dbデータベースに接続
  2. 2 商品テーブルを作成（配列型とJSONB型を含む）
  3. 3 サンプルデータを挿入し、検索クエリを実行して確認
実習用SQL 
-- 商品カタログテーブル作成
CREATE TABLE product_catalog (
    id SERIAL PRIMARY KEY,
    name VARCHAR(200),
    tags TEXT[],           -- 商品タグ（配列型）
    specs JSONB,           -- 商品仕様（JSONB型）
    price DECIMAL(10, 2)
);

-- サンプルデータ挿入
INSERT INTO product_catalog (name, tags, specs, price) VALUES
('ノートPC X1',
 ARRAY['electronics', 'computer', 'laptop'],
 '{"cpu": "Core i7", "ram": "16GB", "storage": "512GB SSD"}',
 129800.00),
('スマートフォン Y2',
 ARRAY['electronics', 'mobile', 'android'],
 '{"os": "Android 13", "screen": "6.5inch", "camera": "50MP"}',
 89800.00);

-- タグで検索
SELECT name FROM product_catalog WHERE 'laptop' = ANY(tags);

-- JSONB型から特定の値を検索
SELECT name, specs->>'cpu' AS cpu
FROM product_catalog
WHERE specs->>'cpu' = 'Core i7';

理解度確認クイズ

  1. Q1. 金額計算に適したデータ型はどれですか？また、その理由を説明してください。

    回答を表示

    答え: DECIMAL型（またはNUMERIC型）
    理由: DECIMAL型は任意精度の小数を扱え、丸め誤差が発生しません。 REAL型やDOUBLE PRECISION型は浮動小数点のため丸め誤差が発生し、金額計算には不向きです。 例: DECIMAL(10, 2)は最大10桁で小数点以下2桁を表します。

  2. Q2. JSON型とJSONB型の主な違いを3つ挙げてください。

    回答を表示

    答え:
    1. 保存形式: JSON型はテキストのまま、JSONB型はバイナリ形式に変換して保存
    2. インデックス: JSON型はインデックス作成不可、JSONB型はGINインデックスが作成可能
    3. 検索速度: JSON型は毎回パースが必要で遅い、JSONB型は検索が高速
    （他にも、書き込み速度、キーの順序保持などの違いがあります）

  3. Q3. 配列型でタグ「postgresql」を含む記事を検索するSQLを書いてください。

    回答を表示

    答え:
    SELECT * FROM articles WHERE 'postgresql' = ANY(tags);
    解説: ANY演算子を使うと、配列の中に指定した値が含まれるかを簡単にチェックできます。

第3章のまとめ

  • PostgreSQLは基本型（数値、文字列、日付時刻）に加え、豊富な独自データ型を提供
  • 配列型を使うと複数の値を1つのカラムに格納でき、タグやラベル管理に便利
  • JSONB型を使うと柔軟なスキーマレスデータを効率的に扱え、インデックスも作成可能
  • UUID型、範囲型などの独自データ型を活用することで、より表現力豊かなデータモデリングが可能
