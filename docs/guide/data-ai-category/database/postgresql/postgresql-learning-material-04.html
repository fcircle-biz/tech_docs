PostgreSQLのテーブル設計 - 制約とインデックスの基礎

PostgreSQLでのテーブル作成と各種制約（PRIMARY KEY、FOREIGN KEY、UNIQUE、CHECK、NOT NULL）の設定方法を学習します。インデックスの基本概念と作成方法、パフォーマンスへの影響について理解します。

この章で学ぶこと

  • PostgreSQLでのテーブル作成の基本構文を理解する
  • 各種制約（PRIMARY KEY、FOREIGN KEY、UNIQUE、CHECK、NOT NULL）の役割と設定方法を学ぶ
  • インデックスの基本概念とパフォーマンスへの影響を理解する
  • 適切なインデックスの作成方法を習得する

4.1 テーブル作成の基本

PostgreSQLでのテーブル作成は、CREATE TABLE文を使用します。 テーブル設計では、適切なデータ型の選択と制約の設定が重要です。 まずは基本的なテーブル作成から始めましょう。

基本構文

基本構文 
-- 基本的なテーブル作成
CREATE TABLE テーブル名 (
    列名1 データ型 制約,
    列名2 データ型 制約,
    ...
    テーブル制約
);

具体例: 社員テーブル

employee.sql 
-- 社員テーブルの作成
CREATE TABLE employees (
    employee_id SERIAL,           -- 連番（自動採番）
    name VARCHAR(100),            -- 名前（最大100文字）
    email VARCHAR(255),           -- メールアドレス
    birth_date DATE,              -- 生年月日
    hire_date DATE,               -- 入社日
    salary NUMERIC(10, 2),        -- 給与（10桁、小数点以下2桁）
    department_id INTEGER         -- 部署ID
);

SERIAL型について

SERIAL型は、PostgreSQL独自の型で、自動的に連番を生成します。 内部的には整数型（INTEGER）のシーケンスが作成され、INSERT時に自動で値が割り当てられます。 主キーとして使用されることが多い型です。

4.2 制約の種類と使い方

制約は、データベースの整合性を保つために重要な機能です。 不正なデータの登録を防ぎ、データの品質を維持します。

flowchart TD A[データベース制約] --> B[PRIMARY KEY] A --> C[FOREIGN KEY] A --> D[UNIQUE] A --> E[CHECK] A --> F[NOT NULL] B --> B1["一意性を保証
テーブルの主キー"] C --> C1["参照整合性を保証
他テーブルとの関連"] D --> D1["重複を防止
一意な値を保証"] E --> E1["値の範囲を制限
条件式で検証"] F --> F1["NULL値を禁止
必須項目に設定"]

PRIMARY KEY（主キー）

PRIMARY KEYは、テーブル内の各行を一意に識別するための制約です。 PRIMARY KEYを設定すると、自動的にNOT NULLとUNIQUE制約も適用されます。

PRIMARY KEY設定例 
-- 方法1: 列定義時に指定
CREATE TABLE employees (
    employee_id SERIAL PRIMARY KEY,  -- 主キー指定
    name VARCHAR(100) NOT NULL
);

-- 方法2: テーブル制約として指定
CREATE TABLE employees (
    employee_id SERIAL,
    name VARCHAR(100) NOT NULL,
    PRIMARY KEY (employee_id)  -- テーブル制約として指定
);

-- 方法3: 複合主キー（複数列の組み合わせ）
CREATE TABLE order_details (
    order_id INTEGER,
    product_id INTEGER,
    quantity INTEGER,
    PRIMARY KEY (order_id, product_id)  -- 2列の組み合わせで一意
);

FOREIGN KEY（外部キー）

FOREIGN KEYは、他のテーブルとの関連性を定義し、参照整合性を保証します。 親テーブルに存在しない値を子テーブルに登録することを防ぎます。

FOREIGN KEY設定例 
-- 部署テーブル（親テーブル）
CREATE TABLE departments (
    department_id SERIAL PRIMARY KEY,
    department_name VARCHAR(100) NOT NULL
);

-- 社員テーブル（子テーブル）
CREATE TABLE employees (
    employee_id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    department_id INTEGER,
    FOREIGN KEY (department_id)
        REFERENCES departments(department_id)
);

参照動作の指定

FOREIGN KEYには、親テーブルのデータが削除・更新された時の動作を指定できます。

参照動作の指定 
CREATE TABLE employees (
    employee_id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    department_id INTEGER,
    FOREIGN KEY (department_id)
        REFERENCES departments(department_id)
        ON DELETE CASCADE      -- 親削除時、子も削除
        ON UPDATE CASCADE      -- 親更新時、子も更新
);
参照動作 説明
CASCADE 親行が削除・更新されると、子行も削除・更新される
RESTRICT 子行が存在する場合、親行の削除・更新を拒否（デフォルト）
SET NULL 親行が削除・更新されると、子行の外部キーをNULLに設定
SET DEFAULT 親行が削除・更新されると、子行の外部キーをデフォルト値に設定
NO ACTION RESTRICTと同様だが、チェックのタイミングが異なる

UNIQUE制約

UNIQUE制約は、列の値が一意であることを保証します。PRIMARY KEYと異なり、NULL値を許可できます。

UNIQUE制約設定例 
CREATE TABLE employees (
    employee_id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(255) UNIQUE,     -- メールアドレスは一意
    phone VARCHAR(20) UNIQUE       -- 電話番号も一意
);

-- 複数列の組み合わせで一意性を保証
CREATE TABLE user_roles (
    user_id INTEGER,
    role_id INTEGER,
    UNIQUE (user_id, role_id)  -- ユーザーとロールの組み合わせで一意
);

CHECK制約

CHECK制約は、列の値が特定の条件を満たすことを保証します。値の範囲や形式を制限できます。

CHECK制約設定例 
CREATE TABLE employees (
    employee_id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    age INTEGER CHECK (age >= 18 AND age <= 65),  -- 年齢は18歳以上65歳以下
    salary NUMERIC(10, 2) CHECK (salary > 0),     -- 給与は正の数
    email VARCHAR(255) CHECK (email LIKE '%@%')   -- メールには@が必要
);

-- 複数列にまたがる制約
CREATE TABLE orders (
    order_id SERIAL PRIMARY KEY,
    order_date DATE,
    ship_date DATE,
    CHECK (ship_date >= order_date)  -- 発送日は注文日以降
);

NOT NULL制約

NOT NULL制約は、列にNULL値を許可しないことを指定します。必須項目に設定します。

NOT NULL制約設定例 
CREATE TABLE employees (
    employee_id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,        -- 名前は必須
    email VARCHAR(255) NOT NULL,       -- メールアドレスは必須
    phone VARCHAR(20),                 -- 電話番号は任意（NULL可）
    hire_date DATE NOT NULL            -- 入社日は必須
);

制約設計のベストプラクティス

制約は「データベース側で保証する」ことが重要です。アプリケーション側だけで検証すると、 直接SQLで操作された場合に不正なデータが入る可能性があります。 必須項目にはNOT NULL、一意性が必要な列にはUNIQUEまたはPRIMARY KEY、 値の範囲制限にはCHECK、テーブル間の関連にはFOREIGN KEYを適切に設定しましょう。

4.3 インデックスの基本

インデックスは、データベースの検索速度を向上させるための仕組みです。 本の索引のように、特定のデータを素早く見つけるための目次の役割を果たします。

あり?} B -->|あり| C[インデックスを使用
高速検索] B -->|なし| D[全行スキャン
フルスキャン] C --> E[結果返却
速い] D --> F[結果返却
遅い] style C fill:#d1fae5 style E fill:#d1fae5 style D fill:#fee2e2 style F fill:#fee2e2

インデックスの作成

CREATE INDEX文を使用してインデックスを作成します。

インデックス作成例 
-- 基本的なインデックス作成
CREATE INDEX idx_employee_name ON employees(name);

-- 複数列のインデックス（複合インデックス）
CREATE INDEX idx_employee_dept_name
    ON employees(department_id, name);

-- ユニークインデックス
CREATE UNIQUE INDEX idx_employee_email
    ON employees(email);

-- 部分インデックス（条件付き）
CREATE INDEX idx_active_employees
    ON employees(name) WHERE active = true;

インデックスを使用すべき場合

インデックスが効果的

  • • WHERE句で頻繁に検索される列
  • • JOIN条件に使われる列
  • • ORDER BY句で使われる列
  • • FOREIGN KEYの列
  • • 大量データから少数を検索

インデックスが不要

  • • 小規模テーブル（数百行程度）
  • • 頻繁に更新される列
  • • カーディナリティが低い列
  • • ほとんどの行を返す検索
  • • INSERT/UPDATE中心の処理

カーディナリティとは

カーディナリティは「値の種類の多さ」を表します。 例えば、性別（男/女）はカーディナリティが低く、メールアドレスはカーディナリティが高いです。 カーディナリティが高い列ほど、インデックスの効果が大きくなります。

インデックスの確認

インデックスの確認方法 
-- テーブルのインデックス一覧を確認
SELECT
    indexname,
    indexdef
FROM pg_indexes
WHERE tablename = 'employees';

-- インデックスのサイズを確認
SELECT
    indexrelname,
    pg_size_pretty(pg_relation_size(indexrelid))
FROM pg_stat_user_indexes
WHERE schemaname = 'public';

インデックスの削除

インデックスの削除 
-- インデックスを削除
DROP INDEX idx_employee_name;

-- 存在する場合のみ削除
DROP INDEX IF EXISTS idx_employee_name;

インデックスの注意点

インデックスは検索を高速化しますが、デメリットもあります。 ①ディスク容量を消費する、②INSERT/UPDATE/DELETE時の処理が遅くなる、③メンテナンスが必要。 「とりあえず全ての列にインデックス」は避け、本当に必要な列だけに設定しましょう。

4.4 実習: ECサイトのテーブル設計

これまで学んだ制約とインデックスを活用して、簡単なECサイトのテーブルを設計してみましょう。

実習 4-1: ECサイトのテーブル作成

商品テーブル、カテゴリテーブル、注文テーブルを作成し、適切な制約とインデックスを設定します。

手順

  1. 1 pgAdminを起動し、新しいデータベース「ec_db」を作成
  2. 2 カテゴリテーブル、商品テーブル、注文テーブルを順に作成
  3. 3 サンプルデータを挿入して、制約が正しく動作するか確認
ec_tables.sql 
-- カテゴリテーブル
CREATE TABLE categories (
    category_id SERIAL PRIMARY KEY,
    category_name VARCHAR(100) NOT NULL UNIQUE,
    description TEXT
);

-- 商品テーブル
CREATE TABLE products (
    product_id SERIAL PRIMARY KEY,
    product_name VARCHAR(200) NOT NULL,
    category_id INTEGER NOT NULL,
    price NUMERIC(10, 2) NOT NULL CHECK (price >= 0),
    stock INTEGER NOT NULL CHECK (stock >= 0),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (category_id) REFERENCES categories(category_id)
);

-- 注文テーブル
CREATE TABLE orders (
    order_id SERIAL PRIMARY KEY,
    product_id INTEGER NOT NULL,
    quantity INTEGER NOT NULL CHECK (quantity > 0),
    order_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (product_id) REFERENCES products(product_id)
);

-- インデックスの作成
CREATE INDEX idx_products_category ON products(category_id);
CREATE INDEX idx_products_name ON products(product_name);
CREATE INDEX idx_orders_date ON orders(order_date);

動作確認

以下のSQLを実行して、制約が正しく動作するか確認しましょう。

    p.product_name,
    c.category_name,
    p.price,
    p.stock
FROM products p
JOIN categories c ON p.category_id = c.category_id;

VALUES ('テスト商品', 1, -100, 10);  -- CHECK制約違反

理解度確認クイズ

  1. Q1. PRIMARY KEYとUNIQUE制約の違いは何ですか?

    回答を表示

    答え:
    主な違いは3つあります。
    ①PRIMARY KEYは1テーブルに1つだけ、UNIQUEは複数設定可能
    ②PRIMARY KEYは自動的にNOT NULL、UNIQUEはNULL値を許可可能
    ③PRIMARY KEYはテーブルの主キーとして自動的にインデックスが作成される

  2. Q2. FOREIGN KEYの参照動作で「ON DELETE CASCADE」とはどういう意味ですか?

    回答を表示

    答え:
    親テーブルのレコードが削除された時に、それを参照している子テーブルのレコードも自動的に削除される動作です。 例えば、部署テーブルのレコードを削除すると、その部署に所属する社員レコードも一緒に削除されます。 データの整合性は保たれますが、意図しないデータ削除を招く可能性もあるため、慎重に使用する必要があります。

  3. Q3. インデックスを作成すべき列の特徴を3つ挙げてください。

    回答を表示

    答え（例）:
    ①WHERE句で頻繁に検索条件として使われる列
    ②JOIN条件で使われる列（特にFOREIGN KEY）
    ③カーディナリティが高い列（値の種類が多い列）
    （他にも、ORDER BY句で使われる列、大量データから少数の行を検索する列なども正解です）
