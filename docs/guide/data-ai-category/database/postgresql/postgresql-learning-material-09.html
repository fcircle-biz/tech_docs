PostgreSQLのインデックス詳解 - クエリを高速化する

B-treeインデックス、GINインデックス、GiSTインデックスなど、PostgreSQLが提供する様々なインデックスの種類と使い分けを学習します。EXPLAINコマンドでクエリの実行計画を分析する方法も習得します。

この章で学ぶこと

  • インデックスの基本概念と動作原理を理解する
  • B-tree、GIN、GiST、BRIN、Hashなど各種インデックスの特徴と使い分けを習得する
  • EXPLAINコマンドを使ったクエリ実行計画の読み方を理解する
  • 部分インデックスや複合インデックスの効果的な活用方法を習得する

9.1 インデックスとは何か

インデックスは、データベース内のデータを高速に検索するための補助的なデータ構造です。 本棚の索引や辞書の見出しに似ており、データを効率的に探すための「目印」を提供します。

インデックスがない場合、PostgreSQLはテーブル内のすべての行を順番に調べる「フルスキャン」を行います。 これは小さなテーブルでは問題ありませんが、数百万行のテーブルでは非常に時間がかかります。 適切なインデックスを作成することで、必要なデータに直接アクセスできるようになり、クエリのパフォーマンスが劇的に向上します。

インデックスの基本原理

インデックスは検索を速くする代わりに、追加のディスク容量とデータ挿入・更新時のオーバーヘッドが発生します。 すべてのカラムにインデックスを作成すればよいというものではなく、検索頻度やデータの特性を考慮して適切に設計する必要があります。

9.2 B-treeインデックス（デフォルト）

B-tree（Balanced Tree）は、PostgreSQLのデフォルトのインデックスタイプです。 ほとんどの一般的な検索条件に対応でき、等号比較（=）、範囲検索（<、>、BETWEEN）、並び替え（ORDER BY）などで効率的に動作します。

SQL 
-- 基本的なB-treeインデックスの作成
-- ユーザーテーブルのメールアドレスにインデックスを作成
CREATE INDEX idx_users_email ON users(email);

-- 複数カラムの複合インデックス
-- 注文テーブルのユーザーIDと注文日時にインデックスを作成
CREATE INDEX idx_orders_user_date ON orders(user_id, order_date);

-- ユニークインデックス（重複を許さない）
CREATE UNIQUE INDEX idx_products_code ON products(product_code);

B-treeが適している場面

  • • 等号検索: WHERE user_id = 100
  • • 範囲検索: WHERE age BETWEEN 20 AND 30
  • • パターンマッチ（前方一致）: WHERE name LIKE 'Tanaka%'
  • • 並び替え: ORDER BY created_at DESC
  • • MIN/MAXの取得

9.3 GINインデックス（配列・JSON・全文検索）

GIN（Generalized Inverted Index）は、配列、JSON/JSONB、全文検索など、複数の値を含むデータに対して効率的に動作するインデックスです。 PostgreSQLの高度な機能を活用する際に重要なインデックスタイプです。

SQL 
-- 配列型カラムへのGINインデックス
-- タグ検索を高速化
CREATE INDEX idx_articles_tags ON articles USING GIN(tags);

-- JSONBカラムへのGINインデックス
-- JSON内の任意のキーを高速検索
CREATE INDEX idx_products_attrs ON products USING GIN(attributes);

-- 全文検索用のGINインデックス
-- 日本語テキストの全文検索を高速化
CREATE INDEX idx_documents_search ON documents
USING GIN(to_tsvector('japanese', content));

GINの利点

  • • 複数値の高速検索
  • • JSON内の部分一致検索
  • • 全文検索のパフォーマンス向上

GINの注意点

  • • インデックスサイズが大きい
  • • 更新時のオーバーヘッド
  • • B-treeより作成に時間がかかる
SQL - GINインデックスの使用例 
-- 配列型の検索（タグ検索）
SELECT * FROM articles
WHERE tags @> ARRAY['PostgreSQL', 'Database'];

-- JSONBの検索
SELECT * FROM products
WHERE attributes @> '{"color": "red", "size": "L"}';

-- 全文検索
SELECT * FROM documents
WHERE to_tsvector('japanese', content) @@ to_tsquery('japanese', 'データベース');

9.4 その他のインデックスタイプ

PostgreSQLには、B-tree、GIN以外にも特定の用途に特化したインデックスタイプがあります。 データの特性とクエリパターンに応じて適切なタイプを選択しましょう。

GiST（地理空間・範囲検索）

GiST（Generalized Search Tree）は、地理空間データ、範囲型、テキスト検索など、複雑なデータ構造の検索に対応したインデックスです。

使用例:

-- 地理空間検索用
CREATE INDEX idx_locations_geom ON locations USING GIST(geom);

-- 範囲型の重なり検索
CREATE INDEX idx_events_period ON events USING GIST(event_period);

BRIN（大規模テーブル・時系列データ）

BRIN（Block Range Index）は、物理的に順序付けられた大規模テーブルに対して、非常に小さなサイズで効率的な範囲検索を提供します。 時系列データやログデータに最適です。

使用例:

-- 時系列データの範囲検索用（非常に小さいインデックス）
CREATE INDEX idx_logs_timestamp ON access_logs USING BRIN(timestamp);

Hash（等号検索専用）

Hashインデックスは等号検索（=）専用のインデックスです。 以前はWALログに記録されず信頼性が低かったですが、PostgreSQL 10以降は改善され実用的になりました。 ただし、ほとんどの場合はB-treeで十分です。

使用例:

-- 等号検索のみの場合
CREATE INDEX idx_users_uuid ON users USING HASH(user_uuid);
タイプ 主な用途 サイズ 推奨度
B-tree 一般的な検索、範囲検索、並び替え 中 デフォルト
GIN 配列、JSON、全文検索 大 高機能データ
GiST 地理空間、範囲型、近傍検索 中 特定用途
BRIN 時系列、大規模テーブル 極小 大規模データ
Hash 等号検索のみ 中 稀

9.5 EXPLAINコマンド - 実行計画の分析

EXPLAINコマンドは、PostgreSQLがクエリをどのように実行するかを表示する非常に重要なツールです。 インデックスが使われているか、フルスキャンが発生していないかを確認し、パフォーマンス問題を特定できます。

SQL - EXPLAIN基本構文 
-- 基本的なEXPLAIN（実行計画を表示、クエリは実行しない）
EXPLAIN SELECT * FROM users WHERE email = 'test@example.com';

-- EXPLAIN ANALYZE（実際にクエリを実行して詳細な統計を表示）
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 100;

-- より詳細な情報を表示
EXPLAIN (ANALYZE, BUFFERS, VERBOSE)
SELECT u.name, COUNT(o.id)
FROM users u
JOIN orders o ON u.id = o.user_id
GROUP BY u.name;

実習: EXPLAINで実行計画を確認しよう

ステップ1: サンプルテーブル作成

-- テーブル作成
CREATE TABLE customers (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100),
    age INT,
    city VARCHAR(50)
);

-- サンプルデータ挿入（10万件）
INSERT INTO customers (name, email, age, city)
SELECT
    'Customer ' || i,
    'user' || i || '@example.com',
    20 + (i % 60),
    CASE (i % 5)
        WHEN 0 THEN '東京'
        WHEN 1 THEN '大阪'
        WHEN 2 THEN '名古屋'
        WHEN 3 THEN '福岡'
        ELSE '札幌'
    END
FROM generate_series(1, 100000) AS i;

ステップ2: インデックスなしでクエリを実行

EXPLAIN ANALYZE
SELECT * FROM customers WHERE email = 'user50000@example.com';

-- 結果例: Seq Scan（順次スキャン = 遅い）
-- Seq Scan on customers (actual time=10.2..25.4 rows=1)
-- 全行をスキャンしているため時間がかかる

ステップ3: インデックスを作成して再実行

-- インデックス作成
CREATE INDEX idx_customers_email ON customers(email);

-- 再度実行
EXPLAIN ANALYZE
SELECT * FROM customers WHERE email = 'user50000@example.com';

-- 結果例: Index Scan（インデックススキャン = 速い）
-- Index Scan using idx_customers_email (actual time=0.05..0.06 rows=1)
-- インデックスを使って直接アクセス！

主な実行計画のタイプ

  • Index Scan インデックスを使った高速検索
  • Seq Scan 全行スキャン（遅い）
  • Bitmap Scan 複数インデックスの組み合わせ
  • Hash Join ハッシュ表を使った結合

コスト値の見方

cost=0.00..100.00

開始コスト..完了コスト（小さいほど速い）

rows=1000

推定行数

actual time=0.5..10.2

実際の実行時間（ミリ秒）

9.6 複合インデックスと部分インデックス

より高度なインデックス設計テクニックとして、複合インデックスと部分インデックスがあります。 これらを適切に活用することで、インデックスのサイズを抑えながら検索パフォーマンスを最大化できます。

複合インデックス（複数カラム）

複合インデックスは、複数のカラムを組み合わせたインデックスです。 カラムの順序が重要で、最初のカラムから順番に条件を指定するクエリで効果を発揮します。

SQL - 複合インデックス 
-- 複合インデックスの作成（user_id, created_atの順）
CREATE INDEX idx_orders_user_created ON orders(user_id, created_at);

-- このクエリは複合インデックスを使える
SELECT * FROM orders
WHERE user_id = 100 AND created_at > '2024-01-01';

-- このクエリも複合インデックスを使える（最初のカラムのみ）
SELECT * FROM orders WHERE user_id = 100;

-- このクエリは複合インデックスを使えない（2番目のカラムのみ）
SELECT * FROM orders WHERE created_at > '2024-01-01';

複合インデックスの順序の決め方

一般的には、「等号条件（=）のカラム → 範囲条件（<、>）のカラム → 並び替え（ORDER BY）のカラム」の順で配置すると効果的です。 また、選択性が高い（値の種類が多い）カラムを先に配置するのも有効です。

部分インデックス（条件付きインデックス）

部分インデックスは、WHERE句で条件を指定して、テーブルの一部の行だけにインデックスを作成する機能です。 インデックスサイズを削減しながら、特定の検索条件に最適化できます。

SQL - 部分インデックス 
-- アクティブなユーザーだけにインデックスを作成
CREATE INDEX idx_users_active ON users(email)
WHERE is_active = true;

-- 未完了の注文だけにインデックスを作成
CREATE INDEX idx_orders_pending ON orders(user_id, created_at)
WHERE status = 'pending';

-- NULL以外の値だけにインデックスを作成
CREATE INDEX idx_products_discount ON products(discount_rate)
WHERE discount_rate IS NOT NULL;

9.7 インデックスのメンテナンスとベストプラクティス

インデックスは作成して終わりではありません。 適切なメンテナンスとベストプラクティスに従うことで、長期的に高いパフォーマンスを維持できます。

インデックスの再構築（REINDEX）

時間が経つとインデックスが肥大化したり、断片化したりします。 REINDEXコマンドでインデックスを再構築することで、パフォーマンスを回復できます。

SQL - インデックスのメンテナンス 
-- 特定のインデックスを再構築
REINDEX INDEX idx_users_email;

-- テーブルの全インデックスを再構築
REINDEX TABLE users;

-- データベース全体のインデックスを再構築（注意: 時間がかかる）
REINDEX DATABASE mydb;

インデックスの統計情報更新（ANALYZE）

PostgreSQLのクエリプランナーは、統計情報をもとに最適な実行計画を選択します。 データが大きく変化したら、ANALYZEコマンドで統計情報を更新しましょう。

SQL - 統計情報の更新 
-- 特定のテーブルの統計情報を更新
ANALYZE users;

-- データベース全体の統計情報を更新
ANALYZE;

インデックス設計のベストプラクティス

  • 頻繁に検索されるカラムにインデックスを作成: WHERE句やJOIN条件でよく使われるカラムが対象
  • 選択性の高いカラムを優先: 値の種類が多い（ユニークに近い）カラムほど効果が高い
  • 不要なインデックスは削除: インデックスは更新時のオーバーヘッドになる
  • EXPLAINで効果を確認: 作成したインデックスが実際に使われているか検証する
  • 定期的にメンテナンス: REINDEX、ANALYZE、VACUUMを定期実行

よくあるアンチパターン

  • • すべてのカラムにインデックスを作成する（更新が遅くなる）
  • • 複合インデックスの順序を間違える（インデックスが使われない）
  • • EXPLAINで確認せずにインデックスを作成する（効果不明）
  • • 小さなテーブルにインデックスを作成する（オーバーヘッドの方が大きい）

動作確認: インデックスの効果を測定しよう

1. 大量データのテーブルを作成

CREATE TABLE performance_test (
    id SERIAL PRIMARY KEY,
    user_id INT,
    product_id INT,
    purchase_date DATE,
    amount NUMERIC(10, 2)
);

-- 100万件のサンプルデータ挿入
INSERT INTO performance_test (user_id, product_id, purchase_date, amount)
SELECT
    (random() * 10000)::INT,
    (random() * 1000)::INT,
    CURRENT_DATE - (random() * 365)::INT,
    (random() * 10000)::NUMERIC(10, 2)
FROM generate_series(1, 1000000);

2. インデックスなしでクエリを実行

EXPLAIN ANALYZE
SELECT * FROM performance_test WHERE user_id = 5000;

-- 実行時間をメモしておく（例: 150ms）

3. インデックスを作成して再実行

CREATE INDEX idx_perf_user ON performance_test(user_id);

EXPLAIN ANALYZE
SELECT * FROM performance_test WHERE user_id = 5000;

-- 実行時間を比較（例: 5ms → 約30倍高速化！）

4. 複合インデックスの効果を確認

CREATE INDEX idx_perf_user_date
ON performance_test(user_id, purchase_date);

EXPLAIN ANALYZE
SELECT * FROM performance_test
WHERE user_id = 5000 AND purchase_date > '2024-01-01';

理解度確認クイズ

  1. Q1. PostgreSQLのデフォルトのインデックスタイプは何ですか？どのような検索に適していますか？

    回答を表示

    答え: B-treeインデックス
    等号検索（=）、範囲検索（<、>、BETWEEN）、前方一致検索（LIKE 'text%'）、並び替え（ORDER BY）、MIN/MAX取得など、ほとんどの一般的な検索に適しています。

  2. Q2. JSON/JSONBカラムの検索を高速化するには、どのタイプのインデックスを使うべきですか？

    回答を表示

    答え: GINインデックス
    GINインデックスは、JSON/JSONB型、配列型、全文検索など、複数の値を含むデータに対して効率的に動作します。 JSON内の任意のキーを高速に検索できます。

  3. Q3. EXPLAINコマンドで「Seq Scan」が表示されました。これは何を意味しますか？

    回答を表示

    答え: 順次スキャン（Sequential Scan）を意味し、テーブルの全行を順番にスキャンしていることを示します。
    大量データの場合は遅くなるため、適切なインデックスを作成することで「Index Scan」に変更できないか検討すべきです。 ただし、小さなテーブルや条件に合う行が多い場合は、Seq Scanの方が効率的なこともあります。

  4. Q4. 複合インデックス `CREATE INDEX idx ON orders(user_id, order_date)` は、どのクエリで使われますか？

    回答を表示

    答え:
    使われる: `WHERE user_id = 100`, `WHERE user_id = 100 AND order_date > '2024-01-01'`
    使われない: `WHERE order_date > '2024-01-01'`（2番目のカラムのみ）
    複合インデックスは、最初のカラムから順番に条件を指定するクエリで効果を発揮します。

第9章のまとめ

  • インデックスはデータを高速に検索するための補助的なデータ構造で、適切に使うことでクエリパフォーマンスが劇的に向上する
  • B-tree（デフォルト）、GIN（JSON・配列）、GiST（地理空間）、BRIN（時系列）、Hash（等号検索）など、用途に応じた選択が重要
  • EXPLAINコマンドでクエリの実行計画を分析し、インデックスが効果的に使われているか確認できる
  • 複合インデックスはカラムの順序が重要で、部分インデックスは条件付きでインデックスサイズを削減できる
  • REINDEX、ANALYZE、VACUUMなどのメンテナンスを定期的に実行し、長期的なパフォーマンスを維持する
