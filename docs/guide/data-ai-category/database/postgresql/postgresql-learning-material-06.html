JSONデータの操作 - モダンなデータ形式を扱う

PostgreSQLの強力なJSON/JSONB型の活用方法を学習します。JSONデータの格納、検索、更新、JSONパス式による要素の抽出など、Webアプリケーション開発で頻出するJSONデータの操作を実践します。

この章で学ぶこと

  • JSON型とJSONB型の違いと使い分けを理解する
  • JSONデータの格納と基本的な検索操作を習得する
  • JSON演算子とJSONパス式を使った要素の抽出ができる
  • JSONデータの更新・削除・結合など応用操作を実践する

6.1 JSON型とJSONB型の違い

PostgreSQLは、JSONデータを格納するために「JSON型」と「JSONB型」の2種類のデータ型を提供しています。 どちらもJSONデータを扱えますが、内部的な処理方法やパフォーマンス特性が異なります。

項目 JSON型 JSONB型
保存形式 テキストとして保存（入力そのまま） バイナリ形式で保存（解析済み）
挿入速度 高速（変換不要） やや低速（変換処理あり）
検索速度 やや低速（毎回解析） 高速（解析済み）
インデックス 未対応 GINインデックス対応
キーの順序 保持される 保持されない（再構築）
推奨用途 書き込み重視、順序保持が必要 検索重視、一般的な用途

推奨: JSONB型を使おう

ほとんどの場合、検索パフォーマンスとインデックス対応の点でJSONB型の使用が推奨されます。 キーの順序保持が必須の場合や、書き込みが極端に多い場合のみJSON型を検討しましょう。

6.2 JSONデータの格納

まずはテーブルを作成し、JSONBデータを格納してみましょう。 ここでは、商品情報を管理するテーブルを例として使用します。

テーブル作成とデータ挿入 
-- 商品テーブルの作成
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    details JSONB  -- 商品詳細情報をJSONB型で格納
);

-- データの挿入
INSERT INTO products (name, details) VALUES
('ノートパソコン', '{
    "brand": "TechCorp",
    "price": 120000,
    "specs": {
        "cpu": "Intel Core i7",
        "memory": "16GB",
        "storage": "512GB SSD"
    },
    "tags": ["PC", "ビジネス", "携帯"]
}'::JSONB);

INSERT INTO products (name, details) VALUES
('ワイヤレスマウス', '{
    "brand": "MouseTech",
    "price": 3500,
    "specs": {
        "type": "Bluetooth",
        "battery": "単3×2本"
    },
    "tags": ["マウス", "周辺機器"]
}'::JSONB);

::JSONB によるキャスト

JSONテキストをJSONB型に変換するには、文字列の後に「::JSONB」を付けます。 これにより、PostgreSQLが文字列をJSONB形式として解析・保存します。

6.3 JSON演算子による要素の取得

PostgreSQLには、JSONデータから特定の要素を取り出すための便利な演算子が用意されています。

演算子 説明 例 結果の型
-> キーでJSONオブジェクトを取得 details->'brand' JSON/JSONB
->> キーでテキストを取得 details->>'brand' TEXT
#> パスでJSONオブジェクトを取得 details#>'{specs,cpu}' JSON/JSONB
#>> パスでテキストを取得 details#>>'{specs,cpu}' TEXT
@> JSONが指定の値を含むか判定 details@>'{"brand":"TechCorp"}' BOOLEAN
JSON演算子の使用例 
-- ブランド名を取得（JSON型として）
SELECT name, details->'brand' AS brand_json FROM products;
-- 結果: "TechCorp" （ダブルクォート付き）

-- ブランド名を取得（テキスト型として）
SELECT name, details->>'brand' AS brand_text FROM products;
-- 結果: TechCorp （ダブルクォートなし）

-- ネストした値を取得（CPU名）
SELECT name, details#>>'{specs,cpu}' AS cpu FROM products;
-- 結果: Intel Core i7

-- 特定のブランドを検索
SELECT name FROM products
WHERE details @> '{"brand": "TechCorp"}'::JSONB;

6.4 JSONデータの検索と条件指定

JSONデータに対して、様々な条件での検索が可能です。 値の比較、配列要素の検索、部分一致検索など、実践的な例を見ていきましょう。

検索の実例 
-- 価格が10万円以下の商品を検索
SELECT name, details->>'price' AS price FROM products
WHERE (details->>'price')::INTEGER <= 100000;

-- 配列にタグ「PC」を含む商品を検索
SELECT name FROM products
WHERE details->'tags' @> '["PC"]'::JSONB;

-- ブランド名が「Tech」で始まる商品を検索
SELECT name, details->>'brand' AS brand FROM products
WHERE details->>'brand' LIKE 'Tech%';

-- CPUが存在する商品を検索
SELECT name FROM products
WHERE details#>'{specs,cpu}' IS NOT NULL;

注意: 型変換が必要な場合

->> 演算子で取得した値はTEXT型です。数値として比較する場合は ::INTEGER や ::NUMERIC で型変換が必要です。 型変換を忘れると、文字列として比較されるため意図しない結果になります（例: "9" > "10" が TRUE になる）。

実習: JSON検索を試してみよう

  1. 1. テーブル作成: 上記のproductsテーブルを作成し、サンプルデータを挿入してください。
  2. 2. 基本検索: すべての商品のブランド名を取得するクエリを実行してください。
  3. 3. 価格検索: 価格が5000円以下の商品を検索してください。
  4. 4. タグ検索: タグに「周辺機器」を含む商品を検索してください。

6.5 JSONデータの更新

PostgreSQLには、JSONデータを部分的に更新するための関数が用意されています。 全体を置き換えずに、特定のキーだけを更新できます。

関数 説明 使用例
jsonb_set 指定パスの値を設定 jsonb_set(data, '{key}', '"value"')
|| JSONオブジェクトをマージ data || '{"key":"value"}'
- 指定キーを削除 data - 'key'
JSON更新の実例 
-- 価格を更新（jsonb_setを使用）
UPDATE products
SET details = jsonb_set(details, '{price}', '98000')
WHERE name = 'ノートパソコン';

-- 新しいキーを追加（||演算子を使用）
UPDATE products
SET details = details || '{"warranty": "3年保証"}'::JSONB
WHERE name = 'ノートパソコン';

-- ネストした値を更新
UPDATE products
SET details = jsonb_set(details, '{specs,memory}', '"32GB"')
WHERE name = 'ノートパソコン';

-- キーを削除
UPDATE products
SET details = details - 'warranty'
WHERE name = 'ノートパソコン';

jsonb_setの注意点

jsonb_setの第3引数（設定する値）は、JSONB型である必要があります。 文字列を設定する場合は '"value"' のようにダブルクォートで囲んだ上でシングルクォートで囲みます。 数値の場合は '100' のようにします。

6.6 JSON関数の活用

PostgreSQLには、JSONデータを扱うための豊富な関数が用意されています。 配列の展開、キーの一覧取得、型判定など、便利な関数を紹介します。

便利なJSON関数 
-- JSON配列を行に展開
SELECT name, jsonb_array_elements_text(details->'tags') AS tag
FROM products
WHERE details ? 'tags';

-- すべてのキーを取得
SELECT name, jsonb_object_keys(details) AS keys
FROM products;

-- キーの存在確認（?演算子）
SELECT name FROM products WHERE details ? 'warranty';

-- JSON型の判定
SELECT
    name,
    jsonb_typeof(details->'price') AS price_type,
    jsonb_typeof(details->'tags') AS tags_type
FROM products;

jsonb_array_elements

JSON配列の各要素を個別の行として展開します。 タグやカテゴリのような配列データを検索する際に便利です。

jsonb_object_keys

JSONオブジェクトのすべてのキーを取得します。 動的なスキーマを持つJSONデータの構造を調べる際に有用です。

? 演算子

指定したキーがJSONオブジェクト内に存在するかを判定します。 オプショナルなフィールドの有無で条件分岐する際に使用します。

jsonb_typeof

JSON要素の型（object、array、string、number、boolean、null）を返します。 動的なデータの型チェックに使用します。

6.7 JSONインデックスの作成

JSONB型のデータには、GINインデックスを作成することで検索パフォーマンスを大幅に改善できます。

GINインデックスの作成 
-- JSONB全体に対するGINインデックス
CREATE INDEX idx_products_details ON products USING GIN (details);

-- 特定のパスに対するインデックス
CREATE INDEX idx_products_brand
ON products ((details->>'brand'));

-- 配列要素に対するインデックス
CREATE INDEX idx_products_tags
ON products USING GIN ((details->'tags'));

インデックスのメリット

GINインデックスを作成すると、@>、?、?&、?| などの演算子を使った検索が高速化されます。 大量のJSONデータを扱う場合、インデックスの有無で数十倍〜数百倍の性能差が出ることもあります。

実習: JSON更新とインデックスを試してみよう

  1. 1. 価格更新: ノートパソコンの価格を110000円に更新してください。
  2. 2. キー追加: ワイヤレスマウスに "color": "ブラック" を追加してください。
  3. 3. 配列展開: すべての商品のタグを1行ずつ展開して表示してください。
  4. 4. インデックス作成: detailsカラムにGINインデックスを作成してください。

6.8 実践例: ユーザー設定管理

JSONBを活用した実践的な例として、ユーザーごとの設定情報を管理するテーブルを作成してみましょう。

ユーザー設定テーブルの例 
-- ユーザー設定テーブルの作成
CREATE TABLE user_settings (
    user_id INTEGER PRIMARY KEY,
    username VARCHAR(50) NOT NULL,
    preferences JSONB DEFAULT '{}'::JSONB,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- インデックスの作成
CREATE INDEX idx_user_preferences
ON user_settings USING GIN (preferences);

-- データの挿入
INSERT INTO user_settings (user_id, username, preferences) VALUES
(1, 'tanaka', '{
    "theme": "dark",
    "language": "ja",
    "notifications": {
        "email": true,
        "push": false
    },
    "display": {
        "itemsPerPage": 20,
        "fontSize": "medium"
    }
}'::JSONB);

-- テーマがダークモードのユーザーを検索
SELECT username FROM user_settings
WHERE preferences->>'theme' = 'dark';

-- メール通知を有効にしているユーザーを検索
SELECT username FROM user_settings
WHERE preferences#>>'{notifications,email}' = 'true';

JSONBを使うメリット

ユーザー設定のような、項目が頻繁に追加・変更される可能性のあるデータは、JSONB型で管理すると便利です。 新しい設定項目を追加する際に、テーブル構造を変更（ALTER TABLE）する必要がなく、柔軟に対応できます。

理解度確認クイズ

  1. Q1. JSON型とJSONB型の主な違いを2つ挙げてください。

    回答を表示

    答え（例）:
    1. 保存形式 - JSON型はテキストとして保存、JSONB型はバイナリ形式で保存される
    2. インデックス - JSONB型はGINインデックスに対応しているが、JSON型は未対応
    （他にも、検索速度、挿入速度、キーの順序保持なども正解です）

  2. Q2. 演算子 -> と ->> の違いは何ですか？

    回答を表示

    答え:
    -> はJSON/JSONB型を返し、->> はTEXT型を返します。
    -> で取得した値はさらにJSON演算子でアクセスできますが、->> で取得した値は文字列として扱われます。
    数値として比較する場合は ->> で取得後に ::INTEGER などで型変換が必要です。

  3. Q3. JSONB型のカラムに作成すべきインデックスの種類は何ですか？また、どのような検索で効果がありますか？

    回答を表示

    答え:
    GINインデックスを作成します。
    効果がある検索: @>（包含チェック）、?（キー存在チェック）、?&（複数キーすべて存在）、?|（複数キーいずれか存在）などの演算子を使った検索で高速化されます。

よくあるエラーと対処法

ERROR: invalid input syntax for type json

原因: JSON形式が正しくありません。
対処法: キーと値を必ずダブルクォートで囲む、カンマの位置を確認する、最後の要素にカンマをつけないなど、JSON形式を確認してください。

ERROR: operator does not exist: jsonb = text

原因: JSONB型と文字列を直接比較しようとしています。
対処法: ->> 演算子でTEXT型に変換するか、比較する値を ::JSONB でキャストしてください。

数値比較が文字列として比較される

原因: ->> で取得した値はTEXT型です。
対処法: ::INTEGER や ::NUMERIC で型変換してから比較してください。
例: (details->>'price')::INTEGER > 10000

第6章のまとめ

  • JSONB型は検索パフォーマンスとインデックス対応の点でJSON型より優れている
  • ->、->>、#>、#>> などの演算子でJSON要素にアクセスできる
  • jsonb_set、||、- などを使ってJSONデータを部分的に更新できる
  • GINインデックスを作成することでJSON検索を高速化できる
  • ユーザー設定など動的なスキーマにはJSONBが適している
