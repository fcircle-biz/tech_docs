バックアップとリストア - データを安全に守る

pg_dump、pg_restore、pg_basebackupを使ったバックアップとリストアの方法を学習します。論理バックアップと物理バックアップの違い、定期バックアップの自動化について理解します。

この章で学ぶこと

  • 論理バックアップと物理バックアップの違いを理解する
  • pg_dumpでデータベースをバックアップできるようになる
  • pg_restoreでデータを復元できるようになる
  • pg_basebackupで物理バックアップを取得する方法を知る
  • 定期バックアップの自動化手法を理解する

10.1 バックアップの重要性

データベースのバックアップは、企業のデータ資産を守るための最重要施策です。 ハードウェア障害、人為的ミス、サイバー攻撃など、様々な理由でデータが失われるリスクがあります。 定期的なバックアップと、確実に復元できる体制を整えることで、万一の事態にも迅速に対応できます。

ハードウェア障害

ディスク故障、サーバーダウンによるデータ消失

人為的ミス

誤ったDELETE/UPDATE操作によるデータ破損

サイバー攻撃

ランサムウェア、不正アクセスによるデータ改ざん

バックアップの3-2-1ルール

3つのコピーを、2種類の異なる媒体で保管し、1つは別の場所に保管するというルールです。 例えば、本番サーバー（1つ目）、同じサーバーの別ディスク（2つ目、異なる媒体）、クラウドストレージ（3つ目、別の場所）という構成が理想的です。

10.2 論理バックアップと物理バックアップ

PostgreSQLには大きく分けて2種類のバックアップ方法があります。それぞれの特徴を理解して、用途に応じて使い分けましょう。

flowchart TD A[PostgreSQLのバックアップ] subgraph Logical[論理バックアップ] B1[pg_dump] B2[pg_dumpall] B3[SQLファイルとして出力] B4[テーブル単位で復元可能] end subgraph Physical[物理バックアップ] C1[pg_basebackup] C2[ファイルシステムコピー] C3[データベースクラスタ全体] C4[高速バックアップ＆リストア] end A --> Logical A --> Physical
項目 論理バックアップ 物理バックアップ
主なツール pg_dump、pg_dumpall pg_basebackup
バックアップ内容 SQLコマンド（CREATE、INSERT等） データファイル全体
復元の柔軟性 高い（テーブル単位可能） 低い（全体復元のみ）
バックアップ速度 遅い（データ量に比例） 速い（ファイルコピー）
バージョン互換性 高い（異なるバージョン間でも可能） 低い（同じバージョンのみ）
用途 定期バックアップ、テーブル単位復元 災害対策、サーバー移行

10.3 pg_dumpによる論理バックアップ

pg_dumpは、データベースの内容をSQLコマンドとして出力するツールです。 最も一般的なバックアップ方法で、テーブル単位での復元や異なるバージョン間での移行に適しています。

基本的なpg_dumpコマンド

データベース全体をバックアップ 
# 平文SQLファイルで出力
pg_dump -U postgres -d mydb -f mydb_backup.sql

# 圧縮カスタム形式で出力（推奨）
pg_dump -U postgres -d mydb -Fc -f mydb_backup.dump
特定のテーブルのみバックアップ 
# usersテーブルのみバックアップ
pg_dump -U postgres -d mydb -t users -f users_backup.sql
スキーマのみバックアップ（データなし） 
# テーブル定義のみバックアップ
pg_dump -U postgres -d mydb --schema-only -f schema_only.sql

pg_dumpの主なオプション

オプション 説明
-Fc カスタム形式（圧縮、並列復元可能）
-Ft tar形式（アーカイブ）
-t テーブル名 特定のテーブルのみバックアップ
--schema-only スキーマのみ（CREATE文のみ）
--data-only データのみ（INSERT文のみ）
-Z 0-9 圧縮レベル（0=なし、9=最大）

カスタム形式（-Fc）の利点

カスタム形式は圧縮されているためファイルサイズが小さく、pg_restoreで並列復元や選択的復元が可能です。 特別な理由がなければ、カスタム形式を使用することを推奨します。

10.4 pg_restoreによる復元

pg_restoreは、pg_dumpで作成したバックアップファイルからデータベースを復元するツールです。 カスタム形式（-Fc）のバックアップに対して、柔軟な復元オプションを提供します。

基本的なpg_restoreコマンド

データベース全体を復元 
# 既存のデータベースに復元
pg_restore -U postgres -d mydb mydb_backup.dump

# 新しいデータベースを作成して復元
createdb -U postgres mydb_new
pg_restore -U postgres -d mydb_new mydb_backup.dump
特定のテーブルのみ復元 
# usersテーブルのみ復元
pg_restore -U postgres -d mydb -t users mydb_backup.dump
並列復元（高速化） 
# 4つの並列ジョブで復元（大規模DBで有効）
pg_restore -U postgres -d mydb -j 4 mydb_backup.dump

復元前の注意事項

既存のデータがあるデータベースに復元すると、データが重複したりエラーが発生する可能性があります。 復元前に必ず現在のデータベースのバックアップを取り、テスト環境で復元を確認してから本番環境に適用しましょう。

バックアップファイルの内容確認

バックアップファイルに含まれる内容をリスト表示 
# バックアップファイルの内容を確認
pg_restore -l mydb_backup.dump

# 出力例：
# Archive created at 2025-01-15 10:30:00 JST
# dbname: mydb
# 208; 1259 16385 TABLE public users postgres
# 209; 1259 16386 TABLE public posts postgres
# ...

10.5 pg_dumpallによる全データベースバックアップ

pg_dumpallは、PostgreSQLサーバー上のすべてのデータベースとグローバルオブジェクト（ロール、テーブルスペースなど）をまとめてバックアップするツールです。 サーバー移行や災害対策用のバックアップに適しています。

全データベースをバックアップ 
# すべてのデータベースをバックアップ
pg_dumpall -U postgres -f all_databases.sql

# ロール（ユーザー）のみバックアップ
pg_dumpall -U postgres --roles-only -f roles.sql
pg_dumpallのバックアップを復元 
# psqlコマンドで復元（平文SQLのため）
psql -U postgres -f all_databases.sql

pg_dumpとpg_dumpallの使い分け

通常のバックアップにはpg_dumpを使用します。 サーバー全体の移行や、グローバルオブジェクト（ロール定義など）も含めてバックアップする場合はpg_dumpallを使用します。

10.6 pg_basebackupによる物理バックアップ

pg_basebackupは、PostgreSQLのデータディレクトリ全体をコピーする物理バックアップツールです。 大規模なデータベースでも高速にバックアップでき、災害対策やスタンバイサーバーの構築に利用されます。

物理バックアップの取得 
# 物理バックアップを取得
pg_basebackup -U postgres -D /backup/pgdata -Ft -z -P

# オプション説明：
# -D: バックアップ先ディレクトリ
# -Ft: tar形式で出力
# -z: gzip圧縮
# -P: 進捗状況を表示

物理バックアップの前提条件

pg_basebackupを使用するには、postgresql.confで以下の設定が必要です：

postgresql.confの設定 
# WAL（Write-Ahead Logging）レベルを設定
wal_level = replica

# レプリケーション用の接続数を確保
max_wal_senders = 3

# WALセグメントの保持設定
wal_keep_size = 1GB

物理バックアップの注意点

物理バックアップは、同じPostgreSQLバージョン、同じアーキテクチャ（64bit/32bit）でのみ復元可能です。 また、データベース全体のバックアップであり、特定のテーブルのみを復元することはできません。

10.7 定期バックアップの自動化

バックアップは定期的に実行することが重要です。 Linuxのcronやタスクスケジューラーを使って、バックアップを自動化しましょう。

バックアップスクリプトの作成

backup_db.sh - 自動バックアップスクリプト 
#!/bin/bash
# 日付を含むファイル名を生成
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_DIR="/backup/postgresql"
DB_NAME="mydb"

# バックアップ実行
pg_dump -U postgres -d $DB_NAME -Fc -f $BACKUP_DIR/${DB_NAME}_${DATE}.dump

# 7日以上前のバックアップを削除
find $BACKUP_DIR -name "*.dump" -mtime +7 -delete

echo "Backup completed: ${DB_NAME}_${DATE}.dump"

cronで定期実行

crontabの設定（毎日深夜2時に実行） 
# crontab -e で編集
0 2 * * * /path/to/backup_db.sh >> /var/log/pg_backup.log 2>&1

# 説明：
# 0 2 * * *  : 毎日2時0分に実行
# >>         : ログファイルに追記
# 2>&1       : エラー出力も同じファイルに出力

クラウドストレージへの転送

AWS S3にバックアップをアップロード 
# AWS CLIを使ってS3にアップロード
aws s3 cp $BACKUP_DIR/${DB_NAME}_${DATE}.dump s3://my-backup-bucket/postgresql/

# rsyncで別サーバーに転送
rsync -avz $BACKUP_DIR/${DB_NAME}_${DATE}.dump backup-server:/backup/

バックアップ運用のベストプラクティス

  • • 定期的にバックアップが成功しているかログを確認する
  • • 定期的に復元テストを実施し、バックアップが正常に使えることを確認する
  • • バックアップファイルを別の場所（クラウドストレージ、別サーバー）にも保管する
  • • バックアップファイルの暗号化を検討する（機密データの場合）

実習: バックアップとリストアを試そう

  1. 1. テスト用データベースを作成 
    -- pgAdminまたはpsqlで実行
CREATE DATABASE test_backup;
\c test_backup

CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100)
);

INSERT INTO users (name, email) VALUES
    ('山田太郎', 'yamada@example.com'),
    ('鈴木花子', 'suzuki@example.com'),
    ('佐藤次郎', 'sato@example.com');
  2. 2. pg_dumpでバックアップを取得 
    pg_dump -U postgres -d test_backup -Fc -f test_backup.dump
  3. 3. データを削除してみる 
    -- 誤ってデータを削除したと仮定
DELETE FROM users;
  4. 4. pg_restoreでデータを復元 
    pg_restore -U postgres -d test_backup test_backup.dump
  5. 5. データが復元されたか確認 
    SELECT * FROM users;
-- 3件のデータが復元されているはず

理解度確認クイズ

  1. Q1. 論理バックアップと物理バックアップの違いを説明してください。

    回答を表示

    答え:
    論理バックアップ（pg_dump）: データベースの内容をSQLコマンドとして出力。テーブル単位での復元や異なるバージョン間での移行が可能。
    物理バックアップ（pg_basebackup）: データファイル全体をコピー。高速だが、同じバージョンでのみ復元可能で、データベース全体の復元のみ対応。

  2. Q2. pg_dumpでカスタム形式（-Fc）を使うメリットは何ですか？

    回答を表示

    答え:
    1. ファイルサイズが圧縮される（ディスク容量削減）
    2. pg_restoreで並列復元が可能（復元時間短縮）
    3. 特定のテーブルのみを選択的に復元できる
    4. バックアップファイルの内容をリスト表示できる（pg_restore -l）

  3. Q3. バックアップの3-2-1ルールとは何ですか？

    回答を表示

    答え:
    3つのコピーを、2種類の異なる媒体で保管し、1つは別の場所に保管するルールです。
    例: 本番サーバー（1つ目）、同じサーバーの別ディスク（2つ目）、クラウドストレージ（3つ目、別の場所）
    これにより、ハードウェア障害や災害時にもデータを保護できます。

第10章のまとめ

  • バックアップはデータ資産を守るための最重要施策。3-2-1ルールに従って保管する
  • 論理バックアップ（pg_dump）は柔軟性が高く、テーブル単位での復元やバージョン間移行に適している
  • 物理バックアップ（pg_basebackup）は高速だが、同じバージョンでのみ復元可能
  • pg_restoreで並列復元や選択的復元が可能。カスタム形式（-Fc）の使用を推奨
  • cronやスクリプトでバックアップを自動化し、定期的に復元テストを実施する
