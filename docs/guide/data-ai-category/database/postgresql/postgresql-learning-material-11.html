ユーザーと権限管理 - セキュリティの基礎

PostgreSQLのロールとユーザー管理、GRANT/REVOKEによる権限設定を学習します。本番環境で必要となるセキュリティ設定の基本と、最小権限の原則について理解します。

この章で学ぶこと

  • PostgreSQLのロールとユーザーの概念を理解する
  • ロールの作成・変更・削除の基本操作を習得する
  • GRANT/REVOKEによる権限管理を理解する
  • 最小権限の原則に基づくセキュリティ設計を学ぶ

11.1 PostgreSQLのロールとユーザー

PostgreSQLでは、ロール（Role）という概念でユーザー管理を行います。 ロールは「ユーザー」と「グループ」の両方の役割を持つことができる柔軟な仕組みです。

他のRDBMSでは「ユーザー」と「ロール」が別の概念として扱われることが多いですが、PostgreSQLでは統一されています。 ログイン可能なロールが「ユーザー」、権限をまとめたロールが「グループロール」として機能します。

ロールの特徴

ロールにはLOGIN属性を付けることでログイン可能になります。 また、他のロールを継承することで権限をグループ化でき、複雑な権限管理をシンプルに実現できます。

11.2 ロールの作成と管理

ロールの作成・変更・削除の基本操作を学びましょう。

11.2.1 ロールの作成（CREATE ROLE）

ロールを作成するにはCREATE ROLEコマンドを使用します。

SQL 
-- 基本的なロール作成
CREATE ROLE app_user;

-- ログイン可能なロール（ユーザー）作成
CREATE ROLE app_user WITH LOGIN PASSWORD 'password123';

-- より詳細な属性を指定したロール作成
CREATE ROLE developer WITH
  LOGIN                    -- ログイン可能
  PASSWORD 'dev_pass'      -- パスワード設定
  CREATEDB                 -- データベース作成権限
  VALID UNTIL '2025-12-31';-- パスワード有効期限

11.2.2 ロール属性の種類

ロールには様々な属性を設定できます。

属性 説明
LOGIN / NOLOGIN ログイン可能/不可を設定（デフォルト: NOLOGIN）
SUPERUSER / NOSUPERUSER スーパーユーザー権限（全ての制限を回避）
CREATEDB / NOCREATEDB データベース作成権限
CREATEROLE / NOCREATEROLE ロール作成権限
REPLICATION / NOREPLICATION レプリケーション接続権限
PASSWORD 'password' ログインパスワードの設定

11.2.3 ロールの変更（ALTER ROLE）

SQL 
-- パスワード変更
ALTER ROLE app_user WITH PASSWORD 'new_password456';

-- ロール属性の変更
ALTER ROLE developer WITH NOCREATEDB;

-- パスワード有効期限の延長
ALTER ROLE app_user VALID UNTIL '2026-12-31';

11.2.4 ロールの削除（DROP ROLE）

SQL 
-- ロールの削除
DROP ROLE app_user;

-- 所有オブジェクトがある場合は先に譲渡または削除
REASSIGN OWNED BY app_user TO postgres;
DROP OWNED BY app_user;
DROP ROLE app_user;

ロール削除の注意点

ロールがテーブルやビューなどのオブジェクトを所有している場合、そのままでは削除できません。 先にREASSIGN OWNEDで所有権を他のロールに譲渡するか、DROP OWNEDで所有オブジェクトを削除する必要があります。

11.3 権限の種類と管理

PostgreSQLでは、データベース・スキーマ・テーブルなど、様々なオブジェクトに対して細かく権限を設定できます。

11.3.1 主な権限の種類

テーブル権限

  • SELECT - データの参照
  • INSERT - データの挿入
  • UPDATE - データの更新
  • DELETE - データの削除
  • TRUNCATE - テーブルの全削除
  • REFERENCES - 外部キー作成

データベース権限

  • CONNECT - DB接続
  • CREATE - スキーマ作成
  • TEMP - 一時テーブル作成

スキーマ権限

  • USAGE - スキーマ内オブジェクトへのアクセス
  • CREATE - オブジェクト作成

その他の権限

  • EXECUTE - 関数・プロシージャ実行
  • USAGE - シーケンス使用
  • ALL PRIVILEGES - 全権限

11.4 GRANT - 権限の付与

GRANTコマンドで、ロールに対して権限を付与します。

11.4.1 基本的なGRANT構文

SQL 
-- テーブルへのSELECT権限付与
GRANT SELECT ON products TO app_user;

-- テーブルへの複数権限付与
GRANT SELECT, INSERT, UPDATE ON orders TO app_user;

-- データベース接続権限
GRANT CONNECT ON DATABASE mydb TO app_user;

-- スキーマ使用権限
GRANT USAGE ON SCHEMA public TO app_user;

11.4.2 すべてのテーブルに権限を付与

SQL 
-- スキーマ内の全テーブルにSELECT権限付与
GRANT SELECT ON ALL TABLES IN SCHEMA public TO app_user;

-- スキーマ内の全テーブルに読み書き権限付与
GRANT SELECT, INSERT, UPDATE, DELETE
ON ALL TABLES IN SCHEMA public TO app_user;

11.4.3 デフォルト権限の設定

将来作成されるオブジェクトに対する権限を事前に設定できます。

SQL 
-- 将来作成されるテーブルへの権限設定
ALTER DEFAULT PRIVILEGES IN SCHEMA public
GRANT SELECT ON TABLES TO readonly_user;

-- 特定ロールが作成するテーブルへの権限設定
ALTER DEFAULT PRIVILEGES FOR ROLE developer IN SCHEMA public
GRANT ALL ON TABLES TO app_user;

デフォルト権限のメリット

デフォルト権限を設定しておくことで、新しくテーブルを作成するたびに権限を付与する手間が省けます。 開発環境で特に便利な機能です。

11.5 REVOKE - 権限の剥奪

REVOKEコマンドで、付与した権限を取り消します。

SQL 
-- テーブルのSELECT権限剥奪
REVOKE SELECT ON products FROM app_user;

-- 複数の権限を剥奪
REVOKE INSERT, UPDATE, DELETE ON orders FROM app_user;

-- 全権限の剥奪
REVOKE ALL PRIVILEGES ON customers FROM app_user;

-- スキーマ内の全テーブルから権限剥奪
REVOKE SELECT ON ALL TABLES IN SCHEMA public FROM app_user;

権限の継承に注意

ロールが他のロールを継承している場合、継承元の権限も有効になります。 特定のロールから権限を剥奪しても、継承している別のロール経由でアクセスできる可能性があることに注意しましょう。

11.6 グループロールによる権限管理

グループロールを使うことで、権限管理を効率的に行えます。

11.6.1 グループロールの作成と権限付与

SQL 
-- 読み取り専用グループロール作成
CREATE ROLE readonly_group NOLOGIN;

-- 読み書き可能グループロール作成
CREATE ROLE readwrite_group NOLOGIN;

-- 各グループに権限付与
GRANT CONNECT ON DATABASE mydb TO readonly_group;
GRANT USAGE ON SCHEMA public TO readonly_group;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly_group;

GRANT CONNECT ON DATABASE mydb TO readwrite_group;
GRANT USAGE ON SCHEMA public TO readwrite_group;
GRANT SELECT, INSERT, UPDATE, DELETE
ON ALL TABLES IN SCHEMA public TO readwrite_group;

11.6.2 ユーザーにグループロールを付与

SQL 
-- ユーザー作成
CREATE ROLE alice WITH LOGIN PASSWORD 'alice_pass';
CREATE ROLE bob WITH LOGIN PASSWORD 'bob_pass';

-- グループロールを付与（ロールの継承）
GRANT readonly_group TO alice;
GRANT readwrite_group TO bob;

-- 複数のグループロールを付与することも可能
GRANT readonly_group, readwrite_group TO admin_user;

グループロールのメリット

グループロールを使うことで、「読み取り専用ユーザー」「アプリケーション用ユーザー」など、役割ごとに権限を一元管理できます。 新しいユーザーを追加する際も、グループロールを付与するだけで必要な権限がすべて設定されます。

11.7 権限の確認方法

設定した権限が正しく反映されているか確認する方法を学びましょう。

11.7.1 ロール一覧の確認

SQL 
-- すべてのロールを確認
\du

-- SQLクエリでロール確認
SELECT rolname, rolsuper, rolinherit, rolcreaterole,
       rolcreatedb, rolcanlogin
FROM pg_roles
ORDER BY rolname;

11.7.2 テーブル権限の確認

SQL 
-- テーブルのアクセス権限を確認
\dp products

-- SQLクエリでテーブル権限確認
SELECT grantee, privilege_type
FROM information_schema.table_privileges
WHERE table_name = 'products'
ORDER BY grantee, privilege_type;

11.7.3 現在のロール確認

SQL 
-- 現在のロール確認
SELECT current_user;

-- セッション情報確認
SELECT current_user, session_user, current_database();

11.8 セキュリティのベストプラクティス

本番環境でPostgreSQLを安全に運用するためのベストプラクティスを紹介します。

1. 最小権限の原則

各ユーザーには業務に必要最小限の権限のみを付与します。 特にSUPERUSER権限は極力使わず、必要に応じて個別の権限を付与しましょう。

2. 強力なパスワードポリシー

パスワードは十分に長く（12文字以上）、大小英字・数字・記号を組み合わせたものを使用します。 VALID UNTILで有効期限を設定し、定期的な変更を促しましょう。

3. 役割ベースのアクセス制御

グループロールを活用し、「開発者」「アプリケーション」「分析担当者」など、役割ごとに権限を管理します。 個別ユーザーへの直接的な権限付与は避けましょう。

4. postgresロールの使用制限

デフォルトのpostgresロールは管理用途のみに使用し、アプリケーションから直接接続しないようにします。 アプリケーション専用のロールを作成しましょう。

5. 定期的な権限の棚卸し

使われなくなったロールや不要な権限は定期的に見直し、削除します。 退職者のアカウントは即座に無効化しましょう。

実習: Webアプリケーション向けの権限設定

以下のシナリオで権限管理を実践してみましょう。

シナリオ:

  • Webアプリケーション用の読み書き可能ロール（app_user）
  • 分析担当者用の読み取り専用ロール（analyst）
  • バッチ処理用のロール（batch_user）
SQL 
-- グループロール作成
CREATE ROLE app_role NOLOGIN;
CREATE ROLE analyst_role NOLOGIN;
CREATE ROLE batch_role NOLOGIN;

-- 権限付与
GRANT CONNECT ON DATABASE mydb TO app_role;
GRANT USAGE ON SCHEMA public TO app_role;
GRANT SELECT, INSERT, UPDATE, DELETE
ON ALL TABLES IN SCHEMA public TO app_role;

GRANT CONNECT ON DATABASE mydb TO analyst_role;
GRANT USAGE ON SCHEMA public TO analyst_role;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO analyst_role;

-- ユーザー作成とグループ付与
CREATE ROLE app_user WITH LOGIN PASSWORD 'app_secure_pass123';
CREATE ROLE analyst WITH LOGIN PASSWORD 'analyst_pass123';
CREATE ROLE batch_user WITH LOGIN PASSWORD 'batch_pass123';

GRANT app_role TO app_user;
GRANT analyst_role TO analyst;
GRANT batch_role TO batch_user;

理解度確認クイズ

  1. Q1. PostgreSQLでロールとユーザーの違いは何ですか？

    回答を表示

    答え: PostgreSQLでは「ロール」が統一的な概念として使われます。
    - ログイン可能なロール（LOGIN属性あり）がユーザー相当
    - ログイン不可のロール（NOLOGIN）がグループロール相当
    他のRDBMSのように「ユーザー」と「ロール」が分かれていないのがPostgreSQLの特徴です。

  2. Q2. 最小権限の原則とは何ですか？なぜ重要ですか？

    回答を表示

    答え: 最小権限の原則とは、各ユーザーに業務遂行に必要最小限の権限のみを付与する考え方です。
    重要な理由:
    - セキュリティリスクの最小化（不正アクセスの影響範囲を限定）
    - 誤操作による事故の防止（削除権限がなければ誤ってDELETEできない）
    - 監査とトラブルシューティングの容易化（誰が何をできるか明確）

  3. Q3. グループロールを使うメリットを2つ挙げてください。

    回答を表示

    答え（例）:
    1. 権限の一元管理: 役割ごとに権限をまとめて管理でき、変更時も1箇所の修正で済む
    2. 新規ユーザー追加の効率化: グループロールを付与するだけで必要な権限がすべて設定される
    （他にも、権限設定の標準化、ヒューマンエラーの削減なども正解です）

第11章のまとめ

  • PostgreSQLではロールがユーザーとグループの両方の役割を持つ
  • CREATE ROLE、ALTER ROLE、DROP ROLEでロールを管理する
  • GRANTで権限を付与、REVOKEで権限を剥奪する
  • グループロールを使うことで効率的な権限管理が可能
  • 最小権限の原則に基づくセキュリティ設計が重要
