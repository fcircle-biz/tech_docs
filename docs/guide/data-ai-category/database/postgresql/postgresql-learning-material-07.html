ビューとマテリアライズドビュー - 複雑なクエリを簡単に

ビューの作成と活用方法、マテリアライズドビューによるパフォーマンス改善を学習します。複雑なクエリを再利用可能な形で保存し、日常的なデータ分析業務を効率化する方法を習得します。

この章で学ぶこと

  • ビューとは何か、なぜ便利なのかを理解する
  • ビューの作成・更新・削除の方法を習得する
  • マテリアライズドビューとビューの違いを理解する
  • マテリアライズドビューでパフォーマンスを改善する方法を学ぶ

7.1 ビューとは

ビュー（View）は、1つまたは複数のテーブルから必要なデータだけを取り出したSELECT文を保存したものです。 ビューを使うことで、複雑なクエリを「仮想的なテーブル」として扱えるようになります。

ビューは実際にデータを保存しているわけではなく、ビューにアクセスするたびに元のテーブルから最新データを取得します。 これにより、元のテーブルが更新されれば、ビューの結果も自動的に更新されます。

ビューのメリット

複雑なJOINやSUBQUERYを毎回書く必要がなく、再利用できます。また、特定のカラムだけを見せることで、セキュリティ向上にも役立ちます。

7.2 ビューの作成と利用

ビューを作成するには、CREATE VIEW文を使います。実際に例を見ながら学びましょう。

基本的なビューの作成

まず、社員テーブルと部署テーブルがあるとします。部署名と社員名を結合したビューを作成してみましょう。

employees.sql（サンプルテーブル作成） 
-- 部署テーブル
CREATE TABLE departments (
    department_id SERIAL PRIMARY KEY,
    department_name VARCHAR(100) NOT NULL
);

-- 社員テーブル
CREATE TABLE employees (
    employee_id SERIAL PRIMARY KEY,
    employee_name VARCHAR(100) NOT NULL,
    department_id INTEGER REFERENCES departments(department_id),
    salary INTEGER NOT NULL,
    hire_date DATE NOT NULL
);

-- サンプルデータ挿入
INSERT INTO departments VALUES
(1, '営業部'),
(2, '開発部'),
(3, '人事部');

INSERT INTO employees VALUES
(1, '山田太郎', 1, 350000, '2020-04-01'),
(2, '佐藤花子', 2, 420000, '2019-07-15'),
(3, '鈴木一郎', 2, 480000, '2018-05-10'),
(4, '田中美咲', 3, 380000, '2021-09-01');
employee_view.sql（ビュー作成） 
-- 社員と部署を結合したビューを作成
CREATE VIEW employee_department_view AS
SELECT
    e.employee_id,
    e.employee_name,
    d.department_name,
    e.salary,
    e.hire_date
FROM employees e
INNER JOIN departments d ON e.department_id = d.department_id;
query_view.sql（ビューの利用） 
-- ビューを通常のテーブルと同じように利用
SELECT * FROM employee_department_view;

-- ビューに対してWHERE句やORDER BYも使える
SELECT * FROM employee_department_view
WHERE department_name = '開発部'
ORDER BY salary DESC;

集計を含むビューの作成

部署ごとの平均給与と人数を集計したビューを作成してみましょう。

department_summary_view.sql（集計ビュー） 
-- 部署ごとの統計情報ビュー
CREATE VIEW department_summary_view AS
SELECT
    d.department_name,
    COUNT(e.employee_id) AS employee_count,
    AVG(e.salary)::INTEGER AS avg_salary,
    MIN(e.salary) AS min_salary,
    MAX(e.salary) AS max_salary
FROM departments d
LEFT JOIN employees e ON d.department_id = e.department_id
GROUP BY d.department_name;

-- ビューを利用して統計情報を取得
SELECT * FROM department_summary_view
ORDER BY avg_salary DESC;

7.3 ビューの管理

ビューを削除したり、定義を変更したりする方法を学びましょう。

ビューの削除

-- ビューを削除
DROP VIEW employee_department_view;

-- 存在しない場合エラーを防ぐ
DROP VIEW IF EXISTS employee_department_view;

ビューの定義変更

-- ビューを置き換え（OR REPLACE）
CREATE OR REPLACE VIEW employee_department_view AS
SELECT
    e.employee_id,
    e.employee_name,
    d.department_name
    -- salaryカラムを削除した定義に変更
FROM employees e
INNER JOIN departments d ON e.department_id = d.department_id;
view_info.sql（ビュー情報の確認） 
-- データベース内の全ビューを確認
SELECT table_name
FROM information_schema.views
WHERE table_schema = 'public';

-- ビューの定義を確認
SELECT pg_get_viewdef('employee_department_view'::regclass, true);

7.4 マテリアライズドビューとは

マテリアライズドビュー（Materialized View）は、ビューと異なり、クエリ実行結果を実際にディスク上に保存します。 通常のビューは毎回元のテーブルからデータを取得しますが、マテリアライズドビューは結果を「スナップショット」として保存するため、高速にデータを取得できます。

ただし、元のテーブルが更新されても自動的には反映されないため、定期的に「REFRESH」する必要があります。

7.5 マテリアライズドビューの作成と利用

マテリアライズドビューを作成して、パフォーマンスを改善する方法を学びましょう。

基本的なマテリアライズドビューの作成

materialized_view.sql（マテリアライズドビュー作成） 
-- マテリアライズドビューの作成
CREATE MATERIALIZED VIEW department_stats_mv AS
SELECT
    d.department_name,
    COUNT(e.employee_id) AS employee_count,
    AVG(e.salary)::INTEGER AS avg_salary,
    SUM(e.salary) AS total_salary
FROM departments d
LEFT JOIN employees e ON d.department_id = e.department_id
GROUP BY d.department_name;

-- 通常のSELECT文と同じように使える（高速）
SELECT * FROM department_stats_mv;

-- データを最新化（元のテーブルが更新された後に実行）
REFRESH MATERIALIZED VIEW department_stats_mv;

マテリアライズドビューのインデックス

マテリアライズドビューにはインデックスを作成できます。これにより、さらに高速なデータ取得が可能になります。

mv_index.sql（インデックス作成） 
-- マテリアライズドビューにインデックスを作成
CREATE INDEX idx_mv_dept_name
ON department_stats_mv(department_name);

-- 特定の部署を高速に検索できる
SELECT * FROM department_stats_mv
WHERE department_name = '開発部';

CONCURRENT REFRESHの利用

通常のREFRESHは、更新中にビューへのアクセスがブロックされます。REFRESH MATERIALIZED VIEW CONCURRENTLYを使うと、更新中もビューにアクセスできます（ユニークインデックスが必要）。

concurrent_refresh.sql（CONCURRENT REFRESH） 
-- ユニークインデックスを作成（CONCURRENT REFRESHに必要）
CREATE UNIQUE INDEX idx_mv_dept_name_unique
ON department_stats_mv(department_name);

-- 並行更新（アクセスをブロックしない）
REFRESH MATERIALIZED VIEW CONCURRENTLY department_stats_mv;

CONCURRENT REFRESHの制約

CONCURRENT REFRESHを使うには、マテリアライズドビューにユニークインデックスが必要です。また、通常のREFRESHよりも時間がかかる場合があります。

7.7 ビューとマテリアライズドビューの使い分け

ビューとマテリアライズドビューをどのように使い分けるべきか、具体的な状況を見てみましょう。

通常のビューを使うケース

  • 常に最新データが必要な場合（リアルタイム性重視）
  • 元のクエリが十分高速な場合
  • 頻繁に更新されるデータの表示
  • ディスク容量を節約したい場合

マテリアライズドビューを使うケース

  • 複雑で時間がかかる集計処理（パフォーマンス重視）
  • 日次・週次レポートなど、更新頻度が低いデータ
  • 大量のJOINや集計を含むクエリ
  • データ分析・BIツール用のサマリーテーブル

トラブルシューティング

エラー: ビューを削除できない

他のビューやマテリアライズドビューがそのビューを参照している可能性があります。CASCADE オプションを使って依存するビューも一緒に削除するか、先に依存しているビューを削除してください。

DROP VIEW ビュー名 CASCADE;

マテリアライズドビューが更新されない

マテリアライズドビューは自動更新されません。元のテーブルを更新した後、必ず REFRESH MATERIALIZED VIEW を実行してください。定期的な更新が必要な場合は、cronやpg_cronを使ったスケジュール実行を検討しましょう。

CONCURRENT REFRESHが使えない

CONCURRENT REFRESHを使うには、マテリアライズドビューにユニークインデックスが必要です。CREATE UNIQUE INDEX でユニークインデックスを作成してから再度実行してください。

理解度確認クイズ

  1. Q1. 通常のビューとマテリアライズドビューの最大の違いは何ですか？

    回答を表示

    答え: 通常のビューはクエリ実行結果を保存せず、アクセスするたびに元のテーブルからデータを取得します。一方、マテリアライズドビューは実行結果をディスクに保存するため、高速にデータを取得できますが、元のテーブルが更新されても自動的には反映されません（REFRESH必要）。

  2. Q2. ビューを使うメリットを3つ挙げてください。

    回答を表示

    答え（例）:
    1. 複雑なクエリを再利用できる - 長いJOINやSUBQUERYを毎回書く必要がない
    2. データの抽象化 - 特定のカラムだけを見せることでセキュリティ向上
    3. データの整合性 - 同じクエリロジックを複数の場所で再利用できる
    （他にも、可読性向上、メンテナンス性向上なども正解です）

  3. Q3. マテリアライズドビューを使うべき状況はどのようなときですか？

    回答を表示

    答え（例）:
    1. 複雑で時間がかかる集計処理をする場合（パフォーマンス重視）
    2. 日次・週次レポートなど、リアルタイム性が求められない場合
    3. 大量のJOINや集計を含むクエリで、頻繁にアクセスされる場合
    4. データ分析・BIツール用のサマリーデータを用意する場合

第7章のまとめ

  • ビューは複雑なクエリを「仮想テーブル」として保存し、再利用できる機能
  • マテリアライズドビューはクエリ結果をディスクに保存し、高速アクセスを実現
  • 通常のビューは常に最新データ、マテリアライズドビューはREFRESHが必要
  • リアルタイム性が必要なら通常のビュー、パフォーマンスが必要ならマテリアライズドビューを選択
  • CONCURRENT REFRESHを使うには、ユニークインデックスが必要
