MySQLのデータ型 - 適切な型選びでパフォーマンスを最適化

MySQLで使用できる数値型、文字列型、日付時刻型の特徴と使い分け、データ型とストレージエンジンの関係を理解し、適切な型選びでパフォーマンスを最適化する方法を学びます。

この章で学ぶこと

  • 数値型(INT、BIGINT、DECIMAL)の特徴と使い分け
  • 文字列型(VARCHAR、TEXT、ENUM)の特徴と使い分け
  • 日付時刻型(DATE、DATETIME、TIMESTAMP)の違い
  • データ型がストレージとパフォーマンスに与える影響

3.1 MySQLのデータ型の分類

MySQLでは、データの種類に応じて適切なデータ型を選択することが重要です。適切な型を選ぶことで、ストレージ容量の節約、クエリのパフォーマンス向上、データの整合性確保が実現できます。

MySQLのデータ型は大きく以下の5つのカテゴリに分類されます。

flowchart TD A[MySQLのデータ型] --> B[数値型] A --> C[文字列型] A --> D[日付時刻型] A --> E[バイナリ型] A --> F[空間データ型] B --> B1["整数型: INT, BIGINT, TINYINT"] B --> B2["固定小数点: DECIMAL, NUMERIC"] B --> B3["浮動小数点: FLOAT, DOUBLE"] C --> C1["可変長: VARCHAR"] C --> C2["固定長: CHAR"] C --> C3["長文: TEXT, LONGTEXT"] C --> C4["列挙: ENUM, SET"] D --> D1["日付: DATE"] D --> D2["日時: DATETIME, TIMESTAMP"] D --> D3["時刻: TIME"] D --> D4["年: YEAR"]

データ型選びの基本原則

格納するデータの性質を理解し、必要最小限のサイズの型を選ぶことが基本です。例えば、0-100の範囲の数値を格納する場合、BIGINTではなくTINYINTを使うことでストレージを節約できます。

3.2 数値型の種類と使い分け

数値型は整数型、固定小数点型、浮動小数点型の3つに分けられます。それぞれの特徴を理解して使い分けましょう。

3.2.1 整数型

型名 バイト数 符号付き範囲 使用例
TINYINT 1 -128 〜 127 年齢、フラグ
SMALLINT 2 -32,768 〜 32,767 小規模カウンター
MEDIUMINT 3 -8,388,608 〜 8,388,607 中規模カウンター
INT 4 -21億 〜 21億 ID、一般的な数値
BIGINT 8 -922京 〜 922京 大規模ID、タイムスタンプ
整数型の定義例 
-- ユーザーテーブルの例
CREATE TABLE users (
    user_id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,  -- 0〜42億まで対応
    age TINYINT UNSIGNED,                              -- 0〜255(年齢に十分)
    login_count INT UNSIGNED DEFAULT 0,                -- ログイン回数
    points BIGINT UNSIGNED DEFAULT 0                   -- ポイント(大きな値に対応)
);

UNSIGNED属性の活用

負の値が不要な場合はUNSIGNEDを指定することで、格納できる正の値の範囲が2倍になります。IDやカウンター、金額などはUNSIGNEDを使うのが一般的です。

3.2.2 固定小数点型と浮動小数点型

小数を扱う場合は、精度が重要な場合はDECIMAL、科学計算など精度より範囲が重要な場合はFLOAT/DOUBLEを使います。

型名 精度 用途
DECIMAL(M,D) 正確 金額、正確な計算が必要
FLOAT 近似(単精度) 科学計算、座標
DOUBLE 近似(倍精度) 高精度な科学計算
小数型の定義例 
-- 商品テーブルの例
CREATE TABLE products (
    product_id INT UNSIGNED PRIMARY KEY,
    price DECIMAL(10,2),       -- 金額(例: 99999999.99)
    weight FLOAT,               -- 重さ(近似値でOK)
    latitude DOUBLE,            -- 緯度(高精度)
    longitude DOUBLE            -- 経度(高精度)
);

金額はDECIMALを使う

FLOATやDOUBLEは近似値を扱うため、金額の計算で誤差が生じる可能性があります。ECサイトや会計システムでは必ずDECIMAL型を使用してください。

3.3 文字列型の種類と使い分け

文字列型は、固定長のCHAR、可変長のVARCHAR、長文用のTEXT系、選択肢型のENUM/SETがあります。

3.3.1 CHAR vs VARCHAR

項目 CHAR(M) VARCHAR(M)
長さ 固定長(M文字) 可変長(最大M文字)
ストレージ 常にM文字分確保 実際の長さ+1-2バイト
パフォーマンス 高速(固定長) やや遅い
用途 国コード、郵便番号 氏名、メールアドレス
CHAR/VARCHARの定義例 
CREATE TABLE customers (
    customer_id INT UNSIGNED PRIMARY KEY,
    country_code CHAR(2),           -- 'JP', 'US'など固定2文字
    postal_code CHAR(7),             -- '1234567'固定7文字
    name VARCHAR(100),               -- 氏名(可変長)
    email VARCHAR(255),              -- メールアドレス(可変長)
    phone VARCHAR(20)                -- 電話番号(可変長)
);

3.3.2 TEXT系の型

長文を格納する場合はTEXT系の型を使います。4種類のサイズがあり、用途に応じて選択します。

型名 最大長 使用例
TINYTEXT 255文字 短いメモ
TEXT 65,535文字 記事、コメント
MEDIUMTEXT 16MB 長文記事
LONGTEXT 4GB 書籍全文、ログ

3.3.3 ENUM型とSET型

選択肢が固定されている場合は、ENUM型(単一選択)またはSET型(複数選択)を使うことでストレージを節約し、データの整合性を保てます。

ENUM/SETの定義例 
CREATE TABLE articles (
    article_id INT UNSIGNED PRIMARY KEY,
    title VARCHAR(200),
    -- ENUMは単一選択(1つだけ選べる)
    status ENUM('draft', 'published', 'archived') DEFAULT 'draft',
    -- SETは複数選択(複数選べる)
    tags SET('tech', 'news', 'tutorial', 'review')
);

-- 使用例
INSERT INTO articles (title, status, tags)
VALUES ('MySQL入門', 'published', 'tech,tutorial');

ENUMのメリット・デメリット

メリット: ストレージ効率が良く、不正な値の入力を防げる
デメリット: 選択肢を追加・変更する際にテーブル構造の変更(ALTER TABLE)が必要

3.4 日付時刻型の種類と使い分け

日付や時刻を扱う型として、DATE、TIME、DATETIME、TIMESTAMP、YEARがあります。特にDATETIMEとTIMESTAMPの違いは重要です。

型名 範囲 特徴 使用例
DATE 1000-01-01 〜 9999-12-31 日付のみ 誕生日、記念日
TIME -838:59:59 〜 838:59:59 時刻のみ 営業時間、経過時間
DATETIME 1000-01-01 〜 9999-12-31 タイムゾーン非対応 イベント日時
TIMESTAMP 1970-01-01 〜 2038-01-19 タイムゾーン対応 作成日時、更新日時
YEAR 1901 〜 2155 年のみ 卒業年、製造年

3.4.1 DATETIMEとTIMESTAMPの違い

DATETIME

  • 広い範囲(1000年〜9999年)
  • タイムゾーン変換なし
  • 8バイト使用
  • イベント日時向き

TIMESTAMP

  • 範囲制限(1970年〜2038年)
  • タイムゾーン自動変換
  • 4バイト使用
  • 作成/更新日時向き
日付時刻型の定義例 
CREATE TABLE events (
    event_id INT UNSIGNED PRIMARY KEY,
    title VARCHAR(200),
    event_date DATE,                    -- イベント日付
    start_time TIME,                     -- 開始時刻
    event_datetime DATETIME,             -- イベント日時(タイムゾーン固定)
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,  -- 作成日時(自動設定)
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
               ON UPDATE CURRENT_TIMESTAMP            -- 更新日時(自動更新)
);

created_at/updated_atの自動管理

TIMESTAMPにDEFAULT CURRENT_TIMESTAMPとON UPDATE CURRENT_TIMESTAMPを指定することで、レコードの作成日時と更新日時を自動的に記録できます。アプリケーション側で日時を設定する必要がなく便利です。

2038年問題

TIMESTAMPは2038年1月19日までしか扱えません(32ビット符号付き整数の上限)。2038年以降の日時を扱う必要がある場合はDATETIMEを使用してください。

3.5 データ型とパフォーマンス

適切なデータ型を選ぶことは、単にデータを正しく格納するだけでなく、ストレージ効率とクエリパフォーマンスに大きく影響します。

3.5.1 ストレージ効率

データ型によるストレージサイズの違い

データ型の選択による100万件の年齢データのストレージサイズ比較:

BIGINT(8バイト):      8MB
INT(4バイト):         4MB
SMALLINT(2バイト):    2MB
TINYINT(1バイト):     1MB  ← 最適な選択

年齢は0〜150程度なので、TINYINTが最適です。

3.5.2 インデックスとデータ型

インデックスのサイズもデータ型に依存します。小さいデータ型を使うことで、インデックスもコンパクトになり、検索が高速化されます。

flowchart LR A[適切な型選択] --> B[ストレージ節約] A --> C[インデックス効率化] B --> D[ディスクI/O削減] C --> D D --> E[クエリ高速化]

データ型選択のベストプラクティス

  • • データの範囲に合った最小の型を選ぶ
  • • 固定長データはCHARを、可変長はVARCHARを使う
  • • 金額は必ずDECIMALを使う
  • • created_at/updated_atはTIMESTAMPを活用
  • • 将来の拡張性も考慮する(ただし過剰には大きくしない)

実習: 適切なデータ型を選んでテーブルを作ろう

以下のブログシステムのテーブルを、適切なデータ型を選んで作成してみましょう。

実習コード 
-- データベース作成
CREATE DATABASE blog_db CHARACTER SET utf8mb4;
USE blog_db;

-- ブログ記事テーブル
CREATE TABLE posts (
    post_id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    title VARCHAR(200) NOT NULL,
    content TEXT NOT NULL,
    view_count INT UNSIGNED DEFAULT 0,
    is_published TINYINT(1) DEFAULT 0,        -- 0=下書き, 1=公開
    published_at DATETIME,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- データ挿入テスト
INSERT INTO posts (title, content, is_published, published_at)
VALUES
('MySQL入門', 'MySQLのデータ型について解説します。', 1, NOW()),
('データベース設計', 'テーブル設計の基本を学びましょう。', 0, NULL);

-- 確認
SELECT * FROM posts;

理解度確認クイズ

  1. Q1. ECサイトで商品価格を格納する場合、どのデータ型を使うべきですか?またその理由は?

    回答を表示

    DECIMAL型を使うべきです。FLOATやDOUBLEは浮動小数点型で近似値を扱うため、金額計算で誤差が生じる可能性があります。金額のような正確な計算が必要なデータには、固定小数点型のDECIMALを使用します。例: DECIMAL(10,2)で99999999.99まで格納可能。

  2. Q2. DATETIMEとTIMESTAMPの違いを3つ挙げてください。

    回答を表示

    1. 範囲: DATETIMEは1000年〜9999年、TIMESTAMPは1970年〜2038年
    2. タイムゾーン: DATETIMEは非対応、TIMESTAMPは自動変換対応
    3. ストレージ: DATETIMEは8バイト、TIMESTAMPは4バイト
    一般的に、イベント日時にはDATETIME、作成/更新日時にはTIMESTAMPを使用します。

  3. Q3. なぜ適切なデータ型を選ぶことが重要なのですか?

    回答を表示

    適切なデータ型を選ぶことで、ストレージ効率の向上、クエリパフォーマンスの改善、データ整合性の確保が実現できます。例えば、年齢にBIGINTではなくTINYINTを使うことで、ストレージサイズが8分の1になり、インデックスもコンパクトになるため検索が高速化されます。

この章のまとめ

  • MySQLのデータ型は数値型、文字列型、日付時刻型、バイナリ型、空間データ型の5つに分類される
  • 整数型はデータ範囲に応じて最小の型を選び、負の値が不要な場合はUNSIGNEDを使う
  • 金額など正確な計算が必要な場合はDECIMAL型を使い、FLOATやDOUBLEは避ける
  • 文字列型は固定長ならCHAR、可変長ならVARCHAR、長文ならTEXTを選択する
  • DATETIMEはタイムゾーン非対応で広範囲、TIMESTAMPはタイムゾーン対応で作成/更新日時向き
  • 適切な型選びでストレージ効率とクエリパフォーマンスを大幅に改善できる
