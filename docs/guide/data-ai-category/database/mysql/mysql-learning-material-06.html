MySQLのインデックス - クエリを高速化する仕組み

インデックスの基本概念とMySQLでの作成方法を学習します。B-treeインデックス、複合インデックス、フルテキストインデックスの使い分け、EXPLAINコマンドでクエリの実行計画を分析する方法を習得します。

この章で学ぶこと

  • インデックスの基本概念と役割
  • B-treeインデックスの仕組みと作成方法
  • 複合インデックスとフルテキストインデックスの使い分け
  • EXPLAINコマンドでクエリの実行計画を分析する方法

6.1 インデックスとは - 本の索引のような仕組み

インデックス（索引）は、データベーステーブル内のデータを高速に検索するための仕組みです。本の巻末にある索引と同じように、特定のキーワードが「どこにあるか」を素早く見つけるために使用されます。

例えば、100万行のテーブルから特定の顧客情報を探す場合、インデックスがなければ全行をスキャン（フルテーブルスキャン）する必要がありますが、インデックスがあれば数十行程度のアクセスで済みます。

flowchart TD A[検索クエリ実行] --> B{インデックスあり?} B -->|なし| C[フルテーブルスキャン
全行を順番に検索] B -->|あり| D[インデックスを使用
必要な行だけアクセス] C --> E[処理時間: 数秒〜数十秒] D --> F[処理時間: ミリ秒単位]

インデックスのメリット

  • 検索速度の向上 - WHERE句、JOIN、ORDER BYが高速化
  • ユニーク制約 - UNIQUE INDEXで重複データを防止
  • データ整合性 - PRIMARY KEYやFOREIGN KEYにも使用

インデックスのデメリット

  • ストレージ容量の増加 - インデックスもディスク領域を消費
  • INSERT/UPDATE/DELETEの速度低下 - データ変更時にインデックスも更新が必要
  • メンテナンスコスト - 適切な設計と定期的な見直しが必要

6.2 B-treeインデックス - MySQLの標準インデックス

MySQLのInnoDBストレージエンジンでは、B-tree（Balanced Tree）というデータ構造を使用してインデックスを管理しています。B-treeは木構造でデータを保持し、高速な検索・挿入・削除を実現します。

インデックスの作成

基本的なインデックス作成 
-- 単一カラムインデックス
CREATE INDEX idx_lastname ON employees(last_name);

-- ユニークインデックス(重複を許可しない)
CREATE UNIQUE INDEX idx_email ON users(email);

-- 既存インデックスの確認
SHOW INDEX FROM employees;
テーブル作成時にインデックスを定義 
-- PRIMARY KEYは自動的にインデックスが作成される
CREATE TABLE products (
    product_id INT PRIMARY KEY,  -- 主キーインデックス
    product_name VARCHAR(100),
    category_id INT,
    price DECIMAL(10, 2),
    created_at DATETIME,

    INDEX idx_category (category_id),  -- カテゴリ検索用
    INDEX idx_price (price)            -- 価格検索用
) ENGINE=InnoDB;

インデックスの削除

インデックス削除の方法 
-- DROP INDEX文
DROP INDEX idx_lastname ON employees;

-- ALTER TABLE文でも削除可能
ALTER TABLE employees DROP INDEX idx_lastname;

6.3 複合インデックス - 複数カラムを組み合わせる

複合インデックス（Composite Index）は、複数のカラムを組み合わせたインデックスです。複数の条件で検索することが多い場合に有効です。

複合インデックスの作成 
-- 姓と名の複合インデックス
CREATE INDEX idx_name ON employees(last_name, first_name);

-- 都道府県と市区町村の複合インデックス
CREATE INDEX idx_location ON addresses(prefecture, city);

複合インデックスの使用ルール（重要）

複合インデックス(last_name, first_name)の場合：

  • ✅ 使用される - WHERE last_name = '山田'
  • ✅ 使用される - WHERE last_name = '山田' AND first_name = '太郎'
  • ❌ 使用されない - WHERE first_name = '太郎' （左端カラムがない）

左端一致の原則: 複合インデックスは最初のカラムから順番に使用される必要があります。

複合インデックスの効果的な使用例 
-- 商品検索で使用する複合インデックス
CREATE INDEX idx_product_search
ON products(category_id, price, created_at);

-- 以下のクエリで効果的に使用される
SELECT * FROM products
WHERE category_id = 5           -- 1番目のカラム
  AND price BETWEEN 1000 AND 5000  -- 2番目のカラム
ORDER BY created_at DESC;       -- 3番目のカラム

6.4 フルテキストインデックス - 文書検索に特化

フルテキストインデックス（Full-Text Index）は、文章や長いテキストから特定のキーワードを検索するためのインデックスです。ブログ記事、商品説明、コメントなどの全文検索に使用します。

フルテキストインデックスの作成 
-- フルテキストインデックスの作成
CREATE TABLE articles (
    article_id INT PRIMARY KEY,
    title VARCHAR(200),
    content TEXT,
    created_at DATETIME,

    FULLTEXT INDEX ft_content (title, content)
) ENGINE=InnoDB;
フルテキスト検索の実行 
-- MATCH...AGAINST構文を使用
SELECT article_id, title
FROM articles
WHERE MATCH(title, content) AGAINST('MySQL インデックス');

-- ブーリアンモード（AND, ORなどが使える）
SELECT article_id, title
FROM articles
WHERE MATCH(title, content)
      AGAINST('+MySQL +パフォーマンス -Oracle' IN BOOLEAN MODE);

ブーリアンモードの演算子

  • + - 必須キーワード（AND条件）
  • - - 除外キーワード（NOT条件）
  • * - ワイルドカード（前方一致）
  • "" - フレーズ検索（完全一致）

6.5 EXPLAINコマンド - クエリの実行計画を確認

EXPLAINコマンドは、MySQLがクエリをどのように実行するか（実行計画）を表示します。インデックスが正しく使用されているか、パフォーマンスのボトルネックがどこにあるかを確認できます。

EXPLAINの基本的な使い方 
-- SELECT文の前にEXPLAINをつける
EXPLAIN SELECT * FROM employees WHERE last_name = '山田';

-- より詳細な情報を表示
EXPLAIN ANALYZE SELECT * FROM employees WHERE last_name = '山田';

EXPLAIN結果の主要カラム

カラム 意味 重要な値
type アクセス方法 const > ref > range > index > ALL
possible_keys 使用可能なインデックス 候補となるインデックス一覧
key 実際に使用されたインデックス NULLならフルスキャン
rows スキャン行数の推定 少ないほど良い
Extra 追加情報 Using where, Using index等

typeカラムの良い順序（高速順）

  1. const - PRIMARY KEYやUNIQUEインデックスで1行だけ
  2. ref - 通常のインデックス使用（複数行）
  3. range - 範囲検索（BETWEEN、<、>など）
  4. index - インデックス全スキャン
  5. ALL - フルテーブルスキャン（最も遅い）

実習: インデックスのパフォーマンス比較

実際にインデックスの有無でクエリ速度がどう変わるか確認してみましょう。

サンプルデータの作成 
-- 大量データを持つテーブル作成
CREATE TABLE test_performance (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_name VARCHAR(50),
    email VARCHAR(100),
    age INT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 10万行のデータ挿入（プロシージャ使用）
DELIMITER //
CREATE PROCEDURE insert_test_data()
BEGIN
    DECLARE i INT DEFAULT 1;
    WHILE i <= 100000 DO
        INSERT INTO test_performance (user_name, email, age)
        VALUES (CONCAT('user', i), CONCAT('user', i, '@example.com'),
                FLOOR(20 + RAND() * 50));
        SET i = i + 1;
    END WHILE;
END//
DELIMITER ;

CALL insert_test_data();
インデックスなしでの検索 
-- 実行時間を計測
SET profiling = 1;

-- インデックスなしで検索
SELECT * FROM test_performance WHERE age = 30;

-- 実行計画確認
EXPLAIN SELECT * FROM test_performance WHERE age = 30;

-- 実行時間確認
SHOW PROFILES;
インデックスありでの検索 
-- ageカラムにインデックス作成
CREATE INDEX idx_age ON test_performance(age);

-- 同じクエリを実行
SELECT * FROM test_performance WHERE age = 30;

-- 実行計画確認（インデックスが使われているか）
EXPLAIN SELECT * FROM test_performance WHERE age = 30;

-- 実行時間確認
SHOW PROFILES;

6.6 インデックス設計のベストプラクティス

やるべきこと

  • WHERE句でよく使うカラムにインデックス
  • JOIN条件のカラムにインデックス
  • ORDER BYで使うカラムにインデックス
  • 選択性の高いカラムを優先
  • 複合インデックスは左端一致を意識

避けるべきこと

  • すべてのカラムにインデックス
  • 小さなテーブルへの過剰なインデックス
  • 選択性の低いカラム（性別など）
  • 頻繁に更新されるカラム
  • 使用されないインデックスの放置

選択性（Selectivity）とは

カラム内のユニークな値の割合を示す指標です。選択性が高いほどインデックスの効果が大きくなります。

  • 高選択性（インデックス向き）: メールアドレス、注文番号、ユーザーID
  • 低選択性（インデックス不向き）: 性別、都道府県、フラグ（true/false）

理解度確認クイズ

  1. Q1. インデックスのメリットとデメリットをそれぞれ2つずつ挙げてください。

    回答を表示

    メリット: ①検索速度が大幅に向上する ②ユニーク制約やデータ整合性の維持に使える
    デメリット: ①ストレージ容量を消費する ②INSERT/UPDATE/DELETE時の処理が遅くなる

  2. Q2. 複合インデックス(last_name, first_name)が使用される検索条件と使用されない検索条件を説明してください。

    回答を表示

    使用される: WHERE last_name = '山田' または WHERE last_name = '山田' AND first_name = '太郎'
    使用されない: WHERE first_name = '太郎' （左端カラムがない）
    複合インデックスは左端一致の原則により、最初のカラムから順番に使用される必要があります。

  3. Q3. EXPLAIN結果の「type」カラムで、最も高速なアクセス方法と最も遅いアクセス方法は何ですか?

    回答を表示

    最も高速: const（PRIMARY KEYやUNIQUE INDEXで1行だけアクセス）
    最も遅い: ALL（フルテーブルスキャン、すべての行を読み取る）

この章のまとめ

  • インデックスは検索速度を大幅に向上させるが、ストレージ消費と更新処理の低下というトレードオフがある
  • B-treeインデックスがMySQLの標準で、CREATE INDEXまたはテーブル定義時に作成できる
  • 複合インデックスは左端一致の原則に従い、最初のカラムから順番に条件指定する必要がある
  • フルテキストインデックスはMATCH...AGAINST構文で文書検索に使用できる
  • EXPLAINコマンドでクエリの実行計画を確認し、インデックスが正しく使われているか検証できる
