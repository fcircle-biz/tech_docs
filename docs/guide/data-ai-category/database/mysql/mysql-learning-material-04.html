MySQLのテーブル設計 - 制約とストレージエンジンの基礎

MySQLでのテーブル作成と各種制約の設定方法を学習します。PRIMARY KEY、FOREIGN KEY、UNIQUE、CHECK、NOT NULLなどの制約と、InnoDB、MyISAMなどストレージエンジンの違いと選び方について理解します。

この章で学ぶこと

  • MySQLでのテーブル作成構文とデータ型の指定方法
  • PRIMARY KEY、FOREIGN KEY、UNIQUE、CHECK、NOT NULL制約の使い方
  • ストレージエンジン（InnoDB、MyISAM）の違いと選び方
  • テーブル設計のベストプラクティス

4.1 テーブルの作成基本

MySQLでテーブルを作成するには、CREATE TABLE文を使用します。SQL入門で学んだ基本構文に加えて、MySQLならではの設定オプションがいくつかあります。

基本的なテーブル作成構文 
-- シンプルなテーブル作成例
CREATE TABLE users (
    user_id INT,              -- 整数型のユーザーID
    username VARCHAR(50),     -- 最大50文字のユーザー名
    email VARCHAR(100),       -- 最大100文字のメールアドレス
    created_at DATETIME       -- 作成日時
);

-- データ型とデフォルト値を指定した例
CREATE TABLE products (
    product_id INT,
    product_name VARCHAR(100),
    price DECIMAL(10, 2),     -- 最大10桁、小数点以下2桁
    stock INT DEFAULT 0,      -- デフォルト値を0に設定
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLEのポイント

  • カラム名 - 半角英数字とアンダースコアを使用（日本語も可能だが推奨されない）
  • データ型 - 適切なデータ型を選択することでストレージ容量を節約
  • DEFAULT値 - INSERT時に値を指定しなかった場合のデフォルト値を設定可能
  • TIMESTAMP - CURRENT_TIMESTAMPで自動的に現在時刻を記録

4.2 PRIMARY KEY制約 - 主キーの設定

PRIMARY KEY（主キー）は、テーブル内の各行を一意に識別するためのカラムです。MySQLでは主キーを設定すると、自動的にインデックスが作成され、検索が高速化されます。

PRIMARY KEYの設定方法 
-- 方法1: カラム定義時に指定
CREATE TABLE users (
    user_id INT PRIMARY KEY,  -- user_idを主キーに設定
    username VARCHAR(50),
    email VARCHAR(100)
);

-- 方法2: AUTO_INCREMENTで自動採番
CREATE TABLE users (
    user_id INT AUTO_INCREMENT PRIMARY KEY,  -- 1から自動採番
    username VARCHAR(50),
    email VARCHAR(100)
);

-- 方法3: テーブル制約として指定
CREATE TABLE users (
    user_id INT AUTO_INCREMENT,
    username VARCHAR(50),
    email VARCHAR(100),
    PRIMARY KEY (user_id)     -- テーブル末尾で指定
);

AUTO_INCREMENTのメリット

AUTO_INCREMENTを使用すると、INSERT時にIDを指定する必要がなく、MySQLが自動的に連番を割り当てます。削除された行のIDは再利用されません。

複合主キー（2つ以上のカラム） 
-- 複数のカラムを組み合わせて主キーとする
CREATE TABLE order_details (
    order_id INT,             -- 注文ID
    product_id INT,           -- 商品ID
    quantity INT,
    PRIMARY KEY (order_id, product_id)  -- 2つの組み合わせを主キーに
);

4.3 FOREIGN KEY制約 - 参照整合性の確保

FOREIGN KEY（外部キー）は、他のテーブルの主キーを参照するカラムです。外部キー制約を設定することで、参照整合性を保ち、存在しないデータへの参照を防ぎます。

erDiagram USERS ||--o{ ORDERS : "places" USERS { int user_id PK varchar username varchar email } ORDERS { int order_id PK int user_id FK datetime order_date decimal total_amount }
FOREIGN KEYの設定方法 
-- まず親テーブル（参照される側）を作成
CREATE TABLE users (
    user_id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50),
    email VARCHAR(100)
);

-- 次に子テーブル（参照する側）を作成
CREATE TABLE orders (
    order_id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT,
    order_date DATETIME,
    total_amount DECIMAL(10, 2),
    FOREIGN KEY (user_id) REFERENCES users(user_id)
);

外部キー制約の注意点

外部キー制約を設定すると、親テーブルに存在しないIDを子テーブルに挿入できなくなります。また、親テーブルのレコードを削除する際も制約が働きます。

ON DELETEとON UPDATEオプション 
-- 親レコード削除時の動作を指定
CREATE TABLE orders (
    order_id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT,
    order_date DATETIME,
    FOREIGN KEY (user_id)
        REFERENCES users(user_id)
        ON DELETE CASCADE      -- 親削除時に子も削除
        ON UPDATE CASCADE      -- 親更新時に子も更新
);

-- その他のオプション
-- ON DELETE RESTRICT  : 親を削除できない（デフォルト）
-- ON DELETE SET NULL  : 親削除時に子をNULLに設定
-- ON DELETE NO ACTION : RESTRICTと同じ

4.4 その他の制約 - UNIQUE、NOT NULL、CHECK

UNIQUE制約 - 重複を許さない

UNIQUE制約は、カラムの値が重複しないことを保証します。例えば、メールアドレスやユーザー名など、同じ値が複数存在してはいけない場合に使用します。

UNIQUE制約の設定 
-- カラム定義時に指定
CREATE TABLE users (
    user_id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) UNIQUE,      -- 重複不可
    email VARCHAR(100) UNIQUE,        -- 重複不可
    password_hash VARCHAR(255)
);

-- テーブル制約として指定
CREATE TABLE users (
    user_id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50),
    email VARCHAR(100),
    UNIQUE (username),
    UNIQUE (email)
);

NOT NULL制約 - NULL値を許さない

NOT NULL制約は、カラムにNULL値（値が未設定の状態）を許さないことを指定します。必須項目に対して設定します。

NOT NULL制約の設定 
CREATE TABLE products (
    product_id INT AUTO_INCREMENT PRIMARY KEY,
    product_name VARCHAR(100) NOT NULL,  -- 必須項目
    price DECIMAL(10, 2) NOT NULL,       -- 必須項目
    description TEXT,                    -- NULL許可（任意項目）
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CHECK制約 - 値の範囲を制限

CHECK制約は、カラムに格納できる値の条件を指定します。MySQL 8.0.16以降でサポートされています。

CHECK制約の設定 
-- 値の範囲を制限
CREATE TABLE products (
    product_id INT AUTO_INCREMENT PRIMARY KEY,
    product_name VARCHAR(100) NOT NULL,
    price DECIMAL(10, 2) NOT NULL CHECK (price >= 0),  -- 0以上
    stock INT CHECK (stock >= 0),                      -- 0以上
    rating DECIMAL(2, 1) CHECK (rating >= 0.0 AND rating <= 5.0)
);
制約 役割 使用例
PRIMARY KEY 各行を一意に識別 user_id、product_id
FOREIGN KEY 他テーブルとの関連付け ordersテーブルのuser_id
UNIQUE 値の重複を防ぐ email、username
NOT NULL NULL値を許さない 氏名、価格など必須項目
CHECK 値の範囲を制限 価格 >= 0、評価 0-5

4.5 ストレージエンジン - InnoDBとMyISAMの違い

MySQLの大きな特徴の一つが、テーブルごとに異なるストレージエンジンを選択できることです。ストレージエンジンは、データの保存方法や検索方法を決定する仕組みです。

（デフォルト）"] B --> D["MyISAM"] B --> E["Memory"] B --> F["その他"] C --> C1["トランザクション対応"] C --> C2["外部キー制約対応"] C --> C3["行ロック"] D --> D1["高速読み取り"] D --> D2["全文検索"] D --> D3["テーブルロック"]

InnoDB - 現在の標準エンジン

MySQL 5.5以降のデフォルトストレージエンジンです。トランザクション処理や外部キー制約に対応しており、データの整合性を重視する用途に適しています。

InnoDBの特徴

  • トランザクション対応 - COMMIT、ROLLBACKが使える
  • 外部キー制約 - FOREIGN KEY制約に対応
  • 行レベルロック - 同時アクセス時の性能が高い
  • クラッシュリカバリ - データの安全性が高い
  • ACID準拠 - データの整合性を保証

MyISAM - 読み取り特化エンジン

MySQL 5.5より前のデフォルトエンジンです。読み取り性能は高いですが、トランザクションに対応していません。現在は特別な理由がない限りInnoDBを使用することが推奨されています。

MyISAMの特徴

  • 高速な読み取り - SELECT文の実行が速い
  • 全文検索 - FULLTEXT INDEXに対応
  • テーブルレベルロック - 書き込み時は全体がロック
  • トランザクション非対応 - COMMIT/ROLLBACKが使えない
  • 外部キー非対応 - FOREIGN KEY制約が使えない
項目 InnoDB MyISAM
トランザクション 対応 非対応
外部キー制約 対応 非対応
ロック単位 行ロック テーブルロック
読み取り速度 高速 より高速
適した用途 一般的な用途全般 読み取り専用システム
全文検索 対応（5.6以降） 対応
ストレージエンジンの指定方法 
-- InnoDBで作成（デフォルトなので省略可）
CREATE TABLE users (
    user_id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50)
) ENGINE=InnoDB;

-- MyISAMで作成
CREATE TABLE logs (
    log_id INT AUTO_INCREMENT PRIMARY KEY,
    log_message TEXT,
    created_at TIMESTAMP
) ENGINE=MyISAM;

-- 既存テーブルのエンジンを変更
ALTER TABLE users ENGINE=InnoDB;

推奨：特別な理由がない限りInnoDBを使用

現在のMySQLではInnoDBがデフォルトであり、ほとんどの用途に適しています。トランザクション、外部キー制約、クラッシュリカバリなど、信頼性の高いシステムを構築できます。MyISAMは読み取り専用ログテーブルなど限定的な用途でのみ検討してください。

4.6 テーブル設計のベストプラクティス

主キーは必ず設定

すべてのテーブルにPRIMARY KEYを設定しましょう。AUTO_INCREMENTを使った連番IDが一般的です。

外部キーで整合性確保

関連テーブル間はFOREIGN KEY制約を設定し、データの整合性を保ちましょう。

適切なデータ型を選択

必要以上に大きなデータ型を使わず、ストレージ容量と検索速度を最適化しましょう。

制約で不正データを防ぐ

NOT NULL、UNIQUE、CHECK制約を活用し、アプリケーション側だけでなくDB側でもデータの妥当性を保証しましょう。

ベストプラクティスを適用したテーブル例 
-- ユーザーテーブル
CREATE TABLE users (
    user_id INT AUTO_INCREMENT PRIMARY KEY,     -- 主キー
    username VARCHAR(50) NOT NULL UNIQUE,       -- 必須・重複不可
    email VARCHAR(100) NOT NULL UNIQUE,         -- 必須・重複不可
    password_hash VARCHAR(255) NOT NULL,        -- 必須
    is_active BOOLEAN DEFAULT TRUE,             -- デフォルト値
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
               ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB;                                 -- トランザクション対応

-- 注文テーブル
CREATE TABLE orders (
    order_id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    order_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    total_amount DECIMAL(10, 2) NOT NULL CHECK (total_amount >= 0),
    status ENUM('pending', 'processing', 'completed', 'cancelled')
           DEFAULT 'pending',
    FOREIGN KEY (user_id) REFERENCES users(user_id)
           ON DELETE RESTRICT ON UPDATE CASCADE
) ENGINE=InnoDB;

ハンズオン：制約付きテーブルを作成してみよう

これまで学んだ制約を使って、ブログシステムのテーブルを作成してみましょう。

  1. 1. データベース作成
    CREATE DATABASE blog_system;
    USE blog_system;
  2. 2. authorsテーブル作成
    author_id（主キー、自動採番）、name（必須・最大50文字）、email（必須・重複不可）、created_at（現在時刻デフォルト）
  3. 3. articlesテーブル作成
    article_id（主キー、自動採番）、author_id（外部キー）、title（必須・最大200文字）、content（TEXT型）、view_count（0以上のINT、デフォルト0）、created_at、updated_at
  4. 4. データ挿入とエラー確認
    正常なデータを挿入後、制約違反するデータを挿入してエラーメッセージを確認しましょう。

理解度確認クイズ

  1. Q1. AUTO_INCREMENTの役割と利点を説明してください。

    回答を表示

    AUTO_INCREMENTは、INSERT時にIDを自動的に採番する機能です。1から始まる連番を自動割り当てし、削除された行のIDは再利用されません。開発者がIDを管理する必要がなく、重複を防げるメリットがあります。

  2. Q2. FOREIGN KEY制約のON DELETE CASCADEとON DELETE RESTRICTの違いは何ですか？

    回答を表示

    ON DELETE CASCADEは、親テーブルのレコードを削除すると、それを参照している子テーブルのレコードも自動的に削除されます。一方、ON DELETE RESTRICT（デフォルト）は、子テーブルに参照されている親レコードを削除できません。データの整合性を保つ方法が異なります。

  3. Q3. InnoDBとMyISAMの主な違いを3つ挙げてください。

    回答を表示

    1. トランザクション：InnoDBは対応、MyISAMは非対応
    2. 外部キー制約：InnoDBは対応、MyISAMは非対応
    3. ロック単位：InnoDBは行ロック、MyISAMはテーブルロック
    現在はInnoDBがデフォルトであり、ほとんどの用途に適しています。
