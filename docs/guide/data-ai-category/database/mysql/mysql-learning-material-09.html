MySQLの文字コードと照合順序 - 日本語を正しく扱う

文字コード（utf8mb4、utf8）と照合順序（collation）の概念を学習し、日本語データを正しく格納・検索するための設定、文字化けの原因と対策について理解します。

この章で学ぶこと

  • 文字コードとは何か、なぜ重要なのか
  • utf8とutf8mb4の違いと選び方
  • 照合順序（collation）の役割と種類
  • 日本語を正しく扱うための設定方法
  • 文字化けの原因と対策

9.1 文字コードとは - コンピュータが文字を扱う仕組み

文字コードとは、文字をコンピュータで扱うための符号化方式です。コンピュータ内部ではすべてのデータが0と1の数値で表現されるため、「あ」や「A」といった文字を数値に変換する必要があります。この変換規則が文字コードです。

flowchart LR A["文字: あ"] --> B["符号化"] B --> C["数値: E38182"] C --> D["復号化"] D --> E["文字: あ"] style A fill:#e0f2fe style E fill:#e0f2fe style C fill:#fef3c7

MySQLでは、データベース、テーブル、カラムのそれぞれに文字コードを設定できます。日本語を扱う場合、適切な文字コードを選択しないと文字化けが発生したり、一部の文字が保存できなかったりする問題が起こります。

文字コード設定の重要性

文字コードの設定ミスは、データベース構築後に修正するのが非常に困難です。データベース作成時に正しい文字コードを設定することが重要です。特に絵文字（😀など）を扱う場合は、utf8mb4の使用が必須です。

9.2 MySQLの主な文字コード - utf8とutf8mb4

MySQLで日本語を扱う場合、主に使用される文字コードはutf8とutf8mb4の2つです。名前が似ていますが、重要な違いがあります。

項目 utf8 utf8mb4
正式名称 UTF-8の3バイト版 UTF-8の4バイト版
最大バイト数 3バイト/文字 4バイト/文字
絵文字対応 ❌ 非対応 ✅ 対応
日本語ひらがな ✅ 対応 ✅ 対応
日本語漢字 ✅ 対応 ✅ 対応
推奨度 △ 古い ◎ 推奨

utf8mb4を選ぶべき理由

MySQLの「utf8」は実はUTF-8の完全版ではなく、3バイトまでの文字しか扱えません。絵文字や一部の特殊な漢字（𠮷など）は4バイト必要なため、utf8では保存できません。現在はutf8mb4の使用が強く推奨されています。

utf8mb4の設定例

データベース作成時にutf8mb4を指定 
-- データベースを作成する際に文字コードを指定
CREATE DATABASE myapp_db
CHARACTER SET utf8mb4
COLLATE utf8mb4_unicode_ci;
テーブル作成時にutf8mb4を指定 
-- テーブル作成時に文字コードを指定
CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100),
    bio TEXT
) CHARACTER SET utf8mb4
  COLLATE utf8mb4_unicode_ci;

9.3 照合順序（Collation） - 文字の並び順と比較ルール

照合順序（Collation）とは、文字列の比較やソートを行う際のルールです。同じ文字コードを使っていても、照合順序が異なると検索結果やソート順が変わることがあります。

flowchart TD A[utf8mb4文字コード] --> B{照合順序} B --> C[utf8mb4_general_ci
高速、簡易比較] B --> D[utf8mb4_unicode_ci
正確、Unicode規格準拠] B --> E[utf8mb4_bin
バイナリ比較、大小文字区別] B --> F[utf8mb4_ja_0900_as_cs
日本語専用、MySQL 8.0+] style A fill:#e0f2fe style C fill:#fef3c7 style D fill:#d1fae5 style E fill:#fce7f3 style F fill:#ddd6fe

主な照合順序の種類

照合順序 特徴 用途
utf8mb4_general_ci 簡易的な比較、高速 一般的な用途
utf8mb4_unicode_ci Unicode規格準拠、正確 国際化対応、推奨
utf8mb4_bin バイナリ比較、大小文字区別 厳密な比較が必要な場合
utf8mb4_ja_0900_as_cs 日本語専用、ひらがな/カタカナ区別 MySQL 8.0以降の日本語DB

照合順序の接尾辞の意味

  • _ci - Case Insensitive（大小文字を区別しない）
  • _cs - Case Sensitive（大小文字を区別する）
  • _bin - Binary（バイナリ比較、最も厳密）
  • _ai - Accent Insensitive（アクセント記号を区別しない）
  • _as - Accent Sensitive（アクセント記号を区別する）

照合順序による検索結果の違い

大小文字区別の違いを確認 
-- utf8mb4_general_ci（大小文字区別なし）の場合
SELECT * FROM users
WHERE name = 'tanaka'
COLLATE utf8mb4_general_ci;
-- 結果: 'Tanaka', 'TANAKA', 'tanaka' すべてマッチ

-- utf8mb4_bin（バイナリ比較）の場合
SELECT * FROM users
WHERE name = 'tanaka'
COLLATE utf8mb4_bin;
-- 結果: 'tanaka' のみマッチ

9.4 文字コードと照合順序の確認・変更方法

現在の設定を確認する

データベースの文字コードを確認 
-- データベースの文字コード設定を確認
SHOW CREATE DATABASE myapp_db;

-- すべてのデータベースの文字コードを一覧表示
SELECT
    SCHEMA_NAME,
    DEFAULT_CHARACTER_SET_NAME,
    DEFAULT_COLLATION_NAME
FROM information_schema.SCHEMATA;
テーブルの文字コードを確認 
-- テーブルの文字コード設定を確認
SHOW CREATE TABLE users;

-- すべてのテーブルの照合順序を確認
SELECT
    TABLE_NAME,
    TABLE_COLLATION
FROM information_schema.TABLES
WHERE TABLE_SCHEMA = 'myapp_db';
カラムの文字コードを確認 
-- 特定のテーブルの全カラム情報を確認
SHOW FULL COLUMNS FROM users;

-- カラムの文字コードのみ抽出
SELECT
    COLUMN_NAME,
    CHARACTER_SET_NAME,
    COLLATION_NAME
FROM information_schema.COLUMNS
WHERE TABLE_SCHEMA = 'myapp_db'
  AND TABLE_NAME = 'users';

既存データベースの文字コードを変更する

注意: データベース変更前に必ずバックアップ

文字コードの変更は既存データに影響を与える可能性があります。実行前に必ずバックアップを取得してください。また、本番環境では事前にテスト環境で動作確認を行いましょう。

データベース全体の文字コードを変更 
-- データベースのデフォルト文字コードを変更
ALTER DATABASE myapp_db
CHARACTER SET utf8mb4
COLLATE utf8mb4_unicode_ci;
テーブルの文字コードを変更 
-- テーブルの文字コードを変更（既存データも変換）
ALTER TABLE users
CONVERT TO CHARACTER SET utf8mb4
COLLATE utf8mb4_unicode_ci;
特定のカラムだけ文字コードを変更 
-- 特定のカラムの文字コードを変更
ALTER TABLE users
MODIFY COLUMN name VARCHAR(100)
CHARACTER SET utf8mb4
COLLATE utf8mb4_unicode_ci;

9.5 文字化けの原因と対策

MySQLで日本語を扱う際によく遭遇する文字化けの原因と、その対策方法を理解しましょう。

flowchart TD A[文字化けが発生] --> B{原因を特定} B --> C[DB/テーブルの
文字コード設定ミス] B --> D[接続時の
文字コード設定ミス] B --> E[アプリケーション側の
文字コード設定ミス] C --> F[ALTER DATABASEで修正] D --> G[SET NAMES utf8mb4で修正] E --> H[アプリの設定を確認] style A fill:#fecaca style C fill:#fef3c7 style D fill:#fef3c7 style E fill:#fef3c7 style F fill:#d1fae5 style G fill:#d1fae5 style H fill:#d1fae5

原因1: データベース/テーブルの文字コード設定ミス

対策: 作成時に正しい文字コードを指定

-- データベース作成時に明示的に指定
CREATE DATABASE myapp_db
CHARACTER SET utf8mb4
COLLATE utf8mb4_unicode_ci;

-- テーブル作成時にも指定
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(100)
) CHARACTER SET utf8mb4;

原因2: MySQL接続時の文字コード設定ミス

対策: 接続後にSET NAMESを実行

-- MySQL接続後すぐに実行
SET NAMES utf8mb4;

-- または詳細に指定
SET character_set_client = utf8mb4;
SET character_set_connection = utf8mb4;
SET character_set_results = utf8mb4;

原因3: アプリケーション側の文字コード設定ミス

対策: 接続文字列にcharsetを指定

PHPの場合 
// PDOの場合
$pdo = new PDO(
    'mysql:host=localhost;dbname=myapp_db;charset=utf8mb4',
    'user',
    'password'
);
Pythonの場合 
# PyMySQLの場合
connection = pymysql.connect(
    host='localhost',
    user='user',
    password='password',
    database='myapp_db',
    charset='utf8mb4'
)

文字化けトラブルシューティングの手順

  1. データベースの文字コード設定を確認（SHOW CREATE DATABASE）
  2. テーブルの文字コード設定を確認（SHOW CREATE TABLE）
  3. 接続時の文字コード設定を確認（SHOW VARIABLES LIKE 'character%'）
  4. アプリケーション側の接続設定を確認
  5. 問題箇所を特定し、該当箇所を修正

実習: 文字コードと照合順序の設定を確認してみよう

実際にデータベースを作成し、文字コードと照合順序の設定を確認してみましょう。

手順

  1. utf8mb4でデータベースを作成する
  2. utf8mb4_unicode_ciとutf8mb4_binの2つの照合順序でテーブルを作成する
  3. それぞれのテーブルに同じデータを挿入する
  4. 大小文字を区別する検索と区別しない検索を実行し、結果の違いを確認する
実習用SQL 
-- 1. データベース作成
CREATE DATABASE charset_test
CHARACTER SET utf8mb4
COLLATE utf8mb4_unicode_ci;

USE charset_test;

-- 2. 照合順序が異なる2つのテーブルを作成
CREATE TABLE users_ci (
    id INT PRIMARY KEY,
    name VARCHAR(50)
) COLLATE utf8mb4_unicode_ci;

CREATE TABLE users_bin (
    id INT PRIMARY KEY,
    name VARCHAR(50)
) COLLATE utf8mb4_bin;

-- 3. 同じデータを挿入
INSERT INTO users_ci VALUES
(1, 'Tanaka'),
(2, 'TANAKA'),
(3, 'tanaka');

INSERT INTO users_bin VALUES
(1, 'Tanaka'),
(2, 'TANAKA'),
(3, 'tanaka');

-- 4. 検索して結果を比較
-- Case Insensitive（大小文字区別なし）
SELECT * FROM users_ci WHERE name = 'tanaka';
-- 結果: 3件すべてマッチ

-- Binary（バイナリ比較、厳密）
SELECT * FROM users_bin WHERE name = 'tanaka';
-- 結果: id=3のみマッチ

理解度確認クイズ

  1. Q1. MySQLでutf8mb4を選ぶべき理由を説明してください。

    回答を表示

    MySQLの「utf8」は3バイトまでの文字しか扱えず、絵文字や一部の特殊な漢字（4バイト文字）を保存できません。utf8mb4は4バイトまで対応しているため、すべてのUnicode文字を正しく扱えます。特にSNSなど絵文字を扱うアプリケーションではutf8mb4が必須です。

  2. Q2. 照合順序（Collation）とは何ですか？

    回答を表示

    照合順序は、文字列の比較やソートを行う際のルールです。例えば、大小文字を区別するかしないか（_ciか_cs）、バイナリ比較をするか（_bin）などを決定します。同じ文字コードでも照合順序が異なると検索結果やソート順が変わります。

  3. Q3. 文字化けの主な原因を3つ挙げてください。

    回答を表示

    1. データベース/テーブルの文字コード設定が不適切（utf8mb4でない等）
    2. MySQL接続時の文字コード設定が不適切（SET NAMESの未実行等）
    3. アプリケーション側の接続設定が不適切（接続文字列にcharset指定がない等）

この章のまとめ

  • 文字コードはコンピュータが文字を扱うための符号化方式で、MySQLではutf8mb4の使用が推奨される
  • utf8は3バイトまで、utf8mb4は4バイトまで対応し、絵文字や特殊漢字を扱うにはutf8mb4が必須
  • 照合順序（Collation）は文字列の比較・ソートのルールを決定し、_ci（大小文字区別なし）や_bin（バイナリ比較）などがある
  • 文字化けの主な原因は、データベース設定、接続時の設定、アプリケーション側の設定のいずれかの不備
  • データベース作成時に文字コードを正しく設定し、接続時にSET NAMES utf8mb4を実行することで文字化けを防げる
