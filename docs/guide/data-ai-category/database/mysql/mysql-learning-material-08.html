ビューとストアドプロシージャ - 再利用可能なSQLを作る

複雑なクエリを再利用可能な形で保存するビューと、処理ロジックをデータベース側に実装するストアドプロシージャの基礎を学びます。日常的なデータ操作を効率化する方法を習得します。

この章で学ぶこと

  • ビューの概念と作成方法
  • ビューの活用場面と注意点
  • ストアドプロシージャとストアドファンクションの基礎
  • パラメータと変数の使い方

8.1 ビューとは - 仮想的なテーブル

ビュー（View）は、SELECT文の結果をテーブルのように扱えるようにしたものです。実際のデータを保存するのではなく、クエリの定義だけを保存する「仮想テーブル」として機能します。

ビューのメリット

  • 複雑なクエリの簡略化 - 長いJOINやサブクエリを1つのビューにまとめられる
  • セキュリティ向上 - 必要なカラムだけを公開し、機密情報を隠せる
  • コードの一貫性 - 同じクエリを複数箇所で使う際、ビューを使えば定義が統一される
  • 抽象化 - テーブル構造の変更をビューで吸収し、アプリ側の変更を最小化
flowchart LR A[実テーブル
products] --> B[ビュー
view_product_list] C[実テーブル
categories] --> B B --> D[SELECT文で参照] D --> E[結果セット
仮想テーブルとして利用]

ビューを作成すると、あたかも実際のテーブルが存在するかのようにSELECT文で参照できます。

SQL 
-- ビューの作成例
CREATE VIEW view_product_list AS
SELECT
    p.id,
    p.name AS product_name,
    c.name AS category_name,
    p.price
FROM products p
INNER JOIN categories c ON p.category_id = c.id;

-- ビューを通常のテーブルのように参照
SELECT * FROM view_product_list;

8.2 ビューの作成と管理

MySQLではCREATE VIEW文でビューを作成し、DROP VIEWで削除、ALTER VIEWで変更できます。

SQL 
-- ビューの作成
CREATE VIEW view_high_price_products AS
SELECT
    id,
    name,
    price
FROM products
WHERE price >= 10000;

-- ビューの参照
SELECT * FROM view_high_price_products;

-- ビューの削除
DROP VIEW IF EXISTS view_high_price_products;

-- ビューの再作成（OR REPLACEで既存を上書き）
CREATE OR REPLACE VIEW view_high_price_products AS
SELECT
    id,
    name,
    price
FROM products
WHERE price >= 5000;  -- 条件を変更

ビューの更新制約

ビューを通じてINSERT、UPDATE、DELETEを行うことも可能ですが、集計関数、GROUP BY、DISTINCT、JOINなどを含むビューでは更新できない場合があります。基本的にビューは参照専用として使うことをお勧めします。

実習: 売上レポート用ビューを作成しよう

ordersテーブルとorder_itemsテーブルを結合し、注文ごとの合計金額を表示するビューを作成してみましょう。

ステップ1: テーブルとデータを準備

SQL 
CREATE TABLE orders (
    id INT PRIMARY KEY,
    customer_name VARCHAR(100),
    order_date DATE
);

CREATE TABLE order_items (
    id INT PRIMARY KEY,
    order_id INT,
    product_name VARCHAR(100),
    quantity INT,
    price DECIMAL(10, 2)
);

INSERT INTO orders VALUES
(1, '山田太郎', '2024-01-10'),
(2, '佐藤花子', '2024-01-12');

INSERT INTO order_items VALUES
(1, 1, 'ノートPC', 1, 80000),
(2, 1, 'マウス', 2, 1500),
(3, 2, 'キーボード', 1, 5000);

ステップ2: 注文ごとの合計金額を表示するビューを作成

SQL 
CREATE VIEW view_order_summary AS
SELECT
    o.id AS order_id,
    o.customer_name,
    o.order_date,
    SUM(oi.quantity * oi.price) AS total_amount
FROM orders o
INNER JOIN order_items oi ON o.id = oi.order_id
GROUP BY o.id, o.customer_name, o.order_date;

ステップ3: ビューを参照

SQL 
SELECT * FROM view_order_summary;

8.3 ストアドプロシージャとは

ストアドプロシージャ（Stored Procedure）は、一連のSQL文をひとまとめにして、データベース内に保存したものです。プログラミング言語の関数に似た役割を果たし、複雑な処理を再利用可能な形で保存できます。

ストアドプロシージャのメリット

  • 処理のカプセル化 - 複雑なビジネスロジックをDB側に集約できる
  • パフォーマンス向上 - コンパイル済みの処理が実行され、ネットワーク通信も削減
  • 保守性向上 - ロジック変更時、プロシージャを修正するだけで全アプリに反映
  • セキュリティ - テーブルへの直接アクセスを制限し、プロシージャ経由のみに限定できる
SQL 
-- シンプルなストアドプロシージャの作成
DELIMITER //

CREATE PROCEDURE GetAllProducts()
BEGIN
    SELECT * FROM products;
END //

DELIMITER ;

-- ストアドプロシージャの実行
CALL GetAllProducts();

DELIMITERとは

MySQLのコマンドラインクライアントでは、通常セミコロン（;）が文の終わりを示します。しかし、ストアドプロシージャ内部でもセミコロンを使うため、一時的に区切り文字を「//」などに変更する必要があります。MySQL Workbenchではこの記述は不要な場合もあります。

8.4 パラメータを持つストアドプロシージャ

ストアドプロシージャはパラメータを受け取ることができます。パラメータには3種類あります：IN（入力）、OUT（出力）、INOUT（入出力）。

SQL 
-- INパラメータ: 価格で商品を検索
DELIMITER //

CREATE PROCEDURE GetProductsByPrice(IN min_price DECIMAL(10, 2))
BEGIN
    SELECT id, name, price
    FROM products
    WHERE price >= min_price;
END //

DELIMITER ;

-- 実行例
CALL GetProductsByPrice(5000);

-- OUTパラメータ: 商品数を取得
DELIMITER //

CREATE PROCEDURE GetProductCount(OUT product_count INT)
BEGIN
    SELECT COUNT(*) INTO product_count FROM products;
END //

DELIMITER ;

-- 実行例（変数に結果を格納）
CALL GetProductCount(@count);
SELECT @count;
パラメータ種類 役割 使用例
IN 入力パラメータ（デフォルト） 検索条件、フィルタ値
OUT 出力パラメータ 集計結果、カウント値
INOUT 入出力パラメータ 計算処理、値の更新

8.5 変数と制御構文

ストアドプロシージャ内では変数を宣言し、IF文やCASE文などの制御構文を使えます。

SQL 
-- 変数とIF文の例
DELIMITER //

CREATE PROCEDURE CheckPrice(IN product_id INT)
BEGIN
    DECLARE v_price DECIMAL(10, 2);
    DECLARE v_message VARCHAR(100);

    -- 価格を取得
    SELECT price INTO v_price
    FROM products
    WHERE id = product_id;

    -- 条件分岐
    IF v_price >= 10000 THEN
        SET v_message = '高価格商品です';
    ELSEIF v_price >= 5000 THEN
        SET v_message = '中価格商品です';
    ELSE
        SET v_message = '低価格商品です';
    END IF;

    SELECT v_message AS message, v_price AS price;
END //

DELIMITER ;

-- 実行例
CALL CheckPrice(1);

よく使う制御構文

  • IF...THEN...ELSE - 条件分岐
  • CASE - 複数条件の分岐
  • WHILE - ループ処理
  • LOOP - 無限ループ（LEAVE文で脱出）

8.6 ストアドファンクション

ストアドファンクション（Stored Function）は、ストアドプロシージャと似ていますが、必ず1つの値を返し、SELECT文の中で直接使えるという特徴があります。

SQL 
-- 消費税を計算するファンクション
DELIMITER //

CREATE FUNCTION CalcTax(price DECIMAL(10, 2))
RETURNS DECIMAL(10, 2)
DETERMINISTIC
BEGIN
    RETURN price * 1.10;  -- 10%の消費税
END //

DELIMITER ;

-- SELECT文内で使用
SELECT
    name,
    price,
    CalcTax(price) AS price_with_tax
FROM products;
項目 ストアドプロシージャ ストアドファンクション
返り値 複数の結果セット可 単一の値
呼び出し方 CALL文 SELECT文内
主な用途 複雑な処理、データ更新 計算、値の変換

DETERMINISTICの指定

ストアドファンクションを作成する際、DETERMINISTIC（同じ入力に対して常に同じ結果を返す）またはNOT DETERMINISTIC（結果が変わる可能性がある）の指定が必要です。MySQL 8.0以降、省略するとエラーになる場合があります。

実習: 在庫更新プロシージャを作成しよう

商品の在庫を更新し、在庫が0以下になった場合は警告メッセージを返すストアドプロシージャを作成してみましょう。

SQL 
-- テーブル作成
CREATE TABLE inventory (
    product_id INT PRIMARY KEY,
    product_name VARCHAR(100),
    stock INT
);

INSERT INTO inventory VALUES
(1, 'ノートPC', 5),
(2, 'マウス', 10);

-- ストアドプロシージャ作成
DELIMITER //

CREATE PROCEDURE UpdateStock(
    IN p_product_id INT,
    IN p_quantity INT,
    OUT p_message VARCHAR(200)
)
BEGIN
    DECLARE v_current_stock INT;

    -- 現在の在庫を取得
    SELECT stock INTO v_current_stock
    FROM inventory
    WHERE product_id = p_product_id;

    -- 在庫を更新
    UPDATE inventory
    SET stock = stock + p_quantity
    WHERE product_id = p_product_id;

    -- 更新後の在庫を確認
    SELECT stock INTO v_current_stock
    FROM inventory
    WHERE product_id = p_product_id;

    -- メッセージ作成
    IF v_current_stock <= 0 THEN
        SET p_message = '警告: 在庫切れです';
    ELSEIF v_current_stock <= 3 THEN
        SET p_message = '注意: 在庫が少なくなっています';
    ELSE
        SET p_message = '在庫は十分にあります';
    END IF;
END //

DELIMITER ;

-- 実行例
CALL UpdateStock(1, -3, @msg);
SELECT @msg;

8.7 ビューとプロシージャの削除・確認

作成したビュー、ストアドプロシージャ、ストアドファンクションを確認・削除する方法を覚えておきましょう。

SQL 
-- ビューの一覧表示
SHOW FULL TABLES WHERE Table_type = 'VIEW';

-- ビューの定義を確認
SHOW CREATE VIEW view_product_list;

-- ビューの削除
DROP VIEW IF EXISTS view_product_list;

-- ストアドプロシージャの一覧表示
SHOW PROCEDURE STATUS WHERE Db = 'データベース名';

-- プロシージャの定義を確認
SHOW CREATE PROCEDURE GetAllProducts;

-- プロシージャの削除
DROP PROCEDURE IF EXISTS GetAllProducts;

-- ストアドファンクションの一覧表示
SHOW FUNCTION STATUS WHERE Db = 'データベース名';

-- ファンクションの削除
DROP FUNCTION IF EXISTS CalcTax;

理解度確認クイズ

  1. Q1. ビューとは何ですか？また、どのようなメリットがありますか?

    回答を表示

    ビューはSELECT文の結果を仮想的なテーブルとして扱えるようにしたものです。メリットは、複雑なクエリの簡略化、セキュリティ向上（必要なカラムだけを公開）、コードの一貫性確保、テーブル構造変更の影響を吸収する抽象化などがあります。

  2. Q2. ストアドプロシージャのパラメータ種類（IN、OUT、INOUT）の違いを説明してください。

    回答を表示

    IN（入力）パラメータは呼び出し側から値を受け取る、OUTパラメータは処理結果を呼び出し側に返す、INOUTパラメータは入力と出力の両方に使える、という違いがあります。INが最も一般的で、OUTは集計結果やカウント値を返す際に使います。

  3. Q3. ストアドプロシージャとストアドファンクションの違いは何ですか?

    回答を表示

    ストアドプロシージャは複数の結果セットを返すことができ、CALL文で呼び出します。一方、ストアドファンクションは単一の値のみを返し、SELECT文の中で直接使えます。ファンクションは計算や値の変換に適しており、プロシージャは複雑な処理やデータ更新に適しています。

この章のまとめ

  • ビューは複雑なクエリを仮想テーブルとして再利用でき、セキュリティや保守性の向上に役立つ
  • ストアドプロシージャは一連のSQL文をカプセル化し、パラメータや制御構文を使った複雑な処理を実装できる
  • パラメータにはIN（入力）、OUT（出力）、INOUT（入出力）の3種類がある
  • ストアドファンクションは単一の値を返し、SELECT文内で直接使える
