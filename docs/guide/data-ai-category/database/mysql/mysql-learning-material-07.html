トランザクションとロック - データの整合性を守る

トランザクション(BEGIN、COMMIT、ROLLBACK)の基本概念と実践的な使い方を学習します。ACID特性、InnoDBの分離レベル、行ロックとテーブルロックの違いを理解し、データの整合性を保つ方法を習得します。

この章で学ぶこと

  • トランザクションの基本概念とACID特性
  • BEGIN、COMMIT、ROLLBACKの使い方
  • InnoDBのトランザクション分離レベル
  • 行ロックとテーブルロックの違いと使い分け

7.1 トランザクションとは

トランザクションとは、複数のSQL操作をひとまとまりの処理単位として扱う仕組みです。銀行のATMでお金を振り込む際、「出金」と「入金」の2つの処理がセットで成功するか、失敗した場合は両方とも実行されないようにする必要があります。このような処理をトランザクションで保証します。

トランザクションの必要性

例えば、オンラインショッピングで商品を購入する際、以下の処理が同時に行われます：

  • 1. 在庫テーブルから商品を減らす
  • 2. 注文テーブルに注文情報を追加する
  • 3. 顧客テーブルのポイントを減算する

もし2番目の処理で失敗した場合、1番目の処理も取り消す必要があります。トランザクションを使えば、全ての処理が成功した場合のみデータベースに反映できます。

flowchart TD A["トランザクション開始
BEGIN"] --> B["SQL操作1
在庫減少"] B --> C["SQL操作2
注文追加"] C --> D["SQL操作3
ポイント減算"] D --> E{全て成功?} E -->|成功| F["確定
COMMIT"] E -->|失敗| G["取り消し
ROLLBACK"] F --> H["データベースに反映"] G --> I["全ての操作を元に戻す"]

7.2 ACID特性 - トランザクションが満たすべき4つの性質

トランザクションが信頼できるものとして機能するためには、ACID特性と呼ばれる4つの性質を満たす必要があります。

A

Atomicity（原子性）

トランザクション内の全ての処理が実行されるか、全く実行されないかのどちらか。途中で失敗した場合は全て取り消される。

C

Consistency（一貫性）

トランザクション実行前後でデータベースの整合性が保たれる。制約違反や矛盾が発生しない。

I

Isolation（独立性）

複数のトランザクションが同時実行されても、互いに影響を与えない。他のトランザクションの途中経過は見えない。

D

Durability（永続性）

一度COMMITされたトランザクションの結果は、システム障害が起きても失われない。

MySQLのACID対応

MySQLのInnoDBストレージエンジンは完全なACID特性をサポートしています。一方、MyISAMはトランザクションをサポートしていないため、ACID特性も保証されません。本章ではInnoDBを前提に解説します。

7.3 トランザクションの基本操作（BEGIN・COMMIT・ROLLBACK）

MySQLでトランザクションを制御するには、以下の3つのコマンドを使用します。

コマンド 意味 別名
BEGIN トランザクションを開始する START TRANSACTION
COMMIT トランザクションを確定してデータベースに反映する -
ROLLBACK トランザクションを取り消して元の状態に戻す -

基本的なトランザクションの例

口座Aから口座Bへ1000円を送金する例を見てみましょう。

SQL 
-- トランザクション開始
BEGIN;

-- 口座Aから1000円引く
UPDATE accounts SET balance = balance - 1000 WHERE account_id = 'A';

-- 口座Bに1000円足す
UPDATE accounts SET balance = balance + 1000 WHERE account_id = 'B';

-- 両方の処理が成功したら確定
COMMIT;

もし途中でエラーが発生した場合は、ROLLBACKを実行して全ての変更を取り消します。

SQL 
-- トランザクション開始
BEGIN;

-- 口座Aから1000円引く
UPDATE accounts SET balance = balance - 1000 WHERE account_id = 'A';

-- エラーが発生したと仮定
-- 全ての変更を取り消す
ROLLBACK;

実習: トランザクションの基本操作

実際にトランザクションを試してみましょう。

  1. テスト用テーブルを作成する
  2. トランザクションを使ってデータを挿入し、COMMITする
  3. トランザクションを使ってデータを挿入し、ROLLBACKする
  4. それぞれの結果を確認する
SQL 
-- 1. テーブル作成
CREATE TABLE test_products (
    id INT PRIMARY KEY,
    name VARCHAR(50)
);

-- 2. COMMITする場合
BEGIN;
INSERT INTO test_products VALUES (1, '商品A');
COMMIT;
-- 確認
SELECT * FROM test_products;

-- 3. ROLLBACKする場合
BEGIN;
INSERT INTO test_products VALUES (2, '商品B');
ROLLBACK;
-- 確認
SELECT * FROM test_products;

期待される結果

  • ・COMMITした商品Aはテーブルに保存される
  • ・ROLLBACKした商品Bはテーブルに保存されない

7.4 トランザクション分離レベル

複数のユーザーが同時にデータベースにアクセスする環境では、トランザクション同士の干渉をどこまで許容するかを設定する必要があります。これをトランザクション分離レベルと呼びます。

分離レベル ダーティリード 非再現リード ファントムリード
READ UNCOMMITTED 発生する 発生する 発生する
READ COMMITTED 防げる 発生する 発生する
REPEATABLE READ（デフォルト） 防げる 防げる 一部発生
SERIALIZABLE 防げる 防げる 防げる

各問題の説明

  • ダーティリード - 他のトランザクションがCOMMITしていない変更を読み取ってしまう
  • 非再現リード - 同じトランザクション内で同じデータを2回読んだ時、結果が異なる
  • ファントムリード - 同じトランザクション内で同じ条件でSELECTしても、行数が変わる

InnoDBのデフォルト: REPEATABLE READ

MySQLのInnoDBはREPEATABLE READをデフォルトの分離レベルとして使用します。これは性能と整合性のバランスが良く、ほとんどのアプリケーションで適切です。

SQL 
-- 現在の分離レベルを確認
SELECT @@transaction_isolation;

-- 分離レベルを変更（セッション単位）
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

7.5 ロックの種類と仕組み

MySQLは、複数のユーザーが同時にデータを変更しようとした際にデータの整合性を保つために、ロックという仕組みを使用します。

行ロックとテーブルロック

行ロック（Row Lock）

特定の行だけをロックする。InnoDBがサポート。

利点: 並行性が高い（他の行は操作可能）

テーブルロック（Table Lock）

テーブル全体をロックする。MyISAMで使用。

欠点: 他のユーザーは待たされる

共有ロックと排他ロック

ロックの種類 説明 使用方法
共有ロック（S） 読み取り専用ロック。複数同時取得可能。 SELECT ... FOR SHARE
排他ロック（X） 読み書き独占ロック。他のロックを許可しない。 SELECT ... FOR UPDATE
SQL 
-- 排他ロックを取得（更新予定のデータを先にロック）
BEGIN;
SELECT * FROM products WHERE id = 1 FOR UPDATE;
-- この行は他のトランザクションから更新できない
UPDATE products SET stock = stock - 1 WHERE id = 1;
COMMIT;

デッドロックに注意

複数のトランザクションが互いにロックを待ち合ってしまい、どちらも進めなくなる状態をデッドロックと呼びます。MySQLは自動的にデッドロックを検出し、一方のトランザクションをロールバックしてデッドロックを解消します。

実習: ロックの動作を確認

2つのMySQLセッションを使って、ロックの動作を確認してみましょう。

準備

CREATE TABLE lock_test (
    id INT PRIMARY KEY,
    value INT
);
INSERT INTO lock_test VALUES (1, 100);

セッション1

BEGIN;
SELECT * FROM lock_test WHERE id = 1 FOR UPDATE;
-- ここで待機（まだCOMMITしない）

セッション2（別のウィンドウで実行）

-- この操作はセッション1がCOMMITするまで待たされる
UPDATE lock_test SET value = 200 WHERE id = 1;

セッション1でCOMMIT

COMMIT;
-- セッション2のUPDATEが実行される

7.6 実践的なトランザクション活用例

例1: ECサイトの注文処理

複数のテーブルを更新する処理は、トランザクションで囲む必要があります。

SQL 
BEGIN;

-- 1. 商品の在庫を減らす
UPDATE products
SET stock = stock - 1
WHERE product_id = 101 AND stock > 0;

-- 2. 注文を登録
INSERT INTO orders (customer_id, product_id, amount)
VALUES (5001, 101, 2980);

-- 3. 顧客のポイントを減算
UPDATE customers
SET points = points - 100
WHERE customer_id = 5001;

-- すべて成功したら確定
COMMIT;

例2: 在庫不足時のロールバック

アプリケーション側で条件をチェックし、問題があればROLLBACKします。

SQL 
BEGIN;

-- 在庫をチェック（排他ロック付き）
SELECT stock FROM products WHERE product_id = 101 FOR UPDATE;

-- アプリケーション側で在庫が足りるか確認
-- もし在庫が足りなければ...
ROLLBACK;  -- トランザクションを取り消す

-- 在庫があれば続行
-- UPDATE products SET stock = stock - 1 ...
-- COMMIT;

トラブルシューティング

Q1. トランザクションが自動でCOMMITされてしまう

MySQLはデフォルトで自動コミットモードが有効です。SET autocommit = 0;で無効化できますが、通常は明示的にBEGINを使う方が安全です。

Q2. デッドロックが頻繁に発生する

複数のテーブルを更新する際は、常に同じ順序でロックを取得するようにしましょう。また、トランザクションをできるだけ短く保つことも重要です。

Q3. トランザクション中に接続が切れた

接続が切れるとトランザクションは自動的にROLLBACKされます。これによりデータの整合性が保たれます。

理解度確認クイズ

  1. Q1. ACID特性の「A」は何を意味しますか？また、それはどういう性質ですか？

    回答を表示

    「A」はAtomicity（原子性）を意味します。トランザクション内の全ての処理が実行されるか、全く実行されないかのどちらかであることを保証する性質です。途中で失敗した場合は全て取り消されます。

  2. Q2. MySQLのInnoDBでデフォルトのトランザクション分離レベルは何ですか？

    回答を表示

    REPEATABLE READです。これは性能と整合性のバランスが良く、ダーティリードと非再現リードを防ぐことができます。

  3. Q3. 排他ロックを明示的に取得するSQLコマンドは何ですか？

    回答を表示

    SELECT ... FOR UPDATEです。このコマンドで取得した行は、他のトランザクションから読み書きできなくなります。更新予定のデータを先にロックしておく際に使用します。

この章のまとめ

  • トランザクションは複数のSQL操作をひとまとまりの処理として扱う仕組みで、BEGIN、COMMIT、ROLLBACKで制御する
  • ACID特性（原子性・一貫性・独立性・永続性）により、データの整合性が保たれる
  • トランザクション分離レベルで、複数ユーザーの同時アクセス時の干渉度を制御できる
  • InnoDBは行ロックをサポートし、共有ロックと排他ロックを使い分けてデータを保護する
