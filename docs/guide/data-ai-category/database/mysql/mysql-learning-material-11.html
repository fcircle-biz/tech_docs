ユーザーと権限管理 - セキュリティの基礎

MySQLのユーザー管理、GRANT/REVOKEによる権限設定を学習します。本番環境で必要となるセキュリティ設定の基本、接続制限、パスワードポリシーについて理解します。

この章で学ぶこと

  • MySQLのユーザー管理の基本
  • GRANT/REVOKEによる権限設定
  • 本番環境でのセキュリティ設定
  • 接続制限とパスワードポリシー

11.1 MySQLのユーザー管理

MySQLでは、セキュリティ上の理由から各ユーザーに適切な権限のみを付与する「最小権限の原則」に従うことが重要です。開発環境と本番環境で異なるユーザーを作成し、それぞれ必要な権限のみを付与することで、データの漏洩や改ざんのリスクを減らします。

MySQLのユーザー管理の基本概念

  • ユーザー識別 - MySQLのユーザーは「ユーザー名@ホスト名」の形式で識別される
  • 権限レベル - グローバル、データベース、テーブル、カラムの4段階がある
  • 認証プラグイン - MySQL 8.0からは「caching_sha2_password」がデフォルト
  • mysql.user テーブル - ユーザー情報はこのシステムテーブルに格納される

ユーザーの作成

新しいユーザーを作成するには CREATE USER 文を使用します。

ユーザー作成の基本構文 
-- ローカルホストからのみ接続可能なユーザー
CREATE USER 'app_user'@'localhost' IDENTIFIED BY 'SecurePassword123!';

-- 任意のホストから接続可能なユーザー（注意: セキュリティリスク高）
CREATE USER 'remote_user'@'%' IDENTIFIED BY 'AnotherSecure456!';

-- 特定のIPアドレスからのみ接続可能なユーザー
CREATE USER 'admin_user'@'192.168.1.100' IDENTIFIED BY 'AdminPass789!';

-- パスワード有効期限を設定（180日）
CREATE USER 'temp_user'@'localhost'
IDENTIFIED BY 'TempPass321!'
PASSWORD EXPIRE INTERVAL 180 DAY;

セキュリティ上の注意

本番環境では '@%'（全ホストからの接続許可）は使用しないでください。必ず接続元のIPアドレスを限定しましょう。また、パスワードは大文字・小文字・数字・記号を含む強固なものを設定し、定期的に変更してください。

ユーザー情報の確認

ユーザー一覧の確認 
-- すべてのユーザーを確認
SELECT user, host FROM mysql.user;

-- 現在のユーザーを確認
SELECT USER(), CURRENT_USER();

-- 特定ユーザーの詳細情報
SHOW CREATE USER 'app_user'@'localhost';

ユーザーの変更・削除

ユーザー管理操作 
-- パスワードの変更
ALTER USER 'app_user'@'localhost' IDENTIFIED BY 'NewPassword999!';

-- ユーザー名の変更
RENAME USER 'old_name'@'localhost' TO 'new_name'@'localhost';

-- ユーザーの削除
DROP USER 'temp_user'@'localhost';

-- アカウントのロック（ログイン不可）
ALTER USER 'app_user'@'localhost' ACCOUNT LOCK;

-- アカウントのアンロック
ALTER USER 'app_user'@'localhost' ACCOUNT UNLOCK;

11.2 権限の種類と設定

MySQLでは、SELECT、INSERT、UPDATE、DELETE などの基本的なデータ操作権限から、CREATE、DROP などのスキーマ変更権限まで、細かく権限を設定できます。

主な権限の種類

権限名 説明 用途例
SELECT データの読み取り 閲覧専用ユーザー
INSERT データの追加 アプリケーション
UPDATE データの更新 アプリケーション
DELETE データの削除 管理者のみ
CREATE テーブル・DBの作成 開発者・管理者
DROP テーブル・DBの削除 管理者のみ
ALTER テーブル構造の変更 開発者・管理者
INDEX インデックスの作成・削除 管理者
EXECUTE ストアドプロシージャ実行 アプリケーション
ALL PRIVILEGES すべての権限（GRANT除く） 管理者のみ

権限の付与（GRANT文）

基本的な権限付与 
-- 特定のデータベースに読み取り権限のみ付与
GRANT SELECT ON company_db.* TO 'readonly_user'@'localhost';

-- データベース全体に基本的な操作権限を付与
GRANT SELECT, INSERT, UPDATE, DELETE ON app_db.*
TO 'app_user'@'localhost';

-- 特定のテーブルにのみ権限を付与
GRANT SELECT, UPDATE (name, email) ON app_db.users
TO 'support_user'@'localhost';

-- すべての権限を付与（管理者用）
GRANT ALL PRIVILEGES ON *.* TO 'admin_user'@'localhost';

-- 権限を他のユーザーに付与する権限（WITH GRANT OPTION）
GRANT SELECT, INSERT ON app_db.* TO 'manager'@'localhost'
WITH GRANT OPTION;

権限の確認

権限の確認方法 
-- 現在のユーザーの権限を確認
SHOW GRANTS;

-- 特定ユーザーの権限を確認
SHOW GRANTS FOR 'app_user'@'localhost';

-- データベースごとの権限を確認
SELECT * FROM mysql.db WHERE User = 'app_user';

権限の取り消し（REVOKE文）

権限の取り消し 
-- 特定の権限を取り消す
REVOKE DELETE ON app_db.* FROM 'app_user'@'localhost';

-- すべての権限を取り消す
REVOKE ALL PRIVILEGES ON *.* FROM 'temp_user'@'localhost';

-- 権限付与の権限を取り消す
REVOKE GRANT OPTION ON app_db.* FROM 'manager'@'localhost';

-- 権限変更を即座に反映（通常は自動だが明示的に実行することも可能）
FLUSH PRIVILEGES;

GRANT] B --> C{権限レベル} C -->|グローバル| D["*.*"] C -->|データベース| E["db_name.*"] C -->|テーブル| F["db.table"] C -->|カラム| G["db.table（column）"] D --> H[権限確認

11.3 実用的な権限設定パターン

実際の開発・運用現場では、役割に応じて複数のユーザーを作成し、それぞれに適切な権限を設定します。以下は代表的なパターンです。

開発者用ユーザー

開発環境での作業用。テーブル作成・変更が可能だが本番DBへはアクセス不可。

GRANT ALL ON dev_db.*

アプリケーション用

本番環境で動作するアプリケーション用。基本操作のみ許可。

SELECT, INSERT,
UPDATE, DELETE

閲覧専用ユーザー

データ分析・レポート作成用。読み取り専用で安全。

SELECT のみ

管理者用ユーザー

すべての操作が可能。バックアップ・メンテナンス用。

ALL PRIVILEGES

実習: 役割別ユーザーの作成

実際に3種類のユーザーを作成して権限を設定してみましょう。

役割別ユーザー作成スクリプト 
-- 1. 開発者用ユーザー（開発DBへの完全アクセス）
CREATE USER 'dev_user'@'localhost' IDENTIFIED BY 'DevPass123!';
GRANT ALL PRIVILEGES ON dev_db.* TO 'dev_user'@'localhost';

-- 2. アプリケーション用ユーザー（基本操作のみ）
CREATE USER 'app_user'@'localhost' IDENTIFIED BY 'AppPass456!';
GRANT SELECT, INSERT, UPDATE, DELETE ON prod_db.*
TO 'app_user'@'localhost';

-- 3. 閲覧専用ユーザー（読み取りのみ）
CREATE USER 'analyst'@'localhost' IDENTIFIED BY 'ReadOnly789!';
GRANT SELECT ON prod_db.* TO 'analyst'@'localhost';

-- 権限確認
SHOW GRANTS FOR 'dev_user'@'localhost';
SHOW GRANTS FOR 'app_user'@'localhost';
SHOW GRANTS FOR 'analyst'@'localhost';

11.4 セキュリティ設定のベストプラクティス

本番環境でMySQLを安全に運用するために、以下のセキュリティ対策を実施しましょう。

セキュリティチェックリスト

  • rootユーザーは必ずパスワードを設定し、リモートアクセスを禁止する
  • 本番環境では'@%'（全ホスト許可）を使わず接続元を限定する
  • パスワードは最低12文字以上で、複雑な組み合わせを使用する
  • 不要なユーザーや匿名ユーザーは削除する
  • SSL/TLS接続を有効にして通信を暗号化する
  • 定期的にユーザーと権限の見直しを行う

パスワードポリシーの設定

MySQL 8.0では、パスワードの強度をチェックする「validate_password コンポーネント」が利用できます。

パスワードポリシーの設定 
-- パスワード検証プラグインのインストール
INSTALL COMPONENT 'file://component_validate_password';

-- パスワードポリシーの確認
SHOW VARIABLES LIKE 'validate_password%';

-- パスワードポリシーの変更（LOW、MEDIUM、STRONG）
SET GLOBAL validate_password.policy = MEDIUM;
SET GLOBAL validate_password.length = 12;

-- パスワード有効期限のデフォルト設定（全ユーザー対象）
SET GLOBAL default_password_lifetime = 180;

接続失敗時のアカウントロック

アカウントロック設定 
-- 3回失敗したら1日ロックする設定
CREATE USER 'secure_user'@'localhost'
IDENTIFIED BY 'SecurePass123!'
FAILED_LOGIN_ATTEMPTS 3
PASSWORD_LOCK_TIME 1;

-- 既存ユーザーに設定
ALTER USER 'app_user'@'localhost'
FAILED_LOGIN_ATTEMPTS 5
PASSWORD_LOCK_TIME 2;

SSL/TLS接続の有効化

リモート接続を行う場合、通信を暗号化することでパスワードやデータの盗聴を防ぎます。

SSL接続の強制 
-- SSL接続を必須にする
CREATE USER 'remote_user'@'%'
IDENTIFIED BY 'RemotePass123!'
REQUIRE SSL;

-- 既存ユーザーにSSL接続を強制
ALTER USER 'app_user'@'%' REQUIRE SSL;

-- SSL設定の確認
SHOW VARIABLES LIKE '%ssl%';

トラブルシューティング

問題: 権限を付与したのにアクセスできない

原因: ホスト名が一致していない可能性があります。
解決策: ユーザー作成時の'@ホスト名'と実際の接続元が一致しているか確認してください。'localhost'と'127.0.0.1'は別物として扱われます。

問題: 「Access denied」エラーが出る

原因: パスワードが間違っているか、権限が不足しています。
解決策: SHOW GRANTS コマンドで権限を確認し、必要に応じて再付与してください。

問題: rootパスワードを忘れた

解決策: MySQLを--skip-grant-tablesオプション付きで起動し、パスワードをリセットできます（ただしセキュリティリスクがあるため慎重に）。

11.5 監査ログとセキュリティ監視

本番環境では、誰がいつどのような操作を行ったかを記録しておくことが重要です。MySQLでは一般クエリログやバイナリログを使って操作を追跡できます。

一般クエリログの有効化

クエリログ設定 
-- 一般クエリログを有効化（すべてのクエリを記録）
SET GLOBAL general_log = 'ON';
SET GLOBAL general_log_file = '/var/log/mysql/general.log';

-- 現在の設定確認
SHOW VARIABLES LIKE 'general_log%';

-- ログサイズが大きくなるため、本番環境では注意して使用

パフォーマンスへの影響

一般クエリログはすべてのクエリを記録するため、大量のディスクI/Oが発生しパフォーマンスに影響します。通常は無効にしておき、トラブルシューティング時のみ一時的に有効化するのが推奨されます。

ログイン履歴の確認

接続履歴の確認 
-- 現在の接続を確認
SHOW PROCESSLIST;

-- より詳細な情報
SELECT * FROM information_schema.PROCESSLIST;

理解度確認クイズ

  1. Q1. ユーザー 'app_user'@'localhost' に company_db データベースの SELECT と INSERT 権限のみを付与するSQL文を書いてください。

    回答を表示 
    GRANT SELECT, INSERT ON company_db.* TO 'app_user'@'localhost';

  2. Q2. 本番環境でユーザーを作成する際に避けるべきホスト指定は何ですか？また、その理由を説明してください。

    回答を表示

    '@%'（全ホストからの接続許可）は避けるべきです。理由は、世界中のどこからでも接続を試みることができてしまい、セキュリティリスクが非常に高くなるためです。本番環境では必ず接続元のIPアドレスやホスト名を限定してください。

  3. Q3. 「最小権限の原則」とは何ですか？具体例を挙げて説明してください。

    回答を表示

    最小権限の原則とは、各ユーザーに必要最低限の権限のみを付与するセキュリティの考え方です。例えば、データ分析専用のユーザーには SELECT 権限のみを付与し、DELETE や DROP などのデータを削除する権限は与えません。これにより、誤操作や不正アクセスによるデータ損失のリスクを最小化できます。

この章のまとめ

  • MySQLのユーザーは「ユーザー名@ホスト名」で識別され、CREATE USER で作成する
  • GRANT文で権限を付与し、REVOKE文で権限を取り消す。権限は役割に応じて適切に設定する
  • 本番環境では '@%' を使わず接続元を限定し、強固なパスワードを設定する
  • パスワードポリシー、アカウントロック、SSL接続などのセキュリティ機能を活用する
  • クエリログやプロセスリストで接続履歴と操作を監視し、セキュリティを維持する
