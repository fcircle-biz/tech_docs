バックアップとリストア - データを安全に守る

mysqldump、mysqlコマンドを使ったバックアップとリストアの方法を学習します。論理バックアップと物理バックアップの違い、定期バックアップの自動化、ポイントインタイムリカバリの基礎について理解します。

この章で学ぶこと

  • mysqldumpコマンドを使った論理バックアップの方法
  • mysqlコマンドを使ったリストア（復元）の方法
  • 論理バックアップと物理バックアップの違いと使い分け
  • 定期バックアップの自動化とバイナリログを使ったポイントインタイムリカバリ

10.1 バックアップの重要性と種類

データベースは企業や組織の重要な資産を格納しています。ハードウェア障害、ソフトウェアのバグ、人的ミス、災害など様々な理由でデータが失われる可能性があります。定期的なバックアップを取得し、確実にリストアできることを確認することは、データベース管理の基本中の基本です。

flowchart TD A[データ損失のリスク] --> B[ハードウェア障害] A --> C[ソフトウェアバグ] A --> D[人的ミス] A --> E[セキュリティ侵害] A --> F[自然災害] G[対策: バックアップ] --> H[定期バックアップ] G --> I[テスト済みリストア手順] G --> J[複数世代保持] G --> K[オフサイト保存]

バックアップの種類

  • 論理バックアップ - SQLのCREATE/INSERT文として出力。mysqldumpコマンドで取得。可読性が高く、異なるバージョン間で移行可能。
  • 物理バックアップ - データファイルをそのままコピー。MySQL Enterprise BackupやPercona XtraBackupで取得。高速で大規模DB向き。
  • フルバックアップ - すべてのデータを取得。復元が簡単だが、時間とストレージを消費。
  • 増分バックアップ - 前回バックアップ以降の変更のみ取得。効率的だが、復元が複雑。
  • バイナリログ - すべてのデータ変更を記録。ポイントインタイムリカバリに利用。
バックアップ種類 メリット デメリット 適用場面
論理バックアップ 可読性が高い、テーブル単位で取得可能、バージョン間移行可能 遅い、サーバー負荷が高い 小〜中規模DB、定期バックアップ
物理バックアップ 高速、一貫性が高い 専用ツールが必要、可読性がない 大規模DB、高速復旧が必要
バイナリログ ポイントインタイムリカバリ可能 単独では使えない（フルバックアップと併用） 特定時点への復元が必要な場合

10.2 mysqldumpによる論理バックアップ

mysqldumpはMySQLに標準で付属する論理バックアップツールです。データベースやテーブルの内容をSQL文として出力し、テキストファイルに保存します。可読性が高く、編集も可能なため、初心者にも扱いやすいバックアップ方法です。

基本的な使い方

データベース全体をバックアップ 
# 単一データベースをバックアップ
mysqldump -u root -p sample_db > backup_sample_db.sql

# 複数データベースをバックアップ
mysqldump -u root -p --databases db1 db2 db3 > backup_multiple.sql

# すべてのデータベースをバックアップ
mysqldump -u root -p --all-databases > backup_all.sql
特定テーブルのみバックアップ 
# 特定のテーブルだけバックアップ
mysqldump -u root -p sample_db users orders > backup_tables.sql

# テーブル構造のみバックアップ（データは含まない）
mysqldump -u root -p --no-data sample_db > backup_schema_only.sql

重要なオプション

オプション 説明 用途
--single-transaction トランザクション内でバックアップを実行（InnoDB推奨） 本番稼働中のバックアップ
--routines ストアドプロシージャと関数もバックアップ 完全なバックアップ
--triggers トリガーもバックアップ 完全なバックアップ
--master-data=2 バイナリログ位置を記録 レプリケーション、PITR
--compress 通信データを圧縮 リモートバックアップ
本番運用に適したバックアップコマンド 
# InnoDB環境で推奨される設定
mysqldump -u root -p \
  --single-transaction \
  --routines \
  --triggers \
  --events \
  --master-data=2 \
  sample_db > backup_$(date +%Y%m%d_%H%M%S).sql

ファイル名に日時を含める

バックアップファイル名に日時を含めることで、複数世代を保持しやすくなります。$(date +%Y%m%d_%H%M%S)でタイムスタンプを追加できます。

10.3 リストア（復元）

mysqldumpで作成したバックアップファイルは、mysqlコマンドを使って復元できます。バックアップファイルにはSQL文が記述されているため、それをそのまま実行することでデータベースが復元されます。

基本的なリストアコマンド 
# データベースをリストア
mysql -u root -p sample_db < backup_sample_db.sql

# 全データベースをリストア
mysql -u root -p < backup_all.sql
flowchart LR A[バックアップファイル
backup.sql] --> B[mysqlコマンド実行] B --> C[SQL文を順次実行] C --> D[CREATE TABLE] C --> E[INSERT INTO] C --> F[CREATE PROCEDURE] D --> G[MySQLデータベース
復元完了] E --> G F --> G

リストア前の注意点

リストアを実行すると、既存のデータは上書きされます。本番環境でリストアする前に、必ずテスト環境でリストアが正常に完了することを確認してください。また、リストア前に現在のデータのバックアップを取ることも推奨されます。

リストア手順の例

安全なリストア手順 
# 1. 現在のデータベースをバックアップ
mysqldump -u root -p sample_db > backup_before_restore.sql

# 2. データベースを削除（既存データをクリア）
mysql -u root -p -e "DROP DATABASE IF EXISTS sample_db;"

# 3. データベースを再作成
mysql -u root -p -e "CREATE DATABASE sample_db;"

# 4. バックアップからリストア
mysql -u root -p sample_db < backup_sample_db.sql

# 5. リストア後の確認
mysql -u root -p sample_db -e "SHOW TABLES;"

10.4 定期バックアップの自動化

手動でバックアップを取得していると、忘れてしまったり、タイミングが遅れたりするリスクがあります。cronやタスクスケジューラを使って定期バックアップを自動化することで、確実にバックアップを取得できます。

バックアップスクリプトの作成

backup.sh 
#!/bin/bash
# MySQLバックアップスクリプト

# 設定
DB_USER="root"
DB_PASS="パスワード"  # 本番環境では.my.cnfを使用
DB_NAME="sample_db"
BACKUP_DIR="/backups/mysql"
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="${BACKUP_DIR}/${DB_NAME}_${DATE}.sql"

# バックアップディレクトリ作成
mkdir -p ${BACKUP_DIR}

# バックアップ実行
mysqldump -u ${DB_USER} -p${DB_PASS} \
  --single-transaction \
  --routines \
  --triggers \
  ${DB_NAME} > ${BACKUP_FILE}

# 圧縮
gzip ${BACKUP_FILE}

# 古いバックアップを削除（30日以前）
find ${BACKUP_DIR} -name "*.sql.gz" -mtime +30 -delete

echo "Backup completed: ${BACKUP_FILE}.gz"

パスワードの安全な管理

スクリプト内に直接パスワードを書くのはセキュリティリスクがあります。~/.my.cnfファイルに認証情報を記述し、パーミッションを600に設定することで、安全に認証できます。

cronで定期実行

crontabの設定 
# crontab編集
crontab -e

# 毎日午前2時にバックアップを実行
0 2 * * * /path/to/backup.sh >> /var/log/mysql_backup.log 2>&1

# 毎週日曜日午前3時にフルバックアップ
0 3 * * 0 /path/to/backup_full.sh >> /var/log/mysql_backup.log 2>&1
flowchart TD A[cronスケジューラ] --> B{指定時刻?} B -- Yes --> C[backup.shを実行] B -- No --> A C --> D[mysqldumpでバックアップ] D --> E[ファイルを圧縮 gzip] E --> F[古いバックアップを削除] F --> G[ログに記録] G --> A

10.5 ポイントインタイムリカバリ（PITR）

ポイントインタイムリカバリ（PITR）は、特定の時点までデータベースを復元する技術です。例えば、午前10時にバックアップを取得し、午後2時に誤ってテーブルを削除してしまった場合、午後1時59分の状態に戻すことができます。これにはバイナリログが必要です。

バイナリログの有効化

my.cnfまたはmy.ini 
[mysqld]
# バイナリログを有効化
log-bin=mysql-bin
server-id=1

# バイナリログの保存期間（日数）
expire_logs_days=7

# バイナリログのフォーマット（ROWが推奨）
binlog_format=ROW
flowchart LR A[フルバックアップ
10:00 AM] --> B[バイナリログ記録開始] B --> C[通常運用
INSERT/UPDATE/DELETE] C --> D[誤操作発生
2:00 PM] D --> E[フルバックアップから復元] E --> F[バイナリログを
10:00〜1:59まで適用] F --> G[1:59 PMの状態に復元完了]

PITRの手順

ポイントインタイムリカバリの実行 
# 1. フルバックアップから復元
mysql -u root -p sample_db < backup_20250115_1000.sql

# 2. バイナリログの位置を確認
mysqlbinlog mysql-bin.000001 | less

# 3. 特定時刻までのバイナリログを適用
mysqlbinlog --stop-datetime="2025-01-15 13:59:00" \
  mysql-bin.000001 mysql-bin.000002 | mysql -u root -p sample_db

# または、特定の位置まで適用
mysqlbinlog --stop-position=12345 \
  mysql-bin.000001 | mysql -u root -p sample_db

mysqlbinlogコマンドの主なオプション

  • --start-datetime - 指定した日時以降のログを出力
  • --stop-datetime - 指定した日時までのログを出力
  • --start-position - 指定した位置以降のログを出力
  • --stop-position - 指定した位置までのログを出力
  • --database - 特定のデータベースのみ出力

PITRは上級者向け

ポイントインタイムリカバリは強力な機能ですが、正確な時刻や位置の指定が必要で、操作ミスのリスクもあります。初心者のうちは定期的なフルバックアップを確実に取得することに集中し、PITRは慣れてから導入しましょう。

実習 10-1: バックアップとリストアを体験しよう

データベースのバックアップを取得し、それをリストアしてデータが復元されることを確認します。

手順

  1. 1 テスト用データベースを作成し、サンプルデータを追加
  2. 2 mysqldumpコマンドでバックアップを取得
  3. 3 データベースを削除して、バックアップからリストアして復元できることを確認
  4. 4 バックアップファイルをテキストエディタで開いて中身を確認

理解度確認クイズ

  1. Q1. 論理バックアップと物理バックアップの違いは何ですか?

    回答を表示

    論理バックアップはデータベースの内容をSQL文（CREATE TABLE、INSERT文など）として出力します。可読性が高く、異なるバージョン間での移行も可能ですが、速度は遅めです。物理バックアップはデータファイルを直接コピーします。高速で一貫性が高いですが、専用ツールが必要で、可読性はありません。

  2. Q2. mysqldumpの--single-transactionオプションは何のために使いますか？

    回答を表示

    --single-transactionオプションは、バックアップをトランザクション内で実行し、一貫性のあるスナップショットを取得するために使います。これによりInnoDBテーブルをロックせずにバックアップできるため、本番稼働中のデータベースでも安全にバックアップが可能です。

  3. Q3. ポイントインタイムリカバリ（PITR）に必要なものは何ですか？

    回答を表示

    ポイントインタイムリカバリには、フルバックアップとバイナリログが必要です。フルバックアップで基準となる時点に戻し、バイナリログを適用することで、特定の時点までの変更を再現します。バイナリログはmy.cnfで「log-bin」を有効にすることで記録されます。

この章のまとめ

  • mysqldumpコマンドで論理バックアップを取得し、mysqlコマンドでリストアできる
  • 論理バックアップは可読性が高く初心者向き、物理バックアップは高速で大規模向き
  • cronやタスクスケジューラで定期バックアップを自動化できる
  • バイナリログを使えばポイントインタイムリカバリで特定時点への復元が可能
