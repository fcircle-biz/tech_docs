10.1 データセット圧縮とインデックスによるI/O削減

大規模データを扱う際、ディスクI/Oがボトルネックとなることがよくあります。COMPRESS=YESオプションを使用することで、データセットのサイズを大幅に削減し、読み書き速度を向上させることができます。また、インデックスを作成することで、特定の条件に合致するレコードを高速に検索できるようになります。

データセット圧縮の基本

DATA データセット名(COMPRESS=YES);
このオプションを指定することで、データは自動的に圧縮されてディスクに保存されます。読み込み時に自動で解凍されるため、プログラムの変更は不要です。データサイズが数GB以上の場合、50%以上の容量削減が期待できます。

例1: 圧縮データセットの作成

データセット圧縮.sas 
/* 圧縮なしのデータセット */
DATA large_data;
    SET original_data;
RUN;

/* 圧縮ありのデータセット */
DATA large_data_compressed(COMPRESS=YES);
    SET original_data;
RUN;

/* データセットサイズの比較 */
PROC CONTENTS DATA=large_data;
RUN;

PROC CONTENTS DATA=large_data_compressed;
RUN;

例2: インデックスの作成と活用

インデックスは、特定の変数に基づいてデータを高速に検索するための仕組みです。WHERE句による絞り込みが頻繁に行われる場合に効果的です。

インデックス作成.sas 
/* 顧客IDにインデックスを作成 */
PROC DATASETS LIBRARY=WORK NOLIST;
    MODIFY customer_data;
    INDEX CREATE customer_id;
QUIT;

/* インデックスを利用した高速検索 */
DATA subset;
    SET customer_data;
    WHERE customer_id = '12345';  /* インデックスが自動的に使用される */
RUN;

インデックス使用時の注意点

インデックスは検索を高速化しますが、データセットの更新（INSERT/UPDATE/DELETE）は遅くなります。また、インデックス自体がディスク容量を消費します。頻繁に検索される変数にのみ作成し、更新が多いデータセットでは慎重に使用してください。

10.2 WHERE句とKEEP/DROPステートメントの最適化

SASプログラムの実行速度は、WHERE句とKEEP/DROPステートメントの配置順序によって大きく変わります。WHERE句でデータを絞り込んでからKEEPで必要な変数だけを残すことで、メモリ使用量とI/Oを削減できます。

最適化の基本原則

  • WHERE句は早く - データをできるだけ早く絞り込む
  • KEEPは多く - 不要な変数は早めに除外する
  • 計算は後で - 必要なデータだけに対して計算を行う

例3: 非効率なコード vs 効率的なコード

非効率なコード.sas 
/* 非効率: すべての変数を読み込んでから絞り込む */
DATA result;
    SET large_dataset;
    /* 100個の変数をすべて読み込む */
    IF age >= 20 AND age <= 30;  /* IF文は遅い */
    /* ここで初めて必要な変数だけを残す */
    KEEP customer_id name age;
RUN;
効率的なコード.sas 
/* 効率的: 先に絞り込み、必要な変数だけ読み込む */
DATA result(KEEP=customer_id name age);
    SET large_dataset;
    WHERE age >= 20 AND age <= 30;  /* WHERE句は高速 */
    /* 必要な3変数だけがメモリに読み込まれる */
RUN;

WHERE句 vs IF文の違い

WHERE句はデータを読み込む前に評価されるため高速ですが、IF文はすべてのデータをメモリに読み込んだ後に評価されます。可能な限りWHERE句を使用しましょう。ただし、WHERE句ではLAG関数やFIRST./LAST.変数は使用できません。

例4: 複数ステップでの段階的絞り込み

段階的絞り込み.sas 
/* ステップ1: 大まかな絞り込み */
DATA temp1(COMPRESS=YES);
    SET huge_dataset(KEEP=customer_id date amount region);
    WHERE region = 'Tokyo' AND date >= '01JAN2024'd;
RUN;

/* ステップ2: さらに詳細な条件で絞り込み */
DATA final_result;
    SET temp1;
    WHERE amount > 10000;
RUN;

10.3 実行時間の計測とボトルネック特定

プログラムの実行時間を計測することで、どの処理に時間がかかっているかを特定できます。OPTIONS FULLSTIMER;を使用すると、各ステップの詳細な実行統計が表示されます。

実行時間計測.sas 
/* 詳細な実行時間計測を有効化 */
OPTIONS FULLSTIMER;

/* 処理の実行 */
DATA result;
    SET large_dataset;
    WHERE age >= 20;
RUN;

/* 実行時間計測を無効化 */
OPTIONS NOFULLSTIMER;

FULLSTIMERの出力例

NOTE: DATA statement used (Total process time):
      real time           2.34 seconds
      cpu time            1.89 seconds

例5: 処理速度の比較

パフォーマンス比較.sas 
OPTIONS FULLSTIMER;

/* 方法1: Merge文による結合 */
DATA result1;
    MERGE master(IN=a) transaction(IN=b);
    BY customer_id;
    IF a AND b;
RUN;

/* 方法2: ハッシュオブジェクトによる結合 */
DATA result2;
    IF _N_ = 1 THEN DO;
        DECLARE HASH h(DATASET: "master");
        h.DefineKey("customer_id");
        h.DefineData(ALL: "YES");
        h.DefineDone();
    END;
    SET transaction;
    IF h.FIND() = 0;
RUN;

パフォーマンス分析のポイント

  • real time - 実際の経過時間（ディスクI/Oを含む）
  • cpu time - CPU処理時間（計算に使われた時間）
  • real time >> cpu time の場合、I/Oがボトルネック
  • cpu time が大きい場合、アルゴリズムの見直しが必要

10.4 コーディング規約とベストプラクティス

可読性の高いコードは、保守性を向上させ、バグを減らします。チーム開発や長期運用を見据えて、一貫したコーディングスタイルを採用しましょう。

ベストプラクティス1: インデントとスタイル

❌ 悪い例 
data x;
set y;
if a>10 then do;
b=a*2;
c=b+5;
end;
run;
✅ 良い例 
DATA result;
    SET source_data;
    IF amount > 10 THEN DO;
        double_amount = amount * 2;
        final_value = double_amount + 5;
    END;
RUN;

ベストプラクティス2: 命名規則

  • 変数名は小文字とアンダースコア - customer_id, total_amount
  • データセット名は明確に - temp_customer_summary, final_sales_report
  • マクロ変数は大文字 - %LET START_DATE = '01JAN2024'd;
  • 意味のある名前を使う - cnt よりも customer_count

ベストプラクティス3: コメントの書き方

適切なコメント例.sas 
/*==============================================================
 * プログラム名: monthly_sales_report.sas
 * 作成日: 2024-01-15
 * 作成者: Data Analysis Team
 * 目的: 月次売上レポートの生成
 * 入力: sales.monthly_data
 * 出力: report.monthly_summary
 ==============================================================*/

/* グローバルオプション設定 */
OPTIONS COMPRESS=YES FULLSTIMER;

/* マクロ変数定義 */
%LET report_month = 202401;  /* レポート対象月（YYYYMM形式） */
%LET threshold = 1000000;     /* 売上閾値（100万円） */

/* データ抽出と集計 */
DATA monthly_summary(COMPRESS=YES);
    SET sales.monthly_data;
    WHERE month = &report_month.;  /* 対象月のデータのみ抽出 */

    /* 売上区分の設定 */
    IF total_sales >= &threshold. THEN sales_category = 'High';
    ELSE sales_category = 'Low';
RUN;

ベストプラクティス4: エラーハンドリング

エラーチェック.sas 
/* データセットの存在確認 */
%MACRO CheckDataset(dsname);
    %IF %SYSFUNC(EXIST(&dsname.)) = 0 %THEN %DO;
        %PUT ERROR: データセット &dsname. が存在しません;
        %ABORT CANCEL;
    %END;
%MEND;

%CheckDataset(sales.customer_data);

/* 処理実行 */
DATA result;
    SET sales.customer_data;
RUN;

/* エラーチェック */
%IF &SYSERR. > 4 %THEN %DO;
    %PUT ERROR: データ処理中にエラーが発生しました;
    %ABORT CANCEL;
%END;

実習 10-1: パフォーマンス最適化の実践

非効率なコードを最適化し、実行時間を計測して改善効果を確認してください。

手順

  1. 1 以下の非効率なコードを実行し、FULLSTIMERで実行時間を記録します。
  2. 2 WHERE句、KEEP、COMPRESSを使ってコードを最適化します。
  3. 3 最適化後のコードを実行し、実行時間を比較します。

非効率なコード（改善前）

DATA result;
    SET large_dataset;
    IF region = 'Tokyo' AND age >= 20 AND age <= 30;
RUN;

理解度確認クイズ

  1. Q1. COMPRESS=YESオプションを使用する主な目的は何ですか？

    回答を表示

    データセットのディスク容量を削減し、I/O速度を向上させることです。圧縮により、読み書き時間が短縮され、特に大規模データセットで効果的です。

  2. Q2. WHERE句とIF文の主な違いは何ですか？

    回答を表示

    WHERE句はデータを読み込む前に評価されるため高速ですが、IF文はデータを読み込んだ後に評価されます。WHERE句ではLAG関数やFIRST./LAST.変数は使用できませんが、通常の条件フィルタリングではWHERE句の方が効率的です。

  3. Q3. FULLSTIMERで表示されるreal timeとcpu timeの違いは何ですか？

    回答を表示

    real timeは実際の経過時間（ディスクI/Oを含む）を示し、cpu timeはCPU処理に使われた時間を示します。real time >> cpu timeの場合、I/Oがボトルネックであり、cpu timeが大きい場合はアルゴリズムの見直しが必要です。

  • FULLSTIMERにより実行時間を計測し、ボトルネックを特定できる。
  • 一貫したコーディング規約により、可読性と保守性が向上する。
