CAS（Cloud Analytic Services）の基礎

SAS Viyaの心臓部であるCASエンジンを深く理解します。インメモリ処理、分散コンピューティング、CASライブラリとCASテーブルの操作方法を習得します。

この章で学ぶこと

  • CASエンジンのアーキテクチャと動作原理を理解する
  • インメモリ処理と分散コンピューティングの仕組みを学ぶ
  • CASセッションの開始・終了と基本操作を習得する
  • CASライブラリとCASテーブルの操作方法を理解する
  • CASアクションの使い方と実践的な活用方法を身につける

3.1 CASエンジンのアーキテクチャ

CAS（Cloud Analytic Services）は、SAS Viyaの最も重要なコンポーネントです。第1章で概要を学びましたが、ここではより詳しく仕組みを理解します。

CASクラスタの構成要素

CASクラスタはControllerとWorkerという2種類のノードで構成されます。

  • CAS Controller: クライアントからのリクエストを受け取り、タスクをWorkerに分散し、結果を集約して返す制御ノード
  • CAS Worker: データの一部を担当し、実際の計算処理を並列実行する処理ノード（複数台で水平スケール可能）

ヒント

CASクラスタは水平スケールが可能です。Workerノードを追加すれば、データ量やユーザー数が増えても性能を維持できます。これがクラウドネイティブの強みです。

インメモリ処理の仕組み

従来のSASはディスクから必要なデータを読み込みながら処理していました。一方、CASはデータ全体をメモリ（RAM）に展開してから処理します。

項目 従来のSAS（ディスクベース） CAS（インメモリ）
データ配置 ディスク（SAS7BDATファイル） RAM（メモリ上）
読み込み速度 遅い（I/O待ちが発生） 非常に高速
処理方式 単一プロセス（逐次処理） 複数Workerで並列処理
同時アクセス 制限あり 複数ユーザー同時利用可能
性能 標準 数十倍～数百倍高速

重要ポイント

インメモリ処理は高速ですが、メモリ容量の制約があります。大規模データの場合は、Workerノードを増やすか、データをフィルタリングしてからCASにロードすることが推奨されます。

3.2 CASセッションの開始と終了

CASを利用するには、まずCASセッションを開始する必要があります。セッションはCASクラスタへの接続チャネルで、セッション内でデータ操作やアクション実行を行います。

CASセッションの開始

cas_session_start.sas 
/* CASセッションの開始 */
cas mysession;

/* セッション情報を確認 */
proc cas;
  session mysession;
  builtins.sessionStatus;
quit;

cas mysession;でセッションが開始され、mysessionという名前でCASクラスタに接続されます。

CASセッションの終了

cas_session_end.sas 
/* CASセッションの終了 */
cas mysession terminate;

注意

セッションを終了すると、そのセッション内でロードしたCASテーブルはメモリから削除されます。重要なデータは永続化（SAVE）してから終了しましょう。

3.3 CASライブラリとCASテーブルの操作

CASでは、データはCASライブラリとCASテーブルという概念で管理されます。

CASライブラリとは

CASライブラリ（Caslib）は、従来のSASライブラリに相当する概念です。データソースへの接続情報をまとめたものです。

  • CASUSER: ユーザー個人用の一時的なCASライブラリ（セッション終了時に削除）
  • Public: すべてのユーザーが共有できる永続的なCASライブラリ
  • カスタムCaslib: 特定のデータソース（CSV、Hadoop、データベースなど）に接続するライブラリ

CASテーブルとは

CASテーブルは、CASライブラリ内に存在する、メモリ上に展開されたデータセットです。従来のSASデータセットに相当します。

CASライブラリとCASテーブルの関係

データのロード

外部データをCASテーブルとしてメモリにロードするにはPROC CASUTILまたはtable.loadTableアクションを使用します。

load_data_to_cas.sas 
/* SASデータセットをCASにロード */
proc casutil;
  load data=sashelp.cars
       casout="cars_in_cas"
       outcaslib="casuser"
       replace;
quit;

/* ロードされたテーブルを確認 */
proc cas;
  table.tableInfo / caslib="casuser";
quit;

データの保存と削除

save_and_drop_table.sas 
/* CASテーブルをSASデータセットとして保存 */
proc casutil;
  save casdata="cars_in_cas"
       incaslib="casuser"
       outcaslib="casuser"
       casout="cars_saved.sashdat"
       replace;
quit;

/* CASテーブルをメモリから削除 */
proc casutil;
  droptable casdata="cars_in_cas"
            incaslib="casuser";
quit;

3.4 CASアクションの実践的な使い方

CASアクションは、CASエンジン上で実行される処理単位です。従来のPROCプロシジャに相当しますが、分散処理に最適化されています。

主要なCASアクションセット

simple

基本統計量の計算（summary, freq, distinctなど）

aggregation

集計処理（GROUP BY、ピボット、集約など）

dataPreprocess

データ前処理（欠損値補完、標準化、変換など）

regression

回帰分析（線形回帰、ロジスティック回帰など）

実践例: データ集計

cas_aggregation_example.sas 
/* データをCASにロード */
proc casutil;
  load data=sashelp.cars casout="cars";
quit;

/* simple.summaryアクションで基本統計量を計算 */
proc cas;
  simple.summary /
    table={name="cars"}
    inputs={"msrp", "horsepower", "mpg_city"}
    casout={name="summary_stats", replace=true};
quit;

/* 結果を表示 */
proc print data=summary_stats;
run;

実践例: グループ別集計

cas_groupby_example.sas 
/* aggregation.aggregateアクションでグループ別集計 */
proc cas;
  aggregation.aggregate /
    table={name="cars"}
    varSpecs={
      {name="msrp", summarySubset={"MEAN", "MAX", "MIN"}},
      {name="horsepower", summarySubset={"MEAN"}}
    }
    casOut={name="cars_by_type", replace=true}
    groupBy={"type"};
quit;

/* 結果を確認 */
proc print data=cars_by_type;
run;

ヒント

CASアクションは、従来のPROCプロシジャより高速に動作します。特に大規模データの集計では、数十倍の速度向上が期待できます。既存のSASコードをCASアクションに置き換えると、パフォーマンスが大幅に改善します。

実習 3-1: CASテーブルの作成と基本操作

実際にCASセッションを開始し、データをロードして基本的な集計処理を行ってみましょう。

手順

  1. 1 SAS Studioにログインし、新しいプログラムファイルを作成します
  2. 2 CASセッションを開始します（cas mysession;）
  3. 3 sashelp.carsデータセットをCASにロードします
  4. 4 simple.summaryアクションで基本統計量を計算します
  5. 5 メーカー（Make）別の平均価格を集計してみましょう
  6. 6 最後にCASセッションを終了します

第3章のまとめ

  • CASエンジンはControllerとWorkerで構成され、データを分散並列処理する
  • インメモリ処理により従来のSASより数十倍～数百倍高速に動作する
  • CASセッションを開始してからCASライブラリとCASテーブルを操作する
  • データのロード・保存・削除はPROC CASUTILで実行する
  • CASアクションは従来のPROCプロシジャに代わる高速な処理単位である

理解度確認クイズ

  1. Q1. CASクラスタを構成する2種類のノードの名前と役割を説明してください。

    回答を表示

    ①CAS Controller：クライアントからのリクエストを受け取り、タスクをWorkerに分散し、結果を集約する制御ノード。②CAS Worker：データの一部を担当し、実際の計算処理を並列実行する処理ノード。

  2. Q2. インメモリ処理と従来のディスクベース処理の主な違いは何ですか？

    回答を表示

    インメモリ処理はデータ全体をRAM（メモリ）に展開してから処理するため、ディスクI/O待ちが発生せず非常に高速です。一方、従来のディスクベース処理は必要なデータをディスクから読み込みながら処理するため、I/O待ちで遅くなります。

  3. Q3. CASセッションを終了するとCASテーブルはどうなりますか？

    回答を表示

    セッション内でロードしたCASテーブルはメモリから削除されます。データを永続化したい場合は、終了前にSAVEアクションで保存する必要があります。

  4. Q4. CASアクションと従来のPROCプロシジャの違いを説明してください。

    回答を表示

    CASアクションはCASエンジン上で分散並列処理されるため、従来のPROCプロシジャより高速に動作します。また、API経由で呼び出せるため、SAS以外の言語（Python、Rなど）からも利用可能です。
