RAG（検索拡張生成）の実装

これまで学んだ技術を組み合わせて、独自のドキュメントに基づいて回答するRAGシステムを構築します。RetrievalQAチェーンの実装方法を習得します。

この章で学ぶこと

  • RAG（Retrieval-Augmented Generation）の基本概念と仕組みを理解する
  • RetrievalQAチェーンを使った基本的なRAGシステムの実装方法を習得する
  • LCELを使ったカスタムRAGパイプラインの構築方法を学ぶ
  • RAGシステムの精度を向上させるためのチューニング手法を理解する

13.1 RAGとは何か

RAG（Retrieval-Augmented Generation: 検索拡張生成）は、外部のデータソースから関連情報を検索し、それをLLMに渡すことで、より正確で文脈に即した回答を生成する技術です。

LLMは学習時点のデータに基づく知識しか持たないため、最新情報や企業固有の情報については回答できません。RAGはこの制限を克服し、LLMに「外部知識」を与える仕組みです。

RAGの重要性

ファインチューニングと異なり、RAGはモデルの再学習なしで知識を更新できます。社内文書、製品マニュアル、最新ニュースなど、動的に変化する情報を扱う際に非常に有効です。

flowchart TD Q[ユーザーの質問] --> E[Embeddingsでベクトル化] E --> VS[(Vector Store
類似検索)] VS --> R[関連ドキュメント取得] R --> P[プロンプト作成
質問+ドキュメント] P --> LLM[LLM] LLM --> A[回答生成] subgraph 事前準備 D[ドキュメント] --> TS[Text Splitter] TS --> EMB[Embeddings] EMB --> VS end

図13-1: RAGの基本フロー

RAGの3つのステップ

1 Indexing（インデックス化）

ドキュメントを読み込み、分割し、Embeddingsでベクトル化してVector Storeに保存

2 Retrieval（検索）

ユーザーの質問をベクトル化し、Vector Storeから類似度の高いドキュメントを検索

3 Generation（生成）

検索したドキュメントを質問と一緒にLLMに渡し、根拠のある回答を生成

13.2 RetrievalQAチェーンの基本

LangChainのRetrievalQAチェーンは、RAGシステムを簡単に構築できる高レベルのコンポーネントです。Retriever、LLM、プロンプトテンプレートを組み合わせて、数行のコードでRAGを実装できます。

基本的な実装例

basic_rag.py 
from langchain.chains import RetrievalQA
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_community.vectorstores import FAISS
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.document_loaders import TextFileLoader

# ドキュメント読み込み
loader = TextFileLoader("company_policy.txt")
documents = loader.load()

# テキスト分割
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50
)
splits = text_splitter.split_documents(documents)

# Vector Store作成
embeddings = OpenAIEmbeddings()
vectorstore = FAISS.from_documents(splits, embeddings)

# RetrievalQAチェーン構築
llm = ChatOpenAI(model="gpt-4", temperature=0)
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",  # ドキュメントの結合方法
    retriever=vectorstore.as_retriever()
)

# 質問実行
response = qa_chain.invoke({"query": "有給休暇の取得方法を教えてください"})
print(response["result"])

重要ポイント

chain_type="stuff"は、取得したすべてのドキュメントを1つのプロンプトに結合します。シンプルで高速ですが、ドキュメント量が多いとトークン制限を超える可能性があります。

chain_typeの種類

stuff（シンプル結合）

すべてのドキュメントを1つのプロンプトに結合してLLMに渡す

メリット: 最も速く、シンプル
デメリット: ドキュメントが多いとトークン制限に引っかかる

map_reduce（分割処理）

各ドキュメントごとにLLMを呼び出し、最後に結果を統合

メリット: 大量ドキュメントに対応
デメリット: LLM呼び出しが増えてコストと時間がかかる

refine（段階的改善）

最初のドキュメントで回答を生成し、次のドキュメントで改善を繰り返す

メリット: 精度が高い
デメリット: 処理に時間がかかる

map_rerank（ランキング）

各ドキュメントで回答を生成し、最も確信度の高い回答を選択

メリット: 複数候補から最適解を選べる
デメリット: コストが高い

13.3 LCELによるカスタムRAGパイプライン

LCEL（LangChain Expression Language）を使うことで、より柔軟でカスタマイズ可能なRAGシステムを構築できます。プロンプトやドキュメントフォーマットを自由に制御できます。

LCEL版RAGの実装

lcel_rag.py 
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

# プロンプトテンプレート
template = """以下のコンテキストを参考に、質問に回答してください。
コンテキストに含まれていない情報については「わかりません」と答えてください。

コンテキスト:
{context}

質問: {question}

回答:"""

prompt = ChatPromptTemplate.from_template(template)

# ドキュメントフォーマット関数
def format_docs(docs):
    return "\n\n".join([f"[文書{i+1}]\n{doc.page_content}" for i, doc in enumerate(docs)])

# LCELでRAGチェーン構築
rag_chain = (
    {
        "context": vectorstore.as_retriever() | format_docs,
        "question": RunnablePassthrough()
    }
    | prompt
    | llm
    | StrOutputParser()
)

# 実行
response = rag_chain.invoke("有給休暇の取得方法を教えてください")
print(response)

LCELのメリット

LCELを使うと、ドキュメントのフォーマット、プロンプトの構造、出力の処理など、RAGパイプラインの各ステップを細かく制御できます。デバッグもしやすくなります。

実習 13-1: 基本的なRAGシステムの構築

自分のテキストファイルを使って、RetrievalQAチェーンベースのRAGシステムを構築し、質問応答を実装してみましょう。

手順

  1. 1 サンプルテキストファイル（会社規定、製品マニュアルなど）を準備する
  2. 2 TextFileLoaderでドキュメントを読み込み、RecursiveCharacterTextSplitterで分割する
  3. 3 OpenAIEmbeddingsとFAISSを使ってVector Storeを作成する
  4. 4 RetrievalQA.from_chain_type()でRAGチェーンを構築し、質問を実行する
  5. 5 異なるchain_type（stuff、map_reduce、refine）を試して結果を比較する

13.4 RAGの精度向上テクニック

基本的なRAGシステムを構築した後は、精度を向上させるためのチューニングが重要です。以下のテクニックを組み合わせることで、より正確で有用な回答を生成できます。

1. 検索パラメータの調整

tuning_retrieval.py 
# 検索ドキュメント数を調整
retriever = vectorstore.as_retriever(
    search_type="similarity",  # 類似度検索
    search_kwargs={"k": 4}  # 上位4件を取得（デフォルトは4）
)

# MMR検索を使用（多様性を重視）
retriever_mmr = vectorstore.as_retriever(
    search_type="mmr",
    search_kwargs={
        "k": 4,  # 最終的に返すドキュメント数
        "fetch_k": 20,  # 初期検索数
        "lambda_mult": 0.5  # 多様性（0-1、1に近いほど類似性重視）
    }
)

# スコア閾値による絞り込み
retriever_threshold = vectorstore.as_retriever(
    search_type="similarity_score_threshold",
    search_kwargs={
        "score_threshold": 0.7  # 類似度スコアが0.7以上のもののみ
    }
)

2. チャンクサイズの最適化

optimize_chunks.py 
# 小さめのチャンク（詳細な検索向け）
text_splitter_small = RecursiveCharacterTextSplitter(
    chunk_size=300,
    chunk_overlap=50
)

# 大きめのチャンク（文脈保持優先）
text_splitter_large = RecursiveCharacterTextSplitter(
    chunk_size=1000,
    chunk_overlap=200
)

# セマンティック分割（文の区切りを重視）
from langchain.text_splitter import SentenceTransformersTokenTextSplitter
semantic_splitter = SentenceTransformersTokenTextSplitter(
    chunk_overlap=50,
    tokens_per_chunk=256
)

3. プロンプトエンジニアリング

custom_prompt.py 
from langchain.prompts import PromptTemplate

# カスタムプロンプト（出典明記版）
template = """あなたは企業の社内規定に詳しいアシスタントです。
以下のコンテキストを参考に、質問に正確に回答してください。

コンテキスト:
{context}

質問: {question}

回答する際の注意事項:
1. コンテキストに基づいた情報のみを使用してください
2. 情報の出典（文書番号）を明記してください
3. コンテキストに情報がない場合は「提供された情報では回答できません」と答えてください
4. 推測や一般的な知識ではなく、コンテキストの内容に忠実に回答してください

回答:"""

custom_prompt = PromptTemplate(
    template=template,
    input_variables=["context", "question"]
)

# RetrievalQAに適用
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",
    retriever=retriever,
    chain_type_kwargs={"prompt": custom_prompt}
)

4. メタデータフィルタリング

from langchain.schema import Document

    }

13.5 ソース情報の表示

RAGシステムでは、回答の根拠となったドキュメントを表示することで、信頼性を高めることができます。return_source_documentsオプションを使うと、検索されたドキュメントを取得できます。

source_display.py 
# ソース情報を含めて返す
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",
    retriever=retriever,
    return_source_documents=True  # ソース情報を返す
)

# 実行
result = qa_chain.invoke({"query": "有給休暇の取得方法を教えてください"})

# 回答表示
print("【回答】")
print(result["result"])

# ソース情報表示
print("\n【参照ドキュメント】")
for i, doc in enumerate(result["source_documents"], 1):
    print(f"\n--- 文書 {i} ---")
    print(f"内容: {doc.page_content[:200]}...")  # 最初の200文字
    print(f"出典: {doc.metadata.get('source', '不明')}")
    print(f"ページ: {doc.metadata.get('page', '不明')}")

実習 13-2: LCELでカスタムRAGを実装

LCELを使って、プロンプトとドキュメントフォーマットをカスタマイズしたRAGシステムを構築してみましょう。

手順

  1. 1 ドキュメントのフォーマット関数を作成し、文書番号と出典を明記するようにする
  2. 2 カスタムプロンプトテンプレートを作成し、回答形式を指定する
  3. 3 LCELのパイプ演算子を使ってRAGチェーンを構築する
  4. 4 複数の質問で実行し、回答の品質を確認する

トラブルシューティング

RAGの回答が不正確または「わかりません」と返される

原因: 検索されたドキュメントに質問に関連する情報が含まれていない、またはチャンクサイズが不適切

解決方法:

  • search_kwargsの"k"を増やして、より多くのドキュメントを取得する
  • chunk_sizeとchunk_overlapを調整し、より適切な分割を行う
  • MMR検索を使って、多様な視点のドキュメントを取得する
  • return_source_documentsで実際に検索されたドキュメントを確認する
トークン制限エラー（context length exceeded）

原因: 検索されたドキュメントの合計トークン数がLLMの入力制限を超えている

解決方法:

  • chain_type="stuff"から"map_reduce"や"refine"に変更する
  • search_kwargsの"k"を減らして、取得ドキュメント数を削減する
  • chunk_sizeを小さくして、各ドキュメントを短くする
  • トークン数の多いモデル（gpt-4-turboなど）を使用する
検索速度が遅い

原因: Vector Storeのデータ量が多い、またはクラウドベースのベクトルDBを使用している

解決方法:

  • FAISSなどローカルベクトルストアを使用する
  • メタデータフィルタリングで検索範囲を限定する
  • インデックス作成時に最適化（FAISS.save_local()で保存し再利用）
  • 検索パラメータfetch_kを調整する

この章のまとめ

  • RAGは外部データから情報を検索し、LLMに渡すことで正確な回答を生成する技術
  • RetrievalQAチェーンを使うと、簡単にRAGシステムを構築できる
  • chain_typeにはstuff、map_reduce、refine、map_rerankがあり、用途に応じて使い分ける
  • LCELを使うと、プロンプトやドキュメントフォーマットをカスタマイズできる
  • 検索パラメータ、チャンクサイズ、プロンプトエンジニアリングで精度を向上できる
  • return_source_documentsで回答の根拠となったドキュメントを表示できる

理解度確認クイズ

  1. Q1. RAGの3つのステップ（Indexing、Retrieval、Generation）をそれぞれ説明してください。

    回答を表示

    ①Indexing（インデックス化）: ドキュメントを読み込み、Text Splitterで分割し、Embeddingsでベクトル化してVector Storeに保存する。②Retrieval（検索）: ユーザーの質問をベクトル化し、Vector Storeから類似度の高いドキュメントを検索する。③Generation（生成）: 検索したドキュメントを質問と一緒にプロンプトに埋め込み、LLMに渡して回答を生成する。

  2. Q2. RetrievalQAのchain_type="stuff"と"map_reduce"の違いは何ですか？

    回答を表示

    "stuff"は、検索したすべてのドキュメントを1つのプロンプトに結合してLLMに1回だけ渡します。シンプルで高速ですが、ドキュメントが多いとトークン制限を超える可能性があります。"map_reduce"は、各ドキュメントごとに個別にLLMを呼び出して回答を生成し、最後にそれらを統合します。大量のドキュメントに対応できますが、LLM呼び出しが増えるためコストと時間がかかります。

  3. Q3. RAGシステムの精度を向上させるための主な手法を3つ挙げてください。

    回答を表示

    ①検索パラメータの調整: search_kwargsの"k"を調整して適切な数のドキュメントを取得する、MMR検索で多様性を確保する、score_thresholdで低品質な結果を除外する。②チャンクサイズの最適化: chunk_sizeとchunk_overlapを調整して、適切な粒度でドキュメントを分割する。③プロンプトエンジニアリング: カスタムプロンプトテンプレートを作成し、回答形式や注意事項を明確に指示する、出典明記を促す。
