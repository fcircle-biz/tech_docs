ナレッジベースの構築 - ドキュメントのアップロードと管理

PDF、Word、テキストファイルなど、様々なドキュメントをナレッジベースに追加する方法を学び、チャンク分割、インデックス設定、メタデータ管理の実践を行います。

この章で学ぶこと

  • ドキュメントをナレッジベースにアップロードする具体的な手順
  • チャンク分割の設定とドキュメント特性に応じた最適化
  • インデックス方法の選択とベクトル埋め込みモデルの設定
  • メタデータの活用とドキュメント管理のベストプラクティス

7.1 ナレッジベースへのドキュメントアップロード

Difyのナレッジベースには、様々な形式のドキュメントをアップロードして、AIに参照させることができます。 まずは基本的なアップロード手順を確認しましょう。

実習：ドキュメントをアップロードする

ステップ1: ナレッジベースの作成

  1. 1. Difyのダッシュボードで「Knowledge」タブをクリック
  2. 2. 右上の「Create Knowledge」ボタンをクリック
  3. 3. ナレッジベース名を入力（例：「会社の規定集」）
  4. 4. 説明を入力（任意）し、「Create」をクリック

ステップ2: ドキュメントのアップロード

  1. 1. 作成したナレッジベースを開く
  2. 2. 「Add Document」ボタンをクリック
  3. 3. 「Upload Files」を選択、またはドラッグ&ドロップ
  4. 4. 対応ファイル形式を選択してアップロード

Difyがサポートするファイル形式

PDF

マニュアル、報告書など（最も一般的）

Word (.docx)

企画書、仕様書など

テキスト (.txt, .md)

README、技術文書など

CSV

データリスト、一覧表など

HTML

Webページ、ドキュメントサイト

Excel (.xlsx)

スプレッドシート、データ集

ファイルサイズ制限

Dify Cloudでは、1ファイルあたり最大15MBまでアップロード可能です。 大きなファイルは分割するか、セルフホスト版での設定変更を検討してください。 複数ファイルを一度にアップロードすることも可能です。

7.2 チャンク分割の理解と設定

チャンク（Chunk）とは、長いドキュメントを小さな塊に分割したものです。 RAGでは、ドキュメント全体ではなく、関連性の高いチャンクだけを検索して参照します。

チャンク分割の仕組み

（500文字）"] B --> C2["チャンク2

（500文字）"] B --> C4["..."] C1 --> D1["ベクトル埋め込み1"] C2 --> D2["ベクトル埋め込み2"] C3 --> D3["ベクトル埋め込み3"] D1 --> E["ベクトルDB"] D2 --> E D3 --> E

なぜチャンク分割が必要かというと、LLMにはコンテキスト長の制限があり、 長いドキュメント全体を一度に処理できないためです。また、関連性の高い部分だけを参照することで、回答の精度が向上します。

チャンク分割の主要設定

1. チャンクサイズ（Chunk Size）

1つのチャンクに含める文字数またはトークン数。デフォルトは500トークン。

推奨: 一般的な文書は500-800、技術文書は800-1200

2. オーバーラップ（Overlap）

隣接するチャンク間で重複させる文字数。コンテキストの連続性を保つために重要。

推奨: チャンクサイズの10-20%（例：500トークンなら50-100）

3. セパレーター（Separator）

チャンクの区切り方。段落、セクション、文単位など。

推奨: 自動（Automatic）が最適なセパレーターを判断

実習：チャンク設定を調整する

  1. 1. アップロード画面で「Settings」または「Advanced」を展開
  2. 2. 「Chunk Size」を設定（例：800トークン）
  3. 3. 「Chunk Overlap」を設定（例：100トークン）
  4. 4. 「Separator」は「Automatic」を選択
  5. 5. プレビューでチャンクの分割結果を確認
ドキュメントタイプ 推奨チャンクサイズ 推奨オーバーラップ 理由
FAQ、短文集 200-400トークン 20-50トークン 各Q&Aが独立しているため小さく
一般文書、記事 500-800トークン 50-100トークン 標準的な段落サイズに対応
技術文書、仕様書 800-1200トークン 100-150トークン 複雑な概念を保持するため大きく
コード、API文書 300-600トークン 50-80トークン 関数・クラス単位で分割

7.3 インデックス方法とベクトル埋め込みモデル

チャンクに分割したテキストは、ベクトル埋め込み（Embedding）と呼ばれる数値化処理を経て、 ベクトルデータベースにインデックスされます。

Difyのインデックス方法

1. ハイクオリティインデックス（High Quality）

推奨: 外部の埋め込みモデル（OpenAI、Cohere等）を使用し、高精度な検索を実現。 APIコストが発生するが、検索品質が高い。

2. エコノミーインデックス（Economy）

キーワードベースの検索。埋め込みモデル不要でコストゼロ。 短文やFAQには十分だが、意味的な類似性検索には不向き。

主要な埋め込みモデル

OpenAI text-embedding-3-small: コストパフォーマンスが高く、多言語対応。一般的な用途に推奨。
OpenAI text-embedding-3-large: 最高精度。重要なドキュメントやコンテキストが複雑な場合に使用。
Cohere embed-multilingual-v3.0: 多言語性能に優れる。日本語と英語が混在する文書に有効。
Azure OpenAI Embeddings: エンタープライズ向け。セキュリティとコンプライアンス重視。
モデル コスト（1M トークン） 次元数 用途
text-embedding-3-small $0.02 1536 一般的な用途（推奨）
text-embedding-3-large $0.13 3072 高精度が必要な場合
Cohere embed-multilingual $0.10 1024 多言語ドキュメント

埋め込みモデルは後から変更できない

一度インデックスを作成すると、埋め込みモデルの変更にはナレッジベースの再作成が必要です。 最初に適切なモデルを選択してください。テスト用には小さなドキュメントで試してから本番データをアップロードすることを推奨します。

7.4 メタデータ管理とドキュメント整理

メタデータ（Metadata）は、ドキュメントに関する追加情報です。 メタデータを活用することで、検索精度の向上や、ドキュメントの管理が容易になります。

Difyで管理できるメタデータ

タイトル（Title）: ドキュメント名。検索結果に表示されます。
カテゴリ（Category）: ドキュメントを分類。例：「マニュアル」「FAQ」「規定」
タグ（Tags）: 複数付与可能。例：「人事」「給与」「休暇」
作成日・更新日: ドキュメントの鮮度管理。
カスタムフィールド: 任意のキー・バリューペアを追加可能。

メタデータを活用した高度な検索

例1: カテゴリフィルタ

「人事規定に関する質問にのみ答える」チャットボットを作る場合、カテゴリ「人事」のドキュメントのみを検索対象にできます。

例2: 日付フィルタ

「最新版のマニュアルのみを参照」するよう設定し、古い情報を除外できます。

例3: タグの組み合わせ

「製品A」かつ「トラブルシューティング」のタグを持つドキュメントだけを検索対象にできます。

実習：メタデータを設定する

  1. 1. ナレッジベースのドキュメント一覧で、編集したいドキュメントを選択
  2. 2. 「Edit Metadata」または歯車アイコンをクリック
  3. 3. カテゴリを設定（例：「社内規定」）
  4. 4. タグを追加（例：「人事」「給与」）
  5. 5. カスタムフィールドを追加（例：「部門: 総務」）
  6. 6. 保存して、検索で活用できることを確認

7.5 ドキュメント管理のベストプラクティス

ナレッジベースを効果的に管理するためのベストプラクティスを紹介します。

1

定期的な更新管理

ドキュメントが古くなると、AIが誤った情報を提供するリスクがあります。 更新日を記録し、定期的にレビューして最新情報を反映してください。

2

ドキュメントの粒度を統一

非常に大きなファイル（例：100ページの総合マニュアル）と小さなファイル（例：1ページのFAQ）を混在させると、 検索精度が低下します。大きなファイルは章ごとに分割することを推奨します。

3

テキスト品質を確保

スキャンPDFの場合はOCR（文字認識）が正確か確認してください。 画像やグラフは、説明文を追加してテキスト情報を補完すると効果的です。

4

ナレッジベースの分割

用途が異なる場合は、ナレッジベースを分けることを検討してください。 例：「製品マニュアル用」「社内FAQ用」「法務文書用」など。目的に応じて検索対象を絞れます。

5

バージョン管理

重要なドキュメントの更新時は、古いバージョンを削除する前に、新バージョンで検証してください。 必要に応じてバージョン番号をファイル名やメタデータに記録します。

理解度確認クイズ

  1. Q1. チャンク分割で「オーバーラップ」を設定する理由は何ですか？

    回答を表示

    隣接するチャンク間で内容を重複させることで、コンテキストの連続性を保ち、チャンクの境界で情報が分断されることを防ぐためです。 これにより、検索精度が向上します。

  2. Q2. 一般的な文書のチャンクサイズとして推奨される範囲はどのくらいですか？

    回答を表示

    500-800トークンが推奨されます。FAQなどの短文集は200-400トークン、技術文書や仕様書は800-1200トークンが適切です。 ドキュメントの特性に応じて調整してください。

  3. Q3. 埋め込みモデルを後から変更したい場合、どうする必要がありますか？

    回答を表示

    埋め込みモデルは一度インデックスを作成すると変更できません。モデルを変更するには、ナレッジベース全体を再作成し、 全ドキュメントを再アップロードする必要があります。そのため、最初に適切なモデルを選択することが重要です。

第7章のまとめ

  • PDF、Word、テキストなど様々な形式のドキュメントをナレッジベースにアップロードできる
  • チャンク分割はRAGの精度を左右する重要な設定。サイズとオーバーラップをドキュメント特性に応じて調整する
  • ハイクオリティインデックスでは埋め込みモデルを選択。一般用途にはtext-embedding-3-smallが推奨
  • メタデータ（カテゴリ、タグ、カスタムフィールド）を活用して、検索精度を向上させ、ドキュメント管理を効率化できる
  • 定期的な更新、粒度の統一、テキスト品質の確保がナレッジベース管理のベストプラクティス
