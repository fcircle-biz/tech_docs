第8章

RAGアプリの実装 - 社内文書に答えるAIを作る

ナレッジベースを活用した実用的なRAGアプリケーションを構築し、検索設定の最適化と回答品質の向上テクニックを習得します。

この章で学ぶこと

  • ナレッジベースを活用したRAGアプリケーションの構築方法
  • 検索設定の最適化（トップK、スコア閾値、ハイブリッド検索）
  • 回答品質を向上させるプロンプトエンジニアリングの実践
  • 引用元の表示とRAG特有のエラー対策

8.1 RAGアプリとは何か

RAG（Retrieval Augmented Generation）は、検索拡張生成と呼ばれる技術で、 LLMに独自のドキュメントを参照させながら回答を生成させる仕組みです。

通常のLLMは訓練データまでの知識しか持っていませんが、RAGを使うことで、 最新の社内文書や製品マニュアルなど、LLMが学習していない情報に基づいた回答を生成できるようになります。

RAGが解決する問題

LLMの訓練データに含まれない情報（社内文書、最新製品情報、個別の業務知識など）を参照させることで、 ハルシネーション（事実と異なる回答）を減らし、正確な情報提供が可能になります。

8.2 RAGアプリの作成手順

Difyで社内文書に基づくFAQ対応チャットボットを作成する手順を見ていきましょう。

1

新しいアプリケーションを作成

Difyの「スタジオ」から「アプリを作成」→「最初から作成」→「チャットアプリ」を選択します。

2

ナレッジベースを追加

「コンテキスト」セクションで、既に作成したナレッジベースを選択します（第7章で作成したもの）。

ナレッジベース設定例 
# ナレッジベース: 社内FAQ文書集
# 設定:
トップK: 3
スコア閾値: 0.5
検索モード: ベクトル検索
リランク: 有効
3

プロンプトを設計

システムプロンプトでRAGの振る舞いを定義します。参照した情報源を明示するよう指示するのがポイントです。

システムプロンプト例 
あなたは社内FAQ対応アシスタントです。

以下のルールに従って回答してください：
1. 提供されたナレッジベースの情報のみを使用する
2. ナレッジベースに情報がない場合は「情報がありません」と答える
3. 回答の最後に参照した文書名を記載する
4. 簡潔で分かりやすい日本語で回答する
5. 不確実な情報は推測せず、確認を促す
4

テストと調整

右側のプレビュー画面で実際に質問を投げてみて、期待通りの回答が得られるか確認します。 必要に応じてトップKやプロンプトを調整します。

8.3 検索設定の最適化

RAGアプリの精度を左右する重要な設定パラメータについて理解しましょう。

パラメータ 説明 推奨値
トップK 検索結果から取得するチャンク数 3〜5（多すぎると無関係な情報が混入）
スコア閾値 この値未満の類似度スコアのチャンクを除外 0.5〜0.7（高いほど厳格）
リランク 検索結果を再度スコアリングして順位を最適化 有効（精度向上、ややコスト増）
検索モード ベクトル検索、キーワード検索、ハイブリッド検索 ハイブリッド検索（精度と再現性のバランス）

ハイブリッド検索とは

ベクトル検索とキーワード検索の両方を組み合わせた検索手法です。

  • ベクトル検索: 意味的に類似した内容を見つける（「料金」と「価格」を同一視）
  • キーワード検索: 完全一致や部分一致で検索（固有名詞や型番に強い）
  • ハイブリッド: 両方の利点を組み合わせて、より高精度な検索を実現

8.4 回答品質の向上テクニック

RAGアプリの回答精度を高めるための実践的なテクニックを紹介します。

引用元の明示を指示

プロンプトで「回答の最後に参照した文書名とページ番号を記載してください」と指示することで、 ユーザーが情報源を確認でき、信頼性が向上します。 

回答の最後に以下の形式で出典を記載してください：
【出典】文書名 - ページX

情報がない場合の対応を明確化

ナレッジベースに情報がない場合、LLMが一般知識で勝手に回答しないよう、明示的に指示します。 

ナレッジベースに関連情報がない場合は、
「申し訳ございません。その件に関する情報は社内文書に見つかりませんでした。
担当者に直接お問い合わせください。」と回答してください。

質問の言い換えテクニック

ユーザーの質問をそのまま検索するのではなく、LLMに質問を言い換えさせてから検索することで精度が向上します。 これはワークフローで実装可能です（第9章で詳しく学びます）。

検索"] C --> D["回答生成"]

マルチターン対話の対応

会話履歴を参照できるようにすることで、「それは何ですか？」のような代名詞を含む質問に対応できます。 Difyのチャットアプリでは、会話履歴機能を有効にすることで実現できます。

8.5 RAG特有のエラーと対策

RAGアプリでよく発生する問題とその対策を理解しておきましょう。

よくある問題と対策

問題: 関連する情報があるはずなのに「情報がありません」と回答される

原因: スコア閾値が高すぎる、または質問とドキュメントの表現が大きく異なる

対策: スコア閾値を下げる（0.7 → 0.5）、ハイブリッド検索を有効化、ドキュメントに複数の表現を含める

問題: 無関係な情報を参照して間違った回答をする

原因: トップKが大きすぎて、無関係なチャンクが混入している

対策: トップKを3以下に減らす、スコア閾値を上げる、リランクを有効化

問題: 複数ページにまたがる情報を統合できない

原因: チャンクサイズが小さすぎて、情報が分断されている

対策: チャンクサイズを大きくする（1000トークン → 1500トークン）、 プロンプトで「複数の情報を統合して回答してください」と指示

問題: 古い情報を参照してしまう

原因: ナレッジベースが更新されていない

対策: 定期的にドキュメントを再アップロード、古いバージョンを削除、 メタデータに更新日を含める

実習: 社内FAQ対応チャットボットを作る

第7章で作成したナレッジベースを使って、実用的なFAQチャットボットを構築しましょう。

  1. 1. 新しいチャットアプリを作成し、「社内FAQ対応ボット」と名付けます
  2. 2. ナレッジベースを追加し、トップK=3、スコア閾値=0.5、ハイブリッド検索に設定
  3. 3. システムプロンプトを設定：「ナレッジベースの情報のみを使用」「出典を明示」「情報がない場合の対応」
  4. 4. テスト質問を投げて、回答品質を確認：「休暇申請の方法は？」「経費精算の期限は？」など
  5. 5. 期待通りの回答が得られない場合、パラメータを調整して再テスト

理解度確認クイズ

  1. Q1. トップKとは何を意味するパラメータですか？ また、どのように設定すべきですか？

    回答を表示

    トップKは、検索結果から取得するチャンク数を指定します。 多すぎると無関係な情報が混入し、少なすぎると必要な情報を見逃す可能性があります。 一般的には3〜5が推奨され、FAQのように情報が限定されている場合は3以下が適切です。

  2. Q2. ハイブリッド検索のメリットは何ですか？

    回答を表示

    ハイブリッド検索は、ベクトル検索（意味的類似性）とキーワード検索（完全一致）の両方を組み合わせることで、 より高精度な検索を実現します。固有名詞や型番など正確な一致が必要な場合と、 概念的な質問の両方に対応できる柔軟性があります。

  3. Q3. ナレッジベースに情報があるのに「情報がありません」と回答される場合、どのような対策がありますか？

    回答を表示

    主な対策は以下の通りです： (1) スコア閾値を下げて検索条件を緩和する（0.7 → 0.5など）、 (2) ハイブリッド検索を有効化して検索方法を多様化する、 (3) ドキュメント内に複数の表現や類義語を含めて検索されやすくする、 (4) リランク機能を有効化して検索順位を最適化する。

第8章のまとめ

  • RAGは、ナレッジベースを参照しながらLLMが回答を生成する技術で、独自文書に基づくAIアプリを構築できる
  • トップK、スコア閾値、リランク、検索モード（ハイブリッド）の設定が回答精度に大きく影響する
  • プロンプトで引用元の明示、情報がない場合の対応、質問の言い換えを指示することで回答品質が向上する
  • RAG特有のエラー（情報が見つからない、無関係な情報を参照、古い情報など）への対策を理解することが重要
