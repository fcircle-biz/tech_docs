実践プロジェクト2 - ドキュメント要約ワークフロー

ワークフロー機能を活用して、長文ドキュメントの自動要約システムを構築し、複数ステップの処理設計とエラーハンドリングを実践します。

この章で学ぶこと

  • ワークフローを使った複数段階の文書処理の設計方法
  • 長文ドキュメントのチャンク分割と階層的要約の実装
  • エラーハンドリングとリトライロジックの設計
  • 出力フォーマットの制御と品質検証

17.1 プロジェクトの概要

このプロジェクトでは、長文のドキュメント（レポート、論文、契約書など）を自動的に要約するワークフローを構築します。 単なる要約だけでなく、重要なキーワード抽出、構造化された要約出力、品質検証までを一連のワークフローで自動化します。

実装する機能

  • 文書の自動チャンク分割: 長文ドキュメントを処理可能なサイズに分割
  • 階層的要約: 各チャンクの要約 → 全体の統合要約
  • キーワード抽出: 文書から重要な用語・トピックを自動抽出
  • 構造化出力: JSON形式で要約、キーワード、メタ情報を整理
  • エラーハンドリング: 処理失敗時のリトライとエラー報告

ワークフロー全体構成

このワークフローは、条件分岐、ループ処理、エラーハンドリングといった 実務で必要となる高度なワークフロー設計パターンを網羅しています。

17.2 ワークフローの設計

まず、Difyでワークフローを設計します。ビジュアルエディタで各ステップを配置していきます。

実習: ワークフローの作成

ステップ1: 新規ワークフロー作成

  1. Dify管理画面で「スタジオ」→「ワークフローを作成」をクリック
  2. アプリ名を「ドキュメント要約AI」に設定
  3. アプリタイプ「ワークフロー」を選択
  4. 説明欄に「長文ドキュメントの自動要約と分析」と記入

ステップ2: 入力変数の定義

  1. 「開始」ノードをクリックして設定を開く
  2. 入力変数を追加: document_text（型: テキスト）
  3. 説明に「要約対象のドキュメント本文」と記入
  4. 必須フィールドにチェック

主要ノードの配置

以下のノードを順番に配置し、ワークフローを構築していきます。

ノード名 種類 役割
文字数チェック コードノード 文書の長さを判定
長文分岐 条件分岐 長文か短文かで処理を分岐
チャンク分割 コードノード 文書を分割
チャンク要約ループ 反復ノード 各チャンクを順次処理
部分要約生成 LLMノード 各チャンクを要約
統合要約生成 LLMノード 部分要約を統合
キーワード抽出 LLMノード 重要語句を抽出
構造化出力 コードノード JSON形式で整形

17.3 各ノードの実装

それぞれのノードの具体的な設定内容を見ていきます。

1. 文字数チェック（コードノード）

入力ドキュメントの文字数を計測し、5000文字を超えるかどうかを判定します。

文字数チェック（Python） 
def main(document_text: str) -> dict:
    """
    文書の文字数をチェックし、分割が必要かを判定
    """
    char_count = len(document_text)

    return {
        "char_count": char_count,
        "needs_split": char_count > 5000,
        "document": document_text
    }

2. 長文分岐（条件分岐ノード）

前ノードの判定結果に基づき、処理を分岐します。

条件設定

  • IF条件: 文字数チェック.needs_split が true → 「チャンク分割」へ
  • ELSE条件: それ以外 → 「直接要約」へ

3. チャンク分割（コードノード）

長文を適切なサイズに分割します。文の途中で切れないよう、句読点を考慮します。

チャンク分割（Python） 
def main(document: str) -> dict:
    """
    文書を適切なサイズのチャンクに分割
    """
    chunk_size = 2000  # 1チャンク約2000文字
    chunks = []
    current_pos = 0

    while current_pos < len(document):
        # チャンク終了位置を計算
        end_pos = current_pos + chunk_size

        # 文の途中で切れないよう、句読点を探す
        if end_pos < len(document):
            # 句読点を後方探索
            search_text = document[end_pos-100:end_pos+100]
            for delimiter in ['。', '.\n', '！', '？']:
                idx = search_text.rfind(delimiter)
                if idx != -1:
                    end_pos = end_pos - 100 + idx + 1
                    break

        chunks.append(document[current_pos:end_pos])
        current_pos = end_pos

    return {
        "chunks": chunks,
        "chunk_count": len(chunks)
    }

4. チャンク要約ループ（反復ノード）

分割されたチャンクの配列を順番に処理します。

反復設定

  • 対象配列: チャンク分割.chunks
  • 繰り返し変数名: chunk_text
  • 最大反復数: 50回（デフォルト）

5. 部分要約生成（LLMノード）

ループ内で各チャンクを要約します。

LLM設定

  • モデル: GPT-4o mini（コスト削減）
  • Temperature: 0.3（安定した要約）
  • Max Tokens: 500
部分要約プロンプト 
以下のテキストを簡潔に要約してください。
重要なポイントを3〜5つの箇条書きで抽出してください。

【テキスト】
{{chunk_text}}

【要約の注意点】
- 主要な内容を漏らさない
- 具体的な数値や固有名詞は保持する
- 箇条書き形式で出力する

6. 統合要約生成（LLMノード）

ループで生成された部分要約をすべて統合し、全体の要約を作成します。

LLM設定

  • モデル: GPT-4o（高品質な統合のため）
  • Temperature: 0.5
  • Max Tokens: 1000

これらを統合して、全体の一貫した要約を作成してください。

【部分要約リスト】

【要約要件】
1. 文書全体の主旨を200〜300文字で要約
2. 重要なポイントを5〜7つの箇条書きで列挙
3. 論理的な流れを保つ

7. キーワード抽出（LLMノード）

文書から重要なキーワードやトピックを抽出します。

キーワード抽出プロンプト

【要約】

【抽出条件】
- 文書の中核となる概念や用語
- 固有名詞（人名、組織名、製品名など）
- 技術用語や専門用語
- カンマ区切りで出力

17.4 構造化出力とエラーハンドリング

最後に、要約結果をJSON形式で整形し、エラーハンドリングを実装します。

8. 構造化出力（コードノード）

要約とキーワードをJSON形式で整形します。

構造化出力（Python） 
import json
from datetime import datetime

def main(summary: str, keywords: str, char_count: int) -> dict:
    """
    要約結果を構造化されたJSON形式で出力
    """
    # キーワードを配列に変換
    keyword_list = [kw.strip() for kw in keywords.split(',')]

    # 構造化データを作成
    result = {
        "summary": summary,
        "keywords": keyword_list,
        "metadata": {
            "original_length": char_count,
            "summary_length": len(summary),
            "keyword_count": len(keyword_list),
            "created_at": datetime.now().isoformat()
        }
    }

    return {
        "json_output": json.dumps(result, ensure_ascii=False, indent=2),
        "result": result
    }

エラーハンドリングの実装

LLMノードやコードノードにはエラー時の動作を設定します。

各ノードのエラー設定

  • LLMノード: リトライ回数3回、リトライ間隔5秒
  • コードノード: エラー時はエラーメッセージを変数に格納して次へ
  • 最終ノード: エラーがある場合は専用のエラーレスポンスを返す

エラー報告ノード（コードノード）

エラー報告（Python） 
def main(error_message: str = None) -> dict:
    """
    エラーが発生した場合のレスポンスを生成
    """
    if error_message:
        return {
            "status": "error",
            "message": "要約処理中にエラーが発生しました",
            "error_detail": error_message
        }
    else:
        return {
            "status": "success",
            "message": "要約が正常に完了しました"
        }

17.6 ワークフローの最適化

基本的なワークフローが動作したら、さらに以下のような最適化を検討します。

処理速度の改善

  • • GPT-4o miniを優先的に使用
  • • 並列処理が可能な箇所を特定
  • • キャッシュ機能の活用

コスト削減

  • • 部分要約は小モデルを使用
  • • トークン数制限の最適化
  • • 不要な再生成を削減

品質向上

  • • プロンプトのA/Bテスト
  • • 要約品質の自動評価ノード追加
  • • ユーザーフィードバック機能

機能拡張

  • • 複数言語への翻訳機能追加
  • • メール送信やSlack通知
  • • PDF出力機能の実装

パフォーマンス監視のヒント

Difyの「ログとアノテーション」機能で、各ノードの実行時間やトークン消費量を確認できます。 ボトルネックとなっているノードを特定し、優先的に最適化しましょう。

17.7 実務での活用シーン

このドキュメント要約ワークフローは、以下のような実務シーンで活用できます。

会議議事録の自動要約

長時間の会議の文字起こしテキストを入力し、主要な決定事項やアクションアイテムを自動抽出。 関係者への共有資料として活用できます。

契約書・規約の要約

長大な契約書や利用規約を要約し、重要な条項やリスク事項をキーワードとして抽出。 法務チェックの事前準備に活用できます。

技術ドキュメントの概要作成

新しい技術の公式ドキュメントや論文を要約し、チーム内での知識共有を促進。 学習コストを削減できます。

ニュース記事の自動要約配信

業界ニュースや市場レポートを定期的に要約し、重要なトピックだけをメールやSlackで通知。 情報収集の効率化を実現できます。

理解度確認クイズ

  1. Q1. 長文ドキュメントをチャンク分割する際、なぜ句読点を考慮する必要がありますか？

    回答を表示

    文の途中で切れると、文脈が失われ、各チャンクの要約品質が低下するためです。句読点で区切ることで、各チャンクが意味的に完結したテキストになります。

  2. Q2. 部分要約と統合要約でモデルを変えるメリットは何ですか？

    回答を表示

    部分要約は量が多いため小モデル（GPT-4o mini）でコストを抑え、統合要約は高品質が求められるため大モデル（GPT-4o）を使うことで、コストと品質のバランスを最適化できます。

  3. Q3. ワークフローにエラーハンドリングを実装する理由は何ですか？

    回答を表示

    LLM APIの一時的な障害やレート制限、予期しない入力形式などでワークフローが途中で停止しないようにするためです。 エラーハンドリングとリトライ機能により、実用的なシステムとして安定稼働が可能になります。
