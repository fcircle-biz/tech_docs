第10章

ワークフローノードの活用 - 多様な処理を組み合わせる

LLMノード、条件分岐、ループ、HTTPリクエスト、コードノードなど、各種ノードの使い方と実践的なワークフロー設計パターンを学びます。

この章で学ぶこと

  • LLMノード、条件分岐、ループなどワークフローで使用できる主要ノードの種類と役割
  • HTTPリクエストノードで外部APIと連携する方法
  • コードノード（Python/JavaScript）で高度なデータ処理を実装する方法
  • 実践的なワークフロー設計パターンと組み合わせ方のベストプラクティス

10.1 ワークフローノードの全体像

Difyのワークフローでは、様々なノードを組み合わせて処理を設計します。 各ノードは特定の機能を持ち、それらを連結することで複雑なAI処理フローを実現できます。

第9章で学んだように、ワークフローは「入力」→「処理」→「出力」という流れで設計されます。 この章では、「処理」部分で使用できる様々なノードの種類と活用方法を詳しく見ていきます。

主要なノードカテゴリ

  • 入出力ノード: Start（開始）、End（終了）、Answer（回答）
  • LLMノード: LLM（言語モデル呼び出し）、Knowledge Retrieval（ナレッジ検索）
  • 制御フローノード: IF/ELSE（条件分岐）、Loop（ループ）、Iteration（反復処理）
  • データ処理ノード: Variable Assigner（変数代入）、Template（テンプレート）、Code（コード実行）
  • 外部連携ノード: HTTP Request（HTTPリクエスト）、Tool（ツール呼び出し）

10.2 LLMノードの活用

LLMノードは、OpenAI、Claude、Geminiなどの大規模言語モデルを呼び出して自然言語処理を行うノードです。 ワークフロー内で最も重要なノードの一つで、テキスト生成、分類、要約、翻訳などに使用します。

LLMノードの主要設定項目

  • モデル選択: 使用するLLMモデル（GPT-4、Claude 3.5、Gemini等）
  • システムプロンプト: LLMの振る舞いを定義する指示
  • ユーザープロンプト: 処理対象のテキストや質問（変数を使用可能）
  • 温度（Temperature）: 出力のランダム性（0.0〜1.0）
  • 最大トークン数: 生成される応答の最大長
  • 出力変数: LLMの応答を格納する変数名

活用例：顧客問い合わせの分類

システムプロンプト例 
あなたは顧客問い合わせを分類するエキスパートです。
以下のカテゴリのいずれかに分類してください：

- 技術サポート: 製品の使い方や技術的な問題
- 請求・支払い: 料金や請求に関する質問
- アカウント管理: ログインやアカウント設定
- その他: 上記に該当しないもの

カテゴリ名のみを回答してください。
ユーザープロンプト例 
問い合わせ内容：{{inquiry_text}}

{{inquiry_text}}には前のノードから渡された問い合わせテキストが入ります。

10.3 条件分岐とループ

ワークフローに動的な処理を追加するために、条件分岐（IF/ELSE）とループ（Loop）は欠かせません。

IF/ELSE（条件分岐）ノード

変数の値や条件に基づいて、処理の流れを分岐させます。

条件設定例：

  • • {{category}} が「技術サポート」に等しい → 技術チーム対応
  • • {{sentiment}} が「ネガティブ」を含む → 優先対応フラグON
  • • {{confidence_score}} が0.8以上 → 自動回答
  • • {{user_tier}} が「Premium」に等しい → VIP対応

Loop（ループ）ノード

配列やリストの各要素に対して同じ処理を繰り返し実行します。

活用例：

  • • 複数のドキュメントに対して順次要約処理を実行
  • • 商品リストの各アイテムに対して説明文を生成
  • • メールアドレスリストの各宛先にパーソナライズメッセージを送信

10.4 HTTPリクエストノードで外部連携

HTTPリクエストノードを使用すると、外部のWeb APIを呼び出してデータを取得したり、他のシステムと連携したりできます。

HTTPリクエストノードの設定項目

  • メソッド: GET、POST、PUT、DELETE等
  • URL: APIエンドポイントのURL（変数使用可能）
  • ヘッダー: 認証トークンやContent-Type等
  • ボディ: POSTやPUTの場合にリクエストデータを指定
  • タイムアウト: リクエストのタイムアウト時間
  • 出力変数: レスポンスを格納する変数名

実習：天気情報取得ワークフロー

HTTPリクエストノードで天気APIを呼び出し、取得した情報をLLMで自然な文章に変換する例です。

ワークフロー構成：

  1. Start：都市名を入力 {{city}}
  2. HTTP Request：天気APIに都市名を送信 → {{weather_data}}
  3. LLM：JSONデータを自然な日本語に変換
  4. Answer：「東京は現在晴れで気温は22度です」と回答

HTTPリクエスト設定例

HTTPリクエスト設定（JSON） 
{
  "method": "GET",
  "url": "https://api.openweathermap.org/data/2.5/weather",
  "params": {
    "q": "{{city}}",
    "appid": "YOUR_API_KEY",
    "lang": "ja",
    "units": "metric"
  },
  "headers": {
    "Content-Type": "application/json"
  }
}

10.5 コードノードで高度な処理

コードノードは、PythonまたはJavaScriptのコードを直接実行できるノードです。 ビジュアルノードでは実現できない複雑なデータ処理や計算を行う際に使用します。

コードノードの主な用途

  • • JSONデータの複雑な変換・加工
  • • 数値計算やデータ分析
  • • 正規表現を使ったテキスト処理
  • • 配列やオブジェクトの高度な操作
  • • 日付計算や時刻処理

例：Pythonコードノード - テキストの統計分析

Python 
def main(text: str) -> dict:
    # 文字数、単語数、行数をカウント
    char_count = len(text)
    word_count = len(text.split())
    line_count = len(text.split('\n'))

    return {
        'char_count': char_count,
        'word_count': word_count,
        'line_count': line_count,
        'avg_word_length': char_count / word_count if word_count > 0 else 0
    }

入力: {{input_text}}
出力: {{stats}} に統計情報が格納されます

例：JavaScriptコードノード - 日付計算

JavaScript 
function main(days) {
    // N日後の日付を計算
    const today = new Date();
    const futureDate = new Date(today);
    futureDate.setDate(today.getDate() + parseInt(days));

    return {
        today: today.toISOString().split('T')[0],
        futureDate: futureDate.toISOString().split('T')[0],
        dayOfWeek: futureDate.toLocaleDateString('ja-JP', { weekday: 'long' })
    };
}

入力: {{days_ahead}}
出力: {{date_info}} に日付情報が格納されます

セキュリティとパフォーマンスの注意点

  • • コードノードの実行時間には制限があります（通常10秒程度）
  • • 外部ライブラリは限定的なもののみ使用可能
  • • ユーザー入力をそのままコードに組み込まないようにする（インジェクション対策）
  • • 大量データの処理にはループノードとの併用を検討

10.6 実践的なワークフロー設計パターン

ここまで学んだノードを組み合わせて、実践的なワークフローを設計するパターンを紹介します。

パターン1：マルチステップ処理パイプライン

複数のLLMノードを順次実行し、段階的にデータを加工するパターン

例：ブログ記事の自動生成
  1. LLM1: トピックから記事構成を生成
  2. LLM2: 構成に基づいて各セクションの本文を生成
  3. LLM3: タイトルとメタディスクリプションを生成
  4. テンプレート: HTML形式に整形
  5. Answer: 完成した記事を出力

パターン2：条件分岐による動的処理

入力や中間結果に応じて処理フローを動的に切り替えるパターン

例：問い合わせ自動対応システム
  1. LLM: 問い合わせをカテゴリ分類
  2. IF/ELSE: カテゴリに応じて分岐
  3. → 技術サポート: ナレッジベース検索
  4. → 請求関連: HTTPリクエストで顧客情報取得
  5. → その他: 標準テンプレート回答
  6. Answer: 適切な回答を返す

パターン3：外部APIとの統合処理

外部サービスのAPIを呼び出し、結果をLLMで処理するパターン

例：レストラン推薦システム
  1. LLM: ユーザーの希望を構造化データに変換
  2. HTTP Request: レストラン検索API呼び出し
  3. コードノード: 検索結果をランキング・フィルタリング
  4. Loop: 各レストランに対して詳細情報取得
  5. LLM: 取得情報を自然な文章に整形
  6. Answer: おすすめレストランを提示

10.7 その他の便利なノード

ここまで紹介した以外にも、Difyには便利なノードが多数用意されています。

ノード名 用途 活用例
Variable Assigner 変数に値を代入 カウンター、フラグ設定
Template テンプレート文字列の生成 メール本文、HTML生成
Parameter Extractor テキストから構造化データ抽出 問い合わせから日時・場所抽出

理解度確認クイズ

  1. Q1. LLMノードで設定できる「Temperature（温度）」パラメータは何を制御しますか？

    回答を表示

    出力のランダム性（創造性）を制御します。0.0に近いほど決定論的で一貫した出力になり、1.0に近いほど多様で創造的な出力になります。 事実を答える場合は低め（0.1〜0.3）、創作文を生成する場合は高め（0.7〜0.9）に設定します。

  2. Q2. HTTPリクエストノードで外部APIを呼び出す際に必須の設定項目は何ですか？

    回答を表示

    メソッド（GET、POST等）とURL（APIエンドポイント）が必須です。また、多くのAPIでは認証トークンをヘッダーに含める必要があります。 レスポンスを後続ノードで使用するために、出力変数名の設定も重要です。

  3. Q3. コードノードとLLMノードの使い分けはどのように考えればよいですか？

    回答を表示

    コードノード: 数値計算、データ変換、正規表現処理など、明確なロジックで処理できるもの
    LLMノード: 自然言語理解、文章生成、意味的な分類など、柔軟な理解や創造性が必要なもの
    両者を組み合わせることで、効率的かつ柔軟なワークフローを構築できます。

第10章のまとめ

  • Difyワークフローには、LLMノード、条件分岐、ループ、HTTPリクエスト、コードノードなど多様なノードが用意されている
  • LLMノードで自然言語処理、HTTPノードで外部API連携、コードノードで高度なデータ処理が可能
  • 条件分岐とループを使うことで、動的で柔軟なワークフローを構築できる
  • マルチステップ処理、条件分岐処理、外部API統合などの実践パターンを理解すれば、複雑な業務フローも自動化できる
