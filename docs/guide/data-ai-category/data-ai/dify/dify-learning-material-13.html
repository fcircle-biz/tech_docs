外部API連携 - 既存システムとつなげる

HTTPノードやカスタムAPIを使ったDifyアプリと外部システムの連携方法、MCP（Model Context Protocol）を使った標準化された連携を学びます。

この章で学ぶこと

  • HTTPノードを使った外部APIへのリクエスト送信方法
  • 認証（APIキー、OAuth）を含むAPI連携の実装
  • カスタムAPIツールの作成と利用方法
  • MCP（Model Context Protocol）を使った標準化連携

13.1 外部API連携の重要性

DifyのAIアプリケーションを実用的なビジネスシステムとして活用するには、外部システムとの連携が不可欠です。 社内データベースから情報を取得したり、外部サービスにデータを送信したり、既存の業務システムと統合することで、Difyの価値は大きく高まります。

典型的な連携シナリオ

  • 顧客管理システム連携: CRMから顧客情報を取得し、パーソナライズされた回答を生成
  • 在庫管理システム連携: リアルタイムの在庫状況を確認し、顧客からの問い合わせに回答
  • 通知システム連携: Slack、メール、SMSで処理結果を自動通知
  • 決済システム連携: 注文情報を取得し、請求書を自動生成
  • 外部データ取得: 天気予報API、為替レートAPI、ニュースAPIなどから最新情報を取得

Difyにおける外部連携の3つの方法

13.2 HTTPノードによるAPI連携

HTTPノードは、Difyワークフロー内で外部のREST APIにリクエストを送信し、レスポンスを受け取るためのノードです。 GET、POST、PUT、DELETEなどのHTTPメソッドをサポートし、ヘッダーやボディのカスタマイズが可能です。

HTTPノードの基本設定項目

  • • HTTPメソッド: GET、POST、PUT、DELETE、PATCHなど
  • • URL: リクエスト先のエンドポイント（変数を埋め込み可能）
  • • ヘッダー: Content-Type、Authorization、カスタムヘッダーを設定
  • • ボディ: JSONやフォームデータを送信（POSTやPUTの場合）
  • • 認証: APIキー、Bearerトークン、Basic認証などを設定

実習: OpenWeatherMap APIで天気情報を取得する

HTTPノードを使って、OpenWeatherMap APIから指定した都市の天気情報を取得し、自然な日本語で回答するワークフローを作成します。

  1. 1. OpenWeatherMap APIキーの取得
    • • OpenWeatherMapに無料登録し、APIキーを取得
  2. 2. ワークフローの作成
    • • 新規ワークフローを作成
    • • Startノードで「都市名」を入力変数として定義（例: Tokyo）
  3. 3. HTTPノードの設定
    • • HTTPメソッド: GET
    • • URL: https://api.openweathermap.org/data/2.5/weather?q={{'{'}}{'{'}city{'}'}}{{'}'}}＆appid=YOUR_API_KEY＆units=metric＆lang=ja
    • • {{'{'}}{'{'}city{'}'}}{{'}'}}) にStartノードの都市名変数を接続
    • • レスポンスをJSON形式で受け取る
  4. 4. LLMノードで自然言語化
    • • HTTPノードのレスポンスを入力として受け取る
    • • プロンプト例: 「以下の天気情報を日本語で分かりやすく説明してください: {{'{'}}{'{'}http_response{'}'}}{{'}'}}」
  5. 5. テスト実行
    • • 都市名に「Tokyo」と入力してワークフローを実行
    • • 「東京の天気は晴れで、気温は25度です」のような自然な回答が返ってくるか確認

HTTPレスポンスの処理

HTTPノードのレスポンスはJSON形式で返されます。レスポンスデータから必要な情報を抽出するには、Jinja2テンプレート記法を使います。

JSONレスポンス例 
{
  "name": "Tokyo",
  "weather": [
    {
      "description": "晴天"
    }
  ],
  "main": {
    "temp": 25.5,
    "humidity": 60
  }
}
Jinja2テンプレートでのデータ抽出 
# 都市名を取得
{{'{'}}{'{'}http_response.name{'}'}}{{'}'}}

# 天気の説明を取得
{{'{'}}{'{'}http_response.weather[0].description{'}'}}{{'}'}}

# 気温を取得
{{'{'}}{'{'}http_response.main.temp{'}'}}{{'}'}}度

13.3 認証を含むAPI連携

実際のビジネスAPIでは、セキュリティのために認証が必要です。 Difyでは、APIキー、Bearerトークン、Basic認証など、一般的な認証方式をサポートしています。

認証方式 設定方法 利用例
APIキー（URL） URLパラメータに追加
?api_key=YOUR_KEY		 OpenWeatherMap、Google Maps API
APIキー（ヘッダー） ヘッダーに追加
X-API-Key: YOUR_KEY		 Stripe、SendGrid
Bearerトークン ヘッダーに追加
Authorization: Bearer YOUR_TOKEN		 GitHub API、Twitter API
Basic認証 ヘッダーに追加
Authorization: Basic (Base64エンコード)		 社内API、レガシーシステム

実習: GitHub APIでリポジトリ情報を取得

Bearerトークン認証を使って、GitHub APIからリポジトリの情報を取得します。

  1. 1. GitHub Personal Access Tokenの取得
    • • GitHubの Settings → Developer settings → Personal access tokens → Generate new token
    • • repo スコープを選択してトークンを生成
  2. 2. HTTPノードの設定
    • • HTTPメソッド: GET
    • • URL: https://api.github.com/repos/{{'{'}}{'{'}owner{'}'}}{{'}'}}全角／{{'{'}}{'{'}repo{'}'}}{{'}'}})
    • • ヘッダーに追加: Authorization: Bearer YOUR_TOKEN
  3. 3. レスポンス例
    • • リポジトリ名、説明、スター数、フォーク数などが取得できます
    • • LLMノードで「このリポジトリについて説明してください」と組み合わせて自然な回答を生成

APIキー・トークンの管理に注意

APIキーやトークンは機密情報です。Difyの環境変数機能を使って安全に管理し、ワークフロー内に直接記述しないようにしましょう。 また、不要になったトークンは必ず無効化してください。

13.4 カスタムAPIツールの作成

HTTPノードはワークフロー内で使用しますが、カスタムAPIツールとして登録すると、エージェントが自律的に呼び出せるようになります。 OpenAPI（Swagger）仕様に基づいてツールを定義することで、より柔軟なAPI連携が可能になります。

カスタムAPIツールのメリット

  • • エージェントが必要に応じて自動的にAPIを呼び出せる
  • • 複数のワークフローやアプリで再利用できる
  • • OpenAPI仕様で定義するため、ドキュメントとして機能する
  • • パラメータのバリデーションが自動で行われる

カスタムAPIツールの作成手順

  1. 1. OpenAPI仕様の準備

    APIのエンドポイント、パラメータ、レスポンスをOpenAPI 3.0形式で記述します。

  2. 2. Difyにツールを登録

    「ツール」メニューから「カスタムAPIツールを追加」を選択し、OpenAPI仕様をインポートします。

  3. 3. 認証情報の設定

    APIキーやトークンを環境変数として設定します。

  4. 4. エージェントで利用

    エージェントアプリの「ツール」設定で、作成したカスタムツールを選択します。

OpenAPI 3.0仕様例（簡易版） 
openapi: 3.0.0
info:
  title: 在庫確認API
  version: 1.0.0
servers:
  - url: https://api.example.com/v1
paths:
  /inventory/{productId}:
    get:
      summary: 商品の在庫数を取得
      parameters:
        - name: productId
          in: path
          required: true
          schema:
            type: string
      responses:
        '200':
          description: 成功
          content:
            application/json:
              schema:
                type: object
                properties:
                  productId:
                    type: string
                  stock:
                    type: integer

13.5 MCP（Model Context Protocol）を使った連携

MCP（Model Context Protocol）は、Anthropic社が提唱する、LLMとツールを標準化された方法で接続するためのオープンプロトコルです。 Difyも将来的にMCPのサポートを強化する予定で、現在はClaudeとの連携で利用できます。

MCPの特徴

  • 標準化: プロバイダー間で共通のインターフェース
  • セキュリティ: 安全なコンテキスト共有の仕組み
  • 拡張性: 新しいツールを簡単に追加可能
  • 相互運用性: 異なるLLMプラットフォーム間でのツール再利用

MCPの構造

（Dify等）"] --> B["MCP Client"] B <--> C["MCP Protocol"] C <--> D["MCP Server"] D --> E["ファイルシステム"] D --> F["データベース"] D --> G["外部API"] D --> H["その他のツール"]

DifyでのMCP利用（参考）

現在、DifyでMCPを直接利用するには、カスタム実装が必要ですが、将来的にはGUIから簡単に設定できるようになる見込みです。 MCPサーバーを立てて、Difyからアクセスする形になります。

MCPの詳細情報

MCPについて詳しく知りたい場合は、MCP公式サイトや Anthropic公式ドキュメントを参照してください。

13.6 API連携のベストプラクティス

外部API連携を実装する際には、以下のベストプラクティスを意識しましょう。

セキュリティ

  • • APIキーは環境変数で管理
  • • HTTPS接続を使用
  • • 最小権限の原則を適用
  • • トークンの定期的な更新

エラーハンドリング

  • • タイムアウト設定を適切に行う
  • • リトライロジックを実装
  • • エラーレスポンスを適切に処理
  • • ユーザーへのフィードバックを提供

パフォーマンス

  • • レスポンスのキャッシュを検討
  • • レート制限を遵守
  • • 不要なデータは取得しない
  • • 並列処理を活用

ドキュメント化

  • • API仕様を明確に記録
  • • パラメータの説明を記載
  • • レスポンス例を記録
  • • 変更履歴を管理

トラブルシューティング

問題: 401 Unauthorized エラー

原因: 認証情報が正しくないか、期限切れ
対処法: APIキーやトークンが正しいか確認、有効期限をチェック、ヘッダーの記述方法を確認

問題: 429 Too Many Requests エラー

原因: APIのレート制限に達した
対処法: リクエスト頻度を下げる、リトライ間隔を設ける、キャッシュを活用

問題: レスポンスデータが取得できない

原因: JSONパスの指定が間違っている
対処法: HTTPノードのレスポンスをログ出力し、正しいJSONパスを確認

理解度確認クイズ

  1. Q1. HTTPノードとカスタムAPIツールの違いは何ですか?

    回答を表示

    HTTPノードはワークフロー内で使用する単発のAPI呼び出しです。カスタムAPIツールはOpenAPI仕様で定義し、エージェントが自律的に呼び出せる再利用可能なツールです。 カスタムAPIツールは複数のワークフローで共有でき、パラメータのバリデーションも自動で行われます。

  2. Q2. Bearerトークン認証を設定する場合、どのヘッダーにどのような値を設定しますか?

    回答を表示

    Authorization ヘッダーに Bearer YOUR_TOKEN という形式で設定します。 例: Authorization: Bearer ghp_1234567890abcdefghij

  3. Q3. MCP（Model Context Protocol）の主なメリットは何ですか?

    回答を表示

    MCPは標準化されたプロトコルなので、異なるLLMプラットフォーム間でツールを再利用できます。 また、セキュアなコンテキスト共有の仕組みを持ち、新しいツールの追加が容易です。プロバイダー間の相互運用性が高まります。

第13章のまとめ

  • HTTPノードを使って、外部APIにGET/POST/PUT/DELETEリクエストを送信できる
  • APIキー、Bearerトークン、Basic認証など、様々な認証方式をサポートしている
  • カスタムAPIツールをOpenAPI仕様で定義すると、エージェントが自律的に呼び出せる
  • MCP（Model Context Protocol）は標準化されたLLMとツールの連携プロトコルである
  • APIキーの管理、エラーハンドリング、パフォーマンス最適化を意識した実装が重要
