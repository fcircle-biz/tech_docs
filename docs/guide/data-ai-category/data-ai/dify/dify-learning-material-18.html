第18章 総合演習

総合演習 - 業務効率化AIアシスタントの開発

学んだ全ての技術を活用して、RAG、ワークフロー、エージェント機能を組み合わせた総合的なAIアシスタントを開発します。要件定義から設計、実装、公開までの一連の流れを体験します。

この章で学ぶこと

  • AIアプリケーション開発の全体プロセス（要件定義から公開まで）
  • RAG、ワークフロー、エージェント機能を統合した複合システムの設計
  • 業務シナリオに応じた機能選択とアーキテクチャ設計の実践
  • 運用を見据えたテスト、評価、継続的改善の方法論

18.1 プロジェクト概要：業務効率化AIアシスタント

この総合演習では、企業の業務部門を支援する多機能AIアシスタントを開発します。 RAG、ワークフロー、エージェント機能を組み合わせることで、単一のチャットボットでは実現できない高度な業務支援を実現します。

プロジェクトの特徴

  • RAG: 社内規定、マニュアル、FAQ文書を参照して回答
  • ワークフロー: 文書要約、翻訳、レポート生成などの定型業務を自動化
  • エージェント: ユーザーの質問内容に応じてWeb検索、計算、データベース検索を自律実行
  • 外部連携: 既存業務システムとAPIで連携し、データ取得や更新を実現

18.2 フェーズ1：要件定義と設計

実際の業務改善プロジェクトと同様に、まず要件を整理し、システム設計を行います。

要件定義

機能要件

  • • 社内規定・マニュアルに関する質問応答（RAG）
  • • 長文ドキュメントの要約・翻訳（ワークフロー）
  • • 最新情報のWeb検索と報告（エージェント）
  • • 業務データベースへの検索・集計（API連携）
  • • 多言語対応（日本語・英語）

非機能要件

  • • 回答速度: 5秒以内（RAG検索）
  • • 回答精度: 関連文書のTop-K適合率80%以上
  • • 利用可能時間: 24時間365日
  • • セキュリティ: アクセス制限、ログ記録

設計方針

アーキテクチャ

ルーターパターン: 質問タイプを分類し、適切なサブシステム（RAG/ワークフロー/エージェント）に振り分ける

LLM選定

  • • ルーター: GPT-4o-mini（高速・低コスト）
  • • RAG回答生成: GPT-4o（高精度）
  • • ワークフロー: Claude 3.5 Sonnet（長文処理に強い）
  • • エージェント: GPT-4o（Function Calling精度が高い）

ナレッジベース構成

社内規定、業務マニュアル、FAQを別々のコレクションとして管理。検索時は複数コレクションをハイブリッド検索。

設計書サンプル（YAML形式） 

# 業務効率化AIアシスタント 設計書
project:
  name: "業務効率化AIアシスタント"
  version: "1.0"

architecture:
  type: "Router-based Multi-System"
  components:
    - name: "Router"
      llm: "gpt-4o-mini"
      function: "質問分類"
    - name: "RAG"
      llm: "gpt-4o"
      knowledge_base: ["社内規定", "マニュアル", "FAQ"]
    - name: "Workflow"
      llm: "claude-3-5-sonnet"
      workflows: ["要約", "翻訳", "レポート生成"]
    - name: "Agent"
      llm: "gpt-4o"
      tools: ["Web検索", "計算", "API連携"]

performance:
  response_time: "5秒以内"
  accuracy: "80%以上"
  availability: "99.9%"

設計のポイント

各サブシステムで異なるLLMを使い分けることで、コストと性能のバランスを最適化できます。 例えば、ルーター処理は単純な分類なのでGPT-4o-miniで十分ですが、RAG回答生成は精度が重要なのでGPT-4oを使用します。

18.3 フェーズ2：実装ステップ

設計に基づいて、Difyで実際にシステムを構築していきます。段階的に機能を追加していきましょう。

ステップ1：ナレッジベースの構築

1-1. ナレッジベースを作成

  • • Difyダッシュボードで「ナレッジベース」→「作成」
  • • 名前: 「社内規定」「業務マニュアル」「FAQ」の3つを作成

1-2. ドキュメントをアップロード

  • • 各ナレッジベースに対応するPDF、Word、テキストファイルをアップロード
  • • チャンク設定: 自動（推奨500トークン、オーバーラップ50）
  • • インデックス方式: ハイブリッド検索（キーワード + ベクトル）

1-3. 検索テスト

各ナレッジベースで「検索テスト」を実行し、適切な文書が取得できるか確認

ステップ2：ルーターワークフローの作成

2-1. 新規チャットフローを作成

  • • アプリタイプ: 「チャットフロー」を選択
  • • 名前: 「業務効率化AIアシスタント」

2-2. ルーターノードを設置

  • • LLMノード（GPT-4o-mini）を追加
  • • システムプロンプトで質問タイプを分類（RAG/ワークフロー/エージェント/その他）
  • • 出力をJSON形式で指定（type, reason, context）

2-3. 条件分岐ノードを追加

ルーターの出力（type）に応じて、適切なサブワークフローに分岐

ルーターワークフロー構造

ステップ3：RAGサブフローの構築

3-1. ナレッジベース検索ノード

  • • ナレッジベース検索ノードを追加
  • • 複数コレクション（社内規定、マニュアル、FAQ）を選択
  • • Top-K: 5、スコア閾値: 0.7

3-2. 回答生成LLM

  • • LLMノード（GPT-4o）を追加
  • • コンテキスト: 検索結果の文書
  • • プロンプト: 「検索結果に基づいて質問に回答してください。情報が不十分な場合はその旨を伝えてください。」

ステップ4：ワークフローサブフローの構築

4-1. 処理タイプ判定

LLMノードで要約/翻訳/レポート生成のいずれかを判定

4-2. 各処理のLLMノード

  • • 要約: Claude 3.5 Sonnet（長文に強い）
  • • 翻訳: GPT-4o（多言語対応）
  • • レポート生成: Claude 3.5 Sonnet（構造化文書作成に強い）

ステップ5：エージェントサブフローの構築

5-1. エージェントLLM設定

  • • LLMノード（GPT-4o）をエージェントモードで追加
  • • 推論方式: Function Calling

5-2. ツールの追加

  • • Web検索ツール（Google Search API）
  • • 計算ツール（WolframAlpha）
  • • カスタムAPIツール（業務データベース検索）

5-3. エージェント指示

「あなたは業務サポートエージェントです。必要に応じてツールを使い、正確な情報を提供してください。」

18.4 フェーズ3：テストと評価

構築したシステムが要件を満たしているか、様々な観点からテストを実施します。

テスト項目

機能テスト

  • • RAG: 「経費精算のルールは？」→ 社内規定から正確に回答
  • • ワークフロー: 長文ドキュメントを添付して「要約して」→ 要約が生成される
  • • エージェント: 「2025年の為替レートトレンドは？」→ Web検索して回答
  • • 複合: 「最新の営業データを検索して、レポートにまとめて」

性能テスト

  • • 回答時間を計測（目標: 5秒以内）
  • • 複数ユーザーの同時アクセステスト
  • • RAG検索精度（Top-5適合率を確認）

エラーハンドリングテスト

  • • ナレッジベースに情報がない質問への対応
  • • 外部API接続エラー時の挙動
  • • 不適切な入力（空文字、極端に長い文章）への対応

RAG回答精度 80%以上 Top-5適合率 （測定値を記録）
回答時間 5秒以内 平均レスポンスタイム （測定値を記録）
ルーター精度 95%以上 正しいフローへの振り分け率 （測定値を記録）

動作確認：テストシナリオ実行

シナリオ1：RAG質問応答

入力: 「経費精算の申請期限は何日までですか？」

期待出力: 社内規定から正確な期限を回答

シナリオ2：文書要約

入力: （長文レポートを添付）「このレポートを3つのポイントで要約してください」

期待出力: 3つのポイントで簡潔に要約された内容

シナリオ3：Web検索エージェント

入力: 「2025年の日本のGDP成長率予測を調べてください」

期待出力: Web検索で最新情報を取得し、出典とともに回答

18.5 フェーズ4：公開と運用

テストが完了したら、アプリを公開し、運用フェーズに移行します。

公開設定

公開方法の選択

  • • Webアプリ公開: Difyホスティングで社内URLを発行
  • • API提供: 既存社内システムから呼び出し可能に
  • • iframe埋め込み: イントラネットポータルにチャットウィジェットを配置

アクセス制御

  • • SSO（Single Sign-On）による認証設定
  • • 部署ごとのアクセス権限設定
  • • APIキーによるアクセス制限（API公開時）

利用ガイドの作成

ユーザー向けのマニュアル、サンプル質問集、FAQ、トラブルシューティングガイドを用意

運用体制

モニタリング

  • • Difyログで利用状況を週次確認
  • • エラーログ、低評価回答の分析
  • • コスト（LLM APIトークン使用量）のモニタリング

フィードバック収集

  • • 回答ごとの評価ボタン（Good/Bad）
  • • ユーザーアンケート実施（月次）
  • • 問い合わせ窓口の設置

継続的改善

  • • ナレッジベースの定期更新（新規文書追加、古い文書削除）
  • • プロンプトのチューニング（低評価回答の原因分析）
  • • RAG検索パラメータの最適化（Top-K、スコア閾値）
  • • 新機能の追加検討（ユーザー要望に基づく）

運用成功のポイント

AIアプリは「作って終わり」ではありません。ユーザーフィードバックを収集し、継続的に改善していくことで、 初期の精度70%が数ヶ月で90%以上に向上するケースも珍しくありません。 特にナレッジベースの更新とプロンプトのチューニングは、低コストで大きな効果を生みます。

18.6 発展課題：さらなる高度化

基本的な業務効率化AIアシスタントが完成したら、以下のような発展的な機能を追加してみましょう。

マルチモーダル対応

画像認識機能を追加し、写真から文字を抽出したり、図表を解析して説明する機能を実装。

ヒント: GPT-4 Visionをマルチモーダルノードで使用

会話履歴の活用

過去の会話内容を記憶し、文脈を理解した回答を生成。ユーザーごとにパーソナライズされた対応を実現。

ヒント: 変数ノードで会話履歴を保持、コンテキストに含める

多言語自動切り替え

ユーザーの入力言語を自動検出し、同じ言語で回答するマルチリンガル対応。

ヒント: 言語判定LLMノード → 条件分岐で言語別プロンプト選択

情報セキュリティ強化

機密情報検出フィルター、出力内容のマスキング、アクセスログの暗号化など、セキュリティ機能を強化。

ヒント: コードノードで正規表現による機密情報検出

発展課題に取り組む前に

基本システムを安定稼働させてから発展課題に取り組みましょう。 まずは運用で得られたフィードバックを元に優先度を決め、ユーザーにとって最も価値の高い機能から実装していくことをお勧めします。

理解度確認クイズ

  1. Q1. ルーターパターンを使う主な目的は何ですか？

    回答を表示

    ユーザーの質問内容に応じて、適切なサブシステム（RAG、ワークフロー、エージェント）に振り分けるためです。 これにより、各質問タイプに最適化された処理を提供し、精度とコスト効率を最大化できます。

  2. Q2. ルーター処理にGPT-4o-mini、RAG回答生成にGPT-4oを使い分ける理由は？

    回答を表示

    ルーター処理は単純な質問分類なので、高速で低コストなGPT-4o-miniで十分です。 一方、RAG回答生成は精度が重要なので、より高性能なGPT-4oを使用します。 このように、処理内容に応じてLLMを使い分けることで、コストと性能のバランスを最適化できます。

  3. Q3. AIアプリ運用において、継続的改善が重要な理由は何ですか？

    回答を表示

    AIアプリは公開時点では精度が不十分な場合が多く、ユーザーフィードバックを収集して改善することで、 精度が大きく向上します。また、業務内容や社内規定の変更に追従するためにも、ナレッジベースの定期更新が不可欠です。 「作って終わり」ではなく、継続的に改善するサイクルを回すことが、AIアプリ成功の鍵です。
