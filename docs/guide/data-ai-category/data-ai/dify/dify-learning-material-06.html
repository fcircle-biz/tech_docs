第6章

ナレッジベース入門 - RAGの基礎を理解する

RAG（検索拡張生成）の概念と、Difyでのナレッジベース機能を理解し、独自のドキュメントをAIに参照させる仕組みの基本を学びます。

この章で学ぶこと

  • RAG（検索拡張生成）とは何か、なぜ重要なのかを理解する
  • LLMの限界とRAGによる解決方法
  • ベクトルデータベース、埋め込み、セマンティック検索の基本概念
  • Difyのナレッジベース機能の概要と仕組み

6.1 RAGとは何か

RAG（Retrieval Augmented Generation：検索拡張生成）は、外部の知識ソースから関連情報を検索し、その情報を基にLLMが回答を生成する技術です。

通常のLLMは、学習時点のデータに基づいて回答を生成するため、以下のような限界があります：

LLMの限界

  • • 古い情報: 学習データの時点以降の情報を知らない（知識カットオフ）
  • • 企業固有の情報がない: 社内文書、製品マニュアル、業務規定などは学習していない
  • • ハルシネーション: 知らない情報について事実のように創作してしまう
  • • 根拠が不明: 回答の出典や参照元を示せない

RAGは、これらの限界を解決します。 独自のドキュメント（社内マニュアル、製品文書、最新ニュースなど）から関連情報を検索し、その情報を元にLLMが回答を生成するため、 正確性が高く、根拠が明確な回答が得られます。

RAGのメリット

  • 最新情報に対応: ドキュメントを更新すれば即座に最新情報を参照可能
  • 独自情報の活用: 社内文書や製品マニュアルなど企業固有の情報を利用できる
  • ハルシネーション抑制: 実際のドキュメントに基づいて回答するため、創作が減る
  • 出典の明示: どのドキュメントを参照したかを示せる
  • コスト効率: 大規模モデルの再学習不要

6.2 RAGの仕組み

RAGは、大きく分けて2つのフェーズで動作します。

RAGの動作フロー

PDF、Word、テキスト] --> B[チャンク分割
小さな断片に分ける] B --> C[埋め込み
ベクトル化] C --> D[ベクトルDB保存
検索可能な形で保存] end subgraph 実行時["フェーズ2: 実行時（検索と生成）"] E[ユーザー質問] --> F[質問を埋め込み

類似ベクトルを検索] D --> G G --> H[関連文書断片を取得

コンテキストとして付与] I --> J[LLMが生成
文書を参照して回答] J --> K[回答+出典] end

フェーズ1: 事前準備（インデックス作成）

RAGを使う前に、参照したいドキュメントを検索可能な形に変換します。

  1. ドキュメント読み込み: PDF、Word、テキストファイルなどを読み込む
  2. チャンク分割: 長い文書を小さな断片（チャンク）に分割する

    例: 1000文字ごと、段落ごと、セクションごとなど

  3. 埋め込み（Embedding）: 各チャンクをベクトル（数値の配列）に変換する

    意味が似ている文章は似たベクトルになる

  4. ベクトルデータベース保存: ベクトル化されたチャンクをベクトルデータベースに保存

フェーズ2: 実行時（検索と生成）

ユーザーが質問すると、以下の流れで回答が生成されます。

  1. 質問をベクトル化: ユーザーの質問を埋め込みモデルでベクトルに変換
  2. セマンティック検索: 質問ベクトルと似たベクトルを持つチャンクをベクトルDBから検索

    キーワード一致ではなく、意味的に関連するチャンクを探す

  3. 関連文書を取得: 類似度が高い上位N件のチャンクを取得
  4. プロンプトに挿入: 取得したチャンクをコンテキストとしてプロンプトに挿入
  5. LLMが生成: コンテキスト付きプロンプトをLLMに送り、回答を生成
  6. 回答+出典: 回答とともに参照した文書の出典情報を返す

セマンティック検索の重要性

従来のキーワード検索では「犬」と「イヌ」は別の単語として扱われますが、 セマンティック検索では意味的に同じと理解します。 これにより、質問の表現が異なっても関連する情報を正確に見つけられます。

6.3 重要な構成要素

RAGを理解する上で重要な3つの構成要素を詳しく見ていきましょう。

1. ベクトルデータベース（Vector Database）

ベクトル（数値の配列）を効率的に保存・検索するためのデータベースです。 従来のリレーショナルDBとは異なり、ベクトル間の距離や類似度を高速に計算できます。

代表的なベクトルDB:

  • • Weaviate: Difyのデフォルト、高速でスケーラブル
  • • Qdrant: Rust製、高性能
  • • Chroma: シンプルで使いやすい
  • • Pinecone: クラウドマネージドサービス

2. 埋め込みモデル（Embedding Model）

テキストをベクトルに変換するためのAIモデルです。 意味が似ている文章は似たベクトルになるように学習されています。

代表的な埋め込みモデル:

  • • text-embedding-3-small: OpenAI、コスパ良好
  • • text-embedding-3-large: OpenAI、高精度
  • • voyage-large-2: Voyage AI、最高精度クラス
  • • multilingual-e5-large: 多言語対応、無料で利用可能

3. チャンク分割（Chunking）

長いドキュメントを小さな断片に分割する処理です。 チャンクのサイズと分割方法は、RAGの精度に大きく影響します。

チャンク分割の戦略:

  • • 固定サイズ: 500文字、1000文字など固定長で分割
  • • 段落ベース: 段落の区切りで分割
  • • セクションベース: 見出しや章ごとに分割
  • • オーバーラップ: チャンク間で一部重複させて文脈を保持

6.4 Difyのナレッジベース機能

Difyでは、RAGの複雑な仕組みが「ナレッジベース」という機能として抽象化されています。 開発者はベクトルDBや埋め込みモデルの詳細を意識することなく、ドキュメントをアップロードするだけでRAGを利用できます。

Difyナレッジベースの特徴

  • 多様なファイル形式対応: PDF、Word、Markdown、TXT、HTML、CSVなど
  • 自動チャンク分割: ドキュメントアップロード時に自動で適切なサイズに分割
  • 埋め込みモデル選択: 複数の埋め込みモデルから選択可能
  • ベクトルDB統合: Weaviate、Qdrant、Chromaなどに対応
  • インデックス管理: GUIでインデックスの追加・削除・更新が可能
  • 複数ナレッジベース: 用途ごとに複数のナレッジベースを作成・管理できる

Difyでのナレッジベース利用フロー

graph TB A[ナレッジベース作成] --> B[ドキュメントアップロード
PDF、Word、TXTなど] B --> C[チャンク設定
サイズとオーバーラップ] C --> D[埋め込みモデル選択
text-embedding-3-small等] D --> E[インデックス作成
自動処理] E --> F[ナレッジベース完成
検索可能] F --> G[アプリに接続] G --> H[ユーザー質問] H --> I[セマンティック検索
関連チャンク取得] I --> J[LLMが回答生成
文書参照] J --> K[回答+出典表示]

ナレッジベースの再利用

一度作成したナレッジベースは、複数のアプリで共有・再利用できます。 例えば、社内マニュアルのナレッジベースを作れば、FAQチャットボット、社内検索エンジン、 新人教育アシスタントなど、様々なアプリで活用できます。

6.5 RAGの活用シーン

RAGは実務において非常に幅広い用途で活用されています。

社内FAQ・ヘルプデスク

社内マニュアル、業務手順書、過去のQ&Aをナレッジベース化し、社員からの問い合わせに自動で回答。 サポート担当者の負担を軽減しながら、24時間対応可能なヘルプシステムを構築。

製品ドキュメント検索

製品仕様書、技術マニュアル、APIドキュメントをナレッジベースに登録し、 開発者やサポート担当者が必要な情報を自然言語で検索。キーワード検索より直感的。

法務・契約書分析

契約書、社内規定、法令文書をナレッジベース化し、特定の条項や規定を検索・要約。 法務担当者の調査時間を大幅に短縮し、見落としを防ぐ。

業界ニュース分析

業界ニュース、レポート、論文をナレッジベースに登録し、特定のトピックに関する情報を抽出・要約。 市場調査や競合分析の効率化に活用。

理解度確認クイズ

  1. Q1. RAG（検索拡張生成）とは何ですか？通常のLLMとの違いを説明してください。

    回答を表示

    RAGは、外部の知識ソースから関連情報を検索し、その情報を基にLLMが回答を生成する技術です。 通常のLLMは学習時点のデータのみに依存しますが、RAGは独自のドキュメントを参照できるため、 最新情報や企業固有の情報に対応でき、ハルシネーションが抑制され、出典を明示できます。

  2. Q2. ベクトルデータベースと埋め込みモデルの役割は何ですか？

    回答を表示

    埋め込みモデルは、テキストをベクトル（数値の配列）に変換する役割を持ちます。 ベクトルデータベースは、そのベクトルを効率的に保存・検索し、類似度が高いベクトルを高速に見つけます。 これにより、セマンティック検索（意味的に関連する情報の検索）が可能になります。

  3. Q3. チャンク分割とは何ですか？なぜ重要なのですか？

    回答を表示

    チャンク分割は、長いドキュメントを小さな断片に分割する処理です。 適切なサイズに分割することで、検索精度が向上し、LLMのコンテキスト長制限にも対応できます。 分割サイズが大きすぎると関連性が低い情報が混ざり、小さすぎると文脈が失われるため、バランスが重要です。

  4. Q4. Difyのナレッジベース機能の主なメリットは何ですか？

    回答を表示

    DifyのナレッジベースはRAGの複雑な仕組みを抽象化し、ドキュメントをアップロードするだけで 自動的にチャンク分割、埋め込み、ベクトルDB保存を行います。 技術的な知識がなくても簡単にRAGアプリを構築でき、複数のナレッジベースを管理・再利用できる点が大きなメリットです。

第6章のまとめ

  • RAGは外部文書を検索し参照することで、LLMの限界（古い情報、企業固有情報の欠如、ハルシネーション）を解決する
  • RAGは事前準備（インデックス作成）と実行時（検索と生成）の2つのフェーズで動作する
  • ベクトルデータベース、埋め込みモデル、チャンク分割がRAGの重要な構成要素
  • Difyのナレッジベース機能により、複雑なRAGの仕組みを意識せず簡単に実装できる
  • FAQ、製品ドキュメント検索、法務調査、業界ニュース分析など、幅広い活用シーンがある
