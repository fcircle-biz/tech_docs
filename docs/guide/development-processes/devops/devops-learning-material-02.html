第2章: バージョン管理とGit戦略

効果的なコラボレーションの基盤

この章で学ぶこと
  • 主要なGitワークフロー（Git Flow、GitHub Flow、GitLab Flow等）を理解する
  • プロジェクトに適したブランチ戦略を選択できるようになる
  • プルリクエスト・コードレビューのベストプラクティスを習得する
  • コミットメッセージ規約（Conventional Commits）を学ぶ
  • Git Hooksを活用した品質保証を実践する

2.1 Gitワークフロー

Gitワークフローとは、チームがGitを使ってどのように協力するかを定義した規約・パターンです。プロジェクトの性質やチームの規模に応じて適切なワークフローを選択することが重要です。

主要なGitワークフローの比較

ワークフロー 特徴 適用シーン
Git Flow 複数の長期ブランチ（main、develop）と短期ブランチ（feature、release、hotfix）を使用 定期リリースがあるプロジェクト、複数バージョンのサポート
GitHub Flow mainブランチとfeatureブランチのみのシンプルな構成 継続的デプロイメント、Web アプリケーション
GitLab Flow 環境ブランチ（production、staging）を追加 複数環境へのデプロイが必要なプロジェクト
Trunk Based Development 常にmainブランチに直接コミット、短命なfeatureブランチ 高頻度デプロイ、成熟したCI/CD環境

Git Flow（伝統的なワークフロー）

gitGraph commit id: "initial" branch develop checkout develop commit id: "setup" branch feature/user-auth checkout feature/user-auth commit id: "add login" commit id: "add signup" checkout develop merge feature/user-auth branch release/v1.0 checkout release/v1.0 commit id: "version bump" checkout main merge release/v1.0 tag: "v1.0" checkout develop merge release/v1.0
Git Flowのブランチ構成
  • main: 本番環境にデプロイされる安定版
  • develop: 開発の統合ブランチ
  • feature/*: 新機能開発用の短期ブランチ
  • release/*: リリース準備用のブランチ
  • hotfix/*: 本番の緊急修正用ブランチ

GitHub Flow（シンプルなワークフロー）

gitGraph commit id: "initial" branch feature/add-profile checkout feature/add-profile commit id: "add profile page" commit id: "add tests" checkout main merge feature/add-profile tag: "deployed" branch feature/fix-bug checkout feature/fix-bug commit id: "fix validation" checkout main merge feature/fix-bug tag: "deployed"
GitHub Flowの原則
  1. mainブランチは常にデプロイ可能な状態
  2. 新機能や修正は必ずfeatureブランチで開発
  3. 定期的にmainにプッシュしてフィードバックを得る
  4. プルリクエストを使ってレビューを受ける
  5. レビュー承認後、mainにマージして即座にデプロイ

Trunk Based Development（最も迅速なワークフロー）

Trunk Based Developmentでは、開発者は常にmainブランチ（トランク）に頻繁にコミットし、短命なfeatureブランチ（1日以内）のみを使用します。

Trunk Based Developmentの特徴
  • 高頻度統合: 1日に複数回mainにマージ
  • Feature Flags: 未完成機能は機能フラグで隠蔽
  • 強力なCI/CD: 自動テストとデプロイが必須
  • 迅速なフィードバック: コンフリクトが少なく、問題の早期発見

2.2 ブランチ戦略のベストプラクティス

ブランチ命名規則

一貫した命名規則により、ブランチの目的が明確になり、チーム全体の理解が容易になります。

# 機能開発
feature/user-authentication
feature/payment-integration
feature/JIRA-123-add-dashboard

# バグ修正
fix/login-validation-error
fix/JIRA-456-fix-memory-leak

# リリース準備
release/v1.2.0
release/2024-Q1

# 緊急修正
hotfix/security-patch
hotfix/critical-data-loss

# 実験・試作
experiment/new-algorithm
spike/performance-test

ブランチ保護ルール

重要なブランチ（main、develop）には保護ルールを設定し、品質を担保します。

推奨される保護ルール
  • 直接プッシュの禁止: mainブランチへの直接コミットを防止
  • プルリクエスト必須: すべての変更はPR経由でマージ
  • レビュー承認必須: 最低1名（または2名）の承認が必要
  • CI/CD通過必須: すべてのテストが成功しないとマージ不可
  • コンフリクト解決必須: マージコンフリクトがないことを確認
  • 署名コミット必須: GPG署名でコミット者の身元を保証（オプション）

マージ戦略の選択

flowchart TD A["マージ戦略の選択"] --> B["Merge Commit
（マージコミット）"] A --> C["Squash and Merge
（スカッシュマージ）"] A --> D["Rebase and Merge
（リベースマージ）"] B --> B1["履歴を完全保持
すべてのコミットが残る"] C --> C1["複数コミットを1つに統合
履歴がクリーンになる"] D --> D1["リニアな履歴
マージコミット不要"]
マージ戦略 利点 欠点 推奨シーン
Merge Commit すべての履歴が保持される 履歴が複雑になる 履歴の完全性が重要な場合
Squash and Merge mainブランチの履歴がクリーン featureブランチの履歴が失われる 多くのWIP commitがある場合
Rebase and Merge リニアな履歴で理解しやすい コンフリクト解決が複雑 クリーンなコミット履歴を維持したい場合

2.3 プルリクエストとコードレビュー

プルリクエストの目的

プルリクエスト（PR）/マージリクエスト（MR）は、単なるコードマージの手段ではなく、以下の重要な役割を果たします。

  • 品質保証: コードレビューによるバグやセキュリティ問題の発見
  • 知識共有: チーム全体でコードベースの理解を深める
  • 設計議論: より良い実装方法についての対話
  • ドキュメント: 変更の意図や背景を記録

効果的なプルリクエストの書き方

## 概要
ユーザー認証機能のパスワードリセット機能を追加

## 背景・目的
- ユーザーがパスワードを忘れた際に自己解決できるようにする
- サポートチームの問い合わせ対応を削減

## 変更内容
- パスワードリセットAPIエンドポイントを追加 (`/api/auth/reset-password`)
- メール送信サービスを統合
- フロントエンドにリセットフォームを追加
- セキュリティ: トークン有効期限を1時間に設定

## テスト方法
1. `/reset-password`ページにアクセス
2. 登録済みメールアドレスを入力
3. 受信したメールのリンクをクリック
4. 新しいパスワードを設定

## スクリーンショット
（UIの変更がある場合は画像を添付）

## チェックリスト
- [x] ユニットテストを追加
- [x] 統合テストを追加
- [x] ドキュメントを更新
- [x] セキュリティレビュー完了
- [x] パフォーマンス影響を確認

## 関連Issue
Closes #123

コードレビューのベストプラクティス

レビュアーの心得
  • 建設的なフィードバック: 問題点だけでなく、改善案も提示
  • 質問形式: 「ここは〇〇すべきだ」ではなく「〇〇した理由は？」
  • タイムリーなレビュー: 24時間以内にレビューを開始
  • コンテキストの理解: PRの目的と背景を把握してからレビュー
  • 自動化可能な指摘は避ける: フォーマットやリントはツールに任せる
PR作成者の心得
  • 小さなPR: 300行以下が理想、大きい場合は分割
  • 明確な説明: 何を、なぜ変更したかを記載
  • セルフレビュー: 提出前に自分でコードを見直す
  • CI/CD通過: テストが成功してからレビューを依頼
  • フィードバックへの対応: 建設的に受け止め、必要に応じて議論

2.4 コミット規約とGit Hooks

Conventional Commits（慣例的コミットメッセージ）

Conventional Commitsは、コミットメッセージに構造化されたフォーマットを適用する規約です。これにより、CHANGELOGの自動生成やバージョン管理が容易になります。

# 基本フォーマット
<type>(<scope>): <subject>

<body>

<footer>
Type（種類）
  • feat: 新機能の追加
  • fix: バグ修正
  • docs: ドキュメントのみの変更
  • style: コードの意味に影響しない変更（フォーマット、セミコロン等）
  • refactor: バグ修正や機能追加を伴わないコードの改善
  • perf: パフォーマンス改善
  • test: テストの追加・修正
  • chore: ビルドプロセスやツールの変更
  • ci: CI/CD設定の変更
実例
# 新機能
feat(auth): add password reset functionality

# バグ修正
fix(api): fix null pointer exception in user service

# Breaking Change（破壊的変更）
feat(api): change response format to include metadata

BREAKING CHANGE: API response now includes 'meta' field

# 複数の変更
feat(ui): add dark mode support

- Add theme toggle button
- Update color scheme
- Save preference to localStorage

Closes #234

Git Hooksによる品質保証

Git Hooksは、Gitのイベント（コミット、プッシュ等）の前後に自動実行されるスクリプトです。これにより、コード品質を自動的に保証できます。

主要なGit Hooks
Hook タイミング 用途
pre-commit コミット前 リンター実行、フォーマット確認
commit-msg コミットメッセージ作成後 メッセージ形式の検証
pre-push プッシュ前 ユニットテストの実行
Huskyによる簡単なGit Hooks設定
# Huskyのインストール（Node.jsプロジェクト）
npm install --save-dev husky
npx husky init

# pre-commitフックの作成
npx husky add .husky/pre-commit "npm test"
npx husky add .husky/pre-commit "npm run lint"

# commit-msgフックの作成（Conventional Commits検証）
npx husky add .husky/commit-msg 'npx --no -- commitlint --edit "$1"'
実習 2-1: GitHub Flowの実践

実際にGitHub Flowを使って開発プロセスを体験します。

手順
  1. リポジトリをクローン（または新規作成）
  2. featureブランチを作成
  3. 変更をコミット（Conventional Commits形式）
  4. リモートにプッシュ
  5. プルリクエストを作成
  6. レビューとマージ
実行例
# 1. リポジトリのクローン
git clone https://github.com/your-org/your-repo.git
cd your-repo

# 2. featureブランチの作成
git checkout -b feature/add-user-profile

# 3. ファイルを編集後、コミット
git add src/profile.js
git commit -m "feat(profile): add user profile page

- Add profile component
- Add API integration
- Add unit tests

Closes #45"

# 4. リモートにプッシュ
git push -u origin feature/add-user-profile

# 5. GitHubでプルリクエストを作成（Web UIまたはGH CLI）
gh pr create --title "Add user profile page" --body "See commit message"

# 6. レビュー承認後、マージ
gh pr merge --squash
期待される結果
  • featureブランチがmainにマージされる
  • CI/CDパイプラインが自動実行される
  • 関連Issueが自動クローズされる
  • クリーンな履歴が保たれる
理解度確認クイズ
  1. Q: Git FlowとGitHub Flowの最も大きな違いは何ですか？
    A: Git Flowは複数の長期ブランチ（main、develop）と明確なリリースサイクルを持つのに対し、GitHub Flowはmainブランチのみを長期ブランチとし、継続的デプロイメントに適したシンプルな構成です。
  2. Q: Squash and Mergeのメリットとデメリットを説明してください。
    A: メリット：mainブランチの履歴がクリーンで理解しやすい。デメリット：featureブランチの詳細な履歴が失われる。多くのWIP commitがある場合に有効です。
  3. Q: 効果的なプルリクエストの条件を3つ挙げてください。
    A: (1) 適切なサイズ（300行以下が理想）、(2) 明確な説明（何を、なぜ変更したか）、(3) CI/CDテストの通過。
  4. Q: Conventional Commitsの「feat」と「fix」の違いは何ですか？
    A: 「feat」は新機能の追加、「fix」はバグ修正を表します。これによりCHANGELOGの自動生成やセマンティックバージョニングが可能になります。
  5. Q: Git Hooksの「pre-commit」は何に使用されますか？
    A: コミット前に自動実行されるスクリプトで、リンター実行やフォーマット確認など、コード品質の自動チェックに使用されます。
