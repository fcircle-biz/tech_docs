経験ベースのテスト技法

エラー推測やチェックリストベースのテスト、探索的テストなど、テスト担当者の知識と経験を活かしたテスト技法を学びます。形式的な技法と組み合わせることで、より効果的なテストが可能になります。

この章で学ぶこと

  • 経験ベースのテスト技法とは何か
  • エラー推測の考え方と実践方法
  • チェックリストベースのテスト手法
  • 探索的テストの進め方とメリット

8.1 経験ベースのテスト技法とは

経験ベースのテスト技法とは、テスト担当者や開発者の経験、スキル、直感、業務知識を活用してテストを設計・実施する手法です。ブラックボックステストやホワイトボックステストのように形式的なルールに基づくのではなく、実践的な知識や過去の経験から「ここにバグがありそう」と予測してテストします。

経験ベースのテストの特徴

  • 柔軟性 - 状況に応じて自由にテストを調整できる
  • 効率性 - 経験から「怪しい箇所」に絞ってテストできる
  • 補完性 - 形式的な技法で見逃した欠陥を発見できる
  • 依存性 - テスト担当者のスキルや経験に大きく依存する

両者を組み合わせることで、最も効果的なテストが実現できます

8.2 エラー推測（Error Guessing）

エラー推測とは、過去の経験や直感から「この部分にはバグがありそうだ」と推測してテストする手法です。テスト担当者が過去に遭遇したバグのパターン、よくある間違い、システムの弱点などを考慮して、テストケースを設計します。

エラー推測の例

境界値付近を疑う

入力フィールドの最大文字数、配列のサイズ上限、日付の月末など、境界値付近でバグが起きやすい

空データ・ゼロを疑う

空の文字列、ゼロ、nullなどの特殊な値で処理が正しく動作しないことがある

複雑な条件を疑う

複数の条件が組み合わさったロジックでは、想定外の組み合わせでバグが起きやすい

過去のバグを疑う

以前にバグが発生した箇所や似た機能は、再度バグが発生する可能性が高い

エラー推測のコツ

過去のバグリストを定期的に見返し、「よくあるバグのパターン」をまとめておくことで、エラー推測の精度が向上します。また、他のテスト担当者とバグ情報を共有することも効果的です。

8.3 チェックリストベースのテスト

チェックリストベースのテストとは、あらかじめ作成した確認項目リスト（チェックリスト）に基づいてテストを実施する手法です。過去の経験や業界のベストプラクティスをもとに作成したチェックリストを使用することで、テストの漏れを防ぎ、品質を一定に保つことができます。

チェックリストの例（Webアプリケーション）

確認項目 確認内容
ブラウザ互換性 Chrome、Firefox、Safari、Edgeで正しく表示されるか
レスポンシブ対応 PC、タブレット、スマートフォンで正しく表示されるか
入力検証 不正な入力に対して適切なエラーメッセージが表示されるか
セキュリティ XSS、SQLインジェクション対策が実装されているか
パフォーマンス ページの読み込み時間が許容範囲内か
アクセシビリティ キーボード操作、スクリーンリーダー対応がされているか

チェックリストのメリット

  • テスト漏れの防止 - 確認すべき項目を網羅的にリスト化することで、重要な項目の見落としを防ぐ
  • 品質の均一化 - 経験の浅いテスト担当者でも、チェックリストに従うことで一定の品質を保てる
  • 知識の共有 - 組織内でチェックリストを共有することで、ノウハウが蓄積される
  • 効率化 - よくあるチェック項目を体系化することで、テスト準備時間を短縮できる

注意点

チェックリストはあくまで「確認すべき項目の一覧」であり、詳細なテストケース（入力値、期待結果など）は含まれません。チェックリストだけに頼らず、状況に応じて追加のテストを行うことが重要です。

8.4 探索的テスト（Exploratory Testing）

探索的テストとは、テストの設計、実行、学習を同時に行う動的なテスト手法です。事前に詳細なテストケースを作成せず、システムを操作しながらリアルタイムでテストを考え、実行し、結果から学習して次のテストを決めていきます。

flowchart LR A[テストの設計] --> B[テストの実行] B --> C[結果の学習] C --> A style A fill:#fef3c7 style B fill:#dbeafe style C fill:#d1fae5

探索的テストのサイクル - 設計・実行・学習を繰り返す

探索的テストの進め方

1. チャーター（目的）を決める

「ログイン機能のセキュリティを確認する」など、テストの大まかな目的を設定

2. 実際に操作しながらテストする

システムを自由に操作し、気になる動作や異常を探す

3. 発見したことを記録する

バグや気づいたことをメモし、次のテスト計画に活かす

4. 学習したことをもとに新たなテストを考える

「ここも試してみよう」と新しいアイデアが生まれたら即座に実行

メリット

  • 予期しないバグを発見しやすい
  • テスト準備時間が短縮される
  • テスト担当者の創造性が活かせる
  • 仕様書が不完全でも実施できる

デメリット

  • テスト結果の再現性が低い
  • 担当者のスキルに大きく依存する
  • 網羅性の証明が難しい
  • 進捗管理がしにくい

探索的テストが有効な場面

テスト時間が限られている場合、仕様が曖昧な場合、新機能の初期段階、形式的なテストで見逃したバグを探す場合などに特に有効です。

8.5 経験ベースのテストと形式的なテストの使い分け

経験ベースのテストと形式的なテスト（同値分割法、境界値分析など）は、どちらか一方だけを使うのではなく、両方を組み合わせて使うことが重要です。それぞれの特性を理解し、状況に応じて適切に使い分けましょう。

テスト技法の比較

項目 形式的なテスト 経験ベースのテスト
再現性 高い（誰でも同じテストを実施可能） 低い（担当者により結果が変わる）
網羅性 高い（体系的にテストを網羅） 低い（担当者の経験に依存）
準備時間 長い（詳細な設計が必要） 短い（すぐに開始できる）
予期しないバグ 発見しにくい 発見しやすい
進捗管理 しやすい（ケース数で管理） しにくい（時間や観点で管理）

推奨される組み合わせ方

  • 基本は形式的なテスト - 同値分割法や境界値分析などで体系的なテストを行う
  • 追加で経験ベースのテスト - エラー推測や探索的テストで形式的なテストで見逃した箇所を探す
  • チェックリストで漏れ防止 - 重要な確認項目を漏らさないようチェックリストを併用する

実習: 経験ベースのテストを試してみよう

課題

普段使っているWebサイト（例: ECサイト、SNS、ニュースサイトなど）を1つ選び、以下の手順で経験ベースのテストを実施してみましょう。

手順

  1. エラー推測 - 「ここにバグがありそう」と思う箇所を3つ挙げてみる
  2. チェックリスト作成 - そのサイトで確認すべき項目を5つリストアップする
  3. 探索的テスト - 10分間、自由にサイトを操作しながら気になる点を探す
  4. 記録 - 発見したことや気づいたことをメモする

振り返り

実際にバグを発見できなくても構いません。「どんな視点でテストを考えたか」「どんな操作を試したか」を振り返ることが重要です。
