テストレベル - 単体・結合・システムテスト

テストの段階（レベル）について学びます。小さな部品をテストする「単体テスト」、部品を組み合わせてテストする「結合テスト」、システム全体をテストする「システムテスト」、ユーザーが確認する「受入テスト」の違いと目的を理解します。

この章で学ぶこと

  • テストレベルとは何か、なぜ段階的にテストするのか
  • 単体テスト（Unit Test）の目的と実施方法
  • 結合テスト（Integration Test）の目的と実施方法
  • システムテスト（System Test）の目的と実施方法
  • 受入テスト（Acceptance Test）の目的と実施方法

4.1 テストレベルとは

テストレベルとは、ソフトウェアテストを実施する段階や範囲のことです。開発プロセスの中で、小さな部品から大きなシステム全体まで、段階的にテストを行います。

なぜ段階的にテストするのでしょうか？それは、小さな単位で問題を早期に発見し、修正コストを抑えるためです。小さな部品レベルでバグを発見すれば、影響範囲が限定され、修正も容易になります。

テストレベルの考え方

それぞれのレベルで、異なる観点と粒度でテストを行います。これにより、バグを段階的に除去し、品質を高めていきます。

4.2 単体テスト（Unit Test）

単体テストは、ソフトウェアの最小単位（関数、メソッド、モジュール）を個別にテストする活動です。プログラムの「部品」が正しく動作するかを確認します。

単体テストの特徴

  • テスト対象 - 関数、メソッド、クラス単位の小さなコード
  • 実施者 - 主に開発者（プログラマー）
  • 実施タイミング - コード実装直後、継続的に実施
  • 目的 - 個々の部品が仕様通りに動作することを確認
  • 自動化 - テストフレームワーク（JUnit、pytest等）を使った自動化が一般的

単体テストの具体例

以下は、足し算を行う関数とそのテストコードの例です。

Python 
# テスト対象の関数
def add(a, b):
    """2つの数値を足し算する関数"""
    return a + b

# 単体テストコード
def test_add():
    # 正常ケース
    assert add(2, 3) == 5
    # ゼロを含むケース
    assert add(0, 5) == 5
    # 負の数を含むケース
    assert add(-1, 1) == 0

このように、1つの関数に対して複数のテストケースを用意し、正しく動作するかを確認します。

単体テストのメリット

  • • バグを早期発見でき、修正コストが低い
  • • 自動化により、継続的に品質を保てる
  • • リファクタリング時の安全性が高まる
  • • コードの仕様を文書化する役割も持つ

4.3 結合テスト（Integration Test）

結合テストは、複数のモジュールや部品を組み合わせて、それらの連携が正しく動作するかをテストする活動です。単体テストで検証した部品同士のインターフェースや相互作用を確認します。

結合テストの特徴

  • テスト対象 - モジュール間のインターフェース、データのやり取り
  • 実施者 - 開発者またはテスト担当者
  • 実施タイミング - 単体テスト完了後、モジュール統合時
  • 目的 - モジュール間の連携が正しく機能することを確認
  • 手法 - ビッグバン統合、トップダウン統合、ボトムアップ統合など

結合テストのイメージ

flowchart LR A[ユーザー入力
モジュール] --> B[計算処理
モジュール] B --> C[データベース
モジュール] C --> D[画面表示
モジュール] style A fill:#dbeafe style B fill:#bfdbfe style C fill:#a5f3fc style D fill:#d1fae5

各モジュールが正しく連携しているか、データが正しく受け渡されるかを確認します。

結合テストで見つかるバグの例

インターフェースの不一致

モジュールAは文字列を返すのに、モジュールBは数値を期待している

データフォーマットの齟齬

日付データを「YYYY/MM/DD」形式で渡したが、受け取り側は「YYYY-MM-DD」を期待

タイミングの問題

データがまだ準備されていないのに、次のモジュールが処理を開始してしまう

4.4 システムテスト（System Test）

システムテストは、システム全体が要求仕様を満たしているかを確認するテストです。個々の部品ではなく、統合されたシステム全体の動作を検証します。

システムテストの特徴

  • テスト対象 - システム全体（エンドツーエンド）
  • 実施者 - テスト担当者、QAチーム
  • 実施タイミング - 結合テスト完了後、本番環境に近い環境で
  • 目的 - システムが要求仕様を満たすことを確認
  • 観点 - 機能要件だけでなく、非機能要件（性能、セキュリティ等）も検証

システムテストの主な検証項目

機能テスト

システムが仕様通りの機能を提供するか

性能テスト

レスポンス時間、スループットが要求を満たすか

セキュリティテスト

不正アクセスや脆弱性がないか

使用性テスト

ユーザーが使いやすいか、直感的に操作できるか

互換性テスト

異なるOS、ブラウザで動作するか

回復性テスト

障害発生時に適切に回復できるか

本番環境に近い環境でのテスト

システムテストは、可能な限り本番環境に近い環境（ステージング環境）で実施します。開発用の環境では見つからない問題を発見するためです。

4.5 受入テスト（Acceptance Test）

受入テストは、システムがビジネス要件を満たし、リリース可能な状態であることを確認するテストです。顧客やユーザーの視点で、実際の業務で使えるかを検証します。

受入テストの特徴

  • テスト対象 - システム全体（ビジネス要件の観点から）
  • 実施者 - 顧客、エンドユーザー、業務担当者
  • 実施タイミング - システムテスト完了後、リリース直前
  • 目的 - システムがビジネスニーズを満たすことを確認
  • 種類 - UAT（ユーザー受入テスト）、運用受入テスト、契約受入テストなど

受入テストの種類

ユーザー受入テスト（UAT）

実際のエンドユーザーが、業務で使えるかを確認するテスト

運用受入テスト（OAT）

運用チームが、バックアップ、復旧、監視などの運用手順を確認するテスト

契約受入テスト（CAT）

契約条件や規制要件を満たしているかを確認するテスト

アルファ・ベータテスト

開発者環境（アルファ）や一部ユーザー環境（ベータ）で実施する試験的テスト

受入テストの重要性

受入テストは、システムが実際の業務で使えるかを最終確認する重要なステップです。ユーザーの視点で問題を発見し、リリース後のトラブルを防ぎます。

理解度確認クイズ

  1. Q1. テストレベルとは何ですか？なぜ段階的にテストするのでしょうか？

    回答を表示

    回答:
    テストレベルは、ソフトウェアテストを実施する段階や範囲のことです。小さな部品から大きなシステム全体まで、段階的にテストを行います。
    段階的にテストする理由は、小さな単位でバグを早期に発見し、修正コストを抑えるためです。部品レベルでバグを発見すれば、影響範囲が限定され、修正も容易になります。

  2. Q2. 単体テストの主な実施者は誰ですか？また、主な目的は何ですか？

    回答を表示

    回答:
    実施者は主に開発者（プログラマー）です。目的は、個々の部品（関数、メソッド）が仕様通りに動作することを確認することです。

  3. Q3. 結合テストで見つかりやすいバグの例を2つ挙げてください。

    回答を表示

    回答例:
    1. インターフェースの不一致 - モジュールAは文字列を返すのに、モジュールBは数値を期待している
    2. データフォーマットの齟齬 - 日付データの形式が送信側と受信側で異なる

  4. Q4. システムテストと受入テストの違いは何ですか？

    回答を表示

    回答:
    システムテスト: QAチームが実施し、システムが要求仕様を満たすことを技術的観点から確認します。
    受入テスト: 顧客やエンドユーザーが実施し、システムがビジネス要件を満たし、実際の業務で使えることを確認します。

  5. Q5. ユーザー受入テスト（UAT）とは何ですか？

    回答を表示

    回答:
    実際のエンドユーザーが、システムが業務で使えるかを確認するテストです。システムがビジネスニーズを満たし、リリース可能な状態であることを最終確認します。

第4章のまとめ

  • テストレベルは、小さな部品から大きなシステム全体まで段階的にテストする仕組みです
  • 単体テストは、関数やメソッド単位の動作を開発者が確認します
  • 結合テストは、モジュール間の連携とインターフェースを確認します
  • システムテストは、システム全体が要求仕様を満たすことを確認します
  • 受入テストは、顧客やユーザーがビジネス要件の充足を最終確認します
  • 各レベルで早期にバグを発見することで、修正コストを大幅に削減できます
