ホワイトボックステスト技法

内部構造を理解した上で実施するテスト技法を学びます。ステートメントカバレッジ、ブランチカバレッジなど、コードのどの部分がテストされたかを測定する方法を理解します。

この章で学ぶこと

  • ホワイトボックステストの基本概念と目的
  • ステートメントカバレッジ（命令網羅）の考え方
  • ブランチカバレッジ（分岐網羅）の考え方
  • カバレッジの計算方法と測定ツール
  • カバレッジ基準の選択と実務での活用

7.1 ホワイトボックステストとは

ホワイトボックステスト（White-box Testing）は、ソフトウェアの内部構造（プログラムコード）を理解した上で実施するテスト技法です。「構造ベーステスト」「ガラスボックステスト」とも呼ばれます。

ブラックボックステストが「何をするか」に注目するのに対し、ホワイトボックステストは「どのように実装されているか」に注目します。コードの内部ロジックを見て、どの部分が実行されたか（カバレッジ）を測定します。

ホワイトボックステストの特徴

  • コードを見る - プログラムの内部構造を確認しながらテストする
  • 網羅性を測定 - どの部分がテストされたかを数値で測定する
  • 論理エラーを発見 - 条件分岐やループの誤りを見つける
  • 開発者主体 - プログラミング知識が必要なため、主に開発者が実施する

主な用途

ホワイトボックステストは主に単体テスト（ユニットテスト）で使用されます。個々の関数やメソッドが正しく実装されているかを確認する際に有効です。

7.2 コードカバレッジとは

コードカバレッジ（Code Coverage）は、テストによってプログラムコードのどの程度が実行されたかを示す指標です。カバレッジが高いほど、より多くのコードがテストされたことを意味します。

カバレッジの計算式

カバレッジ (%) = (実行された要素数 ÷ 全体の要素数) × 100

※「要素」とは、命令文、分岐、条件式など、カバレッジの種類によって異なります

カバレッジのメリット

  • テスト漏れを定量的に把握できる
  • テストの進捗を数値で管理できる
  • テストの品質基準として使える

カバレッジの限界

  • 100%でもバグがないとは限らない
  • コード外の問題は発見できない
  • 実行されても正しさは保証されない

重要な注意点

カバレッジは「テストの網羅性」を測る指標であり、「テストの品質」を直接示すものではありません。100%カバレッジでも不適切なテストケースでは意味がありません。

7.3 ステートメントカバレッジ（命令網羅）

ステートメントカバレッジ（Statement Coverage）は、プログラムのすべての命令文（ステートメント）が少なくとも1回は実行されたかを測定するカバレッジ基準です。日本語では「命令網羅」とも呼ばれます。

サンプルコード

JavaScript 
// 割引価格を計算する関数
function calculateDiscount(price, isMember) {
  let discount = 0;

  if (isMember) {
    discount = price * 0.1; // 会員は10%割引
  }

  return price - discount;
}

ステートメントカバレッジの測定

上記コードには実行可能な命令文が4行あります：

  1. let discount = 0;
  2. if (isMember) の条件評価
  3. discount = price * 0.1;
  4. return price - discount;

テストケース1: calculateDiscount(1000, true)

→ 全4行が実行される → カバレッジ100%

テストケース2: calculateDiscount(1000, false)

→ 3行目をスキップ、3行実行 → カバレッジ75%

100%ステートメントカバレッジを達成するには

最低2つのテストケースが必要：
1. isMember = true のパターン
2. isMember = false のパターン

7.4 ブランチカバレッジ（分岐網羅）

ブランチカバレッジ（Branch Coverage）は、プログラム内のすべての分岐（if文、switch文など）について、真（True）と偽（False）の両方のパスが少なくとも1回は実行されたかを測定するカバレッジ基準です。日本語では「分岐網羅」とも呼ばれます。

サンプルコード

JavaScript 
// ユーザー区分を判定する関数
function getUserType(age, isMember) {
  let type = "一般";

  if (age >= 65) {
    type = "シニア";
  } else if (age < 18) {
    type = "未成年";
  }

  if (isMember) {
    type += "（会員）";
  }

  return type;
}

ブランチカバレッジの測定

上記コードには3つの分岐があり、合計6つのパス（各分岐×2）があります：

  1. if (age >= 65) → True / False
  2. else if (age < 18) → True / False
  3. if (isMember) → True / False

100%ブランチカバレッジを達成するテストケース例：

  • getUserType(70, true) → シニア、会員
  • getUserType(15, false) → 未成年、非会員
  • getUserType(30, false) → 一般、非会員
  • getUserType(30, true) → 一般、会員

ステートメントカバレッジとの違い

ブランチカバレッジは、各分岐の両方のパス（TrueとFalse）を必ずテストします。そのため、ステートメントカバレッジよりも厳密な基準です。ブランチカバレッジ100%であれば、ステートメントカバレッジも100%になります。

7.5 その他のカバレッジ基準

ステートメントカバレッジとブランチカバレッジ以外にも、さまざまなカバレッジ基準が存在します。プロジェクトの要求や重要度に応じて適切な基準を選択します。

条件カバレッジ（Condition Coverage）

個々の条件式（a > 0、b == 5など）がTrueとFalseの両方を取ることを確認します。複雑な条件式をテストする際に使用します。

パスカバレッジ（Path Coverage）

プログラム内のすべての実行可能なパス（経路）をテストします。最も厳密な基準ですが、ループがあると組み合わせが無限になるため、実用的には制限を設けます。

関数カバレッジ（Function Coverage）

プログラム内のすべての関数・メソッドが少なくとも1回は呼び出されたかを測定します。大規模なプログラムの全体的なテスト状況を把握する際に有効です。

行カバレッジ（Line Coverage）

ソースコードの各行が実行されたかを測定します。ステートメントカバレッジとほぼ同義ですが、ツールによっては空行やコメントの扱いが異なります。

7.6 カバレッジ測定ツール

カバレッジを手動で測定するのは困難なため、通常は専用のツールを使用します。各プログラミング言語に対応したツールが用意されています。

JavaScript / TypeScript

  • Jest - テストフレームワークに組み込みのカバレッジ機能
  • Istanbul (nyc) - 独立したカバレッジツール
  • c8 - V8エンジンのネイティブカバレッジ使用

Python

  • Coverage.py - Pythonの標準的なカバレッジツール
  • pytest-cov - pytestと統合したカバレッジ測定

Java

  • JaCoCo - Java Code Coverage Library
  • Cobertura - レポート生成に優れたツール

C# / .NET

  • Coverlet - .NET Core対応のカバレッジツール
  • Visual Studio Code Coverage - Visual Studio統合ツール

カバレッジツールの使用例（Jest）

bash 
# Jestでカバレッジを測定
npm test -- --coverage

# 出力例
# File           | % Stmts | % Branch | % Funcs | % Lines
# ---------------|---------|----------|---------|--------
# calculator.js  |   85.71 |    75.00 |   83.33 |   85.71

7.7 実務での活用とベストプラクティス

カバレッジは有用な指標ですが、適切に活用するためにはいくつかの注意点があります。実務でのベストプラクティスを紹介します。

推奨する活用方法

  • 目標設定: プロジェクト開始時にカバレッジ目標を設定する（例: ブランチカバレッジ80%以上）
  • 継続的測定: CI/CDパイプラインに組み込み、常に測定する
  • 未実行パスの確認: カバレッジレポートで未実行部分を確認し、テストを追加する
  • 質とのバランス: カバレッジだけでなく、テストケースの内容も重視する

避けるべき間違い

  • • カバレッジを上げることだけが目的になる（無意味なテストを増やす）
  • • 100%を必須とする（コストと効果のバランスを無視する）
  • • カバレッジ=品質と誤解する（実行されても正しさは別問題）

適切なカバレッジ目標の目安

プロジェクトタイプ 推奨カバレッジ 備考
医療・金融システム 90%以上 高い信頼性が必要
業務システム 80%以上 標準的な品質基準
Webアプリケーション 70%以上 反復開発が多い
プロトタイプ・MVP 50%以上 初期検証フェーズ

ブラックボックステストとの組み合わせ

ホワイトボックステストは「コードの網羅性」を測定しますが、「仕様の網羅性」は測定できません。ブラックボックステストと組み合わせることで、コードと仕様の両面から品質を確保できます。

実習課題: カバレッジを測定してみよう

課題: 以下の関数に対してテストケースを作成し、カバレッジを測定してください。

JavaScript 
function gradeScore(score) {
  if (score >= 90) {
    return "A";
  } else if (score >= 70) {
    return "B";
  } else if (score >= 50) {
    return "C";
  } else {
    return "D";
  }
}

ステップ1: ブランチを特定する

この関数には何個の分岐がありますか？各分岐のTrue/Falseパスを整理してください。

ステップ2: テストケースを設計する

100%ブランチカバレッジを達成するために必要なテストケースを列挙してください。

ステップ3: 実装する（任意）

可能であれば、実際にテストコードを書いてカバレッジツールで測定してみましょう。

理解度確認クイズ

  1. Q1. ホワイトボックステストとブラックボックステストの違いを説明してください。

    回答を表示

    回答:
    ホワイトボックステストはプログラムの内部構造（コード）を見てテストする手法で、「どのように実装されているか」に注目します。一方、ブラックボックステストは内部構造を見ずに仕様書に基づいてテストする手法で、「何をするか」に注目します。

  2. Q2. ステートメントカバレッジ80%の意味を説明してください。

    回答を表示

    回答:
    プログラム内の実行可能な命令文（ステートメント）のうち、80%がテストによって少なくとも1回は実行されたことを意味します。残り20%の命令文はまだテストされていません。

  3. Q3. ブランチカバレッジの方がステートメントカバレッジより厳密な基準である理由を説明してください。

    回答を表示

    回答:
    ブランチカバレッジは各分岐のTrueとFalseの両方のパスを必ずテストするため、ステートメントカバレッジよりも網羅性が高くなります。ステートメントカバレッジでは条件分岐の片方のパスだけを通ることもあり得ますが、ブランチカバレッジでは両方を通る必要があります。

  4. Q4. カバレッジ100%を達成すれば、バグは存在しないと言えますか？理由も含めて答えてください。

    回答を表示

    回答:
    いいえ、言えません。カバレッジ100%は「すべてのコードが実行された」ことを示すだけで、「すべてのコードが正しく動作した」ことは保証しません。また、テストケースの入力値が不適切であれば、実行されても誤りを発見できない可能性があります。カバレッジは網羅性の指標であり、品質を直接示すものではありません。

第7章のまとめ

  • ホワイトボックステストはコードの内部構造を理解した上で実施するテスト技法です
  • コードカバレッジはテストによってどの程度コードが実行されたかを測定する指標です
  • ステートメントカバレッジはすべての命令文が実行されたかを測定します
  • ブランチカバレッジは各分岐のTrue/False両方のパスが実行されたかを測定します
  • カバレッジ測定にはJest、Coverage.py、JaCoCoなどの専用ツールを使用します
  • カバレッジは網羅性の指標であり、100%でもバグがないとは限りません
  • プロジェクトの重要度に応じた適切なカバレッジ目標を設定することが重要です
