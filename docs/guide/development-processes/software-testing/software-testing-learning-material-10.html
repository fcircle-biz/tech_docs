バグ報告と不具合管理

バグを発見した際の報告方法と、不具合の管理プロセスを学びます。再現手順の記載方法、重要度・優先度の判断基準、不具合のライフサイクル管理について理解します。

この章で学ぶこと

  • 効果的なバグ報告書の書き方
  • バグの再現手順の記載方法
  • 重要度と優先度の違いと判断基準
  • 不具合のライフサイクルと管理プロセス

10.1 バグ報告の重要性

バグを発見した際、適切に報告しなければ、開発者はバグを修正できません。効果的なバグ報告は、バグの早期修正と品質向上に不可欠です。

良いバグ報告の特徴

  • 明確 - 誰が読んでも理解できる
  • 簡潔 - 必要な情報が過不足なくまとめられている
  • 再現可能 - 手順に従えば誰でも同じバグを再現できる
  • 客観的 - 事実に基づいた記述で、憶測や感情は含まない

不適切なバグ報告の例

「画面がおかしい」「動かない」「エラーが出た」といった曖昧な報告では、開発者は原因を特定できず、バグ修正に時間がかかってしまいます。

10.2 バグ報告に含めるべき情報

効果的なバグ報告には、以下の情報を含める必要があります。

バグ報告の必須項目

1. タイトル（概要）

バグの内容を一目で理解できる簡潔な表題

例: 「ログイン画面で正しいパスワードを入力してもエラーメッセージが表示される」

2. 再現手順

バグを発生させるための具体的なステップ（番号付きリストで記載）

3. 期待結果

本来どのように動作すべきか

4. 実際の結果

実際にどのような動作・エラーが発生したか

5. テスト環境

OS、ブラウザ、バージョン、デバイスなどの環境情報

6. 添付資料（任意）

スクリーンショット、ログファイル、動画など

重要なポイント

再現手順は、初めてそのシステムを触る人でも再現できるレベルで詳細に書くことが重要です。前提条件（ログイン状態、事前データ登録など）も忘れずに記載しましょう。

10.3 バグ報告書の具体例

以下に、効果的なバグ報告書の例を示します。

良いバグ報告書の例

【タイトル】

ログイン画面で正しいパスワードを入力してもエラーメッセージが表示される

【再現手順】

  1. ログイン画面にアクセスする
  2. ユーザー名: "test_user01" を入力する
  3. パスワード: "Test@1234" を入力する（正しいパスワード）
  4. 「ログイン」ボタンをクリックする

【期待結果】

ダッシュボード画面に遷移する

【実際の結果】

「ユーザー名またはパスワードが正しくありません」というエラーメッセージが表示され、ログインできない

【テスト環境】

  • OS: Windows 11 Pro
  • ブラウザ: Google Chrome バージョン 120.0.6099.130
  • 画面解像度: 1920×1080

【添付資料】

スクリーンショット: login_error.png

【補足情報】

他のユーザーアカウント (test_user02, test_user03) でも同じ現象を確認しました

悪いバグ報告書の例

【タイトル】 ログインできない

【内容】 ログイン画面でエラーが出てログインできませんでした

問題点: 再現手順、期待結果、実際の結果、テスト環境が記載されていないため、開発者が原因を特定できません。

10.4 重要度と優先度の違い

バグには重要度（Severity）と優先度（Priority）という2つの評価軸があります。この2つは混同されがちですが、明確に異なる概念です。

重要度（Severity）

バグがシステムやユーザーに与える影響の大きさ

  • 致命的 - システムが停止、データ損失
  • 重大 - 主要機能が使用不可
  • 中程度 - 一部機能に影響
  • 軽微 - 見た目の不具合など

優先度（Priority）

バグを修正する順序・緊急性

  • 最優先 - 即座に対応が必要
  • 高 - 次のリリースまでに修正
  • 中 - 将来のリリースで修正
  • 低 - 時間があれば修正

重要度と優先度の関係例

ケース1: 重要度「高」+ 優先度「高」

本番環境でシステムが停止するバグ → すぐに修正が必要

ケース2: 重要度「高」+ 優先度「低」

めったに使われない機能の致命的なバグ → 影響範囲が限定的なので後回し

ケース3: 重要度「低」+ 優先度「高」

デモ画面のロゴ表示ミス（翌日の重要プレゼンで使用） → 影響は軽微だが緊急対応

ケース4: 重要度「低」+ 優先度「低」

文字の微妙な位置ずれ → 余裕があれば修正

誰が決めるのか

重要度は主にテスト担当者が評価し、優先度はプロジェクトマネージャーやプロダクトオーナーがビジネス判断に基づいて決定することが一般的です。

10.5 不具合のライフサイクル

バグは発見されてから修正、検証、クローズまで、一連のライフサイクルを経ます。以下の図は、典型的な不具合管理のフローです。

各ステータスの説明

新規（New）

バグが報告された直後の状態

トリアージ（Triage）

バグを受理するか却下するか、重要度・優先度を判断する段階

オープン（Open）

バグが受理され、修正対象として承認された状態

修正中（In Progress）

開発者がバグの修正作業を進めている状態

修正完了（Fixed）

開発者が修正を完了し、テストチームに返却した状態

再テスト（Retest）

テスト担当者が修正内容を検証している状態

検証済み（Verified）

修正が正しく行われたことが確認された状態

再オープン（Reopen）

修正が不十分で、再度修正が必要な状態

クローズ（Closed）

バグが完全に解決され、対応が終了した状態

トリアージの重要性

報告されたすべてのバグを修正するわけではありません。仕様通りの動作であれば却下されたり、リスクや影響が小さいバグは「既知の制限事項」として受け入れられることもあります。

10.6 不具合管理ツール

実際の開発現場では、バグ報告や管理を効率化するために専用ツールを使用します。代表的なツールを紹介します。

Jira

Atlassian社の統合プロジェクト管理ツール

課題管理、バグトラッキング、スプリント管理に広く使用される

GitHub Issues

GitHubに統合された課題管理機能

オープンソースプロジェクトで広く利用される

Bugzilla

Mozilla開発のオープンソースバグ管理ツール

高機能で長い歴史を持つツール

Redmine

オープンソースのプロジェクト管理ツール

日本で人気の高い課題管理システム

ツールの共通機能

  • バグの登録・検索・フィルタリング
  • ステータス管理（新規→修正中→完了など）
  • 担当者の割り当て
  • コメント機能（議論・質問の履歴）
  • 添付ファイルの管理
  • 通知機能（メール、Slack連携など）
  • レポート・統計機能

10.7 バグ報告時の注意点とベストプラクティス

ベストプラクティス

  • 1つのバグ報告には1つの問題 - 複数の問題を1つの報告にまとめない
  • 重複確認 - 報告前に既存のバグと重複していないか確認する
  • 迅速な報告 - バグを発見したらできるだけ早く報告する
  • 客観的な記述 - 感情的な表現や推測は避け、事実のみを記載する
  • 証拠の提供 - スクリーンショットやログファイルを添付する
  • 再現性の確認 - 報告前に自分で再現できることを確認する
  • コミュニケーション - 開発者からの質問には迅速に回答する

避けるべきこと

  • 曖昧な記述 - 「画面がおかしい」「動かない」など具体性のない表現
  • 感情的な表現 - 「ひどいバグ」「信じられない」など主観的な言葉
  • 原因の憶測 - 「たぶん〇〇が原因」など根拠のない推測
  • 重複報告 - 既存のバグを確認せずに同じ内容を報告する
  • 放置 - 開発者からの質問に回答しない、議論に参加しない

実習: バグ報告書を作成してみよう

シナリオ:

あなたはECサイトのテストを行っています。カートに商品を追加後、「購入手続きに進む」ボタンをクリックすると、エラーページが表示されてしまいました。この問題をバグ報告書として記述してください。

報告例を表示

【タイトル】

カートから購入手続きに進むとエラーページが表示される

【再現手順】

  1. ECサイトのトップページにアクセスする
  2. 任意の商品をクリックして商品詳細ページに移動する
  3. 「カートに追加」ボタンをクリックする
  4. 画面右上のカートアイコンをクリックしてカート画面を表示する
  5. 「購入手続きに進む」ボタンをクリックする

【期待結果】

配送先情報の入力画面に遷移する

【実際の結果】

「500 Internal Server Error」と表示されたエラーページが表示される

【テスト環境】

  • OS: macOS Sonoma 14.2
  • ブラウザ: Safari 17.2

【添付資料】

スクリーンショット: error_page.png

【補足情報】

ブラウザのコンソールログには「TypeError: Cannot read property 'id' of undefined」というエラーメッセージが出力されています

理解度確認クイズ

  1. Q1. バグ報告書に必ず含めるべき情報を5つ挙げてください。

    回答を表示

    回答:
    1. タイトル（概要）
    2. 再現手順
    3. 期待結果
    4. 実際の結果
    5. テスト環境（OS、ブラウザ、バージョンなど）

  2. Q2. 重要度（Severity）と優先度（Priority）の違いを説明してください。

    回答を表示

    回答:
    重要度（Severity）は、バグがシステムやユーザーに与える影響の大きさを示します（致命的、重大、中程度、軽微など）。
    優先度（Priority）は、バグを修正する順序や緊急性を示します（最優先、高、中、低など）。
    例えば、影響は大きいが使用頻度が低い機能のバグは「重要度:高、優先度:低」となることがあります。

  3. Q3. バグのライフサイクルで「トリアージ」とは何ですか？

    回答を表示

    回答:
    トリアージとは、報告されたバグを受理するか却下するか、また重要度・優先度をどう設定するかを判断する段階です。すべての報告がバグとして承認されるわけではなく、仕様通りの動作であれば却下されることもあります。

  4. Q4. 効果的なバグ報告を書くために心がけるべきことを3つ挙げてください。

    回答を表示

    回答例:
    1. 再現手順を具体的に書く（初めて触る人でも再現できるレベル）
    2. 客観的な記述を心がけ、感情的な表現や憶測を避ける
    3. スクリーンショットやログファイルなど証拠となる資料を添付する
    （他にも、1つのバグ報告には1つの問題のみ含める、重複確認を行う、などがあります）

第10章のまとめ

  • 効果的なバグ報告には、タイトル、再現手順、期待結果、実際の結果、テスト環境が必要です
  • 再現手順は初めて触る人でも再現できるレベルで詳細に記載することが重要です
  • 重要度はバグの影響の大きさ、優先度は修正する順序・緊急性を示します
  • バグは新規登録からトリアージ、修正、検証、クローズまでのライフサイクルを経ます
  • Jira、GitHub Issues、Bugzilla、Redmineなどのツールを使ってバグを管理します
  • 客観的な記述、証拠の提供、迅速な報告がバグ報告のベストプラクティスです
