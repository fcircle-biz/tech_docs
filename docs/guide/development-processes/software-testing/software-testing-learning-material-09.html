テストドキュメント - テスト計画書・テストケース

テストで使用する各種ドキュメントの書き方を学びます。テスト計画書、テスト設計仕様書、テストケース、テスト報告書など、チームでテストを進める際に必要な文書の作成方法を習得します。

この章で学ぶこと

  • テストドキュメントの種類と目的
  • テスト計画書の構成と書き方
  • テスト設計仕様書とテストケースの作成方法
  • テスト報告書とトレーサビリティマトリクス

9.1 テストドキュメントとは

テストドキュメントとは、テスト活動で使用・作成される各種文書のことです。これらのドキュメントは、テストの計画・実施・管理を体系的に行うために不可欠であり、チーム内での情報共有や、テスト品質の維持に重要な役割を果たします。

主なテストドキュメントの種類

テスト計画書（Test Plan）

テストの全体方針、スコープ、スケジュール、リソース、リスクを定義する文書

テスト設計仕様書（Test Design Specification）

テスト対象の機能、テスト条件、テスト技法を定義する文書

テストケース（Test Case）

具体的なテスト手順、入力値、期待結果を記述した文書

テスト報告書（Test Report）

テスト実施結果、発見された欠陥、品質評価をまとめた文書

トレーサビリティマトリクス（Traceability Matrix）

要件とテストケースの対応関係を管理する文書

ドキュメントの重要性

テストドキュメントは、テスト活動の証跡となり、再現性や保守性を高めます。また、プロジェクトメンバーが変わっても、一貫したテストを継続できるようになります。

9.2 テスト計画書の書き方

テスト計画書は、テストプロジェクト全体の戦略と方針を定義する文書です。テスト活動の「何を」「いつ」「誰が」「どのように」実施するかを明確にします。

テスト計画書の主な構成項目

1. 概要・目的

プロジェクト名、テストの目的、対象システムの概要

2. テスト対象（スコープ）

テストする機能・範囲、テストしない機能・範囲

3. テストアプローチ・戦略

テストレベル（単体・結合・システム等）、テストタイプ（機能・非機能等）、テスト技法

4. 合格基準（終了基準）

テストを終了する条件（例: 重大バグゼロ、カバレッジ80%以上）

5. テストスケジュール

各テストフェーズの開始日・終了日、マイルストーン

6. 体制・役割

テストマネージャー、テストエンジニア等の役割分担

7. テスト環境

使用するハードウェア、ソフトウェア、ネットワーク環境

8. リスクと対応策

想定されるリスク（スケジュール遅延、環境トラブル等）と軽減策

テスト計画書の記述例（抜粋）

【プロジェクト名】ECサイト新規構築プロジェクト

【テストの目的】

本テストは、ECサイトシステムが要求仕様を満たし、安全かつ安定的に稼働することを確認するために実施する。

【テスト対象】

  • 会員登録・ログイン機能
  • 商品検索・カート機能
  • 決済機能

【合格基準】

  • すべての重大バグが修正済み
  • テストケース実行率100%
  • 機能カバレッジ90%以上

9.3 テスト設計仕様書の書き方

テスト設計仕様書は、テスト計画書をより具体化し、「何を」「どのように」テストするかを定義します。テスト対象の機能ごとに、テスト条件やテスト技法を明確にします。

テスト設計仕様書の構成項目

1. テスト対象機能

どの機能をテストするか（例: ユーザー登録機能）

2. テスト条件

テストで確認すべき条件（例: 正常系、異常系、境界値）

3. 適用するテスト技法

同値分割法、境界値分析、デシジョンテーブル等の技法

4. テストデータの考え方

どのようなデータを用意するか（例: 有効なメールアドレス、無効な文字列）

テスト設計仕様書の記述例

【テスト対象機能】ユーザー登録機能

【テスト条件】

  • メールアドレス形式チェック（正常・異常）
  • パスワード長チェック（8文字以上）
  • 同一メールアドレスの重複登録不可

【適用技法】

  • 同値分割法: 有効なメールアドレス形式/無効な形式
  • 境界値分析: パスワード7文字、8文字、9文字

9.4 テストケースの書き方

テストケースは、テストを実施するための具体的な手順書です。誰が実行しても同じ結果が得られるよう、明確かつ詳細に記述する必要があります。

テストケースの構成要素

項目 説明
テストケースID 一意の識別番号（例: TC001）
テスト項目名 何をテストするか（例: 正常なメールアドレスでの登録）
前提条件 テスト実行前の状態（例: ログイン画面が表示されている）
テスト手順 実行する操作の順序（ステップバイステップ）
テストデータ（入力値） 入力する具体的な値（例: user@example.com）
期待結果 正しく動作した場合の結果（例: 登録完了メッセージが表示される）

テストケースの記述例

テストケースID TC_REG_001
テスト項目名 有効なメールアドレスでのユーザー登録
前提条件 ユーザー登録画面が表示されている
テスト手順 1. メールアドレス欄に「user@example.com」を入力
2. パスワード欄に「password123」を入力
3. 確認パスワード欄に「password123」を入力
4. 「登録する」ボタンをクリック
期待結果 ・「登録が完了しました」というメッセージが表示される
・登録完了画面に遷移する
・データベースに新規ユーザーが登録される

テストケース作成の注意点

  • 曖昧な表現を避け、具体的に記述する（「適切に」「正しく」等は不明確）
  • 誰が実行しても同じ結果が得られる再現性を確保する
  • 1つのテストケースで1つの目的を検証する（複数の目的を混在させない）

9.5 テスト報告書の書き方

テスト報告書は、テスト実施結果をまとめた文書です。プロジェクト関係者に、テストの進捗状況、品質状況、残存するリスクを報告します。

テスト報告書の主な内容

1. テスト実施サマリー

テスト期間、実施したテストケース数、合格/不合格数

2. 検出された欠陥の状況

バグの総数、重要度別内訳、未解決バグの状況

3. テストカバレッジ

要件カバレッジ、コードカバレッジ等の指標

4. 品質評価

合格基準との比較、リリース可否の判断

5. 残存リスク

未解決のバグや未実施のテストによるリスク

テスト報告書の記述例（抜粋）

【テスト実施サマリー】

  • テスト期間: 2025年1月10日 〜 1月20日
  • 実施テストケース数: 150件
  • 合格: 145件、不合格: 5件（再テスト後すべて合格）

【検出された欠陥の状況】

  • 総検出バグ数: 23件
  • 重大: 2件（すべて修正済み）
  • 中: 8件（すべて修正済み）
  • 軽微: 13件（7件修正済み、6件は次バージョンで対応）

【品質評価】

合格基準（重大バグゼロ、カバレッジ90%以上）を満たしており、リリース可能と判断します。

9.6 トレーサビリティマトリクス

トレーサビリティマトリクス（追跡可能性マトリクス）とは、要件とテストケースの対応関係を管理する表です。すべての要件がテストケースでカバーされているかを確認し、テスト漏れを防ぎます。

トレーサビリティマトリクスの例

要件ID 要件名 テストケースID テスト結果
REQ-001 ユーザー登録機能 TC_REG_001, TC_REG_002, TC_REG_003 合格
REQ-002 ログイン機能 TC_LOGIN_001, TC_LOGIN_002 合格
REQ-003 商品検索機能 TC_SEARCH_001, TC_SEARCH_002 合格
REQ-004 決済機能 TC_PAY_001, TC_PAY_002, TC_PAY_003 合格

トレーサビリティマトリクスのメリット

  • すべての要件がテストでカバーされているか確認できる（テスト漏れ防止）
  • 要件が変更された際に、影響を受けるテストケースを特定できる
  • テスト進捗と品質状況を可視化できる

実習: テストケースを作成してみよう

以下のシナリオに対して、テストケースを作成してください。

【シナリオ】

Webサイトのお問い合わせフォームには、以下の入力項目があります：

  • お名前（必須、最大50文字）
  • メールアドレス（必須、メール形式チェック）
  • お問い合わせ内容（必須、最大500文字）

送信ボタンを押すと、入力チェックが行われ、問題なければ「送信完了」メッセージが表示されます。

【課題】

「メールアドレス」欄に対するテストケースを2つ作成してください（正常系1つ、異常系1つ）。

理解度確認クイズ

  1. Q1. テスト計画書に含まれる主な項目を5つ挙げてください。

    回答を表示

    回答例:
    1. 概要・目的
    2. テスト対象（スコープ）
    3. テストアプローチ・戦略
    4. 合格基準（終了基準）
    5. テストスケジュール
    （他にも、体制・役割、テスト環境、リスクと対応策などがあります）

  2. Q2. テストケースの構成要素として必要な項目は何ですか？

    回答を表示

    回答:
    ・テストケースID（一意の識別番号）
    ・テスト項目名（何をテストするか）
    ・前提条件（テスト実行前の状態）
    ・テスト手順（実行する操作の順序）
    ・テストデータ（入力値）
    ・期待結果（正しく動作した場合の結果）

  3. Q3. トレーサビリティマトリクスの目的は何ですか？

    回答を表示

    回答:
    要件とテストケースの対応関係を管理し、すべての要件がテストでカバーされているか確認するためです。これにより、テスト漏れを防ぎ、要件変更時に影響を受けるテストケースを特定できます。

  4. Q4. テスト報告書に含まれるべき内容を3つ挙げてください。

    回答を表示

    回答例:
    1. テスト実施サマリー（テスト期間、実施したテストケース数、合格/不合格数）
    2. 検出された欠陥の状況（バグの総数、重要度別内訳、未解決バグの状況）
    3. 品質評価（合格基準との比較、リリース可否の判断）
    （他にも、テストカバレッジ、残存リスクなどがあります）

第9章のまとめ

  • テストドキュメントには、テスト計画書、テスト設計仕様書、テストケース、テスト報告書、トレーサビリティマトリクスなどがあります
  • テスト計画書は、テストの全体方針、スコープ、スケジュール、体制、リスクを定義します
  • テストケースは、具体的な手順、入力値、期待結果を明確に記述し、再現性を確保します
  • テスト報告書は、テスト結果、品質評価、残存リスクをまとめ、リリース可否の判断材料となります
  • トレーサビリティマトリクスにより、要件とテストケースの対応を管理し、テスト漏れを防ぎます
