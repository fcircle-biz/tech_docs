テストプロセスの全体像

テスト計画から実行、完了報告までの一連の流れを学びます。テスト活動がどのように進められるのか、各フェーズで何をするのかを理解し、テスト全体の見通しを持てるようにします。

この章で学ぶこと

  • テストプロセス全体の流れと各フェーズの役割
  • テスト計画の立て方と計画書の重要性
  • テスト設計・実装・実行の具体的な進め方
  • テスト完了条件と報告書の作成方法

3.1 テストプロセスとは

テストプロセスとは、テスト活動を体系的に実施するための一連の流れ・手順のことです。ただ思いつきでテストを実行するのではなく、計画を立て、設計し、実行し、結果を評価するという段階を踏むことで、効率的かつ効果的にテストを行うことができます。

国際的なテスト標準であるISTQB/JSTQBでは、テストプロセスを以下の5つの主要な活動に分類しています。

プロセスを守る意義

テストプロセスを定義し、それに従うことで、テスト漏れの防止、作業の効率化、品質の可視化、チームメンバー間の認識統一が実現されます。特にプロジェクトが大きくなるほど、プロセスの重要性が高まります。

3.2 テスト計画(Test Planning)

テスト計画は、テストプロセスの最初のフェーズです。「何を」「いつまでに」「誰が」「どのように」テストするのかを明確にします。テスト計画書は、プロジェクト全体のテスト戦略を示す重要なドキュメントです。

テスト計画で決めること

  • テスト目標 - 何を確認するのか（機能動作、性能、セキュリティ等）
  • テスト範囲 - どの機能・モジュールをテストするか、どこをテスト対象外とするか
  • テストアプローチ - 手動テストか自動テストか、どのテスト技法を使うか
  • テストスケジュール - いつからいつまで、各フェーズにどれくらいの期間を割くか
  • リソース - 誰がテストを担当するか、どんな機材や環境が必要か
  • リスク - テストできない部分、不確実な要素の洗い出し
  • 完了条件 - どの状態になればテストを完了とみなすか

テスト計画書の例（簡易版）

1. テスト目標: ECサイトの注文機能が正しく動作することを確認する

2. テスト範囲: 商品検索、カート追加、注文確定、決済処理

3. テスト期間: 2025年2月1日〜2月15日

4. 担当者: テスト担当:田中、レビュー担当:佐藤

5. テスト環境: ステージング環境、Chrome最新版

6. 完了条件: 重大バグゼロ、軽微バグ5件以下

3.3 テスト設計(Test Design)

テスト設計では、計画に基づいて「何をテストするか」を具体的に決めます。テスト対象の機能や仕様を分析し、テストケースの大まかな方針（テスト条件）を定義します。

テスト設計で行うこと

  • テスト対象の分析 - 要求仕様書や設計書を読み、何を確認すべきかを理解する
  • テスト条件の抽出 - テストすべき項目をリストアップする
  • テスト技法の選択 - 同値分割法、境界値分析など、どの技法を適用するかを決める
  • 優先順位の決定 - 重要な機能から順にテストする優先度を決める

テスト条件の例

例: ログイン機能のテスト設計

• 正しいID・パスワードでログインできること

• 間違ったパスワードの場合エラーメッセージが表示されること

• 未登録のIDでログインできないこと

• パスワードを空欄にした場合の挙動

• ログイン後、正しいページに遷移すること

3.4 テスト実装(Test Implementation)

テスト実装では、テスト設計で定めた条件をもとに、実際に実行可能なテストケースを作成します。テストケースには、「前提条件」「実行手順」「期待結果」を明確に記載します。

テスト実装で行うこと

  • テストケースの作成 - テスト条件ごとに具体的な手順と期待結果を記述
  • テストデータの準備 - テストに必要なデータ（ユーザー情報、サンプル商品等）を用意
  • テスト環境の構築 - テストを実行する環境をセットアップ
  • テストスクリプトの作成 - 自動テストを行う場合、スクリプトを記述

テストケースの例

項目 内容
テストID TC-LOGIN-001
テスト項目 正しいIDとパスワードでログイン
前提条件 ユーザー「test@example.com」が登録済み
実行手順 1. ログイン画面を開く
2. ID「test@example.com」を入力
3. パスワード「Test1234」を入力
4. ログインボタンをクリック
期待結果 ホーム画面に遷移し、「ようこそ」メッセージが表示される

3.5 テスト実行(Test Execution)

テスト実行では、作成したテストケースに従って実際にテストを行います。手順通りに操作し、期待通りの結果が得られるかを確認します。結果は「合格(Pass)」「不合格(Fail)」として記録します。

テスト実行で行うこと

  • テストケースの実行 - 手順に従って操作し、結果を記録
  • 結果の比較 - 期待結果と実際の結果を比較し、合否を判定
  • バグの記録 - 不具合が見つかった場合、バグ報告書を作成
  • 再テスト - 修正されたバグが本当に直っているか確認
  • 進捗管理 - テストの進行状況をトラッキング

再テストと回帰テスト

再テストは、修正されたバグが直っているかを確認するテストです。回帰テストは、修正によって他の機能に影響が出ていないかを確認するテストです。両方とも重要です。

3.6 テスト完了(Test Completion)

テスト完了フェーズでは、すべてのテストが終了したことを確認し、テスト結果をまとめた報告書を作成します。プロジェクト関係者に対して、品質状況とリリース可否を報告します。

テスト完了で行うこと

  • 完了条件の確認 - 計画時に定めた完了条件を満たしているかチェック
  • テスト報告書の作成 - テスト結果、発見されたバグの数・状況、品質評価をまとめる
  • 成果物の保管 - テストケース、バグ報告書、実行ログなどを整理・保管
  • 振り返りミーティング - プロセスの改善点を話し合う
  • リリース判断 - テスト結果をもとに、本番環境へのリリース可否を判断

テスト報告書に含まれる項目

• テスト概要: 実施期間、対象機能、担当者

• テスト結果: 実行したテストケース数、合格/不合格の内訳

• バグ報告: 発見されたバグの数、重要度別の内訳、未解決バグの状況

• 品質評価: 目標と比較した達成度、品質に関する総合評価

• リスク: 残存する問題点、今後の注意事項

• 推奨事項: リリース可否の判断、今後の改善提案

完了条件の例

「致命的なバグがゼロ」「重大バグがすべて修正済み」「テストカバレッジが90%以上」「スケジュールの期限到達」など、計画段階で明確に定めておきます。

第3章のまとめ

  • テストプロセスは、計画→設計→実装→実行→完了の5つのフェーズで構成されます
  • テスト計画では、目標・範囲・スケジュール・リソース・完了条件を明確にします
  • テスト設計では、仕様を分析しテスト条件を抽出します
  • テスト実装では、具体的なテストケース（前提条件・手順・期待結果）を作成します
  • テスト実行では、テストケースに従ってテストを実施し、バグを報告・再テストします
  • テスト完了では、完了条件を確認し、報告書を作成してリリース可否を判断します
