テストタイプ - 機能・非機能・変更テスト

テストの種類（タイプ）について学びます。「正しく動作するか」を確認する機能テスト、「使いやすいか」「速いか」を確認する非機能テスト、「変更が悪影響を与えていないか」を確認する回帰テストの違いを理解します。

この章で学ぶこと

  • テストタイプとは何か、テストレベルとの違い
  • 機能テストの目的と種類（ブラックボックステストの応用）
  • 非機能テストの種類（性能、セキュリティ、使用性など）
  • 回帰テストと再テストの違いと重要性

5.1 テストタイプとテストレベルの違い

第4章で学んだテストレベルは「いつ・どの段階でテストするか」を表しました（単体テスト、結合テスト、システムテストなど）。一方、テストタイプは「何を確認するか」という観点での分類です。

5.2 機能テスト（Functional Testing）

機能テストは、ソフトウェアが「何をするべきか」（仕様書に記載された機能）を正しく実行できるかを確認するテストです。ブラックボックステスト技法（第6章で学ぶ）を用いて実施されることが多いです。

機能テストの具体例

ECサイトのカート機能

  • 商品をカートに追加できるか
  • カート内の商品数量を変更できるか
  • カートから商品を削除できるか
  • 合計金額が正しく計算されるか

ログイン機能

  • 正しいIDとパスワードでログインできるか
  • 間違ったIDやパスワードではログインできないか
  • パスワードを忘れた場合の再設定ができるか

検索機能

  • キーワードで商品を検索できるか
  • 検索結果が適切にフィルタリングされるか
  • 検索結果の並び替えができるか

機能テストのポイント

機能テストでは「仕様書に書かれている通りに動作するか」を確認します。内部構造は知らなくても実施可能で、ユーザー視点でのテストとも言えます。

5.3 非機能テスト（Non-Functional Testing）

非機能テストは、「どのように動作するか」を確認するテストです。機能が正しく動作しても、遅い、使いにくい、セキュリティが弱いなどの問題があれば、ユーザーは満足しません。

性能テスト（Performance Testing）

レスポンス速度やシステムの処理能力を確認

  • ページの読み込み時間は2秒以内か
  • 1000人同時アクセスで動作するか
  • 長時間稼働でも性能が低下しないか

セキュリティテスト（Security Testing）

情報の保護や不正アクセス防止を確認

  • 不正なログイン試行をブロックできるか
  • 個人情報が暗号化されているか
  • SQLインジェクション攻撃を防げるか

使用性テスト（Usability Testing）

ユーザーにとっての使いやすさを確認

  • 初めてのユーザーでも操作がわかるか
  • エラーメッセージがわかりやすいか
  • 必要な操作を少ないクリックで完了できるか

信頼性テスト（Reliability Testing）

障害の発生頻度や復旧能力を確認

  • 24時間連続稼働しても安定しているか
  • 障害発生時に適切に復旧できるか
  • データの整合性が保たれるか

互換性テスト（Compatibility Testing）

異なる環境での動作を確認

  • 複数のブラウザで正しく表示されるか
  • スマートフォンでも動作するか
  • 古いOSでも動作するか

保守性テスト（Maintainability Testing）

修正や変更のしやすさを確認

  • コードが読みやすく理解しやすいか
  • 設定変更が簡単にできるか
  • バグ修正が局所的に収まるか

非機能テストの重要性

機能が完璧でも、レスポンスが遅い、セキュリティが甘い、使いにくいなどの問題があれば、ユーザーは離れていきます。非機能テストは品質の大きな部分を占めます。

5.4 回帰テスト（Regression Testing）

回帰テストとは、バグ修正や新機能追加などの変更を行った後に、既存の機能が壊れていないか（デグレードしていないか）を確認するテストです。変更による予期しない副作用を検出することが目的です。

回帰テストが必要な理由

例1: ECサイトで決済機能のバグを修正したところ、カート機能が動かなくなってしまった（コードの依存関係による影響）

例2: 新しい検索機能を追加したら、既存のログイン機能のレスポンスが遅くなった（リソースの競合）

例3: データベースの構造を変更したら、以前は正常だったレポート出力機能でエラーが発生した（データ構造の変更影響）

回帰テストの効率化

変更のたびに全機能を手動でテストするのは時間がかかります。そのため、回帰テストはテスト自動化（第11章で学習）の最も重要な適用領域です。

  • 自動テストスクリプトを用意しておく
  • 変更のたびに自動実行して結果を確認
  • 人手のテストは重要部分のみに集中

5.5 再テスト（Re-Testing）と回帰テストの違い

再テストと回帰テストは、どちらも変更後に実施しますが、目的が異なります。

再テスト（Re-Testing）

目的: 修正したバグが正しく直ったかを確認する

範囲: バグが発生した箇所のみ

例: 「ログインボタンが動かない」というバグを修正した後、そのボタンが正常に動作するかをテストする

回帰テスト（Regression Testing）

目的: 変更が他の機能に悪影響を与えていないかを確認する

範囲: 既存の機能全体（特に変更箇所に関連する部分）

例: ログインボタンの修正後、ログイン以外の機能（カート、検索など）も正常に動作するかをテストする

両者の関係

修正箇所の動作確認] C --> D{修正OK?} D -->|No| B D -->|Yes| E[回帰テスト
既存機能への影響確認] E --> F{全テストOK?} F -->|No| G[影響箇所を修正] F -->|Yes| H[完了] G --> C style A fill:#fecaca style B fill:#fef3c7 style C fill:#dbeafe style D fill:#fef3c7 style E fill:#d1fae5 style F fill:#fef3c7 style G fill:#fecaca style H fill:#a7f3d0

両方のテストが必要

再テストで修正が正しいことを確認した後、回帰テストで他の部分に影響がないかを確認します。どちらか一方だけでは不十分です。

実習: 身近なアプリで考えてみよう

あなたが普段使っているアプリ（SNS、ECサイト、メモアプリなど）を1つ選び、以下の観点で考えてみましょう。

  1. 機能テストの観点:

    そのアプリの主要機能を3つ挙げ、それぞれどんなテストケースが必要か考えてみてください。

  2. 非機能テストの観点:

    性能、使用性、セキュリティの面で、どんな問題があると困るか考えてみてください。

  3. 回帰テストの観点:

    アプリがアップデートされた際、どの既存機能が正常に動作することを確認すべきか考えてみてください。

理解度確認クイズ

  1. Q1. テストレベルとテストタイプの違いを説明してください。

    回答を表示

    回答:
    テストレベル: 「いつ・どの段階でテストするか」を表す（例: 単体テスト、結合テスト、システムテスト）
    テストタイプ: 「何を確認するか」を表す（例: 機能テスト、非機能テスト、回帰テスト）
    両者は組み合わせて使用され、例えば「システムテスト（レベル）」の中で「機能テストと非機能テスト（タイプ）」を実施する、といった形になります。

  2. Q2. 機能テストと非機能テストの違いを具体例とともに説明してください。

    回答を表示

    回答:
    機能テスト: 「何をするべきか」（仕様通りに機能するか）を確認する
    例: ログイン機能で正しいID/パスワードでログインできるか

    非機能テスト: 「どのように動作するか」（性能、使いやすさ、セキュリティなど）を確認する
    例: ログイン処理が2秒以内に完了するか、パスワードが暗号化されているか

  3. Q3. 非機能テストの種類を3つ挙げ、それぞれの目的を説明してください。

    回答を表示

    回答例:
    1. 性能テスト: レスポンス速度やシステムの処理能力を確認する
    2. セキュリティテスト: 情報の保護や不正アクセス防止を確認する
    3. 使用性テスト: ユーザーにとっての使いやすさを確認する
    （他にも、信頼性テスト、互換性テスト、保守性テストなどがあります）

  4. Q4. 再テストと回帰テストの違いを説明してください。

    回答を表示

    回答:
    再テスト（Re-Testing）:
    - 目的: 修正したバグが正しく直ったかを確認する
    - 範囲: バグが発生した箇所のみ

    回帰テスト（Regression Testing）:
    - 目的: 変更が他の機能に悪影響を与えていないかを確認する
    - 範囲: 既存の機能全体（特に変更箇所に関連する部分）

    両方のテストが必要で、まず再テストで修正が正しいことを確認し、次に回帰テストで他の部分に影響がないかを確認します。

  5. Q5. なぜ回帰テストにはテスト自動化が適しているのですか？

    回答を表示

    回答:
    回帰テストは変更のたびに既存機能全体をテストする必要があり、手動で実施すると膨大な時間がかかります。自動テストスクリプトを用意しておけば、変更のたびに自動実行して短時間で結果を確認でき、効率的に品質を維持できるためです。また、同じテストを繰り返し実行するため、自動化との相性が非常に良いのも理由の一つです。

第5章のまとめ

  • テストレベルは「いつテストするか」、テストタイプは「何を確認するか」を表す
  • 機能テストは「何をするべきか」（仕様通りの機能）を確認し、ブラックボックステスト技法を用いる
  • 非機能テストは「どのように動作するか」を確認し、性能、セキュリティ、使用性など多様な観点がある
  • 回帰テストは変更による既存機能への悪影響を確認し、テスト自動化との相性が良い
  • 再テストは修正箇所の動作確認、回帰テストは既存機能全体への影響確認であり、両方が必要
