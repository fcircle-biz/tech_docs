テストの基本原則

ISTQB/JSTQBが定めるソフトウェアテストの7つの基本原則を学びます。「テストは欠陥があることを示せるが、欠陥がないことは示せない」「全数テストは不可能」など、テストに取り組む際に知っておくべき重要な考え方を理解します。

この章で学ぶこと

  • ISTQB/JSTQBのテスト7原則とは何か
  • 各原則の意味と実務での適用方法
  • テストの限界と現実的なアプローチ
  • 効果的なテスト戦略の立て方

2.1 テスト7原則とは

ISTQB（International Software Testing Qualifications Board）とその日本支部であるJSTQB（Japan Software Testing Qualifications Board）は、ソフトウェアテストの国際標準資格を提供する組織です。これらの組織が定める「テスト7原則」は、テストに取り組む際の基本的な考え方として、世界中で広く認識されています。

テスト7原則の役割

これらの原則は、テストの限界を理解し、現実的で効果的なテスト戦略を立てるためのガイドラインです。すべてのテスト担当者が知っておくべき基礎知識となります。

2.2 原則1: テストは欠陥があることを示せるが、欠陥がないことは示せない

テストは「バグがある」ことは証明できますが、「バグがない」ことは証明できません。テストで見つからなかったバグが、まだ隠れている可能性は常にあります。

具体例

100個のテストケースをすべて合格しても...

101個目のテストケースでバグが見つかる可能性があります。また、テストで想定していない操作方法や環境で問題が発生することもあります。

テストの役割

テストは「バグがない」と保証するものではなく、「可能な範囲でバグを見つけ出し、品質を高める」活動です。

実務での適用

テスト完了時に「すべてのバグを発見した」と断言せず、「計画したテストを完了し、重大な問題は発見されなかった」と報告します。

2.3 原則2: 全数テストは不可能

すべての入力パターン、すべての操作の組み合わせをテストすることは、時間的にも物理的にも不可能です。そのため、リスクベースでテスト範囲を決定する必要があります。

なぜ全数テストは不可能か

組み合わせの爆発

例: 3つの入力フィールドがあり、それぞれ10パターンの入力値がある場合

10 × 10 × 10 = 1,000通り

実際のシステムでは入力項目が数十～数百あり、組み合わせは数百万通りになることもあります。

現実的な対応

優先度やリスクに基づいて、重要なテストケースを選択します。これをリスクベーステストと呼びます。

テスト技法の活用

同値分割法、境界値分析などのテスト技法を用いることで、効率的に重要なテストケースを選択できます。これらの技法は第6章で詳しく学習します。

2.4 原則3: 早期テストで時間とコストを節約

開発プロセスの早い段階でテスト活動を開始することで、欠陥を早期に発見でき、修正コストを大幅に削減できます。これをシフトレフト（Shift Left）とも呼びます。

2.5 原則4: 欠陥の偏在

バグは均等に分布するのではなく、特定のモジュールや機能に集中する傾向があります。これをパレートの法則（80:20の法則）とも呼びます。

欠陥の偏在の例

パレートの法則

全体の80%のバグは、20%のモジュールに集中していることが多い

バグが集中する理由

  • 複雑な処理ロジック
  • 頻繁に変更される機能
  • 開発者の経験不足
  • 不十分な設計や仕様

実務での適用

過去にバグが多く発見されたモジュールには、重点的にテストリソースを割り当てます。バグの発生履歴を分析し、リスクの高い箇所を優先的にテストする戦略が効果的です。

2.6 原則5: 殺虫剤のパラドックス

同じテストを繰り返し実行し続けると、新しいバグを発見できなくなります。これは、害虫が殺虫剤に対して耐性を持つようになる現象に似ているため、殺虫剤のパラドックス（Pesticide Paradox）と呼ばれます。

なぜ起こるのか

既知のバグは修正済み

同じテストケースでは、すでに見つかって修正されたバグしか確認できません

カバレッジの偏り

同じ範囲ばかりテストしていると、テストされていない部分にバグが残ったままになります

対策

  • テストケースの定期的な見直し - 新しい視点でテストを追加・更新する
  • テスト技法の変更 - 異なるアプローチでテストを実施する
  • 探索的テスト - 予定外の操作や入力を試してみる
  • メンバーのローテーション - 異なる担当者が異なる視点でテストする

2.7 原則6: テストは状況に依存する

テストのアプローチは、対象システムの種類、目的、リスク、開発手法などによって異なります。万能なテスト手法は存在せず、状況に応じて最適なテスト戦略を選択する必要があります。

システムの種類による違い

ECサイト

  • 決済機能のテストを重視
  • 負荷テストが重要
  • セキュリティテストが必須

医療機器ソフトウェア

  • 安全性が最優先
  • 厳格な規制対応が必要
  • 完全なトレーサビリティ

モバイルゲーム

  • ユーザビリティテストを重視
  • 多様なデバイスでのテスト
  • 短いリリースサイクル

基幹システム

  • データ整合性が重要
  • 回帰テストが必須
  • 長期間の運用を想定

実務での適用

プロジェクト開始時に、対象システムの特性を分析し、最適なテスト戦略を立案します。他のプロジェクトで成功したテスト手法をそのまま適用するのではなく、カスタマイズが必要です。

2.8 原則7: 「バグゼロ」の幻想

すべてのバグを修正しても、ユーザーのニーズを満たさなければ、そのシステムは品質が高いとは言えません。テストの目的は単にバグを見つけることではなく、ユーザーの期待に応えるソフトウェアを提供することです。

「バグゼロ」のリスク

ケース1: 仕様のミス

仕様通りに実装されバグもないが、そもそも仕様がユーザーのニーズを満たしていない

ケース2: 使いにくいUI

すべての機能が正しく動作するが、操作が複雑でユーザーが使いこなせない

ケース3: パフォーマンス不足

機能的には正常だが、処理が遅すぎて実用に耐えない

真の品質とは

真の品質は、技術的な正しさだけでなく、ユーザーの満足度やビジネス目標の達成によって測られます。テストは、これらの観点からも評価する必要があります。

2.9 テスト7原則のまとめと実務への活用

これら7つの原則は、テストの計画、実行、報告のあらゆる段階で活用できます。以下の表は、各原則と実務での適用例をまとめたものです。

原則 キーポイント 実務での適用
1. 欠陥の存在を示す 欠陥がないことは証明できない 「バグゼロ」を保証せず、リスク評価を含めて報告
2. 全数テスト不可能 リスクベースで優先順位をつける 重要度・頻度・影響度を評価してテストケースを選択
3. 早期テスト 早期発見で修正コスト削減 要件・設計段階からレビューとテスト計画を開始
4. 欠陥の偏在 バグは特定モジュールに集中 過去の欠陥履歴を分析し、リスク箇所を重点的にテスト
5. 殺虫剤のパラドックス 同じテストでは新規バグを発見できない テストケースを定期的に見直し、更新する
6. 状況依存 システムの特性に応じたテスト戦略 プロジェクトごとにテスト計画をカスタマイズ
7. 「バグゼロ」の幻想 ユーザーニーズを満たすことが重要 機能テストだけでなく、ユーザビリティや満足度も評価

理解度確認クイズ

  1. Q1. 「テストは欠陥があることを示せるが、欠陥がないことは示せない」という原則が意味することを説明してください。

    回答を表示

    回答:
    テストでバグが見つかれば「バグがある」ことは証明できますが、テストで見つからなかったとしても「バグがない」ことは証明できません。まだ発見されていないバグが隠れている可能性は常にあるため、テストは「可能な範囲でバグを見つけ、品質を高める」活動として理解する必要があります。

  2. Q2. なぜ「全数テストは不可能」なのですか?その対策として何をすべきですか?

    回答を表示

    理由: 入力パターンや操作の組み合わせが膨大になり、すべてをテストすることは時間的・物理的に不可能だからです。

    対策: リスクベースでテスト範囲を決定し、重要度・頻度・影響度を評価して優先順位をつけます。同値分割法や境界値分析などのテスト技法を活用して、効率的に重要なテストケースを選択します。

  3. Q3. 「殺虫剤のパラドックス」とは何ですか?その対策を2つ挙げてください。

    回答を表示

    説明: 同じテストを繰り返し実行し続けると、新しいバグを発見できなくなる現象です。

    対策例:
    1. テストケースを定期的に見直し、新しい視点でテストを追加・更新する
    2. テスト技法を変更したり、探索的テストを実施したりして、異なるアプローチを試す

  4. Q4. 「バグゼロの幻想」という原則が伝えたいことは何ですか?

    回答を表示

    回答:
    すべてのバグを修正しても、ユーザーのニーズを満たさなければ品質が高いとは言えません。真の品質は、技術的な正しさだけでなく、ユーザーの満足度やビジネス目標の達成によって測られるべきです。テストは単にバグを見つけることではなく、ユーザーの期待に応えるソフトウェアを提供することが目的です。

第2章のまとめ

  • ISTQB/JSTQBのテスト7原則は、テストの基本的な考え方を示すガイドラインです
  • テストは欠陥の存在を示せますが、欠陥がないことは証明できません
  • 全数テストは不可能なため、リスクベースでテスト範囲を決定します
  • 早期テストにより修正コストを大幅に削減でき、欠陥は特定モジュールに偏在する傾向があります
  • 同じテストを繰り返すと新規バグを発見できなくなるため、定期的な見直しが必要です
  • テスト戦略は状況に応じてカスタマイズし、真の品質はユーザー満足度で測ります
