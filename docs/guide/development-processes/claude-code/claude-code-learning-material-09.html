Git・GitHub連携

Claude Code を使うと、コミット・ブランチ操作・プルリクエスト作成といった Git/GitHub のワークフローも自然言語で依頼できます。この章では、日々の開発に直結するバージョン管理操作を Claude Code とともに進める方法を学びます。

この章で学ぶこと

  • Claude Code に変更のコミットを依頼し、わかりやすいコミットメッセージを生成させる方法を理解する
  • git の状態確認・差分の説明・ブランチ操作を Claude Code に依頼できる
  • gh CLI を使ったプルリクエスト作成・Issue 対応の流れを把握する
  • コミットや push をユーザーの意図・確認のもとで安全に行う考え方を身につける

9.1 Claude Code と Git の関係

Claude Code はファイル編集だけでなく、Git コマンドの実行も担うことができます。「変更をコミットして」「差分を見せて」のように自然言語で指示すると、AI が適切な git コマンドを選んで実行します。

ただし、コミットや push はリポジトリの履歴を変更する操作です。Claude Code はユーザーの意図を確認しながら進める設計になっており、予期しない上書きが起きにくくなっています。安全に使うために、「何を・なぜ行うのか」を自分でも把握しておくことが大切です。

コミット

変更内容を分析し、わかりやすいメッセージを自動で作成します。

ブランチ操作

新しいブランチの作成・切り替えを自然言語で依頼できます。

GitHub 連携

gh CLI があれば PR 作成・Issue 確認も依頼できます。

前提: git と gh CLI

この章のGit操作には git コマンドが、GitHub 連携（PR・Issue）には GitHub CLI（gh） がインストールされている必要があります。gh をインストールしていない場合、PR 作成などの GitHub 操作は行えません。

9.2 状態確認と差分の説明

コミットの前に、まず現在のリポジトリの状態を把握することが重要です。Claude Code に「今の状態を教えて」「差分を説明して」のように依頼すると、git status や git diff を実行し、その結果を自然言語でわかりやすく説明してくれます。

terminal — 状態確認の例 
# ユーザーの入力
> 今どんな変更がある？差分を説明してほしい

# Claude Code の応答（実行内容）
● git status を実行しています…
● git diff を実行しています…
● src/auth/login.js に3行の変更があります。
  - エラーハンドリングの追加（try/catch ブロック）
  - ログメッセージの修正
  未追跡ファイルはありません。コミット可能な状態です。

単に git diff の出力を貼るだけでなく、変更の意味を言葉で解説してくれるのが Claude Code の特徴です。複数ファイルにまたがる変更でも「どのファイルで何が変わったか」を整理して伝えてくれます。

9.3 コミットの依頼とメッセージ生成

Claude Code に「変更をコミットして」と依頼すると、変更内容を分析した上で適切なコミットメッセージを自動生成し、コミットを実行します。コミットメッセージの質は、コードの変更内容とその理由を正確に反映した、読み手にとってわかりやすいものになります。

terminal — コミット依頼の疑似セッション例 
# ユーザーの入力
> 今の変更をコミットしてください

# Claude Code の応答
● 変更内容を確認しています…（git diff を実行）
● 以下のコミットメッセージを作成しました:
  "fix: ログイン処理のエラーハンドリングを追加"
  コミットしてよろしいですか？

# ユーザーが確認・承認
> はい、お願いします

# Claude Code の実行
● git add src/auth/login.js を実行しています…
● git commit -m "fix: ..." を実行しています…
● コミットが完了しました（abc1234）。

メッセージの内容が気に入らない場合は「メッセージを〇〇に変えて」と修正を依頼できます。また、git add する対象ファイルも「src フォルダだけコミットして」のように絞り込みを指定できます。

コミットは確認してから

Claude Code はコミット前に変更内容を提示し、ユーザーの承認を求めます。「何がステージされているか」「メッセージは適切か」を自分でも確認する習慣をつけましょう。承認なしに勝手にコミットが走ることはありません。

9.4 ブランチ操作

新機能の開発やバグ修正では、作業ブランチを切って進めるのがGitの基本スタイルです。Claude Code にブランチ操作を依頼する際は、目的を伝えると適切なブランチ名を提案してくれます。

新しいブランチを作成して切り替える

依頼例 
# 新機能開発ブランチを切る
> ログイン機能の開発用に
  新しいブランチを作って切り替えて

# Claude Code が実行
● git checkout -b feature/login を実行…
● ブランチ feature/login に切り替えました。

既存ブランチに切り替える

依頼例 
# mainブランチへ戻る
> mainブランチに切り替えて

# Claude Code が実行
● git checkout main を実行…
● main ブランチに切り替えました。

ブランチの一覧確認（git branch）や、最新のリモートブランチの取得（git fetch）なども同様に依頼できます。「今どのブランチにいる？」のような質問にも答えてくれます。

9.5 GitHub 連携 — PR 作成・Issue 対応

GitHub CLI（gh）がインストール・認証済みの環境では、Claude Code はプルリクエストの作成や Issue の確認・対応も担うことができます。ターミナルを離れずに GitHub 上の操作が完結する点が大きなメリットです。

依頼例 Claude Code の動き 必要な条件
「PR を作成して」 gh pr create でタイトル・本文を自動生成して PR を作成 gh CLI・push 済みブランチ
「未解決の Issue を教えて」 gh issue list を実行して一覧を表示・説明 gh CLI
「Issue #12 に対応して」 Issue の内容を読み、修正 → コミット → PR 作成まで一貫して進める gh CLI・適切な権限

push と PR 作成は慎重に

リモートへの push や PR 作成は取り消しが難しい操作です。Claude Code は実行前に確認を求めますが、本当に意図した操作であるかを自分でも確かめてから承認するようにしましょう。特にパブリックリポジトリでは注意が必要です。

9.6 典型的な Git ワークフロー

Claude Code を使った実際の開発では、次のような流れが基本になります。「変更 → 確認 → コミット → PR 作成」という一連の操作を、すべてターミナル内で完結させられます。

flowchart TD A["コードを変更する"] --> B["状態・差分を確認する"] B --> C["Claude Code にコミットを依頼"] C --> D["コミットメッセージを確認・承認"] D --> E["コミット完了"] E --> F{"GitHub PR を作る?"} F -->|"作る（gh CLI あり）"| G["Claude Code に PR 作成を依頼"] G --> H["PR タイトル・本文を確認・承認"] H --> I["PR 作成完了"] F -->|"今回はしない"| J["作業終了"]

このフローのポイントは、各ステップでユーザーが確認・承認する機会があることです。Claude Code が勝手にコミットしたり push したりするのではなく、ユーザーの意図に沿って動くよう設計されています。

一度に全部お願いすることもできる

「変更をコミットして、PR も作って」のように一度の依頼で複数の操作をまとめて頼むことも可能です。Claude Code は順番に確認しながら処理を進めます。慣れてきたら、よく使うフローをまとめて依頼してみましょう。

9.7 Git・GitHub をさらに深く学ぶ

Claude Code を使って Git 操作を依頼できるようになっても、Git 自体の仕組みを理解していると、より的確な指示が出せます。コンフリクトの解消やリベースなど、複雑な操作では Claude Code の動きを理解しながら進めることが大切です。

やってみよう 9-1: 変更をコミットして（任意で）PR を作成する

自分の git リポジトリで小さな変更を行い、Claude Code に一連の操作を依頼してみましょう。

手順

  1. 1 git リポジトリ内で任意のファイルに小さな変更（コメントの追加など）を加える。
  2. 2 Claude Code に「今どんな変更がある？」と確認を依頼し、差分の説明を受ける。
  3. 3 「変更をコミットして」と依頼し、生成されたコミットメッセージを確認した上で承認する。
  4. 4 （任意）gh CLI をインストール済みの場合は「PR も作って」と続けて依頼してみる。

確認ポイント: コミットメッセージが変更内容を適切に説明しているかどうかをチェックしてみましょう。気に入らない場合は「メッセージを〇〇に変えて」と修正を依頼できます。

理解度確認クイズ

  1. Q1. Claude Code に「変更をコミットして」と依頼したとき、コミットメッセージはどのように決まりますか？

    回答を表示

    Claude Code が変更内容（git diff の出力）を分析し、変更の内容と目的を反映したコミットメッセージを自動生成します。生成されたメッセージはユーザーに提示され、確認・修正・承認のステップを経てコミットが実行されます。

  2. Q2. Claude Code から GitHub のプルリクエストを作成するために必要なツールは何ですか？

    回答を表示

    GitHub CLI（gh コマンド）のインストールと認証が必要です。gh が入っていない環境では、PR 作成などの GitHub 上の操作を Claude Code から行うことはできません。

  3. Q3. コミットや push を Claude Code に依頼する際に特に注意すべき点は何ですか？

    回答を表示

    コミットや push はリポジトリの履歴を変更する操作で、取り消しが難しい場合があります。Claude Code は実行前に確認を求めますが、「何がステージされているか」「コミットメッセージは適切か」「本当に push してよいか」を自分でも確認してから承認することが重要です。
