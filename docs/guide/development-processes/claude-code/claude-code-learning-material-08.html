CLAUDE.mdとメモリ・コンテキスト管理

プロジェクト固有の指示を記述する CLAUDE.md の役割と書き方を学びます。/init での自動生成、メモリの使い分け、長い会話でのコンテキスト管理、AI に意図を正しく伝える指示の書き方を理解します。

この章で学ぶこと

  • CLAUDE.md の役割と自動読み込みの仕組みを理解する
  • プロジェクトメモリとユーザーメモリの違いと使い分けを説明できる
  • コンテキストウィンドウの概念と長い会話での管理方法を知る
  • AI に意図を正確に伝える指示の書き方のコツを習得する

8.1 CLAUDE.md とは何か

CLAUDE.md は、プロジェクトのルートディレクトリ（./CLAUDE.md）または ./.claude/CLAUDE.md に置くテキストファイルです。Claude Code はセッション起動時にこのファイルを自動的に読み込み、プロジェクト固有のコンテキスト（背景情報）として利用します。

たとえば「このプロジェクトは Python 3.11 を使っています」「コードの関数名はキャメルケースではなくスネークケースで書いてください」「ビルドは make build で実行できます」といった情報を書いておくと、毎回指示しなくても Claude Code が最初からその方針に従って動いてくれます。

自動で読み込まれる

セッション起動時に Claude Code が自動で参照します。毎回説明する手間が省けます。

チームで共有できる

リポジトリにコミットすれば、チーム全員が同じ指示セットで Claude Code を使えます。

プロジェクトに最適化

コーディング規約・ビルド手順・注意点など、そのプロジェクト固有のルールを伝えられます。

CLAUDE.md の読み込みタイミングと置き場所

Claude Code は起動時にカレントディレクトリから上位ディレクトリへ順にたどり、見つかった CLAUDE.md を読み込みます。プロジェクトでは ./CLAUDE.md または ./.claude/CLAUDE.md が使えます。サブディレクトリに置くとそのフォルダを読むときに専用の指示が追加され、自分専用でコミットしないメモは CLAUDE.local.md に書けます。さらに、ファイル内で @パス と書くと別ファイルを読み込む（import）こともできます。

CLAUDE.md は「文脈」であって「強制設定」ではない

CLAUDE.md に書いた内容は Claude の行動を強く誘導するコンテキストであり、必ず守られることを保証する強制ルールではありません。「絶対に実行させたくない／必ず実行させたい」処理は、CLAUDE.md ではなく 権限設定（settings.json の permissions）や フック（PreToolUse など／第12章）で強制してください。

8.2 /init で CLAUDE.md を自動生成する

既存のプロジェクトに新しく CLAUDE.md を作成するとき、一から書く必要はありません。スラッシュコマンド /init を使うと、Claude Code がプロジェクトの構造を分析して、ひな形を自動生成してくれます。

terminal — /init の実行例 
$ claude
> /init

● プロジェクトを分析しています…
● package.json, src/, tests/ を確認しました。
● CLAUDE.md を生成しました。内容を確認して、
  プロジェクト固有の追記をしてください。

生成された CLAUDE.md は「ひな形」です。プロジェクトの技術スタック・ビルドコマンド・テスト方法などが自動で記述されますが、必ずエディタで開いて不足している情報を追記・修正することが重要です。

重要ポイント

/init はあくまで出発点です。チームのコーディング規約、使用禁止のライブラリ、デプロイ手順など、自動分析では分からない情報は手で追記しましょう。CLAUDE.md は「育てていく」ドキュメントです。

8.3 メモリの種類と使い分け

Claude Code には、指示を記憶させる場所が複数あります。大きく「プロジェクトメモリ」と「ユーザーメモリ」の2種類です。それぞれ目的と使い所が違います。

flowchart TD A["Claude Code のメモリ"] --> B["プロジェクトメモリ"] A --> C["ユーザーメモリ"] B --> D["./CLAUDE.md または
./.claude/CLAUDE.md"] B --> E["サブディレクトリの CLAUDE.md"] B --> J["CLAUDE.local.md
（自分専用・コミットしない）"] C --> F["~/.claude/CLAUDE.md
（ホームディレクトリ）"] D --> G["チームで共有・リポジトリにコミット"] E --> H["特定フォルダへの追加指示"] J --> K["個人的なメモ・一時的な指示"] F --> I["個人の好み・全プロジェクト共通の設定"]
種類 場所 共有 用途
プロジェクトメモリ ./CLAUDE.md または ./.claude/CLAUDE.md チーム全員 コーディング規約・ビルド手順・プロジェクトの方針
ローカルメモリ ./CLAUDE.local.md 自分のみ（コミットしない） このプロジェクト限定の個人的なメモ・一時的な指示
ユーザーメモリ ~/.claude/CLAUDE.md 自分のみ 個人の好み・全プロジェクトに適用したいルール

たとえば「常に日本語でコメントを書いてほしい」という個人の好みはユーザーメモリに、「このリポジトリのテストは pytest で実行する」というプロジェクト固有のルールはプロジェクトメモリ（CLAUDE.md）に書くのが適切です。

ユーザーメモリの作り方

ホームディレクトリに ~/.claude/CLAUDE.md というファイルを作成すると、Claude Code がすべてのプロジェクトで読み込みます。個人の開発スタイルや好みを書き込んでおくと、プロジェクトをまたいで一貫した動作を得られます。

8.4 良い CLAUDE.md の書き方

CLAUDE.md に書く内容が多すぎると、かえって重要な情報が埋もれてしまいます。簡潔・具体的・箇条書きが基本です。以下に良い例と避けるべき例を示します。

CLAUDE.md — 良い記述例 
# プロジェクト概要
受発注管理システム（Python 3.11 + FastAPI + PostgreSQL）

## ビルド・実行
- 起動: `make dev`
- テスト: `pytest tests/`（カバレッジ80%以上を維持）

## コーディング規約
- 関数名・変数名はスネークケース（例: `get_order_list`）
- コメントは日本語で書く
- 型ヒントを必ず付ける

## 注意事項
- `config/secrets.yaml` は絶対にコミットしない

良い書き方

  • 箇条書きで端的に書く
  • 具体的なコマンドや例を示す
  • 「絶対に守るルール」を明記する
  • 情報が変わったら都度更新する

避けるべき書き方

  • 長い説明文で埋める（重要点が埋もれる）
  • 「なるべく〜してください」など曖昧な表現
  • 古い情報を放置したまま更新しない
  • 秘密情報（パスワード・トークン）を書く

セキュリティ注意

CLAUDE.md はリポジトリにコミットされるファイルです。API キー・パスワード・シークレット等の秘密情報は絶対に書かないでください。そのような情報は環境変数や .env ファイル（.gitignore に追加済みのもの）で管理します。

8.5 コンテキストウィンドウと会話の管理

Claude Code の内部では、Claude（大規模言語モデル）が会話の履歴をすべて保持しながら動いています。この「会話を保持できる量」のことをコンテキストウィンドウと呼びます。長いセッションではこのコンテキストが増え続け、いくつかの影響が出てきます。

flowchart LR A["会話開始"] --> B["指示・応答が積み重なる"] B --> C{"コンテキストが
増えてきた"} C -->|そのまま続ける| D["古い内容が圧縮・要約される"] C -->|"/clear で整理"| E["会話をリセット
（新鮮な状態で再開）"] D --> F["重要な情報は CLAUDE.md に残す"] E --> F

会話が長くなると、Claude Code は古い内容を自動的に要約（圧縮）して全体のコンテキスト量を調整します。これは仕様上の動作で、完全に透明というわけではありません。重要なルールや方針は必ず CLAUDE.md に記録しておくことで、会話の長さに関係なく Claude Code に伝え続けられます。

/clear — 会話をリセットする

スラッシュコマンド /clear を使うと、現在の会話履歴をすべてリセットし、新鮮な状態でセッションを再開できます。

  • タスクの切り替えで前の文脈を引きずりたくないとき
  • 応答がずれてきて「白紙から整理したい」とき
  • コンテキストを意図的に軽くして処理を速くしたいとき

8.6 AI に意図を正しく伝える指示の工夫

CLAUDE.md でプロジェクト全体のルールを伝えたうえで、セッション中の個別の指示をどう書くかも大切です。以下のポイントを意識すると、Claude Code の応答の精度が上がります。

1

「何を」「どこで」「どうしたいか」を明示する

「直して」よりも「src/auth/login.py の validate_token 関数で、有効期限切れのときに例外を投げるよう修正して」のように具体的に書きます。

2

制約・禁止事項を明記する

「既存のテストを変えないで」「外部ライブラリは追加しないで」のように、やってほしくないことも指示します。

3

背景・目的も伝える

「なぜ変えたいか」を伝えると、Claude Code がより適切な実装を選んでくれます。「パフォーマンス改善のため処理を並列化したい」など。

4

期待する出力形式を指定する

「一覧にして」「ステップごとに説明して」「コードだけ出して」のように、どういう形で答えてほしいかを添えると、応答が読みやすくなります。

思い通りにいかないときは「会話でブラッシュアップ」

一度の指示で完璧な結果が得られなくても大丈夫です。「もう少しシンプルにして」「エラーハンドリングを追加して」と追加指示を重ねながら仕上げるのが、Claude Code の自然な使い方です。

やってみよう 8-1: /init で CLAUDE.md を生成して規約を追記する

手元の任意のプロジェクト（または練習用に空のフォルダ）で /init を実行し、生成された CLAUDE.md に自分のルールを1つ追記してみましょう。

手順

  1. 1 ターミナルでプロジェクトのルートディレクトリへ移動し、claude を起動する。
  2. 2 /init と入力して実行し、CLAUDE.md が生成されることを確認する。
  3. 3 テキストエディタで CLAUDE.md を開き、自分のプロジェクトに合ったコーディング規約を1つ追記する（例: 「コメントは必ず日本語で書く」）。
  4. 4 Claude Code のセッションを再起動し、追記したルールが反映されているか確認するため、「このプロジェクトのコーディング規約を教えて」と質問してみる。

確認ポイント

Claude Code が自分で追記したルールを答えの中で言及すれば、CLAUDE.md が正しく読み込まれています。

理解度確認クイズ

  1. Q1. CLAUDE.md はどこに置き、いつ読み込まれますか？

    回答を表示

    プロジェクトのルート（./CLAUDE.md）または ./.claude/CLAUDE.md に置きます。Claude Code はセッション起動時に自動的に読み込むため、毎回指示しなくてもその内容を参照して動作します。ただしこれは行動を誘導する「コンテキスト」であり、必ず守られる強制ルールではない点に注意します（強制したい場合は権限設定やフックを使う）。

  2. Q2. プロジェクトメモリとユーザーメモリの違いは何ですか？それぞれどんな内容を書くのが適切ですか？

    回答を表示

    プロジェクトメモリ（プロジェクトルートの CLAUDE.md）はチームで共有するもので、コーディング規約・ビルド手順・プロジェクト固有のルールを書きます。ユーザーメモリ（~/.claude/CLAUDE.md）は自分専用で、個人の好みや全プロジェクトに適用したい設定を書きます。

  3. Q3. 会話が長くなったとき、重要な指示を Claude Code に伝え続けるにはどうすればよいですか？

    回答を表示

    重要なルールや方針を CLAUDE.md に記録しておくことで、会話が長くなっても常に読み込まれます。また、/clear で会話履歴をリセットして新鮮な状態で再開することで、コンテキストをすっきり整理できます。
