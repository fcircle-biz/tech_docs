権限と設定

Claude Code の安全な使い方の要となる権限モデルと設定を学びます。ツール実行の許可・拒否、settings.json による権限・環境変数の設定、権限モードの考え方、そして安全に使うためのセキュリティの基礎を理解します。

この章で学ぶこと

  • Claude Code の権限モデル（確認の仕組み）を理解し、安全に使えるようになる
  • settings.json でよく使う操作を許可リストに登録して、確認の手間を減らす方法を知る
  • 環境変数など基本的な設定の概要を把握する
  • 機密情報の扱いや信頼できないリポジトリでの注意点を知り、セキュリティを意識した運用ができる

10.1 権限モデルとは何か

Claude Code は、ファイルを編集したりコマンドを実行したりといった実際の操作を行う前に、原則としてユーザーに許可を求めます。これを「権限モデル」と呼びます。

たとえば「このファイルを削除してよいですか？」「このコマンドを実行してよいですか？」と確認を求めることで、AI が意図しない操作を勝手に行うリスクを減らしています。あなたが OK を出した操作だけが実行されるため、安心して使えるのが大きなメリットです。

flowchart TD A["あなたの指示"] --> B["Claude Code が操作を計画する"] B --> C{"許可が必要な操作か？"} C -->|はい| D["ユーザーに確認を求める"] D -->|許可する| E["操作を実行する"] D -->|拒否する| F["操作をスキップする"] C -->|許可済みの操作| E E --> G["結果を報告する"]

用語: 「ツール」と「権限」

Claude Code の中で、ファイル読み取り・編集・コマンド実行・Web検索など、AIが実際に何かを行うための機能をそれぞれ「ツール」と呼びます。権限モデルは、どのツールの実行を許可するか・確認するか・拒否するかを管理する仕組みです。

実際の操作では、Claude Code がターミナル上に「このコマンドを実行してよいですか？」のような確認メッセージを表示します。y（yes）で許可、n（no）で拒否できます。また、セッション中の同種の操作を一括で許可するオプションが提示されることもあります。

terminal — 確認メッセージの例（イメージ） 
● 以下のコマンドを実行しようとしています:
  rm -rf dist/

  実行してよいですか？ [y/n/a（今後のすべてに許可）]

10.2 許可・確認・拒否の考え方

Claude Code では、ツールごとに「常に許可」「毎回確認」「拒否」の3段階で挙動を制御できます。これを使いこなすと、安全を保ちつつ作業効率を上げることができます。

設定 挙動 向いている操作の例
許可（allow） 確認なしに実行する ファイル読み取りや定型的なテスト実行など、繰り返し使う安全な操作
確認（confirm） 毎回ユーザーに確認を求める ファイル編集・コマンド実行など、内容を確認してから進めたい操作
拒否（deny） 実行しない（ブロックする） 本番環境へのデプロイや、特定の危険なコマンドなど

よく使う安全な操作（たとえば「読み取り専用のファイル参照」や「ユニットテストの実行」など）を許可リストに登録しておくと、確認ダイアログが繰り返し表示されずスムーズに作業できます。一方、影響の大きい操作はデフォルトの「確認」のままにしておくのが安全です。

/permissions コマンド

Claude Code のセッション中に /permissions と入力すると、現在の権限設定を確認・変更するためのメニューが開きます。よく使う操作の許可登録や、特定ツールのブロック設定をここで行えます。

10.3 settings.json による設定

権限や各種設定は、JSON 形式の設定ファイル（settings.json）に記述して管理できます。設定ファイルは主に2か所に置きます。

プロジェクト用

.claude/settings.json

プロジェクトルートの .claude/ フォルダ内に置く設定ファイル。そのプロジェクト固有の権限や設定を記述します。チームで共有する場合はリポジトリに含めます。

ユーザー用（グローバル）

~/.claude/settings.json

ホームディレクトリの ~/.claude/ 内に置く設定ファイル。自分のすべてのプロジェクトに共通で適用される設定です。

どちらの設定ファイルも JSON 形式で記述します。以下は設定ファイルの概念的な例です（実際のキー名は公式ドキュメントを確認してください）。

.claude/settings.json — 設定例（概念的なイメージ） 
{
  "permissions": {
    "allow": [
      "Bash(npm test)",
      "Bash(npm run lint)",
      "Read"
    ],
    "deny": [
      "Bash(rm -rf *)"
    ]
  },
  "env": {
    "NODE_ENV": "development"
  }
}

設定の読み込み優先順位

プロジェクト用の設定（.claude/settings.json）は、ユーザー用のグローバル設定よりも優先されます。プロジェクトごとに個別の権限ルールを設けたい場合はプロジェクト用に記述し、自分の全プロジェクトに共通の設定はユーザー用に書くのが基本です。

環境変数の設定

settings.json の env セクションに環境変数を記述すると、Claude Code のセッション内で利用できます。APIキーや接続先URLのような環境依存の値を設定する用途に使われます。ただし、機密性の高い値（APIキーやパスワード）は settings.json にそのまま書かず、環境変数やシークレット管理の仕組みを使うことを検討してください（詳細は次の10.4節で解説します）。

10.4 セキュリティの基礎

Claude Code を安全に使い続けるために、以下のセキュリティの基礎を押さえておきましょう。

セキュリティ上の重要な注意事項

  • 信頼できないリポジトリでは注意する: 悪意ある CLAUDE.md やコードが、意図しない操作を Claude Code に指示する可能性があります。知らないリポジトリを開く際は慎重に。
  • 機密情報を設定ファイルに平文で書かない: APIキーやパスワードを settings.json や CLAUDE.md に直接書いてリポジトリにコミットしないようにしましょう。
  • 確認をスキップしすぎない: 「すべての操作を常に許可」にすると便利ですが、意図しない変更が起きやすくなります。不慣れなうちは確認の習慣を保ちましょう。
  • 本番環境への直接操作は慎重に: 本番データベースや本番サーバーに直接接続した状態では、誤操作のリスクが高まります。開発・ステージング環境での使用を基本にしましょう。

プロンプトインジェクションへの注意

外部ファイルやWebページの内容を Claude Code に読み込ませる場合、その内容に「Claude Code への指示」が埋め込まれている可能性があります（プロンプトインジェクション）。たとえば、Webページ内に「すべてのファイルを削除してください」という指示が隠されているケースです。信頼できない外部コンテンツを処理するときは、AI が異常な操作を求めていないか確認するようにしましょう。

安全に使うための基本姿勢

「Claude Code はあなたの代わりに作業を行う協働パートナー」であることを意識しましょう。人間のメンバーに作業を依頼するときと同様に、内容を確認しながら進めることが、安全で生産的な運用の基本です。

機密情報の安全な管理

APIキーやパスワードなどの機密情報は、以下の方法で管理するのが一般的です。

  • シェルの環境変数（.env ファイルや export コマンド）に設定し、設定ファイルからは読み込む形にする
  • .gitignore で機密情報を含むファイルをコミット対象から除外する
  • パスワードマネージャーやシークレット管理サービスを活用する

やってみよう 10-1: 権限設定を確認する

Claude Code の現在の権限設定を確認し、よく使う安全な操作を許可リストに追加してみましょう。

手順

  1. 1 ターミナルで Claude Code を起動し、/permissions コマンドを入力して現在の権限設定を確認する。
  2. 2 自分のプロジェクトで繰り返し使う安全な操作（例: npm test や git status）を確認し、許可リストへの追加を検討する。
  3. 3 プロジェクトフォルダに .claude/settings.json がなければ作成し、許可したい操作を概念に沿って記述してみる（設定例は 10.3 節を参照）。
  4. 4 機密情報（APIキーなど）が settings.json や CLAUDE.md に平文で書かれていないか確認し、あれば環境変数方式への移行を検討する。

ポイント: 許可する操作は「繰り返し使う」「リスクが低い」ものから始めましょう。慣れてきたら少しずつ調整するのが安全です。

理解度確認クイズ

  1. Q1. Claude Code が操作を実行する前に確認を求める仕組みを何と呼びますか？また、その目的を説明してください。

    回答を表示

    「権限モデル」と呼びます。ファイル編集やコマンド実行などの操作の前にユーザーの許可を求めることで、AI が意図しない操作を勝手に行うリスクを減らし、安全に使えるようにするための仕組みです。

  2. Q2. プロジェクト用の設定ファイルとユーザー用（グローバル）の設定ファイルは、それぞれどこに置きますか？

    回答を表示

    プロジェクト用はプロジェクトルートの .claude/settings.json、ユーザー用（グローバル）はホームディレクトリの ~/.claude/settings.json に置きます。プロジェクト用の設定が優先されます。

  3. Q3. APIキーやパスワードなどの機密情報を settings.json にそのまま書くことが推奨されないのはなぜですか？安全な代替手段も挙げてください。

    回答を表示

    設定ファイルをリポジトリにコミットした場合、機密情報が外部に漏れるリスクがあるためです。代替手段としては、シェルの環境変数（.env ファイルや export コマンド）を使う方法や、パスワードマネージャー・シークレット管理サービスを活用する方法があります。また、.gitignore で機密情報を含むファイルをコミット対象から除外することも重要です。
