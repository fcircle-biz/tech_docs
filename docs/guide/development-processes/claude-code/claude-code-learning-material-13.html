実践ワークフローとベストプラクティス

Claude Code を日々の開発で使いこなすための実践的なパターンを学びます。プランモードの活用、効果的なプロンプトの書き方、タスクの分割、テスト駆動の進め方、レビュー依頼など、成果を出すための具体的な使い方と、避けるべきアンチパターンを整理します。

この章で学ぶこと

  • プランモードを使って大きな変更を安全に進める方法を理解する
  • 効果的なプロンプトの書き方（具体的・ゴール明示・ファイル指定）を身につける
  • タスク分割・テスト駆動・レビュー依頼など実践パターンを習得する
  • アンチパターンを知り、失敗しやすい使い方を事前に回避できる

13.1 プランモードの活用

Claude Code には「プランモード」があります。通常のモードではすぐに実行に移りますが、プランモードではまず計画だけを提示し、実際の変更は一切行いません。大きな作業を依頼する前に計画を確認することで、意図とずれた変更が施されるリスクを大幅に減らせます。

モード 動作 向いている場面
通常モード 指示を受けたらすぐ実行 小さな修正・単純なタスク
プランモード 計画を提示して確認を待つ（変更なし） 大規模変更・複数ファイル横断・影響範囲が広い作業

プランモードを有効にするには、Shift+Tab キーを押すか、/plan スラッシュコマンドで切り替えます。計画を確認した後、問題なければ通常モードに戻して「実行して」と依頼します。

terminal — プランモードの流れ 
$ claude
# Shift+Tab または /plan でプランモードへ切り替え

> ユーザー認証モジュールをJWTベースに切り替えたい

● [計画] 以下の手順を実行する予定です：
  1. src/auth/session.js を確認
  2. jwt ライブラリを依存関係に追加
  3. login / logout ハンドラを書き換え
  4. 既存のテストを更新
  この計画で進めますか？

重要ポイント

「大きな変更はまずプランを確認する」という習慣がつくと、意図しない大量変更を防ぎ、レビューも楽になります。プランに違和感があればその場で修正できるので、後から巻き戻す手間が省けます。

13.2 効果的なプロンプトの書き方

Claude Code の出力品質は、プロンプト（指示）の書き方に大きく左右されます。「なんかバグってるから直して」では AI は手がかりを自分で探す必要があります。ゴール・制約・参照ファイルを明示するほど、期待どおりの結果が得られます。

良い指示 vs 悪い指示

悪い指示（曖昧）
  • 「なんかエラーが出てる。直して」
  • 「ログイン周りをよくして」
  • 「パフォーマンスを改善して」
良い指示（具体的）
  • 「src/api/user.js の 42 行目で TypeError が出る。原因を調べて修正して」
  • 「src/auth/ 配下のログイン失敗時に400ではなく401を返すよう修正して」
  • 「db/query.js の N+1 クエリを解消して。テーブル構造は schema.sql を参照して」

効果的なプロンプトには、次の3要素を含めると精度が上がります。

ゴールを明示

「何を達成したいか」を最初に書く。期待する動作や完了の定義を具体的に伝える。

参照ファイルを指定

関係するファイルのパスを明記する。AI が広範囲を探し回る手間を省ける。

制約を伝える

「既存のAPIは変えない」「テストは追加しない」など、やってほしくないことも書く。

ヒント: @ でファイルをピン留め

プロンプト内で @src/auth/login.js のように @ファイルパス と書くと、そのファイルをコンテキストに直接読み込ませることができます。広い範囲から探させるよりも正確で高速です。

13.3 大きなタスクを分割する

「新機能を丸ごと実装して」のような巨大な依頼は、期待どおりに完了しないことが多く、途中で迷子になりやすい傾向があります。タスクを小さな単位に分割し、一歩ずつ確認しながら進める方が、最終的に速くかつ正確に目標へたどり着けます。

flowchart LR A["巨大な依頼\n（まとめて）"] -->|失敗しやすい| B["迷子・手戻り多発"] C["分割した依頼\n（ステップ別）"] -->|成功しやすい| D["段階的に完成"] D --> E["確認・修正"] E --> D

たとえば「ショッピングカート機能を追加して」という依頼は、次のように分割できます。

1

データモデルを設計する

「カートとカートアイテムのDBスキーマを schema.sql に追記して」

2

APIエンドポイントを追加する

「カートへの追加・削除・取得APIを src/api/cart.js に実装して」

3

テストを書く

「カートAPIのユニットテストを tests/cart.test.js に追加して」

4

UIコンポーネントを実装する

「カート画面コンポーネントを src/components/Cart.jsx に作成して」

ヒント: 各ステップの後に確認する

ステップが終わったら「変更内容を差分で確認」「テストを走らせて」など、その場で動作確認を挟むと問題の早期発見につながります。後になればなるほど原因の特定が難しくなります。

13.4 テスト駆動的な進め方

Claude Code との相性が特に良い開発スタイルに「テスト駆動」があります。実装の前にテストを用意しておき、「テストが通るまで直し続けて」と依頼する方法です。完了の定義が明確なため、AI が迷いなく作業でき、品質も担保されます。

flowchart TD A["テストを先に書く\n（または既存テストを用意）"] --> B["Claude Codeに実装を依頼"] B --> C["テストを実行"] C -->|"失敗"| D["Claude Codeが実装を修正"] D --> C C -->|"全テスト通過"| E["完成・レビューへ"]
terminal — テスト駆動での依頼例 
> tests/auth.test.js のテストがすべて通るように
  src/auth/login.js を実装して。
  テストを実行して、全件パスしたら完了とする。

重要ポイント

「テストが通ったら完了」という終了条件を与えると、Claude Code は自律的に試行錯誤してくれます。終了条件が曖昧だと、作業が途中で止まったり過剰な変更が加わったりしやすくなります。

13.5 レビュー依頼と改善提案

Claude Code は実装だけでなく、変更内容のレビューや改善提案も得意です。実装後に「このコードをレビューして問題点を指摘して」と依頼することで、見落としを減らせます。

コードレビュー

変更差分に対してレビューを依頼する。

「今の変更内容をレビューして、 潜在的なバグや改善点があれば教えて」

リファクタリング提案

可読性・保守性の改善点を提案させる。

「src/utils/format.js のリファクタリング 案を提案して（動作は変えないこと）」

ヒント: セカンドオピニオンとして活用

自分が書いたコードや、チームメンバーのPRに対して「セキュリティ上の問題がないかチェックして」と依頼するセカンドオピニオン的な使い方も効果的です。ただしレビュー結果は必ず自分で確認・判断してから採用してください。

13.6 CLAUDE.md で一貫性を高める

第8章で学んだ CLAUDE.md は、プロジェクト固有の規約や背景情報を書いておく場所です。毎回プロンプトに書く必要がある情報をまとめておくことで、指示が短くなり、AI の回答も一貫性が高まります。

CLAUDE.md に書くと効果的な内容 効果
コーディング規約（命名・フォーマット） 生成コードがプロジェクトのスタイルに合う
テストの実行コマンド 毎回聞かなくても自動でテストを走らせてくれる
触ってはいけないファイル・ディレクトリ 意図しない変更を防ぐ
主要なアーキテクチャの説明 AI がプロジェクト構造を理解した上で回答できる
外部ライブラリの選定方針 不要なライブラリ追加を防ぐ

重要ポイント

CLAUDE.md はチームで共有・更新するドキュメントです。「AIとのやり取りで気づいた規約」を随時追記していくと、チーム全員の Claude Code 利用の一貫性が上がります。

13.7 アンチパターン（避けるべき使い方）

Claude Code を使いこなすには、「やってはいけないパターン」を知っておくことも重要です。よくある失敗と対策を整理します。

アンチパターン1: 曖昧な丸投げ

「アプリをいい感じにして」「全部やって」のような指示では、AI は何を優先すべきかわからない。

対策

ゴール・対象ファイル・制約を具体的に書く（13.2を参照）。

アンチパターン2: 確認せず適用し続ける

変更内容を確認せずに「OK、次も続けて」と繰り返すと、意図とずれた修正が積み重なってしまう。

対策

各ステップの後に差分を確認する。変更に違和感があればその場で修正を依頼する。

アンチパターン3: 巨大すぎる一度きりの依頼

「全機能を一から作って」のような依頼は途中で失敗しやすく、リカバリも難しい。

対策

小さく分割して一歩ずつ進める（13.3を参照）。プランモードで計画を確認してから進める（13.1を参照）。

アンチパターン4: 出力を盲目的に信頼する

Claude Code の出力は優秀だが完全ではない。レビューなしでそのまま本番に適用するのは危険。

対策

変更は必ず自分の目で確認し、テストを通し、チームレビューを経てからマージする。

13.8 実践ワークフローのまとめ

この章で学んだパターンを組み合わせると、次のような実践的なワークフローが構築できます。

flowchart TD A["タスク開始"] --> B["プランモードで計画確認"] B --> C{"計画はOK？"} C -->|"修正が必要"| D["プロンプトを修正して再確認"] D --> B C -->|"OK"| E["タスクを小さく分割"] E --> F["具体的プロンプトで1ステップ実行"] F --> G["差分とテストで確認"] G -->|"NG"| F G -->|"OK"| H{"全ステップ完了？"} H -->|"いいえ"| F H -->|"はい"| I["レビュー依頼"] I --> J["コミット＆PR作成"]

重要ポイント: Claude Code は協働パートナー

Claude Code は「魔法の自動化ツール」ではなく、あなたの意図と確認のもとで動く協働パートナーです。良い指示・適切な分割・こまめな確認という習慣が、Claude Code の力を最大限に引き出す鍵です。

やってみよう 13-1: プランモード＋具体的プロンプトで1タスクを進める

自分のプロジェクト（または練習用リポジトリ）で、この章で学んだベストプラクティスを実際に試してみましょう。

手順

  1. 1 取り組みたい小さなタスクを1つ決める（例: 既存関数のリファクタリング、小さなバグ修正）。
  2. 2 プランモード（Shift+Tab）に切り替え、「ゴール + 参照ファイルパス + 制約」を含む具体的なプロンプトを入力する。
  3. 3 提示された計画を確認し、問題がなければ通常モードに戻して実行を依頼する。
  4. 4 変更差分を確認し、テストを実行して動作を確かめる。完了したら「このコードをレビューして」と依頼してみる。

チェックポイント: 「曖昧なプロンプトで依頼したとき」と「具体的なプロンプトで依頼したとき」を比較してみると、出力の質の違いがよくわかります。

理解度確認クイズ

  1. Q1. プランモードとはどのようなモードで、どんな場面に向いていますか？

    回答を表示

    実際の変更を加えず「計画だけを提示して確認を待つ」モードです。大規模変更・複数ファイルをまたぐ作業・影響範囲が広い場合に向いています。計画を確認してから実行に移すため、意図とずれた変更を防げます。

  2. Q2. 効果的なプロンプトに含めるべき3つの要素を答えてください。

    回答を表示

    ①ゴールを明示する（何を達成したいか）、②参照すべきファイルパスを指定する、③制約を伝える（やってほしくないことを明示する）。この3要素を含めることで、期待どおりの結果が得られやすくなります。

  3. Q3. Claude Code を使う際のアンチパターンを1つ挙げ、その対策も説明してください。

    回答を表示

    例1: 「確認せず適用し続ける」→ 各ステップの後に差分を確認し、違和感があればその場で修正を依頼する。
    例2: 「巨大すぎる一度きりの依頼」→ タスクを小さく分割し、プランモードで計画を確認してから進める。
    例3: 「曖昧な丸投げ」→ ゴール・対象ファイル・制約を具体的に書いた指示にする。
