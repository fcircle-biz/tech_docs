第2章: スクラムフレームワーク - 基礎編

スクラムとは何か

この章で学ぶこと
  • スクラムの概要と歴史
  • スクラムの3つの柱（透明性、検査、適応）
  • スクラムの5つの価値観（確約、勇気、集中、公開、尊敬）
  • スクラムの3つのロール（プロダクトオーナー、スクラムマスター、開発チーム）
  • スクラムの5つのイベント（スプリント、スプリントプランニング、デイリースクラム、スプリントレビュー、スプリントレトロスペクティブ）
  • スクラムの3つの作成物（プロダクトバックログ、スプリントバックログ、インクリメント）

2.1 スクラムの概要

スクラム（Scrum）は、アジャイル開発手法の中で最も広く使われているフレームワークです。1990年代初頭に竹内弘高と野中郁次郎が提唱した「ザ・ニュー・プロダクト・デベロップメント・ゲーム」という論文から着想を得て、ケン・シュウェイバーとジェフ・サザーランドによって開発されました。

スクラムという名前は、ラグビーのスクラムから来ています。チーム全員が一丸となってゴールに向かう様子を表しています。

スクラムの定義

スクラムは、複雑な問題に対応する適応型のソリューション開発を可能にする軽量なフレームワークです。スクラムは完全で規範的なプロセスや技法ではなく、現在のマネジメント、環境、作業技法が見えるようにすることで、それらの有効性を継続的に改善できるようにするフレームワークです。

2.2 スクラムの3つの柱

スクラムは経験主義（Empiricism）に基づいており、経験から知識を得て、観察に基づいて意思決定を行います。スクラムは以下の3つの柱で支えられています。

1. 透明性（Transparency）

作業のプロセスと成果物が、作業を実行する人と受け取る人の両方に見えるようにする必要があります。スクラムでは、重要な決定は透明性に基づいて行われます。

実践例:
  • バックログを全員が見られる場所に公開
  • 進捗をバーンダウンチャートで可視化
  • 課題や障害をオープンに共有
  • 共通の用語と定義を使用
2. 検査（Inspection）

スクラムチームは、目標に向けた進捗を頻繁に検査し、望ましくない変化や問題を早期に検出します。検査は作業を妨げるものであってはなりません。

実践例:
  • デイリースクラムでの毎日の進捗確認
  • スプリントレビューでのインクリメント検査
  • レトロスペクティブでのプロセス検査
  • 継続的な品質チェック
3. 適応（Adaptation）

検査の結果、プロセスまたは成果物のいずれかの側面が許容範囲を逸脱している場合、できるだけ早く調整を行う必要があります。

実践例:
  • バックログの優先順位を状況に応じて変更
  • レトロスペクティブでの改善アクション
  • スプリントプランニングでの計画調整
  • 継続的なプロセス改善

2.3 スクラムの5つの価値観

スクラムチームが効果的に機能するためには、以下の5つの価値観をチーム全員が実践し、体現する必要があります。

  1. 確約（Commitment）: チームメンバーは、スプリントゴールの達成とチーム全体をサポートすることに確約します。
  2. 勇気（Courage）: スクラムチームメンバーは、正しいことをする勇気と困難な問題に取り組む勇気を持ちます。
  3. 集中（Focus）: すべての人が、スプリントの作業とスクラムチームの目標に集中します。
  4. 公開（Openness）: スクラムチームとステークホルダーは、作業とそれを実行する上での課題についてオープンにします。
  5. 尊敬（Respect）: スクラムチームメンバーは、お互いを能力のある独立した個人として尊敬します。
演習 2-1: スクラムの価値観を実践で考える

目的: スクラムの5つの価値観を実際の状況で理解する

シナリオ

開発チームは、スプリント終了まで残り2日ですが、予定していたストーリーの半分しか完了していません。プロダクトオーナーから「このストーリーは絶対に必要だ」というプレッシャーがかかっています。

考えてみましょう
  1. この状況で、「公開（Openness）」の価値観を実践するとはどういうことですか？
  2. 「勇気（Courage）」を持ってチームが取るべき行動は何ですか？
  3. 「確約（Commitment）」と「現実的でない約束」の違いは何ですか？

2.4 スクラムの3つのロール

スクラムチームは、プロダクトオーナー、スクラムマスター、開発チームの3つのロールで構成されます。それぞれが明確な責任を持ち、相互に補完し合います。

2.4.1 プロダクトオーナー（Product Owner）

役割と責任

プロダクトオーナーは、スクラムチームが創出するプロダクトの価値を最大化することに責任を持ちます。プロダクトの「何を作るか」を決定する唯一の責任者です。

主な責任:
  • プロダクトバックログの管理: バックログアイテムを明確にし、優先順位を付ける
  • ビジョンの伝達: プロダクトのビジョンと目標をチームとステークホルダーに明確に伝える
  • ステークホルダーとの調整: ビジネス側とチームの橋渡し役
  • 受け入れ基準の定義: 完成の定義（Definition of Done）を明確にする
  • ROIの最大化: ビジネス価値を最大化する決定を行う
必要なスキル:
  • ビジネス知識と市場理解
  • 優先順位付けと意思決定能力
  • コミュニケーション能力
  • 利用可能であること（チームからの質問に迅速に対応）

2.4.2 スクラムマスター（Scrum Master）

役割と責任

スクラムマスターは、スクラムガイドに定義されたスクラムを確立することに責任を持ちます。スクラムチームと組織が、スクラムの理論とプラクティスを理解し実践することを支援する「サーバントリーダー」です。

主な責任:
  • プロセスの促進: スクラムイベントの円滑な運営をサポート
  • 障害の除去: チームの生産性を妨げる障害を特定し、除去を支援
  • コーチング: チームとプロダクトオーナーにスクラムの原則を教え、導く
  • 自己組織化の促進: チームが自律的に問題解決できるよう支援
  • 組織への働きかけ: スクラムの導入と実践を組織全体に促進
必要なスキル:
  • スクラムの深い理解
  • ファシリテーション能力
  • サーバントリーダーシップ
  • 問題解決能力
  • 忍耐力と粘り強さ
注意: スクラムマスターはプロジェクトマネージャーではない

スクラムマスターは、伝統的なプロジェクトマネージャーとは異なります。スクラムマスターは指示を出したり、タスクを割り当てたりするのではなく、チームが自己組織化して最高のパフォーマンスを発揮できるよう支援する役割です。

2.4.3 開発チーム（Development Team）

役割と責任

開発チームは、各スプリントで潜在的にリリース可能なプロダクトインクリメントを作成する専門家で構成されます。チームは自己組織化し、「どのように」作業を行うかを決定します。

特徴:
  • 自己組織化: 外部から指示されるのではなく、チーム自身が作業方法を決定
  • クロスファンクショナル: インクリメントを作成するために必要なすべてのスキルを保有
  • 肩書きなし: 個人の専門性に関係なく、全員が「開発チームメンバー」
  • サブチームなし: チーム内にサブチームを作らない
  • 集団責任: 個人ではなくチーム全体が責任を負う
理想的なチームサイズ:
  • 3〜9人（スクラムマスターとプロダクトオーナーは含まない）
  • 小さすぎるとスキルの制約があり、大きすぎるとコミュニケーションが複雑になる

2.5 スクラムの5つのイベント

スクラムでは、規則性を生み出し、スクラムで定義されていない会議の必要性を最小化するために、5つのイベントが定義されています。すべてのイベントはタイムボックス（時間制限）が設定されています。

プランニング] B --> C[デイリー
スクラム] C --> C C --> D{スプリント
終了?} D -->|No| C D -->|Yes| E[スプリント
レビュー] E --> F[スプリント
レトロスペクティブ] F --> G([次のスプリント]) G --> B

2.5.1 スプリント（Sprint）

スプリントは、スクラムの心臓部です。スプリント内で、完成（Done）したインクリメントが作成されます。

  • 期間: 1ヶ月以内（通常2週間または4週間）
  • 固定長: すべてのスプリントは同じ長さ
  • 連続性: 前のスプリント終了後、すぐに次のスプリントが始まる
  • スプリントゴール: 各スプリントには目標が設定される
スプリント中の制約
  • 品質を下げるような変更はしない
  • スプリントゴールを危険にさらす変更は避ける
  • プロダクトオーナーと開発チームが合意すれば、スコープの明確化や再交渉は可能
  • スプリントの中止は、スプリントゴールが無効になった場合のみ、プロダクトオーナーのみが実行可能（非常にまれ）

2.5.2 スプリントプランニング（Sprint Planning）

スプリントプランニングは、スプリントで実行する作業を計画するイベントです。

  • タイムボックス: 2週間スプリントの場合、最大4時間
  • 参加者: スクラムチーム全員
  • 目的: スプリントで何を達成するか（What）と、どのように達成するか（How）を決定
3つの重要な質問:
  1. このスプリントで何を届けられるか？ - スプリントゴールとバックログアイテムの選択
  2. 選択した作業をどのように完成させるか？ - タスク分解と実装方法の決定
  3. なぜこのスプリントが価値があるのか？ - スプリントゴールの明確化

2.5.3 デイリースクラム（Daily Scrum）

デイリースクラムは、開発チームのための15分のタイムボックスイベントです。

  • タイムボックス: 15分
  • 参加者: 開発チーム（必須）、スクラムマスター、プロダクトオーナー（任意）
  • 時間と場所: 毎日同じ時間・同じ場所
  • 目的: 進捗を検査し、次の24時間の作業を計画する
よくある質問フォーマット（必須ではない）:
  1. 昨日、スプリントゴール達成のために何をしましたか？
  2. 今日、スプリントゴール達成のために何をしますか？
  3. スプリントゴール達成を妨げる障害はありますか？

2.5.4 スプリントレビュー（Sprint Review）

スプリントレビューは、スプリント終了時に開催され、インクリメントを検査しプロダクトバックログを適応させます。

  • タイムボックス: 2週間スプリントの場合、最大2時間
  • 参加者: スクラムチーム全員とステークホルダー
  • 目的: 完成したインクリメントをデモし、フィードバックを得る
レビューの内容:
  • 完成したバックログアイテムのデモンストレーション
  • うまくいったこと、問題点の議論
  • プロダクトバックログの現状レビュー
  • 次の可能性が高いことの予測
  • 市場やプロダクトの使用方法の変化のレビュー

2.5.5 スプリントレトロスペクティブ（Sprint Retrospective）

スプリントレトロスペクティブは、スクラムチームが自分たちを検査し、次のスプリントの改善計画を立てる機会です。

  • タイムボックス: 2週間スプリントの場合、最大1.5時間
  • 参加者: スクラムチーム全員（通常、ステークホルダーは不参加）
  • 目的: プロセス、ツール、関係性を改善する
レトロスペクティブで検査すること:
  • 人、関係性、プロセス、ツールの観点から前スプリントを検査
  • うまくいった項目と改善可能な項目を特定
  • チームの作業方法を改善する計画を立てる

2.6 スクラムの3つの作成物

スクラムの作成物は、作業や価値を表現し、透明性と検査・適応の機会を提供します。

2.6.1 プロダクトバックログ（Product Backlog）

プロダクトバックログは、プロダクトに必要なことすべての順序付きリストです。プロダクトへの変更はすべて、プロダクトバックログの変更を通じて行われます。

  • 責任者: プロダクトオーナー
  • 内容: 機能、機能拡張、バグ修正、技術的改善など
  • 特徴:
    • 常に進化し続ける
    • 優先順位が付けられている
    • 詳細度は上位ほど高い

2.6.2 スプリントバックログ（Sprint Backlog）

スプリントバックログは、スプリントで選択されたプロダクトバックログアイテムと、それを完成させるための計画です。

  • 責任者: 開発チーム
  • 内容: 選択されたバックログアイテム + スプリントゴール + 実装計画
  • 特徴: スプリント中に学習に応じて更新される

2.6.3 インクリメント（Increment）

インクリメントは、スプリント中に完成したすべてのプロダクトバックログアイテムの合計です。インクリメントは、「完成の定義（Definition of Done）」に準拠している必要があります。

  • 要件: 潜在的にリリース可能な状態
  • 品質: 完成の定義を満たしている
  • 累積性: 過去のすべてのインクリメントに今回の成果を加えたもの

理解度確認クイズ
  1. スクラムの3つの柱を説明してください。それぞれの柱がなぜ重要なのかも述べてください。
  2. スクラムの5つの価値観を挙げ、それぞれが実際の開発現場でどのように実践されるか例を挙げてください。
  3. プロダクトオーナーとスクラムマスターの役割の違いを説明してください。
  4. デイリースクラムとスプリントレビューの目的の違いは何ですか？
  5. スプリントバックログとプロダクトバックログの違いを説明してください。
  6. あなたのチームでスクラムを導入する場合、最も課題となりそうなことは何ですか？

2.7 まとめ

この章では、スクラムフレームワークの基礎を学びました。スクラムは、ロール、イベント、作成物というシンプルな構造で構成されていますが、それぞれが密接に関連し合い、効果的なチーム運営を実現します。

重要ポイント
  • スクラムの3つの柱（透明性、検査、適応）は、経験主義に基づく継続的改善を支える
  • 5つの価値観（確約、勇気、集中、公開、尊敬）は、効果的なチーム文化の基盤
  • 3つのロールは明確な責任分担を持ち、相互補完的に機能する
  • 5つのイベントは規則性を生み出し、透明性と適応の機会を提供する
  • 3つの作成物は、作業と価値を可視化し、検査・適応を可能にする
