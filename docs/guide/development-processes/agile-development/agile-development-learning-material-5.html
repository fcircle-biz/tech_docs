5.1 スプリントプランニングの概要

スプリントプランニングは、スプリントの開始時に行う重要なイベントです。チーム全体が集まり、次のスプリントで何を達成するか、どのように達成するかを計画します。

スプリントプランニングの基本情報

項目 詳細
時間 最大8時間（1ヶ月スプリントの場合）
2週間スプリントの場合は4時間程度
参加者 スクラムチーム全員（PO、SM、開発チーム）
タイミング スプリントの初日
成果物 スプリントゴール、スプリントバックログ

5.2 スプリントゴールの設定

スプリントゴール（Sprint Goal）は、スプリントで達成したい目標を簡潔に表現したものです。チーム全体が同じ方向を向くための「北極星」のような役割を果たします。

良いスプリントゴールの特徴

  • 簡潔: 1〜2文で表現できる
  • 価値にフォーカス: 技術的なタスクではなく、ビジネス価値や顧客価値を表現
  • 達成可能: スプリント期間内に実現可能な目標
  • 測定可能: 達成したかどうかを判断できる
  • 柔軟性を持つ: 詳細な実装方法は柔軟に変更可能
  • チームの共感を得る: チーム全員がモチベーションを持てる

スプリントゴールの例

良い例

「新規ユーザーが簡単にアカウント登録し、初回ログインできるようにする」

→ 明確な価値とユーザー体験にフォーカス
悪い例

「ユーザーテーブルを作成し、登録APIを実装する」

→ 技術的タスクで、ユーザー価値が不明確
良い例

「管理者が売上レポートをワンクリックで生成し、経営判断に活用できる」

→ 達成したい成果が明確
悪い例

「5つのストーリーを完了する」

→ ストーリー数だけで、価値が見えない

5.3 キャパシティプランニングとベロシティ

キャパシティプランニングは、チームがスプリント中にどれだけの作業ができるかを計画することです。ベロシティ（過去の実績）を参考にしながら、現実的な計画を立てます。

ベロシティ（Velocity）とは

ベロシティは、チームがスプリントで完了したストーリーポイントの合計です。過去数スプリントの平均ベロシティを使って、次のスプリントの計画を立てます。

graph LR A[スプリント1
完了: 25pt] --> B[スプリント2
完了: 30pt] B --> C[スプリント3
完了: 28pt] C --> D[スプリント4
完了: 27pt] D --> E["平均ベロシティ
(25+30+28+27)/4
= 27.5pt"] E --> F["スプリント5
計画: 約28pt"] style E fill:#ffecb3 style F fill:#d1e7dd

キャパシティの計算方法

チームのキャパシティは、以下の要素を考慮して計算します：

  1. 稼働日数: スプリント期間内の実際の稼働日数
  2. メンバー数: チームメンバーの人数
  3. 休暇・予定: 既に予定されている休暇や会議
  4. フォーカスファクター: 実際に開発に集中できる時間の割合（通常60〜80%）
キャパシティ計算例（2週間スプリント）
  • 稼働日数: 10日（土日除く）
  • チームメンバー: 5人
  • 1日の作業時間: 8時間
  • 休暇: メンバーA が3日休暇
  • 定例会議: 全員が週5時間（計10時間/2週間）

計算:

  • 総時間 = 10日 × 8時間 × 5人 = 400時間
  • 休暇減算 = 3日 × 8時間 = 24時間
  • 会議減算 = 10時間 × 5人 = 50時間
  • 利用可能時間 = 400 - 24 - 50 = 326時間
  • フォーカスファクター70%適用 = 326 × 0.7 = 228時間

結果: チームは約228時間分の作業が可能

5.4 タスク分解とタスクボード

選択したユーザーストーリーを、具体的な技術的タスクに分解します。タスクは、個人またはペアで数時間〜1日程度で完了できるサイズにします。

タスク分解の例

ユーザーストーリー

「オンラインショッパーとして、商品をカートに追加したい。なぜなら、複数の商品をまとめて購入したいからだ。」

分解されたタスク:
  1. データベースにcartテーブルとcart_itemsテーブルを作成（2時間）
  2. 「カートに追加」ボタンのUI実装（3時間）
  3. カート追加APIエンドポイント実装（4時間）
  4. カート内アイテム一覧表示UI実装（4時間）
  5. カート取得APIエンドポイント実装（3時間）
  6. 単体テスト作成（3時間）
  7. 統合テスト実施（2時間）
  8. ブラウザ動作確認（1時間）

タスクボードの構造

タスクボードは、ストーリーごとにタスクを管理します。

graph LR A["ストーリー:
カート追加機能"] --> B[ToDo] B --> C[進行中] C --> D[レビュー] D --> E[完了] B1[タスク1: DB設計] -.-> B B2[タスク2: API実装] -.-> C B3[タスク3: UI実装] -.-> C B4[タスク4: テスト] -.-> D B5[タスク5: 動作確認] -.-> E

5.5 スプリント中の進捗管理

スプリント中は、毎日チームで進捗を確認し、問題があれば早期に対処します。

バーンダウンチャート

バーンダウンチャートは、残作業量をグラフ化したものです。理想線と実績線を比較し、スプリントが順調かを視覚的に把握できます。

バーンダウンチャートの活用

  • 毎日更新: デイリースクラム後に残作業を更新
  • トレンドを見る: 数日の傾向から完了予測を立てる
  • 早期警告: 理想線から大きく乖離したら、スコープ調整を検討
  • 学習ツール: スプリント終了後、見積もりと実績の差を分析
注意: バーンダウンチャートの限界

バーンダウンチャートは進捗を可視化する有用なツールですが、以下の点に注意が必要です：

  • 完了したタスクしか反映されないため、80%完了のタスクも0%と表示される
  • スプリント中にストーリーが追加されると、線が上がることがある
  • 数値だけでなく、チームの状況や品質も考慮する必要がある

5.6 スプリントレビュー

スプリントレビュー（Sprint Review）は、スプリント終了時に開催するイベントです。完成したインクリメントをステークホルダーにデモし、フィードバックを得ます。

スプリントレビューの基本情報

項目 詳細
時間 最大4時間（1ヶ月スプリントの場合）
2週間スプリントの場合は2時間程度
参加者 スクラムチーム、ステークホルダー、顧客
目的 インクリメントの確認、フィードバック収集、バックログの調整
形式 インフォーマル、ライブデモ中心

スプリントレビューのアジェンダ

効果的なデモンストレーションのコツ

  • 実際の環境で動かす: スライドではなく、実際に動作するソフトウェアを見せる
  • ユーザーストーリーに沿って: 技術的な詳細ではなく、ユーザー視点で説明
  • シナリオを用意: 「顧客のAさんがこの機能を使う場合...」とストーリーで説明
  • 完成したものだけ: 80%完成は見せない。完全に動作するもののみデモ
  • 対話的に: 一方的な発表ではなく、質問やフィードバックを歓迎
  • 問題も共有: 失敗や学んだことも正直に共有
演習 5-1: スプリントレビューのシミュレーション

目的: スプリントレビューを模擬的に実施し、デモとフィードバックを経験する

準備
  • チームで簡単なWebアプリや機能を開発（または既存のものを使用）
  • ステークホルダー役を数名決める
  • デモシナリオを準備
手順
  1. プロダクトオーナー役がスプリントゴールを説明（3分）
  2. 開発チームが完成した機能をデモ（15分）
  3. ステークホルダー役が質問やフィードバック（10分）
  4. プロダクトバックログへの反映を議論（5分）
振り返りポイント
  • デモは分かりやすかったか？
  • ステークホルダーから有益なフィードバックが得られたか？
  • 改善すべき点は何か？

5.7 完成の定義（Definition of Done）

完成の定義（DoD: Definition of Done）は、ユーザーストーリーが「完成」したと言えるための基準です。チーム全体で合意し、一貫して適用します。

完成の定義の例

DoD チェックリスト
  • ☑ コードがレビューされ、承認されている
  • ☑ 単体テストが書かれ、すべて合格している
  • ☑ 統合テストが実施され、合格している
  • ☑ 受け入れ基準がすべて満たされている
  • ☑ コードがメインブランチにマージされている
  • ☑ ドキュメント（必要最小限）が更新されている
  • ☑ プロダクトオーナーがレビューし、承認している
  • ☑ ステージング環境にデプロイされ、動作確認済み
  • ☑ パフォーマンステストを通過している（必要に応じて）
  • ☑ セキュリティチェックを通過している

DoDの重要性

  • 品質の保証: 最低限の品質基準を保つ
  • 共通理解: 「完成」の定義をチーム全体で共有
  • 技術的負債の防止: 「後でやる」タスクが溜まるのを防ぐ
  • 透明性: ステークホルダーに対して、完成の基準を明確にする
比較 完成の定義（DoD） 受け入れ基準（AC）
適用範囲 すべてのストーリーに共通 各ストーリー固有
内容 技術的品質基準 機能的要件
定義者 開発チーム（組織レベルもあり） プロダクトオーナーと開発チーム
変更頻度 低い（数ヶ月〜年単位） 高い（ストーリーごと）
アンチパターン: 「ほぼ完成」症候群

DoDが不明確だと、以下のような問題が発生します：

  • 「90%完成」のストーリーが複数残る
  • スプリント終了後に追加作業が必要になる
  • 技術的負債が蓄積する
  • リリース直前にバグが発覚する

解決策: 厳格なDoDを設定し、「完成」していないストーリーは未完了として扱う

理解度確認クイズ
  1. スプリントプランニングの2つのパートは何ですか？それぞれ何を決定しますか？
  2. 良いスプリントゴールの特徴を3つ挙げてください。
  3. ベロシティとは何ですか？どのように計算し、何に使用しますか？
  4. バーンダウンチャートで、実績線が理想線の上にある場合、何を意味しますか？どう対処すべきですか？
  5. スプリントレビューとスプリントレトロスペクティブの違いは何ですか？
  6. 完成の定義（DoD）に含めるべき項目を5つ挙げてください。
  7. あなたのチームで完成の定義（DoD）を作成する場合、どのような項目を含めますか？その理由も説明してください。

5.8 まとめ

この章では、スプリント計画と実行について学びました。効果的な計画と実行により、チームは継続的に価値を提供できます。

重要ポイント
  • スプリントプランニングで「何を」「どうやって」を明確にする
  • スプリントゴールを設定し、チーム全体の方向性を統一する
  • ベロシティを参考にキャパシティを計画し、現実的なコミットをする
  • ストーリーを具体的なタスクに分解し、タスクボードで管理する
  • バーンダウンチャートで進捗を可視化し、早期に問題を発見する
  • スプリントレビューで動作するソフトウェアをデモし、フィードバックを得る
  • 完成の定義（DoD）を設定し、品質基準を保つ
