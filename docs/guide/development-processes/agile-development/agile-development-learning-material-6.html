6.1 デイリースタンドアップとは

デイリースタンドアップ（またはデイリースクラム）は、スクラムチームが毎日同じ時間に集まり、15分以内で情報を共有する短いミーティングです。チームの透明性を高め、障害を早期に発見するための重要なプラクティスです。

デイリースタンドアップの基本情報

項目 詳細
時間 固定15分（延長禁止）
参加者 開発チーム（必須）、SM・PO（任意）
頻度 毎日、同じ時間・同じ場所
形式 立ったまま（スタンドアップ）、ボード前
目的 進捗共有、障害発見、協力促進
なぜ「スタンドアップ」なのか？

立ったままミーティングを行うことで、以下の効果があります：

  • 短時間で終わる: 座ると長くなりがち。立つことで自然に簡潔になる
  • エネルギーレベルが上がる: 朝の眠気を払い、活発な議論が促進される
  • 平等性: 全員が同じ姿勢で、リラックスしすぎず集中できる

6.2 3つの質問

デイリースタンドアップでは、各メンバーが以下の3つの質問に答えます：

flowchart LR A[メンバーA] --> B[1. 昨日
やったこと] B --> C[2. 今日
やること] C --> D[3. 障害・
問題] D --> E[次のメンバーへ]
3つの質問の詳細
  1. 昨日私は何をしたか？（What did I do yesterday?）
    前回のデイリースタンドアップ以降に完了した作業を報告します。スプリントゴールへの貢献を意識して説明します。
  2. 今日私は何をするか？（What will I do today?）
    今日取り組む予定の作業を宣言します。チーム全体の計画との整合性を確認します。
  3. 何か障害があるか？（Are there any impediments?）
    作業を進める上で妨げになっていること、助けが必要なことを共有します。スクラムマスターがこれらの障害を取り除くサポートをします。

回答例

良い例

昨日: ユーザー登録APIの実装を完了し、単体テストも書きました。レビューをお願いしています。

今日: レビューのフィードバックを反映して、ログイン機能のフロントエンド実装に着手します。

障害: 開発環境のデータベースが遅いので、インフラチームに相談したいです。

悪い例（詳細すぎる）

昨日: 朝からユーザー登録の設計を見直して、まずデータベーススキーマを変更しました。その後、バリデーションロジックを実装しようとしたんですが、既存のコードが複雑で...（以下5分続く）

→ 詳細すぎて他のメンバーの時間を奪っている

6.3 タイムボックスの重要性

デイリースタンドアップは厳格に15分以内に終了します。これはタイムボックスと呼ばれる、アジャイルの重要な原則の一つです。

15分を守るためのテクニック

  • タイマーを使う: 見えるところにタイマーを設置し、残り時間を意識する
  • 1人あたり1〜2分: 5人チームなら各自1分半程度を目安にする
  • 詳細は後で: 深い議論が必要な場合は「パーキングロット」に記録し、スタンドアップ後に関係者だけで話す
  • ファシリテーターを立てる: スクラムマスターまたは持ち回りで進行役を決める
  • 同じ順番で回る: 毎回異なると時間がかかるため、ボードの順番やアルファベット順など固定する
  • 時間通りに開始: 遅刻者を待たない。遅れた人は次回から気をつける
pie title デイリースタンドアップの理想的な時間配分 "メンバー5人の報告" : 10 "障害の確認・次のアクション" : 3 "予備時間" : 2

6.4 効果的なデイリースタンドアップの進め方

デイリースタンドアップを効果的にするための実践的なアドバイスです。

ステップバイステップ

flowchart TD A[1. 定刻に開始
遅刻者は待たない] --> B[2. カンバンボード前に集合] B --> C[3. スプリントゴールを確認] C --> D[4. 各メンバーが3つの質問に答える] D --> E[5. 障害をパーキングロットに記録] E --> F[6. 15分以内に終了] F --> G[7. 必要に応じて
関係者だけで追加議論]

ファシリテーションのコツ

  • ボードを指す: 各タスクやストーリーを指しながら話すことで、具体的になる
  • チーム全体に向けて話す: スクラムマスターやマネージャーに報告するのではなく、チームメンバー全員に向けて話す
  • ポジティブな雰囲気を作る: 問題を責めるのではなく、協力して解決する姿勢を示す
  • 視覚的に: ボード上のカードを動かしながら進捗を共有する
  • 「私たち」の言葉を使う: 個人ではなくチーム全体の進捗として捉える

6.5 リモートチームでのデイリースタンドアップ

リモートワークやハイブリッドワークが増える中、デイリースタンドアップをオンラインで効果的に実施する方法を学びます。

リモートでの課題

  • 画面越しでは表情や雰囲気が伝わりにくい
  • 接続トラブルで時間がかかる
  • 全員が同じタイミングで参加しづらい（タイムゾーンの違い）
  • ながら参加（メールチェックしながらなど）が発生しやすい

リモートでの実践方法

対策 具体的な方法
ビデオオン必須 全員がカメラをオンにし、顔を見せる。表情から状況を読み取る
デジタルボード Jira、Trello、Miroなどを画面共有し、全員が同じボードを見る
手を挙げる機能 次に話す人をスムーズに決めるため、ビデオ会議の機能を活用
非同期更新 タイムゾーンが大きく異なる場合、Slackなどで書面更新も併用
接続チェック 開始5分前に接続テストし、トラブルを事前解決
録画 欠席者や後から参加する人のために録画する（任意）
演習 6-1: デイリースタンドアップのロールプレイ

目的: 効果的なデイリースタンドアップを体験し、問題点を発見する

準備
  • チームメンバー5〜7人を集める
  • タイマーを用意（15分にセット）
  • カンバンボードを準備（模擬的なもので可）
手順
  1. スクラムマスター役を1名決める
  2. 各メンバーに架空のタスクを割り当てる
  3. 15分間でデイリースタンドアップを実施
  4. 終了後、5分間で振り返り
振り返りポイント
  • 15分以内に終了できたか？
  • 各メンバーの報告は簡潔だったか？
  • 障害や問題は適切に共有されたか？
  • チーム全体の進捗が把握できたか？
  • 改善すべき点は何か？

6.6 よくあるアンチパターン

デイリースタンドアップでよく見られる失敗パターンと、その回避方法を学びます。

アンチパターン 問題 解決策
報告会になる マネージャーへの報告になり、チーム同士の対話がない チームメンバー同士を向いて話す。マネージャーは見守る立場
長すぎる 詳細な議論が始まり、30分以上かかる タイマー設定。詳細は「パーキングロット」に記録し後で議論
問題解決会議になる 障害の解決策をその場で議論し始める 障害を記録するだけ。解決はスタンドアップ後に関係者だけで
座ったまま 会議室で座り、リラックスしすぎて長引く 立ったまま実施。ボード前で行う
参加しない人がいる 一部のメンバーが欠席、または黙って聞くだけ 全員参加の文化を作る。欠席理由を事前共有
曖昧な報告 「いろいろやりました」など具体性がない ボード上のカードを指して具体的に説明
責任追及の場になる 遅延や問題を責める雰囲気 心理的安全性を確保。問題は協力して解決する姿勢
危険信号：デイリースタンドアップの形骸化

以下の兆候が見られたら、デイリースタンドアップが形骸化しています：

  • メンバーが「特にありません」「変わりません」と繰り返す
  • 毎回同じことを報告している
  • 誰も質問やコメントをしない
  • 障害が共有されない（実際には問題があるのに）
  • 欠席者が増える

対策: レトロスペクティブでデイリースタンドアップの進め方を見直し、チーム全体で改善策を議論する

6.7 チーム内の透明性とコラボレーション

デイリースタンドアップは、チームの透明性を高め、協力を促進します。

透明性がもたらす効果

graph TD A[透明性
情報を隠さない] --> B[早期発見
問題を早く見つける] B --> C[迅速な対応
すぐに助け合える] C --> D[信頼構築
チームの結束] D --> E[高いパフォーマンス
目標達成] E -.継続的改善.- A

コラボレーションを促進する方法

  • 「助けが必要」と言いやすい環境: 弱みを見せることが許される文化
  • 積極的に助け合う: 他のメンバーの障害に対して「私が手伝えます」と声をかける
  • 知識の共有: 専門知識を持つメンバーが他のメンバーに教える機会を作る
  • ペアワークの提案: 難しいタスクはペアやモブで取り組むことを提案

6.8 ペアプログラミングとモブプログラミング

チームコラボレーションを最大化する手法として、ペアプログラミングとモブプログラミングがあります。

ペアプログラミング（Pair Programming）

2人の開発者が1つのコンピュータで協力してコードを書く手法です。

役割分担
  • ドライバー（Driver）: キーボードを操作し、コードを書く
  • ナビゲーター（Navigator）: コード全体を俯瞰し、方向性を指示、レビュー

15〜30分ごとに役割を交代します。

メリット デメリット・課題
  • コード品質の向上
  • バグの早期発見
  • 知識の共有
  • 集中力の維持
  • 設計の改善
  • 疲労度が高い
  • 性格の相性が影響
  • 単純作業には不向き
  • 個人作業時間が減る

モブプログラミング（Mob Programming）

チーム全員（3人以上）が同じ問題に取り組む手法です。全員が1つのスクリーンを見ながら、1人がコードを書きます。

flowchart LR A[ドライバー
1人] --> B[大画面モニター] C[ナビゲーター
チーム全員] -.指示.- A B -.見る.- C D[5-10分で
ドライバー交代] -.-> A

いつペア/モブを使うか

  • 複雑な機能: 難易度が高く、設計判断が重要な場合
  • 新しい技術: チーム全員が学ぶ必要がある新技術導入時
  • 知識伝達: 経験豊富なメンバーから新人への教育
  • 重要な意思決定: アーキテクチャの選択など、後戻りできない決定
  • バグ修正: 難解なバグを複数の視点で解決
理解度確認クイズ
  1. デイリースタンドアップの3つの質問を挙げてください。
  2. デイリースタンドアップが15分に制限されている理由を説明してください。
  3. デイリースタンドアップでよくあるアンチパターンを3つ挙げ、それぞれの解決策を説明してください。
  4. リモートチームでデイリースタンドアップを効果的に実施するための工夫を3つ挙げてください。
  5. ペアプログラミングのドライバーとナビゲーターの役割の違いを説明してください。
  6. あなたのチームでデイリースタンドアップを改善するとしたら、どのような点を変更しますか？理由とともに説明してください。

6.9 まとめ

この章では、デイリースタンドアップとチームコミュニケーションについて学びました。毎日の短い同期が、チームの透明性と協力を大きく向上させます。

重要ポイント
  • デイリースタンドアップは15分以内、毎日同じ時間に実施
  • 3つの質問（昨日・今日・障害）に簡潔に答える
  • チームメンバー同士の対話を促進し、報告会にしない
  • 詳細な議論はパーキングロットに記録し、後で実施
  • リモートでもビデオオン、デジタルボード活用で効果的に実施
  • アンチパターンを認識し、継続的に改善する
  • 透明性を高め、問題を早期発見・早期解決する
  • ペアプログラミング・モブプログラミングで協力を最大化
