3.1 カンバンの起源と基本原則

カンバン（Kanban）は、元々トヨタ自動車が開発した「かんばん方式」に由来します。日本語の「看板」から来ており、作業の流れを視覚的に管理する手法です。2004年にDavid J. Andersonがソフトウェア開発に適用し、アジャイル開発の一手法として広まりました。

トヨタ生産方式からの学び

トヨタのカンバン方式は、以下の原則に基づいています：

  • ジャストインタイム（JIT）: 必要なものを、必要なときに、必要な量だけ生産する
  • プルシステム: 後工程が前工程から必要な分だけ引き取る（プッシュではなくプル）
  • 視覚的管理: 作業の状態を誰でも一目で把握できるようにする
  • 継続的改善（カイゼン）: 小さな改善を積み重ね、常に効率化を図る

カンバンの6つの基本原則

変更管理の3原則
  1. 今やっていることから始める
    現在のプロセスや役割をそのまま受け入れ、段階的に改善します。既存の体制を尊重することで、チームの抵抗を減らします。
  2. 段階的な進化を目指す
    急激な変革ではなく、小さな改善を積み重ねていきます。実験と学習を繰り返し、継続的に進化させます。
  3. 現在のプロセス、役割、責任を尊重する
    既存の役割や責任を否定せず、その上でカンバンを導入します。チームの強みを活かしながら改善します。
サービスデリバリーの3原則
  1. 顧客のニーズと期待を理解する
    顧客が求める価値を明確にし、それを提供することに集中します。
  2. 作業を管理し、セルフオーガナイゼーションを発展させる
    チームが自律的に作業を管理し、問題解決できるようにします。
  3. ポリシーを進化させ、サービスの提供を改善する
    作業ルールやポリシーを定期的に見直し、継続的に改善します。

3.2 カンバンボードの作成と活用

カンバンボードは、作業の流れを視覚化する最も重要なツールです。チーム全員が現在の状況を一目で把握でき、問題を早期に発見できます。

基本的なカンバンボードの構造

カンバンボードは通常、以下のカラム（列）で構成されます：

カンバンカードの要素

各カード（タスク）には以下の情報を含めると効果的です：

  • タスク名: 簡潔でわかりやすいタイトル
  • 担当者: アサインされたメンバー名やアバター
  • 優先度: 色分けやラベルで表示（高・中・低）
  • タスクタイプ: 機能開発、バグ修正、技術的負債など
  • 見積もり: ストーリーポイントや時間（オプション）
  • 締め切り: 期限がある場合（オプション）
  • ブロッカー: 作業を阻害している問題（赤い旗アイコンなど）

3.3 WIP（Work In Progress）制限

WIP制限は、カンバンの最も重要な概念の一つです。各カラムで同時に進行できる作業数を制限することで、マルチタスクを防ぎ、フロー効率を向上させます。

なぜWIP制限が重要なのか

WIP制限のメリット

  • フォーカスの向上: 少数のタスクに集中することで、完了までの時間が短縮される
  • ボトルネックの可視化: 作業が滞留する場所が明確になり、問題を早期発見できる
  • 品質の向上: 急いで多数のタスクをこなすより、少数を丁寧に仕上げる
  • スイッチングコストの削減: タスク切り替えによる時間ロスを最小化
  • チーム協力の促進: WIP制限に達したら、チームで協力して既存タスクを完了させる

WIP制限の設定方法

WIP制限は以下の手順で設定します：

  1. チームサイズを基準にする: まずはチームメンバー数と同じか、それより少し少ない数から始める（例：5人チームなら3〜4）
  2. 実験して調整する: 実際に運用しながら、ボトルネックや待ち時間を観察
  3. カラムごとに設定する: 各工程の特性に応じて異なる制限を設ける
  4. 定期的に見直す: チームの成熟度や状況に応じて調整
カラム 推奨WIP制限（5人チーム） 理由
ToDo（準備完了） 制限なし（またはスプリント分） バックログは制限しないが、準備完了は適度に制限
進行中 3〜4 チーム全員が同時作業するのを防ぐ
レビュー 2〜3 レビュー待ちが溜まらないように制限
完了 制限なし 完了タスクは制限不要
演習 3-1: カンバンボードの作成とWIP制限の設定

目的: チームでカンバンボードを作成し、WIP制限を設定する

手順
  1. ホワイトボードまたはオンラインツール（Trello、Jira、Miroなど）を準備
  2. 作業の流れを表すカラムを作成（例：ToDo、進行中、レビュー、完了）
  3. 現在進行中のタスクをカードとして書き出し、適切なカラムに配置
  4. チームで議論し、各カラムのWIP制限を決定
  5. 1週間運用し、制限に達したときの対応方法を観察
考察ポイント
  • WIP制限に達したとき、チームはどう行動したか？
  • ボトルネックはどのカラムで発生したか？
  • WIP制限を変更する必要があるか？

3.4 フロー効率とリードタイム

カンバンでは、作業の「流れ（フロー）」を最適化することが重要です。フロー効率を測定し、継続的に改善します。

リードタイムとサイクルタイム

  • リードタイム（Lead Time）: 要求が発生してから完了するまでの総時間
    例：顧客が機能をリクエストしてから、その機能が本番環境にリリースされるまで
  • サイクルタイム（Cycle Time）: 作業を開始してから完了するまでの時間
    例：タスクを「進行中」に移してから「完了」になるまで

累積フロー図（Cumulative Flow Diagram）

累積フロー図は、時間経過とともに各ステータスのタスク数を積み上げて表示します。これにより、以下のことがわかります：

  • 各ステータスの作業量の推移
  • ボトルネックの位置（幅が広がっている部分）
  • 全体のフローが安定しているか
  • リードタイムの平均値（縦軸の幅）

3.5 ボトルネックの特定と解消

ボトルネックとは、作業の流れを妨げている制約のことです。ボトルネックを特定し解消することで、全体のスループットが向上します。

ボトルネックの兆候

  • 特定のカラムでタスクが滞留している
  • WIP制限に頻繁に達する
  • 後続の工程が待機状態になっている
  • リードタイムが長くなっている

ボトルネック解消の5つのステップ（制約の理論 - TOC）

解消されたか?} E -->|Yes| F[5. 新しい制約を探す] E -->|No| B F --> A
ボトルネック解消の実践的アプローチ
  1. スキルの共有: ボトルネック工程のスキルを他のメンバーに教える
  2. ペアワーク: 経験豊富なメンバーと協力して作業を進める
  3. 自動化: 繰り返し作業をツールで自動化する
  4. プロセスの簡素化: 不要なステップを削減する
  5. リソースの追加: 必要に応じて専門家を一時的に投入する

3.6 プルシステムの理解

プルシステムは、カンバンの核心的な考え方です。作業を前工程から「押し込む（プッシュ）」のではなく、後工程が必要なときに「引き取る（プル）」方式です。

プッシュシステム vs プルシステム

比較項目 プッシュシステム プルシステム
作業の開始 前工程が完了したら自動的に次へ 後工程が準備できたら要求する
WIP 無制限に増加する可能性 制限されている
在庫 各工程で滞留しやすい 最小限に保たれる
問題の発見 遅れて発覚する 早期に発見できる
柔軟性 低い（計画に依存） 高い（需要に応じる）

3.7 カンバンとスクラムの違い

カンバンとスクラムは、どちらもアジャイル開発の手法ですが、異なるアプローチを取ります。状況に応じて適切な手法を選択することが重要です。

比較項目 スクラム カンバン
イテレーション 固定期間のスプリント（1〜4週間） 継続的なフロー（期間なし）
役割 明確に定義（PO、SM、開発チーム） 規定なし（既存の役割を尊重）
変更 スプリント中は変更不可 いつでも変更可能
計画 スプリントプランニングで一括計画 継続的に優先順位を調整
見積もり ストーリーポイントで見積もり 見積もりは任意
WIP制限 間接的（スプリント容量） 明示的にWIP制限を設定
メトリクス ベロシティ、バーンダウン リードタイム、サイクルタイム
適している状況 機能開発、製品開発 運用保守、サポート業務

3.8 スクラムバン（Scrumban）

スクラムバンは、スクラムとカンバンの良い部分を組み合わせたハイブリッドアプローチです。スクラムの構造とカンバンの柔軟性を併せ持ちます。

スクラムバンの特徴

  • スプリントを採用: スクラムと同様に固定期間のスプリントを使用
  • WIP制限を設定: カンバンのWIP制限を導入
  • ボードを視覚化: カンバンボードで作業を管理
  • 柔軟な優先順位変更: スプリント中でも優先順位を調整可能（制限付き）
  • プランニングオンデマンド: バックログが一定レベルまで減ったときに計画会議を開催
スクラムバンが適している状況
  • スクラムを実践しているが、より柔軟性が必要
  • 緊急対応が多い開発チーム（バグ修正、サポート対応）
  • 継続的デリバリーを目指している
  • 複数の優先度が異なる作業を並行処理する必要がある

3.9 継続的な改善（カイゼン）

カンバンでは、プロセスを継続的に改善することが重要です。定期的にカンバンボードを見直し、以下の点を改善します：

カイゼンの実践方法

  1. 定期的なレビュー会議: 週次または隔週でカンバンボードをレビュー
  2. メトリクスの測定: リードタイム、サイクルタイム、スループットを記録
  3. ボトルネックの特定: データをもとに問題箇所を特定
  4. 実験的な変更: 小さな改善策を試し、効果を測定
  5. 学習の共有: チーム全体で学んだことを共有

演習 3-2: カンバンメトリクスの分析

目的: リードタイムとサイクルタイムを測定し、改善点を見つける

手順
  1. 過去2週間に完了したタスクをリストアップ
  2. 各タスクのリードタイムとサイクルタイムを計算
  3. 平均値、最小値、最大値を算出
  4. 特に時間がかかったタスクの原因を分析
  5. 改善策を3つ提案
分析のポイント
  • どのカラムで最も時間がかかっているか？
  • 待ち時間と作業時間の比率は？
  • タスクのサイズは適切か？（大きすぎないか）
理解度確認クイズ
  1. カンバンの6つの基本原則のうち、変更管理に関する3つの原則を挙げてください。
  2. WIP（Work In Progress）制限を設定する目的は何ですか？3つ挙げてください。
  3. リードタイムとサイクルタイムの違いを説明してください。
  4. プルシステムとプッシュシステムの主な違いは何ですか？
  5. スクラムとカンバンの最も大きな違いを3つ挙げてください。
  6. あなたのチームでカンバンを導入する場合、最初のWIP制限をどのように設定しますか？その理由も説明してください。
