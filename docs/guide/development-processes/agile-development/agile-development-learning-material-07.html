7.1 レトロスペクティブとは

スプリントレトロスペクティブ（Sprint Retrospective）は、スクラムの5つのイベントの中で最も重要なものの一つです。スプリント終了後にチーム全体で振り返り、次のスプリントでの改善策を決定します。

レトロスペクティブの基本情報

項目 詳細
時間 最大3時間（1ヶ月スプリントの場合）
2週間スプリントの場合は1.5時間程度
参加者 スクラムチーム全員（PO、SM、開発チーム）
タイミング スプリントレビュー後、次のスプリントプランニング前
目的 チームのプロセス、人間関係、ツールを検査し、改善する
成果物 改善アクションアイテムのリスト
レトロスペクティブの核心

「どうすればもっと良くなるか？」を全員で考え、実行する

レトロスペクティブは単なる反省会ではありません。チームが協力して、より良い働き方を実験的に試し、学習していく場です。失敗を責めるのではなく、学びの機会として捉えます。

7.2 レトロスペクティブの基本的な進め方

Esther DerbyとDiana Larsenが提唱する5つのフェーズに沿って進めることで、効果的なレトロスペクティブを実施できます。

10-15分] B --> C[3. アイデアを出す
10-20分] C --> D[4. 何をするか決める
10-15分] D --> E[5. レトロスペクティブを
終了する
5分] style A fill:#d1e7dd style C fill:#fff3cd style D fill:#cfe2ff

各フェーズの詳細

1. 場を設定する（Set the Stage）

参加者全員が安心して発言できる雰囲気を作ります。

  • チェックイン: 各自が一言ずつ、現在の気分や状態を共有
  • ゴールの共有: 今回のレトロスペクティブで達成したいことを確認
  • ルールの確認: 心理的安全性、批判禁止、建設的な議論などのルール
チェックイン質問例
  • 「今の気分を天気で表すと？」（晴れ、曇り、雨など）
  • 「このスプリントを一言で表すと？」
  • 「今日のエネルギーレベルは？（1〜10）」
2. データを収集する（Gather Data）

スプリント中に起きた事実、感情、データを集めます。

  • 付箋に思いついたことを書き出す（個人作業）
  • タイムラインを作成し、重要な出来事を配置
  • メトリクス（ベロシティ、バーンダウンなど）を確認
  • 各自が書いた付箋を共有し、ボードに貼る
3. アイデアを出す（Generate Insights）

収集したデータを分析し、パターンや根本原因を探ります。

  • 似た付箋をグループ化（親和図法）
  • 「5つのなぜ」で根本原因を探る
  • チーム全体で議論し、インサイトを得る
  • 投票で重要なテーマを選ぶ
4. 何をするか決める（Decide What to Do）

具体的な改善アクションを決定します。

  • SMART（具体的、測定可能、達成可能、関連性、期限）な目標を設定
  • 責任者を明確にする（誰がやるのか）
  • 実験として小さく始める
  • 2〜3個に絞る（多すぎると実行されない）
5. レトロスペクティブを終了する（Close the Retrospective）

前向きに終わり、次のスプリントへの期待を高めます。

  • 決定したアクションを全員で確認
  • チェックアウト（一言ずつ感想を共有）
  • レトロスペクティブ自体のレトロスペクティブ（メタ振り返り）
  • 感謝の共有

7.3 KPT法

KPTは日本発祥のシンプルで効果的なレトロスペクティブ手法です。3つのカテゴリーで振り返ります。

Keep（良かったこと）

継続したい良い習慣や成功したこと

Problem（問題）

改善が必要な問題や課題

Try（試したいこと）

次のスプリントで実験する改善策

KPT法の進め方

  1. 個人作業（5分）: 各自が付箋にK・P・Tを書き出す
  2. 共有（10分）: 各自が書いた付箋をボードに貼り、簡単に説明
  3. グループ化（5分）: 似た内容をまとめる
  4. 議論（15分）: 特に重要なProblemについて議論
  5. Tryの決定（10分）: 2〜3個の具体的なTryを決める
  6. 次回確認: 前回のTryが実行できたかを最初に確認

KPT例（ECサイト開発チーム）

Keep（良かったこと）
  • ペアプログラミングでバグが減った
  • デイリースタンドアップの時間が15分以内に収まった
  • プロダクトオーナーが迅速にフィードバックしてくれた
  • テストカバレッジが80%を超えた
Problem（問題）
  • 本番環境へのデプロイに時間がかかりすぎる（手動作業）
  • レビュー待ちのプルリクエストが溜まっている
  • スプリント終盤にバグが集中して焦った
  • ドキュメントが古くなっている
Try（試したいこと）
  • 優先度1: デプロイスクリプトを作成し、自動化する（担当: Bob、期限: 次スプリント中）
  • 優先度2: 毎日決まった時間をレビュータイムとして確保する（全員、明日から）
  • 優先度3: スプリント前半にテストを書く文化を定着させる（全員、次スプリントから）

7.4 Start・Stop・Continue法

KPTと似ていますが、より行動にフォーカスした手法です。

カテゴリー 質問 例
Start
（始めること）		 何を新しく始めるべきか？
  • 自動テストの導入
  • 週次の技術共有会
Stop
（やめること）		 何をやめるべきか？
  • 不要な定例会議
  • 個人プレー
Continue
（続けること）		 何を続けるべきか？
  • ペアプログラミング
  • 透明性の高い情報共有

7.5 その他のレトロスペクティブ技法

タイムライン法

スプリントの時系列で重要な出来事を並べ、感情の浮き沈みを可視化します。

timeline title スプリントタイムライン（感情の変化） スプリント1日目 : プランニング成功 : 気分良好 3日目 : 本番障害発生 : 焦り・ストレス 6日目 : ペアプロで解決 : 達成感 9日目 : レビュー準備 : 不安 10日目 : デモ成功 : 満足・安心

5つのなぜ（5 Whys）

問題の根本原因を探るために、「なぜ？」を5回繰り返します。

例: デプロイが遅い
  1. 問題: デプロイに2時間かかる
  2. なぜ? 手動でコマンドを実行しているから
  3. なぜ手動? 自動化スクリプトがないから
  4. なぜスクリプトがない? 作る時間がなかったから
  5. なぜ時間がない? 機能開発に追われているから
  6. なぜ追われている? 技術的負債の返済時間が確保されていないから
  7. 根本原因: スプリント計画時に技術的負債返済の時間を確保していない

改善策: 各スプリントの20%を技術的負債返済に充てるルールを作る

感謝のシャワー

メンバー同士で感謝や称賛を伝え合うことで、チームの絆を強めます。

  • 各メンバーの名前を書いた紙を用意
  • 他のメンバー全員が、その人への感謝や称賛を付箋に書いて貼る
  • 最後に各自が自分宛の感謝カードを読み、受け取る

7.6 心理的安全性

心理的安全性（Psychological Safety）は、レトロスペクティブの成功に不可欠です。Googleの研究「Project Aristotle」で、高パフォーマンスチームの最重要要素として特定されました。

心理的安全性とは

「このチームでは、リスクを取っても安全だ」と感じられる状態

失敗を恐れずに意見を言ったり、質問したり、助けを求めたりできる環境です。批判や報復を恐れることなく、本音で話せることが重要です。

心理的安全性を高める方法

  • プライムディレクティブ: 「全員が自分の知識とスキルを最大限活用し、ベストを尽くしたと仮定する」
  • 批判ではなく好奇心: 「なぜそうしたの？（批判）」ではなく「どういう判断だったの？（好奇心）」
  • 失敗の共有: リーダーが自分の失敗を先に共有し、失敗が許される文化を示す
  • 小さな声を拾う: 発言が少ないメンバーに意見を求める
  • 感謝と承認: 良い行動をすぐに認め、感謝を伝える
  • 守秘義務: レトロで話された内容は外部に漏らさない
心理的安全性が低い兆候
  • 誰も問題を指摘しない、または表面的な問題しか出ない
  • 特定のメンバーだけが発言し、他は黙っている
  • 「でも...」「しかし...」という反論が多い
  • 失敗を隠す、責任を押し付け合う
  • レトロスペクティブ自体を嫌がる

7.7 改善アクションの追跡

決定したアクションアイテムは、必ず追跡し実行します。実行されないアクションは、チームの士気を下げます。

アクションアイテムの管理

flowchart TD A[レトロスペクティブ
でアクション決定] --> B[担当者と期限を明確化] B --> C[スプリントバックログ
または別ボードに追加] C --> D[デイリースタンドアップで
進捗確認] D --> E{完了?} E -->|Yes| F[次のレトロで成果を共有] E -->|No| G[障害を特定] G --> H[サポートや調整] H --> D F --> I[学びを記録]

効果的なアクションアイテムの特徴

  • SMART: 具体的、測定可能、達成可能、関連性、期限
  • 少数: 2〜3個に絞る（多すぎると実行されない）
  • 実験的: 「まず試してみる」マインドセット
  • 小さく始める: 最小限の変更から始める
  • 可視化: ボードに貼って全員が見えるようにする

目的: チームでKPT法を使ったレトロスペクティブを体験する

準備
  • ホワイトボードまたはオンラインツール（Miro、Muralなど）
  • 付箋（物理またはデジタル）
  • タイマー

手順（60分）
  1. 場の設定（5分）: チェックイン、ルール確認

  3. 共有（15分）: 各自が付箋を貼り、説明
  4. グループ化（5分）: 似た内容をまとめる
  5. 投票（5分）: 最も重要なProblemに投票

  • 全員が発言できたか？
  • 決定したTryは実行可能か？
  • 心理的安全性は保たれていたか？

7.8 まとめ

この章では、レトロスペクティブと継続的改善について学びました。レトロスペクティブは、チームが学び成長するための最も重要な場です。

重要ポイント
  • レトロスペクティブはチームの継続的改善の核心
  • 5つのフェーズ（場の設定、データ収集、アイデア出し、決定、終了）で構造的に進める
  • KPT法やStart・Stop・Continue法など、状況に応じた手法を選択
  • 心理的安全性を確保し、全員が本音で話せる環境を作る
  • 改善アクションは2〜3個に絞り、確実に実行する
  • 実験的マインドで小さく始め、学習を重視する
  • 前回のアクションを必ず確認し、継続的な改善サイクルを回す
