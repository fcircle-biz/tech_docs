ファイルの読み書きとコード編集

Codex がファイルをどのように読み、編集するかを学びます。ファイルの参照・新規作成・既存ファイルの編集、差分（変更内容）の確認方法、複数ファイルにまたがる修正の進め方を理解し、AI に編集を任せながら自分で変更を確認・承認する流れを身につけます。

この章で学ぶこと

  • Codex がファイルを読み込んで参照する仕組みを理解する
  • 新規ファイルの作成と既存ファイルの編集を依頼できる
  • 差分（diff）を読んで変更内容を自分で確認・承認する流れを身につける
  • 複数ファイルにまたがる修正を安全に進める考え方を知る

5.1 Codex はどうやってファイルを「読む」のか

Codex に指示を出すと、AI はまずプロジェクト内の関連ファイルを自分で見つけて読み込みます。人間が「このファイルを開いて読む」のと同じように、Codex はファイルの内容を確認してから作業を始めます。

ファイルを読む際に Codex が行うことは主に3つです。

ファイルの検索

指示に関係しそうなファイルを自動的に探します。ファイル名・内容・ディレクトリ構造を手がかりにします。

内容の確認

ファイルを読んで「何が書かれているか」を把握し、変更が必要な箇所を特定します。

依存関係の把握

import・require・include などを辿り、変更の影響が及ぶ範囲を調べます。

用語: 「コンテキストウィンドウ」との関係

AI が一度に扱える情報量には上限（コンテキストウィンドウ）があります。プロジェクトが大きい場合、Codex はすべてのファイルを一度に読むのではなく、指示に関連しそうなファイルを絞り込んで読みます。適切なファイル名・ディレクトリ構成にしておくと、Codex が必要なファイルを見つけやすくなります。

実際のセッションでは、Codex がどのファイルを読んでいるかがターミナルに表示されます。「● src/utils/helper.js を読み込んでいます…」のようなメッセージを確認することで、AI が何を手がかりにしているのかを把握できます。

terminal — ファイル読み込みの様子 
$ codex
> src/auth/login.js のバリデーション処理を確認して

● src/auth/login.js を読み込んでいます…
● src/utils/validator.js を読み込んでいます…（依存ファイル）
● バリデーション処理を確認しました。メールアドレスのチェックが抜けています。

5.2 新規ファイルの作成

「新しいファイルを作って」という指示を出すと、Codex はファイルを新規作成します。ファイル名・配置場所・内容を自然言語で伝えるだけで、適切な場所にファイルが生成されます。

terminal — 新規ファイル作成の例 
$ codex
> src/utils/ に日付フォーマット用のユーティリティファイルを作成して

● src/utils/dateFormatter.js を新規作成します。
  内容: formatDate（Date → 文字列）関数を実装します。
  承認しますか？ [y/n]

承認モードが有効な場合（詳しくは第7章）、Codex はファイルを実際に作成する前に内容を提示して承認を求めます。内容を確認してから「y」を押すことで、意図しないファイルが生まれることを防げます。

指示のコツ: 具体的な配置先を伝える

「ユーティリティを作って」より「src/utils/ 以下に作って」のように、配置先のディレクトリを指定すると、意図した場所にファイルが生成されます。ファイル名も具体的に伝えると確実です。

新規作成後のディレクトリ構成イメージ：

src/
└── utils/
    ├── validator.js       ← 既存ファイル
    └── dateFormatter.js   ← Codex が新規作成したファイル

5.3 既存ファイルの編集

既存ファイルを編集させるには、「〇〇を修正して」「〇〇に〇〇を追加して」のように変更内容を具体的に伝えます。Codex はファイルを読んで変更箇所を特定し、編集案を提示します。

terminal — 既存ファイル編集の例 
$ codex
> src/auth/login.js のバリデーションにメールアドレス形式チェックを追加して

● src/auth/login.js を読み込んでいます…
● 変更案を作成しました。差分を確認してください（下記）。
  承認しますか？ [y/n/d（差分詳細表示）]

Codex は編集前に必ず差分（変更内容）を表示します。どの行が変わるのかを確認してから承認する流れが、安全な使い方の基本です。

用語: 「差分（diff）」とは

差分（diff）とは、ファイルの「変更前」と「変更後」の違いを示したものです。- で始まる行が削除される内容、+ で始まる行が追加される内容を表します。差分を読む習慣をつけると、意図しない変更を見落とさずに済みます。

差分の見方（例）：

diff — src/auth/login.js の変更内容 
// メールアドレスのバリデーション
- if (!email) {
+ if (!email || !/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email)) {
    return { valid: false, message: '無効なメールアドレスです' };
  }

5.4 変更内容の確認・承認フロー

Codex を使う上で最も大切な習慣は、「承認する前に必ず差分を読む」ことです。AI が提案した変更を盲目的に承認すると、意図しないコードが混入するリスクがあります。

flowchart TD A["指示を送る"] --> B["Codexがファイルを読む"] B --> C["変更案と差分を提示"] C --> D{"内容を確認する"} D -->|"問題なし（y）"| E["ファイルに変更を適用"] D -->|"修正が必要（n）"| F["追加指示を出す"] D -->|"詳細確認（d）"| G["差分の詳細を表示"] F --> B G --> D E --> H["変更完了"]

確認時に使える主なキー操作は以下のとおりです。

キー 操作
y 変更を承認してファイルに適用する
n 変更を却下して指示を出し直す
d 差分の詳細（全体）を表示する
Ctrl + C 処理を中断する

注意: 差分を確認せず承認しない

AI の提案が常に正しいとは限りません。差分を確認せずに「y」を押し続けると、意図しない変更がコードに混入する場合があります。慣れてきても、変更前に差分を一読する習慣を維持することが安全な開発の基本です。

5.5 複数ファイルにまたがる修正

実際の開発では、1つの変更が複数のファイルに影響することが珍しくありません。Codex は関連するすべてのファイルをまとめて修正することができます。

たとえば「関数名をリネームして」という指示を出すと、Codex はその関数が定義されているファイルだけでなく、呼び出し側のすべてのファイルも調べて変更案を作成します。

terminal — 複数ファイルにまたがる修正の例 
$ codex
> formatDate 関数を formatDateJa にリネームして

● src/utils/dateFormatter.js を読み込んでいます…
● 呼び出し箇所を検索しています… 3ファイルで使用を確認
● 変更対象: dateFormatter.js, dashboard.js, report.js
  すべての差分を承認しますか？ [y/n/d]

複数ファイルの変更時は、変更対象の一覧を確認してから承認しましょう：

変更対象ファイル（3件）:
  1. src/utils/dateFormatter.js  ← 関数定義を変更
  2. src/pages/dashboard.js      ← 呼び出し箇所を変更
  3. src/pages/report.js         ← 呼び出し箇所を変更

重要ポイント: 一括変更前の確認手順

複数ファイルをまとめて変更する際は、①変更対象のファイル数と名前を確認する、②各ファイルの差分をひと通り確認する（d キーで詳細表示）、③問題がなければ承認する、という3ステップを守ると安全です。不安があれば1ファイルずつ順番に変更するよう指示することもできます。

また、複数ファイルの修正は途中でも中断できます。「まず定義ファイルだけ直して、呼び出し側は後で確認してから変更して」のように指示を分けることも有効です。

5.6 ファイル操作の全体像

この章で学んだファイル操作（読む・新規作成・編集・複数ファイル修正）の流れを図でまとめます。

flowchart LR subgraph 指示 A["あなたの指示"] end subgraph Codexの処理 B["ファイルを検索・読む"] C{"操作の種類"} D["新規作成"] E["既存ファイルを編集"] F["複数ファイルを修正"] end subgraph 確認 G["差分を提示"] H{"あなたが承認"} I["変更を適用"] end A --> B B --> C C -->|"新規"| D C -->|"変更"| E C -->|"横断"| F D --> G E --> G F --> G G --> H H -->|"y"| I H -->|"n"| A

やってみよう 5-1: ファイルの新規作成と編集を体験する

第3章・第4章でセットアップしたプロジェクトを使い、ファイルの作成と編集を実際に試してみましょう。差分を確認してから承認する流れを体に覚えさせることが目的です。

手順

  1. 1 プロジェクトフォルダに移動して codex を起動する。
  2. 2 「utils/hello.js を作成して、名前を受け取って挨拶を返す関数を書いて」と依頼する。
  3. 3 提示された差分（内容）を確認し、問題なければ y で承認してファイルを作成する。
  4. 4 続けて「日本語と英語の両方で挨拶できるように修正して」と依頼し、差分を確認してから承認する。
  5. 5 実際にファイルをエディタで開き、Codex が生成・編集した内容を目視で確認する。

  • 承認前に差分を確認できましたか？
  • 一度 n で却下し、追加指示を出して再提案してもらいましたか？

理解度確認クイズ

  1. Q1. 差分（diff）の表示において、- で始まる行と + で始まる行はそれぞれ何を意味しますか？

    回答を表示

    - で始まる行は「削除される内容」（変更前のコード）を、+ で始まる行は「追加される内容」（変更後のコード）を表します。これを確認することで、どのコードが書き換わるのかを事前に把握できます。

  2. Q2. Codex に複数ファイルにまたがる修正を依頼したとき、承認前にすべきことは何ですか？

    回答を表示

    ①変更対象のファイル数と名前を確認する、②各ファイルの差分をひと通り確認する（dキーで詳細表示）、③問題がなければ承認する、という3ステップを踏むことが大切です。不安があれば1ファイルずつ変更するよう指示を分けることもできます。

  3. Q3. Codex が新規ファイルや既存ファイルを変更する際に、作成・変更が実際に反映されるのはいつですか？

    回答を表示

    承認モードが有効な場合、Codex は変更案と差分を提示した後、あなたが y（承認）を入力したときに初めてファイルに変更が適用されます。承認前にはファイルは変更されないため、安心して差分を確認できます。
