AI支援開発の基礎知識

Codex を使いこなすための前提となる基礎知識を学びます。LLM・トークン・コンテキストウィンドウ・エージェント・ツールといった基本用語を平易に解説し、「読む → 計画する → 編集する → 検証する」というエージェントの基本ループを理解します。

この章で学ぶこと

  • 大規模言語モデル（LLM）が何者で、なぜ「言葉でコーディングを任せられるのか」を説明できる
  • トークンとコンテキストウィンドウの概念を理解し、Codex の動作に与える影響を把握する
  • 補完型・チャット型・エージェント型の3タイプの違いを整理できる
  • 「ツール使用」という仕組みを理解し、エージェントがなぜファイルを読めるのかを説明できる
  • 「読む → 計画する → 編集する → 検証する」というエージェントの基本ループを図で説明できる

2.1 大規模言語モデル（LLM）とは

Codex の頭脳となっているのが、大規模言語モデル（LLM: Large Language Model）です。LLM とは、インターネット上の大量のテキスト（文章・コード・書籍など）を学習し、「次にどんな言葉が続くか」を高精度に予測できる AI モデルです。

「次の言葉を予測する」と聞くと単純に見えますが、この仕組みを超大規模に繰り返すことで、LLM は文章の要約・翻訳・質問応答・コード生成など、非常に多様なタスクをこなせるようになります。Codex が使っている OpenAI のモデル（o4-mini など）は、特にコードを多く学習しており、プログラミング関連のタスクに特化しています。

事前学習済み

大量のデータをあらかじめ学習しているため、特定のプロジェクトに限らず幅広い知識を持っています。プログラミング言語・ライブラリ・パターンを深く理解しています。

自然言語を解釈

「このバグを直して」「テストを書いて」という日本語の指示を理解し、適切なコード操作に変換できます。

LLM は「確率的な文章生成機」

LLM が生成する回答は、確率的に選ばれた言葉の連続です。そのため、同じ質問でも回答が毎回少し変わることがあります。また、「自信を持って間違える」ことも起こります（ハルシネーション）。Codex では検証ステップでこうしたリスクを軽減しています。

2.2 トークンとコンテキストウィンドウ

LLM を使う上で欠かせない概念がトークンとコンテキストウィンドウです。これらは Codex を使っていると「大きなファイルを読めない」「会話が長くなると忘れる」といった現象の根本的な理由になります。

トークン：LLM が扱う最小単位

LLM は文章をトークン（token）という単位に分割して処理します。トークンは「単語よりも細かく、文字よりも大きい」単位で、英語では1単語がほぼ1トークン、日本語では1〜3文字で1トークンになることが多いです。

トークン分割の例（イメージ）

英語の場合

Hello , world !

→ 4トークン

日本語の場合

こん にち は

→ 3〜5トークン（分割方法はモデルにより異なる）

コンテキストウィンドウ：LLM の「作業台」

LLM が一度に「見られる」トークンの上限がコンテキストウィンドウ（context window）です。これは人間でいう「短期記憶」や「机の広さ」のようなもので、作業台に乗せられる情報量には限りがあります。

コンテキストウィンドウには、あなたの指示・会話履歴・Codex が読んだファイルの内容・ツールの実行結果などすべてが入ります。上限を超えると古い情報が削除（圧縮）されるため、非常に長いセッションでは初期の指示が薄れることがあります。

flowchart LR A["あなたの指示"] --> W B["会話の履歴"] --> W C["読み込んだファイル"] --> W D["ツール実行結果"] --> W W["コンテキストウィンドウ
（有限の作業台）"] --> M["LLMが回答を生成"]

実践上のポイント

コンテキストウィンドウが埋まると、Codex は古い情報を要約・削除します。長い作業の途中で AI の応答がずれてきたと感じたら、「前提となる情報を改めて共有する」か「新しいセッションを始める」のが有効です。

2.3 ツール使用（Tool Use）と「エージェント」の仕組み

LLM 単体は「テキストを受け取り、テキストを返す」だけの存在です。しかし Codex では、LLM にツール（tools）が与えられており、「ファイルを読む」「コマンドを実行する」「検索する」といった実際の操作を LLM が指示・実行できます。この仕組みをツール使用（Tool Use）と呼びます。

ツール使用を組み合わせることで、LLM は単なる「回答機」からエージェント（agent） — 自律的に行動を起こして目標を達成できる存在 — へと変わります。Codex はまさにこのエージェントの形で設計されています。

Codex が使うツールの例 できること
ファイル読み込み 指定したファイルの内容をコンテキストに取り込む
ファイル書き込み・編集 ファイルを新規作成・上書き・差分適用する
コマンド実行 シェルコマンドを実行し、結果を受け取る
検索・グレップ コードベース内のパターンを検索する
MCP ツール（拡張） ブラウザ・データベース・外部 API など（第11章）

LLM は「判断」し、ツールが「実行」する

LLM 自身がファイルを直接書き換えるわけではありません。LLM は「このファイルをこのように編集してほしい」という指示を出し、Codex の実行エンジンがそれをツールとして実際に実行します。つまり LLM は「頭脳」、ツールは「手足」の役割を担っています。

2.4 エージェントの基本ループと AI ツールの3分類

第1章でも少し触れましたが、この章の知識を踏まえて、エージェントがどのように動作するかをより詳しく見ていきましょう。

エージェントの基本ループ

Codex は「読む → 計画する → 編集する → 検証する」というループを繰り返して作業を進めます。各ステップで LLM が判断し、必要なツールを呼び出します。

flowchart TD S["あなたの指示"] --> R R["読む
ファイル・コードを読み込む"] --> P P["計画する
何をどの順で行うか決める"] --> E E["編集する
ファイルを書き換える・コマンドを実行"] --> V V["検証する
テスト・コンパイル・確認"] V -->|"問題あり"| R V -->|"完了"| D["変更を報告する"]

AI コーディングツールの3分類

現在の AI コーディング支援ツールは、大きく3つのタイプに分類できます。Codex がエージェント型に属することを、改めて整理しましょう。

補完型

例: GitHub Copilot

  • コードの続きを提案する
  • ファイルの読み書きは人間が行う
  • エディタに常駐

チャット型

例: ChatGPT（ブラウザ版）

  • 質問・相談・コード例の生成
  • 実際の編集・実行は人間が行う
  • ブラウザ／アプリ上のUI

エージェント型

例: Codex・Claude Code

  • 指示から完了まで自律実行
  • ファイル読み書き・コマンド実行を自分で行う
  • ターミナル（CLI）で動作

ループの各ステップで何が起きているかを、簡単なコード例のコメントで確認しましょう。

terminal — エージェントループのイメージ（擬似コード） 
# 1. 指示受け取り
> 「login.js のバグを修正してほしい」

# 2. 読む（ツール: ファイル読み込み）
● login.js を読み込み中...

# 3. 計画する（LLM が判断）
● 有効期限チェックが抜けている。修正計画を立てた。

# 4. 編集する（ツール: ファイル書き込み）
● login.js を修正しました。

# 5. 検証する（ツール: コマンド実行）
● npm test 実行中... 全テスト通過。完了。

この章のまとめ

Codex の裏側では、LLM がトークン単位でテキストを処理し、コンテキストウィンドウ内の情報を参照しながら判断を下しています。そしてツール使用によってファイルの読み書きやコマンド実行が可能になり、「読む → 計画する → 編集する → 検証する」というループでタスクを完了させます。この仕組みを知っていると、Codex の動作が直感的に理解できるようになります。

やってみよう 2-1: 用語を自分の言葉で説明する

この章で学んだ用語を「まだ知らない同僚に説明する」という設定で、自分の言葉で書き出してみましょう。書くことで理解が定着します。

手順

  1. 1 ノートやメモアプリに「LLM・トークン・コンテキストウィンドウ・ツール使用・エージェント」の5つの用語を書く。
  2. 2 各用語について「一言で言うと何か」を書いてみる。本文を見ずにまず書き、後から確認する。
  3. 3 「Codex がファイルを編集できる理由」を、学んだ用語を使って2〜3文で説明してみる。
ヒント: うまく説明できない用語があれば、その箇所を読み直しましょう。説明できるまで理解が深まったということが、次のステップに進むサインです。

理解度確認クイズ

  1. Q1. コンテキストウィンドウが埋まると何が起こりますか？また、実際にそうなったときの対処法を1つ挙げてください。

    回答を表示

    コンテキストウィンドウが埋まると、古い情報（会話履歴・読み込んだファイルなど）が要約・削除されます。その結果、AI の応答が初期の指示とずれてくることがあります。対処法として、「前提情報を改めて伝え直す」か「新しいセッション（codex を再起動）を始める」のが有効です。

  2. Q2. 「ツール使用（Tool Use）」とは何ですか？なぜこれがないとエージェントとして機能しないのか説明してください。

    回答を表示

    ツール使用とは、LLM がファイルの読み書きやコマンド実行などの外部操作を行うために用意された仕組みです。LLM 単体は「テキストを返すだけ」で、実際にファイルを変更することはできません。ツールがあることで初めて「実際に作業を完了させる」エージェントとして機能します。

  3. Q3. 補完型・チャット型・エージェント型の違いを一言ずつで説明してください。

    回答を表示

    補完型 — 「コードの続きを提案するだけ」。チャット型 — 「質問に答えるが、編集・実行は人間が行う」。エージェント型（Codex など）— 「指示を受け取り、ファイルの読み書きやコマンド実行も含めて作業を自律的に完了させる」。
