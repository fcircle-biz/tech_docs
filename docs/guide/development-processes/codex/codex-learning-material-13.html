実践ワークフローとベストプラクティス

日々の開発で Codex を使って成果を出すための実践的な使い方を学びます。効果的なプロンプトの書き方、大きなタスクの分割、計画してから実行する進め方、テスト駆動での活用、レビューの依頼など、現場で役立つ活用パターンとアンチパターンを整理します。

この章で学ぶこと

  • AI に伝わる、効果的なプロンプトの書き方を身につける
  • 大きなタスクを適切に分割し、段階的に進める方法を理解する
  • 「計画してから実行」「テスト駆動」など主要なワークフローパターンを知る
  • よくある失敗（アンチパターン）と、その回避策を理解する

13.1 効果的なプロンプトの書き方

Codex の出力品質は、与えるプロンプト（指示）の質で大きく変わります。「何を」「どこで」「どのように」を明確に書くことが、期待どおりの結果を得る近道です。

悪い例（漠然とした指示）

「バグを直して」

  • どのバグかが不明
  • どのファイル・関数かが不明
  • 期待する動作が不明
良い例（具体的な指示）

「src/auth/login.js の verifyToken 関数で、トークン期限切れのときに例外が発生する。期限切れの場合は false を返すよう修正して」

  • 対象ファイル・関数が明確
  • 現在の問題が明確
  • 期待する動作が明確

良いプロンプトには、次の要素を含めることを意識しましょう。

場所（Where）

対象のファイルパスや関数名を明示する。
例: src/utils/format.js の formatDate 関数

問題（What）

何が起きているか、何が問題かを説明する。
例: タイムゾーンがずれて表示される

期待（Goal）

どうなってほしいかを書く。
例: JST で表示するよう直してほしい

制約（Constraint）

触ってはいけない箇所や方針を伝える。
例: 外部ライブラリは追加しないで

ヒント: AGENTS.md に常時ルールを書く

コーディング規約や使用禁止ライブラリなど、毎回伝えなくてよいルールは AGENTS.md に書いておきましょう（第8章参照）。プロンプトはタスクに集中したシンプルな指示だけになり、ミスが減ります。

13.2 大きなタスクの分割

「新機能を一式実装して」のような大きな依頼を一度に出すと、Codex が誤った方向へ進むリスクが高まります。小さな単位に分割して、1つずつ確認しながら進めることが、品質を保つコツです。

flowchart LR A["大きなタスク
（例: ユーザー管理機能）"] --> B["① DB スキーマ設計"] B --> C["② モデル／リポジトリ実装"] C --> D["③ API エンドポイント実装"] D --> E["④ バリデーション追加"] E --> F["⑤ テスト作成・実行"] F --> G["完成"]

各ステップでコードを確認・承認し、問題があれば次に進む前に修正します。この「小さく進む」サイクルにより、後半での大幅な手戻りを防げます。

避けたい進め方

ユーザー管理機能を全部まとめて作って（DB・API・テスト・フロント・認証も含めて）

→ 方向性のズレが後半まで気づけず、大量の手直しが発生しやすい。

推奨する進め方

まず users テーブルの migration ファイルだけ作って

→ 確認・承認後に次のステップへ。問題を早期に発見できる。

重要ポイント: 承認モードを活用する

Auto モードでは変更が自動適用されますが、大きなタスクの序盤は Read Only モードで計画だけ確認し、納得できたら Auto に切り替えるという使い方も有効です（第7章参照）。

13.3 計画してから実行するワークフロー

Codex にいきなり「実装して」と伝えるより、まず計画を立ててもらい、それを確認してから実行に進む流れにすると、手戻りが大きく減ります。「計画フェーズ」と「実行フェーズ」を意識的に分けましょう。

terminal — 計画から実行への流れ 
$ codex
# ステップ1: まず計画だけ聞く
> パスワードリセット機能を追加したい。実装計画を箇条書きで教えて（まだ実装しないで）

● 計画:
  1. reset_tokens テーブルを追加する
  2. /api/password-reset エンドポイントを実装する
  3. メール送信処理を追加する
  4. トークン検証・パスワード更新処理を実装する

# ステップ2: 計画を確認後、実行を指示
> よさそう。では1の migration ファイルから実装して
1

計画を聞く

「まだ実装しないで」と添えて、アプローチを出してもらう

2

計画を確認する

方針や順序が正しいかを自分でチェック。必要なら修正を依頼

3

実行する

合意した計画に沿って、1ステップずつ実装を進める

13.4 テスト駆動での活用

Codex はテストの作成・実行が得意です。先にテストを書いてもらい、そのテストを通過する実装を依頼する「テスト駆動」の流れは、品質の高いコードを得るうえで特に効果的です。

flowchart TD A["テストを書いてもらう"] --> B["テストを確認・承認する"] B --> C["実装を依頼する"] C --> D["テストを実行する"] D -->|すべて通過| E["完成"] D -->|失敗あり| C
terminal — テスト駆動の例 
$ codex
# まずテストを書いてもらう
> calcTax(price, rate) 関数のテストを tests/tax.test.js に書いて
  （実装はまだしないで）

# テストを確認後、実装を依頼する
> いいね。じゃあそのテストが通る実装を src/utils/tax.js に書いて

# テストを実行させる
> npm test を実行して、全件通ることを確認して

ヒント: 既存テストを守る指示を入れる

新機能を追加するとき「既存のテストを壊さないで」と明示するのがおすすめです。Codex は既存テストのことを忘れがちなため、この一言で予期しないリグレッションを防げます。

13.5 コードレビューの依頼

Codex は実装だけでなく、コードのレビューや品質改善にも役立ちます。「実装して」の後に「レビューして」と続けることで、自分が書いたコードや AI が書いたコードをセルフチェックさせることができます。

セキュリティ観点のレビュー

「src/api/user.js をセキュリティ観点でレビューして。脆弱性があれば指摘して」

リファクタリング提案

「このファイルで読みにくいと思う箇所を指摘して。修正は後で行うから今は提案だけ」

パフォーマンス確認

「この SQL クエリにパフォーマンス上の問題はある？改善案があれば教えて」

PR 説明文の作成

「今回の変更内容を元に PR の説明文（日本語）を書いて」

13.6 よくあるアンチパターンと回避策

Codex を使いはじめると、特定のパターンでうまくいかないことがあります。代表的なアンチパターン（避けたい使い方）と、その回避策を整理します。

アンチパターン1: AI の出力を確認せずにマージ

問題

Codex が出力したコードを読まずに承認・コミットすると、バグや意図しない変更が本番に入るリスクがある。

回避策

必ず diff（差分）を自分の目で確認する。「変更内容を表示して」と依頼して確認後に承認する。

アンチパターン2: 文脈なしで「続きを書いて」

問題

セッションが長くなるとコンテキストが薄れ、「続きを書いて」だけでは AI が意図を誤解しやすい。

回避策

要所でゴールを再確認する。「現在の目標は〜で、次に〜をやりたい」と状況を整理して伝える。

アンチパターン3: 全部お任せで手放し運転

問題

AI が誤った方向で大量のコードを生成し、後で全部やり直しになる。責任のあるコードは自分が理解していないと危険。

回避策

小さなステップで進め、各ステップで確認。AI は「協働パートナー」であり、判断はエンジニアが行う。

アンチパターン4: 機密情報をプロンプトに含める

問題

API キー・パスワード・個人情報などをプロンプトに直接書くと、ログや履歴に残るリスクがある。

回避策

機密情報は環境変数（.env）で管理する。プロンプトでは変数名（API_KEY）で参照させる。

注意: AI の「自信あり」な誤りに気をつける

Codex は誤ったコードを自信満々に出力することがあります（ハルシネーション）。「正しそうに見える」だけで信頼しすぎず、動作確認・テスト・コードレビューを必ず行いましょう。

13.7 日々の開発に組み込むためのヒント

Codex を「たまに使う便利ツール」から「日常の開発の一部」にするためのヒントをまとめます。

朝: タスクの計画立案

今日やることをリストアップして Codex に渡し、実装順序や注意点のアドバイスを聞く。計画の抜け漏れを補完してもらう。

昼: コーディング補助

実装の定型部分（バリデーション・エラーハンドリング・テストケース）を任せ、設計判断やビジネスロジックに集中する。

終業前: コミット＆PR 作成

変更内容を Codex に渡し、コミットメッセージや PR 説明文を作成してもらう。第10章の Git 連携と組み合わせると効果的。

繰り返し作業: 自動化

定期的に行う作業（コードの整形・依存関係の更新確認など）は codex exec でスクリプト化する（第12章参照）。

重要ポイント: Codex はスキルアップの道具でもある

Codex が書いたコードを読んで理解することで、知らなかった書き方や設計パターンを学べます。「完成したからOK」ではなく、なぜそのコードになったのかを理解する習慣をつけることが長期的な成長につながります。

やってみよう 13-1: プロンプトを書き直してみる

実際に自分のプロジェクトを使ってプロンプト改善を体験してみましょう。

手順

  1. 1 今後 Codex に依頼したいタスクを1つ選ぶ。
  2. 2 まず「漠然とした指示」でプロンプトを書く（例: 「機能を追加して」）。
  3. 3 13.1 の「Where・What・Goal・Constraint」の4要素を加えて書き直す。
  4. 4 書き直したプロンプトで「まず計画を立てて」と伝えて Codex に送り、計画が的確かを確認する。

理解度確認クイズ

  1. Q1. 効果的なプロンプトに含めると良い4つの要素は何ですか？

    回答を表示

    ①場所（Where）: 対象ファイル・関数名、②問題（What）: 何が起きているか、③期待（Goal）: どうなってほしいか、④制約（Constraint）: 触ってはいけない箇所・守るべき方針。この4要素を含めることで、AI が意図を正確に理解しやすくなります。

  2. Q2. 「計画してから実行する」ワークフローで、最初のステップはどうすればよいですか？

    回答を表示

    「まだ実装しないで」と添えて、実装計画だけを出してもらいます。計画を自分で確認・修正してから、合意した内容で実装を依頼することで、手戻りを大きく減らせます。

  3. Q3. Codex を使ううえでの代表的なアンチパターンを1つ挙げ、その回避策を説明してください。

    回答を表示

    例: 「AI の出力を確認せずにマージする」→ 必ず diff（差分）を自分の目で確認し、テストが通ることを確かめてから承認・コミットする。ほかにも「文脈なしで指示する」「全部お任せにする」「機密情報をプロンプトに含める」などがある。
