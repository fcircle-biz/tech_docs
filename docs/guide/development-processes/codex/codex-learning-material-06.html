コードベースの探索と理解

既存のプロジェクトを Codex に理解させ、調査に活用する方法を学びます。コード検索・プロジェクト構造の把握・効果的な調査依頼の出し方を習得し、「この関数はどこで使われている？」「この処理の流れを説明して」といった質問を Codex に任せられるようになりましょう。

この章で学ぶこと

  • Codex がコードベースをどのように「読んで理解するか」の仕組みを理解する
  • プロジェクト全体の構造把握を Codex に依頼できる
  • 「この関数はどこで使われている？」など、コード検索の依頼を出せる
  • 「この処理の流れを説明して」など、調査依頼を正確に伝えるコツを身につける
  • 効果的な質問の書き方のパターンを覚える

6.1 Codex はどうやってコードを「理解」するのか

Codex に「このプロジェクトについて教えて」と頼むと、ちゃんと答えが返ってきます。これは Codex がファイルを実際に読むからです。人間と同じように、ディレクトリ構造を確認したり、ファイルの中身を開いて読んだりするためのツールを持っています。

重要なのは、Codex は「魔法的に全部を知っている」わけではないという点です。必要になったときに、必要なファイルを読みに行くという、人間の調査に近い動き方をします。だからこそ、どのファイルや範囲を調べてほしいかを的確に伝えることが大切です。

flowchart TD A["あなたの調査依頼"] --> B["Codexが計画を立てる"] B --> C["ディレクトリ構造を確認する"] C --> D["関連ファイルを読み込む"] D --> E["コードの内容を分析する"] E --> F["わかりやすく説明・報告する"] F -->|追加で調べることがあれば| D

Codex が読めるものと読めないもの

Codex はローカルに存在するテキストファイル（.js・.py・.ts・.md など）を読むことができます。一方、コンパイル済みバイナリ・画像・データベースの中身などは直接読めません。また、読むファイルが多すぎると時間がかかるため、範囲を絞った依頼が効果的です。

6.2 プロジェクト全体の構造を把握する

初めて触るプロジェクトに入ったとき、「どのフォルダに何があるの？」と戸惑うことがあります。Codex にプロジェクト構造の概要を教えてもらうことで、全体像を素早くつかめます。

典型的な依頼の例

terminal — 構造把握の依頼例 
$ codex
> このプロジェクトのフォルダ構成を説明してください。
  どのフォルダに何の役割があるかを、初心者にもわかるように教えてください。

Codex はディレクトリを確認し、主要なフォルダと各ファイルの役割をまとめて説明してくれます。さらに踏み込んで「エントリポイント（プログラムの入口）はどこですか？」のように追加質問することも効果的です。

全体構造の把握

「このプロジェクトのディレクトリ構成を教えて」「主要なファイルを一覧して」

エントリポイントの特定

「プログラムはどこから始まる？」「起動ファイルはどれ？」

レイヤー・役割の分類

「フロントエンドとバックエンドはどう分かれている？」「APIはどこで定義されている？」

設定ファイルの確認

「設定ファイルはどこにある？」「環境変数はどう管理している？」

6.3 コード検索：「この関数はどこで使われている？」

プロジェクトに慣れてくると、「この関数（クラス・変数）はどこで呼ばれているんだろう？」という疑問が出てきます。Codex はこのようなコード検索も得意です。grep コマンドなどを使いながら、ファイルをまたいで参照箇所を探し出してくれます。

terminal — コード検索の依頼例 
> `validateUser` 関数がプロジェクト内のどこで呼ばれているか調べてください。
  呼び出し元のファイル名と行番号も一緒に教えてください。

Codex は関連ファイルを横断的に検索し、「src/auth/middleware.js の12行目から呼ばれています」のように具体的に教えてくれます。検索したい対象が明確なほど、精度の高い回答が得られます。

コード検索のコツ：対象を明確に

「どこかで呼ばれているか調べて」より「validateUser 関数がどこで呼ばれているか調べて」のように、関数名・クラス名・変数名を具体的に指定すると Codex が的確に探せます。ファイルの範囲（例: src/ 以下のみ）も指定するとさらに効果的です。

よく使う検索依頼のパターン

知りたいこと 依頼の例
関数の使用箇所 「sendEmail 関数はどこで呼ばれていますか？」
クラスの継承関係 「BaseController を継承しているクラスをすべて教えて」
特定の文字列の場所 「"TODO:" のコメントが残っているファイルを探して」
importの依存関係 「utils/logger.js をimportしているファイルはどれ？」

6.4 処理の流れを説明させる

コード検索だけでなく、「この処理がどのように動くのか」を Codex に説明させることもできます。複雑な処理フローや、複数ファイルにまたがる動作を把握したいときに非常に役立ちます。

terminal — 処理フロー説明の依頼例 
> ユーザーがログインボタンを押してから、
  セッションが確立されるまでの処理の流れを
  ファイルとコードを追いながら説明してください。

このように「〇〇が起きてから、△△になるまで」という形で起点と終点を指定すると、Codex は途中のすべてのステップ（関数呼び出し・ファイル間の連携・状態の変化など）を追って説明してくれます。

flowchart LR A["起点を指定
（例: ボタンを押したとき）"] --> B["経路を追う
（Codexが各ファイルを読む）"] B --> C["終点まで解説
（例: DBに保存されるまで）"]

「初心者向けに」を追加すると効果的

依頼の末尾に「初心者にもわかるように説明してください」「技術用語には補足をつけてください」と加えると、Codex は専門的な言い回しを避けてわかりやすく説明してくれます。慣れてきたら「簡潔に」「箇条書きで」といった出力形式の指定も有効です。

6.5 効果的な質問のコツ

Codex の調査能力を最大限に引き出すには、質問の出し方が重要です。漠然とした質問より、具体的な情報を含んだ質問の方が、精度の高い回答が得られます。

あまりよくない質問

  • 「このコードを説明して」
  • 「バグはどこ？」
  • 「どうなってるか教えて」

対象が広すぎて、Codex も何を調べればよいか判断しにくい。

よい質問

  • 「src/api/users.js の getUser 関数を説明して」
  • 「ログイン後にリダイレクトが起きないバグの原因を auth/ フォルダで調べて」
  • 「order.js で注文確定後にメールが送られる流れを説明して」

ファイル名・関数名・範囲が具体的で、Codex が的確に調べられる。

効果的な質問に含める要素

1

対象を絞る

ファイル名・関数名・クラス名・フォルダ名を具体的に指定する。

2

目的を伝える

「バグを直すために流れを知りたい」「新機能を追加するために構造を把握したい」など、なぜ調べるのかを添える。

3

出力形式を指定する

「箇条書きで」「フロー図で」「表形式で」のように、回答の形式を指定するとわかりやすい結果が得られる。

4

前提知識を共有する

「Node.js の Express を使っているプロジェクトで」のように、技術スタックや背景情報を添えると Codex の理解が深まる。

注意: Codex の回答は必ず確認する

Codex の調査結果は非常に参考になりますが、すべてが正確とは限りません。特に大規模なプロジェクトや、設定に依存する挙動については、必ず自分でもコードを確認する習慣をつけましょう。「この説明は正しそう？」と Codex 自身に検証を依頼することも有効です。

6.6 調査をそのまま作業につなげる

Codex のすぐれている点は、調査を単なる説明で終わらせず、そのまま作業に移れることです。「この関数の使われ方を説明して」と調べた後、「じゃあ、この部分をリファクタリングして」と続けても、Codex はすでにコードを読んでいるので、スムーズに次の作業に取り掛かれます。

terminal — 調査から作業への連携例 
$ codex
> `formatDate` 関数がどこで使われているか調べてください。

● src/utils/date.js, src/components/EventCard.jsx,
  src/pages/calendar.js の3箇所で使われています。

> ありがとう。タイムゾーン対応のために第2引数を追加したいので、
  3箇所すべてを更新してください。

調査と編集をセットで考える

「まず調べる → 次に変更する」というセットで依頼することで、Codex は文脈を理解したまま作業を進められます。大きな変更を加える前に必ず調査を挟む習慣をつけると、ミスが減り安全に開発できます。

やってみよう 6-1: 手元のプロジェクトを Codex に探索させる

第3・4章でセットアップした Codex を使って、実際のプロジェクト（自分の既存コードや練習プロジェクト）を探索してみましょう。

手順

  1. 1 任意のプロジェクトフォルダで codex を起動する。
  2. 2 「このプロジェクトのフォルダ構成と主な役割を教えてください」と依頼して、説明を受け取る。
  3. 3 気になる関数やクラスを一つ選んで「〇〇はどこで使われている？」と検索を依頼する。
  4. 4 Codex の回答と実際のコードを見比べて、説明が正確かどうかを自分でも確認する。

練習プロジェクトがない場合

GitHub 上の有名なオープンソースプロジェクト（例: expressjs/express）を git clone して使ってみましょう。実際の規模のプロジェクトで練習すると理解が深まります。

理解度確認クイズ

  1. Q1. Codex がコードを「理解する」仕組みとして正しいものはどれですか？

    回答を表示

    Codex は「すべてを最初から知っている」わけではなく、必要になったときに実際のファイルを読みに行くことでコードを理解します。ディレクトリ構造の確認・ファイルの読み込み・内容の分析というステップを踏みます。

  2. Q2. 「この関数がどこで呼ばれているか調べて」という依頼をより効果的にするには、どうすればよいですか？

    回答を表示

    関数名を具体的に指定する（例: validateUser 関数）こと、さらに調べる範囲（フォルダ）を絞り込むことが効果的です。「呼び出し元のファイル名と行番号も教えて」のように、出力形式を指定するとさらに使いやすい回答が得られます。

  3. Q3. Codex に処理の流れを説明させるとき、質問に含めると効果的な「起点と終点」とはどういう意味ですか？例を挙げて説明してください。

    回答を表示

    処理の「始まり（起点）」と「終わり（終点）」を明確に指定することです。例えば「ユーザーがログインボタンを押した時点（起点）から、セッションが確立されるまで（終点）の流れを説明して」のように指定すると、Codex はその間のすべてのステップを追って説明してくれます。
