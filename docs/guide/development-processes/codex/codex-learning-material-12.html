非対話実行と自動化（codex exec）

対話 UI を開かずに、1 つのコマンドで Codex にタスクを実行させる「非対話モード」を学びます。スクリプトや CI（継続的インテグレーション）に組み込むことで、繰り返し作業を仕組み化して自動化する考え方と実践を理解しましょう。

この章で学ぶこと

  • codex exec コマンドの基本構文と非対話モードの仕組みを理解する
  • --full-auto などのフラグを使って承認を省略する方法を知る
  • シェルスクリプトに codex exec を組み込んで繰り返し作業を自動化できる
  • GitHub Actions などの CI パイプラインへ Codex を統合する基本的な考え方を説明できる
  • 非対話実行の出力を受け取り、次の処理へ渡す方法を理解する

12.1 非対話モードとは何か

Codex を起動すると、通常はターミナル上に対話 UI（TUI）が開き、あなたが指示を入力しながらやり取りを続けます。これが対話モードです。一方、非対話モードとは、TUI を一切開かずに、コマンド1行でタスクを与えて実行し、結果を受け取るスタイルです。

非対話モードのメインコマンドが codex exec です。「exec」は execute（実行する）の略で、「指定したプロンプトを非対話的に実行する」という意味をもちます。

対話モード（codex）

  • TUI が開き、やり取りを繰り返す
  • 途中で確認・修正の指示が出せる
  • 探索的なタスク・調査に向いている

非対話モード（codex exec）

  • TUI を開かずコマンド1行で完結
  • スクリプトや CI から呼び出せる
  • 繰り返し・定型タスクの自動化に向いている

用語: CI（継続的インテグレーション）とは

CI（Continuous Integration）とは、コードのプッシュやプルリクエストのタイミングで、ビルド・テスト・チェックなどを自動実行する仕組みです。GitHub Actions などのサービスがよく使われます。codex exec を CI に組み込むと、「コードレビューを AI に依頼する」「コーディング規約の違反を自動修正する」といった作業を人手なしで流せます。

12.2 codex exec の基本構文

codex exec の基本的な使い方を見ていきましょう。最もシンプルな形は次のとおりです。

terminal — 基本構文 
# 基本形: プロンプトを引数として渡す
codex exec "README.md に使い方のセクションを追加して"

# --full-auto フラグ: すべての操作を承認なしで自動実行
codex exec --full-auto "テストを実行して失敗箇所をすべて修正して"

ダブルクォートの中に普通の日本語や英語でプロンプト（指示）を書くだけです。Codex はそれを読み取り、必要なファイルの読み書きやコマンド実行をすべて自動で行います。

フラグ 意味 使いどころ
--full-auto ファイル編集・コマンド実行をすべて自動承認 CI や信頼済みスクリプト内での完全自動実行
--model 使用モデルを指定 タスクに応じて高速モデル／高性能モデルを切替
--profile config.toml のプロファイルを指定 CI 用・開発用など環境ごとに設定を切り替える
--output 出力形式を指定（text / json） 後続スクリプトで結果を解析したいとき
--no-sandbox サンドボックスを無効化 ネットワークアクセスが必要なタスク（要注意）

注意: --full-auto の使いどころ

--full-auto は便利ですが、ファイルの削除・上書き・コマンド実行を確認なしで行うため、誤った指示を渡すと予期しない変更が起こる可能性があります。CI 環境のように変更がすぐに本番に影響しない隔離された場所や、事前に動作を確認済みのスクリプトで使うようにしましょう。

12.3 出力の受け取り方

codex exec の実行結果は、通常のコマンドと同じく標準出力（stdout）に書き出されます。これをリダイレクトやパイプで受け取れるため、シェルスクリプトの一部として自然に組み込めます。

terminal — 出力の受け取り方 
# 出力をファイルに保存する
codex exec "src/ 以下のコード品質をレポートして" > report.txt

# 出力を変数に格納して条件分岐に使う
RESULT=$(codex exec --output json "テストを実行して結果を返して")

# 終了コード（0=成功, 非0=失敗）で次の処理を制御する
codex exec --full-auto "lint エラーをすべて修正して" && echo "修正完了"

重要ポイント: 終了コード

codex exec はタスクが正常完了すると終了コード 0 を返し、失敗すると 1 以上の値を返します。シェルスクリプトや CI の後続ステップでこの値を使うことで「修正成功ならコミット、失敗なら通知」といった制御ができます。

terminal — JSON 出力を jq でパース 
# --output json で構造化された結果を取得
# jq コマンドで特定フィールドを抜き出す
codex exec --output json "依存パッケージの脆弱性を調べて" \
  | jq '.summary'

12.4 シェルスクリプトへの組み込み

codex exec は普通のコマンドなので、既存のシェルスクリプト（.sh ファイル）の中に自由に書けます。これにより、「毎朝コードをチェックして結果をスラック通知する」「ファイルが追加されたらドキュメントを自動更新する」といった繰り返し作業を仕組み化できます。

daily-check.sh — 日次コード品質チェック 
#!/bin/bash
# 毎日実行するコード品質チェックスクリプト

# 1. lint エラーを自動修正（--full-auto で承認なし）
codex exec --full-auto "ESLint のエラーをすべて修正して"

# 2. 修正があればコミットする
if ! git diff --quiet; then
  git add -A
  git commit -m "fix: AI による lint 自動修正"
  echo "修正をコミットしました"
fi

ヒント: cron で定期実行する

Linux / macOS では cron や launchd を使ってシェルスクリプトを定期実行できます。たとえば 0 9 * * 1-5（平日毎朝9時）に上記スクリプトを実行するように設定すれば、毎朝コードが自動でチェックされます。

12.5 CI パイプラインへの組み込み（GitHub Actions）

codex exec の強力な活用先のひとつが、GitHub Actions などの CI パイプラインです。プルリクエストが作成されるたびに Codex がコードをレビューしてコメントを残したり、テストが失敗したら自動修正を試みたりする、といったワークフローが実現できます。

CI パイプラインでの codex exec 自動化フロー

flowchart TD A["プルリクエスト作成"] --> B["GitHub Actions 起動"] B --> C["codex exec でコードレビュー"] C --> D{"問題あり？"} D -->|なし| E["レビュー OK コメントを投稿"] D -->|あり| F["codex exec --full-auto で自動修正"] F --> G["修正コミットをプッシュ"] G --> H["CI を再実行"] E --> I["マージ可能な状態へ"] H --> I
.github/workflows/codex-review.yml 
name: Codex AI レビュー

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  ai-review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # Codex CLI をインストール
      - name: Install Codex CLI
        run: npm install -g @openai/codex

      # 非対話モードでコードレビューを実行
      - name: AI コードレビュー
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
        run: |
          codex exec --full-auto \
            "変更差分を確認してバグや問題点を日本語でまとめて"

重要ポイント: API キーの扱い

CI 環境で Codex を使う場合、OpenAI API キー（OPENAI_API_KEY）を環境変数として渡す必要があります。GitHub Actions では「リポジトリの Secrets」に登録し、${{ secrets.OPENAI_API_KEY }} で参照するのが安全な方法です。絶対にコードやログに直接書かないようにしましょう。

ヒント: CI 専用プロファイルを作る

第9章で学んだ config.toml のプロファイル機能を使うと、CI 向けの設定（承認モード・モデル・サンドボックス）をまとめておき、--profile ci で切り替えられます。開発環境と CI 環境で設定を分けることで、誤操作を防ぎつつ自動化をスムーズに進められます。

12.6 自動化設計の考え方

codex exec を使った自動化を設計するときは、次の3ステップで考えるとうまくいきます。

1

繰り返しを見つける

毎日・毎週必ず行う定型タスク（lint・テスト・ドキュメント生成など）を書き出す。

2

プロンプトを固める

まず対話モードで同じ指示を試し、期待どおりに動くプロンプトが定まってから codex exec に切り替える。

3

段階的に自動化する

最初は出力確認ありで運用し、信頼できると分かってから --full-auto に移行する。

自動化のポイントは「まず対話で磨いてから非対話に移す」という段階的アプローチです。対話モードで何度か試してプロンプトが安定したら、それをそのまま codex exec に渡すだけで自動化が完成します。

auto-docs.sh — ドキュメント自動更新スクリプト 
#!/bin/bash
# src/ の変更を検知してドキュメントを自動更新

# 変更があった場合のみ実行する
if git diff --name-only HEAD~1 | grep -q '^src/'; then
  echo "ソース変更を検知。ドキュメントを更新します..."
  codex exec --full-auto \
    "src/ の変更に合わせて docs/API.md を最新化して"
fi

やってみよう 12-1: codex exec を試してみる

手元の任意のプロジェクトで codex exec を実行し、非対話モードの動作を体験しましょう。

手順

  1. 1 ターミナルで任意のプロジェクトフォルダに移動する（cd my-project）。
  2. 2 次のコマンドを実行してみる: codex exec "このプロジェクトの概要を3行で説明して"
  3. 3 TUI が開かずに直接出力が返ってきたことを確認する。
  4. 4 次に > summary.txt を付け加え、結果をファイルに保存してみる。

発展: 上記のコマンドをシェルスクリプトファイル（check.sh）に書き込み、chmod +x check.sh && ./check.sh で実行してみましょう。自動化の第一歩が体験できます。

理解度確認クイズ

  1. Q1. codex exec と通常の codex コマンドの最大の違いは何ですか？

    回答を表示

    codex exec は対話 UI（TUI）を開かずに、コマンド1行でタスクを実行して結果を出力する非対話モードです。通常の codex はターミナル上に対話画面が開き、やり取りを繰り返しながら作業を進めます。codex exec はスクリプトや CI から呼び出せる点が大きな違いです。

  2. Q2. --full-auto フラグを使う際に注意すべきことは何ですか？

    回答を表示

    --full-auto はファイルの編集・削除・コマンド実行をすべて確認なしで自動実行します。誤ったプロンプトを渡すと予期しない変更が起こる危険があります。信頼できる CI 環境や、事前に対話モードで動作確認済みのスクリプトでのみ使うことが推奨されます。

  3. Q3. codex exec を CI に組み込む際、API キーはどのように管理すべきですか？

    回答を表示

    API キー（OPENAI_API_KEY）はコードやログに直接書かず、GitHub Actions であれば「リポジトリの Secrets」に登録し、${{ secrets.OPENAI_API_KEY }} という形で環境変数として参照します。これにより、キーが外部に漏れるリスクを最小限に抑えられます。
