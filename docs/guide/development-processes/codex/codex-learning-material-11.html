MCPによる外部ツール連携

Model Context Protocol（MCP）は、AI エージェントが外部のツールやデータソースと標準化された方法でやり取りするための仕組みです。この章では、MCP の概念を理解し、config.toml への MCP サーバーの追加方法、データベース・外部 API・ブラウザ操作など、Codex の能力を大きく広げる具体的な活用例を学びます。

この章で学ぶこと

  • Model Context Protocol（MCP）とは何かを理解する
  • config.toml の [mcp_servers.xxx] セクションで MCP サーバーを追加できる
  • データベース・外部 API・ブラウザ操作など、代表的な MCP 活用例を説明できる
  • MCP を安全に使うためのポイントを理解する

11.1 MCP とは何か

Model Context Protocol（MCP）とは、AI エージェントが外部のツールやデータソースと通信するための共通の規格（プロトコル）です。Anthropic が提案し、現在は多くの AI エージェントツールに採用されています。Codex もこの MCP に対応しており、サードパーティが提供する多様な「MCP サーバー」を接続することで、標準機能では届かない領域まで Codex の能力を広げられます。

MCP が生まれた背景として、各ツールが独自の連携方法を持つと「AI ごとに設定方法が異なる」「同じデータベースにアクセスするのにツールごとにアダプターを作らなければならない」という問題がありました。MCP はこれを解決する共通言語です。一度 MCP サーバーを作れば、MCP に対応したどの AI エージェントからも利用できます。

共通規格

ツールごとに異なる連携コードを書く必要がなく、MCP 対応サーバーならどれでも同じ方法で接続できます。

能力の拡張

DB アクセス・Web 検索・ブラウザ操作など、Codex 単体にはない機能を追加できます。

エコシステム

公式・サードパーティ問わず多数の MCP サーバーが公開されており、すぐに活用できます。

用語: 「プロトコル」とは

「プロトコル」とは、複数のシステムが通信するときに守る取り決め・規約のことです。たとえば Web サイトの表示に使われる「HTTP」も一種のプロトコルです。MCP も同様に、AI エージェントと外部ツールが「どんな形式でやり取りするか」を定めた規格です。

11.2 MCP の仕組みと構成図

MCP の全体像を図で確認しましょう。Codex（MCP クライアント）が MCP サーバーを介して外部リソースにアクセスします。Codex 自身は「何のサーバーがあるか・どんな操作ができるか」を MCP サーバーから自動的に取得し、必要なときにそのサーバーへリクエストを送ります。

flowchart LR A["あなた\n（ユーザー）"] -->|"自然言語で指示"| B["Codex\n（MCPクライアント）"] B -->|"MCPプロトコル"| C["MCPサーバーA\nデータベース"] B -->|"MCPプロトコル"| D["MCPサーバーB\n外部API"] B -->|"MCPプロトコル"| E["MCPサーバーC\nブラウザ操作"] C -->|"結果を返す"| B D -->|"結果を返す"| B E -->|"結果を返す"| B B -->|"回答・作業結果"| A

重要なのは、Codex（クライアント側）が各外部ツールの詳細な実装を知らなくても、MCP という共通の規格でやり取りできる点です。MCP サーバーは独立したプロセスとして動作し、Codex とは JSON ベースのメッセージでやり取りします。

MCP クライアント（Codex 側）

  • どの MCP サーバーが使えるかを設定ファイルから読み込む
  • ユーザーの指示に応じて必要なサーバーへ要求を送る
  • サーバーからの結果を受け取り、回答や作業に活用する

MCP サーバー（ツール側）

  • 提供できるツール・機能の一覧を Codex に伝える
  • Codex から受け取った要求を実際のリソース（DB・API）に伝える
  • 処理結果を MCP プロトコルの形式で Codex へ返す

11.3 config.toml への MCP サーバー追加方法

MCP サーバーを Codex に認識させるには、第9章で学んだ設定ファイル ~/.codex/config.toml に設定を追記します。書き方のルールはシンプルで、[mcp_servers.サーバー名] というセクションを追加し、その下に起動コマンドなどを指定するだけです。

基本的な書き方

~/.codex/config.toml — MCP サーバーの基本設定 
# [mcp_servers.サーバー名] でサーバーを追加する
# サーバー名は自分で分かりやすい名前を付ける

[mcp_servers.my_database]
# コマンド: MCPサーバーを起動するプログラム
command = "npx"
# 引数: コマンドに渡すオプション（リスト形式）
args = ["-y", "@modelcontextprotocol/server-sqlite", "/path/to/db.sqlite"]

[mcp_servers.web_search]
command = "npx"
args = ["-y", "@modelcontextprotocol/server-brave-search"]
# 環境変数: APIキーなど秘密情報はここで渡す
env = { BRAVE_API_KEY = "YOUR_API_KEY_HERE" }
設定キー 必須 説明
command 必須 MCP サーバーを起動するコマンド（例: npx, python）
args 任意 コマンドへの引数をリスト形式で指定
env 任意 サーバーに渡す環境変数（API キーなど）をテーブル形式で指定

重要ポイント: 設定後に Codex を再起動

config.toml を編集したら、Codex を一度終了して再起動することで設定が反映されます。設定が正しく読み込まれると、Codex のセッション開始時に「利用可能なツール」として MCP サーバーが表示されます。

11.4 代表的な MCP 活用例

MCP によって、Codex はどんなことができるようになるのかを具体的に見ていきましょう。以下は代表的な活用パターンです。

パターン 1: データベースへのアクセス

SQLite や PostgreSQL などのデータベースに接続する MCP サーバーを追加すると、Codex は自然言語でデータを調べたり、テーブル構造を理解した上でコードを書いたりできます。

~/.codex/config.toml — SQLite MCP サーバーの例 
# SQLite データベース用の MCP サーバー
[mcp_servers.sqlite_db]
command = "npx"
# -y: インストール確認をスキップ、第2引数はDBファイルのパス
args = ["-y", "@modelcontextprotocol/server-sqlite", "~/projects/myapp/data.db"]

設定後にできること（Codex への指示例）:

  • 「users テーブルの構造を教えて」
  • 「今月登録したユーザー数を調べて」
  • 「このテーブル構造に合わせたマイグレーションファイルを作って」

パターン 2: 外部 API・Web 検索

Web 検索 API や GitHub API などに接続する MCP サーバーを使うと、Codex が最新情報を調べながらコードを書いたり、Issue を参照しながら修正を進めたりできます。

~/.codex/config.toml — GitHub MCP サーバーの例 
# GitHub 連携用 MCP サーバー
[mcp_servers.github]
command = "npx"
args = ["-y", "@modelcontextprotocol/server-github"]
# GitHub トークンを環境変数で渡す（リポジトリアクセス権限）
env = { GITHUB_PERSONAL_ACCESS_TOKEN = "ghp_XXXXXXXXXXXX" }

設定後にできること（Codex への指示例）:

  • 「Issue #42 の内容を確認して対応するコードを書いて」
  • 「このリポジトリの最近のプルリクエストを一覧表示して」

パターン 3: ブラウザ操作・スクレイピング

Playwright や Puppeteer ベースの MCP サーバーを使うと、Codex が実際にブラウザを操作して Web ページを取得したり、E2E テストのシナリオを書いて実行したりできます。

~/.codex/config.toml — Playwright MCP サーバーの例 
# Playwright ブラウザ操作用 MCP サーバー
[mcp_servers.browser]
command = "npx"
# ヘッドレスブラウザで Web ページの取得・操作が可能になる
args = ["-y", "@playwright/mcp"]

設定後にできること（Codex への指示例）:

  • 「このページの構造を取得して、必要なデータを抽出するコードを書いて」
  • 「ログインフォームへの入力から送信までの E2E テストを作成して」

パターン 4: ファイルシステム拡張

公式が提供するファイルシステム MCP サーバーを使うと、Codex がアクセスできるディレクトリを明示的に制御できます。プロジェクト外のフォルダへのアクセスを管理したいときに役立ちます。

~/.codex/config.toml — ファイルシステム MCP の例 
# ファイルシステムアクセス用 MCP サーバー
[mcp_servers.filesystem]
command = "npx"
# アクセスを許可するディレクトリをパスで指定する
args = ["-y", "@modelcontextprotocol/server-filesystem", "~/projects/myapp"]

11.5 MCP サーバーを探す方法

どんな MCP サーバーが使えるのかは、公式リポジトリや NPM を検索することで調べられます。

公式リファレンスサーバー

Anthropic が管理する公式リポジトリに、よく使われる MCP サーバーのリファレンス実装がまとまっています。

github.com/modelcontextprotocol/servers

NPM パッケージ検索

NPM で @modelcontextprotocol や mcp-server で検索すると多数のサーバーが見つかります。

npmjs.com で "mcp-server" を検索

注意: サードパーティ MCP サーバーの信頼性

サードパーティが公開している MCP サーバーは、公式に検証されているわけではありません。インストールする前に、リポジトリのスター数・メンテナンス状況・コードの中身を確認する習慣をつけましょう。特に API キーやパスワードを扱うサーバーは、信頼できるソースのものだけを使用してください。

11.6 複数の MCP サーバーを組み合わせる

複数の MCP サーバーを同時に設定することもできます。Codex は指示の内容に応じて、必要なサーバーを自動的に選んで使います。下記は実践的な組み合わせ例です。

~/.codex/config.toml — 複数 MCP サーバーの設定例 
# データベース: プロジェクトのローカル DB
[mcp_servers.app_db]
command = "npx"
args = ["-y", "@modelcontextprotocol/server-sqlite", "~/projects/app/db.sqlite"]

# GitHub: Issue・PR の参照
[mcp_servers.github]
command = "npx"
args = ["-y", "@modelcontextprotocol/server-github"]
env = { GITHUB_PERSONAL_ACCESS_TOKEN = "ghp_XXXXXXXXXXXX" }

# ブラウザ: E2Eテストや Web 情報の取得
[mcp_servers.browser]
command = "npx"
args = ["-y", "@playwright/mcp"]

この設定で、Codex は「DB を調べて GitHub Issue を参照しながら、ブラウザで動作確認する」という複合的なタスクを一度の指示でこなせるようになります。

Tips: プロファイルで MCP を使い分ける

第9章で学んだプロファイル機能と組み合わせると、「開発用プロファイルには DB ＋ GitHub」「調査用プロファイルには Web 検索のみ」のように、目的に応じて利用可能なツールを切り替えられます。不要な権限を持つサーバーを常時起動しないことはセキュリティ上も望ましい習慣です。

やってみよう 11-1: SQLite MCP サーバーを試してみる

実際に SQLite 用 MCP サーバーを設定して、Codex からデータベースへ問い合わせてみましょう。

手順

  1. 1 テスト用の SQLite データベースファイルを用意する（例: ~/test.db）。SQLite3 コマンドでサンプルテーブルを作成してもよい。
  2. 2 ~/.codex/config.toml を開き、[mcp_servers.test_db] セクションを追記して DB ファイルパスを指定する。
  3. 3 Codex を再起動し、「テーブル一覧を見せて」や「レコード数を教えて」と指示してみる。
  4. 4 Codex が MCP サーバーを経由してデータベースを参照し、結果を返すことを確認する。

うまく動いたら、「このテーブル構造に合った INSERT 文を 5 件分作って」など、DB の内容を活かした指示にも挑戦してみましょう。
