承認モードとサンドボックス

Codex は強力なエージェントです。ファイルを書き換えたり、コマンドを実行したりできるからこそ、「どこまで自由に動かすか」を制御する仕組みが重要になります。この章では、安全な利用の要となる承認モード（3段階の権限設定）とサンドボックス（実行を隔離する仕組み）を学びます。

この章で学ぶこと

  • 承認モード（Read Only / Auto / Full Access）の3段階の違いを説明できる
  • ファイル編集やコマンド実行の承認の流れを理解する
  • サンドボックスの役割と実行隔離の仕組みを理解する
  • ネットワークアクセスの扱いを把握し、安全に Codex を運用できる

7.1 承認モードとは

Codex はファイルを読み書きしたり、ターミナルでコマンドを実行したりできます。それは便利ですが、間違った指示を与えてしまったときや、AI が予期しない操作をしようとしたときのために、どの操作まで自動で実行させるかを段階的に制御できる仕組みが用意されています。それが承認モード（Approval Mode）です。

承認モードは大きく 3段階 に分かれており、自分の状況や目的に合わせて切り替えながら使います。まず概要を表で把握しましょう。

モード名 ファイル読み込み ファイル編集 コマンド実行 こんな時に使う
Read Only 自動で実行 承認が必要 承認が必要 コード調査・質問だけ。書き換えは絶対させたくないとき
Auto 自動で実行 自動で実行 承認が必要 ファイル編集は任せるが、コマンドは確認したいとき（デフォルト）
Full Access 自動で実行 自動で実行 自動で実行 CI/CD など自動化パイプラインで使うとき。慎重に検討が必要

初めて使うときは Auto がおすすめ

Codex のデフォルトは Auto モードです。ファイルの読み書きは自動で進めてもらいつつ、シェルコマンドの実行前には確認が入るので、慣れないうちでも安心して使えます。

7.2 各モードの詳細

R Read Only モード

Read Only（読み取り専用）モードでは、Codex はファイルを読んだり検索したりすることは自由に行いますが、ファイルの書き換えやコマンドの実行には毎回あなたの承認が求められます。コードベースを調査させたい、「この処理を説明して」と質問したいだけのときに最適です。

モードの指定は起動時に --approval-mode オプションで行います。

terminal — Read Only モードで起動 
# read-only モードで起動（書き換えを一切させない）
codex --approval-mode read-only

# または省略形
codex -a read-only

A Auto モード（デフォルト）

Auto モードは Codex のデフォルト設定です。ファイルの読み込みと編集（書き込み）は自動で進み、シェルコマンドの実行のみ承認が求められます。「編集はしてほしいけど、変なコマンドを実行される前には確認したい」という、日常的な開発作業にバランスよく対応します。

terminal — Auto モードで起動 
# auto モードで起動（省略時のデフォルトと同じ）
codex --approval-mode auto

# または何も指定しなければ auto になる
codex

F Full Access モード

Full Access モードでは、ファイル編集もコマンド実行もすべて承認なしで自動的に行われます。手を止めずに大量の作業を一気に進めたいとき、またはサンドボックス内で CI/CD パイプラインに組み込むときに使います。自分のマシンで直接使う場合は特に注意が必要です。

注意: Full Access は慎重に使う

Full Access では、誤った指示を与えてしまうと重要なファイルが削除されたり、意図しないコマンドが実行されたりする可能性があります。初めて使う作業・慣れないプロジェクト・本番環境に近い環境では使わないようにしましょう。サンドボックスと組み合わせるか、Read Only / Auto から始めることをお勧めします。

7.3 承認の流れ（実際の操作感）

Codex が承認を求める場面では、ターミナル上に操作の内容（差分やコマンド）が表示され、あなたが「承認（y）」か「拒否（n）」を選ぶことで処理が進みます。以下のフローで承認がどのように流れるかを確認しましょう。

承認画面では、変更するファイルのパス・差分（変更前後の内容）・実行予定のコマンドが表示されます。内容を確認してから「y」で承認するか「n」で拒否できます。

terminal — 承認プロンプトのイメージ 
● 次のコマンドを実行しようとしています:

  npm test

承認しますか？ [y/n]:

差分は必ず確認する習慣を

承認画面には変更の差分（追加行・削除行）が色付きで表示されます。「y で承認」を押す前に内容を読む習慣をつけると、意図しない変更を防げます。

7.4 サンドボックスとは

承認モードは「操作をするたびに確認する」仕組みですが、サンドボックスはそもそも「Codex が動く環境を隔離する」という別の安全層です。

サンドボックス（Sandbox）とは、Codex のエージェントプロセスを隔離された空間の中で動かす仕組みです。コンテナや OS の機能を使って、指定されたワークスペース（作業フォルダ）の外へのアクセスや、外部ネットワークへの通信を制限できます。

ワークスペース書き込み制限

Codex が書き込めるのは指定したワークスペース（プロジェクトフォルダ）の中だけ。システムファイルや他のフォルダを誤って書き換えるリスクがなくなります。

ネットワーク遮断

サンドボックスが有効な場合、エージェントプロセスから外部ネットワークへの接続が遮断されます（許可リストによる制御も可能）。

flowchart LR subgraph SB["サンドボックス内（隔離空間）"] direction TB A["Codex エージェント"] --> B["ワークスペース\nフォルダへの書き込み"] A --> C["システムファイル\n（書き込み不可）"] A --> D["外部ネットワーク\n（遮断）"] end A --> E["ターミナル UI\n（あなたとの対話）"] style C fill:#fecaca,stroke:#ef4444 style D fill:#fecaca,stroke:#ef4444 style B fill:#d1fae5,stroke:#10b981

重要ポイント: 承認モードとサンドボックスは別の概念

承認モードは「操作するたびに確認する」人間による確認の仕組み。サンドボックスは「そもそも動ける範囲を OS レベルで制限する」技術的な隔離の仕組みです。両方を組み合わせることで、より高い安全性が確保できます。

7.5 ネットワークアクセスの扱い

サンドボックスが有効な場合、エージェントが外部サービスに接続しようとしても通信が遮断されます。一方、Codex 本体が OpenAI の API サーバーと通信する部分（あなたの指示を AI が処理する部分）はサンドボックスの外で動いているため、正常に機能します。

ネットワーク通信の範囲（サンドボックス有効時）

あなたの PC
├── Codex CLI 本体
│   └── OpenAI API（クラウド）との通信  ← サンドボックス外 [許可]
│
└── [サンドボックス内] エージェントプロセス
    ├── ワークスペース読み書き           ← 許可
    ├── パッケージマネージャ（npm等）     ← 許可リスト次第
    └── 外部 URL への HTTP リクエスト    ← 遮断

テストの実行や npm パッケージのダウンロードなど、一部の操作では外部ネットワークが必要になる場合があります。その場合は設定で許可リスト（allowlist）に必要なホストを追加することで、最低限の通信だけを許可できます（詳しくは第9章の設定ファイルで扱います）。

注意: 機密情報をワークスペースに置かない

API キーやパスワードなどの機密情報をワークスペース内のファイルに保存しているとき、Codex はその内容を読むことができます。秘匿情報は .env ファイルなどで管理し、.gitignore に追加してリポジトリに含めないようにしましょう。

7.6 安全な使い方のポイント

承認モードとサンドボックスの知識を踏まえて、実際に安全に使うためのポイントをまとめます。

1

慣れるまでは Auto または Read Only を使う

新しいプロジェクトや初めての操作では、まず Read Only で調査させ、問題なければ Auto に切り替えるのが安全です。

2

差分・コマンドを必ず確認する

承認プロンプトが表示されたとき、「y」を押す前に変更内容やコマンドをよく読みましょう。慣れてきたら流さずに確認する習慣がつきます。

3

作業前に Git でコミットしておく

Codex に大きな変更を任せる前に、現時点のコードを Git にコミットしておくと、問題があったときに元に戻せます。

4

本番環境では Full Access を避ける

本番データベースや本番サーバーに影響するフォルダでは Full Access を使わず、まず Read Only か Auto で動作を確認しましょう。

注意: サンドボックスが使えない環境について

サンドボックスは macOS（Apple Silicon）や Linux では Codex が自動で有効にしますが、Windows ネイティブ環境では制限がある場合があります。Windows を使っている場合は WSL2（Windows Subsystem for Linux）の利用を強くおすすめします。サンドボックスが使えない環境では、承認モードによる確認が唯一の安全層になるため、Auto または Read Only を使いましょう。

やってみよう 7-1: 承認モードを切り替えて使い心地を確認する

実際にモードを切り替えながら、承認の流れを体感してみましょう。

手順

  1. 1 適当なプロジェクトフォルダで codex --approval-mode read-only を起動し、「このフォルダにあるファイルの一覧を教えて」と指示する。承認なしに答えが返ってくることを確認する。
  2. 2 続けて「README.md に一行追記して」と指示し、承認プロンプトが表示されることを確認する（「n」で拒否してファイルを変更しないでよい）。
  3. 3 次に codex（Auto モード）で起動し直し、同じ「README.md に一行追記して」を試す。ファイル編集は自動で進み、コマンドを実行しようとしたときだけ承認が入ることを確認する。
  4. 4 2つのモードの使い心地の違いをメモし、自分がどちらを普段使いにしたいかを考える。

理解度確認クイズ

  1. Q1. Codex の Auto モードでは、何の操作をするときに承認が求められますか？

    回答を表示

    シェルコマンドの実行を行うときに承認が求められます。ファイルの読み込みと編集（書き込み）は自動で進みます。Auto はデフォルトのモードで、日常的な開発作業にバランスよく対応します。

  2. Q2. サンドボックスはどのような仕組みで安全を確保しますか？承認モードとの違いも含めて説明してください。

    回答を表示

    サンドボックスは、Codex のエージェントプロセスを隔離された空間で動かし、指定したワークスペース外へのファイル書き込みや外部ネットワークへの通信を OS レベルで制限します。一方、承認モードは操作のたびにユーザーが確認・承認する人間による制御です。両者は補完的な関係にあり、組み合わせることでより高い安全性が得られます。

  3. Q3. Full Access モードを使う際に特に注意すべきことは何ですか？

    回答を表示

    Full Access ではすべての操作が承認なしで自動実行されるため、誤った指示によってファイルが削除されたり意図しないコマンドが実行されたりするリスクがあります。慣れないプロジェクトや本番環境に近い場所では使わず、事前に Git でコミットしてからサンドボックスと組み合わせて使うことが推奨されます。
