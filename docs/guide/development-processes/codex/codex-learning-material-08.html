AGENTS.mdとメモリ・コンテキスト管理

プロジェクト固有のルールを Codex に覚えさせる仕組みが AGENTS.md です。この章では、AGENTS.md の役割・書き方・配置場所による適用範囲の違い、そしてメモリ（記憶）とコンテキスト（文脈）の考え方、長い会話での管理方法まで、Codex を「自分のチームの一員」として動かすための知識をまとめて学びます。

この章で学ぶこと

  • AGENTS.md とは何か、どんな役割を持つかを理解する
  • 配置場所（リポジトリルート・サブディレクトリ・ホーム）による適用範囲の違いを説明できる
  • 効果的なルールや規約を AGENTS.md に書けるようになる
  • メモリとコンテキストウィンドウの違いを理解し、長い会話でコンテキストを管理できる
  • AI に意図を正しく伝えるための指示の書き方のコツを身につける

8.1 AGENTS.md とは何か

毎回の会話で「このプロジェクトはPythonのコーディング規約を守ってほしい」「テストは必ず書いて」と指示するのは手間がかかります。AGENTS.md は、そういったプロジェクト固有のルール・規約・背景情報を、あらかじめファイルに書いておくことで Codex に自動的に読み込ませる仕組みです。

Codex はセッション開始時に、作業ディレクトリおよびその親ディレクトリに存在する AGENTS.md を自動的に読み込みます。これにより「毎回同じ前置きを言わなくてよい」状態を作れます。

Markdownで書ける

普通のMarkdownテキストで記述できます。複雑な設定ファイルの書き方を覚える必要はありません。

毎回自動で読まれる

セッション開始時に自動読み込みされます。指示を毎回繰り返す必要がなくなります。

階層で適用範囲を制御

配置場所によって「全プロジェクト共通」「このフォルダだけ」と使い分けられます。

注意: AGENTS.md と CLAUDE.md は別物です

AGENTS.md は Codex（OpenAI）が読み込む指示ファイルです。一方 CLAUDE.md は Claude Code（Anthropic）が読み込む指示ファイルです。両者はよく似た仕組みですが、それぞれ別のツール向けのファイルです。同じリポジトリで両方使うこともできますが、混同しないように注意してください。

8.2 配置場所と適用範囲

AGENTS.md は「どこに置くか」によって、どの範囲の作業に適用されるかが変わります。Codex は複数の場所にある AGENTS.md をすべて読み込み、内容をマージして使います。

AGENTS.md の配置場所と適用範囲

flowchart TD A["~/.codex/AGENTS.md
ホームディレクトリ"] -->|"全プロジェクトに適用"| B["共通ルール・個人設定"] C["~/myproject/AGENTS.md
リポジトリルート"] -->|"このプロジェクト全体に適用"| D["プロジェクト共通ルール"] E["~/myproject/src/AGENTS.md
サブディレクトリ"] -->|"src/ 以下の作業に適用"| F["特定フォルダ固有のルール"] A --> C C --> E

複数の AGENTS.md が存在する場合、より上位のものから順に読み込まれ、下位のファイルが上書き・補足します。

配置場所 パス例 適用範囲 主な用途
ホームディレクトリ ~/.codex/AGENTS.md すべてのプロジェクト 個人の好み、共通スタイル、言語設定など
リポジトリルート ./AGENTS.md プロジェクト全体 コーディング規約、テスト方針、チーム共通ルール
サブディレクトリ ./src/AGENTS.md そのフォルダ以下 モジュール固有のルール、特定技術の制約

チームでの運用ヒント

リポジトリルートの AGENTS.md はチームの共通ルールとして Git で管理できます（.gitignore に入れなければ自動的にコミットされます）。チームメンバー全員が同じルールで Codex を使えるようになるため、コードスタイルの統一に役立ちます。

8.3 AGENTS.md の書き方

AGENTS.md は普通の Markdown ファイルです。「Codex に守らせたいこと」を自然な文章で書きます。何を書けばよいか、代表的なカテゴリを見てみましょう。

コーディング規約

  • ・使用言語・フレームワークのバージョン
  • ・インデント・命名規則
  • ・型アノテーションの有無

テスト方針

  • ・使用するテストフレームワーク
  • ・テストを必ず書くかどうか
  • ・カバレッジ目標

プロジェクト構造

  • ・ディレクトリ構成の説明
  • ・各フォルダの役割
  • ・ファイル配置のルール

禁止・制約事項

  • ・触ってはいけないファイル
  • ・使用禁止のライブラリ
  • ・本番環境への直接操作禁止

実際の AGENTS.md の例を見てみましょう。

AGENTS.md — リポジトリルート例 
# プロジェクトルール

## 言語・スタック
- Python 3.11、FastAPI、SQLAlchemy 2.x を使用
- 型アノテーションを必ず付けること

## コーディング規約
- インデントはスペース4つ
- 変数・関数名はスネークケース（snake_case）

## テスト方針
- pytest を使う。新機能は必ずテストを書く
- テストファイルは tests/ ディレクトリに配置

## 禁止事項
- .env ファイルは絶対に編集しない
- production ブランチへの直接プッシュ禁止

ホームディレクトリの ~/.codex/AGENTS.md には、個人の設定を書きます。

~/.codex/AGENTS.md — 個人設定例 
# 個人設定

## コミュニケーション
- 説明は日本語で返してほしい
- コードのコメントも日本語で書くこと

## 作業スタイル
- 変更前に何をするか必ず説明する
- 大きな変更はステップごとに確認を取る

効果的な書き方のコツ

  • ・具体的に書く: 「良いコードを書いて」より「型アノテーションを付けてください」の方が伝わる
  • ・禁止事項はリストで: 「〜しないこと」を箇条書きにすると見落としが減る
  • ・理由も添える: 「なぜそのルールか」を書くと、例外対応でも適切に判断してもらいやすい

8.4 メモリとコンテキストの考え方

Codex（の背後にある AI モデル）には「コンテキストウィンドウ」という制限があります。これは AI が一度に「覚えておける」情報量の上限（トークン数）です。会話が長くなったり、大きなファイルを多数読んだりすると、古い情報がウィンドウから押し出されてしまいます。

コンテキストウィンドウのイメージ

flowchart LR A["AGENTS.md
（自動読み込み）"] --> W["コンテキストウィンドウ
（AI の作業記憶）"] B["会話履歴"] --> W C["読み込んだファイル"] --> W D["あなたの新しい指示"] --> W W -->|"上限を超えると"| E["古い情報が押し出される"]

AGENTS.md の内容は毎回読み込まれるため、「忘れてほしくないルール」を書いておくのに最適です。

「メモリ」と「コンテキスト」の違い

メモリ（記憶）

セッションをまたいで保持したい情報。AGENTS.md に書くことで「永続するルール」として機能させます。セッションが終わってもファイルは残ります。

コンテキスト（文脈）

現在のセッション内で AI が保持している会話の流れ・ファイル内容・指示。セッションが終わると消えます。上限を超えると古いものから忘れます。

8.5 長い会話でのコンテキスト管理

大きなプロジェクトで長時間作業していると、コンテキストウィンドウが埋まってきます。Codex はコンテキストが満杯に近づくと「コンテキスト圧縮」を行い、会話の要点を自動的にまとめて古い詳細を省略します。これはある程度自動で行われますが、以下の対策を知っておくと安心です。

セッションを分割する

「機能Aの実装」「テストの追加」のように作業を小さく区切り、新しいセッションを開始します。コンテキストがリフレッシュされます。

重要情報はAGENTS.mdへ

「毎回伝えたいこと」はAGENTS.mdに書いておきます。コンテキストが圧縮されても、次のセッションで再び読み込まれます。

必要なファイルだけ渡す

「プロジェクト全体を読んで」ではなく「src/auth.py を見て」のように、作業に関係するファイルだけ参照させます。

要約を依頼する

長い会話の途中で「ここまでの作業内容をまとめて」と頼んでおくと、自分の確認にもなり、後で参照しやすくなります。

セッション再開（codex resume）との使い分け

codex resume コマンドは直前のセッションを再開します（第4章参照）。ただし、コンテキストが大量に積み重なったセッションを再開すると、また同じコンテキスト圧縮の問題が起きます。作業単位ごとに新しいセッションを立ち上げ、継続事項は AGENTS.md に書いておくのがバランスよい方法です。

8.6 AI に意図を正しく伝えるコツ

AGENTS.md だけでなく、その場その場の指示（プロンプト）の書き方も重要です。AI の応答の質は、指示の明確さに大きく左右されます。

曖昧な指示（NG）

  • 「ちょっと直して」
  • 「いい感じにして」
  • 「なんかおかしいから見て」

どこを・どのように・なぜ直すか、AI には分かりません。

具体的な指示（OK）

  • 「src/auth.py の login 関数のバグを直して。ログイン後にセッションが切れる」
  • 「コメントをすべて日本語に書き換えて」

対象ファイル・箇所・問題・期待結果が明確です。

よい指示の4要素

1

対象を明示する

「どのファイルの」「どの関数の」を具体的に指定します。

2

問題・現状を説明する

「〜というエラーが出る」「〜の動作がおかしい」と現状を伝えます。

3

期待する結果を伝える

「〜のように動いてほしい」「テストが全件通る状態にしてほしい」と目標を示します。

4

制約・条件を添える

「既存のテストは壊さないで」「外部ライブラリは追加しないで」などの制約を付けると、意図しない変更を防げます。

まとめ: AGENTS.md と指示の使い分け

毎回変わらないルール → AGENTS.md に書く。 今日この作業だけの指示 → その場で伝える。この使い分けが Codex を効率よく使うための核心です。AGENTS.md はプロジェクトの「チームルールブック」として育てていくものです。

やってみよう 8-1: AGENTS.md を作成する

自分のプロジェクト（または練習用ディレクトリ）に AGENTS.md を作り、Codex に読み込ませてみましょう。

手順

  1. 1 プロジェクトのルートディレクトリに AGENTS.md を新規作成する。
  2. 2 「使用言語・コーディング規約・テスト方針・禁止事項」を2〜3行ずつ書く。
  3. 3 codex を起動し「AGENTS.md のルールを確認して」と尋ねる。正しく読み込まれているか確認する。
  4. 4 次に ~/.codex/AGENTS.md を作成して、個人設定（言語・スタイル）を追加してみる。

期待される結果

Codex が「AGENTS.md に書かれているルールを守りながら」作業を進めるようになります。毎回前置きを言わなくてよい快適さを実感できます。

理解度確認クイズ

  1. Q1. ~/.codex/AGENTS.md（ホームディレクトリ）に書いた内容は、どの範囲に適用されますか？

    回答を表示

    そのユーザーが Codex を使うすべてのプロジェクトに適用されます。個人の好みや共通スタイルを書くのに適した場所です。

  2. Q2. コンテキストウィンドウが満杯に近づいたとき、Codex が自動的に行う処理を何と呼びますか？

    回答を表示

    コンテキスト圧縮と呼びます。会話の要点を自動的にまとめ、古い詳細を省略してコンテキストウィンドウを確保します。

  3. Q3. AGENTS.md と CLAUDE.md の違いを説明してください。

    回答を表示

    AGENTS.md は Codex（OpenAI）が読み込む指示ファイルで、CLAUDE.md は Claude Code（Anthropic）が読み込む指示ファイルです。仕組みはよく似ていますが、それぞれ異なるツール専用のファイルです。同じリポジトリに両方置くことも可能ですが、混同しないよう注意が必要です。
