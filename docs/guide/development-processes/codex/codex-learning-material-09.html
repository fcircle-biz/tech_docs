設定ファイル（config.toml）とプロファイル

Codex の挙動を細かく調整できる設定ファイル ~/.codex/config.toml を学びます。承認モードやサンドボックスの既定値、使用モデルの選択、用途別にまとめるプロファイルの作り方と切り替え方を理解し、自分の開発スタイルに合った環境を整えましょう。

この章で学ぶこと

  • ~/.codex/config.toml の役割と基本的な書き方を理解する
  • 承認モード・サンドボックス・使用モデルを設定ファイルで既定値として指定できる
  • プロファイル（[profiles.xxx]）を作成して用途別に設定を使い分けられる
  • --profile フラグで起動時にプロファイルを切り替えられる

9.1 設定ファイルとは何か

Codex を起動するたびにオプションを手入力するのは手間がかかります。設定ファイル（config.toml）を使うと、よく使う設定をあらかじめ書いておき、起動時に自動で読み込ませることができます。

設定ファイルは ~/.codex/config.toml（ホームディレクトリの .codex フォルダ）に置きます。.codex フォルダが存在しない場合は、手動で作成します。

ファイル形式は TOML（Tom's Obvious Minimal Language）です。TOML は設定ファイルに広く使われるシンプルな形式で、キー = "値" という読みやすい書き方が特徴です。

一度書けば自動適用

起動のたびにオプションを入力しなくてもよくなります。

細かく挙動を調整

モデル選択・承認モード・サンドボックスを既定値として設定できます。

用途別にプロファイル

業務・学習・実験など、目的ごとに設定セットを切り替えられます。

用語: TOML とは

TOML は設定ファイル用の軽量テキスト形式です。キー = "値" で文字列、キー = true で真偽値、[セクション名] でグループをまとめます。JSON や YAML よりも人が読みやすい点が特徴で、Rust や Python のツールでよく使われます。

9.2 設定ファイルの作成と基本的な書き方

まず設定ファイルを置くフォルダを作成し、ファイルを用意します。

terminal — フォルダとファイルの作成 
# .codex フォルダを作成（すでにある場合はスキップ）
mkdir -p ~/.codex

# 設定ファイルをエディタで開く（なければ新規作成）
nano ~/.codex/config.toml

次に設定の内容を書きます。以下は最もよく使う基本設定の例です。

~/.codex/config.toml — 基本設定の例 
# 使用するモデルを指定（既定: o4-mini）
model = "o4-mini"

# 承認モードの既定値（suggest / auto-edit / full-auto）
approval_policy = "suggest"

# サンドボックスを有効にするか（true / false）
sandbox = true

# ターミナルの表示言語（任意）
locale = "ja"

それぞれのキーについて詳しく見ていきましょう。

キー名 型 説明 主な値の例
model 文字列 使用する AI モデルを指定 "o4-mini"、"o3"
approval_policy 文字列 承認モードの既定値 "suggest"、"auto-edit"、"full-auto"
sandbox 真偽値 サンドボックスを有効にするか true、false
locale 文字列 表示言語・ロケール "ja"、"en"

注意: キー名のスペルミスに気をつける

TOML は存在しないキーを書いてもエラーにならず、単に無視されることがあります。設定が反映されない場合は、まずキー名のスペルを確認してください。公式ドキュメントで最新のキー一覧を確かめるのが確実です。

9.3 モデルの選択（model 設定）

model キーで、Codex が使う AI モデルを指定できます。モデルによって応答速度・処理の精度・API コストが異なるため、用途に応じて使い分けることが重要です。

o4-mini（既定）

  • 応答が速く、コスト効率が良い
  • 日常的なコーディング作業に最適
  • 初学者・実験的な使用に向いている
model = "o4-mini"

o3

  • 高度な推論と複雑なタスクに強い
  • 難しいバグ調査や設計判断に向いている
  • 応答に時間がかかることがある
model = "o3"

コマンドライン上で --model オプションを使うと、設定ファイルの値を一時的に上書きできます。

terminal — モデルを一時的に上書きして起動 
# 設定ファイルの model を上書きして o3 で起動
codex --model o3

重要ポイント: コマンドラインはファイルより優先

コマンドラインオプション（例: --model o3）は、設定ファイルの同じキーより優先されます。「いつもは o4-mini、今回だけ o3 で試したい」というときに便利です。

9.4 承認モードとサンドボックスの既定値を設定する

第7章で学んだ承認モードとサンドボックスも、設定ファイルで既定値を決めておけます。毎回コマンドラインで指定する手間が省けます。

flowchart LR A["config.toml
既定値を設定"] -->|"approval_policy = suggest"| B["suggest モード
（毎回承認）"] A -->|"approval_policy = auto-edit"| C["auto-edit モード
（編集は自動・実行は承認）"] A -->|"approval_policy = full-auto"| D["full-auto モード
（すべて自動）"]
~/.codex/config.toml — 承認モードとサンドボックスの設定 
# 承認モード: suggest（毎回確認）が最も安全な既定値
approval_policy = "suggest"

# サンドボックス: true にしてネットワーク＆システム保護
sandbox = true

approval_policy の3つの値の意味を改めて整理します。

値 動作 向いている場面
"suggest" すべての操作を承認してから実行 学習中・重要なコードベース
"auto-edit" ファイル編集は自動・コマンド実行は承認 日常開発で効率を上げたいとき
"full-auto" すべて自動で実行 CI 自動化・信頼済みスクリプト

初心者は "suggest" から始めよう

設定ファイルに慣れていないうちは approval_policy = "suggest" と sandbox = true の組み合わせが最も安全です。AI が何をしようとしているかを毎回確認できるため、誤操作のリスクを抑えられます。

9.5 プロファイルを作る

プロファイルは、用途ごとに異なる設定セットを1つの config.toml にまとめる仕組みです。たとえば「業務では厳格な承認モード・学習中は自動モード」のように、切り替えながら使えます。

プロファイルは [profiles.プロファイル名] というセクションとして書きます。

~/.codex/config.toml — プロファイルの定義 
# ---- グローバル既定値（プロファイル未指定時に使われる） ----
model = "o4-mini"
approval_policy = "suggest"
sandbox = true

# ---- 業務用プロファイル: 安全重視 ----
[profiles.work]
model = "o3"
approval_policy = "suggest"
sandbox = true

# ---- 学習・実験用プロファイル: 速度重視 ----
[profiles.learning]
model = "o4-mini"
approval_policy = "auto-edit"
sandbox = true

プロファイル名は自由に決められます（英数字・ハイフン・アンダースコアを推奨）。グローバル設定はプロファイルを指定しなかった場合の既定値として機能します。

プロファイルの構造イメージ

flowchart TD A["config.toml"] --> B["グローバル設定
（既定値）"] A --> C["[profiles.work]
業務用"] A --> D["[profiles.learning]
学習用"] A --> E["[profiles.ci]
自動化用"] B -->|"--profile 未指定"| F["Codex 起動"] C -->|"--profile work"| F D -->|"--profile learning"| F E -->|"--profile ci"| F

9.6 プロファイルを切り替えて起動する（--profile）

作成したプロファイルは、--profile（または -p）フラグで指定して起動します。

terminal — プロファイルを指定して起動 
# 業務用プロファイルで起動
codex --profile work

# 学習用プロファイルで起動（省略形）
codex -p learning

# グローバル既定値で起動（プロファイル指定なし）
codex

さらに、--profile と他のオプションを組み合わせることもできます。この場合、コマンドラインオプションがプロファイルの値を上書きします。

terminal — プロファイル＋追加オプションの組み合わせ 
# learning プロファイルで起動しつつ、モデルだけ o3 に上書き
codex --profile learning --model o3

設定の優先順位（高い順）

  1. 1 コマンドラインオプション（--model o3 など）
  2. 2 指定プロファイルの設定（[profiles.work] など）
  3. 3 グローバル設定（config.toml のトップレベル）
  4. 4 Codex の組み込み既定値

9.7 設定例: 開発スタイル別のおすすめ構成

よくある開発スタイルごとに、おすすめの config.toml の書き方をまとめます。自分の状況に合うものを参考にカスタマイズしてください。

入門者・学習中の方向け
~/.codex/config.toml 
# 安全最優先・毎回確認
model = "o4-mini"
approval_policy = "suggest"
sandbox = true
日常開発者向け（プロファイル活用）
~/.codex/config.toml 
# 既定: 安全重視
model = "o4-mini"
approval_policy = "suggest"
sandbox = true

[profiles.fast]
# 信頼できるリポジトリ作業向け: 編集は自動
approval_policy = "auto-edit"
sandbox = true

設定を気軽に試そう

config.toml はいつでも編集できます。「しばらく使ってみてから調整する」という姿勢で始めると、自分に合った設定が自然に見えてきます。変更後は次回の codex 起動から反映されます。

やってみよう 9-1: config.toml を作成してプロファイルを試す

自分の設定ファイルを作り、プロファイルを切り替えて起動する流れを体験しましょう。

手順

  1. 1 mkdir -p ~/.codex を実行し、フォルダを作成する。
  2. 2 テキストエディタで ~/.codex/config.toml を開き、グローバル設定と [profiles.fast] を書いて保存する。
  3. 3 codex で起動し（グローバル設定が使われる）、設定が反映されているか確認する。
  4. 4 codex --profile fast で起動し、承認モードが変わっていることを確かめる。
  5. 5 自分の開発スタイルに合うよう設定を調整してみる。

チェックポイント

  • ~/.codex/config.toml が作成されている
  • プロファイルの切り替えで動作が変わることを確認できた

理解度確認クイズ

  1. Q1. Codex の設定ファイルは通常どこに置きますか？また、ファイル形式は何ですか？

    回答を表示

    ~/.codex/config.toml（ホームディレクトリの .codex フォルダ内）に置きます。ファイル形式は TOML（Tom's Obvious Minimal Language）です。

  2. Q2. config.toml に複数のプロファイルを定義するとき、どのような書き方をしますか？

    回答を表示

    [profiles.プロファイル名] というセクションを作り、その下に設定キーを書きます。例: [profiles.work] のあとに approval_policy = "suggest" のように記述します。

  3. Q3. config.toml の設定と、コマンドラインオプション（例: --model o3）が両方指定されたとき、どちらが優先されますか？

    回答を表示

    コマンドラインオプションが優先されます。優先順位は「コマンドラインオプション → 指定プロファイルの設定 → グローバル設定 → Codex の組み込み既定値」の順です。これにより、設定ファイルの値を一時的に上書きして試すことができます。
