12.1 git rebase - 履歴を整理する

git rebaseは、コミット履歴を「きれいに整理」するための強力なコマンドです。 マージとは異なり、直線的な履歴を作ることができます。

rebase と merge の違い

  • merge: 2つの履歴を合流させる（分岐が残る）
  • rebase: 履歴の「起点」を付け替える（直線になる）

merge と rebase の履歴の違い

gitGraph commit id: "A" commit id: "B" branch feature commit id: "C" commit id: "D" checkout main commit id: "E" merge feature id: "Merge"

上図: merge の場合 - 分岐が残り、マージコミットが作成される

gitGraph commit id: "A" commit id: "B" commit id: "E" commit id: "C'" commit id: "D'"

上図: rebase の場合 - 直線的な履歴になる（C, Dが C', D' として付け替えられる）

基本的な使い方

git rebase の基本 
# feature ブランチを main の最新に付け替える
git checkout feature
git rebase main

# コンフリクトが発生した場合
# 1. コンフリクトを解決
# 2. git add でステージング
# 3. git rebase --continue で続行

# rebase を中断したい場合
git rebase --abort

rebase の黄金ルール

プッシュ済みのコミットを rebase してはいけません！
rebase は履歴を書き換えるため、他の人がすでに取得したコミットを rebase すると、 チームメンバーに混乱を引き起こします。
ローカルでの作業にのみ使用しましょう。

12.2 git stash - 作業を一時退避する

「今の作業を中断して、別のブランチで緊急対応したい」という場面で便利なのがgit stashです。 コミットせずに、作業中の変更を一時的に保存できます。

よくある使用シーン

  • 作業途中で別のブランチに切り替えたい時
  • コミットするほどでもない変更を一時的に保存したい時
  • git pullする前に、ローカルの変更を退避したい時
git stash の基本操作 
# 現在の変更を退避（メッセージ付き）
git stash save "作業途中: ログイン機能"

# または単純に
git stash

# 退避した一覧を確認
git stash list
# 出力例:
# stash@{0}: On feature: 作業途中: ログイン機能
# stash@{1}: WIP on main: abc1234 前回の退避

# 最新の退避を復元（stash から削除）
git stash pop

# 最新の退避を復元（stash に残す）
git stash apply

# 特定の退避を復元
git stash apply stash@{1}

# 退避を削除
git stash drop stash@{0}

# 全ての退避を削除
git stash clear

ヒント: 新規ファイルも stash する

デフォルトでは、まだ追跡されていない新規ファイルは stash されません。 git stash -u（または --include-untracked）で新規ファイルも含めることができます。

12.3 git cherry-pick - 特定のコミットを取り込む

git cherry-pickは、別のブランチから特定のコミットだけを 現在のブランチに取り込むコマンドです。「あのブランチのあの修正だけ欲しい」という時に便利です。

cherry-pick のイメージ

flowchart LR subgraph Before[cherry-pick前] B1[main: A - B - C] B2[feature: A - B - X - Y] end subgraph After[cherry-pick後] A1["main: A - B - C - Y'"] A2[feature: A - B - X - Y] end Before --> After

feature ブランチの Y コミットだけを main に取り込む

git cherry-pick の使い方 
# 特定のコミットを取り込む
git cherry-pick a1b2c3d

# 複数のコミットを取り込む
git cherry-pick a1b2c3d e4f5g6h

# 範囲指定（終点を含まない）
git cherry-pick A..B

# コンフリクト発生時
git cherry-pick --continue  # 解決後に続行
git cherry-pick --abort     # 中断

使用例

  • 本番環境用ブランチに、特定のバグ修正だけを適用したい時
  • 別の開発者の特定の変更を自分のブランチに取り込みたい時
  • 間違えて別のブランチにコミットした変更を正しいブランチに移動したい時

12.4 Git Flow - ブランチ戦略の定番

Git Flowは、Vincent Driessen氏が提唱したブランチ運用の規則です。 大規模なプロジェクトやリリースサイクルが決まっているプロジェクトでよく使われます。

Git Flow のブランチ構成

flowchart TB subgraph Branches[ブランチ構成] M[main
本番リリース用] D[develop
開発統合用] F[feature/*
機能開発用] R[release/*
リリース準備用] H[hotfix/*
緊急修正用] end F -->|機能完成| D D -->|リリース準備| R R -->|リリース| M H -->|緊急修正| M H -->|修正をdevelopにも| D
ブランチ 用途 派生元
main 本番環境にデプロイされるコード -
develop 開発中の統合ブランチ main
feature/* 新機能の開発 develop
release/* リリース準備（バグ修正など） develop
hotfix/* 本番の緊急バグ修正 main

Git Flow の注意点

Git Flow は強力ですが、ブランチが多く複雑です。 小規模なプロジェクトや頻繁にリリースするプロジェクトでは、次に紹介するGitHub Flowの方がシンプルで適している場合があります。

12.5 GitHub Flow - よりシンプルなワークフロー

GitHub Flowは、GitHubが提唱するシンプルなブランチ戦略です。 Git Flowに比べて非常にシンプルで、継続的デリバリーに適しています。

GitHub Flow のワークフロー

flowchart LR A[1. main から
ブランチ作成] --> B[2. 変更を
コミット] B --> C[3. プルリクエスト
を作成] C --> D[4. レビュー
& 議論] D --> E[5. mainに
マージ] E --> F[6. デプロイ]

GitHub Flow のルール

1

main は常にデプロイ可能

mainブランチは常に本番環境にデプロイできる状態を保つ

2

説明的な名前のブランチを作成

例: add-login-feature、fix-header-bug

3

定期的にプッシュ

作業中も定期的にリモートにプッシュして進捗を共有

4

プルリクエストでレビュー

マージ前に必ずプルリクエストを作成し、レビューを受ける

5

マージ後すぐにデプロイ

mainにマージされたら、すぐに本番環境にデプロイ

初心者にはGitHub Flowがおすすめ

GitHub Flowは覚えるルールが少なく、シンプルです。 まずはGitHub Flowで経験を積み、必要に応じてGit Flowを学ぶのがおすすめです。

12.6 GitHub Actions入門 - 自動化の世界への第一歩

GitHub Actionsは、GitHubに組み込まれたCI/CD（継続的インテグレーション/継続的デリバリー）ツールです。 プッシュやプルリクエストをトリガーに、テストやデプロイを自動実行できます。

CI/CD とは？

  • CI（継続的インテグレーション）: コードの変更を頻繁に統合し、自動テストを実行
  • CD（継続的デリバリー）: テスト通過後、自動的に本番環境にデプロイ

GitHub Actions の基本構成

.github/workflows/test.yml 
# ワークフロー名
name: Run Tests

# トリガー（いつ実行するか）
on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

# ジョブ（実行する処理）
jobs:
  test:
    runs-on: ubuntu-latest  # 実行環境
    steps:
      - uses: actions/checkout@v4  # リポジトリをチェックアウト

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm install

      - name: Run tests
        run: npm test

よく使われるワークフロー例

自動テスト

プッシュごとにテストを自動実行

コード品質チェック

Linter や Formatter の自動実行

自動デプロイ

mainへのマージ時に自動で本番環境にデプロイ

ドキュメント生成

コードからドキュメントを自動生成・公開

12.7 チーム開発でのGit運用ルール

チームでGitを使う際には、共通のルールを決めておくことが重要です。 以下は、多くのチームで採用されている一般的なルールです。

ブランチ命名規則の例

プレフィックス 用途 例
feature/ 新機能の開発 feature/user-authentication
fix/ バグ修正 fix/login-error
hotfix/ 緊急の修正 hotfix/security-patch
docs/ ドキュメント更新 docs/update-readme
refactor/ リファクタリング refactor/clean-up-utils

コミットメッセージの規約

Conventional Commitsという規約がよく使われます。

コミットメッセージの形式 
# 形式
: 

# type の種類
feat:     新機能
fix:      バグ修正
docs:     ドキュメントのみの変更
style:    コードの意味に影響しない変更（空白、フォーマットなど）
refactor: リファクタリング
test:     テストの追加・修正
chore:    ビルドプロセスやツールの変更

# 例
feat: ユーザー登録機能を追加
fix: ログイン時のエラーメッセージを修正
docs: READMEにインストール手順を追加

その他のベストプラクティス

  • main への直接プッシュは禁止 - 必ずプルリクエストを経由
  • プルリクエストには説明を書く - 何をなぜ変更したか
  • 小さく頻繁にコミット - 大きな変更は分割して
  • コードレビューは必須 - 最低1人のレビューを受ける
  • テストが通るまでマージしない - CI/CDの結果を確認

実習 12-1: git stash を使った作業の退避

実際に作業を中断して、別のブランチで作業する流れを体験しましょう。

手順

  1. 1 feature ブランチでファイルを編集（コミットしない）
  2. 2 git stash save "作業中" で退避
  3. 3 main ブランチに切り替えて別の作業をする
  4. 4 feature ブランチに戻り、git stash pop で復元

実行例

# 1. feature ブランチで作業
git checkout -b feature/test-stash
echo "作業中の内容" > work.txt

# 2. 退避
git stash save "テスト作業を退避"
git stash list  # 退避を確認

# 3. main で別作業
git checkout main
echo "緊急修正" > hotfix.txt
git add hotfix.txt && git commit -m "緊急修正"

# 4. feature に戻って復元
git checkout feature/test-stash
git stash pop
cat work.txt  # 作業内容が復元されている

理解度確認クイズ

  1. Q1. git merge と git rebase の違いは何ですか？

    回答を表示

    mergeは2つのブランチの履歴を合流させ、マージコミットを作成します（分岐が残る）。
    rebaseはブランチの起点を付け替え、直線的な履歴を作ります（履歴を書き換える）。
    rebaseはローカルでの作業にのみ使用し、プッシュ済みのコミットには使用しません。

  2. Q2. git stash はどのような場面で使用しますか？

    回答を表示

    git stash は、作業途中の変更をコミットせずに一時的に退避したい場面で使用します。 例えば、作業中に別のブランチで緊急対応が必要になった時や、 git pull する前にローカルの変更を退避したい時などに便利です。 git stash popで退避した変更を復元できます。

  3. Q3. GitHub Flow の基本的なワークフローを説明してください。

    回答を表示

    GitHub Flow は以下の流れで行います：
    1. mainブランチから説明的な名前のブランチを作成
    2. 変更をコミットし、定期的にリモートにプッシュ
    3. プルリクエストを作成
    4. レビューを受けて議論
    5. mainにマージ
    6. デプロイ
    mainは常にデプロイ可能な状態を保ち、マージしたらすぐにデプロイします。
