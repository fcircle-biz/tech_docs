プロジェクト管理 - .gitignore、タグ、リリース

実際のプロジェクトでGit/GitHubを活用するための知識を学びます。.gitignoreでファイルを除外し、タグでバージョンを管理し、リリース機能で成果物を公開する方法を理解しましょう。

この章で学ぶこと

  • .gitignoreファイルの作成と設定方法を理解する
  • 効果的なコミットメッセージの書き方を習得する
  • git tagでバージョン番号を付ける方法を学ぶ
  • GitHubのリリース機能を活用する
  • README、ライセンス、Issues機能の使い方を知る

12.1 .gitignoreファイル - 不要なファイルを除外する

プロジェクトには、Gitで管理したくないファイルやフォルダが存在します。例えば、開発環境によって自動生成されるファイルや、個人設定、機密情報を含むファイルなどです。.gitignoreファイルを使うと、これらを自動的に除外できます。

Gitで管理すべきでないファイルの例

分類 ファイル例 理由
ビルド成果物 *.o, *.class, dist/, build/ ソースから自動生成できる
依存ライブラリ node_modules/, venv/ パッケージマネージャで復元可能
機密情報 .env, secrets.json, *.key セキュリティリスク
OS自動生成 .DS_Store, Thumbs.db OS固有で不要
IDE設定 .vscode/, .idea/ 個人の環境設定
ログ・一時ファイル *.log, tmp/, cache/ 動作中に自動生成される

.gitignoreファイルの作成

.gitignoreファイルは、リポジトリのルートディレクトリ（.gitフォルダと同じ階層）に配置します。

.gitignore作成例 
# リポジトリのルートに.gitignoreファイルを作成
touch .gitignore

# エディタで開く
nano .gitignore

.gitignoreの記述パターン

.gitignore 
# コメント行（#で始まる）

# 特定のファイルを除外
secrets.json
config.local.yml

# ワイルドカード（*.拡張子で全てのファイルを除外）
*.log
*.tmp

# ディレクトリ全体を除外（/で終わる）
node_modules/
dist/
build/

# ルートディレクトリのみの特定ファイル（/で始まる）
/TODO.txt

# 除外パターンの例外（!で始まる）
*.log
!important.log

# ネストしたディレクトリ（**でどの階層でもマッチ）
**/tmp/
**/*.bak

ヒント: テンプレートを活用

各言語やフレームワークの.gitignoreテンプレートは、github/gitignoreリポジトリで公開されています。自分のプロジェクトに合ったテンプレートをコピーして使うと便利です。

既に追跡されているファイルを除外する

既にGitで管理されているファイルは、.gitignoreに追加しても除外されません。一度追跡を解除する必要があります。

追跡解除のコマンド 
# 特定のファイルをGitの追跡から削除（ファイル自体は残す）
git rm --cached secrets.json

# ディレクトリごと削除
git rm -r --cached node_modules/

# コミット
git commit -m "Remove ignored files from tracking"

12.2 効果的なコミットメッセージの書き方

コミットメッセージは、変更内容を後から理解するための重要な記録です。適切に書くことで、チーム開発の効率が大きく向上します。

コミットメッセージの基本構造

コミットメッセージの基本形 
[種類] 件名（50文字以内）

本文（72文字で改行、必要に応じて）
- 変更の理由
- 変更の内容
- 注意点やトレードオフ

関連するIssue番号: #123

コミットメッセージの種類（プレフィックス）

プレフィックス 意味 例
feat 新機能の追加 feat: ユーザー検索機能を追加
fix バグ修正 fix: ログイン時のエラーを修正
docs ドキュメント変更 docs: READMEのインストール手順を更新
style コード整形（動作変更なし） style: インデントを統一
refactor リファクタリング refactor: 認証ロジックを関数に分離
test テスト追加・修正 test: ユーザー登録のテストを追加
chore ビルド・設定変更 chore: 依存パッケージを更新

良いコミットメッセージの例

悪い例

修正
update
fix bug
色々変更

何を変更したか不明確

良い例

fix: ログイン失敗時のエラー表示を修正

パスワードが間違っている場合に、
エラーメッセージが表示されない
問題を修正しました。

Fixes #45

変更内容と理由が明確

コミットメッセージのベストプラクティス

  • 件名は50文字以内、本文は72文字で改行
  • 件名は現在形・命令形で書く（「追加した」ではなく「追加」）
  • 「何を」変更したかより、「なぜ」変更したかを書く
  • 1コミットは1つの変更に集中する
  • 関連するIssue番号を記載する

12.3 git tag - バージョン番号をつける

タグ（Tag）は、特定のコミットに「目印」をつける機能です。主にリリースバージョン（v1.0.0など）を記録するために使われます。

タグの種類

軽量タグ

単純な目印。タグ名だけを記録。一時的なマーキングに使用。 

git tag v1.0.0

注釈付きタグ（推奨）

メッセージ、作成者、日時を記録。リリースに推奨。 

git tag -a v1.0.0 -m "Release v1.0.0"

タグの基本操作

タグの作成と管理 
# 注釈付きタグを作成（推奨）
git tag -a v1.0.0 -m "Initial release"

# 既存のタグ一覧を表示
git tag

# 特定のパターンでタグを検索
git tag -l "v1.*"

# タグの詳細情報を表示
git show v1.0.0

# 過去のコミットにタグを付ける
git tag -a v0.9.0 9fceb02 -m "Beta release"

# タグを削除
git tag -d v1.0.0

タグをリモートにプッシュする

デフォルトでは、タグはgit pushで自動的にリモートに送信されません。明示的にプッシュする必要があります。

タグのリモート操作 
# 特定のタグをプッシュ
git push origin v1.0.0

# すべてのタグをプッシュ
git push origin --tags

# リモートのタグを削除
git push origin --delete v1.0.0

セマンティックバージョニング

バージョン番号はセマンティックバージョニング（Semantic Versioning）という規則に従うのが一般的です。

セマンティックバージョニング（v X.Y.Z）

  • X（メジャーバージョン）: 後方互換性のない大きな変更
  • Y（マイナーバージョン）: 後方互換性のある新機能追加
  • Z（パッチバージョン）: 後方互換性のあるバグ修正

例: v1.2.3 → v1.2.4（バグ修正）、v1.3.0（新機能）、v2.0.0（大きな変更）

12.4 GitHubのリリース機能 - 成果物の公開

GitHubのリリース（Release）機能を使うと、タグに加えて、リリースノート（変更内容の説明）やビルド済みファイルを添付できます。

リリースの作成手順

GitHubでリリースを作成

  1. 1 GitHubリポジトリページの「Releases」をクリック
  2. 2 「Create a new release」ボタンをクリック
  3. 3 「Choose a tag」でタグを選択（または新規作成）
  4. 4 リリースタイトルと説明文を入力（Markdown形式）
  5. 5 必要に応じてファイルを添付（実行ファイル、ZIPなど）
  6. 6 「Publish release」をクリック

リリースノートの書き方

リリースノートの例 
## v1.2.0 - 2025年1月15日リリース

### 新機能
- ユーザー検索機能を追加 (#123)
- ダークモードに対応 (#145)

### 改善
- レスポンス速度を30%向上
- エラーメッセージをより分かりやすく改善

### バグ修正
- ログイン失敗時のエラー表示を修正 (#156)
- 画像アップロード時のクラッシュを修正

### 既知の問題
- Safari 15でレイアウトが崩れる場合があります

### アップグレード手順
1. 依存パッケージを更新: `npm install`
2. データベースマイグレーション: `npm run migrate`

ヒント

GitHubは「Generate release notes」ボタンで、前回のリリース以降のコミットメッセージやプルリクエストから自動的にリリースノートを生成してくれます。

12.5 プロジェクト管理の追加機能

README.mdの書き方

README.mdは、プロジェクトの「顔」です。リポジトリのトップページに表示され、プロジェクトの概要や使い方を説明します。

README.mdの基本構成 
# プロジェクト名

## 概要
このプロジェクトの目的と主な機能を簡潔に説明

## 主な機能
- 機能1
- 機能2
- 機能3

## インストール
```bash
git clone https://github.com/username/project.git
cd project
npm install
```

## 使い方
```bash
npm start
```

## 開発環境
- Node.js 18以上
- npm 9以上

## ライセンス
MIT License

## 貢献
プルリクエストを歓迎します！

ライセンスの選び方

オープンソースプロジェクトには、適切なライセンスを設定することが重要です。

ライセンス 特徴 適している場合
MIT 最も制限が少ない 自由に使ってほしい
Apache 2.0 特許権の明示的な許諾 特許が関係する場合
GPL v3 コピーレフト（改変版も公開必須） 常にオープンであってほしい

ライセンス選びはchoosealicense.comを参考にしてください。

GitHub Issuesの活用

Issuesは、バグ報告、機能リクエスト、質問などを管理する機能です。

バグ報告

  • 発生した問題を記録
  • 再現手順を明記
  • ラベルで分類

機能リクエスト

  • 新機能のアイデア
  • 改善提案
  • コミュニティの意見収集

タスク管理

  • TODO管理
  • マイルストーンで整理
  • 進捗の可視化

プルリクエストとの連携

  • コミットメッセージに「Fixes #123」
  • PRマージ時に自動クローズ
  • 変更履歴の追跡

GitHub Pages - 静的サイトの公開

GitHub Pagesを使うと、リポジトリから無料で静的Webサイトを公開できます。

GitHub Pagesの設定

  1. 1. リポジトリの「Settings」→「Pages」を開く
  2. 2. Sourceで「main」ブランチを選択
  3. 3. フォルダを選択（root または /docs）
  4. 4. 「Save」をクリック
  5. 5. 数分後に https://username.github.io/repo でアクセス可能に

実習 12-1: プロジェクト管理機能を使う

実際に.gitignoreを作成し、タグをつけて、GitHubでリリースを公開してみましょう。

手順

  1. 1 .gitignoreファイルを作成し、node_modules/や.env等を除外する
  2. 2 git tag -a v1.0.0 -m "Initial release"でタグを作成
  3. 3 git push origin --tagsでタグをプッシュ
  4. 4 GitHubの「Releases」からリリースを作成し、リリースノートを記入
  5. 5 README.mdを編集し、プロジェクトの説明を追加

理解度確認クイズ

  1. Q1. .gitignoreファイルで除外すべきファイルの種類を3つ挙げてください。

    回答を表示

    以下のようなファイルを除外します（3つ挙げる）：

    • ビルド成果物（*.o、dist/など）
    • 依存ライブラリ（node_modules/、venv/など）
    • 機密情報（.env、secrets.jsonなど）
    • OS自動生成ファイル（.DS_Store、Thumbs.dbなど）
    • IDE設定ファイル（.vscode/、.idea/など）

  2. Q2. セマンティックバージョニングのv1.2.3で、各数字が何を意味するか説明してください。

    回答を表示
    • 1（メジャーバージョン）: 後方互換性のない大きな変更
    • 2（マイナーバージョン）: 後方互換性のある新機能追加
    • 3（パッチバージョン）: 後方互換性のあるバグ修正

  3. Q3. 良いコミットメッセージの特徴を3つ挙げてください。

    回答を表示

    良いコミットメッセージの特徴：

    • 件名は50文字以内で簡潔に
    • 「何を」変更したかより「なぜ」変更したかを記述
    • プレフィックス（feat、fix等）で種類を明示
    • 1コミットは1つの変更に集中
    • 関連するIssue番号を記載

  4. Q4. GitHubのリリース機能でできることを2つ挙げてください。

    回答を表示

    リリース機能でできること：

    • タグにリリースノート（変更内容の説明）を添付できる
    • ビルド済みファイルやZIPなどを添付して配布できる
    • 前回リリースからの変更を自動生成できる
    • リリース履歴を一覧で確認できる

  5. Q5. GitHub Issuesの活用方法を3つ挙げてください。

    回答を表示

    Issuesの活用方法：

    • バグ報告と再現手順の記録
    • 機能リクエストやアイデアの収集
    • TODOやタスクの管理
    • プルリクエストと連携した自動クローズ
    • 質問やディスカッションの場
