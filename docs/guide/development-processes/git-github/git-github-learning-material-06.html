第6章

リモート操作 - コマンドラインでのGitHub連携

コマンドラインでGitHubと連携する方法を学びます。push、pull、cloneなどのリモート操作コマンドと、認証設定の方法を習得しましょう。

この章で学ぶこと

  • git remoteでリモートリポジトリを設定する方法
  • git push / git pull / git cloneの使い方
  • SSH認証の設定方法（パスワードなしで接続）
  • HTTPS vs SSH接続の違いと選び方

6.1 リモートリポジトリの概念

第5章ではローカル（自分のPC）でのGit操作を学びました。この章では、リモートリポジトリ（GitHub上のリポジトリ）とやり取りする方法を学びます。

用語解説: ローカルとリモート

  • ローカルリポジトリ: 自分のパソコン上にあるリポジトリ
  • リモートリポジトリ: GitHub等のサーバー上にあるリポジトリ

ローカルとリモートの関係

主要なリモート操作コマンドは以下の3つです：

コマンド 役割 イメージ
git push ローカルの変更をリモートに送信 クラウドへアップロード
git pull リモートの変更をローカルに取得 クラウドからダウンロード
git clone リモートリポジトリを丸ごと複製 初めてダウンロード

6.2 git remote - リモートリポジトリの設定

ローカルリポジトリとリモートリポジトリを紐付けるには、git remoteコマンドを使います。

リモートの確認

まず、現在設定されているリモートを確認してみましょう。

Terminal 
# 設定されているリモートを表示
git remote

# リモートの詳細（URLも表示）
git remote -v

リモートの追加

新規にローカルリポジトリを作成した場合、リモートを追加する必要があります。

Terminal 
# リモートリポジトリを追加
# 「origin」は慣習的に使われるリモート名
git remote add origin https://github.com/ユーザー名/リポジトリ名.git

# 追加されたか確認
git remote -v
# 出力例:
# origin  https://github.com/username/my-project.git (fetch)
# origin  https://github.com/username/my-project.git (push)

「origin」とは？

originは「起源」という意味で、リモートリポジトリのエイリアス（別名）です。URLを毎回入力する代わりに「origin」と書くだけで済みます。別の名前を付けることもできますが、originが広く使われている慣習です。

その他のremoteコマンド

Terminal 
# リモートのURLを変更
git remote set-url origin 新しいURL

# リモートを削除
git remote remove origin

# リモート名を変更
git remote rename origin upstream

6.3 git push - 変更をGitHubにアップロード

ローカルでコミットした変更をGitHubに送信するには、git pushを使います。

Terminal 
# 基本形: git push リモート名 ブランチ名
git push origin main

# 初回push時は -u オプションを付ける（追跡ブランチを設定）
git push -u origin main

# -u を付けた後は、ブランチ名を省略可能
git push

-u オプションとは？

-u（または--set-upstream）は、ローカルブランチとリモートブランチを関連付けるオプションです。

  • 一度設定すれば、以降はgit pushだけでOK
  • git pullも同様にブランチ名省略可能に

git pushの流れ

6.4 git pull - GitHubから変更を取得

リモートリポジトリの最新の変更を取得し、ローカルに反映するにはgit pullを使います。

Terminal 
# 基本形: git pull リモート名 ブランチ名
git pull origin main

# 追跡ブランチが設定されていれば省略可能
git pull

pull = fetch + merge

git pullは、実際には2つのコマンドを組み合わせたものです。

  • git fetch リモートの情報を取得（ローカルには反映しない）
  • git merge 取得した変更をローカルに統合

チーム開発での注意点

作業を始める前にgit pullで最新の状態を取得する習慣をつけましょう。古い状態で作業を進めると、後でコンフリクト（競合）が発生しやすくなります。

6.5 git clone - リポジトリを複製する

GitHubにある既存のリポジトリを自分のパソコンにダウンロードするには、git cloneを使います。

Terminal 
# 基本形: git clone リポジトリURL
git clone https://github.com/ユーザー名/リポジトリ名.git

# 別の名前でクローン
git clone https://github.com/ユーザー名/リポジトリ名.git my-folder

# クローン後はそのフォルダに移動して作業
cd リポジトリ名

git clone

既存のリモートリポジトリをコピー

  • - リモートの設定済み
  • - 履歴も全てコピー
  • - すぐにpush/pull可能

git init

新規にローカルリポジトリを作成

  • - リモートは自分で設定
  • - 履歴は空の状態
  • - remote addが必要

GitHubからURLを取得する方法

  1. GitHubでリポジトリのページを開く
  2. 緑色の「Code」ボタンをクリック
  3. HTTPS（またはSSH）のURLをコピー
  4. ターミナルでgit clone URLを実行

6.6 HTTPS vs SSH - 接続方式の選び方

GitHubとの通信にはHTTPSとSSHの2つの方式があります。それぞれの特徴を理解して使い分けましょう。

項目 HTTPS SSH
URL形式 https://github.com/... git@github.com:...
初期設定 簡単（そのまま使える） SSH鍵の生成と登録が必要
認証方法 Personal Access Token SSH鍵（自動認証）
毎回の認証 トークン入力（または保存） 不要（鍵で自動認証）
おすすめ 初心者、一時的な利用 日常的に使う場合

初心者へのおすすめ

最初はHTTPSで始めて、慣れてきたらSSHに移行するのがおすすめです。SSHは一度設定すれば毎回の認証が不要になり、非常に便利です。

6.7 SSH認証の設定

SSHを使うと、パスワードやトークンを入力せずにGitHubと通信できます。以下の手順で設定しましょう。

Step 1: SSH鍵の生成

Terminal 
# SSH鍵を生成（メールアドレスは自分のものに置き換え）
ssh-keygen -t ed25519 -C "your_email@example.com"

# 保存場所を聞かれたらEnterでデフォルトのまま
# パスフレーズは任意（空でもOK、セキュリティのため設定推奨）

Step 2: SSH鍵の確認とコピー

Terminal 
# 公開鍵の内容を表示
cat ~/.ssh/id_ed25519.pub

# 出力例（これをコピー）:
# ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAI... your_email@example.com

Step 3: GitHubに公開鍵を登録

  1. GitHubにログイン
  2. 右上のアイコン → Settings
  3. 左メニューの「SSH and GPG keys」
  4. 「New SSH key」ボタンをクリック
  5. Titleに任意の名前（例: My PC）を入力
  6. Keyにコピーした公開鍵を貼り付け
  7. 「Add SSH key」をクリック

Step 4: 接続テスト

Terminal 
# GitHubとの接続をテスト
ssh -T git@github.com

# 成功時のメッセージ:
# Hi ユーザー名! You've successfully authenticated, ...

SSH設定完了後

SSHでクローンする場合は、URLをgit@github.com:ユーザー名/リポジトリ.gitの形式で指定します。

6.8 認証情報の管理（HTTPS利用時）

HTTPSを使う場合、2021年8月以降はパスワードの代わりにPersonal Access Token（PAT）が必要です。

Personal Access Tokenの作成

  1. GitHubにログイン → Settings
  2. 左メニュー下部の「Developer settings」
  3. 「Personal access tokens」→「Tokens (classic)」
  4. 「Generate new token」→「Generate new token (classic)」
  5. Note（名前）を入力、Expiration（有効期限）を設定
  6. 「repo」にチェックを入れる
  7. 「Generate token」をクリック
  8. 表示されたトークンをコピーして安全に保管（再表示不可）

トークンの取り扱い注意

Personal Access Tokenはパスワードと同等の機密情報です。コードに直接書いたり、公開リポジトリにアップロードしないでください。

認証情報の保存（credential helper）

毎回トークンを入力するのは面倒なので、credential helperを使って保存できます。

Terminal 
# macOS
git config --global credential.helper osxkeychain

# Windows
git config --global credential.helper wincred

# Linux
git config --global credential.helper store
# または cache（一時的に保存、デフォルト15分）
git config --global credential.helper cache

6.9 よくあるエラーと解決方法

トラブルシューティング

fatal: remote origin already exists.

原因: 既にoriginという名前のリモートが設定されている

解決方法: git remote set-url origin 新しいURLでURLを変更するか、git remote remove originで削除してから再度追加

error: failed to push some refs to ...

原因: リモートにローカルにない変更がある

解決方法: まずgit pullでリモートの変更を取得してから、再度push

Permission denied (publickey).

原因: SSH鍵が正しく設定されていない

解決方法:

  • SSH鍵が存在するか確認: ls ~/.ssh/id_ed25519.pub
  • GitHubに公開鍵が登録されているか確認
  • ssh-agentに鍵が追加されているか確認
Authentication failed

原因: HTTPSでのパスワード認証が廃止されている（トークンが必要）

解決方法: Personal Access Tokenを作成し、パスワードの代わりに使用。または、SSHに切り替え

実習 6-1: リモートリポジトリとの連携

ローカルで作成したリポジトリをGitHubにpushしてみましょう。

手順

  1. 1 GitHubで新しいリポジトリを作成（READMEは追加しない）
  2. 2 ローカルでgit initしたフォルダに移動
  3. 3 git remote add originでリモートを追加
  4. 4 git push -u origin mainで初回push

実行例

# 1. リモートを追加
git remote add origin https://github.com/あなたのユーザー名/test-repo.git

# 2. リモートの確認
git remote -v

# 3. 初回push（-uで追跡設定）
git push -u origin main

確認ポイント

  • GitHubのリポジトリページでファイルが表示されるか
  • コミット履歴がGitHubでも確認できるか

理解度確認クイズ

  1. Q1. git pushとgit pullの違いを説明してください。

    回答を表示

    git pushは、ローカルリポジトリの変更をリモートリポジトリ（GitHub）に送信（アップロード）するコマンドです。
    git pullは、リモートリポジトリの変更をローカルリポジトリに取得（ダウンロード）して統合するコマンドです。
    方向が逆で、pushは「ローカル→リモート」、pullは「リモート→ローカル」です。

  2. Q2. git cloneとgit initの違いは何ですか？

    回答を表示

    git cloneは、既存のリモートリポジトリを丸ごとコピーするコマンドで、履歴もリモート設定も含まれます。
    git initは、新規に空のローカルリポジトリを作成するコマンドで、リモートの設定は自分で行う必要があります。
    既存プロジェクトを取得する場合はclone、新規プロジェクトを始める場合はinitを使います。

  3. Q3. HTTPSとSSHの主な違いは何ですか？

    回答を表示

    HTTPSは設定が簡単ですが、毎回Personal Access Tokenでの認証が必要です（credential helperで保存可能）。
    SSHは初期設定（鍵の生成と登録）が必要ですが、一度設定すれば毎回の認証が不要で便利です。
    初心者はHTTPSから始め、慣れたらSSHに移行するのがおすすめです。
