ブランチの基本 - 並行作業の概念と操作

ブランチの概念と基本的な使い方を学びます。ブランチを使うことで、複数の作業を並行して進めたり、実験的な変更を安全に試したりできます。

この章で学ぶこと

  • ブランチとは何か、木の枝のイメージで理解する
  • なぜブランチが必要か、並行開発と実験的変更のメリット
  • mainブランチの役割と重要性
  • git branchでブランチを作成・一覧表示する
  • git checkout/git switchでブランチを切り替える
  • ブランチの削除と命名規則のベストプラクティス

8.1 ブランチとは何か

ブランチ（Branch）とは、日本語で「枝」という意味です。Gitでは、開発の歴史が分岐して並行に進む仕組みを「ブランチ」と呼びます。

木の枝のイメージで理解する

ブランチは、木の幹から枝が分かれるイメージで理解できます：

  • 幹（main/master）: 本流の開発ライン、常に安定した状態を保つ
  • 枝（branch）: 幹から分岐した開発ライン、新機能や実験的な変更を行う
  • 合流（merge）: 枝で完成した変更を幹に戻す

ブランチの仕組み

gitGraph commit id: "初期コミット" commit id: "機能A完成" branch feature-b checkout feature-b commit id: "機能B開発中" commit id: "機能B完成" checkout main commit id: "バグ修正" merge feature-b commit id: "リリース準備"

mainブランチから枝分かれしてfeature-bブランチで作業し、完成後にmainブランチに統合（マージ）する流れです。

ブランチを使うことで、複数の作業を同時並行で進めることができます。例えば：

  • Aさんは新機能の開発ブランチで作業
  • Bさんはバグ修正ブランチで作業
  • それぞれが完成したらmainブランチに統合

8.2 なぜブランチが必要なのか

ブランチがないと、以下のような問題が発生します：

ブランチなしの開発の問題点

  • 作業途中のコードが本番に混入 - 未完成の機能がリリースされてしまう
  • 複数人が同じファイルで競合 - 誰かの作業を上書きしてしまう
  • 実験的な変更が危険 - 試してみたいことができない
  • バグ修正と機能開発を同時にできない - 作業が順番待ちになる

ブランチを使うことで、これらの問題を解決できます：

並行開発

複数の機能を同時に開発できます。それぞれのブランチは独立しているため、お互いに影響しません。

実験的な変更

試してみたいアイデアを専用ブランチで試せます。うまくいかなければブランチごと削除すればOK。

安定性の確保

mainブランチは常に動作する状態を保ち、開発はブランチで行うことで安定性を維持できます。

チーム協業

各メンバーが自分のブランチで作業し、完成したらレビューを経て統合する効率的なワークフローが実現できます。

8.3 mainブランチの役割

Gitリポジトリを作成すると、デフォルトでmainブランチ（またはmasterブランチ）が作成されます。これは「本流」となるブランチです。

mainとmasterの違い

以前は「master」という名前が標準でしたが、最近は「main」に変更されています。どちらも同じ意味で、デフォルトの主要ブランチを指します。本教材では「main」を使用します。

mainブランチの特徴

項目 説明
役割 本番環境にデプロイする安定したコード
状態 常に動作する、テスト済みの状態を保つ
直接変更 基本的に行わない（ブランチで開発→マージ）
保護 チーム開発ではプルリクエスト必須にすることが多い

ベストプラクティス

mainブランチは「いつでもデプロイできる状態」を保つことが重要です。新機能の開発やバグ修正は必ず専用ブランチを作成して行い、レビューとテストを経てからmainにマージします。

8.4 ブランチの作成と一覧表示

git branchコマンドを使って、ブランチの作成と一覧表示を行います。

現在のブランチを確認する

ターミナル 
# ブランチの一覧を表示
git branch

# 出力例：
# * main
#   feature-login
#   bugfix-header

現在いるブランチには * マークが付きます。

新しいブランチを作成する

ターミナル 
# 新しいブランチを作成（現在のブランチから分岐）
git branch feature-new

# ブランチが作成されたか確認
git branch

# 出力例：
# * main
#   feature-new

重要なポイント

git branch ブランチ名は、ブランチを作成するだけで、そのブランチに切り替わりません。作成後、まだmainブランチにいる状態です。

リモートブランチも含めて表示

ターミナル 
# ローカルとリモートの全ブランチを表示
git branch -a

# 出力例：
# * main
#   feature-new
#   remotes/origin/main
#   remotes/origin/feature-login

8.5 ブランチの切り替え

作成したブランチに切り替えるには、git checkoutまたはgit switchコマンドを使います。

git switchコマンド（推奨）

Git 2.23以降では、ブランチ切り替え専用のgit switchコマンドが推奨されています。

ターミナル 
# feature-newブランチに切り替え
git switch feature-new

# 出力：
# Switched to branch 'feature-new'

# 確認
git branch
# 出力：
#   main
# * feature-new

git checkoutコマンド（従来の方法）

ターミナル 
# checkoutでもブランチ切り替え可能
git checkout main

# 出力：
# Switched to branch 'main'

switchとcheckoutの違い

git checkoutは、ブランチ切り替え以外にも様々な用途に使える多機能コマンドです。そのため、ブランチ切り替え専用のgit switchが新たに追加されました。

  • git switch: ブランチ切り替え専用（わかりやすい）
  • git checkout: 多機能（ブランチ切り替え、ファイル復元など）

ブランチ作成と切り替えを同時に行う

実際の開発では、ブランチを作成したらすぐに切り替えることがほとんどです。これを1つのコマンドで行えます。

ターミナル 
# switchの場合（推奨）
git switch -c feature-profile
# 出力：
# Switched to a new branch 'feature-profile'

# checkoutの場合
git checkout -b feature-settings
# 出力：
# Switched to a new branch 'feature-settings'

-c（switch）または-b（checkout）オプションで、作成と切り替えを同時に行えます。

実習: ブランチを作成して切り替える

実際にブランチを作成して切り替えを体験してみましょう。

手順

  1. 現在のブランチを確認する（git branch）
  2. 新しいブランチ「feature-about」を作成する
  3. ブランチ一覧を確認する
  4. 「feature-about」ブランチに切り替える
  5. README.mdに「About Page」という行を追加する
  6. 変更をコミットする
  7. mainブランチに戻る
  8. README.mdを確認する（追加した行がないことを確認）
実行例 
# 1. 現在のブランチ確認
git branch

# 2. ブランチ作成と切り替えを同時に
git switch -c feature-about

# 3. ファイルを編集
echo "About Page" >> README.md

# 4. 変更をコミット
git add README.md
git commit -m "Add About Page to README"

# 5. mainブランチに戻る
git switch main

# 6. README.mdを確認（About Pageの行がない）
cat README.md

8.6 ブランチの削除と名前変更

不要になったブランチは削除できます。また、ブランチ名を変更することもできます。

ブランチの削除

ターミナル 
# マージ済みのブランチを削除（-d）
git branch -d feature-about
# 出力：
# Deleted branch feature-about (was a1b2c3d).

# マージしていないブランチを強制削除（-D）
git branch -D feature-experimental
# 出力：
# Deleted branch feature-experimental (was x9y8z7w).

注意

-dオプションは、マージ済みのブランチのみ削除できます。マージしていないブランチを削除する場合は-D（大文字）を使いますが、コミットが失われるため注意が必要です。

ブランチ名の変更

ターミナル 
# 現在いるブランチの名前を変更
git branch -m new-branch-name

# 別のブランチの名前を変更
git branch -m old-name new-name

8.7 ブランチの命名規則とベストプラクティス

ブランチ名は、何の作業をしているかがわかるように命名することが重要です。一般的な命名規則を紹介します。

よく使われるプレフィックス（接頭辞）

プレフィックス 用途 例
feature/ 新機能の開発 feature/user-login
bugfix/ バグ修正 bugfix/header-display
hotfix/ 緊急のバグ修正 hotfix/security-patch
release/ リリース準備 release/v1.2.0
docs/ ドキュメント更新 docs/update-readme
refactor/ リファクタリング refactor/optimize-query

命名のベストプラクティス

良い例

  • feature/user-authentication
  • bugfix/cart-total-calculation
  • docs/api-documentation

悪い例

  • test（何のテストか不明）
  • fix（何を修正したか不明）
  • new_feature（スペースでなく-を使う）

ポイント

  • 英小文字とハイフン（-）を使う
  • 何の作業か一目でわかるようにする
  • 長すぎず、短すぎない（3〜5単語程度）
  • チームで命名規則を統一する

理解度確認クイズ

  1. Q1. ブランチを使うメリットを3つ挙げてください。

    回答を表示

    以下のようなメリットがあります（3つ挙げる）：

    • 複数の機能を同時並行で開発できる
    • 実験的な変更を安全に試せる
    • mainブランチの安定性を保てる
    • チームメンバーが独立して作業できる

  2. Q2. mainブランチの役割は何ですか？

    回答を表示

    mainブランチは本番環境にデプロイする安定したコードを保持する「本流」のブランチです。常に動作する状態を保ち、基本的には直接変更せず、ブランチで開発した内容をマージして更新します。

  3. Q3. 「feature-login」というブランチを作成して、そのブランチに切り替えるコマンドを1つで実行するにはどうすればよいですか？

    回答を表示

    以下のいずれかのコマンドを使います：

    git switch -c feature-login

    または

    git checkout -b feature-login

  4. Q4. git branchコマンドで、現在いるブランチはどのように表示されますか？

    回答を表示

    現在いるブランチには * マークが付きます。例えば、mainブランチにいる場合は「* main」と表示されます。

  5. Q5. ブランチ名の命名規則として推奨されることを2つ挙げてください。

    回答を表示

    以下のような規則が推奨されます（2つ挙げる）：

    • 英小文字とハイフン（-）を使う
    • プレフィックス（feature/, bugfix/等）を付けて用途を明確にする
    • 何の作業か一目でわかる名前にする
    • 長すぎず短すぎない（3〜5単語程度）
    • チームで命名規則を統一する

第8章のまとめ

  • ブランチは開発の歴史を分岐させ、並行作業を可能にする仕組み
  • ブランチを使うことで、並行開発・実験的変更・安定性確保が実現できる
  • mainブランチは常に動作する安定した状態を保つ本流のブランチ
  • git branchでブランチの作成・一覧表示、git switch/checkoutで切り替え
  • ブランチ名には用途がわかるプレフィックスを付けるのがベストプラクティス
