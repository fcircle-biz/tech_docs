第7章

リモートリポジトリとの連携 - clone, push, pull

コマンドラインでGitHubと連携する方法を学びます。ローカルで作成したリポジトリをGitHubにアップロードしたり、既存のリポジトリを複製して作業を開始する方法を習得しましょう。

この章で学ぶこと

  • git remoteでリモートリポジトリを設定する方法
  • git pushでGitHubにアップロードする手順
  • git pullでGitHubから最新の変更を取得する方法
  • git cloneで既存リポジトリを複製する方法
  • HTTPS接続とSSH接続の認証設定を理解する
  • よくあるエラーとその解決方法を知る

7.1 リモートリポジトリとは

これまでの章では、自分のパソコン（ローカル）でGitを使ってファイルの履歴を管理してきました。しかし、ローカルだけで管理していると、以下のような問題があります：

ローカルのみでの管理の問題点

  • バックアップがない - パソコンが壊れるとデータが全て失われる
  • 他の人と共有できない - チーム開発ができない
  • 別のパソコンで作業できない - 会社と自宅で同じコードを扱えない
  • 公開できない - オープンソースとして公開したい場合に困る

リモートリポジトリとは、インターネット上（クラウド）に保存されるGitリポジトリのことです。GitHub、GitLab、Bitbucketなどのサービスが、リモートリポジトリのホスティング（保管・公開）機能を提供しています。

ローカルとリモートの関係

ローカルリポジトリは自分のパソコン上、リモートリポジトリはGitHub等のクラウド上にあります。git pushでアップロード、git pullでダウンロードします。

リモートリポジトリのメリット

  • 安全なバックアップ - クラウド上に履歴が保存される
  • チーム開発が可能 - 複数人で同じリポジトリを共有できる
  • どこからでもアクセス - インターネットがあればどのパソコンからでも作業できる
  • コードレビュー - プルリクエスト機能で変更内容を確認してもらえる
  • 公開・共有 - オープンソースとして世界に公開できる

7.2 git remote - リモートリポジトリの設定

ローカルリポジトリとリモートリポジトリを連携するには、まずGitHub上にリポジトリを作成し、その後ローカルから「どのリモートリポジトリに接続するか」を設定する必要があります。

リモートリポジトリの設定手順

実習 7-1: リモートリポジトリの設定

既存のローカルリポジトリをGitHubのリモートリポジトリと連携させます。

手順

  1. 1 GitHubにログインし、画面右上の「+」→「New repository」をクリック
  2. 2 リポジトリ名を入力（例: my-first-repo）、「Create repository」をクリック
  3. 3 表示されたHTTPS URLをコピー（例: https://github.com/username/my-first-repo.git）
  4. 4 ローカルのGitリポジトリフォルダに移動し、以下のコマンドを実行
リモートリポジトリを追加 
# リモートリポジトリを「origin」という名前で追加
git remote add origin https://github.com/username/my-first-repo.git

# 設定されたリモートリポジトリを確認
git remote -v

「origin」とは？

originは、リモートリポジトリに付けるニックネーム（別名）です。慣例的に、メインのリモートリポジトリには「origin」という名前を使います。

複数のリモートリポジトリを設定することもできますが、通常は1つのリモート（origin）だけを使います。

git remoteの主なコマンド

コマンド 説明
git remote -v 設定されているリモートリポジトリを一覧表示
git remote add origin URL 新しいリモートリポジトリを追加
git remote remove origin リモートリポジトリを削除
git remote rename old new リモートの名前を変更
git remote show origin リモートの詳細情報を表示

7.3 git push - GitHubにアップロード

git pushは、ローカルリポジトリのコミット履歴をリモートリポジトリ（GitHub等）にアップロードするコマンドです。

初回のpush

初回push（上流ブランチを設定） 
# -u オプションで上流ブランチを設定しながらpush
git push -u origin main

-u オプションとは？

-u（または--set-upstream）は、ローカルブランチとリモートブランチの「紐付け」を設定するオプションです。

一度設定すると、次回以降はgit pushだけで同じリモートブランチにpushできるようになります。

2回目以降のpush

2回目以降のpush 
# 上流ブランチ設定後は、これだけでOK
git push

pushの流れ

エリア participant L as ローカル

git addでステージング、git commitでローカルに記録、git pushでリモートにアップロードする流れです。

動作確認

GitHubのWebサイトでリポジトリのページを開き、pushした内容が反映されているか確認しましょう。

期待される結果

  • GitHubのリポジトリページにファイルが表示される
  • コミット履歴が「XX commits」として表示される
  • 最新のコミットメッセージが表示される

7.4 git pull - GitHubから最新の変更を取得

git pullは、リモートリポジトリの最新の変更をローカルリポジトリにダウンロード・統合するコマンドです。

リモートから最新の変更を取得 
# リモートの変更をローカルに取り込む
git pull

pullが必要な場面

  • チーム開発 - 他の人がpushした変更を取り込む
  • 別のパソコン - 会社でpushした内容を自宅のパソコンで取得
  • GitHub上で編集 - GitHub Webエディタで変更した内容をローカルに反映

git pull = git fetch + git merge

実は、git pullは以下の2つの操作を同時に行っています：

  1. git fetch - リモートの変更をダウンロード
  2. git merge - ダウンロードした変更をローカルブランチに統合

通常はgit pullを使えば十分ですが、慎重に確認してからマージしたい場合は、fetchとmergeを分けて実行することもできます。

pullとpushの使い分け

コマンド 方向 タイミング
git push ローカル → リモート（アップロード） 自分が変更をコミットした後
git pull リモート → ローカル（ダウンロード） 作業を始める前、他の人の変更を取り込むとき

ベストプラクティス

作業を始める前にgit pullで最新の状態を取得する習慣をつけましょう。これにより、他の人の変更と競合するリスクを減らせます。

7.5 git clone - 既存リポジトリの複製

git cloneは、GitHub上に既に存在するリポジトリを、自分のパソコンに丸ごとコピーするコマンドです。全てのコミット履歴も一緒にダウンロードされます。

リポジトリをクローン 
# GitHubのリポジトリをクローン
git clone https://github.com/username/repository-name.git

# クローンしたフォルダに移動
cd repository-name

# リモートが自動的に設定されている
git remote -v

cloneとremote addの違い

項目 git clone git init + git remote add
使う場面 既存のリモートリポジトリを複製する ローカルで新規作成してからリモートと連携
初期状態 全てのファイルと履歴がダウンロード済み 空のリポジトリから開始
リモート設定 自動的に「origin」として設定される 手動でgit remote addが必要
よくある例 オープンソースプロジェクトに参加、チーム開発開始 新しいプロジェクトを自分で立ち上げる

実習 7-2: 公開リポジトリをクローン

GitHubの公開リポジトリをクローンして、中身を確認してみましょう。

手順

  1. 1 任意のGitHubリポジトリページを開く（例: 有名なOSSプロジェクト）
  2. 2 緑色の「Code」ボタンをクリックし、HTTPS URLをコピー
  3. 3 ターミナルでgit cloneコマンドを実行
  4. 4 クローンしたフォルダに移動し、git logで履歴を確認

7.6 認証設定 - HTTPS接続とSSH接続

GitHubにpushやpullをする際、GitHubが「本当にあなたか」を確認するための認証が必要です。認証方式には主にHTTPSとSSHの2種類があります。

項目 HTTPS SSH
URL形式 https://github.com/... git@github.com:...
認証方法 Personal Access Token（PAT） SSH鍵ペア
初心者向け ◎ 設定が簡単 △ 鍵生成の知識が必要
パスワード入力 都度必要（キャッシュ可） 不要
セキュリティ 高（トークンに有効期限設定可） 高（秘密鍵の管理が重要）

HTTPS認証 - Personal Access Token（PAT）の作成

GitHubは2021年8月以降、HTTPSでのパスワード認証を廃止しました。代わりにPersonal Access Token（PAT）を使います。

実習 7-3: Personal Access Tokenの作成

手順

  1. 1 GitHubにログイン → 右上のアイコン → Settings
  2. 2 左メニュー最下部の「Developer settings」をクリック
  3. 3 「Personal access tokens」→「Tokens (classic)」→「Generate new token (classic)」
  4. 4 Note（メモ）に「Git CLI」などと入力、有効期限を選択
  5. 5 「repo」にチェックを入れる（リポジトリへのフルアクセス権限）
  6. 6 「Generate token」をクリック
  7. 7 表示されたトークン（ghp_xxxxx...）をコピーして安全な場所に保存

重要: トークンの保管

Personal Access Tokenは一度しか表示されません。必ずコピーして安全な場所（パスワードマネージャー等）に保存してください。紛失した場合は新しいトークンを再発行する必要があります。

トークンを使ったpush

git pushを実行すると、ユーザー名とパスワードを求められます。パスワードの欄にPATを貼り付けてください。

PATを使ったpush 
# pushを実行
git push -u origin main

# プロンプトが表示される
# Username: あなたのGitHubユーザー名
# Password: (ここにPATを貼り付け) ← 画面には表示されません

SSH認証の設定（参考）

SSH認証を使うと、毎回トークンを入力せずにpush/pullできます。ただし、SSH鍵の生成と登録が必要です。

初心者へのアドバイス

最初はHTTPS（PAT）を使うことをお勧めします。SSH鍵の設定は、Gitの基本操作に慣れてから学習しても遅くありません。SSH設定の詳細は、GitHubの公式ドキュメントを参照してください。

7.7 よくあるエラーと解決方法

トラブルシューティング

error: failed to push some refs

原因: リモートに自分のローカルにない変更がある（他の人がpushした、または別のパソコンで変更した）

解決方法:

# まずpullして最新の状態を取得
git pull

# コンフリクトがあれば解決して、再度コミット
# その後、改めてpush
git push
fatal: remote origin already exists

原因: 既に「origin」という名前のリモートが設定されている

解決方法:

# 既存のoriginを削除
git remote remove origin

# 改めて追加
git remote add origin https://github.com/username/repo.git

# または、URLを変更
git remote set-url origin https://github.com/username/repo.git
Authentication failed（認証失敗）

原因: Personal Access Token（PAT）が正しくない、または期限切れ

解決方法:

  • GitHubでPATを新規作成または再発行
  • Passwordの欄に正しいPATを貼り付ける
  • キャッシュされた古い認証情報をクリア（credential helperの設定を確認）
fatal: refusing to merge unrelated histories

原因: ローカルとリモートのコミット履歴が完全に異なる（別々に作られた）

解決方法:

# 無関係な履歴の統合を許可してpull
git pull origin main --allow-unrelated-histories

# コンフリクトがあれば解決してコミット
# その後、push
git push -u origin main
Permission denied (publickey)

原因: SSH接続でSSH鍵が正しく設定されていない

解決方法:

  • SSH鍵をGitHubに登録しているか確認
  • または、HTTPS URLに変更して接続（PATを使用）
  • git remote set-url origin https://github.com/username/repo.git

理解度確認クイズ

  1. Q1. リモートリポジトリを使うメリットを3つ挙げてください。

    回答を表示

    以下のようなメリットがあります（3つ挙げる）：

    • クラウド上に安全なバックアップが取られる
    • 複数人でリポジトリを共有してチーム開発ができる
    • どのパソコンからでもアクセスして作業できる
    • プルリクエストでコードレビューができる
    • オープンソースとして世界に公開できる

  2. Q2. git push と git pull の違いを説明してください。

    回答を表示

    git pushは、ローカルリポジトリのコミット履歴をリモートリポジトリ（GitHub等）にアップロードするコマンドです。自分が変更をコミットした後に使います。

    git pullは、リモートリポジトリの最新の変更をローカルリポジトリにダウンロード・統合するコマンドです。作業を始める前や、他の人の変更を取り込むときに使います。

  3. Q3. git clone と git init + git remote add の違いは何ですか？

    回答を表示

    git cloneは、GitHub上に既に存在するリポジトリを自分のパソコンに丸ごとコピーします。全てのファイルとコミット履歴がダウンロードされ、リモートも自動的に設定されます。

    git init + git remote addは、ローカルで新規にリポジトリを作成してから、後でリモートリポジトリと連携します。空のリポジトリから開始し、リモート設定は手動で行う必要があります。

  4. Q4. HTTPS認証でパスワードの代わりに何を使いますか？

    回答を表示

    Personal Access Token（PAT）を使います。GitHubは2021年8月以降、HTTPSでの通常のパスワード認証を廃止しました。GitHubの設定画面でPATを生成し、git pushやpull時のパスワード欄に貼り付けて認証します。

  5. Q5. 「origin」とは何ですか？

    回答を表示

    「origin」は、リモートリポジトリに付けるニックネーム（別名）です。慣例的に、メインのリモートリポジトリには「origin」という名前を使います。git remote add origin URLで設定し、以降はoriginという名前でリモートを参照できます。

第7章のまとめ

  • リモートリポジトリはクラウド上に保存されるGitリポジトリで、バックアップやチーム開発に役立つ
  • git remote add origin URLでリモートリポジトリを設定する
  • git pushでローカルの変更をGitHubにアップロード、git pullでGitHubから最新の変更を取得
  • git cloneで既存のリポジトリを丸ごと複製できる
  • HTTPS認証ではPersonal Access Token（PAT）を使う
  • よくあるエラーは落ち着いてメッセージを読み、適切な対処をすれば解決できる
