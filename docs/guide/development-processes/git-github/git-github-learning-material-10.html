プルリクエスト - チーム開発の基本ワークフロー

GitHubのプルリクエスト機能を学び、チーム開発の基本を理解します。コードレビューと変更の可視化により、品質の高い開発を実現する方法を習得しましょう。

この章で学ぶこと

  • プルリクエスト（Pull Request）とは何かを理解する
  • なぜプルリクエストを使うのか、そのメリットを知る
  • ブランチをプッシュしてプルリクエストを作成する
  • レビューとコメント機能の使い方を習得する
  • プルリクエストのマージ方法（Merge, Squash, Rebase）を理解する

10.1 プルリクエスト（Pull Request）とは

プルリクエスト（Pull Request、略してPR）は、GitHubが提供する機能で、自分が行った変更を他の人にレビューしてもらい、メインブランチに統合してもらうための「依頼」です。

プルリクエストの本質

プルリクエストは、単なる「マージの依頼」ではありません。チーム内でのコミュニケーションツールであり、コードの品質を保つための重要なプロセスです。

「この変更を取り込んでください」というシンプルな依頼から、「なぜこの変更が必要か」「どのように実装したか」「何をテストしたか」といった情報共有まで、開発における多くの情報がプルリクエストに集約されます。

10.2 なぜプルリクエストを使うのか

プルリクエストには、コードの品質を高め、チーム開発を円滑にするための多くのメリットがあります。

1. コードレビューの促進

他の開発者に変更内容を見てもらうことで、バグの早期発見や品質向上が期待できます。

  • 見落としていたバグを発見できる
  • より良い実装方法の提案を受けられる
  • コーディング規約の統一ができる

2. 変更内容の可視化

何が変わったのか、なぜ変えたのかが一目でわかります。

  • 変更されたファイルと行数が明確に表示される
  • コミット履歴とともに変更の経緯がわかる
  • 過去のPRを見れば、なぜその変更をしたかがわかる

3. チーム内コミュニケーション

プルリクエストを通じて、チームメンバー間で技術的な議論ができます。

  • コード上でコメントを付けて議論できる
  • 実装方法について意見交換ができる
  • 知識の共有と学習の機会になる

4. 本番環境の安全性

いきなりmainブランチに変更を加えず、レビュー後にマージすることで安全性が高まります。

  • 本番環境に問題のあるコードが混入するリスクを減らせる
  • CI/CDツールと連携して自動テストを実行できる
  • 承認プロセスを設けることでガバナンスを強化できる

10.3 プルリクエストの作成手順

実際にプルリクエストを作成してみましょう。ここでは、新機能を追加する例で説明します。

ステップ1: ブランチ作成と変更のコミット

まず、新しいブランチを作成して変更を加えます。

ブランチ作成と切り替え 
# 新しいブランチを作成して切り替え
git switch -c feature/add-user-profile

# または
git checkout -b feature/add-user-profile
ファイルを編集してコミット 
# ファイルを編集（例: user.py）
# エディタで変更を加える

# 変更をステージング
git add user.py

# コミット
git commit -m "Add user profile feature"

ステップ2: ブランチをGitHubにプッシュ

作成したブランチをリモートリポジトリにプッシュします。

ブランチのプッシュ 
# ブランチをリモートにプッシュ
git push -u origin feature/add-user-profile

# -u オプションで、ローカルブランチとリモートブランチを紐付ける
# 次回からは git push だけでOK

ヒント: プッシュ後、ターミナルに「Create a pull request for 'feature/add-user-profile'」というメッセージとURLが表示されることがあります。このURLをクリックすると、すぐにプルリクエスト作成画面に移動できます。

ステップ3: GitHubでプルリクエストを作成

  1. GitHubのリポジトリページにアクセス
  2. 黄色い「Compare & pull request」ボタンをクリック（ブランチプッシュ直後に表示される）
  3. または「Pull requests」タブ → 「New pull request」をクリック
  4. ベースブランチ（main）とコンペアブランチ（feature/add-user-profile）を選択
  5. タイトルと説明を記入
  6. 「Create pull request」をクリック

10.4 プルリクエストの説明の書き方

良いプルリクエストの説明は、レビュアーが変更内容を理解しやすくし、レビューを効率的に進めるために重要です。

良いPR説明に含めるべき情報

  • 何を変更したか - 変更の概要
  • なぜ変更したか - 変更の理由・背景
  • どのように実装したか - 実装の方針・アプローチ
  • 何をテストしたか - テスト内容や確認事項
  • スクリーンショット - UI変更の場合

プルリクエストのテンプレート例

PR説明のテンプレート 
## 概要
ユーザープロフィール機能を追加しました。

## 変更内容
- ユーザープロフィールページの作成
- プロフィール編集フォームの実装
- プロフィール画像のアップロード機能

## 理由
Issue #123 で要望があったため、ユーザーが自分の情報を
編集できる機能を追加しました。

## テスト内容
- [ ] プロフィールページが正しく表示される
- [ ] プロフィール情報を編集・保存できる
- [ ] 画像アップロードが正常に動作する
- [ ] モバイル表示でもレイアウトが崩れない

## スクリーンショット
（UI変更がある場合は画像を添付）

## 関連Issue
Closes #123

10.5 レビューとコメント機能

GitHubのプルリクエストには、コードレビューを支援する様々な機能があります。

コメントの種類

一般コメント

PR全体に対する質問や感想を書くコメント。タイムライン上に表示されます。

行コメント

特定のコード行に対するコメント。該当行に直接紐付けられます。

承認（Approve）

変更内容に問題がなく、マージしても良いと判断した場合に使用します。

変更要求（Request changes）

修正が必要な場合に使用。修正されるまでマージをブロックできます。

レビューの基本手順

  1. 1. 「Files changed」タブを開いて変更内容を確認
  2. 2. コード行の横の「+」アイコンをクリックしてコメントを追加
  3. 3. 「Start a review」を選択（コメントをまとめて投稿できる）
  4. 4. すべてのコメントを書き終えたら「Review changes」をクリック
  5. 5. Approve / Request changes / Comment を選択して送信

良いレビューコメントのポイント

  • 具体的な指摘を心がける（「ここが悪い」ではなく「こうするとより良い」）
  • 建設的なフィードバックをする（批判ではなく改善提案）
  • 質問形式を使う（「なぜこの実装にしましたか？」）
  • 良い点も積極的に褒める（モチベーション向上）

10.6 変更の追加とレビューへの対応

レビューでコメントや修正依頼を受けた場合、追加の変更をコミット・プッシュすることで、同じプルリクエストに反映されます。

修正を追加コミット 
# レビュー指摘に基づいて修正
# ファイルを編集

# 修正をコミット
git add .
git commit -m "Fix review comments: improve error handling"

# プッシュ（ブランチ指定は省略可能）
git push

# 自動的に既存のPRに追加される

重要なポイント

  • 同じブランチにプッシュすれば、自動的にPRが更新される
  • 新しいPRを作る必要はない
  • コメントに返信して、どう対応したかを説明すると親切
  • 「Resolve conversation」で対応済みコメントをクローズできる

10.7 プルリクエストのマージ方法

GitHubでは、プルリクエストをマージする際に3つの方法から選択できます。それぞれに特徴があり、プロジェクトの方針に応じて使い分けます。

1. Merge commit（マージコミット）

全てのコミット履歴を保持したまま、マージコミットを作成してmainブランチに統合します。

メリット：

  • 全てのコミット履歴が残る
  • ブランチの分岐・統合が明確

デメリット：

  • 履歴が複雑になりがち
  • 細かいコミットも全て残る

2. Squash and merge（スカッシュマージ）

複数のコミットを1つにまとめて（squash）、mainブランチに統合します。

メリット：

  • 履歴がシンプルで読みやすい
  • 1つのPRが1つのコミットになる

デメリット：

  • 細かい変更の履歴が失われる
  • コミット単位でのrevertが難しい

3. Rebase and merge（リベースマージ）

ブランチのコミットをmainブランチの先頭に追加します。マージコミットは作成されません。

メリット：

  • 履歴が一直線でシンプル
  • 各コミットの履歴が残る

デメリット：

  • マージした事実が履歴に残らない
  • コミットハッシュが変わる

マージ方法の比較

マージ方法 履歴の見やすさ 詳細な履歴 推奨ケース
Merge commit △ ◎ 詳細履歴が重要な大規模プロジェクト
Squash and merge ◎ △ シンプルな履歴が好ましい小〜中規模プロジェクト
Rebase and merge ◎ ○ 一直線の履歴を保ちたいプロジェクト

初心者へのおすすめ

最初はSquash and mergeを使うのがおすすめです。履歴がシンプルで理解しやすく、1つのPRが1つのコミットになるため、管理が容易です。

10.8 ドラフトプルリクエストの活用

ドラフトプルリクエスト（Draft Pull Request）は、まだレビューの準備ができていない、作業途中のプルリクエストを作成する機能です。

ドラフトPRの用途

  • 早期のフィードバック収集
  • 進捗の可視化
  • 実装方針の相談
  • CI/CDの動作確認

通常PRとの違い

  • マージができない（保護）
  • 「Draft」のラベルが付く
  • レビュー依頼の通知が送られない
  • 準備ができたら通常PRに変換できる
ドラフトPRの作成 
1. 通常のPR作成画面で「Create pull request」の横の▼をクリック
2. 「Create draft pull request」を選択
3. 準備ができたら「Ready for review」をクリックして通常PRに変換

実習: プルリクエストの作成とマージ

実際にプルリクエストを作成してマージするまでの流れを体験しましょう。

  1. 新しいブランチを作成 
    git switch -c feature/update-readme
  2. README.mdを編集

    ファイルに好きな内容を追加してください

  3. 変更をコミット 
    git add README.md
git commit -m "Update README with project description"
  4. ブランチをプッシュ 
    git push -u origin feature/update-readme
  5. GitHubでプルリクエストを作成

    タイトルと説明を記入しましょう

  6. 自分でレビューしてマージ

    「Files changed」を確認後、Squash and mergeでマージ

  7. ブランチを削除 
    git switch main
git pull
git branch -d feature/update-readme

理解度確認クイズ

  1. Q1. プルリクエストを使う主なメリットを3つ挙げてください。

    回答を表示

    以下のようなメリットがあります（3つ挙げる）：

    • コードレビューによる品質向上とバグの早期発見
    • 変更内容の可視化と履歴管理
    • チーム内でのコミュニケーションと知識共有
    • 本番環境への安全な変更適用

  2. Q2. ブランチをプッシュした後、プルリクエストを作成する手順を説明してください。

    回答を表示
    1. GitHubのリポジトリページにアクセス
    2. 「Compare & pull request」ボタンをクリック（またはPull requestsタブから新規作成）
    3. ベースブランチとコンペアブランチを確認
    4. タイトルと説明を記入
    5. 「Create pull request」をクリック

  3. Q3. GitHubの3つのマージ方法の違いを説明してください。

    回答を表示
    • Merge commit: 全てのコミット履歴を保持し、マージコミットを作成。履歴が複雑になるが詳細が残る。
    • Squash and merge: 複数のコミットを1つにまとめてマージ。履歴がシンプルになる。
    • Rebase and merge: コミットをmainの先頭に追加。一直線の履歴になる。

  4. Q4. ドラフトプルリクエストはどのような場合に使いますか？

    回答を表示

    作業途中で早期のフィードバックが欲しい場合や、実装方針を相談したい場合、進捗を可視化したい場合などに使います。マージはできず、準備ができたら通常のプルリクエストに変換できます。

  5. Q5. レビューでコメントを受けた後、どのように対応しますか？

    回答を表示

    指摘された内容を修正し、同じブランチにコミット・プッシュします。自動的に既存のPRに追加されます。コメントに返信して対応内容を説明し、対応済みの場合は「Resolve conversation」をクリックします。

第10章のまとめ

  • プルリクエストは変更をレビューしてもらい、マージしてもらう依頼
  • コードレビュー、変更の可視化、チーム内コミュニケーションを促進する
  • ブランチをプッシュ後、GitHubでPRを作成し、タイトルと説明を記入する
  • レビューでのコメントに対応し、同じブランチにプッシュすると自動的にPRが更新される
  • マージ方法はMerge commit、Squash and merge、Rebase and mergeの3種類がある
