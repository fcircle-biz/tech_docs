実践Git/GitHub - 個人プロジェクトでの活用

これまで学んだ内容を統合し、実際の個人プロジェクトでGit/GitHubを活用する方法を学びます。.gitignore、README、ライセンス、タグ、Issues、GitHub Pagesなど、プロジェクト運用に必要な知識を身につけましょう。

この章で学ぶこと

  • .gitignoreファイルで不要なファイルを管理から除外する方法
  • README.mdの書き方とMarkdown記法の基本
  • ライセンスの選び方とプロジェクトへの追加方法
  • 効果的なコミットメッセージの書き方
  • タグとリリースの作成、Issues機能、GitHub Pagesの活用

10.1 .gitignoreファイル - 管理対象外のファイルを指定する

プロジェクトには、Gitで管理すべきでないファイルが存在します。例えば、パスワードなどの機密情報、ビルドで自動生成されるファイル、OSが作成する一時ファイルなどです。.gitignoreファイルを使うと、これらのファイルをGitの管理対象から除外できます。

Gitで管理してはいけないファイル

  • 機密情報: パスワード、APIキー、秘密鍵（.env, credentials.json等）
  • 依存パッケージ: node_modules/, vendor/, venv/など（サイズが大きい）
  • ビルド成果物: dist/, build/, *.classなど（再生成可能）
  • OS/エディタの一時ファイル: .DS_Store, Thumbs.db, .ideaなど

.gitignoreの基本的な書き方

.gitignore 
# コメントは # で始める

# 特定のファイル名を指定
.env
secret.key

# ワイルドカードでパターン指定
*.log
*.tmp

# ディレクトリ全体を除外（末尾に/）
node_modules/
dist/
build/

# 特定のディレクトリ内のファイル
logs/*.log

# 否定パターン（除外の例外）
!important.log

# ディレクトリ階層を問わず除外
**/.DS_Store
パターン 意味 例
*.log .logで終わるファイル全て error.log, app.log
build/ buildディレクトリ全体 build/配下の全ファイル
**/temp どの階層のtempも対象 src/temp, lib/temp
!keep.log keep.logは除外しない *.logの例外として

gitignore.ioを活用しよう

gitignore.ioでは、使用言語やフレームワークを選ぶだけで、適切な.gitignoreファイルを自動生成できます。

例: 「Python」「Node」「VisualStudioCode」と入力すると、それぞれに適した除外設定が生成されます。

10.2 README.mdの書き方 - プロジェクトの顔を作る

README.mdは、プロジェクトの「顔」となるファイルです。GitHubでリポジトリを開くと最初に表示され、プロジェクトの概要、使い方、インストール方法などを伝えます。「.md」はMarkdown形式を意味し、シンプルな記法で見やすい文書を作成できます。

Markdown記法の基本

README.md（Markdown記法） 
# プロジェクト名（h1見出し）

## 概要（h2見出し）
このプロジェクトは○○するためのツールです。

### 特徴（h3見出し）
- リスト項目1
- リスト項目2
- リスト項目3

## インストール方法

```bash
npm install my-project
```

## 使い方

**太字**のテキストや*斜体*のテキストが使えます。

[リンクテキスト](https://example.com)

![画像の説明](image.png)

良いREADMEの構成要素

flowchart TB subgraph README[README.mdの構成] A[プロジェクト名] --> B[概要・説明] B --> C[機能・特徴] C --> D[インストール方法] D --> E[使い方・例] E --> F[設定方法] F --> G[コントリビュート方法] G --> H[ライセンス] end
README.md（実践的なテンプレート） 
# My Awesome Project

シンプルで使いやすいToDoアプリです。

## 特徴

- シンプルなUI
- ローカルストレージに保存
- レスポンシブデザイン

## 必要条件

- Node.js 18以上
- npm または yarn

## インストール

```bash
git clone https://github.com/username/my-project.git
cd my-project
npm install
```

## 使い方

```bash
npm start
```

ブラウザで http://localhost:3000 を開いてください。

## ライセンス

MIT License

10.3 ライセンスの選び方 - コードの利用条件を明示する

ライセンスは、あなたのコードを他の人がどのように使っていいかを定めるものです。ライセンスがないと、法的には「誰も使ってはいけない」と解釈される可能性があります。オープンソースとして公開する場合は、必ずライセンスを明示しましょう。

ライセンス 特徴 おすすめ用途
MIT License 最も緩い。商用利用、改変、再配布すべてOK 広く使ってほしい場合（最も人気）
Apache 2.0 MITに似るが、特許に関する条項あり 企業での利用を想定する場合
GPL 3.0 改変版も同じライセンスで公開必須 派生作品もオープンソースにしたい場合
BSD 3-Clause MITに似るが、推奨条項がより明確 学術・研究プロジェクト

迷ったらMITライセンス

個人プロジェクトで迷った場合は、MITライセンスがおすすめです。最もシンプルで理解しやすく、世界で最も使われているオープンソースライセンスです。著作権表示さえすれば、商用利用、改変、再配布が自由にできます。

GitHubでライセンスを追加する方法

  1. リポジトリのトップページで「Add file」→「Create new file」をクリック
  2. ファイル名に「LICENSE」と入力
  3. 右側に「Choose a license template」ボタンが表示される
  4. ライセンスを選択し、年と名前を入力
  5. 「Commit new file」でコミット

10.4 効果的なコミットメッセージの書き方

コミットメッセージは「何をしたか」「なぜしたか」を将来の自分やチームメンバーに伝える重要な情報です。適当に書いてしまうと、後から履歴を追うのが困難になります。

悪い例

  • fix
  • update
  • 修正
  • asdfgh
  • WIP

良い例

  • Add user login feature
  • Fix null pointer exception in search
  • Update README with install guide
  • Remove deprecated API calls
  • Refactor database connection logic

コミットメッセージの規約（Conventional Commits）

多くのプロジェクトでは、コミットメッセージに「プレフィックス」を付ける規約が使われています。

コミットメッセージの形式 
# 基本形式
type: 変更の要約

# 例
feat: ユーザー登録機能を追加
fix: ログイン時のエラーハンドリングを修正
docs: READMEにインストール手順を追加
style: コードフォーマットを統一
refactor: 認証処理をモジュール化
test: ユーザー登録のテストを追加
chore: 依存パッケージを更新
プレフィックス 用途
feat: 新機能の追加
fix: バグの修正
docs: ドキュメントのみの変更
style: コードの意味に影響しない変更（空白、フォーマット等）
refactor: 機能追加でもバグ修正でもないコード変更
test: テストの追加・修正
chore: ビルドプロセスや補助ツールの変更

10.5 タグとリリース - バージョンを管理する

タグは、特定のコミットに「名前」を付ける機能です。通常、バージョン番号（v1.0.0など）を付けて、リリースポイントをマークするために使います。GitHubのリリース機能と組み合わせると、ダウンロード可能なパッケージを配布できます。

タグの作成方法

ターミナル 
# 軽量タグの作成（単純なポインタ）
git tag v1.0.0

# 注釈付きタグの作成（推奨：メッセージ付き）
git tag -a v1.0.0 -m "最初の正式リリース"

# タグの一覧表示
git tag

# 特定のコミットにタグを付ける
git tag -a v0.9.0 abc1234 -m "ベータ版"

# タグをリモートにプッシュ
git push origin v1.0.0

# 全てのタグをプッシュ
git push origin --tags

セマンティックバージョニング

バージョン番号の付け方にはセマンティックバージョニング（SemVer）という規約があります。

flowchart LR V["v1.2.3"] --> Major["1 = メジャー
後方互換性のない変更"] V --> Minor["2 = マイナー
後方互換性のある機能追加"] V --> Patch["3 = パッチ
バグ修正"]

GitHubでリリースを作成

  1. リポジトリページで「Releases」をクリック
  2. 「Create a new release」をクリック
  3. タグを選択または新規作成
  4. リリースタイトルと説明を入力
  5. 必要に応じてバイナリファイルを添付
  6. 「Publish release」をクリック

10.6 Issues機能 - タスクと課題を管理する

Issuesは、GitHubに組み込まれたタスク管理・バグ追跡システムです。個人プロジェクトでも、やるべきことをIssueとして管理すると、作業の見通しが良くなります。

Issuesの活用方法

バグ報告

発見したバグを記録。再現手順、期待する動作、実際の動作を書く。

機能要望

追加したい機能をIssueとして登録。実装の優先度を決められる。

タスク管理

やるべきことをIssueで管理。チェックリストで進捗を可視化。

質問・議論

設計の相談や方針の議論をIssueで行い、記録を残す。

コミットでIssueを閉じる

コミットメッセージに特定のキーワードとIssue番号を含めると、自動的にIssueをクローズできます。

コミットメッセージ 
# Issue #123 を自動的にクローズするキーワード
fix: ログインエラーを修正 (closes #123)
fix: バグを修正 (fixes #123)
feat: 新機能を追加 (resolves #123)

# 複数のIssueを閉じる
fix: 複数のバグを修正 (closes #123, #124, #125)

10.7 GitHub Pages - 静的サイトを無料で公開

GitHub Pagesは、リポジトリのファイルをそのままWebサイトとして公開できる機能です。HTMLファイルやJekyllを使ったブログ、ドキュメントサイトなどを無料でホスティングできます。

GitHub Pagesの有効化手順

  1. リポジトリの「Settings」タブを開く
  2. 左メニューから「Pages」を選択
  3. 「Source」でブランチ（通常main）とフォルダ（/ または /docs）を選択
  4. 「Save」をクリック
  5. 数分後、https://username.github.io/repository-name/でアクセス可能に
flowchart LR A[リポジトリの
index.html] -->|GitHub Pages有効化| B[Webサイトとして公開] B --> C["https://username.github.io/repo/"]

GitHub Pagesの活用例

  • ポートフォリオサイト: 自己紹介や作品集を公開
  • プロジェクトのドキュメント: 使い方やAPIリファレンス
  • ブログ: Jekyllと組み合わせて技術ブログを運営

10.8 日常的なGitワークフロー - 実践的な使い方

個人プロジェクトでの典型的なGitワークフローを紹介します。このパターンを身につければ、効率的にプロジェクトを管理できます。

flowchart TD A[1. 作業開始] --> B[2. 変更を加える] B --> C[3. git status で確認] C --> D[4. git add で追加] D --> E[5. git commit で記録] E --> F{まだ作業ある？} F -->|Yes| B F -->|No| G[6. git push で共有]

個人プロジェクトの推奨ワークフロー

日常的なGit操作 
# 朝: 最新を取得
git pull

# 機能開発開始: ブランチを作成
git checkout -b feature/new-function

# 作業中: こまめにコミット
git add .
git commit -m "feat: 新機能の基本実装"

# 作業完了: mainにマージ
git checkout main
git merge feature/new-function

# GitHubに反映
git push

# 不要になったブランチを削除
git branch -d feature/new-function

実習 10-1: 個人プロジェクトをセットアップする

新しいプロジェクトに必要なファイルを一通り作成してみましょう。

手順

  1. 1 GitHubで新しいリポジトリを作成（README.mdを追加）
  2. 2 ローカルにクローンする
  3. 3 .gitignoreファイルを作成し、適切な除外設定を追加
  4. 4 README.mdを編集し、プロジェクト概要を記述
  5. 5 LICENSEファイルを追加（MITライセンス推奨）
  6. 6 コミットしてGitHubにプッシュ

コマンド例

# リポジトリをクローン
git clone https://github.com/username/my-project.git
cd my-project

# .gitignoreを作成
echo "node_modules/" >> .gitignore
echo ".env" >> .gitignore

# コミットしてプッシュ
git add .
git commit -m "chore: プロジェクト初期設定"
git push

第10章のまとめ

  • .gitignoreで機密情報や不要ファイルを管理対象外にする
  • README.mdはプロジェクトの顔。Markdown記法で見やすく書く
  • ライセンスはオープンソース公開に必須。迷ったらMIT
  • コミットメッセージは「何を」「なぜ」が伝わるように書く
  • タグ/リリースでバージョン管理、Issuesでタスク管理
  • GitHub Pagesで静的サイトを無料公開できる

理解度確認クイズ

  1. Q1. .gitignoreファイルの役割は何ですか？

    回答を表示

    .gitignoreは、Gitの管理対象から除外したいファイルやディレクトリを指定するファイルです。機密情報（パスワード、APIキーなど）、依存パッケージ（node_modules/など）、ビルド成果物、OS固有の一時ファイルなどを除外することで、リポジトリを安全かつ効率的に管理できます。

  2. Q2. セマンティックバージョニングで「v2.1.3」の各数字は何を表しますか？

    回答を表示

    2（メジャー）: 後方互換性のない大きな変更
    1（マイナー）: 後方互換性のある新機能の追加
    3（パッチ）: 後方互換性のあるバグ修正
    この規約に従うことで、バージョン番号を見るだけで変更の影響度が分かります。

  3. Q3. コミットメッセージで「closes #123」と書くとどうなりますか？

    回答を表示

    そのコミットがmainブランチにマージされると、GitHubのIssue #123が自動的にクローズ（閉じられ）ます。closes、fixes、resolvesなどのキーワードが使えます。これにより、コードの変更とIssueの関連付けが明確になり、プロジェクト管理が効率化されます。
