GitHub Desktop入門 - GUIで学ぶ基本操作

GitHub Desktopを使って、マウス操作でGitの基本概念を視覚的に理解しましょう。コマンドを使わずに、リポジトリの作成からコミット、プッシュまでの一連の流れを体験します。

この章で学ぶこと

  • GitHub Desktopの画面構成と各部の役割を理解する
  • リポジトリの作成とファイル変更の記録（コミット）ができる
  • 変更履歴の確認と過去の状態への復元方法を理解する
  • GitHubへのプッシュ・プルの基本操作ができる

3.1 GitHub Desktopとは

GitHub Desktopは、GitHub社が提供する無料のGUI（グラフィカルユーザーインターフェース）ツールです。コマンドラインを使わずに、視覚的な操作でGitとGitHubの機能を利用できます。

GitHub Desktopを使うメリット

  • コマンドを覚えなくても操作できる
  • 変更内容が視覚的に分かりやすい
  • GitHubとの連携がスムーズ
  • 操作ミスを減らせる（確認画面がある）

GUIツールとコマンドラインの比較

Gitを操作する方法は大きく2つあります。それぞれに特徴があるため、状況に応じて使い分けると効率的です。

項目 GUI（GitHub Desktop） コマンドライン
操作方法 マウスでボタンをクリック キーボードでコマンドを入力
学習コスト 低い（直感的） 高い（コマンドを覚える必要あり）
高度な操作 一部制限あり すべての機能を使える
自動化 難しい スクリプト化が可能
おすすめ 初心者、日常的な操作 上級者、複雑な操作

学習のポイント

本ガイドでは、まずGitHub Desktopで概念を理解し、その後コマンドラインを学びます。両方を使えるようになると、状況に応じて最適な方法を選べるようになります。

3.2 GitHub Desktopの画面構成

GitHub Desktopを起動すると、以下のような画面が表示されます。各部分の役割を理解しておくと、操作がスムーズになります。

各部分の役割

リポジトリ選択

現在作業中のリポジトリ（プロジェクト）を表示・切り替えできます。複数のプロジェクトを管理している場合に便利です。

ブランチ選択

現在のブランチを表示・切り替えできます。ブランチについては第7章で詳しく学びます。

変更リスト

変更されたファイルの一覧が表示されます。ファイルを選択すると、右側に詳細な変更内容が表示されます。

差分表示

選択したファイルの変更箇所を色分けして表示します。追加された行は緑、削除された行は赤で表示されます。

用語解説

ステージング（Staging）
コミットに含める変更を選ぶ作業。買い物かごに商品を入れるイメージです。
差分（Diff）
ファイルの変更前と変更後の違い。何をどう変えたかが一目で分かります。
Fetch（フェッチ）
GitHubに新しい変更がないか確認する操作。メールの受信確認のようなものです。

3.3 リポジトリの作成

まず、練習用のリポジトリ（プロジェクト）を作成してみましょう。GitHub Desktopでは、「Create New Repository」からリポジトリを作成できます。

実習 3-1: 新しいリポジトリを作成する

GitHub Desktopで新しいリポジトリを作成し、Git管理を始めてみましょう。

手順

  1. 1 GitHub Desktopを起動し、メニューから「File」→「New Repository」を選択（またはCtrl+N / Cmd+N）
  2. 2 「Name」に my-first-repo と入力
  3. 3 「Local Path」で保存先フォルダを選択（デスクトップやドキュメントフォルダがおすすめ）
  4. 4 「Initialize this repository with a README」にチェックを入れる
  5. 5 「Create Repository」ボタンをクリック

確認ポイント

  • GitHub Desktopの画面に「my-first-repo」が表示されている
  • 指定したフォルダに「my-first-repo」フォルダが作成されている
  • フォルダ内に「README.md」と「.git」フォルダがある

リポジトリ作成時の設定項目

項目 説明 推奨
Name リポジトリ名（フォルダ名にもなる） 英数字とハイフンを使用
Description プロジェクトの説明（省略可） 簡潔に内容を記述
Local Path 保存先フォルダのパス 管理しやすい場所を選択
Git Ignore 無視するファイルのテンプレート 開発言語に合わせて選択
License ライセンスの種類 公開時に適切なものを選択

3.4 変更を記録する（コミット）

コミット（Commit）とは、ファイルの変更を記録する操作です。ゲームのセーブポイントのように、いつでもその時点の状態に戻ることができます。

実習 3-2: ファイルを編集してコミットする

README.mdファイルを編集し、変更をコミットしてみましょう。

手順

  1. 1 GitHub Desktopで「Repository」→「Open in Explorer/Finder」を選択し、フォルダを開く
  2. 2 README.mdをテキストエディタ（VS Code等）で開く
  3. 3 以下のように編集して保存：
README.md 
# my-first-repo

はじめてのGitリポジトリです。

## 学習内容
- Git/GitHubの基本操作
- GitHub Desktopの使い方
  1. 4 GitHub Desktopに戻ると、左側の「Changes」に変更が表示される
  2. 5 「Summary」欄に READMEを更新 と入力
  3. 6 「Commit to main」ボタンをクリック

確認ポイント

  • 「Changes」タブが空になっている（変更がコミットされた）
  • 「History」タブに新しいコミットが追加されている

良いコミットメッセージの書き方

  • 何をしたかを簡潔に書く（例：「ログイン機能を追加」）
  • 現在形または命令形で書く（例：「修正」「追加」）
  • 1行目は50文字以内を目安に
  • 詳細が必要な場合はDescription欄に記載

3.5 変更履歴を確認する（History）

Gitの大きな特徴の1つは、過去のすべての変更履歴を確認できることです。「History」タブでは、いつ、誰が、何を変更したかを時系列で見ることができます。

Historyタブの見方

コミット一覧（左側）

過去のコミットが新しい順に表示されます。各コミットのメッセージ、作成者、日時が確認できます。

変更内容（右側）

選択したコミットで変更されたファイルと、その差分（追加・削除された行）が表示されます。

履歴を活用するシーン

  • 1
    バグの原因調査

    いつからバグが発生したか、どの変更が原因かを特定できます。

  • 2
    コードの意図を理解

    なぜこのように書かれているかをコミットメッセージから把握できます。

  • 3
    過去の状態に戻す

    問題が起きた場合に、動いていた時点の状態に復元できます。

3.6 過去の状態に戻す

間違えて変更してしまった場合や、以前の状態に戻したい場合、GitHub Desktopでは簡単に復元操作ができます。

復元方法の種類

操作 説明 使用場面
Discard Changes まだコミットしていない変更を破棄 編集を取り消したい時
Revert Commit 特定のコミットを打ち消す新しいコミットを作成 すでにコミットした変更を取り消したい時
Checkout（参照） 過去のコミット時点のファイル内容を確認 過去の状態を一時的に見たい時

実習 3-3: 変更を破棄してみる

まだコミットしていない変更を破棄する方法を練習しましょう。

手順

  1. 1 README.mdに適当なテキスト（「テスト」など）を追加して保存
  2. 2 GitHub Desktopの「Changes」タブで変更が表示されていることを確認
  3. 3 変更されたファイル「README.md」を右クリック
  4. 4 「Discard Changes」を選択し、確認ダイアログで「Discard Changes」をクリック

確認ポイント

  • 「Changes」タブが空になっている
  • README.mdの内容が編集前の状態に戻っている

注意

「Discard Changes」を実行すると、変更は完全に削除され元に戻せません。大切な変更を誤って破棄しないよう、操作前に確認しましょう。

3.7 GitHubへのプッシュとプル

ローカル（自分のパソコン）で作成したリポジトリをGitHubに公開することで、バックアップや他の人との共有ができます。

sequenceDiagram participant PC as 自分のパソコン
（ローカル） participant GH as GitHub
（リモート） PC->>GH: Push（プッシュ）
ローカルの変更をGitHubに送る Note over PC,GH: 自分の変更を反映 GH->>PC: Pull（プル）
GitHubの変更をローカルに取り込む Note over PC,GH: 他の人の変更を取得

用語解説

Push（プッシュ）
ローカルのコミットをリモートリポジトリ（GitHub）に送信する操作。「アップロード」のイメージです。
Pull（プル）
リモートリポジトリの変更をローカルに取り込む操作。「ダウンロード＋統合」のイメージです。
Publish（パブリッシュ）
ローカルのみのリポジトリをGitHubに初めて公開する操作。

実習 3-4: リポジトリをGitHubに公開する

作成したリポジトリをGitHubに公開（Publish）してみましょう。

手順

  1. 1 GitHub Desktopの上部にある「Publish repository」ボタンをクリック
  2. 2 「Keep this code private」のチェックを確認（プライベートにする場合はチェックを入れる）
  3. 3 「Publish Repository」ボタンをクリック
  4. 4 「Repository」→「View on GitHub」でブラウザでGitHubを開いて確認

確認ポイント

  • GitHub.comに「my-first-repo」が表示されている
  • README.mdの内容がGitHub上で確認できる
  • コミット履歴がGitHub上でも確認できる

プッシュとプルのタイミング

プッシュするタイミング

  • 作業が一段落したとき
  • バックアップを取りたいとき
  • 他の人に変更を共有したいとき

プルするタイミング

  • 作業を始める前
  • 他の人が変更を加えたとき
  • Fetchで新しい変更があると分かったとき

3.8 ブランチの基本操作

ブランチは、開発の「枝分かれ」を作る機能です。メインの開発とは別に、新機能の開発や修正作業を独立して行えます。詳しくは第7章で学びますが、ここではGitHub Desktopでの基本操作を体験してみましょう。

gitGraph commit id: "初期コミット" commit id: "機能A追加" branch feature checkout feature commit id: "新機能開発中" commit id: "新機能完成" checkout main merge feature id: "マージ" commit id: "リリース"

GitHub Desktopでのブランチ操作

操作 方法
新規ブランチ作成 「Branch」→「New Branch」または現在のブランチ名をクリック→「New Branch」
ブランチ切り替え 現在のブランチ名をクリック→切り替えたいブランチを選択
ブランチのマージ 「Branch」→「Merge into Current Branch」
ブランチの削除 「Branch」→「Delete」

ブランチを使うメリット

ブランチを使うと、メインの作業に影響を与えずに新しい機能を試すことができます。うまくいったらマージ、失敗したらブランチを削除するだけです。

理解度確認クイズ

  1. Q1. コミット（Commit）とは何ですか？

    回答を表示

    コミットとは、ファイルの変更を記録（保存）する操作です。ゲームのセーブポイントのように、その時点の状態を保存し、後からいつでも確認したり戻ったりすることができます。

  2. Q2. プッシュ（Push）とプル（Pull）の違いは何ですか？

    回答を表示

    プッシュは、ローカル（自分のパソコン）の変更をリモート（GitHub）に送信する操作です。プルは、リモートの変更をローカルに取り込む操作です。プッシュは「アップロード」、プルは「ダウンロード＋統合」のイメージです。

  3. Q3. ステージング（Staging）とは何ですか？

    回答を表示

    ステージングとは、次のコミットに含める変更を選択する作業です。買い物かごに商品を入れるイメージで、複数の変更があっても、コミットに含めたいものだけを選んで記録できます。

  4. Q4. まだコミットしていない変更を取り消したいときは、どの操作を使いますか？

    回答を表示

    「Discard Changes」（変更を破棄）を使います。変更したファイルを右クリックして選択できます。ただし、この操作は取り消せないので注意が必要です。

  • プルでGitHubの変更をローカルに取り込む
  • ブランチで独立した作業環境を作成できる
