コマンドライン入門 - ターミナルでのGit操作の基礎

GUIで学んだGitの概念を、コマンドラインで実行できるようになりましょう。ターミナルの基本操作から、主要なGitコマンドまでを丁寧に解説します。

この章で学ぶこと

  • コマンドラインとは何か、なぜ必要なのか
  • ターミナル/Git Bashの基本操作（cd, ls, pwd, mkdir）
  • git init, git status, git add, git commitの使い方
  • git log, git diffで履歴と差分を確認する方法

5.1 コマンドラインとは - なぜGUIだけでは不十分なのか

前章まではGitHub Desktopというグラフィカルなツール（GUI）でGitを操作してきました。では、なぜわざわざ黒い画面でコマンドを打つ必要があるのでしょうか？

用語解説

  • コマンドライン: 文字（コマンド）を入力してパソコンを操作する方法
  • ターミナル: コマンドを入力するためのアプリケーション（Macの場合）
  • Git Bash: Windows用のターミナル（Gitインストール時に一緒にインストールされる）
  • GUI: Graphical User Interface。マウスでボタンをクリックする操作方法
  • CUI/CLI: Character/Command Line Interface。コマンドで操作する方法

コマンドラインを学ぶ理由

観点 GUI（GitHub Desktop等） コマンドライン
学習コスト 低い（直感的） やや高い（慣れが必要）
機能の網羅性 基本機能のみ 全ての機能が使える
作業スピード クリック回数が多い 慣れると高速
自動化 難しい スクリプト化が容易
開発現場での利用 一部の企業 ほぼ全ての企業

プロからのアドバイス

開発現場では、サーバー（遠隔のコンピュータ）にはGUIがないことがほとんどです。 そのため、コマンドライン操作は必須スキルとなります。 最初は難しく感じるかもしれませんが、一度慣れると「GUIより楽」と感じるようになります。

5.2 ターミナルの起動と基本操作

ターミナル/Git Bashの起動方法

Windows

Git Bashを使用
  1. 1. スタートメニューを開く
  2. 2. 「Git Bash」と入力して検索
  3. 3. Git Bashをクリックして起動

フォルダ内で右クリック→「Git Bash Here」でも起動可能

Mac

ターミナルを使用
  1. 1. Finderを開く
  2. 2. アプリケーション→ユーティリティ
  3. 3. 「ターミナル」をダブルクリック

Spotlight（Cmd+Space）で「ターミナル」と検索しても起動可能

ターミナルの画面の見方

ターミナル画面の例 
# Macの場合
username@MacBook-Pro ~ %

# Windowsの場合（Git Bash）
username@DESKTOP-ABC123 MINGW64 ~
$

username: あなたのユーザー名

~: 現在いる場所（ホームディレクトリを意味する）

% または $: コマンド入力待ちの記号（プロンプト）

最低限覚えるべき基本コマンド

Gitを使う前に、まずはターミナルでフォルダを移動したり、中身を確認したりする基本操作を覚えましょう。

pwd - 現在地を確認

Print Working Directory の略。今自分がどのフォルダにいるかを表示します。 

# 現在地を表示
pwd

# 出力例
/Users/username/Documents

ls - ファイル一覧を表示

List の略。現在のフォルダ内にあるファイルやフォルダを表示します。 

# ファイル一覧を表示
ls

# 詳細情報も表示（-lオプション）
ls -l

# 隠しファイルも表示（-aオプション）
ls -a

# 両方のオプションを組み合わせる
ls -la

cd - フォルダを移動

Change Directory の略。指定したフォルダに移動します。 

# Documentsフォルダに移動
cd Documents

# 1つ上のフォルダに戻る
cd ..

# ホームディレクトリに戻る
cd ~

# 絶対パスで移動
cd /Users/username/Documents/projects

mkdir - フォルダを作成

Make Directory の略。新しいフォルダを作成します。 

# my-projectというフォルダを作成
mkdir my-project

# 作成したフォルダに移動
cd my-project

実習 5-1: ターミナルの基本操作を練習しよう

実際にターミナルを開いて、基本コマンドを試してみましょう。

手順

  1. 1 ターミナル（Git Bash）を起動する
  2. 2 pwd で現在地を確認
  3. 3 ls でファイル一覧を表示
  4. 4 cd Documents でDocumentsフォルダに移動
  5. 5 mkdir git-practice で練習用フォルダを作成
  6. 6 cd git-practice で作成したフォルダに移動

実行例

$ pwd
/Users/username
$ ls
Desktop  Documents  Downloads  Pictures
$ cd Documents
$ mkdir git-practice
$ cd git-practice
$ pwd
/Users/username/Documents/git-practice

5.3 git init - リポジトリを作成する

git init は、現在のフォルダをGitリポジトリとして初期化するコマンドです。 GitHub Desktopで「Create New Repository」をクリックしたのと同じ操作です。

git init の使い方 
# フォルダをGitリポジトリとして初期化
git init

# 実行結果
Initialized empty Git repository in /Users/username/Documents/git-practice/.git/

何が起きたのか？

.git という隠しフォルダが作成されました。このフォルダにGitの全ての情報（変更履歴など）が保存されます。 通常は触る必要はありませんが、これを削除するとGitリポジトリでなくなります。

.gitフォルダを確認 
# 隠しファイルも含めて一覧表示
ls -la

# 出力例
total 0
drwxr-xr-x   3 username  staff   96 Jan 15 10:00 .
drwx------+ 12 username  staff  384 Jan 15 09:55 ..
drwxr-xr-x   9 username  staff  288 Jan 15 10:00 .git  ← これが作成された

5.4 git status - 現在の状態を確認する

git status は、最も頻繁に使うコマンドです。 リポジトリの現在の状態（どのファイルが変更されたか、コミット待ちのファイルは何か等）を確認できます。

git status の使い方 
# 状態を確認
git status

# 初期状態の出力例
On branch main

No commits yet

nothing to commit (create/copy files and use "git add" to track)

まだ何もファイルがないので「nothing to commit」と表示されています。 ファイルを作成して、もう一度確認してみましょう。

ファイルを作成してステータスを確認 
# テスト用のファイルを作成（echoコマンドでファイルに書き込み）
echo "# My Project" > README.md

# ステータスを確認
git status

# 出力
On branch main

No commits yet

Untracked files:
  (use "git add ..." to include in what will be committed)
        README.md

nothing added to commit but untracked files present (use "git add" to track)

git statusの出力の読み方

  • Untracked files Gitがまだ追跡していない新しいファイル
  • Changes not staged 変更されたが、まだステージングされていないファイル
  • Changes to be committed ステージング済みで、次のコミットに含まれるファイル

5.5 git add - 変更をステージングする

git add は、変更をステージングエリアに追加するコマンドです。 ステージングとは「次のコミットに含める変更を選択する」ことです。

ステージングの流れ

flowchart LR A[作業ディレクトリ
Working Directory] -->|git add| B[ステージングエリア
Staging Area] B -->|git commit| C[リポジトリ
Repository] style A fill:#fee2e2,stroke:#ef4444 style B fill:#fef3c7,stroke:#f59e0b style C fill:#d1fae5,stroke:#10b981
git add の使い方 
# 特定のファイルをステージング
git add README.md

# 複数のファイルをステージング
git add file1.txt file2.txt

# 全ての変更をステージング（よく使う）
git add .

# ステータスを確認
git status
ステージング後のステータス 
# git add . 実行後
git status

# 出力（緑色で表示される）
On branch main

No commits yet

Changes to be committed:
  (use "git rm --cached ..." to unstage)
        new file:   README.md

なぜステージングが必要なのか？

複数のファイルを変更した時、「関連する変更だけ」をまとめてコミットできます。 例えば、「ログイン機能の追加」と「バグ修正」を別々のコミットにすることで、履歴が分かりやすくなります。

5.6 git commit - 変更を記録する

git commit は、ステージングした変更を履歴として記録するコマンドです。 コミットには必ずメッセージを付けます。

git commit の使い方 
# メッセージを付けてコミット（-mオプション）
git commit -m "最初のコミット: READMEを追加"

# 出力例
[main (root-commit) abc1234] 最初のコミット: READMEを追加
 1 file changed, 1 insertion(+)
 create mode 100644 README.md

良いコミットメッセージを書くコツ

  • 「何をしたか」を簡潔に書く（例: 「ログイン機能を追加」）
  • 1行目は50文字以内を目安にする
  • 動詞から始める（「追加」「修正」「削除」「更新」など）

良い例: 「ユーザー認証機能を追加」「トップページのレイアウトを修正」
悪い例: 「更新」「修正」「aaa」（何をしたか分からない）

コミット後のステータス 
git status

# 出力
On branch main
nothing to commit, working tree clean

5.7 git log - 履歴を確認する

git log は、コミット履歴を表示するコマンドです。 いつ、誰が、何を変更したかを確認できます。

git log の使い方 
# 履歴を表示
git log

# 出力例
commit abc1234567890abcdef1234567890abcdef12345 (HEAD -> main)
Author: Your Name 
Date:   Mon Jan 15 10:30:00 2024 +0900

    最初のコミット: READMEを追加
よく使うオプション 
# 1行で簡潔に表示
git log --oneline

# 出力例
abc1234 最初のコミット: READMEを追加

# 最新3件だけ表示
git log -3

# グラフ表示（ブランチの分岐が視覚的に分かる）
git log --oneline --graph

ヒント: ログ表示の終了方法

git log の表示が長い場合、q キーを押すと終了できます。 矢印キーやスペースキーでスクロールできます。

5.8 git diff - 差分を確認する

git diff は、変更内容の差分を表示するコマンドです。 何が変わったかを確認してからコミットする習慣をつけましょう。

ファイルを編集して差分を確認 
# README.mdに1行追加
echo "This is my first Git project." >> README.md

# 差分を確認
git diff

# 出力例
diff --git a/README.md b/README.md
index 7e59600..a097810 100644
--- a/README.md
+++ b/README.md
@@ -1 +1,2 @@
 # My Project
+This is my first Git project.

git diffの出力の読み方

  • + 緑色の行 追加された行
  • - 赤色の行 削除された行
  • 白色の行 変更がない行（前後の文脈として表示）
様々なdiffの使い方 
# 作業ディレクトリの変更を表示（ステージング前）
git diff

# ステージング済みの変更を表示
git diff --staged

# 特定のファイルの差分だけ表示
git diff README.md

実習 5-2: 一連のGit操作を練習しよう

学んだコマンドを使って、ファイルの作成からコミットまでの流れを実践しましょう。

手順

  1. 1 先ほど作成したgit-practiceフォルダで作業
  2. 2 echo "Hello Git" > hello.txt でファイルを作成
  3. 3 git status で状態を確認
  4. 4 git add hello.txt でステージング
  5. 5 git status で状態を再確認（緑色になる）
  6. 6 git commit -m "hello.txtを追加" でコミット
  7. 7 git log --oneline で履歴を確認

期待される結果

$ git log --oneline
def5678 hello.txtを追加
abc1234 最初のコミット: READMEを追加

理解度確認クイズ

  1. Q1. 現在いるフォルダを確認するコマンドは何ですか？

    回答を表示

    pwd（Print Working Directory）です。 現在自分がどのディレクトリ（フォルダ）にいるかを表示します。

  2. Q2. git add と git commit の違いは何ですか？

    回答を表示

    git addは変更を「ステージングエリア」に追加するコマンドで、次のコミットに含める変更を選択します。
    git commitはステージングエリアの変更を「履歴として記録」するコマンドです。
    つまり、add は「選ぶ」、commit は「記録する」という役割の違いがあります。

  3. Q3. コミット履歴を1行ずつ簡潔に表示するコマンドは何ですか？

    回答を表示

    git log --oneline です。 各コミットを1行で表示するため、履歴の全体像を把握しやすくなります。
