第8章

マージとコンフリクト - 変更の統合

ブランチを統合（マージ）する方法と、コンフリクト（競合）が発生した場合の解決方法を学びます。チーム開発で避けて通れないスキルをマスターしましょう。

この章で学ぶこと

  • マージとは何か、なぜ必要なのか
  • Fast-forwardマージと3-wayマージの違い
  • コンフリクト（競合）の発生原因と確認方法
  • コンフリクトの解決手順（手動での編集）
  • GitHub Desktopでのマージとコンフリクト解決

8.1 マージとは - ブランチを一つにまとめる

前章でブランチの作成と切り替えを学びました。開発が完了したブランチは、最終的にメインのブランチ（main）に取り込む必要があります。この「ブランチを統合する」操作をマージ（Merge）と呼びます。

マージの基本イメージ

gitGraph commit id: "A" commit id: "B" branch feature checkout feature commit id: "C" commit id: "D" checkout main merge feature id: "E" tag: "マージ" commit id: "F"

上の図では、featureブランチで開発した変更（C, D）をmainブランチに統合しています。マージすることで、featureブランチの変更がmainブランチに反映されます。

用語解説: マージの方向

マージには「どこに」「何を」統合するかという方向があります。

  • マージ先（ベースブランチ）: 変更を受け入れるブランチ（例: main）
  • マージ元（トピックブランチ）: 統合したい変更があるブランチ（例: feature）

マージの基本コマンド

コマンドラインでマージを行う場合、まずマージ先のブランチに移動してから、マージコマンドを実行します。

Terminal 
# 1. マージ先のブランチ（main）に移動
git checkout main

# 2. featureブランチをmainにマージ
git merge feature

# マージが成功すると以下のようなメッセージが表示される
# Updating abc1234..def5678
# Fast-forward
#  index.html | 10 ++++++++++
#  1 file changed, 10 insertions(+)

8.2 マージの種類 - Fast-forwardと3-wayマージ

Gitのマージには主に2つの方式があります。状況に応じてGitが自動的に選択しますが、それぞれの違いを理解しておくと、履歴の見方が変わります。

Fast-forward（ファストフォワード）マージ

ブランチを「早送り」するだけ

マージ先のブランチがマージ元のブランチから分岐後に変更されていない場合に発生します。単純にポインタを移動するだけなので、新しいコミットは作成されません。

Fast-forwardマージの流れ

gitGraph commit id: "A" commit id: "B" branch feature checkout feature commit id: "C" commit id: "D" checkout main merge feature

特徴: 履歴が一直線になる。シンプルで分かりやすいが、ブランチがあったことが分からなくなる。

3-way（スリーウェイ）マージ

マージコミットを作成

マージ先のブランチも分岐後に変更されている場合に発生します。両方の変更を統合した新しい「マージコミット」が作成されます。

3-wayマージの流れ

gitGraph commit id: "A" commit id: "B" branch feature checkout feature commit id: "C" checkout main commit id: "D" checkout feature commit id: "E" checkout main merge feature id: "M" tag: "マージコミット"

特徴: マージコミットが履歴に残る。いつどのブランチをマージしたかが明確。

ヒント: 常にマージコミットを作成したい場合

--no-ffオプションを使うと、Fast-forward可能な場合でも強制的にマージコミットを作成できます。 

# 常にマージコミットを作成
git merge --no-ff feature

8.3 コンフリクト（競合）とは - 同じ場所を別々に編集した場合

マージは通常Gitが自動的に行いますが、同じファイルの同じ行を異なる内容で編集していた場合、Gitはどちらの変更を採用すべきか判断できません。この状態をコンフリクト（Conflict）または競合と呼びます。

コンフリクトが発生する条件

  • 同じファイルの同じ行を、両方のブランチで異なる内容に変更
  • 一方のブランチでファイルを編集し、他方で削除
  • 両方のブランチで同名のファイルを新規作成（内容が異なる）

コンフリクト発生のシナリオ

flowchart LR subgraph main[mainブランチ] M1[index.html
3行目: Hello] end subgraph feature[featureブランチ] F1[index.html
3行目: Welcome] end M1 -->|マージ| C{コンフリクト!} F1 -->|マージ| C C -->|どちらを採用？| R[手動で解決が必要] style C fill:#fee2e2,stroke:#ef4444,color:#7f1d1d style R fill:#fef3c7,stroke:#f59e0b,color:#78350f

コンフリクトが発生すると、Gitはマージを途中で止め、「どちらの変更を採用するか」を人間に判断してもらいます。コンフリクトは怖いものではありません。落ち着いて対処すれば必ず解決できます。

8.4 コンフリクトの確認方法

マージ時にコンフリクトが発生すると、Gitは明確なメッセージを表示します。また、git statusコマンドで詳細を確認できます。

Terminal - コンフリクト発生時 
$ git merge feature
Auto-merging index.html
CONFLICT (content): Merge conflict in index.html
Automatic merge failed; fix conflicts and then commit the result.

# git status で状態を確認
$ git status
On branch main
You have unmerged paths.
  (fix conflicts and run "git commit")
  (use "git merge --abort" to abort the merge)

Unmerged paths:
  (use "git add ..." to mark resolution)
        both modified:   index.html

no changes added to commit (use "git add" and/or "git commit -a")

コンフリクトが発生したファイルを開くと、Gitが特殊なコンフリクトマーカーを挿入しています。これがコンフリクト箇所を示しています。

index.html（コンフリクト状態） 
<html>
<body>
<<<<<<< HEAD
    <h1>Hello World</h1>
=======
    <h1>Welcome to my site</h1>
>>>>>>> feature
</body>
</html>

コンフリクトマーカーの読み方

<<<<<<< HEAD 現在のブランチ（マージ先）の変更の開始
======= 2つの変更の区切り
>>>>>>> feature マージしようとしているブランチの変更の終了

8.5 コンフリクトの解決手順（手動での編集）

コンフリクトの解決は、以下の3ステップで行います。

1

コンフリクト箇所を確認する

git statusでコンフリクトが発生したファイルを確認し、エディタで開きます。

2

マーカーを削除し、正しい内容に編集する

コンフリクトマーカー（<<<<<<<、=======、>>>>>>>）を削除し、最終的に残したい内容だけにします。

解決例: 両方の良い部分を組み合わせる場合

<html>
<body>
    <h1>Welcome to Hello World</h1>
</body>
</html>
3

解決を完了してコミットする

編集が終わったら、git addでステージングし、git commitでマージを完了します。 

# 解決したファイルをステージング
git add index.html

# マージコミットを作成（メッセージは自動で入る）
git commit

# または明示的にメッセージを指定
git commit -m "Merge feature branch, resolve conflict in index.html"

コンフリクト解決の3パターン

自分の変更を採用

HEAD側（現在のブランチ）の変更だけを残す

相手の変更を採用

マージ元ブランチの変更だけを残す

両方を組み合わせ

両方の変更を活かした新しい内容にする

8.6 GitHub Desktopでのマージとコンフリクト解決

GitHub Desktopでは、GUIを使って視覚的にマージとコンフリクト解決ができます。コマンドラインに慣れていない方は、まずこちらで練習すると良いでしょう。

GitHub Desktopでマージする手順

  1. 1 マージ先のブランチに切り替え: 画面上部の「Current Branch」からmainを選択
  2. 2 マージを実行: メニュー「Branch」→「Merge into current branch...」を選択
  3. 3 ブランチを選択: マージしたいブランチ（例: feature）を選んで「Create a merge commit」をクリック

GitHub Desktopでコンフリクトが発生した場合

  1. 1 コンフリクトが発生すると、「1 conflicted file」などと表示されます
  2. 2 「Open in Visual Studio Code」または「Open in External Editor」をクリックしてエディタで開く
  3. 3 VS Codeではコンフリクト箇所がハイライト表示され、「Accept Current Change」「Accept Incoming Change」「Accept Both Changes」ボタンで解決できます
  4. 4 ファイルを保存後、GitHub Desktopに戻り「Continue Merge」をクリック

8.7 マージを取り消す方法

「やっぱりマージをやめたい」という場面もあります。状況に応じた取り消し方法を覚えておきましょう。

マージ途中で中止する（コンフリクト解決前）

コンフリクトが発生して、まだ解決していない状態でマージ自体を取りやめたい場合。 

# マージを中止してマージ前の状態に戻す
git merge --abort

マージ完了後に取り消す

マージコミットを作成した後で取り消したい場合。まだpushしていなければ、resetで戻せます。 

# マージコミット直前の状態に戻す
# （まだpushしていない場合のみ！）
git reset --hard HEAD~1

# 注意: pushした後は git revert を使う（第11章で学習）

注意: git reset --hard は要注意

--hardオプションは作業中の変更も全て消えます。また、pushした後のコミットをresetすると履歴の不整合が起きるため、使わないでください。

実習 8-1: コンフリクトを発生させて解決する

意図的にコンフリクトを発生させ、解決する練習をしましょう。

手順

  1. 1 練習用リポジトリを作成し、greeting.txtを作成して「Hello」と書いてコミット
  2. 2 featureブランチを作成し、greeting.txtを「Welcome」に変更してコミット
  3. 3 mainブランチに戻り、greeting.txtを「Hi」に変更してコミット
  4. 4 mainブランチでgit merge featureを実行（コンフリクト発生！）
  5. 5 ファイルを開いてコンフリクトマーカーを確認し、「Hello, Welcome!」に編集して解決
  6. 6 git addとgit commitでマージを完了

実行例

# 1. 準備
mkdir merge-practice && cd merge-practice
git init
echo "Hello" > greeting.txt
git add greeting.txt
git commit -m "Initial commit"

# 2. featureブランチで編集
git checkout -b feature
echo "Welcome" > greeting.txt
git commit -am "Change to Welcome"

# 3. mainで別の編集
git checkout main
echo "Hi" > greeting.txt
git commit -am "Change to Hi"

# 4. マージ（コンフリクト発生）
git merge feature
# CONFLICT と表示される

# 5. コンフリクトを解決
echo "Hello, Welcome!" > greeting.txt
git add greeting.txt
git commit -m "Resolve conflict"

# 6. 履歴を確認
git log --oneline --graph

第8章のまとめ

  • マージは、分岐したブランチを統合する操作
  • Fast-forwardマージ: ブランチを早送りするだけ（新しいコミットなし）
  • 3-wayマージ: マージコミットを作成して統合
  • コンフリクトは同じ箇所を別々に編集した時に発生
  • コンフリクト解決: マーカーを削除 → 正しい内容に編集 → add → commit
  • git merge --abortでマージを中止できる

理解度確認クイズ

  1. Q1. Fast-forwardマージと3-wayマージの違いを説明してください。

    回答を表示

    Fast-forwardマージは、マージ先のブランチが分岐後に変更されていない場合に発生します。単純にブランチのポインタを移動するだけで、新しいコミットは作成されません。

    3-wayマージは、マージ先のブランチも分岐後に変更されている場合に発生します。両方の変更を統合した新しい「マージコミット」が作成されます。

  2. Q2. コンフリクトが発生するのはどのような場合ですか？

    回答を表示

    コンフリクトは、同じファイルの同じ行を異なるブランチで別々に編集した場合に発生します。Gitはどちらの変更を採用すべきか判断できないため、人間による手動での解決が必要になります。

  3. Q3. コンフリクトを解決する基本的な手順を3ステップで説明してください。

    回答を表示

    1. コンフリクト箇所を確認する（git statusでファイルを特定し、エディタで開く）

    2. コンフリクトマーカーを削除し、正しい内容に編集する（<<<<<<<、=======、>>>>>>>を削除）

    3. 解決を完了してコミットする（git add → git commit）

  4. Q4. マージを途中で中止するコマンドは何ですか？

    回答を表示

    git merge --abortです。コンフリクトが発生して、まだ解決していない状態でマージ自体を取りやめたい場合に使用します。このコマンドを実行すると、マージ前の状態に戻ります。
