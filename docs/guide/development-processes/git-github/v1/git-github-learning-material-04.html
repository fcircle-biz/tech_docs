GitHubの基本 - リモートリポジトリと公開・共有

GitHubのWebサイトの使い方を学び、リモートリポジトリの概念を理解します。コードの公開や共有の方法をマスターしましょう。

この章で学ぶこと

  • GitHubの画面構成とダッシュボードの見方
  • Webブラウザからリポジトリを作成・管理する方法
  • ローカルとリモートの関係（push/pull/fetch）を理解する
  • 他の人のプロジェクトを見る・参加する方法（Fork、Star、Watch）

4.1 GitHubのダッシュボードと画面構成

GitHubにログインすると表示されるダッシュボードは、あなたの活動の中心となる画面です。ここから様々な操作を行うことができます。

主要な画面要素

  • 検索バー: リポジトリ、ユーザー、コードを検索
  • Pull requests: 自分に関係するプルリクエスト一覧
  • Issues: 自分に割り当てられた課題一覧
  • 通知: メンションや更新のお知らせ

プロフィールメニュー

右上のアイコンをクリックすると、以下の項目にアクセスできます：

メニュー項目 説明
Your profile 自分のプロフィールページを表示
Your repositories 自分が作成・参加しているリポジトリ一覧
Your stars スターを付けたリポジトリ一覧
Settings アカウント設定（プロフィール、セキュリティなど）

4.2 新しいリポジトリの作成（Web版）

GitHub Desktopからだけでなく、GitHubのWebサイトから直接リポジトリを作成することもできます。この方法は、ブラウザさえあればどこからでもリポジトリを作れるため便利です。

実習 4-1: Webからリポジトリを作成する

GitHubのWebサイトから新しいリポジトリを作成してみましょう。

手順

  1. 1 GitHubにログインし、右上の「+」ボタンをクリック
  2. 2 「New repository」を選択
  3. 3 リポジトリ名を入力（例: my-first-web-repo）
  4. 4 「Add a README file」にチェックを入れる
  5. 5 「Create repository」ボタンをクリック

リポジトリ作成時の設定項目

Repository name（リポジトリ名）

URLの一部になるため、英数字とハイフンで命名するのがおすすめです。

Description（説明）

任意ですが、何のプロジェクトか分かるように書くと便利です。

Public / Private

公開（Public）か非公開（Private）かを選択します。後から変更も可能です。

Add a README file

README.mdファイルを自動生成します。チェックを入れるのがおすすめです。

4.3 README.mdファイルの作成と編集

README.mdは、リポジトリの「説明書」のようなファイルです。GitHubでは、このファイルがあるとリポジトリのトップページに自動的に表示されます。

README.mdの「.md」って何？

「.md」はMarkdown（マークダウン）形式のファイルを表します。 Markdownは、簡単な記号で見出しや箇条書きが書ける軽量な文書記法です。 GitHubでは自動的にきれいに整形して表示してくれます。

Markdownの基本記法

README.md 
# プロジェクト名（見出し1）

## 概要（見出し2）
このプロジェクトは○○するためのツールです。

## 機能
- 機能1（箇条書き）
- 機能2
- 機能3

## インストール方法
1. リポジトリをクローン（番号付きリスト）
2. 依存関係をインストール
3. 起動

## 使い方
`コマンド` はこのように書きます（インラインコード）

```python
# コードブロックの例
print("Hello, World!")
```

## ライセンス
MIT License

実習 4-2: README.mdを編集する

先ほど作成したリポジトリのREADME.mdを編集してみましょう。

手順

  1. 1 リポジトリページでREADME.mdをクリック
  2. 2 右上の鉛筆アイコン（Edit）をクリック
  3. 3 内容を編集（上のMarkdown例を参考に）
  4. 4 「Preview」タブで表示を確認
  5. 5 「Commit changes」ボタンで保存

4.4 ローカルとリモートの関係 - push/pull/fetch

Gitでは、ローカルリポジトリ（自分のPC）とリモートリポジトリ（GitHub）の2つの場所でファイルを管理します。この2つを同期させる操作が非常に重要です。

ローカルとリモートの同期

flowchart LR subgraph Local[自分のパソコン] LR[(ローカル
リポジトリ)] end subgraph Remote[GitHub クラウド] RR[(リモート
リポジトリ)] end LR -->|push
アップロード| RR RR -->|pull
ダウンロード+統合| LR RR -->|fetch
ダウンロードのみ| LR

Push（プッシュ）

ローカルの変更をリモートにアップロードする操作。 自分の作業を他の人と共有するときに使います。

「自分のPC → GitHub」への一方通行

Pull（プル）

リモートの変更をローカルにダウンロードして統合する操作。 他の人の変更を取り込むときに使います。

「GitHub → 自分のPC」+ 自動でマージ

Fetch（フェッチ）

リモートの変更を確認だけする操作。 ダウンロードはするが、ローカルの作業には影響しません。

「確認してから取り込みたい」ときに便利

よくある混乱ポイント

「プッシュできない！」というエラーの多くは、リモートに新しい変更があるのに、ローカルで取り込んでいないことが原因です。

解決策: プッシュの前に、まずプルしてリモートの変更を取り込みましょう。

4.5 パブリックとプライベートリポジトリ

GitHubのリポジトリには、Public（公開）とPrivate（非公開）の2種類があります。用途に応じて使い分けましょう。

Public（パブリック）

世界中に公開
  • 誰でも閲覧・クローン可能
  • オープンソースプロジェクト向き
  • ポートフォリオとして公開できる
  • 無料で無制限に作成可能

向いている用途: 個人プロジェクト、学習用、OSS活動

Private（プライベート）

招待された人のみ
  • 招待されたユーザーのみアクセス可能
  • 企業の内部プロジェクト向き
  • 機密情報を含むコードに適切
  • 無料プランでも作成可能（共同作業者数に制限あり）

向いている用途: 仕事のコード、個人的なメモ、未完成のプロジェクト

ヒント

公開/非公開の設定はリポジトリ作成後でも変更できます。 Settings → Danger Zone → Change repository visibility で変更可能です。

4.6 他の人のリポジトリを見る・参加する

GitHubでは、他の人が公開しているリポジトリを見たり、参加したりできます。以下の3つの機能を使いこなしましょう。

Fork（フォーク）

他の人のリポジトリを自分のアカウントにコピーする機能。 コピーしたリポジトリは自由に編集でき、元のプロジェクトに貢献することもできます。

オープンソースへの貢献は、まずForkから始めます

Star（スター）

リポジトリに「いいね！」を付ける機能。ブックマークとして使えます。 スター数は、そのプロジェクトの人気度を示す指標にもなります。

後で見返すときに便利 作者への応援にもなる

Watch（ウォッチ）

リポジトリの更新を通知で受け取る機能。 新しいIssueやプルリクエスト、リリースなどの情報を受け取れます。

「All Activity」で全ての更新、「Releases only」でリリースのみなど選択可能

4.7 GitHubプロフィールのカスタマイズ

GitHubプロフィールは、あなたの「技術名刺」のような存在です。転職活動やポートフォリオとしても活用できるため、しっかり設定しておきましょう。

実習 4-3: プロフィールを設定する

自分のGitHubプロフィールを充実させてみましょう。

手順

  1. 1 右上のアイコン → 「Your profile」をクリック
  2. 2 「Edit profile」をクリック
  3. 3 名前、自己紹介、所在地などを入力
  4. 4 プロフィール画像（アバター）を設定

上級者向け: プロフィールREADME

自分のユーザー名と同じ名前のリポジトリを作成し、README.mdを追加すると、 プロフィールページに自動的に表示されます。

例: ユーザー名が「tanaka」なら、「tanaka/tanaka」というリポジトリを作成します。 これにより、スキルバッジや活動グラフなど、個性的なプロフィールを作れます。

理解度確認クイズ

  1. Q1. push と pull の違いは何ですか？

    回答を表示

    pushは、ローカルリポジトリの変更をリモートリポジトリ（GitHub）にアップロードする操作です。
    pullは、リモートリポジトリの変更をローカルにダウンロードして、自動的に統合（マージ）する操作です。
    つまり、pushは「自分 → GitHub」、pullは「GitHub → 自分」の方向です。

  2. Q2. Public と Private リポジトリの違いを説明してください。

    回答を表示

    Publicは誰でも閲覧・クローンできる公開リポジトリです。オープンソースプロジェクトやポートフォリオに向いています。
    Privateは招待された人だけがアクセスできる非公開リポジトリです。企業の内部コードや機密情報を含むプロジェクトに適しています。

  3. Q3. Fork と Clone の違いは何ですか？

    回答を表示

    Forkは、他の人のリポジトリを自分のGitHubアカウントにコピーする操作です。GitHub上でコピーが作られます。
    Cloneは、リポジトリを自分のローカルPC（パソコン）にコピーする操作です。
    オープンソースへの貢献では、まずForkして自分のアカウントにコピーし、それをCloneしてローカルで編集するという流れになります。
