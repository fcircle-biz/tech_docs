ブランチ入門 - 並行作業の基本

ブランチの概念と基本的な使い方を学びます。複数の作業を安全に並行して進めるための重要な機能です。

この章で学ぶこと

  • ブランチとは何か、なぜ必要なのかを理解する
  • mainブランチの役割と重要性
  • コマンドラインでのブランチ操作（作成・切り替え・削除）
  • GitHub Desktopでのブランチ操作
  • ブランチ運用のベストプラクティス

7.1 ブランチとは - 木の枝のイメージで理解する

ブランチ（Branch）とは、英語で「枝」を意味します。Gitにおけるブランチは、開発の流れを「枝分かれ」させる機能です。これにより、メインの作業に影響を与えることなく、新しい機能の開発や実験的な変更を行うことができます。

ブランチのイメージ

gitGraph commit id: "初期コミット" commit id: "機能A追加" branch feature-login checkout feature-login commit id: "ログイン画面作成" commit id: "認証機能追加" checkout main commit id: "バグ修正" checkout feature-login commit id: "ログイン完成" checkout main merge feature-login id: "マージ" commit id: "リリース"

身近な例えで理解する

ブランチは「下書きノート」のようなものです。

  • main（本番ノート）: 完成した内容だけを書く清書用ノート
  • ブランチ（下書きノート）: 試行錯誤しながら書く練習用ノート
  • マージ: 下書きが完成したら、本番ノートに書き写す

なぜブランチが必要なのか

ブランチがない場合、以下のような問題が発生します。

ブランチなしの問題点

  • 開発中の不完全なコードが本番環境に影響する
  • 複数人が同時に作業すると、お互いの変更が競合する
  • 実験的な変更を試しにくい（失敗したら全体に影響）
  • 緊急のバグ修正と新機能開発を同時に進められない

ブランチを使うメリット

  • 安全な開発: メインのコードに影響を与えずに作業できる
  • 並行作業: 複数の機能を同時に開発できる
  • 実験的な変更: 気軽に試して、ダメなら捨てられる
  • コードレビュー: 変更をレビューしてからマージできる

7.2 mainブランチとは - 最も重要なブランチ

main（またはmaster）ブランチは、リポジトリを作成した時に自動的に作られるデフォルトのブランチです。これは「本番環境のコード」「リリース可能な状態のコード」を保持する最も重要なブランチです。

mainとmasterの違い

以前はデフォルトブランチの名前は「master」でしたが、2020年以降、GitHubでは「main」がデフォルトになりました。どちらも同じ役割を持ちますが、新しいリポジトリでは「main」が使われます。

mainブランチの基本ルール

  • 1 常に動作する状態を保つ: mainブランチのコードは、いつでも本番環境にデプロイできる状態にする
  • 2 直接コミットしない: 新機能や修正は別ブランチで作業し、完成したらマージする
  • 3 レビューを経てからマージ: チーム開発では、プルリクエストでレビューを受けてからマージする

開発の流れ

flowchart LR A[mainブランチ] -->|ブランチ作成| B[featureブランチ] B -->|開発作業| C[コミット] C -->|テスト| D[動作確認] D -->|完成| E[プルリクエスト] E -->|レビュー| F[承認] F -->|マージ| A

7.3 ブランチ操作（コマンドライン）

コマンドラインでブランチを操作する方法を学びましょう。基本的なコマンドを覚えれば、効率的にブランチを管理できます。

ブランチの一覧を確認する

まず、現在のブランチ一覧を確認してみましょう。

ターミナル 
# ローカルブランチの一覧を表示
git branch

# 出力例（*は現在いるブランチ）
# * main
#   feature-login
#   fix-bug

# リモートブランチも含めて表示
git branch -a

新しいブランチを作成する

新しい機能を開発する時は、まずブランチを作成します。

ターミナル 
# 方法1: ブランチを作成（作成のみ、移動はしない）
git branch feature-login

# 方法2: ブランチを作成して、すぐに移動（おすすめ）
git checkout -b feature-login

# 方法3: 新しいコマンド（Git 2.23以降）
git switch -c feature-login

コマンドの覚え方

git checkout -bの「-b」は「branch」の略です。「checkout（移動）しながら、branch（ブランチ）を作る」と覚えましょう。

ブランチを切り替える

既存のブランチに移動する場合は、以下のコマンドを使います。

ターミナル 
# 方法1: checkoutコマンド（従来の方法）
git checkout main
git checkout feature-login

# 方法2: switchコマンド（Git 2.23以降、推奨）
git switch main
git switch feature-login

# 現在のブランチを確認
git branch
# * feature-login  ← 現在のブランチに*がつく
#   main

checkout vs switch

コマンド 特徴 推奨度
git checkout 従来のコマンド。多機能だが分かりにくい まだ広く使われている
git switch ブランチ切り替え専用。シンプルで分かりやすい 推奨

ブランチを削除する

マージが完了して不要になったブランチは削除しましょう。

ターミナル 
# ブランチを削除（マージ済みのブランチのみ）
git branch -d feature-login

# 強制削除（マージしていなくても削除）
git branch -D feature-login

# 注意: 現在いるブランチは削除できません
# 削除する前に、mainブランチなどに移動してください
git switch main
git branch -d feature-login

注意

-D（大文字）は強制削除です。マージしていない変更がある場合でも削除されてしまうため、慎重に使用してください。

7.4 GitHub Desktopでのブランチ操作

GitHub Desktopを使えば、コマンドを覚えなくてもGUI（マウス操作）でブランチを操作できます。

ブランチの作成

  1. 1 画面上部の「Current Branch」をクリック
  2. 2 「New Branch」ボタンをクリック
  3. 3 ブランチ名を入力（例: feature-login）
  4. 4 「Create Branch」をクリック

ブランチの切り替え

  1. 1 画面上部の「Current Branch」をクリック
  2. 2 ブランチ一覧から、移動したいブランチをクリック

現在のブランチにはチェックマークが表示されます。

ブランチの削除

  1. 1 削除したいブランチ以外に切り替える（mainなど）
  2. 2 メニューから「Branch」→「Delete...」を選択
  3. 3 確認ダイアログで「Delete」をクリック
操作 コマンドライン GitHub Desktop
ブランチ一覧 git branch Current Branch クリック
ブランチ作成 git checkout -b 名前 New Branch ボタン
ブランチ切り替え git switch 名前 ブランチ名をクリック
ブランチ削除 git branch -d 名前 Branch → Delete...

7.5 ブランチ命名規則 - わかりやすい名前の付け方

ブランチ名は、その作業内容が一目でわかるように付けることが重要です。チーム開発では、命名規則を決めておくと管理がしやすくなります。

feature/

新機能の開発

feature/user-login feature/shopping-cart

fix/

バグ修正

fix/login-error fix/typo-header

hotfix/

緊急のバグ修正

hotfix/security-patch hotfix/db-connection

docs/

ドキュメントの更新

docs/api-reference docs/readme-update

命名のベストプラクティス

  • 小文字を使う: 大文字と小文字の混在を避ける
  • ハイフンで区切る: スペースの代わりにハイフン（-）を使用
  • 簡潔に: 長すぎず、内容が伝わる名前にする
  • Issue番号を含める: feature/123-user-login のように番号を付けると追跡しやすい

避けるべき命名

test - 何のテストか不明

fix - 何の修正か不明

work - 作業内容が不明

tanaka-branch - 人名だけでは内容が不明

実習 7-1: ブランチを作成して作業する

実際にブランチを作成し、そこで作業してみましょう。

手順

  1. 1 練習用リポジトリに移動（または新規作成）
  2. 2 新しいブランチを作成して移動
  3. 3 ファイルを編集してコミット
  4. 4 mainブランチに戻って、変更が反映されていないことを確認

実行例

# 1. リポジトリに移動
cd ~/my-git-practice

# 2. 新しいブランチを作成して移動
git checkout -b feature/add-greeting

# 3. ファイルを編集
echo "こんにちは！" >> greeting.txt

# 4. 変更をコミット
git add greeting.txt
git commit -m "挨拶メッセージを追加"

# 5. mainブランチに戻る
git switch main

# 6. greeting.txtを確認（変更が反映されていない）
cat greeting.txt

確認ポイント

  • feature/add-greetingブランチでの変更が、mainブランチには反映されていない
  • git branchで両方のブランチが表示される

第7章のまとめ

  • ブランチは、開発の流れを「枝分かれ」させる機能
  • mainブランチは、本番環境のコードを保持する最重要ブランチ
  • git checkout -bまたはgit switch -cでブランチを作成・移動
  • GitHub Desktopでも同じ操作がGUIで可能
  • ブランチ名はfeature/、fix/などの接頭辞を付けると分かりやすい

理解度確認クイズ

  1. Q1. ブランチを使う主なメリットを2つ挙げてください。

    回答を表示

    1. 安全な開発: メインのコードに影響を与えずに新機能の開発や実験ができる
    2. 並行作業: 複数の機能を同時に開発できる
    その他: 実験的な変更が気軽にできる、コードレビューしてからマージできる、など

  2. Q2. 新しいブランチを作成して、そのブランチに移動するコマンドは何ですか？

    回答を表示

    git checkout -b ブランチ名
    または
    git switch -c ブランチ名（Git 2.23以降）

  3. Q3. mainブランチの役割と、守るべきルールを説明してください。

    回答を表示

    役割: 本番環境のコード、リリース可能な状態のコードを保持する最重要ブランチ

    守るべきルール:
    1. 常に動作する状態を保つ
    2. 直接コミットせず、別ブランチで作業してからマージする
    3. チーム開発では、レビューを経てからマージする

  4. Q4. 「feature/user-login」というブランチ名が良い理由を説明してください。

    回答を表示

    1. feature/接頭辞により、新機能の開発であることが明確
    2. user-loginにより、ユーザーログイン機能の開発だと具体的にわかる
    3. 小文字とハイフンを使用し、命名規則に従っている
    4. 簡潔で、ブランチの目的が一目でわかる
