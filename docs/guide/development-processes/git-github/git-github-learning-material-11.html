第11章

トラブルシューティング - よくある問題と解決方法

Git/GitHubでよく発生する問題と、その解決方法を学びます。エラーが起きても慌てずに対処できるようになりましょう。

この章で学ぶこと

  • 「変更が消えた」と思った時の対処法（git reflog）
  • 間違えてコミットした時の修正方法（git commit --amend）
  • コミットの取り消し方法（git reset、git revert）
  • よくあるエラーメッセージの意味と対処法

はじめに

Gitを使っていると、さまざまなトラブルに遭遇します。「変更が消えた！」「コミットを間違えた！」と焦ることもあるでしょう。 しかし、Gitは変更履歴を保持するツールです。ほとんどの場合、データは完全には失われていません。 この章では、よくあるトラブルとその解決方法を学び、落ち着いて対処できるようになりましょう。

11.1 「変更が消えた！」と思った時 - git reflog

ブランチを削除した、resetを実行した、rebaseで失敗した...そんな時、「消えた」と思ったコミットを復活させる最後の砦がgit reflogです。

用語解説: reflog とは？

reflog（Reference Log）は、HEADやブランチの参照がいつ、どこを指していたかの履歴を記録したものです。

  • git log: コミットの履歴（枝分かれの歴史）
  • git reflog: 操作の履歴（あなたが何をしたか）

reflogの使い方

reflogを確認 
# reflogの履歴を表示
git reflog

# 出力例:
# a1b2c3d (HEAD -> main) HEAD@{0}: commit: 新機能追加
# e4f5g6h HEAD@{1}: reset: moving to HEAD~1
# i7j8k9l HEAD@{2}: commit: 削除したいコミット
# m0n1o2p HEAD@{3}: commit: 正常なコミット

この例では、HEAD@{2}の「削除したいコミット」がresetで消されたことがわかります。 しかし、そのコミット自体はi7j8k9lというハッシュでまだ存在しています。

消えたコミットを復活させる

コミットの復活 
# 方法1: 特定のコミットに戻る
git reset --hard i7j8k9l

# 方法2: reflogの参照を使って戻る
git reset --hard HEAD@{2}

# 方法3: 新しいブランチとして復活させる（安全）
git branch recovered-branch i7j8k9l

注意

  • reflogはローカルのみに保存されます（リモートにはありません）
  • デフォルトで90日間保持されます（古いものは消えます）
  • コミットしていない変更はreflogでも復活できません

11.2 間違えてコミットした時 - git commit --amend

「コミットメッセージを間違えた」「ファイルを追加し忘れた」という経験はありませんか？ 直前のコミットであれば、git commit --amendで修正できます。

コミットメッセージの修正

メッセージを修正 
# 直前のコミットメッセージを修正
git commit --amend -m "正しいコミットメッセージ"

# エディタでメッセージを編集する場合
git commit --amend

ファイルの追加忘れを修正

ファイルを追加して修正 
# 追加し忘れたファイルをステージング
git add forgotten-file.txt

# 直前のコミットに追加（メッセージはそのまま）
git commit --amend --no-edit

重要ポイント

--amendは新しいコミットを作成して古いコミットを置き換えます。 そのため、プッシュ済みのコミットを修正すると、リモートとの不整合が起きます。 プッシュ前のコミットにのみ使用しましょう。

11.3 コミットを取り消したい時 - git reset vs git revert

コミットを「なかったこと」にしたい場合、git resetとgit revertという2つの方法があります。 それぞれの違いを理解して、適切な方法を選びましょう。

reset と revert の違い

flowchart TB subgraph Before[変更前] B1[コミット A] --> B2[コミット B] --> B3[コミット C] end subgraph AfterReset["reset後（履歴を書き換え）"] R1[コミット A] --> R2[コミット B] R3["コミット C（消える）"] end subgraph AfterRevert["revert後（打ち消しを追加）"] V1[コミット A] --> V2[コミット B] --> V3[コミット C] --> V4["Revert C（打ち消し）"] end
項目 git reset git revert
動作 履歴を巻き戻す 打ち消しコミットを追加
履歴 変更される（消える） 残る（追加される）
プッシュ済み 非推奨（--force が必要） 安全に使用可能
使用場面 ローカルでの作業 共有リポジトリでの作業

git reset の3つのモード

git reset のモード 
# --soft: コミットだけ取り消し（変更は残る）
git reset --soft HEAD~1
# ステージング状態のまま残る

# --mixed: コミットとステージングを取り消し（デフォルト）
git reset HEAD~1
# 作業ディレクトリには変更が残る

# --hard: 全て取り消し（変更も消える）
git reset --hard HEAD~1
# 作業ディレクトリの変更も消える（危険）

git revert の使い方

git revert 
# 直前のコミットを打ち消す
git revert HEAD

# 特定のコミットを打ち消す
git revert a1b2c3d

# 複数のコミットを打ち消す（範囲指定）
git revert HEAD~3..HEAD

11.4 プッシュできない時のエラー対処

git pushを実行して「rejected」というエラーが出ることがあります。 これは、リモートに自分が持っていない変更があることを意味します。

よくあるエラーと解決方法

error: failed to push some refs to...

原因: リモートに新しい変更があり、ローカルが古い状態です

解決方法:

# リモートの変更を取り込んでからプッシュ
git pull --rebase origin main
git push origin main
error: src refspec main does not match any

原因: 指定したブランチが存在しないか、コミットがありません

解決方法:

# 現在のブランチを確認
git branch

# 最初のコミットがあるか確認
git log

# ブランチ名を確認してプッシュ
git push origin $(git branch --show-current)
Permission denied (publickey)

原因: SSH認証に失敗しています

解決方法:

# SSHキーがあるか確認
ls -la ~/.ssh/

# SSH接続をテスト
ssh -T git@github.com

# 必要に応じてSSHキーを再生成
ssh-keygen -t ed25519 -C "your_email@example.com"

11.5 プル時のコンフリクト解決

git pullを実行した時にコンフリクト（競合）が発生することがあります。 第8章で学んだマージのコンフリクトと同様に解決しましょう。

コンフリクト解決の手順

コンフリクト解決 
# 1. コンフリクトが発生するファイルを確認
git status

# 2. コンフリクトマーカーを含むファイルを編集
# <<<<<<< HEAD
# あなたの変更
# =======
# リモートの変更
# >>>>>>> origin/main

# 3. 編集後、ステージングして完了
git add conflicted-file.txt
git commit -m "Merge conflict resolved"

ヒント: プルを中断する

コンフリクトの解決が難しい場合、git merge --abortでプル（マージ）を中断し、元の状態に戻ることができます。

11.6 「detached HEAD」状態からの復帰

「detached HEAD」というメッセージを見たことはありますか？ これはどのブランチにも属していない状態を意味します。

detached HEAD とは

flowchart LR subgraph Normal[通常の状態] N1[コミット A] --> N2[コミット B] --> N3[コミット C] NM["main"] --> N3 NH["HEAD"] --> NM end subgraph Detached[detached HEAD状態] D1[コミット A] --> D2[コミット B] --> D3[コミット C] DM["main"] --> D3 DH["HEAD"] --> D2 end

通常、HEADはブランチを指し、ブランチがコミットを指します。 detached HEAD状態では、HEADが直接コミットを指しています。

detached HEAD になる原因

  • 特定のコミットを直接チェックアウトした時（git checkout a1b2c3d）
  • タグをチェックアウトした時（git checkout v1.0）
  • リモートブランチを直接チェックアウトした時

復帰方法

detached HEADからの復帰 
# 方法1: 既存のブランチに戻る
git checkout main
# または
git switch main

# 方法2: 現在の状態を新しいブランチとして保存
git checkout -b new-branch-name
# または
git switch -c new-branch-name

11.7 よくあるエラーメッセージの対処法

Gitで表示されるエラーメッセージと、その対処法をまとめました。

fatal: not a git repository

意味: 現在のディレクトリがGitリポジトリではありません

対処: git initでリポジトリを作成するか、正しいディレクトリに移動してください

error: Your local changes would be overwritten

意味: コミットしていない変更があり、操作によって上書きされます

対処: 変更をコミットするか、git stashで一時退避してから操作してください

fatal: refusing to merge unrelated histories

意味: 共通の祖先がない2つの履歴をマージしようとしています

対処: git pull origin main --allow-unrelated-historiesで強制的にマージできます（注意して使用）

error: pathspec 'xxx' did not match any file(s)

意味: 指定したファイルやブランチが見つかりません

対処: ファイル名やブランチ名のスペルを確認してください。git branch -aやgit statusで確認できます

実習 11-1: reflogを使った復旧体験

実際にコミットを「消して」から復旧する体験をしてみましょう。

手順

  1. 1 練習用のリポジトリで、ファイルを作成してコミットします
  2. 2 git reset --hard HEAD~1でコミットを消します
  3. 3 git reflogで消えたコミットのハッシュを確認します
  4. 4 git reset --hard [ハッシュ]で復旧します

実行例

# ファイル作成とコミット
echo "大切なデータ" > important.txt
git add important.txt
git commit -m "大切なデータを追加"

# コミットを消す（意図的に）
git reset --hard HEAD~1

# reflogで確認
git reflog
# 出力: xxxx HEAD@{1}: commit: 大切なデータを追加

# 復旧
git reset --hard HEAD@{1}

# 確認
cat important.txt
# 出力: 大切なデータ

第11章のまとめ

  • git reflogで操作履歴を確認し、「消えた」コミットを復活できる
  • git commit --amendで直前のコミットを修正できる（プッシュ前のみ）
  • git resetは履歴を書き換え、git revertは打ち消しを追加する
  • プッシュ済みのコミットにはrevertを使うのが安全
  • エラーメッセージを読めば、多くの問題は自己解決できる

理解度確認クイズ

  1. Q1. git reflogはどのような情報を記録していますか？

    回答を表示

    git reflogは、HEADやブランチの参照が「いつ、どこを指していたか」の履歴を記録しています。 コミットの履歴（git log）とは異なり、reset、checkout、mergeなど、あなたが行った操作の履歴を90日間保持します。 これにより、誤って消してしまったコミットも復旧できます。

  2. Q2. git resetとgit revertの違いは何ですか？

    回答を表示

    git resetは履歴を巻き戻し、指定したコミット以降の履歴を消します。
    git revertは打ち消しコミットを新たに追加し、履歴は残ります。
    プッシュ済みのコミットにはrevertを使い、ローカルでの作業にはresetを使うのが一般的です。

  3. Q3. detached HEAD状態とは何ですか？どうすれば復帰できますか？

    回答を表示

    detached HEAD状態は、HEADがブランチではなく、直接コミットを指している状態です。 特定のコミットやタグをチェックアウトした時に発生します。
    復帰するには、git checkout mainで既存のブランチに戻るか、 git checkout -b new-branchで現在の状態を新しいブランチとして保存します。
