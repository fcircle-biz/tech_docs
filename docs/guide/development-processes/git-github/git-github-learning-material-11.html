第11章

実践的なGit操作 - 履歴の操作とトラブル対処

より実践的なGit操作とトラブルシューティングを学びます。直前のコミットを修正する方法、変更を取り消す方法、作業を一時退避する方法など、実務でよく使う操作を習得しましょう。

この章で学ぶこと

  • git commit --amendで直前のコミットを修正する
  • git resetで変更を取り消す（soft, mixed, hard）
  • git revertでコミットを打ち消す
  • git stashで作業を一時退避する
  • git reflogで失った変更を復元する
  • git cherry-pickで特定のコミットを取り込む
  • よくあるエラーメッセージと対処法を理解する
  • detached HEAD状態からの復帰方法を知る

11.1 直前のコミットの修正 - git commit --amend

コミットした直後に「しまった、タイポがあった!」「ファイルを追加し忘れた!」と気づくことはよくあります。git commit --amendを使えば、直前のコミットを修正できます。

git commit --amendの用途

  • コミットメッセージの修正
  • ファイルの追加忘れの修正
  • 直前のコミットへの小さな変更の追加

コミットメッセージの修正

コミットメッセージにタイポがあった場合や、より良い説明に変更したい場合：

ターミナル 
# コミットメッセージを修正（エディタが開く）
git commit --amend

# エディタを開かずに直接メッセージを指定
git commit --amend -m "新しいコミットメッセージ"

ファイルの追加忘れの修正

コミットした後に「あのファイルを追加し忘れた!」と気づいた場合：

ターミナル 
# 追加忘れたファイルをステージング
git add forgotten_file.txt

# 直前のコミットに追加（メッセージは変更しない）
git commit --amend --no-edit

重要な注意点

git commit --amendは履歴を書き換えます。すでにプッシュ済みのコミットに対して--amendを使うと、他の開発者に迷惑をかける可能性があります。プッシュ前のコミットに対してのみ使用してください。

11.2 変更の取り消し - git reset

git resetは、コミット履歴を巻き戻すコマンドです。3つのモード（soft, mixed, hard）があり、それぞれ動作が異なります。

ステージングは残す

ステージングも取り消し
ファイル変更は残す"] E[git reset --hard HEAD~1] --> F["コミットを取り消し
ステージングも取り消し
ファイル変更も破棄"] style A fill:#dbeafe,stroke:#3b82f6 style B fill:#dbeafe,stroke:#3b82f6 style C fill:#fef3c7,stroke:#f59e0b style D fill:#fef3c7,stroke:#f59e0b style E fill:#fecaca,stroke:#ef4444 style F fill:#fecaca,stroke:#ef4444

git reset --soft（コミットのみ取り消し）

コミットを取り消しますが、ステージングエリアとファイルの変更は残ります。コミットをやり直したい場合に便利です。

ターミナル 
# 直前のコミットを取り消し（ファイルとステージングは残る）
git reset --soft HEAD~1

# コミットし直す
git commit -m "新しいコミットメッセージ"

git reset --mixed（デフォルト、ステージングも取り消し）

コミットとステージングを取り消しますが、ファイルの変更は残ります。オプションを指定しない場合のデフォルト動作です。

ターミナル 
# 直前のコミットとステージングを取り消し（ファイルは残る）
git reset HEAD~1
# または
git reset --mixed HEAD~1

# ファイルは変更されたまま残るので、再度addとcommitできる
git add .
git commit -m "修正後のコミット"

git reset --hard（全て破棄）

コミット、ステージング、ファイルの変更をすべて破棄します。完全に元の状態に戻るため、使用には注意が必要です。

ターミナル 
# 直前のコミットとファイルの変更を完全に破棄
git reset --hard HEAD~1

# 特定のコミットまで戻る
git reset --hard a1b2c3d

git reset --hardの危険性

--hardオプションはファイルの変更を完全に削除します。通常のファイル削除と異なり、ゴミ箱にも入りません。使用前に本当に削除して良いか確認してください。ただし、git reflog（後述）を使えば復元できる場合もあります。

HEAD~1とは？

  • HEAD - 現在のブランチの最新コミット
  • HEAD~1 - 1つ前のコミット
  • HEAD~2 - 2つ前のコミット
  • HEAD^ - HEAD~1と同じ（1つ前）

11.3 安全な取り消し - git revert

git revertは、特定のコミットの変更を打ち消す新しいコミットを作成します。git resetと異なり、履歴を書き換えないため、プッシュ済みのコミットを取り消す場合に適しています。

git resetとgit revertの違い

resetは履歴を削除しますが、revertは新しいコミットで変更を打ち消すため、履歴が残ります。

git revertの基本的な使い方

ターミナル 
# 直前のコミットを打ち消す
git revert HEAD

# 特定のコミットを打ち消す
git revert a1b2c3d

# コミットメッセージを編集せずに自動で作成
git revert HEAD --no-edit
項目 git reset git revert
履歴の扱い 履歴を削除（書き換え） 新しいコミットを追加（保持）
プッシュ済み 使用すべきでない 使用可能（推奨）
用途 ローカルの未プッシュコミット プッシュ済みコミット
安全性 注意が必要 安全

使い分けのポイント

  • ローカルのみの変更 → git reset
  • プッシュ済みの変更 → git revert
  • チーム開発 → git revert（履歴を保持）

11.4 作業の一時退避 - git stash

作業中に突然「緊急のバグ修正をしてほしい」と頼まれたとき、今の変更をコミットせずに別の作業をしたいことがあります。git stashは、現在の変更を一時的に退避させ、後で復元できるようにします。

基本的な使い方

ターミナル 
# 現在の変更を退避
git stash

# メッセージ付きで退避
git stash save "作業途中の機能A"

# 退避した一覧を表示
git stash list

# 最新の退避を復元
git stash pop

# 特定の退避を復元
git stash apply stash@{1}

stashの典型的な流れ

全角括弧作業を退避全角括弧 Note over WD: クリーンな状態に WD->>Branch: git checkout main
全角括弧バグ修正全角括弧 Branch->>WD: バグ修正完了 Stash->>WD: git stash pop
全角括弧作業を復元全角括弧 Note over WD: 機能Aの開発を再開

その他の便利なstashコマンド

ターミナル 
# 退避内容を確認
git stash show

# 退避内容を詳細表示
git stash show -p

# 特定の退避を削除
git stash drop stash@{0}

# 全ての退避を削除
git stash clear

git stash popとapplyの違い

  • git stash pop - 復元後、stash領域から削除される
  • git stash apply - 復元後、stash領域に残る（複数ブランチで適用可能）

11.5 失った変更の復元 - git reflog

「git reset --hardで間違ってコミットを削除してしまった!」という場合でも、git reflogを使えば復元できる可能性があります。reflogは、HEADの移動履歴を記録しています。

reflogで履歴を確認

ターミナル 
# HEADの移動履歴を表示
git reflog

# 出力例：
# a1b2c3d HEAD@{0}: reset: moving to HEAD~1
# e4f5g6h HEAD@{1}: commit: 重要な機能を追加
# i7j8k9l HEAD@{2}: commit: バグ修正

失ったコミットを復元

reflogで削除されたコミットのハッシュを見つけたら、そこにresetで戻れます：

ターミナル 
# reflogで見つけたコミットに戻る
git reset --hard e4f5g6h

# または、HEAD@{番号}で指定
git reset --hard HEAD@{1}

reflogの保持期間

reflogは通常、30日間保持されます（デフォルト設定）。そのため、ほとんどの誤操作は復元可能です。ただし、確実ではないので、重要な変更は定期的にプッシュするのがベストプラクティスです。

11.6 特定のコミットを取り込む - git cherry-pick

git cherry-pickは、他のブランチから特定のコミットだけを取り込むコマンドです。「あのブランチのあのコミットだけ欲しい」という場合に使います。

gitGraph commit id: "A" commit id: "B" branch feature commit id: "C" commit id: "D全角括弧これが欲しい全角括弧" type: HIGHLIGHT commit id: "E" checkout main commit id: "F" cherry-pick id: "D"

featureブランチのコミットDだけをmainブランチに取り込む例

基本的な使い方

ターミナル 
# 他のブランチのコミットを取り込む
git cherry-pick a1b2c3d

# 複数のコミットを取り込む
git cherry-pick a1b2c3d e4f5g6h

# コミット範囲を指定
git cherry-pick a1b2c3d..e4f5g6h

cherry-pickの典型的な使用例

  • 開発ブランチのバグ修正を本番ブランチに緊急適用
  • 間違ったブランチにコミットした変更を正しいブランチに移動
  • 実験的なブランチから一部の変更だけを取り込む

11.7 よくあるエラーメッセージと対処法

Gitを使っていると、様々なエラーメッセージに遭遇します。ここでは、特によく見るエラーとその対処法を紹介します。

エラー1: 「Your local changes would be overwritten」

error: Your local changes to the following files would be overwritten by checkout

原因: ローカルに未コミットの変更があり、ブランチ切り替えで上書きされてしまう

対処法:

  • 変更をコミットする: git commit -am "WIP"
  • 変更を退避する: git stash
  • 変更を破棄する: git checkout -- ファイル名

エラー2: 「fatal: refusing to merge unrelated histories」

fatal: refusing to merge unrelated histories

原因: 共通の履歴を持たないリポジトリをマージしようとしている

対処法:

  • 強制的にマージを許可: git pull origin main --allow-unrelated-histories

エラー3: 「error: failed to push some refs」

error: failed to push some refs to 'https://github.com/...'

原因: リモートに自分が持っていない新しいコミットがある

対処法:

  • まずpullしてからpush: git pull origin main → git push
  • rebaseでpull（履歴を一直線に）: git pull --rebase origin main

エラー4: 「Permission denied (publickey)」

Permission denied (publickey).

原因: SSH鍵の設定が正しくない

対処法:

  • SSH鍵を再設定（第7章参照）
  • または、HTTPS接続に切り替える

11.8 detached HEAD状態からの復帰

detached HEADとは、ブランチではなく特定のコミットに直接チェックアウトしている状態です。この状態で新しいコミットを作っても、ブランチに紐付かないため、ブランチを切り替えると見失ってしまいます。

detached HEAD警告メッセージ

You are in 'detached HEAD' state.

detached HEAD状態になる原因

ターミナル 
# 特定のコミットハッシュにチェックアウト
git checkout a1b2c3d

# タグにチェックアウト
git checkout v1.0.0

対処法1: 変更を保存したい場合

detached HEAD状態で行った変更を保存したい場合は、新しいブランチを作成します：

ターミナル 
# 新しいブランチを作成して移動
git switch -c new-branch

# または
git checkout -b new-branch

対処法2: 変更を破棄してブランチに戻る

ターミナル 
# 元のブランチに戻る（detached HEADの変更は破棄）
git switch main

# または
git checkout main

detached HEADを防ぐには

特定のコミットを見たいだけの場合は、git logやgit show コミットハッシュで内容を確認し、チェックアウトしないようにしましょう。

実習: 実践的なGit操作を体験

この章で学んだコマンドを実際に試してみましょう。

  1. 1. commit --amendの練習
    • 何か変更をコミット
    • コミットメッセージを修正
    • 追加のファイルをコミットに含める
  2. 2. git stashの練習
    • ファイルを編集（コミットしない）
    • git stashで退避
    • 別の作業をする
    • git stash popで復元
  3. 3. git resetの練習
    • 3つのコミットを作成
    • git reset --soft HEAD~1で1つ戻る
    • 再度コミット

理解度確認クイズ

  1. Q1. git commit --amendを使うべきでない状況は？

    回答を表示

    すでにリモートリポジトリにプッシュ済みのコミットに対しては使うべきではありません。--amendは履歴を書き換えるため、プッシュ済みのコミットを修正すると、他の開発者との履歴が食い違い、問題が発生します。

  2. Q2. git reset --soft, --mixed, --hardの違いを説明してください。

    回答を表示
    • --soft: コミットだけを取り消し、ステージングとファイル変更は残る
    • --mixed（デフォルト）: コミットとステージングを取り消し、ファイル変更は残る
    • --hard: コミット、ステージング、ファイル変更をすべて破棄

  3. Q3. プッシュ済みのコミットを取り消したい場合、resetとrevertのどちらを使うべきですか？

    回答を表示

    git revertを使うべきです。revertは新しいコミットで変更を打ち消すため、履歴を書き換えません。resetは履歴を書き換えるため、プッシュ済みのコミットに対して使うと他の開発者に問題が発生します。

  4. Q4. git stashの主な用途を2つ挙げてください。

    回答を表示
    • 作業中に緊急の別作業が入った時、現在の変更を一時退避する
    • ブランチ切り替え前に、未コミットの変更を一時的に保存する
    • 実験的な変更を一時的に退避して、後で復元する

    （上記のうち2つ）

  5. Q5. git reflogは何を記録していますか？また、どのような場面で役立ちますか？

    回答を表示

    reflogはHEADの移動履歴を記録しています。git reset --hardで誤ってコミットを削除してしまった場合や、ブランチを間違って削除した場合など、失った変更を復元する際に役立ちます。通常30日間履歴が保持されます。

第11章のまとめ

  • git commit --amendで直前のコミットを修正できる（プッシュ前のみ）
  • git resetには--soft, --mixed, --hardの3モードがあり、取り消す範囲が異なる
  • git revertはプッシュ済みのコミットを安全に打ち消せる
  • git stashで作業を一時退避し、後で復元できる
  • git reflogで失った変更を復元できる可能性がある
  • git cherry-pickで他のブランチの特定のコミットだけを取り込める
  • よくあるエラーメッセージの意味と対処法を理解した
  • detached HEAD状態からの復帰方法を学んだ
