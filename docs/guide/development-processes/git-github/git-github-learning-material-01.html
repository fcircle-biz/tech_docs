第1章

はじめてのバージョン管理 - なぜGitが必要なのか

バージョン管理の必要性とGitの基本概念を学びます。「なぜファイル名に日付をつけて保存するのは良くないのか」という身近な問題から始め、Gitが生まれた背景と特徴を理解しましょう。

この章で学ぶこと

  • バージョン管理とは何か、なぜ必要なのかを理解する
  • Gitが生まれた背景と歴史を知る
  • Gitの特徴（分散型バージョン管理）を理解する
  • GitとGitHubの違いを明確に説明できる
  • バージョン管理がもたらすメリットを具体的に知る

1.1 バージョン管理とは何か

プログラムや文書を作成していると、こんな経験はありませんか？

よくあるファイル管理の例

  • レポート_最終版.docx
  • レポート_最終版2.docx
  • レポート_本当に最終版.docx
  • レポート_20250115.docx
  • レポート_20250115_修正.docx

このように、ファイル名に「最終版」や日付を付けて保存する方法には、いくつかの問題があります：

ファイル名での管理の問題点

  • どれが本当に最新版かわからない - ファイル名だけでは判断が難しい
  • 変更内容が不明 - 何を変更したのかファイル名からはわからない
  • 過去の状態に戻せない - 間違って上書きすると元に戻せない
  • 複数人での作業が困難 - 誰がいつ何を変更したかわからない
  • ディスク容量の無駄 - 同じファイルのコピーが大量に増える

バージョン管理システム（Version Control System, VCS）は、こうした問題を解決するための仕組みです。ファイルの変更履歴を自動的に記録し、いつでも過去の状態に戻したり、誰が何を変更したかを追跡したりできます。

バージョン管理システムができること

  • 変更履歴の記録 - いつ・誰が・何を変更したかが全て記録される
  • 過去の状態に戻す - 任意の時点の状態に簡単に戻せる
  • 変更内容の比較 - 前後の変更点を一目で確認できる
  • 並行作業の統合 - 複数人が同時に作業しても、変更を統合できる
  • 効率的なストレージ - 差分だけを保存するため容量を節約できる

1.2 Gitの誕生 - Linuxカーネル開発から生まれたツール

Gitは、2005年にリーナス・トーバルズ（Linus Torvalds）によって開発されました。リーナスは、オペレーティングシステム「Linux」の開発者としても知られています。

Gitが生まれた背景

Linuxカーネルは、世界中の数千人の開発者が協力して開発している巨大プロジェクトです。当初は別のバージョン管理システムを使っていましたが、2005年にそのツールが無料で使えなくなるという問題が発生しました。

そこでリーナスは、Linuxカーネル開発に必要な要件を満たす全く新しいバージョン管理システムを、わずか数日で開発しました。それがGitです。

Gitの名前の由来については諸説ありますが、リーナス本人によると「頑固者」という意味の英国スラング（git）から来ているとのことです。

Gitが重視した設計思想

高速性

大規模プロジェクトでも高速に動作する

分散型

各開発者が完全な履歴を持つ

データ整合性

全てのデータはチェックサムで保護

並行開発

ブランチによる効率的な並行作業

1.3 分散型バージョン管理システムとは

バージョン管理システムには、主に「集中型」と「分散型」の2つの方式があります。Gitは分散型バージョン管理システムです。

集中型バージョン管理の仕組み

集中型では、中央サーバーに全ての履歴が保存されます。開発者は必要な部分だけをチェックアウトして作業します。

分散型バージョン管理の仕組み（Git）

分散型では、各開発者のパソコンに完全な履歴を持つリポジトリがあります。オフラインでも作業でき、必要に応じてリモートと同期します。

特徴 集中型（SVN等） 分散型（Git）
履歴の保存場所 中央サーバーのみ 各開発者のローカル
オフライン作業 不可能 可能
コミットの速度 遅い（ネットワーク経由） 高速（ローカル）
バックアップ 中央サーバーのみ 全開発者が保持
ブランチ作成 遅く、コストが高い 高速で軽量

1.4 GitとGitHubの違い

GitとGitHubは名前が似ていますが、全く異なるものです。この違いを理解することは非常に重要です。

項目 Git GitHub
種類 ソフトウェア（ツール） Webサービス（プラットフォーム）
役割 ファイルの変更履歴を管理 Gitで管理したファイルをネット上で共有・公開
動作場所 自分のパソコン（ローカル） インターネット上（クラウド）
費用 無料（オープンソース） 無料プランあり（有料プランも）
提供元 オープンソースコミュニティ Microsoft社
例え 日記帳（変更の記録を残す） 日記を保管する図書館（共有・公開）

簡単に言うと

Gitは「変更履歴を記録するツール」、GitHubは「その記録をネット上で保存・共有するサービス」です。Gitがなくても生活できますが、GitHubを使うにはGitが必要です。

GitHub以外のGitホスティングサービス

GitHubが最も有名ですが、Gitのリポジトリをホスティング（保管・公開）できるサービスは他にもあります：

  • GitLab - オープンソースで、自社サーバーにも構築可能
  • Bitbucket - Atlassian社が提供、Jiraとの連携が強力
  • Azure DevOps - Microsoft社が提供する開発プラットフォーム

どのサービスを使っても、基本的なGitの操作は同じです。本教材ではGitHubを例に説明しますが、学んだ知識は他のサービスでも活用できます。

1.5 バージョン管理がもたらすメリット

バージョン管理システムを使うことで、個人開発でもチーム開発でも、多くのメリットが得られます。

1. 履歴の追跡と記録

全ての変更が記録されるため、「いつ」「誰が」「何を」「なぜ」変更したかが明確になります。

  • バグが発生した時、いつから問題があったか特定できる
  • 過去の意思決定の理由を確認できる
  • チーム全員の作業状況を把握できる

2. ミスからの復旧

間違った変更をしても、簡単に過去の正常な状態に戻すことができます。

  • 誤って削除したファイルを復元できる
  • 動いていた状態に一瞬で戻せる
  • 実験的な変更を安心して試せる

3. チーム開発の効率化

複数人が同時に作業しても、変更を安全に統合できます。

  • ファイルを送り合う必要がない
  • 誰かの変更を上書きしてしまう心配がない
  • コードレビューを通じて品質を向上できる

4. 並行作業とブランチ

複数の機能を同時並行で開発できます。

  • 新機能の開発中も、本番のバグ修正ができる
  • リリース版と開発版を分けて管理できる
  • 実験的な機能を本体に影響なく試せる

5. 自動的なバックアップ

リモートリポジトリにプッシュすることで、自動的にバックアップが取られます。

  • パソコンが壊れてもデータは安全
  • 別のパソコンでも同じ環境で作業できる
  • 災害対策にもなる

1.6 Git/GitHubでよく使う用語

Git/GitHubを学ぶ上で、いくつかの重要な用語があります。ここでは初心者向けに、日本語でわかりやすく説明します。

リポジトリ（Repository）

ファイルや変更履歴を保存する「保管庫」のこと。プロジェクト全体を管理する入れ物です。

コミット（Commit）

変更を記録すること。ゲームのセーブポイントのようなもので、いつでもこの時点に戻れます。

ブランチ（Branch）

開発の「枝分かれ」。木の枝のように、本体から分岐して別々の作業ができます。

マージ（Merge）

ブランチを統合すること。別々に作業した変更を1つにまとめます。

プッシュ（Push）

ローカルの変更をリモートリポジトリ（GitHub等）にアップロードすること。

プル（Pull）

リモートリポジトリの最新の変更をローカルにダウンロードすること。

クローン（Clone）

リモートリポジトリを自分のパソコンに複製すること。全ての履歴もコピーされます。

プルリクエスト（Pull Request）

「この変更を取り込んでください」という依頼。コードレビューを受けてからマージします。

ポイント

これらの用語は最初は難しく感じるかもしれませんが、実際に手を動かして操作していくうちに自然と理解できるようになります。次の章から、実際にGitとGitHubを使っていきましょう。

理解度確認クイズ

  1. Q1. ファイル名に「最終版」や日付をつけて保存する方法の問題点を3つ挙げてください。

    回答を表示

    以下のような問題があります（3つ挙げる）：

    • どれが本当に最新版かわからない
    • 変更内容がファイル名からはわからない
    • 過去の状態に戻せない
    • 複数人での作業が困難
    • ディスク容量の無駄

  2. Q2. GitとGitHubの違いを説明してください。

    回答を表示

    Gitは、ファイルの変更履歴を管理するソフトウェア（ツール）です。自分のパソコン上で動作します。

    GitHubは、Gitで管理したファイルをインターネット上で保存・共有できるWebサービス（プラットフォーム）です。

    簡単に言うと、Gitは「変更履歴を記録するツール」、GitHubは「その記録をネット上で保存・共有するサービス」です。

  3. Q3. 分散型バージョン管理システムの利点を2つ挙げてください。

    回答を表示

    以下のような利点があります（2つ挙げる）：

    • 各開発者が完全な履歴を持つため、オフラインでも作業できる
    • ローカルでコミットできるため、高速に動作する
    • 全開発者がバックアップを持つため、データ消失のリスクが低い
    • ブランチの作成・マージが軽量で高速

  4. Q4. 「コミット」とは何ですか？簡単に説明してください。

    回答を表示

    コミットとは、ファイルの変更を記録することです。ゲームのセーブポイントのようなもので、コミットした時点の状態を保存し、いつでもその時点に戻ることができます。

  5. Q5. バージョン管理システムを使うメリットを3つ挙げてください。

    回答を表示

    以下のようなメリットがあります（3つ挙げる）：

    • 全ての変更履歴が記録され、いつ・誰が・何を変更したかわかる
    • 間違った変更をしても、過去の正常な状態に簡単に戻せる
    • 複数人が同時に作業しても、変更を安全に統合できる
    • ブランチを使って並行開発ができる
    • リモートリポジトリで自動的にバックアップが取られる

第1章のまとめ

  • バージョン管理システムは、ファイル名に日付を付ける方法の問題を解決する
  • Gitは2005年にリーナス・トーバルズがLinuxカーネル開発のために作成した
  • Gitは分散型バージョン管理システムで、各開発者が完全な履歴を持つ
  • Gitはツール、GitHubはそれを使ったWebサービス（プラットフォーム）
  • バージョン管理により、履歴追跡・安全な変更・チーム開発の効率化などが実現できる
