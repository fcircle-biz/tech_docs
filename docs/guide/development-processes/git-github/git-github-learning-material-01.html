第1章

はじめてのバージョン管理 - なぜGitが必要なのか

バージョン管理の必要性とGitの基本概念を学びます。ファイル管理の日常的な問題から、Gitがどのように解決してくれるのかを理解しましょう。

この章で学ぶこと

  • バージョン管理システムとは何か、なぜ必要なのか
  • Gitの誕生背景と特徴（分散型バージョン管理）
  • GitとGitHubの違いを明確に理解する
  • 基本用語（リポジトリ、コミット、ブランチなど）の意味

1.1 バージョン管理とは - よくある失敗から考える

こんな経験はありませんか？

  • ファイル名に「最終版」「最終版2」「本当の最終版」と付けたことがある
  • 「昨日まで動いていたのに...」と困ったことがある
  • 誰がいつ何を変更したか分からなくなった

多くの人がファイル管理で経験する悩みです。特にチームで作業すると、「どれが最新？」「この変更は誰がした？」という混乱が頻繁に起こります。

バージョン管理システム（Version Control System / VCS）とは、ファイルの変更履歴を記録・管理するためのシステムです。以下のような問題を解決してくれます。

問題 従来の方法 バージョン管理
最新版が分からない ファイル名に日付を付ける 常に最新版が明確
過去に戻したい バックアップフォルダを探す ワンクリックで復元
誰が変更したか不明 ファイル名に名前を付ける 変更者・日時を自動記録
複数人で編集 順番待ち・手作業で統合 同時編集・自動統合

日付ファイル名管理の問題点

1.2 Gitとは - 世界で最も使われているバージョン管理システム

Git（ギット）は、2005年にLinuxカーネルの開発者リーナス・トーバルズによって開発された、分散型バージョン管理システムです。

用語解説: 分散型とは？

「分散型」とは、すべての開発者が完全な履歴を持つことを意味します。

  • 集中型: サーバー1台に履歴がある（サーバーが壊れると全て失う）
  • 分散型: 各自のパソコンに完全な履歴がある（誰かのPCがあれば復旧可能）

集中型 vs 分散型バージョン管理

Gitの誕生背景

Linuxカーネル（OSの中心部分）の開発には、世界中の何千人もの開発者が参加しています。以前使っていたツールが使えなくなったことをきっかけに、リーナス・トーバルズは自ら新しいツールを作ることを決意しました。

彼が求めた条件は以下の3つでした：

  • 高速: 巨大なプロジェクトでも快適に動作する
  • 分散型: オフラインでも作業でき、誰でも完全な履歴を持てる
  • データの整合性: 履歴が破壊されたり改ざんされたりしない

これらを満たすGitは、今や世界で最も使われているバージョン管理システムとなりました。個人開発からGoogle、Microsoft、Facebookなどの大企業まで、あらゆる規模のプロジェクトで利用されています。

1.3 GitとGitHubの違い - 混同しやすい2つを整理

「Git」と「GitHub」は名前が似ていますが、全く別のものです。この違いを理解することは、学習を進める上で非常に重要です。

Git

ソフトウェア（ツール）
  • 自分のパソコンにインストール
  • ファイルの変更履歴を記録
  • オフラインで使用可能
  • 無料で使用可能

例え: 日記帳（変更の記録を残すもの）

GitHub

Webサービス（プラットフォーム）
  • インターネット上のサービス
  • Gitで管理したファイルを保存・共有
  • チームでの共同作業を支援
  • 無料プランあり（有料プランも）

例え: 図書館（日記を保管・共有できる場所）

覚えておきたいポイント

  • GitはGitHub「なし」でも使える（自分のPCだけで完結）
  • GitHubはGitの「上に」成り立つサービス
  • GitHub以外にも GitLab、Bitbucket など類似サービスがある

1.4 覚えておきたい基本用語

Gitを学ぶ上で最初に押さえておきたい用語を紹介します。今は完全に理解できなくても大丈夫です。学習を進めながら、何度も出てくるうちに自然と身につきます。

リポジトリ (Repository)

ファイルと変更履歴を保存する「貯蔵庫」のこと。略して「リポ」や「レポ」とも呼ばれます。

ローカルリポジトリ: 自分のPC上 リモートリポジトリ: GitHub上

コミット (Commit)

変更を「記録する」こと。ゲームで言うと「セーブ」のようなもの。いつでもコミットした時点に戻れます。

「ログイン機能を追加」などのメッセージと共に記録します

ブランチ (Branch)

作業を「枝分かれ」させる機能。メインの作業に影響を与えずに、新機能の開発や実験ができます。

例: 「main」ブランチは本番用、「feature-login」ブランチでログイン機能を開発

マージ (Merge)

分岐した作業を「統合」すること。開発が完了したブランチをメインに取り込みます。

プッシュ (Push) / プル (Pull)

リモートリポジトリとのやり取り。

Push: ローカル → リモート（アップロード） Pull: リモート → ローカル（ダウンロード）

1.5 バージョン管理で得られるメリット

Gitを使ったバージョン管理がもたらす具体的なメリットを見ていきましょう。

履歴の追跡

誰が、いつ、何を変更したかが全て記録されます。「いつから動かなくなったか」も簡単に特定できます。

ミスからの復旧

間違えて削除したり、壊してしまっても、いつでも過去の状態に戻せます。実験も安心して行えます。

チーム開発の効率化

複数人が同時に作業しても、変更を自動的に統合。順番待ちや手作業での統合が不要になります。

バックアップと安全性

GitHubにプッシュしておけば、PCが壊れても安心。複数箇所に同じデータがあるため、データ損失リスクが低減します。

第1章のまとめ

  • バージョン管理は、ファイルの変更履歴を記録・管理するシステム
  • Gitは、自分のPCにインストールする分散型バージョン管理ツール
  • GitHubは、Gitで管理したファイルを保存・共有するWebサービス
  • リポジトリ、コミット、ブランチ、マージなど基本用語を学んだ

理解度確認クイズ

  1. Q1. GitとGitHubの違いを説明してください。

    回答を表示

    Gitは自分のパソコンにインストールする「ソフトウェア（ツール）」で、ファイルの変更履歴を管理します。
    GitHubはインターネット上の「Webサービス（プラットフォーム）」で、Gitで管理したファイルを保存・共有するために使います。
    つまり、GitはGitHubなしでも使えますが、GitHubはGitの上に成り立つサービスです。

  2. Q2. 「コミット」とは何ですか？

    回答を表示

    コミットとは、ファイルの変更を「記録する」ことです。ゲームの「セーブ」のようなもので、コミットすることで、いつでもその時点の状態に戻ることができます。通常、「ログイン機能を追加」などの説明メッセージと共に記録します。

  3. Q3. 「分散型」バージョン管理の特徴は何ですか？

    回答を表示

    分散型バージョン管理の特徴は、すべての開発者が「完全な履歴」を持つことです。中央サーバーだけでなく、各自のパソコンに完全な履歴があるため、オフラインでも作業ができ、サーバーが壊れても誰かのPCから復旧できます。
