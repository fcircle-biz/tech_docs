Gitの仕組み - ローカルとリモートの関係

Gitの内部構造とデータの流れを図解で理解します。ワーキングディレクトリ、ステージングエリア、ローカルリポジトリの3つの領域と、ローカルとリモートの関係を視覚的に学びましょう。

この章で学ぶこと

  • ワーキングディレクトリ、ステージングエリア、ローカルリポジトリの3つの領域を理解する
  • ローカルリポジトリとリモートリポジトリの関係を理解する
  • push/pull/fetchの概念を理解する
  • Gitオブジェクト（blob、tree、commit）の基礎を知る
  • HEADとブランチポインタの仕組みを理解する
  • .gitディレクトリの役割を知る

4.1 Gitの3つの領域 - ワーキングディレクトリ、ステージングエリア、リポジトリ

Gitは、ファイルを管理するために3つの異なる領域を使用します。この仕組みを理解することで、Gitの操作が直感的に理解できるようになります。

Gitの3つの領域とデータの流れ

ファイルは「ワーキングディレクトリ」→「ステージングエリア」→「ローカルリポジトリ」の順に進みます。

1. ワーキングディレクトリ（Working Directory）

実際にファイルを編集する作業フォルダです。パソコンのエクスプローラーやFinderで見えるフォルダと同じです。

  • ファイルを作成・編集・削除する場所
  • まだGitには記録されていない状態
  • git statusで変更状況を確認できる

2. ステージングエリア（Staging Area / Index）

次のコミットに含める変更を準備する領域です。「インデックス」とも呼ばれます。

  • git addコマンドでファイルをステージング
  • コミットする内容を選別できる（一部だけコミット可能）
  • 「コミット予定リスト」のようなもの

3. ローカルリポジトリ（Local Repository）

変更履歴が保存される場所です。.gitフォルダの中に全ての履歴が記録されます。

  • git commitコマンドで変更を記録
  • 過去のコミット履歴を全て保持
  • いつでも過去の状態に戻せる

なぜ3つの領域に分かれているのか？

ステージングエリアがあることで、「この変更はコミットに含めるが、この変更はまだ含めない」という細かい調整ができます。例えば、バグ修正と新機能を同時に作業していても、バグ修正だけを先にコミットすることが可能です。

4.2 ローカルリポジトリとリモートリポジトリ

Gitは分散型バージョン管理システムなので、ローカル（自分のパソコン）とリモート（GitHubなどのサーバー）の両方にリポジトリが存在します。

ローカルとリモートの関係

ローカルで変更を記録（commit）した後、リモートにアップロード（push）します。リモートの変更はダウンロード（pull）して取り込みます。

項目 ローカルリポジトリ リモートリポジトリ

アクセス オフラインでも利用可能 インターネット接続が必要
役割 個人の作業環境 バックアップ・共有の中心
操作 add, commit, branch等 push, pull, fetch等

4.3 push/pull/fetchの概念理解

ローカルとリモートを同期するための3つの重要なコマンドがあります。

git push - ローカルからリモートへアップロード

ローカルでコミットした変更を、リモートリポジトリにアップロードします。

  • ローカルの変更を他の人と共有する
  • バックアップとして保存する
  • コミット履歴も一緒にアップロード

git pull - リモートからローカルへダウンロード＋マージ

リモートリポジトリの最新の変更を、ローカルにダウンロードして統合（マージ）します。

  • 他の人の変更を取り込む
  • fetch + mergeを一度に実行
  • 作業前に実行して最新状態にする

git fetch - リモートの情報を取得（マージはしない）

リモートリポジトリの最新情報を取得しますが、ローカルのファイルには反映しません。

  • リモートの状態を確認する
  • マージ前に変更内容を確認できる
  • pullより安全（自動マージしない）

git pull と git fetch の違い

flowchart TB subgraph Pull["git pull の動作"] P1[リモートから取得] --> P2[自動的にマージ] end subgraph Fetch["git fetch の動作"] F1[リモートから取得] --> F2[手動でマージ判断] end style P1 fill:#bfdbfe,stroke:#3b82f6 style P2 fill:#fde68a,stroke:#f59e0b style F1 fill:#bfdbfe,stroke:#3b82f6 style F2 fill:#d1fae5,stroke:#10b981

pullは取得と統合を自動で行い、fetchは取得のみで統合は手動です。

使い分けのポイント

  • git pull: 最新の変更をすぐに取り込みたい時（日常作業）
  • git fetch: リモートの状態を確認してから慎重に統合したい時
  • git push: 自分の作業を共有・バックアップしたい時

4.4 Gitオブジェクトの基礎 - blob、tree、commit

Gitは内部でファイルや変更履歴を「オブジェクト」として管理しています。主に3種類のオブジェクトがあります。

Gitオブジェクトの構造

commitオブジェクトがtreeを参照し、treeが複数のblobを参照する階層構造です。

blob（ブロブ）

ファイルの内容を保存するオブジェクト

  • ファイル名は含まない
  • 内容だけを保存
  • 同じ内容なら1つだけ保存

tree（ツリー）

ディレクトリ構造を保存するオブジェクト

  • ファイル名とblobの対応
  • ディレクトリ構造を表現
  • 他のtreeも参照可能

commit（コミット）

スナップショットとメタ情報を保存

  • treeオブジェクトを参照
  • 作成者、日時、メッセージ
  • 親コミットへの参照

オブジェクトとSHA-1ハッシュ

全てのGitオブジェクトは、その内容から生成されるSHA-1ハッシュ値（40文字の英数字）で識別されます。同じ内容なら必ず同じハッシュ値になるため、データの整合性が保証されます。

4.5 HEADとブランチポインタの仕組み

Gitでは「HEAD」と「ブランチ」という概念を使って、現在の作業位置と履歴を管理します。

HEADとブランチの関係

gitGraph commit id: "Initial commit" commit id: "Add feature A" commit id: "Fix bug" branch develop checkout develop commit id: "Add feature B" commit id: "Update docs" checkout main commit id: "Hotfix"

HEADは現在作業中のブランチを指し、ブランチは最新のコミットを指します。

HEAD - 現在の作業位置

HEADは「今、どこで作業しているか」を示すポインタです。通常は現在のブランチを指しています。

  • 現在作業中のブランチを指す
  • 新しいコミットはHEADの位置に作成される
  • git checkoutでHEADが移動する

ブランチポインタ - 最新コミットへの参照

ブランチは単なるポインタで、そのブランチの最新のコミットを指しています。

  • ブランチは特定のコミットを指す軽量なポインタ
  • 新しいコミットが作成されると自動的に更新される
  • ブランチを切り替えるとHEADが移動する

HEADの状態確認

# HEADが指しているブランチを確認
git branch

# 現在のHEADの詳細情報
cat .git/HEAD

detached HEAD状態とは

HEADが特定のコミットを直接指している状態を「detached HEAD」と言います。この状態でコミットを作成すると、ブランチに紐付かないため、後で見失う可能性があります。通常はブランチ上で作業しましょう。

4.6 .gitディレクトリの役割

Gitリポジトリの全ての情報は、.gitディレクトリという隠しフォルダに保存されています。

.gitディレクトリの主な構造

.git/
├── HEAD              # 現在のブランチへの参照
├── config            # リポジトリの設定ファイル
├── description       # リポジトリの説明
├── hooks/            # Gitフック（イベント時に実行されるスクリプト）
├── index             # ステージングエリアの内容
├── logs/             # 操作履歴（reflog）
├── objects/          # Gitオブジェクト（blob、tree、commit）の保存場所
├── refs/             # ブランチとタグの参照
│   ├── heads/        # ローカルブランチ
│   └── remotes/      # リモートブランチ
└── ...
HEAD

現在作業中のブランチを指すファイル

index

ステージングエリアの内容を保存（バイナリ形式）

objects/

全てのGitオブジェクト（コミット履歴、ファイル内容）を保存

refs/heads/

各ブランチが指すコミットのハッシュ値を保存

refs/remotes/

リモートブランチの情報を保存

重要な注意

.gitディレクトリを削除すると、全ての履歴が失われます。通常は直接編集する必要はありません。Gitコマンドを使って操作しましょう。

.gitディレクトリの確認方法

通常、.gitディレクトリは隠しフォルダになっています。Windows/macOSのエクスプローラーやFinderで隠しファイルを表示する設定にすると確認できます。ただし、中身を直接編集することは推奨されません。

理解度確認クイズ

  1. Q1. Gitの3つの領域の名前を挙げ、それぞれの役割を簡潔に説明してください。

    回答を表示
    • ワーキングディレクトリ: 実際にファイルを編集する作業フォルダ
    • ステージングエリア: 次のコミットに含める変更を準備する領域
    • ローカルリポジトリ: 変更履歴が保存される場所（.gitフォルダ内）

  2. Q2. git pullとgit fetchの違いを説明してください。

    回答を表示

    git pullは、リモートリポジトリの変更を取得して、自動的にローカルブランチにマージします。

    git fetchは、リモートリポジトリの変更を取得しますが、自動的にはマージしません。手動で変更内容を確認してからマージできるため、より安全です。

    簡単に言うと、pull = fetch + mergeです。

  3. Q3. Gitの3種類のオブジェクト（blob、tree、commit）の役割を説明してください。

    回答を表示
    • blob（ブロブ）: ファイルの内容を保存するオブジェクト（ファイル名は含まない）
    • tree（ツリー）: ディレクトリ構造を保存するオブジェクト（ファイル名とblobの対応）
    • commit（コミット）: スナップショットとメタ情報（作成者、日時、メッセージ、親コミット）を保存するオブジェクト

  4. Q4. HEADとは何ですか？その役割を説明してください。

    回答を表示

    HEADは「現在の作業位置」を示すポインタです。通常は現在のブランチを指しており、「今、どのブランチで作業しているか」を示します。新しいコミットはHEADの位置に作成され、git checkoutやgit switchでHEADが移動します。

  5. Q5. .gitディレクトリの役割を説明してください。また、削除するとどうなりますか？

    回答を表示

    .gitディレクトリは、Gitリポジトリの全ての情報（コミット履歴、ブランチ情報、設定、ステージングエリア、Gitオブジェクトなど）を保存する隠しフォルダです。

    .gitディレクトリを削除すると、全ての履歴が失われ、通常のフォルダに戻ります（Gitリポジトリではなくなります）。そのため、絶対に削除してはいけません。

第4章のまとめ

  • Gitは「ワーキングディレクトリ」「ステージングエリア」「ローカルリポジトリ」の3つの領域でファイルを管理する
  • ローカルリポジトリとリモートリポジトリは、push/pull/fetchで同期する
  • git pullは自動マージ、git fetchは取得のみで手動マージが可能
  • Gitオブジェクト（blob、tree、commit）がファイルと履歴を保存する
  • HEADは現在の作業位置を示し、ブランチは最新コミットを指すポインタ
  • .gitディレクトリに全ての履歴とGit情報が保存される
