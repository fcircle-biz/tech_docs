マージとコンフリクト - 変更の統合と競合解決

ブランチを統合（マージ）する方法と、コンフリクト（競合）の解決方法を学びます。複数の変更を安全に統合するための基礎を理解しましょう。

この章で学ぶこと

  • マージとは何か、どのように動作するかを理解する
  • git mergeコマンドの使い方を習得する
  • Fast-forwardマージと3-wayマージの違いを理解する
  • コンフリクト（競合）の仕組みと発生原因を理解する
  • コンフリクトマーカーを読み解き、手動で解決できる
  • マージを取り消す方法を知る

9.1 マージとは - ブランチを1つにまとめる

マージ（Merge）は、2つのブランチの変更履歴を統合する操作です。例えば、新機能を開発したfeatureブランチをmainブランチに統合したい場合にマージを使います。

マージの基本的なイメージ

木の枝が幹に戻るイメージです。枝（featureブランチ）で作業した内容を、幹（mainブランチ）に統合します。

マージの流れ

gitGraph commit id: "Initial" commit id: "A" branch feature checkout feature commit id: "B" commit id: "C" checkout main commit id: "D" merge feature id: "Merge feature" commit id: "E"

mainブランチとfeatureブランチが並行して進み、最終的にマージによって1つに統合されます。

マージの基本的な手順

  1. マージ先のブランチに切り替える - 例: mainブランチに切り替え
  2. git mergeコマンドを実行 - 例: git merge feature
  3. マージコミットが作成される - 2つのブランチの変更が統合される

9.2 git mergeコマンドの使い方

マージはgit mergeコマンドで実行します。基本的な使い方を見てみましょう。

実習: 基本的なマージを実行する

まず、新しいリポジトリでブランチを作成し、マージを体験してみましょう。

手順1: 新しいディレクトリとリポジトリを作成

# 新しいディレクトリを作成
mkdir merge-practice
cd merge-practice

# Gitリポジトリを初期化
git init

手順2: mainブランチで最初のファイルを作成

# README.mdを作成
echo "# マージの練習" > README.md

# コミット
git add README.md
git commit -m "initial commit"

手順3: 新しいブランチを作成して変更を追加

# featureブランチを作成して切り替え
git switch -c feature

# 新しいファイルを追加
echo "新機能の説明" > feature.txt
git add feature.txt
git commit -m "add feature.txt"

手順4: mainブランチに戻ってマージ

# mainブランチに切り替え
git switch main

# featureブランチをマージ
git merge feature

# ログを確認
git log --oneline --graph

9.3 Fast-forwardマージと3-wayマージの違い

Gitのマージには、主に2つのタイプがあります。どちらのタイプになるかは、ブランチの状況によって自動的に決まります。

Fast-forwardマージ

Fast-forward（早送り）マージは、マージ先のブランチが全く進んでいない場合に発生します。ブランチポインタを単純に前に進めるだけで済みます。

gitGraph commit id: "A" commit id: "B" branch feature checkout feature commit id: "C" commit id: "D" checkout main merge feature tag: "Fast-forward"

Fast-forwardマージの特徴

  • 新しいマージコミットは作成されない
  • mainブランチのポインタがfeatureブランチの位置まで移動するだけ
  • 履歴が一直線になり、シンプルで見やすい

3-wayマージ

3-wayマージは、両方のブランチが分岐後に独立して進んでいる場合に発生します。2つのブランチの変更を統合した新しいマージコミットが作成されます。

gitGraph commit id: "A" commit id: "B" branch feature checkout feature commit id: "C" commit id: "D" checkout main commit id: "E" commit id: "F" merge feature id: "Merge commit"

3-wayマージの特徴

  • 新しいマージコミットが作成される
  • マージコミットは2つの親コミットを持つ
  • 並行開発の履歴が明確に残る
  • 共通の祖先（分岐点）、2つのブランチの先端の3つを使って統合（3-way）
特徴 Fast-forward 3-way
発生条件 マージ先が分岐後に進んでいない 両方のブランチが進んでいる
マージコミット 作成されない 作成される
履歴の見た目 一直線 分岐と統合

9.4 コンフリクト（競合）とは

コンフリクト（競合、Conflict）は、2つのブランチで同じファイルの同じ場所を異なる方法で編集した場合に発生します。Gitは自動的に統合できないため、手動で解決する必要があります。

コンフリクトが発生する例

  • 同じ行を異なる内容で編集 - mainで「こんにちは」、featureで「Hello」に変更
  • 隣接する行の編集 - 近い位置で異なる変更をした場合も競合する可能性がある
  • 一方が削除、一方が編集 - mainで削除、featureで編集した場合
gitGraph commit id: "A: ファイルの初期状態" branch feature checkout feature commit id: "B: 同じ行をHelloに変更" checkout main commit id: "C: 同じ行をこんにちはに変更" merge feature id: "コンフリクト発生！"

上記の例では、同じファイルの同じ行をmainブランチとfeatureブランチで異なる内容に編集しているため、マージ時にコンフリクトが発生します。

9.5 コンフリクトの発生と確認方法

実際にコンフリクトを発生させて、その確認方法を学びましょう。

実習: コンフリクトを発生させる

手順1: 新しいリポジトリを作成

mkdir conflict-practice
cd conflict-practice
git init

# greeting.txtを作成
echo "おはよう" > greeting.txt
git add greeting.txt
git commit -m "initial greeting"

手順2: featureブランチで同じ行を編集

# featureブランチを作成
git switch -c feature

# greeting.txtを編集（内容を「Hello」に変更）
echo "Hello" > greeting.txt
git add greeting.txt
git commit -m "change to English"

手順3: mainブランチでも同じ行を編集

# mainブランチに戻る
git switch main

# greeting.txtを編集（内容を「こんにちは」に変更）
echo "こんにちは" > greeting.txt
git add greeting.txt
git commit -m "change to polite Japanese"

手順4: マージを試みる（コンフリクト発生）

# featureブランチをマージ
git merge feature

# コンフリクトが発生し、以下のようなメッセージが表示される
# Auto-merging greeting.txt
# CONFLICT (content): Merge conflict in greeting.txt
# Automatic merge failed; fix conflicts and then commit the result.

コンフリクトの状態を確認する

# コンフリクトが発生しているファイルを確認
git status

# 以下のような出力が表示される
# On branch main
# You have unmerged paths.
#   (fix conflicts and run "git commit")
#   (use "git merge --abort" to abort the merge)
#
# Unmerged paths:
#   (use "git add ..." to mark resolution)
#         both modified:   greeting.txt

both modifiedは、両方のブランチで同じファイルが編集されたことを示しています。

9.6 コンフリクトマーカーの読み方

コンフリクトが発生すると、Gitはファイル内に特殊なマーカーを挿入します。これをコンフリクトマーカーと呼びます。

コンフリクトマーカーの構造

<<<<<<< HEAD
こんにちは
=======
Hello
>>>>>>> feature

マーカーの意味

  • <<<<<<< HEAD - 現在のブランチ（HEAD）の内容の開始
  • ======= - 2つの変更内容の区切り
  • >>>>>>> feature - マージしようとしているブランチの内容の終了

読み方のポイント

<<<<<<<と=======の間が「現在のブランチの変更」、 =======と>>>>>>>の間が「マージしようとしているブランチの変更」です。

9.7 コンフリクトの解決手順

コンフリクトを解決するには、マーカーを手動で編集し、どちらの変更を採用するかを決定します。

解決方法の3つの選択肢

現在のブランチを採用

mainブランチの変更を残す

こんにちは

相手のブランチを採用

featureブランチの変更を残す

Hello

両方を組み合わせる

両方の変更を取り入れる

こんにちは / Hello

実習: コンフリクトを解決する

手順1: ファイルを開いてコンフリクトマーカーを確認

# VS Codeやテキストエディタでgreeting.txtを開く
cat greeting.txt

# 以下のような内容が表示される
# <<<<<<< HEAD
# こんにちは
# =======
# Hello
# >>>>>>> feature

手順2: マーカーを削除し、望む内容に編集

ここでは両方を組み合わせてみましょう。

# greeting.txtを編集して以下の内容に変更
echo "こんにちは / Hello" > greeting.txt

手順3: 変更をステージングしてコミット

# 解決したファイルをステージング
git add greeting.txt

# マージコミットを作成
git commit -m "resolve conflict: combine greetings"

手順4: マージが完了したか確認

# ステータスを確認
git status

# ログを確認
git log --oneline --graph

コンフリクト解決のポイント

  • コンフリクトマーカー（<<<, ===, >>>）は必ず削除する
  • どちらの変更を残すか、慎重に判断する
  • 解決後は必ずgit addでステージングする
  • 複数ファイルでコンフリクトが発生した場合は、全て解決してからコミット

9.8 マージを取り消す方法

コンフリクトが複雑で解決が難しい場合、または間違ってマージを開始してしまった場合は、マージを取り消すことができます。

git merge --abort

マージ中（コンフリクト発生中）の状態から、マージ前の状態に戻すことができます。 

# マージを中止して元の状態に戻す
git merge --abort

# ステータスを確認
git status

注意

git merge --abortは、マージ中（コンフリクト発生中）にのみ使用できます。マージが完了してコミットされた後は使えません。

完了したマージを取り消す

マージが完了してしまった後に取り消したい場合は、git resetを使用します。 

# マージ前の状態に戻す（直前のコミットに戻す）
git reset --hard HEAD~1

# または、特定のコミットIDに戻す
git reset --hard コミットID

警告

git reset --hardは危険なコマンドです。作業ディレクトリの変更も全て失われます。リモートにプッシュ済みの場合は使用しないでください。

よくあるトラブルと対処法

Q: マージ後にファイルが意図した内容になっていない

A: コンフリクトマーカーを削除し忘れている可能性があります。ファイルを開いて、<<<、===、>>>が残っていないか確認してください。

Q: git commitができない（コンフリクト未解決）

A: コンフリクトが解決されていません。git statusで「both modified」と表示されているファイルを全て解決し、git addでステージングしてからコミットしてください。

Q: どちらの変更を残すべきか判断できない

A: チームメンバーに相談するか、git logやgit showでコミット履歴を確認して、変更の意図を理解しましょう。必要であれば、両方の変更を組み合わせる方法も検討してください。

理解度確認クイズ

  1. Q1. マージとは何ですか？簡潔に説明してください。

    回答を表示

    マージとは、2つのブランチの変更履歴を統合する操作です。例えば、新機能を開発したfeatureブランチをmainブランチに統合する際に使用します。

  2. Q2. Fast-forwardマージと3-wayマージの違いを説明してください。

    回答を表示

    Fast-forwardマージ: マージ先のブランチが分岐後に進んでいない場合に発生します。ブランチポインタを前に移動させるだけで、新しいマージコミットは作成されません。

    3-wayマージ: 両方のブランチが分岐後に進んでいる場合に発生します。2つのブランチの変更を統合した新しいマージコミットが作成されます。

  3. Q3. コンフリクトが発生する原因を2つ挙げてください。

    回答を表示
    • 同じファイルの同じ行を異なる内容で編集した場合
    • 一方のブランチでファイルを削除し、もう一方で編集した場合
    • 隣接する行を異なる方法で編集した場合（Gitが自動統合できない場合）

  4. Q4. コンフリクトマーカーの3つの記号とその意味を説明してください。

    回答を表示
    • <<<<<<< HEAD: 現在のブランチ（HEAD）の変更内容の開始
    • =======: 2つの変更内容の区切り
    • >>>>>>> ブランチ名: マージしようとしているブランチの変更内容の終了

  5. Q5. マージ中にコンフリクトが発生した場合、マージを取り消すコマンドは何ですか？

    回答を表示

    git merge --abort - このコマンドでマージ前の状態に戻すことができます。ただし、マージ中（コンフリクト発生中）にのみ使用できます。

第9章のまとめ

  • マージは2つのブランチの変更履歴を統合する操作
  • Fast-forwardマージは新しいコミットを作らず、3-wayマージはマージコミットを作成する
  • コンフリクトは同じファイルの同じ場所を異なる方法で編集した場合に発生する
  • コンフリクトマーカー（<<<, ===, >>>）を手動で編集して解決する
  • マージを取り消すにはgit merge --abortを使用する
