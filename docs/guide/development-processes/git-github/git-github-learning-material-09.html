第9章

プルリクエスト入門 - チーム開発の第一歩

GitHubのプルリクエスト機能を学び、チーム開発の基本を理解します。コードレビューのやり方や、プルリクエストを通じた効果的なコミュニケーションの方法を身につけましょう。

この章で学ぶこと

  • プルリクエスト（Pull Request）とは何か、なぜ使うのか
  • プルリクエストの作成方法と説明の書き方
  • コードレビューの基本とコメント機能の使い方
  • プルリクエストのマージとベストプラクティス

9.1 プルリクエストとは - チーム開発の心臓部

プルリクエスト（Pull Request、略してPR）とは、「自分の変更をプロジェクトに取り込んでほしい」というリクエスト（依頼）のことです。GitHubが提供するチーム開発のための最も重要な機能の一つです。

用語解説: プルリクエストの名前の由来

「プル（Pull）」＋「リクエスト（Request）」という名前は、「自分の変更を取り込んで（Pull）ほしいという依頼（Request）」という意味から来ています。

  • GitLabでは「マージリクエスト（Merge Request）」と呼ばれます
  • どちらも同じ概念を指しています

プルリクエストの基本フロー

flowchart LR A[1. ブランチ作成] --> B[2. 変更をコミット] B --> C[3. GitHubにプッシュ] C --> D[4. プルリクエスト作成] D --> E[5. レビュー] E --> F{6. 承認?} F -->|OK| G[7. マージ] F -->|修正依頼| H[修正してコミット] H --> E G --> I[8. ブランチ削除]

なぜプルリクエストを使うのか？

直接mainブランチに変更をマージすることもできますが、プルリクエストを使うことで多くのメリットがあります。

コードレビュー

他の人に変更内容を確認してもらえます。バグの早期発見や、より良いコードへの改善提案を受けられます。

変更の可視化

何が変更されたかが一覧で見られます。変更の意図や背景も説明できるため、チーム全体で理解を共有できます。

議論の場

コードの特定の行についてコメントでき、議論の履歴が残ります。後から「なぜこうなったか」を追跡できます。

品質の担保

レビューを経ないとマージできない設定にすることで、一定の品質を保ったコードのみが本番に反映されます。

9.2 プルリクエストの作成方法

プルリクエストを作成する手順を、実際の操作に沿って見ていきましょう。

前提条件

プルリクエストを作成する前に、以下の準備が必要です：

  1. 作業用のブランチを作成している
  2. 変更をコミットしている
  3. GitHubにプッシュしている

実習 9-1: プルリクエストを作成する

ブランチの変更をGitHubにプッシュし、プルリクエストを作成してみましょう。

手順

  1. 1 ブランチを作成し、ファイルを変更してコミット
  2. 2 GitHubにプッシュ
  3. 3 GitHubでプルリクエストを作成

コマンド例

Terminal 
# 1. 新しいブランチを作成して切り替え
git checkout -b feature/add-greeting

# 2. ファイルを編集（例: README.mdに挨拶を追加）
echo "Hello, Team!" >> README.md

# 3. 変更をステージングしてコミット
git add README.md
git commit -m "READMEに挨拶を追加"

# 4. GitHubにプッシュ
git push -u origin feature/add-greeting

GitHubでのプルリクエスト作成

プッシュした後、GitHubのリポジトリページを開くと、黄色い通知バナーが表示されます。

プルリクエスト作成画面の要素

flowchart TB subgraph PR[プルリクエスト作成画面] direction TB T[タイトル] --> D[説明 Description] D --> R[レビュアー Reviewers] R --> A[担当者 Assignees] A --> L[ラベル Labels] L --> P[プロジェクト Projects] P --> M[マイルストーン Milestone] end

GitHub Desktopからの作成

GitHub Desktopを使っている場合、プッシュ後に「Create Pull Request」ボタンが表示されます。 クリックするとブラウザでGitHubのプルリクエスト作成画面が開きます。

9.3 良いプルリクエストの説明の書き方

プルリクエストの説明（Description）は、レビュアーが変更内容を理解するための重要な情報源です。良い説明を書くことで、レビューがスムーズに進み、マージまでの時間が短縮されます。

説明に含めるべき項目

何を変更したか（What）

変更の概要を簡潔に説明します。どのファイルを変更したか、どんな機能を追加・修正したかを記載します。

なぜ変更したか（Why）

変更の理由や背景を説明します。関連するIssue番号があれば、「#123」のように記載してリンクします。

どうやって変更したか（How）

技術的なアプローチや実装方法を説明します。特殊な対応をした場合は、その理由も記載します。

テスト方法

変更をどうやって確認したか、レビュアーがどうやって動作確認できるかを記載します。

プルリクエストの説明テンプレート 
## 概要

ユーザー登録フォームにメールアドレスの形式チェックを追加しました。

## 変更理由

- 関連Issue: #42
- 不正なメールアドレスでの登録を防ぐため

## 変更内容

- `validateEmail()` 関数を追加
- フォーム送信前にバリデーションを実行
- エラーメッセージを表示

## テスト方法

1. 登録フォームを開く
2. 不正なメールアドレス（例: "test"）を入力
3. エラーメッセージが表示されることを確認

## スクリーンショット

ヒント

プロジェクトによっては、プルリクエストのテンプレートが用意されています。 .github/PULL_REQUEST_TEMPLATE.md というファイルがあれば、 自動的にテンプレートが読み込まれます。

9.4 レビューとコメント機能

プルリクエストが作成されると、チームメンバーは変更内容を確認（レビュー）し、コメントを残すことができます。

レビューの種類

種類 意味 使う場面
Approve 承認 問題なし、マージOK
Request changes 修正依頼 修正が必要な箇所がある
Comment コメントのみ 質問や提案がある場合

コメントの種類

一般コメント

プルリクエスト全体に対するコメント。「Conversation」タブから投稿します。全体的な感想や質問に使います。

行コメント

特定のコード行に対するコメント。「Files changed」タブで、行番号の「+」ボタンをクリックして投稿します。

レビューの流れ

sequenceDiagram participant A as 作成者 participant R as レビュアー participant GH as GitHub A->>GH: プルリクエスト作成 GH->>R: レビュー依頼通知 R->>GH: コードを確認 R->>GH: コメント/レビュー投稿 GH->>A: コメント通知 A->>GH: 修正をコミット&プッシュ A->>GH: コメントに返信 R->>GH: 再確認 R->>GH: Approve A->>GH: マージ

良いレビューコメントの例

  • 「この変数名をuserEmailにすると意図が明確になると思います」
  • 「このループはforEachよりmapの方が適切かもしれません」
  • 「この実装、分かりやすくて良いですね！」

9.5 レビュー指摘への対応

レビューで修正依頼を受けた場合、どのように対応すればよいでしょうか。プルリクエストは一度作成した後も、追加のコミットをプッシュすることで更新できます。

修正の手順

Terminal 
# 1. プルリクエスト用のブランチに切り替え
git checkout feature/add-greeting

# 2. 指摘された箇所を修正
# （ファイルを編集）

# 3. 修正をコミット
git add .
git commit -m "レビュー指摘対応: 変数名を修正"

# 4. GitHubにプッシュ（同じブランチへ）
git push origin feature/add-greeting

プッシュすると、プルリクエストに新しいコミットが自動的に追加されます。レビュアーは新しいコミットを確認し、再度レビューできます。

注意

修正後は、レビューで指摘されたコメントに対して「対応しました」などと返信しましょう。 コミュニケーションを取ることで、レビュアーが再確認しやすくなります。

Resolveボタンの活用

レビューコメントに対応したら、「Resolve conversation」ボタンをクリックして、 対応済みであることを示すことができます。これにより、未対応のコメントが把握しやすくなります。

9.6 プルリクエストのマージ

レビューが承認されたら、いよいよマージ（統合）です。GitHubでは3種類のマージ方法が選べます。

マージの種類

Create a merge commit

マージコミットを作成して統合します。全てのコミット履歴が残り、いつマージされたかが明確になります。

最も一般的

Squash and merge

複数のコミットを1つにまとめてマージします。細かいコミットが多い場合に履歴をスッキリさせられます。

履歴を整理したい場合

Rebase and merge

コミットをmainブランチの先頭に直線的に配置します。マージコミットが作られず、履歴が直線的になります。

上級者向け

マージ方法の違い

gitGraph commit id: "A" commit id: "B" branch feature commit id: "C" commit id: "D" checkout main merge feature id: "Merge commit" commit id: "E"

初心者へのおすすめ

迷ったら「Create a merge commit」を選びましょう。履歴が全て残り、何が起きたか追跡しやすくなります。 チームで方針が決まっている場合は、その方針に従ってください。

マージ後のブランチ削除

マージが完了すると、「Delete branch」ボタンが表示されます。作業が完了したブランチは削除してリポジトリを整理しましょう。 削除しても、プルリクエストの履歴やコミット履歴は残ります。

9.7 ドラフトプルリクエストの活用

ドラフトプルリクエスト（Draft Pull Request）は、「まだ作業中」であることを示すプルリクエストです。 正式なレビューを依頼する前に、早い段階でフィードバックを得たい場合に便利です。

ドラフトPRの使いどころ

早期フィードバック

実装の方向性が正しいか、完成前に確認したい場合。大きな手戻りを防げます。

作業の可視化

チームに「今この作業をしている」ことを共有したい場合。進捗の見える化に役立ちます。

ドラフトPRの作成方法

プルリクエスト作成時に、「Create pull request」ボタンの横にある矢印をクリックし、 「Create draft pull request」を選択します。

作業が完了したら、「Ready for review」ボタンをクリックして正式なプルリクエストに変更できます。

9.8 プルリクエストのベストプラクティス

効果的なプルリクエストを作成するためのポイントをまとめます。

1

小さく保つ

1つのプルリクエストは1つの目的に絞りましょう。大きなPRはレビューが難しく、バグの見落としが増えます。 目安として、変更行数は200-400行程度が適切です。

2

わかりやすいタイトル

タイトルを見ただけで何のPRかわかるようにしましょう。 「修正」「更新」ではなく、「ログイン画面にパスワード表示切替ボタンを追加」のように具体的に。

3

セルフレビューをする

PRを作成したら、まず自分で「Files changed」を確認しましょう。 デバッグ用のコードや、不要なファイルが含まれていないかチェックします。

4

レビューに感謝する

レビューは時間と労力がかかります。指摘を受けたら、感謝の気持ちを伝えましょう。 建設的なコミュニケーションがチームの雰囲気を良くします。

避けるべきこと

  • 「いろいろ修正」のような曖昧なタイトル
  • 説明なしのプルリクエスト
  • 複数の無関係な変更を1つのPRにまとめる
  • レビュー指摘を無視する

第9章のまとめ

  • プルリクエストは、変更をプロジェクトに取り込む依頼で、チーム開発の中心的な機能
  • 説明文には「何を」「なぜ」「どうやって」変更したかを書く
  • レビューではApprove（承認）、Request changes（修正依頼）、Comment（コメント）が選べる
  • マージ方法は3種類あり、初心者は「Create a merge commit」がおすすめ
  • ドラフトPRは作業中に早期フィードバックを得るのに便利

理解度確認クイズ

  1. Q1. プルリクエストを使うメリットを3つ挙げてください。

    回答を表示

    1. コードレビュー：他の人に変更内容を確認してもらえる
    2. 変更の可視化：何が変更されたか一覧で見られる
    3. 議論の場：コードの特定の行についてコメントでき、議論の履歴が残る
    4. 品質の担保：レビューを経ないとマージできない設定にできる

  2. Q2. レビューの3種類（Approve、Request changes、Comment）はそれぞれどのような場合に使いますか？

    回答を表示

    Approve（承認）：問題がなく、マージしてOKな場合
    Request changes（修正依頼）：修正が必要な箇所がある場合
    Comment（コメント）：承認も却下もせず、質問や提案だけしたい場合

  3. Q3. ドラフトプルリクエストはどのような場合に使うと便利ですか？

    回答を表示

    1. 早期フィードバック：実装の方向性が正しいか、完成前に確認したい場合
    2. 作業の可視化：チームに「今この作業をしている」ことを共有したい場合
    作業が完了したら「Ready for review」で正式なPRに変更できます。
