コンテナ技術入門 - アプリを「箱」に入れて運ぶ

「私のパソコンでは動いたのに、サーバーでは動かない...」という問題を解決するコンテナ技術を、引っ越しの段ボール箱に例えて理解します。

この章で学ぶこと

  • 「私のパソコンでは動いたのに...」問題がなぜ起こるのか
  • コンテナとは何か、従来の方法と何が違うのか
  • Docker（ドッカー）の基本的な仕組みと用語
  • コンテナがDevOpsにもたらすメリット

10.1 「私のパソコンでは動いたのに...」問題

ソフトウェア開発の現場では、こんな会話がよく起こります：

よくある困った場面

開発者: 「新しい機能を作りました！私のパソコンではちゃんと動いています。」
運用担当: 「サーバーにアップロードしたら動きません...エラーが出ています。」
開発者: 「え？私のパソコンでは動くのに...なんでだろう...」

この問題はなぜ起こるのでしょうか？理由はシンプルです： 開発者のパソコンとサーバーの「環境」が違うからです。

環境が違うとは？

コンピュータで「環境」というのは、次のようなものを指します：

  • OS（オペレーティングシステム）: Windows、macOS、Linuxのどれを使っているか
  • ソフトウェアのバージョン: Pythonのバージョンが3.9なのか3.11なのか、など
  • 設定ファイル: データベースの接続先などの設定
  • 依存関係（ライブラリ）: アプリが動くために必要な追加ソフトウェア

開発者のパソコンでは「Python 3.11」が入っているのに、サーバーには「Python 3.8」しか入っていない… こういった小さな違いが、「動かない」原因になります。

環境の違いによる問題

graph LR subgraph 開発者のパソコン D1[OS: Windows] D2[Python 3.11] D3[ライブラリ A v2.0] D4["アプリ ✓ 動く"] end subgraph サーバー S1[OS: Linux] S2[Python 3.8] S3[ライブラリ A v1.5] S4["アプリ ✗ 動かない"] end D4 -.コピー.-> S4

開発環境とサーバー環境の違いが、「動かない」問題を引き起こします。

10.2 コンテナとは何か - 引っ越しの段ボール箱に例えると

この問題を解決するのがコンテナという技術です。 コンテナを理解するために、引っ越しの「段ボール箱」に例えてみましょう。

段ボール箱の例

引っ越しのとき、食器を段ボール箱に入れますよね。 箱の中には、食器だけでなく、緩衝材（プチプチ）や仕切りも一緒に入れます。 こうすることで、どこに運んでも、開けたらすぐに使える状態になります。

コンテナも同じです。アプリケーションを動かすために必要なものを全部「箱」に入れます：

  • アプリケーション本体（食器に相当）
  • 必要なライブラリやソフトウェア（緩衝材や仕切りに相当）
  • 設定ファイル（取扱説明書に相当）

この「箱」を、開発者のパソコンから、テストサーバーへ、本番サーバーへと運べば、 どこでも同じように動くのです。

コンテナとは？

アプリケーションとその実行に必要なすべてをひとまとめにした「箱」です。 この箱は、どのコンピュータに持っていっても、同じように動きます。

コンテナの仕組み

graph TB subgraph コンテナ APP[アプリケーション] LIB[必要なライブラリ] CFG[設定ファイル] OS_PART[軽量OS部分] end subgraph どこでも動く DEV["開発者PC ✓"] TEST["テストサーバー ✓"] PROD["本番サーバー ✓"] end コンテナ --> DEV コンテナ --> TEST コンテナ --> PROD

コンテナは必要なものをすべて含むため、どこに持っていっても同じように動きます。

10.3 従来の方法（仮想マシン）との違い

コンテナが登場する前は、仮想マシン（Virtual Machine, VM）という技術が使われていました。 仮想マシンも「環境をまるごと用意する」技術ですが、コンテナとは違いがあります。

仮想マシンとは？

仮想マシンは、「1台のコンピュータの中に、もう1台のコンピュータを作る」技術です。 例えば、Windows PCの中に、Linuxコンピュータを仮想的に作ることができます。

家に例えると

仮想マシン: 大きな土地に、家を丸ごと複数建てる（それぞれに基礎、壁、屋根がある）
コンテナ: 大きな家の中を、部屋で区切る（基礎や壁は共有、各部屋は独立）

コンテナと仮想マシンの比較

項目 コンテナ 仮想マシン
起動速度 数秒（速い） 数分（遅い）
サイズ 数百MB（軽い） 数GB～数十GB（重い）
OSの扱い ホストOSを共有 完全に独立したOS
リソース効率 高い（無駄が少ない） 低い（オーバーヘッド大）
用途 アプリの軽量実行環境 完全に独立した環境が必要な場合

コンテナは仮想マシンよりも軽量で速いのが特徴です。 そのため、DevOpsでは「素早くアプリを立ち上げる」ことが重要なので、コンテナがよく使われます。

10.4 Docker（ドッカー）とは - コンテナの代表格

Docker（ドッカー）は、コンテナを作って動かすための最も有名なツールです。 コンテナ技術といえばDockerを指すほど、広く使われています。

Dockerとは？

アプリケーションを「コンテナ」という形で作成・実行・管理するためのプラットフォームです。 開発者が作ったアプリを、どんな環境でも同じように動かせるようにするツールです。

Dockerの重要な用語

Dockerを理解するために、3つの重要な用語を覚えましょう：

Dockerイメージ

コンテナの「設計図」です。アプリと必要なものが全部入った「型紙」のようなもの。 この設計図をもとに、コンテナを作ります。

Dockerコンテナ

イメージから作られた「実際に動く環境」です。設計図をもとに組み立てた「完成品」のようなもの。 複数のコンテナを同時に動かせます。

Dockerfile

イメージを作るための「レシピ」です。「Pythonをインストールして、このファイルをコピーして...」という手順を書いたテキストファイルです。

Docker用語の関係

graph LR A[Dockerfile
レシピ] -->|ビルド| B[Dockerイメージ
設計図] B -->|実行| C1[コンテナ1
実行環境] B -->|実行| C2[コンテナ2
実行環境] B -->|実行| C3[コンテナ3
実行環境]

Dockerfileからイメージを作り（ビルド）、イメージから複数のコンテナを起動できます。

10.5 Dockerの基本的な使い方（概要）

Dockerを使う基本的な流れを、簡単に見てみましょう。 実際のコマンドは後の章で学ぶので、ここでは全体の流れを理解してください。

Dockerを使う流れ

1

Dockerfileを書く

どんなアプリをどう動かすかのレシピを書きます

2

イメージをビルドする

Dockerfileをもとに、イメージ（設計図）を作ります

3

コンテナを起動する

イメージからコンテナを作って実行します

4

アプリが動く！

コンテナの中でアプリケーションが動きます

Dockerfileの例

実際のDockerfileがどんなものか、簡単な例を見てみましょう：

Dockerfile 
# ベースとなるイメージを指定（Pythonが入った環境）
FROM python:3.11

# 作業ディレクトリを作成
WORKDIR /app

# 必要なライブラリをインストール
COPY requirements.txt .
RUN pip install -r requirements.txt

# アプリケーションをコピー
COPY . .

# アプリを起動するコマンド
CMD ["python", "app.py"]

このように、「何をインストールして、何をコピーして、どうやって起動するか」を、 1行ずつ書いていきます。このレシピを使えば、誰でも同じ環境を作れます。

10.6 コンテナがDevOpsにもたらすメリット

DevOpsにとって、コンテナ技術は大きな助けになります。 なぜなら、DevOpsの「速く、安全にリリースする」という目標にぴったり合っているからです。

素早い起動

コンテナは数秒で起動するため、テストやデプロイが高速になります。 開発サイクルが速くなり、アイデアを素早く形にできます。

一貫性

開発、テスト、本番で全く同じ環境を使えるため、「私のパソコンでは動いたのに...」問題がなくなります。

簡単な複製

同じコンテナを複数起動することで、簡単にアプリをスケールアウト（台数を増やす）できます。

リソースの効率化

仮想マシンより軽量なため、1台のサーバーでたくさんのコンテナを動かせます。 コストの削減につながります。

10.7 コンテナの実際の使われ方

コンテナは、実際のソフトウェア開発現場でどのように使われているのでしょうか？

1. マイクロサービスアーキテクチャ

大きなアプリを「小さな部品」に分けて、それぞれをコンテナで動かします。 例えば、ECサイトなら「ユーザー管理」「商品管理」「決済処理」を別々のコンテナで動かす、といった具合です。

2. 開発環境の統一

チームメンバー全員が同じDockerイメージを使うことで、 「Aさんのパソコンでは動くけど、Bさんのパソコンでは動かない」という問題がなくなります。

3. CI/CDパイプライン

自動テストや自動デプロイ（第9章で学んだCI/CD）でコンテナを使うことで、 テスト環境と本番環境を完全に同じにできます。

マイクロサービスでのコンテナ利用例

graph TB USER[ユーザー] subgraph コンテナ群 C1[ユーザー管理
コンテナ] C2[商品管理
コンテナ] C3[決済処理
コンテナ] C4[通知
コンテナ] end USER --> C1 USER --> C2 USER --> C3 C3 --> C4

各機能を独立したコンテナで動かすことで、個別にアップデートや拡張ができます。

10.8 コンテナオーケストレーション（次のステップ）

コンテナが数個なら手動で管理できますが、数百個、数千個になると大変です。 そこでコンテナオーケストレーションという技術が登場します。

オーケストレーションとは？

「オーケストラ（orchestra）」のように、複数のコンテナを調和させて管理する技術です。 自動でコンテナを起動したり、停止したり、負荷に応じて数を増減させたりします。

最も有名なオーケストレーションツールはKubernetes（クーベネティス、略してK8s）です。 Kubernetesは、GoogleやAmazon、Microsoftなど、大手IT企業も使っている強力なツールです。

Kubernetesは少し難しいので、まずはDockerの基本を理解することが大切です。 この教材の後に、さらに学習を深めていくときに登場します。

理解度確認クイズ

  1. Q1. 「私のパソコンでは動いたのに、サーバーでは動かない」問題はなぜ起こるのですか？

    回答を表示

    回答: 開発者のパソコンとサーバーで「環境」が違うからです。 OSのバージョン、インストールされているソフトウェア、設定などが異なると、同じコードでも動作が変わることがあります。

  2. Q2. コンテナと仮想マシンの主な違いは何ですか？

    回答を表示

    回答:
    ・コンテナ: 軽量で起動が速い（数秒）。ホストOSを共有する。
    ・仮想マシン: 重く起動が遅い（数分）。完全に独立したOSを持つ。
    コンテナの方がリソース効率が良く、DevOpsでよく使われます。

  3. Q3. Dockerにおける「イメージ」と「コンテナ」の違いを説明してください。

    回答を表示

    回答:
    ・イメージ: コンテナの「設計図」。アプリと必要なものが全部入った雛形。
    ・コンテナ: イメージから作られた「実際に動く環境」。イメージをもとに起動した実体。
    1つのイメージから、複数のコンテナを作ることができます。

  4. Q4. コンテナがDevOpsにもたらす主なメリットを2つ挙げてください。

    回答を表示

    回答例:
    ① 環境の一貫性: 開発、テスト、本番で同じ環境を使えるため、予期しないエラーが減る。
    ② 素早いデプロイ: コンテナは起動が速いため、素早くアプリをリリースできる。
    （他にも、簡単な複製、リソース効率化などが挙げられます）
