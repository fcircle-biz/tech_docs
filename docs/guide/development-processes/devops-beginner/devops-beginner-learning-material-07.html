サーバーとは何か - Webサービスの心臓部

私たちが使うWebサービスやアプリは、どこかにある「サーバー」というコンピュータで動いています。サーバーの役割と種類を学びます。

この章で学ぶこと

  • サーバーとは何か、なぜ24時間365日動き続ける必要があるのか
  • Webサーバー、データベースサーバー、アプリケーションサーバーの役割と違い
  • オンプレミス（自社設置）とクラウド（レンタル）の違い
  • DevOpsエンジニアがサーバーをどのように管理するのか

7.1 サーバーとは何か

YouTubeで動画を見る、LINEでメッセージを送る、Amazonで買い物をする…。これらのサービスは、すべてサーバーというコンピュータで動いています。

サーバーとは？

サーバーとは、インターネット経由で「サービスを提供する」コンピュータのことです。 私たちが使うパソコンやスマートフォンは「クライアント（サービスを受ける側）」、サーバーは「サービスを提供する側」と呼ばれます。

サーバーを銀行に例えると

サーバーを理解するために、銀行の仕組みに例えてみましょう。

  • お客さん（あなた） = クライアント（パソコン/スマホ）
  • 銀行の建物と窓口 = サーバー
  • お金を預ける・引き出す = データの保存・取得
  • 銀行員が対応してくれる = サーバーがリクエストに応答する

銀行は24時間365日、お金を安全に保管し続けます。同じように、サーバーも24時間365日動き続け、ユーザーのデータを保存し、リクエストに応答します。

クライアント・サーバーモデル

flowchart LR subgraph クライアント側 A[パソコン/スマホ] end subgraph サーバー側 B[Webサーバー] C[データベース] end A -->|リクエスト: データをください| B B -->|応答: データを送ります| A B <-->|データ取得| C

クライアント（パソコン/スマホ）がサーバーにリクエストを送り、サーバーがデータベースからデータを取得して応答を返します。

7.2 サーバーの種類

一口に「サーバー」と言っても、実は役割ごとにいくつかの種類があります。 レストランに例えると、「厨房（調理場）」「ホール（接客）」「倉庫（在庫管理）」のように、それぞれ役割が分かれています。

Webサーバー

役割：Webページやファイルを配信する「接客係」のようなサーバーです。

例：ブラウザで「https://example.com」にアクセスしたとき、HTMLファイルや画像を送ってくれるのがWebサーバーです。

代表的なソフトウェア：Apache, Nginx, IIS

アプリケーションサーバー

役割：プログラムを実行する「頭脳」のようなサーバーです。

例：「ログイン処理」「商品検索」「決済処理」など、複雑な処理を実行します。

代表的なソフトウェア：Tomcat, Node.js, Gunicorn

データベースサーバー

役割：データを保存・管理する「倉庫」のようなサーバーです。

例：ユーザー情報、商品情報、注文履歴などを保存し、必要なときに取り出します。

代表的なソフトウェア：MySQL, PostgreSQL, MongoDB

Webサービスにおけるサーバーの連携

パソコン/スマホ]) -->|"HTTPリクエスト"| Web[Webサーバー
Nginx/Apache] Web -->|"HTMLページ配信"| User Web -->|"動的処理依頼"| App[アプリケーションサーバー
Python/Node.js] App -->|"処理結果"| Web App <-->|"データ取得/保存"| DB[(データベースサーバー
MySQL/PostgreSQL)]

実際のWebサービスでは、Webサーバー、アプリケーションサーバー、データベースサーバーが連携して動きます。

7.3 オンプレミスとクラウドの違い

サーバーを用意する方法には、大きく分けて2つの選択肢があります：オンプレミス（自社設置）とクラウド（レンタル）です。

オンプレミス（自社設置）

会社の中にサーバー専用の部屋を作り、物理的なサーバー機器を購入・設置して運用する方法です。 持ち家のようなイメージです。

メリット

  • 自社で完全にコントロールできる
  • 外部ネットワークに依存しない（セキュリティ面で有利な場合も）
  • 長期的に見ると、大規模な場合はコストが安くなることも

デメリット

  • 初期費用が高い（機器購入、専用部屋の設置など）
  • サーバーの保守・管理に専門人材が必要
  • 故障したら自分たちで対応しなければならない
  • スケール（拡張）に時間がかかる

クラウド（レンタル）

AWS、Azure、Google Cloudなどのクラウド事業者から、インターネット経由でサーバーを借りる方法です。 賃貸アパートのようなイメージです。

メリット

  • 初期費用が安い（使った分だけ支払う）
  • 必要に応じてすぐにスケール（拡張）できる
  • ハードウェアの保守はクラウド事業者が担当
  • 世界中のどこからでもアクセスできる

デメリット

  • 長期的に見ると、大規模な場合はコストが高くなることも
  • インターネット接続に依存する
  • クラウド事業者のサービス障害の影響を受ける
  • データの物理的な保存場所が外部になる

オンプレミス vs クラウド 比較表

項目 オンプレミス クラウド
初期費用 高い（数百万円〜） 低い（数千円〜）
運用費用 電気代、人件費など 使った分だけ支払い
スケール 機器購入が必要（時間かかる） 数分で拡張可能
保守管理 自社で対応 クラウド事業者が対応
導入期間 数週間〜数ヶ月 数分〜数時間

7.4 DevOpsエンジニアとサーバー管理

従来のサーバー管理は、人間が手動でサーバーにログインし、コマンドを1つ1つ実行していました。 しかし、DevOpsでは自動化とコード化を重視します。

Infrastructure as Code (IaC)

サーバーの設定を「コード」として記述し、自動的にサーバーを構築・管理する考え方です。 これにより、以下のようなメリットがあります：

  • 再現性: 同じコードを実行すれば、いつでも同じサーバーが作れる
  • バージョン管理: Gitで管理できるので、変更履歴が残る
  • スピード: 手動設定より圧倒的に速い
  • ミス防止: 人間の手作業によるミスが減る

代表的なIaCツール

Terraform: クラウドのインフラをコードで管理
Ansible: サーバーの設定を自動化
CloudFormation: AWS専用のIaCツール
Docker: アプリケーション環境をコンテナで管理

監視と自動復旧

DevOpsエンジニアは、サーバーが正常に動いているかを常に監視し、問題があれば自動的に復旧する仕組みを構築します。

  • 監視ツール: Prometheus, Grafana, CloudWatch など
  • アラート: サーバーの負荷が高くなったら通知
  • 自動スケール: アクセスが増えたら自動的にサーバーを追加
  • 自動復旧: サーバーが停止したら自動的に再起動

DevOpsにおけるサーバー管理フロー

flowchart TD A([コードでインフラ定義
Terraform/Ansible]) --> B[自動でサーバー構築] B --> C[アプリケーションデプロイ] C --> D[監視開始
Prometheus/Grafana] D --> E{異常検知?} E -->|正常| D E -->|異常| F[アラート通知
Slack/メール] F --> G[自動復旧/スケール] G --> D

DevOpsでは、サーバーの構築から監視、復旧まで、すべてを自動化・コード化します。

実習 7-1: クラウドサービスを見てみよう（オプション）

実際のクラウドサービスの管理画面を見てみましょう。無料枠があるサービスを使えば、費用をかけずに試すことができます。

手順

  1. 1 AWS、Azure、Google Cloudのいずれかの無料枠アカウントを作成（クレジットカード登録が必要ですが、無料枠内なら課金されません）
  2. 2 管理画面にログインし、「仮想マシン」のページを見てみる（AWS: EC2、Azure: Virtual Machines、GCP: Compute Engine）
  3. 3 サーバーのスペック（CPU、メモリ、ストレージ）や料金を確認してみる

注意：実際にサーバーを起動すると料金が発生する場合があるので、今は「見るだけ」にしておきましょう。無料枠の範囲を超えないよう注意してください。

    これら3つが連携して、Webサービスを提供します。

    他にも「保守管理の負担」「導入期間」などの違いがあります。

    回答を表示

    回答: Infrastructure as Code (IaC) とは、サーバーの設定を「コード」として記述し、自動的にサーバーを構築・管理する考え方です。 これにより、再現性が高まり、バージョン管理ができ、手作業によるミスが減ります。 代表的なツールに Terraform、Ansible などがあります。

この章のまとめ

  • サーバーとは、24時間365日稼働し続け、インターネット経由でサービスを提供するコンピュータ
  • Webサーバー（配信）、アプリケーションサーバー（処理）、データベースサーバー（保存）が連携して動く
  • オンプレミス（自社設置）は初期費用が高いが完全にコントロールでき、クラウド（レンタル）は初期費用が安く柔軟
  • DevOpsでは Infrastructure as Code (IaC) で、サーバーの構築・管理を自動化・コード化する
  • 監視ツールを使ってサーバーの状態を常に確認し、問題があれば自動的に復旧する仕組みを構築する
