ファイルのバージョン管理 - 「保存」の進化版

Wordで文書を作るときのように、ファイルがどんどん増えていく経験はありませんか？Gitを使えば1つのファイルで全ての変更履歴を管理できます。

この章で学ぶこと

  • ファイルを「上書き保存」するだけでは不便な理由
  • バージョン管理システム（Git）の基本的な考え方
  • Gitの「コミット」「ブランチ」という概念
  • GitとGitHubの違い、なぜDevOpsで重要なのか

5.1 ファイル管理のよくある悩み

Wordで企画書を作っているとき、こんな経験はありませんか？

  • 「報告書_v1.docx」「報告書_v2.docx」「報告書_最終版.docx」「報告書_最終版_修正.docx」…とどんどんファイルが増える
  • 「昨日までの内容に戻したい」と思っても、どのファイルが該当するか分からない
  • 複数人で編集したとき、誰がどこを変更したのか分からず、上書きしてしまう
  • 間違って最新版を削除してしまい、元に戻せない

このような問題は、バージョン管理システムを使えば解決できます。 特にエンジニアの世界では、Git（ギット）というツールが、事実上の標準として使われています。

従来は手動でファイルをコピーして管理していましたが、Gitは自動で変更履歴を記録し、1つのファイルで全てのバージョンを管理できます。

5.2 バージョン管理システムとは

バージョン管理システム（Version Control System, VCS）とは、 ファイルの変更履歴を自動で記録し、管理してくれるツールです。

バージョン管理システムでできること

  • ファイルの変更履歴を自動で記録する
  • 誰が、いつ、どこを変更したかを記録する
  • 過去の任意の時点に簡単に戻せる
  • 複数人が同時に編集しても、変更を統合できる
  • 実験的な変更を「ブランチ」として分離できる

タイムマシンのような仕組み

バージョン管理システムは、「ファイルのタイムマシン」のようなものです。 ゲームのセーブデータのように、好きなタイミングで「この時点を保存」し、 後から「あの時点に戻る」ことができます。

5.3 Gitとは何か

Git（ギット）は、 世界中のエンジニアが使っているバージョン管理システムです。 2005年にリーナス・トーバルズ（Linuxの開発者）によって作られました。

Gitの特徴

  • 分散型: 各人のパソコンに全ての履歴が保存される（サーバーに依存しない）
  • 高速: ほとんどの操作がローカル（自分のPC）で完結するため、非常に高速
  • ブランチが軽い: 実験的な変更を気軽に試せる
  • 無料でオープンソース: 誰でも無料で使える

なぜGitが標準になったのか？

Gitは、他のバージョン管理システム（SVNなど）と比べて、ブランチ機能が非常に強力で、 複数人での並行開発がしやすいため、現在ではほぼすべてのソフトウェア開発で使われています。

5.4 Gitの基本概念

リポジトリ（Repository）

リポジトリは、プロジェクトのファイルと全ての変更履歴が保存される「倉庫」のようなものです。 1つのプロジェクトに1つのリポジトリを作成します。

コミット（Commit）

コミットは、「この時点のファイル状態を保存する」という操作です。 ゲームのセーブポイントのようなもので、後からこの時点に戻ることができます。

コミットには、コミットメッセージという「何を変更したか」を説明する短いメモを付けます。 例: 「ログイン機能を追加」「バグを修正」など。

ブランチ（Branch）

ブランチは、「開発の分岐」を作る機能です。 例えば、既存の機能を保ちながら、新しい機能を実験的に作りたいときに使います。

デフォルトではmain（またはmaster）というブランチが存在し、 そこから新しいブランチを作成して、開発が終わったら元に戻す（マージする）ことができます。

Gitのコミット履歴イメージ

Gitでは、コミットが時系列で記録され、ブランチを使って並行して開発を進めることができます。

5.5 GitとGitHubの違い

初心者が混乱しやすいのが、GitとGitHubの違いです。

Git

バージョン管理「ツール」そのもの。あなたのパソコンにインストールして使います。 自分のPC上でファイルの履歴を管理します。

GitHub

Gitで管理しているプロジェクトを、インターネット上で共有できる「サービス」。 複数人で協力したり、コードを公開したりできます。

例えるなら、Git = カメラ（写真を撮る道具）、GitHub = Instagram（写真を共有するサービス）のような関係です。

GitHubの他にも選択肢がある

GitLabやBitbucketなど、Gitのリポジトリをホスティングするサービスは他にもあります。 ただし、GitHubが最も広く使われています。

5.6 Gitの基本操作の流れ

Gitを使った開発の基本的な流れは、以下のようになります。

Git操作の基本フロー

flowchart LR A[ファイルを編集] --> B[変更をステージング
git add] B --> C[コミット
git commit] C --> D[リモートへプッシュ
git push] D -.次の作業.-> A

基本的な流れは「編集 → ステージング → コミット → プッシュ」の繰り返しです。

1

ファイルを編集

普段通り、ファイルを作成・編集します。

2

ステージング（git add）

コミットしたいファイルを「ステージングエリア」に追加します。「次のコミットに含めるファイルを選ぶ」イメージです。

3

コミット（git commit）

ステージングした変更を、履歴として保存します。コミットメッセージで「何を変更したか」を記録します。

4

プッシュ（git push）

ローカル（自分のPC）のコミットを、リモート（GitHubなど）にアップロードします。これで他のメンバーと共有できます。

基本的なGitコマンド 
# リポジトリを初期化（最初の1回だけ）
git init

# 変更をステージングエリアに追加
git add ファイル名

# コミット（履歴として保存）
git commit -m "変更内容の説明"

# リモートリポジトリにプッシュ
git push origin main

注意

この章ではコマンドを紹介していますが、実際に手を動かす詳しい説明は第6章以降で行います。 ここでは「こういう流れなんだ」と全体像を理解することが目的です。

5.7 DevOpsでGitが重要な理由

DevOpsでは、開発と運用が協力して、素早くソフトウェアをリリースすることが求められます。 その中心にあるのがGitです。

自動化の起点

GitHubにコードをプッシュすると、自動でテストを実行したり、自動でサーバーにデプロイ（公開）したりする仕組みが作れます。 これをCI/CD（継続的インテグレーション/継続的デリバリー）と呼びます（詳しくは後の章で説明します）。

チームでの協力

複数人で開発する際、Gitがあれば誰がどこを変更したかが明確になり、変更の衝突を自動で検出してくれます。 開発チームと運用チームが同じリポジトリを見ることで、コミュニケーションもスムーズになります。

安全性と信頼性

もし本番環境に問題が起きた場合、すぐに前のバージョンに戻すことができます。 これをロールバックと呼びます。Gitがあれば、安心して新しい機能をリリースできます。
