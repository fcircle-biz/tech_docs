テスト〜ちゃんと動くか確認する〜

作ったプログラムが正しく動くかをチェックする「テスト」を学びます。単体テスト、結合テスト、総合テストという3つの段階があることを説明します。

この章で学ぶこと

  • なぜテストが必要なのかを理解する
  • 単体テスト、結合テスト、総合テストの違いを知る
  • テストで見つけるべきバグの種類を理解する

7.1 テストとは？なぜ必要？

プログラムを書き終えたら、それで開発が終わりというわけではありません。テストという工程で、「ちゃんと動くか」「正しく動くか」を確認する必要があります。

これは、車を組み立てた後に試運転や検査を行うのと同じです。車が完成しても、走ってみないと「ブレーキが効くか」「ハンドルがスムーズか」は分かりませんよね。

テストの目的

  • バグを見つける: プログラムの間違いや不具合を発見する
  • 品質を保証する: お客様に安心して使ってもらえるようにする
  • 仕様通りか確認: 要件定義や設計書通りに動いているかチェックする
  • リリース前の最終確認: 本番稼働前に問題がないか確かめる

もしテストをしないで本番稼働させてしまうと、お客様がシステムを使っているときにエラーが起きて、大きな問題になる可能性があります。

7.2 テストの3つの段階

ウォーターフォール開発では、テストを3つの段階に分けて実施します。小さい部分から順番に確認していき、最後に全体をチェックします。

7.3 単体テスト（ユニットテスト）

単体テストは、プログラムの一つひとつの部品（関数やメソッド）が正しく動くかをチェックするテストです。

家を建てる例で言えば、「窓がちゃんと開閉するか」「ドアの鍵がかかるか」を個別に確認するようなものです。

単体テストのポイント

  • 担当者: プログラマー自身が実施することが多い
  • 対象: 個別の関数やメソッド
  • 確認内容: 計算結果が正しいか、エラー処理が動くか、など
  • ツール: テスト用のプログラム（テストコード）を使うこともある

単体テストの例

「消費税を計算する関数」があるとします。この関数が正しく動くか、以下のようにテストします。

入力: 商品価格 1000円

期待する出力: 消費税額 100円（税率10%の場合）

結果: 実際に100円が返ってきたら ✓ テスト合格

7.4 結合テスト（インテグレーションテスト）

結合テストは、複数の部品を組み合わせたときに、正しく連携して動くかをチェックするテストです。

家の例では、「窓とカーテンレールがちゃんと組み合わさっているか」「ドアと床の段差が問題ないか」を確認するようなイメージです。

flowchart LR A[画面入力] --> B[データ検証] B --> C[データ保存] C --> D[確認メッセージ表示] style A fill:#e0e7ff style B fill:#ddd6fe style C fill:#c4b5fd style D fill:#a78bfa

結合テストでは、複数の機能がつながって動くかを確認します

結合テストのポイント

  • 担当者: プログラマーやテスト担当者
  • 対象: 複数の機能をつなげた部分
  • 確認内容: データの受け渡しが正しいか、画面遷移が正しいか、など
  • よくあるバグ: 単体では動いても、組み合わせるとエラーになる

結合テストの例

「商品購入」の流れをテストします。

  1. 1. 商品選択画面で商品を選ぶ
  2. 2. カート追加処理で商品がカートに入る
  3. 3. 決済画面で金額が正しく表示される
  4. 4. 決済完了後に確認メールが送られる

すべての工程が正しくつながっていれば ✓ テスト合格

7.5 総合テスト（システムテスト）

総合テストは、システム全体が要件定義書通りに動くかを確認する最終的なテストです。

家の例では、「家全体が設計図通りに完成しているか」「実際に住んでみて問題ないか」を確認する完成検査のイメージです。

総合テストのポイント

  • 担当者: テスト専門チームやお客様
  • 対象: システム全体
  • 確認内容: 要件通りか、性能は十分か、セキュリティは大丈夫か
  • 実施場所: 本番環境に近い環境でテストする

機能テスト

要件定義で決めた機能がすべて動くか確認

性能テスト

たくさんの人が同時に使っても遅くならないか確認

セキュリティテスト

個人情報が漏れないか、不正アクセスに強いか確認

操作性テスト

使いやすいか、分かりやすい画面になっているか確認

7.6 バグとは？よくあるバグの例

テストで見つかるバグとは、プログラムの間違いや不具合のことです。英語で「虫」という意味ですが、コンピューター業界では「プログラムの欠陥」を指します。

よくあるバグの種類

  • 計算ミス: 合計金額が間違っている、税金の計算が違う
  • 表示エラー: 画面に文字化けが起きる、ボタンが押せない
  • データの不整合: 保存したデータが消える、重複して登録される
  • エラーメッセージが出る: 操作中に突然エラーが表示されて先に進めない
  • 動作が遅い: ボタンを押してから反応するまで時間がかかりすぎる

バグを放置すると...

  • お客様がシステムを使えなくなる
  • 会社の信用を失う
  • 修正するのに余計な時間とお金がかかる
  • 場合によっては損害賠償の問題になることも

7.7 なぜ何度もテストするのか？

「単体テスト」「結合テスト」「総合テスト」と何度もテストを行うのには、理由があります。

段階的にテストする理由

  • 早く問題を見つけられる: 単体テストで小さなバグを早めに発見できる
  • 原因を特定しやすい: どの部品が悪いのかを絞り込める
  • 修正コストが安い: 小さい段階で直すほうが、後から直すより簡単
  • 品質が安定する: 何度も確認することで、見逃しが減る

テストの比喩: 車の検査

テストを車の検査に例えると、こうなります。

  • 単体テスト: エンジン単体、ブレーキ単体を工場で検査
  • 結合テスト: エンジンとブレーキを組み合わせて動かしてみる
  • 総合テスト: 完成した車を実際の道路で試運転する

すべての段階で問題がなければ、安心してお客様に納車できます。

理解度確認クイズ

  1. Q1. テストの目的を2つ挙げてください。

    回答例を表示

    ①バグを見つけること、②品質を保証すること、③仕様通りか確認すること、④リリース前の最終確認。どれでも正解です。

  2. Q2. 単体テスト、結合テスト、総合テストの違いを簡単に説明してください。

    回答例を表示

    単体テストは個別の部品の動作確認、結合テストは複数の部品の連携確認、総合テストはシステム全体が要件通りかの確認です。

  3. Q3. バグを放置するとどんな問題が起きますか？

    回答例を表示

    お客様がシステムを使えなくなったり、会社の信用を失ったり、修正に余計な時間とお金がかかったりする。場合によっては損害賠償の問題になることもあります。

第7章のまとめ

  • テストは、プログラムが正しく動くかを確認する重要な工程
  • テストは単体テスト→結合テスト→総合テストの3段階で実施する
  • バグは早期に発見・修正することが重要
  • 段階的にテストすることで、品質を高く保ち、問題を早く見つけられる
  • 総合テストでは、機能・性能・セキュリティ・操作性などを幅広く確認する
