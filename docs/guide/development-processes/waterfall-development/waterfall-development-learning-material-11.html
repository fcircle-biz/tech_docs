ドキュメント（文書）の重要性

ウォーターフォール開発では「ドキュメント」（設計書や報告書などの文書）がとても重要です。なぜ文書を残すのか、どんな文書があるのか、文書があることで何が助かるのかを学びます。

この章で学ぶこと

  • ドキュメント（文書）がなぜ必要なのかを理解する
  • ウォーターフォール開発でよく使われる文書の種類を知る
  • 文書があることで得られるメリットを理解する
  • 良いドキュメントの書き方のポイントを学ぶ

11.1 ドキュメントとは何か？

「ドキュメント」とは、プロジェクトで作成する様々な文書のことです。 設計書、報告書、マニュアル、議事録など、プロジェクトに関わる情報を文字や図で記録したものを指します。

ウォーターフォール開発では、各フェーズで必ずドキュメントを作成します。 なぜなら、ドキュメントが「次のフェーズへの引継ぎ資料」になるからです。

身近な例: レシピ本

料理のレシピ本は「料理のドキュメント」です。材料、手順、注意点が書かれているので、初めて作る人でも同じ料理が作れます。システム開発のドキュメントも同じで、「このシステムはどう作られたか」「どう動くか」が書かれているので、誰でも理解できるのです。

11.2 なぜドキュメントが重要なのか？

システム開発でドキュメントが重要な理由は、以下の5つです。

1. 引継ぎがスムーズになる

ドキュメントがあれば、担当者が変わっても前の人がやった作業の続きができます。口頭での説明だけだと、聞き間違いや忘れてしまう危険があります。

2. チーム全員が同じ理解を持てる

ドキュメントを読むことで、「今何を作っているのか」「どうやって作るのか」をチーム全員が同じように理解できます。誤解やすれ違いを防げます。

3. 後から確認できる

「あれ、なんでこの仕様にしたんだっけ？」と疑問に思ったとき、ドキュメントを読めば当時の判断理由が分かります。記憶に頼らず、正確な情報が得られます。

4. 保守・改修がしやすくなる

システムは完成後も修正や機能追加が必要です。ドキュメントがあれば、後からシステムを改修する人が「どこを変えればいいか」を理解しやすくなります。

5. 証拠になる（トラブル対応）

もしトラブルが起きたとき、「このときこう決めました」という記録があると、原因究明や責任の所在が明確になります。お客様との合意の証拠にもなります。

11.3 ウォーターフォール開発で使われる主なドキュメント

ウォーターフォール開発では、各フェーズで以下のようなドキュメントが作成されます。

フェーズごとのドキュメント一覧

フェーズ ドキュメント名 内容
要件定義 要件定義書 何を作るのか、どんな機能が必要かを記載
基本設計 基本設計書 画面レイアウト、データ構造などの全体設計
詳細設計 詳細設計書 プログラムの具体的な処理の流れ

全フェーズ共通 議事録

重要ポイント

すべてのドキュメントは、「後で読む人が理解できる」ことを最優先に書きます。自分だけが分かる書き方ではなく、他の人が読んでも分かるように書くことが大切です。

11.4 良いドキュメントの書き方

ドキュメントは「書けばいい」というものではありません。読む人にとって役立つ、良いドキュメントを作ることが重要です。

1. 分かりやすい言葉を使う

専門用語ばかりだと、初心者は理解できません。なるべく簡単な言葉で書き、必要な専門用語には説明を付けましょう。

2. 図や表を使う

文章だけだと分かりにくいことも、図や表を使うとすぐに理解できます。フローチャートや画面イメージを入れると効果的です。

3. 具体的に書く

「いい感じにする」ではなく、「ボタンを画面右下に配置し、青色にする」のように、誰が読んでも同じように理解できる書き方をします。

4. 最新の情報を保つ

仕様が変わったら、必ずドキュメントも更新しましょう。古い情報が残っていると、誤解やトラブルの原因になります。

5. 構造を整理する

目次を付けたり、章立てを明確にしたりして、読みたい情報がすぐに見つかるようにします。

6. レビューしてもらう

自分で書いた文章は、自分では分かりにくさに気づきません。他の人に読んでもらい、意見をもらいましょう。

ヒント: 「5W1H」を意識する

ドキュメントを書くときは、「誰が（Who）」「何を（What）」「いつ（When）」「どこで（Where）」「なぜ（Why）」「どうやって（How）」を意識すると、抜け漏れが減ります。

11.5 ドキュメントの課題と注意点

ドキュメントは便利ですが、いくつか注意すべき点もあります。

課題1: 作成に時間がかかる

詳細なドキュメントを作るには時間が必要です。急いでいるプロジェクトでは、ドキュメント作成が負担になることがあります。

課題2: 更新が追いつかないことがある

仕様変更が多いと、ドキュメントの更新が追いつかず、古い情報が残ってしまうことがあります。気づかずに古い情報を信じてしまうと、間違った作業をしてしまいます。

課題3: 書きすぎても読まれない

詳しすぎるドキュメントは、逆に読むのが大変になります。「必要な情報だけを分かりやすく」書くバランスが重要です。

対策: ドキュメントの「粒度」を調整する

すべてを詳細に書く必要はありません。プロジェクトの規模や重要度に応じて、「どこまで詳しく書くか」を調整しましょう。小さなプロジェクトなら、簡潔なドキュメントでも十分です。

11.6 アジャイル開発との違い

ウォーターフォール開発では「詳細なドキュメント」を重視しますが、アジャイル開発では「動くソフトウェア」を重視し、ドキュメントは最小限にすることが多いです。

どちらが良い・悪いではなく、プロジェクトの性質に合わせて選ぶことが大切です。 大規模で長期間運用するシステムなら、詳細なドキュメントがあるウォーターフォール開発が向いています。 小規模で変化が多いシステムなら、アジャイル開発の方が効率的なこともあります。

理解度確認クイズ

  1. Q1. ドキュメントが重要な理由を3つ挙げてください。

    回答例を表示

    ①引継ぎがスムーズになる、②チーム全員が同じ理解を持てる、③後から確認できる、④保守・改修がしやすくなる、⑤証拠になる（トラブル対応）のうち、どれでも3つ正解です。

  2. Q2. 良いドキュメントの書き方のポイントを2つ挙げてください。

    回答例を表示

    分かりやすい言葉を使う、図や表を使う、具体的に書く、最新の情報を保つ、構造を整理する、レビューしてもらう、のうちどれでも2つ正解です。

  3. Q3. ウォーターフォール開発とアジャイル開発のドキュメントに対する考え方の違いは何ですか？

    回答例を表示

    ウォーターフォール開発は「詳細なドキュメント」を重視するのに対し、アジャイル開発は「動くソフトウェア」を重視し、ドキュメントは最小限にすることが多いです。プロジェクトの性質に応じて使い分けることが大切です。

第11章のまとめ

  • ドキュメントは、プロジェクトで作成する設計書、報告書、マニュアルなどの文書
  • ドキュメントが重要な理由: 引継ぎ、共有理解、後からの確認、保守・改修、証拠
  • 各フェーズで要件定義書、設計書、テスト報告書、マニュアルなどを作成する
  • 良いドキュメントのポイント: 分かりやすく、図表を使い、具体的で、最新を保つ
  • 課題: 作成に時間がかかる、更新が追いつかない、詳しすぎると読まれない
  • プロジェクトに応じて、ドキュメントの詳細度を調整することが重要
