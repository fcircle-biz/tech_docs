ウォーターフォール開発のメリットとデメリット

ここまで学んだウォーターフォール開発の良いところと課題を整理します。どんなプロジェクトに向いているのか、逆にどんな場合は別の方法が良いのかを学びます。

この章で学ぶこと

  • ウォーターフォール開発のメリット（良い点）を理解する
  • ウォーターフォール開発のデメリット（課題）を知る
  • どんなプロジェクトに向いているかを判断できるようになる

9.1 ウォーターフォール開発のメリット

ウォーターフォール開発には、以下のような優れた点があります。

計画が明確

最初にすべての要件を決めるので、「何を作るか」「いつ完成するか」がはっきりしています。スケジュールや予算も立てやすくなります。

ドキュメントが充実

各フェーズで詳細な設計書や仕様書を作成します。担当者が変わっても、ドキュメントがあれば引き継ぎがスムーズです。

品質管理がしやすい

各フェーズごとにチェックポイントがあり、問題があれば次に進めません。段階的に確認するので品質が保たれます。

役割分担が明確

要件定義は営業、設計は設計者、プログラミングはプログラマーと役割がはっきりしているので、誰が何をするか迷いません。

顧客の承認を得やすい

設計段階で顧客に承認をもらうので、「こんなはずじゃなかった」というトラブルを防げます。契約もしやすくなります。

大規模プロジェクト向き

銀行システムや政府のシステムなど、多くの人が関わる大規模プロジェクトでは、しっかりした計画と管理が必要です。

9.2 ウォーターフォール開発のデメリット

一方で、ウォーターフォール開発にはいくつかの課題もあります。

後戻りが難しい

一度次のフェーズに進むと前に戻りにくいです。例えば、実装中に「やっぱり設計を変更したい」となると、大きなコストと時間がかかります。

例: 実装フェーズで「この機能は不要だった」と気づいても、設計書を作り直すのは大変です。

完成までに時間がかかる

すべてのフェーズを順番に進めるので、最初に動くものが見えるまで何ヶ月もかかることがあります。

例: 要件定義に2ヶ月、設計に3ヶ月かけると、プログラミングを始めるまで5ヶ月もかかります。

変更に弱い

プロジェクトの途中で要件が変わると対応しにくいです。「こう作ってほしい」という顧客の要望が変わると、すべて作り直しになることもあります。

例: スマホアプリのような、流行やユーザーの反応を見て変更したいプロジェクトには向きません。

顧客が実物を見るのが遅い

設計書だけでは実際の画面や操作感がイメージしにくいです。完成してから「思っていたのと違う」となることがあります。

例: 設計書では良さそうに見えても、実際に操作してみると使いにくいことがあります。

ドキュメント作成の負担

詳細な設計書を作るのに時間がかかります。小規模なプロジェクトでは、ドキュメント作成だけで疲れてしまうこともあります。

例: 3ページの機能を作るのに、50ページの設計書を書くのは大変です。

9.3 ウォーターフォール開発が向いているプロジェクト

以下のような特徴があるプロジェクトでは、ウォーターフォール開発が適しています。

ウォーターフォールが向いているケース

  • 要件が固まっている: 最初から「何を作るか」がはっきり決まっている
  • 変更が少ない: プロジェクト中に大きな仕様変更が起きにくい
  • 大規模プロジェクト: 多くの人が関わる、予算や期間が長い
  • 厳格な管理が必要: 銀行、医療、政府など規制が厳しい業界
  • 契約がしっかりしている: 最初に契約書で範囲と費用を決める必要がある

具体例

  • 銀行の勘定系システム: 要件が法律で決まっていて、変更が少ない
  • 病院の電子カルテシステム: 医療法の規制があり、厳密な管理が必要
  • 政府の住民基本台帳システム: 大規模で、仕様変更は議会承認が必要

9.4 ウォーターフォール開発が向いていないプロジェクト

逆に、以下のようなプロジェクトではウォーターフォールは適していません。

ウォーターフォールが向いていないケース

  • 要件が曖昧: 作り始めてから「何を作るか」を考える
  • 変更が多い: ユーザーの反応を見て頻繁に改善したい
  • 小規模プロジェクト: 少人数で、すぐに完成させたい
  • 新しい技術を試す: 実験的な要素が多く、試行錯誤が必要
  • スピード重視: とにかく早く市場に出したい

具体例（アジャイル開発の方が適している）

  • スタートアップのWebサービス: ユーザーの反応を見て頻繁に改善
  • スマホアプリ: 短期間で市場に出して、レビューを見て改良
  • AIを使った新サービス: 技術が新しく、試行錯誤が必要

9.5 メリット・デメリットの整理

ここまでの内容を表でまとめます。

項目 メリット デメリット
計画性 スケジュールや予算が明確 計画変更が難しい
ドキュメント 詳細で引き継ぎしやすい 作成に時間がかかる
品質 段階的チェックで品質安定 問題発見が遅れることも
変更対応 最初に承認を得るので安心 途中の変更に弱い
スピード 大規模でも管理しやすい 完成まで時間がかかる
顧客確認 設計段階で承認 実物を見るのが遅い

理解度確認クイズ

  1. Q1. ウォーターフォール開発のメリットを3つ挙げてください。

    回答例を表示

    ①計画が明確（スケジュールや予算が立てやすい）、②ドキュメントが充実（引き継ぎがスムーズ）、③品質管理がしやすい（段階的にチェック）など。他にも「役割分担が明確」「顧客の承認を得やすい」などもあります。

  2. Q2. ウォーターフォール開発の最大のデメリットは何ですか？

    回答例を表示

    「後戻りが難しい」「変更に弱い」ことです。一度次のフェーズに進むと前に戻りにくく、途中で要件が変わると大きなコストがかかります。

  3. Q3. 以下のプロジェクトのうち、ウォーターフォール開発が向いているのはどれですか？
    A: 新しいスマホゲームアプリ
    B: 銀行の勘定系システム
    C: スタートアップのWebサービス

    回答例を表示

    正解はB: 銀行の勘定系システムです。銀行システムは要件が法律で決まっており、変更が少なく、厳格な管理が必要なため、ウォーターフォール開発が適しています。AとCはユーザーの反応を見て頻繁に改善する必要があるため、アジャイル開発の方が向いています。

第9章のまとめ

  • ウォーターフォール開発のメリットは、計画が明確、ドキュメント充実、品質管理がしやすいこと
  • デメリットは、後戻りが難しい、完成まで時間がかかる、変更に弱いこと
  • 要件が固まっていて変更が少ない大規模プロジェクトに向いている
  • 要件が曖昧で変更が多い小規模プロジェクトには向いていない
  • 銀行、医療、政府など厳格な管理が必要な業界で多く使われている
