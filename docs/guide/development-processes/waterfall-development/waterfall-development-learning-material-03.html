要件定義〜「何を作るか」を決める〜

システム開発の最初のステップ「要件定義」を学びます。お客様の「こんなものが欲しい」という希望を聞き取り、「何を作るのか」を明確にする作業です。

この章で学ぶこと

  • 要件定義とは何か、なぜ重要なのかを理解する
  • お客様の要望を聞き取り、整理する方法を学ぶ
  • 要件定義書に含まれる主な内容を知る
  • 要件定義がうまくいかないとどうなるかを学ぶ

3.1 要件定義とは？

要件定義（ようけんていぎ）とは、「何を作るのか」を明確に決める作業です。 ウォーターフォール開発の最初のステップであり、最も重要なフェーズの一つです。

ケーキを注文する場面を想像してください。ただ「ケーキを作ってください」と伝えるだけでは、お店の人は困ってしまいます。

ケーキの注文で考える要件定義

  • 何のために？ 誕生日のお祝い用
  • 何人分？ 6人分
  • どんな味？ チョコレート味
  • 形は？ 丸い形で、直径18cm
  • デコレーションは？ 「Happy Birthday」のメッセージと、ろうそく付き
  • いつまでに？ 明後日の午後3時まで
  • 予算は？ 5,000円まで

これと同じように、システム開発でも「何を作るのか」を具体的に決める必要があります。 これが要件定義です。

3.2 なぜ要件定義が重要なのか？

要件定義がしっかりできていないと、以下のような問題が起きます。

要件定義がうまくいかないと...

  • 完成してから「こんなはずじゃなかった」 となる
  • 何度も作り直しになり、時間とお金が無駄になる
  • お客様と開発チームの認識がずれる
  • プロジェクトが途中で失敗するリスクが高まる

逆に、要件定義をしっかり行うと、以下のメリットがあります。

要件定義をしっかり行うメリット

  • 全員が同じゴールを目指せる - お客様と開発チームの認識が一致する
  • 手戻りが少なくなる - 作り直しが減り、時間とコストを節約できる
  • スケジュールと予算を立てやすい - 何を作るかが明確なので計画が立てられる
  • 品質が安定する - 明確な基準があるのでブレない

3.3 要件定義の進め方

要件定義は、以下のようなステップで進めます。

flowchart LR A[1. ヒアリング] --> B[2. 整理・分析] B --> C[3. 要件の文書化] C --> D[4. 合意・承認] style A fill:#ddd6fe style B fill:#c4b5fd style C fill:#a78bfa,color:#fff style D fill:#8b5cf6,color:#fff
1

ヒアリング（聞き取り）

お客様に直接会って、要望や困っていることを聞き取ります。
「どんなシステムが欲しいですか？」「今何に困っていますか？」などを質問します。

例: 図書館システムの場合

  • 本の貸出・返却を自動化したい
  • 利用者が自分で本を検索できるようにしたい
  • 返却期限を自動で通知してほしい
2

整理・分析

聞いた内容を整理し、「本当に必要なこと」と「あったら良いこと」を分けます。
また、矛盾がないかチェックします。

整理のポイント

  • 必須機能: 貸出・返却・検索
  • あると便利: おすすめ本の表示
  • 優先順位をつける
3

要件の文書化

整理した内容を文書（要件定義書）にまとめます。
誰が読んでも同じ理解ができるように、具体的に書きます。

要件定義書の例

  • システムの目的・背景
  • 機能一覧（何ができるか）
  • 画面イメージ
  • データの種類
  • 利用者の想定人数
4

合意・承認

作成した要件定義書をお客様に見せて、「これで間違いないか」を確認します。
お客様が「OK」と言ったら、要件定義が完了です。

合意のポイント

  • お客様と一緒に内容を確認する
  • 疑問点はその場で解消する
  • 署名やサインをもらって正式に承認

3.4 要件定義書に含まれる主な内容

要件定義書には、以下のような内容が含まれます。

mindmap root((要件定義書)) 機能要件 どんな機能が必要か ユーザーが何をできるか データの入力と出力 非機能要件 性能: 応答速度 セキュリティ 使いやすさ 可用性: 稼働時間 システム構成 ハードウェア ソフトウェア ネットワーク 制約条件 予算 スケジュール 技術的制約

機能要件

「システムが何をするか」を定義します。

  • ・貸出機能
  • ・返却機能
  • ・検索機能
  • ・会員登録機能

非機能要件

「どれくらいの性能・品質か」を定義します。

  • ・応答速度: 3秒以内
  • ・同時利用者: 100人まで
  • ・稼働時間: 24時間365日
  • ・セキュリティ対策

3.5 要件定義のコツ

良い要件定義を行うためのポイントを紹介します。

要件定義を成功させるコツ

  • お客様の本当の困りごとを理解する: 「何が欲しいか」だけでなく「なぜ欲しいか」を聞く
  • 具体的に書く: 曖昧な表現（「速い」「使いやすい」）は避け、数字で表現する
  • 優先順位をつける: すべてを一度に実現するのは難しいので、優先順位を決める
  • 何度も確認する: お客様と繰り返し確認しながら進める
  • できないことも伝える: 技術的に実現不可能なことは正直に伝える

よくある失敗例

  • お客様の話を鵜呑みにする: 言われたことをそのまま書くのではなく、本質を理解する
  • 専門用語を使いすぎる: お客様が理解できる言葉で書く
  • 曖昧な表現を使う: 「なるべく」「できるだけ」などの表現は避ける
  • 変更を嫌がる: 要件定義中に変更が出ることは自然。柔軟に対応する

簡単な実習: 要件を考えてみよう

以下のシナリオを読んで、要件定義で確認すべき項目を考えてみましょう。

シナリオ:

あなたは、小さなパン屋さんからシステム開発を依頼されました。
店主は「インターネットでパンを注文できるシステムが欲しい」と言っています。

考えるべきポイントを表示
  • 1. どんな機能が必要？
    ・商品一覧表示、カートに追加、注文確定、支払い方法、配達か店頭受取か、など
  • 2. 誰が使う？
    ・お客様、店主（管理者）
  • 3. どんな情報を扱う？
    ・パンの種類、価格、在庫数、お客様情報、注文履歴、など
  • 4. どれくらいの規模？
    ・1日何件の注文を想定？同時アクセス数は？
  • 5. いつまでに必要？
    ・納期、予算はいくらか？

理解度確認クイズ

  1. Q1. 要件定義とは何ですか？自分の言葉で説明してみましょう。

    回答例を表示

    「何を作るのか」を明確に決める作業。お客様の要望を聞き取り、整理して、文書にまとめ、合意を得るプロセスです。

  2. Q2. 機能要件と非機能要件の違いは何ですか？

    回答例を表示

    機能要件: システムが「何をするか」（例: 貸出機能、検索機能）
    非機能要件: システムの「性能や品質」（例: 応答速度、同時利用者数、セキュリティ）

  3. Q3. 要件定義がうまくいかないと、どんな問題が起きますか？

    回答例を表示

    完成してから「こんなはずじゃなかった」となったり、何度も作り直しになったり、お客様と開発チームの認識がずれたり、プロジェクトが失敗したりする可能性が高まります。

第3章のまとめ

  • 要件定義とは、「何を作るのか」を明確に決める作業で、ウォーターフォール開発の最初のステップ
  • 要件定義がしっかりできていないと、手戻りやトラブルが発生しやすい
  • 要件定義は「ヒアリング」→「整理・分析」→「文書化」→「合意・承認」の順で進める
  • 機能要件（何をするか）と非機能要件（性能や品質）を明確にする
  • 曖昧な表現を避け、具体的・数値的に書くことが大切
