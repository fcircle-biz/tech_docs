ウォーターフォール開発の実践例〜架空のプロジェクトで体験〜

学んだ知識を使って、架空の「図書館の本の貸出システム」を開発するプロジェクトをシミュレーションします。各フェーズでどんな作業をするのか、具体的な例を通じて理解を深めます。

この章で学ぶこと

  • 実際のプロジェクトでウォーターフォール開発の各フェーズをシミュレーション
  • 要件定義から運用まで、一連の流れを具体例で理解
  • 各フェーズで何を作成し、どんな成果物が残るのかを体験

12.1 プロジェクト概要：図書館貸出システム

これから、架空のプロジェクトを通じてウォーターフォール開発の全体像を体験します。

プロジェクト設定

  • 依頼主: ○○市立図書館
  • 目的: 紙の貸出カードを廃止し、コンピュータで貸出・返却を管理したい
  • 課題: 現在は手書きの台帳で管理しており、貸出中の本が分からなくなることがある
  • 予算: 500万円
  • 納期: 6ヶ月後
  • チーム: プロジェクトマネージャー1名、システムエンジニア2名、プログラマー3名

このプロジェクトを、ウォーターフォール開発の手順に沿って進めていきましょう。

12.2 フェーズ1：要件定義（1ヶ月目）

まず、図書館の担当者からヒアリングを行い、「何を作るか」を決めます。

ヒアリング内容

図書館職員さんのコメント：

「今は本を貸し出すとき、手書きの台帳に利用者名と本のタイトルを書いています。返却されたら線を引いて消すんですが、時々書き忘れたり、字が読めなかったりして困っています。また、どの本が今借りられているかをすぐに調べられないので、利用者から問い合わせがあったときに時間がかかってしまいます。」

要件定義書の内容（抜粋）

  • 機能1: 利用者情報の登録・検索ができる
  • 機能2: 本の情報（タイトル、著者、ISBNなど）を登録・検索できる
  • 機能3: 貸出処理ができる（本のバーコードをスキャン）
  • 機能4: 返却処理ができる
  • 機能5: 貸出中の本を一覧表示できる
  • 機能6: 延滞している本を表示できる
  • 制約: バーコードリーダーは既存のものを使用
  • 制約: パソコンはWindows PCを使用

要件定義が完了したら、図書館の担当者に内容を確認してもらい、承認をもらいます。この承認がないと次の工程に進めません。

12.4 フェーズ3：詳細設計（3ヶ月目）

基本設計を、プログラマーが実装できるレベルまで具体化します。

処理フロー例：貸出処理

flowchart TD A["開始"] --> B{"利用者ID入力済？"} B -->|No| C["エラー：IDを入力"] C --> Z["終了"] B -->|Yes| D{"利用者DB存在？"} D -->|No| E["エラー：利用者なし"] E --> Z D -->|Yes| F["利用者情報表示"] F --> G{"バーコードスキャン？"} G -->|No| G G -->|Yes| H{"貸出可能？"} H -->|No| I["エラー：貸出中"] I --> Z H -->|Yes| J["貸出テーブル登録"] J --> K["日付設定（14日後）"] K --> L["完了メッセージ"] L --> Z

詳細設計書に含まれる内容

  • 各画面の入力チェック仕様（必須項目、文字数制限など）
  • データベースのSQL文の例
  • エラーメッセージの一覧
  • 処理の順序を示すフローチャート

12.5 フェーズ4：実装（4ヶ月目）

詳細設計書に基づいて、プログラマーが実際にコードを書きます。

実装作業の分担例

  • プログラマーA: 利用者登録・検索機能
  • プログラマーB: 書籍登録・検索機能
  • プログラマーC: 貸出・返却機能

実装中は、詳細設計書の通りにコードを書くことが重要です。もし設計書に不明な点があれば、勝手に判断せず、設計担当者に確認します。

実装例（イメージ）

※これは概念を理解するための簡略化した例です

貸出処理のコード（疑似コード） 
# 利用者IDを取得
user_id = 入力欄から取得()

# 利用者が存在するかチェック
if user_id が空:
    エラー表示("利用者IDを入力してください")
    終了

# データベースから利用者情報を取得
user = データベースから検索(user_id)
if user が存在しない:
    エラー表示("利用者が見つかりません")
    終了

# 貸出処理
book_id = バーコードリーダーから取得()
if 本が既に貸出中:
    エラー表示("既に貸出中です")
    終了

# 貸出情報を登録
貸出日 = 今日の日付
返却予定日 = 今日の日付 + 14日
データベースに登録(user_id, book_id, 貸出日, 返却予定日)

完了メッセージ表示("貸出が完了しました")

12.6 フェーズ5：テスト（5ヶ月目）

作成したシステムが正しく動作するか、何度もチェックします。

単体テスト

各機能が個別に正しく動くか確認

結合テスト

機能を組み合わせて動かして確認

総合テスト

実際の運用を想定して確認

テスト項目例

テスト項目 期待される結果 結果
利用者IDが空欄のまま貸出ボタンを押す エラーメッセージが表示される ✓ OK
存在しない利用者IDを入力する 「利用者が見つかりません」と表示される ✓ OK
既に貸出中の本を貸し出そうとする 「既に貸出中です」と表示される ✓ OK
正常に貸出処理を行う 貸出が登録され、完了メッセージが表示される ✓ OK

テストで見つかったバグ（不具合）は、すべて修正してから次の工程に進みます。

12.7 フェーズ6：リリースと運用（6ヶ月目〜）

すべてのテストが完了したら、図書館でシステムを本番稼働させます。

リリース作業

  • 図書館のパソコンにシステムをインストール
  • バーコードリーダーとの接続テスト
  • 既存の図書データをシステムに登録
  • 図書館職員への操作研修
  • 操作マニュアルの提供

運用・保守

システムが稼働した後も、以下のような対応を続けます。

  • 問い合わせ対応: 図書館職員からの質問に答える
  • バグ修正: 運用中に見つかった不具合を直す
  • 機能追加: 「こんな機能が欲しい」という要望に対応
  • セキュリティ対応: システムを安全に保つための更新

第12章のまとめ

  • ウォーターフォール開発は、要件定義→設計→実装→テスト→リリースの順番で進む
  • 各フェーズで成果物（ドキュメント）を作成し、次の工程に引き継ぐ
