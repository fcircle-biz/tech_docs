第5章: 実装フェーズ - 設計からコードへ

実装フェーズ - 設計をプログラムコードに変換する

5.1 実装フェーズの目的と成果物

実装フェーズでは、詳細設計書に基づいて実際のプログラムコードを作成します。このフェーズの品質が、システムの保守性と拡張性に大きく影響します。

実装フェーズの成果物

  • ソースコード：実装したプログラム
  • 単体テストコード：各モジュールのテスト
  • ビルドスクリプト：コンパイル・デプロイの自動化
  • コードレビュー記録：品質確認の証跡

5.2 コーディング規約の重要性

コーディング規約は、チーム全体でコードの書き方を統一するルールです。可読性と保守性を高めるために不可欠です。

コーディング規約の例（Java）
  • 命名規則：クラス名はPascalCase、メソッド名はcamelCase
  • インデント：スペース4つ（タブ禁止）
  • 行の長さ：最大120文字
  • コメント：public メソッドには必ずJavadoc
  • 例外処理：空のcatchブロック禁止
/**
 * 経費申請サービスクラス
 * 経費申請の登録・承認・却下を管理します
 */
public class ExpenseService {

    private final ExpenseRepository expenseRepository;

    /**
     * 経費申請を新規登録します
     * @param request 経費申請リクエスト
     * @return 登録された経費申請
     * @throws ValidationException 入力値が不正な場合
     */
    public ExpenseResponse createExpense(ExpenseRequest request) {
        // バリデーション
        validateRequest(request);

        // エンティティ変換
        Expense expense = convertToEntity(request);

        // DB保存
        Expense saved = expenseRepository.save(expense);

        return convertToResponse(saved);
    }
}

5.3 バージョン管理システムの活用

Gitなどのバージョン管理システムを使うことで、コードの変更履歴を追跡し、チームでの協調開発を円滑にします。

Gitの基本ワークフロー

flowchart LR A[ローカルリポジトリ] -->|git add| B[ステージング] B -->|git commit| C[コミット] C -->|git push| D[リモートリポジトリ] D -->|git pull| A

ブランチ戦略

  • main/master：本番環境のコード
  • develop：開発中のコード
  • feature/*：機能開発用ブランチ
  • hotfix/*：緊急バグ修正用

5.4 コードレビューの実践

コードレビューは、コードの品質を保証し、チーム内で知識を共有する重要なプロセスです。

コードレビューのチェックポイント
観点チェック内容
設計との整合性詳細設計書の通りに実装されているか
コーディング規約規約に従っているか
可読性他の開発者が理解できるか
例外処理適切にエラーハンドリングされているか
セキュリティSQLインジェクション等の脆弱性がないか
性能無駄なループや重複処理がないか
テスト単体テストが十分か

5.5 単体テストの実装

実装と並行して単体テストを作成します。JUnitなどのテストフレームワークを使用します。

JUnitテストの例
@Test
public void testCreateExpense_正常系() {
    // 準備
    ExpenseRequest request = new ExpenseRequest();
    request.setEmployeeId(1);
    request.setAmount(5000);
    request.setCategory("交通費");

    // 実行
    ExpenseResponse response = expenseService.createExpense(request);

    // 検証
    assertNotNull(response);
    assertEquals(5000, response.getAmount());
    assertEquals("SUBMITTED", response.getStatus());
}

@Test(expected = ValidationException.class)
public void testCreateExpense_金額が負の場合() {
    ExpenseRequest request = new ExpenseRequest();
    request.setEmployeeId(1);
    request.setAmount(-1000); // 負の金額

    expenseService.createExpense(request);
}

5.6 静的解析ツールの活用

SonarQubeやCheckstyleなどの静的解析ツールで、コードの品質を自動チェックします。

検出できる問題

  • コーディング規約違反
  • 潜在的なバグ（nullチェック漏れ等）
  • セキュリティ脆弱性
  • コードの複雑度（循環的複雑度）
  • 重複コード
理解度確認クイズ
  1. コーディング規約が必要な理由を2つ挙げてください。
  2. Gitでコードをリモートリポジトリに反映する手順を説明してください。
  3. コードレビューの主な目的は何ですか？
  4. 単体テストで確認すべきことを3つ挙げてください。
解答例
  1. (1)コードの可読性を高める、(2)チーム全体で統一された品質を保つ
  2. git add でステージング → git commit でコミット → git push でリモートに反映
  3. コードの品質を保証し、バグを早期発見し、チーム内で知識を共有することです。
  4. 例：(1)正常系の動作、(2)異常系の例外処理、(3)境界値の動作
