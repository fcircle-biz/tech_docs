8.1 プロジェクト管理の基本

ウォーターフォール開発では、プロジェクト全体を計画的に管理することが成功の鍵です。PMBOK（プロジェクトマネジメント知識体系ガイド）に基づく体系的な管理が重要です。

プロジェクト管理の5つのプロセス群

graph LR A[立ち上げ] --> B[計画] B --> C[実行] C --> D[監視・コントロール] D --> C D --> E[終結]

8.2 WBS（Work Breakdown Structure）

WBSは、プロジェクト全体を作業単位に分解し、階層構造で整理したものです。これにより、作業の漏れを防ぎ、工数見積もりを精緻化できます。

工数見積もり手法

  • 類推見積もり：過去の類似プロジェクトから推定
  • ボトムアップ見積もり：WBSの各作業を積み上げ
  • 三点見積もり：楽観値、最頻値、悲観値の3つから算出
  • ファンクションポイント法：機能数から算出

マイルストーン管理

マイルストーンは、プロジェクトの重要な節目（成果物完成、レビュー完了等）を示します。

8.4 進捗管理と報告

定期的に進捗を確認し、計画との差異を早期に検出します。

EVM（Earned Value Management）

EVMは、計画、実績、コストを統合的に管理する手法です。

主要指標
  • PV（Planned Value）：計画時点での予定作業量
  • EV（Earned Value）：実際に完了した作業の価値
  • AC（Actual Cost）：実際にかかったコスト
  • SPI（Schedule Performance Index） = EV / PV（1.0以上で進捗良好）
  • CPI（Cost Performance Index） = EV / AC（1.0以上でコスト良好）

8.6 リソース管理とチーム編成

プロジェクトに必要な人員を適切に配置し、効率的なチーム編成を行います。

典型的なチーム構成

  • プロジェクトマネージャー（PM）：プロジェクト全体の責任者
  • サブリーダー（PL）：技術面のリーダー
  • 設計担当：システム設計
  • 開発担当：プログラミング
  • テスト担当：品質保証
  • インフラ担当：環境構築・運用

8.7 ドキュメント管理と構成管理

プロジェクトで作成する大量のドキュメントを体系的に管理します。

ドキュメント管理のポイント

  • バージョン管理：V1.0、V1.1等で版管理
  • 命名規則：ファイル名に日付やバージョンを含める
  • 格納場所の統一：共有フォルダを整理
  • アクセス権限：必要な人だけがアクセス
  • 変更履歴：誰がいつ何を変更したか記録

8.8 変更管理プロセス

要件や仕様の変更を適切に管理し、影響範囲を最小化します。

理解度確認クイズ

  2. ガントチャートで表現される情報を3つ挙げてください。
  3. EVMにおけるSPIが1.0未満の場合、何を意味しますか？
  4. プロジェクトチームの典型的な役割を5つ挙げてください。
  5. ドキュメント管理で重要なポイントを3つ挙げてください。
解答例
  1. WBS（Work Breakdown Structure）は、プロジェクト全体を作業単位に分解し階層構造で整理したものです。作業の漏れ防止と工数見積もりの精緻化が目的です。
  2. 作業の開始日、終了日、期間（または作業名、依存関係）
  3. 計画より遅れていることを意味します（EV < PV）。
  4. 例：プロジェクトマネージャー、プロジェクトリーダー、設計担当、開発担当、テスト担当
  5. 例：バージョン管理、命名規則の統一、アクセス権限の設定
第8章のまとめ
  • プロジェクト管理は立ち上げ→計画→実行→監視・コントロール→終結のプロセス
  • WBSで作業を分解し、工数を見積もる
  • ガントチャートでスケジュールを視覚化
  • EVMで計画・実績・コストを統合管理
  • 課題は台帳で管理し、確実に解決
  • チーム編成で適材適所の配置
  • ドキュメントはバージョン管理と命名規則で整理
  • 変更は影響分析と承認プロセスを経て実施
