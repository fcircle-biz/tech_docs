3.1 基本設計（外部設計）の目的と位置づけ

基本設計フェーズは、要件定義で明確にした「何を作るか」を「どう実現するか」に変換する重要なフェーズです。別名「外部設計」とも呼ばれ、ユーザーから見える部分（画面、帳票、外部インターフェース等）を中心に設計します。

graph LR A["要件定義
（何を作るか）"] --> B["基本設計
（どう実現するか）"] B --> C["詳細設計
（どう実装するか）"] B1["要件定義書"] --> B B --> B2["基本設計書"] B2 --> C

基本設計と詳細設計の違い

観点 基本設計（外部設計） 詳細設計（内部設計）
視点 ユーザー視点 開発者視点
焦点 「何が見えるか」「どう動くか」 「内部でどう処理するか」
主な成果物 画面設計書、DB論理設計書、システム構成図 クラス設計書、処理ロジック仕様書、DB物理設計書
レビュー参加者 ユーザー、プロジェクトマネージャー、設計者 設計者、開発者、テスト担当者

3.2 システム方式設計とアーキテクチャの決定

システム方式設計では、システム全体のアーキテクチャ（構成）を決定します。これはプロジェクト全体の技術的基盤となる重要な意思決定です。

一般的なシステムアーキテクチャパターン

1. 3層アーキテクチャ（3-Tier Architecture）
flowchart TB subgraph プレゼンテーション層 A[Webブラウザ
モバイルアプリ] end subgraph アプリケーション層 B[Webサーバー
ビジネスロジック] end subgraph データ層 C[データベースサーバー] end A --> B B --> C

特徴：各層の責務が明確で保守性が高い。エンタープライズシステムで広く採用されています。

2. マイクロサービスアーキテクチャ
flowchart LR A[APIゲートウェイ] --> B[ユーザー
サービス] A --> C[経費申請
サービス] A --> D[承認
サービス] A --> E[通知
サービス] B --> B1[(DB)] C --> C1[(DB)] D --> D1[(DB)] E --> E1[(DB)]

特徴：機能ごとに独立したサービスとして実装。スケーラビリティと独立性が高い。

技術スタックの選定

システムアーキテクチャに基づいて、使用する技術スタックを決定します。

技術スタック選定例（経費精算システム）
  • フロントエンド：React + TypeScript（SPA）、スマートフォン対応にはReact Native
  • バックエンド：Java Spring Boot（REST API）
  • データベース：PostgreSQL（メインDB）、Redis（キャッシュ）
  • 認証：OAuth 2.0 + JWT
  • インフラ：AWS（EC2、RDS、S3）
  • CI/CD：GitHub Actions、Docker

3.3 画面設計・帳票設計・インターフェース設計

画面設計の基本

画面設計では、ユーザーが操作する画面の構成、レイアウト、操作フローを定義します。

画面設計書に含めるべき情報
  • 画面ID：画面を一意に識別するID（例：SCR-001）
  • 画面名称：画面の名前（例：「経費申請登録画面」）
  • 画面レイアウト：ワイヤーフレームまたはモックアップ
  • 入力項目：各項目の名称、データ型、必須/任意、桁数、初期値
  • ボタン・リンク：各ボタンの動作と遷移先
  • バリデーションルール：入力チェックの条件とエラーメッセージ
  • 表示条件：権限による表示/非表示の制御
実習 3-1: 画面項目定義

シナリオ：経費申請登録画面の項目を定義してください。

項目名 データ型 桁数 必須 バリデーション
申請日 日付 - 必須 過去日のみ選択可、1年以内
経費種別 選択リスト - 必須 交通費/宿泊費/会議費/その他
金額 数値 最大10桁 必須 1円以上、半角数字のみ
用途 テキスト 最大200文字 必須 -
領収書画像 ファイル 最大5MB 任意 JPG/PNG/PDF形式

3.4 データベース概念設計と論理設計

基本設計フェーズでは、データベースの概念設計と論理設計を行います。物理設計は詳細設計フェーズで実施します。

概念設計（ER図）

エンティティ（実体）とリレーションシップ（関連）を定義します。

erDiagram EMPLOYEE ||--o{ EXPENSE : submits EMPLOYEE ||--o{ APPROVAL : approves EXPENSE ||--|| APPROVAL : has EXPENSE ||--o{ EXPENSE_ITEM : contains EMPLOYEE { int employee_id PK string employee_name string email string department } EXPENSE { int expense_id PK int employee_id FK date application_date decimal total_amount string status } EXPENSE_ITEM { int item_id PK int expense_id FK string category decimal amount string description string receipt_image_url } APPROVAL { int approval_id PK int expense_id FK int approver_id FK date approval_date string result string comment }

論理設計（テーブル定義）

概念設計を基に、具体的なテーブル構造を定義します。

テーブル定義例：経費申請テーブル（EXPENSE）
カラム名 データ型 Null デフォルト 説明
expense_id INT NOT NULL AUTO_INCREMENT 経費申請ID（主キー）
employee_id INT NOT NULL - 申請者社員ID（外部キー）
application_date DATE NOT NULL - 申請日
total_amount DECIMAL(10,0) NOT NULL 0 合計金額
status VARCHAR(20) NOT NULL 'DRAFT' ステータス（DRAFT/SUBMITTED/APPROVED/REJECTED）
created_at TIMESTAMP NOT NULL CURRENT_TIMESTAMP 作成日時
updated_at TIMESTAMP NOT NULL CURRENT_TIMESTAMP 更新日時

3.6 外部システム連携仕様の定義

他のシステムと連携する場合、インターフェース仕様を明確に定義します。

連携方式の種類

  • REST API：HTTP/HTTPSを使用したリアルタイム連携
  • ファイル連携：CSV/XMLファイルを介したバッチ連携
  • メッセージキュー：非同期メッセージング（RabbitMQ、Kafka等）
  • データベース連携：共通DBへの直接アクセス
外部連携仕様例：会計システム連携（REST API）

エンドポイント：POST /api/v1/accounting/expenses

リクエスト形式：JSON

{
  "expense_id": 12345,
  "employee_id": "E001",
  "application_date": "2025-10-20",
  "total_amount": 5000,
  "items": [
    {
      "category": "交通費",
      "amount": 3000,
      "description": "東京→大阪 新幹線"
    },
    {
      "category": "宿泊費",
      "amount": 2000,
      "description": "ビジネスホテル"
    }
  ]
}

レスポンス形式：JSON

{
  "status": "success",
  "accounting_id": "ACC-67890",
  "message": "経費が会計システムに登録されました"
}

エラーハンドリング：HTTPステータスコード 400（入力エラー）、500（サーバーエラー）

3.7 非機能要件の具体化

要件定義で定義した非機能要件を、システムアーキテクチャに落とし込みます。

性能要件の具体化

要件 設計での対応
画面表示3秒以内 ・Redisでキャッシュ導入
・CDNで静的ファイル配信
・DBインデックス最適化
同時100ユーザー対応 ・Webサーバー2台構成（ロードバランサー）
・接続プール最大200

セキュリティ要件の具体化

要件 設計での対応
社員ID・パスワード認証 ・OAuth 2.0 + JWT実装
・パスワードはbcryptでハッシュ化
他人の経費情報保護 ・APIで申請者IDをチェック
・DBで行レベルセキュリティ
理解度確認クイズ
  1. 基本設計（外部設計）と詳細設計（内部設計）の主な違いは何ですか？
  2. 3層アーキテクチャの3つの層を挙げてください。
  3. 画面設計書に含めるべき情報を5つ挙げてください。
  4. データベース設計において、「概念設計」と「論理設計」の違いを説明してください。
  5. シーケンス図は何を表現するための図ですか？
  6. 外部システム連携の方式を3つ挙げてください。
解答例
  1. 基本設計はユーザー視点で「何が見えるか」を設計し、詳細設計は開発者視点で「内部でどう処理するか」を設計します。
  2. プレゼンテーション層、アプリケーション層、データ層の3つです。
  3. 例：(1)画面レイアウト、(2)入力項目定義、(3)ボタン・リンクの動作、(4)バリデーションルール、(5)画面遷移
  4. 概念設計はエンティティとリレーションシップを定義し（ER図）、論理設計は具体的なテーブル構造を定義します。
  5. システムの複数のコンポーネント間の処理フロー、メッセージのやり取りを時系列で表現します。
  6. 例：REST API、ファイル連携、メッセージキュー
第3章のまとめ
  • 基本設計はユーザー視点で「どう実現するか」を設計するフェーズ
  • システムアーキテクチャと技術スタックを決定する
  • 画面設計では項目定義、バリデーション、遷移を明確化
  • データベースは概念設計（ER図）→論理設計（テーブル定義）の順で進める
  • シーケンス図で処理フローを可視化する
  • 外部システム連携はインターフェース仕様を明確に定義
  • 非機能要件を具体的な設計に落とし込む
