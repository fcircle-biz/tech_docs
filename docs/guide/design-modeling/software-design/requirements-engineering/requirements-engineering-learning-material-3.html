第3章: EARS記法による要件記述

シンプルで明確な要件記述のための構造化記法

この章で学ぶこと
  • EARS（Easy Approach to Requirements Syntax）記法の概要と利点
  • 5つの基本パターン（Ubiquitous、Event-driven、State-driven、Optional、Unwanted）
  • 各パターンの構文と適用場面
  • EARS記法を使った明確で検証可能な要件文の作成技法
  • 曖昧な要件文をEARS記法で改善する実践的スキル
  • 複合的な要件をEARS記法で効果的に表現する方法
  • EARS記法による要件品質の向上

3.1 EARS（Easy Approach to Requirements Syntax）記法とは

EARS記法は、ロールス・ロイス社とBAEシステムズ社によって開発された要件記述のための構造化記法です。複雑な要件を明確で理解しやすい形で表現することを目的とし、要件の曖昧性を大幅に削減します。

なぜEARS記法が必要なのか

従来の要件記述の問題点
  • 曖昧性：「適切な速度で」「必要に応じて」など解釈の余地がある表現
  • 条件の不明確性：いつ、どのような条件で動作するかが不明
  • 主語の欠如：誰が何をするのかが不明確
  • 検証困難性：実装結果が要件を満たしているか判断できない

EARS記法の基本原則

  • シンプル性：覚えやすく適用しやすい5つのパターン
  • 一貫性：すべての要件が同じ構造で記述される
  • 明確性：条件、主体、動作が明確に分離される
  • 検証可能性：各要件がテスト可能な形で記述される
  • 追跡可能性：要件の構造により依存関係が明確

3.2 EARS記法の5つの基本パターン

EARS記法では、システムの動作を5つの基本パターンで分類し、それぞれに対応した構文テンプレートを提供します。

パターン1: Ubiquitous（常時動作）

構文テンプレート
システムは [動作] する
適用場面

システムが常に実行すべき基本的な機能や継続的な動作を記述する場合

良い例
  • システムはユーザーのログイン状態を保持する
  • システムは入力データの形式を検証する
  • システムは全ての取引ログを記録する
悪い例（曖昧な記述）

「システムはユーザーフレンドリーである」

問題点：「ユーザーフレンドリー」が主観的で測定不可能

改善例（EARS記法）

「システムは新規ユーザーが基本操作を30分以内に習得できるよう支援機能を提供する」

パターン2: Event-driven（イベント駆動）

構文テンプレート
[イベント・条件] のとき、システムは [動作] する
適用場面

特定のイベントや条件が発生した際にシステムが実行すべき動作を記述する場合

良い例
  • ユーザーがログインボタンをクリックしたとき、システムは認証処理を開始する
  • 注文総額が1万円を超えたとき、システムは送料を無料に設定する
  • パスワード入力を3回間違えたとき、システムはアカウントを30分間ロックする

パターン3: State-driven（状態駆動）

構文テンプレート
[状態] である間、システムは [動作] する
適用場面

システムが特定の状態にある間継続的に実行すべき動作を記述する場合

良い例
  • ユーザーがログイン状態である間、システムはセッションを維持する
  • バッテリー残量が20%以下である間、システムは省電力モードで動作する
  • メンテナンスモードが有効である間、システムは一般ユーザーのアクセスを拒否する

パターン4: Optional（オプション）

構文テンプレート
[条件] の場合、システムは [動作] すべきである（should）
適用場面

必須ではないが、特定の条件下で実装が望ましい機能を記述する場合

良い例
  • ネットワーク接続が利用可能な場合、システムは自動的にデータを同期すべきである
  • ユーザーが管理者権限を持つ場合、システムは詳細ログを表示すべきである
  • 画面サイズが十分大きい場合、システムはサイドメニューを展開すべきである

パターン5: Unwanted（望ましくない動作）

構文テンプレート
[条件] のとき、システムは [動作] してはならない
適用場面

特定の条件下でシステムが実行してはいけない動作を明確に禁止する場合

良い例
  • ユーザーが一般権限である場合、システムは管理画面にアクセスしてはならない
  • データベース接続が失われたとき、システムは重要なデータを削除してはならない
  • セキュリティ警告が発生したとき、システムは自動的にログアウトしてはならない

3.3 実践的な要件記述技法

曖昧な要件文の改善プロセス

既存の曖昧な要件文をEARS記法で明確化するための段階的アプローチを学びます。

graph TD A["曖昧な要件文"] --> B["要件の意図分析"] B --> C["適切なパターン選択"] C --> D["構文テンプレート適用"] D --> E["具体的・測定可能な表現"] E --> F["検証可能な要件文"] B1["誰が？"] --> B B2["いつ？"] --> B B3["何を？"] --> B B4["なぜ？"] --> B style A fill:#ffebee style F fill:#e8f5e8

改善例の詳細分析

改善前（曖昧な要件）

「システムは高速で動作する必要がある」

問題分析
  • 主語不明：システムのどの部分が？
  • 条件不明：いつ、どのような状況で？
  • 基準不明：「高速」とは具体的にどの程度？
  • 検証不可：何をもって「高速」と判断するか？
改善後（EARS記法適用）
  • Ubiquitous：「システムはデータベース検索結果を2秒以内に表示する」
  • Event-driven：「ユーザーが検索ボタンをクリックしたとき、システムは3秒以内に結果を表示する」
  • State-driven：「ピーク時間帯である間、システムは通常の1.5倍以内の応答時間で処理する」

複合要件の記述技法

複雑な業務ルールや複数の条件を含む要件も、EARS記法の組み合わせで明確に表現できます。

複合条件の例

業務ルール：VIP顧客の大口注文は特別割引を適用し、在庫不足の場合は優先手配する

EARS記法での記述：

  • 「顧客がVIP顧客かつ注文金額が10万円以上のとき、システムは10%割引を適用する」
  • 「VIP顧客の注文商品が在庫不足のとき、システムは優先手配フラグを設定する」
  • 「優先手配フラグが設定されている間、システムは出荷予定を他の注文より優先する」

3.4 EARS記法による要件品質の向上

要件品質の評価基準

EARS記法で記述された要件は、以下の品質特性を満たすことができます。

EARS記法による品質向上
品質特性 従来の問題 EARS記法の改善
明確性 解釈の余地がある表現 構造化された一意の表現
完全性 条件や主体の記述漏れ テンプレートによる網羅的記述
一貫性 記述スタイルのばらつき 統一された構文パターン
検証可能性 テスト設計が困難 条件と動作が明確で測定可能
追跡可能性 要件間の関係が不明 構造化により依存関係が明確

要件レビューでのEARS記法活用

EARS記法で記述された要件は、レビューでの品質確認が効率的に行えます。

EARS記法チェックリスト
Ubiquitousパターンのチェック項目
  • ☐ 主語（システム）が明確に記述されているか
  • ☐ 動作が具体的で測定可能か
  • ☐ 常時実行すべき動作として適切か
Event-drivenパターンのチェック項目
  • ☐ トリガーとなるイベントが明確に定義されているか
  • ☐ 条件が具体的で判定可能か
  • ☐ 実行される動作が明確か
共通チェック項目
  • ☐ 曖昧な形容詞・副詞が使用されていないか
  • ☐ 数値基準が適切に設定されているか
  • ☐ 例外条件が考慮されているか
実習 3-2: 曖昧な要件文の改善

以下の曖昧な要件文をEARS記法を使って改善してください。

改善対象の要件文
  1. 「システムはセキュアである必要がある」
  2. 「ユーザーにとって使いやすいインターフェースを提供する」
  3. 「エラーが発生した場合は適切に対処する」
  4. 「パフォーマンスを最適化する」
  5. 「データのバックアップを定期的に行う」
改善のポイント
  • 具体的な数値基準を設定する
  • 条件と動作を明確に分離する
  • 主語と目的語を明確にする
  • 測定可能な表現を使用する
改善例

  3. 「データベース接続エラーが発生したとき、システムはユーザーにエラーメッセージを表示し、自動的に再接続を3回試行する」
  4. 「ページ読み込み要求があったとき、システムは2秒以内にコンテンツを表示する」
