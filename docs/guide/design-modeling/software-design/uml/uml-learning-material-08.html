アクティビティ図 - 業務フローとアルゴリズムを描く

業務プロセスやプログラムの処理手順を表現するアクティビティ図を学びます。フローチャートに似た直感的な記法で、分岐・ループ・並行処理などを表現する方法を習得します。

この章で学ぶこと

  • アクティビティ図の基本要素（アクション、フロー、決定ノード）を理解する
  • 分岐・ループ・並行処理の表現方法を習得する
  • スイムレーンを使った担当者別の業務フローの整理方法を学ぶ

8.1 アクティビティ図とは

アクティビティ図（Activity Diagram）は、業務プロセスやプログラムの処理手順を表現するUML図です。 フローチャートに似た直感的な記法を使い、作業の流れ、分岐、並行処理などを視覚化します。

シーケンス図が「オブジェクト間のやり取り」に注目するのに対し、アクティビティ図は「作業の流れ」そのものに注目します。 特に業務分析や要件定義のフェーズで、「誰が何をするのか」「どのような順番で処理が進むのか」を整理するのに役立ちます。

アクティビティ図の主な用途

業務プロセスの可視化

申請承認フロー、発注処理、顧客対応など業務の流れを整理

アルゴリズムの設計

プログラムのロジックや処理手順を設計段階で可視化

担当者間の作業分担

スイムレーンを使って部門や担当者ごとの役割を明確化

8.2 アクティビティ図の基本要素

アクティビティ図を構成する基本要素を見ていきましょう。

開始ノード（Initial Node）

表記：黒い塗りつぶし円 ●
意味：フローの開始地点を表します。1つのアクティビティ図に1つだけ配置します。

アクション（Action）

表記：角丸長方形 ┌─────┐
意味：実行される作業や処理を表します。動詞で記述します（例：「注文を受け付ける」「在庫を確認する」）。

制御フロー（Control Flow）

表記：矢印 →
意味：アクション間の実行順序を表します。矢印の向きに処理が進みます。

終了ノード（Final Node）

表記：二重円 ◎
意味：フローの終了地点を表します。複数の終了ノードを持つこともできます。

基本的なアクティビティ図の例

flowchart TD Start([開始]) --> A1[注文を受け付ける] A1 --> A2[商品の在庫を確認する] A2 --> A3[配送手配をする] A3 --> A4[注文確認メールを送信する] A4 --> End([終了])

最もシンプルなアクティビティ図です。一連の作業が順番に実行されます。

8.3 決定ノードとマージノード（分岐と合流）

実際の業務やプログラムでは、条件によって処理の流れが分かれることがよくあります。 アクティビティ図では決定ノードとマージノードを使って分岐と合流を表現します。

決定ノード（Decision Node）

表記：ひし形 ◇

条件によってフローが分岐する箇所を表します。各分岐には条件を記述します（ガード条件）。

マージノード（Merge Node）

表記：ひし形 ◇

分岐したフローが再び合流する箇所を表します。決定ノードと同じ形状ですが、用途が異なります。

決定ノードとマージノードの例（ECサイトの注文処理）

flowchart TD Start([開始]) --> A1[注文を受け付ける] A1 --> A2[在庫を確認する] A2 --> D1{在庫は十分?} D1 -->|はい| A3[配送手配をする] D1 -->|いいえ| A4[入荷予定を確認する] A4 --> A5[入荷待ちメールを送信] A3 --> M1{マージ} A5 --> M1 M1 --> A6[注文をデータベースに保存] A6 --> End([終了])

在庫状況によって処理が分岐し、最終的にマージノードで合流します。

ガード条件の書き方

分岐の矢印には条件を記述します。角括弧 [条件] で囲むのが正式ですが、 初心者向けには「はい/いいえ」「OK/NG」などシンプルな記述でも問題ありません。

8.4 フォークとジョイン（並行処理）

複数の作業が同時に進行する場合、フォークノードとジョインノードを使って並行処理を表現します。

フォークノード（Fork Node）

表記：太い横線（バー）━━━

1つのフローが複数の並行フローに分かれる箇所を表します。すべてのフローが同時に開始されます。

ジョインノード（Join Node）

表記：太い横線（バー）━━━

並行フローがすべて完了するのを待って合流する箇所を表します。すべてのフローが揃うまで次に進みません。

並行処理の例（商品発送準備）

flowchart TD Start([開始]) --> A1[注文を確定する] A1 --> Fork{フォーク} Fork --> B1[商品をピッキング] Fork --> B2[送り状を印刷] Fork --> B3[梱包材を準備] B1 --> Join{ジョイン} B2 --> Join B3 --> Join Join --> A2[商品を梱包する] A2 --> A3[配送業者に引き渡す] A3 --> End([終了])

「ピッキング」「送り状印刷」「梱包材準備」の3つの作業は並行して進められます。すべて完了後に梱包作業に進みます。

フォーク/ジョインとマージ/決定の違い

  • フォーク/ジョイン：すべてのフローが同時に実行される（AND条件）
  • 決定/マージ：条件によっていずれか1つのフローが実行される（OR条件）

8.5 スイムレーン（担当者・部門別の整理）

スイムレーン（Swimlane）は、複数の担当者や部門が関わる業務フローを整理するための領域分けです。 プールで泳ぐレーンのように、縦または横に領域を分割し、「誰が何をするか」を明確にします。

スイムレーンには以下の2種類があります：

縦レーン（Vertical Swimlane）

縦に領域を分割します。左から右へフローが進むため、時間の流れが分かりやすくなります。

横レーン（Horizontal Swimlane）

横に領域を分割します。上から下へフローが進みます。縦長のページに適しています。

スイムレーンの例（購買申請プロセス）

flowchart LR subgraph 申請者 A1[購買申請を作成] --> A2[申請を提出] end subgraph 上長 B1[申請を確認] --> B2{承認する?} B2 -->|承認| B3[承認印を押す] B2 -->|却下| B4[却下理由を記入] end subgraph 購買部門 C1[発注先を選定] --> C2[発注書を作成] C2 --> C3[発注を実行] end A2 --> B1 B3 --> C1 B4 --> A1

申請者、上長、購買部門のそれぞれの役割が明確に分かれています。

スイムレーンの効果

スイムレーンを使うことで、責任の所在が明確になり、業務の引き継ぎポイントが分かりやすくなります。 また、特定の部門に作業が集中していないか、待ち時間が発生していないかなど、業務改善のポイントも見つけやすくなります。

8.6 アクティビティ図とフローチャートの違い

アクティビティ図はフローチャートと似ていますが、UMLの一部として標準化されており、より多機能です。

比較項目 フローチャート アクティビティ図
目的 アルゴリズムの可視化 業務プロセス全体の可視化
並行処理 表現が難しい フォーク/ジョインで明示的に表現
担当者の区別 通常なし スイムレーンで明確に区別
標準化 組織ごとに記法が異なる場合がある UMLとして国際標準化されている
適用範囲 主にプログラムのロジック 業務フローからアルゴリズムまで幅広い

どちらを使うべきか

シンプルなアルゴリズムを説明するだけなら、従来のフローチャートでも十分です。 しかし、複数部門が関わる業務プロセス、並行処理を含むシステム、他のUML図と連携した設計などでは、 アクティビティ図を使うことで表現力が大きく向上します。

実習 8-1: コンビニレジ業務のアクティビティ図を描く

コンビニのレジ業務をアクティビティ図で表現してみましょう。分岐や並行処理を意識して描いてください。

手順

  1. 1 開始ノードから「商品をスキャンする」アクションを配置
  2. 2 「年齢確認が必要か?」という決定ノードを追加し、分岐させる
  3. 3 「合計金額を表示」「ポイントカード処理」を並行処理として配置
  4. 4 「支払い方法」で決定ノードを追加（現金/カード/電子マネー）
  5. 5 すべてのフローをマージし、「レシート発行」→終了ノードへ

ヒント

draw.ioを使って実際に描いてみましょう。菱形が決定ノード、太い横線がフォーク/ジョインです。 各アクションには動詞を使って「〜する」と書くと分かりやすくなります。

理解度確認クイズ

  1. Q1. 決定ノードとフォークノードの違いを説明してください。

    回答を表示

    決定ノード：条件によっていずれか1つのフローを選択します（OR条件）。ひし形で表します。
    フォークノード：すべてのフローが同時に実行されます（AND条件）。太い横線で表します。

  2. Q2. スイムレーンを使う主な目的は何ですか？

    回答を表示

    複数の担当者や部門が関わる業務フローを整理するためです。 「誰が何をするか」を明確にし、責任の所在や作業の引き継ぎポイントを分かりやすくします。

  3. Q3. アクティビティ図がフローチャートより優れている点を2つ挙げてください。

    回答を表示

    主な利点は以下です：
    1. 並行処理の表現：フォーク/ジョインを使って明示的に表現できる
    2. スイムレーン：担当者や部門ごとに領域を分けて責任を明確化できる
    3. 標準化：UMLとして国際標準化されているため、ツール間の互換性が高い

第8章のまとめ

  • アクティビティ図は業務プロセスやアルゴリズムを表現するUML図です
  • 基本要素は開始ノード、アクション、制御フロー、終了ノードです
  • 決定ノードとマージノードで条件分岐と合流を表現します
  • フォークとジョインで並行処理を明示的に表現できます
  • スイムレーンを使うことで担当者別の作業を整理できます
  • フローチャートより表現力が高く、業務分析から設計まで幅広く活用できます
