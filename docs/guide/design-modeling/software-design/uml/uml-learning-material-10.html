第10章

コンポーネント図・パッケージ図 - システムの構成部品を描く

システムを構成する部品（コンポーネント）とその依存関係を表現する方法を学びます。大規模なシステムを整理して把握するためのパッケージ図も合わせて習得します。

この章で学ぶこと

  • コンポーネント図の基本要素と表記法を理解する
  • パッケージ図による大規模システムの整理方法を習得する
  • Webアプリケーションの構成を図で表現できるようになる

10.1 コンポーネント図とは

コンポーネント図（Component Diagram）は、システムを構成する再利用可能な部品（コンポーネント）と、 それらの間の依存関係を表現するUML図です。ソフトウェアの物理的な構造を理解するために使用されます。

例えば、Webアプリケーションでは「フロントエンド」「バックエンドAPI」「データベース接続」などの部品に分かれています。 コンポーネント図を使うと、これらの部品がどのように組み合わさってシステム全体を構成しているかを視覚化できます。

コンポーネント図が役立つ場面

  • アーキテクチャ設計：システムを部品に分割し、全体構成を整理する
  • 依存関係の把握：どの部品がどの部品を使っているかを明確にする
  • 再利用の促進：独立した部品として設計することで、別プロジェクトでの再利用が可能になる

10.2 コンポーネント図の基本要素

コンポーネント

システムを構成する部品。右上に小さなアイコン（矩形2つ）が付いた四角形で表現します。

インターフェース

提供される機能（円記号）と必要とする機能（半円記号）で表現します。

依存関係

点線矢印で、あるコンポーネントが別のコンポーネントに依存していることを示します。

ポート

コンポーネントの境界上の小さな四角。外部とのやり取りの窓口を表します。

コンポーネント図の基本記法

flowchart TB subgraph legend[凡例] comp1["<<component>>
コンポーネント名"] comp2["<<component>>
別のコンポーネント"] iface((提供インターフェース)) end comp1 -.依存.-> comp2 comp1 --> iface

重要ポイント

コンポーネントは「独立して動作可能な部品」という意味を持ちます。例えば、ライブラリ、モジュール、サービス、JARファイル、DLLなどが該当します。

10.4 パッケージ図とは

パッケージ図（Package Diagram）は、システムを論理的なグループ（パッケージ）に分割し、 それらの間の依存関係を表現する図です。大規模なシステムを整理して理解しやすくするために使用されます。

パッケージは、クラスやコンポーネント、あるいは他のパッケージを含むことができる「フォルダ」のようなものです。 例えば、Javaのcom.example.serviceのようなパッケージ名が該当します。

パッケージ図の用途

  • モジュール分割：システムを機能や責務ごとに整理する
  • 依存関係の管理：循環依存を避け、クリーンな構造を維持する
  • チーム分担：パッケージ単位で開発担当を割り振る

10.5 パッケージ図の基本要素

パッケージ

タブ付きフォルダのような形で表現します。名前空間やモジュールを表します。

依存関係

点線矢印で、あるパッケージが別のパッケージに依存していることを示します。

入れ子構造

パッケージの中に別のパッケージを含めることができます（階層構造）。

要素

パッケージ内にクラス、インターフェース、コンポーネントなどを含めることができます。

パッケージ図の基本記法

flowchart TB subgraph pkgA[パッケージA] classA1[クラスA1] classA2[クラスA2] end subgraph pkgB[パッケージB] classB1[クラスB1] end pkgA -.依存.-> pkgB

ECサイトのバックエンドシステムを例に、パッケージ図を見てみましょう。 機能ごとにパッケージを分割し、依存関係を整理しています。

ECサイトのパッケージ構成

設計のポイント

  • レイヤー化：プレゼンテーション層、ビジネスロジック層、データアクセス層に分離
  • 単方向依存：上位層から下位層への依存のみ（逆方向の依存を避ける）
  • 責務の分離：各パッケージが明確な役割を持つ

10.8 コンポーネント図とパッケージ図の使い分け

観点 コンポーネント図 パッケージ図
焦点 物理的な部品構成 論理的なグループ分け
表現対象 実行可能な部品、ライブラリ 名前空間、モジュール、フォルダ
粒度 比較的粗い（サブシステムレベル） 細かくも粗くも調整可能
使用場面 アーキテクチャ設計、デプロイメント検討 コード構成、モジュール分割

実務での使い分け

アーキテクチャ全体を説明するときはコンポーネント図、 コードの構成を詳細に説明するときはパッケージ図を使うと効果的です。 また、両方を組み合わせて使うこともよくあります。

理解度確認クイズ

  1. Q1. コンポーネント図とパッケージ図の違いを説明してください。

    回答を表示

    コンポーネント図は、システムを構成する物理的な部品（ライブラリ、モジュール、サービスなど）とその依存関係を表現します。
    パッケージ図は、システムを論理的なグループ（パッケージ、名前空間）に分割し、それらの依存関係を表現します。
    コンポーネント図は「実行可能な部品」、パッケージ図は「論理的な整理」に焦点を当てています。

  2. Q2. パッケージ図で「単方向依存」が推奨される理由は何ですか？

    回答を表示

    単方向依存（上位層から下位層への依存のみ）にすることで、以下のメリットがあります：
    1. 循環依存の防止：パッケージ間の複雑な相互依存を避ける
    2. 保守性の向上：下位層を変更しても上位層への影響が少ない
    3. 再利用性：下位層を他のプロジェクトで再利用しやすい
    4. テストのしやすさ：下位層から独立してテスト可能

  3. Q3. Webアプリケーションをコンポーネント図で表現する際、どのような部品に分割するのが一般的ですか？

    回答を表示

    一般的には以下のようなコンポーネントに分割します：
    1. フロントエンド：ユーザーインターフェース（React、Vue.jsなど）
    2. バックエンドAPI：ビジネスロジック、API提供（Express、Djangoなど）
    3. データアクセス層：データベース接続、ORM（Sequelize、Hibernateなど）
    4. データベース：データストレージ（MySQL、PostgreSQLなど）
    これらの依存関係は、フロントエンド → バックエンドAPI → データアクセス層 → データベース の順になります。

第10章のまとめ

  • コンポーネント図は、システムを構成する物理的な部品とその依存関係を表現します
  • パッケージ図は、システムを論理的なグループに分割し、依存関係を整理します
  • コンポーネント図はアーキテクチャ設計に、パッケージ図はコード構成に有効です
  • Webアプリケーションは、フロントエンド、バックエンドAPI、データアクセス層などに分割するのが一般的です
  • 依存関係は単方向（上位層→下位層）にすることで、保守性と再利用性が向上します
