14.2 各図の使い分け方

UMLには多くの図がありますが、「どの図をいつ使うか」を適切に選択することが重要です。 以下に、代表的な場面ごとの図の選び方をまとめます。

「システムの目的と機能」を伝えたい

→ ユースケース図

  • 顧客や非技術者への説明に最適
  • システムの「誰が」「何をするか」が一目で分かる
  • 要件定義フェーズで活躍

「データ構造と関係性」を整理したい

→ クラス図

  • データベース設計の前段階として有効
  • オブジェクト指向設計の中心的な図
  • エンティティの属性と関連を明確化

「処理の流れと呼び出し順序」を示したい

→ シーケンス図

  • API呼び出しやメッセージパッシングの可視化
  • 時系列に沿った処理フローが分かりやすい
  • 詳細設計やレビューで効果的

「業務プロセス全体」を俯瞰したい

→ アクティビティ図

  • 業務フロー分析やBPR（業務改革）に活用
  • 分岐・並行処理が分かりやすい
  • スイムレーンで担当者別の役割を明確化

「状態の変化とトリガー」を表現したい

→ ステートマシン図

  • 注文ステータス、承認フローなどの状態管理
  • 「何をきっかけに状態が変わるか」が明確
  • 仕様の抜け漏れを発見しやすい

「システムの部品構成」を整理したい

→ コンポーネント図 / パッケージ図

  • マイクロサービス構成やモジュール分割の可視化
  • 依存関係の整理とアーキテクチャレビュー
  • 大規模システムの構造把握に有効

「インフラ構成と配置」を記録したい

→ デプロイメント図

  • サーバー、ネットワーク、クラウドリソースの配置
  • 本番環境の構成管理やインフラドキュメント
  • 運用チームとの情報共有に最適

重要ポイント

1つのプロジェクトで全ての図を作成する必要はありません。目的に応じて適切な図を2～4種類選択し、組み合わせて使うことが実務では一般的です。

14.3 よくある間違いと対処法

UML初心者がつまずきやすいポイントと、その改善方法を紹介します。

よくある間違い集

間違い1: 詳細すぎる図を描いてしまう

問題: すべての属性・メソッドを書き込み、図が複雑になりすぎて本質が伝わらない。

解決方法:

  • 目的に応じて「重要な部分だけ」を記載する
  • 詳細はコードやAPIドキュメントに任せる
  • 図は「全体像の理解」を第一目的とする
間違い2: クラス図とER図を混同する

問題: データベーステーブルをそのままクラス図に書いてしまう。

解決方法:

  • クラス図 = ビジネスロジックを含むオブジェクト設計
  • ER図 = データベースのテーブル構造
  • 両者は似ているが目的が異なることを理解する
間違い3: シーケンス図に制御フローを詰め込みすぎる

問題: 1つのシーケンス図に複数の条件分岐やループを詰め込み、読みにくくなる。

解決方法:

  • 「正常フロー」と「異常フロー」で図を分ける
  • 複雑な条件はaltフラグメントの代わりにアクティビティ図を使用
  • 1つの図で伝えることを1つに絞る
間違い4: 図の更新を怠る（ドキュメント腐敗）

問題: コードは変更したが図は古いまま。実装と図が乖離してしまう。

解決方法:

  • PlantUMLなどテキストベースツールでバージョン管理
  • 重要な部分だけを図にして、更新コストを下げる
  • 設計変更時に図の更新をタスクに含める
間違い5: 関連と依存の違いを理解していない

問題: すべての矢印を実線で引いてしまい、関係性の種類が不明瞭。

解決方法:

  • 関連 (実線): 永続的な関係（顧客が注文を「持つ」）
  • 依存 (点線): 一時的な使用（メソッド引数で渡される程度）
  • 集約・コンポジション: ライフサイクルの強弱を表現

ヒント

UMLは「完璧な正確さ」よりも「チームでの理解共有」が目的です。厳密さにこだわりすぎず、相手に伝わることを最優先にしましょう。

14.4 業界でのUMLの使われ方

実際のソフトウェア開発現場では、UMLはどのように活用されているのでしょうか？ 業界やプロジェクトの規模によって使われ方が異なります。

大企業・SIer

  • 要件定義書や設計書の一部として正式にUMLを使用
  • クラス図、シーケンス図、ER図が特に重視される
  • Enterprise Architectなど専用ツールを導入

スタートアップ・アジャイル開発

  • 必要な部分だけを簡易的に描く（軽量UML）
  • ホワイトボードやMiro等で議論しながら作成
  • PlantUMLでコードと一緒にバージョン管理

組み込み・制御系

  • ステートマシン図が中心的に使われる
  • デプロイメント図でハードウェア構成を管理
  • SysML（UMLの拡張）も併用されることが多い

Webサービス開発

  • API設計でシーケンス図を活用
  • マイクロサービス構成をコンポーネント図で整理
  • draw.ioやMermaid.jsなど軽量ツールが人気

UMLの現状と今後

かつてのUMLブーム（2000年代）ほどの厳密な使用は減りましたが、UMLの考え方は現在も広く浸透しています。

  • 簡易UML: 正式な記法にこだわらず、図の本質だけを活用
  • As-Code化: PlantUMLやMermaid.jsでテキストから図を生成
  • 自動生成: コードから図を自動生成するツールの発展

GoFの23パターンなど、よくある設計課題に対する解決策の「型」を学びます。 UMLのクラス図を使ってパターンを理解し、実装に活かせます。

学習を始める

ドメイン駆動設計（DDD）

ビジネスロジックを中心に据えた設計手法です。 UMLのクラス図やコンポーネント図を活用して、ドメインモデルを表現します。

学習を始める

データベース設計（ER図）

UMLのクラス図と似た記法で、データベースのテーブル構造を設計します。 正規化理論やパフォーマンスチューニングも学べます。

学習を始める

アジャイル開発入門

UMLを活用しながら、スプリント単位で設計と開発を反復する手法を学びます。 ユーザーストーリーとユースケース図の関連も理解できます。

14.6 実践に向けたアドバイス

最後に、UMLを実務で活用するための実践的なアドバイスをお伝えします。

継続的な練習

日常的に触れるシステム（Webサービス、アプリ等）を自分なりにUMLで描いてみましょう。 習うより慣れろです。

レビューを受ける

作成した図を先輩エンジニアやチームメンバーに見てもらい、 フィードバックをもらうことで理解が深まります。

完璧を求めすぎない

厳密なUML記法にこだわるより、 「相手に伝わること」「議論のきっかけになること」を優先しましょう。

コードとセットで学ぶ

UMLで設計した後、実際にコードに落とし込む練習をすることで、 設計とプログラミングの橋渡しができます。

理解度確認クイズ

  1. Q1. 「API呼び出しの流れを時系列で示したい」場合、どの図を使うべきですか？

    回答を表示

    シーケンス図を使用します。シーケンス図は時間の流れに沿ってメッセージのやり取りを表現するため、API呼び出しや処理フローの可視化に最適です。

  2. Q2. よくある間違いの1つとして「詳細すぎる図を描く」がありますが、どう改善すべきですか？

    回答を表示

    目的に応じて「重要な部分だけ」を記載します。図は全体像の理解を第一目的とし、詳細はコードやドキュメントに任せることで、伝わりやすい図になります。
