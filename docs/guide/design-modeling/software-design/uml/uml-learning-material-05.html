第5章

クラス図の詳細 - 属性・操作・関連の表現

クラス図をより詳細に表現する方法を学びます。属性（データ）と操作（機能）の書き方、可視性（+/-/#）の意味、関連・集約・コンポジションの違い、多重度（1対多など）の表記方法を実例を通じて理解します。

この章で学ぶこと

  • 属性（データ）と操作（メソッド）の記述方法を理解する
  • 可視性（+/-/#）の意味とアクセス制御を理解する
  • 関連・集約・コンポジションの違いを正確に使い分けられる
  • 多重度の表記法（1対多、多対多など）をマスターする

5.1 属性の記述

クラス図では、クラスが持つ属性（Attribute）を記述することで、そのクラスがどのようなデータを保持するかを表現します。 属性は、クラスの長方形の中段に記述します。

属性の基本構文

属性は以下の形式で記述します： 

[可視性] 属性名: データ型 [= デフォルト値]

属性の例

  • - name: string - 名前（文字列型、private）
  • - age: int - 年齢（整数型、private）
  • + email: string - メールアドレス（文字列型、public）
  • - isActive: boolean = true - 有効フラグ（デフォルト値true）

図：User クラスの属性記述例

classDiagram class User { - userId: string - name: string - email: string - age: int - isActive: boolean - createdAt: datetime }

データ型の表記

一般的に使用されるデータ型の例：

基本型

  • string - 文字列
  • int - 整数
  • float / double - 小数
  • boolean - 真偽値
  • date / datetime - 日付・日時

複合型

  • List<string> - 文字列のリスト
  • Map<string, int> - 辞書型
  • Address - カスタムクラス
  • Order[] - 配列

5.2 可視性（アクセス修飾子）

可視性（Visibility）は、属性や操作にアクセスできる範囲を制御します。 UMLでは4つの可視性が定義されており、記号で表現します。

4つの可視性記号

+ public（公開）

どこからでもアクセス可能。外部に公開するメソッドなど

- private（非公開）

そのクラス内部からのみアクセス可能。内部データの隠蔽

# protected（保護）

そのクラスと継承したサブクラスからアクセス可能

~ package（パッケージ）

同じパッケージ内からアクセス可能（Java等）

図：可視性の使い分け例

classDiagram class BankAccount { - accountNumber: string - balance: decimal # ownerName: string + accountType: string + getBalance(): decimal + deposit(amount: decimal): void + withdraw(amount: decimal): boolean - validateAmount(amount: decimal): boolean }

設計原則：カプセル化

基本的に属性は-（private）にして、必要に応じてpublicなメソッド（getter/setter）でアクセスするのが一般的です。 これにより、内部実装を隠蔽し、データの整合性を保つことができます。

5.3 操作（メソッド）の記述

操作（Operation）は、クラスが実行できる機能や振る舞いを表します。 プログラミングにおけるメソッド（関数）に相当します。操作はクラスの長方形の下段に記述します。

操作の基本構文

[可視性] 操作名(引数リスト): 戻り値の型

操作の例

  • + getName(): string - 名前を取得（引数なし、戻り値string）
  • + setAge(age: int): void - 年齢を設定（引数1つ、戻り値なし）
  • + calculateTotal(price: decimal, qty: int): decimal - 合計計算
  • - validateEmail(email: string): boolean - メール検証（private）

図：Product クラスの属性と操作

classDiagram class Product { - productId: string - name: string - price: decimal - stock: int + getName(): string + getPrice(): decimal + setPrice(price: decimal): void + isInStock(): boolean + addStock(quantity: int): void + reduceStock(quantity: int): boolean - validatePrice(price: decimal): boolean }

引数（パラメータ）の記述

引数が複数ある場合は、カンマで区切って記述します： 

+ register(name: string, email: string, password: string): boolean
+ search(keyword: string, category: string, maxResults: int): List<Product>

5.4 関連・集約・コンポジション

クラス間の関係性を表現するには、3つの主要な表記法があります。それぞれ「モノ同士のつながりの強さ」を表現します。

関連（Association）

最も基本的な関係性で、実線で結びます。「AはBを知っている」「AはBを使う」といった関係を表します。

図：関連の例（顧客と注文）

classDiagram class Customer { - customerId: string - name: string + placeOrder(): Order } class Order { - orderId: string - orderDate: date - totalAmount: decimal } Customer "1" -- "0..*" Order : places

集約（Aggregation）

「全体と部分」の関係で、部分は全体とは独立して存在できます。白抜きのひし形で表現します。 例：「部署と社員」（社員は部署が無くなっても存在できる）

図：集約の例（部署と社員）

classDiagram class Department { - deptId: string - deptName: string } class Employee { - empId: string - name: string - position: string } Department "1" o-- "0..*" Employee : has

コンポジション（Composition）

より強い「全体と部分」の関係で、部分は全体に依存します。黒塗りのひし形で表現します。 例：「注文と注文明細」（注文が削除されると明細も削除される）

図：コンポジションの例（注文と注文明細）

classDiagram class Order { - orderId: string - orderDate: date + calculateTotal(): decimal } class OrderLine { - lineNumber: int - productId: string - quantity: int - unitPrice: decimal } Order "1" *-- "1..*" OrderLine : contains

関連・集約・コンポジションの使い分け

関連（Association）

独立した2つのクラスがお互いを参照する関係

例：顧客と注文、学生と履修科目

集約（Aggregation）

全体と部分の関係だが、部分は独立して存在できる

例：部署と社員、チームとメンバー

コンポジション（Composition）

全体と部分の強い依存関係、部分は全体なしに存在できない

例：注文と注文明細、書籍と章

5.5 多重度の表記

多重度（Multiplicity）は、関連の両端に記述し、 「1つのAに対して、いくつのBが関連付けられるか」を表現します。

主な多重度の表記

固定数の多重度

  • 1 - ちょうど1つ
  • 0..1 - 0または1つ（オプション）
  • * または 0..* - 0個以上（任意の数）
  • 1..* - 1個以上（必須で複数可）

範囲指定の多重度

  • 2..5 - 2個から5個
  • 3, 5, 7 - 3個、5個、または7個
  • 0..10 - 0個から10個まで

図：様々な多重度の例

classDiagram class Company { - companyId: string - name: string } class Department { - deptId: string - deptName: string } class Employee { - empId: string - name: string } class Project { - projectId: string - projectName: string } Company "1" -- "1..*" Department : has Department "1" -- "0..*" Employee : employs Employee "1..*" -- "0..*" Project : works on

多重度の読み方

Company "1" -- "1..*" Department の場合：
「1つの会社（Company）は、1つ以上の部署（Department）を持つ」
「1つの部署（Department）は、ちょうど1つの会社（Company）に所属する」
と読みます。関連線の両端を見て、双方向から理解することが重要です。

実践例：ECサイトのクラス図

図：多重度を含む完全なクラス図例

classDiagram class Customer { - customerId: string - name: string - email: string + placeOrder(): Order } class Order { - orderId: string - orderDate: date - status: string + calculateTotal(): decimal + cancel(): boolean } class OrderItem { - itemId: string - quantity: int - unitPrice: decimal + getSubtotal(): decimal } class Product { - productId: string - name: string - price: decimal - stock: int + isAvailable(): boolean } class ShoppingCart { - cartId: string - createdAt: date + addItem(product: Product): void + removeItem(itemId: string): void } Customer "1" -- "0..*" Order : places Customer "1" -- "0..1" ShoppingCart : has Order "1" *-- "1..*" OrderItem : contains OrderItem "0..*" -- "1" Product : references ShoppingCart "1" o-- "0..*" Product : contains

これまで学んだ属性、操作、可視性、多重度を使って、図書館システムのクラス図を作成してみましょう。

作成するクラス

  2. 2 Memberクラス：属性（会員ID、氏名、メールアドレス、電話番号）、操作（本を借りる、本を返す）
  3. 3 Loanクラス（貸出記録）：属性（貸出ID、貸出日、返却予定日、実際の返却日）
  4. 4 クラス間の関連と多重度を考えて線で結ぶ（1人の会員は複数の本を借りられる、1冊の本は複数回貸し出される可能性がある）

ヒント：draw.ioやPlantUMLを使って実際に図を描いてみましょう。 可視性は、外部に公開すべき操作は+、内部データは-にしましょう。

理解度確認クイズ

  1. Q1. 可視性記号「-」と「+」の違いを説明してください。

    回答を表示

    「-」はprivate（非公開）で、そのクラス内部からのみアクセス可能です。 「+」はpublic（公開）で、どこからでもアクセス可能です。 一般的に、データを保持する属性は「-」で隠蔽し、外部とのインターフェースとなる操作は「+」にします。

  2. Q2. 集約（Aggregation）とコンポジション（Composition）の違いを具体例で説明してください。

    回答を表示

    集約は「全体と部分」の関係ですが、部分は独立して存在できます（例：部署と社員。社員は部署が無くなっても別の部署に移れる）。白抜きのひし形で表現します。

    コンポジションはより強い依存関係で、部分は全体なしには存在できません（例：注文と注文明細。注文が削除されると明細も削除される）。黒塗りのひし形で表現します。

  3. Q3. 多重度「1..*」と「0..*」の違いを説明してください。

    回答を表示

    1..*は「1つ以上（必須）」を意味し、最低でも1つは必要です（例：注文には最低1つの注文明細が必要）。

    0..*は「0個以上（任意）」を意味し、0個でも構いません（例：顧客は注文を1つも持っていなくても良い）。

  4. Q4. 属性「- balance: decimal = 0.00」の各要素の意味を説明してください。

    回答を表示

    -：可視性がprivate（非公開）
    balance：属性名（残高）
    decimal：データ型（小数点を含む数値）
    = 0.00：デフォルト値（初期値は0.00）

第5章のまとめ

  • 属性は「可視性 属性名: 型 = デフォルト値」の形式で記述します
  • 可視性記号：+（public）、-（private）、#（protected）、~（package）
  • 操作は「可視性 操作名(引数): 戻り値型」の形式で記述します
  • 関連は実線、集約は白抜きひし形、コンポジションは黒塗りひし形で表現します
  • 多重度で関係の数を表現：1、0..1、*（0..*）、1..*、範囲指定も可能
  • カプセル化の原則：属性はprivate、必要に応じてpublicメソッドで公開
