第9章

ステートマシン図 - 状態の変化を追跡する

「注文中」「発送済み」「配達完了」といった状態の変化を表現するステートマシン図を学びます。状態、遷移、トリガー、ガード条件などの基本要素を、ECサイトの注文管理やエレベーターの動作など身近な例で理解します。

この章で学ぶこと

  • ステートマシン図の基本概念と用途を理解する
  • 状態、遷移、トリガー、ガード条件の表記方法を習得する
  • ECサイトやエレベーターなど身近な例でステートマシン図を描けるようになる

9.1 ステートマシン図とは

ステートマシン図（State Machine Diagram）は、オブジェクトやシステムが持つ「状態」と、 ある状態から別の状態への「遷移」を表現するUML図です。時間の経過やイベントの発生によって、 システムがどのように状態を変化させるかを視覚的に示します。

身近な例として、信号機を考えてみましょう。信号機は「赤」「黄」「青」という3つの状態を持ち、 時間経過に応じて状態が遷移します。このような状態の変化を図で表現するのがステートマシン図です。

stateDiagram-v2 [*] --> 赤 赤 --> 青: 60秒経過 青 --> 黄: 40秒経過 黄 --> 赤: 5秒経過

ステートマシン図が活躍する場面

  • 注文処理やワークフローの管理
  • デバイスや機器の動作制御
  • ユーザーインターフェースの画面遷移
  • ゲームキャラクターの行動パターン

9.2 基本要素：状態と遷移

ステートマシン図の基本要素は、状態（State）と遷移（Transition）です。

状態（State）

オブジェクトが特定の条件を満たしている間の状況を表します。 角丸四角形で表現します。

例：注文中、発送済み、配達完了

遷移（Transition）

ある状態から別の状態への移り変わりを表します。 矢印で表現します。

例：支払い完了 → 注文確定

特殊な状態

開始状態（Initial State）

黒丸で表現。ステートマシンの開始点を示します。

終了状態（Final State）

二重円（黒丸を囲む円）で表現。処理の完了を示します。

実習：簡単なドアの状態遷移

ドアの「開いている」「閉じている」という2つの状態を表現してみましょう。

stateDiagram-v2 [*] --> 閉じている 閉じている --> 開いている: 開ける 開いている --> 閉じている: 閉める 開いている --> [*]: 最終的に閉める

9.3 トリガーとイベント

状態遷移は、何らかのイベント（Event）やトリガー（Trigger）によって引き起こされます。 遷移の矢印に沿って、「どのイベントで遷移するか」を記述します。

トリガーの書き方

イベント名 [ガード条件] / アクション
  • イベント名：遷移のきっかけとなる出来事
  • ガード条件（オプション）：遷移が実行される追加条件
  • アクション（オプション）：遷移時に実行される処理

ECサイトの注文状態遷移

stateDiagram-v2 [*] --> 注文作成中 注文作成中 --> 注文確定: 注文送信 注文確定 --> 支払い待ち: 注文確認完了 支払い待ち --> 支払い完了: 支払い実行 支払い待ち --> キャンセル: キャンセル要求 支払い完了 --> 発送準備中: 在庫確保 発送準備中 --> 発送済み: 発送処理完了 発送済み --> 配達完了: 配達確認 配達完了 --> [*] キャンセル --> [*]

重要ポイント

イベント名は動詞形（「支払う」「キャンセルする」）で記述し、 状態名は名詞または状態を表す形容詞（「支払い待ち」「発送済み」）で記述します。

9.4 ガード条件

ガード条件（Guard Condition）は、 遷移が実行されるための追加条件を表します。イベントが発生しても、 ガード条件が満たされない場合は遷移が行われません。角括弧 [条件] で記述します。

実習：ATMの出金処理

残高の有無によって遷移が変わるATMの動作を表現してみましょう。

stateDiagram-v2 [*] --> 待機中 待機中 --> カード挿入済み: カード挿入 カード挿入済み --> PIN入力中: カード読取成功 PIN入力中 --> メニュー表示: PIN正常 PIN入力中 --> カード返却: PIN異常 メニュー表示 --> 出金処理中: 出金選択 出金処理中 --> 出金成功: 残高十分 出金処理中 --> 出金失敗: 残高不足 出金成功 --> カード返却 出金失敗 --> メニュー表示 カード返却 --> 待機中 カード返却 --> [*]

ガード条件の例：

  • 出金処理中 → 出金成功：[残高 >= 出金額]
  • 出金処理中 → 出金失敗：[残高 < 出金額]

注意点

同じイベントで複数の遷移がある場合、ガード条件は相互に排他的（どれか1つだけが真）になるように設定します。 複数の条件が同時に真になると、どの遷移を選ぶか曖昧になってしまいます。

9.5 エントリー/イグジットアクション

状態には、その状態に入るとき（エントリー）や出るとき（イグジット）に実行される処理を定義できます。

状態内アクションの記述方法

entry / アクション

状態に入るときに実行される処理

exit / アクション

状態から出るときに実行される処理

do / アクション

状態に留まっている間に継続的に実行される処理

エレベーターの動作例

stateDiagram-v2 [*] --> 停止中 停止中: entry / ドア開く 停止中: exit / ドア閉じる 停止中 --> 上昇中: 上階ボタン押下 停止中 --> 下降中: 下階ボタン押下 上昇中: do / 上方向に移動 上昇中: entry / 上昇モーター起動 上昇中: exit / モーター停止 下降中: do / 下方向に移動 下降中: entry / 下降モーター起動 下降中: exit / モーター停止 上昇中 --> 停止中: 目的階到達 下降中 --> 停止中: 目的階到達

エントリー/イグジットアクションのメリット

  • 状態に入る/出るたびに必ず実行される処理を1箇所に集約できる
  • 遷移ごとにアクションを書く必要がなくなり、図がシンプルになる
  • 初期化処理やクリーンアップ処理の記述に適している

9.6 複合状態と階層化

複雑なシステムでは、状態の中にさらに細かい状態（サブステート）を持つことがあります。 このような複合状態（Composite State）を使うと、 階層的に状態を整理できます。

音楽プレーヤーの例

stateDiagram-v2 [*] --> 電源OFF 電源OFF --> 電源ON: 電源ボタン 電源ON --> 電源OFF: 電源ボタン state 電源ON { [*] --> 停止中 停止中 --> 再生中: 再生ボタン 再生中 --> 停止中: 停止ボタン 再生中 --> 一時停止中: 一時停止ボタン 一時停止中 --> 再生中: 再生ボタン 一時停止中 --> 停止中: 停止ボタン }

複合状態のメリット

  • 関連する状態をグループ化して、図の見通しを良くできる
  • 親状態への遷移を定義すれば、子状態すべてに適用される
  • 共通の処理（エントリー/イグジット）を親状態にまとめられる

実習：スマートフォンの画面状態

スマートフォンの画面が「ロック状態」と「ロック解除状態」を持ち、 ロック解除状態の中で「ホーム画面」「アプリ実行中」などのサブステートがある状況を考えてみましょう。

stateDiagram-v2 [*] --> ロック状態 ロック状態 --> ロック解除状態: 認証成功 ロック解除状態 --> ロック状態: タイムアウト state ロック解除状態 { [*] --> ホーム画面 ホーム画面 --> アプリ実行中: アプリ起動 アプリ実行中 --> ホーム画面: ホームボタン }

9.7 実践例：会員登録のワークフロー

これまで学んだ要素を組み合わせて、Webサービスの会員登録フローをステートマシン図で表現してみましょう。

stateDiagram-v2 [*] --> 未登録 未登録 --> 情報入力中: 登録開始 情報入力中: entry / フォーム表示 情報入力中 --> 確認中: 送信ボタン押下 情報入力中 --> 未登録: キャンセル 確認中: entry / バリデーション実行 確認中 --> 情報入力中: 入力エラーあり 確認中 --> メール送信済み: 入力正常 メール送信済み: entry / 確認メール送信 メール送信済み: do / 認証待機 メール送信済み --> 登録完了: メールリンククリック メール送信済み --> 期限切れ: 24時間経過 期限切れ --> 未登録: 再登録要求 登録完了: entry / アカウント有効化 登録完了 --> [*]

この図のポイント

  • 各状態でのエントリーアクション（entry）を明記している
  • バリデーションエラー時は情報入力中に戻る
  • メール認証待ちの状態（do）を表現している
  • タイムアウトによる期限切れの遷移を定義している

理解度確認クイズ

  1. Q1. ステートマシン図における「状態」と「遷移」をそれぞれ説明してください。

    回答を表示

    状態（State）：オブジェクトが特定の条件を満たしている間の状況。角丸四角形で表現します。
    遷移（Transition）：ある状態から別の状態への移り変わり。矢印で表現します。

  2. Q2. ガード条件とは何ですか？またどのように記述しますか？

    回答を表示

    ガード条件は、遷移が実行されるための追加条件です。 イベントが発生しても、ガード条件が満たされない場合は遷移が行われません。 角括弧 [条件] で記述します。
    例：出金要求 [残高 >= 出金額]

  3. Q3. entry、exit、do の各アクションの違いを説明してください。

    回答を表示

    entry：状態に入るときに1回だけ実行される処理
    exit：状態から出るときに1回だけ実行される処理
    do：状態に留まっている間、継続的に実行される処理

よくある間違いと対処法

❌ 状態名と遷移ラベルが曖昧

状態名は名詞または形容詞（「待機中」「処理済み」）、遷移は動詞（「処理する」「キャンセル」）で明確に区別しましょう。

❌ ガード条件が排他的でない

同じイベントで複数の遷移がある場合、ガード条件が同時に真にならないよう注意してください。 例：[x > 10] と [x <= 10]

❌ 開始状態・終了状態の欠落

ステートマシン図には必ず開始状態（黒丸）を設けます。 終了状態は必須ではありませんが、明確な終了がある場合は記述すると分かりやすくなります。

第9章のまとめ

  • ステートマシン図は、オブジェクトやシステムの状態とその遷移を表現する図です
  • 状態は角丸四角形、遷移は矢印で表現します
  • 遷移には、トリガー（イベント）、ガード条件、アクションを記述できます
  • entry/exit/doアクションを使って、状態に関連する処理を定義できます
  • 複合状態を使うと、階層的に状態を整理できます
  • ワークフロー、デバイス制御、画面遷移など、幅広い場面で活用されます
