第4章: シーケンス図による相互作用モデリング

時系列で捉えるオブジェクト間の相互作用

この章で学ぶこと

  • ライフラインとアクティベーションボックスの概念

  • 複合フラグメント（alt、opt、loop、par）の使用法
  • 相互作用の使用（interaction use）による図の構造化

  • 実際のユースケースからシーケンス図への変換手法

4.1 シーケンス図とは何か

シーケンス図（Sequence Diagram）は、オブジェクト間の相互作用を時系列で表現するUMLの振る舞い図です。特定のシナリオにおいて、どのオブジェクトがいつ、どのような順序でメッセージをやり取りするかを視覚的に示します。

シーケンス図が重要な理由
  • 動的振る舞いの可視化：静的なクラス図では表現できない時系列の処理フローを表現
  • 処理の流れの確認：複雑な業務処理やアルゴリズムの理解を促進
  • 設計の妥当性検証：想定したオブジェクト設計が実際に機能するかを確認
  • 開発者間の共有：処理の詳細を開発チーム全体で共有しやすい

シーケンス図は、まるで演劇の台本のようなものです。舞台（システム）に登場する役者（オブジェクト）が、どのタイミングでどのセリフ（メッセージ）を言うかが詳細に記されています。

sequenceDiagram participant 顧客 as 顧客 participant システム as ECサイト participant 決済 as 決済システム participant 在庫 as 在庫管理 顧客->>システム: 商品注文 システム->>在庫: 在庫確認 在庫-->>システム: 在庫あり システム->>決済: 決済処理 決済-->>システム: 決済完了 システム-->>顧客: 注文確認

4.2 シーケンス図の基本構成要素

シーケンス図は4つの主要な構成要素から成り立っています。

1. ライフライン（Lifeline）

参加するオブジェクトやアクターを表します。図の上部に配置され、下向きの点線はそのオブジェクトの存在期間を表します。

ライフラインの表記法
  • 構文：「オブジェクト名:クラス名」または「:クラス名」
  • 例：「customer:Customer」、「:PaymentService」
  • 匿名オブジェクト：「:ClassName」形式で名前を省略可能
  • アクター：人間のアイコンで表現することもある

2. メッセージ（Message）

オブジェクト間で交換される通信を表します。矢印の種類により意味が異なります。

メッセージ種類 矢印 意味 例
同期メッセージ → 呼び出し元が応答を待つ メソッド呼び出し
非同期メッセージ → 呼び出し元が応答を待たない イベント送信
リターンメッセージ - -> 処理結果の返却 戻り値
セルフメッセージ → 自分自身への呼び出し 内部メソッド

3. アクティベーションボックス（Activation Box）

オブジェクトが処理を実行している期間を縦の矩形で表現します。

sequenceDiagram participant A as クライアント participant B as サーバー participant C as データベース A->>+B: リクエスト送信 B->>+C: データ検索 C-->>-B: 結果返却 B-->>-A: レスポンス送信

4. 複合フラグメント（Combined Fragment）

条件分岐やループなどの制御構造を表現します。

4.3 メッセージの詳細

シーケンス図では、様々な種類のメッセージを使い分けることで、正確な相互作用を表現できます。

同期メッセージと非同期メッセージ

同期メッセージ（Synchronous Message）

送信者が受信者の処理完了まで待機するメッセージです。

  • 実例：関数呼び出し、データベースクエリ
  • 特徴：ブロッキング操作、順序が保証される
  • リスク：応答が遅いと全体の処理が遅延
非同期メッセージ（Asynchronous Message）

送信者が受信者の処理を待たずに次の処理に進むメッセージです。

  • 実例：イベント送信、メール送信、ログ出力
  • 特徴：ノンブロッキング操作、並行処理可能
  • メリット：性能向上、レスポンス時間改善
sequenceDiagram participant U as ユーザー participant W as Webアプリ participant M as メールサービス participant D as データベース U->>W: 会員登録 W->>D: ユーザー情報保存 D-->>W: 保存完了 W->>M: 確認メール送信（非同期） W-->>U: 登録完了画面 Note over M: バックグラウンドで
メール送信処理

セルフメッセージ

オブジェクトが自分自身にメッセージを送る場合に使用します。内部的な処理やヘルパーメソッドの呼び出しを表現します。

実習 4-1: ATMシステムのシーケンス図作成

ATMでの現金引き出しプロセスをシーケンス図で表現してみましょう。

処理の流れ
  1. 利用者がカードを挿入
  2. ATMがPINの入力を要求
  3. 利用者がPINを入力
  4. ATMが銀行システムに認証を依頼
  5. 銀行システムが認証結果を返す
  6. ATMが引き出し金額の入力を要求
  7. 利用者が金額を入力
  8. ATMが銀行システムに残高確認を依頼
  9. 銀行システムが残高を確認し、引き出し処理
  10. ATMが現金を放出
PlantUMLの例
@startuml
actor "利用者" as User
participant "ATM" as ATM
participant "銀行システム" as Bank

User -> ATM: カード挿入
ATM --> User: PIN入力要求
User -> ATM: PIN入力
ATM -> Bank: 認証要求(カード番号, PIN)
Bank --> ATM: 認証OK
ATM --> User: 金額入力要求
User -> ATM: 引き出し金額入力
ATM -> Bank: 残高確認・引き出し要求
Bank -> Bank: 残高確認
Bank -> Bank: 口座から減額
Bank --> ATM: 処理完了
ATM -> ATM: 現金準備
ATM --> User: 現金放出
@enduml
期待される結果
  • 同期メッセージと非同期メッセージが適切に使い分けられている
  • セルフメッセージで内部処理が表現されている
  • 処理の時系列が正確に表現されている

4.4 複合フラグメント

複合フラグメントは、条件分岐やループなどの制御構造をシーケンス図で表現するための仕組みです。

alt（Alternative）- 条件分岐

if-else文に相当する条件分岐を表現します。

sequenceDiagram participant U as ユーザー participant S as システム participant D as データベース U->>S: ログイン要求 S->>D: ユーザー認証 D-->>S: 認証結果 alt 認証成功 S-->>U: ログイン成功 S->>D: ログ記録 else 認証失敗 S-->>U: エラーメッセージ S->>S: 失敗カウント増加 end

opt（Optional）- オプション処理

特定の条件下でのみ実行される処理を表現します。

sequenceDiagram participant C as 顧客 participant E as ECサイト participant P as ポイント C->>E: 商品購入 E->>E: 購入金額計算 opt 会員の場合 E->>P: ポイント付与 P-->>E: 付与完了 end E-->>C: 購入完了

loop（Loop）- 繰り返し処理

ループ処理を表現します。繰り返し条件を記述できます。

sequenceDiagram participant B as バッチ処理 participant D as データベース participant M as メール B->>D: 対象ユーザー取得 D-->>B: ユーザーリスト loop 各ユーザーに対して B->>M: メール送信 M-->>B: 送信完了 B->>D: 送信履歴更新 end B->>D: バッチ完了記録

par（Parallel）- 並行処理

並行して実行される処理を表現します。

並行処理の例：オンライン注文処理

注文確定後、以下の処理が並行して実行される場合

  • 在庫システムでの在庫更新
  • 会計システムでの売上記録
  • 顧客への確認メール送信

4.5 相互作用の使用（Interaction Use）

複雑なシーケンス図は、小さな単位に分割して管理することができます。相互作用の使用により、他のシーケンス図を参照できます。

相互作用の使用のメリット
  • 再利用性：共通処理を複数の図で再利用
  • 保守性：変更が一箇所の修正で済む
  • 可読性：図が複雑になりすぎることを防ぐ
  • 責任分担：チームメンバーがそれぞれ担当部分を作成
sequenceDiagram participant U as ユーザー participant S as ショッピング participant A as 認証 participant P as 支払い U->>S: 商品注文 rect rgb(200, 220, 240) note over S,A: ref: ユーザー認証 S->>A: 認証処理呼び出し A-->>S: 認証結果 end rect rgb(240, 220, 200) note over S,P: ref: 決済処理 S->>P: 決済処理呼び出し P-->>S: 決済結果 end S-->>U: 注文完了

4.6 時間制約とタイミング

シーケンス図では、時間的な制約やタイミング要件も表現できます。

時間制約の表現方法

時間制約の例
  • 応答時間：「{応答時間 < 3秒}」
  • タイムアウト：「{タイムアウト = 30秒}」
  • 待機時間：「{待機 = 1秒}」
  • 処理時間：「{処理時間 < 100ms}」
実習 4-2: オンライン会議システムのシーケンス図

オンライン会議への参加プロセスを複合フラグメントを使って表現してみましょう。

要件
  • 参加者は会議IDとパスワードを入力
  • 認証に成功すれば会議に参加、失敗なら3回まで再入力可能
  • 会議参加後、音声・ビデオの設定を並行して行う
  • 他の参加者がいれば、参加者リストを更新
PlantUMLの例
@startuml
participant "参加者" as User
participant "会議アプリ" as App
participant "認証サーバー" as Auth
participant "音声システム" as Audio
participant "ビデオシステム" as Video

User -> App: 会議ID・パスワード入力

loop 最大3回
    App -> Auth: 認証要求
    Auth --> App: 認証結果
    alt 認証成功
        App --> User: 会議参加成功
        break
    else 認証失敗
        App --> User: エラー・再入力要求
    end
end

par 音声設定
    App -> Audio: 音声デバイス設定
    Audio --> App: 設定完了
and ビデオ設定  
    App -> Video: ビデオデバイス設定
    Video --> App: 設定完了
end

opt 他の参加者がいる場合
    App -> App: 参加者リスト更新
end

App --> User: 会議開始準備完了
@enduml
期待される結果
  • loop、alt、par、optフラグメントが適切に使用されている
  • エラー処理とリトライ機能が表現されている
  • 並行処理が正しくモデリングされている

4.7 シーケンス図作成のベストプラクティス

効果的なシーケンス図を作成するための実践的なガイドラインです。

設計時のポイント
  • 適切な粒度：1つの図で1つの主要なシナリオに集中する
  • アクターの識別：システム外部のアクターから開始する
  • メッセージの命名：動詞で始まる分かりやすい名前を付ける
  • エラー処理：正常系だけでなく例外系も検討する
よくある間違いと対策
  • 間違い：すべての処理を1つの図に詰め込む
    対策：相互作用の使用で分割する
  • 間違い：技術的な詳細を過度に記述
    対策：ビジネスロジックに焦点を当てる
  • 間違い：時系列が曖昧
    対策：メッセージの順序を明確にする

4.8 他の図との関係

シーケンス図は他のUML図と組み合わせることで、より完全な設計を表現できます。

図間の関係性
  • ユースケース図 → シーケンス図：各ユースケースの詳細な実現方法を記述
  • クラス図 ↔ シーケンス図：静的構造と動的振る舞いの整合性を確認
  • アクティビティ図 ↔ シーケンス図：ワークフローの詳細な相互作用を表現
  • ステートマシン図 → シーケンス図：状態遷移を引き起こすメッセージの流れ

応用問題

オンラインバンキングでの振込処理を想定し、以下を含むシーケンス図を作成してください：

  • ユーザー認証（失敗時は3回までリトライ）
  • 振込先口座の確認
  • 残高確認と振込実行
  • 取引履歴の記録と確認メールの送信（並行処理）
