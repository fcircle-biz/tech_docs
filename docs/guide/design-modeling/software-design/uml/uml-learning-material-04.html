クラス図の基礎 - 「モノ」と「関係」を描く

実務で最も頻繁に使用されるクラス図の基礎を学びます。「顧客」「商品」「注文」といった概念を四角形で表し、それらの関係を線で結ぶ方法を習得します。難しい専門用語は後回しにして、まずは「読める」「描ける」を目指します。

この章で学ぶこと

  • クラス図とは何か、何を表現するのかを理解する
  • クラスの基本的な表記方法（四角形と3つの区画）を習得する
  • クラス間の関連を線で表現できる
  • 身近な例（顧客、商品、注文）でクラス図を読み書きできる

4.1 クラス図とは何か

クラス図（Class Diagram）は、システムを構成する「モノ」とその「関係」を表現するUML図です。 実務で最も頻繁に使用される図の一つで、データベース設計、オブジェクト指向プログラミング、システム全体の構造設計など、幅広い場面で活用されます。

クラス図は以下を視覚化します：

クラス（概念・モノ）

顧客、商品、注文など、システムで扱う「モノ」を四角形で表現

関連（関係性）

クラス同士のつながり（顧客が注文する、など）を線で表現

クラス図の特徴

  • 静的な構造を表現（時間の流れは含まない）
  • データベース設計のER図に似た考え方
  • プログラムの設計書としても使用される
  • 初心者でも理解しやすいシンプルな表記法

4.2 クラスの表記方法

クラスは3つの区画を持つ四角形で表現します。まずは一番上の「クラス名」だけに注目しましょう。

classDiagram class 顧客 { } class 商品 { } class 注文 { }

クラスの3つの区画（詳細は次章で学習）

第1区画：クラス名

システムで扱う「モノ」の名前（例：顧客、商品、注文）

第2区画：属性（データ）

そのモノが持つ情報（例：顧客名、価格、注文日）
※詳細は第5章で学習します

第3区画：操作（メソッド）

そのモノができる動作（例：注文する、キャンセルする）
※詳細は第5章で学習します

この章では名前だけに集中

この章では、まず「クラス名」だけを使ったシンプルなクラス図に慣れましょう。 属性や操作の詳細な書き方は、次の第5章で学習します。

4.3 クラス間の関連

クラス同士の関係を関連（Association）と呼び、線で結んで表現します。 線の上にラベルを書くと、どのような関係かが分かりやすくなります。

classDiagram 顧客 --> 注文 : 注文する 注文 --> 商品 : 含む

この図は以下を表現しています：

  • 「顧客」は「注文」を注文する
  • 「注文」は「商品」を含む

関連の読み方のコツ

矢印の向きに沿って読みます。「AからBへ」の矢印があれば、「AはBを～する」と読めます。 ラベルがあれば、その動詞を使って「AはBを[ラベル]する」と読みます。

4.4 実例：ECサイトのクラス図

オンラインショッピングサイトを例に、実際のクラス図を見てみましょう。

classDiagram class 顧客 { +顧客ID +氏名 +メールアドレス } class 注文 { +注文ID +注文日 +合計金額 } class 商品 { +商品ID +商品名 +価格 } class カート { +カートID +作成日 } 顧客 "1" --> "0..*" 注文 : 注文する 顧客 "1" --> "0..1" カート : 持つ 注文 "1" --> "1..*" 商品 : 含む カート "1" --> "0..*" 商品 : 入れる

この図の読み方

1人の顧客は、0個以上の注文を持つことができる

1人の顧客は、0個または1個のカートを持つ

1つの注文は、1個以上の商品を含む

1つのカートは、0個以上の商品を含むことができる

多重度の表記（数字の意味）

  • 1：必ず1つ
  • 0..1：0個または1個
  • 0..*：0個以上（いくつでも可）
  • 1..*：1個以上（必ず1つは存在）

実習 4-1: 図書館システムのクラス図を描く

draw.ioまたはPlantUMLを使って、簡単な図書館システムのクラス図を作成してみましょう。

手順

  1. 1 以下のクラスを四角形で描く：「会員」「本」「貸出」
  2. 2 「会員」から「貸出」へ線を引き、ラベルに「借りる」と書く
  3. 3 「貸出」から「本」へ線を引き、ラベルに「含む」と書く
  4. 4 完成したら、図を声に出して読んでみましょう（「会員は貸出を借りる」など）

4.5 クラス図の読み方・描き方のコツ

読み方のコツ

  • まずクラス名だけを見て、どんな「モノ」が登場するかを把握
  • 次に線（関連）を見て、どのモノとどのモノがつながっているかを確認
  • 矢印の向きに沿って声に出して読む（「顧客は注文を注文する」など）
  • 多重度（数字）があれば、「何個持てるか」を確認

描き方のコツ

  • まず登場する「モノ」をリストアップ（名詞を探す）
  • 各「モノ」を四角形で描く
  • 「AはBを～する」という関係を考えて線で結ぶ
  • 最初から完璧を目指さず、何度も描き直す前提で描く
  • 同僚やレビュー者に声に出して読んでもらう（理解されるか確認）

よくある間違い

  • クラス名が動詞になっている（正しくは名詞）
  • 関連の向きが逆（「注文が顧客を持つ」になっている）
  • 1つの図に詰め込みすぎ（見やすさ重視で分割しましょう）
  • 専門用語を使いすぎ（まずは日常語で表現）

つまずきポイント

「クラスと関連だけでは物足りない」と感じるかもしれませんが、まずはこの基本を確実に習得することが重要です。 属性（データ）や操作（メソッド）の詳細な書き方は、次の第5章でじっくり学びます。 焦らず、まずは「読める」「描ける」を目指しましょう。

理解度確認クイズ

  1. Q1. クラス図は何を表現するための図ですか？

    回答を表示

    クラス図は、システムを構成する「モノ」とその「関係」を表現するための図です。 静的な構造を視覚化し、データベース設計やプログラム設計に活用されます。

  2. Q2. クラスは何個の区画を持つ四角形で表現されますか？また、それぞれ何を表しますか？

    回答を表示

    クラスは3つの区画を持つ四角形で表現されます：
    1. 第1区画：クラス名（モノの名前）
    2. 第2区画：属性（そのモノが持つデータ）
    3. 第3区画：操作（そのモノができる動作・メソッド）

  3. Q3. 多重度「0..*」は何を意味しますか？

    回答を表示

    「0..*」は「0個以上」を意味します。つまり、1つも持たない場合もあれば、 何個でも持てることを表します（無制限）。例えば「顧客 1 --> 0..* 注文」なら、 「1人の顧客は、注文を1つも持っていない場合もあれば、何個でも持てる」ことを示します。

第4章のまとめ

  • クラス図は、システムの「モノ」と「関係」を表現する最も重要なUML図です
  • クラスは3つの区画（名前、属性、操作）を持つ四角形で表現されます
  • クラス間の関連は線で結び、ラベルで関係性を説明します
  • 多重度（1、0..1、0..*、1..*）で「何個持てるか」を表現します
  • まずは「読める」「描ける」を目指し、詳細な書き方は次章で学びます
