ドメインモデルの基礎 - ビジネスを図で表現する

ドメインモデルとは何か、どのように作成するかを学びます。ビジネスの概念を図（クラス図やオブジェクト図）で表現し、関係者全員が理解できるモデルを作る方法を習得します。ECサイトの注文管理を例に、モデリングの実践を行います。

この章で学ぶこと

  • ドメインモデルとは何か、その役割と目的を理解する
  • ビジネスの概念を図で表現する方法を習得する
  • クラス図とオブジェクト図の使い分けを理解する
  • ECサイトの注文管理を題材に、実際にモデリングを体験する

3.1 ドメインモデルとは何か

ドメインモデルとは、ビジネス領域（ドメイン）の概念や関係性を図や言葉で表現したものです。システムが扱う「注文」「顧客」「商品」といったビジネス上の重要な概念を整理し、それらがどのように関係しているかを視覚化します。

ドメインモデルは、開発者とビジネス担当者をつなぐ「共通の地図」の役割を果たします。この地図があることで、チーム全員が同じ理解のもとで議論し、設計を進められるようになります。

ドメインモデルの3つの役割

  • ① 共通理解の形成: 開発者とビジネス担当者が同じ言葉で話せるようになる
  • ② 設計の基盤: モデルをそのままコードに反映することで、ビジネスとコードが一致する
  • ③ ドキュメント: システムの中核となるビジネスルールを可視化し、新しいメンバーの理解を助ける

なぜ図で表現するのか

文章だけで「注文は顧客が行い、商品を複数含む」と書いても、人によって解釈が異なる可能性があります。図で表現すると、以下のメリットがあります：

視覚的に理解しやすい

複雑な関係性も一目で把握できる。「注文」と「顧客」の関係、「注文」と「商品」の関係が視覚的に明確になる。

共通認識を作りやすい

図を見ながら議論することで、認識のズレを早期に発見し、修正できる。曖昧さが減る。

コードへの変換が容易

クラス図はそのままプログラムのクラスに変換できる。設計とコードの一貫性が保ちやすい。

変更に強い

仕様変更時に図を更新することで、影響範囲を素早く把握できる。チーム全体で変更を共有しやすい。

3.2 ドメインモデルの構成要素

ドメインモデルは主に以下の3つの要素で構成されます：

① 概念（Concept）

ビジネス上の重要な「もの」や「こと」を表します。ECサイトなら「注文」「顧客」「商品」「カート」などが概念に当たります。これらはユビキタス言語（第2章で学んだ共通言語）の主要な用語でもあります。

② 関連（Relationship）

概念同士のつながりを表します。「顧客が注文を行う」「注文は複数の商品を含む」といった関係性を線で表現します。関連には以下の種類があります：

関連の種類

  • 1対1: 1人の顧客は1つのカートを持つ
  • 1対多: 1人の顧客は複数の注文を行える
  • 多対多: 1つの注文は複数の商品を含み、1つの商品は複数の注文に含まれる

③ 属性（Attribute）

概念が持つ具体的な情報です。「注文」なら「注文番号」「注文日」「合計金額」「配送先住所」などが属性に当たります。属性は概念の特性を表現します。

3.3 モデリングの基本手順

ドメインモデルを作成する際は、以下の手順で進めます：

ステップ1: ビジネスヒアリング

ドメイン専門家（ビジネス担当者）から業務の流れやルールを聞き出します。このとき、専門用語をそのまま使い、ユビキタス言語を形成していきます。

ヒアリング例（ECサイト）:

「お客様がサイトで商品を選んでカートに追加します。カート内の商品を確認後、注文を確定すると注文番号が発行されます。注文には配送先住所と支払い方法が必要です。」

ステップ2: 概念抽出

ヒアリング内容から、ビジネス上重要な「名詞」を抽出します。これらが概念（クラス）の候補になります。

抽出した概念:

  • • 顧客（お客様）
  • • 商品
  • • カート
  • • 注文
  • • 配送先住所
  • • 支払い方法

ステップ3: 関連の整理

抽出した概念同士の関係性を整理します。「誰が何をする」「何が何を含む」といった視点で考えます。

関連の例:

  • • 顧客は注文を行う（1対多）
  • • 顧客は1つのカートを持つ（1対1）
  • • 注文は複数の商品を含む（多対多）
  • • 注文は1つの配送先住所を持つ（1対1）

ステップ4: 図の作成

整理した概念と関連を、クラス図やオブジェクト図として描きます。ツールは何でも構いません（draw.io、Miro、手書きでも可）。

次のセクションで実際にECサイトの注文管理モデルを作成してみましょう。

反復的なプロセス

モデリングは一度で完璧にする必要はありません。ビジネス担当者と図を見ながら議論し、何度も修正・改善を繰り返すことで、より正確なモデルが完成します。

3.4 ECサイトの注文管理モデリング実践

それでは、ECサイトの注文管理システムを題材に、実際にドメインモデルを作成してみましょう。

ビジネスシナリオ

シナリオ: 顧客がECサイトで商品を購入する流れ

  1. 顧客がサイトにログインする
  2. 商品を検索・閲覧し、気に入った商品をカートに追加する
  3. カート内の商品を確認し、数量を調整する
  4. 注文を確定し、配送先住所と支払い方法を入力する
  5. 注文が確定され、注文番号が発行される

クラス図による表現

クラス図は、概念（クラス）の構造と関係性を表現する図です。各クラスが持つ属性とメソッド、そしてクラス間の関連を示します。

ECサイト注文管理のクラス図

classDiagram class 顧客 { +顧客ID: string +氏名: string +メールアドレス: string +パスワード: string +注文する() +カートに追加する() } class 注文 { +注文番号: string +注文日時: datetime +合計金額: decimal +ステータス: string +注文を確定する() +合計金額を計算する() } class 注文明細 { +商品ID: string +数量: int +単価: decimal +小計を計算する() } class 商品 { +商品ID: string +商品名: string +価格: decimal +在庫数: int +在庫を確認する() } class 配送先住所 { +郵便番号: string +都道府県: string +市区町村: string +番地: string +建物名: string } 顧客 "1" --> "0..*" 注文: 行う 注文 "1" --> "1..*" 注文明細: 含む 注文明細 "0..*" --> "1" 商品: 参照する 注文 "1" --> "1" 配送先住所: 配送される

クラス図の読み方

  • "1" → "0..*": 1人の顧客は0件以上の注文を持つ
  • "1" → "1..*": 1つの注文は1件以上の注文明細を含む
  • 属性: 各クラスが持つデータ項目
  • メソッド: 各クラスが持つ振る舞い（処理）

オブジェクト図による表現

クラス図とオブジェクト図の使い分け

  • クラス図: システム全体の構造を示す。設計段階で使用
  • オブジェクト図: 具体例を示して理解を助ける。ビジネス担当者との議論で有効

3.5 モデリングのベストプラクティス

効果的なドメインモデルを作成するためのポイントを紹介します。

誰にでも分かる図を描く

技術者だけでなく、ビジネス担当者も理解できるシンプルな図を心がけましょう。

  • • 専門用語を避け、ユビキタス言語を使う
  • • 1つの図に詰め込みすぎない
  • • 適度に分割し、焦点を明確にする

ユビキタス言語との一貫性

モデル内の用語は、ユビキタス言語と完全に一致させます。

  • • クラス名はビジネス用語そのまま
  • • 「Order」ではなく「注文」と日本語で書いてもOK
  • • チーム内で用語集を共有する

モデルとコードの同期

ドメインモデルとコードは常に一致させます。モデルを変更したらコードも更新します。

  • • モデル駆動設計を意識する
  • • リファクタリング時はモデルも更新
  • • ドキュメントの自動生成も検討

継続的な対話と改善

モデルは完成品ではなく、進化し続けるものです。

  • • 定期的にビジネス担当者とレビュー
  • • 新しい要件が出たらモデルを更新
  • • モデルの正確さを常に検証

よくある失敗

  • 技術詳細を入れすぎる: データベース設計とドメインモデルを混同しない
  • モデルを放置する: コードだけ更新してモデルを更新しないと、ドキュメントが役に立たなくなる
  • 完璧主義: 最初から完璧を目指さず、反復的に改善する

理解度確認クイズ

  1. Q1. ドメインモデルの3つの役割を挙げてください。

    回答を表示

    ① 共通理解の形成: 開発者とビジネス担当者が同じ言葉で話せる
    ② 設計の基盤: モデルをコードに反映し、一貫性を保つ
    ③ ドキュメント: ビジネスルールを可視化し、新メンバーの理解を助ける

  2. Q2. ドメインモデルの構成要素である「概念」「関連」「属性」について、ECサイトの例を1つずつ挙げてください。

    回答を表示

    概念（Concept）: 注文、顧客、商品など
    関連（Relationship）: 顧客が注文を行う、注文が商品を含むなど
    属性（Attribute）: 注文番号、顧客名、商品価格など

  3. Q3. クラス図とオブジェクト図の違いは何ですか？

    回答を表示

    クラス図は、概念（クラス）の構造と関係性を示す設計図です。システム全体の構造を表現します。
    オブジェクト図は、特定の時点における具体的なデータの例を示します。実際のインスタンスを表現し、理解を助けます。

  4. Q4. モデリングで「技術詳細を入れすぎない」とはどういう意味ですか？

    回答を表示

    ドメインモデルはビジネスの概念を表現するものであり、データベース設計（テーブル構造、外部キー、インデックスなど）や技術的な実装詳細（フレームワーク、ライブラリなど）は含めません。ビジネス担当者が理解できる純粋なビジネスロジックに焦点を当てます。

この章のまとめ

  • ドメインモデルは、ビジネス領域の概念や関係性を図で表現したもので、共通理解・設計基盤・ドキュメントの役割を果たす
  • ドメインモデルは「概念（Concept）」「関連（Relationship）」「属性（Attribute）」の3要素で構成される
  • モデリングは「ビジネスヒアリング → 概念抽出 → 関連整理 → 図の作成」の4ステップで進める
  • クラス図は構造を示す設計図、オブジェクト図は具体例を示す実例として使い分ける
  • 効果的なモデリングには、シンプルさ・ユビキタス言語との一貫性・継続的な改善が重要
