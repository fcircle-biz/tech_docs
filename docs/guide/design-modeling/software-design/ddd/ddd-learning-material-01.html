DDDとは何か - ビジネスとソフトウェアをつなぐ設計手法

複雑なビジネスロジックを持つシステムで、なぜドメインを中心に設計することが重要なのかを理解します。DDDの歴史、基本的な考え方、適用すべき場面を具体例とともに紹介します。

この章で学ぶこと

  • DDDとは何か、その基本的な考え方を理解する
  • なぜDDDが必要なのか、従来の設計との違いを知る
  • DDDを適用すべき場面と、適用しない方が良い場面を区別できるようになる
  • DDDの歴史的背景と現代における意義を理解する

1.1 DDD（ドメイン駆動設計）とは何か

DDD（Domain-Driven Design / ドメイン駆動設計）は、複雑なビジネスロジックを持つソフトウェアシステムを設計・開発するための方法論です。2003年にEric Evansが著書「Domain-Driven Design」で提唱しました。

DDDの最大の特徴は、「ドメイン（業務領域）をソフトウェア設計の中心に置く」という考え方です。従来の設計では、データベース構造や技術的な制約から設計を始めることが多かったのに対し、DDDでは「ビジネスで何が重要か」を最優先に考えます。

ドメインとは

ドメインとは、ソフトウェアが対象とする「業務領域」や「問題領域」のことです。例えば、ECサイトなら「商品管理」「在庫管理」「注文処理」「配送」などがドメインに含まれます。それぞれのドメインには、独自のルール（ビジネスルール）や概念が存在します。

1.2 DDDの歴史的背景

2000年代初頭、ソフトウェア開発の現場では複雑なビジネスロジックを扱うシステムが増加していました。しかし、当時主流だった設計手法では以下のような問題が発生していました：

従来の課題①: 技術とビジネスの乖離

データベース設計やフレームワークの都合が優先され、ビジネスの本質が見えなくなっていた。開発者とビジネス担当者の間でコミュニケーションが困難だった。

従来の課題②: 保守性の低下

ビジネスルールがコード全体に散らばり、変更が困難に。仕様変更のたびに予期しない箇所が壊れるリスクが増大していた。

Eric Evansは、これらの課題を解決するために、ビジネスドメインの知識をソフトウェア設計の中核に据えるという新しいアプローチを提唱しました。これがDDDの始まりです。

1.3 なぜDDDが必要なのか

DDDは、以下のような複雑性を持つシステムで特に有効です：

DDDが力を発揮する場面

  • 複雑なビジネスルールがある: 単純なCRUD操作だけでなく、業務固有の複雑なルールや制約が多数存在する
  • 長期間の保守が必要: システムが5年、10年と使われ続け、仕様変更が頻繁に発生する
  • ドメイン専門家との協業が重要: ビジネス担当者や業務の専門家と密に連携する必要がある
  • 複数のチームで開発: 大規模なシステムで、複数のチームが異なる部分を担当する

1.4 DDDを適用すべき場面・適用しない方が良い場面

DDDは強力な設計手法ですが、すべてのプロジェクトに適しているわけではありません。適用すべき場面と、そうでない場面を理解することが重要です。

DDDを適用すべき場面

  • ● ECサイトの注文管理システム（複雑な在庫管理、配送ルール）
  • ● 金融システム（複雑な取引ルール、リスク管理）
  • ● 医療システム（患者管理、診療フロー）
  • ● 業務システム（複雑な承認フロー、権限管理）

DDDを適用しない方が良い場面

  • ● 単純なCRUDアプリケーション（マスタデータ管理など）
  • ● 短期間で使い捨てるプロトタイプ
  • ● 小規模で複雑性が低いシステム
  • ● 技術検証のみが目的のプロジェクト

判断基準

「ビジネスルールが複雑か」「長期保守が必要か」「ドメイン専門家との協業が重要か」の3つの観点で判断しましょう。これらすべてに「はい」と答えられる場合、DDDの適用を検討する価値があります。

DDDは、大きく分けて戦術的パターンと戦略的パターンの2つに分類されます。これから学んでいく要素の全体像を把握しておきましょう。

戦術的パターン（第4-9章）

実装レベルの設計パターンです。コードをどのように書くか、オブジェクトをどう設計するかを扱います。

エンティティ、値オブジェクト、集約、リポジトリなど

戦略的パターン（第10-11章）

大規模システムの設計パターンです。システム全体をどう分割するか、チーム間でどう協業するかを扱います。

境界づけられたコンテキスト、コンテキストマップなど

理解度確認クイズ

  1. Q1. DDDの最大の特徴は何ですか？

    回答を表示

    「ドメイン（業務領域）をソフトウェア設計の中心に置く」という考え方です。データベース構造や技術的な制約からではなく、ビジネスで何が重要かを最優先に考えます。

  2. Q2. DDDを適用すべきプロジェクトの特徴を3つ挙げてください。

    回答を表示

    ① 複雑なビジネスルールがある
    ② 長期間の保守が必要
    ③ ドメイン専門家との協業が重要
    （その他、複数チームでの開発なども該当）

  3. Q3. DDDの「戦術的パターン」と「戦略的パターン」の違いは何ですか？

    回答を表示

    戦術的パターンは実装レベルの設計パターン（エンティティ、値オブジェクトなど）で、コードをどう書くかを扱います。戦略的パターンは大規模システムの設計パターン（境界づけられたコンテキストなど）で、システム全体をどう分割するかを扱います。

  4. Q4. 単純なCRUDアプリケーションにDDDを適用すべきでないのはなぜですか？

    回答を表示

    DDDは複雑なビジネスルールを扱うための設計手法です。単純なCRUD操作だけであれば、DDDを適用することで逆に複雑性が増し、開発効率が下がる可能性があります。シンプルな設計で十分な場合は、シンプルな手法を選ぶべきです。

この章のまとめ

  • DDDは「ドメインを中心に設計する」ことで、複雑なビジネスロジックを持つシステムを効果的に開発する手法
  • 2003年にEric Evansが提唱し、現代ではマイクロサービスやクリーンアーキテクチャと組み合わせて使われる
  • 複雑なビジネスルール、長期保守、ドメイン専門家との協業が重要な場面で力を発揮する
  • 戦術的パターン（実装レベル）と戦略的パターン（大規模設計）の2つの側面がある
  • すべてのプロジェクトに適用すべきではなく、適用場面を見極めることが重要
