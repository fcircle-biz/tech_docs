ユビキタス言語 - チーム全員で共有する共通言語

DDDの最も重要な概念であるユビキタス言語を学びます。開発者とビジネス担当者が同じ言葉を使うことで、認識のズレを防ぎ、正確なソフトウェアを作る方法を習得します。用語集の作り方、モデルとコードの一貫性を保つ方法を実践的に学びます。

この章で学ぶこと

  • ユビキタス言語とは何か、なぜ必要なのかを理解する
  • チーム内で共通言語を作り、育てていく方法を学ぶ
  • 用語集の作成と管理方法を習得する
  • コード・ドキュメント・会話でユビキタス言語を一貫して使う方法を理解する

2.1 ユビキタス言語とは何か

ユビキタス言語（Ubiquitous Language）は、DDDで最も重要な概念の一つです。「ユビキタス（Ubiquitous）」とは「どこにでも存在する」という意味で、チーム全体（開発者、ビジネス担当者、ドメインエキスパート）が同じ言葉を使って同じ概念を表現することを指します。

例えば、ECサイトの開発では「注文」という概念を扱います。この「注文」を、プログラマーは「Order」、営業担当者は「受注」、配送担当者は「出荷指示」と呼んでいたらどうなるでしょうか？会話やドキュメント、コードの間で認識のズレが生まれ、誤解や手戻りが発生してしまいます。

ユビキタス言語の基本原則

ユビキタス言語は、コード、ドキュメント、会話のすべてで同じ用語を使うことを徹底します。クラス名、メソッド名、変数名、データベーステーブル名、ドキュメントの見出し、会議での発言――すべてで一貫した用語を使用します。

2.2 なぜユビキタス言語が必要なのか

ユビキタス言語がない場合、以下のような問題が発生します：

問題①: コミュニケーションのコスト増大

「これはOrderのことですか？それともPurchase Requestのことですか？」といった確認が毎回必要になり、開発速度が低下します。

問題②: 誤解による手戻り

同じ概念を違う言葉で表現することで、認識のズレが生まれ、間違った機能を実装してしまいます。

問題③: コードの可読性低下

ビジネス用語とコード上の用語が一致しないため、コードを読んでも業務の流れが理解できません。

問題④: ドキュメントとコードの乖離

ドキュメントとコードで異なる用語を使うため、どちらが正しいのか判断できなくなります。

ユビキタス言語の効果

ユビキタス言語を導入することで、チーム全体の認識が一致し、コミュニケーションコストが劇的に下がります。また、コードがビジネスロジックを直接表現するため、読みやすく保守しやすいコードになります。

2.3 ユビキタス言語の作り方

ユビキタス言語は、一度決めたら終わりではありません。チーム全体で継続的に育てていくプロセスが重要です。

ユビキタス言語の作成プロセス

  1. 1 ドメインエキスパートとの対話: ビジネス担当者や業務の専門家と会話し、彼らが使っている用語を集めます。「このプロセスは何と呼ばれていますか？」「この概念の正式名称は何ですか？」と積極的に質問します。
  2. 2 用語の定義を明確化: 集めた用語について、「具体的にはどういう意味か」「どのような条件を満たすとそう呼ばれるのか」を明確にします。曖昧な定義は誤解の元です。
  3. 3 用語集（Glossary）の作成: 定義した用語を文書化し、チーム全体で共有します。GitHubのWikiやConfluenceなど、誰でもアクセスできる場所に置きます。
  4. 4 コードへの反映: 用語集に記載した用語を、クラス名、メソッド名、変数名として使用します。技術用語に翻訳せず、そのまま使うことが重要です。
  5. 5 継続的な改善: 開発を進める中で新しい概念が見つかったら、すぐに用語集に追加します。また、不適切な用語があれば、チーム全体で議論して変更します。

具体例: ECサイトの注文管理でのユビキタス言語

❌ ユビキタス言語を使わない場合

# ビジネス用語と乖離
class PurchaseRequest:
    def submit(self):
        # 送信処理
        pass

    def check_status(self):
        # ステータス確認
        pass

ビジネス側は「注文（Order）」と呼んでいるのに、コードは「購入リクエスト（PurchaseRequest）」という異なる用語を使用。

✅ ユビキタス言語を使う場合

# ビジネス用語と一致
class Order:
    def place(self):
        # 注文確定処理
        pass

    def check_status(self):
        # ステータス確認
        pass

ビジネス側と同じ「注文（Order）」という用語を使用。クラス名、メソッド名もビジネスプロセスと一致。

2.4 用語集（Glossary）の管理方法

用語集は、ユビキタス言語を記録・共有するための重要なドキュメントです。以下のような形式で管理します：

用語集サンプル（ECサイト）

用語 定義 英語表記
注文 顧客が商品を購入する意思を確定し、配送先・支払い方法を指定した状態 Order
注文明細 注文に含まれる個々の商品と数量の組み合わせ OrderLine
カート 顧客が購入を検討中の商品を一時的に保存する場所。まだ注文は確定していない Cart
在庫引当 注文確定時に、在庫を確保し他の注文では使用できないようにする処理 StockAllocation

用語集の管理場所

用語集は、Confluence、Notion、GitHubのWiki、リポジトリ内のMarkdownファイルなど、チーム全員がアクセスできる場所に置きます。コードと同じリポジトリで管理すれば、バージョン管理も容易です。

用語集管理のベストプラクティス

  • 定義は具体的に: 「〇〇な状態」「〇〇を満たすもの」など、明確な条件を記載する
  • 例を追加する: 具体例を併記することで、理解しやすくなる
  • 関連用語にリンクする: 用語間の関係を明示し、体系的に理解できるようにする
  • 更新履歴を残す: 用語の定義が変更された理由と時期を記録する

2.5 コードとの一貫性を保つ

ユビキタス言語は、単にドキュメントに書くだけでは不十分です。コードそのものがユビキタス言語を表現していることが重要です。

ユビキタス言語をコードで表現する例

ECサイトで「注文を確定する」という業務プロセスをコードで表現します。ビジネス担当者が使う言葉をそのままコードに反映させることがポイントです。

order.py 
# ユビキタス言語を使ったクラス設計
class Order:
    """注文: 顧客が商品を購入する意思を確定した状態"""

    def __init__(self, order_id, customer, order_lines):
        self.order_id = order_id
        self.customer = customer
        self.order_lines = order_lines  # 注文明細のリスト
        self.status = OrderStatus.DRAFT  # 下書き状態

    def place(self):
        """注文を確定する"""
        # 在庫引当を実行
        self._allocate_stock()
        # ステータスを「確定済み」に変更
        self.status = OrderStatus.PLACED

    def _allocate_stock(self):
        """在庫引当: 注文明細の商品を在庫から確保する"""
        for line in self.order_lines:
            line.allocate_stock()

クラス名 Order、メソッド名 place()、変数名 order_lines など、すべてがビジネス用語と一致しています。

よくある間違い

技術用語に翻訳してしまうことです。例えば、ビジネス側が「注文を確定する」と言っているのに、コードで submitRequest() や confirmPurchase() といったメソッド名にすると、ユビキタス言語が壊れてしまいます。

ユビキタス言語の一貫性チェックリスト

理解度確認クイズ

  1. Q1. ユビキタス言語とは何ですか？

    回答を表示

    チーム全体（開発者、ビジネス担当者、ドメインエキスパート）が同じ言葉を使って同じ概念を表現することです。コード、ドキュメント、会話のすべてで一貫した用語を使用します。

  2. Q2. ユビキタス言語がない場合、どのような問題が発生しますか？3つ挙げてください。

    回答を表示

    ① コミュニケーションのコスト増大（毎回確認が必要）
    ② 誤解による手戻り（認識のズレが生まれる）
    ③ コードの可読性低下（業務の流れが理解できない）
    ④ ドキュメントとコードの乖離（どちらが正しいか判断できない）

  3. Q3. 用語集（Glossary）に記載すべき情報は何ですか？

    回答を表示

    ① 用語（日本語）
    ② 明確な定義（具体的な条件や状態）
    ③ 英語表記（コードで使用する名前）
    ④ 具体例（あれば）
    ⑤ 関連用語へのリンク（あれば）

  4. Q4. ビジネス担当者が「注文を確定する」と言っている場合、コードのメソッド名は何が適切ですか？

    回答を表示

    place() が適切です（「注文を確定する」= "place an order"）。
    技術用語に翻訳した submitRequest() や confirmPurchase() は、ユビキタス言語と一致しないため不適切です。
