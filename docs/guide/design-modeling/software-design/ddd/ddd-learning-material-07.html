リポジトリ - 集約の永続化と取得

集約をデータベースに保存し、取得するためのリポジトリパターンを学びます。永続化の詳細をドメイン層から隠蔽し、ドメインモデルの純粋性を保つ方法を習得します。インターフェースの設計、実装の分離、テスタビリティの向上について理解します。

この章で学ぶこと

  • リポジトリパターンとは何か、その役割を理解する
  • リポジトリのインターフェース設計と実装の分離の重要性を学ぶ
  • 集約単位でのデータ永続化と取得方法を習得する
  • リポジトリを活用したテスタビリティの向上方法を理解する

7.1 リポジトリとは何か

リポジトリ（Repository）は、集約の永続化と取得を担当するオブジェクトです。データベースへのアクセス方法をカプセル化し、ドメイン層からインフラストラクチャ層の詳細を隠蔽します。

リポジトリは、「コレクション風のインターフェース」を提供します。ドメイン層のコードからは、あたかもメモリ上のコレクション（ListやSetなど）を操作しているかのように見えますが、実際にはデータベースに対する永続化操作が行われます。

リポジトリの役割

リポジトリは集約ごとに1つ定義します。例えば、注文集約には「OrderRepository」、顧客集約には「CustomerRepository」を用意します。これにより、集約の境界が明確になり、データアクセスのロジックが整理されます。

7.2 なぜリポジトリが必要なのか

リポジトリパターンを使わずにドメイン層から直接データベースにアクセスすると、以下のような問題が発生します：

問題①: ドメインロジックの汚染

SQL文やORM固有のコードがドメインモデルに混入し、ビジネスルールが見えにくくなります。ドメインモデルの純粋性が失われます。

問題②: テストが困難

ドメインロジックのテストにデータベースが必須になり、テストの実行速度が遅くなります。また、テストデータの準備が煩雑になります。

問題③: 技術的依存関係

特定のデータベースやORMに強く依存し、技術スタックの変更が困難になります。例えば、PostgreSQLからMySQLへの移行などが難しくなります。

問題④: 重複コード

同じデータアクセスロジックが複数箇所に散らばり、保守性が低下します。変更時に修正漏れが発生しやすくなります。

リポジトリのメリット

  • ドメインモデルの純粋性: ドメイン層がインフラストラクチャの詳細に依存せず、ビジネスロジックに集中できます
  • テスタビリティの向上: モックやスタブを使って、データベースなしでドメインロジックをテストできます
  • 柔軟な実装変更: 永続化技術を変更してもドメイン層への影響がありません
  • データアクセスの一元管理: 集約へのアクセスロジックが1箇所に集約され、保守性が向上します

7.3 リポジトリのインターフェース設計

リポジトリのインターフェースは、ドメイン層に配置します。これにより、ドメイン層がインフラストラクチャ層に依存することなく、必要なデータアクセス機能を定義できます。

IOrderRepository.java (ドメイン層) 
// ドメイン層に配置するインターフェース
public interface IOrderRepository {
    // IDによる検索
    Optional<Order> findById(OrderId id);

    // 顧客IDによる検索
    List<Order> findByCustomerId(CustomerId customerId);

    // 保存（新規作成・更新どちらも対応）
    void save(Order order);

    // 削除
    void delete(OrderId id);

    // 次のID取得（採番）
    OrderId nextId();
}

インターフェース設計のポイント

リポジトリのメソッドは、ドメインの言葉で定義します。「SELECT文を実行する」ではなく「顧客IDで注文を検索する」のように、ビジネスの意図が明確になる命名を心がけましょう。

基本的なメソッド

  • 1
    findById: IDを指定して集約を取得します。存在しない場合はOptional.empty()を返します。
  • 2
    save: 集約を永続化します。新規作成と更新を区別せず、同じメソッドで処理します（Upsert）。
  • 3
    delete: 集約を削除します。IDを指定して削除対象を特定します。
  • 4
    検索メソッド: ビジネスで必要な検索条件に応じたメソッドを定義します（例: findByCustomerId、findByStatus）。

7.4 リポジトリの実装

リポジトリの実装クラスは、インフラストラクチャ層に配置します。ここで初めてデータベースアクセスの具体的な処理を記述します。

OrderRepositoryImpl.java (インフラ層) 
// インフラストラクチャ層の実装
public class OrderRepositoryImpl implements IOrderRepository {
    private final EntityManager entityManager;

    @Override
    public Optional<Order> findById(OrderId id) {
        // JPAを使った検索
        Order order = entityManager.find(Order.class, id.getValue());
        return Optional.ofNullable(order);
    }

    @Override
    public List<Order> findByCustomerId(CustomerId customerId) {
        // JPQLで条件検索
        return entityManager
            .createQuery("SELECT o FROM Order o WHERE o.customerId = :customerId", Order.class)
            .setParameter("customerId", customerId)
            .getResultList();
    }

    @Override
    public void save(Order order) {
        // 新規か更新かを自動判定
        if (entityManager.contains(order)) {
            entityManager.merge(order);
        } else {
            entityManager.persist(order);
        }
    }
}

注意: 集約境界の尊重

リポジトリは常に集約ルート単位で保存・取得します。集約内部のエンティティ（例: 注文明細）を個別に保存するメソッドは作りません。これにより、集約の整合性が保証されます。

7.5 テスタビリティの向上

リポジトリパターンの大きなメリットの1つが、テスタビリティの向上です。インターフェースと実装を分離することで、テスト用の実装（モックやスタブ）を簡単に差し替えられます。

InMemoryOrderRepository.java (テスト用) 
// テスト用のインメモリ実装
public class InMemoryOrderRepository implements IOrderRepository {
    // メモリ上にデータを保持
    private final Map<OrderId, Order> orders = new HashMap<>();

    @Override
    public Optional<Order> findById(OrderId id) {
        return Optional.ofNullable(orders.get(id));
    }

    @Override
    public void save(Order order) {
        // メモリ上のMapに保存（高速）
        orders.put(order.getId(), order);
    }

    // テスト用の便利メソッド
    public void clear() {
        orders.clear();
    }
}
OrderServiceTest.java 
public class OrderServiceTest {
    private IOrderRepository orderRepository;
    private OrderService orderService;

    @BeforeEach
    void setUp() {
        // インメモリ実装を使用（DBなしでテスト可能）
        orderRepository = new InMemoryOrderRepository();
        orderService = new OrderService(orderRepository);
    }

    @Test
    void testOrderCreation() {
        // 高速にテストを実行できる
        Order order = orderService.createOrder(...);

        // 検証
        Order saved = orderRepository.findById(order.getId()).get();
        assertEquals(OrderStatus.PENDING, saved.getStatus());
    }
}

テストの高速化

インメモリ実装を使うことで、データベース接続やトランザクション処理が不要になり、テストが数秒から数ミリ秒に短縮されます。これにより、TDD（テスト駆動開発）のサイクルを高速に回せます。

7.6 リポジトリ設計のベストプラクティス

1. 集約ごとに1つのリポジトリ

リポジトリは集約単位で定義します。集約内部のエンティティに対して個別のリポジトリは作りません。

良い設計

  • ● IOrderRepository（注文集約用）
  • ● ICustomerRepository（顧客集約用）
  • ● IProductRepository（商品集約用）

悪い設計

  • ● IOrderLineRepository（注文明細は集約内部）
  • ● IAddressRepository（住所は値オブジェクト）
  • ● IGenericRepository（汎用リポジトリは避ける）

2. ドメインの言葉でメソッドを命名

技術的な用語ではなく、ビジネスの言葉でメソッド名を付けます。

良い命名例 
// ビジネスの意図が明確
List<Order> findByCustomerId(CustomerId customerId);
List<Order> findPendingOrders();
List<Order> findOrdersShippedIn(DateRange dateRange);

3. 検索条件はドメインオブジェクトで表現

複雑な検索条件は、専用のクラス（仕様パターン）でカプセル化します。

仕様パターンの例 
// 検索条件を表すクラス
public class OrderSearchCriteria {
    private CustomerId customerId;
    private OrderStatus status;
    private DateRange orderDateRange;
    // ゲッター・セッター等
}

// リポジトリで使用
public interface IOrderRepository {
    List<Order> findByCriteria(OrderSearchCriteria criteria);
}

4. トランザクション管理は外部で行う

リポジトリ内部でトランザクションを開始・コミットせず、呼び出し側（アプリケーション層のサービス）で管理します。

トランザクション境界

複数のリポジトリ操作を1つのトランザクションでまとめる必要がある場合、アプリケーションサービスでトランザクション境界を定義します。リポジトリはその中で動作する部品として設計します。

理解度確認クイズ

  1. Q1. リポジトリパターンの主な目的は何ですか？

    回答を表示

    永続化の詳細をドメイン層から隠蔽し、データアクセスロジックをカプセル化することです。これにより、ドメインモデルの純粋性を保ち、テスタビリティを向上させ、永続化技術の変更を容易にします。

  2. Q2. リポジトリのインターフェースはどの層に配置しますか？また、実装クラスはどの層に配置しますか？

    回答を表示

    インターフェースはドメイン層に配置します。実装クラスはインフラストラクチャ層に配置します。これにより、ドメイン層がインフラストラクチャ層に依存することなく、依存性逆転の原則を実現できます。

  3. Q3. なぜ集約内部のエンティティに対して個別のリポジトリを作ってはいけないのですか？

    回答を表示

    集約は整合性の境界を表しており、常に集約ルート経由でアクセスする必要があります。集約内部のエンティティに個別のリポジトリを作ると、集約の不変条件を守れなくなり、データの整合性が崩れる可能性があります。

  4. Q4. テスト用にインメモリリポジトリを実装するメリットは何ですか？

    回答を表示

    データベース接続が不要になり、テストの実行速度が大幅に向上します（数秒→数ミリ秒）。また、テストデータの準備や後処理が簡単になり、TDDのサイクルを高速に回せるようになります。さらに、インターフェースが正しく定義されているかの検証にもなります。

この章のまとめ

  • リポジトリは集約の永続化と取得を担当し、データアクセスの詳細をドメイン層から隠蔽する
  • インターフェースをドメイン層、実装をインフラ層に配置することで依存性を逆転させる
  • 集約ごとに1つのリポジトリを定義し、集約境界を尊重する
  • テスト用のインメモリ実装を用意することで、テスタビリティが大幅に向上する
  • メソッド名はドメインの言葉で表現し、ビジネスの意図を明確にする
