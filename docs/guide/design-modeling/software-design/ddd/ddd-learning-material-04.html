エンティティ - 識別子を持つビジネスオブジェクト

DDDの戦術的パターンの学習を開始します。「同じ注文番号の注文は同一である」のように、識別子によってアイデンティティが決まるエンティティの概念と実装方法を学びます。

この章で学ぶこと

  • エンティティとは何か、その本質的な特徴を理解する
  • 識別子（ID）による同一性の判定方法を学ぶ
  • エンティティのライフサイクルと変更可能性を理解する
  • エンティティを適切に実装し、設計できるようになる

4.1 エンティティとは何か

エンティティは、DDDの戦術的パターンの中で最も基本的な概念の一つです。エンティティとは、識別子（ID）によってアイデンティティが決まるオブジェクトを指します。

例えば、ECサイトの「注文」を考えてみましょう。注文番号「ORD-12345」の注文と、注文番号「ORD-67890」の注文は、たとえ商品内容や金額が同じであっても、別々の注文として扱われます。このように、属性が同じでも識別子が異なれば別物とみなされるのがエンティティの特徴です。

識別子（ID）とは

識別子は、オブジェクトを一意に識別するための値です。注文番号、ユーザーID、商品コードなど、そのオブジェクトのライフサイクル全体で変わらない値を使用します。識別子は通常、システムが自動生成するか、ビジネス上の意味を持つ値を使用します。

4.2 エンティティの3つの特徴

エンティティには、以下の3つの重要な特徴があります。これらを理解することで、エンティティを適切に設計できるようになります。

① 同一性（Identity）- 識別子で判定する

エンティティの同一性は、識別子によって判定されます。属性が変わっても、識別子が同じであれば同じオブジェクトです。

例：

ユーザーID「USER-001」のユーザーは、名前を「田中太郎」から「田中花子」に変更しても、同じユーザーとして扱われます。

② ライフサイクル（Lifecycle）- 時間とともに変化する

エンティティは時間とともに状態が変化します。作成、更新、削除といったライフサイクルを持ちます。

例：

注文は「受付中」→「確定済」→「配送中」→「配送完了」とステータスが変化します。各段階でオブジェクトの状態は変わりますが、注文IDは変わりません。

③ 変更可能性（Mutability）- 属性を変更できる

エンティティは属性を変更できるオブジェクトです。ただし、ビジネスルールに従った変更のみが許可されます。

例：

注文の配送先住所を変更できますが、「確定済」の注文に対してのみ変更可能とするなど、ビジネスルールで制御します。

4.3 エンティティと値オブジェクトの違い

DDDでは、エンティティと値オブジェクトを明確に区別します。この区別を理解することで、適切なモデリングができるようになります。

観点 エンティティ 値オブジェクト
同一性の判定 識別子で判定 すべての属性で判定
変更可能性 変更可能（Mutable） 不変（Immutable）
ライフサイクル ある（作成→更新→削除） ない（作成したら変えない）
例 注文、ユーザー、商品 金額、住所、メールアドレス

判断基準

「このオブジェクトを追跡する必要があるか？」「時間とともに状態が変わるか？」と問いかけましょう。どちらかに「はい」と答えられる場合、エンティティとして設計すべきです。

4.4 エンティティの実装例

エンティティを実装する際の基本的なパターンを見ていきましょう。ここではJavaとPythonの例を示します。

Java実装例

Java 
// 注文エンティティ
public class Order {
    // 識別子（変更不可）
    private final OrderId id;

    // 属性（変更可能）
    private CustomerId customerId;
    private Money totalAmount;
    private OrderStatus status;
    private ShippingAddress shippingAddress;

    // コンストラクタ
    public Order(OrderId id, CustomerId customerId) {
        this.id = id;
        this.customerId = customerId;
        this.status = OrderStatus.PENDING;
    }

    // 同一性判定はIDで行う
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Order)) return false;
        Order other = (Order) o;
        return this.id.equals(other.id);
    }
}

Python実装例

Python 
from dataclasses import dataclass
from enum import Enum

class OrderStatus(Enum):
    PENDING = "受付中"
    CONFIRMED = "確定済"
    SHIPPED = "配送中"

@dataclass
class Order:
    # 識別子（変更不可）
    order_id: str
    customer_id: str

    # 属性（変更可能）
    total_amount: float = 0.0
    status: OrderStatus = OrderStatus.PENDING

    # 同一性判定はIDで行う
    def __eq__(self, other):
        if not isinstance(other, Order):
            return False
        return self.order_id == other.order_id

    # ビジネスロジック
    def confirm(self):
        if self.status != OrderStatus.PENDING:
            raise ValueError("受付中の注文のみ確定可能")
        self.status = OrderStatus.CONFIRMED

実装のポイント

  • 識別子は変更不可（final / 読み取り専用）にする
  • equals/hashCodeメソッドは識別子で判定する
  • ビジネスロジックはエンティティ内にメソッドとして実装する
  • 不正な状態変更を防ぐバリデーションを含める

4.5 エンティティ設計のベストプラクティス

① 識別子は不変にする

識別子は一度設定したら変更できないようにします。識別子が変わると、オブジェクトの追跡ができなくなります。

② ビジネスロジックを含める

エンティティには、そのオブジェクトに関連するビジネスロジックをメソッドとして実装します。データとロジックを一体化させます。

③ 不正な状態を防ぐ

コンストラクタやメソッド内でバリデーションを行い、不正な状態になることを防ぎます。常に正しい状態を保ちます。

④ 適切な粒度を保つ

エンティティが肥大化しないよう、責務を明確にします。必要に応じて値オブジェクトを活用し、複雑性を抑えます。

よくあるアンチパターン

  • 貧血ドメインモデル

    エンティティをデータの入れ物だけにして、ロジックをサービス層に書いてしまう。オブジェクト指向の利点が失われます。

  • 識別子の変更を許可

    識別子を変更可能にすると、オブジェクトの追跡ができなくなり、データの一貫性が損なわれます。

  • すべてをエンティティにする

    値オブジェクトで十分な概念までエンティティにすると、不必要な複雑性が生まれます。

実習：エンティティを設計してみよう

以下のシナリオに基づいて、「ユーザー」エンティティを設計してみましょう。

シナリオ：

  • ユーザーは一意のユーザーIDを持つ
  • 名前、メールアドレス、生年月日の情報を持つ
  • アカウントのステータス（アクティブ/停止中）を持つ
  • 名前とメールアドレスは変更できる
  • 停止中のアカウントはアクティブに戻せる
設計例を表示
Python 
from dataclasses import dataclass
from datetime import date
from enum import Enum

class AccountStatus(Enum):
    ACTIVE = "アクティブ"
    SUSPENDED = "停止中"

@dataclass
class User:
    user_id: str  # 識別子
    name: str
    email: str
    birth_date: date
    status: AccountStatus = AccountStatus.ACTIVE

    def change_name(self, new_name: str):
        if not new_name:
            raise ValueError("名前は空にできません")
        self.name = new_name

    def activate(self):
        self.status = AccountStatus.ACTIVE

    def __eq__(self, other):
        if not isinstance(other, User):
            return False
        return self.user_id == other.user_id

理解度確認クイズ

  1. Q1. エンティティの同一性は何によって判定されますか？

    回答を表示

    識別子（ID）によって判定されます。属性がすべて同じでも、識別子が異なれば別のオブジェクトとして扱われます。

  2. Q2. エンティティと値オブジェクトの最も大きな違いは何ですか？

    回答を表示

    エンティティは識別子で同一性を判定し変更可能ですが、値オブジェクトはすべての属性で同一性を判定し不変です。また、エンティティにはライフサイクルがありますが、値オブジェクトにはありません。

  3. Q3. エンティティの識別子を変更可能にすると、どのような問題が発生しますか？

    回答を表示

    オブジェクトの追跡ができなくなり、データの一貫性が損なわれます。また、他のオブジェクトとの関連が壊れる可能性があります。

  4. Q4. 「貧血ドメインモデル」とは何ですか？なぜ避けるべきですか？

    回答を表示

    貧血ドメインモデルとは、エンティティをデータの入れ物だけにして、ビジネスロジックを外部のサービス層に書いてしまう設計です。これはオブジェクト指向の利点（データとロジックのカプセル化）が失われ、コードの保守性が低下します。

この章のまとめ

  • エンティティは識別子によってアイデンティティが決まるオブジェクト
  • 同一性（識別子で判定）、ライフサイクル（時間で変化）、変更可能性の3つの特徴を持つ
  • 値オブジェクトとは同一性の判定方法と変更可能性が異なる
  • 識別子は不変にし、ビジネスロジックをエンティティ内に含める
  • 貧血ドメインモデルや識別子の変更許可などのアンチパターンを避ける
