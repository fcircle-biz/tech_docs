ドメインサービス - エンティティに属さないビジネスロジック

複数のエンティティにまたがる処理や、特定のエンティティに属さないビジネスロジックを扱うドメインサービスを学びます。「送金処理」「在庫確認」などの例を通じて、ドメインサービスの適切な使い方と、過度な使用を避ける方法を理解します。

この章で学ぶこと

  • ドメインサービスとは何か、どのような場面で使うのかを理解する
  • エンティティとドメインサービスの使い分け基準を習得する
  • 送金処理、在庫確認など具体的な実装例を学ぶ
  • 過度な使用による貧血ドメインモデルを避ける方法を知る

8.1 ドメインサービスとは何か

ドメインサービスは、エンティティや値オブジェクトに自然に属さないビジネスロジックを扱うオブジェクトです。DDDの戦術的パターンの一つで、複数の集約をまたがる処理や、概念的に特定のエンティティに属さない操作を表現します。

これまで学んだエンティティや値オブジェクトは、「注文」「顧客」「金額」など「モノ」を表現していました。一方、ドメインサービスは「送金する」「在庫を確認する」「重複をチェックする」といった「行為」や「処理」を表現します。

ドメインサービスの特徴

ドメインサービスは「無状態（ステートレス）」であることが基本です。つまり、インスタンス変数を持たず、メソッドのパラメータとして必要な情報を受け取り、処理結果を返します。これにより、再利用性が高く、テストしやすい設計になります。

8.2 なぜドメインサービスが必要なのか

すべてのビジネスロジックをエンティティに詰め込もうとすると、以下のような問題が発生します：

問題①: 不自然な責務の割り当て

「口座Aから口座Bへの送金」を口座エンティティのメソッドにすると、「どちらの口座の責務か？」という不自然な問題が発生します。

問題②: 集約境界を越えた依存

複数の集約をまたがる処理をエンティティに持たせると、集約の独立性が失われ、循環依存が発生する可能性があります。

ドメインサービスを使うことで、これらの問題を解決し、ビジネスロジックをドメイン層に保ちながら、自然で保守しやすい設計を実現できます。

8.3 ドメインサービスの具体例

ドメインサービスが使われる代表的な場面を、具体例とともに見ていきましょう。

例1: 送金処理

銀行システムで、口座Aから口座Bへ送金する処理は、どちらの口座エンティティに属するとも言えません。この場合、送金サービスとして独立させます。

MoneyTransferService.java 
// ドメインサービス: 送金処理
public class MoneyTransferService {

    // 送金処理を実行
    public void transfer(
        Account fromAccount,
        Account toAccount,
        Money amount
    ) {
        // ビジネスルール検証
        if (fromAccount.getBalance().lessThan(amount)) {
            throw new InsufficientBalanceException(
                "残高不足です"
            );
        }

        // 送金実行
        fromAccount.withdraw(amount);
        toAccount.deposit(amount);
    }
}

重要ポイント

送金サービスは、2つの口座エンティティを協調させる「オーケストレーター」の役割を果たします。エンティティ自身は単純な引き出し・預け入れのみを担当し、送金という複雑な処理はサービスが担います。

例2: 在庫確認サービス

ECサイトで、複数の倉庫をまたいで在庫を確認する処理も、ドメインサービスとして表現できます。

InventoryCheckService.java 
// ドメインサービス: 在庫確認
public class InventoryCheckService {

    // 複数倉庫の在庫を確認
    public boolean isAvailable(
        ProductId productId,
        Quantity requestedQty,
        List<Warehouse> warehouses
    ) {
        // 全倉庫の在庫数を合計
        Quantity totalStock = warehouses.stream()
            .map(w -> w.getStock(productId))
            .reduce(Quantity.zero(), Quantity::add);

        // 要求数量が在庫内か判定
        return totalStock.greaterThanOrEqual(requestedQty);
    }
}

例3: ユーザー名重複チェックサービス

ユーザー登録時に、既に同じユーザー名が存在しないかチェックする処理は、リポジトリを使った検索が必要なため、ドメインサービスとして実装します。

UserDuplicationCheckService.java 
// ドメインサービス: ユーザー重複チェック
public class UserDuplicationCheckService {

    private final UserRepository userRepository;

    public boolean isDuplicate(Username username) {
        // リポジトリで既存ユーザーを検索
        return userRepository.findByUsername(username)
            .isPresent();
    }
}

8.4 エンティティ vs ドメインサービスの判断基準

ビジネスロジックをエンティティに持たせるべきか、ドメインサービスにすべきか、判断に迷うことがあります。以下の基準を参考にしてください。

エンティティに持たせる場合

  • ● 単一のエンティティの状態変更: 「注文をキャンセルする」など
  • ● エンティティの不変条件を守る処理: 「注文明細を追加する」など
  • ● 概念的にそのエンティティに属する: 「顧客の住所を変更する」など

ドメインサービスにする場合

  • ● 複数のエンティティをまたがる処理: 「送金処理」など
  • ● 特定のエンティティに属さない処理: 「在庫確認」など
  • ● リポジトリを使った検索が必要: 「重複チェック」など

8.5 過度な使用を避ける - 貧血ドメインモデルの問題

ドメインサービスは便利ですが、使いすぎると「貧血ドメインモデル」という問題が発生します。これは、エンティティがデータの入れ物だけになり、ビジネスロジックがすべてサービスに集まってしまう状態です。

❌ 貧血ドメインモデル（悪い例） 

// エンティティがデータのみ
class Order {
    private Money total;

    // getter/setterのみ
    public Money getTotal() {
        return total;
    }
    public void setTotal(Money m) {
        this.total = m;
    }
}

✓ 適切な設計（良い例） 

// エンティティが振る舞いを持つ
class Order {
    private Money total;

    // ビジネスロジックを持つ
    public void cancel() {
        if (isShipped()) {
            throw new Exception(
                "発送済みはキャンセル不可"
            );
        }
        this.status = CANCELLED;
    }
}

注意

「迷ったらエンティティに配置する」を基本方針にしましょう。本当にエンティティに属さない場合のみ、ドメインサービスを使います。これにより、ドメインモデルが豊かな振る舞いを持ち、ビジネスロジックが分散しすぎることを防げます。

8.6 ドメインサービス vs アプリケーションサービス

DDDには「ドメインサービス」の他に「アプリケーションサービス」という概念もあります。名前が似ていますが、役割は全く異なります。

比較項目 ドメインサービス アプリケーションサービス
配置レイヤ ドメイン層 アプリケーション層
役割 ビジネスロジックの実行 ユースケースの調整・制御
具体例 送金処理、在庫確認 注文受付処理、ユーザー登録処理
依存関係 エンティティ、値オブジェクトに依存 ドメインサービス、リポジトリに依存

使い分けのポイント

ドメインサービスは「ビジネスの専門家と話す言葉」で書きます。アプリケーションサービスは「ユースケースの流れ」を調整します。「送金する」はビジネス用語なのでドメインサービス、「注文受付処理」はユースケースなのでアプリケーションサービスです。

実習 8-1: ドメインサービスの実装

ECサイトで、注文時に在庫を予約するドメインサービスを実装してみましょう。

手順

  1. 1 ProductクラスとWarehouseクラスを作成し、基本的な在庫管理機能を実装する
  2. 2 InventoryReservationServiceを作成し、複数倉庫から在庫を予約するメソッドを実装する
  3. 3 在庫不足時の例外処理を実装し、テストコードで動作確認する

実装のヒント

  • • ドメインサービスはステートレス（状態を持たない）にする
  • • 複数の倉庫エンティティを引数で受け取る設計にする
  • • ビジネスルールの検証をサービス内で行う

理解度確認クイズ

  1. Q1. ドメインサービスが必要になるのはどのような場面ですか？3つ挙げてください。

    回答を表示

    ① 複数のエンティティをまたがる処理（例: 送金処理）
    ② 特定のエンティティに概念的に属さない処理（例: 在庫確認）
    ③ リポジトリを使った検索が必要な処理（例: 重複チェック）

  2. Q2. 「貧血ドメインモデル」とは何ですか？なぜ避けるべきですか？

    回答を表示

    貧血ドメインモデルとは、エンティティがデータの入れ物だけになり、ビジネスロジックがすべてドメインサービスに集まってしまう状態です。これを避けるべき理由は、エンティティが本来持つべき振る舞いを失い、オブジェクト指向の利点が損なわれるためです。ビジネスロジックが分散しすぎて保守性が低下します。

  3. Q3. ドメインサービスとアプリケーションサービスの違いを説明してください。

    回答を表示

    ドメインサービスはドメイン層に配置され、ビジネスロジックそのものを実装します（例: 送金処理）。アプリケーションサービスはアプリケーション層に配置され、ユースケースの流れを調整・制御します（例: 注文受付処理）。ドメインサービスは「ビジネスの専門家と話す言葉」、アプリケーションサービスは「システムのユースケース」を表現します。

  4. Q4. 「注文をキャンセルする」処理は、エンティティとドメインサービスのどちらに配置すべきですか？理由も答えてください。

    回答を表示

    エンティティ（Orderクラス）に配置すべきです。理由は、「注文をキャンセルする」は単一のエンティティ（注文）の状態を変更する処理であり、概念的に注文エンティティに属するためです。ドメインサービスは複数のエンティティをまたがる処理や、特定のエンティティに属さない処理のために使います。

この章のまとめ

  • ドメインサービスは、エンティティや値オブジェクトに自然に属さないビジネスロジックを扱う
  • 複数のエンティティをまたがる処理、特定のエンティティに属さない処理で使用する
  • 送金処理、在庫確認、重複チェックなどが代表的な使用例
  • 過度な使用は「貧血ドメインモデル」を招くため、基本はエンティティに配置する
  • ドメインサービスはビジネスロジック、アプリケーションサービスはユースケース調整を担当する
