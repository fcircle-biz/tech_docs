ドメインイベント - システム内外への変更の通知

「注文が確定された」「在庫が減少した」など、ドメインで発生した重要な出来事を表現するドメインイベントを学びます。イベントの設計、発行、購読の方法を理解し、疎結合なシステム設計への応用を習得します。

この章で学ぶこと

  • ドメインイベントとは何か、なぜ必要なのかを理解する
  • ドメインイベントの設計方法と命名規則を習得する
  • イベントの発行（Publish）と購読（Subscribe）パターンを実装できるようになる
  • 疎結合なシステムを実現するイベント駆動設計の基礎を理解する

13.1 ドメインイベントとは何か

ドメインイベント（Domain Event）は、ドメインで発生した重要な出来事を表現するオブジェクトです。ビジネス的に意味のある何かが起こったことを他の部分に伝えるための仕組みです。

例えば、ECサイトでは以下のような出来事が重要です：

注文ドメインでのイベント例

  • ● 注文が確定された（OrderPlaced）
  • ● 注文がキャンセルされた（OrderCancelled）
  • ● 支払いが完了した（PaymentCompleted）

在庫ドメインでのイベント例

  • ● 在庫が減少した（StockDecreased）
  • ● 在庫切れが発生した（OutOfStock）
  • ● 在庫が補充された（StockReplenished）

イベントは「過去形」で命名する

ドメインイベントは「既に起きたこと」を表現するため、過去形（～された、～が完了した）で命名します。これにより、イベントが発生した時点で状態変更が完了していることが明確になります。

13.2 なぜドメインイベントが必要なのか

ドメインイベントを使うことで、以下のようなメリットが得られます：

ドメインイベントのメリット

  • 疎結合化: イベント発行側と購読側が直接依存しないため、システムが柔軟になる
  • ビジネスロジックの明確化: 重要な出来事がコードで明示的に表現される
  • 監査・ログ記録: 何が起こったかを時系列で記録できる
  • 非同期処理: イベントをキューに入れて後で処理できる

イベントを使わない場合の問題

イベントを使わずに直接処理を呼び出すと、以下のような問題が発生します：

密結合な設計の例（問題あり） 

// 注文確定時に様々な処理を直接呼び出す
public class Order {
    public void place() {
        // 注文確定
        this.status = OrderStatus.PLACED;

        // 在庫サービスを直接呼び出し
        inventoryService.decreaseStock(this.items);

        // 配送サービスを直接呼び出し
        shippingService.createShipment(this);

        // メール通知サービスを直接呼び出し
        emailService.sendOrderConfirmation(this);

        // ポイントサービスを直接呼び出し
        pointService.addPoints(this.customerId);
    }
}

この設計の問題点

  • 注文クラスが多数のサービスに依存している（密結合）
  • 新しい処理を追加するたびに注文クラスを変更する必要がある
  • 各サービスの処理が失敗した時のエラーハンドリングが複雑
  • テストが困難（すべてのサービスをモック化する必要がある）

イベント駆動な設計の例（改善版） 

// 注文確定時にイベントを発行するだけ
public class Order {
    public void place() {
        // 注文確定
        this.status = OrderStatus.PLACED;

        // イベントを発行（他のサービスへの依存なし）
        DomainEventPublisher.publish(
            new OrderPlacedEvent(this.orderId, this.customerId, this.items)
        );
    }
}

// 各サービスは独立してイベントを購読
@EventHandler
public class InventoryEventHandler {
    public void handle(OrderPlacedEvent event) {
        inventoryService.decreaseStock(event.getItems());
    }
}

@EventHandler
public class ShippingEventHandler {
    public void handle(OrderPlacedEvent event) {
        shippingService.createShipment(event.getOrderId());
    }
}

改善のポイント

  • 注文クラスは他のサービスに依存しない（疎結合）
  • 新しい処理を追加しても注文クラスを変更しなくて良い
  • 各イベントハンドラーは独立してエラーハンドリング可能
  • テストが容易（イベント発行を確認するだけ）

13.3 ドメインイベントの設計

ドメインイベントを設計する際には、以下の点に注意します：

イベントの構造

ドメインイベントは通常、以下の要素を持ちます：

必須の属性

  • ● イベントID: イベントを一意に識別するID
  • ● 発生日時: イベントがいつ発生したか
  • ● 集約ID: どの集約で起きたイベントか

ドメイン固有の属性

  • ● ビジネス情報: イベント処理に必要な情報
  • ● 変更前の状態: 必要に応じて含める
  • ● 変更後の状態: 必要に応じて含める

ドメインイベントの実装例 

// 注文確定イベント
public class OrderPlacedEvent implements DomainEvent {
    private final String eventId;        // イベントID
    private final LocalDateTime occurredOn; // 発生日時
    private final String orderId;        // 注文ID（集約ID）
    private final String customerId;     // 顧客ID
    private final List<OrderItem> items; // 注文明細
    private final Money totalAmount;     // 合計金額

    public OrderPlacedEvent(String orderId, String customerId,
                           List<OrderItem> items, Money totalAmount) {
        this.eventId = UUID.randomUUID().toString();
        this.occurredOn = LocalDateTime.now();
        this.orderId = orderId;
        this.customerId = customerId;
        this.items = new ArrayList<>(items);
        this.totalAmount = totalAmount;
    }

    // Getterのみ（イベントは不変）
    public String getEventId() { return eventId; }
    public LocalDateTime getOccurredOn() { return occurredOn; }
    public String getOrderId() { return orderId; }
    public String getCustomerId() { return customerId; }
    public List<OrderItem> getItems() { return items; }
    public Money getTotalAmount() { return totalAmount; }
}

イベントは不変（Immutable）にする

ドメインイベントは「過去に起きたこと」を表すため、一度作成したら変更できないようにします。すべてのフィールドをfinalにし、Setterを提供しません。

13.4 イベントの発行と購読パターン

ドメインイベントを活用するには、発行（Publish）と購読（Subscribe）の仕組みが必要です。これを「Pub/Subパターン」と呼びます。

イベント発行・購読のフロー

sequenceDiagram participant Order as 注文集約 participant Publisher as イベント発行者 participant Handler1 as 在庫ハンドラー participant Handler2 as 配送ハンドラー participant Handler3 as 通知ハンドラー Order->>Order: 注文確定処理 Order->>Publisher: OrderPlacedEventを発行 Publisher->>Handler1: イベントを通知 Publisher->>Handler2: イベントを通知 Publisher->>Handler3: イベントを通知 Handler1->>Handler1: 在庫減少処理 Handler2->>Handler2: 配送手配処理 Handler3->>Handler3: メール送信処理

イベント発行者（Publisher）の実装

// シンプルなイベント発行者
public class DomainEventPublisher {
    private static final List<DomainEventHandler> handlers = new ArrayList<>();

    // ハンドラーを登録
    public static void subscribe(DomainEventHandler handler) {
        handlers.add(handler);
    }

    // イベントを発行
    public static void publish(DomainEvent event) {
        for (DomainEventHandler handler : handlers) {
            if (handler.canHandle(event)) {
                handler.handle(event);
            }
        }
    }
}

イベントハンドラー（Subscriber）の実装

// イベントハンドラーのインターフェース
public interface DomainEventHandler<T extends DomainEvent> {
    boolean canHandle(DomainEvent event);
    void handle(T event);
}

// 在庫減少ハンドラーの実装
public class InventoryEventHandler
    implements DomainEventHandler<OrderPlacedEvent> {

    private final InventoryService inventoryService;

    @Override
    public boolean canHandle(DomainEvent event) {
        return event instanceof OrderPlacedEvent;
    }

    @Override
    public void handle(OrderPlacedEvent event) {
        // 注文確定イベントを受け取って在庫を減らす
        inventoryService.decreaseStock(event.getItems());
    }
}

13.5 実践例: ECサイトでのドメインイベント活用

ECサイトの注文処理で、ドメインイベントを使った実装例を見てみましょう。

注文確定時のイベントフロー

flowchart TB A[顧客が注文確定] --> B[注文集約] B --> C{OrderPlacedEvent発行} C --> D1[在庫ハンドラー] C --> D2[配送ハンドラー] C --> D3[通知ハンドラー] C --> D4[ポイントハンドラー] D1 --> E1[在庫減少] D2 --> E2[配送伝票作成] D3 --> E3[確認メール送信] D4 --> E4[ポイント付与] E1 --> F{在庫不足?} F -->|はい| G[OutOfStockEvent発行] F -->|いいえ| H[処理完了] G --> I[発注ハンドラー] I --> J[自動発注]

統合例: 注文から発送までの一連の流れ

// 1. 注文集約でイベントを発行
public class Order {
    public void place() {
        // ビジネスルール検証
        if (this.items.isEmpty()) {
            throw new IllegalStateException("注文明細が空です");
        }

        // 注文確定
        this.status = OrderStatus.PLACED;

        // イベント発行
        DomainEventPublisher.publish(
            new OrderPlacedEvent(orderId, customerId, items, totalAmount)
        );
    }
}

// 2. 各ハンドラーが独立して処理
@Component
public class ShippingEventHandler
    implements DomainEventHandler<OrderPlacedEvent> {

    public void handle(OrderPlacedEvent event) {
        // 配送伝票を作成
        Shipment shipment = new Shipment(event.getOrderId());
        shipmentRepository.save(shipment);

        // 配送伝票作成イベントを発行
        DomainEventPublisher.publish(
            new ShipmentCreatedEvent(shipment.getId(), event.getOrderId())
        );
    }
}

13.6 イベントストーミング - イベントでビジネスを理解する

イベントストーミングは、ビジネスプロセスを「ドメインイベント」の視点で洗い出すワークショップ手法です。DDDを実践する際の強力なツールです。

イベントストーミングの進め方

  1. ドメインイベントを洗い出す: オレンジ色の付箋で「～された」という出来事を書き出す
  2. 時系列に並べる: イベントを発生順に壁に貼っていく
  3. コマンドを追加: 青色の付箋でイベントを引き起こす「アクション」を書く
  4. 集約を識別する: 黄色の付箋で関連するイベント群をまとめる
  5. 境界を見つける: 境界づけられたコンテキストの境界を引く

イベントストーミングの効果

開発者とビジネス担当者が同じ場所で付箋を貼りながら議論することで、ドメインへの理解が深まり、ユビキタス言語が自然に形成されます。また、システムの複雑な部分や境界が可視化されます。

13.7 イベント駆動アーキテクチャ（EDA）への発展

ドメインイベントの考え方は、イベント駆動アーキテクチャ（Event-Driven Architecture / EDA）へと発展します。

同期的なイベント処理

同じプロセス内で即座に処理

  • ✓ 実装がシンプル
  • ✓ トランザクション管理が容易
  • ✗ 処理が遅くなる可能性

非同期的なイベント処理

メッセージキューを使って後で処理

  • ✓ レスポンスが速い
  • ✓ スケーラビリティが高い
  • ✗ 実装が複雑

メッセージキューを使った非同期イベント処理

flowchart LR A[注文サービス] -->|OrderPlacedEvent| B[メッセージキュー
RabbitMQ/Kafka] B -->|イベント配信| C[在庫サービス] B -->|イベント配信| D[配送サービス] B -->|イベント配信| E[通知サービス] C --> F[在庫減少] D --> G[配送手配] E --> H[メール送信]

マイクロサービスとの親和性

ドメインイベントを使うことで、マイクロサービス間の連携が疎結合になります。各サービスは他のサービスの内部実装を知らなくても、イベントを通じて協調動作できます。

13.8 ドメインイベントのベストプラクティス

ベストプラクティス

  • ● イベント名は過去形で明確に（OrderPlaced, PaymentCompleted）
  • ● イベントは不変（Immutable）にする
  • ● イベントにビジネス的意味を持たせる
  • ● 必要な情報だけを含める（肥大化させない）
  • ● イベントハンドラーは冪等性を保つ

避けるべきアンチパターン

  • ● 技術的な名前をつける（DataSaved, RecordUpdated）
  • ● イベントを変更可能にする
  • ● イベントに処理ロジックを含める
  • ● すべての変更をイベント化する（過剰設計）
  • ● イベントハンドラーで別のイベントに依存する

イベントの粒度に注意

すべての変更をイベント化する必要はありません。ビジネス的に重要な出来事だけをイベントにしましょう。例えば「住所が変更された」はイベント化すべきですが、「画面のボタンがクリックされた」は技術的な出来事であり、ドメインイベントではありません。

理解度確認クイズ

  1. Q1. ドメインイベントを使う最大のメリットは何ですか？

    回答を表示

    疎結合化です。イベント発行側と購読側が直接依存しないため、システムが柔軟になり、新しい機能の追加や変更が容易になります。また、ビジネスロジックが明確になり、監査ログとしても活用できます。

  2. Q2. ドメインイベントを過去形で命名する理由は何ですか？

    回答を表示

    ドメインイベントは「既に起きたこと」を表現するためです。イベントが発行される時点で状態変更が完了していることを明確にします。例: OrderPlaced（注文された）、PaymentCompleted（支払いが完了した）

  3. Q3. イベントストーミングとは何ですか？また、その効果は何ですか？

    回答を表示

    ビジネスプロセスを「ドメインイベント」の視点で洗い出すワークショップ手法です。開発者とビジネス担当者が付箋を使って議論することで、ドメインへの理解が深まり、ユビキタス言語が自然に形成されます。システムの複雑な部分や境界も可視化されます。

  4. Q4. ドメインイベントを不変（Immutable）にする理由は何ですか？

    回答を表示

    イベントは「過去に起きたこと」を表すため、後から変更されるべきではありません。不変にすることで、イベントの内容が改ざんされず、監査ログとしても信頼性が保たれます。また、並行処理でも安全に使用できます。

  5. Q5. 同期的なイベント処理と非同期的なイベント処理の使い分けの基準は何ですか？

    回答を表示

    同期処理は実装がシンプルで、トランザクション管理が容易ですが、処理が遅くなる可能性があります。非同期処理はレスポンスが速くスケーラビリティが高いですが、実装が複雑になります。ビジネス要件（レスポンス時間、処理量、整合性の要求度合い）に応じて選択します。
