DDDと他の設計パターンの組み合わせ

DDDとCQRS（コマンドクエリ責務分離）、イベントソーシング、マイクロサービスなどの組み合わせ方を学びます。それぞれのパターンがDDDをどのように補完するか、いつ採用すべきかを理解し、より高度な設計への道筋を把握します。

この章で学ぶこと

  • DDDとCQRSを組み合わせるメリットと実装パターンを理解する
  • イベントソーシングとDDDの相性と、採用する際の考慮点を学ぶ
  • DDDがマイクロサービスアーキテクチャをどう支援するかを理解する
  • ヘキサゴナルアーキテクチャとクリーンアーキテクチャの活用方法を把握する

16.1 DDD + CQRS（コマンドクエリ責務分離）

CQRS（Command Query Responsibility Segregation）は、データの更新（Command）と参照（Query）の責務を分離するアーキテクチャパターンです。DDDと組み合わせることで、より柔軟で拡張性の高いシステムを構築できます。

CQRSの基本概念

CRUDの世界では、読み書きで同じモデルを使用しますが、CQRSでは書き込み用モデル（Command Model）と読み取り用モデル（Query Model）を分離します。これにより、それぞれの責務に最適化された設計が可能になります。

DDDとCQRSを組み合わせるメリット

コマンド側（書き込み）

  • ● DDDの集約でビジネスルールを厳密に保護
  • ● トランザクション整合性を維持
  • ● ドメインイベントで変更を通知

クエリ側（読み取り）

  • ● 表示用に最適化された非正規化データ
  • ● 複雑な結合を避けた高速な読み取り
  • ● 画面ごとに専用のデータモデルを用意可能
TypeScript 
// コマンド側: DDDの集約を使ってビジネスルールを保護
class Order {
  // ビジネスルールを厳密に実装
  confirm(): void {
    if (this.items.length === 0) {
      throw new Error('注文明細が空です');
    }
    if (this.status !== 'DRAFT') {
      throw new Error('下書き状態の注文のみ確定できます');
    }
    this.status = 'CONFIRMED';
    // ドメインイベント発行
    this.addDomainEvent(new OrderConfirmedEvent(this.id));
  }
}

// クエリ側: 表示用に最適化されたモデル
interface OrderListItem {
  orderId: string;
  customerName: string; // 正規化を解除して結合済み
  totalAmount: number;
  itemCount: number;
  statusText: string; // 画面表示用のテキスト
  formattedDate: string; // フォーマット済みの日付
}

16.2 DDD + イベントソーシング

イベントソーシングは、エンティティの状態変更をイベントのシーケンスとして保存する手法です。現在の状態だけでなく、「どのような変更が行われたか」の履歴をすべて保持します。

DDDとイベントソーシングのシナジー

相性が良い理由

  • ドメインイベントが自然に保存される: DDDで発行したドメインイベントがそのまま永続化される
  • 完全な監査証跡: ビジネス上重要な「誰が・いつ・何をしたか」を完全に記録
  • 時間旅行デバッグ: 過去の任意の時点での状態を再現可能
  • ビジネス分析: イベント履歴から業務プロセスの分析が可能
TypeScript - イベントソーシングの実装例 
// イベント定義
interface DomainEvent {
  eventId: string;
  aggregateId: string;
  occurredAt: Date;
}

class OrderCreatedEvent implements DomainEvent {
  constructor(
    public eventId: string,
    public aggregateId: string,
    public customerId: string,
    public occurredAt: Date
  ) {}
}

class OrderConfirmedEvent implements DomainEvent {
  constructor(
    public eventId: string,
    public aggregateId: string,
    public occurredAt: Date
  ) {}
}

// 集約: イベントから状態を再構築
class Order {
  private events: DomainEvent[] = [];

  // イベントリプレイで状態を復元
  static fromEvents(events: DomainEvent[]): Order {
    const order = new Order();
    events.forEach(event => order.apply(event));
    return order;
  }

  private apply(event: DomainEvent): void {
    if (event instanceof OrderCreatedEvent) {
      this.id = event.aggregateId;
      this.customerId = event.customerId;
    } else if (event instanceof OrderConfirmedEvent) {
      this.status = 'CONFIRMED';
    }
  }
}

イベントソーシングの注意点

  • • イベントのスキーマ変更管理が複雑になる（バージョニング必須）
  • • イベントリプレイに時間がかかる場合がある（スナップショット併用が必要）
  • • 学習コストが高く、チームの理解が必要
  • • すべてのシステムに適しているわけではない

16.3 DDD + マイクロサービスアーキテクチャ

DDDの境界づけられたコンテキストは、マイクロサービスの境界を決定する際の優れた指針となります。ビジネスドメインに基づいたサービス分割により、疎結合で変更に強いシステムを構築できます。

DDDがマイクロサービスを支援する方法

サービス境界の決定

境界づけられたコンテキストがそのままマイクロサービスの境界になる。ビジネス的に意味のある単位で分割できるため、適切な粒度のサービスを設計できる。

データの独立性

各コンテキストが独自のデータベースを持つことで、サービス間の結合度を下げる。集約の境界がトランザクション境界となり、整合性管理が明確になる。

チーム編成

コンテキスト単位でチームを編成することで、各チームが独立して開発・デプロイ可能。ユビキタス言語がチーム内のコミュニケーションを円滑にする。

サービス間連携

ドメインイベントを使った非同期通信により、サービス間の疎結合を実現。コンテキストマップで連携パターンを明確化できる。

段階的なマイクロサービス化

最初からマイクロサービスで始める必要はありません。モノリシックなアプリケーションの中で境界づけられたコンテキストを明確にしておけば、将来的にマイクロサービスに分割する際の設計が容易になります。

16.4 DDD + ヘキサゴナルアーキテクチャ / クリーンアーキテクチャ

DDDは「何を設計するか」を定義しますが、ヘキサゴナルアーキテクチャやクリーンアーキテクチャは「どのように構造化するか」を定義します。これらを組み合わせることで、テスタブルで保守性の高いシステムを実現できます。

DDDとアーキテクチャパターンの統合

ヘキサゴナルアーキテクチャ

  • ● 中心にドメインモデルを配置
  • ● ポートでインターフェースを定義
  • ● アダプターで外部システムと接続
  • ● フレームワークやDBへの依存を排除

クリーンアーキテクチャ

  • ● 依存性の方向が内側（ドメイン）に向かう
  • ● レイヤー分離を明確化
  • ● テスタビリティが向上
  • ● 技術選択を後回しにできる
TypeScript - ポート＆アダプターの実装例 
// ポート（インターフェース）: ドメイン層が定義
interface OrderRepository {
  save(order: Order): Promise<void>;
  findById(id: string): Promise<Order | null>;
}

// ドメインモデル: ポートに依存（具象には依存しない）
class OrderService {
  constructor(private repository: OrderRepository) {}

  async confirmOrder(orderId: string): Promise<void> {
    const order = await this.repository.findById(orderId);
    if (!order) throw new Error('注文が見つかりません');
    order.confirm(); // ドメインロジック
    await this.repository.save(order);
  }
}

  }

  async findById(id: string): Promise<Order | null> {
    // PostgreSQL固有の実装
  }
}

16.5 パターン選択のガイドライン

これらのパターンをすべて採用する必要はありません。プロジェクトの特性に応じて、適切なパターンを選択しましょう。

パターン選択マトリックス

パターン 採用すべき場面 避けるべき場面 複雑度
DDD基本 複雑なビジネスルールがある 単純なCRUDアプリ ★★☆☆☆
CQRS 読み書きの性能要件が大きく異なる 小規模システム ★★★☆☆
イベントソーシング 監査証跡が必須、時系列分析が重要 履歴不要なシステム ★★★★☆
マイクロサービス 大規模、複数チーム、独立デプロイ必須 小規模チーム、モノリスで十分 ★★★★★
ヘキサゴナル テスタビリティ重視、フレームワーク独立性 プロトタイプ、短期プロジェクト ★★★☆☆

推奨される組み合わせ

  • • DDD + ヘキサゴナルアーキテクチャ: 中規模以上のほとんどのプロジェクトで推奨
  • • DDD + CQRS: 読み書きの性能要件が異なる場合に有効
  • • DDD + CQRS + イベントソーシング: 監査証跡が重要な金融・医療系システム
  • • DDD + マイクロサービス: 大規模システム、複数チーム編成の場合

実装のステップバイステップ

  1. 1
    DDDの基本から始める

    まずはユビキタス言語、エンティティ、値オブジェクト、集約などDDDの基本パターンを適用する

  2. 2
    レイヤー構造を整理する

    ヘキサゴナルアーキテクチャやクリーンアーキテクチャでドメイン層を保護する

  3. 3
    必要に応じてCQRSを導入

    読み書きの要件が大きく異なる場合に、段階的にCQRSパターンを適用する

  4. 4
    高度なパターンは慎重に検討

    イベントソーシングやマイクロサービスは本当に必要な場合のみ採用する

理解度確認クイズ

  1. Q1. CQRSとは何の略で、どのような責務を分離しますか？

    回答を表示

    CQRS（Command Query Responsibility Segregation / コマンドクエリ責務分離）は、データの更新（Command）と参照（Query）の責務を分離するパターンです。書き込み用モデルと読み取り用モデルを別々に設計することで、それぞれの要件に最適化できます。

  2. Q2. イベントソーシングの主なメリットを3つ挙げてください。

    回答を表示

    ① 完全な監査証跡（すべての変更履歴を保存）
    ② 時間旅行デバッグ（過去の任意の時点の状態を再現可能）
    ③ ビジネス分析（イベント履歴から業務プロセスを分析可能）

  3. Q3. DDDの「境界づけられたコンテキスト」は、マイクロサービスアーキテクチャとどのように関連しますか？

    回答を表示

    境界づけられたコンテキストは、マイクロサービスの境界を決定する際の優れた指針となります。各コンテキストが独立したマイクロサービスとなることで、ビジネスドメインに基づいた自然な分割が可能になり、疎結合で変更に強いシステムを構築できます。

  4. Q4. ヘキサゴナルアーキテクチャにおける「ポート」と「アダプター」の役割を説明してください。

    回答を表示

    「ポート」はドメイン層が定義するインターフェース（抽象）で、外部との通信方法を規定します。「アダプター」はポートを実装する具象クラスで、実際に外部システム（データベース、外部API、UI等）と接続します。これにより、ドメインロジックを外部技術から独立させることができます。

この章のまとめ

  • DDDとCQRSを組み合わせることで、書き込みはビジネスルール重視、読み取りは性能重視の設計が可能
  • イベントソーシングはDDDのドメインイベントと相性が良く、完全な監査証跡を実現できる
  • 境界づけられたコンテキストがマイクロサービスの自然な境界となり、適切な粒度でサービスを分割できる
  • ヘキサゴナルアーキテクチャやクリーンアーキテクチャでドメイン層を外部技術から保護できる
  • すべてのパターンを採用する必要はなく、プロジェクトの特性に応じて適切に選択することが重要
