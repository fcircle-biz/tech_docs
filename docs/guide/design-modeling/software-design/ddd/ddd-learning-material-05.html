値オブジェクト - 属性の組み合わせで意味を持つオブジェクト

「金額」「住所」「メールアドレス」など、その値自体が意味を持つ値オブジェクトを学びます。不変性、等価性、自己検証などの特性を理解し、エンティティとの使い分け方を習得します。コードの品質を高める値オブジェクトの効果的な活用法を学びます。

この章で学ぶこと

  • 値オブジェクトとは何か、その基本概念を理解する
  • 値オブジェクトの3つの重要な特性（不変性、等価性、自己検証）を習得する
  • エンティティと値オブジェクトの使い分け方を身につける
  • 値オブジェクトを実装し、コードの品質を向上させる方法を学ぶ

5.1 値オブジェクトとは何か

値オブジェクト（Value Object）は、その値自体が意味を持つオブジェクトです。例えば、「金額」「住所」「メールアドレス」「日付範囲」など、複数の属性を組み合わせて一つの概念を表現します。

値オブジェクトは、第4章で学んだエンティティと並ぶ、DDDの重要な戦術的パターンの一つです。エンティティが「識別子によってアイデンティティを持つ」のに対し、値オブジェクトは「値そのものが意味を持つ」という点が大きな違いです。

具体例：金額（Money）

「1000円」という金額を表現する際、単に数値「1000」だけでは不十分です。「金額（amount）」と「通貨（currency）」の組み合わせで初めて「1000円」という意味になります。この「金額 + 通貨」をまとめたものが値オブジェクトです。

値オブジェクトの具体例

金額（Money）

属性: 金額、通貨

例: 1000円、$50、€30

住所（Address）

属性: 郵便番号、都道府県、市区町村、番地

例: 〒100-0001 東京都千代田区千代田1-1

メールアドレス（Email）

属性: メールアドレス文字列

例: user@example.com（形式検証を含む）

日付範囲（DateRange）

属性: 開始日、終了日

例: 2025-01-01 ～ 2025-12-31

5.2 値オブジェクトの3つの重要な特性

値オブジェクトには、3つの重要な特性があります。これらの特性を正しく実装することで、堅牢で保守しやすいコードを書くことができます。

特性①: 不変性（Immutability）

値オブジェクトは、一度作成したら内容を変更できないという特性を持ちます。値を変更したい場合は、新しい値オブジェクトを作成します。

理由:

  • ● 予期しない変更を防ぎ、バグを減らせる
  • ● マルチスレッド環境でも安全に使える
  • ● 値の同一性が保証される
Money.java - 不変性の実装 
// 値オブジェクトは不変（Immutable）
public final class Money {
    private final BigDecimal amount;  // finalで変更不可
    private final String currency;    // finalで変更不可

    public Money(BigDecimal amount, String currency) {
        this.amount = amount;
        this.currency = currency;
    }

    // 値を変更するのではなく、新しいインスタンスを返す
    public Money add(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new IllegalArgumentException("通貨が異なります");
        }
        return new Money(
            this.amount.add(other.amount),
            this.currency
        );
    }
}

特性②: 等価性（Equality）

値オブジェクトは、すべての属性が同じであれば同じ値として扱います。識別子（ID）ではなく、値そのもので比較します。

例:

「1000円」という金額オブジェクトが2つあった場合、どちらも「金額=1000、通貨=JPY」であれば、同じ値として扱います。オブジェクトのインスタンスが異なっていても、値が同じなら等しいと判定します。

Money.java - 等価性の実装 
// equals()とhashCode()をオーバーライド
@Override
public boolean equals(Object obj) {
    if (this == obj) return true;
    if (!(obj instanceof Money)) return false;
    Money other = (Money) obj;
    // すべての属性を比較
    return this.amount.equals(other.amount) &&
           this.currency.equals(other.currency);
}

@Override
public int hashCode() {
    return Objects.hash(amount, currency);
}

特性③: 自己検証（Self-Validation）

値オブジェクトは、自身の妥当性を保証します。コンストラクタで検証を行い、無効な値を持つオブジェクトを作成できないようにします。

メリット:

  • ● 値オブジェクトが存在すれば必ず正しい状態であると保証される
  • ● 使う側で毎回バリデーションする必要がなくなる
  • ● ビジネスルールが値オブジェクト内に集約される
Email.java - 自己検証の実装 
public final class Email {
    private final String value;

    public Email(String value) {
        // コンストラクタで検証
        if (value == null || value.isEmpty()) {
            throw new IllegalArgumentException(
                "メールアドレスは必須です");
        }
        if (!value.matches("^[\\w.-]+@[\\w.-]+\\.[a-z]{2,}$")) {
            throw new IllegalArgumentException(
                "メールアドレスの形式が不正です");
        }
        this.value = value;
    }
    // Emailオブジェクトが存在すれば、必ず正しい形式
}

5.3 エンティティと値オブジェクトの使い分け

エンティティ（第4章）と値オブジェクトは、どちらもDDDの重要な要素ですが、使い分けの基準を理解することが重要です。

エンティティ vs 値オブジェクト

観点 エンティティ 値オブジェクト
同一性 識別子（ID）で判断 すべての属性の値で判断
変更可能性 可変（状態が変化する） 不変（変更不可）
ライフサイクル 長期間存在し、追跡される 短期的、使い捨て可能
具体例 注文、顧客、商品 金額、住所、メールアドレス
質問 「誰?」「どれ?」 「いくら?」「何?」

判断のポイント

「時間が経過しても同じものとして追跡する必要があるか？」と問いかけてみましょう。答えが「はい」ならエンティティ、「いいえ」なら値オブジェクトです。例えば、「顧客」は住所が変わっても同じ顧客として追跡しますが、「住所」自体は変わったら別の住所になります。

5.4 値オブジェクトの実装例

実際に値オブジェクトを実装する例を見てみましょう。ECサイトの「住所（Address）」値オブジェクトを作成します。

実習 5-1: 住所（Address）値オブジェクトの実装

値オブジェクトの3つの特性を持つAddressクラスを実装します。

要件

  • ● 郵便番号、都道府県、市区町村、番地を持つ
  • ● すべてのフィールドはfinalで不変にする
  • ● 郵便番号の形式をコンストラクタで検証する
  • ● equals()とhashCode()を実装する
Address.java 
public final class Address {
    // 不変性: すべてfinal
    private final String postalCode;
    private final String prefecture;
    private final String city;
    private final String street;

    public Address(String postalCode, String prefecture,
                   String city, String street) {
        // 自己検証
        if (!postalCode.matches("\\d{3}-\\d{4}")) {
            throw new IllegalArgumentException(
                "郵便番号はXXX-XXXX形式です");
        }
        this.postalCode = postalCode;
        this.prefecture = prefecture;
        this.city = city;
        this.street = street;
    }
}
Address.java - 等価性の実装 
    // 等価性: すべての属性を比較
    @Override
    public boolean equals(Object obj) {
        if (!(obj instanceof Address)) return false;
        Address other = (Address) obj;
        return this.postalCode.equals(other.postalCode) &&
               this.prefecture.equals(other.prefecture) &&
               this.city.equals(other.city) &&
               this.street.equals(other.street);
    }

    @Override
    public int hashCode() {
        return Objects.hash(
            postalCode, prefecture, city, street);
    }
}

5.5 値オブジェクトがもたらす効果

値オブジェクトを適切に使うことで、コードの品質が大幅に向上します。

型安全性の向上

プリミティブ型（String、int等）の代わりに値オブジェクトを使うことで、型の間違いをコンパイル時に検出できます。例えば、「金額」と「数量」を間違えて渡すことを防げます。

ビジネスロジックの集約

値に関するビジネスルール（検証、計算等）を値オブジェクト内に集約できるため、重複コードが減り、変更が容易になります。

ドメイン知識の明示化

「金額」「住所」といった概念が明示的にクラスとして表現されるため、コードの可読性が向上し、ドメインの理解が深まります。

バグの削減

不変性により予期しない変更が防がれ、自己検証により無効な値が存在しなくなるため、バグが大幅に減少します。

値オブジェクト導入前後の比較

導入前（プリミティブ型） 

// 問題が多いコード
class Customer {
    String name;
    String email;  // 検証なし
    String postalCode;
    String prefecture;
    String city;
    String street;
}

// 使う側で毎回検証が必要
if (!email.matches("...")) {
    // エラー
}

導入後（値オブジェクト） 

// 安全で明確なコード
class Customer {
    Name name;
    Email email;  // 自己検証済み
    Address address;
}

// 値オブジェクトが存在すれば
// 常に正しい状態が保証される
Email email = new Email(
    "test@example.com");
// これだけでOK!

5.6 値オブジェクト実装時のよくある間違い

つまずきポイント

間違い①: セッターメソッドを作ってしまう

問題: 値オブジェクトに setXxx() メソッドを作ると不変性が失われます。

解決方法: 値を変更したい場合は、新しい値オブジェクトを返すメソッドを作成します（例: withXxx()、add()等）。

間違い②: equals()とhashCode()を実装し忘れる

問題: これらを実装しないと、値が同じでも異なるオブジェクトとして扱われます。

解決方法: 必ずequals()とhashCode()をオーバーライドし、すべての属性を比較対象に含めます。

間違い③: すべてを値オブジェクトにしてしまう

問題: 単純な文字列や数値まで値オブジェクト化すると、逆に複雑になります。

解決方法: ビジネスルールを持つ概念や、複数の属性を組み合わせる場合にのみ値オブジェクトを使います。

理解度確認クイズ

  1. Q1. 値オブジェクトの3つの重要な特性を挙げてください。

    回答を表示

    ① 不変性（Immutability）: 一度作成したら変更できない
    ② 等価性（Equality）: すべての属性が同じであれば同じ値
    ③ 自己検証（Self-Validation）: 自身の妥当性を保証する

  2. Q2. エンティティと値オブジェクトを区別する判断基準は何ですか？

    回答を表示

    「時間が経過しても同じものとして追跡する必要があるか？」が判断基準です。追跡が必要ならエンティティ（識別子で判断）、不要なら値オブジェクト（値で判断）です。例: 顧客はエンティティ、住所は値オブジェクト。

  3. Q3. 値オブジェクトで不変性を実装するには、どのようにしますか？

    回答を表示

    すべてのフィールドをfinalにして、セッターメソッドを作らないようにします。値を変更したい場合は、新しい値オブジェクトを生成して返すメソッド（add()、withXxx()等）を用意します。

  4. Q4. 値オブジェクトを使うことで、どのような効果が得られますか？（3つ挙げてください）

    回答を表示

    ① 型安全性の向上（コンパイル時に型の間違いを検出）
    ② ビジネスロジックの集約（重複コードの削減）
    ③ バグの削減（不変性と自己検証による）
    その他: ドメイン知識の明示化、コードの可読性向上なども該当

この章のまとめ

  • 値オブジェクトは、値自体が意味を持つオブジェクトで、金額・住所・メールアドレスなどがその例
  • 不変性、等価性、自己検証の3つが値オブジェクトの重要な特性
  • エンティティと値オブジェクトは「追跡の必要性」で使い分ける
  • 値オブジェクトを使うことで型安全性が向上し、バグが減少する
  • すべてをfinalにし、equals()とhashCode()を実装することが実装の鍵
