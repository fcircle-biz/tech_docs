境界づけられたコンテキスト - ドメインの分割と統治

DDDの戦略的パターンの学習を開始します。大規模なシステムをどのように分割するか、「顧客」という同じ言葉が販売・配送・サポートで異なる意味を持つ場合にどう対処するかを学びます。コンテキストの境界を見極め、適切に分割する方法を習得します。

この章で学ぶこと

  • 境界づけられたコンテキストとは何か、なぜ必要なのかを理解する
  • 同じ言葉が異なる意味を持つ問題とその解決方法を学ぶ
  • コンテキストの境界を見極める方法を習得する
  • 適切なコンテキスト分割の指針とパターンを理解する

10.1 境界づけられたコンテキストとは何か

境界づけられたコンテキスト（Bounded Context）は、DDDの戦略的パターンの中で最も重要な概念です。これは、「特定のドメインモデルが適用される明確な境界を持った領域」を意味します。

大規模なシステムでは、すべてを単一のモデルで表現しようとすると、矛盾や複雑性が急激に増大します。境界づけられたコンテキストは、システムを小さく管理可能な部分に分割することで、この問題を解決します。

重要ポイント

境界づけられたコンテキストの中では、ユビキタス言語が一貫性を持ちます。異なるコンテキスト間では、同じ言葉が異なる意味を持つことが許されます。これが「境界」の本質です。

10.2 なぜコンテキストを分割する必要があるのか

コンテキストを分割せず、すべてを単一のモデルで表現しようとすると、以下のような問題が発生します：

問題①: 概念の衝突

「顧客」という言葉が、販売部門では「購入者」、配送部門では「配送先」、サポート部門では「問い合わせ元」と異なる意味を持つため、単一のモデルで統一しようとすると混乱が生じます。

問題②: モデルの肥大化

すべての要件を1つのモデルに詰め込むと、エンティティや関連が膨れ上がり、理解・保守が困難になります。すべてのチームが同じコードベースで作業することで、変更のリスクも増大します。

問題③: チーム間の調整コスト

複数のチームが単一のモデルを共有すると、変更のたびに全チームでの合意が必要になり、開発スピードが低下します。

問題④: 技術的制約の混在

リアルタイム性が必要な部分とバッチ処理で良い部分、高負荷な部分と低負荷な部分が混在し、適切な技術選定が困難になります。

コンテキスト分割のメリット

  • 明確な責任範囲: 各コンテキストが特定のビジネス機能に専念できる
  • チームの自律性: 他のチームに影響を与えずに変更・進化できる
  • 技術の多様性: コンテキストごとに最適な技術スタックを選択できる
  • スケーラビリティ: 負荷の高いコンテキストだけを独立してスケールできる

販売コンテキストの「顧客」

// 販売コンテキスト
class Customer {
  String id;
  String name;
  int points; // ポイント残高
  List<Order> orders; // 購買履歴
  String memberRank; // 会員ランク
}

配送コンテキストの「顧客」

Recipient.java 
// 配送コンテキスト
class Recipient {
  String id;
  String name;
  Address address; // 配送先住所
  String phone;
  String deliveryTime; // 配送希望時間
}

サポートコンテキストの「顧客」

SupportCustomer.java 
// サポートコンテキスト
class SupportCustomer {
  String id;
  String name;
  List<Inquiry> inquiries; // 問い合わせ履歴
  int priority; // 対応優先度
  String assignedAgent; // 担当者
}

無理に統一しようとすると...

すべての属性を持つ巨大な「Customer」クラスを作ると、販売では使わない配送先情報、配送では使わない問い合わせ履歴など、不要な情報が混在し、複雑性が増大します。境界づけられたコンテキストでは、それぞれの文脈で必要な情報だけを持つモデルを作ります。

10.4 コンテキストの境界を見極める方法

では、どのようにしてコンテキストの境界を見極めればよいのでしょうか。以下の観点で検討します。

コンテキスト境界を見極める5つの観点

  1. 1
    ビジネス機能の違い

    「販売」「配送」「サポート」など、異なるビジネス機能は異なるコンテキストになることが多い。それぞれが独立した価値を提供している。

  2. 2
    言葉の意味の違い

    同じ言葉でも異なる意味や属性を持つ場合は、別のコンテキストである可能性が高い。ユビキタス言語が一貫性を保てる範囲が1つのコンテキスト。

  3. 3
    チームの境界

    異なるチームが担当する領域は、異なるコンテキストになることが多い。チームの自律性を保つためにも境界を明確にする。

  4. 4
    変更の独立性

    ある機能の変更が他の機能に影響を与えない場合、それらは別のコンテキストとして分割できる。独立して進化できる単位。

  5. 5
    技術的要件の違い

    リアルタイム性が必要な部分とバッチ処理で良い部分、高負荷な部分と低負荷な部分など、技術的要件が大きく異なる場合も分割の指標になる。

10.6 コンテキストの大きさと粒度

コンテキストをどのくらいの大きさに分割すべきかは、プロジェクトの規模やチーム構成によって異なります。

コンテキストが大きすぎる場合

  • ● 複数のビジネス機能が混在し、責務が不明確になる
  • ● ユビキタス言語の一貫性を保つのが困難になる
  • ● 変更の影響範囲が広くなり、リスクが増大する
  • ● チーム間の調整コストが増える

→ さらに分割を検討する

コンテキストが小さすぎる場合

  • ● コンテキスト間の連携が複雑になり、オーバーヘッドが増える
  • ● データの重複や不整合が発生しやすくなる
  • ● システム全体の把握が困難になる
  • ● 統合テストやデプロイの複雑性が増す

→ 統合を検討する

適切な粒度の目安

1つのチーム（5〜9人程度）が担当できる範囲を1つのコンテキストとするのが目安です。また、「2週間〜1ヶ月で機能追加できる」程度の大きさが理想的です。マイクロサービスの場合は、さらに小さく分割することもあります。

理解度確認クイズ

  1. Q1. 境界づけられたコンテキストとは何ですか？簡単に説明してください。

    回答を表示

    境界づけられたコンテキストとは、特定のドメインモデルが適用される明確な境界を持った領域のことです。その境界の中では、ユビキタス言語が一貫性を持ち、同じ言葉が同じ意味で使われます。

  2. Q2. なぜコンテキストを分割する必要があるのですか？主な理由を3つ挙げてください。

    回答を表示

    ① 概念の衝突を防ぐ（同じ言葉が異なる意味を持つ問題を解決）
    ② モデルの肥大化を防ぐ（理解・保守が困難になるのを防ぐ）
    ③ チームの自律性を高める（他のチームに影響を与えずに変更できる）
    （その他、技術的制約の分離、スケーラビリティなども理由）

  3. Q3. 販売コンテキストの「顧客」と配送コンテキストの「顧客」では、どのように意味が異なりますか？

    回答を表示

    販売コンテキストでは「顧客 = 購入者」であり、購買履歴、ポイント残高、会員ランクなどの属性を持ちます。一方、配送コンテキストでは「顧客 = 配送先」であり、住所、電話番号、配送時間帯指定などの属性を持ちます。同じ「顧客」という言葉でも、関心事や必要な情報が全く異なります。

  4. Q4. コンテキストの境界を見極めるための5つの観点のうち、3つを挙げてください。

    回答を表示

    ① ビジネス機能の違い
    ② 言葉の意味の違い
    ③ チームの境界
    （その他、変更の独立性、技術的要件の違いも観点）

  5. Q5. コンテキストが小さすぎる場合、どのような問題が発生しますか？

    回答を表示

    コンテキスト間の連携が複雑になり、オーバーヘッドが増えます。また、データの重複や不整合が発生しやすくなり、システム全体の把握が困難になります。統合テストやデプロイの複雑性も増します。

  • 境界づけられたコンテキストは、特定のドメインモデルが適用される明確な境界を持った領域
  • コンテキスト内ではユビキタス言語が一貫性を持ち、異なるコンテキスト間では同じ言葉が異なる意味を持つことが許される
  • コンテキスト分割により、概念の衝突を防ぎ、モデルの肥大化を回避し、チームの自律性を高めることができる
  • 境界を見極めるには、ビジネス機能、言葉の意味、チームの境界、変更の独立性、技術的要件の5つの観点で検討する
  • コンテキストの大きさは、1チームが担当できる範囲を目安とし、大きすぎても小さすぎても問題が発生する
