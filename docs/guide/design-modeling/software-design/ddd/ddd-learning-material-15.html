第15章

既存システムへのDDD適用 - 段階的なリファクタリング

レガシーシステムにDDDを適用する方法を学びます。腐敗防止層を使った段階的な移行、ストラングラーパターン、バブルコンテキストなど、現実的なアプローチを習得します。既存コードを活かしながらドメインモデルを導入する戦略を理解します。

この章で学ぶこと

  • 既存システムにDDDを適用する際の課題と現実的なアプローチを理解する
  • 腐敗防止層(ACL)の役割と実装方法を習得する
  • ストラングラーパターンによる段階的な移行戦略を学ぶ
  • バブルコンテキストを使った局所的なDDD導入手法を理解する

15.1 レガシーシステムへのDDD適用が難しい理由

新規開発では理想的なDDD設計を目指すことができますが、既存システムに後からDDDを適用するには多くの課題があります。現実の開発現場でよくある状況を理解しましょう。

課題①: 技術的負債

長年の開発でビジネスロジックがデータベース層、UI層、サービス層に散在。責任境界が不明確で、どこから手をつければよいか分からない状態。

課題②: ドメイン知識の欠如

仕様書が古い、または存在しない。ビジネスルールがコードに埋もれており、なぜそうなっているのか誰も分からない状態。

課題③: リスクの高さ

既存システムは稼働中のため、大規模なリファクタリングは業務への影響が大きい。テストが不足している場合、変更による影響範囲が予測不可能。

課題④: リソース制約

新機能開発と並行して行う必要があり、リファクタリングだけに専念できない。短期的な成果が見えにくく、経営層の理解が得にくい。

現実的なアプローチが必要

これらの課題から、既存システムへのDDD適用は「すべてを一度にリファクタリングする」のではなく、「段階的に、リスクを抑えながら進める」戦略が重要です。

15.2 腐敗防止層(Anti-Corruption Layer / ACL)

腐敗防止層(ACL)は、新しいDDDモデルをレガシーシステムから守るための「防護壁」です。レガシーシステムの複雑さや技術的負債が、新しいドメインモデルに侵入するのを防ぎます。

ACLの3つの主要機能

① データ変換: レガシーのデータ構造をドメインモデルに適した形に変換

② 概念の翻訳: レガシー用語をユビキタス言語に翻訳

③ プロトコル変換: 異なる通信方式やAPIを橋渡し

実装例: 腐敗防止層

レガシー顧客テーブルから新しいCustomerエンティティへの変換

LegacyCustomerAdapter.java 
// 腐敗防止層: レガシーからドメインモデルへの変換
public class LegacyCustomerAdapter {
    // レガシーDBアクセス
    private final LegacyCustomerRepository legacyRepo;

    // ドメインモデルに変換
    public Customer findById(CustomerId id) {
        // レガシーから取得
        LegacyCustomerData data = legacyRepo
            .findByOldId(id.getValue());

        // ドメインモデルに変換
        return new Customer(
            id,
            new CustomerName(data.getFullName()),
            new EmailAddress(data.getEmail()),
            // レガシーの顧客レベルを新モデルに変換
            convertToCustomerType(data.getLevel())
        );
    }

    // レガシーの概念を新モデルに翻訳
    private CustomerType convertToCustomerType(int level) {
        // レガシー: 1=一般、2=ゴールド、3=プラチナ
        // 新モデル: STANDARD、PREMIUM、VIP
        return switch(level) {
            case 1 -> CustomerType.STANDARD;
            case 2 -> CustomerType.PREMIUM;
            case 3 -> CustomerType.VIP;
            default -> CustomerType.STANDARD;
        };
    }
}

15.3 ストラングラーパターン - 段階的な置き換え

ストラングラーパターン(Strangler Fig Pattern)は、レガシーシステムを段階的に新しいシステムで置き換えていく戦略です。名前の由来は、寄生植物が宿主を徐々に覆い尽くす様子からきています。

ストラングラーパターンの利点

✓ リスクの分散: 小さな単位で移行するため、失敗時の影響が限定的

✓ 継続的な価値提供: 移行中も新機能を追加できる

✓ 学習機会: 初期の移行から学び、後続の移行に活かせる

✓ ロールバック可能: 問題が発生したら元のシステムに戻せる

移行の5ステップ

  1. 1
    移行対象の選定

    最もシンプルで、ビジネス影響が小さく、境界が明確な機能から始める。例: 商品カタログ表示、ユーザープロフィール参照など。

  2. 2
    腐敗防止層の構築

    新しいドメインモデルとレガシーシステムの間にACLを配置し、データ変換を行う。

  3. 3
    新機能の実装

    DDDパターンを使って新しいモデルを実装。この段階ではレガシーと並行稼働。

  4. 4
    段階的な切り替え

    フィーチャートグル(機能フラグ)を使い、一部ユーザーから徐々に新システムに切り替え。

  5. 5
    レガシーコードの削除

    新システムが安定したら、対応するレガシーコードを削除。データも移行する。

15.4 バブルコンテキスト - 局所的なDDD導入

バブルコンテキスト(Bubble Context)は、レガシーシステムの中に「泡(バブル)」のように小さなDDDモデルを作り、徐々に拡大していくアプローチです。システム全体を置き換えるのではなく、局所的にDDDを適用します。

バブルコンテキストのイメージ

バブルコンテキストの特徴

  • ● レガシーの中に小さな「島」を作る
  • ● 境界が明確で独立性が高い機能から始める
  • ● レガシーとは疎結合に保つ
  • ● 成功したら隣接する機能に拡大

適用に向いている機能

  • ● 新規機能の追加
  • ● 頻繁に変更される機能
  • ● ビジネス価値が高い機能
  • ● 独立してテスト可能な機能

15.5 実践的な移行ロードマップ

既存システムへのDDD適用を成功させるための、段階的なロードマップを示します。

移行ロードマップ(6ヶ月〜1年)

第1フェーズ: 調査と準備(1〜2ヶ月)

  • • レガシーコードの理解とドメイン知識の収集
  • • ドメインエキスパートへのインタビュー
  • • 境界づけられたコンテキストの暫定的な特定
  • • 移行対象機能の優先順位付け

第2フェーズ: 最初のバブル作成(2〜3ヶ月)

  • • 最もシンプルな機能でバブルコンテキスト作成
  • • 腐敗防止層の実装
  • • ユビキタス言語の定義と文書化
  • • エンティティ、値オブジェクト、集約の実装

第3フェーズ: ストラングラー適用(3〜4ヶ月)

  • • フィーチャートグルで段階的切り替え
  • • A/Bテストで新旧システムの動作比較
  • • レガシーコードの段階的削除
  • • チームメンバーへのDDDトレーニング

第4フェーズ: 拡大と最適化(4〜6ヶ月)

  • • 成功した領域を隣接機能に拡大
  • • リファクタリングと設計改善
  • • ドメインイベントの導入
  • • 完全なテストカバレッジの確立

15.6 よくある失敗とその回避策

既存システムへのDDD適用でよく見られる失敗パターンと、その対策を理解しましょう。

トラブルシューティング

失敗①: ビッグバンリファクタリング

症状: 一度にすべてを書き換えようとして、数ヶ月経っても完成せず、途中で挫折する。

対策: 小さく始める。最小限の機能でバブルを作り、成功体験を積む。週単位でデプロイできる粒度を目指す。

失敗②: ドメイン知識の不足

症状: ビジネスルールを理解せずに設計し、間違ったモデルを作ってしまう。

対策: ドメインエキスパートとの定期的なセッション。イベントストーミングなどで知識を共有。ユビキタス言語を文書化。

失敗③: テスト不足

症状: リファクタリングで既存機能を壊してしまい、本番で不具合が発生する。

対策: リファクタリング前に特性テスト(Characterization Test)を作成。新しいモデルには単体テストと統合テストを追加。

失敗④: 理想を追い求めすぎる

症状: 完璧なドメインモデルを目指して、実装が進まない。現実のビジネス要求に対応できない。

対策: 「動くソフトウェア」を優先。最初から完璧を目指さず、継続的にリファクタリングして改善する。

理解度確認クイズ

  1. Q1. 腐敗防止層(ACL)の主な役割を3つ挙げてください。

    回答を表示

    ① データ変換: レガシーのデータ構造をドメインモデルに適した形に変換
    ② 概念の翻訳: レガシー用語をユビキタス言語に翻訳
    ③ プロトコル変換: 異なる通信方式やAPIを橋渡し

  2. Q2. ストラングラーパターンの利点を2つ説明してください。

    回答を表示

    ① リスクの分散: 小さな単位で移行するため、失敗時の影響が限定的
    ② 継続的な価値提供: 移行中も新機能を追加でき、ビジネスを止めずに済む
    （その他、学習機会、ロールバック可能性なども正解）

  3. Q3. バブルコンテキストを適用するのに向いている機能の特徴を2つ挙げてください。

    回答を表示

    ① 境界が明確で独立性が高い機能
    ② 新規機能の追加や、頻繁に変更される機能
    （その他、ビジネス価値が高い、独立してテスト可能なども正解）

  4. Q4. 既存システムへのDDD適用で「ビッグバンリファクタリング」が失敗しやすいのはなぜですか?

    回答を表示

    一度にすべてを書き換えようとすると、リスクが大きく、完成までに長時間かかります。その間、ビジネス要求への対応ができず、途中で挫折しやすくなります。小さく始めて成功体験を積み、段階的に拡大する方が成功率が高くなります。

この章のまとめ

  • 既存システムへのDDD適用は、技術的負債、知識不足、リスク、リソース制約などの課題がある
  • 腐敗防止層(ACL)は、新しいドメインモデルをレガシーの影響から守る防護壁
  • ストラングラーパターンで段階的に置き換えることで、リスクを分散し継続的に価値を提供できる
  • バブルコンテキストで局所的にDDDを適用し、成功したら隣接機能に拡大する
  • ビッグバンリファクタリングを避け、小さく始めて段階的に拡大することが成功の鍵
