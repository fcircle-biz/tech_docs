属性の表現 - エンティティの詳細情報を記録する

エンティティが持つ個々のデータ項目「属性」を正しく表現する方法を学びましょう。

この章で学ぶこと

  • 属性（アトリビュート）とは何か、その役割を理解する
  • 属性の種類（単純属性・複合属性・導出属性）を学ぶ
  • IE記法（鳥の足記法）とChen記法の違いを知る
  • draw.ioを使ってエンティティに属性を追加する方法を習得する

4.1 属性（アトリビュート）とは

前章でエンティティ（実体）を学びました。エンティティは「顧客」「商品」「注文」といった管理したい「もの」や「こと」でした。 しかし、エンティティだけでは「具体的にどんな情報を記録するのか」が分かりません。

属性（Attribute）とは

エンティティが持つ個々のデータ項目のこと。 例えば「顧客」エンティティなら、「顧客名」「メールアドレス」「電話番号」「住所」などが属性にあたります。

具体例で理解する

エンティティ「顧客」の属性の例

属性名 説明 データ例
顧客ID 顧客を一意に識別する番号 C001
顧客名 顧客の氏名 山田太郎
メールアドレス 連絡先メール yamada@example.com
電話番号 連絡先電話 090-1234-5678
住所 配送先住所 東京都渋谷区...

このように、属性はエンティティに関する具体的な情報を表します。 データベースでは、エンティティがテーブルになり、属性が列（カラム）になります。

4.2 属性の種類

属性には、その性質によっていくつかの分類があります。初心者のうちは「単純属性」を中心に理解し、 必要に応じて「複合属性」「導出属性」を使い分けられるようになりましょう。

1. 単純属性（Simple Attribute）

単純属性とは

それ以上分割できない基本的な属性です。 例：顧客名、メールアドレス、電話番号、商品価格など。

2. 複合属性（Composite Attribute）

複合属性とは

複数の要素に分割できる属性です。 例：「住所」は「郵便番号」「都道府県」「市区町村」「番地」「建物名」に分割できます。

例：

  • 住所 → 郵便番号、都道府県、市区町村、番地
  • 氏名 → 姓、名

3. 導出属性（Derived Attribute）

導出属性とは

他の属性から計算や導出できる属性です。 例：「年齢」は「生年月日」から計算できるため、導出属性として扱えます。

例：

  • 年齢 ← 生年月日から計算
  • 合計金額 ← 単価 × 数量で計算
  • 在庫日数 ← 入庫日から現在日までの日数

導出属性の注意点

導出属性は、データベースに保存せず毎回計算する方法と、計算結果を保存する方法があります。 どちらを選ぶかは、処理速度やデータの鮮度の要件によります。初心者のうちは「計算で求められる属性」という理解で十分です。

4.3 IE記法とChen記法の違い

ER図には複数の表記法がありますが、代表的なものがIE記法（鳥の足記法）とChen記法です。 draw.ioではどちらの記法も使えますが、実務ではIE記法がよく使われます。

IE記法（Information Engineering記法）

IE記法の特徴

  • エンティティを長方形（テーブル形式）で表す
  • 属性をエンティティ内部にリスト形式で記載
  • 主キー（PK）や外部キー（FK）を明示できる
  • 関連性を「鳥の足」のような記号で表現するため「鳥の足記法」とも呼ばれる
erDiagram CUSTOMER { string customer_id PK string customer_name string email string phone string address }

Chen記法

Chen記法の特徴

  • エンティティを長方形で表す
  • 属性を楕円形で表し、エンティティと線で結ぶ
  • リレーションシップをひし形で表現
  • 視覚的には分かりやすいが、属性が多いと図が複雑になる

このガイドではIE記法を採用

IE記法は属性が多くても見やすく、データベース設計の実務でよく使われます。 draw.ioでもIE記法が標準的に使われているため、本ガイドではIE記法を中心に学習します。

4.4 draw.ioで属性を描く方法

それでは、実際にdraw.ioを使ってエンティティに属性を追加する方法を学びましょう。 IE記法では、エンティティをテーブル形式で表現し、その中に属性をリスト形式で記載します。

手順1: ERD図形ライブラリを開く

  1. 1 draw.ioを開く（https://app.diagrams.net/）
  2. 2 画面左側の「その他の図形」をクリック
  3. 3 「ソフトウェア」カテゴリの中から「Entity Relation」にチェックを入れて「適用」をクリック

手順2: テーブル図形を配置

  1. 1 左側パネルの「Entity Relation」セクションから「Table」図形をキャンバスにドラッグ
  2. 2 テーブル図形がデフォルトで3行（ヘッダー＋行2つ）表示される

手順3: エンティティ名を入力

  1. 1 テーブルのヘッダー部分（一番上）をダブルクリック
  2. 2 エンティティ名（例：「顧客」「CUSTOMER」）を入力

手順4: 属性を追加

  1. 1 各行をダブルクリックして属性名を入力（例：「顧客ID」「顧客名」など）
  2. 2 行を追加したい場合：テーブルを右クリック → 「行を挿入」を選択
  3. 3 主キー（PK）を示す場合：属性名の横に「（PK）」や鍵マークを追加

ポイント

最初は主要な属性だけを記載し、詳細は後から追加していくのが効率的です。 概念設計の段階では、細かいデータ型は省略して構いません。

4.5 実習：エンティティに属性を追加しよう

実習 4-1: 「図書館システム」の属性を設計する

図書館システムを題材に、エンティティとその属性を設計してみましょう。

課題

以下の3つのエンティティに必要な属性を考えて、draw.ioで描いてみましょう。

エンティティ1: 会員

図書館の利用者情報

エンティティ2: 書籍

図書館が所蔵する本の情報

エンティティ3: 貸出

本の貸出履歴

手順

  1. 1 draw.ioを開き、「Entity Relation」ライブラリを有効にする
  2. 2 3つのテーブル図形を配置し、それぞれに「会員」「書籍」「貸出」という名前を付ける
  3. 3 各エンティティに必要な属性を考えて追加する（最低5個ずつ）
  4. 4 完成したら保存する（ファイル名：「図書館システムER図.drawio」）

ヒント

属性の例を表示

会員エンティティの属性例：

  • 会員ID（PK）
  • 氏名
  • 生年月日
  • 電話番号
  • メールアドレス
  • 住所
  • 会員登録日

書籍エンティティの属性例：

  • 書籍ID（PK）
  • 書籍名
  • 著者名
  • 出版社
  • ISBN
  • 出版年
  • 分類コード

貸出エンティティの属性例：

  • 貸出ID（PK）
  • 会員ID（FK）
  • 書籍ID（FK）
  • 貸出日
  • 返却予定日
  • 返却日

理解度確認クイズ

  1. Q1. 属性（Attribute）とは何ですか？簡潔に説明してください。

    回答を表示

    エンティティが持つ個々のデータ項目のこと。例えば「顧客」エンティティの「顧客名」「メールアドレス」「電話番号」などが属性にあたります。データベースではテーブルの列（カラム）として実装されます。

  2. Q2. 次のうち、導出属性（Derived Attribute）はどれですか？

    1. 顧客名
    2. 生年月日
    3. 年齢（生年月日から計算）
    4. 電話番号
    回答を表示

    3. 年齢（生年月日から計算）が導出属性です。

    年齢は生年月日と現在日から計算できるため、他の属性から導出可能な属性です。そのため、データベースには保存せず、必要な時に計算して求めることもできます。

  3. Q3. IE記法（鳥の足記法）とChen記法の違いについて、正しい説明はどれですか？

    1. IE記法では属性を楕円形で表し、Chen記法ではテーブル内に記載する
    2. IE記法では属性をテーブル内に記載し、Chen記法では楕円形で表す
    3. 両方とも属性の表現方法は同じである
    4. IE記法では属性を記載せず、Chen記法のみ記載する
    回答を表示

    2. IE記法では属性をテーブル内に記載し、Chen記法では楕円形で表すが正しいです。

    IE記法（Information Engineering記法）はテーブル形式でエンティティを表現し、属性をリスト形式で内部に記載します。一方、Chen記法は属性を楕円形で表し、エンティティと線で結びます。実務ではIE記法がより広く使われています。

第4章のまとめ

  • 属性（Attribute）は、エンティティが持つ個々のデータ項目である
  • 属性には、単純属性、複合属性、導出属性の3種類がある
  • IE記法ではテーブル内に属性をリスト形式で記載し、Chen記法では楕円形で表現する
  • draw.ioでは「Entity Relation」ライブラリのテーブル図形を使って、属性を効率的に記述できる
  • 導出属性は他の属性から計算できるため、データベースに保存するかどうかを検討する必要がある
