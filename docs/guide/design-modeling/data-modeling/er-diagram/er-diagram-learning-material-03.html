第3章

エンティティの基本 - 「もの」を四角形で表す

ER図の最も基本的な要素「エンティティ」を学び、データの管理対象を明確にしましょう。

この章で学ぶこと

  • エンティティとは何か、その役割を理解する
  • 業務要件からエンティティを見つける方法を学ぶ
  • 適切なエンティティ名の付け方を習得する
  • draw.ioでエンティティを描く方法を実践する

3.1 エンティティとは何か

エンティティ（Entity）とは、データベースで管理したい「もの」や「こと」を表す概念です。 日本語では「実体」と訳されます。

エンティティの定義

業務やシステムでデータとして記録・管理する必要がある対象のことです。 例えば「顧客」「商品」「注文」「社員」など、具体的に存在し、情報を記録する価値のあるものがエンティティになります。

身近な例で理解する

オンラインショップを例に考えてみましょう。どのような「もの」や「こと」を管理する必要があるでしょうか？

エンティティ: 顧客

サイトに登録したユーザー。氏名、メールアドレス、住所などの情報を管理します。

エンティティ: 商品

販売する商品。商品名、価格、在庫数、カテゴリなどの情報を管理します。

エンティティ: 注文

顧客が行った購入行為。注文日時、合計金額、配送先などの情報を管理します。

エンティティ: 注文明細

注文に含まれる個別の商品情報。どの注文に、どの商品が何個含まれるかを管理します。

ER図でのエンティティ表現

erDiagram CUSTOMER { string customer_id PK string name string email string address } PRODUCT { string product_id PK string product_name decimal price int stock } ORDER { string order_id PK date order_date decimal total_amount } ORDER_LINE { string order_line_id PK int quantity decimal unit_price }

ER図では、エンティティを四角形（長方形）で表現します。

3.2 エンティティの見つけ方

業務要件やシステムの仕様から、どのようにエンティティを抽出すればよいのでしょうか？ 以下のステップで考えていきましょう。

ステップ1: 業務の流れを把握する

まず、対象となる業務やシステムの流れを整理します。 例えば、図書館システムの場合：

図書館システムの業務フロー

flowchart LR A([会員登録]) --> B[本を検索] B --> C[貸出申込] C --> D[貸出記録] D --> E[返却] E --> F[返却記録]

ステップ2: 名詞を抜き出す

業務の説明文や仕様書から、名詞を抜き出します。 多くの場合、名詞がエンティティの候補になります。

図書館システムの説明文例

「会員は図書館に登録し、本を借りることができます。 貸出の記録には貸出日と返却予定日が含まれます。 本にはタイトル、著者、ISBNなどの情報があります。」

抽出された名詞（エンティティ候補）:

  • 会員
  • 本
  • 貸出

ステップ3: エンティティとして適切か判断する

抽出した名詞すべてがエンティティになるわけではありません。以下の基準で判断します。

エンティティとして適切な条件

✓

複数のインスタンスが存在する

「会員」には山田さん、佐藤さんなど複数の人が該当

✓

独自の属性（情報）を持つ

「会員」は会員番号、氏名、連絡先などの情報を持つ

✓

業務上重要な管理対象

システムとして記録・追跡する必要がある

✓

一意に識別できる

会員番号、ISBNなど、他と区別できる識別子を持つ

エンティティにならないもの

  • 単なる属性（例: 「氏名」「価格」は属性であり、エンティティではない）
  • 一時的な処理結果（例: 「合計金額」は計算結果）
  • 動作や処理（例: 「検索する」「登録する」は動詞）

3.3 良いエンティティ名の付け方

エンティティには、誰が見ても理解できる明確な名前を付けることが重要です。

エンティティ命名の原則

1. 名詞の単数形を使う

○ 「顧客」「商品」
× 「顧客たち」「商品群」

2. 業務用語を使う

業務担当者が日常的に使う用語を採用します。
例: 「受注」「発注」「在庫」など

3. 具体的で分かりやすい名前にする

○ 「会員」「書籍」
× 「データ」「情報」「マスタ」

4. 略語は避ける（必要な場合は注釈を付ける）

○ 「顧客」「注文」
△ 「CUST」「ORD」（英語システムの場合）

英語表記の場合

英語でER図を作成する場合は、以下の慣例に従います。

日本語 英語（推奨） 備考
顧客 CUSTOMER 大文字・単数形
商品 PRODUCT 大文字・単数形
注文 ORDER 大文字・単数形
注文明細 ORDER_LINE 複数単語はアンダースコアで接続

3.5 draw.ioでエンティティを描く

実際にdraw.ioを使ってエンティティを描いてみましょう。

実習 3-1: エンティティを描いてみよう

図書館システムの基本エンティティをdraw.ioで作成します。

手順

  1. 1 draw.io（https://app.diagrams.net/）を開く
  2. 2 左パネルから「Entity Relation」を選択（なければ「その他の図形」→「Entity Relation」をチェック）
  3. 3 「Table」または「Entity」図形をキャンバスにドラッグ＆ドロップ
  4. 4 エンティティ名を「会員」「書籍」「貸出」に変更
  5. 5 3つのエンティティを横に並べて配置

ヒント: エンティティ図形の選び方

draw.ioのEntity Relationライブラリには複数のエンティティ図形があります。 シンプルな「Table」図形を使うのがおすすめです。後の章で属性を追加する際に便利です。

動作確認

以下のようなER図が作成できていれば成功です。

期待される結果

  • ✓ 「会員」「書籍」「貸出」という3つのエンティティが配置されている
  • ✓ 各エンティティが四角形で表現されている
  • ✓ エンティティ名が明確に読み取れる

理解度確認クイズ

  1. Q1. エンティティの定義として最も適切なものはどれですか？

    1. データベースで管理したい「もの」や「こと」
    2. 業務の処理手順
    3. データの計算結果
    4. システムの画面デザイン
    回答を表示

    1. データベースで管理したい「もの」や「こと」が正解です。

    エンティティは、業務やシステムでデータとして記録・管理する必要がある対象（顧客、商品、注文など）を表します。処理手順や計算結果、画面デザインはエンティティではありません。

  2. Q2. 以下の中で、エンティティとして「適切でない」ものはどれですか？

    1. 顧客
    2. 商品価格
    3. 注文
    4. 社員
    回答を表示

    2. 商品価格が不適切です。

    「商品価格」は「商品」エンティティが持つ属性（情報）であり、それ自体がエンティティになるわけではありません。「顧客」「注文」「社員」はそれぞれ独自の属性を持ち、複数のインスタンスが存在する適切なエンティティです。

  3. Q3. エンティティ名を付けるときの原則として正しいものをすべて選んでください。

    1. 名詞の単数形を使う
    2. 業務用語を使う
    3. できるだけ短い略語にする
    4. 具体的で分かりやすい名前にする
    回答を表示

    1, 2, 4が正解です。

    エンティティ名は名詞の単数形を使い、業務担当者が理解できる業務用語で、具体的で分かりやすい名前を付けるのが原則です。略語は避けるべきです（必要な場合は注釈を付ける）。

第3章のまとめ

  • エンティティとは、データベースで管理したい「もの」や「こと」を表す概念
  • 業務の流れを把握し、名詞を抜き出すことでエンティティを見つけることができる
  • エンティティとして適切な条件は、複数のインスタンスが存在し、独自の属性を持ち、業務上重要な管理対象であること
  • エンティティ名は名詞の単数形、業務用語、具体的で分かりやすい名前を付ける
  • ER図ではエンティティを四角形（長方形）で表現し、draw.ioで簡単に作成できる
