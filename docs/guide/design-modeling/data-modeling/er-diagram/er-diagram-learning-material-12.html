実践演習1 - 小規模システムをER図で設計する

これまで学んだ知識を総合して、実際のシステムをER図で設計する演習を行います。

この章で学ぶこと

  • 要件定義からER図を作成する一連のプロセスを実践する
  • エンティティの抽出から正規化まで、段階的な設計手法を習得する
  • 小規模システムのER図を自力で完成させる経験を積む
  • draw.ioを使った実践的なER図作成技術を身につける

12.1 実践演習の進め方

この章では、これまでに学んだER図の知識を総合して、実際のシステムを設計する演習を行います。 単にER図を描くだけでなく、要件を理解し、段階的に設計を進めるプロセス全体を体験することが重要です。

ER図設計の5つのステップ

1

要件の理解

システムで管理したいデータや業務フローを把握する

2

概念ER図の作成

エンティティとリレーションシップを抽出し、全体像を描く

3

論理ER図への展開

属性を追加し、主キー・外部キーを明確化する

4

正規化の適用

データの重複や矛盾を排除し、設計を改善する

5

設計の検証とレビュー

要件が満たされているか、漏れがないかを確認する

演習のポイント

最初から完璧なER図を作ろうとせず、何度も見直しながら改善していくことが大切です。 実務でも、初版から何度も修正を重ねるのが一般的です。

12.2 演習テーマの選択

以下の3つのテーマから1つを選んで、ER図の設計に挑戦してください。 身近で理解しやすいテーマを選ぶことで、要件の理解が深まり、より効果的な学習になります。

テーマA

タスク管理アプリ

個人やチームでタスクを管理するシンプルなアプリケーション。

  • 難易度: ★☆☆
  • エンティティ数: 4-5個
  • 推奨学習時間: 40分

テーマB

レシピ共有サイト

料理レシピを投稿・検索・共有できるコミュニティサイト。

  • 難易度: ★★☆
  • エンティティ数: 6-7個
  • 推奨学習時間: 50分

テーマC

イベント予約システム

セミナーやイベントの予約を管理するシステム。

  • 難易度: ★★☆
  • エンティティ数: 5-6個
  • 推奨学習時間: 50分

初めての方へ

初めてER図設計に挑戦する場合は、テーマA（タスク管理アプリ）から始めることをお勧めします。 シンプルな構造なので、設計プロセス全体を無理なく体験できます。

12.3 テーマA: タスク管理アプリの設計

ステップ1: 要件の理解

まず、タスク管理アプリで管理したい情報と、実現したい機能を整理しましょう。

システム要件

  • ユーザーはアカウントを持ち、複数のタスクを作成・管理できる
  • タスクはプロジェクト（複数のタスクをまとめる単位）に所属する
  • タスクにはタイトル、説明、期限、優先度、ステータスを設定できる
  • タスクには複数のタグを付けて分類できる
  • タスクにはコメントを追加できる

ステップ2: エンティティの抽出

要件から、管理すべき「もの」や「こと」を抽出します。

抽出されたエンティティ

1. ユーザー（User）

システムを利用するユーザー情報

2. プロジェクト（Project）

タスクをまとめる単位

3. タスク（Task）

管理する個々のタスク

4. タグ（Tag）

タスクを分類するラベル

5. コメント（Comment）

タスクに付けるコメント

ステップ3: リレーションシップの特定

エンティティ同士の関係性とカーディナリティを考えます。

主要なリレーションシップ

  • ユーザー → プロジェクト

    一人のユーザーは複数のプロジェクトを作成できる（1対多）

  • プロジェクト → タスク

    一つのプロジェクトには複数のタスクが所属する（1対多）

  • タスク → タグ

    一つのタスクに複数のタグ、一つのタグが複数のタスクに使われる（多対多）

  • タスク → コメント

    一つのタスクに複数のコメントが付けられる（1対多）

  • ユーザー → コメント

    一人のユーザーが複数のコメントを投稿できる（1対多）

ステップ4: 概念ER図の作成

エンティティとリレーションシップを図で表現します。

erDiagram USER ||--o{ PROJECT : creates USER ||--o{ COMMENT : posts PROJECT ||--o{ TASK : contains TASK ||--o{ COMMENT : has TASK }o--o{ TAG : tagged_with USER { int user_id PK string username string email string password_hash datetime created_at } PROJECT { int project_id PK int user_id FK string project_name string description datetime created_at } TASK { int task_id PK int project_id FK string title text description date due_date enum priority enum status datetime created_at } TAG { int tag_id PK string tag_name string color } COMMENT { int comment_id PK int task_id FK int user_id FK text content datetime created_at }

多対多リレーションシップの処理

タスクとタグの多対多関係は、実際のデータベースでは中間テーブル（TASK_TAG）を作成して解消します。 この図ではMermaidの記法で多対多を表現していますが、物理ER図では中間テーブルが必要になります。

12.4 draw.ioでの実践演習

ここからは、実際にdraw.ioを使って、選択したテーマのER図を作成してみましょう。

実習: ER図の作成

以下の手順でdraw.ioを使ってER図を作成してください。

  1. 手順1: draw.ioを開き、新しいダイアグラムを作成（Blank Diagramを選択）
  2. 手順2: 左側のシェイプライブラリから「Entity Relation」を追加
  3. 手順3: エンティティを配置（四角形で表現）
  4. 手順4: 各エンティティに属性を追加（主キーはPK、外部キーはFKと記載）
  5. 手順5: リレーションシップを線で表現し、カーディナリティを設定
  6. 手順6: 全体のレイアウトを調整し、見やすく整理
  7. 手順7: ファイルを保存（「ファイル」→「名前を付けて保存」）

draw.ioでのエンティティ作成のコツ

見やすい配置

エンティティ同士が重ならないよう、適切な間隔を保ちます。 主要なエンティティを中心に、関連するエンティティを周囲に配置すると分かりやすくなります。

色の活用

エンティティの種類や重要度に応じて色分けすると、図が見やすくなります。 例：マスターデータは青、トランザクションデータは緑など。

属性の整理

主キーは最上部に配置し、重要な属性を上に、派生属性やオプショナルな属性を下に並べます。

関連線の引き方

リレーションシップの線は、可能な限り直角または直線で引き、交差を最小限にします。

12.5 他のテーマのヒント

テーマBとテーマCに挑戦する場合のヒントを紹介します。

テーマB: レシピ共有サイト

考慮すべきエンティティ候補

ユーザー（User）

レシピ投稿者・閲覧者

レシピ（Recipe）

料理のレシピ情報

材料（Ingredient）

レシピで使用する材料

調理手順（Step）

レシピの作り方

カテゴリ（Category）

レビュー（Review）

レシピへの評価とコメント

ポイント: レシピと材料の関係は多対多です。中間テーブル（Recipe_Ingredient）で分量も管理しましょう。

テーマC: イベント予約システム

考慮すべきエンティティ候補

ユーザー（User）

予約者情報

イベント（Event）

開催されるイベント情報

会場（Venue）

イベント開催場所

予約（Reservation）

ユーザーのイベント予約記録

主催者（Organizer）

イベントを主催する団体・個人

チケット種別（TicketType）

一般、学生など料金区分

ポイント: 予約のキャンセル履歴や座席番号も管理するかを検討しましょう。

12.6 設計のチェックリスト

ER図が完成したら、以下のチェックリストで設計を確認しましょう。

設計チェックポイント

よくある設計ミス

  • 主キーが設定されていないエンティティがある
  • 多対多リレーションシップを直接結んでしまう（中間テーブルが必要）
  • 同じ属性が複数のエンティティに重複して存在する
  • 外部キーが参照先の主キーと型が一致していない

12.7 発展課題

基本的なER図が完成したら、以下の発展的な要素を追加してみましょう。

課題1: 監査ログの追加

データの作成日時、更新日時、更新者を記録する属性を追加してみましょう。

ヒント: created_at, updated_at, created_by などの属性を検討

課題2: 論理削除の実装

データを物理的に削除せず、削除フラグで管理する仕組みを追加してみましょう。

ヒント: is_deleted (Boolean), deleted_at (Timestamp) などの属性

課題3: 権限管理の追加

ユーザーに役割（管理者、一般ユーザーなど）を持たせる仕組みを追加してみましょう。

ヒント: Role（役割）エンティティとUser_Roleの関連テーブル

課題4: 通知機能の設計

ユーザーへの通知を管理するエンティティを追加してみましょう。

ヒント: Notification エンティティ、既読/未読の状態管理

動作確認: 設計の妥当性検証

作成したER図が正しく機能するか、以下のシナリオで確認してみましょう。

シナリオ1: データの登録

新しいユーザーが登録し、タスクを作成する流れをER図上でたどれるか確認

シナリオ2: データの検索

特定のプロジェクトに含まれるすべてのタスクを取得できるか確認

シナリオ3: データの更新

ユーザー情報を変更したとき、関連するデータが正しく更新されるか確認

シナリオ4: データの削除

プロジェクトを削除したとき、関連するタスクやコメントの扱いが明確か確認

トラブルシューティング

Q1. エンティティが多すぎて図が複雑になってしまう

エンティティをグループ化して、段階的に表示することを検討してください。

例えば、「コアエンティティ図」と「詳細エンティティ図」に分けて作成すると分かりやすくなります。

Q2. 多対多リレーションシップの処理方法が分からない

多対多は中間テーブル（関連テーブル）を作成して、2つの1対多に分解します。

例: Task ⇔ Tag → Task → Task_Tag ← Tag

中間テーブルには両方のエンティティの主キーを外部キーとして持たせます。

Q3. 主キーに何を設定すべきか迷う

多くの場合、自動採番のID（user_id, task_id など）を主キーにするのが無難です。

メールアドレスなど変更の可能性がある属性を主キーにすると、後で困ることがあります。

Q4. draw.ioで線が複雑に交差してしまう

エンティティの配置を変更してみましょう。関連の強いエンティティを近くに配置すると線が短くなります。

また、線のスタイルを「直角」に設定すると、きれいに整理しやすくなります。
