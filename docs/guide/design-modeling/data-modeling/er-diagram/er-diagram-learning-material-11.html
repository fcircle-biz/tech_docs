11.1 物理ER図とは何か - 論理ER図との違い

これまで学んできた論理ER図は、ビジネス要件を整理し、データの関係性を明確にするための設計図でした。 一方、物理ER図は、論理ER図を基に「実際のデータベース（RDBMS）でどう実装するか」を表現する設計図です。

論理ER図と物理ER図の比較

観点 論理ER図 物理ER図
目的 ビジネス要件の整理 データベース実装の設計
エンティティ名 業務用語（顧客、注文） テーブル名（customers、orders）
属性名 日本語可（顧客名、登録日） カラム名（customer_name、created_at）
データ型 抽象的（文字列、数値、日付） 具体的（VARCHAR(100)、INT、DATETIME）
インデックス 考慮しない 検討・定義する
制約 概念的（必須/任意） 具体的（NOT NULL、UNIQUE、DEFAULT）

物理ER図では、データベース管理システム（MySQL、PostgreSQL、Oracle等）で実際に動くテーブルを定義するため、 より技術的で詳細な情報を含みます。

11.2 命名規則 - テーブル名・カラム名のルール

物理設計では、統一された命名規則に従うことが非常に重要です。 チーム全体で同じルールを守ることで、可読性が高く、メンテナンスしやすいデータベースになります。

一般的な命名規則

テーブル名の規則

  • 小文字のスネークケースを使用：customer_orders
  • 複数形を推奨：users、products
  • 意味が分かる名前：orders ○、tbl1 ✕
  • 予約語を避ける：user → users

カラム名の規則

  • 小文字のスネークケース：first_name、order_date
  • 主キーは id または テーブル名_id：customer_id
  • 外部キーは参照元テーブル名_id：customer_id（customersテーブルへの外部キー）
  • 日時カラムは目的を明示：created_at、updated_at
  • フラグは is_ または has_ で始める：is_active

11.3 データ型の選択 - 適切な型を選ぶ基準

論理ER図では「文字列」「数値」といった抽象的な型で十分でしたが、 物理設計では、データベースの具体的なデータ型（VARCHAR、INT、DATETIMEなど）を選択する必要があります。

主なデータ型と使い分け

文字列型

  • VARCHAR(n)：可変長文字列（名前、メールアドレスなど）
  • CHAR(n)：固定長文字列（郵便番号、都道府県コードなど）
  • TEXT：長文テキスト（商品説明、コメントなど）

VARCHAR(100)のように、想定される最大文字数を指定します。

数値型

  • INT：整数（ID、数量、年齢など）
  • BIGINT：大きな整数（アクセス数、ポイント累計など）
  • DECIMAL(p, s)：固定小数点（金額、税率など）
  • FLOAT/DOUBLE：浮動小数点（座標、統計値など）

金額はDECIMAL(10, 2)のように精度を指定して正確に扱います。

日時型

  • DATE：日付のみ（生年月日、注文日など）
  • TIME：時刻のみ（営業時間、レッスン開始時刻など）
  • DATETIME：日時（注文日時、更新日時など）
  • TIMESTAMP：タイムスタンプ（作成日時、更新日時など）

作成日時・更新日時はTIMESTAMPで自動記録できます。

真偽値型

  • BOOLEANまたはTINYINT(1)：フラグ（有効/無効、公開/非公開など）

is_active、has_permissionなどのフラグに使用します。

データ型選択のポイント

  • データの性質を考える：数値なのか、文字なのか、日時なのか
  • サイズを適切に：大きすぎるとストレージを無駄にし、小さすぎるとデータが入らない
  • 計算の精度を考慮：金額計算はFLOATではなくDECIMALを使う

11.4 インデックスの基本 - パフォーマンスを考慮する

インデックス（索引）は、データ検索を高速化するための仕組みです。 本の巻末にある索引と同じように、データベースのインデックスは特定のカラムの値を素早く見つけるために使われます。

インデックスとは

例えば、100万件の顧客データから「メールアドレスがexample@test.comの顧客」を探す場合、 インデックスがないと全件を順番にチェックする必要があります（フルスキャン）。 インデックスがあれば、すぐに該当レコードを見つけられます。

インデックスを設定すべきカラム

主キー

主キーには自動的にユニークインデックスが作成されます。

外部キー

JOINでよく使うため、インデックスを設定すると検索が高速化します。

検索条件でよく使うカラム

WHERE句、ORDER BY句で頻繁に使用されるカラムにはインデックスが有効です。

ユニーク制約のあるカラム

メールアドレス、ユーザー名など、重複を許さないカラムには効果的です。

インデックスの注意点

  • インデックスは万能ではない：むやみに設定すると、INSERT/UPDATEが遅くなる
  • ストレージを消費する：インデックス自体もディスク容量を使用する
  • 適切な選択が重要：頻繁に検索するカラムにのみ設定する

11.5 制約の定義 - データの整合性を保つ

物理設計では、データの整合性を保つために各種制約（Constraint）を定義します。 制約により、不正なデータの登録を防ぎ、データベースの品質を保証できます。

主な制約の種類

PRIMARY KEY 主キー制約

テーブル内でレコードを一意に識別するカラム。NULL不可、重複不可。

FOREIGN KEY 外部キー制約

他のテーブルの主キーを参照。参照整合性を保証し、存在しないIDの登録を防ぐ。

NOT NULL 非NULL制約

カラムに必ず値を入れる必要がある。NULL（空）を許可しない。

UNIQUE ユニーク制約

カラムの値が重複しないことを保証（メールアドレス、ユーザー名など）。

DEFAULT デフォルト値

値が指定されなかった場合のデフォルト値を設定（例：is_active DEFAULT TRUE）。

CHECK チェック制約

カラムの値が特定の条件を満たすことを保証（例：age >= 0）。

11.6 ER図からCREATE TABLE文への変換 - 実践演習

ここまで学んだ知識を使って、実際にER図からCREATE TABLE文を生成してみましょう。 例として、「顧客」「注文」「商品」のシンプルなER図を使います。

実習 11-1: 論理ER図から物理設計へ

以下の論理ER図を、物理設計に落とし込みます。

erDiagram CUSTOMER ||--o{ ORDER : places ORDER ||--|{ ORDER_LINE : contains PRODUCT ||--o{ ORDER_LINE : referenced_by CUSTOMER { int customer_id PK string name string email date registered_date } ORDER { int order_id PK date order_date int customer_id FK string status } ORDER_LINE { int order_line_id PK int order_id FK int product_id FK int quantity decimal unit_price } PRODUCT { int product_id PK string product_name decimal price int stock_quantity }

手順

  1. 1 各エンティティをテーブル名に変換（複数形、スネークケース）
  2. 2 各属性に適切なデータ型を割り当てる
  3. 3 主キー・外部キー・制約を定義する
  4. 4 CREATE TABLE文を作成する

解答例：customersテーブル

customers.sql 
-- 顧客テーブル
CREATE TABLE customers (
    customer_id INT AUTO_INCREMENT PRIMARY KEY,
    customer_name VARCHAR(100) NOT NULL,
    email VARCHAR(255) NOT NULL UNIQUE,
    registered_date DATE NOT NULL DEFAULT CURRENT_DATE,
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- インデックス追加
CREATE INDEX idx_email ON customers(email);

解答例：ordersテーブル

orders.sql 
-- 注文テーブル
CREATE TABLE orders (
    order_id INT AUTO_INCREMENT PRIMARY KEY,
    order_date DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    customer_id INT NOT NULL,
    status VARCHAR(20) NOT NULL DEFAULT 'pending',
    total_amount DECIMAL(10, 2),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
        ON DELETE RESTRICT ON UPDATE CASCADE
);

-- インデックス追加
CREATE INDEX idx_customer_id ON orders(customer_id);
CREATE INDEX idx_order_date ON orders(order_date);
CREATE INDEX idx_status ON orders(status);

解答例：productsテーブル

products.sql 
-- 商品テーブル
CREATE TABLE products (
    product_id INT AUTO_INCREMENT PRIMARY KEY,
    product_name VARCHAR(255) NOT NULL,
    price DECIMAL(10, 2) NOT NULL,
    stock_quantity INT NOT NULL DEFAULT 0,
    is_available BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    CHECK (price >= 0),
    CHECK (stock_quantity >= 0)
);

-- インデックス追加
CREATE INDEX idx_product_name ON products(product_name);

解答例：order_linesテーブル

order_lines.sql 
-- 注文明細テーブル
CREATE TABLE order_lines (
    order_line_id INT AUTO_INCREMENT PRIMARY KEY,
    order_id INT NOT NULL,
    product_id INT NOT NULL,
    quantity INT NOT NULL,
    unit_price DECIMAL(10, 2) NOT NULL,
    subtotal DECIMAL(10, 2) GENERATED ALWAYS AS (quantity * unit_price) STORED,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (order_id) REFERENCES orders(order_id)
        ON DELETE CASCADE ON UPDATE CASCADE,
    FOREIGN KEY (product_id) REFERENCES products(product_id)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    CHECK (quantity > 0),
    CHECK (unit_price >= 0)
);

-- インデックス追加
CREATE INDEX idx_order_id ON order_lines(order_id);
CREATE INDEX idx_product_id ON order_lines(product_id);

CREATE TABLE文のポイント

  • AUTO_INCREMENTで主キーを自動採番
  • NOT NULLで必須項目を明示
  • DEFAULTでデフォルト値を設定
  • FOREIGN KEYで参照整合性を保証
  • CHECKで値の範囲を制限
  • created_at/updated_atで変更履歴を自動記録

11.7 draw.ioでの物理ER図の描き方

draw.ioで物理ER図を描く際は、論理ER図にデータ型や制約情報を追加します。 以下のポイントに注意して、実装を意識した図を作成しましょう。

実習 11-2: draw.ioで物理ER図を描く

customersテーブルの物理ER図をdraw.ioで描いてみましょう。

手順

  1. 1 draw.ioを開き、「Entity Relation」テンプレートを選択
  2. 2 テーブルを表す矩形を配置し、テーブル名「customers」を記入
  3. 3 カラム名とデータ型を記載（例：customer_id INT PK、customer_name VARCHAR(100) NOT NULL）
  4. 4 主キーには「PK」、外部キーには「FK」、必須項目には「NOT NULL」を明記
  5. 5 他のテーブル（orders、products、order_lines）も同様に作成
  6. 6 外部キーの線を引き、カーディナリティを表現

理解度確認クイズ

  1. Q1. 論理ER図と物理ER図の違いを3つ挙げてください。

    回答を表示
    1. 目的：論理ER図はビジネス要件の整理、物理ER図はデータベース実装の設計
    2. 表記：論理ER図は業務用語（顧客、注文）、物理ER図はテーブル名（customers、orders）
    3. 詳細度：論理ER図は抽象的、物理ER図は具体的なデータ型や制約を含む

  2. Q2. 「顧客メールアドレス」を物理設計する場合、適切なカラム定義はどれですか？

    1. email TEXT
    2. email VARCHAR(255) NOT NULL UNIQUE
    3. email INT PRIMARY KEY
    4. email BOOLEAN DEFAULT TRUE
    回答を表示

    2. email VARCHAR(255) NOT NULL UNIQUEが正解です。

    メールアドレスは可変長文字列（VARCHAR）で、必須（NOT NULL）かつ重複不可（UNIQUE）にすべきです。長さは一般的に255文字で十分です。

  3. Q3. インデックスを設定すべきカラムとして適切でないものはどれですか？

    1. 主キー（customer_id）
    2. 外部キー（order_id）
    3. 検索条件でよく使うカラム（email）
    4. テキスト型の長文カラム（商品説明）
    回答を表示

    4. テキスト型の長文カラム（商品説明）が不適切です。

    長文テキストは検索条件として使われることが少なく、インデックスのサイズも大きくなるため、通常はインデックスを設定しません。全文検索が必要な場合は専用の機能を使います。

第11章のまとめ

  • 物理ER図は論理ER図を基に、実際のデータベース実装を設計する図である
  • 命名規則：テーブル名・カラム名は小文字のスネークケースで統一する
  • データ型は適切に選択し、VARCHAR、INT、DATE、DECIMAL等を使い分ける
  • インデックスは検索性能を向上させるが、過度な設定は避ける
  • 制約（PRIMARY KEY、FOREIGN KEY、NOT NULL、UNIQUE等）でデータの整合性を保証する
  • 物理ER図からCREATE TABLE文を生成し、実際のデータベースを構築できる
