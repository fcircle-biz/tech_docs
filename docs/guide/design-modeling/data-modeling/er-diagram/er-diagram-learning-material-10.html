正規化入門 - データの重複と矛盾を防ぐ

データベース設計の品質を高める「正規化」の基本を学び、第1〜第3正規形を習得します。

この章で学ぶこと

  • 正規化とは何か、その目的を理解する
  • データの重複と矛盾がもたらす問題を知る
  • 第1正規形、第2正規形、第3正規形の概念を習得する
  • 正規化の手順をdraw.ioで実践する方法を学ぶ

10.1 データの重複と矛盾の問題 - なぜ正規化が必要か

ER図を作成する際、「このままのテーブル設計で良いのだろうか？」と疑問に思うことがあります。 実は、適切に設計されていないテーブルには、データの重複や矛盾といった問題が潜んでいます。

問題のあるテーブル設計の例

以下は、注文情報をすべて1つのテーブルで管理しようとした例です。

非正規化テーブルの例（注文テーブル）

注文ID 顧客名 顧客住所 商品名 商品価格 数量
O001 山田太郎 東京都渋谷区... ノートPC 80,000円 1
O002 山田太郎 東京都渋谷区... マウス 2,000円 2
O003 佐藤花子 大阪府大阪市... ノートPC 85,000円 1

このテーブルの問題点

  • データの重複：山田太郎の住所が複数行に重複している
  • 更新の異常：山田太郎の住所が変わったとき、すべての行を更新しなければならない
  • 削除の異常：注文O002を削除すると、マウスの価格情報も失われる可能性がある
  • データの矛盾：同じノートPCなのに価格が80,000円と85,000円で異なっている

10.2 正規化とは何か - 定義と目的

正規化（Normalization）とは、データベースのテーブル設計を改善し、 データの重複や矛盾を防ぐための体系的な手法です。

正規化の定義

データベーステーブルをより小さく、関連性の高い複数のテーブルに分割することで、 データの一貫性を保ち、更新・削除時の異常を防ぐプロセスです。

正規化の目的

1. データの重複を最小化

同じ情報を複数の場所に保存しないようにします。

2. データの一貫性を保証

矛盾したデータが混在しないようにします。

3. 更新の容易性

一箇所の変更で、関連するすべてのデータに反映されます。

4. ストレージの効率化

重複がなくなることで、必要な記憶容量が減少します。

正規化の段階

正規化には段階があり、それぞれ「正規形（Normal Form）」と呼ばれます。 本章では、実務で最もよく使われる第1正規形、第2正規形、第3正規形を学びます。

2NF] C --> D[第3正規形
3NF] D --> E[ボイスコッド正規形
BCNF] E -.-> F[第4正規形

実務では第3正規形（3NF）まで適用することが一般的です。

10.3 第1正規形（1NF） - 繰り返し項目の排除

第1正規形（First Normal Form: 1NF）は、正規化の最初のステップです。 「1つのセルに1つの値だけ」という原則を守ります。

第1正規形の条件

  • 各カラムが原子的（分割不可能）な値を持つ
  • 繰り返しグループがない
  • 各行が一意に識別できる（主キーが存在する）

第1正規形違反の例

❌ 第1正規形に違反（繰り返し項目あり）

顧客ID 顧客名 電話番号
C001 山田太郎 090-1111-2222, 03-3333-4444

問題点：電話番号カラムに複数の値がカンマ区切りで入っている

✅ 第1正規形に適合（別テーブルに分離）

顧客テーブル

顧客ID 顧客名
C001 山田太郎

電話番号テーブル

電話番号ID 顧客ID 電話番号 種別
P001 C001 090-1111-2222 携帯
P002 C001 03-3333-4444 自宅

10.4 第2正規形（2NF） - 部分関数従属の排除

第2正規形（Second Normal Form: 2NF）は、第1正規形を満たした上で、 「主キーの一部だけに依存する属性」を排除します。

第2正規形の条件

  • 第1正規形を満たしている
  • すべての非キー属性が、主キー全体に完全関数従属する
  • （主キーが単一カラムの場合は自動的に第2正規形を満たす）

部分関数従属とは？

複合主キー（2つ以上のカラムで構成される主キー）の場合に、 主キーの一部のカラムだけで他のカラムの値が決まってしまう状態を指します。

第2正規形違反の例

❌ 第2正規形に違反（部分関数従属あり）

注文明細テーブル（主キー：注文ID + 商品ID）

注文ID PK 商品ID PK 商品名 商品価格 数量
O001 P001 ノートPC 80,000円 1
O001 P002 マウス 2,000円 2

問題点：商品名と商品価格は「商品ID」だけで決まる（注文IDは不要）

✅ 第2正規形に適合（テーブル分割）

商品テーブル

商品ID PK 商品名 商品価格
P001 ノートPC 80,000円
P002 マウス 2,000円

注文明細テーブル

注文ID PK 商品ID PK/FK 数量
O001 P001 1
O001 P002 2

10.5 第3正規形（3NF） - 推移的関数従属の排除

第3正規形（Third Normal Form: 3NF）は、第2正規形を満たした上で、 「主キー以外のカラムに依存する属性」を排除します。

第3正規形の条件

  • 第2正規形を満たしている
  • 非キー属性が他の非キー属性に依存していない
  • （推移的関数従属がない）

推移的関数従属とは？

主キー → カラムA → カラムB のように、 間に別のカラムを経由して値が決まる状態を指します。 例：社員ID → 部署ID → 部署名（社員IDから部署名が間接的に決まる）

第3正規形違反の例

❌ 第3正規形に違反（推移的関数従属あり）

社員テーブル（主キー：社員ID）

社員ID PK 社員名 部署ID 部署名 部署の電話番号
E001 山田太郎 D01 営業部 03-1111-2222
E002 佐藤花子 D01 営業部 03-1111-2222

問題点：部署名と部署の電話番号は「部署ID」だけで決まる（社員IDを経由している）

✅ 第3正規形に適合（テーブル分割）

部署テーブル

部署ID PK 部署名 部署の電話番号
D01 営業部 03-1111-2222

社員テーブル

社員ID PK 社員名 部署ID FK
E001 山田太郎 D01
E002 佐藤花子 D01

10.6 正規化の実践 - draw.ioで描いてみよう

正規化の概念を理解したら、実際にdraw.ioでER図を描きながら正規化を実践してみましょう。

実習 10-1: 正規化前後のER図を描く

「学生の履修科目管理」を題材に、非正規形から第3正規形までのER図を段階的に作成します。

題材：学生履修管理システム

学生が複数の科目を履修し、各科目には担当教員がいる。 同じ科目でも異なる教員が担当する場合があります。

手順

  1. 1 draw.ioで新規図を作成し、「ER図」テンプレートを選択
  2. 2 非正規形のテーブルを描く（学生ID、学生名、科目ID、科目名、教員名、成績を1つのテーブルに）
  3. 3 第1正規形に修正（繰り返し項目を排除し、主キーを設定）
  4. 4 第2正規形に修正（部分関数従属を排除し、学生テーブルと科目テーブルを分離）
  5. 5 第3正規形に修正（推移的関数従属を排除し、教員テーブルを分離）
  6. 6 各テーブル間のリレーションシップを線で接続し、カーディナリティを設定

正規化後のER図イメージ

erDiagram STUDENT ||--o{ ENROLLMENT : enrolls COURSE ||--o{ ENROLLMENT : contains TEACHER ||--o{ COURSE : teaches STUDENT { string student_id PK string student_name } TEACHER { string teacher_id PK string teacher_name } COURSE { string course_id PK string course_name string teacher_id FK } ENROLLMENT { string enrollment_id PK string student_id FK string course_id FK string grade }

学生、教員、科目、履修の4テーブルに正規化され、データの重複がなくなりました。

正規化のポイント

  • 学生情報は学生テーブルに一箇所だけ保存
  • 科目情報は科目テーブルに一箇所だけ保存
  • 教員情報は教員テーブルに一箇所だけ保存
  • 履修関係は履修テーブルで管理（成績もここに記録）

10.7 過度な正規化の弊害 - 非正規化の検討

正規化は重要ですが、やりすぎると逆にパフォーマンスが悪化する場合があります。 実務では、状況に応じて意図的に「非正規化」を行うこともあります。

過度な正規化の問題点

1. JOINの増加で遅くなる

テーブルを細かく分けすぎると、データ取得時に多くのテーブルを結合する必要があり、処理が遅くなります。

2. SQLが複雑になる

シンプルなデータ取得でも、複雑なJOIN文を書かなければならなくなります。

3. 理解しづらくなる

テーブル数が多すぎると、全体像の把握が難しくなります。

4. メンテナンスコストの増加

テーブル数が増えると、変更時の影響範囲が広がります。

非正規化を検討すべきケース

非正規化が有効な場合

  • 読み取り専用のレポート用テーブル：更新異常の心配がない場合
  • 頻繁にアクセスされるデータ：JOINのコストが大きい場合
  • 集計値の保存：毎回計算するよりも、結果を保存した方が効率的な場合
  • 履歴データ：過去のスナップショットとして保存する場合

注意

非正規化は、正規化を理解した上で、パフォーマンスやビジネス要件のために意図的に行うものです。 最初から非正規化されたテーブルを作るのではなく、まずは正規化し、必要に応じて非正規化を検討するのが基本です。

理解度確認クイズ

  1. Q1. 正規化の主な目的を3つ挙げてください。

    回答を表示
    1. データの重複を最小化：同じ情報を複数の場所に保存しない
    2. データの一貫性を保証：矛盾したデータが混在しないようにする
    3. 更新の容易性：一箇所の変更で関連するすべてのデータに反映される

  2. Q2. 第1正規形、第2正規形、第3正規形のそれぞれで排除する問題を説明してください。

    回答を表示
    • 第1正規形（1NF）：繰り返し項目の排除。1つのセルに複数の値を入れない。
    • 第2正規形（2NF）：部分関数従属の排除。複合主キーの一部だけに依存する属性を別テーブルに分離。
    • 第3正規形（3NF）：推移的関数従属の排除。主キー以外のカラムに依存する属性を別テーブルに分離。

  3. Q3. 非正規化を検討すべきケースとして適切なものはどれですか？（複数選択可）

    1. 頻繁に更新される顧客マスタテーブル
    2. 読み取り専用のレポート用テーブル
    3. リアルタイム性が重視される在庫管理テーブル
    4. 過去の注文データの履歴スナップショット
    回答を表示

    2と4が正解です。

    • 2. 読み取り専用のレポート用テーブル：更新がないため、非正規化によるパフォーマンス向上が有効
    • 4. 過去の注文データの履歴スナップショット：履歴として固定されるため、その時点の状態を保存する非正規化が適切

    1と3は更新が頻繁に発生するため、正規化を維持すべきです。

第10章のまとめ

  • 正規化とは、データの重複と矛盾を防ぐための体系的な手法である
  • 第1正規形（1NF）は、繰り返し項目を排除し、各セルに1つの値だけを格納する
  • 第2正規形（2NF）は、部分関数従属を排除し、複合主キーの一部だけに依存する属性を分離する
  • 第3正規形（3NF）は、推移的関数従属を排除し、主キー以外のカラムに依存する属性を分離する
  • 過度な正規化はパフォーマンス低下を招くため、実務では意図的な非正規化も検討する
  • draw.ioで正規化前後のER図を描くことで、データ構造の改善を視覚的に確認できる
