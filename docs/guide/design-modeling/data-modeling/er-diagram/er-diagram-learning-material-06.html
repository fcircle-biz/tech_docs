第6章

カーディナリティの理解 - 1対1、1対多、多対多

エンティティ間の数量関係を正しく表現し、IE記法（鳥の足記法）で図示する方法を学びます。

この章で学ぶこと

  • カーディナリティ（数量関係）とは何かを理解する
  • 1対1、1対多、多対多の違いを具体例で学ぶ
  • IE記法（鳥の足記法）での表記方法をマスターする
  • オプショナリティ（必須/任意）の概念を理解する
  • draw.ioでカーディナリティを正しく描く方法を実践する

6.1 カーディナリティとは何か

前章でリレーションシップ（エンティティ間の関係）を学びましたが、それだけでは不十分です。 「一人の顧客は何件の注文ができるのか？」「一つの注文は何人の顧客に属するのか？」といった数量的な関係も明確にする必要があります。

カーディナリティ（Cardinality）とは

リレーションシップにおける数量的な関係を表すものです。 一方のエンティティの1つのインスタンスが、他方のエンティティのいくつのインスタンスと関連するかを示します。

カーディナリティの主な種類

1対1（One-to-One）

一方の1つのインスタンスが、他方の1つのインスタンスとのみ関連する

1対多（One-to-Many）

一方の1つのインスタンスが、他方の複数のインスタンスと関連する（最も一般的）

多対多（Many-to-Many）

一方の複数のインスタンスが、他方の複数のインスタンスと関連する

6.2 1対1（One-to-One）の関係

1対1の関係は、一方のエンティティの1つのインスタンスが、他方のエンティティの1つのインスタンスとのみ関連する関係です。 実務では比較的レアなケースですが、重要な概念です。

具体例：社員と社員証

一人の社員は一つの社員証を持ち、一つの社員証は一人の社員に属します。

erDiagram EMPLOYEE ||--|| EMPLOYEE_CARD : has EMPLOYEE { string employee_id PK string name string department } EMPLOYEE_CARD { string card_id PK string employee_id FK date issue_date date expiry_date }

||--||が1対1の関係を示しています。両端の||は「1」を意味します。

1対1の典型的なユースケース

  • 社員と社員証（一人の社員に一つの社員証）
  • 国と首都（一つの国に一つの首都）
  • ユーザーとユーザー設定（一人のユーザーに一つの設定データ）
  • 商品と商品詳細（基本情報と詳細情報を分離する場合）

6.3 1対多（One-to-Many）の関係

1対多の関係は、最も頻繁に登場するカーディナリティです。 一方のエンティティの1つのインスタンスが、他方のエンティティの複数のインスタンスと関連します。

具体例：顧客と注文

一人の顧客は複数の注文を行うことができますが、一つの注文は一人の顧客にのみ属します。

erDiagram CUSTOMER ||--o{ ORDER : places CUSTOMER { string customer_id PK string name string email string phone } ORDER { string order_id PK date order_date decimal total_amount string customer_id FK }

||--o{が1対多の関係を示しています。||は「1」、o{は「0個以上」を意味します。

1対多の典型的なパターン

1

部署と社員

一つの部署に複数の社員が所属、一人の社員は一つの部署に所属

2

カテゴリと商品

一つのカテゴリに複数の商品が属し、一つの商品は一つのカテゴリに属する

3

ブログ記事とコメント

一つの記事に複数のコメントが付き、一つのコメントは一つの記事に属する

4

親フォルダと子ファイル

一つのフォルダに複数のファイルが含まれ、一つのファイルは一つのフォルダに属する

6.4 多対多（Many-to-Many）の関係

多対多の関係は、一方の複数のインスタンスが、他方の複数のインスタンスと関連する関係です。 実務では非常によく登場しますが、データベースの実装時には中間テーブルを使って表現します。

具体例：学生と授業

一人の学生は複数の授業を受講でき、一つの授業には複数の学生が参加できます。

erDiagram STUDENT }o--o{ ENROLLMENT : enrolls ENROLLMENT }o--o{ COURSE : contains STUDENT { string student_id PK string name string major } ENROLLMENT { string enrollment_id PK string student_id FK string course_id FK date enrollment_date string grade } COURSE { string course_id PK string course_name string instructor int credits }

多対多の関係を実装するため、ENROLLMENT（受講）という中間テーブルを使用しています。

多対多は直接実装できない！

データベース（リレーショナルデータベース）では、多対多の関係を直接実装することはできません。 必ず中間テーブル（ブリッジテーブル、連関エンティティ）を作成し、2つの1対多の関係に分解する必要があります。

多対多の典型例

1. 商品とタグ

一つの商品に複数のタグ、一つのタグが複数の商品に付く

中間テーブル: 商品タグ関連

2. 著者と書籍

一人の著者が複数の本を執筆、一つの本に複数の著者

中間テーブル: 著作

3. プロジェクトと社員

一つのプロジェクトに複数の社員、一人の社員が複数のプロジェクト参加

中間テーブル: プロジェクト参加

4. 映画と俳優

一つの映画に複数の俳優、一人の俳優が複数の映画に出演

中間テーブル: 出演

6.5 オプショナリティ（必須/任意）

カーディナリティに加えて、オプショナリティ（Optionality）も重要な概念です。 リレーションシップが必須（Mandatory）なのか、任意（Optional）なのかを表します。

オプショナリティの種類

  • 必須（Mandatory）：関連が必ず存在する（最小カーディナリティ = 1）
  • 任意（Optional）：関連が存在しない場合もある（最小カーディナリティ = 0）

IE記法での表現

IE記法（鳥の足記法）では、以下のように表記します：

記号の意味

|| 必須で1（Exactly One）
|o 任意で0または1（Zero or One）
|{ 必須で1以上（One or More）
o{ 任意で0以上（Zero or More）

具体例：部署と社員

erDiagram DEPARTMENT ||--o{ EMPLOYEE : employs DEPARTMENT { string dept_id PK string dept_name } EMPLOYEE { string emp_id PK string name string dept_id FK }

||--o{の意味：
• 左側（部署）：|| = 社員は必ず1つの部署に所属
• 右側（社員）：o{ = 部署には0人以上の社員が所属（社員がいない部署もあり得る）

ビジネスルールを正確に反映する

「顧客は注文がなくても存在できるか？」「社員は必ず部署に所属するか？」といったビジネスルールを、 オプショナリティを使って正確に表現しましょう。

6.6 draw.ioでカーディナリティを描く実習

draw.ioを使って、カーディナリティを正しく表現するER図を描いてみましょう。

実習：ブログシステムのER図作成

以下の要件を満たすER図をdraw.ioで作成してみましょう。

要件：

  • ユーザー（USER）エンティティ：user_id、username、email
  • 記事（POST）エンティティ：post_id、title、content、user_id
  • コメント（COMMENT）エンティティ：comment_id、post_id、user_id、comment_text
  • 一人のユーザーは複数の記事を投稿できる（0件でも可）
  • 一つの記事は必ず一人のユーザーによって書かれる
  • 一つの記事には複数のコメントが付けられる（0件でも可）
  • 一つのコメントは必ず一つの記事に属する
  • 一人のユーザーは複数のコメントを投稿できる（0件でも可）
  • 一つのコメントは必ず一人のユーザーによって書かれる

手順

1

draw.ioを開き、ER図テンプレートを選択

https://app.diagrams.net/ にアクセスし、「Entity Relation」シェイプライブラリを有効化します。

2

3つのエンティティを配置

「Entity Relation」パレットから「Table」シェイプをドラッグし、USER、POST、COMMENTを作成します。 各エンティティに属性（PK、FK付き）を追加します。

3

リレーションシップを接続

「Entity Relation」パレットの接続線を使用して、以下を接続します：
• USER → POST（一人のユーザーが複数の記事を投稿）
• POST → COMMENT（一つの記事に複数のコメント）
• USER → COMMENT（一人のユーザーが複数のコメントを投稿）

4

カーディナリティを設定

各リレーションシップ線を選択し、右側のプロパティパネルで「Line end」（線の端点）を変更します：
• USER → POST：||--o{（必須1対任意多）
• POST → COMMENT：||--o{（必須1対任意多）
• USER → COMMENT：||--o{（必須1対任意多）

完成イメージ

erDiagram USER ||--o{ POST : writes USER ||--o{ COMMENT : posts POST ||--o{ COMMENT : has USER { string user_id PK string username string email } POST { string post_id PK string title string content string user_id FK } COMMENT { string comment_id PK string comment_text string post_id FK string user_id FK }

理解度確認クイズ

  1. Q1. IE記法で||--o{が表すカーディナリティを説明してください。

    回答を表示

    1対多（One-to-Many）の関係です。

    • 左側（||）：必ず1つ（Exactly One）

    • 右側（o{）：0個以上（Zero or More）

    例：一人の顧客（1）が複数の注文（0以上）を行う関係

  2. Q2. 多対多の関係をデータベースで実装する際、なぜ中間テーブルが必要ですか？

    回答を表示

    リレーショナルデータベースでは、多対多の関係を直接実装できないからです。

    中間テーブルを作成することで、多対多の関係を2つの1対多の関係に分解し、正しくデータを管理できます。

    例：学生（STUDENT）と授業（COURSE）の多対多を、受講（ENROLLMENT）という中間テーブルで、STUDENT→ENROLLMENT（1対多）とCOURSE→ENROLLMENT（1対多）に分解します。

  3. Q3. 以下のシナリオのカーディナリティを答えてください：「一つの国には複数の都市があり、一つの都市は必ず一つの国に属する。ただし、国には都市が登録されていない場合もある。」

    回答を表示

    1対多（One-to-Many）で、オプショナリティを含む関係です。

    IE記法で表すと：COUNTRY ||--o{ CITY

    • 左側（国）：|| = 都市は必ず1つの国に属する

    • 右側（都市）：o{ = 国には0個以上の都市がある（都市がない国もあり得る）

  4. Q4. 「社員証」と「社員」の関係が1対1である理由を説明してください。また、他に1対1の関係が適切な例を1つ挙げてください。

    回答を表示

    理由：一人の社員は一つの社員証のみを持ち、一つの社員証は一人の社員にのみ発行されるためです。

    他の1対1の例：

    • 国と首都（一つの国に一つの首都）
    • ユーザーとユーザー設定（一人のユーザーに一つの設定データ）
    • パスポートと所有者（一つのパスポートは一人の所有者）

第6章のまとめ

  • カーディナリティは、エンティティ間の数量関係を表す重要な概念
  • 1対1：一方の1つが他方の1つとのみ関連（例：社員と社員証）
  • 1対多：一方の1つが他方の複数と関連（最も一般的、例：顧客と注文）
  • 多対多：両方が複数と関連し、中間テーブルで実装（例：学生と授業）
  • オプショナリティは必須（|）か任意（o）かを表す
  • IE記法（鳥の足記法）で||、|o、|{、o{を使い分ける
  • draw.ioでリレーションシップ線の端点を変更してカーディナリティを表現できる
