第9章

論理ER図への展開 - 詳細な設計を進める

概念ER図を基に、属性の詳細化、データ型の検討、主キー・外部キーの明確化、多対多リレーションシップの解消を学びます。

この章で学ぶこと

  • 概念ER図と論理ER図の違いを理解する
  • 属性の詳細化とデータ型の選定方法を習得する
  • 主キー・外部キーを明確に表現する方法を学ぶ
  • 多対多リレーションシップを中間テーブルで解消する手法を実践する

9.1 概念ER図と論理ER図の違い

第8章で作成した概念ER図は、ビジネス要件を整理し、 エンティティとその関係性を大まかに表現したものでした。 これに対して論理ER図は、より詳細な設計を行い、 実際のデータベース実装に近づけたものです。

概念ER図と論理ER図の比較

項目 概念ER図 論理ER図
目的 ビジネス要件の整理 データベース設計の詳細化
エンティティ 主要なもののみ 中間テーブルを含むすべて
属性 主要な属性のみ 全属性＋データ型
主キー 暗黙的 明示的に表記
外部キー 記載なし 明示的に表記
多対多 そのまま表現 中間テーブルで解消

論理ER図の役割

論理ER図は、概念ER図で定義したビジネス要件を、 データベースに実装できる形に変換する橋渡しの役割を果たします。 次のステップである物理ER図（第11章）では、 さらに具体的なDBMS固有の仕様を反映します。

9.2 属性の詳細化とデータ型の検討

論理ER図では、各エンティティのすべての属性を洗い出し、 それぞれに適切なデータ型を割り当てます。

主なデータ型の種類

文字列型

  • VARCHAR(n): 可変長文字列（例: 顧客名、メールアドレス）
  • CHAR(n): 固定長文字列（例: 都道府県コード）
  • TEXT: 長文テキスト（例: 商品説明）

数値型

  • INTEGER: 整数（例: 数量、年齢）
  • DECIMAL(p,s): 固定小数点数（例: 価格、金額）
  • FLOAT: 浮動小数点数（例: 割合）

日付・時刻型

  • DATE: 日付（例: 注文日）
  • TIME: 時刻（例: 営業開始時刻）
  • TIMESTAMP: 日時（例: 登録日時）

その他の型

  • BOOLEAN: 真偽値（例: 有効フラグ）
  • ENUM: 列挙型（例: ステータス）
  • BLOB: バイナリデータ（例: 画像）

実例: 書籍エンティティの属性詳細化

erDiagram BOOK { INTEGER book_id PK "書籍ID（主キー）" VARCHAR title "タイトル（最大200文字）" VARCHAR isbn "ISBN（13桁）" DATE publication_date "出版日" DECIMAL price "価格（10桁、小数点以下2桁）" INTEGER stock_quantity "在庫数" TEXT description "説明文" BOOLEAN is_available "販売可能フラグ" }

データ型選定のポイント

  • 価格や金額はDECIMALを使う（浮動小数点数は誤差が生じるため）
  • 文字列の長さは実際のデータを考慮して設定（余裕を持たせすぎない）
  • 将来の拡張性も考慮する（例: 電話番号は国際化を想定）

9.3 主キー・外部キーの明確化

論理ER図では、主キー（PK: Primary Key）と外部キー（FK: Foreign Key）を 属性として明示的に記載します。これにより、エンティティ間の関連性がより明確になります。

主キー・外部キーの表記方法

オンライン書店システムの例

erDiagram CUSTOMER ||--o{ ORDER : "注文する" ORDER ||--|{ ORDER_DETAIL : "含む" BOOK ||--o{ ORDER_DETAIL : "参照される" AUTHOR ||--o{ BOOK : "執筆する" CUSTOMER { INTEGER customer_id PK "顧客ID（主キー）" VARCHAR name "顧客名" VARCHAR email "メールアドレス" VARCHAR phone "電話番号" } ORDER { INTEGER order_id PK "注文ID（主キー）" INTEGER customer_id FK "顧客ID（外部キー）" DATE order_date "注文日" DECIMAL total_amount "合計金額" } ORDER_DETAIL { INTEGER order_detail_id PK "注文明細ID（主キー）" INTEGER order_id FK "注文ID（外部キー）" INTEGER book_id FK "書籍ID（外部キー）" INTEGER quantity "数量" DECIMAL unit_price "単価" } BOOK { INTEGER book_id PK "書籍ID（主キー）" INTEGER author_id FK "著者ID（外部キー）" VARCHAR title "タイトル" DECIMAL price "価格" } AUTHOR { INTEGER author_id PK "著者ID（主キー）" VARCHAR name "著者名" TEXT biography "略歴" }

主キー・外部キーの命名規則

  • 主キー: エンティティ名の単数形 + _id（例: customer_id, order_id）
  • 外部キー: 参照先の主キーと同じ名前を使う（関連性が一目で分かる）
  • 複合主キー: 複数の属性を組み合わせて主キーとする場合もある

draw.ioでの主キー・外部キーの表現

draw.ioでの設定手順

  1. 1.
    エンティティ（テーブル）を配置

    左側のパネルから「ER図」または「Entity Relation」を選択し、テーブル図形を配置

  2. 2.
    属性を追加

    テーブルをダブルクリックして、属性名とデータ型を入力

  3. 3.
    主キーをマーク

    属性の横に「PK」または鍵アイコンを追加（多くのテンプレートには自動で設定可能）

  4. 4.
    外部キーをマーク

    属性の横に「FK」を追加し、参照先との関連線を引く

9.4 多対多リレーションシップの解消

概念ER図では「多対多」の関係性をそのまま表現しましたが、 実際のデータベースでは多対多の関係は直接実装できません。 そのため、中間テーブルを作成して、2つの「1対多」の関係に分解します。

多対多の問題点

多対多の関係をそのまま実装できない理由

  • 一つのレコードが複数の関連先を持つ場合、データの保存方法が曖昧になる
  • 関連の属性（例: 注文数量）をどちらのテーブルに保存すべきか不明確
  • データの整合性を保つことが困難

中間テーブルによる解消

例: 学生と授業の多対多関係

問題: 一人の学生は複数の授業を受講し、一つの授業には複数の学生が参加する（多対多）

❌ 概念ER図（多対多）
erDiagram STUDENT }o--o{ COURSE : "受講する" STUDENT { INTEGER student_id PK VARCHAR name } COURSE { INTEGER course_id PK VARCHAR course_name }

このままでは実装できない

✅ 論理ER図（中間テーブル）
erDiagram STUDENT ||--o{ ENROLLMENT : "登録" COURSE ||--o{ ENROLLMENT : "登録される" STUDENT { INTEGER student_id PK VARCHAR name } ENROLLMENT { INTEGER enrollment_id PK INTEGER student_id FK INTEGER course_id FK DATE enrollment_date VARCHAR grade } COURSE { INTEGER course_id PK VARCHAR course_name }

中間テーブルENROLLMENTで2つの1対多に分解

中間テーブルの利点

  • 関連に関する追加情報（例: 受講日、成績）を保存できる
  • データの整合性を保ちやすい
  • 検索や集計が容易になる

もう一つの例: 商品とタグの関係

ECサイトでの商品タグ付け

erDiagram PRODUCT ||--o{ PRODUCT_TAG : "タグ付けされる" TAG ||--o{ PRODUCT_TAG : "商品に付与される" PRODUCT { INTEGER product_id PK VARCHAR product_name DECIMAL price } PRODUCT_TAG { INTEGER product_tag_id PK INTEGER product_id FK INTEGER tag_id FK TIMESTAMP tagged_at } TAG { INTEGER tag_id PK VARCHAR tag_name }

一つの商品には複数のタグ（例: 「新作」「セール」「おすすめ」）が付けられ、 一つのタグは複数の商品に付けられます。 中間テーブルPRODUCT_TAGで、 いつタグ付けされたかの情報も保存できます。

draw.ioでの表現のコツ

  • 色分け: エンティティを機能ごとに色分けすると見やすい（例: マスターデータ=青、トランザクション=緑）
  • 配置: 主要なエンティティを中央に、関連エンティティを周囲に配置
  • 線の引き方: 交差を最小限にし、直線または直角の線で接続
  • 注釈: 必要に応じてメモやコメントを追加（ビジネスルールなど）

確認ポイント

  • すべてのエンティティに主キーが設定されているか
  • 外部キーが参照先の主キーと整合しているか
  • 多対多の関係が中間テーブルで解消されているか
  • 各属性に適切なデータ型が割り当てられているか
  • ビジネス要件を満たすすべての属性が含まれているか

理解度確認クイズ

  1. Q1. 概念ER図と論理ER図の最も大きな違いは何ですか？

    回答を表示

    論理ER図では、属性の詳細化、主キー・外部キーの明示、多対多リレーションシップの解消（中間テーブル）が行われます。

    概念ER図はビジネス要件の整理が目的ですが、論理ER図は実際のデータベース実装に向けた詳細設計が目的です。

  2. Q2. 価格や金額を保存する際に、なぜDECIMAL型を使うべきですか？

    回答を表示

    DECIMAL型は固定小数点数であり、正確な値を保存できるためです。

    FLOAT型（浮動小数点数）は計算時に誤差が生じる可能性があり、金額計算では問題になります。例えば、0.1 + 0.2が正確に0.3にならないことがあります。

  3. Q3. 「学生」と「サークル」の関係が多対多の場合、中間テーブルの名前としてふさわしいものはどれですか？

    1. STUDENT_CIRCLE
    2. MEMBERSHIP
    3. ENROLLMENT
    4. 上記のいずれも正解
    回答を表示

    4. 上記のいずれも正解

    中間テーブルの命名には以下の選択肢があります：

    • STUDENT_CIRCLE: 2つのエンティティ名を組み合わせる（分かりやすい）
    • MEMBERSHIP: 関係を表す名詞を使う（より自然）
    • ENROLLMENT: 「登録」という意味（サークルへの加入を表す）

    いずれも適切ですが、ビジネスの文脈に合った名前を選ぶとより分かりやすくなります。

第9章のまとめ

  • 論理ER図は、概念ER図を基にデータベース実装に向けた詳細設計を行う
  • 各属性に適切なデータ型を割り当てることで、データの整合性を保つ
  • 主キー（PK）と外部キー（FK）を明示することで、エンティティ間の関連性を明確にする
  • 多対多リレーションシップは、中間テーブルを作成して2つの1対多の関係に分解する
  • draw.ioを使って、視覚的に分かりやすい論理ER図を作成できる
