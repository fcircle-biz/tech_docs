ER図とUMLの関係 - クラス図との比較

ER図と似た役割を持つUMLクラス図との関係を学び、設計スキルの幅を広げましょう。

この章で学ぶこと

  • UMLとクラス図の基本的な概念を理解する
  • ER図とクラス図の類似点と相違点を知る
  • どのような場面でどちらを使うべきか判断できるようになる
  • オブジェクト指向設計とデータベース設計を連携させる視点を養う

14.1 UMLとは何か - 統一モデリング言語

ER図を学んできた皆さんは、「図でシステムを表現する」ことに慣れてきたと思います。 実は、ソフトウェア開発の世界にはUML（Unified Modeling Language、統一モデリング言語）という、より広範囲なシステムを表現するための図式表記法があります。

UMLとは

UMLは、ソフトウェアシステムの構造や動作を視覚的に表現するための標準的な図式言語です。 1つの図だけでなく、14種類の図を使い分けて、システムのさまざまな側面を表現します。

  • 構造図：クラス図、オブジェクト図、コンポーネント図など（システムの構成要素）
  • 振る舞い図：ユースケース図、シーケンス図、状態遷移図など（システムの動き）

この中で、クラス図は特にER図と似た役割を持ち、データの構造を表現する図です。 今回は、このクラス図を中心にER図との関係を見ていきましょう。

14.2 クラス図の基本 - オブジェクト指向設計の中心

クラス図は、オブジェクト指向プログラミングにおけるクラスとその関係を表現する図です。 「クラス」とは、データ（属性）と処理（メソッド）をまとめたものです。

クラス図の基本構成要素

classDiagram class Customer { -customerId String -name String -email String +placeOrder() Order +updateProfile() void } class Order { -orderId String -orderDate Date -totalAmount Decimal +addProduct() void +calculateTotal() Decimal } Customer "1" --> "*" Order : places

クラス図では、データ（属性）だけでなく、処理（メソッド）も記載します。

クラス図の要素

1

クラス名

例：Customer、Order（通常は先頭が大文字）

2

属性（Attribute）

データの項目。例：customerId、name、email

3

メソッド（Method）

処理・操作。例：placeOrder、updateProfile

4

関連（Association）

クラス間の関係を線で表現

14.3 ER図とクラス図の類似点 - 共通する考え方

ER図とクラス図には、多くの共通点があります。両方とも「データの構造を視覚化する」という目的を持っているためです。

1. データの集まりを表現

ER図：エンティティ（実体）
クラス図：クラス

どちらも「顧客」「商品」など、管理したいデータのまとまりを四角形で表現します。

2. 詳細情報を持つ

ER図：属性（Attribute）
クラス図：属性（Attribute）

どちらも「顧客ID」「顧客名」「メールアドレス」といった詳細データを持ちます。

3. 関係性を表現

ER図：リレーションシップ
クラス図：関連（Association）

どちらも「顧客が注文する」といった関係性を線で表します。

4. 多重度の表現

ER図：カーディナリティ（1対多、多対多など）
クラス図：多重度（1、*、0..1など）

どちらも「1人の顧客が複数の注文をする」といった数量関係を表現できます。

14.4 ER図とクラス図の相違点 - 目的の違い

一方で、ER図とクラス図には重要な違いもあります。それは設計の目的と焦点の違いです。

項目 ER図 クラス図
主な目的 データベースの設計 プログラムの設計
焦点 データの保存構造 データ＋処理（振る舞い）
処理の記述 記載しない メソッドとして記載する
実装先 データベース（SQL） プログラミング言語（Java、Pythonなど）
主キー 必須（主キーを明示） 必須ではない
正規化 重視する（データの重複排除） 柔軟（設計の自由度が高い）
継承 表現しづらい 明確に表現できる

最大の違い：処理（振る舞い）の表現

ER図は「データをどう保存するか」に焦点を当てており、処理は記載しません。 一方、クラス図は「データをどう扱うか（処理）」も含めて設計するため、メソッド（関数）を記載します。

14.5 使い分けの指針 - どちらを使うべきか

ER図とクラス図、どちらを使うべきかは、設計のフェーズと目的によって変わります。

使い分けのガイドライン

ER図を使うべき場面

  • データベース設計が主な目的の場合
  • データの保存構造を明確にしたい場合
  • 正規化を重視したい場合
  • SQLデータベースに直接マッピングしたい場合
  • 業務担当者とデータの関係性を議論する場合

クラス図を使うべき場面

  • オブジェクト指向プログラムの設計をする場合
  • データだけでなく処理（メソッド）も設計したい場合
  • 継承やポリモーフィズムを表現したい場合
  • APIやライブラリの設計をドキュメント化する場合
  • 開発者間で実装方針を共有する場合

ヒント：両方を組み合わせる

実際のプロジェクトでは、ER図でデータベースを設計し、クラス図でアプリケーションの構造を設計するなど、 両方を組み合わせて使うことが一般的です。

14.6 連携した設計の実践 - ER図からクラス図へ

ER図とクラス図は、互いに補完し合う関係にあります。 実際のシステム開発では、ER図で設計したデータベースをクラス図で表現されたプログラムから操作することになります。

実践例：オンライン書店システム

ステップ1：ER図でデータベース設計

erDiagram CUSTOMER ||--o{ ORDER : places ORDER ||--|{ ORDER_ITEM : contains BOOK ||--o{ ORDER_ITEM : referenced_by CUSTOMER { string customer_id PK string name string email } ORDER { string order_id PK date order_date string customer_id FK } ORDER_ITEM { string order_item_id PK string order_id FK string book_id FK int quantity } BOOK { string book_id PK string title string author decimal price }

ステップ2：クラス図でアプリケーション設計

classDiagram class Customer { -customerId String -name String -email String +placeOrder() Order +getOrderHistory() List } class Order { -orderId String -orderDate Date -items List +addItem() void +calculateTotal() Decimal +submit() boolean } class Book { -bookId String -title String -author String -price Decimal +getDetails() String +isAvailable() boolean } Customer "1" --> "*" Order : places Order "1" --> "*" Book : contains

対応関係

  • ER図のエンティティ → クラス図のクラス（データ構造は同じ）
  • ER図の属性 → クラス図の属性（フィールド名が一致）
  • クラス図のメソッド：データベース操作やビジネスロジックを追加

連携設計のメリット

一貫性の確保

データベースとプログラムのデータ構造が一致するため、開発がスムーズに進みます。

チーム間の連携

データベース設計者とアプリ開発者が同じ図を見て協力できます。

保守性の向上

データ構造の変更時に、両方の図を参照することで影響範囲を把握しやすくなります。

ドキュメント充実

システムの全体像を、データ層とアプリ層の両方から理解できます。

理解度確認クイズ

  1. Q1. UMLとは何の略で、どのような目的で使われますか？

    回答を表示

    Unified Modeling Language（統一モデリング言語）の略です。

    ソフトウェアシステムの構造や動作を視覚的に表現するための標準的な図式言語で、14種類の図を使い分けてシステムのさまざまな側面を表現します。

  2. Q2. ER図とクラス図の最大の違いは何ですか？

    回答を表示

    処理（振る舞い）を表現するかどうかが最大の違いです。

    ER図はデータの保存構造に焦点を当て、処理は記載しません。一方、クラス図はデータだけでなく、メソッド（処理）も含めて設計します。

  3. Q3. 以下のうち、ER図を使うべき場面として適切なものはどれですか？

    1. オブジェクト指向プログラムの設計をする場合
    2. データベースの保存構造を明確にしたい場合
    3. 継承やポリモーフィズムを表現したい場合
    4. APIの設計をドキュメント化する場合
    回答を表示

    2. データベースの保存構造を明確にしたい場合が正解です。

    ER図はデータベース設計が主な目的であり、データの保存構造や関係性を明確にするのに適しています。他の選択肢はクラス図が適している場面です。

第14章のまとめ

  • UMLは、ソフトウェアシステムを視覚的に表現する標準的な図式言語である
  • クラス図は、データの構造だけでなく、処理（メソッド）も表現するUMLの図である
  • ER図とクラス図は、データの集まりや関係性を表現する点で類似している
  • 最大の違いは、処理を表現するかどうか（ER図は記載しない、クラス図は記載する）
  • ER図はデータベース設計、クラス図はプログラム設計に適している
  • 実際のプロジェクトでは、両方を連携させて使うことでシステム全体の一貫性を保てる
