主キーと外部キー - データを識別し関連付ける

データベース設計の核心となる主キーと外部キーの概念を理解し、ER図での表記方法を習得しましょう。

この章で学ぶこと

  • 主キー（Primary Key）の役割と重要性を理解する
  • 適切な主キーの選び方と設計指針を学ぶ
  • 外部キー（Foreign Key）によるエンティティの関連付け方法を理解する
  • ER図での主キー・外部キーの表記方法をdraw.ioで実践する

7.1 主キーとは何か - データを一意に識別する

データベースには、同じような情報がたくさん保存されます。 例えば、顧客テーブルには何百人、何千人もの顧客情報が入ります。 そんな中で、「この人は誰なのか」を確実に特定する方法が必要です。

主キー（Primary Key）とは

エンティティの中で、各レコードを一意に識別するための属性です。 同じ値を持つレコードは存在せず、必ず値が入っている（NULL不可）という特徴があります。

なぜ主キーが必要なのか

例えば、「山田太郎」という名前の顧客が複数いたらどうでしょうか？ 名前だけでは、どの山田太郎さんの注文なのか分かりません。 そこで、顧客IDのような一意の識別子が必要になります。

主キーがない場合の問題例

顧客名 メールアドレス 電話番号
山田太郎 yamada@example.com 090-1234-5678
山田太郎 yamada2@example.com 080-9876-5432

同姓同名の人がいると、どちらの山田太郎さんか特定できません！

主キーがある場合の改善例

顧客ID 顧客名 メールアドレス 電話番号
C001 山田太郎 yamada@example.com 090-1234-5678
C002 山田太郎 yamada2@example.com 080-9876-5432

顧客IDがあれば、同姓同名でも確実に識別できます！

7.2 主キーの選び方と設計

主キーには、どの属性を選べばよいのでしょうか？ 主キーとして適切な属性には、いくつかの重要な条件があります。

主キーの必須条件

1. 一意性（Uniqueness）

同じ値を持つレコードが絶対に存在しないこと。 例：顧客IDは必ず異なる値

2. NOT NULL制約

主キーには必ず値が入っていること。 空欄（NULL）は許されません。

3. 不変性（Immutability）

一度決めたら変更しないこと。 メールアドレスは変わる可能性があるので不適切。

4. シンプルさ

できるだけ単純な値が望ましい。 通常は単一の属性（複合キーは避ける）。

主キーの種類

代理キー（Surrogate Key）

ビジネス的な意味を持たない、システムが自動生成する連番ID。 最も推奨される方法です。

例：

  • 顧客ID：C001, C002, C003...
  • 商品ID：P001, P002, P003...
  • 注文ID：1, 2, 3...（自動増分）

自然キー（Natural Key）

ビジネス的な意味を持つ既存の属性を主キーとする。 変更の可能性があるため、慎重に選ぶ必要があります。

例：

  • 社員番号（変更されないことが保証されている場合）
  • ISBN（書籍）
  • 製品コード（規格で定められている場合）

主キーとして避けるべき属性

  • メールアドレス：ユーザーが変更する可能性がある
  • 電話番号：機種変更などで変わる
  • 名前：同姓同名の可能性、結婚などで変わる
  • 住所：引っ越しで変わる

7.3 外部キーとは何か - エンティティをつなぐ

主キーで各エンティティのレコードを識別できるようになりました。 次は、エンティティ同士をどうやって関連付けるかを考えます。 ここで登場するのが外部キー（Foreign Key）です。

外部キー（Foreign Key）とは

別のエンティティの主キーを参照する属性です。 エンティティ同士を関連付け、「どのレコードとどのレコードがつながっているか」を示します。

外部キーの具体例

例えば、「注文」エンティティには「この注文はどの顧客が行ったのか」という情報が必要です。 これを表現するために、注文エンティティに顧客ID（外部キー）を持たせます。

顧客テーブル（親テーブル）

顧客ID（PK） 顧客名 メールアドレス
C001 山田太郎 yamada@example.com
C002 佐藤花子 sato@example.com

注文テーブル（子テーブル）

注文ID（PK） 顧客ID（FK） 注文日 合計金額
1 C001 2025-01-10 5,000円
2 C001 2025-01-15 8,000円
3 C002 2025-01-20 3,500円

注文テーブルの「顧客ID（FK）」は、顧客テーブルの「顧客ID（PK）」を参照しています。 これにより、「注文1と注文2は山田太郎さんの注文」とわかります。

外部キーの役割

1. 関連性の保持

どのレコードとどのレコードが関連しているかを明確にする。

2. 参照整合性の保証

存在しない顧客IDへの参照を防ぎ、データの整合性を保つ。

3. データの正規化

顧客情報の重複を避け、一箇所で管理できる。

4. 効率的な検索

「この顧客の全注文を取得」といった検索が簡単にできる。

7.4 主キーと外部キーの関係性

主キーと外部キーは、データベース設計において切っても切れない関係です。 この2つがどのように連携するのかを理解しましょう。

主キーと外部キーの関係を図で理解する

flowchart LR A["顧客テーブル
顧客ID（PK）: C001
顧客名: 山田太郎"] B["注文テーブル
注文ID（PK）: 1
顧客ID（FK）: C001
注文日: 2025-01-10"] C["注文テーブル
注文ID（PK）: 2
顧客ID（FK）: C001
注文日: 2025-01-15"] A -.参照.-> B A -.参照.-> C

注文テーブルの「顧客ID（FK）」は、顧客テーブルの「顧客ID（PK）」を参照します。

参照整合性制約

外部キーには参照整合性制約というルールがあります。 これは、「外部キーの値は、必ず参照先の主キーに存在する値でなければならない」というものです。

参照整合性が守られる例

注文テーブルに「顧客ID: C001」を登録しようとする場合、 顧客テーブルに「C001」が存在すれば登録できます。

参照整合性違反の例

顧客テーブルに「C999」が存在しないのに、 注文テーブルに「顧客ID: C999」を登録しようとすると、 データベースがエラーを返して登録を拒否します。

カスケード操作

外部キーには、関連するレコードを自動的に処理するカスケード操作を設定できます。

ON DELETE CASCADE

親テーブルのレコードを削除すると、子テーブルの関連レコードも自動的に削除される。 例：顧客を削除すると、その顧客の注文も自動削除される。

ON UPDATE CASCADE

親テーブルの主キーが更新されると、子テーブルの外部キーも自動的に更新される。 例：顧客IDをC001→C100に変更すると、注文の顧客IDも自動的にC100に変わる。

7.5 ER図での主キー・外部キーの表記方法

ER図では、主キーと外部キーを視覚的に区別できるように表記します。 一般的な表記方法をいくつか見てみましょう。

IE記法（鳥の足記法）での表記

erDiagram CUSTOMER ||--o{ ORDER : places CUSTOMER { string customer_id PK "顧客ID（主キー）" string name "顧客名" string email "メールアドレス" string phone "電話番号" } ORDER { int order_id PK "注文ID（主キー）" string customer_id FK "顧客ID（外部キー）" date order_date "注文日" decimal total_amount "合計金額" }

PKは主キー（Primary Key）、FKは外部キー（Foreign Key）を表します。

主キー・外部キーの表記ルール

主キー（PK）の表記

  • 属性名の後ろに「PK」と記載
  • または、属性名の前に鍵マーク（🔑）を付ける
  • エンティティの最上部に配置することが多い
  • 太字や下線で強調することもある

外部キー（FK）の表記

  • 属性名の後ろに「FK」と記載
  • 参照先のエンティティ名を添えることもある（例：FK → CUSTOMER）
  • リレーションシップの線で参照元・参照先を明確にする
  • 斜体やカラーで区別することもある

7.6 draw.ioでの実践 - 主キー・外部キーを描いてみよう

それでは、draw.ioを使って、主キーと外部キーを含むER図を実際に描いてみましょう。

実習 7-1: 図書館システムのER図を描く

図書館システムを題材に、主キーと外部キーを含むER図をdraw.ioで作成します。

要件

  • 会員（MEMBER）エンティティ：会員ID（PK）、会員名、入会日
  • 本（BOOK）エンティティ：本ID（PK）、書名、著者、ISBN
  • 貸出（LOAN）エンティティ：貸出ID（PK）、会員ID（FK）、本ID（FK）、貸出日、返却予定日
  • 一人の会員は複数の本を借りることができる
  • 一冊の本は複数回貸出履歴を持つことができる

手順

  1. 1 draw.ioを開き、新しい図を作成する
  2. 2 左サイドバーから「Entity Relation」を選択
  3. 3 3つのエンティティ（MEMBER、BOOK、LOAN）を配置する
  4. 4 各エンティティに属性を追加し、主キー（PK）と外部キー（FK）を明記する
  5. 5 エンティティ間のリレーションシップを線で結ぶ（MEMBER → LOAN、BOOK → LOAN）
  6. 6 カーディナリティ（1対多）を表現する

ヒント

  • 主キーは必ず先頭に配置し、「PK」または鍵マークで明示する
  • 外部キーは「FK」と記載し、どのエンティティを参照しているか明確にする
  • 外部キーの属性名は、参照先の主キーと同じ名前にするとわかりやすい
  • リレーションシップの線を外部キーから参照先の主キーに向けて引く

完成例の確認ポイント

  • 各エンティティに主キー（PK）が明記されている
  • LOANエンティティに会員ID（FK）と本ID（FK）が含まれている
  • MEMBER → LOAN、BOOK → LOANのリレーションシップが表現されている
  • 1対多のカーディナリティが正しく設定されている

理解度確認クイズ

  1. Q1. 主キー（Primary Key）の必須条件を3つ挙げてください。

    回答を表示
    1. 一意性（Uniqueness）：同じ値を持つレコードが存在しない
    2. NOT NULL制約：主キーには必ず値が入っている（空欄不可）
    3. 不変性（Immutability）：一度決めたら変更しない

  2. Q2. 外部キー（Foreign Key）の役割を説明してください。

    回答を表示

    外部キーは、別のエンティティの主キーを参照する属性です。

    主な役割：

    • エンティティ同士を関連付ける
    • 参照整合性を保証する（存在しないレコードへの参照を防ぐ）
    • データの重複を防ぎ、正規化を実現する

  3. Q3. 以下のうち、主キーとして最も適切でないものはどれですか？

    1. 自動増分の顧客ID（1, 2, 3...）
    2. メールアドレス
    3. ISBN（書籍の国際標準図書番号）
    4. システムが生成するUUID
    回答を表示

    2. メールアドレスが最も不適切です。

    理由：

    • ユーザーが変更する可能性がある（不変性に欠ける）
    • 機種変更やプロバイダ変更でアドレスが変わることがある
    • 主キーは一度決めたら変更しないことが原則

    他の選択肢は、不変で一意な値を保証できるため主キーとして適しています。

第7章のまとめ

  • 主キー（PK）は、各レコードを一意に識別する属性で、一意性・NOT NULL・不変性が必須条件
  • 主キーは代理キー（自動生成ID）を使うのが一般的で最も推奨される
  • 外部キー（FK）は、別のエンティティの主キーを参照してエンティティ同士を関連付ける
  • 外部キーには参照整合性制約があり、存在しないレコードへの参照を防ぐ
  • ER図では、主キーは「PK」、外部キーは「FK」と明記して区別する
  • draw.ioで主キー・外部キーを含むER図を作成し、実践的なスキルを身につけた
