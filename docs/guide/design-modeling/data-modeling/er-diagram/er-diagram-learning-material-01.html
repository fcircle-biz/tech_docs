第1章

ER図とは何か - データベース設計の第一歩

データベース設計に欠かせないER図の基本概念を、身近な例から理解しましょう。

この章で学ぶこと

  • ER図とは何か、その役割と目的を理解する
  • データベースの基本概念とExcelとの違いを知る
  • なぜ図を描いてデータを設計する必要があるのかを理解する
  • ER図の実際の活用シーンを具体例で学ぶ

1.1 身近な例から始めよう - Excelでの顧客管理

まずは、多くの人が使ったことのあるExcelを例に考えてみましょう。 例えば、あなたがオンラインショップを運営していて、顧客情報をExcelで管理しているとします。

顧客管理表（Excel）の例

顧客ID 顧客名 メールアドレス 電話番号 住所
C001 山田太郎 yamada@example.com 090-1234-5678 東京都渋谷区...
C002 佐藤花子 sato@example.com 080-9876-5432 大阪府大阪市...

この表は分かりやすくて便利ですが、顧客が商品を注文したときの情報も記録したいとしたらどうでしょうか？ 注文日、注文商品、数量、金額...といった情報も追加する必要があります。

Excelだけでは管理が難しくなる理由

  • 同じ顧客が複数回注文すると、顧客情報を何度も入力する必要がある（データの重複）
  • 顧客の住所が変わったとき、すべての行を修正しなければならない（更新の手間）
  • 表が横に長くなりすぎて見づらくなる（可読性の低下）

1.2 データベースとは何か

こうした問題を解決するために登場したのが「データベース」です。 データベースは、データを効率的に保存・検索・更新するための仕組みです。

データベースの特徴

  • データの分割管理：情報を複数の表（テーブル）に分けて管理
  • 関連性の定義：テーブル同士の関係を明確にする
  • 重複の排除：同じ情報を何度も入力する必要がない
  • 一貫性の保持：データの矛盾を防ぐ仕組みがある

例えば、先ほどの顧客管理をデータベースで行う場合、以下のように複数のテーブルに分けて管理します。

flowchart LR A[顧客テーブル
顧客ID、顧客名、メール] --> B{関連} B --> C[注文テーブル
注文日、顧客ID] C --> D{関連} D --> E[商品テーブル
商品ID、商品名、価格]

このように、データを適切に分割して関連付けることで、効率的で矛盾のないデータ管理が可能になります。

1.3 ER図が必要な理由 - なぜ図を描くのか

データベースを設計する際、「どんなテーブルが必要か」「テーブル同士をどう関連付けるか」を考える必要があります。 しかし、頭の中だけで考えたり、文章で説明したりするのは難しいですよね。

ER図（Entity-Relationship Diagram）とは

データベースの構造を視覚的に表現する図です。 どんなデータ（エンティティ）があり、それらがどう関連しているか（リレーションシップ）を一目で理解できるようにします。

ER図を描くメリット

1. 視覚的に理解しやすい

複雑なデータ構造も図で表現することで、一目で全体像を把握できます。

2. チーム間のコミュニケーション

技術者だけでなく、業務担当者とも同じ図を見ながら議論できます。

3. 設計ミスの早期発見

図を描くことで、データの重複や関連の矛盾に気づきやすくなります。

4. ドキュメントとして活用

システムの仕様書として、後から見直すときにも役立ちます。

1.4 ER図の活用シーン - 具体例で理解する

ER図は、さまざまなシステムやサービスの設計で活用されています。 身近な例で、どのようにER図が使われているか見てみましょう。

例1: オンラインショッピングサイト

erDiagram CUSTOMER ||--o{ ORDER : places ORDER ||--|{ ORDER_LINE : contains PRODUCT ||--o{ ORDER_LINE : referenced_by CUSTOMER { string customer_id PK string name string email } ORDER { string order_id PK date order_date string customer_id FK } ORDER_LINE { string order_line_id PK string order_id FK string product_id FK int quantity } PRODUCT { string product_id PK string product_name decimal price }

顧客（CUSTOMER）が注文（ORDER）を行い、注文には複数の商品（PRODUCT）が含まれる関係性を図で表現しています。

例2: 図書館システム

erDiagram MEMBER ||--o{ LOAN : borrows BOOK ||--o{ LOAN : referenced_by MEMBER { string member_id PK string name date join_date } BOOK { string book_id PK string title string author } LOAN { string loan_id PK string member_id FK string book_id FK date loan_date date return_date }

会員（MEMBER）が本（BOOK）を借りる（LOAN）という関係性を明確にしています。

1.5 データの関係性を可視化する

ER図の最大の特徴は、「データ同士の関係性」を視覚化できることです。 例えば、「一人の顧客は複数の注文ができる」「一つの注文は複数の商品を含む」といった関係性を、線やシンボルで表現します。

ER図の主要な構成要素

1

エンティティ（Entity）

管理したい「もの」や「こと」。例：顧客、商品、注文

2

属性（Attribute）

エンティティが持つ詳細情報。例：顧客名、メールアドレス、電話番号

3

リレーションシップ（Relationship）

エンティティ同士の関連性。例：「顧客が注文する」「注文に商品が含まれる」

これらの要素を組み合わせることで、複雑なデータ構造もわかりやすく表現できます。 次の章からは、実際にER図を描く練習を始めていきましょう。

理解度確認クイズ

  1. Q1. ER図の「ER」は何の略ですか？また、その意味を説明してください。

    回答を表示

    Entity-Relationship（実体関連）の略です。

    「Entity（エンティティ）」は管理したいデータの対象（顧客、商品など）を表し、「Relationship（リレーションシップ）」はそれらの関連性（顧客が注文する、など）を表します。

  2. Q2. Excelでのデータ管理と比べて、データベースを使う主なメリットを3つ挙げてください。

    回答を表示
    1. データの重複を排除：同じ情報を何度も入力する必要がない
    2. 更新の一貫性：一箇所を修正すれば、関連するすべてのデータに反映される
    3. 複雑な関連性の管理：複数のテーブル間の関係を明確に定義・管理できる

  3. Q3. ER図を描くメリットとして適切でないものはどれですか？

    1. データ構造を視覚的に理解しやすい
    2. チーム間でのコミュニケーションが円滑になる
    3. 設計ミスを早期に発見できる
    4. プログラミング言語を学ぶ必要がなくなる
    回答を表示

    4. プログラミング言語を学ぶ必要がなくなるが不適切です。

    ER図はデータベース設計のためのツールであり、プログラミング言語の学習とは直接関係ありません。実際のシステム開発では、ER図で設計したデータベースを操作するためにSQLなどの言語を学ぶ必要があります。

第1章のまとめ

  • ER図は、データベースの構造を視覚的に表現する図である
  • Excelでの管理には限界があり、データベースを使うことでデータの重複や矛盾を防げる
  • ER図を描くことで、設計の全体像を把握しやすく、チーム間のコミュニケーションも円滑になる
  • ER図の主要な要素は、エンティティ、属性、リレーションシップの3つ
  • オンラインショップや図書館など、身近なシステムもER図で設計されている
