7.1 システム構成

クライアントサーバシステム

クライアントサーバシステムは、サービスを提供するサーバと、サービスを利用するクライアントで構成されます。

2層クライアントサーバ
  • 構成：クライアント、データベースサーバ
  • 特徴：シンプルだが、クライアント側の負荷が大きい
3層クライアントサーバ
  • 構成：クライアント（プレゼンテーション層）、アプリケーションサーバ（ビジネスロジック層）、データベースサーバ（データ層）
  • 特徴：役割分担が明確、クライアントの負荷が軽い

RAID（Redundant Array of Independent Disks）

RAIDは、複数のハードディスクを組み合わせて信頼性や性能を向上させる技術です。

RAID レベル 構成 特徴
RAID 0 ストライピング 性能向上、冗長性なし
RAID 1 ミラーリング 同じデータを2つのディスクに記録、高い信頼性
RAID 5 分散パリティ 性能と信頼性のバランス、1台故障しても復旧可能

システムの性能指標

スループット

単位時間あたりに処理できる仕事量

レスポンスタイム（応答時間）

処理要求から結果が返るまでの時間

ターンアラウンドタイム

処理要求から処理完了までの総時間

システムの信頼性指標

  • MTBF（平均故障間隔）：故障から次の故障までの平均時間
  • MTTR（平均修理時間）：故障発生から修理完了までの平均時間
  • 稼働率 = MTBF ÷ (MTBF + MTTR)
直列接続と並列接続の稼働率

直列接続：稼働率 = A × B（両方稼働していないと動かない）

並列接続：稼働率 = 1 - (1-A) × (1-B)（どちらかが稼働していれば動く）

7.1.1 クラウドコンピューティング

クラウドコンピューティングとは、インターネット経由でコンピューティングリソース（サーバ、ストレージ、ソフトウェアなど）を利用するサービス形態です。

クラウドコンピューティングの主な特徴
  • オンデマンド：必要な時に必要なだけリソースを利用できる
  • スケーラビリティ：需要に応じて柔軟にリソースを拡張・縮小できる
  • 従量課金：使用した分だけ料金を支払う（Pay-as-you-go）
  • インターネット経由のアクセス：場所を問わずサービスを利用できる
  • リソース共有：複数のユーザーが物理リソースを共有
クラウドサービスモデル

サービスモデル 提供されるもの ユーザーが管理するもの 主な例
IaaS
（Infrastructure as a Service）		 サーバ、ストレージ、ネットワークなどの基盤 OS、ミドルウェア、アプリケーション AWS EC2、Azure VM、Google Compute Engine
PaaS
（Platform as a Service）		 OS、開発環境、実行環境 アプリケーション、データ AWS Elastic Beanstalk、Google App Engine、Heroku
SaaS
（Software as a Service）		 完成されたアプリケーション データの入力・設定のみ Microsoft 365、Salesforce、Gmail、Dropbox
サービスモデルの詳細
IaaS（Infrastructure as a Service）
  • 特徴：仮想サーバやストレージなどのインフラを提供
  • メリット：高い柔軟性、自由なカスタマイズが可能
  • 利用シーン：独自のシステム構築、開発・テスト環境
  • 主要サービス：
    • AWS EC2（Amazon Elastic Compute Cloud）
    • Azure Virtual Machines
    • Google Compute Engine
PaaS（Platform as a Service）
  • 特徴：アプリケーション開発・実行環境を提供
  • メリット：インフラ管理不要、開発に集中できる
  • 利用シーン：Webアプリケーション開発、APIサービス
  • 主要サービス：
    • AWS Elastic Beanstalk
    • Google App Engine
    • Microsoft Azure App Service
SaaS（Software as a Service）
  • 特徴：完成されたソフトウェアをサービスとして提供
  • メリット：導入が簡単、メンテナンス不要
  • 利用シーン：メール、CRM、グループウェア
  • 主要サービス：
    • Microsoft 365（旧Office 365）
    • Salesforce（CRM）
    • Google Workspace
    • Slack、Zoom
クラウドコンピューティングの利点
利点 説明
コスト削減 初期投資不要、従量課金で無駄なコストを削減
柔軟性・拡張性 需要に応じてリソースを迅速に増減できる
災害対策（BCP） データの自動バックアップ、地理的分散
迅速な展開 新しいサービスを短時間で開始できる
自動アップデート 常に最新のソフトウェアを利用できる
場所を問わないアクセス インターネット環境があればどこからでも利用可能
クラウドコンピューティングの課題
課題 説明
セキュリティリスク データの外部保管、不正アクセスのリスク
ベンダーロックイン 特定のクラウド事業者に依存し、移行が困難
インターネット依存 ネットワーク障害時にサービスが利用できない
コスト管理 従量課金のため、使いすぎによる予想外のコスト
コンプライアンス データの保管場所や法規制への対応

7.1.2 仮想化技術

仮想化技術とは、物理的なハードウェアリソースを論理的に分割し、複数の仮想的な環境を作り出す技術です。

サーバ仮想化の概念

1台の物理サーバ上で、複数の仮想サーバ（仮想マシン：VM）を稼働させる技術です。

ハイパーバイザー

ハイパーバイザーは、仮想マシンを管理・制御するソフトウェアです。

種類 特徴 例
Type 1
（ネイティブ型）		 ハードウェア上で直接動作
高性能、サーバ用途に適する		 VMware ESXi、Microsoft Hyper-V、Xen
Type 2
（ホスト型）		 OS上で動作
導入が簡単、デスクトップ用途に適する		 VMware Workstation、Oracle VirtualBox、Parallels Desktop
仮想マシン（VM）の特徴
  • 独立性：各VMは独立したOSとアプリケーションを持つ
  • 分離性：1つのVMの障害が他のVMに影響しない
  • 可搬性：VM全体をファイルとして移動・複製できる
  • スナップショット：特定時点の状態を保存・復元できる
仮想化のメリット
メリット 説明
リソース効率化 1台の物理サーバで複数のシステムを稼働、ハードウェア投資削減
柔軟性 必要に応じてVMを追加・削除、リソース配分を変更可能
BCP対策 VMのバックアップ・復元が容易、災害対策に有効
開発・テスト環境 様々な環境を簡単に構築・破棄できる
運用コスト削減 電力消費、冷却コスト、設置スペースの削減

7.1.3 コンテナ技術

コンテナ技術とは、アプリケーションとその実行に必要な環境を1つのパッケージにまとめ、軽量で高速に起動できる仮想化技術です。

コンテナとは

OSカーネルを共有しながら、アプリケーションとその依存関係を隔離された環境で実行する技術です。

Dockerの概要

Dockerは、最も普及しているコンテナプラットフォームです。

  • Dockerイメージ：アプリケーションと環境を含むテンプレート
  • Dockerコンテナ：イメージから起動された実行環境
  • Docker Hub：コンテナイメージの公開リポジトリ
  • Dockerfile：イメージを作成するための設定ファイル
コンテナと仮想マシンの比較
比較項目 コンテナ 仮想マシン（VM）
仮想化レベル アプリケーションレベル ハードウェアレベル
OSカーネル ホストOSと共有 各VMが独自のOS
起動時間 数秒 数分
リソース使用量 軽量（MB単位） 重量（GB単位）
ポータビリティ 非常に高い 高い
分離性 プロセスレベル 完全分離
用途 マイクロサービス、CI/CD 異なるOS、完全分離が必要な場合
コンテナのメリット
メリット 説明
軽量 OSカーネルを共有するため、リソース消費が少ない
高速起動 数秒で起動・停止が可能
一貫性 開発・テスト・本番環境で同じ環境を再現できる
移植性 どの環境でも同じように動作する
スケーラビリティ 必要に応じて簡単にコンテナを増減できる
CI/CD適性 継続的インテグレーション・デリバリーに最適
仮想化とコンテナの使い分け
  • 仮想マシンが適している場合：
    • 異なるOSを動かす必要がある
    • 完全な分離が必要（セキュリティ要件が高い）
    • 長時間稼働するサーバ
  • コンテナが適している場合：
    • マイクロサービスアーキテクチャ
    • 頻繁にスケールする必要がある
    • 開発・テスト環境の統一
    • CI/CDパイプライン

7.2 ソフトウェアとOS

OS（オペレーティングシステム）

OSは、コンピュータのハードウェアとソフトウェアを管理する基本ソフトウェアです。

OSの主な機能
  • プロセス管理：CPUの割り当て、プログラムの実行管理
  • メモリ管理：主記憶装置の割り当てと管理
  • ファイル管理：ファイルシステムの管理
  • 入出力管理：周辺機器の制御

ファイルシステム

ファイルを管理するための仕組みです。

ディレクトリ構造
  • ルートディレクトリ：最上位のディレクトリ
  • カレントディレクトリ：現在作業中のディレクトリ
  • 絶対パス：ルートディレクトリからの完全なパス
  • 相対パス：カレントディレクトリからのパス

ソフトウェアの種類

種類 説明
オープンソースソフトウェア（OSS） ソースコードが公開され、自由に利用・改変できる
フリーウェア 無料で利用できるソフトウェア
シェアウェア 試用後に購入するソフトウェア
商用ソフトウェア 購入して利用するソフトウェア

代表的なOSS

  • Linux：Unix系OS
  • Apache：Webサーバ
  • MySQL、PostgreSQL：データベース管理システム
  • Firefox：Webブラウザ

7.3 記憶装置

記憶装置の階層

コンピュータは、速度と容量のバランスを取るため、複数の記憶装置を階層的に使用します。

記憶装置 特徴 速度 容量
レジスタ CPU内部の超高速メモリ 最速 最小
キャッシュメモリ CPUと主記憶の間の高速メモリ 高速 小
主記憶装置（RAM） プログラムとデータを一時保存 中速 中
補助記憶装置（HDD、SSD） データを永続的に保存 低速 大

主記憶装置

RAM（Random Access Memory）
  • DRAM：Dynamic RAM、安価、定期的なリフレッシュが必要
  • SRAM：Static RAM、高速、高価、キャッシュメモリに使用
ROM（Read Only Memory）
  • マスクROM：製造時に書き込み、書き換え不可
  • PROM：一度だけ書き込み可能
  • EPROM：紫外線で消去可能
  • フラッシュメモリ：電気的に消去・書き込み可能

補助記憶装置

HDD（ハードディスクドライブ）
  • 特徴：大容量、低価格、機械的動作で衝撃に弱い
  • 構成：プラッタ、磁気ヘッド、アーム
SSD（ソリッドステートドライブ）
  • 特徴：高速、静音、耐衝撃性、高価
  • 技術：フラッシュメモリを使用
光ディスク
  • CD：650～700MB
  • DVD：4.7～17GB
  • Blu-ray：25～128GB
