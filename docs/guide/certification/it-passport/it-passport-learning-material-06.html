システム開発

システム開発の基本、システム設計、プログラミング、アルゴリズムとデータ構造、システムのテスト、開発モデルについて学習します。

この章で学ぶこと

  • システム開発のライフサイクル（SLCP）を理解する
  • 外部設計と内部設計の違いを学ぶ
  • プログラミングの基本（コーディング、デバッグ）を習得する
  • 基本的なデータ構造（スタック、キュー、木構造）を理解する
  • 探索アルゴリズムと整列アルゴリズムの違いを学ぶ
  • 各種テスト手法（単体・統合・システム・受入れテスト）を理解する
  • 代表的な開発モデル（ウォーターフォール、アジャイル、DevOps）を学ぶ

6.1 システム開発のライフサイクル（SLCP）

SLCP（System Life Cycle Process）は、システムの企画から開発、運用、保守、廃棄までの一連のプロセスを定義したものです。システム開発は段階的に進められ、各段階で成果物を作成します。

graph TD Start["企画"] --> A["要件定義"] A --> B["システム設計"] B --> C["プログラミング"] C --> D["テスト"] D --> E["移行"] E --> F["運用・保守"] F --> End["廃棄"]

SLCPの主な段階

  • 企画: システム化の目的を明確にする
  • 要件定義: 必要な機能や性能を定義する
  • システム設計: システムの構造を設計する
  • プログラミング: プログラムコードを作成する
  • テスト: システムが正しく動作するか検証する
  • 移行: 本番環境へシステムを導入する
  • 運用・保守: システムの稼働状況を監視し、修正や改善を行う
  • 廃棄: システムの役割終了後、適切に廃棄する

重要ポイント

要件定義が不十分だと、後の工程で大きな手戻りが発生します。初期段階で利用者のニーズを正確に把握することが重要です。

6.2 システム設計

システム設計は、外部設計と内部設計の2段階に分かれます。外部設計は「何を作るか」、内部設計は「どう作るか」を定義します。

外部設計

外部設計は、利用者から見たシステムの機能や画面、帳票などを設計する工程です。主に以下の設計を行います。

  • 画面設計: 入力画面や表示画面の構成を設計
  • 帳票設計: 印刷される帳票のレイアウトを設計
  • インタフェース設計: 外部システムとのデータのやり取りを設計
  • データベース概念設計: 必要なデータ項目を洗い出す

内部設計

内部設計は、システムの内部構造や処理の詳細を設計する工程です。主に以下の設計を行います。

  • プログラム構造設計: プログラムの構成とモジュール分割
  • データベース論理設計・物理設計: テーブル構造とインデックスの設計
  • 処理フロー設計: 処理の流れを詳細に設計
  • アルゴリズム設計: データ処理の手順を設計
項目 外部設計 内部設計
視点 利用者視点 開発者視点
定義内容 何を作るか どう作るか
成果物例 画面設計書、帳票設計書 プログラム仕様書、データ設計書

6.3 プログラミング

プログラミング工程では、設計書に基づいてプログラムコードを作成します。主な活動としてコーディングとデバッグがあります。

コーディング

コーディングは、設計書をもとにプログラミング言語でソースコードを記述する作業です。コーディング規約に従い、読みやすく保守しやすいコードを書くことが重要です。

コーディング規約の例

  • 変数名や関数名は意味のある名前をつける
  • インデント（字下げ）を適切に行う
  • コメントを適切に記述する
  • 1つの関数は1つの機能に限定する

デバッグ

デバッグは、プログラムの誤り（バグ）を発見し、修正する作業です。デバッガツールを使ってプログラムの実行状況を確認しながら、問題箇所を特定します。

6.4 アルゴリズムとデータ構造

アルゴリズムは問題を解決するための手順、データ構造はデータを効率よく扱うための格納方法です。

基本的なデータ構造

スタック（Stack）

スタックは、データの追加と取り出しを同じ側（先端）で行うデータ構造です。LIFO（Last In First Out）方式で、最後に入れたデータを最初に取り出します。

graph TD subgraph "スタックの動作" A["データ3（最新）"] B["データ2"] C["データ1（最古）"] end A --> B B --> C D["↓ Push（追加）
↑ Pop（取り出し）"]

キュー（Queue）

キューは、データの追加と取り出しを別の側で行うデータ構造です。FIFO（First In First Out）方式で、最初に入れたデータを最初に取り出します。

graph LR A["Enqueue（追加）"] --> B["データ1"] B --> C["データ2"] C --> D["データ3"] D --> E["Dequeue（取り出し）"]

木構造（Tree）

木構造は、階層的にデータを表現するデータ構造です。最上位のノードを根（ルート）、末端のノードを葉（リーフ）と呼びます。

graph TD Root["根（ルート）"] --> A["ノードA"] Root --> B["ノードB"] A --> C["葉C"] A --> D["葉D"] B --> E["葉E"]

代表的なアルゴリズム

探索アルゴリズム

データの中から目的の値を見つけ出すアルゴリズムです。

手法 説明 前提条件
線形探索 先頭から順番に調べる なし
2分探索 中央値と比較して範囲を半分ずつ絞り込む データが整列済み

整列アルゴリズム（ソート）

データを昇順や降順に並べ替えるアルゴリズムです。

手法 説明
バブルソート 隣接する要素を比較して交換を繰り返す
選択ソート 最小値を見つけて先頭に配置を繰り返す
クイックソート 基準値を決めて分割統治法で整列（高速）

6.5 システムのテスト

テストは、システムが要件を満たし、正しく動作することを確認する重要な工程です。テストは段階的に行われます。

テストの段階

テスト種類 目的 実施者
単体テスト 個々のプログラム単位（モジュール）の動作確認 開発者
統合テスト 複数のモジュールを結合した状態での動作確認 開発者
システムテスト システム全体としての動作確認 テスト担当者
受入れテスト 利用者の要件を満たしているか確認 利用者

テスト手法

手法 内容 観点
ホワイトボックステスト プログラムの内部構造を考慮したテスト コードの論理的な経路を網羅
ブラックボックステスト プログラムの外部仕様のみに着目したテスト 入力と出力の関係を確認

重要ポイント

単体テストや統合テストはホワイトボックステスト、システムテストや受入れテストはブラックボックステストで実施されることが一般的です。

6.6 開発モデル

システム開発にはいくつかの開発モデルがあり、プロジェクトの特性に応じて適切なモデルを選択します。

ウォーターフォールモデル

ウォーターフォールモデルは、各工程を順番に進める伝統的な開発手法です。前の工程が完了してから次の工程に進むため、計画的に開発を進められます。

graph TD A["要件定義"] --> B["設計"] B --> C["実装"] C --> D["テスト"] D --> E["運用"]
メリット デメリット
• 各工程の成果物が明確
• 進捗管理がしやすい
• 大規模開発に適している 		• 要件変更に対応しにくい
• テスト段階でバグが発見されると手戻りが大きい
• 利用者の反応が分かるのが遅い

アジャイル開発

アジャイル開発は、短い期間（イテレーション）で計画・設計・実装・テストを繰り返し、段階的にシステムを完成させる手法です。変化に柔軟に対応できます。

（1-4週間）"]
メリット デメリット
• 要件変更に柔軟に対応
• 早期に動作するシステムを確認できる
• 利用者の意見を取り入れやすい 		• 全体計画が見えにくい
• ドキュメント整備が不十分になりがち
• チーム全体の協力が必須

DevOps

DevOpsは、開発（Development）と運用（Operations）を統合し、自動化やCI/CD（継続的インテグレーション/継続的デリバリー）を活用して、迅速かつ頻繁にリリースを行う文化・手法です。

DevOpsの主な実践

  • CI/CD: コードの変更を自動的にビルド・テスト・デプロイ
  • インフラのコード化（IaC）: インフラ構成をコードで管理
  • モニタリング・ログ管理: システムの状態を常時監視
  • 自動化: テストやデプロイを自動化して効率化

理解度確認クイズ

  1. Q1. システム開発のライフサイクルで、最も早い段階に実施するのはどれか。

    回答を表示

    解答: 企画
    解説: SLCPの順序は、企画→要件定義→システム設計→プログラミング→テスト→移行→運用・保守→廃棄です。

  2. Q2. 利用者から見たシステムの機能や画面を設計する工程はどれか。

    回答を表示

    解答: 外部設計
    解説: 外部設計は利用者視点で「何を作るか」を設計します。内部設計は開発者視点で「どう作るか」を設計します。

  3. Q3. 最後に入れたデータを最初に取り出すデータ構造はどれか。

    回答を表示

    解答: スタック（Stack）
    解説: スタックはLIFO方式で、最後に入れたデータを最初に取り出します。キューはFIFO方式で、最初に入れたデータを最初に取り出します。

  4. Q4. プログラムの内部構造を考慮したテスト手法はどれか。

    回答を表示

    解答: ホワイトボックステスト
    解説: ホワイトボックステストは内部構造を考慮します。ブラックボックステストは外部仕様のみに着目します。

  5. Q5. 短い期間で開発を繰り返し、変化に柔軟に対応できる開発モデルはどれか。

    回答を表示

    解答: アジャイル開発
    解説: アジャイル開発は短いイテレーションで開発を繰り返し、要件変更に柔軟に対応できます。ウォーターフォールモデルは各工程を順番に進めます。
