実践プロジェクト - シンプルなWebアプリをコンテナ化しよう

これまで学んだ知識を総動員して、シンプルなTodoアプリをコンテナ化します。フロントエンド、バックエンド、データベースの3層構成を実際に構築し、開発環境としてのDockerの便利さを体感します。

この章で学ぶこと

  • 3層アーキテクチャ（フロントエンド、バックエンド、DB）を理解する
  • シンプルなWebアプリのDocker化を体験する
  • 複数のDockerfileとdocker-compose.ymlを組み合わせる
  • 開発環境としてのDockerの便利さを体感する

9.1 実践プロジェクトの概要

この章では、シンプルなTodoアプリを題材に、これまで学んだDockerの知識を統合して実践的なプロジェクトを構築します。

構築するTodoアプリの機能

Todoの追加

テキストを入力して新しいTodoを追加できます

Todoの一覧表示

登録されているTodoをリスト形式で表示します

Todoの削除

不要になったTodoを削除できます

機能はシンプルですが、フロントエンド・バックエンド・データベースの3層構成を学ぶには十分です。

Todoアプリは3つの層で構成されます。各層を独立したコンテナにすることで、保守性と拡張性が高まります。

9.3 バックエンドのDocker化（Flask API）

まず、PythonのFlaskフレームワークを使ったシンプルなAPIサーバーを作成します。

backend/app.py 

# Flask APIサーバー（シンプル版）
from flask import Flask, jsonify, request
import mysql.connector

app = Flask(__name__)

# データベース接続設定
db_config = {
    'host': 'db',  # docker-composeのサービス名
    'user': 'root',
    'password': 'rootpass',
    'database': 'todoapp'
}

@app.route('/todos', methods=['GET'])
def get_todos():
    # Todo一覧を取得
    conn = mysql.connector.connect(**db_config)
    cursor = conn.cursor()
    cursor.execute("SELECT * FROM todos")
    todos = cursor.fetchall()
    return jsonify(todos)

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000)

backend/Dockerfile 

# Python公式イメージを使用
FROM python:3.11-slim

# 作業ディレクトリを設定
WORKDIR /app

# 依存関係ファイルをコピー
COPY requirements.txt .

# 依存関係をインストール
RUN pip install --no-cache-dir -r requirements.txt

# アプリケーションコードをコピー
COPY app.py .

# ポート5000で起動
EXPOSE 5000
CMD ["python", "app.py"]

Pythonの公式イメージをベースに、FlaskとMySQLドライバをインストールします

シンプルなHTMLフォームを作成し、Nginxで配信します。

<script>

    .then(res => res.json())

frontend/Dockerfile 

# Nginx公式イメージを使用

# HTMLファイルをコピー

9.5 データベースの設定（MySQL + ボリューム）

MySQL公式イメージを使い、データを永続化するためにボリュームを設定します。

データベースの初期設定 

-- Todoテーブルの作成
CREATE TABLE todos (
  id INT AUTO_INCREMENT PRIMARY KEY,
  title VARCHAR(255) NOT NULL,
  completed BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

ボリュームを使う理由

データベースコンテナを削除すると、保存されているTodoデータも消えてしまいます。 ボリュームを使うことで、コンテナを削除してもデータが残ります。 これにより、安心してコンテナの更新や再起動ができます。

9.6 docker-compose.ymlで全体を統合

3つのコンテナを1つのコマンドで起動できるように、docker-compose.ymlを作成します。

docker-compose.yml 

version: '3.8'

services:
  # フロントエンド（Nginx）
  frontend:
    build: ./frontend
    ports:
      - "8080:80"  # ホストの8080番をコンテナの80番に
    depends_on:
      - backend

  # バックエンド（Flask API）
  backend:
    build: ./backend
    ports:
      - "5000:5000"
    depends_on:
      - db

  # データベース（MySQL）
  db:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: rootpass
      MYSQL_DATABASE: todoapp
    volumes:
      - todo_data:/var/lib/mysql  # データ永続化

volumes:
  todo_data:  # 名前付きボリューム定義

depends_onで起動順序を制御し、volumesでデータを永続化します

理解度確認クイズ

  1. Q1. 3層アーキテクチャにおいて、各層（フロントエンド、バックエンド、データベース）を独立したコンテナにするメリットは何ですか？

    回答を表示

    各層を独立したコンテナにすることで、保守性・拡張性・スケーラビリティが向上します。例えば、フロントエンドだけを更新したい場合、バックエンドやデータベースに影響を与えずに変更できます。また、負荷に応じてバックエンドコンテナだけを複数起動することも可能です。

  2. Q2. データベースコンテナにボリュームを設定する理由は何ですか？

    回答を表示

    コンテナを削除すると、その中のデータも消えてしまいます。ボリュームを使うことで、コンテナとは独立してデータを保存でき、コンテナを削除・再作成してもデータが保持されます。これにより、安心してコンテナのアップデートや再起動ができます。

  3. Q3. docker-compose.ymlでdepends_onを設定する理由は何ですか？

    回答を表示

    コンテナの起動順序を制御するためです。例えば、バックエンドはデータベースが起動していないと接続できないため、データベースを先に起動する必要があります。depends_onを設定することで、依存関係を明示し、適切な順序でコンテナが起動されます。
