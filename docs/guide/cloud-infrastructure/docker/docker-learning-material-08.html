第8章

Docker Composeで複数のコンテナを管理しよう

複数のコンテナを連携させる必要性を理解し、Docker Composeの基本を学びます。docker-compose.ymlファイルの書き方を丁寧に解説し、Webアプリケーション + データベースという実践的な構成を一つのコマンドで起動する方法を習得します。

この章で学ぶこと

  • 複数のコンテナを連携させる必要性を理解する
  • docker-compose.ymlファイルの基本構造を習得する
  • docker compose up/downコマンドを使いこなす
  • WebアプリケーションとDBを連携させる構成を作る

8.1 なぜ複数のコンテナが必要なのか

実際のWebアプリケーションは、一つのコンテナだけでは完結しません。たとえば、以下のような構成が一般的です。

典型的なWebアプリケーション構成

1
Webサーバー（例: Nginx）

ユーザーからのアクセスを受け付ける玄関口

2
アプリケーションサーバー（例: WordPress, Django）

ビジネスロジックを処理する頭脳

3
データベース（例: MySQL, PostgreSQL）

データを保存・管理する倉庫

複数コンテナ構成の例

Nginx/WordPress] Web -->|データ要求| DB[(DBコンテナ

各コンテナは独立していますが、ネットワーク経由でデータをやり取りします

これまで学んだdocker runコマンドでも複数のコンテナを起動できますが、以下のような課題があります。

docker runだけでは管理が大変

  • コンテナごとにコマンドを実行する必要がある
  • 起動順序やネットワーク設定を手動で管理
  • 一括停止・削除が面倒
  • 設定を他の人と共有しづらい

8.2 Docker Composeとは - オーケストラの指揮者

Docker Compose（ドッカー・コンポーズ）は、複数のコンテナをまとめて管理するためのツールです。

オーケストラの指揮者の例え

Docker Composeは、オーケストラの指揮者のようなものです。

演奏者（各コンテナ）: それぞれが異なる役割を持つ
指揮者（Docker Compose）: 全員を統率し、タイミングよく演奏開始・停止させる
楽譜（docker-compose.yml）: 誰がいつどう演奏するかが書かれている

Docker Composeを使うと、複数のコンテナの設定を1つのファイルにまとめ、1つのコマンドで起動・停止できます。

簡単起動

複数のコンテナをdocker compose up一発で起動できます。

自動ネットワーク

コンテナ間のネットワークを自動で構築し、名前で通信できます。

設定の共有

YAMLファイルをGitで共有すれば、チーム全員が同じ環境を再現できます。

起動順序制御

依存関係を設定して、DBが起動してからアプリを起動するなどの制御が可能。

8.3 docker-compose.ymlの基本構造

Docker Composeの設定は、docker-compose.ymlというYAML形式のファイルに記述します。基本的な構造を見てみましょう。

docker-compose.yml（基本構造） 
version: '3.8'  # Docker Composeのバージョン

services:  # コンテナの定義を記述
  web:  # サービス名（任意の名前）
    image: nginx:latest  # 使用するイメージ
    ports:
      - "8080:80"  # ポートマッピング（ホスト:コンテナ）
    volumes:
      - ./html:/usr/share/nginx/html  # ボリュームマウント

  db:  # 2つ目のサービス
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: example  # 環境変数

主要な設定項目

項目 説明
services 起動するコンテナのリスト。各サービスが1つのコンテナに対応
image 使用するDockerイメージを指定（Docker Hubから取得）
ports ポートマッピング。「ホスト:コンテナ」の形式で指定
volumes ボリュームマウント。データを永続化する場所を指定
environment 環境変数を設定。パスワードやDBホスト名などを指定
networks コンテナ間のネットワークを定義（省略時は自動作成）

YAMLの注意点

YAMLファイルはインデント（字下げ）が重要です。スペース2つまたは4つで統一し、タブは使わないでください。インデントがずれると正しく動きません。

8.4 はじめてのdocker-compose.yml - WordPress + MySQL

実際にWordPress（ブログシステム）とMySQLデータベースを組み合わせた構成を作ってみましょう。

実習 8-1: WordPress環境の構築

手順

  1. 1 作業用フォルダを作成し、移動します
  2. 2 docker-compose.ymlファイルを作成します
  3. 3 Docker Composeで起動します
ターミナル（作業フォルダ作成） 
# 作業フォルダを作成
mkdir wordpress-demo
cd wordpress-demo
docker-compose.yml 
version: '3.8'

services:
  # データベース（MySQL）
  db:
    image: mysql:8.0
    volumes:
      - db_data:/var/lib/mysql  # DBデータを永続化
    environment:
      MYSQL_ROOT_PASSWORD: somewordpress
      MYSQL_DATABASE: wordpress
      MYSQL_USER: wordpress
      MYSQL_PASSWORD: wordpress

  # WordPress本体
  wordpress:
    image: wordpress:latest
    ports:
      - "8000:80"  # ポート8000でアクセス可能
    environment:
      WORDPRESS_DB_HOST: db:3306  # dbサービスに接続
      WORDPRESS_DB_USER: wordpress
      WORDPRESS_DB_PASSWORD: wordpress
      WORDPRESS_DB_NAME: wordpress
    depends_on:
      - db  # dbが起動してから起動

volumes:
  db_data:  # 名前付きボリュームの定義

設定のポイント

  • depends_on: WordPressをDBより後に起動
  • WORDPRESS_DB_HOST: db: サービス名でDB接続
  • volumes: DBデータを削除しても残るように永続化

8.5 docker compose up/downで起動・停止

docker-compose.ymlファイルがあるフォルダで、以下のコマンドを実行します。

起動コマンド 
# すべてのサービスを起動（バックグラウンドで実行）
docker compose up -d

# 出力例:
# [+] Running 3/3
#  ✔ Network wordpress-demo_default  Created
#  ✔ Container wordpress-demo-db-1   Started
#  ✔ Container wordpress-demo-wordpress-1  Started

動作確認

ブラウザでhttp://localhost:8000にアクセスすると、WordPressの初期設定画面が表示されます。

期待される結果

「WordPressへようこそ」という画面が表示され、言語選択やサイト設定ができる状態になっています。

停止・削除コマンド 
# すべてのサービスを停止・削除
docker compose down

# ボリュームも含めて完全に削除する場合
docker compose down -v

注意

docker compose down -vを実行すると、DBのデータも削除されます。データを残したい場合は-vオプションを付けないでください。

8.6 よく使うDocker Composeコマンド

Docker Composeには、起動・停止以外にも便利なコマンドがあります。

コマンド 説明
docker compose up -d すべてのサービスをバックグラウンドで起動
docker compose down すべてのサービスを停止・削除
docker compose ps 起動中のサービス一覧を表示
docker compose logs すべてのサービスのログを表示
docker compose logs -f ログをリアルタイム表示（Ctrl+Cで終了）
docker compose exec [サービス名] bash コンテナ内でコマンド実行（例: bashシェル起動）
docker compose restart すべてのサービスを再起動
docker compose stop サービスを停止（コンテナは削除しない）
ログ確認の例 
# 特定のサービスのログだけ見る
docker compose logs wordpress

# 最新100行だけ表示
docker compose logs --tail=100

# WordPressコンテナ内でbashを起動
docker compose exec wordpress bash

理解度確認クイズ

  1. Q1. Docker Composeを使う最大のメリットは何ですか？

    回答を表示

    複数のコンテナを1つのファイル（docker-compose.yml）で管理し、1つのコマンド（docker compose up）で一括起動・停止できることです。これにより、複雑な環境でも簡単に再現でき、チームでの共有も容易になります。

  2. Q2. docker-compose.ymlで「depends_on」を設定する理由は何ですか？

    回答を表示

    サービスの起動順序を制御するためです。たとえば、WordPressはデータベースが必要なので、DBコンテナが起動してからWordPressコンテナを起動させることで、エラーを防ぎます。

  3. Q3. docker compose downとdocker compose down -vの違いは何ですか？

    回答を表示

    docker compose downはコンテナとネットワークを削除しますが、ボリューム（データ）は残します。docker compose down -vはボリュームも含めて完全に削除します。データを保持したい場合は-vオプションを付けないようにします。

第8章のまとめ

  • 実際のWebアプリは複数のコンテナ（Web、アプリ、DB）を組み合わせて構成する
  • Docker Composeは複数コンテナをまとめて管理するオーケストラの指揮者
  • docker-compose.ymlにservices、image、ports、volumesなどを記述する
  • docker compose up/downで一括起動・停止ができる
  • logs、ps、execなどのコマンドで運用・デバッグができる
