10.1 よくあるトラブルと解決方法

Dockerを使っていると、様々なエラーに遭遇することがあります。ここでは、初心者が遭遇しやすい典型的なトラブルとその解決方法を紹介します。

トラブル1: コンテナが起動しない

症状

docker runを実行してもコンテナがすぐに停止してしまう

よくある原因

  • • コンテナ内で実行されるプログラムがエラーで終了している
  • • Dockerfileの記述にミスがある
  • • 必要なファイルやライブラリが不足している

解決方法

1. ログを確認する

# コンテナのログを表示
docker logs [コンテナ名またはID]

# リアルタイムでログを監視
docker logs -f [コンテナ名またはID]

2. コンテナの状態を確認する

# すべてのコンテナを表示（停止中も含む）
docker ps -a

# 特定のコンテナの詳細情報
docker inspect [コンテナ名またはID]

トラブル2: ポートが既に使用中

症状

Bind for 0.0.0.0:8080 failed: port is already allocatedというエラーが表示される

よくある原因

  • • 他のコンテナが同じポートを使っている
  • • PCで別のアプリケーションが同じポートを使っている
  • • 以前起動したコンテナが停止していない

解決方法

1. 使用中のポートを確認する

# 動作中のコンテナを確認
docker ps

# 特定のポートを使用しているコンテナを停止
docker stop [コンテナ名またはID]

2. 別のポート番号を使う

# 例: ポート8080の代わりに8081を使う
docker run -p 8081:80 nginx

トラブル3: ディスク容量不足

症状

no space left on deviceというエラーが表示される、または動作が遅くなる

よくある原因

  • • 使っていないイメージやコンテナが溜まっている
  • • ボリュームに大量のデータが保存されている
  • • ビルドキャッシュが肥大化している

解決方法

1. ディスク使用量を確認する

# Docker全体のディスク使用量を表示
docker system df

# 詳細な内訳を表示
docker system df -v

2. 不要なデータを削除する

# 停止中のコンテナを削除
docker container prune

# 使っていないイメージを削除
docker image prune

# 使っていないボリュームを削除
docker volume prune

# すべてを一括クリーンアップ（注意: データが消えます）
docker system prune -a --volumes

トラブル4: イメージのダウンロードエラー

症状

Error response from daemon: pull access deniedなどのエラーが表示される

よくある原因

  • • イメージ名のスペルミス
  • • ネットワーク接続の問題
  • • Docker Hubのレート制限に達した

解決方法

1. イメージ名を確認する

# Docker Hubで正しいイメージ名を検索
docker search nginx

# タグを指定してpull
docker pull nginx:latest

2. ネットワーク接続を確認する

インターネット接続が正常か確認し、ファイアウォール設定もチェックします

10.2 デバッグの基本テクニック

トラブルが起きたとき、原因を特定するためのデバッグ技術を身につけましょう。

ログの確認方法

コンテナのログは、エラーの原因を特定する最も重要な情報源です。 

# 基本的なログ表示
docker logs [コンテナ名]

# 最新100行のみ表示
docker logs --tail 100 [コンテナ名]

# リアルタイムでログを監視
docker logs -f [コンテナ名]

# タイムスタンプ付きで表示
docker logs -t [コンテナ名]

コンテナ内部の確認

コンテナの中に入って、直接状態を確認することができます。 

# 実行中のコンテナ内でbashを起動
docker exec -it [コンテナ名] bash

# shを使用する場合（bashが無い場合）
docker exec -it [コンテナ名] sh

# コンテナ内で特定のコマンドを実行
docker exec [コンテナ名] ls -la /app

# コンテナ内のプロセスを確認
docker top [コンテナ名]

パス1: より深いDocker知識

Dockerをより効率的に使うための高度なテクニックを学びます。

マルチステージビルド

イメージサイズを大幅に削減する技術。ビルド用と実行用のステージを分けることで、本番環境で必要なファイルだけを含めることができます。

Dockerネットワーク

コンテナ間の通信を詳細に制御する方法。ブリッジ、ホスト、オーバーレイなど、様々なネットワークモードを理解します。

セキュリティベストプラクティス

本番環境で安全にDockerを使うための設定。rootユーザーを使わない、イメージの脆弱性スキャン、シークレット管理などを学びます。

パス2: Kubernetes入門

大量のコンテナを管理するためのオーケストレーションツールを学びます。

Kubernetesとは

Googleが開発した、コンテナを大規模に管理するためのシステム。自動的なスケーリング、ローリングアップデート、自己修復機能などを提供します。

主要な概念

  • • Pod: 1つ以上のコンテナをまとめた単位
  • • Deployment: Podの管理と更新を自動化
  • • Service: Podへのアクセスを提供

学習リソース

  • • Kubernetes公式サイト（日本語）
  • • Minikube（ローカルでKubernetesを試せるツール）

パス3: クラウドへのデプロイ

作成したコンテナを本番環境で動かすための、クラウドサービスを学びます。

AWS ECS / Fargate

Amazonのコンテナ実行サービス。Fargateを使えばサーバー管理不要でコンテナを動かせます。

Google Cloud Run

Googleのサーバーレスコンテナサービス。リクエストが来たときだけ起動するため、コストを抑えられます。

Azure Container Instances

Microsoftのコンテナ実行サービス。シンプルな構成で素早くコンテナをデプロイできます。

パス4: CI/CD自動化

コードの変更から本番デプロイまでを自動化する仕組みを学びます。

CI/CDとは

  • • CI（継続的インテグレーション）: コードをプッシュすると自動でテスト実行
  • • CD（継続的デリバリー）: テストが通ったら自動でデプロイ

GitHub Actions

GitHubに統合された自動化ツール。プッシュ時に自動でDockerイメージをビルドし、Docker Hubにプッシュできます。

実践例

コードをGitHubにプッシュ → 自動でテスト → Dockerイメージをビルド → クラウドにデプロイ、という一連の流れを自動化します。
