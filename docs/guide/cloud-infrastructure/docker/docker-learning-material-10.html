10.1 よくあるエラーと解決方法

Docker使用中に遭遇しやすいエラーとその対処法を紹介します。エラーメッセージを恐れず、冷静に対処しましょう。

エラー1: コンテナがすぐに停止する

症状:

docker run でコンテナを起動しても、すぐに停止してしまう

原因:

  • メインプロセスがすぐに終了してしまう
  • 起動時にエラーが発生している

解決方法:

  1. ログを確認:
# コンテナ名またはIDを確認
docker ps -a

# ログを確認
docker logs [コンテナ名またはID]
  1. フォアグラウンドで実行して詳細を確認:
# -itオプションで対話モードで起動
docker run -it [イメージ名] /bin/sh

エラー2: ポートがすでに使用中

エラーメッセージ:

Error: bind: address already in use

原因:

指定したホストポート(例: 8080)がすでに他のプロセスやコンテナで使用されている

解決方法:

  1. 使用中のポートを確認:
# すべてのコンテナを確認(停止中も含む)
docker ps -a

# 使用中のポートを確認(Mac/Linux)
lsof -i :8080

# 使用中のポートを確認(Windows)
netstat -ano | findstr :8080
  1. 解決策を選択:
# 方法1: 別のポート番号を使う
docker run -p 8081:80 nginx

# 方法2: 使用中のコンテナを停止
docker stop [コンテナ名]

エラー3: ディスク容量不足

症状:

イメージのビルドやpullが失敗する、または no space left on device エラーが出る

原因:

使わなくなったイメージ、コンテナ、ボリュームが蓄積してディスク容量を圧迫している

解決方法:

  1. ディスク使用状況を確認:
# Docker全体のディスク使用状況
docker system df

# 詳細表示
docker system df -v
  1. 不要なリソースをクリーンアップ:
# すべての停止中コンテナ、未使用ネットワーク、
# タグ無しイメージ、ビルドキャッシュを削除
docker system prune

# ボリュームも含めてすべて削除(注意!)
docker system prune -a --volumes

エラー4: イメージがダウンロードできない

エラーメッセージ:

manifest unknown または pull access denied

原因:

  • イメージ名やタグが間違っている
  • プライベートイメージで認証が必要
  • ネットワーク接続の問題

解決方法:

# Docker Hubで正式なイメージ名を確認
# 例: https://hub.docker.com/

# タグを明示的に指定
docker pull nginx:latest

# プライベートリポジトリの場合はログイン
docker login

10.2 デバッグの基本テクニック

問題が発生したときに役立つデバッグコマンドとテクニックを紹介します。

よく使うデバッグコマンド

1. ログを確認する

# コンテナのログをリアルタイムで表示
docker logs -f [コンテナ名]

# 最新100行だけ表示
docker logs --tail 100 [コンテナ名]

2. コンテナ内部に入る

# 実行中のコンテナに入る
docker exec -it [コンテナ名] /bin/bash

# bashがない場合はsh
docker exec -it [コンテナ名] /bin/sh

3. コンテナの詳細情報を確認

# コンテナの詳細設定を表示
docker inspect [コンテナ名]

# 特定の情報だけ取得(IPアドレスの例)
docker inspect -f '{{.NetworkSettings.IPAddress}}' [コンテナ名]

4. リソース使用状況を確認

# CPU、メモリ使用状況をリアルタイム表示
docker stats

# 特定のコンテナのみ
docker stats [コンテナ名]

5. ネットワーク接続を確認

# すべてのネットワークを確認
docker network ls

# ネットワークの詳細と接続コンテナ
docker network inspect [ネットワーク名]

デバッグのコツ

  • エラーメッセージを最後まで読む - 解決のヒントが含まれています
  • 公式ドキュメントを確認 - docker.comには詳しい説明があります
  • 検索エンジンでエラーメッセージを検索 - 同じ問題に遭遇した人がいるはず

10.3 環境のクリーンアップとメンテナンス

定期的にDocker環境をクリーンアップすることで、ディスク容量を節約し、パフォーマンスを維持できます。

クリーンアップコマンド一覧

停止中のコンテナを削除

# すべての停止中コンテナを削除
docker container prune

# 確認なしで実行
docker container prune -f

未使用のイメージを削除

# タグなしイメージを削除
docker image prune

# すべての未使用イメージを削除
docker image prune -a

未使用のボリュームを削除

# 未使用のボリュームを削除
docker volume prune

すべてをまとめて削除

# 停止コンテナ、未使用ネットワーク、タグ無しイメージ、
# ビルドキャッシュを一括削除
docker system prune

# ボリュームも含めて削除(注意!)
docker system prune --volumes

注意

docker system prune -a --volumes はすべての未使用リソースを削除します。実行前に必ず以下を確認してください:
- 保存が必要なデータはバックアップしたか
- 削除されるイメージを把握しているか(再ダウンロードが必要)

理解度確認クイズ

  1. Q1. コンテナが起動直後に停止してしまう場合、最初に確認すべきことは何ですか?

    回答を表示

    答え: docker logs [コンテナ名] でコンテナのログを確認します。エラーメッセージや起動失敗の原因がログに記録されているはずです。ログを読むことで、設定ミス、不足している環境変数、起動スクリプトのエラーなどを特定できます。

  2. Q2. ディスク容量を節約するために定期的に実行すべきコマンドは何ですか?

    回答を表示

    答え: docker system prune で停止中のコンテナ、未使用ネットワーク、タグ無しイメージ、ビルドキャッシュを削除できます。さらに徹底的にクリーンアップする場合は docker system prune -a --volumes を使いますが、必要なデータのバックアップを忘れずに。

  3. Q3. Docker学習の次のステップとして、どのような選択肢がありますか?

    回答を表示

    答え: 主に3つの学習パスがあります。(1) Docker深掘り: マルチステージビルド、ネットワーク詳細、セキュリティなどDockerそのものをさらに学ぶ。(2) オーケストレーション: Kubernetesなどで複数コンテナを大規模管理する方法を学ぶ。(3) クラウドデプロイ: AWS、GCP、Azureなどクラウド環境でのコンテナ運用を学ぶ。自分の目標に合わせて選択しましょう。
