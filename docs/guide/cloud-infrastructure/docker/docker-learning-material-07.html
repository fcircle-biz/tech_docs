第7章

データを保存しよう - ボリュームの基本

コンテナを削除するとデータが消えてしまう問題を理解し、データを永続化する方法（ボリューム）を学びます。USBメモリの例えでボリュームの概念を説明し、実際にMySQLを使ってデータが保存される様子を確認します。

この章で学ぶこと

  • コンテナのデータは揮発性であることを理解する
  • ボリュームの概念をUSBメモリの例えで理解する
  • 名前付きボリュームの作成と使用方法を学ぶ
  • バインドマウントとボリュームの違いを理解する
  • MySQLでデータ永続化を実践する

7.1 コンテナを削除するとデータが消える問題

これまでの章では、Dockerコンテナを起動・停止・削除する方法を学びました。しかし、コンテナには重要な特徴があります。それは、コンテナを削除すると、その中のデータもすべて消えてしまうということです。

例えば、データベースをコンテナで動かしている場合、コンテナを削除すると登録したデータがすべて消えてしまいます。これでは困りますよね。

コンテナのデータは揮発性

コンテナ内に保存したファイルやデータベースのレコードは、コンテナを削除（docker rm）すると完全に消えます。コンテナの停止（docker stop）だけでは消えませんが、削除すると復元できません。

7.2 ボリュームとは - USBメモリの例えで理解しよう

この問題を解決するのがボリューム（Volume）です。ボリュームを使うと、コンテナを削除してもデータを残すことができます。

ボリュームは、USBメモリのようなものと考えると分かりやすいです。

USBメモリの例え

1
パソコン本体 = コンテナ

パソコン本体に保存したファイルは、パソコンを廃棄すると消えてしまいます

2
USBメモリ = ボリューム

USBメモリに保存したファイルは、パソコンを変えても使えます

3
USBメモリを挿す = ボリュームをマウント

USBメモリをパソコンに挿して使うように、ボリュームをコンテナに接続して使います

ボリュームを使うとデータが残る

ボリュームはコンテナとは別に存在するため、コンテナを削除してもデータは残ります

7.3 名前付きボリュームの使い方

Dockerでは、名前付きボリュームを作成して、複数のコンテナで共有することができます。

ボリュームを作成する

まず、docker volume createコマンドでボリュームを作成します。

ボリューム作成 
# my-dataという名前のボリュームを作成
docker volume create my-data

# ボリューム一覧を確認
docker volume ls

コンテナにボリュームをマウントする

コンテナ起動時に-vオプションでボリュームをマウントします。

ボリュームをマウント 
# -v ボリューム名:コンテナ内のパス
docker run -d \
  --name my-container \
  -v my-data:/data \
  ubuntu:22.04 sleep infinity

マウントの構文

-v ボリューム名:コンテナ内のパス

  • ボリューム名: my-data - 作成したボリュームの名前
  • コンテナ内のパス: /data - コンテナ内でデータを保存する場所

7.4 バインドマウントとボリュームの違い

Dockerには、データを永続化する方法が2つあります：ボリュームとバインドマウントです。

7.5 実践: MySQLでデータを永続化してみよう

実際にMySQLデータベースを使って、ボリュームでデータが保存される様子を確認してみましょう。

ハンズオン: MySQLデータの永続化

手順1: ボリュームを作成

ボリューム作成 
# MySQLのデータ用ボリュームを作成
docker volume create mysql-data

# 作成確認
docker volume ls

手順2: MySQLコンテナをボリューム付きで起動

MySQLコンテナ起動 
# MySQLコンテナを起動（ボリュームをマウント）
docker run -d \
  --name my-mysql \
  -e MYSQL_ROOT_PASSWORD=password \
  -v mysql-data:/var/lib/mysql \
  mysql:8.0

重要: /var/lib/mysql はMySQLがデータを保存するディレクトリです

手順3: データベースとテーブルを作成

データベース作成 
# MySQLコンテナに接続
docker exec -it my-mysql mysql -uroot -ppassword

# MySQL内で以下のSQLを実行
CREATE DATABASE shop;
USE shop;
CREATE TABLE products (id INT, name VARCHAR(50));
INSERT INTO products VALUES (1, 'りんご'), (2, 'みかん');
SELECT * FROM products;
# 確認後、exit で抜ける
exit

手順4: コンテナを削除

コンテナ削除 
# コンテナを停止・削除
docker stop my-mysql
docker rm my-mysql

# コンテナが削除されたことを確認
docker ps -a

手順5: 新しいコンテナで同じボリュームをマウント

新コンテナ起動 
# 同じボリュームで新しいコンテナを起動
docker run -d \
  --name my-mysql-2 \
  -e MYSQL_ROOT_PASSWORD=password \
  -v mysql-data:/var/lib/mysql \
  mysql:8.0

手順6: データが残っているか確認

データ確認 
# 新しいコンテナに接続
docker exec -it my-mysql-2 mysql -uroot -ppassword

# データが残っているか確認
USE shop;
SELECT * FROM products;
# りんごとみかんが表示されれば成功！
exit

動作確認のポイント

  • 最初のコンテナで作成したデータ（りんご、みかん）が表示される
  • コンテナを削除してもボリュームは残っている
  • 新しいコンテナでも同じデータにアクセスできる

7.6 ボリューム管理コマンド

ボリュームを管理するための便利なコマンドをまとめます。

コマンド 説明
docker volume create ボリュームを作成
docker volume ls ボリューム一覧を表示
docker volume inspect ボリュームの詳細情報を表示
docker volume rm ボリュームを削除
docker volume prune 使用されていないボリュームをすべて削除
ボリューム管理の実例 
# ボリューム一覧を確認
docker volume ls

# 特定のボリュームの詳細を確認
docker volume inspect mysql-data

# ボリュームを削除（使用中のボリュームは削除できない）
docker volume rm mysql-data

# 使われていないボリュームをまとめて削除
docker volume prune

ボリューム削除時の注意

docker volume rm でボリュームを削除すると、中のデータもすべて消えます。重要なデータは必ずバックアップを取ってから削除してください。

よくあるトラブルと解決法

ボリュームが削除できない

原因: コンテナがボリュームを使用中です

解決法: まずコンテナを停止・削除してから、ボリュームを削除してください 

# コンテナを削除してからボリュームを削除
docker rm -f コンテナ名
docker volume rm ボリューム名

データが保存されない

原因: マウント先のパスが間違っている可能性があります

解決法: アプリケーションのドキュメントで、データ保存パスを確認してください（MySQLは /var/lib/mysql）

ボリュームがどこにあるか分からない

解決法: docker volume inspect で場所を確認できます 

# ボリュームの保存場所を確認
docker volume inspect mysql-data

理解度確認クイズ

  1. Q1. コンテナを削除すると、中のデータはどうなりますか？ボリュームを使わない場合と使う場合で説明してください。

    回答を表示

    ボリュームを使わない場合: コンテナを削除すると、中のデータもすべて消えてしまいます。
    ボリュームを使う場合: コンテナを削除してもボリューム内のデータは残るため、新しいコンテナで同じデータを使い続けることができます。

  2. Q2. ボリュームをUSBメモリに例えたとき、「ボリュームをマウントする」とはどういう意味ですか？

    回答を表示

    USBメモリをパソコンに挿して使えるようにすることと同じです。ボリューム（USBメモリ）をコンテナ（パソコン）に接続することで、コンテナからボリュームにデータを保存したり読み込んだりできるようになります。

  3. Q3. ボリュームとバインドマウントの使い分けを説明してください。

    回答を表示

    ボリューム: データベースのデータなど、Dockerに管理を任せたい場合に使います。Dockerが自動的に適切な場所にデータを保存してくれます。
    バインドマウント: ソースコードなど、ホストから直接編集したい場合に使います。ホストの特定のフォルダをコンテナにマウントするため、ホスト側で編集すると即座にコンテナに反映されます。

第7章のまとめ

  • コンテナを削除すると、中のデータもすべて消えてしまう（揮発性）
  • ボリュームを使うと、コンテナとは別にデータを永続化できる
  • ボリュームはUSBメモリのようなもので、コンテナにマウントして使う
  • ボリューム（Docker管理）とバインドマウント（ユーザー管理）の2種類がある
  • データベースはボリューム、ソースコードはバインドマウントが適している
