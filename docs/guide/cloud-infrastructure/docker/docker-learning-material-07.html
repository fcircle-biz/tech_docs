第7章

データを保存しよう - ボリュームの基本

コンテナを削除するとデータが消えてしまう問題を理解し、データを永続化する方法（ボリューム）を学びます。USBメモリの例えでボリュームの概念を説明し、実際にデータベース（MySQL）を使ってデータが保存される様子を確認します。バインドマウントとの違いも図解で理解します。

この章で学ぶこと

  • コンテナのデータが消える問題を理解する
  • ボリュームの概念を日常の例え（USBメモリ）で理解する
  • docker volumeコマンドでボリュームを管理する
  • バインドマウントとボリュームの違いを理解する

7.1 コンテナのデータは消える？

これまでの章で、コンテナを起動・停止・削除してきました。実は、コンテナを削除すると、コンテナ内に保存されていたデータもすべて消えてしまいます。

たとえば、データベース（MySQL）をコンテナで動かして、たくさんのデータを登録したとします。しかし、そのコンテナを削除すると、せっかく登録したデータもすべて消えてしまうのです。

コンテナを削除すると、中に保存されていたデータもすべて消えてしまいます

実験 7-1: データが消えることを確認しよう

簡単なコンテナでファイルを作成し、削除後に消えることを確認します。

手順

  1. 1 Ubuntuコンテナを起動し、中でファイルを作成します
  2. 2 コンテナを停止・削除し、再度起動してファイルが消えていることを確認します 
# 1. Ubuntuコンテナを起動して中に入る
docker run -it --name test-data ubuntu bash

# コンテナ内で実行: ファイルを作成
echo "重要なデータ" > mydata.txt
cat mydata.txt  # 「重要なデータ」と表示される
exit  # コンテナから出る

# 2. コンテナを削除
docker rm test-data

# 3. 再度同じコンテナを起動
docker run -it --name test-data ubuntu bash
cat mydata.txt  # エラー: ファイルが存在しない!

このように、コンテナを削除すると中のデータも消えてしまいます。これではデータベースやファイル保存が必要なアプリケーションで困りますね。そこで登場するのが「ボリューム」です。

7.2 ボリュームとは - USBメモリの例え

ボリューム（Volume）は、コンテナとは独立した「データの保存場所」です。これを使うと、コンテナを削除してもデータを残すことができます。

USBメモリの例え

1
パソコン = コンテナ

パソコン本体にデータを保存すると、パソコンを処分したらデータも消えます

2
USBメモリ = ボリューム

USBメモリにデータを保存すれば、パソコンを処分してもデータは残ります

3
別のパソコンでも使える

USBメモリを別のパソコンに挿せば、同じデータが使えます（別のコンテナでも使える）

ボリュームの仕組み

データ保存領域)] end subgraph container1[コンテナ1

ボリュームはコンテナの外に存在し、コンテナを削除してもデータは残ります

7.3 ボリュームを作成・使用する

ボリュームはdocker volumeコマンドで管理します。

ボリュームの作成

# ボリュームを作成
docker volume create my-data

# ボリューム一覧を表示
docker volume ls

# ボリュームの詳細情報を表示
docker volume inspect my-data

ボリュームをコンテナにマウントする

-vオプションで「ボリューム名:コンテナ内のパス」を指定します。 

# ボリュームをマウントしてUbuntuコンテナを起動
docker run -it --name test-volume -v my-data:/data ubuntu bash

# コンテナ内で実行: ファイルを作成
echo "永続化されたデータ" > /data/mydata.txt
cat /data/mydata.txt
exit

# コンテナを削除
docker rm test-volume

# 再度同じボリュームをマウントして起動
docker run -it --name test-volume2 -v my-data:/data ubuntu bash
cat /data/mydata.txt  # データが残っている!

/dataディレクトリに保存したデータは、ボリュームに保存されるため、コンテナを削除してもデータは残ります。

ボリュームの削除

# ボリュームを削除（コンテナが使用中でないこと）
docker volume rm my-data

# 使われていないボリュームをすべて削除
docker volume prune

7.4 バインドマウントとは

ボリュームの他に、バインドマウント（Bind Mount）という方法もあります。これは、ホストマシンの特定のフォルダをコンテナ内にマウントする方法です。

バインドマウントの使い方

ホストマシンのフルパスを指定します。 

# ホストの現在のディレクトリをコンテナの/appにマウント
docker run -it --name bind-test -v $(pwd):/app ubuntu bash

# Windowsの場合
docker run -it --name bind-test -v %cd%:/app ubuntu bash

# PowerShellの場合
docker run -it --name bind-test -v ${PWD}:/app ubuntu bash

ホストのフォルダとコンテナ内のフォルダが同期されます。ホスト側でファイルを編集すると、コンテナ内でもすぐに反映されます。

7.5 ボリュームとバインドマウントの使い分け

それぞれに適した使い方があります。

項目 ボリューム バインドマウント
管理 Dockerが管理
（場所を気にしなくて良い） 		ユーザーが管理
（ホストのパスを指定）
推奨用途 データベースのデータ保存
（永続化が目的） 		開発中のソースコード共有
（リアルタイム同期が必要）
移植性 高い
（どの環境でも動く） 		低い
（ホストのパスに依存）
ホスト側からのアクセス 難しい
（Docker管理下） 		簡単
（直接編集可能）

ボリュームを使う場合

  • ・データベースのデータ保存
  • ・ログファイルの永続化
  • ・アプリケーションの設定ファイル

バインドマウントを使う場合

  • ・開発中のソースコードの共有
  • ・設定ファイルの動的な変更
  • ・ホストのツールとの連携

7.6 実践: データベースでボリュームを使う

MySQLデータベースをコンテナで起動し、ボリュームでデータを永続化してみましょう。

実習 7-2: MySQLでボリュームを使ってみよう

MySQLコンテナにボリュームをマウントし、データが永続化されることを確認します。

手順

  1. 1 ボリュームを作成してMySQLコンテナを起動
  2. 2 データベースにデータを登録
  3. 3 コンテナを削除し、新しいコンテナで同じボリュームを使ってデータが残っていることを確認 
# 1. ボリュームを作成
docker volume create mysql-data

# 2. MySQLコンテナを起動（ボリュームをマウント）
docker run -d \
  --name mysql-test \
  -e MYSQL_ROOT_PASSWORD=mypassword \
  -v mysql-data:/var/lib/mysql \
  mysql:8.0

# 3. MySQLに接続してデータを登録
docker exec -it mysql-test mysql -uroot -pmypassword -e "CREATE DATABASE testdb;"
docker exec -it mysql-test mysql -uroot -pmypassword -e "USE testdb; CREATE TABLE users (id INT, name VARCHAR(50));"
docker exec -it mysql-test mysql -uroot -pmypassword -e "USE testdb; INSERT INTO users VALUES (1, 'Alice');"

# 4. データを確認
docker exec -it mysql-test mysql -uroot -pmypassword -e "USE testdb; SELECT * FROM users;"

# 5. コンテナを停止・削除
docker stop mysql-test
docker rm mysql-test

# 6. 新しいコンテナで同じボリュームを使って起動
docker run -d \
  --name mysql-test2 \
  -e MYSQL_ROOT_PASSWORD=mypassword \
  -v mysql-data:/var/lib/mysql \
  mysql:8.0

# 7. データが残っているか確認
docker exec -it mysql-test2 mysql -uroot -pmypassword -e "USE testdb; SELECT * FROM users;"
# Aliceのデータが表示される!

重要ポイント

MySQLのデータは/var/lib/mysqlに保存されます。このディレクトリをボリュームにマウントすることで、コンテナを削除してもデータが残ります。

理解度確認クイズ

  1. Q1. コンテナを削除すると、コンテナ内に保存されていたデータはどうなりますか？

    回答を表示

    コンテナを削除すると、コンテナ内に保存されていたデータもすべて消えてしまいます。これはコンテナの設計思想で、コンテナは一時的な実行環境として扱われるためです。データを永続化したい場合は、ボリュームやバインドマウントを使う必要があります。

  2. Q2. ボリュームとバインドマウントの主な違いは何ですか？

    回答を表示

    ボリュームはDockerが管理する専用の保存領域で、場所を気にせずに使えます（移植性が高い）。一方、バインドマウントはホストマシンの特定のフォルダを指定する方法で、ホスト側から直接ファイルを編集できますが、ホストのパスに依存するため移植性は低くなります。データベースのデータ保存にはボリューム、開発中のソースコード共有にはバインドマウントが適しています。

  3. Q3. MySQLコンテナでデータを永続化するには、どのディレクトリをボリュームにマウントすれば良いですか？

    回答を表示

    MySQLのデータは/var/lib/mysqlディレクトリに保存されます。このディレクトリをボリュームにマウントすることで、コンテナを削除してもデータベースのデータを保持できます。例: -v mysql-data:/var/lib/mysql

第7章のまとめ

  • コンテナを削除すると中のデータも消えてしまう
  • ボリュームを使うとデータを永続化できる（USBメモリのイメージ）
  • docker volumeコマンドでボリュームを作成・管理する
  • バインドマウントはホストのフォルダを直接マウントする方法
  • データベースにはボリューム、開発コードにはバインドマウントが適している
