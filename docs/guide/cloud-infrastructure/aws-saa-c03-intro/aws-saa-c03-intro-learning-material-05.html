EC2 Auto Scaling - 自動スケーリングの設計

需要の変動に応じてEC2インスタンスを自動調整するAuto Scalingの設計パターンと、スケーリングポリシーの使い分けを学びます。

この章で学ぶこと

  • Auto Scalingの基本概念とメリット
  • 起動テンプレートと起動設定の違い
  • Auto Scalingグループの構成要素
  • 4種類のスケーリングポリシー（ターゲット追跡・ステップ・シンプル・スケジュールド）
  • インスタンスのライフサイクルとフックの活用

5.1 Auto Scalingの基本概念

EC2 Auto Scalingは、アプリケーションの負荷に応じてEC2インスタンスの数を自動的に調整するサービスです。トラフィックが増加したときは自動的にインスタンスを追加し、減少したときは削減することで、コスト効率と可用性の両立を実現します。

Auto Scalingの3つのメリット

弾力性の確保

需要の急増にも自動的に対応し、パフォーマンスを維持

コスト最適化

不要なときはインスタンスを削減し、無駄なコストを削減

可用性の向上

異常インスタンスを自動検出・置換し、サービスを継続

Auto Scalingの動作フロー

重要ポイント

Auto Scalingは単なるインスタンス数の調整だけでなく、ヘルスチェックを通じた異常インスタンスの自動置換機能も提供します。これにより、手動介入なしで高い可用性を維持できます。

5.2 起動テンプレートと起動設定

Auto Scalingグループでインスタンスを起動する際に使用する設定には、「起動テンプレート（Launch Template）」と「起動設定（Launch Configuration）」の2種類があります。現在は起動テンプレートの使用が推奨されています。

起動テンプレート vs 起動設定

項目 起動テンプレート（推奨） 起動設定（旧型）
バージョン管理 可能（複数バージョン保持） 不可（イミュータブル）
スポットインスタンス 複数インスタンスタイプ指定可 1種類のみ
最新機能対応 新機能が順次追加 新機能なし
EC2単体起動 可能 不可

起動テンプレートの構成要素

AMI ID

起動するインスタンスのベースイメージ

インスタンスタイプ

t3.micro、m5.large など（複数指定も可能）

キーペア

SSH接続用の認証キー

セキュリティグループ

ファイアウォールルールの設定

IAMロール

インスタンスに付与する権限

ユーザーデータ

起動時に実行するスクリプト（初期設定の自動化）

ヒント

起動テンプレートはバージョン管理できるため、設定変更時も過去のバージョンにロールバック可能です。本番環境では新しいバージョンをテストしてから、デフォルトバージョンを切り替える運用が推奨されます。

5.3 Auto Scalingグループの構成

Auto Scalingグループ（ASG）は、EC2インスタンスの集合を管理する論理的な単位です。どこで、何台のインスタンスを、どのように維持するかを定義します。

Auto Scalingグループの主要設定

最小

最小キャパシティ

常に維持する最小インスタンス数（例: 2台）

可用性確保のため通常2以上

希望

希望キャパシティ

通常時に維持したいインスタンス数（例: 4台）

スケーリングポリシーで自動調整

最大

最大キャパシティ

スケールアウトの上限（例: 10台）

コスト上限の制御

サブネット配置

複数AZのサブネットを指定

高可用性のためマルチAZ配置

ヘルスチェックの設定

EC2ヘルスチェック

EC2インスタンスのステータスチェック（システムステータス・インスタンスステータス）で判定します。ハードウェア障害やOS障害を検出します。

ELBヘルスチェック

ELBが実施するアプリケーションレベルのヘルスチェック（HTTPレスポンス確認）で判定します。アプリケーション異常を検出できます。

注意

ELBヘルスチェックを有効化すると、アプリケーション異常時に自動的にインスタンスが置換されます。ヘルスチェックの猶予期間（Health Check Grace Period）を適切に設定し、起動直後のインスタンスが誤って終了されないようにしましょう（推奨: 300秒）。

5.4 スケーリングポリシーの種類

Auto Scalingには、トラフィックパターンや要件に応じて選択できる4種類のスケーリングポリシーがあります。それぞれの特徴と使い分けを理解しましょう。

1. ターゲット追跡スケーリング（最も推奨）

概要

目標とするメトリクス値（例: CPU使用率50%）を指定すると、Auto Scalingがその値を維持するように自動調整します。最もシンプルで設定が簡単です。

使用例

ターゲット: 平均CPU使用率 50%

  • CPU使用率が50%を超える → インスタンス追加
  • CPU使用率が50%を下回る → インスタンス削減

推奨メトリクス

  • 平均CPU使用率
  • ターゲットあたりのリクエスト数（ALB使用時）
  • ネットワークイン/アウト

2. ステップスケーリング

概要

メトリクス値の範囲に応じて、異なる数のインスタンスを追加・削減します。負荷の大きさに応じたきめ細かい調整が可能です。

使用例

スケールアウトポリシー:

  • CPU 50-60% → 1台追加
  • CPU 60-70% → 2台追加
  • CPU 70%以上 → 3台追加

適用シーン

トラフィックの急激な変動があり、負荷レベルに応じた柔軟なスケーリングが必要な場合に有効です。

3. シンプルスケーリング

概要

アラームが発生したら固定数のインスタンスを追加・削減します。最もシンプルですが、柔軟性に欠けるため、現在はあまり推奨されません。

使用例

ポリシー:

  • CPU 70%超過 → 1台追加
  • 次のスケーリングまでクールダウン期間が必要

制約

クールダウン期間中は次のスケーリングができないため、急激なトラフィック変動には対応しにくい。

4. スケジュールドスケーリング

概要

事前に予測可能なトラフィックパターンに基づいて、特定の時刻・曜日にスケーリングを実行します。メトリクスに依存しない予測型スケーリングです。

使用例

スケジュール:

  • 平日 9:00 → 最小10台、希望15台
  • 平日 18:00 → 最小5台、希望8台
  • 土日 → 最小2台、希望3台

適用シーン

ビジネスアワーと夜間でトラフィックパターンが明確に異なる場合や、定期的なバッチ処理がある場合に最適です。

スケーリングポリシーの選択ガイド

ポリシー 推奨シーン 設定難易度
ターゲット追跡 一般的なWebアプリケーション（最も推奨） ★☆☆ 簡単
ステップ 負荷レベルに応じた細かい調整が必要 ★★☆ 中程度
シンプル レガシーシステム（非推奨） ★☆☆ 簡単
スケジュールド 予測可能なトラフィックパターン ★☆☆ 簡単

ベストプラクティス

ターゲット追跡スケーリングとスケジュールドスケーリングを組み合わせることで、予測可能なパターンには事前対応し、予期しない負荷にはメトリクスベースで対応する、ハイブリッド戦略が最も効果的です。

5.5 インスタンスのライフサイクルとフック

Auto Scalingグループ内のインスタンスには、起動から終了までの明確なライフサイクルがあります。ライフサイクルフックを活用することで、起動・終了時に独自の処理を挿入できます。

インスタンスのライフサイクル

stateDiagram-v2 [*] --> Pending: スケールアウト Pending --> InService: 起動完了 InService --> Terminating: スケールイン InService --> Terminating: ヘルスチェック失敗 Terminating --> Terminated Terminated --> [*] note right of Pending ライフサイクルフック挿入可能 起動後処理を実行 end note note right of Terminating ライフサイクルフック挿入可能 終了前処理を実行 end note

ライフサイクルフックの活用例

起動時フック

実行タイミング: Pending → InService

  • アプリケーションのウォームアップ
  • 設定ファイルのダウンロード
  • 監視エージェントの登録
  • データベース接続プールの事前確立

終了時フック

実行タイミング: InService → Terminating

  • ログのS3アップロード
  • 処理中リクエストの完了待ち
  • 外部サービスからの登録解除
  • クリーンアップ処理

ライフサイクルフックの実装例

lifecycle-hook-example.sh 
#!/bin/bash
# 起動時フック: アプリケーションのウォームアップ

# 1. 設定ファイルをS3から取得
aws s3 cp s3://my-bucket/config/app.conf /etc/myapp/

# 2. データベース接続確認
until mysql -h ${DB_HOST} -u ${DB_USER} -p${DB_PASS} -e "SELECT 1"; do
  echo "Waiting for database..."
  sleep 5
done

# 3. キャッシュのプリロード
curl http://localhost:8080/warmup

# 4. ライフサイクルアクション完了を通知
aws autoscaling complete-lifecycle-action \
  --lifecycle-action-result CONTINUE \
  --lifecycle-hook-name my-launch-hook \
  --auto-scaling-group-name my-asg \
  --instance-id $(ec2-metadata --instance-id | cut -d " " -f 2)

ヒント

ライフサイクルフックのタイムアウトはデフォルト3600秒（1時間）ですが、処理時間に応じて調整可能です。処理が完了したら必ず `complete-lifecycle-action` APIを呼び出し、タイムアウトを待たずに次の状態に進めましょう。

5.6 スポットインスタンスとの統合

起動テンプレートを使用することで、Auto Scalingグループでスポットインスタンス（最大90%割引）を活用し、大幅なコスト削減が可能です。

スポットインスタンスのメリットと注意点

メリット

  • 最大90%のコスト削減
  • 同じ構成でより多くのインスタンス起動可能
  • 複数インスタンスタイプを組み合わせて可用性向上

注意点

  • 2分前予告で中断される可能性
  • ステートレスなワークロードに限定
  • オンデマンドとの混合構成を推奨

スポット・オンデマンド混合構成のベストプラクティス

推奨配分

基本ベースライン（オンデマンド）: 30%

最小キャパシティはオンデマンドで確保し、安定性を担保

スケールアウト分（スポット）: 70%

需要増加時の追加容量はスポットでコスト削減

インスタンスタイプ: 複数指定

例: m5.large、m5a.large、m5n.large（中断リスク分散）

重要ポイント

スポットインスタンスの中断通知（2分前）をEventBridgeで検知し、ライフサイクルフックで処理中のタスクを別のインスタンスに移行する実装を行うことで、中断の影響を最小化できます。

実習 5-1: Auto Scalingグループの作成とターゲット追跡スケーリング

ターゲット追跡スケーリングポリシーを使用したAuto Scalingグループを作成し、負荷テストで動作を確認します。

手順

  1. 1 起動テンプレートを作成（Amazon Linux 2、t3.micro、ユーザーデータでApache自動起動）
  2. 2 Auto Scalingグループを作成（最小2、希望2、最大5、複数AZのサブネット指定）
  3. 3 Application Load Balancerを作成し、Auto Scalingグループをターゲットグループに登録
  4. 4 ターゲット追跡スケーリングポリシーを設定（ターゲット: 平均CPU使用率50%）
  5. 5 インスタンスにSSH接続し、CPU負荷をかけるコマンド実行（stress-ng等）
  6. 6 CloudWatchメトリクスでCPU使用率を確認、スケールアウトが発生することを確認
  7. 7 負荷を停止し、スケールインが発生することを確認

期待される結果

  • CPU使用率が50%を超えると、自動的にインスタンスが追加される
  • 負荷が下がると、数分後にインスタンスが削減される（スケールインは慎重）
  • ALBが新しいインスタンスをヘルスチェック後、トラフィックを分散する

理解度確認クイズ

  1. Q1. 起動テンプレートと起動設定の最大の違いは何ですか？

    回答を表示

    起動テンプレートはバージョン管理が可能で、複数のインスタンスタイプを指定できます。また、EC2単体起動にも使用でき、最新機能に対応しています。起動設定は変更不可（イミュータブル）で、1種類のインスタンスタイプしか指定できません。

  2. Q2. 最も推奨されるスケーリングポリシーはどれで、その理由は何ですか？

    回答を表示

    ターゲット追跡スケーリングが最も推奨されます。目標とするメトリクス値を指定するだけで、Auto Scalingが自動的に最適なインスタンス数を計算・調整してくれるため、設定が簡単かつ効果的です。一般的なWebアプリケーションの大半のユースケースに対応できます。

  3. Q3. ライフサイクルフックを使用する主な目的は何ですか？

    回答を表示

    インスタンスの起動時や終了時に独自の処理を挿入するためです。起動時には設定ファイルの取得やウォームアップ処理を実行し、終了時にはログのバックアップや処理中リクエストの完了待ちなどを行うことで、Auto Scalingの柔軟性と信頼性を高めます。

  4. Q4. スポットインスタンスとオンデマンドインスタンスの推奨配分比率は？

    回答を表示

    基本ベースライン（最小キャパシティ）はオンデマンド30%で安定性を確保し、スケールアウト分はスポット70%でコスト削減を図る配分が推奨されます。また、複数のインスタンスタイプを指定することで、スポット中断のリスクを分散できます。

この章のまとめ

  • Auto Scalingは弾力性・コスト最適化・可用性向上の3つのメリットを提供
  • 起動テンプレートはバージョン管理可能で、起動設定より柔軟（推奨）
  • Auto Scalingグループは最小・希望・最大キャパシティとサブネット配置を定義
  • スケーリングポリシーは4種類あり、ターゲット追跡が最も推奨される
  • ライフサイクルフックで起動・終了時に独自処理を挿入可能
  • スポットインスタンス混合でコスト削減（ベースラインはオンデマンドで安定性確保）
