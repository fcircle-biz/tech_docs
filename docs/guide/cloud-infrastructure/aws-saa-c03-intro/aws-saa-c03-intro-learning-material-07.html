VPC設計 - セキュアなネットワーク構築

Amazon VPCを使用したセキュアなネットワーク設計、サブネット構成のベストプラクティス、VPCエンドポイントなどの高度なネットワーク機能を学びます。

この章で学ぶこと

  • VPCの設計原則とCIDR設計
  • パブリックサブネットとプライベートサブネット
  • インターネットゲートウェイとNATゲートウェイ
  • セキュリティグループとネットワークACL
  • VPCピアリング

  • VPC Flow Logsによるトラフィック監視

7.1 VPCの基礎とCIDR設計

Amazon Virtual Private Cloud（VPC）は、AWSクラウド内に論理的に分離されたプライベートネットワークを作成するサービスです。VPCを適切に設計することで、セキュアで拡張性の高いネットワークインフラを構築できます。

VPCの基本概念

プライベートネットワーク空間

IPアドレス範囲、サブネット、ルートテーブルを完全に制御可能

論理的な分離

他のAWSアカウントやVPCから完全に隔離された環境

リージョン単位

1つのVPCは1つのリージョン内に存在し、複数のAZに跨ることができる

CIDR設計のベストプラクティス

VPC作成時に指定するCIDRブロックは後から変更できないため、慎重な設計が必要です。

推奨されるCIDRサイズ

  • /16 - 最も一般的（65,536 IP）
  • /20 - 中規模環境（4,096 IP）
  • /24 - 小規模環境（256 IP）

避けるべきCIDR範囲

  • オンプレミス環境と重複する範囲
  • 192.168.0.0/16 - 家庭用ルータで一般的
  • /28以下 - IP数が少なすぎる

7.2 サブネット設計とルーティング

VPC内のサブネットは、パブリックサブネットとプライベートサブネットに分類され、それぞれ異なる役割と用途を持ちます。

パブリックサブネット vs プライベートサブネット

項目 パブリックサブネット プライベートサブネット
インターネット接続 直接可能（IGW経由） NAT経由のみ
パブリックIP 自動割り当て可能 割り当てなし
用途 ALB、踏み台サーバー EC2、RDS、Lambda
セキュリティレベル 低（外部公開） 高（内部のみ）

インターネットゲートウェイとNATゲートウェイ

インターネットゲートウェイ（IGW）

  • VPCとインターネット間の双方向通信を可能にする
  • パブリックサブネットに配置されたリソースに必要
  • 1つのVPCに1つのみアタッチ可能
  • 追加料金なし

NATゲートウェイ

  • プライベートサブネットからインターネットへの単方向通信
  • パブリックサブネットに配置
  • 高可用性のため各AZに配置推奨
  • 時間単位+データ転送量課金

7.3 セキュリティグループとネットワークACL

VPCには2層のファイアウォールが存在します。セキュリティグループ（インスタンスレベル）とネットワークACL（サブネットレベル）です。

セキュリティグループ vs ネットワークACL

項目 セキュリティグループ ネットワークACL
動作レベル インスタンス サブネット
ルール 許可ルールのみ 許可・拒否の両方
ステート ステートフル（戻りトラフィック自動許可） ステートレス（明示的設定必要）
ルール評価 すべてのルールを評価 番号順に評価（最初にマッチ）
適用対象 指定したインスタンス サブネット内すべて

ベストプラクティス

ほとんどのケースではセキュリティグループのみで十分です。ネットワークACLは、明示的な拒否ルールが必要な場合や、サブネット全体のトラフィック制御が必要な場合に使用します。

セキュリティグループの設計例

（デフォルト: すべて拒否）

7.4 VPCピアリング

VPCピアリングは、2つのVPC間でプライベートIPアドレスを使用した通信を可能にする機能です。

VPCピアリングの特徴

利点

  • 低レイテンシ通信
  • インターネットを経由しないセキュアな通信
  • クロスアカウント・クロスリージョン対応
  • データ転送料金のみ

制約

  • CIDR範囲が重複していると接続不可
  • 推移的ピアリング不可（A-B、B-C接続時、A-Cは不可）
  • 各VPCのルートテーブル設定が必要

7.5 VPCエンドポイント

VPCエンドポイントは、インターネットゲートウェイやNATゲートウェイを経由せずに、AWSサービスにプライベート接続する機能です。

2種類のVPCエンドポイント

Gateway Endpoint

対応サービス

  • • Amazon S3
  • • DynamoDB

特徴

  • • ルートテーブルにターゲットを追加
  • • 追加料金なし
  • • 水平スケーリング自動

Interface Endpoint（PrivateLink）

対応サービス

  • • その他多数のAWSサービス
  • • EC2、SNS、SQS、Secrets Manager等

特徴

  • • ENI（Elastic Network Interface）を作成
  • • 時間単位課金
  • • プライベートDNS名使用可能

VPCエンドポイントの利点

セキュリティ向上

トラフィックがインターネットを経由しないため、外部からの攻撃リスクを軽減

パフォーマンス向上

AWSバックボーンネットワークを使用し、低レイテンシで高スループット通信

コスト削減（Gateway Endpoint）

S3・DynamoDBへのアクセスでNAT Gateway料金を削減可能

7.6 VPC Flow Logs

VPC Flow Logsは、VPC内のネットワークインターフェイスとの間で送受信されるIPトラフィックに関する情報をキャプチャする機能です。

Flow Logsの用途

トラブルシューティング

  • • 接続問題の診断
  • • セキュリティグループルールの検証
  • • ネットワークACLの動作確認

セキュリティ監視

  • • 不審なトラフィックの検出
  • • アクセスパターンの分析
  • • コンプライアンス要件対応

Flow Logsの出力先

CloudWatch Logs

リアルタイム監視・アラート設定・長期保存

Amazon S3

コスト効率的な長期保存・ビッグデータ分析

Kinesis Data Firehose

リアルタイムストリーミング分析

ハンズオン実習: 3層アーキテクチャVPCの構築

パブリックサブネット・プライベートサブネット・データベースサブネットを持つ3層アーキテクチャVPCを構築します。

構成要素

  • • VPC: 10.0.0.0/16
  • • パブリックサブネット（2つのAZ）: 10.0.1.0/24, 10.0.2.0/24
  • • プライベートサブネット（2つのAZ）: 10.0.11.0/24, 10.0.12.0/24
  • • インターネットゲートウェイ
  • • NAT Gateway（各AZ）
  • • S3 Gateway Endpoint
手順を表示

ステップ1: VPC作成

  1. VPCコンソールで「VPCを作成」
  2. IPv4 CIDR: 10.0.0.0/16
  3. テナンシー: デフォルト

ステップ2: サブネット作成

  1. パブリックサブネット2つ（ap-northeast-1a, 1c）
  2. プライベートサブネット2つ（ap-northeast-1a, 1c）
  3. 各サブネットにCIDRを割り当て

ステップ3: インターネットゲートウェイ作成

  1. IGWを作成しVPCにアタッチ
  2. パブリックサブネットのルートテーブルに追加

ステップ4: NAT Gateway作成

  1. 各AZのパブリックサブネットにNAT Gateway配置
  2. Elastic IPを割り当て
  3. プライベートサブネットのルートテーブルに追加

ステップ5: S3 Gateway Endpoint作成

  1. VPCエンドポイント作成でS3を選択
  2. プライベートサブネットのルートテーブルに関連付け

理解度確認クイズ

  1. Q1. パブリックサブネットとプライベートサブネットの主な違いは何ですか？

    回答を表示

    パブリックサブネットはインターネットゲートウェイへのルートを持ち、インスタンスにパブリックIPが割り当てられます。プライベートサブネットはNAT Gatewayを経由してのみインターネットアクセスが可能で、パブリックIPは割り当てられません。

  2. Q2. セキュリティグループとネットワークACLの違いを説明してください。

    回答を表示

    セキュリティグループはインスタンスレベルでステートフルに動作し、許可ルールのみを設定します。ネットワークACLはサブネットレベルでステートレスに動作し、許可・拒否の両方のルールを番号順に評価します。

  3. Q3. VPC Gateway EndpointとInterface Endpointの違いと使い分けを説明してください。

    回答を表示

    Gateway EndpointはS3とDynamoDBのみ対応し、ルートテーブルベースで無料です。Interface Endpointはその他多数のAWSサービスに対応し、ENIを使用して時間単位で課金されます。コスト削減が必要な場合はGateway Endpointを優先します。

この章のまとめ

  • VPCはAWS内に論理的に分離されたプライベートネットワークを構築し、CIDR設計は後から変更できないため慎重に
  • パブリックサブネット（IGW経由）とプライベートサブネット（NAT経由）を適切に分離してセキュリティを向上
  • セキュリティグループ（インスタンスレベル・ステートフル）とネットワークACL（サブネットレベル・ステートレス）の2層防御
  • VPCピアリングは推移的接続不可、多数のVPC接続にはTransit Gatewayを検討
  • VPCエンドポイント（Gateway/Interface）でAWSサービスにプライベート接続し、セキュリティとコストを最適化
  • VPC Flow Logsでトラフィックを監視し、トラブルシューティングとセキュリティ分析を実施
