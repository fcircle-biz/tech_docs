Elastic Load Balancing - 負荷分散の設計

ELBを使用したトラフィック分散と、用途に応じたロードバランサーの選択を学びます。

この章で学ぶこと

  • ロードバランサーの役割と必要性
  • Application Load Balancer（ALB）の特徴とユースケース
  • Network Load Balancer（NLB）の特徴とユースケース
  • Gateway Load Balancer（GWLB）の特徴
  • ターゲットグループとヘルスチェック
  • スティッキーセッションとSSL終端

4.1 ロードバランサーの基本概念

ロードバランサーは、複数のサーバーに対してクライアントからのトラフィックを分散させるサービスです。高可用性を実現し、システムのスケーラビリティとパフォーマンスを向上させるために不可欠なコンポーネントです。

ロードバランサーの役割

トラフィック分散

複数のサーバーに負荷を均等に分散し、単一サーバーの過負荷を防ぎます。

ヘルスチェック

バックエンドサーバーの稼働状態を監視し、異常なサーバーへのトラフィックを自動的に停止します。

SSL/TLS終端

HTTPS通信の暗号化/復号化をロードバランサーで処理し、バックエンドサーバーの負荷を軽減します。

高可用性の実現

複数のアベイラビリティゾーンにまたがってトラフィックを分散し、障害時の影響を最小化します。

ELBの基本構成

4.2 Application Load Balancer（ALB）

ALBはOSI参照モデルの第7層（アプリケーション層）で動作するロードバランサーで、HTTP/HTTPSトラフィックの高度なルーティングが可能です。Webアプリケーションやマイクロサービスアーキテクチャに最適です。

ALBの主な機能

パスベースルーティング

URLパス（/api、/images など）に基づいて異なるターゲットグループにリクエストを振り分けられます。

example.com/api/* → APIサーバー群

example.com/images/* → 画像サーバー群

ホストベースルーティング

ホスト名（サブドメイン）に基づいてトラフィックを振り分けます。

api.example.com → APIサーバー群

www.example.com → Webサーバー群

WebSocketサポート

リアルタイム通信に必要なWebSocketプロトコルをネイティブサポートします。

SSL/TLS終端

ACM（AWS Certificate Manager）と統合し、SSL証明書の管理と自動更新が可能です。

ALBのルーティング例

flowchart LR U[ユーザー] --> ALB[Application
Load Balancer] ALB -->|"/api/*"| TG1[ターゲットグループ1
APIサーバー] ALB -->|"/images/*"| TG2[ターゲットグループ2
画像サーバー] ALB -->|"/"| TG3[ターゲットグループ3
Webサーバー] style ALB fill:#ffd,stroke:#fa0

ユースケース

マイクロサービスアーキテクチャ、コンテナベースアプリケーション、複数のWebアプリケーションを1つのドメインでホストする場合に最適です。

4.3 Network Load Balancer（NLB）

NLBはOSI参照モデルの第4層（トランスポート層）で動作するロードバランサーで、超低レイテンシと毎秒数百万リクエストの処理が可能です。TCP/UDP/TLSトラフィックに対応し、高性能が求められるアプリケーションに最適です。

NLBの特徴

超低レイテンシ

ミリ秒単位のレイテンシ

レイヤー4で動作するため、ALBよりも高速な処理が可能です。

静的IPアドレス

AZごとに固定IP割り当て

Elastic IPを使用して、各AZに静的IPアドレスを割り当て可能です。

高スループット

毎秒数百万リクエスト対応

急激なトラフィックの変動にも対応できる高いスケーラビリティを提供します。

送信元IP保持

クライアントIPそのまま

クライアントの送信元IPアドレスがバックエンドサーバーに届きます。

対応プロトコル

プロトコル ポート 用途
TCP 任意 汎用TCP通信
UDP 任意 ゲームサーバー、VoIPなど
TLS 443 暗号化TCP通信

ユースケース

ゲームサーバー、IoTアプリケーション、高頻度取引システム、VoIP、静的IPアドレスが必要なアプリケーションに最適です。

4.4 Gateway Load Balancer（GWLB）とロードバランサーの選択

Gateway Load Balancer（GWLB）は、サードパーティ製のネットワークアプライアンス（ファイアウォール、IDS/IPS、DPIなど）をデプロイ・管理するためのロードバランサーです。

GWLBの特徴

  • 仮想アプライアンス（ファイアウォール、IPS/IDSなど）の負荷分散
  • トラフィックの透過的な検査・処理
  • GENEVEプロトコルを使用したカプセル化

ロードバランサーの選択基準

GWLB レイヤー3 仮想アプライアンス、セキュリティ検査

4.5 ターゲットグループとヘルスチェック

ターゲットグループは、ロードバランサーがトラフィックを転送する先のリソース（EC2インスタンス、コンテナ、Lambdaなど）をまとめた論理的なグループです。

ターゲットタイプ

インスタンス

EC2インスタンスID

IPアドレス

プライベートIP（オンプレ連携可）

Lambda関数

サーバーレス統合（ALBのみ）

ヘルスチェックの設定

プロトコル: HTTP、HTTPS、TCP
パス: /health、/ping など
間隔: 5〜300秒（デフォルト30秒）
タイムアウト: 2〜120秒（デフォルト5秒）
正常しきい値: 2〜10回（デフォルト5回）
異常しきい値: 2〜10回（デフォルト2回）

注意

ヘルスチェックパスは、アプリケーションの本質的な動作確認ができるエンドポイントに設定しましょう。単にHTTP 200を返すだけでなく、データベース接続などの依存関係も確認するのが理想的です。

4.6 スティッキーセッションとクロスゾーン負荷分散

スティッキーセッション（セッションアフィニティ）

スティッキーセッションは、同一クライアントからのリクエストを常に同じターゲットに転送する機能です。ユーザーセッション情報をサーバー側で保持するアプリケーションで有用です。

ALBのスティッキーセッション

  • Duration-based cookies: ELBが生成するクッキー（AWSELB）
  • Application-based cookies: アプリケーションが生成するカスタムクッキー

クロスゾーン負荷分散

クロスゾーン負荷分散を有効にすると、ロードバランサーはすべてのアベイラビリティゾーンの登録済みターゲットに対してトラフィックを均等に分散します。

無効の場合

flowchart TD ELB[ELB] --> AZ1[AZ-A: 50%] ELB --> AZ2[AZ-B: 50%] AZ1 --> E1[EC2 25%] AZ1 --> E2[EC2 25%] AZ2 --> E3[EC2 50%] style E3 fill:#fdd,stroke:#f66

各AZに50%ずつ分散。AZ-Bのインスタンスに負荷集中。

有効の場合

flowchart TD ELB[ELB] --> E1[EC2 33%] ELB --> E2[EC2 33%] ELB --> E3[EC2 33%] style E1 fill:#dfd,stroke:#0a0 style E2 fill:#dfd,stroke:#0a0 style E3 fill:#dfd,stroke:#0a0

すべてのインスタンスに均等に分散。理想的な負荷分散。

デフォルト設定

  • ALB 常に有効（無効化不可）
  • NLB デフォルト無効（手動で有効化可能）
  • GWLB デフォルト無効（手動で有効化可能）

理解度確認クイズ

  1. Q1. URLパスに基づいて異なるターゲットグループにリクエストを振り分けたい場合、どのロードバランサーを選択すべきですか？

    回答を表示

    Application Load Balancer（ALB）です。ALBはレイヤー7で動作し、パスベースルーティングやホストベースルーティングなど、HTTPリクエストの詳細に基づいた高度なルーティングが可能です。

  2. Q2. 静的IPアドレスが必要で、超低レイテンシが求められるTCPアプリケーションに最適なロードバランサーは？

    回答を表示

    Network Load Balancer（NLB）です。NLBは各AZに静的IPアドレス（Elastic IP）を割り当て可能で、レイヤー4で動作するため超低レイテンシを実現します。

  3. Q3. クロスゾーン負荷分散が常に有効で無効化できないロードバランサーはどれですか？

    回答を表示

    Application Load Balancer（ALB）です。ALBは常にクロスゾーン負荷分散が有効で、無効化できません。NLBとGWLBはデフォルトでは無効ですが、手動で有効化できます。

この章のまとめ

  • ロードバランサーはトラフィック分散、ヘルスチェック、SSL終端、高可用性の実現を担う
  • ALB（レイヤー7）はWebアプリやマイクロサービスに最適で、パスベース・ホストベースルーティングが可能
  • NLB（レイヤー4）は超低レイテンシ・高スループット・静的IP対応で、高性能アプリに最適
  • GWLBはセキュリティアプライアンスの負荷分散に使用
  • ターゲットグループとヘルスチェックでバックエンドの健全性を監視
  • スティッキーセッションとクロスゾーン負荷分散で柔軟な負荷分散設計が可能
