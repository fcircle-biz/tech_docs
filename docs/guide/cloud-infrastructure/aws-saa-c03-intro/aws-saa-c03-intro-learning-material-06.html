Route 53 - DNS設計とトラフィックルーティング

Amazon Route 53を使用したDNS管理と、様々なルーティングポリシーを活用したトラフィック制御の設計を習得します。

この章で学ぶこと

  • DNSの基礎知識とRoute 53の役割
  • ホストゾーン（パブリック・プライベート）の違いと用途
  • レコードタイプ（A、AAAA、CNAME、Alias）の特徴
  • 6つのルーティングポリシーの使い分け
  • ヘルスチェックの設定とフェイルオーバー
  • Route 53とELBの連携パターン

6.1 DNSの基礎とRoute 53の役割

DNS（Domain Name System）は、人間が理解しやすいドメイン名（例: example.com）をIPアドレス（例: 192.0.2.1）に変換するシステムです。Route 53は、AWSが提供する高可用性でスケーラブルなDNSサービスで、ドメイン名の管理とトラフィックルーティングを統合的に提供します。

Route 53の主要機能

DNSサービス

権威DNSサーバーとして動作し、ドメイン名の名前解決を提供

ドメイン登録

新しいドメイン名の登録と既存ドメインの移管

ヘルスチェック

エンドポイントの正常性を監視し、自動フェイルオーバーを実現

トラフィックルーティング

複数のルーティングポリシーによる高度なトラフィック制御

DNS名前解決の流れ

sequenceDiagram autonumber participant U as ユーザー participant R as リゾルバ participant Root as ルートDNSサーバー participant TLD as TLDサーバー participant Auth as 権威DNSサーバー participant S as Webサーバー U->>R: www.example.com にアクセス R->>Root: .comのTLDサーバーを問い合わせ Root-->>R: .comのTLDサーバー情報 R->>TLD: example.comの権威DNSサーバーを問い合わせ TLD-->>R: example.comの権威DNSサーバー情報 R->>Auth: www.example.comのIPアドレスを問い合わせ Auth-->>R: 192.0.2.1 R-->>U: IPアドレスを返却 U->>S: 192.0.2.1にHTTPリクエスト

重要ポイント

Route 53は「権威DNSサーバー」として動作します。つまり、あなたが管理するドメイン名に対する「正式な回答」を提供する役割を担います。

6.2 ホストゾーンの種類

ホストゾーンは、ドメイン名とそのサブドメインのDNSレコードを管理するコンテナです。Route 53には2種類のホストゾーンがあります。

パブリックホストゾーン

パブリックホストゾーンの特徴

  • インターネット上からアクセス可能なドメインを管理
  • 外部ユーザーが企業のWebサイトにアクセスする際に使用
  • ドメイン名をRoute 53に登録または移管して使用

プライベートホストゾーン

プライベートホストゾーンの特徴

  • VPC内でのみ名前解決可能な内部ドメインを管理
  • 内部サービス間通信に使用（例: db.internal.example.com）
  • 複数のVPCに関連付け可能（クロスアカウント対応）

6.3 レコードタイプの種類と使い分け

DNSレコードは、ドメイン名とリソース情報を紐付ける情報です。Route 53では様々なレコードタイプをサポートしています。

レコードタイプ 用途 例
A ドメイン名をIPv4アドレスにマッピング example.com → 192.0.2.1
AAAA ドメイン名をIPv6アドレスにマッピング example.com → 2001:0db8::1
CNAME ドメイン名を別のドメイン名にマッピング www.example.com → example.com
Alias AWSリソースへのマッピング（Route 53独自） example.com → ALB
MX メールサーバーの指定 example.com → mail.example.com
TXT テキスト情報の記録（SPF、DKIM等） ドメイン認証情報など

Aliasレコード vs CNAMEレコード

Aliasレコードの利点

  • ゾーンAPEX（example.comなど）でも使用可能（CNAMEは不可）
  • Route 53の問い合わせ料金が無料
  • AWSリソース（ELB、CloudFront、S3など）の変更に自動追随

6.4 ルーティングポリシーの種類と選択

Route 53は6種類のルーティングポリシーを提供し、トラフィックの分散方法を柔軟に制御できます。

1. シンプルルーティング

特徴

1つのドメイン名に対して1つまたは複数のIPアドレスを返します。複数ある場合はランダムに選択されます。

ユースケース: 単一のリソースへの名前解決、最もシンプルな構成

2. 加重ルーティング

特徴

各リソースに重み（0-255）を割り当て、その比率でトラフィックを分散します。

ユースケース: A/Bテスト、カナリアリリース、段階的な新バージョン展開

例: 新旧システムの段階的移行

旧システム（重み: 80） / 新システム（重み: 20） → 80%/20%の割合でトラフィック分散

3. レイテンシーベースルーティング

特徴

ユーザーから最も低レイテンシーのAWSリージョンにトラフィックをルーティングします。

ユースケース: グローバルアプリケーションのパフォーマンス最適化

4. フェイルオーバールーティング

特徴

プライマリリソースが正常な場合はそちらに、障害時はセカンダリリソースに自動的に切り替えます。

ユースケース: アクティブ/スタンバイ構成、DR（災害復旧）

5. 位置情報ルーティング

特徴

ユーザーの地理的位置（国、大陸など）に基づいてトラフィックをルーティングします。

ユースケース: コンテンツローカライゼーション、法規制対応

例: 地域別コンテンツ配信

日本からのアクセス → 東京リージョン（日本語サイト）
米国からのアクセス → バージニアリージョン（英語サイト）

6. 地理的近接性ルーティング

特徴

リソースとユーザーの地理的距離に基づいてルーティングします。バイアス値で調整可能。

ユースケース: トラフィックの地域的シフト、リソース利用の最適化

ヒント: ルーティングポリシーの選択基準

  • シンプル: 単一リソース、テスト環境
  • 加重: A/Bテスト、段階的移行
  • レイテンシー: グローバル展開、性能重視
  • フェイルオーバー: 高可用性、DR
  • 位置情報: コンプライアンス、ローカライズ
  • 地理的近接性: トラフィック最適化

6.5 ヘルスチェックとフェイルオーバー

ヘルスチェックは、エンドポイントの正常性を定期的に監視し、異常時に自動的にトラフィックを切り替える仕組みです。

ヘルスチェックの種類

エンドポイント監視

WebサーバーやAPIエンドポイントの応答を監視

他ヘルスチェック監視

複数のヘルスチェックを組み合わせて判定

CloudWatchアラーム監視

CloudWatchメトリクスに基づいて判定

ヘルスチェックの設定項目

プロトコル

HTTP、HTTPS、TCP

パス

監視対象のURL（例: /health）

間隔

30秒または10秒（高速）

失敗しきい値

異常と判定するまでの連続失敗回数（デフォルト: 3回）

stateDiagram-v2 [*] --> Healthy: ヘルスチェック開始 Healthy --> Checking: チェック実行 Checking --> Healthy: 成功 Checking --> FailCount: 失敗 FailCount --> Checking: 次回チェック FailCount --> Unhealthy: しきい値超過 Unhealthy --> Checking: チェック実行 Checking --> Unhealthy: 失敗継続 Checking --> Recovering: 成功 Recovering --> Healthy: しきい値回復 Healthy --> [*]: 監視終了

注意

ヘルスチェックはグローバルに分散した複数のロケーションから実行されます。セキュリティグループやファイアウォールでRoute 53のヘルスチェッカーIPアドレスを許可する必要があります。

Aliasレコードを使用すると、ELBのIPアドレス変更に自動追随し、Route 53の問い合わせ料金も無料になります。必ずAliasレコードを使用しましょう。

実習 6-1: フェイルオーバールーティングの設定

Route 53でフェイルオーバールーティングを設定し、プライマリサーバーの障害時に自動的にセカンダリサーバーに切り替わることを確認します。

前提条件

  • 2つの異なるリージョンにEC2インスタンスが起動している
  • ドメイン名を所有しているか、Route 53でテスト用ドメインを登録済み

手順

  1. 1 Route 53コンソールでホストゾーンを開き、「レコードを作成」をクリック
  2. 2 レコード名に「www」、レコードタイプに「A」を選択
  3. 3 「ルーティングポリシー」で「フェイルオーバー」を選択
  4. 4 プライマリレコード: 東京リージョンのEC2インスタンスのパブリックIP、フェイルオーバータイプ「プライマリ」
  5. 5 ヘルスチェックを作成し、プライマリレコードに関連付け
  6. 6 セカンダリレコード: 大阪リージョンのEC2インスタンスのパブリックIP、フェイルオーバータイプ「セカンダリ」
  7. 7 ブラウザでwww.example.comにアクセスし、プライマリサーバーのコンテンツが表示されることを確認
  8. 8 プライマリEC2インスタンスを停止し、数分待機後、セカンダリサーバーに自動切り替わることを確認

理解度確認クイズ

  1. Q1. AliasレコードとCNAMEレコードの違いで正しいのはどれですか？

    回答を表示

    AliasレコードはゾーンAPEX（example.comなど）でも使用でき、Route 53の問い合わせ料金が無料です。また、AWSリソースのIPアドレス変更に自動追随します。CNAMEはゾーンAPEXでは使用できません。

  2. Q2. グローバルにサービスを展開し、ユーザーに最も低レイテンシーのリージョンに接続させたい場合、どのルーティングポリシーを使用すべきですか？

    回答を表示

    「レイテンシーベースルーティング」を使用します。これにより、ユーザーから最も低レイテンシーのAWSリージョンに自動的にトラフィックがルーティングされ、パフォーマンスが最適化されます。

  3. Q3. プライベートホストゾーンの用途として正しいものはどれですか？

    回答を表示

    プライベートホストゾーンは、VPC内でのみ名前解決可能な内部ドメインを管理するために使用します。例えば、db.internal.example.comのような内部サービス間通信に使用され、外部からはアクセスできません。
