データベースサービスの選択と設計

ユースケースに応じた最適なデータベースサービスの選択と、高可用性・高性能なデータベース設計を学びます。

この章で学ぶこと

  • RDSのマルチAZデプロイとリードレプリカ
  • Amazon Aurora の特徴と利点
  • DynamoDB の基本概念とユースケース
  • ElastiCacheの活用
  • データベース選択の判断基準

8.1 データベースサービスの全体像

AWSは、用途に応じて最適化された多様なデータベースサービスを提供しています。適切なデータベースを選択することが、アプリケーションのパフォーマンス・可用性・コストに大きく影響します。

データベース選択の基本原則

「1つのデータベースですべてを解決」ではなく、ユースケースごとに最適なデータベースを選択する「Purpose-built database」の考え方が重要です。

8.2 Amazon RDS - マネージドリレーショナルデータベース

Amazon RDSは、MySQL、PostgreSQL、Oracle、SQL Serverなどの主要なリレーショナルデータベースをマネージドサービスとして提供します。パッチ適用、バックアップ、高可用性設定などの運用管理をAWSが自動化します。

RDSマルチAZデプロイ

マルチAZ配置は、高可用性を実現するための構成です。プライマリDBインスタンスのデータを、別のアベイラビリティゾーン（AZ）にあるスタンバイレプリカに同期レプリケーションします。

メリット

  • 自動フェイルオーバー（1-2分）
  • データ損失なし（同期レプリケーション）
  • 自動バックアップ作成

注意点

  • スタンバイは読み取り不可
  • コストは約2倍
  • 同一リージョン内のみ

RDSリードレプリカ

リードレプリカは、読み取り専用のコピーを作成することで、読み取り負荷を分散させる仕組みです。非同期レプリケーションを使用します。

flowchart LR App[アプリケーション] Primary[(プライマリDB)] RR1[(リードレプリカ1)] RR2[(リードレプリカ2)] App -->|書き込み| Primary App -->|読み取り| RR1 App -->|読み取り| RR2 Primary -.->|非同期レプリケーション| RR1 Primary -.->|非同期レプリケーション| RR2

重要ポイント

リードレプリカは別リージョンに配置できます（クロスリージョンレプリカ）。これにより、ディザスタリカバリや地理的に近い場所からの読み取り性能向上が可能です。

8.3 Amazon Aurora - AWS最適化されたデータベース

Amazon Auroraは、MySQL・PostgreSQLと互換性があり、クラウド向けに設計されたAWS独自のデータベースエンジンです。標準的なMySQLやPostgreSQLの5倍以上のスループットを実現します。

Auroraの特徴

高性能

MySQL/PostgreSQLの5倍以上のスループット。分散ストレージによる自動スケーリング

高可用性

データを3つのAZに6箇所コピー。自動フェイルオーバー（30秒以内）

自動スケーリング

ストレージは10GBから128TBまで自動拡張。最大15個のリードレプリカ

バックアップ

自動バックアップ（常時有効）。35日間保持。ポイントインタイムリカバリ対応

Auroraのユースケース

Auroraが適している場合

  • ミッションクリティカルなアプリケーション（金融、ECなど）
  • 高いスループットが必要（数万トランザクション/秒）
  • ダウンタイムを最小化したい（99.99%可用性）
  • データベース管理の運用負荷を削減したい

8.4 Amazon DynamoDB - マネージドNoSQLデータベース

DynamoDBは、フルマネージドなキーバリュー型・ドキュメント型NoSQLデータベースです。ミリ秒単位の低レイテンシで、無制限にスケールできます。

DynamoDBの料金モデル

オンデマンドモード

  • 使用量に応じた従量課金
  • キャパシティプランニング不要
  • 予測不可能なワークロード向き

プロビジョンドモード

  • 読み書きキャパシティを事前設定
  • 予測可能な負荷に最適
  • オンデマンドより低コスト

重要ポイント

DynamoDBはサーバーレスアーキテクチャと非常に相性が良く、Lambda、API Gateway、S3と組み合わせることで、フルマネージドなWebアプリケーションを構築できます。

8.5 Amazon ElastiCache - インメモリキャッシュ

ElastiCacheは、RedisまたはMemcachedをマネージドサービスとして提供します。頻繁にアクセスされるデータをメモリ上にキャッシュすることで、データベースの負荷を軽減し、レスポンス時間を大幅に短縮します。

RedisとMemcachedの比較

機能 Redis Memcached
データ構造 文字列、リスト、セット、ハッシュ 文字列のみ
永続化 対応（スナップショット、AOF） 非対応
レプリケーション 対応（マルチAZ） 非対応
ユースケース 高度なデータ構造、Pub/Sub、永続化 シンプルなキャッシュ、水平スケール

ユースケース別推奨データベース

ECサイト（注文管理）

推奨: Amazon Aurora（MySQL互換）+ ElastiCache（セッション管理）

理由: トランザクション整合性が必須。高可用性と高性能が求められる。

モバイルアプリ（ユーザープロファイル）

推奨: DynamoDB

理由: キーバリューアクセス、無制限スケール、低レイテンシ、グローバルテーブル対応。

ゲームリーダーボード

推奨: ElastiCache for Redis

理由: ソート済みセットでリアルタイムランキング実装。ミリ秒レスポンス。

コンテンツ管理システム（CMS）

推奨: RDS（PostgreSQL）

理由: 複雑なクエリ、JOIN処理、トランザクション必要。標準的な性能で十分。

8.7 バックアップとリストア戦略

データベースのバックアップ戦略は、RPO（Recovery Point Objective：目標復旧時点）とRTO（Recovery Time Objective：目標復旧時間）の要件に基づいて設計します。

RDSのバックアップオプション

自動バックアップ

  • 日次自動スナップショット
  • トランザクションログ保存（5分ごと）
  • 保持期間: 0-35日（デフォルト7日）
  • ポイントインタイムリカバリ対応

手動スナップショット

  • ユーザーが任意のタイミングで作成
  • 削除されるまで永続的に保持
  • メンテナンス前に推奨
  • 別リージョンへのコピー可能

注意

自動バックアップを無効化（保持期間0日）すると、既存のバックアップもすべて削除されます。本番環境では必ず自動バックアップを有効化してください。

理解度確認クイズ

  1. Q1. RDSマルチAZとリードレプリカの違いは何ですか？

    回答を表示

    マルチAZは高可用性を目的とした同期レプリケーションで、スタンバイは読み取り不可。リードレプリカは読み取りスケーリングを目的とした非同期レプリケーションで、読み取り専用アクセスが可能です。

  2. Q2. Amazon Auroraの主な利点を3つ挙げてください。

    回答を表示

    1) 標準MySQLの5倍以上の性能、2) 3つのAZに6箇所のデータコピーによる高可用性、3) 最大128TBまでの自動ストレージスケーリング

  3. Q3. DynamoDBのプライマリキーに使用できる2つの種類は何ですか？

    回答を表示

    1) シンプルプライマリキー（パーティションキーのみ）、2) 複合プライマリキー（パーティションキー + ソートキー）

この章のまとめ

  • RDSマルチAZは高可用性、リードレプリカは読み取りスケーリングを実現
  • Auroraはクラウド最適化されたRDBMSで、標準MySQLの5倍以上の性能と高可用性を提供
  • DynamoDBはキーバリュー型NoSQLで、無制限スケールと低レイテンシを実現
  • ElastiCacheはRedis/Memcachedによるインメモリキャッシュで、DB負荷軽減と高速化に貢献
  • ユースケースに応じた適切なデータベース選択が、性能・コスト・運用性に大きく影響
