高可用性とスケーラビリティの基礎

システムの可用性とスケーラビリティを確保するための基本概念と、AWSのリージョン・アベイラビリティゾーン（AZ）を活用した設計原則を学びます。

この章で学ぶこと

  • 高可用性（HA）の定義と測定指標
  • SLA、RTO、RPOの概念
  • 単一障害点（SPOF）の特定と排除
  • リージョンとアベイラビリティゾーン（AZ）
  • マルチAZ構成の設計原則
  • 水平スケーリングと垂直スケーリング

3.1 高可用性（HA）とは

高可用性（High Availability, HA）とは、システムが長時間にわたって継続的に稼働する能力のことです。AWSでは、複数のアベイラビリティゾーン（AZ）を活用することで、ハードウェア障害やデータセンターの障害に対する耐性を持たせることができます。

可用性の測定指標

SLA（Service Level Agreement）とアップタイム

可用性は通常、年間のアップタイム（稼働時間）のパーセンテージで表現されます。

可用性 年間ダウンタイム 月間ダウンタイム
99% 3.65日 7.2時間
99.9%（スリーナイン） 8.76時間 43.2分
99.99%（フォーナイン） 52.56分 4.32分
99.999%（ファイブナイン） 5.26分 25.9秒

RTO と RPO

災害復旧計画では、以下の2つの重要な指標を定義します。

RTO（Recovery Time Objective）

目標復旧時間

障害発生から復旧までに許容される最大時間。例: 「4時間以内にサービスを復旧する」

RPO（Recovery Point Objective）

目標復旧時点

障害発生時に失われても許容できるデータ量（時間）。例: 「1時間分のデータ損失まで許容」

3.2 単一障害点（SPOF）の排除

単一障害点（Single Point of Failure, SPOF）とは、それが故障するとシステム全体が停止してしまう要素のことです。高可用性を実現するには、SPOFを特定し、冗長化によって排除することが重要です。

SPOFの例と対策

各コンポーネントを冗長化し、1つが故障してもサービス継続可能。

注意: ネットワーク経路もSPOFになりうる

インターネット接続経路、VPN接続、Direct Connectなど、ネットワーク経路も複数確保することが重要です。

3.3 リージョンとアベイラビリティゾーン（AZ）

AWSのグローバルインフラストラクチャは、リージョンとアベイラビリティゾーン（AZ）の2層構造で設計されています。この構造を理解することが、高可用性設計の基礎となります。

リージョン

リージョンとは

地理的に分離された場所にある、複数のデータセンターの集合体です。各リージョンは完全に独立しており、他のリージョンと物理的に分離されています。

  • 例: 東京リージョン（ap-northeast-1）、バージニア北部（us-east-1）
  • 各リージョンは最低2つ以上のAZで構成
  • データはリージョン間で自動複製されない（意図的な分離）

アベイラビリティゾーン（AZ）

アベイラビリティゾーン（AZ）とは

1つ以上のデータセンターから構成される、リージョン内の物理的に分離された場所です。各AZは独立した電源、冷却、ネットワークを持ちます。

  • 東京リージョンには4つのAZ（ap-northeast-1a, 1b, 1c, 1d）
  • AZ間は低レイテンシの専用ネットワークで接続
  • 1つのAZで障害が発生しても、他のAZは影響を受けない

重要ポイント: リージョン選択の考慮事項

  • レイテンシ: ユーザーに近いリージョンを選択
  • コンプライアンス: データ保存場所の法的要件
  • サービス可用性: 全てのサービスが全リージョンで利用可能とは限らない
  • 料金: リージョンによって価格が異なる

3.4 マルチAZ構成の設計原則

マルチAZ構成とは、複数のアベイラビリティゾーン（AZ）にリソースを分散配置することで、1つのAZで障害が発生してもサービスを継続できるようにする設計パターンです。

AWSサービスごとのマルチAZ対応

サービス マルチAZ対応 設定方法
ELB 自動 作成時に複数AZを選択
RDS オプション Multi-AZデプロイメントを有効化
EC2 手動 各AZでインスタンス起動
S3 自動 最低3つのAZに自動複製
DynamoDB 自動 リージョン内の複数AZに自動複製

ヒント: コストとのバランス

マルチAZ構成はコストが増加します（例: RDS Multi-AZは約2倍）。システムの重要度とコストのバランスを考慮して設計しましょう。開発環境ではシングルAZ、本番環境ではマルチAZというように使い分けることが一般的です。

3.5 スケーラビリティの種類

スケーラビリティとは、需要の変化に応じてシステムの処理能力を調整できる能力のことです。AWSでは、垂直スケーリングと水平スケーリングの2つの方法があります。

垂直スケーリング（Scale Up/Down）

インスタンスサイズの変更

概要:

既存のインスタンスのスペック（CPU、メモリ）を増減させる方法

メリット:

  • 実装がシンプル
  • アプリケーションの変更不要

デメリット:

  • 上限がある（最大インスタンスサイズまで）
  • ダウンタイムが発生する
  • SPOFが残る

水平スケーリング（Scale Out/In）

インスタンス数の増減

概要:

インスタンスの台数を増減させる方法（Auto Scalingなど）

メリット:

  • ほぼ無制限にスケール可能
  • 高可用性を実現できる
  • ダウンタイムなし

デメリット:

  • アプリケーションの設計が必要
  • セッション管理などの考慮が必要

重要ポイント: SAA-C03試験での選択基準

試験では「高可用性」「スケーラビリティ」が求められる場合、ほぼ常に水平スケーリングが正解です。垂直スケーリングは一時的な対処として使用しますが、長期的なソリューションとしては推奨されません。

理解度確認クイズ

  1. Q1. 可用性99.99%のシステムで許容される年間ダウンタイムは何時間ですか？

    回答を表示

    約52.56分（≒0.88時間）です。99.99%（フォーナイン）は非常に高い可用性で、年間1時間未満のダウンタイムしか許容されません。

  2. Q2. RTO（Recovery Time Objective）とRPO（Recovery Point Objective）の違いを説明してください。

    回答を表示

    RTOは「目標復旧時間」で、障害からサービスを復旧するまでの許容時間。RPOは「目標復旧時点」で、失われても許容できるデータ量（時間）です。例: RTO 4時間、RPO 1時間の場合、4時間以内に復旧し、最大1時間分のデータ損失は許容する、という意味です。

  3. Q3. 東京リージョン（ap-northeast-1）には何個のアベイラビリティゾーン（AZ）がありますか？

    回答を表示

    4つのAZ（ap-northeast-1a, 1b, 1c, 1d）があります。マルチAZ構成では、最低でも2つのAZにリソースを配置することが推奨されます。

  4. Q4. 垂直スケーリングと水平スケーリングのうち、高可用性とスケーラビリティの両方を実現できるのはどちらですか？

    回答を表示

    水平スケーリング（Scale Out/In）です。インスタンス数を増やすことで、ほぼ無制限にスケール可能で、かつ複数インスタンスで冗長化できるため高可用性も実現できます。垂直スケーリングは上限があり、SPOFが残ります。

この章のまとめ

  • 高可用性（HA）は、SLA、RTO、RPOなどの指標で測定される
  • 単一障害点（SPOF）を特定し、冗長化によって排除することが重要
  • AWSのグローバルインフラはリージョンとアベイラビリティゾーン（AZ）の2層構造
  • マルチAZ構成により、1つのAZで障害が発生してもサービス継続可能
  • 垂直スケーリング（Scale Up）より水平スケーリング（Scale Out）が推奨
  • ELB、S3、DynamoDBは自動的にマルチAZ対応、RDSはオプション、EC2は手動設定
