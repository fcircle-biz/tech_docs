AWSにおけるセキュリティのベストプラクティスとアイデンティティ管理を習得します。IAM、KMS、Secrets Manager、CloudTrail、GuardDutyなどを活用したセキュアなアーキテクチャ設計を学びます。

10.1 IAMのベストプラクティス

AWS Identity and Access Management（IAM）は、AWSリソースへのアクセスを安全に制御するためのサービスです。適切なIAM設定はセキュアなアーキテクチャの基盤となります。

最小権限の原則（Principle of Least Privilege）

ユーザーやサービスには、タスクを実行するために必要な最小限の権限のみを付与します。これにより、セキュリティ侵害時の影響範囲を最小化できます。

重要ポイント

最初は必要最小限の権限から始め、必要に応じて段階的に追加していくアプローチを取りましょう。広範な権限を付与してから削減する方法は、既存機能への影響が大きく推奨されません。

IAMポリシーの種類

アイデンティティベースポリシー

  • IAMユーザー、グループ、ロールに添付
  • 管理ポリシーとインラインポリシー

リソースベースポリシー

  • S3バケット、SNSトピックなどに添付
  • クロスアカウントアクセスの許可

IAMポリシー例: S3読み取り専用アクセス

S3読み取り専用ポリシー 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ]
    }
  ]
}

10.2 IAMロールとクロスアカウントアクセス

IAMロールは、一時的な認証情報を使用してAWSリソースにアクセスするためのメカニズムです。特にEC2インスタンスやLambda関数、クロスアカウントアクセスで重要な役割を果たします。

IAMロールのユースケース

EC2インスタンスロール

EC2インスタンスからS3やDynamoDBにアクセスする際、認証情報をハードコードせずに安全にアクセス

Lambda実行ロール

Lambda関数がCloudWatch Logs、DynamoDB、S3などのサービスにアクセス

クロスアカウントアクセス

アカウントA（本番環境）のリソースにアカウントB（開発環境）からアクセス

クロスアカウントアクセスの設計

sequenceDiagram participant U as ユーザー（開発アカウント） participant STS as AWS STS participant R as IAMロール（本番アカウント） participant S3 as S3バケット（本番アカウント） U->>STS: AssumeRole リクエスト STS->>R: ロール権限確認 R-->>STS: 権限確認OK STS-->>U: 一時認証情報（15分～12時間） U->>S3: 一時認証情報でアクセス S3-->>U: データ取得

ヒント: AssumeRoleとは

AssumeRoleは、IAMロールを引き受けて一時的な認証情報を取得するAPI操作です。これにより、長期的な認証情報（アクセスキー）を使わずに安全にリソースにアクセスできます。

10.3 AWS OrganizationsとSCP

AWS Organizationsは、複数のAWSアカウントを一元管理するサービスです。Service Control Policy（SCP）を使用して、組織内のアカウントに対する権限の上限を設定できます。

SCPの動作原理

1

権限の上限設定

IAMポリシーで許可されていても、SCPで拒否されていれば実行不可

2

継承

OU（組織単位）に適用されたSCPは、配下の全アカウントに継承される

3

ルートユーザーも制限

SCPはアカウントのルートユーザーにも適用される（IAMポリシーとの違い）

SCP例: 特定リージョン以外での操作を禁止

リージョン制限SCP 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Deny",
      "Action": "*",
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "ap-northeast-1",
            "us-east-1"
          ]
        }
      }
    }
  ]
}

10.4 暗号化とキー管理（KMS）

AWS Key Management Service（KMS）は、データ暗号化に使用する暗号化キーの作成と管理を簡素化するマネージドサービスです。

KMSの主要概念

カスタマーマスターキー（CMK）

  • AWS管理のCMK: AWSサービスが自動作成
  • カスタマー管理のCMK: ユーザーが作成・管理

エンベロープ暗号化

  • データキーでデータを暗号化
  • マスターキーでデータキーを暗号化
sequenceDiagram participant App as アプリケーション participant KMS as AWS KMS participant S3 as Amazon S3 App->>KMS: データキー生成リクエスト KMS-->>App: 平文データキー + 暗号化データキー App->>App: 平文データキーでデータ暗号化 App->>S3: 暗号化データ + 暗号化データキー保存 App->>App: 平文データキーをメモリから削除 Note over App,S3: --- 復号時 --- S3-->>App: 暗号化データ + 暗号化データキー取得 App->>KMS: 暗号化データキーの復号リクエスト KMS-->>App: 平文データキー App->>App: 平文データキーでデータ復号

S3暗号化オプション

暗号化方式 キー管理 ユースケース
SSE-S3 AWS管理 シンプルな暗号化
SSE-KMS KMS（カスタマー管理可） 監査ログ必要
SSE-C 顧客管理 独自キー管理

10.5 シークレット管理（Secrets ManagerとParameter Store）

データベース認証情報やAPIキーなどの機密情報を安全に保存・取得するための2つのサービスを比較します。

Secrets Manager vs Parameter Store

機能 Secrets Manager Parameter Store
自動ローテーション ⭕ 対応 ❌ 非対応
料金 シークレットごとに課金 基本無料（高度なパラメータは有料）
推奨用途 DBパスワード、APIキー 設定値、非機密情報
サイズ制限 64KB 8KB（高度: 8KB）

選択の基準

自動ローテーションが必要なRDSパスワードや機密度の高いAPIキーはSecrets Manager、アプリケーション設定やエンドポイントURLなどはParameter Storeを使用するのが一般的です。

10.6 DDoS対策（AWS ShieldとAWS WAF）

AWS ShieldとAWS WAFは、DDoS攻撃やWebアプリケーションへの攻撃からリソースを保護するサービスです。

AWS Shieldの2つのレベル

Shield Standard

無料・自動有効

  • レイヤー3/4のDDoS保護
  • CloudFront、Route 53で利用可能

Shield Advanced

月額3,000ドル

  • 高度なDDoS保護とリアルタイム通知
  • DDoS Response Team（DRT）サポート
  • コスト保護（DDoS攻撃時のスケーリング費用補償）

AWS WAF（Web Application Firewall）

AWS WAFは、Webアプリケーションを一般的な攻撃（SQLインジェクション、XSSなど）から保護するファイアウォールです。

WAFルールの種類

  • IPセットルール: 特定IPアドレスのブロック/許可
  • 地理的マッチング: 特定国からのアクセス制限
  • レートベースルール: 5分間に2,000リクエスト超過でブロック
  • マネージドルール: AWSやサードパーティ提供のルールセット

10.7 監査とコンプライアンス（CloudTrail）

AWS CloudTrailは、AWSアカウントのガバナンス、コンプライアンス、運用監査、リスク監査を可能にするサービスです。すべてのAPI呼び出しをログとして記録します。

CloudTrailの主要機能

イベント履歴

過去90日間のAPIアクティビティを自動記録（無料）

証跡（Trail）

ログをS3バケットに長期保存、CloudWatch Logsに送信

ログファイル検証

ログの改ざん検知のためのデジタル署名

CloudTrailイベントの種類

イベントタイプ 説明 例
管理イベント リソース操作 EC2インスタンス起動、IAMユーザー作成
データイベント データアクセス S3オブジェクトの取得、Lambda関数実行
Insightsイベント 異常検知 通常より多いEC2起動アクティビティ

注意事項

CloudTrailは証跡を作成しない限り、90日を超えるログは自動削除されます。コンプライアンス要件で長期保存が必要な場合は、必ず証跡を作成してS3に保存してください。

10.8 脅威検知（Amazon GuardDuty）

Amazon GuardDutyは、機械学習と脅威インテリジェンスを使用して、AWSアカウントとワークロードを継続的に監視する脅威検出サービスです。

GuardDutyが分析するデータソース

VPC Flow Logs

ネットワークトラフィックの異常検知

DNS Logs

不審なドメインへのクエリ検知

CloudTrail Logs

不正なAPI呼び出し検知

GuardDutyの検出例

UnauthorizedAccess:EC2/MaliciousIPCaller

既知の悪意あるIPからEC2インスタンスへのアクセス

CryptoCurrency:EC2/BitcoinTool

EC2インスタンスでの仮想通貨マイニング検知

Stealth:IAMUser/PasswordPolicyChange

パスワードポリシーの弱体化

flowchart TD A[VPC Flow Logs] --> D[Amazon GuardDuty] B[DNS Logs] --> D C[CloudTrail Logs] --> D D -->|機械学習分析| E{脅威検出} E -->|検出あり| F[EventBridge] F --> G[Lambda関数] F --> H[SNS通知] G --> I[自動対処
セキュリティグループ更新等]

自動対処の実装

GuardDutyの検出結果をEventBridgeでキャッチし、Lambda関数で自動対処（該当インスタンスの隔離、セキュリティグループの更新など）を実装できます。

実習: IAMロールとCloudTrailの設定

ステップ1: EC2インスタンスロールの作成

  1. 1. IAMコンソール → ロール → ロールを作成
  2. 2. 信頼されたエンティティタイプ: AWS サービス → EC2
  3. 3. 許可ポリシー: AmazonS3ReadOnlyAccess を選択
  4. 4. ロール名: EC2-S3-ReadOnly-Role

ステップ2: CloudTrail証跡の作成

  1. 1. CloudTrailコンソール → 証跡の作成
  2. 2. 証跡名: my-security-trail
  3. 3. S3バケット: 新規作成または既存バケット選択
  4. 4. CloudWatch Logs: 有効化（オプション）
  5. 5. 管理イベントとデータイベント（S3）を記録

ステップ3: GuardDutyの有効化

  1. 1. GuardDutyコンソール → 今すぐ始める
  2. 2. GuardDutyを有効にする
  3. 3. 検出結果を確認（サンプル検出結果の生成）

理解度確認クイズ

  1. Q1. IAMポリシーで明示的にDenyされているアクションは、他のポリシーでAllowされていても実行できません。この原則を何と呼びますか？

    回答を表示

    「明示的なDenyは常に優先される（Explicit Deny always wins）」原則です。セキュリティを最優先するため、Denyは他のすべてのAllowよりも優先されます。

  2. Q2. RDSのパスワードを30日ごとに自動ローテーションしたい場合、Secrets ManagerとParameter Storeのどちらを使うべきですか？

    回答を表示

    Secrets Managerです。Parameter Storeは自動ローテーション機能を持たないため、定期的なパスワード更新が必要な場合はSecrets Managerを使用します。

  3. Q3. CloudTrailのイベント履歴は無料で何日間保存されますか？

    回答を表示

    90日間です。それ以上保存したい場合は、証跡を作成してS3バケットにログを保存する必要があります。
