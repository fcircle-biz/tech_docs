アクセス権限を管理しよう - IAM入門

「誰が何をできるか」を管理する仕組みを学びます。会社でいう「入館証」や「アクセス権限」のようなものです。

この章で学ぶこと

  • IAMは「AWSの入館証システム」
  • ルートユーザーは「社長」（普段は使わない）
  • IAMユーザーは「社員証を持った従業員」
  • グループは「部署」（まとめて権限を設定）
  • ポリシーは「できることリスト」
  • MFA（多要素認証）でセキュリティ強化

8.1 IAMとは？

IAM（Identity and Access Management）は、AWSリソースへのアクセスを安全に管理するためのサービスです。 「誰が（Who）」「何を（What）」「どのようにできるか（How）」を細かく設定できます。

身近な例え

IAM = オフィスビルのセキュリティシステム
ルートユーザー = ビルオーナー（全フロアに入れる）
IAMユーザー = 従業員（自分のフロアだけ入れる）
ポリシー = 入館証に設定された「入れる部屋リスト」

IAMを使うことで、チームメンバーごとに適切な権限を与え、セキュリティを高めることができます。

8.2 ルートユーザーとIAMユーザーの違い

AWSには2種類のユーザーがあります。ルートユーザーとIAMユーザーです。

ルートユーザー

AWSアカウントを作成したときに最初に作られるユーザーです。すべての権限を持つ「社長」のような存在です。

重要な注意事項

ルートユーザーは日常的に使わないでください！
乗っ取られると全てのリソースが危険にさらされます。アカウント設定の変更など、どうしても必要な時だけ使います。

IAMユーザー

日常業務で使うための「従業員アカウント」です。必要な権限だけを与えることができます。

ルートユーザー vs IAMユーザー

項目 ルートユーザー IAMユーザー
権限 すべて（無制限） 必要なものだけ
使用頻度 月1回程度 毎日
ログイン方法 メールアドレス ユーザー名
MFA推奨度 必須 強く推奨

ベストプラクティス

アカウント作成後、すぐにIAMユーザーを作成し、普段はそちらを使いましょう。ルートユーザーには必ずMFA（多要素認証）を設定してください。

8.3 IAMの主要コンポーネント

IAMは4つの主要な要素で構成されています。

1. ユーザー

個人を表すアカウント。それぞれ独自のログイン情報を持つ。

2. グループ

ユーザーの集まり。部署やチームごとにまとめて権限を管理。

3. ポリシー

権限の定義書。「何ができるか/できないか」を記述したルール。

4. ロール

一時的な権限セット。サービス間の連携やアプリケーションで使用。

権限の付与パターン

ヒント

ユーザーに直接ポリシーをつけるより、グループに入れて管理する方が便利です。人が増えても楽に管理できます。

8.4 IAMユーザーを作成しよう

実際にIAMユーザーを作成してみましょう。

実習 8-1: 管理者権限を持つIAMユーザーを作成

普段使いのIAMユーザーを作成し、管理者権限を付与します。

手順

  1. 1 AWSマネジメントコンソールにルートユーザーでログイン
  2. 2 検索バーに「IAM」と入力し、IAMサービスを開く
  3. 3 左メニューから「ユーザー」を選択 → 「ユーザーを追加」ボタンをクリック
  4. 4 ユーザー名を入力（例: admin-user）
    「AWSマネジメントコンソールへのユーザーアクセスを提供する」にチェック
  5. 5 「カスタムパスワード」を選択し、複雑なパスワードを設定
    「次へ」をクリック
  6. 6 「ポリシーを直接アタッチする」を選択
    「AdministratorAccess」を検索してチェック → 「次へ」
  7. 7 内容を確認し「ユーザーの作成」をクリック
  8. 8 表示されるログイン情報（アカウントID、ユーザー名、パスワード）を安全な場所にメモ

つまずきポイント

  • AdministratorAccessは強力な権限。学習用途だけで使い、本番環境では最小権限の原則を守る
  • ログイン情報は一度しか表示されないので必ずメモする
  • IAMユーザーのログインURLは通常のAWSログインとは異なる（専用URLが発行される）

8.5 グループとポリシーで権限を管理

複数のユーザーを効率的に管理するには、グループを使います。

よく使うポリシーの例

AdministratorAccess

すべてのAWSサービスとリソースへのフルアクセス。社長レベル。

ReadOnlyAccess

すべてのリソースを「見る」だけ。変更や削除はできない。

AmazonEC2FullAccess

EC2（サーバー）の完全な操作権限。起動・停止・削除が可能。

AmazonS3ReadOnlyAccess

S3のファイルを読むだけ。アップロードや削除はできない。

最小権限の原則

「必要最小限の権限だけを与える」ことがセキュリティの基本です。全員に管理者権限を与えるのは避けましょう。

8.6 MFA（多要素認証）でセキュリティ強化

MFA（Multi-Factor Authentication）は、パスワードに加えて「もう1つの認証手段」を使う仕組みです。 スマートフォンのアプリなどで生成される6桁のコードを入力します。

身近な例え

MFA = 銀行のキャッシュカード＋暗証番号
パスワードだけ（1つ） → カードだけでは引き出せない
MFA（2つ） → カード＋暗証番号で安全性UP

MFAの仕組み

ステップ1

パスワードを入力（知っているもの）

ステップ2

スマホで6桁コード生成（持っているもの）

ステップ3

コードを入力してログイン成功

MFAアプリの例

  • Google Authenticator - iPhoneとAndroid両方で使える無料アプリ
  • Microsoft Authenticator - Microsoftアカウントとも連携可能
  • Authy - 複数デバイスで同期できる

注意

MFAを設定したら、スマートフォンの紛失・故障に備えて「バックアップコード」を印刷して安全な場所に保管してください。

理解度確認クイズ

  1. Q1. ルートユーザーとIAMユーザーの主な違いは何ですか？

    回答を表示

    ルートユーザーはすべての権限を持ち、アカウント作成時に自動で作られる。IAMユーザーは必要な権限だけを持つ日常使い用のアカウント。セキュリティのため、普段はIAMユーザーを使うべき。

  2. Q2. グループを使うメリットは何ですか？

    回答を表示

    複数のユーザーにまとめて権限を設定できる。新しいメンバーが増えても、グループに追加するだけで同じ権限を付与できるため管理が楽になる。

  3. Q3. MFA（多要素認証）を設定すべき理由は何ですか？

    回答を表示

    パスワードが漏洩しても、スマートフォンがないとログインできないため、アカウントの乗っ取りを防げる。特にルートユーザーには必須。

この章のまとめ

  • IAMはAWSの「入館証システム」。誰が何をできるかを細かく管理できる。
  • ルートユーザーは全権限を持つが日常的には使わない。IAMユーザーを作って普段はそちらを使う。
  • グループを使うことで複数のユーザーを効率的に管理できる。
  • ポリシーは「できることリスト」。必要最小限の権限だけを与える（最小権限の原則）。
  • MFA（多要素認証）でセキュリティを大幅に強化できる。特にルートユーザーには必須。
