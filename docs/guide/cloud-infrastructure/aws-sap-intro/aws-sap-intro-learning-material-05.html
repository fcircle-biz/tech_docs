Route 53 - DNSとトラフィック管理

Route 53を使用したDNS管理とトラフィックルーティングの高度な設計手法を習得します。

  • プライベートホストゾーンとVPC統合
  • Route 53 Resolverとハイブリッド環境

5.1 DNSの基礎とRoute 53の役割

DNS（Domain Name System）は、人間が理解しやすいドメイン名（例: example.com）を、コンピュータが通信に使用するIPアドレス（例: 192.0.2.1）に変換する仕組みです。 Route 53は、AWSが提供する高可用性でスケーラブルなDNSサービスで、ドメイン登録、DNSルーティング、ヘルスチェック機能を統合しています。

DNSの動作の仕組み

sequenceDiagram participant User as ユーザー participant Browser as Webブラウザ participant Resolver as DNSリゾルバ participant Route53 as Route 53 participant Web as Webサーバー User->>Browser: example.com にアクセス Browser->>Resolver: example.com のIPアドレスは？ Resolver->>Route53: example.com のAレコードを照会 Route53->>Resolver: IPアドレス: 203.0.113.10 Resolver->>Browser: IPアドレス: 203.0.113.10 Browser->>Web: 203.0.113.10 に接続 Web->>Browser: Webページを返す Browser->>User: Webページを表示

Route 53の主要機能

  • ドメイン登録: .com、.org、.jpなど、様々なTLDのドメイン名を登録可能
  • DNSルーティング: 7種類のルーティングポリシーで高度なトラフィック制御が可能
  • ヘルスチェック: エンドポイントの正常性を監視し、自動フェイルオーバー
  • 100% SLA: 高可用性を保証（パブリックホストゾーン）

Route 53の名称の由来

Route 53という名称は、DNSサーバーが使用する標準ポート番号「53」に由来しています。また、「Route」は米国の国道「Route 66」を連想させ、高速で信頼性の高いトラフィックルーティングを意味しています。

5.2 レコードタイプの使い分け（A、AAAA、CNAME、Alias）

Route 53では、様々なレコードタイプを使用してDNS名前解決を設定します。それぞれのレコードタイプの特徴と使い分けを理解することが重要です。

レコードタイプ 用途 例
A ドメイン名をIPv4アドレスにマッピング example.com → 203.0.113.10
AAAA ドメイン名をIPv6アドレスにマッピング example.com → 2001:0db8::1
CNAME ドメイン名を別のドメイン名にマッピング（ゾーン頂点では使用不可） www.example.com → example.com
Alias AWSリソースにマッピング（ゾーン頂点でも使用可、無料） example.com → ELB、CloudFront

Aliasレコードの重要性

Aliasレコードは、Route 53独自の拡張機能で、AWSリソース（ELB、CloudFront、S3、API Gatewayなど）に対して使用します。 通常のCNAMEレコードと異なり、以下の利点があります。

Aliasレコードの利点

  • • ゾーン頂点（example.com）で使用可能
  • • DNSクエリ料金が無料
  • • ヘルスチェックが自動
  • • TTLが自動設定される

CNAMEレコードの制限

  • • ゾーン頂点では使用不可
  • • DNSクエリごとに課金
  • • ヘルスチェックを別途設定
  • • TTLを手動設定する必要

ベストプラクティス

AWSリソースへのマッピングには、常にAliasレコードを使用してください。 CNAMEレコードは、AWS外部のリソースや、サブドメイン（www.example.com）にのみ使用します。

5.3 ルーティングポリシーの種類と選択基準

Route 53では、7種類のルーティングポリシーを提供しており、トラフィックの分散方法を細かく制御できます。 ビジネス要件に応じて最適なポリシーを選択することが、SAP試験でも重要なポイントです。

7つのルーティングポリシー詳細

1. シンプルルーティング（Simple Routing）

単一のリソースに対してDNSレコードを設定する最も基本的なポリシー。

  • • 用途: 単一のWebサーバー、単一のロードバランサー
  • • 複数のIPアドレスを指定可能（ランダムに1つが返却される）
  • • ヘルスチェック機能なし

2. 加重ルーティング（Weighted Routing）

複数のリソースに対して、指定した重み付けに応じてトラフィックを分散。

  • • 用途: ブルー/グリーンデプロイ、A/Bテスト、段階的移行
  • • 例: リソースA（重み70）、リソースB（重み30）→ 70%と30%に分散
  • • ヘルスチェックと統合可能

3. レイテンシールーティング（Latency Routing）

ユーザーに最も低いレイテンシーを提供できるリージョンのリソースを返す。

  • • 用途: グローバル展開されたアプリケーション
  • • Route 53が自動的にユーザーからの遅延を測定
  • • 例: 東京リージョンとバージニアリージョンで最適なものを自動選択

4. 位置情報ルーティング（Geolocation Routing）

ユーザーの地理的位置（国、大陸、都道府県）に基づいてリソースを返す。

  • • 用途: コンテンツのローカライゼーション、地域制限、コンプライアンス
  • • 例: 日本からのアクセス→日本語サイト、米国→英語サイト
  • • デフォルトロケーションを設定可能

5. 地理的近接性ルーティング（Geoproximity Routing）

リソースの物理的な場所とユーザーの場所の距離に基づいてルーティング。バイアス値で調整可能。

  • • 用途: トラフィックの地理的分散を細かく制御
  • • バイアス値（-99～+99）でトラフィック範囲を拡大/縮小
  • • Traffic Flow機能を使用して設定

6. フェイルオーバールーティング（Failover Routing）

アクティブ-スタンバイ構成で、プライマリリソースが異常時にセカンダリリソースに自動切り替え。

  • • 用途: 災害復旧（DR）、高可用性設計
  • • プライマリとセカンダリを定義し、ヘルスチェックで監視
  • • 例: プライマリ（東京リージョン）→ セカンダリ（大阪リージョン）

7. 複数値回答ルーティング（Multivalue Answer Routing）

複数のリソース（最大8個）のIPアドレスをランダムに返す。各リソースにヘルスチェックを関連付け可能。

  • • 用途: シンプルな負荷分散とヘルスチェックの組み合わせ
  • • シンプルルーティングにヘルスチェックを追加した拡張版
  • • ELBの代替としては不適切（ELB + Aliasレコードを推奨）

5.4 ヘルスチェックとDNSフェイルオーバー

Route 53のヘルスチェック機能は、エンドポイントの正常性を監視し、異常を検出した場合にトラフィックを正常なリソースに自動的にルーティングします。 高可用性アーキテクチャの重要な構成要素です。

ヘルスチェックの3つのタイプ

エンドポイント監視

指定したIPアドレスやドメイン名に対してHTTP/HTTPS/TCPリクエストを送信し、正常性を確認。

計算されたヘルスチェック

複数の子ヘルスチェックの結果を組み合わせて、全体の正常性を判断（AND/OR条件）。

CloudWatch アラーム監視

CloudWatchアラームの状態を監視し、アラームがOKかALARMかで正常性を判断。

フェイルオーバー構成の例

sequenceDiagram participant User as ユーザー participant R53 as Route 53 participant HC as ヘルスチェック participant Primary as プライマリ (東京) participant Secondary as セカンダリ (大阪) HC->>Primary: ヘルスチェック Primary->>HC: 正常応答 Note over HC: ステータス: Healthy User->>R53: example.com の解決 R53->>HC: プライマリの状態確認 HC->>R53: Healthy R53->>User: プライマリのIP返却 User->>Primary: アクセス Note over Primary: 障害発生 HC->>Primary: ヘルスチェック Primary--xHC: 応答なし Note over HC: ステータス: Unhealthy User->>R53: example.com の解決 R53->>HC: プライマリの状態確認 HC->>R53: Unhealthy R53->>User: セカンダリのIP返却 User->>Secondary: アクセス

ヘルスチェックの注意点

ヘルスチェックのリクエストは、世界中のRoute 53ヘルスチェッカーから送信されます。 そのため、ファイアウォールやセキュリティグループで、Route 53のIPアドレス範囲（公開されている）からのトラフィックを許可する必要があります。

5.5 プライベートホストゾーンとVPC統合

Route 53プライベートホストゾーンは、VPC内でのみ有効なプライベートDNS名前空間を提供します。 インターネットに公開されていない内部リソース（RDS、内部ELB、EC2インスタンス）に対してわかりやすい名前を付けることができます。

パブリックホストゾーン vs プライベートホストゾーン

項目 パブリックホストゾーン プライベートホストゾーン
名前解決の範囲 インターネット全体 特定のVPC内のみ
用途 公開Webサイト、API 内部アプリケーション、RDS、内部ELB
関連付け なし 1つ以上のVPCに関連付け
ヘルスチェック 利用可能 利用不可（VPC内からアクセスできないため）

ベストプラクティス

  • • 内部リソースには必ずプライベートホストゾーンを使用
  • • 内部ドメイン名には「.internal」などの明確な接尾辞を使用
  • • 複数のVPCで共通のプライベートDNS名前空間が必要な場合、VPCピアリングを検討

5.6 Route 53 Resolverとハイブリッド環境

Route 53 Resolverは、VPCとオンプレミスネットワーク間でDNSクエリを相互に転送する機能を提供します。 ハイブリッドクラウド環境において、AWSリソースとオンプレミスリソース間でシームレスな名前解決を実現します。

Route 53 Resolverの2つのエンドポイント

インバウンドエンドポイント

オンプレミスからVPC内のDNS名を解決できるようにする

  • • オンプレミスDNSサーバー → Route 53 Resolver
  • • 例: オンプレミスから「rds.internal.aws.com」を解決

アウトバウンドエンドポイント

VPCからオンプレミスのDNS名を解決できるようにする

  • • Route 53 Resolver → オンプレミスDNSサーバー
  • • 例: VPCから「app.onprem.local」を解決

理解度確認クイズ

  1. Q1. AliasレコードとCNAMEレコードの主な違いを3つ挙げてください。

    回答を表示

    1. Aliasレコードはゾーン頂点（example.com）で使用可能、CNAMEは不可
    2. Aliasレコードは無料、CNAMEは有料（クエリごとに課金）
    3. Aliasレコードは自動ヘルスチェック、CNAMEは手動設定が必要

  2. Q2. グローバルに展開されたWebアプリケーションで、ユーザーに最も低いレイテンシーを提供したい場合、どのルーティングポリシーを使用すべきですか？

    回答を表示

    レイテンシールーティング（Latency Routing）を使用します。 Route 53が自動的にユーザーから各リージョンへのレイテンシーを測定し、最も低いレイテンシーのリソースを返します。

  3. Q3. プライベートホストゾーンでヘルスチェックが利用できない理由は何ですか？

    回答を表示

    Route 53のヘルスチェッカーは、インターネット上から対象リソースにアクセスして正常性を確認します。 プライベートホストゾーンはVPC内でのみ有効なため、外部からアクセスできず、ヘルスチェックが実行できません。

  4. Q4. Route 53 Resolverのアウトバウンドエンドポイントは何のために使用されますか？

    回答を表示

    VPC内のリソースから、オンプレミスのDNS名（例: app.onprem.local）を解決するために使用されます。 DNSクエリをオンプレミスのDNSサーバーに転送し、ハイブリッド環境での名前解決を実現します。

第5章のまとめ

  • Route 53は高可用性でスケーラブルなDNSサービスで、100% SLAを提供
  • AWSリソースへのマッピングには、ゾーン頂点で使用可能で無料のAliasレコードを使用
  • 7種類のルーティングポリシー（シンプル、加重、レイテンシー、位置情報、地理的近接性、フェイルオーバー、複数値）を要件に応じて使い分ける
  • ヘルスチェックで自動フェイルオーバーを実現し、高可用性を確保
  • プライベートホストゾーンでVPC内部のリソースにわかりやすいDNS名を付与
  • Route 53 Resolverでハイブリッド環境のDNS統合を実現
