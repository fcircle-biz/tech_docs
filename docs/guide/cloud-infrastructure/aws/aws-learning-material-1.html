第1章: AWS入門とアカウント設定 - クラウドコンピューティングの世界へようこそ

この章で学ぶこと
  • クラウドコンピューティングの基本概念と利点を理解する
  • AWSのグローバルインフラストラクチャ(リージョン、アベイラビリティゾーン)を理解する
  • AWSアカウントを作成し、セキュアに設定できる
  • AWSマネジメントコンソールの基本操作を習得する
  • 責任共有モデルの概念を理解する

1.1 クラウドコンピューティングとは

1.1.1 クラウドコンピューティングの定義

クラウドコンピューティングとは、インターネット経由でコンピューティングリソース(サーバー、ストレージ、データベース、ネットワーク、ソフトウェアなど)を必要に応じて利用できるサービスモデルです。

従来は、企業が自社でサーバーを購入し、データセンターを構築・運用する必要がありました(オンプレミス環境)。しかし、クラウドでは物理的なインフラを所有せず、必要なときに必要な分だけリソースを借りて使うことができます。

1.1.2 オンプレミスとクラウドの違い

オンプレミス環境とクラウドの主な違いを以下の表で確認しましょう。

項目 オンプレミス クラウド
初期投資 高額(サーバー購入、データセンター構築) 低額(使った分だけ課金)
導入スピード 数週間〜数ヶ月 数分〜数時間
スケーラビリティ 困難(物理的なサーバー増設が必要) 容易(数クリックで拡張可能)
保守・運用 自社で全て実施 クラウド事業者が一部を担当
災害対策 自社で別拠点を用意する必要がある 複数リージョンに簡単に展開可能

1.1.3 サービスモデル: IaaS、PaaS、SaaS

クラウドサービスは、提供される機能レベルによって3つのモデルに分類されます。

flowchart TD subgraph SaaS["SaaS (Software as a Service)"] S1["アプリケーション
例: Gmail, Office 365"] end subgraph PaaS["PaaS (Platform as a Service)"] P1["ランタイム環境
例: AWS Elastic Beanstalk"] end subgraph IaaS["IaaS (Infrastructure as a Service)"] I1["仮想サーバー、ストレージ
例: AWS EC2, S3"] end SaaS --> PaaS --> IaaS
  • IaaS (Infrastructure as a Service): 仮想サーバー、ストレージ、ネットワークなどのインフラを提供。ユーザーはOS以上を管理。例: AWS EC2、S3
  • PaaS (Platform as a Service): アプリケーション実行環境を提供。ユーザーはアプリケーションのみを管理。例: AWS Lambda、Elastic Beanstalk
  • SaaS (Software as a Service): 完成したソフトウェアをブラウザ経由で提供。ユーザーは設定とデータのみを管理。例: Gmail、Salesforce

1.1.4 クラウドの6つの利点

AWSが掲げるクラウドコンピューティングの6つの利点は以下の通りです。

  1. 固定費から変動費へ: サーバーを購入する代わりに、使用した分だけ支払う
  2. スケールのメリット: AWS全体の利用量が増えると、単価が下がる恩恵を受けられる
  3. キャパシティの推測が不要: 必要に応じてリソースを増減できるため、過剰投資や不足の心配がない
  4. スピードと俊敏性の向上: 数分で新しいサーバーを起動でき、迅速な実験が可能
  5. データセンター運用コストの削減: インフラ管理から解放され、ビジネス価値の創出に集中できる
  6. グローバル展開が容易: 世界中のリージョンにわずか数クリックでアプリケーションをデプロイ可能

1.2 AWSの全体像

1.2.1 AWSとは

AWS (Amazon Web Services)は、Amazon.comが2006年から提供しているクラウドコンピューティングプラットフォームです。現在、世界で最も広く採用されているクラウドプラットフォームであり、200以上のサービスを提供しています。

AWSは、スタートアップから大企業、政府機関まで幅広い組織に利用されており、Netflix、Airbnb、NASAなども顧客として知られています。

1.2.2 AWSのサービスカテゴリー

AWSは多数のサービスを提供していますが、主要なカテゴリーは以下の通りです。

  • コンピューティング: EC2(仮想サーバー)、Lambda(サーバーレス)
  • ストレージ: S3(オブジェクトストレージ)、EBS(ブロックストレージ)
  • データベース: RDS(リレーショナルDB)、DynamoDB(NoSQL)
  • ネットワーキング: VPC(仮想ネットワーク)、CloudFront(CDN)
  • セキュリティ: IAM(アクセス管理)、KMS(暗号鍵管理)
  • 監視・運用: CloudWatch(監視)、CloudTrail(監査ログ)

1.2.3 グローバルインフラストラクチャ

AWSは世界中にデータセンターを配置しており、以下の階層構造で構成されています。

flowchart TD R["AWSグローバルインフラストラクチャ"] R --> R1["リージョン
(Region)"] R --> R2["リージョン
(Region)"] R --> R3["リージョン
(Region)"] R1 --> AZ1["アベイラビリティゾーン
(AZ-1a)"] R1 --> AZ2["アベイラビリティゾーン
(AZ-1b)"] R1 --> AZ3["アベイラビリティゾーン
(AZ-1c)"] AZ1 --> DC1["データセンター群"] AZ2 --> DC2["データセンター群"] AZ3 --> DC3["データセンター群"]

リージョン (Region)

  • 地理的に独立した地域(例: 東京、オレゴン、アイルランド)
  • 2024年現在、世界に33のリージョンが存在
  • 日本には東京リージョン(ap-northeast-1)と大阪リージョン(ap-northeast-3)がある
  • リージョン間は物理的に完全に分離されており、災害時の影響を最小化

アベイラビリティゾーン (Availability Zone, AZ)

  • 1つのリージョン内に複数存在する独立したデータセンター群
  • 各AZは物理的に離れており、電力・ネットワークが独立
  • 複数AZにリソースを分散することで高可用性を実現
  • 東京リージョンには4つのAZがある

エッジロケーション

  • CloudFront(CDN)やRoute 53(DNS)のための拠点
  • 世界中に400以上のエッジロケーションが存在
  • ユーザーに最も近い拠点からコンテンツを配信し、低レイテンシーを実現

1.2.4 責任共有モデル

AWSでは、セキュリティと運用の責任をAWSとユーザーで分担する「責任共有モデル」を採用しています。

flowchart TD subgraph Customer["ユーザーの責任
(Security IN the Cloud)"] C1["アプリケーションのセキュリティ"] C2["データの暗号化"] C3["OSやミドルウェアのパッチ適用"] C4["IAMユーザー・権限管理"] C5["ネットワーク設定"] end subgraph AWS["AWSの責任
(Security OF the Cloud)"] A1["物理的なデータセンターセキュリティ"] A2["ハードウェアの保守"] A3["ネットワークインフラ"] A4["ハイパーバイザーの管理"] A5["グローバルインフラの運用"] end
  • AWSの責任 (Security OF the Cloud): 物理的なインフラ、ハードウェア、ネットワーク、データセンターのセキュリティ
  • ユーザーの責任 (Security IN the Cloud): アプリケーション、データ、OS、アクセス権限、暗号化設定など
重要ポイント

サービスによって責任範囲が異なります。例えば、EC2ではOSの管理はユーザーの責任ですが、LambdaではAWSが管理します。各サービスの責任範囲を理解することが重要です。

1.3 AWSアカウントの作成と初期設定

1.3.1 AWSアカウントの作成手順

AWSアカウントを作成するには、以下の情報が必要です。

  • メールアドレス(アカウント登録用)
  • 電話番号(SMS/音声による本人確認用)
  • クレジットカード情報(無料利用枠内では課金されません)

作成手順の概要:

  1. AWS公式サイト(https://aws.amazon.com/jp/)にアクセス
  2. 「無料でアカウント作成」をクリック
  3. メールアドレスとアカウント名を入力
  4. 連絡先情報(氏名、住所、電話番号)を入力
  5. クレジットカード情報を登録
  6. SMS/音声による本人確認
  7. サポートプラン選択(最初は無料のベーシックプランを選択)
  8. アカウント作成完了

1.3.2 ルートユーザーとIAMユーザー

AWSアカウントを作成すると、ルートユーザーが生成されます。

  • ルートユーザー: アカウント作成時に登録したメールアドレスでログイン。全ての権限を持つ
  • IAMユーザー: 後から作成する制限された権限を持つユーザー
セキュリティのベストプラクティス

ルートユーザーは日常的に使用しないでください。アカウント作成後すぐに、IAMユーザーを作成し、そちらを使用することが強く推奨されます。ルートユーザーはアカウント設定変更など、限られた管理タスクのみに使用します。

1.3.3 多要素認証(MFA)の設定

多要素認証(MFA: Multi-Factor Authentication)は、パスワードに加えて、スマートフォンなどの物理デバイスによる認証を追加するセキュリティ機能です。

MFAの種類:

  • 仮想MFAデバイス: スマートフォンアプリ(Google Authenticator、Microsoft Authenticatorなど)を使用
  • ハードウェアMFAデバイス: 専用の物理トークンを使用
  • U2Fセキュリティキー: YubiKeyなどのUSBデバイスを使用

最も手軽で一般的なのは、スマートフォンアプリを使った仮想MFAデバイスです。ルートユーザーには必ずMFAを有効化してください。

1.3.4 請求アラートの設定

予期しない課金を防ぐために、アカウント作成後すぐに請求アラートを設定しましょう。

  1. CloudWatchで請求アラームを作成
  2. 予算(例: 月10ドル)を設定
  3. 予算を超えた場合にメール通知を受け取る

また、AWS Budgetsを使用すると、より詳細な予算管理とコスト予測が可能です。

1.3.5 無料利用枠の理解

AWSは新規アカウント向けに12ヶ月間の無料利用枠を提供しています。

主な無料利用枠:

  • EC2: t2.micro/t3.microインスタンスを月750時間(約1ヶ月間1台稼働可能)
  • S3: 5GBの標準ストレージ
  • RDS: db.t2.micro/db.t3.microインスタンスを月750時間
  • Lambda: 月100万リクエスト(永久無料)
  • DynamoDB: 月25GBのストレージ(永久無料)
注意事項
  • 無料利用枠を超えると課金が発生します
  • 一部サービス(NAT Gateway、Elastic IPの未使用など)は無料利用枠がありません
  • 必ず請求アラートを設定し、定期的にコストを確認しましょう

1.4 AWSマネジメントコンソールの基本操作

1.4.1 コンソールへのログイン

AWSマネジメントコンソール(https://console.aws.amazon.com/)にアクセスし、メールアドレスとパスワードでログインします。MFAを有効化している場合は、ワンタイムパスワードの入力も必要です。

1.4.2 コンソールの基本構成

AWSマネジメントコンソールは、以下の主要要素で構成されています。

  • ナビゲーションバー(上部): サービス検索、リージョン選択、アカウント情報
  • サービスメニュー: 左上の「サービス」から全サービスにアクセス可能
  • 最近使用したサービス: よく使うサービスに素早くアクセス
  • お気に入り: 頻繁に使うサービスをピン留め可能
  • リソースグループ: 関連リソースをまとめて管理

1.4.3 リージョンの選択

コンソール右上のドロップダウンメニューから、使用するリージョンを選択できます。

重要: リージョンによるサービスの違い
  • リソース(EC2インスタンス、S3バケットなど)はリージョンごとに作成されます
  • リージョンを切り替えると、別のリソースセットが表示されます
  • 日本で使用する場合は、通常「アジアパシフィック(東京) ap-northeast-1」を選択します
  • 一部サービス(IAM、Route 53など)はグローバルサービスで、リージョン選択不要です

1.4.4 AWS CLI と AWS SDK

AWSの操作は、Webコンソール以外にも以下の方法があります。

AWS CLI (Command Line Interface)

  • コマンドラインからAWSサービスを操作するツール
  • スクリプトによる自動化が可能
  • Windows、macOS、Linuxで使用可能
# AWS CLIの使用例
# S3バケット一覧を表示
aws s3 ls

# EC2インスタンス一覧を表示
aws ec2 describe-instances

# 東京リージョンのEC2インスタンス起動
aws ec2 run-instances --region ap-northeast-1 --image-id ami-xxxxx --instance-type t3.micro

AWS SDK (Software Development Kit)

  • プログラミング言語からAWSを操作するライブラリ
  • Python(Boto3)、JavaScript、Java、C#、Go、Ruby、PHPなどに対応
  • アプリケーションにAWS機能を組み込む際に使用

1.4.5 CloudShell の基本的な使い方

AWS CloudShellは、ブラウザ上で利用できるLinuxシェル環境です。AWS CLIがプリインストールされており、追加設定なしでAWS操作が可能です。

  • マネジメントコンソール右上のアイコンからアクセス
  • 無料で使用可能(データ保存は1GBまで無料)
  • ホームディレクトリ(/home/cloudshell-user)のデータは保持される
  • Python、Node.js、AWS CLI v2などがプリインストール
演習 1-1: AWSアカウントの作成とセキュリティ設定

目的: AWSアカウントを作成し、セキュリティのベストプラクティスに従って初期設定を行います。

手順
  1. AWS公式サイトからAWSアカウントを作成する
  2. ルートユーザーに仮想MFAデバイス(Google Authenticatorなど)を設定する
  3. CloudWatchで請求アラームを作成し、予算を設定する(例: 月10ドル)
  4. IAMユーザーを1つ作成し、管理者権限を付与する
  5. 作成したIAMユーザーでログインできることを確認する
確認ポイント
  • ルートユーザーにMFAが有効化されているか
  • 請求アラートが正しく設定されているか
  • IAMユーザーで正常にログインできるか
重要: この演習後、日常的にはルートユーザーを使用せず、作成したIAMユーザーでログインしてください。
演習 1-2: リージョンとサービスの確認

目的: AWSマネジメントコンソールの基本操作を習得し、リージョンの概念を理解します。

手順
  1. マネジメントコンソールにログインする
  2. コンソール右上でリージョンを「アジアパシフィック(東京)」に切り替える
  3. 「サービス」メニューから「EC2」にアクセスし、ダッシュボードを確認する
  4. リージョンを「米国東部(バージニア北部)」に切り替え、同じくEC2ダッシュボードを確認する
  5. CloudShellを起動し、以下のコマンドを実行する: 
    # 現在のリージョンを確認
aws ec2 describe-regions --output table

# 自分のアカウントIDを確認
aws sts get-caller-identity
確認ポイント
  • リージョンを切り替えると、表示されるリソースが異なることを理解できたか
  • CloudShellで基本的なAWS CLIコマンドを実行できたか
理解度確認クイズ
  1. 問題1: クラウドコンピューティングの6つの利点のうち、3つを挙げてください。
  2. 問題2: IaaS、PaaS、SaaSの違いを説明してください。具体的なAWSサービス例も挙げてください。
  3. 問題3: AWSのリージョンとアベイラビリティゾーンの違いは何ですか?なぜ複数のAZを使用することが推奨されるのですか?
  4. 問題4: 責任共有モデルにおいて、AWSの責任とユーザーの責任をそれぞれ3つずつ挙げてください。
  5. 問題5: ルートユーザーとIAMユーザーの違いは何ですか?日常的にルートユーザーを使用すべきではない理由を説明してください。
  6. 問題6: 多要素認証(MFA)とは何ですか?なぜルートユーザーにMFAを設定する必要があるのですか?
  7. 問題7: AWS無料利用枠について、EC2とLambdaの無料利用内容の違いを説明してください。
