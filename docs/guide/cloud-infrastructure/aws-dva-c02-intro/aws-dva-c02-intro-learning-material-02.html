サーバーレスコンピューティング基礎 - AWS Lambdaの仕組み

AWS Lambdaの動作原理とサーバーレスアーキテクチャの基本概念を深く理解します。

2.1 サーバーレスアーキテクチャとは何か

サーバーレスアーキテクチャは、インフラストラクチャの管理をクラウドプロバイダーに委ね、開発者がビジネスロジックの実装に集中できるアプローチです。「サーバーレス」という名前ですが、実際にはサーバーが存在します。ただし、その管理責任がAWSに移譲されます。

サーバーレスのメリット

  • 運用負荷の削減: サーバーのプロビジョニング、パッチ適用、スケーリングが不要
  • コスト最適化: 実行時間とリクエスト数に応じた従量課金（アイドル時のコストがゼロ）
  • 自動スケーリング: トラフィックに応じて自動的にスケールアウト/スケールイン
  • 高可用性: AWSが複数AZにまたがって冗長化を管理
  • 開発スピード向上: インフラ管理からの解放により、ビジネスロジックに集中

サーバーレスの制約

  • 実行時間制限: Lambda関数は最大15分まで（長時間実行にはStep Functions等を使用）
  • コールドスタート: 初回起動時のレイテンシが発生する可能性
  • メモリ制限: 最大10GBまで（大規模な処理には向かない場合がある）
  • ステートレス: 関数間で状態を保持できない（外部ストレージが必要）
  • ベンダーロックイン: AWS固有のサービスに依存する

重要ポイント

サーバーレスは「万能」ではありません。短時間で完了するイベント駆動型の処理に最適です。長時間実行バッチ処理や常時起動が必要なアプリケーションには、ECSやEC2の方が適している場合があります。

2.2 Lambda関数のライフサイクル

Lambda関数は、イベントによって起動されるたびに以下のライフサイクルを経ます。このライフサイクルを理解することで、パフォーマンス最適化の鍵となるコールドスタートとウォームスタートの違いが分かります。

sequenceDiagram autonumber participant イベント participant Lambda participant 実行環境 participant ハンドラー イベント->>Lambda: 1. 関数呼び出し Lambda->>実行環境: 2. 実行環境の準備（コールドスタート時のみ） 実行環境->>実行環境: 3. コンテナ起動・ランタイム初期化 実行環境->>ハンドラー: 4. ハンドラー関数外のコード実行（初期化） ハンドラー->>ハンドラー: 5. ハンドラー関数の実行 ハンドラー-->>Lambda: 6. レスポンス返却 Lambda->>実行環境: 7. 実行環境を一時保持（次回ウォームスタート）

コールドスタート（Cold Start）

Lambda関数が初めて呼び出される場合、または長時間アイドル状態だった場合、AWSは新しい実行環境を作成します。この初期化プロセスをコールドスタートと呼びます。

  • 実行環境（コンテナ）の起動
  • ランタイム（Python、Node.js等）の初期化
  • 関数コードのダウンロード
  • 依存ライブラリのロード
  • ハンドラー関数外のコード実行（グローバル変数の初期化等）

コールドスタートには通常100ms〜数秒かかります。特にJavaやC#のようなランタイムは初期化が遅い傾向があります。

ウォームスタート（Warm Start）

Lambda関数が実行された後、AWSは実行環境を一定時間（通常5〜15分程度）保持します。この間に同じ関数が再度呼び出されると、既存の実行環境を再利用します。これをウォームスタートと呼びます。

  • 実行環境の初期化が不要
  • ハンドラー関数のみが実行される
  • レイテンシが大幅に低減（数ms〜数十ms）
  • グローバル変数やDB接続が再利用可能

ベストプラクティス: 初期化コードの配置

データベース接続やSDKクライアントの初期化は、ハンドラー関数の外（グローバルスコープ）で行うことで、ウォームスタート時に再利用でき、パフォーマンスが向上します。

2.3 実行ロールとリソースベースポリシー

Lambda関数がAWSリソースにアクセスするには、適切な権限設定が必要です。Lambdaには2種類の権限モデルがあります。

flowchart TD subgraph 実行ロール["実行ロール（Execution Role）"] direction TB E1["Lambda関数が他のAWSサービスにアクセス"] E2["例: S3バケット読み取り、DynamoDB書き込み"] E1 --> E2 end subgraph リソースベース["リソースベースポリシー"] direction TB R1["他のAWSサービスがLambda関数を呼び出し"] R2["例: S3イベント、API Gateway、SNS"] R1 --> R2 end 実行ロール -.Lambda関数が主体.- リソースベース

実行ロール（Execution Role）

実行ロールは、Lambda関数が他のAWSサービスにアクセスするための権限を定義するIAMロールです。関数が実行される際、このロールが自動的に引き受けられます。

実行ロールのポリシー例（JSON） 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::my-bucket/*"
    },
    {
      "Effect": "Allow",
      "Action": [
        "dynamodb:PutItem",
        "dynamodb:GetItem"
      ],
      "Resource": "arn:aws:dynamodb:ap-northeast-1:123456789012:table/MyTable"
    }
  ]
}

リソースベースポリシー

リソースベースポリシーは、どのAWSサービスやアカウントがLambda関数を呼び出せるかを定義します。S3、API Gateway、EventBridgeなどからのトリガーを設定する際に使用します。

リソースベースポリシー例（S3からの呼び出し許可） 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "s3.amazonaws.com"
      },
      "Action": "lambda:InvokeFunction",
      "Resource": "arn:aws:lambda:ap-northeast-1:123456789012:function:MyFunction",
      "Condition": {
        "ArnLike": {
          "AWS:SourceArn": "arn:aws:s3:::my-bucket"
        }
      }
    }
  ]
}

2.4 メモリ・タイムアウト設定とパフォーマンス

Lambda関数のパフォーマンスは、メモリ設定とタイムアウト設定に大きく影響されます。これらの設定を適切に調整することで、コストとパフォーマンスの最適なバランスを実現できます。

メモリ設定

Lambdaでは、128MB〜10,240MB（10GB）の範囲でメモリを1MB単位で設定できます。重要なポイントは、**メモリ設定はCPU性能にも影響する**ことです。

  • メモリを増やすと、割り当てられるCPU性能も比例して向上
  • 1,769MB以上に設定すると、1つの仮想CPUコアが割り当てられる
  • メモリ使用量が少ない処理でも、CPU性能向上のためにメモリを増やすと実行時間が短縮されることがある
  • 料金はメモリ量と実行時間の積で決まる（GB秒）

コスト最適化の考え方

メモリを増やすと1ms当たりの料金は上がりますが、実行時間が短縮されるため、トータルコストが下がる場合があります。

例: 128MBで10秒 vs 1024MBで2秒 → 後者の方がコストが安くなる可能性がある

タイムアウト設定

Lambda関数の最大実行時間は1秒〜900秒（15分）の範囲で設定できます。デフォルトは3秒です。

  • タイムアウトが短すぎると、処理が完了する前に強制終了される
  • タイムアウトが長すぎると、エラー時に無駄な課金が発生する
  • API Gatewayと統合する場合、API Gatewayのタイムアウト（29秒）も考慮が必要

注意

15分を超える処理が必要な場合は、Step Functionsを使用して複数のLambda関数に処理を分割するか、ECS/Fargateなどのコンテナサービスの利用を検討しましょう。

2.5 同時実行数とスロットリング

Lambda関数は自動スケーリングしますが、無制限にスケールするわけではありません。同時実行数（Concurrency）の制限と、スロットリングの仕組みを理解することが重要です。

同時実行数の制限

  • リージョンレベルの制限: デフォルトで1000（AWSサポートに申請して増やすことも可能）
  • 予約された同時実行数（Reserved Concurrency）: 特定の関数に最低限の実行枠を確保
  • プロビジョニングされた同時実行数（Provisioned Concurrency）: 事前にウォームスタート状態の実行環境を準備（コールドスタート対策）

TooManyRequestsException"] D --> E["リトライ or エラー処理"] C --> F["レスポンス返却"]

スロットリングの動作

同時実行数が制限を超えた場合、Lambda関数は**スロットリング**され、リクエストが拒否されます。呼び出し元によって動作が異なります。

  • 同期呼び出し（API Gateway等）: 429エラー（TooManyRequestsException）が返される
  • 非同期呼び出し（S3イベント等）: 自動的に最大6時間リトライされる
  • ストリームベース呼び出し（Kinesis、DynamoDB Streams）: Lambda側でリトライし、成功するまでレコードをブロック

ベストプラクティス: 予約済み同時実行数の設定

重要な本番関数には予約済み同時実行数を設定することで、他の関数がリソースを使い切っても、確実に実行できる枠を確保できます。

2.6 Lambda Layers（レイヤー）による共通コード管理

Lambda Layersは、複数のLambda関数で共有できるライブラリやカスタムランタイムをパッケージ化する機能です。依存ライブラリを各関数にバンドルする必要がなくなり、デプロイサイズを削減できます。

flowchart TB subgraph Layer["Lambda Layer"] direction TB L1["共通ライブラリ（boto3、requests等）"] L2["カスタムヘルパー関数"] L3["設定ファイル"] end subgraph Functions["Lambda関数"] F1["関数A"] F2["関数B"] F3["関数C"] end Layer --> F1 Layer --> F2 Layer --> F3

Layersのメリット

  • デプロイパッケージのサイズを削減（デプロイ時間の短縮）
  • 共通ライブラリのバージョン管理が一元化される
  • 最大5つのLayerを1つの関数にアタッチ可能
  • 各Layerは最大50MBまで（解凍後250MB）

Layersの使用例

Lambda関数でLayerを使用する例（Python） 
import json
# Layerに含まれるライブラリをインポート
import requests  # Layerに含まれている

def lambda_handler(event, context):
    # Layer内の共通ライブラリを使用
    response = requests.get('https://api.example.com/data')

    return {
        'statusCode': 200,
        'body': json.dumps(response.json())
    }

公式LayerとサードパーティLayer

AWSは公式にいくつかのLayerを提供しています（X-Ray SDK、Parameters and Secrets Lambda Extension等）。また、サードパーティが提供するLayerもあります（Datadog、New Relic等）。

2.7 Lambda Extensions

Lambda Extensionsは、Lambda関数の実行環境をカスタマイズするための仕組みです。関数コードと並行して実行され、モニタリング、セキュリティ、ガバナンスなどの機能を追加できます。

Extensionsの種類

  • 外部Extension: 別プロセスとして実行され、ライフサイクル全体にわたって動作
  • 内部Extension: Lambda関数と同じプロセス内で実行（Layerとして提供）

Extensionsの主な用途

  • モニタリング: CloudWatch、Datadog、New Relicなどへのメトリクス送信
  • セキュリティ: Secrets ManagerやParameter Storeからのシークレット取得キャッシュ
  • トレーシング: AWS X-Rayによる分散トレーシング
  • ログ集約: カスタムログ収集・転送

代表的な公式Extension

AWS Parameters and Secrets Lambda Extension: Secrets ManagerやParameter Storeの値をキャッシュし、関数実行時の取得をローカルHTTPエンドポイント経由で高速化します。

理解度確認クイズ

  1. Q1. コールドスタートとウォームスタートの違いを説明してください。

    回答を表示

    コールドスタートは、Lambda関数が初めて呼び出される際に実行環境を新規作成するプロセスで、数百ms〜数秒のレイテンシが発生します。
    ウォームスタートは、既存の実行環境を再利用するため、初期化処理が不要でレイテンシが数ms〜数十msに短縮されます。DB接続やSDKクライアントをグローバルスコープで初期化することで、ウォームスタート時に再利用できます。

  2. Q2. Lambda関数のメモリ設定を増やすと、どのような影響がありますか？

    回答を表示

    メモリを増やすと、CPU性能も比例して向上します。1,769MB以上に設定すると1つの仮想CPUコアが割り当てられます。料金はメモリ量×実行時間（GB秒）で計算されますが、実行時間が短縮されるため、トータルコストが下がる場合があります。

  3. Q3. Lambda Layersを使用するメリットは何ですか？

    回答を表示

    Lambda Layersを使用すると、デプロイパッケージのサイズを削減でき、複数の関数で共通ライブラリを共有できます。これにより、デプロイ時間が短縮され、共通ライブラリのバージョン管理が一元化されます。最大5つのLayerを1つの関数にアタッチでき、各Layerは最大50MBまでです。
