4.1 DynamoDBとは - NoSQLデータベースの特徴

Amazon DynamoDBは、AWSが提供する**フルマネージドNoSQLデータベース**です。RDB（リレーショナルデータベース）とは異なり、スキーマレスで水平スケーリングが容易な設計となっています。

重要ポイント

DynamoDBは、**一貫したミリ秒台のレスポンスタイム**を実現するため、高トラフィックのWebアプリケーションやモバイルバックエンドに最適です。

DynamoDBの主な特徴

flowchart TD A["DynamoDB"] --> B["フルマネージド"] A --> C["サーバーレス"] A --> D["自動スケーリング"] A --> E["高可用性"] B --> B1["インフラ管理不要"] C --> C1["容量計画なし"] D --> D1["負荷に応じた自動調整"] E --> E1["複数AZへの自動レプリケーション"]
  • スキーマレス: テーブル作成時にプライマリキー以外のカラム定義は不要
  • 水平スケーリング: パーティション分散により自動的にスケールアウト
  • 低レイテンシ: 一貫したミリ秒台のレスポンス時間
  • 高可用性: 3つのAZにデータを自動レプリケーション
  • 柔軟な課金モデル: オンデマンドまたはプロビジョニング済みキャパシティ

RDBとの違い

項目 RDB（RDS等） DynamoDB
データモデル テーブル・行・列（固定スキーマ） テーブル・アイテム・属性（スキーマレス）
スケーリング 垂直スケーリング（インスタンス拡張） 水平スケーリング（パーティション追加）
クエリ SQL、JOIN、集計関数 キーバリューアクセス、Query、Scan
トランザクション ACID準拠 限定的なACIDサポート
課金モデル インスタンス稼働時間ベース リクエスト数とストレージ容量ベース

4.2 プライマリキーの設計 - データモデリングの基礎

DynamoDBでは、**プライマリキー**が最も重要な設計要素です。プライマリキーには2つのタイプがあります。

パーティションキー（Partition Key）のみ

最もシンプルな構成。パーティションキーの値が一意であれば、そのまま主キーとして機能します。

パーティションキーのみの例 
{
  "TableName": "Users",
  "KeySchema": [
    { "AttributeName": "userId", "KeyType": "HASH" }
  ],
  "AttributeDefinitions": [
    { "AttributeName": "userId", "AttributeType": "S" }
  ]
}

パーティションキー + ソートキー（Composite Key）

パーティションキーとソートキーを組み合わせた複合主キー。同じパーティションキーを持つ複数のアイテムをソートキーで区別できます。

複合キーの例 
{
  "TableName": "Orders",
  "KeySchema": [
    { "AttributeName": "customerId", "KeyType": "HASH" },
    { "AttributeName": "orderDate", "KeyType": "RANGE" }
  ],
  "AttributeDefinitions": [
    { "AttributeName": "customerId", "AttributeType": "S" },
    { "AttributeName": "orderDate", "AttributeType": "S" }
  ]
}
flowchart LR subgraph PK["パーティションキーのみ"] A1["userId: U001
name: Alice"] A2["userId: U002
name: Bob"] end subgraph CPK["複合キー（Partition + Sort）"] B1["customerId: C001
orderDate: 2025-01-15"] B2["customerId: C001
orderDate: 2025-02-01"] B3["customerId: C002
orderDate: 2025-01-20"] end

設計のヒント

パーティションキーは**均等に分散する値**を選びましょう。例えば、日付をパーティションキーにすると特定日にアクセスが集中し、ホットパーティション問題が発生します。

4.3 キャパシティユニット - RCUとWCU

DynamoDBのスループットは、**読み取り容量ユニット（RCU）**と**書き込み容量ユニット（WCU）**で測定されます。

読み取り容量ユニット（RCU）

  • 1 RCU = 1秒あたり最大4KBのアイテムを1回読み取り（強い整合性）
  • 結果整合性読み取りの場合、1 RCUで2回読み取り可能
  • アイテムサイズが4KBを超える場合、追加のRCUが必要
RCU計算例 
# 例: 3KBのアイテムを100回/秒読み取り（強い整合性）
# 必要なRCU = 100 × (3KB / 4KB) = 100 × 1 = 100 RCU

# 例: 6KBのアイテムを50回/秒読み取り（結果整合性）
# 必要なRCU = 50 × (6KB / 4KB) / 2 = 50 × 2 / 2 = 50 RCU

書き込み容量ユニット（WCU）

  • 1 WCU = 1秒あたり最大1KBのアイテムを1回書き込み
  • アイテムサイズが1KBを超える場合、追加のWCUが必要
WCU計算例 
# 例: 2KBのアイテムを100回/秒書き込み
# 必要なWCU = 100 × (2KB / 1KB) = 100 × 2 = 200 WCU

# 例: 500バイトのアイテムを50回/秒書き込み
# 必要なWCU = 50 × (0.5KB / 1KB) = 50 × 1 = 50 WCU

4.4 キャパシティモード - オンデマンド vs プロビジョニング済み

DynamoDBには2つのキャパシティモードがあります。ワークロードの予測可能性に応じて選択します。

flowchart LR A["キャパシティモード"] --> B["オンデマンド"] A --> C["プロビジョニング済み"] B --> B1["トラフィック予測不要"] B --> B2["リクエスト単位の課金"] C --> C1["RCU/WCU事前設定"] C --> C2["予測可能なコスト"]

オンデマンドモード

  • 適用ケース: トラフィックが予測不可能、新規アプリケーション
  • 課金: リクエスト数に応じた従量課金
  • メリット: 事前のキャパシティ計画不要、自動スケーリング
  • デメリット: プロビジョニング済みより割高

プロビジョニング済みモード

  • 適用ケース: 予測可能なトラフィック、コスト最適化が必要
  • 課金: 設定したRCU/WCUに基づく固定費用
  • メリット: コスト予測可能、Auto Scalingで調整可能
  • デメリット: キャパシティ計画が必要

注意

プロビジョニング済みモードで、設定したキャパシティを超えるリクエストが来るとスロットリングエラーが発生します。Auto Scalingの設定を推奨します。

4.5 整合性モデル - 強い整合性と結果整合性

DynamoDBは分散データベースであるため、読み取り時に**整合性レベル**を選択できます。

結果整合性読み取り（Eventual Consistency）

  • デフォルト設定: 通常の読み取りは結果整合性
  • 特徴: レプリケーションが完了していない可能性があるため、最新データでない場合がある
  • コスト: RCU消費が少ない（強い整合性の半分）
  • 用途: リアルタイム性が不要なデータ取得

強い整合性読み取り（Strong Consistency）

  • 明示的指定: `ConsistentRead=true`パラメータで指定
  • 特徴: 必ず最新のデータを返す
  • コスト: RCU消費が多い（結果整合性の2倍）
  • 用途: 金融データなど、最新状態が必須のケース
強い整合性読み取りの例 
import boto3

# DynamoDBクライアント作成
dynamodb = boto3.resource('dynamodb')
table = dynamodb.Table('Users')

# 強い整合性読み取り
response = table.get_item(
    Key={'userId': 'U001'},
    ConsistentRead=True  # 強い整合性を指定
)

print(response['Item'])

4.6 セカンダリインデックス - GSIとLSI

プライマリキー以外の属性で検索したい場合、**セカンダリインデックス**を作成します。2つのタイプがあります。

グローバルセカンダリインデックス（GSI）

  • 特徴: 任意の属性をパーティションキーとして使用可能
  • 作成タイミング: テーブル作成後でも追加可能
  • キャパシティ: GSI独自のRCU/WCUを設定
  • 整合性: 結果整合性のみ
GSI作成例 
{
  "GlobalSecondaryIndexes": [
    {
      "IndexName": "EmailIndex",
      "KeySchema": [
        { "AttributeName": "email", "KeyType": "HASH" }
      ],
      "Projection": {
        "ProjectionType": "ALL"
      }
    }
  ]
}

ローカルセカンダリインデックス（LSI）

  • 特徴: パーティションキーは同じ、ソートキーのみ別属性を使用
  • 作成タイミング: テーブル作成時のみ
  • キャパシティ: テーブルのRCU/WCUを共有
  • 整合性: 強い整合性と結果整合性の両方をサポート

ヒント

GSIは柔軟性が高く後から追加可能なため、多くの場合GSIが推奨されます。LSIは強い整合性が必要で、テーブル設計時に要件が明確な場合に使用します。

理解度確認クイズ

  1. Q1. 3KBのアイテムを毎秒100回、強い整合性で読み取る場合、必要なRCUはいくつですか？

    回答を表示

    100 RCU
    計算式: 100回/秒 × (3KB / 4KB) = 100 × 1 = 100 RCU
    3KBは4KB以下なので、1RCUあたり1回読み取り可能です。

  2. Q2. GSIとLSIの最も大きな違いは何ですか？

    回答を表示

    最も大きな違いは作成タイミングとパーティションキーです。
    - GSI: テーブル作成後でも追加可能、任意の属性をパーティションキーに使用
    - LSI: テーブル作成時のみ、パーティションキーはテーブルと同じ

  3. Q3. どのようなケースでオンデマンドモードを選択すべきですか？

    回答を表示

    以下のケースでオンデマンドモードが適しています：
    - トラフィックが予測不可能なアプリケーション
    - 新規サービスでアクセスパターンが未知
    - スパイク的なアクセスが発生する
    - キャパシティ計画の工数を削減したい
