11.1 CI/CDの基本概念とAWS開発者ツール

CI/CD（Continuous Integration / Continuous Deployment）は、コードの変更を継続的にビルド、テスト、デプロイするプラクティスです。AWSは、コード管理からデプロイまでの全プロセスを自動化できるツールセットを提供しています。

変更] --> B[CodeCommit
または
GitHub] B --> C[CodeBuild
ビルド・テスト] C --> D[CodeDeploy
デプロイ] D --> E[本番環境] style B fill:#ff9900,color:#fff style C fill:#ff9900,color:#fff style D fill:#ff9900,color:#fff style E fill:#4a5568,color:#fff

AWSの主要なCI/CDサービス

  • AWS CodeCommit: マネージドなGitリポジトリサービス（GitHub代替）
  • AWS CodeBuild: フルマネージドなビルド・テストサービス
  • AWS CodeDeploy: EC2、Lambda、ECSへの自動デプロイサービス
  • AWS CodePipeline: CI/CDパイプラインのオーケストレーションサービス
  • AWS CodeArtifact: パッケージ管理サービス（npm、Maven等）

重要ポイント

DVA-C02試験では、各サービスの役割と組み合わせ方が問われます。CodePipelineが全体を統合し、CodeBuildでビルド、CodeDeployでデプロイという流れを理解しましょう。

11.2 AWS CodeCommitとGitHub連携

CodeCommitは、AWSが提供するフルマネージドなGitリポジトリサービスです。GitHubやGitLabと同様に、ソースコードのバージョン管理を行います。

CodeCommitの特徴

  • AWS IAMによるアクセス制御
  • リポジトリデータの暗号化（保存時、転送時）
  • Pull Request、コードレビュー機能
  • CloudWatch EventsやLambdaとの統合
  • 無料利用枠: アクティブユーザー5人まで

CodeCommit vs GitHub

サービス選択} A -->|AWS統合重視| B[CodeCommit] A -->|オープンソース
コミュニティ重視| C[GitHub] B --> D[IAM認証] B --> E[VPC内からアクセス可] C --> F[OAuth認証] C --> G[GitHub Actions] style B fill:#ff9900,color:#fff style C fill:#2ea44f,color:#fff

ヒント: GitHubとの統合

CodePipelineはGitHubとの直接統合をサポートしており、GitHubリポジトリをソースとして使用できます。多くの組織では、既存のGitHub環境を活用しながらAWSのCI/CDツールを利用しています。

CodeCommitへのアクセス方法

11.3 AWS CodeBuild - ビルドとテストの自動化

CodeBuildは、ソースコードをコンパイルし、テストを実行し、デプロイ可能なアーティファクトを生成するフルマネージドなビルドサービスです。

CodeBuildの動作フロー

sequenceDiagram participant S as ソースリポジトリ participant CB as CodeBuild participant ECR as Docker Image participant S3 as S3 Bucket S->>CB: コード変更をトリガー CB->>CB: buildspec.yml読み込み CB->>CB: ビルド環境起動 CB->>CB: ビルド・テスト実行 CB->>ECR: Dockerイメージをプッシュ CB->>S3: アーティファクトをアップロード CB->>CB: ビルド環境削除

buildspec.ymlの基本構造

CodeBuildは、buildspec.ymlファイルに記述されたビルド手順を実行します。このファイルはリポジトリのルートに配置します。

buildspec.ymlの例（Node.jsアプリ） 
version: 0.2

phases:
  # インストールフェーズ: ランタイム環境のセットアップ
  install:
    runtime-versions:
      nodejs: 18
    commands:
      - echo "Installing dependencies..."
      - npm install

  # ビルド前フェーズ: テストやリンターの実行
  pre_build:
    commands:
      - echo "Running tests..."
      - npm test
      - echo "Running linter..."
      - npm run lint

  # ビルドフェーズ: アプリケーションのビルド
  build:
    commands:
      - echo "Building application..."
      - npm run build

  # ビルド後フェーズ: アーティファクトの準備
  post_build:
    commands:
      - echo "Build completed on $(date)"

# 成果物の定義
artifacts:
  files:
    - '**/*'
  base-directory: 'build'

環境変数とSecrets Managerの統合

buildspec.ymlでのシークレット参照 
version: 0.2

env:
  # プレーンテキストの環境変数
  variables:
    NODE_ENV: "production"

  # Parameter Storeから取得
  parameter-store:
    DB_HOST: /myapp/db/host
    DB_PORT: /myapp/db/port

  # Secrets Managerから取得
  secrets-manager:
    DB_PASSWORD: myapp/db:password
    API_KEY: myapp/api:key

phases:
  build:
    commands:
      - echo "Connecting to $DB_HOST:$DB_PORT"
      - npm run build

注意事項

buildspec.ymlに直接シークレット（APIキー、パスワード等）を書いてはいけません。必ずSecrets ManagerまたはParameter Storeを使用してください。

11.4 AWS CodeDeploy - デプロイ戦略の実装

CodeDeployは、EC2インスタンス、Lambda関数、ECSサービスへのアプリケーションデプロイを自動化するサービスです。段階的なデプロイやロールバックをサポートします。

デプロイ対象とデプロイタイプ

デプロイ戦略の詳細

ブルー/グリーンデプロイ

新旧2つの環境を用意し、トラフィックを一度に切り替えるデプロイ方法です。

  • メリット: ダウンタイムなし、即座にロールバック可能
  • デメリット: リソースコストが2倍（一時的）
  • 適用例: EC2、ECS、Lambda

Canary（カナリア）デプロイ

最初に一部のトラフィックのみ新バージョンに流し、問題なければ全体に展開する方法です。

  • Canary10Percent5Minutes: 10%のトラフィックを5分間流し、その後100%に移行
  • Canary10Percent10Minutes: 10%のトラフィックを10分間流し、その後100%に移行
  • 適用例: Lambda関数のエイリアス更新

Linear（段階的）デプロイ

一定間隔でトラフィックを段階的に増やしていく方法です。

  • Linear10PercentEvery1Minute: 1分ごとに10%ずつ増やす（10分で完了）
  • Linear10PercentEvery3Minutes: 3分ごとに10%ずつ増やす（30分で完了）
  • 適用例: Lambda関数のエイリアス更新

Lambda関数のデプロイ設定

appspec.yml（Lambda用） 
version: 0.0
Resources:
  - myLambdaFunction:
      Type: AWS::Lambda::Function
      Properties:
        Name: "my-function"
        Alias: "live"
        CurrentVersion: "1"
        TargetVersion: "2"

# デプロイ設定
Hooks:
  - BeforeAllowTraffic: "BeforeAllowTrafficHookFunction"
  - AfterAllowTraffic: "AfterAllowTrafficHookFunction"

11.5 AWS CodePipeline - CI/CDパイプラインの構築

CodePipelineは、ソースコードの変更検知から本番デプロイまでのワークフロー全体をオーケストレーションするサービスです。

パイプラインの基本構造

flowchart LR A[ソースステージ
CodeCommit/GitHub] --> B[ビルドステージ
CodeBuild] B --> C[承認ステージ
手動承認] C --> D[デプロイステージ
CodeDeploy] style A fill:#ff9900,color:#fff style B fill:#ff9900,color:#fff style C fill:#ffc107,color:#000 style D fill:#ff9900,color:#fff

パイプラインステージの種類

  • ソースステージ: CodeCommit、GitHub、S3からソースコードを取得
  • ビルドステージ: CodeBuildでビルド・テストを実行
  • テストステージ: 統合テスト、E2Eテストを実行
  • 承認ステージ: 手動承認（任意）
  • デプロイステージ: CodeDeployでデプロイ

手動承認ステージの設定

本番環境へのデプロイ前に、人間による承認を挟むことができます。

AWS CLIでのパイプライン作成（抜粋） 
{
  "name": "ApprovalStage",
  "actions": [
    {
      "name": "ManualApproval",
      "actionTypeId": {
        "category": "Approval",
        "owner": "AWS",
        "provider": "Manual",
        "version": "1"
      },
      "configuration": {
        "CustomData": "本番環境へのデプロイを承認してください",
        "NotificationArn": "arn:aws:sns:region:account-id:topic-name"
      }
    }
  ]
}

重要ポイント

手動承認ステージを設定すると、SNSトピック経由でメール通知を送信できます。承認または却下は、CodePipelineコンソールから行います。

11.6 実習: Lambda関数のCI/CDパイプライン構築

実習: Lambda関数の自動デプロイパイプライン

CodePipelineを使用して、GitHub → CodeBuild → Lambda関数デプロイの自動化パイプラインを構築します。

ステップ1: Lambda関数の準備

index.js 
// シンプルなLambda関数
exports.handler = async (event) => {
    const response = {
        statusCode: 200,
        body: JSON.stringify({
            message: 'Hello from Lambda v2.0!',
            timestamp: new Date().toISOString()
        })
    };
    return response;
};

ステップ2: buildspec.ymlの作成

buildspec.yml 
version: 0.2

phases:
  install:
    runtime-versions:
      nodejs: 18
  build:
    commands:
      - echo "Packaging Lambda function..."
      - zip function.zip index.js

artifacts:
  files:
    - function.zip

ステップ3: CodePipelineの作成

  1. CodePipelineコンソールで「パイプラインを作成」
  2. ソースプロバイダー: GitHub（またはCodeCommit）
  3. ビルドプロバイダー: CodeBuild
  4. デプロイプロバイダー: Lambda
  5. デプロイ設定: Canary10Percent5Minutesを選択

動作確認

  1. GitHubリポジトリにコードをプッシュ
  2. CodePipelineが自動的にトリガーされることを確認
  3. 各ステージ（ソース→ビルド→デプロイ）の進行を確認
  4. Lambda関数が正常にデプロイされたことを確認
  5. CloudWatch Logsでログを確認

11.7 ベストプラクティスとトラブルシューティング

CI/CDのベストプラクティス

  • 小さく頻繁にデプロイする: 大きな変更を一度にデプロイするとリスクが高まります
  • 自動テストを組み込む: ビルドステージにユニットテスト、統合テストを含める
  • 段階的デプロイを活用する: カナリアデプロイで影響を最小化
  • ロールバック手順を用意する: 問題発生時の切り戻しを計画
  • パイプラインをコードで管理する: CloudFormationやCDKでパイプラインを定義

トラブルシューティング

問題: CodeBuildでビルドが失敗する

  • buildspec.ymlの構文エラーを確認
  • CodeBuildのサービスロールに必要な権限があるか確認
  • CloudWatch Logsでビルドログを確認

問題: CodeDeployでデプロイが失敗する

  • appspec.ymlの構文エラーを確認
  • Lambda関数のIAMロールを確認
  • デプロイ設定のタイムアウト時間を確認

問題: パイプラインが自動トリガーされない

  • GitHub Webhookが正しく設定されているか確認
  • CodePipelineのサービスロールにソースリポジトリへのアクセス権限があるか確認
  • CloudTrailでイベント履歴を確認

理解度確認クイズ

  1. Q1. CodeBuildでビルド手順を定義するファイルは何ですか？

    回答を表示

    buildspec.yml
    buildspec.ymlファイルには、install、pre_build、build、post_buildの各フェーズにおける実行コマンドを記述します。このファイルはリポジトリのルートに配置します。

  2. Q2. Lambda関数のデプロイで、10%のトラフィックを5分間流してから100%に移行するデプロイ戦略はどれですか？

    回答を表示

    Canary10Percent5Minutes
    カナリアデプロイは、最初に一部のトラフィックのみを新バージョンに流し、問題がなければ全体に展開する方式です。Canary10Percent5Minutesは、DVA-C02試験でよく問われるデプロイ設定の1つです。

  3. Q3. CodePipelineの手動承認ステージで通知を送信するAWSサービスはどれですか？

    回答を表示

    Amazon SNS（Simple Notification Service）
    手動承認ステージでは、SNSトピックを指定することで、承認が必要になったタイミングでメール通知を送信できます。承認または却下は、CodePipelineコンソールから行います。
