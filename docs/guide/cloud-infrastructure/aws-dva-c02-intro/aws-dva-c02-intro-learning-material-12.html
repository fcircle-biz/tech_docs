12.1 CloudWatch Logs - ログ管理の基礎

CloudWatch Logsは、AWSアプリケーションのログを一元管理し、リアルタイムで監視・分析するためのサービスです。Lambda関数、ECSコンテナ、EC2インスタンスなど、さまざまなAWSリソースからログを収集できます。

ロググループとログストリーム

  • ロググループ: 関連するログストリームをまとめる論理的なコンテナ（例: /aws/lambda/my-function）
  • ログストリーム: 同じソースからの一連のログイベント（例: Lambda関数の各実行インスタンス）
  • ログイベント: タイムスタンプとメッセージを含む個別のログエントリ
  • 保持期間: ログの保存期間（1日〜無期限、デフォルトは無期限）

重要ポイント

Lambda関数は自動的にCloudWatch Logsにログを出力します。関数にCloudWatch Logs書き込み権限（logs:CreateLogGroup、logs:CreateLogStream、logs:PutLogEvents）を持つIAMロールをアタッチする必要があります。

12.2 構造化ログとJSON形式

構造化ログ（JSON形式）を使用することで、CloudWatch Logs Insightsでのクエリが容易になり、ログ分析の効率が大幅に向上します。

Lambda関数での構造化ログ実装（Python）

Python 
import json
import logging

# 構造化ログの設定
logger = logging.getLogger()
logger.setLevel(logging.INFO)

def lambda_handler(event, context):
    # 構造化ログ（JSON形式）で出力
    log_entry = {
        "level": "INFO",
        "message": "処理開始",
        "request_id": context.request_id,
        "user_id": event.get("userId"),
        "action": "process_order"
    }
    logger.info(json.dumps(log_entry))

    try:
        # 処理実行
        result = process_order(event)

        # 成功ログ
        success_log = {
            "level": "INFO",
            "message": "処理成功",
            "request_id": context.request_id,
            "order_id": result["order_id"]
        }
        logger.info(json.dumps(success_log))

        return {"statusCode": 200, "body": json.dumps(result)}
    except Exception as e:
        # エラーログ
        error_log = {
            "level": "ERROR",
            "message": "処理失敗",
            "request_id": context.request_id,
            "error": str(e)
        }
        logger.error(json.dumps(error_log))
        raise

ヒント: 構造化ログのメリット

JSON形式のログは、CloudWatch Logs Insightsで簡単にフィルタ・集計できます。例えば、特定のuser_idやerrorレベルだけを抽出するクエリが書けます。

12.3 CloudWatch Logs Insights - ログクエリ

CloudWatch Logs Insightsは、大量のログデータから必要な情報を素早く検索・分析できる強力なクエリ機能です。

よく使うクエリパターン

エラーログの抽出

CloudWatch Logs Insights Query 
fields @timestamp, @message
| filter @message like /ERROR/
| sort @timestamp desc
| limit 100

直近100件のエラーログを時系列で表示します。

レスポンスタイムの統計

CloudWatch Logs Insights Query 
fields @timestamp, @duration
| stats avg(@duration) as avg_duration,
        max(@duration) as max_duration,
        pct(@duration, 95) as p95_duration
by bin(5m)

5分間隔で平均、最大、95パーセンタイルのレスポンスタイムを計算します。

特定ユーザーの操作履歴

CloudWatch Logs Insights Query 
fields @timestamp, user_id, action, order_id
| filter user_id = "user123"
| sort @timestamp desc
| limit 50

特定ユーザーの直近50件の操作履歴を取得します（構造化ログが前提）。

12.4 CloudWatchメトリクスとアラーム

CloudWatchメトリクスは、AWSリソースのパフォーマンスデータを収集し、可視化します。カスタムメトリクスを発行することで、アプリケーション固有のメトリクスも監視できます。

カスタムメトリクスの発行（Python）

Python 
import boto3

# CloudWatchクライアント
cloudwatch = boto3.client('cloudwatch')

def publish_custom_metric(metric_name, value, unit='Count'):
    """
    カスタムメトリクスをCloudWatchに発行
    """
    cloudwatch.put_metric_data(
        Namespace='MyApp/Orders',  # カスタム名前空間
        MetricData=[
            {
                'MetricName': metric_name,
                'Value': value,
                'Unit': unit,
                'Dimensions': [
                    {
                        'Name': 'Environment',
                        'Value': 'Production'
                    }
                ]
            }
        ]
    )

# 使用例: 注文完了数を記録
publish_custom_metric('OrdersCompleted', 1, 'Count')

12.5 AWS X-Ray - 分散トレーシングの基礎

AWS X-Rayは、マイクロサービスやサーバーレスアーキテクチャにおいて、リクエストの流れを可視化し、ボトルネックやエラーを特定するための分散トレーシングサービスです。

flowchart LR C[クライアント] -->|1. リクエスト| AG[API Gateway] AG -->|2. 呼び出し| L1[Lambda 1] L1 -->|3. クエリ| DB[DynamoDB] L1 -->|4. 呼び出し| L2[Lambda 2] L2 -->|5. 送信| SQS[SQS] subgraph xray["X-Ray トレース"] direction TB T1[セグメント: API Gateway] T2[サブセグメント: Lambda 1] T3[サブセグメント: DynamoDB] T4[サブセグメント: Lambda 2] T5[サブセグメント: SQS] T1 --> T2 T2 --> T3 T2 --> T4 T4 --> T5 end

X-Rayの主要概念

  • トレース: エンドツーエンドのリクエストの流れ全体
  • セグメント: 各サービスでの処理単位（Lambda関数、API Gateway等）
  • サブセグメント: セグメント内の詳細な処理（DynamoDBクエリ、HTTP呼び出し等）
  • アノテーション: フィルタリング可能なキーバリューペア（indexed: true）
  • メタデータ: 追加情報（indexed: false）

重要ポイント

Lambda関数でX-Rayを有効化するには、関数の設定で「アクティブトレーシングを有効化」をオンにし、実行ロールに「AWSXRayDaemonWriteAccess」ポリシーをアタッチする必要があります。

12.6 X-Ray SDKの組み込み

Lambda関数にX-Ray SDKを組み込むことで、詳細なトレース情報を収集できます。

Lambda関数へのX-Ray SDK組み込み（Python）

Python 
import boto3
from aws_xray_sdk.core import xray_recorder
from aws_xray_sdk.core import patch_all

# AWSサービスのSDKをパッチ（自動トレース）
patch_all()

dynamodb = boto3.resource('dynamodb')
table = dynamodb.Table('Orders')

def lambda_handler(event, context):
    # カスタムサブセグメントの作成
    with xray_recorder.capture('ProcessOrder'):
        order_id = event['order_id']

        # DynamoDBクエリ（自動的にトレース）
        response = table.get_item(Key={'order_id': order_id})

        # アノテーション追加（検索可能）
        xray_recorder.put_annotation('order_id', order_id)
        xray_recorder.put_annotation('status', 'success')

        # メタデータ追加（詳細情報）
        xray_recorder.put_metadata('order_details', response['Item'])

    return {"statusCode": 200, "body": "Success"}

トレース分析のポイント

  • レスポンスタイム分布: 遅いリクエストを特定してボトルネックを発見
  • エラー率: どのサービスでエラーが発生しているか可視化
  • スロットリング: DynamoDBやAPI Gatewayのスロットリングを検出
  • フィルタ: アノテーションを使用して特定条件のトレースのみ表示

ヒント: トレースフィルタの活用

X-Rayコンソールで「annotation.status = "error"」のようなフィルタ式を使用すると、エラーが発生したリクエストのみを抽出できます。パフォーマンス調査に非常に有効です。

理解度確認クイズ

  1. Q1. CloudWatch Logsで構造化ログ（JSON形式）を使用するメリットは何ですか？

    回答を表示

    CloudWatch Logs Insightsで効率的にクエリできること
    JSON形式のログは、フィールド名を指定してフィルタ・集計が可能です。例えば「user_id = "user123"」のように特定条件でログを抽出でき、分析が容易になります。

  2. Q2. X-Rayのアノテーションとメタデータの違いは何ですか？

    回答を表示

    アノテーションは検索可能、メタデータは検索不可
    アノテーションはX-Rayコンソールでフィルタリングに使用できるキーバリューペア（indexed: true）です。一方、メタデータは追加の詳細情報を保存しますが検索には使用できません（indexed: false）。

  3. Q3. Lambda関数でX-Rayを有効化するために必要な設定は何ですか？

    回答を表示

    アクティブトレーシングの有効化とIAMロールへのポリシーアタッチ
    Lambda関数の設定で「アクティブトレーシングを有効化」をオンにし、実行ロールに「AWSXRayDaemonWriteAccess」ポリシーをアタッチする必要があります。
