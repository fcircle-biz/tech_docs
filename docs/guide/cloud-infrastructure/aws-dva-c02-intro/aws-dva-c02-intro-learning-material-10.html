アプリケーションセキュリティ - Secrets Manager、KMS、パラメータストア

アプリケーション開発におけるセキュリティのベストプラクティスを学びます。シークレット管理、暗号化、環境変数の安全な取り扱い方法を習得します。

この章で学ぶこと

  • AWS Secrets Managerによるシークレット管理
  • シークレットのローテーション設定
  • AWS KMSの暗号化キー管理
  • カスタマーマネージドキー（CMK）とAWSマネージドキー
  • Parameter Storeとの使い分け
  • Lambda環境変数の暗号化
  • コード内でのセキュリティベストプラクティス

10.1 アプリケーションセキュリティの基本原則

アプリケーション開発において、機密情報（データベース認証情報、APIキー、暗号化キーなど）を適切に管理することは最も重要なセキュリティ対策の一つです。これらの情報を**コード内にハードコーディング**したり、**平文で保存**することは重大なセキュリティリスクとなります。

絶対にやってはいけないこと

  • コード内にパスワードやAPIキーをハードコーディング
  • GitHub等のバージョン管理システムにシークレットをコミット
  • 環境変数に平文で機密情報を保存（暗号化なし）
  • アプリケーションログにシークレットを出力

Secrets Manager

データベースパスワード、APIキーなど**動的に変更が必要**なシークレットを管理。自動ローテーション機能を提供。

Parameter Store

設定値、環境変数など**静的な設定情報**を管理。無料枠が大きく、コスト効率が高い。

KMS

暗号化キーを一元管理。Secrets ManagerやParameter Storeの暗号化にも使用される。

10.2 AWS Secrets Manager - シークレットの保管と自動ローテーション

AWS Secrets Managerは、データベース認証情報、APIキー、OAuth トークンなどの機密情報を安全に保管し、自動的にローテーション（定期的な変更）する機能を提供します。

Secrets Managerの主な機能

  • 自動ローテーション: RDS、Redshift、DocumentDBのパスワードを自動的に変更
  • 暗号化保存: KMSを使用してシークレットを暗号化
  • きめ細かなアクセス制御: IAMポリシーとリソースポリシーによる制御
  • 監査ログ: CloudTrailでシークレットアクセスを記録
  • バージョン管理: シークレットの変更履歴を保持

シークレットの作成

AWS CLI - Secrets Manager シークレット作成 
# データベース認証情報をシークレットとして保存
aws secretsmanager create-secret \
    --name prod/myapp/db-password \
    --description "Production database password" \
    --secret-string '{
        "username": "admin",
        "password": "MyS3cr3tP@ssw0rd",
        "engine": "mysql",
        "host": "mydb.abc123.us-east-1.rds.amazonaws.com",
        "port": 3306
    }'

Lambda関数からシークレットを取得

Python - Secrets Manager から認証情報を取得 
import boto3
import json
from botocore.exceptions import ClientError

def get_secret(secret_name):
    """Secrets Managerからシークレットを取得"""
    client = boto3.client('secretsmanager', region_name='us-east-1')

    try:
        response = client.get_secret_value(SecretId=secret_name)
        # JSON形式のシークレットをパース
        return json.loads(response['SecretString'])
    except ClientError as e:
        # エラーハンドリング
        raise e

def lambda_handler(event, context):
    # データベース認証情報を取得
    db_secret = get_secret('prod/myapp/db-password')

    # 取得した認証情報でデータベース接続
    username = db_secret['username']
    password = db_secret['password']
    host = db_secret['host']
    # ... データベース接続処理

ベストプラクティス: シークレットのキャッシュ

Lambda関数では、Secrets Managerへのアクセス回数を減らすため、取得したシークレットをグローバル変数にキャッシュすることが推奨されます。ただし、ローテーション期間を考慮してTTLを設定しましょう。

自動ローテーションの仕組み

sequenceDiagram participant SM as Secrets Manager participant LR as Lambda (ローテーション関数) participant RDS as RDS Database SM->>SM: ローテーション期間到達 SM->>LR: ローテーション関数を呼び出し LR->>SM: 新しいパスワードを生成 LR->>RDS: データベースのパスワードを変更 RDS-->>LR: 変更完了 LR->>SM: 新しいシークレット値を保存 SM->>SM: 新バージョンを AWSCURRENT にマーク SM->>SM: 旧バージョンを AWSPREVIOUS に移動

ローテーション戦略

RDS、Redshift、DocumentDBのシークレットは、AWS提供のローテーション関数を使用可能。カスタムシークレット（APIキー等）は、独自のLambda関数を実装する必要があります。

10.3 AWS KMS - 暗号化キーの一元管理

AWS Key Management Service (KMS)は、暗号化キーの作成と管理を一元化するサービスです。S3、EBS、RDS、Secrets Managerなど、多くのAWSサービスがKMSと統合されています。

KMSキーの種類

AWSマネージドキー

  • AWS側が自動で作成・管理
  • キー名: aws/s3, aws/rds など
  • 無料で使用可能
  • ローテーションは自動（3年ごと）
  • 削除不可

カスタマーマネージドキー (CMK)

  • ユーザーが作成・管理
  • きめ細かなアクセス制御が可能
  • 月額 $1/キー + API利用料
  • 自動ローテーション設定可能（1年ごと）
  • 削除可能（削除待機期間7-30日）

KMSキーの作成

AWS CLI - CMKキー作成 
# カスタマーマネージドキーを作成
aws kms create-key \
    --description "Key for encrypting application secrets" \
    --key-usage ENCRYPT_DECRYPT \
    --origin AWS_KMS

# エイリアスを作成（キーを識別しやすくする）
aws kms create-alias \
    --alias-name alias/myapp-secrets \
    --target-key-id 1234abcd-12ab-34cd-56ef-1234567890ab

KMSによるデータの暗号化・復号化

sequenceDiagram participant App as アプリケーション participant KMS as AWS KMS Note over App,KMS: データ暗号化フロー App->>KMS: Encrypt API (平文データ + キーID) KMS->>KMS: CMKで暗号化 KMS-->>App: 暗号化済みデータ App->>App: 暗号化データを保存 Note over App,KMS: データ復号化フロー App->>KMS: Decrypt API (暗号化データ) KMS->>KMS: CMKで復号化 KMS-->>App: 平文データ
Python - KMS による暗号化・復号化 
import boto3
import base64

kms_client = boto3.client('kms')

def encrypt_data(plaintext, key_id):
    """KMSでデータを暗号化"""
    response = kms_client.encrypt(
        KeyId=key_id,
        Plaintext=plaintext
    )
    # バイナリデータをBase64エンコード
    return base64.b64encode(response['CiphertextBlob']).decode()

def decrypt_data(ciphertext_base64):
    """KMSでデータを復号化"""
    # Base64デコード
    ciphertext = base64.b64decode(ciphertext_base64)
    response = kms_client.decrypt(CiphertextBlob=ciphertext)
    return response['Plaintext'].decode()

# 使用例
api_key = "my-secret-api-key-12345"
encrypted = encrypt_data(api_key, 'alias/myapp-secrets')
decrypted = decrypt_data(encrypted)

Envelope Encryption（エンベロープ暗号化）

KMSは4KB以上のデータを直接暗号化できません。大きなデータはデータキーで暗号化し、そのデータキーをKMSのCMKで暗号化する「エンベロープ暗号化」パターンを使用します。

10.4 Systems Manager Parameter Store - 設定値の一元管理

AWS Systems Manager Parameter Storeは、設定値や機密情報を階層的に保存できるサービスです。Secrets Managerと似ていますが、より低コストで、自動ローテーション機能はありません。

Parameter Store vs Secrets Manager

機能 Parameter Store Secrets Manager
料金 10,000パラメータまで無料 $0.40/シークレット/月
自動ローテーション なし あり（RDS等に対応）
パラメータサイズ 標準: 4KB、高度: 8KB 最大 64KB
暗号化 SecureString（KMS使用） 常にKMSで暗号化
適用例 環境変数、設定値 DB認証情報、APIキー

パラメータの階層構造

/myapp/
├── dev/
│   ├── db-host            (String)
│   ├── db-port            (String)
│   └── api-key            (SecureString - KMS暗号化)
├── staging/
│   ├── db-host
│   ├── db-port
│   └── api-key
└── prod/
    ├── db-host
    ├── db-port
    └── api-key

パラメータの作成と取得

AWS CLI - Parameter Store パラメータ作成 
# 平文パラメータ
aws ssm put-parameter \
    --name /myapp/prod/db-host \
    --value "prod-db.abc123.us-east-1.rds.amazonaws.com" \
    --type String

# 暗号化パラメータ (SecureString)
aws ssm put-parameter \
    --name /myapp/prod/api-key \
    --value "sk-1234567890abcdef" \
    --type SecureString \
    --key-id alias/myapp-secrets
Python - Parameter Store からパラメータ取得 
import boto3

ssm_client = boto3.client('ssm')

def get_parameter(name, with_decryption=True):
    """Parameter Storeからパラメータを取得"""
    response = ssm_client.get_parameter(
        Name=name,
        WithDecryption=with_decryption  # SecureStringを復号化
    )
    return response['Parameter']['Value']

# 使用例
db_host = get_parameter('/myapp/prod/db-host')
api_key = get_parameter('/myapp/prod/api-key')  # 自動復号化

推奨: パラメータの階層取得

get_parameters_by_path APIを使用すると、階層下のすべてのパラメータを一度に取得できます。Lambda起動時に環境全体の設定を一括取得する際に便利です。

10.5 Lambda環境変数の暗号化

Lambda関数の環境変数は、デフォルトでAWSマネージドキーで暗号化されますが、より高度なセキュリティ要件がある場合は、カスタマーマネージドキー（CMK）を使用して暗号化できます。

AWS CLI - Lambda環境変数をCMKで暗号化 
# Lambda関数の環境変数を更新（CMKで暗号化）
aws lambda update-function-configuration \
    --function-name my-function \
    --environment Variables={API_KEY=abc123,DB_HOST=mydb.example.com} \
    --kms-key-arn arn:aws:kms:us-east-1:123456789012:key/1234abcd-12ab-34cd-56ef-1234567890ab

注意: 環境変数の制約

  • 環境変数は実行時に平文で復号化されます
  • ログ出力時に環境変数を表示しないよう注意
  • 機密性の高い情報は Secrets Manager を使用推奨

環境変数 vs Secrets Manager の使い分け

環境変数を使用する場合

  • 静的な設定値（リージョン名、バケット名）
  • アプリケーション設定（ログレベル、タイムアウト）
  • 頻繁に変更されない値
  • 複数Lambda間で共有しない値

Secrets Manager を使用する場合

  • データベース認証情報
  • APIキー、OAuthトークン
  • 自動ローテーションが必要な値
  • 複数サービス間で共有する機密情報

10.6 セキュリティベストプラクティス

アプリケーション開発において、機密情報を安全に扱うためのベストプラクティスをまとめます。

1. 最小権限の原則

IAMポリシー例 - Secrets Manager へのアクセス制限 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "secretsmanager:GetSecretValue"
      ],
      "Resource": [
        "arn:aws:secretsmanager:us-east-1:123456789012:secret:prod/myapp/*"
      ]
    },
    {
      "Effect": "Allow",
      "Action": [
        "kms:Decrypt"
      ],
      "Resource": "arn:aws:kms:us-east-1:123456789012:key/1234abcd-12ab-34cd-56ef-1234567890ab"
    }
  ]
}

2. ログにシークレットを出力しない

Python - シークレットをマスクしたログ出力 
import logging

logger = logging.getLogger()
logger.setLevel(logging.INFO)

def lambda_handler(event, context):
    api_key = get_secret('prod/myapp/api-key')

    # ❌ 悪い例: シークレットをログに出力
    # logger.info(f"API Key: {api_key}")

    # ✅ 良い例: シークレットをマスク
    logger.info("API Key retrieved successfully")
    logger.info(f"API Key prefix: {api_key[:4]}****")  # 先頭4文字のみ表示

3. シークレットのローテーション

  • RDS/Redshift: Secrets Managerの自動ローテーションを有効化（30-90日ごと推奨）
  • APIキー: カスタムLambda関数でローテーションロジックを実装
  • KMSキー: 自動ローテーションを有効化（1年ごと）

4. CloudTrailによる監査

すべてのシークレットアクセスをCloudTrailで記録し、定期的に監査します。

トラブルシューティング

問題: Lambda関数がSecrets Managerにアクセスできない

解決策: Lambda実行ロールに secretsmanager:GetSecretValue と kms:Decrypt 権限を付与。リソースARNを確認。

問題: Parameter Store の SecureString が復号化されない

解決策: get_parameter APIで WithDecryption=True を指定。Lambda実行ロールにKMS復号化権限を付与。

実習: Secrets Managerを使用したLambda関数

  1. 1. KMSキーの作成
    カスタマーマネージドキーを作成し、エイリアス alias/myapp-secrets を設定
  2. 2. Secrets Managerでシークレット作成
    JSON形式でデータベース認証情報を保存（username, password, host, port）
  3. 3. Lambda関数の作成
    Secrets Managerからシークレットを取得し、CloudWatch Logsに接続情報（hostのみ）を出力
  4. 4. IAMロールの設定
    Lambda実行ロールに secretsmanager:GetSecretValue と kms:Decrypt 権限を付与
  5. 5. 動作確認
    Lambda関数をテスト実行し、シークレットが正しく取得できることを確認

理解度確認クイズ

  1. Q1. Secrets Manager と Parameter Store の使い分けとして適切なものはどれですか?

    回答を表示

    Secrets Manager: データベース認証情報やAPIキーなど、自動ローテーションが必要な機密情報
    Parameter Store: 環境変数や設定値など、静的な設定情報（コスト効率重視）
    Secrets Managerは自動ローテーション機能があり、RDS/Redshift/DocumentDBとの統合が容易です。Parameter Storeは無料枠が大きく、コスト効率が高いです。

  2. Q2. Lambda関数がKMSで暗号化されたSecrets Managerのシークレットにアクセスするために必要なIAM権限はどれですか?

    回答を表示

    1. secretsmanager:GetSecretValue（シークレット取得）
    2. kms:Decrypt（KMS復号化）
    Secrets Managerのシークレットは必ずKMSで暗号化されているため、取得時にKMS復号化権限が必要です。

  3. Q3. Lambda環境変数に機密情報を保存する際のベストプラクティスはどれですか?

    回答を表示

    環境変数には静的な設定値のみを保存し、機密性の高い情報はSecrets Managerを使用する
    Lambda環境変数はCMKで暗号化できますが、実行時には平文で復号化されます。データベース認証情報やAPIキーなど、ローテーションが必要な機密情報はSecrets Managerを使用することが推奨されます。
