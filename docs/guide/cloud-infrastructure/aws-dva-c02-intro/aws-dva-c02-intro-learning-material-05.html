5.1 Query操作とScan操作の使い分け

DynamoDBでデータを検索する方法には、QueryとScanの2つがあります。それぞれ異なる特性を持ち、適切な使い分けがパフォーマンスとコストに大きく影響します。

Query操作の特徴

Query操作は、パーティションキーを指定して特定のパーティション内のアイテムを取得します。ソートキーを使用して範囲検索やフィルタリングも可能です。

flowchart LR subgraph Query["Query操作（効率的）"] direction TB Q1["パーティションキー指定必須"] Q2["ソートキーで範囲検索可能"] Q3["インデックスを利用"] Q4["消費RCU少ない"] end subgraph Scan["Scan操作（非効率）"] direction TB S1["全テーブル/パーティションスキャン"] S2["フィルター式で事後絞り込み"] S3["並列スキャン可能"] S4["消費RCU多い"] end Query -.推奨.- Scan

重要ポイント

Query操作はパーティションキー指定が必須で、特定のパーティション内のみを検索するため効率的です。一方、Scan操作は全テーブルを読み込むため、大規模テーブルではコストとレイテンシが増大します。

Query操作の例

Python 
# UserIdでクエリし、Timestamp範囲で絞り込む例
import boto3
from boto3.dynamodb.conditions import Key

dynamodb = boto3.resource('dynamodb')
table = dynamodb.Table('Orders')

# パーティションキー（UserId）指定は必須
# ソートキー（Timestamp）で範囲検索
response = table.query(
    KeyConditionExpression=Key('UserId').eq('user123') &
                          Key('Timestamp').between('2025-01-01', '2025-01-31')
)

items = response['Items']
print(f"取得アイテム数: {len(items)}")

Scan操作の例と注意点

Scan操作は全テーブルスキャン後にフィルタリングするため、**フィルタ前のアイテム数に基づいてRCUが消費**されます。

Python 
# Scan操作（非推奨: 全テーブルスキャン）
response = table.scan(
    FilterExpression=Attr('Status').eq('Active')
)

# 並列スキャンでパフォーマンス向上（大規模テーブル向け）
response = table.scan(
    FilterExpression=Attr('Status').eq('Active'),
    Segment=0,        # セグメント番号（0～TotalSegments-1）
    TotalSegments=4   # 並列度（4つのスレッドでスキャン）
)

注意事項

Scan操作は**最後の手段**として使用してください。可能な限りQueryを使用するようにテーブル設計（パーティションキー、GSI）を工夫しましょう。

5.2 フィルター式とプロジェクション式

DynamoDBでは、フィルター式（FilterExpression）で取得後にアイテムを絞り込み、プロジェクション式（ProjectionExpression）で返す属性を制限できます。

フィルター式（FilterExpression）

フィルター式は、QueryまたはScanで取得したアイテムを事後的にフィルタリングします。フィルタ前のアイテム数でRCUが消費される点に注意が必要です。

Python 
from boto3.dynamodb.conditions import Attr

# フィルター式の例（金額100以上のアイテム）
response = table.query(
    KeyConditionExpression=Key('UserId').eq('user123'),
    FilterExpression=Attr('Amount').gte(100)
)

# 複数条件の組み合わせ（AND条件）
response = table.query(
    KeyConditionExpression=Key('UserId').eq('user123'),
    FilterExpression=Attr('Amount').gte(100) & Attr('Status').eq('Completed')
)

# OR条件の例
response = table.query(
    KeyConditionExpression=Key('UserId').eq('user123'),
    FilterExpression=Attr('Status').eq('Pending') | Attr('Status').eq('InProgress')
)

プロジェクション式（ProjectionExpression）

プロジェクション式は、返す属性を制限することで**ネットワーク転送量を削減**します。RCU消費には影響しません。

Python 
# 特定の属性のみ取得
response = table.query(
    KeyConditionExpression=Key('UserId').eq('user123'),
    ProjectionExpression='OrderId, Amount, Status'
)

# 予約語を含む属性名はExpressionAttributeNamesで指定
response = table.query(
    KeyConditionExpression=Key('UserId').eq('user123'),
    ProjectionExpression='#dt, Amount',
    ExpressionAttributeNames={'#dt': 'Date'}  # Dateは予約語
)

5.3 ページネーションとLimit

DynamoDBのQueryとScanは、**1回のリクエストで最大1MB**のデータしか返しません。より多くのデータを取得する場合は、ページネーションを使用します。

LastEvaluatedKeyとExclusiveStartKey

レスポンスにLastEvaluatedKeyが含まれている場合、さらにアイテムが存在することを示します。次のリクエストでExclusiveStartKeyとして指定することで続きを取得できます。

Python 
# ページネーションの実装例
def query_all_items(user_id):
    items = []
    last_key = None

    while True:
        # ExclusiveStartKeyで続きから取得
        if last_key:
            response = table.query(
                KeyConditionExpression=Key('UserId').eq(user_id),
                ExclusiveStartKey=last_key
            )
        else:
            response = table.query(
                KeyConditionExpression=Key('UserId').eq(user_id)
            )

        items.extend(response['Items'])

        # LastEvaluatedKeyがなければ終了
        last_key = response.get('LastEvaluatedKey')
        if not last_key:
            break

    return items

Limitパラメータ

Limitパラメータは、**フィルタ前に評価されるアイテム数**を制限します。フィルタ後の結果数を保証するものではありません。

Python 
# Limit指定（フィルタ前に最大10アイテムを評価）
response = table.query(
    KeyConditionExpression=Key('UserId').eq('user123'),
    FilterExpression=Attr('Status').eq('Active'),
    Limit=10  # フィルタ前に10アイテムまで評価
)

# 注意: FilterExpressionで絞り込んだ結果、
# 実際に返却されるアイテムは10個未満になる可能性がある

5.4 DynamoDB Streams - 変更データキャプチャ

DynamoDB Streamsは、テーブルに対する変更（作成、更新、削除）を時系列順にキャプチャするストリームです。Lambda関数をトリガーしてリアルタイム処理を実装できます。

flowchart LR A["DynamoDBテーブル"] -->|変更イベント| B["DynamoDB Streams"] B --> C["Lambda関数
トリガー"] C --> D1["別テーブルへ集計"] C --> D2["検索エンジンへ同期"] C --> D3["通知送信（SNS）"] C --> D4["監査ログ保存（S3）"] style B fill:#fef3c7

Streamsの4つのビュータイプ

  • KEYS_ONLY: 変更されたアイテムのキーのみ
  • NEW_IMAGE: 変更後の新しいアイテム全体
  • OLD_IMAGE: 変更前の古いアイテム全体
  • NEW_AND_OLD_IMAGES: 変更前と変更後の両方
Python (Lambda Handler) 
import json

def lambda_handler(event, context):
    # DynamoDB Streamsイベント処理
    for record in event['Records']:
        event_name = record['eventName']  # INSERT, MODIFY, REMOVE

        # 新しいアイテム（INSERTまたはMODIFY時）
        if 'NewImage' in record['dynamodb']:
            new_item = record['dynamodb']['NewImage']
            print(f"新しいアイテム: {new_item}")

        # 古いアイテム（MODIFYまたはREMOVE時）
        if 'OldImage' in record['dynamodb']:
            old_item = record['dynamodb']['OldImage']
            print(f"古いアイテム: {old_item}")

        # イベントタイプ別処理
        if event_name == 'INSERT':
            handle_insert(new_item)
        elif event_name == 'MODIFY':
            handle_modify(old_item, new_item)
        elif event_name == 'REMOVE':
            handle_remove(old_item)

ユースケース例

DynamoDB Streamsは、テーブル間データ同期、検索エンジン（Elasticsearch）へのインデックス更新、リアルタイム分析、監査ログ作成など、多様なシナリオで活用できます。

5.5 トランザクション操作

DynamoDBのトランザクション操作により、複数のアイテムに対する書き込みまたは読み込みを**ACID特性を保証しながら**実行できます。

TransactWriteItems（書き込みトランザクション）

最大25個のアクション（Put、Update、Delete、ConditionCheck）をアトミックに実行します。すべて成功するかすべて失敗するかのいずれかです。

Python 
import boto3
from botocore.exceptions import ClientError

client = boto3.client('dynamodb')

try:
    # 銀行送金のトランザクション例
    response = client.transact_write_items(
        TransactItems=[
            {
                'Update': {
                    'TableName': 'Accounts',
                    'Key': {'AccountId': {'S': 'account-A'}},
                    'UpdateExpression': 'SET Balance = Balance - :amount',
                    'ExpressionAttributeValues': {':amount': {'N': '100'}},
                    'ConditionExpression': 'Balance >= :amount'
                }
            },
            {
                'Update': {
                    'TableName': 'Accounts',
                    'Key': {'AccountId': {'S': 'account-B'}},
                    'UpdateExpression': 'SET Balance = Balance + :amount',
                    'ExpressionAttributeValues': {':amount': {'N': '100'}}
                }
            }
        ]
    )
    print("トランザクション成功")
except ClientError as e:
    print(f"トランザクション失敗: {e.response['Error']['Message']}")

TransactGetItems（読み込みトランザクション）

最大25個のアイテムを**強い整合性**で同時に取得します。

Python 
# 複数アイテムの一貫性のある読み取り
response = client.transact_get_items(
    TransactItems=[
        {'Get': {'TableName': 'Accounts', 'Key': {'AccountId': {'S': 'account-A'}}}},
        {'Get': {'TableName': 'Accounts', 'Key': {'AccountId': {'S': 'account-B'}}}},
    ]
)

for item in response['Responses']:
    print(item['Item'])

コスト注意

トランザクション操作は通常の操作の2倍のWCU/RCUを消費します。頻繁に使用する場合はコストを考慮してください。

5.6 TTL（Time To Live）によるデータ自動削除

TTL機能は、指定した有効期限を過ぎたアイテムを自動的に削除します。セッションデータや一時データの管理に最適で、**追加コストなし**で利用できます。

TTLの設定

AWS CLI 
# TTL属性を設定（ExpirationTimeという属性名を使用）
aws dynamodb update-time-to-live \
  --table-name Sessions \
  --time-to-live-specification \
    "Enabled=true,AttributeName=ExpirationTime"
Python 
import time

# 現在時刻から1時間後に期限切れ
expiration_time = int(time.time()) + 3600

# アイテム挿入
table.put_item(
    Item={
        'SessionId': 'session-12345',
        'UserId': 'user123',
        'Data': 'session-data',
        'ExpirationTime': expiration_time  # TTL属性
    }
)

ヒント

TTLによる削除は即座ではなく、期限切れ後**48時間以内**にバックグラウンドで実行されます。削除イベントはDynamoDB Streamsでキャプチャできます。

5.7 DAX（DynamoDB Accelerator）

DAXは、DynamoDB専用のインメモリキャッシュサービスです。読み取り性能を劇的に向上させ、レイテンシを**ミリ秒からマイクロ秒**に短縮します。

マイクロ秒レイテンシ"] B -->|キャッシュミス| D["DynamoDBテーブル
ミリ秒レイテンシ"] D -->|データ取得| B B --> A style C fill:#a7f3d0 style D fill:#fef3c7

DAXの主な特徴

  • 読み取り集中型ワークロードに最適（書き込み集中型には不向き）
  • 完全マネージドで、DynamoDB APIと互換性あり
  • アイテムキャッシュとクエリキャッシュの2種類
  • TTL（デフォルト5分）によるキャッシュ有効期限管理
Python 
# DAXクライアントの使用例
import amazondax

# DAXクラスターエンドポイント
dax_endpoint = 'mycluster.abc123.dax-clusters.us-east-1.amazonaws.com:8111'
dax_client = amazondax.AmazonDaxClient(endpoint_url=dax_endpoint)

# DynamoDB互換APIでクエリ（キャッシュから取得）
response = dax_client.query(
    TableName='Products',
    KeyConditionExpression='CategoryId = :catid',
    ExpressionAttributeValues={':catid': {'S': 'electronics'}}
)

DAXの使いどころ

向いている: ゲームのリーダーボード、商品カタログの頻繁な参照、セッションストアの高速読み取り

向いていない: 書き込み集中型ワークロード、強い整合性が必須の場合

理解度確認クイズ

  1. Q1. Query操作とScan操作の最も重要な違いは何ですか？

    回答を表示

    Query操作はパーティションキー指定が必須で、特定のパーティション内のみを効率的に検索します。一方、Scan操作は全テーブルをスキャンするため、大規模テーブルではRCU消費が多く、レイテンシも増大します。

  2. Q2. DynamoDB Streamsの主なユースケースを3つ挙げてください。

    回答を表示

    1. テーブル間データ同期: 別テーブルへの集計データ作成
    2. 検索エンジン同期: Elasticsearchへのインデックス更新
    3. 監査ログ: すべての変更をS3へ保存

  3. Q3. DAXが適しているワークロードと適していないワークロードを説明してください。

    回答を表示

    適している: 読み取り集中型ワークロード、同じデータへの頻繁なアクセス、マイクロ秒レベルのレイテンシが必要な場合（ゲームのリーダーボード等）
    適していない: 書き込み集中型ワークロード、強い整合性が必須の場合
