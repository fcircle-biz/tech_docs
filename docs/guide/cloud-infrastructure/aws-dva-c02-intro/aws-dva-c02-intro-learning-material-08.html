AWS Step Functions - ワークフローオーケストレーション

Step Functionsを使用したサーバーレスワークフローの設計を学びます。

この章で学ぶこと

  • ステートマシンとASL（Amazon States Language）
  • ステートタイプ（Task、Choice、Parallel、Map、Wait、Pass、Fail、Succeed）
  • 標準ワークフローとExpressワークフロー
  • エラーハンドリングとリトライ戦略

  • 長時間実行ワークフローの設計
  • Step Functions + Lambda/ECS/SNSの連携

8.1 AWS Step Functionsとは

AWS Step Functionsは、複数のAWSサービスを組み合わせた**ワークフロー**を視覚的に設計・管理できるサーバーレスオーケストレーションサービスです。個別のLambda関数やマイクロサービスを1つのアプリケーションフローとして統合できます。

重要ポイント

Step Functionsを使用することで、複雑な分岐や並列処理、エラーハンドリングをコードではなく**ステートマシン定義**として宣言的に記述できます。これにより、ビジネスロジックとワークフロー制御を分離でき、保守性が向上します。

Step Functionsの主な用途

  • データ処理パイプライン: ETL処理、画像・動画処理、レポート生成
  • 注文処理フロー: 在庫確認 → 決済処理 → 発送手配の自動化
  • 承認ワークフロー: 人間の承認待ちを含む長時間実行処理
  • 機械学習パイプライン: データ準備 → モデル学習 → デプロイまでの自動化
flowchart LR A[リクエスト開始] --> B[Lambda: データ検証] B --> C{データは有効?} C -->|Yes| D[Lambda: データ処理] C -->|No| E[SNS: エラー通知] D --> F[DynamoDB: 結果保存] F --> G[SNS: 成功通知] E --> H[処理終了] G --> H

8.2 ステートマシンとASL（Amazon States Language）

Step Functionsのワークフローは**ステートマシン**として定義され、**Amazon States Language（ASL）**というJSON形式のDSL（Domain Specific Language）で記述します。

基本的なステートマシン定義

simple-state-machine.json 
{
  "Comment": "シンプルなステートマシンの例",
  "StartAt": "HelloWorld",
  "States": {
    "HelloWorld": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:REGION:ACCOUNT:function:HelloFunction",
      "End": true
    }
  }
}

ASLの主要フィールド

  • Comment: ステートマシンの説明（任意）
  • StartAt: 最初に実行されるステート名
  • States: 各ステートの定義（キー: ステート名、値: ステート設定）
  • Type: ステートの種類（Task、Choice、Parallel等）
  • End: このステートで終了する場合はtrue

8.3 ステートタイプ一覧

Step Functionsには8種類のステートタイプがあり、それぞれ異なる役割を持ちます。

Task

Lambda関数、ECSタスク、AWS SDK呼び出しなど、実際の作業を実行します。 

"Type": "Task",
"Resource": "arn:aws:..."

Choice

条件分岐を行い、次のステートを動的に決定します。 

"Type": "Choice",
"Choices": [...]

Parallel

複数の処理を並列実行します。全て完了するまで待機します。 

"Type": "Parallel",
"Branches": [...]

Map

配列の各要素に対して同じ処理を反復実行します（並列可能）。 

"Type": "Map",
"ItemsPath": "$.items"

Wait

指定された時間待機します（秒数指定またはタイムスタンプ指定）。 

"Type": "Wait",
"Seconds": 60

Pass

入力をそのまま出力に渡すか、データを変換します（デバッグ用にも利用）。 

"Type": "Pass",
"Result": {...}

Fail

実行を失敗で終了します。エラーメッセージを指定できます。 

"Type": "Fail",
"Error": "ValidationError"

Succeed

実行を成功で終了します。 

"Type": "Succeed"

8.4 標準ワークフローとExpressワークフロー

Step Functionsには2種類のワークフロータイプがあり、ユースケースに応じて選択します。

標準ワークフロー（Standard Workflows）

  • 実行時間: 最大1年間
  • 実行履歴: 完全な実行履歴が保存される（監査可能）
  • 実行保証: Exactly-once（正確に1回実行）
  • 用途: 長時間実行処理、監査が必要な処理
  • 料金: ステート遷移数に応じた課金

Expressワークフロー（Express Workflows）

  • 実行時間: 最大5分間
  • 実行履歴: CloudWatch Logsに出力（オプション）
  • 実行保証: At-least-once（最低1回実行、重複の可能性あり）
  • 用途: 高頻度・短時間処理（IoTデータ処理、ストリーム処理）
  • 料金: 実行回数と実行時間に応じた課金

選択のヒント

1日数千回以上実行される短時間処理にはExpressワークフローを、承認フローや数時間〜数日かかる処理には標準ワークフローを選択します。

8.5 エラーハンドリングとリトライ戦略

Step Functionsでは、**Retry（リトライ）**と**Catch（エラーキャッチ）**の2つの仕組みでエラーハンドリングを行います。

Retry（自動リトライ）

retry-example.json 
{
  "ProcessData": {
    "Type": "Task",
    "Resource": "arn:aws:lambda:REGION:ACCOUNT:function:ProcessFunction",
    "Retry": [
      {
        "ErrorEquals": ["Lambda.ServiceException", "Lambda.TooManyRequestsException"],
        "IntervalSeconds": 2,
        "MaxAttempts": 3,
        "BackoffRate": 2.0
      }
    ],
    "End": true
  }
}
  • ErrorEquals: リトライ対象のエラータイプ（States.ALL で全エラー対象）
  • IntervalSeconds: 最初のリトライまでの待機時間（秒）
  • MaxAttempts: 最大リトライ回数
  • BackoffRate: リトライごとの待機時間の倍率（指数バックオフ）

Catch（エラーキャッチ）

catch-example.json 
{
  "ProcessData": {
    "Type": "Task",
    "Resource": "arn:aws:lambda:REGION:ACCOUNT:function:ProcessFunction",
    "Catch": [
      {
        "ErrorEquals": ["CustomError"],
        "Next": "HandleCustomError"
      },
      {
        "ErrorEquals": ["States.ALL"],
        "Next": "HandleGenericError"
      }
    ],
    "End": true
  },
  "HandleCustomError": {
    "Type": "Task",
    "Resource": "arn:aws:states:::sns:publish",
    "Parameters": {
      "TopicArn": "arn:aws:sns:REGION:ACCOUNT:error-topic",
      "Message": "カスタムエラーが発生しました"
    },
    "End": true
  }
}

8.6 サービス統合（Optimized vs SDK統合）

Step Functionsは200以上のAWSサービスと統合可能です。統合方式には**最適化統合（Optimized）**と**SDK統合**の2種類があります。

最適化統合（Optimized Integrations）

主要なAWSサービスに対して、Step Functionsが非同期処理の完了を自動的に待機します。

optimized-integration.json 
{
  "RunECSTask": {
    "Type": "Task",
    "Resource": "arn:aws:states:::ecs:runTask.sync",
    "Parameters": {
      "Cluster": "my-cluster",
      "TaskDefinition": "my-task"
    },
    "End": true
  }
}
  • .sync: タスク完了まで待機（Lambda、ECS、Batch等）
  • .waitForTaskToken: コールバックトークンによる手動完了待機

SDK統合

AWS SDK APIを直接呼び出します。DynamoDB、S3、SNS等あらゆるサービスに対応。

sdk-integration.json 
{
  "PutItemToDynamoDB": {
    "Type": "Task",
    "Resource": "arn:aws:states:::dynamodb:putItem",
    "Parameters": {
      "TableName": "MyTable",
      "Item": {
        "id": {"S": "12345"},
        "name": {"S": "Sample Item"}
      }
    },
    "End": true
  }
}

8.7 実践: 注文処理ワークフローの設計

ECサイトの注文処理フロー（在庫確認 → 決済 → 発送）をStep Functionsで実装する例を見てみましょう。

flowchart TD A[注文受付] --> B[在庫確認 Lambda] B --> C{在庫あり?} C -->|Yes| D[決済処理 Lambda] C -->|No| E[在庫切れ通知 SNS] E --> F[処理終了] D --> G{決済成功?} G -->|Yes| H[発送手配 Lambda] G -->|No| I[決済失敗通知 SNS] I --> F H --> J[注文完了通知 SNS] J --> F
order-workflow.json（抜粋） 
{
  "Comment": "注文処理ワークフロー",
  "StartAt": "CheckInventory",
  "States": {
    "CheckInventory": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:REGION:ACCOUNT:function:CheckInventory",
      "Next": "InventoryAvailable"
    },
    "InventoryAvailable": {
      "Type": "Choice",
      "Choices": [
        {
          "Variable": "$.inventoryAvailable",
          "BooleanEquals": true,
          "Next": "ProcessPayment"
        }
      ],
      "Default": "NotifyOutOfStock"
    },
    "ProcessPayment": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:REGION:ACCOUNT:function:ProcessPayment",
      "Next": "PaymentSucceeded"
    }
  }
}

理解度確認クイズ

  1. Q1. 標準ワークフローとExpressワークフローの最大実行時間の違いは？

    回答を表示

    標準ワークフロー: 最大1年間
    Expressワークフロー: 最大5分間
    長時間実行される承認フローや監査が必要な処理には標準ワークフローを、高頻度・短時間処理にはExpressワークフローを使用します。

  2. Q2. 配列の各要素に対して同じ処理を反復実行するステートタイプは？

    回答を表示

    Map ステートです。配列の各要素に対して並列または順次実行が可能で、大量のデータ処理に適しています。

  3. Q3. エラーハンドリングでリトライの待機時間を指数的に増やすパラメータは？

    回答を表示

    BackoffRateです。例えば2.0に設定すると、1回目2秒→2回目4秒→3回目8秒のように待機時間が倍増します。
