Amazon API Gateway - RESTful API の構築

API Gatewayを使用したRESTful APIの設計と実装を学びます。

この章で学ぶこと

  • リソースとメソッドの設計

6.1 Amazon API Gatewayとは

Amazon API Gatewayは、あらゆる規模のREST、HTTP、およびWebSocket APIを作成、公開、維持、監視、保護するためのフルマネージドサービスです。開発者はAPI Gatewayを使用して、AWS Lambda関数、EC2インスタンス、その他のWebサービスにアクセスするAPIを作成できます。

重要ポイント

API Gatewayは**バックエンドサービスへの入口**として機能し、リクエストのルーティング、認証・認可、スロットリング、モニタリングといった機能を提供します。これにより、開発者はビジネスロジックに集中できます。

API Gatewayの主な機能

  • APIの作成と管理: RESTful API、HTTP API、WebSocket APIの作成
  • 統合: Lambda、EC2、その他のAWSサービスとの統合
  • セキュリティ: IAM、Cognito、Lambdaオーソライザーによる認証・認可
  • スロットリング: リクエスト数の制限によるバックエンド保護
  • モニタリング: CloudWatchによるメトリクスとログの記録
  • キャッシング: レスポンスのキャッシュによるパフォーマンス向上

6.2 API Gatewayの3つのタイプ - REST、HTTP、WebSocket

API Gatewayには3つのタイプがあり、それぞれ異なるユースケースに適しています。適切なタイプを選択することで、コストとパフォーマンスを最適化できます。

1. REST API - フル機能型

特徴

  • API Gatewayの**フル機能**を利用可能
  • リクエスト・レスポンスの変換（マッピングテンプレート）
  • APIキーと使用量プランによるアクセス制御
  • キャッシング機能
  • プライベートAPIエンドポイント

適したユースケース

エンタープライズ向けAPI、詳細なアクセス制御が必要な場合、レガシーシステムとの統合

2. HTTP API - シンプル・低コスト型

特徴

  • REST APIの**約70%のコスト**
  • 低レイテンシ（REST APIより高速）
  • JWT認証（Cognito、OAuth 2.0）をネイティブサポート
  • CORS設定が簡単
  • マッピングテンプレートやAPIキーは**非対応**

適したユースケース

マイクロサービス、モバイルバックエンド、サーバーレスアプリケーション

3. WebSocket API - 双方向通信型

特徴

  • **双方向のリアルタイム通信**をサポート
  • ステートフル接続（持続的接続）
  • サーバーからクライアントへのプッシュ通知
  • 接続管理（$connect、$disconnect、$default ルート）

適したユースケース

チャットアプリケーション、リアルタイムダッシュボード、オンラインゲーム

選択のポイント

新規プロジェクトでは、まず**HTTP API**を検討しましょう。低コストで高速、かつ多くのユースケースに対応できます。高度な機能が必要になった場合のみREST APIに移行を検討します。

6.3 REST APIの構成要素 - リソースとメソッド

REST APIは**リソース**と**メソッド**という2つの基本要素で構成されます。リソースはAPIエンドポイントのパス（例: /users、/users/{id}）を表し、メソッドはHTTP動詞（GET、POST、PUT、DELETE）を表します。

リソース設計のベストプラクティス

# RESTful APIのリソース設計例

# 良い設計（名詞を使用、階層構造を反映）
GET    /users              # ユーザー一覧を取得
GET    /users/{id}         # 特定ユーザーを取得
POST   /users              # 新規ユーザーを作成
PUT    /users/{id}         # ユーザー情報を更新
DELETE /users/{id}         # ユーザーを削除

# ネストされたリソース
GET    /users/{id}/orders  # 特定ユーザーの注文一覧
GET    /users/{id}/orders/{orderId}  # 特定注文の詳細

# 悪い設計（動詞を使用、REST原則に反する）
GET    /getUsers           # 非推奨
POST   /createUser         # 非推奨
POST   /deleteUser         # 非推奨（DELETEを使うべき）

注意事項

パスパラメータ（{id}など）は、メソッドリクエストで定義し、統合リクエストでバックエンドに渡す必要があります。Lambda統合の場合、event.pathParameters.idで取得できます。

6.4 統合タイプ - バックエンドとの接続方法

API Gatewayのメソッドは、バックエンドサービスと統合する必要があります。統合タイプには、Lambda、HTTP、AWS、Mockの4種類があり、それぞれ異なる用途に適しています。

1. Lambda統合（最も一般的）

Lambda プロキシ統合（推奨）

API Gatewayはリクエスト全体をLambda関数に渡し、Lambda関数は特定の形式でレスポンスを返します。設定が簡単で、最も広く使われています。 

# Lambdaプロキシ統合のイベント構造
import json

def lambda_handler(event, context):
    # APIGatewayから渡される情報
    http_method = event['httpMethod']          # GET、POST等
    path = event['path']                       # /users/{id}
    path_params = event['pathParameters']      # {id: '123'}
    query_params = event['queryStringParameters']  # ?name=value
    headers = event['headers']                 # リクエストヘッダー
    body = event['body']                       # リクエストボディ（JSON文字列）

    # レスポンスの形式（必須）
    return {
        'statusCode': 200,
        'headers': {
            'Content-Type': 'application/json',
            'Access-Control-Allow-Origin': '*'  # CORS対応
        },
        'body': json.dumps({'message': 'Success'})
    }

Lambda カスタム統合

マッピングテンプレートを使用してリクエスト・レスポンスを変換します。Lambdaに渡すデータ形式を細かく制御したい場合に使用しますが、設定が複雑です。

2. HTTP統合

既存のHTTPエンドポイント（オンプレミスAPI、サードパーティAPI）をAPI Gateway経由で公開します。API Gatewayが**プロキシ**として機能します。

  • HTTPプロキシ統合: リクエストをそのまま転送
  • HTTPカスタム統合: マッピングテンプレートで変換

3. AWS統合

Lambda以外のAWSサービス（DynamoDB、S3、SNS、SQS等）に直接統合します。Lambdaを経由せずにAWSサービスを呼び出せるため、コストとレイテンシを削減できます。

例: API GatewayからDynamoDBに直接アクセス

4. Mock統合

バックエンドを呼び出さず、固定のレスポンスを返します。**APIのプロトタイピング**やCORS設定のテストに使用します。

レスポンス例: {"message": "This is a mock response"}

統合リクエストのマッピングテンプレート例

クライアントから送られたリクエストを、バックエンドが期待する形式に変換します。 

## Content-Type: application/json
## クライアントリクエスト: {"name": "John", "age": 30}
## 変換後: {"user": {"name": "John", "age": 30}, "timestamp": "2025-01-15"}

{
  "user": {
    "name": "$input.path('$.name')",
    "age": $input.path('$.age')
  },
  "timestamp": "$context.requestTime",
  "sourceIp": "$context.identity.sourceIp"
}

VTLの主な構文

## リクエストボディからJSONパスで値を取得
$input.path('$.fieldName')

## クエリパラメータを取得
$input.params('queryParam')

## パスパラメータを取得
$input.params('pathParam')

## リクエストコンテキスト変数
$context.requestId          # リクエストID
$context.requestTime        # リクエスト時刻
$context.identity.sourceIp  # クライアントIP
$context.httpMethod         # HTTPメソッド

## 条件分岐
#if($input.path('$.type') == "premium")
  "discount": 20
#else
  "discount": 0
#end

実務での推奨

新規プロジェクトでは**Lambdaプロキシ統合**を使用し、変換処理はLambda関数内で行うことを推奨します。VTLは強力ですが、デバッグが難しく、保守性が低下します。

ステージの主な設定項目

API GatewayのAPIキーと使用量プランを使用すると、APIの利用者を識別し、リクエスト数を制限できます。これにより、有料APIの提供や、顧客ごとの利用制限を実装できます。

使用量プランの設定項目

  • レート（Rate）: 1秒あたりのリクエスト数の上限（例: 100 req/秒）
  • バースト（Burst）: 短時間に許容されるリクエスト数の上限（例: 200リクエスト）
  • クォータ（Quota）: 日・週・月単位のリクエスト総数の上限（例: 10,000 req/月）

APIキーの使用方法

  -H "x-api-key: abcdefghijklmnop123456789"

# APIキーが無効または未指定の場合
# {"message":"Forbidden"}

セキュリティ上の注意

APIキーは**認証（Authentication）ではなく、識別（Identification）**のための仕組みです。セキュリティが重要なAPIでは、IAM認証、Cognito、Lambdaオーソライザーを使用してください。APIキーは主に使用量の管理に使用します。

6.8 スロットリングとクォータ - バックエンド保護

API Gatewayには、バックエンドサービスを過負荷から保護するための**スロットリング**機能が組み込まれています。スロットリングは、アカウントレベル、ステージレベル、メソッドレベルで設定できます。

スロットリングの2つのパラメータ

レート（Rate）

定常状態での1秒あたりのリクエスト数。この値を超えるリクエストは拒否されます。

例: 1,000 req/秒 = 毎秒1,000リクエストまで許可

バースト（Burst）

短時間に処理できるリクエストの総数。トークンバケットアルゴリズムで制御されます。

例: 2,000 = 短時間に2,000リクエストまでバースト可能

スロットリング設定の優先順位

# 優先順位: メソッド > ステージ > アカウント

1. メソッドレベル（最優先）
   POST /users: 100 req/秒、200 バースト

2. ステージレベル（メソッド未設定の場合）
   prod ステージ: 1,000 req/秒、2,000 バースト

3. アカウントレベル（デフォルト）
   全リージョン合計: 10,000 req/秒
   ※ AWS サポートに申請して上限緩和可能

# 結果
POST /users      -> 100 req/秒（メソッド設定優先）
GET /users       -> 1,000 req/秒（ステージ設定を継承）
その他のエンドポイント -> 1,000 req/秒

スロットリングエラーのハンドリング

# クライアント側でのリトライ処理例
import time
import requests
from requests.adapters import HTTPAdapter
from requests.packages.urllib3.util.retry import Retry

def create_session_with_retry():
    """429エラーに対応したセッションを作成"""
    session = requests.Session()

    # リトライ戦略: 429エラーで指数バックオフ
    retry = Retry(
        total=5,                    # 最大5回リトライ
        status_forcelist=[429, 500, 502, 503, 504],
        backoff_factor=1,           # 1秒、2秒、4秒...と増加
        allowed_methods=["GET", "POST", "PUT", "DELETE"]
    )

    adapter = HTTPAdapter(max_retries=retry)
    session.mount("https://", adapter)

    return session

# 使用例
session = create_session_with_retry()
response = session.get(
    "https://api.example.com/prod/users",
    headers={"x-api-key": "your-api-key"}
)

ベストプラクティス

本番環境では、**ステージレベルのスロットリング**を必ず設定し、重要なエンドポイント（書き込み操作など）には**メソッドレベルのスロットリング**を追加します。これにより、予期しないトラフィック急増からバックエンドを保護できます。

ハンズオン実習: Lambda統合REST APIの作成

API Gatewayを使用して、Lambdaプロキシ統合のREST APIを作成し、基本的なCRUD操作を実装してみましょう。

ステップ1: Lambda関数の作成

# Lambda関数: user-api-handler
import json

def lambda_handler(event, context):
    """APIGatewayからのリクエストを処理"""
    http_method = event['httpMethod']
    path = event['path']

    # ルーティング
    if path == '/users' and http_method == 'GET':
        return get_users()
    elif path == '/users' and http_method == 'POST':
        return create_user(event)
    else:
        return {
            'statusCode': 404,
            'headers': {'Content-Type': 'application/json'},
            'body': json.dumps({'error': 'Not Found'})
        }

def get_users():
    """ユーザー一覧を取得（モックデータ）"""
    users = [
        {'id': 1, 'name': 'Alice', 'email': 'alice@example.com'},
        {'id': 2, 'name': 'Bob', 'email': 'bob@example.com'}
    ]
    return {
        'statusCode': 200,
        'headers': {
            'Content-Type': 'application/json',
            'Access-Control-Allow-Origin': '*'
        },
        'body': json.dumps(users)
    }

def create_user(event):
    """新規ユーザーを作成"""
    try:
        body = json.loads(event['body'])
        # 本来はDynamoDBに保存する処理
        return {
            'statusCode': 201,
            'headers': {
                'Content-Type': 'application/json',
                'Access-Control-Allow-Origin': '*'
            },
            'body': json.dumps({
                'message': 'User created',
                'user': body
            })
        }
    except Exception as e:
        return {
            'statusCode': 400,
            'headers': {'Content-Type': 'application/json'},
            'body': json.dumps({'error': str(e)})
        }

理解度確認クイズ

  1. Q1. REST APIとHTTP APIの主な違いは何ですか？

    回答を表示

    HTTP APIは REST APIの約70%のコストで、低レイテンシを実現しますが、マッピングテンプレートやAPIキーなどの高度な機能は非対応です。新規プロジェクトではHTTP APIを検討し、必要に応じてREST APIを選択します。

  2. Q2. Lambdaプロキシ統合とLambdaカスタム統合の違いは？

    回答を表示

    Lambdaプロキシ統合は、API Gatewayがリクエスト全体をLambdaに渡し、Lambda関数が特定の形式でレスポンスを返します。設定が簡単で推奨されます。

    Lambdaカスタム統合は、マッピングテンプレートでリクエスト・レスポンスを変換しますが、設定が複雑でデバッグが困難です。

  3. Q3. スロットリングのレートとバーストの違いは？

    回答を表示

    レートは定常状態での1秒あたりのリクエスト数の上限です（例: 1,000 req/秒）。

    バーストは短時間に処理できるリクエストの総数です（例: 2,000リクエスト）。トークンバケットアルゴリズムで制御され、一時的なトラフィック急増を吸収します。
