イベント駆動アーキテクチャ - EventBridge、SQS、SNS

イベント駆動型アプリケーションの設計パターンを習得します。疎結合で拡張性の高いシステムを構築するためのAWSメッセージングサービスを理解します。

この章で学ぶこと

  • Amazon EventBridgeのイベントバスとルール
  • スケジュール式とイベントパターン
  • Amazon SQSの標準キューとFIFOキュー
  • デッドレターキュー（DLQ）とリドライブポリシー
  • Amazon SNSのトピックとサブスクリプション
  • ファンアウトパターンとメッセージフィルタリング
  • Lambda + SQS/SNS/EventBridgeの統合

9.1 イベント駆動アーキテクチャとは

イベント駆動アーキテクチャ（Event-Driven Architecture）は、サービス間の結合度を下げ、拡張性と保守性を向上させる設計パターンです。従来の同期的なAPI呼び出しとは異なり、**イベントの発生と処理を非同期で行う**ことで、システムの柔軟性を高めます。

flowchart LR subgraph Traditional["従来型（同期処理）"] direction LR A1["クライアント"] -->|API呼び出し| B1["サービスA"] B1 -->|API呼び出し| C1["サービスB"] C1 -->|API呼び出し| D1["サービスC"] D1 -->|レスポンス| C1 C1 -->|レスポンス| B1 B1 -->|レスポンス| A1 end subgraph EventDriven["イベント駆動型（非同期処理）"] direction LR A2["クライアント"] -->|イベント発行| E["イベントバス"] E -->|イベント配信| B2["サービスA"] E -->|イベント配信| C2["サービスB"] E -->|イベント配信| D2["サービスC"] end

イベント駆動アーキテクチャのメリット

  • 疎結合: サービス間の依存関係が最小化され、独立した開発・デプロイが可能
  • 拡張性: 新しいサービスを追加しても既存のサービスに影響しない
  • 耐障害性: 一部のサービスが停止しても全体が停止しない
  • スケーラビリティ: 各サービスを個別にスケーリング可能

イベント駆動アーキテクチャの3つの主要サービス

EventBridge

イベントバスとルールを使用したイベントルーティング。AWSサービスやSaaSとの統合が容易。

SQS

メッセージキューサービス。非同期処理とバッファリングに使用。標準キューとFIFOキューの2種類。

SNS

Pub/Subメッセージングサービス。1対多の通知配信に使用。ファンアウトパターンで活用。

9.2 Amazon EventBridge - イベントバスとルール

Amazon EventBridgeは、AWSサービス、SaaSアプリケーション、カスタムアプリケーション間でイベントをルーティングするサーバーレスイベントバスサービスです。CloudWatch Eventsの後継サービスとして、より強力なイベント管理機能を提供します。

イベントパターンの例

EventBridgeルールでは、イベントパターンを使用して特定のイベントをフィルタリングできます。以下はS3バケットへのオブジェクト作成イベントを検出する例です。

スケジュール式による定期実行

EventBridgeは、cron式またはrate式を使用して定期的にLambda関数を実行できます。

式の種類 構文例 説明
rate式 rate(5 minutes) 5分ごとに実行
rate式 rate(1 hour) 1時間ごとに実行
cron式 cron(0 9 * * ? *) 毎日午前9時（UTC）に実行
cron式 cron(0 0 1 * ? *) 毎月1日の午前0時に実行

9.3 Amazon SQS - メッセージキューサービス

Amazon Simple Queue Service（SQS）は、フルマネージドメッセージキューサービスです。アプリケーション間でメッセージを非同期に送受信し、疎結合なシステムを構築できます。

標準キューとFIFOキューの比較

スループット ほぼ無制限 最大3,000メッセージ/秒
（バッチ処理で30,000/秒）
順序保証 ベストエフォート 厳密な順序保証（FIFO）
重複排除 メッセージが重複する可能性あり 5分間の重複排除
ユースケース 高スループットが必要で、順序が重要でない処理 金融取引、注文処理など順序が重要な処理

Lambda関数 participant DLQ as デッドレター
キュー Producer->>SQS: メッセージ送信 Note over SQS: メッセージを保持
最大14日間 SQS->>Consumer: メッセージ受信 Consumer->>Consumer: 処理成功 Consumer->>SQS: メッセージ削除 SQS->>Consumer: 別メッセージ受信 Consumer->>Consumer: 処理失敗 Note over SQS: 可視性タイムアウト後

DLQへ移動

デッドレターキュー（DLQ）の設定

処理に失敗したメッセージを別のキュー（DLQ）に移動することで、エラー分析と再処理が可能になります。

sqs-dlq-config.py 
import boto3

# SQSクライアント作成
sqs = boto3.client('sqs')

# DLQのARNを指定してリドライブポリシーを設定
response = sqs.set_queue_attributes(
    QueueUrl='https://sqs.ap-northeast-1.amazonaws.com/123456789012/MyQueue',
    Attributes={
        'RedrivePolicy': json.dumps({
            'deadLetterTargetArn': 'arn:aws:sqs:ap-northeast-1:123456789012:MyDLQ',
            'maxReceiveCount': '3'  # 3回失敗したらDLQへ移動
        })
    }
)

ベストプラクティス: 可視性タイムアウトの設定

可視性タイムアウトは、Lambda関数のタイムアウト値の6倍に設定することが推奨されます。例えば、Lambda関数のタイムアウトが30秒の場合、可視性タイムアウトは180秒（3分）に設定します。

9.4 Amazon SNS - Pub/Subメッセージング

Amazon Simple Notification Service（SNS）は、Pub/Sub（パブリッシュ/サブスクライブ）モデルに基づくメッセージングサービスです。1つのメッセージを複数のサブスクライバーに配信する**ファンアウトパターン**に最適です。

メッセージフィルタリング

SNSサブスクリプションにフィルタポリシーを設定することで、特定の条件に一致するメッセージのみを受信できます。

sns-filter-policy.json 
{
  "eventType": ["order"],
  "price": [{"numeric": [">", 100]}],
  "region": ["ap-northeast-1", "us-east-1"]
}

上記のフィルタポリシーは、以下の条件をすべて満たすメッセージのみを配信します：

  • eventTypeが「order」
  • priceが100より大きい
  • regionが「ap-northeast-1」または「us-east-1」

9.5 ファンアウトパターン - SNS + SQSの組み合わせ

SNSとSQSを組み合わせることで、**耐障害性の高いファンアウトパターン**を実装できます。SNSトピックに複数のSQSキューをサブスクライブさせることで、各サービスが独立してメッセージを処理できます。

ファンアウトパターンのメリット

  • 疎結合: 各サービスが独立して動作し、相互依存しない
  • 耐障害性: 1つのサービスが停止しても他のサービスは継続
  • 独立したスケーリング: 各サービスのトラフィックに応じてスケール
  • メッセージ永続化: SQSがメッセージを保持するため処理の再試行が可能

ファンアウトパターンの実装例

publish-to-sns.py 
import boto3
import json

# SNSクライアント作成
sns = boto3.client('sns')

# 注文完了イベントを発行
response = sns.publish(
    TopicArn='arn:aws:sns:ap-northeast-1:123456789012:OrderCompletedTopic',
    Message=json.dumps({
        'orderId': 'ORD-12345',
        'customerId': 'CUST-67890',
        'items': [
            {'productId': 'PROD-001', 'quantity': 2},
            {'productId': 'PROD-002', 'quantity': 1}
        ],
        'totalAmount': 15000
    }),
    MessageAttributes={
        'eventType': {'DataType': 'String', 'StringValue': 'order'},
        'priority': {'DataType': 'String', 'StringValue': 'high'}
    }
)

9.6 Lambda統合パターン

Lambda関数は、EventBridge、SQS、SNSすべてと統合できます。各サービスとの統合パターンを理解することで、適切なアーキテクチャを選択できます。

Lambda + EventBridge

EventBridgeルールのターゲットとしてLambda関数を指定することで、特定のイベント発生時に自動的に関数を実行できます。

lambda-eventbridge-handler.py 
def lambda_handler(event, context):
    # EventBridgeから渡されたイベントを処理
    detail = event['detail']
    source = event['source']

    # S3オブジェクト作成イベントの場合
    if source == 'aws.s3' and event['detail-type'] == 'Object Created':
        bucket_name = detail['bucket']['name']
        object_key = detail['object']['key']

        print(f"新しいオブジェクトが作成されました: {bucket_name}/{object_key}")
        # 処理ロジックをここに記述

    return {'statusCode': 200}

Lambda + SQS（ポーリング）

LambdaはSQSキューを自動的にポーリングし、メッセージを受信して処理します。バッチサイズを調整することで処理効率を最適化できます。

lambda-sqs-handler.py 
import json

def lambda_handler(event, context):
    # SQSから受信したメッセージをバッチ処理
    for record in event['Records']:
        # メッセージボディを取得
        body = json.loads(record['body'])

        # メッセージIDと受信時刻を取得
        message_id = record['messageId']
        receipt_handle = record['receiptHandle']

        print(f"メッセージ処理: {message_id}")
        print(f"内容: {body}")

        # 処理ロジックをここに記述
        # 処理が成功すると、Lambdaが自動的にメッセージを削除

    return {'statusCode': 200}

注意: バッチ処理の部分失敗

Lambda関数がSQSメッセージをバッチ処理する場合、1件でも処理に失敗するとバッチ全体が再試行されます。部分失敗を許容する場合は、ReportBatchItemFailuresを有効化してください。

Lambda + SNS（プッシュ）

SNSトピックのサブスクリプションとしてLambda関数を登録することで、メッセージ発行時に即座に関数を実行できます。

lambda-sns-handler.py 
import json

def lambda_handler(event, context):
    # SNSから受信したメッセージを処理

        sns_message = record['Sns']

実習: SNS + SQS + Lambdaでファンアウトパターン構築

目標

SNSトピックに2つのSQSキューをサブスクライブさせ、それぞれのキューにLambda関数を接続して、ファンアウトパターンを実装します。

手順

  1. SNSトピック「OrderEventsTopic」を作成
  2. 2つのSQSキュー「InventoryQueue」「NotificationQueue」を作成
  3. 各キューをSNSトピックにサブスクライブ
  4. 各キューにトリガーされるLambda関数を作成
  5. SNSトピックにテストメッセージを発行し、両方のLambda関数が実行されることを確認

確認ポイント

  • 1つのメッセージが2つのキューに配信されるか
  • 各Lambda関数が独立して実行されるか
  • CloudWatch Logsで実行ログを確認できるか

理解度確認クイズ

  1. Q1. SQS標準キューとFIFOキューの主な違いは何ですか?

    回答を表示

    標準キューは、ほぼ無制限のスループットを提供しますが、メッセージの順序保証やベストエフォートであり、重複が発生する可能性があります。FIFOキューは、厳密な順序保証と重複排除を提供しますが、スループットは最大3,000メッセージ/秒（バッチ処理で30,000/秒）に制限されます。金融取引や注文処理など順序が重要な場合はFIFOキューを使用します。

  2. Q2. SNS + SQSのファンアウトパターンを使用する主なメリットは何ですか?

    回答を表示

    ファンアウトパターンの主なメリットは、疎結合、耐障害性、独立したスケーリング、メッセージ永続化です。SNSが1つのメッセージを複数のSQSキューに配信し、各サービスが独立してメッセージを処理できます。1つのサービスが停止しても他のサービスは影響を受けず、SQSがメッセージを保持するため処理の再試行が可能です。

  3. Q3. EventBridgeのルールで使用できる2種類の式は何ですか?

    回答を表示

    EventBridgeルールでは、イベントパターンとスケジュール式の2種類を使用できます。イベントパターンは特定のイベントをフィルタリングし、スケジュール式（rate式またはcron式）は定期的にターゲットを実行します。例: rate(5 minutes)は5分ごと、cron(0 9 * * ? *)は毎日午前9時（UTC）に実行します。
