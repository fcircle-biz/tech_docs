7.1 API Gatewayの認証・認可オプション

API Gatewayは複数の認証・認可メカニズムを提供しており、アプリケーションの要件に応じて適切な方法を選択できます。セキュリティは**多層防御**の考え方で設計することが重要です。

flowchart TD A[クライアント] --> B{認証方式の選択} B -->|AWSサービス間通信| C[IAM認証] B -->|エンドユーザー認証| D[Cognito ユーザープール] B -->|カスタム認証ロジック| E[Lambda オーソライザー] B -->|公開API| F[APIキー] C --> G[SigV4署名検証] D --> H[JWTトークン検証] E --> I[カスタム検証ロジック] F --> J[使用量プラン] G --> K[バックエンド] H --> K I --> K J --> K

認証方式の比較

認証方式 ユースケース メリット 注意点
IAM認証 AWSサービス間通信 追加設定不要、強力な認証 エンドユーザーには不向き
Cognito モバイル・Webアプリ ユーザー管理機能内蔵 Cognito設定が必要
Lambdaオーソライザー カスタム認証ロジック 柔軟性が高い 実装・管理コストあり
APIキー パートナーAPI、使用量制限 シンプル 認可のみ（認証なし）

重要な注意事項

APIキーは**認証（Authentication）**ではなく**認可（Authorization）**の仕組みです。APIキーだけでセキュリティを確保することはできません。必ず他の認証方式と組み合わせて使用してください。

7.2 IAM認証とSigV4署名

IAM認証は、AWSの認証情報を使用してAPIにアクセスする方式です。Lambda、EC2、ECSなどのAWSサービスから呼び出す際に最適です。**Signature Version 4（SigV4）**という署名プロセスを使用します。

SigV4署名のしくみ

sequenceDiagram participant Client as クライアント participant APIGateway as API Gateway participant IAM as IAM participant Lambda as Lambda関数 Client->>Client: アクセスキーで
リクエストに署名 Client->>APIGateway: 署名付きリクエスト送信 APIGateway->>IAM: 署名検証 IAM-->>APIGateway: 検証結果 alt 検証成功 APIGateway->>Lambda: リクエスト転送 Lambda-->>APIGateway: レスポンス APIGateway-->>Client: レスポンス返却 else 検証失敗 APIGateway-->>Client: 403 Forbidden end

IAM認証の設定例

API Gatewayメソッド設定（Terraform） 
# API Gatewayメソッドの作成
resource "aws_api_gateway_method" "get_items" {
  rest_api_id   = aws_api_gateway_rest_api.api.id
  resource_id   = aws_api_gateway_resource.items.id
  http_method   = "GET"
  authorization = "AWS_IAM"  # IAM認証を有効化
}

# Lambda関数との統合
resource "aws_api_gateway_integration" "lambda" {
  rest_api_id             = aws_api_gateway_rest_api.api.id
  resource_id             = aws_api_gateway_resource.items.id
  http_method             = aws_api_gateway_method.get_items.http_method
  integration_http_method = "POST"
  type                    = "AWS_PROXY"
  uri                     = aws_lambda_function.handler.invoke_arn
}

クライアント側からのIAM認証付きリクエスト

Python + boto3での署名付きリクエスト 
import boto3
from botocore.auth import SigV4Auth
from botocore.awsrequest import AWSRequest
import requests

# AWS認証情報を取得
session = boto3.Session()
credentials = session.get_credentials()

# リクエストを作成
url = 'https://abc123.execute-api.ap-northeast-1.amazonaws.com/prod/items'
request = AWSRequest(method='GET', url=url)

# SigV4署名を追加
SigV4Auth(credentials, 'execute-api', 'ap-northeast-1').add_auth(request)

# リクエスト送信
response = requests.get(url, headers=dict(request.headers))
print(response.json())

ヒント: IAM認証のベストプラクティス

IAM認証は**AWSサービス間の通信**に最適です。Lambda、EC2、ECSからAPI Gatewayを呼び出す場合、IAMロールを使用すればアクセスキーをハードコードする必要がありません。

7.3 Amazon Cognito ユーザープールとの統合

Amazon Cognitoは、ユーザーのサインアップ、サインイン、アクセス制御を提供するマネージドサービスです。モバイルアプリやWebアプリケーションのユーザー認証に最適です。

Cognitoユーザープール認証フロー

sequenceDiagram participant User as ユーザー participant App as アプリ participant Cognito as Cognito ユーザープール participant APIGateway as API Gateway participant Lambda as Lambda User->>App: ログイン App->>Cognito: 認証リクエスト Cognito-->>App: JWTトークン発行 App->>APIGateway: JWTトークン付きリクエスト APIGateway->>Cognito: トークン検証 Cognito-->>APIGateway: 検証結果 APIGateway->>Lambda: リクエスト転送 Lambda-->>APIGateway: レスポンス APIGateway-->>App: レスポンス App-->>User: データ表示

API GatewayでのCognito認証設定

Cognito Authorizerの設定（AWS CLI） 
# Cognito AuthorizerをAPI Gatewayに作成
aws apigateway create-authorizer \
  --rest-api-id abc123 \
  --name CognitoAuthorizer \
  --type COGNITO_USER_POOLS \
  --provider-arns arn:aws:cognito-idp:ap-northeast-1:123456789012:userpool/ap-northeast-1_ABC123 \
  --identity-source method.request.header.Authorization

# メソッドに認証を設定
aws apigateway update-method \
  --rest-api-id abc123 \
  --resource-id xyz789 \
  --http-method GET \
  --patch-operations op=replace,path=/authorizationType,value=COGNITO_USER_POOLS \
    op=replace,path=/authorizerId,value=authorizer-id

クライアント側からのCognito認証付きリクエスト

JavaScriptでのトークン付きリクエスト 
// Cognitoログインで取得したIDトークンを使用
const idToken = 'eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...';

// API Gatewayにリクエスト
const response = await fetch(
  'https://abc123.execute-api.ap-northeast-1.amazonaws.com/prod/items',
  {
    method: 'GET',
    headers: {
      'Authorization': idToken  // IDトークンをAuthorizationヘッダーに含める
    }
  }
);

const data = await response.json();
console.log(data);

重要ポイント

Cognitoは**IDトークン**、**アクセストークン**、**リフレッシュトークン**の3種類のトークンを発行します。API Gatewayでの認証には**IDトークン**を使用します。

7.4 Lambda オーソライザー

Lambdaオーソライザーは、独自の認証ロジックを実装できる柔軟な認証方式です。サードパーティのOAuth、SAML、カスタムトークンなど、Cognitoではカバーできない認証要件に対応できます。

Lambda オーソライザーの種類

トークンベース

Authorizationヘッダーのトークンを検証します。JWTトークンなどに適しています。

ユースケース:
OAuth 2.0、JWT、カスタムトークン

リクエストベース

リクエスト全体（ヘッダー、パス、クエリパラメータ）を検証します。複雑な認証ロジックに適しています。

ユースケース:
IPアドレス制限、複数ヘッダー検証

Lambda オーソライザーの実装例

トークンベースオーソライザー（Python） 
import json

def lambda_handler(event, context):
    # トークンを取得
    token = event['authorizationToken']

    # トークン検証（簡易例）
    if token == 'allow':
        effect = 'Allow'
    else:
        effect = 'Deny'

    # IAMポリシーを返す
    return {
        'principalId': 'user123',
        'policyDocument': {
            'Version': '2012-10-17',
            'Statement': [{
                'Action': 'execute-api:Invoke',
                'Effect': effect,
                'Resource': event['methodArn']
            }]
        },
        # コンテキスト（Lambda関数に渡される）
        'context': {
            'userId': 'user123',
            'email': 'user@example.com'
        }
    }

重要: Lambda オーソライザーのキャッシュ

Lambda オーソライザーの結果は、デフォルトで5分間キャッシュされます。認証結果がすぐに反映されない場合は、キャッシュ設定（TTL）を確認してください。

7.5 CORS（Cross-Origin Resource Sharing）の設定

WebブラウザからAPI Gatewayにアクセスする場合、**CORS（クロスオリジンリソース共有）**を正しく設定する必要があります。CORSが正しく設定されていないと、ブラウザがリクエストをブロックします。

CORSのしくみ

sequenceDiagram participant Browser as ブラウザ participant APIGateway as API Gateway Note over Browser,APIGateway: プリフライトリクエスト Browser->>APIGateway: OPTIONS /items APIGateway-->>Browser: Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: GET, POST Note over Browser,APIGateway: 実際のリクエスト Browser->>APIGateway: GET /items APIGateway-->>Browser: データ + CORSヘッダー

API GatewayでのCORS設定

Lambda関数でのCORSヘッダー返却 
def lambda_handler(event, context):
    return {
        'statusCode': 200,
        'headers': {
            'Access-Control-Allow-Origin': '*',  # 許可するオリジン
            'Access-Control-Allow-Headers': 'Content-Type,Authorization',
            'Access-Control-Allow-Methods': 'GET,POST,PUT,DELETE,OPTIONS'
        },
        'body': json.dumps({'message': 'Hello'})
    }

セキュリティ上の注意

本番環境ではAccess-Control-Allow-Origin: *を使用せず、特定のドメインのみを許可してください。例: 'https://example.com'

7.6 リソースポリシーによるアクセス制御

リソースポリシーは、API Gateway全体またはステージ単位でアクセス制御を定義できる仕組みです。特定のIPアドレス、VPC、AWSアカウントからのアクセスのみを許可できます。

リソースポリシーの例

特定IPアドレスからのみアクセスを許可 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": "*",
      "Action": "execute-api:Invoke",
      "Resource": "arn:aws:execute-api:ap-northeast-1:123456789012:abc123/*",
      "Condition": {
        "IpAddress": {
          "aws:SourceIp": ["203.0.113.0/24"]
        }
      }
    }
  ]
}

ヒント: 多層防御の実践

リソースポリシー + IAM認証/Cognito認証を組み合わせることで、より強固なセキュリティを実現できます。例えば、社内ネットワークからのみアクセス可能にし、さらにCognito認証を必須とする設定が可能です。

7.7 AWS WAF統合によるセキュリティ強化

AWS WAF（Web Application Firewall）をAPI Gatewayと統合することで、SQLインジェクション、XSS、DDoS攻撃などから保護できます。

WAFで実装できる保護

  • レート制限: 同一IPアドレスからの過度なリクエストをブロック
  • 地理的制限: 特定の国からのアクセスを許可/拒否
  • SQLインジェクション対策: 悪意のあるSQLパターンを検出
  • XSS対策: クロスサイトスクリプティングパターンを検出
  • IPレピュテーションリスト: 既知の悪意のあるIPをブロック
flowchart LR A[クライアント] --> B[AWS WAF] B -->|許可| C[API Gateway] B -->|ブロック| D[403 Forbidden] C --> E[Lambda関数] style B fill:#ff9999 style D fill:#ff6666

WAF WebACLのAPI Gatewayへの関連付け

AWS CLI でのWAF関連付け 
# WAF WebACLをAPI Gatewayに関連付け
aws wafv2 associate-web-acl \
  --web-acl-arn arn:aws:wafv2:ap-northeast-1:123456789012:regional/webacl/MyWebACL/abc123 \
  --resource-arn arn:aws:apigateway:ap-northeast-1::/restapis/abc123/stages/prod

7.8 プライベートAPIエンドポイント

プライベートAPIエンドポイントを使用すると、VPC内からのみアクセス可能なAPIを作成できます。インターネットに公開せず、社内システムからのみアクセス可能なAPIに適しています。

flowchart TD subgraph VPC["VPC（社内ネットワーク）"] A[EC2インスタンス] --> B[VPCエンドポイント] C[Lambda in VPC] --> B end B --> D[プライベート API Gateway] D --> E[Lambda関数] F[インターネット] -.-x|アクセス不可| D style D fill:#99ccff style F fill:#ffcccc

プライベートAPIの作成手順

  1. VPCエンドポイント（PrivateLink）を作成
  2. API Gatewayでエンドポイントタイプを「プライベート」に設定
  3. リソースポリシーでVPCエンドポイントからのアクセスのみを許可
プライベートAPIのリソースポリシー 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": "*",
      "Action": "execute-api:Invoke",
      "Resource": "arn:aws:execute-api:ap-northeast-1:123456789012:abc123/*",
      "Condition": {
        "StringEquals": {
          "aws:SourceVpce": "vpce-0123456789abcdef0"
        }
      }
    }
  ]
}

実習: Cognito認証付きAPIの構築

  1. 1. Cognitoユーザープールの作成
    マネジメントコンソールでCognitoユーザープールを作成し、テストユーザーを追加します。
  2. 2. API Gateway + Lambdaの作成
    簡単なLambda関数を作成し、API Gatewayで公開します。
  3. 3. Cognito Authorizerの設定
    API GatewayにCognito Authorizerを追加し、メソッドに適用します。
  4. 4. 認証テスト
    Postmanまたはcurlで、IDトークンなしのリクエスト（401エラー）とIDトークン付きリクエスト（200成功）を試します。

理解度確認クイズ

  1. Q1. IAM認証が最適なユースケースはどれですか?

    回答を表示

    AWSサービス間の通信（Lambda→API Gateway、EC2→API Gateway等）。IAM認証はSigV4署名を使用し、AWSの認証情報で強力な認証を実現できます。エンドユーザー認証にはCognitoが適しています。

  2. Q2. CognitoユーザープールでAPI Gatewayを認証する際、どのトークンを使用しますか?

    回答を表示

    IDトークン。Cognitoは3種類のトークン（IDトークン、アクセストークン、リフレッシュトークン）を発行しますが、API Gateway認証にはIDトークンを使用します。

  3. Q3. Lambda オーソライザーの認証結果はどのくらいキャッシュされますか?

    回答を表示

    デフォルトで5分間キャッシュされます。キャッシュTTLは0〜3600秒の範囲で設定可能です。認証結果をすぐに反映したい場合はTTLを短く設定しますが、Lambda呼び出し回数が増えるためコストに注意が必要です。
