10.1 クラウド移行戦略（6R）

既存システムをクラウドに移行する際の6つの主要な戦略（6R）があります。

戦略説明メリット適用例
1. Rehost
(Lift and Shift)		 変更なしでそのままクラウドへ移行 迅速、低リスク レガシーシステムの移行
2. Replatform
(Lift, Tinker, and Shift)		 最小限の最適化を行って移行 クラウドの一部機能を活用 データベースをRDSに変更
3. Repurchase
(Drop and Shop)		 別の製品・SaaSに切り替え 最新機能、運用負荷削減 自社CRMをSalesforceに
4. Refactor
(Re-architect)		 クラウドネイティブに再設計 最大限のクラウド活用 マイクロサービス化
5. Retire
(廃止)		 不要なシステムを廃止 コスト削減、簡素化 使われていないアプリ
6. Retain
(保持)		 現状維持（オンプレミス継続） 規制対応、移行不要 移行が困難なシステム

10.2 Infrastructure as Code（IaC）

IaCの利点

  • 再現性: 同じ環境を何度でも構築可能
  • バージョン管理: Gitでインフラの変更履歴を管理
  • 自動化: 手動作業を削減、ヒューマンエラーを防止
  • ドキュメント: コード自体がドキュメントとなる
  • コードレビュー: インフラ変更もレビュー可能

主要なIaCツール

ツールプロバイダー言語特徴
TerraformマルチクラウドHCL最も人気、宣言的、状態管理
CloudFormationAWSYAML/JSONAWSネイティブ、無料
ARM Templates / BicepAzureJSON/BicepAzureネイティブ
Cloud Deployment ManagerGCPYAML/PythonGCPネイティブ
PulumiマルチクラウドPython/TypeScript/Goプログラミング言語で記述
実習 10-1: Terraformで基本インフラを構築
# main.tf
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = "ap-northeast-1"
}

# VPCの作成
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
  tags = {
    Name = "terraform-vpc"
  }
}

# パブリックサブネット
resource "aws_subnet" "public" {
  vpc_id            = aws_vpc.main.id
  cidr_block        = "10.0.1.0/24"
  availability_zone = "ap-northeast-1a"
  tags = {
    Name = "public-subnet"
  }
}

# EC2インスタンス
resource "aws_instance" "web" {
  ami           = "ami-0d52744d6551d851e"
  instance_type = "t3.micro"
  subnet_id     = aws_subnet.public.id
  tags = {
    Name = "web-server"
  }
}

output "instance_ip" {
  value = aws_instance.web.public_ip
}
# Terraformコマンド
terraform init      # 初期化
terraform plan      # 実行計画を表示
terraform apply     # リソースを作成
terraform destroy   # リソースを削除

10.3 CI/CDパイプライン

CI/CDとは

  • CI（Continuous Integration）: コードの変更を頻繁に統合し、自動テストを実行
  • CD（Continuous Delivery/Deployment）: テスト済みコードを自動的にデプロイ
graph LR A[コード変更] --> B[Git Push] B --> C[CI: ビルド] C --> D[CI: テスト] D --> E[CD: ステージング環境へデプロイ] E --> F[手動承認] F --> G[CD: 本番環境へデプロイ]
サービスプロバイダー特徴
GitHub ActionsGitHubGitHubネイティブ、YAML設定
AWS CodePipelineAWSAWSサービスとの統合
Azure DevOpsAzureエンタープライズ向け
Cloud BuildGCPGCPネイティブ、高速ビルド
GitLab CI/CDGitLab包括的なDevOpsプラットフォーム

10.4 セキュリティベストプラクティス

クラウドセキュリティの黄金律
  • 最小権限の原則: 必要最小限の権限のみ付与
  • 多層防御: 複数のセキュリティレイヤーを実装
  • 暗号化の徹底: 保存時・転送時ともに暗号化
  • 定期的な監査: ログの確認とセキュリティ評価
  • パッチ管理: 脆弱性に迅速に対応
  • セキュリティ自動化: 手動作業を減らす
  • インシデント対応計画: 侵害発生時の対応手順を準備

10.5 実践プロジェクト1: 3層Webアプリケーション

プロジェクト概要

高可用性とスケーラビリティを備えた3層Webアプリケーションを構築します。

アーキテクチャ

graph TB User[ユーザー] --> CF[CloudFront CDN] CF --> ALB[Application Load Balancer] subgraph "Multi-AZ" ALB --> Web1[Webサーバー AZ-1a] ALB --> Web2[Webサーバー AZ-1c] Web1 --> App1[アプリサーバー AZ-1a] Web2 --> App2[アプリサーバー AZ-1c] App1 --> RDS[(RDS MySQL Multi-AZ)] App2 --> RDS App1 --> Redis[ElastiCache Redis] App2 --> Redis end RDS --> S3[S3バックアップ]

必要なリソース

  • ネットワーク: VPC、パブリック/プライベートサブネット（2AZ）
  • コンピューティング: EC2 Auto Scaling Group、Application Load Balancer
  • データベース: RDS MySQL（Multi-AZ）、ElastiCache Redis
  • ストレージ: S3（静的ファイル）、CloudFront（CDN）
  • セキュリティ: Security Groups、IAM Roles
  • 監視: CloudWatch、CloudTrail

構築手順

  1. VPCとサブネットの設計・作成（マルチAZ）
  2. セキュリティグループの設定（最小権限）
  3. RDSデータベースの作成（Multi-AZ、自動バックアップ有効）
  4. ElastiCacheの設定（セッション管理用）
  5. EC2起動テンプレートの作成
  6. Auto Scaling Groupの設定（2〜10インスタンス）
  7. Application Load Balancerの作成
  8. CloudFrontディストリビューションの設定
  9. CloudWatchアラームの設定
  10. 負荷テストと動作確認

10.6 実践プロジェクト2: サーバーレスアーキテクチャ

プロジェクト概要

完全サーバーレスのイベント処理システムを構築します。

アーキテクチャ

graph LR User[ユーザー] --> API[API Gateway] API --> Lambda1[Lambda: 認証] Lambda1 --> Lambda2[Lambda: ビジネスロジック] Lambda2 --> DynamoDB[(DynamoDB)] Lambda2 --> SQS[SQS キュー] SQS --> Lambda3[Lambda: 非同期処理] Lambda3 --> S3[S3] DynamoDB --> Stream[DynamoDB Streams] Stream --> Lambda4[Lambda: イベント処理]

使用するサービス

  • API Gateway: RESTful APIの公開
  • Lambda: ビジネスロジックの実行
  • DynamoDB: NoSQLデータベース
  • SQS: 非同期処理用のキュー
  • S3: ファイルストレージ
  • Cognito: ユーザー認証
  • CloudWatch: ログとメトリクス
