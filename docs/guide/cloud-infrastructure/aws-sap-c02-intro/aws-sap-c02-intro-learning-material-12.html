障害復旧（DR）戦略とマルチリージョン設計

12.1 DR戦略の基礎とRPO/RTO

災害復旧（Disaster Recovery, DR）は、予期しない障害や災害からシステムを復旧し、ビジネス継続性を確保するための重要な戦略です。 AWSでは、柔軟なDRソリューションを低コストで構築できます。

RPOとRTOの定義

RPOとRTO

RPO (Recovery Point Objective)
目標復旧時点。障害発生時に許容できるデータ損失の最大時間。
例: RPO=1時間 → 最大1時間分のデータ損失を許容
RTO (Recovery Time Objective)
目標復旧時間。障害発生からサービス復旧までの目標時間。
例: RTO=4時間 → 4時間以内にサービスを復旧

RPO/RTOとコストのトレードオフ

RPOとRTOを短くするほど、コストが増加します。ビジネス要件（許容できるダウンタイムとデータ損失）に基づいて、 最適なDR戦略を選択することが重要です。

1. バックアップ&リストア

概要

定期的なバックアップをS3やAWS Backupに保存し、障害時に別リージョンでリストアする最もシンプルな戦略。

メリット

  • • 最も低コスト
  • • 実装がシンプル
  • • 複数世代のバックアップ保持可能

デメリット

  • • RPO/RTOが長い
  • • 復旧に時間がかかる
  • • 手動作業が多い
主要サービス: S3, AWS Backup, EBS Snapshots, RDS Snapshots

2. パイロットライト

概要

DRリージョンでコアシステム（データベース等）を最小構成で常時稼働させ、障害時に素早くスケールアップする戦略。

メリット

  • • RPOを短縮可能（分単位）
  • • RTOを短縮可能（時間単位）
  • • コアデータは常に同期

デメリット

  • • 中程度のコスト
  • • 復旧時に手動作業が必要
  • • スケールアップに時間がかかる
主要サービス: RDS リードレプリカ, Aurora Global Database, DynamoDB Global Tables

3. ウォームスタンバイ

概要

DRリージョンで縮小版の本番環境を常時稼働させ、障害時に即座にスケールアップする戦略。

メリット

  • • RPO/RTOが短い（分単位）
  • • 迅速な復旧が可能
  • • 定期的なDRテスト可能

デメリット

  • • 高コスト
  • • 複雑な構成管理
  • • 両環境の同期が必要
主要サービス: Auto Scaling, Route 53 Health Checks, Aurora Global Database

4. マルチサイトアクティブ-アクティブ

概要

複数リージョンで本番環境を同時稼働させ、トラフィックを分散する最高レベルのDR戦略。

メリット

  • • RPO/RTOがほぼゼロ
  • • 自動フェイルオーバー
  • • グローバルパフォーマンス向上

デメリット

  • • 最高コスト
  • • 複雑なアーキテクチャ
  • • データ整合性の課題
主要サービス: Route 53, Global Accelerator, Aurora Global Database, DynamoDB Global Tables
戦略 RPO RTO コスト 適用シーン
バックアップ&リストア 時間単位 時間～日単位 低 開発環境、非クリティカルシステム

12.3 AWS Elastic Disaster Recovery（AWS DRS）

AWS Elastic Disaster Recovery（旧CloudEndure Disaster Recovery）は、オンプレミスやAWS上のサーバーを 低コストで継続的にレプリケートし、迅速な復旧を実現するマネージドDRサービスです。

AWS DRSの主要機能

継続的データレプリケーション

ブロックレベルの継続的レプリケーションにより、RPOを秒単位まで短縮できます。

ポイントインタイムリカバリ

過去の任意の時点にリカバリできるため、ランサムウェア攻撃などからの復旧に有効です。

自動化されたDRテスト

本番環境に影響を与えずに、定期的なDRテストを実行できます。

クロスリージョン/クロスアカウント復旧

異なるリージョンやAWSアカウントへの復旧が可能です。

AWS DRSの料金モデル

AWS DRSは、レプリケート対象サーバーの台数とステージングエリアのストレージ使用量に基づいて課金されます。 復旧時のインスタンス起動コストは別途必要です。低コストで高度なDR環境を構築できます。

12.4 マルチリージョンアクティブ-アクティブ構成の設計

主要コンポーネント

Route 53 地理的ルーティング

ユーザーの地理的位置に基づいて、最も近いリージョンにトラフィックをルーティングします。

Aurora Global Database

複数リージョン間でデータを1秒未満でレプリケートし、読み取りスケーリングとDRを実現します。

S3 クロスリージョンレプリケーション

静的コンテンツや画像などのオブジェクトを自動的に複数リージョンにレプリケートします。

ヘルスチェックと自動フェイルオーバー

Route 53ヘルスチェックにより、障害リージョンを検出し、自動的に正常なリージョンにルーティングします。

12.5 DRテストとRunbook

DR戦略は、定期的なテストと明確な手順書（Runbook）がなければ、実際の障害時に機能しません。 DRテストとRunbookの作成は、DR戦略の成功に不可欠です。

DRテストの種類

1. ペーパーテスト（机上演習）

Runbookを読み合わせ、手順を確認するテスト。

  • • 頻度: 四半期ごと
  • • 目的: 手順の理解と改善点の洗い出し
  • • コスト: ゼロ

2. 部分的DRテスト

DRリージョンでデータベースやアプリケーションサーバーを起動し、接続性を確認するテスト。

  • • 頻度: 半年ごと
  • • 目的: 復旧手順の実行可能性確認
  • • コスト: 低～中

3. フルDRテスト

実際にプライマリリージョンを停止し、DRリージョンへの完全なフェイルオーバーを実施するテスト。

  • • 頻度: 年1回以上
  • • 目的: DR計画の有効性の完全検証
  • • コスト: 中～高

Runbookに含めるべき項目

  1. 1. 障害検知と初動対応
    • • CloudWatchアラーム、SNS通知の確認手順
    • • 障害の影響範囲の特定方法
    • • 関係者への連絡フロー
  2. 2. DR実行判断基準
    • • DR発動の意思決定者
    • • DR発動の判断基準（リージョン障害、データセンター障害等）
  3. 3. 詳細な復旧手順
    • • DRリージョンでのリソース起動手順（スクリーンショット付き）
    • • Route 53フェイルオーバーの切り替え手順
    • • データベースのプロモーション手順
    • • 各手順の実行者と所要時間
  4. 4. 動作確認とロールバック
    • • サービス復旧後の動作確認項目
    • • ロールバック手順（問題が発生した場合）
  5. 5. 事後対応
    • • ステークホルダーへの報告
    • • 障害の根本原因分析（RCA）
    • • Runbookの更新

Runbookのメンテナンス

システム構成の変更やAWSサービスのアップデートに伴い、Runbookも定期的に更新する必要があります。 DRテスト実施時に発見された問題点や改善点を必ずRunbookに反映しましょう。

実習: Route 53フェイルオーバーの設定

Route 53のヘルスチェックとフェイルオーバールーティングポリシーを使用して、シンプルなDR構成を構築してみましょう。

ステップ1: プライマリとセカンダリのリソース作成

2つの異なるリージョン（例: 東京とバージニア）にEC2インスタンスとALBを作成します。

  • • プライマリ: 東京リージョン（ap-northeast-1）
  • • セカンダリ: バージニアリージョン（us-east-1）

ステップ2: Route 53ヘルスチェックの作成

Route 53コンソールで、プライマリALBのヘルスチェックを作成します。

  • • エンドポイント: プライマリALBのDNS名
  • • プロトコル: HTTP/HTTPS
  • • パス: /health（ヘルスチェックエンドポイント）
  • • 間隔: 30秒

ステップ3: フェイルオーバールーティングポリシーの設定

Route 53レコード設定例 
レコード1 (プライマリ):
  名前: www.example.com
  タイプ: A - IPv4アドレス
  値/エイリアス: プライマリALB
  ルーティングポリシー: フェイルオーバー
  フェイルオーバーレコードタイプ: プライマリ
  ヘルスチェック: （ステップ2で作成）
  レコードID: primary

レコード2 (セカンダリ):
  名前: www.example.com
  タイプ: A - IPv4アドレス
  値/エイリアス: セカンダリALB
  ルーティングポリシー: フェイルオーバー
  フェイルオーバーレコードタイプ: セカンダリ
  ヘルスチェック: なし
  レコードID: secondary

ステップ4: フェイルオーバーのテスト

プライマリALBを停止し、フェイルオーバーが自動的に行われることを確認します。

  1. 1. ブラウザで www.example.com にアクセス（プライマリに接続）
  2. 2. プライマリALBを停止
  3. 3. 60-90秒待つ（ヘルスチェック失敗検出）
  4. 4. 再度 www.example.com にアクセス（セカンダリに接続）

注意: 実習後は必ずリソースを削除してください。Route 53ヘルスチェックも課金対象です。

理解度確認クイズ

  1. Q1. RPOとRTOの違いを説明してください。

    回答を表示

    RPO（Recovery Point Objective）は、障害発生時に許容できるデータ損失の最大時間です。 RTO（Recovery Time Objective）は、障害発生からサービス復旧までの目標時間です。 例: RPO=1時間は最大1時間分のデータ損失を許容、RTO=4時間は4時間以内にサービスを復旧することを意味します。

  2. Q2. DR戦略の4パターンを、コストが低い順に挙げてください。

    回答を表示

    1. バックアップ&リストア（最低コスト）
    2. パイロットライト（中程度のコスト）
    3. ウォームスタンバイ（高コスト）
    4. マルチサイトアクティブ-アクティブ（最高コスト）

  3. Q3. AWS Elastic Disaster Recovery（AWS DRS）の主な利点は何ですか？

    回答を表示

    AWS DRSは、ブロックレベルの継続的レプリケーションによりRPOを秒単位まで短縮し、 ポイントインタイムリカバリ機能により過去の任意の時点への復旧が可能です。 また、自動化されたDRテストにより本番環境に影響を与えずにDR計画の検証ができます。

  4. Q4. マルチリージョンアクティブ-アクティブ構成で、データベースのグローバルレプリケーションに適したAWSサービスは何ですか？

    回答を表示

    Aurora Global DatabaseとDynamoDB Global Tablesです。 Aurora Global Databaseは1秒未満でデータをレプリケートし、 DynamoDB Global Tablesはマルチリージョン間でマルチマスター書き込みを実現します。

第12章のまとめ

  • RPOはデータ損失の許容時間、RTOはサービス復旧までの目標時間
  • DR戦略は4パターン: バックアップ&リストア、パイロットライト、ウォームスタンバイ、マルチサイトアクティブ
  • AWS DRSは継続的レプリケーションとポイントインタイムリカバリを提供するマネージドDRサービス
  • マルチリージョンアクティブ-アクティブ構成は、Route 53、Aurora Global Database、S3 CRRを組み合わせて実現
  • DRテストとRunbookの作成・メンテナンスはDR戦略の成功に不可欠
