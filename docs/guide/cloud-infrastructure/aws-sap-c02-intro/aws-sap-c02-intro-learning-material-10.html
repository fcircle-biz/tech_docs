10.1 RDSマルチAZ構成の仕組み

Amazon RDS（Relational Database Service）のマルチAZ構成は、高可用性を実現するための重要な機能です。 プライマリDBインスタンスに障害が発生した場合、自動的にスタンバイDBインスタンスにフェイルオーバーし、ダウンタイムを最小限に抑えます。

マルチAZの主要な特徴

1. 同期レプリケーション

プライマリDBインスタンスへの書き込みは、スタンバイインスタンスに同期的にレプリケートされます。 これにより、データの整合性が保証され、フェイルオーバー時のデータ損失を防ぎます。

2. 自動フェイルオーバー

プライマリインスタンスに障害が発生すると、60-120秒以内にスタンバイインスタンスに自動的にフェイルオーバーします。 アプリケーションは同じDNSエンドポイントを使用し続けられるため、コード変更は不要です。

3. 自動バックアップとメンテナンス

スタンバイインスタンスからバックアップが取得されるため、プライマリインスタンスへの影響を回避できます。 また、OSパッチやデータベースアップグレードもスタンバイで先に実行されます。

マルチAZのユースケース

本番環境のRDSインスタンスでは、マルチAZ構成を有効化することが強く推奨されます。 特に、高い可用性が求められるビジネスクリティカルなアプリケーションでは必須の設定です。

マルチAZ有効化の手順

AWS CLI - RDSマルチAZ有効化 
# 既存のRDSインスタンスをマルチAZに変更
aws rds modify-db-instance \
    --db-instance-identifier mydb \
    --multi-az \
    --apply-immediately

# マルチAZ構成で新規作成
aws rds create-db-instance \
    --db-instance-identifier mydb \
    --db-instance-class db.t3.medium \
    --engine postgres \
    --master-username admin \
    --master-user-password mypassword \
    --allocated-storage 20 \
    --multi-az

10.2 Aurora Global DatabaseとAuroraレプリカ

Amazon Auroraは、MySQLおよびPostgreSQLと互換性のあるクラウドネイティブなリレーショナルデータベースです。 Aurora Global Databaseは、複数のリージョンにまたがるデータベースを提供し、グローバルな高可用性とディザスタリカバリを実現します。

Auroraの主要機能

Auroraレプリカ

  • • 最大15個のリードレプリカ
  • • ミリ秒単位のレプリケーション遅延
  • • 読み取りワークロードの分散
  • • 自動フェイルオーバー対象

Aurora Global Database

  • • 最大5つのセカンダリリージョン
  • • 1秒未満のクロスリージョンレプリケーション
  • • リージョン障害時の高速復旧（RPO: 1秒、RTO: 1分未満）
  • • グローバルな読み取りスケーリング

共有ストレージアーキテクチャ

  • • 3つのAZに6つのコピー
  • • 自動修復とバックアップ
  • • 最大128TBまで自動拡張
  • • ストレージレイヤーでのレプリケーション

自動フェイルオーバー

  • • 優先度ティアによる制御
  • • 通常30秒以内にフェイルオーバー
  • • アプリケーションの再接続のみ
  • • データ損失なし
AWS CLI - Aurora Global Database作成 
# Global Databaseクラスター作成
aws rds create-global-cluster \
    --global-cluster-identifier my-global-db \
    --engine aurora-mysql

# プライマリクラスター作成
aws rds create-db-cluster \
    --db-cluster-identifier primary-cluster \
    --engine aurora-mysql \
    --global-cluster-identifier my-global-db \
    --master-username admin \
    --master-user-password mypassword

# セカンダリリージョンにクラスター追加
aws rds create-db-cluster \
    --db-cluster-identifier secondary-cluster \
    --engine aurora-mysql \
    --global-cluster-identifier my-global-db \
    --region ap-southeast-1

10.3 RDSプロキシによる接続管理

Amazon RDS Proxyは、データベース接続のプーリングと管理を行うフルマネージド型のプロキシサービスです。 サーバーレスアプリケーション（AWS Lambda）や頻繁にスケールするアプリケーションでの接続管理を効率化します。

RDS Proxyの利点

1. 接続プーリングによるスケーラビリティ向上

多数のLambda関数や動的にスケールするアプリケーションからの接続を集約し、データベースへの接続数を削減します。 これにより、データベースのメモリとCPUリソースを節約できます。

2. フェイルオーバー時間の短縮

マルチAZフェイルオーバー時、RDS Proxyは新しいプライマリインスタンスへの接続を自動的に維持します。 アプリケーションは接続エラーを受け取ることなく、透過的にフェイルオーバーを経験できます。

3. IAM認証とSecrets Manager統合

IAM認証を使用してデータベースアクセスを管理でき、認証情報はSecrets Managerに安全に保存されます。 アプリケーションコード内にパスワードをハードコードする必要がありません。

RDS Proxyのユースケース

  • • サーバーレスアプリケーション（Lambda + RDS）
  • • 頻繁にスケールアップ・ダウンするアプリケーション
  • • 短時間の接続が多数発生するワークロード
  • • フェイルオーバー時のダウンタイムを最小化したい場合

DynamoDBグローバルテーブルは、複数のAWSリージョンにまたがるマルチマスター、マルチリージョンレプリケーションを提供します。 各リージョンで読み書きが可能で、数秒以内に他のリージョンに変更がレプリケートされます。

グローバルテーブルの特徴

マルチマスターレプリケーション

すべてのリージョンで読み書きが可能。ローカルリージョンへの書き込みは、通常1秒以内に他のリージョンにレプリケートされます。

自動競合解決

複数のリージョンで同じアイテムが同時に更新された場合、「最終書き込み優先（LWW）」ルールで競合が解決されます。

高可用性とディザスタリカバリ

1つのリージョンで障害が発生しても、他のリージョンのテーブルが引き続きサービスを提供します。RPO/RTOはほぼゼロです。

グローバルパフォーマンス

ユーザーの地理的に最も近いリージョンからデータにアクセスでき、低レイテンシーを実現します。

AWS CLI - DynamoDBグローバルテーブル作成 
# 東京リージョンでテーブル作成
aws dynamodb create-table \
    --table-name GlobalUsers \
    --attribute-definitions \
        AttributeName=UserId,AttributeType=S \
    --key-schema \
        AttributeName=UserId,KeyType=HASH \
    --billing-mode PAY_PER_REQUEST \
    --stream-specification StreamEnabled=true,StreamViewType=NEW_AND_OLD_IMAGES \
    --region ap-northeast-1

# グローバルテーブルに変換
aws dynamodb create-global-table \
    --global-table-name GlobalUsers \
    --replication-group \
        RegionName=ap-northeast-1 \
        RegionName=ap-southeast-1 \
        RegionName=us-east-1

10.5 バックアップ戦略とポイントインタイムリカバリ

データベースのバックアップ戦略は、データ損失のリスクを最小化し、ビジネス継続性を確保するための重要な要素です。 RDS、Aurora、DynamoDBはそれぞれ、自動バックアップとポイントインタイムリカバリ（PITR）をサポートしています。

手動スナップショット 任意のタイミングで作成。無期限保持 オンデマンドバックアップ。無期限保持
PITR 5分前から最大35日前の任意の時点 1秒前から35日前の任意の時点
クロスリージョンコピー スナップショットを他リージョンにコピー可能 AWS Backupまたはカスタムソリューション

バックアップのベストプラクティス

1. 自動バックアップを必ず有効化する

本番環境では自動バックアップを有効化し、保持期間を最低7日間（推奨は30日間）に設定します。

2. 重要な変更前に手動スナップショットを取得

データベーススキーマ変更、大規模なデータ移行、アプリケーションのメジャーアップデートの前には、必ず手動スナップショットを取得します。

3. クロスリージョンバックアップ

災害復旧のため、重要なスナップショットは別のリージョンにコピーします。リージョン全体の障害に備えることができます。

4. 定期的なリストアテスト

バックアップから正常に復元できることを定期的にテストします。バックアップが破損していたり、復元プロセスが不明確だったりすると、緊急時に役立ちません。

AWS CLI - RDSスナップショット管理 
# 手動スナップショット作成
aws rds create-db-snapshot \
    --db-instance-identifier mydb \
    --db-snapshot-identifier mydb-before-migration-2025-12-24

# スナップショットをクロスリージョンコピー
aws rds copy-db-snapshot \
    --source-db-snapshot-identifier arn:aws:rds:ap-northeast-1:123456789012:snapshot:mydb-snapshot \
    --target-db-snapshot-identifier mydb-dr-snapshot \
    --region us-east-1

# ポイントインタイムリカバリで復元
aws rds restore-db-instance-to-point-in-time \
    --source-db-instance-identifier mydb \
    --target-db-instance-identifier mydb-restored \
    --restore-time 2025-12-24T10:30:00Z

10.6 災害復旧のRTO/RPO設計

災害復旧（DR）戦略を設計する際、RTO（Recovery Time Objective）とRPO（Recovery Point Objective）は最も重要な指標です。 これらの目標値に基づいて、適切なデータベースアーキテクチャとバックアップ戦略を選択します。

RTO と RPO の定義

  • RTO（Recovery Time Objective）: システムを復旧させるまでに許容される最大時間
  • RPO（Recovery Point Objective）: データ損失として許容される最大時間（最後のバックアップからの経過時間）

データベースサービス別のRTO/RPO

構成 RPO RTO ユースケース
RDS シングルAZ 5分〜35日（バックアップ頻度次第） 数時間（手動復元） 開発・テスト環境
RDS マルチAZ 0（同期レプリケーション） 60-120秒 本番環境（単一リージョン）
Aurora + レプリカ 0（共有ストレージ） 30秒未満 高可用性が求められる本番環境
Aurora Global Database 1秒 1分未満 グローバルDR、マルチリージョン
DynamoDB Global Table 1秒 0（自動フェイルオーバー） グローバルアプリケーション

RTO/RPO要件に基づくアーキテクチャ選択

要件: RPO < 1分、RTO < 5分

推奨構成: Aurora Global Database または DynamoDB Global Table

  • • クロスリージョンレプリケーションで1秒未満のRPO
  • • 手動または自動フェイルオーバーで1分未満のRTO

要件: RPO = 0、RTO < 2分

推奨構成: RDS マルチAZ + RDS Proxy

  • • 同期レプリケーションでRPO = 0
  • • 自動フェイルオーバーで60-120秒のRTO
  • • RDS Proxyでさらにフェイルオーバー時間短縮

要件: RPO < 1時間、RTO < 4時間

推奨構成: RDS 自動バックアップ + クロスリージョンスナップショット

  • • 5分ごとのトランザクションログバックアップ
  • • 日次スナップショットを他リージョンにコピー
  • • 手動復元で4時間以内にサービス再開

コストとRTO/RPOのトレードオフ

RTO/RPOを小さくするほど、コストは高くなります。ビジネス影響を分析し、適切なバランスを見つけることが重要です。 例えば、全システムを最高レベルのDR構成にする必要はなく、重要度に応じて段階的に設計します。

実習: RDSマルチAZ構成の作成

以下の手順で、RDSマルチAZ構成を作成し、高可用性を体験してみましょう。

手順:

  1. AWSマネジメントコンソールでRDSサービスを開く
  2. PostgreSQLまたはMySQLでマルチAZ構成のDBインスタンスを作成
  3. EC2インスタンスから接続し、テストデータを挿入
  4. 「フェイルオーバー」操作を実行してスタンバイへの切り替えを体験
  5. フェイルオーバー前後でデータが保持されていることを確認

注意事項:

  • • db.t3.microインスタンスを使用してコストを抑える
  • • 実習後は必ずDBインスタンスを削除する
  • • マルチAZは通常のシングルAZの約2倍のコスト

理解度確認クイズ

  1. Q1. RDSマルチAZ構成の自動フェイルオーバーにかかる時間はどのくらいですか？

    回答を表示

    通常60-120秒です。DNSエンドポイントが新しいプライマリインスタンスを指すように自動的に更新されます。

  2. Q2. Aurora Global Databaseの主な利点は何ですか？

    回答を表示

    複数リージョンにまたがる高速レプリケーション（1秒未満）により、グローバルな読み取りスケーリングと高速ディザスタリカバリ（RPO: 1秒、RTO: 1分未満）を実現します。

  3. Q3. RDS Proxyが最も有効なユースケースは何ですか？

    回答を表示

    AWS Lambdaのようなサーバーレスアプリケーションや、頻繁にスケールするアプリケーションです。接続プーリングにより、データベースへの接続数を削減し、フェイルオーバー時のダウンタイムも短縮できます。

  4. Q4. DynamoDBグローバルテーブルの競合解決メカニズムは何ですか？

    回答を表示

    「最終書き込み優先（Last Writer Wins: LWW）」ルールです。複数のリージョンで同時に更新された場合、最も新しいタイムスタンプを持つ書き込みが優先されます。

第10章のまとめ

  • RDSマルチAZは同期レプリケーションにより、RPO = 0、RTO = 60-120秒を実現
  • Aurora Global Databaseは1秒未満のクロスリージョンレプリケーションでグローバルDRを提供
  • RDS Proxyはサーバーレスアプリケーションでの接続管理とフェイルオーバー時間短縮に有効
  • DynamoDBグローバルテーブルはマルチマスターでRPO/RTOほぼゼロを実現
  • 自動バックアップとPITRでデータ損失リスクを最小化し、クロスリージョンバックアップで災害対策を強化
  • RTO/RPO要件に基づき、適切なデータベースアーキテクチャとバックアップ戦略を選択
