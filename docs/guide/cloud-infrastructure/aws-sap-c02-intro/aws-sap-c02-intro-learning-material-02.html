高可用性設計の基礎 - 単一障害点の排除

エンタープライズシステムに求められる高可用性（HA）設計の基本原則を学びます。可用性指標、単一障害点の排除、マルチAZ構成の設計原則を理解します。

この章で学ぶこと

  • 可用性の定義と測定指標（SLA、RTO、RPO）
  • 単一障害点（SPOF）の特定と排除
  • マルチAZ構成の設計原則
  • アクティブ-アクティブ vs アクティブ-スタンバイ
  • 障害ドメインとブラスト半径の概念

2.1 可用性の定義と測定指標

高可用性（High Availability, HA）とは、システムが継続的に稼働し、ユーザーにサービスを提供し続ける能力のことです。 エンタープライズシステムでは、ビジネス要件に基づいた適切な可用性レベルの設定が不可欠です。

可用性の指標

可用性は通常、パーセンテージで表現され、「稼働時間 / (稼働時間 + 停止時間)」で計算されます。

可用性レベル 許容ダウンタイム（年間） 許容ダウンタイム（月間） 一般的な用途
99% 3.65日 7.2時間 内部システム
99.9%（スリーナイン） 8.76時間 43.8分 一般的なWebアプリケーション
99.99%（フォーナイン） 52.6分 4.38分 ビジネスクリティカルシステム
99.999%（ファイブナイン） 5.26分 26.3秒 金融取引、通信基盤

SLAとの関係

SLA（Service Level Agreement）は、サービス提供者と顧客の間で合意される稼働保証レベルです。 例えば、「月間99.9%の可用性を保証」という契約の場合、月に43.8分以内のダウンタイムが許容されます。

災害復旧の指標: RTO と RPO

障害発生時の復旧計画では、以下の2つの重要指標を定義します。

RTO (Recovery Time Objective)

目標復旧時間: 障害発生からシステムが復旧するまでの許容時間

例: 「障害発生から4時間以内にサービスを復旧する」

RPO (Recovery Point Objective)

目標復旧時点: 許容されるデータ損失の時間範囲

例: 「最大1時間分のデータ損失まで許容する」

RTO/RPOと設計の関係

RTO/RPOが厳しいほど、より高度な設計とコストが必要になります。 例えば、RPO=0（データ損失ゼロ）を実現するには、リアルタイムレプリケーションが必要です。 ビジネス要件と予算のバランスを考慮して目標値を設定しましょう。

2.2 単一障害点（SPOF）の特定と排除

単一障害点（Single Point of Failure, SPOF）とは、システム内の1つのコンポーネントが故障すると、 システム全体が停止してしまう箇所のことです。高可用性設計では、SPOFを特定し排除することが最優先課題です。

各層が冗長化され、1つのAZ障害に耐えられる

SPOFを排除するための戦略

1. 冗長性の導入

すべてのコンポーネントを複数配置し、1つが故障しても残りで継続稼働できるようにします。

  • • EC2インスタンスを複数のAZに配置
  • • ELBで複数ターゲットに負荷分散
  • • RDSをマルチAZ構成にする

2. 自動フェイルオーバー

障害検出時に自動的に正常なコンポーネントに切り替える仕組みを導入します。

  • • ELBのヘルスチェック機能
  • • RDSマルチAZの自動フェイルオーバー
  • • Route 53のDNSフェイルオーバー

3. マネージドサービスの活用

AWSのマネージドサービスは、高可用性が組み込まれています。

  • • S3: 11ナインの耐久性、複数AZに自動レプリケート
  • • DynamoDB: 3つのAZに自動レプリケート
  • • ELB: AWSが冗長性を管理

見落としやすいSPOF

アプリケーション層やデータベース層だけでなく、NATゲートウェイ、Direct Connect接続、 VPN接続、DNS設定なども潜在的なSPOFになり得ます。 システム全体のアーキテクチャ図を描き、すべてのコンポーネントを確認しましょう。

2.3 マルチAZ構成の設計原則

AWSリージョンは複数のアベイラビリティゾーン（AZ）で構成されています。 マルチAZ構成は、AZ障害からシステムを保護する最も基本的で効果的な高可用性設計です。

リージョンとAZの概念

AZの特徴

  • 物理的な分離: 各AZは独立した電源、ネットワーク、冷却設備を持つ
  • 低レイテンシー: AZ間は専用線で接続され、レイテンシーは数ミリ秒
  • 独立障害: 1つのAZの障害が他のAZに波及しない設計
  • 数量: 各リージョンは最低2つ、通常3つ以上のAZを持つ

マルチAZ設計のベストプラクティス

原則1: 最低2つのAZにリソースを配置

すべてのコンポーネントを最低2つのAZに分散配置します。

Auto Scaling設定例 
# マルチAZ構成のAuto Scalingグループ
AutoScalingGroup:
  DesiredCapacity: 4
  MinSize: 2
  MaxSize: 10
  AvailabilityZones:
    - ap-northeast-1a  # AZ-A
    - ap-northeast-1c  # AZ-C
  # 各AZに均等に配置
  DistributeOverAZs: true

原則2: サブネット設計でAZを意識する

各AZに対して、パブリックサブネットとプライベートサブネットを作成します。

サブネット AZ CIDR 用途
public-1a ap-northeast-1a 10.0.1.0/24 ELB、NAT Gateway
public-1c ap-northeast-1c 10.0.2.0/24 ELB、NAT Gateway
private-1a ap-northeast-1a 10.0.11.0/24 EC2、RDS
private-1c ap-northeast-1c 10.0.12.0/24 EC2、RDS

原則3: ELBのクロスゾーン負荷分散を有効化

クロスゾーン負荷分散を有効にすると、各AZ内のターゲット数が異なっても、 すべてのターゲットに均等にトラフィックが分散されます。

コストとのバランス

マルチAZ構成はリソースが倍になるためコストも増加します。 しかし、ビジネスクリティカルなシステムでは、ダウンタイムのビジネス損失がコストを上回るため、 マルチAZ構成は不可欠です。開発環境など重要度が低い環境では、単一AZ構成でコストを抑えることも検討しましょう。

2.4 アクティブ-アクティブ vs アクティブ-スタンバイ

高可用性構成には、「アクティブ-アクティブ」と「アクティブ-スタンバイ」の2つの主要なパターンがあります。 それぞれの特徴とユースケースを理解し、適切に選択することが重要です。

アクティブ-アクティブ

アクティブ"] LB -->|50%| Server2["サーバー2
アクティブ"] style Server1 fill:#86efac style Server2 fill:#86efac style LB fill:#bfdbfe

特徴:

  • • すべてのリソースが常時稼働
  • • トラフィックを複数のリソースに分散
  • • リソースを無駄なく活用
  • • 障害時の影響が小さい（部分的な性能低下のみ）

適用例:

  • • ELB配下の複数EC2インスタンス
  • • DynamoDBグローバルテーブル
  • • マルチリージョン構成

アクティブ-スタンバイ

アクティブ"] LB -.待機.-> Server2["サーバー2
スタンバイ"] Server1 -.レプリケーション.-> Server2 style Server1 fill:#86efac style Server2 fill:#fef3c7 style LB fill:#bfdbfe

特徴:

  • • 1つがアクティブ、残りはスタンバイ
  • • 障害時に自動フェイルオーバー
  • • スタンバイリソースは通常時は待機
  • • シンプルな構成

適用例:

  • • RDSマルチAZ構成
  • • Route 53フェイルオーバールーティング
  • • NAT Gatewayのバックアップ

選択基準

観点 アクティブ-アクティブ アクティブ-スタンバイ
リソース効率 高い（すべて活用） 低い（スタンバイは待機）
フェイルオーバー時間 瞬時（既に稼働中） 数秒〜数分
構成の複雑さ 複雑（負荷分散、セッション管理） シンプル
データ整合性 複雑（競合解決が必要な場合あり） シンプル（アクティブのみ書き込み）
推奨用途 ステートレスアプリケーション ステートフルアプリケーション
（データベース等）

ハイブリッドアプローチ

実際のシステムでは、両方のパターンを組み合わせることが一般的です。 例えば、Webサーバー層はアクティブ-アクティブ、データベース層はアクティブ-スタンバイという構成です。

2.5 障害ドメインとブラスト半径の概念

大規模システムの設計では、「障害ドメイン」と「ブラスト半径」という概念が重要です。 これらを理解することで、障害の影響範囲を最小化する設計が可能になります。

障害ドメイン（Fault Domain）

障害ドメインとは、共通の障害により同時に影響を受ける可能性があるコンポーネントの集合です。

設計原則: リソースを異なる障害ドメインに分散配置することで、 1つの障害ドメインで障害が発生しても、システム全体は継続稼働できます。

ブラスト半径（Blast Radius）

ブラスト半径とは、1つの障害やインシデントが影響を及ぼす範囲のことです。 ブラスト半径を小さくすることで、障害の影響を最小限に抑えることができます。

影響範囲が1つのサービスに限定される

ブラスト半径を小さくする戦略

1. マイクロサービスアーキテクチャ

システムを小さな独立したサービスに分割し、各サービスが独自のデータベースを持つ構成にします。

2. セルベースアーキテクチャ

システムを独立した「セル」に分割し、各セルが完全に自己完結する構成にします。 1つのセルで障害が発生しても、他のセルは影響を受けません。

3. リージョン分離

顧客やワークロードを複数のリージョンに分散配置し、 1つのリージョンの障害が他のリージョンに波及しないようにします。

4. デプロイメントの段階化

新しいコードを一度に全環境にデプロイせず、段階的にロールアウトします（カナリアリリース、Blue/Greenデプロイ等）。

Amazon の「Shuffle Sharding」

Amazonでは、顧客を複数のシャード（分割されたリソースプール）にランダムに割り当てる 「Shuffle Sharding」という手法を使用しています。 これにより、1つのリソースプールで問題が発生しても、影響を受ける顧客数を最小限に抑えられます。

理解度確認クイズ

  1. Q1. 99.99%の可用性を保証する場合、月間で許容されるダウンタイムは何分ですか？

    回答を表示

    約4.38分です。99.99%の可用性では、年間52.6分のダウンタイムが許容されます。

  2. Q2. RTOとRPOの違いを説明してください。

    回答を表示

    RTO（Recovery Time Objective）: 障害発生からシステムが復旧するまでの許容時間
    RPO（Recovery Point Objective）: 許容されるデータ損失の時間範囲
    例えば、RTO=4時間、RPO=1時間の場合、「1時間分のデータ損失は許容し、4時間以内にシステムを復旧する」という意味です。

  3. Q3. 単一障害点（SPOF）とは何ですか？具体例を挙げてください。

    回答を表示

    SPOFとは、1つのコンポーネントが故障するとシステム全体が停止してしまう箇所のことです。
    例:
    • 単一のEC2インスタンスでアプリケーションを稼働
    • RDSをシングルAZ構成で運用
    • 1つのNAT Gatewayのみを使用
    これらはすべて、障害時にシステムが停止するSPOFです。

  4. Q4. アクティブ-アクティブとアクティブ-スタンバイ構成の主な違いは何ですか？

    回答を表示

    アクティブ-アクティブ: すべてのリソースが常時稼働し、トラフィックを分散処理。リソース効率が高い。
    アクティブ-スタンバイ: 1つがアクティブで稼働し、残りはスタンバイで待機。障害時に自動フェイルオーバー。構成がシンプル。
    Webサーバー層はアクティブ-アクティブ、データベース層はアクティブ-スタンバイが一般的です。

  5. Q5. ブラスト半径を小さくすることがなぜ重要ですか？

    回答を表示

    ブラスト半径を小さくすることで、1つの障害やインシデントが影響を及ぼす範囲を最小限に抑えられます。 これにより、システム全体の可用性が向上し、障害発生時の影響を受けるユーザー数や機能を限定できます。 マイクロサービス、セルベースアーキテクチャ、段階的デプロイなどが有効な戦略です。

第2章のまとめ

  • 可用性はSLA、RTO、RPOで定義され、ビジネス要件に基づいて設定する
  • 単一障害点（SPOF）を特定し排除することが高可用性設計の最優先課題
  • マルチAZ構成は、AZ障害からシステムを保護する基本的で効果的な設計
  • アクティブ-アクティブはリソース効率が高く、アクティブ-スタンバイはシンプルな構成
  • 障害ドメインの分離とブラスト半径の最小化により、障害の影響範囲を制御する
