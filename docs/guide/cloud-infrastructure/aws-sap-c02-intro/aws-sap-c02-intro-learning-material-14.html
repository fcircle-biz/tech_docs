監視・ログ・トラブルシューティング

大規模環境における監視設計とトラブルシューティング手法を習得します。CloudWatch、X-Ray、EventBridgeを活用した運用設計を学びます。

14.1 監視の基本原則と設計思想

エンタープライズ環境における監視は、単なるシステムの死活監視だけでなく、パフォーマンス、セキュリティ、コスト、ユーザー体験を包括的に可視化する必要があります。 AWS Well-Architectedの「運用上の優秀性」の柱では、観測性（Observability）が重視されています。

メトリクス

時系列の数値データ。CPU使用率、メモリ使用量、レスポンスタイムなど。

  • • CloudWatchメトリクス
  • • カスタムメトリクス
  • • アラーム設定

ログ

  • • CloudWatch Logs

トレース

SAP試験でのポイント

問題文で「リクエストが遅い原因を特定したい」→X-Ray、「ログから特定パターンを抽出したい」→Logs Insights、「CPU使用率が高いときにアラートを出したい」→CloudWatchアラーム、というように、要件に応じた適切なサービスを選択できるようにしましょう。

14.2 CloudWatchメトリクスとアラーム設計

CloudWatchは、AWSリソースとアプリケーションのメトリクスを収集・追跡するサービスです。 標準メトリクスに加えて、カスタムメトリクスを送信することで、ビジネス固有の指標も監視できます。

メトリクスの種類

メトリクスタイプ 説明 例
標準メトリクス AWSサービスが自動的に送信 EC2のCPU使用率、RDSの接続数
詳細モニタリング 1分間隔のメトリクス（有料） EC2の詳細メトリクス
カスタムメトリクス アプリケーションから送信 購入数、ログイン数、エラー率

カスタムメトリクスの送信

Python - カスタムメトリクス送信例 
import boto3
from datetime import datetime

# CloudWatchクライアント作成
cloudwatch = boto3.client('cloudwatch')

# カスタムメトリクス送信
cloudwatch.put_metric_data(
    Namespace='MyApp/Business',  # 名前空間（必須）
    MetricData=[
        {
            'MetricName': 'OrderCount',  # メトリクス名
            'Value': 10,  # 値
            'Unit': 'Count',  # 単位
            'Timestamp': datetime.utcnow(),
            'Dimensions': [  # ディメンション（フィルタ用）
                {'Name': 'Environment', 'Value': 'Production'},
                {'Name': 'Region', 'Value': 'ap-northeast-1'}
            ]
        }
    ]
)

Terraform - CloudWatchアラーム設定例 
resource "aws_cloudwatch_metric_alarm" "high_cpu" {
  alarm_name          = "web-high-cpu"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 2  # 2回連続で閾値超え
  metric_name         = "CPUUtilization"
  namespace           = "AWS/EC2"
  period              = 300  # 5分間隔
  statistic           = "Average"
  threshold           = 80  # 閾値80%
  alarm_description   = "CPU使用率が80%を超えた場合"

  # アクション
  alarm_actions = [
    aws_sns_topic.alerts.arn  # SNS通知
  ]

  # ディメンション（どのEC2インスタンスか）
  dimensions = {
    InstanceId = aws_instance.web.id
  }
}

アラーム設計のベストプラクティス

  • • 複数の評価期間: 一時的なスパイクでアラームが鳴らないよう、2回以上の連続で閾値超えを条件にする
  • • 適切な統計: Average、Sum、Maximum、Minimumなど、メトリクスの性質に応じて選択
  • • 複合アラーム: 複数のアラームを組み合わせて、より正確な異常検知を実現
  • • アラーム疲労の防止: 重要度に応じてアラームを分類し、通知先を分ける

14.3 CloudWatch Logs Insightsによるログ分析

CloudWatch Logs Insightsは、SQLライクなクエリ言語を使用して大量のログデータを高速に検索・分析できるサービスです。 従来のgrepやawkよりも柔軟で強力な分析が可能です。

Logs Insightsクエリ例

例1: エラーログの集計

fields @timestamp, @message
| filter @message like /ERROR/
| stats count() by bin(5m)
| sort @timestamp desc

「ERROR」を含むログを5分ごとに集計し、時系列で表示

例2: レスポンスタイムの分析

fields @timestamp, @message
| parse @message /duration: (?<duration>\d+) ms/
| stats avg(duration), max(duration), pct(duration, 95)
| sort @timestamp desc

ログからレスポンスタイムを抽出し、平均、最大、95パーセンタイルを計算

例3: ユーザー別のアクセス集計

fields @timestamp, userId
| stats count() as accessCount by userId
| sort accessCount desc
| limit 10

ユーザーIDごとのアクセス数を集計し、上位10件を表示

コスト最適化のポイント

CloudWatch Logsは、保存データ量とクエリのスキャン量で課金されます。 不要なログは早めに削除し、保存期間を適切に設定しましょう。また、S3にエクスポートして長期保存すると、コストを削減できます。

14.4 X-Rayによる分散トレーシング

AWS X-Rayは、マイクロサービスアーキテクチャや分散システムにおいて、リクエストの経路を追跡し、ボトルネックやエラーの発生箇所を特定するサービスです。

X-Rayの仕組み

サービスマップ"] User --> ALB ALB --> API API --> Lambda1 Lambda1 --> DDB Lambda1 --> Lambda2 Lambda2 --> DDB ALB -.-> XRay API -.-> XRay Lambda1 -.-> XRay Lambda2 -.-> XRay DDB -.-> XRay style User fill:#fed7aa style ALB fill:#dbeafe style API fill:#d1fae5 style Lambda1 fill:#e9d5ff style Lambda2 fill:#e9d5ff style DDB fill:#fef3c7 style XRay fill:#fecaca

X-Rayの主要機能

1. サービスマップ

リクエストが通過するサービス間の依存関係を視覚化。各サービスの応答時間とエラー率も表示。

2. トレース分析

個々のリクエストの詳細な実行時間と、各コンポーネントでの処理時間を確認。

3. セグメントとサブセグメント

セグメント = サービス単位の処理、サブセグメント = 関数やDBクエリなどの細かい処理単位。

X-Ray SDKの統合例（Lambda）

Python - Lambda関数でX-Rayを使用 
from aws_xray_sdk.core import xray_recorder
from aws_xray_sdk.core import patch_all
import boto3

# AWS SDKをパッチしてX-Rayトレースを有効化
patch_all()

dynamodb = boto3.resource('dynamodb')
table = dynamodb.Table('Users')

def lambda_handler(event, context):
    # サブセグメントを作成（任意の処理を追跡）
    with xray_recorder.capture('DynamoDB Query'):
        response = table.get_item(Key={'userId': '123'})

    # アノテーション（フィルタ可能）
    xray_recorder.put_annotation('userId', '123')

    # メタデータ（検索不可、詳細情報用）
    xray_recorder.put_metadata('response', response)

    return response

X-Rayの活用シーン

  • • パフォーマンス問題の特定: どのサービスが遅いかを一目で把握
  • • エラーの原因調査: エラーが発生したリクエストの経路を追跡
  • • 依存関係の可視化: マイクロサービス間の通信パターンを理解
  • • 異常検知: 通常と異なるパターンを自動検出

14.5 EventBridgeによるイベント駆動アーキテクチャ

Amazon EventBridgeは、AWSサービスやアプリケーションからのイベントを受け取り、ルールに基づいてターゲット（Lambda、SQS、SNSなど）にルーティングするサーバーレスイベントバスサービスです。

EventBridgeルールの設定例

Terraform - EC2停止時にSlack通知 
resource "aws_cloudwatch_event_rule" "ec2_stopped" {
  name        = "ec2-stopped-notification"
  description = "EC2インスタンスが停止したときに通知"

  event_pattern = jsonencode({
    source      = ["aws.ec2"]
    detail-type = ["EC2 Instance State-change Notification"]
    detail = {
      state = ["stopped"]
    }
  })
}

resource "aws_cloudwatch_event_target" "lambda" {
  rule      = aws_cloudwatch_event_rule.ec2_stopped.name
  target_id = "SendToLambda"
  arn       = aws_lambda_function.notify_slack.arn
}

EventBridgeの活用パターン

インシデント対応の自動化

EC2障害、セキュリティグループ変更、IAMポリシー変更などのイベントを検知し、自動的にチケット作成やSlack通知を実行。

ワークフローの自動化

S3にファイルがアップロードされたらLambdaで処理し、DynamoDBに保存し、SNSで通知する一連のフローを自動化。

クロスアカウントイベント

異なるAWSアカウント間でイベントを共有し、統合的な監視やワークフローを実現。

スケジュールベースの実行

cron式またはrate式で定期実行（例: 毎日0時にバックアップLambdaを実行）。

Run Commandの活用例

AWS CLI - 複数インスタンスでディスク使用率確認 
aws ssm send-command \
  --document-name "AWS-RunShellScript" \
  --targets "Key=tag:Environment,Values=Production" \
  --parameters 'commands=["df -h"]' \
  --region ap-northeast-1

# 実行結果の取得
aws ssm get-command-invocation \
  --command-id <command-id> \
  --instance-id <instance-id>

Parameter Storeの活用

Parameter Storeは、設定値やシークレット情報を集中管理できるサービスです。SecureStringタイプを使用すると、KMSで暗号化されます。 

# パラメータの作成
aws ssm put-parameter \
  --name "/myapp/db/password" \
  --value "MySecurePassword123" \
  --type "SecureString" \
  --key-id "alias/aws/ssm"

# パラメータの取得
aws ssm get-parameter \
  --name "/myapp/db/password" \
  --with-decryption

Systems Managerのメリット

  • • SSHポート不要: Session Managerを使用すれば、SSHポート22を開放せずにリモート接続可能
  • • 一元管理: 複数のEC2インスタンスやオンプレミスサーバーを統合的に管理
  • • 監査証跡: すべての操作がCloudTrailに記録され、コンプライアンス要件に対応
  • • 自動化: State Managerで設定状態を維持し、ドリフト（設定の逸脱）を防止

実習: CloudWatchアラームとEventBridgeの設定

EC2インスタンスのCPU使用率が高い場合にSNS通知を送り、EventBridgeでインスタンスの状態変更を検知する設定を行います。

ステップ1: SNSトピックの作成

  1. SNSコンソールで新しいトピック「ec2-alerts」を作成
  2. サブスクリプションを作成し、メールアドレスを登録
  3. 確認メールからサブスクリプションを承認

ステップ2: CloudWatchアラームの作成

  1. CloudWatchコンソールで「アラーム」→「アラームの作成」
  2. メトリクス選択: EC2 → Per-Instance Metrics → CPUUtilization
  3. 条件設定: 閾値 > 80、期間 5分、評価期間 2
  4. 通知設定: 作成したSNSトピックを選択

ステップ3: EventBridgeルールの作成

  1. EventBridgeコンソールで「ルールを作成」
  2. イベントパターン: EC2 Instance State-change Notification
  3. ターゲット: SNSトピック「ec2-alerts」を選択
  4. ルールを有効化

ステップ4: 動作確認

  1. EC2インスタンスでstressコマンドなどを実行してCPU使用率を上げる
  2. CloudWatchアラームが「ALARM」状態になることを確認
  3. SNS経由でメール通知が届くことを確認
  4. EC2インスタンスを停止し、EventBridge経由で通知が届くことを確認

理解度確認クイズ

  1. Q1. CloudWatchメトリクスの標準解像度と詳細モニタリングの違いは何ですか？

    回答を表示

    標準解像度は5分間隔、詳細モニタリングは1分間隔でメトリクスを収集します。 詳細モニタリングは有料ですが、より迅速な異常検知とAuto Scalingの応答性向上が可能です。

  2. Q2. X-Rayのサービスマップで確認できる情報を3つ挙げてください。

    回答を表示

    1. サービス間の依存関係（呼び出し関係）
    2. 各サービスの平均応答時間とレイテンシー
    3. エラー率とスロットリング率

  3. Q3. EventBridgeとCloudWatch Eventsの違いは何ですか？

    回答を表示

    EventBridgeはCloudWatch Eventsの後継サービスで、基本機能は同じですが、以下の拡張機能があります： 1. カスタムイベントバスの作成が可能、2. SaaSアプリケーション（Salesforce、Zendeskなど）との統合、3. スキーマレジストリによるイベント構造の管理。

  4. Q4. Systems Manager Parameter StoreとSecrets Managerの使い分け基準を説明してください。

    回答を表示

    Parameter Store: 設定値の管理に適しており、無料枠が大きい（標準パラメータは無料、高度なパラメータは有料）。
    Secrets Manager: シークレット情報（パスワード、APIキー）の管理に特化しており、自動ローテーション機能がある。コストはParameter Storeより高いが、セキュリティ機能が充実。

第14章のまとめ

  • 監視の3本柱は、メトリクス（CloudWatch）、ログ（CloudWatch Logs）、トレース（X-Ray）
  • CloudWatchアラームは、複数の評価期間と適切な統計を使用して、アラーム疲労を防止
  • Logs InsightsはSQLライクなクエリでログを高速分析、S3エクスポートでコスト最適化
  • X-Rayは分散システムのリクエスト経路を追跡し、ボトルネックとエラーの発生箇所を特定
  • EventBridgeはイベント駆動アーキテクチャを実現し、インシデント対応を自動化
  • Systems ManagerはEC2やオンプレミスサーバーの運用管理を統合的に実施
