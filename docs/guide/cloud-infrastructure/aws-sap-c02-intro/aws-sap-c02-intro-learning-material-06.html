CloudFront - グローバルコンテンツ配信

CloudFrontを使用したCDN設計と、パフォーマンス最適化の手法を学びます。

6.1 CDNの基礎とCloudFrontの概要

CDN（Content Delivery Network）は、コンテンツを世界中のエッジロケーションにキャッシュし、 ユーザーに最も近い場所からコンテンツを配信することで、レイテンシーを削減し、パフォーマンスを向上させるサービスです。

CloudFrontの主な利点

  • 低レイテンシー: ユーザーに近いエッジロケーションから配信
  • 高可用性: 450以上のグローバルエッジロケーション
  • オリジンの負荷軽減: キャッシュにより直接アクセスを削減
  • DDoS保護: AWS ShieldとWAF統合による保護
  • コスト削減: データ転送コストの削減

エッジロケーションとリージョナルエッジキャッシュ

CloudFrontは、エッジロケーションとリージョナルエッジキャッシュの2層キャッシュ構造を持ちます。

6.2 オリジンの種類と設定

CloudFrontは、複数のオリジンタイプをサポートしており、用途に応じて適切なオリジンを選択できます。

サポートされるオリジンタイプ

S3オリジン

  • • 静的コンテンツ配信
  • • OAC/OAIで保護
  • • バケットポリシー自動設定
  • • S3バケット専用

ALBオリジン

  • • 動的コンテンツ配信
  • • カスタムヘッダーで保護
  • • HTTP/HTTPSサポート
  • • ヘルスチェック統合

カスタムオリジン

  • • オンプレミスサーバー
  • • EC2インスタンス
  • • 任意のHTTPサーバー
  • • MediaPackage等

複数オリジンとオリジングループ

CloudFrontは、単一のディストリビューションで複数のオリジンを設定でき、パスパターンに基づいて異なるオリジンにルーティングできます。

オリジングループによるフェイルオーバー

オリジングループを使用すると、プライマリオリジンが失敗した場合に自動的にセカンダリオリジンにフェイルオーバーできます。 これにより、高可用性を実現できます。

6.3 キャッシュ動作とTTL設定

キャッシュの効果的な設定は、パフォーマンスとコストの最適化に不可欠です。

TTL（Time To Live）の種類

TTL種類 説明 優先順位
オリジンヘッダー Cache-Control, Expiresヘッダー 高（最優先）
CloudFront設定 最小/デフォルト/最大TTL 中（オリジンヘッダーがない場合）
デフォルト値 24時間（86400秒） 低（すべて未設定の場合）

キャッシュポリシーとオリジンリクエストポリシー

CloudFrontは、キャッシュポリシーとオリジンリクエストポリシーを使用して、キャッシュ動作をきめ細かく制御できます。

キャッシュポリシー

キャッシュキーに含める要素を定義

  • • ヘッダー（Host, Authorization等）
  • • クエリ文字列
  • • Cookie
  • • TTL設定
  • • 圧縮の有効化

オリジンリクエストポリシー

オリジンに転送する情報を定義

  • • カスタムヘッダー
  • • すべてのクエリ文字列
  • • すべてのCookie
  • • CloudFront-Viewer-* ヘッダー
Cache-Controlヘッダーの例 
# 静的コンテンツ（画像、CSS、JS）
Cache-Control: public, max-age=31536000, immutable

# 動的コンテンツ（頻繁に更新）
Cache-Control: public, max-age=300, s-maxage=600

# キャッシュ不可
Cache-Control: no-cache, no-store, must-revalidate

6.4 オリジンアクセスコントロール（OAC）

OAC（Origin Access Control）は、S3バケットへの直接アクセスを防ぎ、CloudFront経由でのみアクセスを許可するセキュリティ機能です。

OACとOAIの違い

機能 OAC（推奨） OAI（レガシー）
リリース 2022年 2009年
SSE-KMS暗号化 ✅ サポート ❌ 未サポート
動的リクエスト ✅ PUT/POST/DELETE ❌ GETのみ
署名方式 AWS Signature Version 4 CloudFront署名
推奨度 ✅ 新規はすべてOAC ⚠️ 移行推奨
OAC用のS3バケットポリシー 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudFrontServicePrincipal",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudfront.amazonaws.com"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*",
      "Condition": {
        "StringEquals": {
          "AWS:SourceArn": "arn:aws:cloudfront::123456789012:distribution/EXAMPLEID"
        }
      }
    }
  ]
}

6.5 Lambda@EdgeとCloudFront Functions

Lambda@EdgeとCloudFront Functionsは、エッジロケーションでコードを実行し、リクエスト/レスポンスをカスタマイズできる機能です。

Lambda@Edge vs CloudFront Functions

観点 CloudFront Functions Lambda@Edge
実行場所 450以上のエッジロケーション 13のリージョナルエッジキャッシュ
実行時間 1ミリ秒未満 最大5秒（Viewer）、30秒（Origin）
メモリ 2MB 128MB - 10GB
言語 JavaScript（ES5/6） Node.js、Python
ネットワーク 不可 可（API呼び出し等）
用途 軽量な変換・検証 複雑な処理・API連携

ユースケース例

CloudFront Functions

  • • HTTPヘッダーの追加/変更
  • • URLリライト/リダイレクト
  • • リクエストの検証（Authorizationヘッダー等）
  • • A/Bテストのルーティング

Lambda@Edge

  • • 画像リサイズ（外部API呼び出し）
  • • ユーザー認証（DynamoDBやCognito連携）
  • • レスポンスの動的生成
  • • ボット検出と防御

6.6 署名付きURLと署名付きCookie

署名付きURLとCookieは、プライベートコンテンツへのアクセスを制御し、特定のユーザーのみがコンテンツにアクセスできるようにする機能です。

署名付きURLと署名付きCookieの使い分け

署名付きURL

使用場面: 単一ファイルへのアクセス制御

  • ✅ 個別ファイルのダウンロード
  • ✅ モバイルアプリでのアクセス
  • ✅ ファイルごとに異なる有効期限
  • ❌ 複数ファイルへのアクセス

署名付きCookie

使用場面: 複数ファイルへのアクセス制御

  • ✅ HLS動画ストリーミング（複数セグメント）
  • ✅ Webサイト全体の保護
  • ✅ URLを変更したくない場合
  • ❌ Cookie非対応クライアント

署名付きURLの生成

署名付きURL生成（Python + boto3） 
import boto3
from datetime import datetime, timedelta

# CloudFrontクライアント作成
cloudfront_client = boto3.client('cloudfront')

# 署名付きURL生成
url = cloudfront_client.generate_presigned_url(
    'get_object',
    Params={
        'Bucket': 'my-bucket',
        'Key': 'private/video.mp4'
    },
    ExpiresIn=3600  # 1時間有効
)

セキュリティのベストプラクティス

署名付きURLを使用する場合、秘密鍵は安全に管理してください。 AWS Secrets Managerやパラメータストアを使用して、鍵をコード外で管理することを推奨します。

CloudFrontは、AWS WAF、AWS Shield、AWS Certificateと統合して、包括的なセキュリティ対策を実現できます。

AWS Shield（DDoS保護）

  • • Shield Standard: すべてのCloudFrontディストリビューションに無料で自動適用
  • • Shield Advanced: 高度なDDoS保護と24/7サポート（有料）

AWS WAF（Webアプリケーションファイアウォール）

  • • SQLインジェクション、XSS攻撃のブロック
  • • IPアドレスベースのアクセス制御
  • • レート制限（DDoS対策）
  • • マネージドルール（OWASP Top 10等）

AWS Certificate Manager（SSL/TLS証明書）

  • • 無料のSSL/TLS証明書
  • • 自動更新（手動更新不要）
  • • CloudFrontディストリビューションに直接統合
  • • カスタムドメインのHTTPS対応

実習: S3 + CloudFrontでWebサイト配信

S3バケットに静的Webサイトをホストし、CloudFront経由で配信してみましょう。

  1. 1. S3バケット作成:
    • • パブリックアクセスをブロック（CloudFront経由のみ許可）
    • • index.htmlをアップロード
  2. 2. CloudFrontディストリビューション作成:
    • • オリジン: S3バケット
    • • OAC設定（自動作成）
    • • デフォルトルートオブジェクト: index.html
    • • キャッシュポリシー: CachingOptimized
  3. 3. S3バケットポリシー更新:
    • • CloudFrontからのアクセスのみ許可
    • • OACのARNを含める
  4. 4. 動作確認:
    • • CloudFrontドメイン名でアクセス
    • • S3直接アクセスがブロックされることを確認
    • • キャッシュ動作の確認（2回目以降の高速化）

理解度確認クイズ

  1. Q1. CloudFrontのキャッシュ階層は何層ですか？

    回答を表示

    2層です。エッジロケーション（450以上）とリージョナルエッジキャッシュ（13拠点）の2層構造です。

  2. Q2. OACとOAIの最も大きな違いは何ですか？

    回答を表示

    OACはSSE-KMS暗号化とPUT/POST/DELETEメソッドをサポートしています。OAIはこれらをサポートしておらず、GETのみです。新規はすべてOACを使用すべきです。

  3. Q3. CloudFront FunctionsとLambda@Edgeの選択基準は？

    回答を表示

    CloudFront Functionsは、軽量な処理（ヘッダー変更、URLリライト等）に適しています。 Lambda@Edgeは、外部API呼び出しや複雑な処理（認証、画像リサイズ等）に適しています。

  4. Q4. 署名付きURLと署名付きCookieの使い分けは？

    回答を表示

    署名付きURLは単一ファイルへのアクセス制御に適しています。 署名付きCookieは複数ファイル（動画ストリーミング等）やWebサイト全体の保護に適しています。

第6章のまとめ

  • CloudFrontは2層キャッシュ構造（エッジロケーション + リージョナルエッジキャッシュ）
  • オリジンはS3、ALB、カスタムの3種類。OAC（推奨）でS3を保護
  • キャッシュポリシーとオリジンリクエストポリシーで細かい制御が可能
  • Lambda@Edge（複雑）とCloudFront Functions（軽量）を使い分け
  • 署名付きURL/Cookieでプライベートコンテンツを保護
  • AWS WAF、Shield、ACMと統合して包括的なセキュリティを実現
