8.1 VPCピアリングとその制限事項

VPCピアリングは、2つのVPC間でプライベートな通信を可能にする接続方式です。 シンプルな構成に適していますが、大規模環境では制限事項を理解する必要があります。

VPCピアリングの仕組み

graph LR subgraph VPC_A["VPC A (10.0.0.0/16)"] A1[Subnet A1] A2[Subnet A2] end subgraph VPC_B["VPC B (10.1.0.0/16)"] B1[Subnet B1] B2[Subnet B2] end VPC_A -."|VPCピアリング接続|".- VPC_B style VPC_A fill:#dbeafe style VPC_B fill:#d1fae5

VPCピアリングの特徴

  • • プライベートIPアドレスでの通信（インターネットを経由しない）
  • • 異なるAWSアカウント・リージョン間でも接続可能
  • • データ転送コストはクロスAZ・クロスリージョン転送と同じ
  • • 1対1の接続（1つのピアリング接続で2つのVPCのみ接続）

VPCピアリングの制限事項

重要な制限事項

  • • 推移的ピアリング不可: VPC A-B、B-Cがピアリングされていても、A-C間の通信は不可
  • • CIDRブロックの重複不可: ピアリングするVPC間でIPアドレス範囲が重複してはいけない
  • • フルメッシュ接続が必要: N個のVPCを全て接続するには N×(N-1)/2 個のピアリング接続が必要
  • • 最大接続数: 1つのVPCあたり最大125個のピアリング接続

フルメッシュ接続の課題

6個のピアリング接続が必要"] style Note fill:#fef3c7

VPCの数が増えると、必要なピアリング接続数が指数関数的に増加します。 このような場合は、次節で説明するTransit Gatewayの利用を検討すべきです。

8.2 Transit Gatewayによるハブ&スポーク構成

AWS Transit Gateway（TGW）は、複数のVPCとオンプレミスネットワークを中央ハブ経由で接続できるマネージドサービスです。 大規模なネットワーク環境において、VPCピアリングの制限を解消します。

Transit Gatewayのアーキテクチャ

graph TB TGW["Transit Gateway
（ハブ）"] VPC_A[VPC A] VPC_B[VPC B] VPC_C[VPC C] VPC_D[VPC D] OnPrem[オンプレミス
VPN/Direct Connect] VPC_A --- TGW VPC_B --- TGW VPC_C --- TGW VPC_D --- TGW OnPrem --- TGW Note["全てのVPCがTransit Gatewayを
経由して相互通信可能"] style TGW fill:#fed7aa style Note fill:#d1fae5

Transit Gatewayの利点

  • • スケーラビリティ: 1つのTransit Gatewayに最大5,000個のVPCをアタッチ可能
  • • シンプルな管理: フルメッシュ接続不要。各VPCは1本の接続のみ
  • • 推移的ルーティング: VPC間の推移的な通信が可能
  • • ルートテーブル分離: 複数のルートテーブルで通信を制御可能

ルートテーブルによるセグメンテーション

Transit Gatewayでは、複数のルートテーブルを使用してトラフィックを分離できます。 これにより、セキュリティやコンプライアンス要件に応じたネットワーク分離が可能です。

ルートテーブル"] VPC_Prod1[VPC 本番1] VPC_Prod2[VPC 本番2] VPC_Dev1[VPC 開発1] VPC_Dev2[VPC 開発2] TGW -.-> RT_Prod TGW -.-> RT_Dev RT_Prod --- VPC_Prod1 RT_Prod --- VPC_Prod2 RT_Dev --- VPC_Dev1 RT_Dev --- VPC_Dev2 Note["本番環境と開発環境を
ルートテーブルで分離"] style RT_Prod fill:#fecaca style RT_Dev fill:#bfdbfe style Note fill:#fef3c7

Transit Gatewayの料金

料金体系

  • • アタッチメント時間課金: VPCアタッチメント1時間あたり約0.05 USD
  • • データ転送課金: Transit Gateway経由のデータ転送1GBあたり約0.02 USD
  • • VPCピアリングよりコストは高いが、管理の簡素化とスケーラビリティで相殺

8.3 VPCエンドポイント（Gateway、Interface）

VPCエンドポイントは、インターネットゲートウェイやNATゲートウェイを経由せずに、AWSサービスにプライベートに接続できる機能です。 セキュリティ向上とコスト削減に貢献します。

VPCエンドポイントの種類

種類 Gateway Endpoint Interface Endpoint
対象サービス S3、DynamoDB 多くのAWSサービス（EC2、SNS、SQS等）
実装方式 ルートテーブルにエントリを追加 ENI（Elastic Network Interface）をサブネットに配置
料金 無料 時間課金 + データ転送課金
アクセス元 VPC内のみ VPC内、オンプレミス、他VPC（PrivateLink経由）

Gateway Endpointの仕組み

graph LR subgraph VPC["VPC"] EC2[EC2 Instance] RT[Route Table] GW_EP["Gateway Endpoint
for S3"] end S3[(S3 Bucket)] EC2 --> RT RT -."|プライベート接続|".- GW_EP GW_EP --> S3 Internet[インターネット経由なし] style GW_EP fill:#d1fae5 style Internet fill:#fecaca

Interface Endpointの仕組み

graph LR subgraph VPC["VPC"] subgraph Subnet["Subnet"] EC2[EC2 Instance] ENI["Interface Endpoint
（ENI）"] end end SNS[Amazon SNS] EC2 -."|Private IP|".- ENI ENI --> SNS style ENI fill:#bfdbfe

VPCエンドポイントのメリット

  • • インターネットゲートウェイ不要 → セキュリティ向上
  • • NATゲートウェイ不要 → コスト削減
  • • AWSバックボーンネットワーク経由 → 低レイテンシー
  • • エンドポイントポリシーでアクセス制御可能

実習: S3向けGateway Endpointの作成

ハンズオン

VPC内のEC2インスタンスからS3へプライベートにアクセスできるGateway Endpointを作成します。

bash 
# VPC IDとルートテーブルIDを取得
VPC_ID=$(aws ec2 describe-vpcs \
  --filters "Name=tag:Name,Values=MyVPC" \
  --query "Vpcs[0].VpcId" --output text)

RT_ID=$(aws ec2 describe-route-tables \
  --filters "Name=vpc-id,Values=$VPC_ID" \
  --query "RouteTables[0].RouteTableId" --output text)

# S3向けGateway Endpointを作成
aws ec2 create-vpc-endpoint \
  --vpc-id $VPC_ID \
  --service-name com.amazonaws.ap-northeast-1.s3 \
  --route-table-ids $RT_ID

8.4 PrivateLinkによるサービス公開

AWS PrivateLinkは、VPC内のサービスを他のVPCやAWSアカウントに安全に公開できる機能です。 VPCピアリングやTransit Gatewayを使わずに、特定のサービスのみを公開できます。

PrivateLinkのアーキテクチャ

graph LR subgraph Provider["サービス提供側VPC"] NLB["Network Load
Balancer"] EP_Service["Endpoint Service"] App[アプリケーション] App --> NLB NLB --> EP_Service end subgraph Consumer["サービス利用側VPC"] Interface_EP["Interface Endpoint"] Client[クライアント] Client --> Interface_EP end Interface_EP -."|PrivateLink|".- EP_Service style EP_Service fill:#fed7aa style Interface_EP fill:#bfdbfe

PrivateLinkのユースケース

  • • SaaS提供者が顧客VPCにサービスを提供
  • • マルチアカウント環境で特定のサービスのみを共有
  • • パートナー企業とのプライベートAPI連携
  • • オンプレミスから特定のAWSサービスへのプライベートアクセス

PrivateLinkの利点

  • • セキュリティ: インターネット経由不要、プライベートIPのみ
  • • CIDRブロック重複OK: VPC間でIPアドレス範囲が重複しても利用可能
  • • 最小権限: 特定のサービスのみを公開、VPC全体の公開不要
  • • スケーラビリティ: 数千のコンシューマーVPCから接続可能

8.5 VPC Flow Logsとネットワーク分析

VPC Flow Logsは、VPC内のネットワークインターフェースを通過するIPトラフィックに関する情報をキャプチャする機能です。 セキュリティ分析、トラブルシューティング、ネットワーク最適化に活用できます。

8.6 NATゲートウェイの高可用性設計

NAT（Network Address Translation）ゲートウェイは、プライベートサブネット内のインスタンスがインターネットにアクセスするために使用されます。 高可用性を確保するには、各アベイラビリティゾーンにNATゲートウェイを配置する必要があります。

単一AZのNATゲートウェイ（非推奨）

AZ Bもインターネット接続不可"] style Warning fill:#fecaca style NAT_A fill:#fef3c7

マルチAZのNATゲートウェイ（推奨）

graph TB subgraph VPC["VPC"] subgraph AZ_A["AZ A"] PubSub_A["Public Subnet A"] PrivSub_A["Private Subnet A"] NAT_A["NAT Gateway A"] end subgraph AZ_B["AZ B"] PubSub_B["Public Subnet B"] PrivSub_B["Private Subnet B"] NAT_B["NAT Gateway B"] end IGW["Internet Gateway"] NAT_A --> PubSub_A NAT_B --> PubSub_B PubSub_A --> IGW PubSub_B --> IGW PrivSub_A --> NAT_A PrivSub_B --> NAT_B end Note["✅ 各AZが独立して
インターネットにアクセス"] style Note fill:#d1fae5 style NAT_A fill:#bfdbfe style NAT_B fill:#bfdbfe

NATゲートウェイ高可用性のベストプラクティス

  • • 各AZに1つのNATゲートウェイを配置
  • • 各プライベートサブネットのルートテーブルは、同じAZ内のNATゲートウェイを参照
  • • クロスAZトラフィック（データ転送料金）を回避
  • • NATゲートウェイは自動的に冗長化されている（単一AZ内での高可用性）

NATゲートウェイのコスト最適化

コスト考慮事項

  • • NATゲートウェイは時間課金（約0.045 USD/時間）+ データ処理課金（約0.045 USD/GB）
  • • VPCエンドポイントの活用でNATゲートウェイ経由のトラフィックを削減
  • • 開発環境では、必要な時間帯のみNATゲートウェイを起動する運用も検討

8.7 ネットワークACLとセキュリティグループの使い分け

AWSでは、セキュリティグループとネットワークACLの2つのファイアウォール機能が提供されています。 それぞれの特性を理解し、多層防御を実現することが重要です。

セキュリティグループ vs ネットワークACL

観点 セキュリティグループ ネットワークACL
適用レベル インスタンス（ENI）レベル サブネットレベル
ルールタイプ 許可ルールのみ 許可・拒否ルール両方
ステートフル/レス ステートフル（戻りトラフィック自動許可） ステートレス（明示的に戻りルール必要）
ルール評価 全ルールを評価（OR条件） ルール番号順に評価（最初にマッチしたルールを適用）
デフォルト動作 全て拒否（明示的に許可が必要） デフォルトACL: 全て許可

多層防御のアーキテクチャ

（サブネットレベル）"] SG["Security Group
（インスタンスレベル）"] EC2[EC2 Instance] end Internet --> IGW IGW --> NACL NACL --> SG SG --> EC2 Note["2層のファイアウォールで
多層防御を実現"] style NACL fill:#fed7aa style SG fill:#bfdbfe style Note fill:#d1fae5

使い分けのベストプラクティス

セキュリティグループの用途

  • • アプリケーションレベルのアクセス制御（通常のファイアウォール）
  • • きめ細かいインスタンス単位の制御
  • • 動的なルール管理（IPアドレス変更が頻繁な環境）

ネットワークACLの用途

  • • サブネット全体に対する追加の防御層
  • • 特定IPアドレスからのアクセス拒否（ブラックリスト）
  • • コンプライアンス要件でサブネットレベルの制御が必要な場合

ネットワークACLのルール例

ルール番号 種類 プロトコル ポート範囲 送信元 許可/拒否
100 インバウンド TCP 80 0.0.0.0/0 許可
110 インバウンド TCP 443 0.0.0.0/0 許可
120 インバウンド TCP 22 192.0.2.100/32 拒否
130 インバウンド TCP 22 10.0.0.0/8 許可
* インバウンド すべて すべて 0.0.0.0/0 拒否

ルール番号のベストプラクティス

ルール番号は100, 110, 120...のように10刻みで設定します。 これにより、後から中間にルールを追加する余地が生まれます。

理解度確認クイズ

  1. Q1. VPCピアリングの推移的ルーティングが不可とはどういう意味ですか？

    回答を表示

    VPC A-B、B-Cがピアリング接続されていても、A-C間の直接通信はできません。 A-C間で通信したい場合は、別途A-Cのピアリング接続を作成する必要があります。

  2. Q2. Transit GatewayとVPCピアリングの主な違いは何ですか？

    回答を表示

    Transit Gatewayは中央ハブとして機能し、推移的ルーティングが可能です。 多数のVPCを接続する場合、VPCピアリングではフルメッシュ接続が必要ですが、Transit Gatewayでは各VPCが1本の接続のみで済みます。

  3. Q3. Gateway EndpointとInterface Endpointの違いを説明してください。

    回答を表示

    Gateway Endpointは、S3とDynamoDBのみに対応し、ルートテーブルにエントリを追加します（無料）。
    Interface Endpointは、多くのAWSサービスに対応し、ENIをサブネットに配置します（有料）。

  4. Q4. なぜNATゲートウェイを各AZに配置する必要がありますか？

    回答を表示

    単一AZにNATゲートウェイを配置すると、そのAZが障害した場合、他のAZのプライベートサブネットもインターネットにアクセスできなくなります（単一障害点）。 各AZにNATゲートウェイを配置することで、AZ障害時でも他のAZは影響を受けません。

  5. Q5. セキュリティグループとネットワークACLの最も大きな違いは何ですか？

    回答を表示

    セキュリティグループはステートフルで、戻りトラフィックを自動的に許可します。 ネットワークACLはステートレスで、明示的に戻りトラフィックのルールを追加する必要があります。 また、セキュリティグループは許可ルールのみですが、ネットワークACLは拒否ルールも設定できます。

第8章のまとめ

  • VPCピアリングはシンプルだが推移的ルーティング不可、フルメッシュ接続が必要
  • Transit Gatewayはハブ&スポーク構成で大規模環境に対応、推移的ルーティング可能
  • VPCエンドポイントでAWSサービスへプライベート接続、セキュリティ向上とコスト削減
  • PrivateLinkで特定のサービスのみを他VPCに安全に公開
  • VPC Flow Logsでネットワークトラフィックを記録、セキュリティ分析とトラブルシューティングに活用
  • NATゲートウェイは各AZに配置して高可用性を確保
  • セキュリティグループ（ステートフル）とネットワークACL（ステートレス）で多層防御
