Auto Scaling - 需要に応じた自動スケーリング

トラフィックの変動に応じてリソースを自動調整するAuto Scalingの設計パターンを学びます。

この章で学ぶこと

  • Auto Scalingグループの構成要素
  • 起動テンプレートと起動設定の違い
  • スケーリングポリシーの種類（ターゲット追跡、ステップ、シンプル）
  • スケジュールベーススケーリング
  • 予測スケーリングの活用
  • ウォームプールとインスタンスのライフサイクル

4.1 Auto Scalingの基礎

Auto Scalingは、トラフィックの変動に応じてEC2インスタンス数を自動的に調整することで、可用性を確保しながらコストを最適化する仕組みです。 需要が増えれば自動的にインスタンスを追加（スケールアウト）し、需要が減れば削除（スケールイン）します。

Auto Scalingの利点

需要への自動対応

トラフィック増加時に自動的にキャパシティを追加し、パフォーマンスを維持します。

高可用性

異常なインスタンスを自動的に検出し、新しいインスタンスに置き換えます。

コスト最適化

必要な時だけインスタンスを起動し、アイドル時のコストを削減します。

マルチAZ対応

複数のアベイラビリティゾーンにインスタンスを分散配置し、障害に備えます。

4.2 Auto Scalingグループの構成要素

Auto Scalingグループ（ASG）は、複数のEC2インスタンスを論理的なグループとして管理します。

主要な構成要素

起動テンプレート / 起動設定

新しいインスタンスを起動する際の設定（AMI、インスタンスタイプ、セキュリティグループ等）を定義します。

  • • 起動テンプレート: バージョン管理可能、推奨される方法
  • • 起動設定: レガシー、変更不可（新規作成が必要）

希望するキャパシティ・最小・最大

インスタンス数の範囲を制御します。

  • • 最小キャパシティ: 常に維持するインスタンス数の下限
  • • 希望するキャパシティ: 通常時のインスタンス数
  • • 最大キャパシティ: スケールアウト時の上限

ヘルスチェック

インスタンスの正常性を監視し、異常なインスタンスを置き換えます。

  • • EC2ヘルスチェック: インスタンスのステータスチェック
  • • ELBヘルスチェック: ELBからの応答確認（推奨）

ターゲットグループ

Auto Scalingグループが起動したインスタンスを自動的にELBのターゲットグループに登録します。

起動テンプレート vs 起動設定

起動テンプレートはバージョン管理が可能で、複数のインスタンスタイプを指定できます。 新規作成時は起動テンプレートを使用することが推奨されます。

4.3 スケーリングポリシーの種類

Auto Scalingは、複数のスケーリングポリシーをサポートしています。用途に応じて適切なポリシーを選択します。

1. ターゲット追跡スケーリング（推奨）

特定のメトリクスを目標値に維持するよう、自動的にスケールアウト/インします。最もシンプルで推奨される方法です。

ターゲット追跡の例

「平均CPU使用率を50%に維持する」ように設定すると、Auto Scalingが自動的にインスタンス数を調整します。

target-tracking-policy.json 
{
  "TargetValue": 50.0,
  "PredefinedMetricSpecification": {
    "PredefinedMetricType": "ASGAverageCPUUtilization"
  }
}

サポートされる定義済みメトリクス

  • • ASGAverageCPUUtilization - 平均CPU使用率
  • • ASGAverageNetworkIn - 平均ネットワーク受信バイト数
  • • ASGAverageNetworkOut - 平均ネットワーク送信バイト数
  • • ALBRequestCountPerTarget - ターゲットあたりのリクエスト数

2. ステップスケーリング

CloudWatchアラームに基づき、メトリクスの変化の大きさに応じて段階的にスケーリングします。 より細かい制御が必要な場合に使用します。

ステップスケーリングの例

CPU使用率に応じて、異なるインスタンス数を追加します。

  • • CPU使用率 50-60%: +1インスタンス
  • • CPU使用率 60-80%: +2インスタンス
  • • CPU使用率 80%以上: +3インスタンス

3. シンプルスケーリング（レガシー）

CloudWatchアラームに基づき、単一のスケーリングアクションを実行します。 現在はターゲット追跡またはステップスケーリングの使用が推奨されます。

ポリシータイプ 使いやすさ 柔軟性 推奨度
ターゲット追跡 高い 標準的 最推奨
ステップスケーリング 中程度 高い 推奨
シンプルスケーリング 低い 低い 非推奨

推奨事項

ほとんどのユースケースでは、ターゲット追跡スケーリングが最適です。 設定が簡単で、自動的にスケールアウトとスケールインの両方を管理します。

4.4 スケジュールベーススケーリングと予測スケーリング

スケジュールベーススケーリング

予測可能なトラフィックパターンに対して、事前にスケジュールを設定してキャパシティを調整します。

スケジュールスケーリングの例

営業時間（9:00-18:00）に自動的にインスタンス数を増やす設定

  • • 平日9:00: 最小2→希望5に変更
  • • 平日18:00: 最小2→希望2に戻す
  • • 休日: 最小1→希望1に縮小

予測スケーリング

機械学習を使用して、過去のトラフィックパターンから将来の需要を予測し、事前にキャパシティを確保します。

予測スケーリングの仕組み

  • • 過去のCloudWatchメトリクスを分析
  • • 将来の負荷を最大2日間予測
  • • 予測に基づいて事前にスケールアウト
  • • ターゲット追跡と併用可能

予測スケーリングの注意点

予測スケーリングは、最低14日間の履歴データが必要です。 また、予測が外れることもあるため、ターゲット追跡スケーリングと併用することが推奨されます。

4.5 ウォームプールとインスタンスライフサイクル

ウォームプール

事前に初期化されたインスタンスをプールしておくことで、スケールアウト時の応答時間を短縮します。

ウォームプールの利点

  • • 起動時間の短縮: AMI起動+UserData実行済みのインスタンスを待機
  • • コスト削減: 停止状態（Stopped）でプール可能（EBSのみ課金）
  • • 予測可能な応答時間: 急激なトラフィック増加に対応

インスタンスライフサイクル

Auto Scalingグループ内のインスタンスは、特定のライフサイクルを経ます。

stateDiagram-v2 [*] --> Pending: スケールアウト Pending --> InService: 起動完了 InService --> Terminating: スケールイン Terminating --> Terminated Terminated --> [*] InService --> Standby: メンテナンス Standby --> InService: メンテナンス完了

Pending（起動中）

インスタンスが起動中。ライフサイクルフックでUserDataの完了を待つことができます。

InService（稼働中）

インスタンスがELBに登録され、トラフィックを受け付けている状態。

Terminating（終了中）

インスタンスが削除される前の状態。ライフサイクルフックでログの保存等が可能。

Standby（スタンバイ）

メンテナンスやトラブルシューティングのため、一時的にトラフィックから除外された状態。

ライフサイクルフックの活用

起動時にアプリケーション初期化を確実に完了させたり、終了時にログをS3に保存したりするために、ライフサイクルフックを使用できます。

4.6 スポットインスタンスとの統合

Auto Scalingグループでスポットインスタンスを活用することで、大幅なコスト削減が可能です。

スポットインスタンスとは

AWSの余剰キャパシティを入札形式で利用するインスタンス。オンデマンド価格の最大90%割引で利用できますが、 AWSがキャパシティを必要とした場合、2分前通知で中断される可能性があります。

スポットインスタンス活用のベストプラクティス

  • • 複数のインスタンスタイプ: 多様なインスタンスタイプを指定し、中断リスクを分散
  • • オンデマンドとの混在: ベースキャパシティはオンデマンド、追加分はスポット
  • • キャパシティ最適化: 中断リスクが低いインスタンスタイプを自動選択
  • • 中断通知の処理: 2分前通知を受け取り、グレースフルシャットダウン

Auto Scalingでのスポット設定例

mixed-instances-policy.json 
{
  "MixedInstancesPolicy": {
    "InstancesDistribution": {
      "OnDemandBaseCapacity": 2,
      "OnDemandPercentageAboveBaseCapacity": 30,
      "SpotAllocationStrategy": "capacity-optimized"
    },
    "LaunchTemplate": {
      "Overrides": [
        {"InstanceType": "t3.medium"},
        {"InstanceType": "t3a.medium"},
        {"InstanceType": "t2.medium"}
      ]
    }
  }
}

OnDemandBaseCapacity: 常にオンデマンドで維持する最小インスタンス数（この例では2）

OnDemandPercentageAboveBaseCapacity: ベースキャパシティ超過分のうちオンデマンドで起動する割合（30%）

SpotAllocationStrategy: capacity-optimized（最も中断されにくいインスタンスタイプを自動選択）

スポットインスタンス中断への対応

スポットインスタンスが中断された場合、Auto Scalingは自動的に新しいインスタンスを起動して希望するキャパシティを維持します。 ステートレスなアプリケーションに最適です。

理解度確認クイズ

  1. Q1. Auto Scalingの主な利点を3つ挙げてください。

    回答を表示

    1. 需要への自動対応（トラフィック増加時に自動スケールアウト）
    2. 高可用性（異常なインスタンスを自動置き換え）
    3. コスト最適化（必要な時だけリソースを使用）

  2. Q2. 最も推奨されるスケーリングポリシーは何ですか？その理由も答えてください。

    回答を表示

    ターゲット追跡スケーリングが最も推奨されます。
    理由: 設定が簡単で、目標メトリクス値を指定するだけで自動的にスケールアウト/インの両方を管理してくれるためです。

  3. Q3. 予測スケーリングを使用するための前提条件は何ですか？

    回答を表示

    最低14日間の履歴データが必要です。
    また、予測が外れる可能性があるため、ターゲット追跡スケーリングと併用することが推奨されます。

  4. Q4. スポットインスタンスをAuto Scalingで活用する際のベストプラクティスを2つ挙げてください。

    回答を表示

    1. 複数のインスタンスタイプを指定して中断リスクを分散する
    2. オンデマンドとスポットを混在させ、ベースキャパシティはオンデマンドで確保する
    3. キャパシティ最適化戦略を使用して中断リスクが低いインスタンスタイプを自動選択する

第4章のまとめ

  • Auto Scalingは需要に応じて自動的にキャパシティを調整し、可用性とコスト効率を両立
  • 起動テンプレートはバージョン管理可能で、起動設定より推奨される
  • ターゲット追跡スケーリングが最もシンプルで推奨される方法
  • スケジュールベーススケーリングは予測可能なパターンに、予測スケーリングは機械学習で将来の需要を予測
  • ウォームプールで起動時間を短縮、ライフサイクルフックでカスタム処理を実行
  • スポットインスタンスとの混在でコストを最大90%削減可能
