11.1 S3のストレージクラスと可用性設計

Amazon S3は、オブジェクトストレージとして業界最高レベルの耐久性（99.999999999%、イレブンナイン）を提供します。 データのアクセスパターンとコスト要件に応じて、適切なストレージクラスを選択することが重要です。

耐久性と可用性の違い

耐久性（Durability）: データが失われない確率。S3は全クラスで11ナインの耐久性を提供。
可用性（Availability）: データにアクセスできる確率。ストレージクラスによって異なり、Standard（99.99%）からOne Zone-IA（99.5%）まで。

1. デフォルトでマルチAZ

S3 Standard、Standard-IA、Glacierは、複数のアベイラビリティゾーンにデータを自動的にレプリケート。 単一AZ障害でもデータにアクセス可能。

2. リージョン内の自動レプリケーション

S3は、同一リージョン内の複数のアベイラビリティゾーンに少なくとも3つのコピーを保存。 ハードウェア障害からデータを保護。

3. 自動的なデータ整合性チェック

S3は定期的にチェックサムを検証し、破損したデータを検出・修復。 データの完全性を継続的に保証。

11.2 S3クロスリージョンレプリケーション（CRR）

S3クロスリージョンレプリケーション（CRR）を使用すると、バケット内のオブジェクトを異なるAWSリージョンに自動的にレプリケートできます。 災害復旧、レイテンシー削減、コンプライアンス要件への対応に有効です。

CRRの主要機能

自動レプリケーション

新規オブジェクトはバージョン管理有効時に自動的にレプリケート。 既存オブジェクトのレプリケーションも可能（Batch Replication）。

フィルタリング

プレフィックス、タグに基づいてレプリケート対象を指定可能。 特定のフォルダやオブジェクトのみをレプリケート。

ストレージクラス変更

レプリケーション先で異なるストレージクラスに変更可能。 例: Standard → Standard-IA でコスト削減。

所有権の変更

宛先バケットの所有者に所有権を変更可能（Replica Ownership Override）。 クロスアカウントレプリケーションで有用。

CRR設定の前提条件

CRRの前提条件

  • ✓ ソースバケットとレプリケーション先バケットの両方でバージョニングが有効
  • ✓ ソースバケットと宛先バケットが異なるリージョンに存在
  • ✓ S3がソースバケットから宛先バケットにオブジェクトをレプリケートする権限を持つIAMロールが必要
  • ✓ ソースバケットの所有者がレプリケーションロールに対して適切な権限を付与

CRRのユースケース

災害復旧（DR）

プライマリリージョンで障害が発生した場合、別リージョンのバケットにフェイルオーバー。 RPOをほぼゼロに近づけることが可能。

レイテンシー削減

グローバルユーザーに対して、地理的に近いリージョンからコンテンツを提供。 ダウンロード速度の向上。

コンプライアンス

法規制により、データを特定の地理的場所に保管する必要がある場合、 該当リージョンにデータをレプリケート。

レプリケーションの注意点

レプリケーションは非同期で行われるため、わずかな遅延が発生します。 また、削除マーカーや特定バージョンの削除はデフォルトではレプリケートされません。 必要に応じて、削除マーカーのレプリケーションを有効化してください。

11.3 S3バージョニングとオブジェクトロック

S3バージョニングとオブジェクトロックは、データの保護と不変性を保証するための重要な機能です。 誤削除や悪意のある変更からデータを保護できます。

バージョニングの仕組み

バケットでバージョニングを有効にすると、オブジェクトの全バージョンが保存されます。

  • • 同じキー名でアップロードすると新しいバージョンが作成される
  • • 削除すると削除マーカーが追加される（実際には削除されない）
  • • 削除マーカーを削除すると元のオブジェクトが復元される
  • • 特定バージョンを指定して取得・削除が可能

バージョニングの利点

  • • 誤削除からの保護: 削除マーカーを削除すれば復元可能
  • • 誤上書きからの保護: 過去のバージョンを取得可能
  • • レプリケーションの前提条件: CRR/SRRにはバージョニングが必須

S3オブジェクトロック

S3オブジェクトロックは、WORM（Write Once Read Many）モデルを実装し、 指定期間または無期限にオブジェクトの削除や上書きを防ぎます。 金融、医療、法律分野のコンプライアンス要件に対応。

ガバナンスモード

特別な権限を持つユーザーはロック設定を変更または削除可能。

用途: 開発環境、テスト環境でのデータ保護

コンプライアンスモード

ルートユーザーを含むすべてのユーザーがロック期間中は削除・変更不可。

用途: 金融記録、医療記録の保管（SEC 17a-4等）

リーガルホールド

保持期間とは独立して、無期限にオブジェクトを保護できる機能。 訴訟や調査が進行中の場合に使用。 s3:PutObjectLegalHold 権限を持つユーザーのみが解除可能。

オブジェクトロックの設定例

バケット作成時にオブジェクトロックを有効化 
# 新規バケット作成時にオブジェクトロック有効化
aws s3api create-bucket \
  --bucket my-compliance-bucket \
  --region ap-northeast-1 \
  --create-bucket-configuration LocationConstraint=ap-northeast-1 \
  --object-lock-enabled-for-bucket

# デフォルト保持期間を設定（30日間のコンプライアンスモード）
aws s3api put-object-lock-configuration \
  --bucket my-compliance-bucket \
  --object-lock-configuration '{
    "ObjectLockEnabled": "Enabled",
    "Rule": {
      "DefaultRetention": {
        "Mode": "COMPLIANCE",
        "Days": 30
      }
    }
  }'

バージョニングとオブジェクトロックの組み合わせ

オブジェクトロックを使用するには、バケットでバージョニングが有効である必要があります。 両機能を組み合わせることで、最高レベルのデータ保護を実現できます。

11.4 EBSスナップショットとマルチアタッチ

Amazon EBSは、EC2インスタンス用のブロックストレージを提供します。 スナップショット機能とマルチアタッチ機能により、データの保護と高可用性を実現します。

増分バックアップ

初回スナップショットはフルバックアップ、以降は変更分のみを保存。 ストレージコストと作成時間を削減。

リージョンレベルの可用性

スナップショットはS3に保存され、リージョン内の複数AZに自動的にレプリケート。 任意のAZに復元可能。

クロスリージョンコピー

スナップショットを別リージョンにコピーして、リージョン障害からのDRを実現。 暗号化や暗号化キーの変更も可能。

EBSスナップショットのベストプラクティス

スナップショット作成のベストプラクティス

  • • 定期的なスナップショット作成を自動化（Data Lifecycle Manager使用）
  • • ボリュームのアタッチ中でもスナップショット作成可能（オンラインバックアップ）
  • • 重要なスナップショットにはタグを付けて管理
  • • 不要なスナップショットは削除してコスト削減

EBSマルチアタッチ

EBSマルチアタッチを使用すると、単一のProvisioned IOPS SSD（io1/io2）ボリュームを 同一アベイラビリティゾーン内の最大16個のEC2インスタンスに同時にアタッチできます。

ユースケース

  • • クラスタファイルシステム（Oracle RAC等）
  • • 高可用性アプリケーション
  • • 共有ストレージが必要なアプリケーション

制限事項

  • • io1/io2ボリュームのみサポート
  • • 同一AZ内のインスタンスのみ
  • • クラスタ対応ファイルシステムが必要
  • • 最大16インスタンスまで

ファイルシステムの注意点

マルチアタッチを使用する場合、通常のファイルシステム（ext4、XFS等）では データ破損のリスクがあります。 必ずクラスタ対応ファイルシステム（GFS2、OCFS2等）またはアプリケーションレベルの同期制御を使用してください。

11.5 EFSのパフォーマンスモードとバックアップ

Amazon EFS（Elastic File System）は、フルマネージド型のNFSファイルシステムを提供します。 マルチAZ構成によりデフォルトで高可用性を実現し、自動バックアップ機能でデータを保護します。

マルチAZアーキテクチャ

EFSは複数のAZにデータを自動的にレプリケート。 各AZにマウントターゲットを作成し、AZ障害時でも他のAZからアクセス可能。

自動スケーリング

ファイルの追加・削除に応じて自動的に容量が増減。 事前のキャパシティプロビジョニング不要。

EFSパフォーマンスモード

モード 特徴 用途
General Purpose 低レイテンシー（ミリ秒単位）を提供 Webサーバー、CMS、開発環境
Max I/O 高スループット、並列処理に最適化 ビッグデータ分析、メディア処理

EFSスループットモード

Bursting Throughput

ファイルシステムのサイズに応じてスループットがスケール。 小規模ファイルシステムでも一時的に高スループットを実現（バースト）。

適用: 小〜中規模のワークロード

Provisioned Throughput

必要なスループットを事前に指定。 ファイルシステムのサイズに関係なく、一定のスループットを保証。

適用: 高スループットが必要なワークロード

EFSバックアップ

AWS Backupとの統合

AWS Backupを使用して、EFSファイルシステムの自動バックアップを設定。 バックアップスケジュール、保持期間、ライフサイクルポリシーを一元管理。

増分バックアップ

初回はフルバックアップ、以降は変更分のみをバックアップ。 バックアップ時間とストレージコストを削減。

クロスリージョンバックアップ

AWS Backupでバックアップを別リージョンにコピー。 リージョン障害からのDRを実現。

EFSとEBSの使い分け

EFS: 複数のEC2インスタンスから同時アクセスが必要な場合、マルチAZ構成のファイル共有。
EBS: 単一のEC2インスタンス専用のブロックストレージ、低レイテンシーが必要な場合。

11.6 AWS Backupによる統合バックアップ管理

AWS Backupは、複数のAWSサービスのバックアップを一元管理できるフルマネージド型サービスです。 バックアップポリシーの一元管理、コンプライアンス対応、クロスリージョンバックアップを簡単に実現します。

AWS Backupの主要機能

バックアッププラン

バックアップのスケジュール、保持期間、ライフサイクルルールを定義。 リソースタグに基づいて自動的にバックアップ対象を選択。

バックアップボールト

バックアップを保存するコンテナ。 暗号化、アクセス制御、ボールトロックポリシーでデータを保護。

クロスリージョンバックアップ

バックアップを別リージョンに自動的にコピー。 リージョン障害からの復旧を実現。

コンプライアンスレポート

バックアップの成功/失敗を監視。 監査証跡として、バックアップのコンプライアンス状況をレポート。

バックアッププランの構成要素

1. バックアップルール

バックアップのスケジュール、開始時刻、頻度を定義。

  • • スケジュール: 日次、週次、月次（cron式で指定可能）
  • • バックアップウィンドウ: バックアップ開始の時間帯
  • • 保持期間: バックアップの保存期間（日数、週数、月数、年数）

2. ライフサイクルポリシー

バックアップを低コストストレージに移行。

  • • コールドストレージへの移行: 一定期間後にコストの低いストレージに移行
  • • 自動削除: 保持期間終了後に自動削除

3. リソース割り当て

バックアップ対象リソースの指定方法。

  • • リソースID: 特定のリソースを直接指定
  • • タグベース: 特定のタグを持つすべてのリソースを自動選択

バックアップボールトロック

バックアップボールトロックは、WORM（Write Once Read Many）ポリシーを実装し、 バックアップの削除や変更を防ぎます。ランサムウェア攻撃や悪意のある削除からバックアップを保護します。

ボールトロックの仕組み

  • • 最小保持期間を設定（例: 30日間）
  • • 保持期間中はルートユーザーでもバックアップを削除不可
  • • 最大保持期間を設定して、保持期間の延長を制限
  • • コンプライアンス要件（SEC 17a-4、HIPAA等）に対応

AWS Backupのメリット

AWS Backupを使用することで、複数のAWSサービスのバックアップを一元管理でき、 バックアップポリシーの統一、監査証跡の確保、運用負荷の軽減を実現できます。 特に大規模環境では、タグベースのリソース割り当てにより、バックアップの自動化と標準化が容易になります。

11.7 データライフサイクル管理

データライフサイクル管理は、データのアクセス頻度や保存期間に基づいて、 ストレージコストを最適化する重要な戦略です。S3ライフサイクルポリシーとEBSライフサイクル管理を活用します。

移行アクション

オブジェクトを別のストレージクラスに自動的に移行。

  • • Standard → Standard-IA（30日後）
  • • Standard-IA → Glacier（90日後）
  • • Glacier → Glacier Deep Archive（180日後）

削除アクション

一定期間経過後にオブジェクトを自動削除。

  • • ログファイルを365日後に削除
  • • 古いバージョンを90日後に削除（バージョニング有効時）
  • • 不完全なマルチパートアップロードを7日後に削除

ライフサイクルポリシーの設定例

        "Prefix": "logs/"

        },
        {

          "StorageClass": "GLACIER"
        }

        "Days": 365
      }
    }
  ]
}

EBSデータライフサイクル管理

Amazon Data Lifecycle Manager（DLM）を使用して、EBSスナップショットとEBS-backed AMIのライフサイクルを自動化します。

自動スナップショット作成

タグに基づいてEBSボリュームを自動的にスナップショット。 毎日、毎週、毎月のスケジュールを設定可能。

保持期間管理

スナップショットの保持期間を設定（例: 7日、30日、90日）。 期間経過後に自動削除してストレージコストを削減。

クロスリージョンコピー

スナップショットを別リージョンに自動的にコピー。 DRバックアップとして保存し、異なる保持期間を設定可能。

ライフサイクル管理のベストプラクティス

  • • アクセスパターンが不明な場合は、S3 Intelligent-Tieringを使用
  • • ライフサイクルポリシーは、プレフィックスやタグで細かく制御
  • • 不完全なマルチパートアップロードを定期的に削除
  • • 古いバージョンの自動削除でストレージコストを削減

理解度確認クイズ

  1. Q1. S3の耐久性と可用性の違いを説明してください。

    回答を表示

    耐久性（Durability）: データが失われない確率。S3は全ストレージクラスで99.999999999%（イレブンナイン）の耐久性を提供。
    可用性（Availability）: データにアクセスできる確率。ストレージクラスによって異なり、S3 Standard（99.99%）からS3 One Zone-IA（99.5%）まで。

  2. Q2. S3クロスリージョンレプリケーション（CRR）の主なユースケースを3つ挙げてください。

    回答を表示

    1. 災害復旧（DR）: プライマリリージョン障害時に別リージョンにフェイルオーバー
    2. レイテンシー削減: グローバルユーザーに地理的に近いリージョンからコンテンツを提供
    3. コンプライアンス: 法規制により特定地域にデータを保管する必要がある場合

  3. Q3. EBSマルチアタッチの制限事項を説明してください。

    回答を表示

    • io1/io2ボリュームのみサポート
    • 同一アベイラビリティゾーン内のインスタンスのみ
    • 最大16インスタンスまでアタッチ可能
    • クラスタ対応ファイルシステム（GFS2、OCFS2等）が必要

  4. Q4. AWS Backupのバックアップボールトロックの目的は何ですか？

    回答を表示

    バックアップボールトロックは、WORM（Write Once Read Many）ポリシーを実装し、 最小保持期間中にルートユーザーを含むすべてのユーザーがバックアップを削除または変更できないようにします。 ランサムウェア攻撃や悪意のある削除からバックアップを保護し、コンプライアンス要件（SEC 17a-4、HIPAA等）に対応します。

第11章のまとめ

  • S3は全ストレージクラスで11ナインの耐久性を提供、可用性はクラスによって異なる
  • S3 CRRで災害復旧、レイテンシー削減、コンプライアンス要件に対応
  • S3バージョニングとオブジェクトロックで誤削除・悪意のある変更からデータを保護
  • EBSスナップショットは増分バックアップ、クロスリージョンコピーでDRを実現
  • EFSはマルチAZ構成でデフォルトで高可用性、AWS Backupで自動バックアップ
  • AWS Backupで複数サービスのバックアップを一元管理、ボールトロックでWORM実装
  • ライフサイクルポリシーでストレージコストを最適化
