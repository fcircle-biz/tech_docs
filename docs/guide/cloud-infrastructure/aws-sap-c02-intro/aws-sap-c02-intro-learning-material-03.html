3.1 Elastic Load Balancingとは

Elastic Load Balancing（ELB）は、AWSが提供するマネージドロードバランサーサービスです。 受信トラフィックを複数のターゲット（EC2インスタンス、コンテナ、IPアドレス、Lambda関数など）に自動的に分散し、 アプリケーションの可用性とスケーラビリティを向上させます。

ロードバランサーの主な利点

  • 高可用性: 複数のAZにまたがってトラフィックを分散し、単一障害点を排除
  • 自動スケーリング: トラフィック量に応じて自動的にキャパシティを調整
  • ヘルスチェック: 異常なターゲットを自動検出し、トラフィックを健全なターゲットにのみ送信
  • セキュリティ: SSL/TLS終端、WAF統合、セキュリティグループ統合
  • マネージドサービス: パッチ適用、スケーリング、障害対応をAWSが自動実施

ELBの重要な用語

用語 説明
リスナー 指定したプロトコルとポートでクライアント接続リクエストをチェックするプロセス
ターゲットグループ リクエストのルーティング先となるターゲット（EC2、IP、Lambda等）のグループ

ルール リスナーで受信したリクエストをどのターゲットグループにルーティングするかを定義

3.2 ロードバランサーの種類と選択基準

AWSは用途に応じて4種類のロードバランサーを提供しています。それぞれの特性を理解し、要件に最も適したものを選択することが重要です。

Application Load Balancer (ALB)

OSI層: レイヤー7（アプリケーション層）

主な用途: HTTP/HTTPS Webアプリケーション

  • • パスベース・ホストベースルーティング
  • • WebSocketとHTTP/2対応
  • • Lambda関数のターゲット対応
  • • コンテンツベースルーティング

Network Load Balancer (NLB)

OSI層: レイヤー4（トランスポート層）

主な用途: 高性能TCP/UDP通信

  • • 超低レイテンシー（100マイクロ秒以下）
  • • 毎秒数百万リクエスト対応
  • • 静的IPアドレス対応
  • • クライアントIPアドレス保持

Gateway Load Balancer (GLB)

OSI層: レイヤー3（ネットワーク層）

主な用途: サードパーティ仮想アプライアンス統合

  • • ファイアウォール、IDS/IPS統合
  • • GENEVEプロトコル対応
  • • トラフィック透過的転送
  • • スケーラブルなセキュリティ検査

Classic Load Balancer (CLB)

OSI層: レイヤー4/7

主な用途: 旧世代（新規推奨されず）

  • • EC2-Classic対応
  • • 基本的な負荷分散機能
  • • 現在は非推奨（既存システムのみ）
  • • 新規はALB/NLBを推奨

選択のベストプラクティス

Webアプリケーションには原則ALBを選択してください。 パスベースルーティング、ホストベースルーティング、Lambda統合などの高度な機能が利用できます。 TCP/UDPアプリケーション、ゲームサーバー、IoTには原則NLBを選択してください。 超低レイテンシーと高スループットが実現できます。

3.3 ターゲットグループとヘルスチェック設計

ターゲットグループは、ロードバランサーがトラフィックをルーティングする宛先の論理的なグループです。 ヘルスチェックにより、正常なターゲットにのみトラフィックが送信されます。

ヘルスチェックの動作

ロードバランサーは、ターゲットグループに登録された各ターゲットに対して定期的にヘルスチェックを実行します。

ヘルスチェック設定の主要パラメータ

  • ヘルスチェックパス: 例: /health、/api/health
  • 間隔: ヘルスチェックの実行間隔（5-300秒、デフォルト30秒）
  • タイムアウト: レスポンス待機時間（2-120秒、デフォルト5秒）
  • 正常閾値: 正常と判断するまでの連続成功回数（2-10回、デフォルト5回）
  • 異常閾値: 異常と判断するまでの連続失敗回数（2-10回、デフォルト2回）
  • 成功コード: 正常と判断するHTTPステータスコード（例: 200、200-299）

ヘルスチェック設計の注意点

  • • ヘルスチェックパスは軽量な処理にすること（DB接続チェック等は避ける）
  • • 間隔を短くしすぎない（不要な負荷を避ける）
  • • 異常閾値を低く設定して、素早く異常を検出する
  • • 正常閾値は慎重に設定して、一時的な障害での復帰を避ける

3.4 スティッキーセッションとセッション管理

スティッキーセッション（セッションアフィニティ）は、同一ユーザーからのリクエストを同じターゲットにルーティングする機能です。 セッション状態をサーバー側で管理するアプリケーションで必要となります。

スティッキーセッションの仕組み

sequenceDiagram participant Client as クライアント participant ALB as ALB participant EC2_1 as EC2-1 participant EC2_2 as EC2-2 Client->>ALB: 1回目のリクエスト ALB->>EC2_1: ルーティング EC2_1->>ALB: レスポンス ALB->>Client: レスポンス + Cookieセット Client->>ALB: 2回目のリクエスト（Cookie付き） ALB->>EC2_1: 同じターゲットにルーティング EC2_1->>ALB: レスポンス ALB->>Client: レスポンス

スティッキーセッションの種類（ALB）

種類 説明 用途
Duration-based ALBが生成したCookie（AWSALB）で有効期限を管理 一般的なセッション管理
Application-based アプリケーションが生成したCookieを使用 既存Cookieとの統合

セッション管理のベストプラクティス

可能な限り、ステートレスなアーキテクチャを目指しましょう。 セッション情報はElastiCache（Redis/Memcached）やDynamoDBなどの共有ストレージに保存することで、 スティッキーセッションへの依存を減らし、スケーラビリティと可用性を向上できます。

3.5 SSL/TLS終端とセキュリティ設定

ALBとNLBは、SSL/TLS通信の暗号化・復号化（SSL/TLS終端）をロードバランサーで実行できます。 これにより、バックエンドサーバーのCPU負荷を軽減し、証明書管理を一元化できます。

証明書の管理

AWS Certificate Manager (ACM) 統合

ACMでSSL/TLS証明書を発行・管理し、ELBに直接アタッチできます。

  • • 無料: パブリックSSL/TLS証明書が無料で発行可能
  • • 自動更新: 証明書の有効期限が近づくと自動的に更新
  • • ワイルドカード: *.example.com のようなワイルドカード証明書対応
  • • SNI対応: 1つのロードバランサーで複数の証明書を使用可能

セキュリティポリシー

セキュリティポリシーは、クライアントとロードバランサー間で使用されるSSL/TLSプロトコルと暗号スイートを定義します。

推奨セキュリティポリシー

  • • ELBSecurityPolicy-TLS13-1-2-2021-06: TLS 1.3対応（最新推奨）
  • • ELBSecurityPolicy-2016-08: 幅広い互換性が必要な場合
  • • 古いポリシーは使用しない: TLS 1.0/1.1は非推奨

3.6 クロスゾーン負荷分散の仕組み

クロスゾーン負荷分散は、ロードバランサーが有効化されているすべてのアベイラビリティゾーンにわたって、 登録されたすべてのターゲットにトラフィックを均等に分散する機能です。

ロードバランサー別のクロスゾーン負荷分散設定

  • ALB: デフォルトで有効（無効化不可）
  • NLB: デフォルトで無効（有効化推奨、AZ間データ転送料金に注意）
  • CLB: デフォルトで無効（有効化可能）

ベストプラクティス

クロスゾーン負荷分散は原則有効化してください。 これにより、各AZのターゲット数に偏りがあっても、トラフィックが均等に分散され、 リソースの効率的な活用と一貫したパフォーマンスが実現できます。

3.7 Connection Drainingとデプロイ戦略

Connection Draining（接続ドレイン）は、ターゲットの登録解除や異常状態になった際に、 既存の接続を正常に終了させるための機能です。現在はDeregistration Delayという名称で提供されています。

Deregistration Delayの動作

sequenceDiagram participant ALB as ALB participant EC2 as EC2インスタンス participant Client as クライアント Note over EC2: 登録解除開始 ALB->>EC2: 新規リクエスト停止 Client->>ALB: 既存接続のリクエスト ALB->>EC2: 既存接続は継続 EC2->>ALB: レスポンス ALB->>Client: レスポンス Note over EC2: Deregistration Delay期間
（デフォルト300秒） Note over EC2: すべての接続終了または
タイムアウト後に完全登録解除

Deregistration Delayの設定

設定パラメータ

  • Deregistration Delay: 0-3600秒（デフォルト300秒）
  • 推奨値: アプリケーションの平均リクエスト処理時間 + バッファ
  • 短い値（30-60秒）: 高速デプロイが必要な場合
  • 長い値（300秒以上）: 長時間実行される処理がある場合

デプロイ戦略とELBの活用

ELBを活用することで、ダウンタイムなしのデプロイが可能になります。

1. ローリングデプロイメント

ターゲットを順次更新していく方法。ELBが新旧バージョンを共存させながら段階的に切り替え。

  • • 古いインスタンスを1つずつ登録解除
  • • 新しいバージョンをデプロイして登録
  • • ヘルスチェック通過後、次のインスタンスへ

2. Blue/Greenデプロイメント

新環境（Green）を完全に準備してから、一括で切り替える方法。

  • • 新環境を別ターゲットグループに構築
  • • リスナールールを変更してトラフィック切り替え
  • • 問題があれば即座にロールバック可能

3. カナリアデプロイメント

トラフィックの一部を新バージョンに流して、段階的に切り替える方法。

  • • ALBのターゲットグループの重み付けを活用
  • • 例: 旧90% / 新10% → 旧50% / 新50% → 新100%
  • • リスクを最小化しながら段階的にリリース

理解度確認クイズ

  1. Q1. HTTP/HTTPS Webアプリケーションに最適なロードバランサーはどれですか？

    回答を表示

    Application Load Balancer（ALB）です。ALBはレイヤー7で動作し、パスベースルーティング、ホストベースルーティング、 Lambda統合など、Webアプリケーションに必要な高度な機能を提供します。

  2. Q2. クロスゾーン負荷分散が有効な場合と無効な場合で、トラフィック分散はどう異なりますか？

    回答を表示

    有効な場合は、すべてのAZのターゲットに均等に分散されます。無効な場合は、各AZに50%ずつ振り分けられ、 その後各AZ内のターゲットに分散されるため、AZごとのターゲット数が異なると負荷が偏ります。

  3. Q3. スティッキーセッションに依存しないアーキテクチャにする利点は何ですか？

    回答を表示

    セッション情報を共有ストレージ（ElastiCache、DynamoDB等）に保存することで、 どのターゲットでもリクエストを処理できるステートレス設計が実現できます。 これにより、スケーラビリティの向上、負荷分散の均等化、障害時の影響範囲縮小が可能になります。

  4. Q4. Deregistration Delayの設定値を決める際の考慮点は何ですか？

    回答を表示

    アプリケーションの平均リクエスト処理時間を基準に設定します。 処理時間が短い場合は短めに（30-60秒）、長時間実行される処理がある場合は長めに（300秒以上）設定します。 高速デプロイとリクエストの完全処理のバランスを考慮することが重要です。

第3章のまとめ

  • ELBは用途に応じて4種類のロードバランサーを提供（ALB、NLB、GLB、CLB）
  • WebアプリケーションにはALB、高性能TCP/UDP通信にはNLBを選択
  • ターゲットグループとヘルスチェックにより、正常なターゲットにのみトラフィックを送信
  • ステートレスアーキテクチャを目指し、スティッキーセッションへの依存を減らす
  • SSL/TLS終端とACM統合により、証明書管理を一元化
  • クロスゾーン負荷分散を有効化し、AZ間でトラフィックを均等に分散
  • Deregistration DelayとBlue/Greenデプロイでダウンタイムゼロのリリースを実現
