13.1 AWS Organizationsとマルチアカウント戦略

AWS Organizationsは、複数のAWSアカウントを一元管理するサービスです。 エンタープライズ環境では、セキュリティ境界の分離、請求の統合、ポリシーの一元管理のために、マルチアカウント戦略が推奨されます。

AWS Organizationsの構成要素

主要な概念

  • Organization: 複数のAWSアカウントを統合管理する単位
  • Management Account: Organizationを管理する親アカウント（旧Master Account）
  • Member Account: Organizationに参加する子アカウント
  • OU (Organizational Unit): アカウントをグループ化する単位
  • SCP (Service Control Policy): アカウント権限を制限するポリシー

サービスコントロールポリシー（SCP）

SCPは、OUまたは個別アカウントに適用できるIAMポリシーで、そのアカウント内のすべてのプリンシパル（IAMユーザー、ロール含む）の最大権限を制限します。

例: 特定リージョン以外での操作を拒否するSCP 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyAllOutsideApprovalRegions",
      "Effect": "Deny",
      "NotAction": [
        "cloudfront:*",
        "iam:*",
        "route53:*",
        "support:*"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "ap-northeast-1",
            "us-east-1"
          ]
        }
      }
    }
  ]
}

SCPのベストプラクティス

  • • グローバルサービス（IAM、CloudFront、Route 53など）はリージョン制限から除外
  • • 開発環境では高コストサービスの起動を制限
  • • セキュリティ関連サービス（CloudTrail、Configなど）の無効化を拒否
  • • 本番環境ではルートユーザーの使用を拒否

マルチアカウント戦略の設計パターン

アカウント種別 目的 例
Management Account Organization管理、請求統合 組織全体の管理用
Log Archive Account ログ集約、長期保管 CloudTrail、Config、VPC Flow Logs
Security Account セキュリティ監査、脅威検出 Security Hub、GuardDuty集約
Shared Services Account 共通サービスのホスト Active Directory、DNS、プロキシ
Network Account ネットワークハブ管理 Transit Gateway、Direct Connect
Workload Accounts アプリケーション実行環境 本番、ステージング、開発

13.2 AWS Control Towerによるランディングゾーン

AWS Control Towerは、ベストプラクティスに基づいたマルチアカウント環境（ランディングゾーン）を自動でセットアップするサービスです。 Organizations、Config、CloudTrail、Security Hubなどを統合し、ガバナンスとコンプライアンスを自動化します。

ガードレールの種類

予防的ガードレール (Preventive)

SCPを使用して、特定の操作を事前に拒否

  • • ルートユーザーでのアクセスを拒否
  • • CloudTrailの無効化を拒否
  • • 承認されていないリージョンでの操作を拒否
  • • MFAなしのS3バケット削除を拒否

検出的ガードレール (Detective)

Config Rulesを使用して、コンプライアンス違反を検出

  • • パブリックアクセス可能なS3バケットを検出
  • • 暗号化されていないEBSボリュームを検出
  • • 未承認のIAMポリシーを検出
  • • セキュリティグループの過度な許可を検出

ガードレールの適用レベル

  • • 必須 (Mandatory): すべてのアカウントに自動適用、無効化不可
  • • 強く推奨 (Strongly recommended): ベストプラクティス、手動で有効化可能
  • • 選択的 (Elective): 特定の要件に応じて選択的に有効化

Account Factory

Account Factoryは、新規AWSアカウントの作成を標準化・自動化する機能です。 事前定義されたベースライン（VPC構成、セキュリティ設定、ログ設定など）を適用した新規アカウントを数分で作成できます。

13.3 AWS Configとコンプライアンス監視

AWS Configは、AWSリソースの構成変更を記録・評価するサービスです。 Config Rulesを使用して、リソースがセキュリティ要件やコンプライアンス基準に準拠しているかを継続的に監視できます。

主要なマネージドConfig Rules

s3-bucket-public-read-prohibited

S3バケットがパブリック読み取りアクセスを許可していないか確認

encrypted-volumes

アタッチされたEBSボリュームが暗号化されているか確認

iam-password-policy

IAMパスワードポリシーが要件を満たしているか確認

cloudtrail-enabled

CloudTrailが有効化されているか確認

ec2-instance-managed-by-ssm

EC2インスタンスがSystems Managerで管理されているか確認

自動修復（Remediation）

Config Rulesの違反を検出した際、自動的に修正アクションを実行できます。 Systems Manager Automation DocumentsまたはカスタムLambda関数を使用します。

例: パブリックS3バケットを自動的に非公開にする修復アクション 
{
  "RemediationConfiguration": {
    "ConfigRuleName": "s3-bucket-public-read-prohibited",
    "TargetType": "SSM_DOCUMENT",
    "TargetIdentifier": "AWS-PublishSNSNotification",
    "TargetVersion": "1",
    "Parameters": {
      "AutomationAssumeRole": {
        "StaticValue": {
          "Values": ["arn:aws:iam::123456789012:role/ConfigRemediationRole"]
        }
      },
      "BucketName": {
        "ResourceValue": {
          "Value": "RESOURCE_ID"
        }
      }
    },
    "Automatic": true,
    "MaximumAutomaticAttempts": 5,
    "RetryAttemptSeconds": 60
  }
}

13.5 脅威検出・脆弱性管理サービス

Amazon GuardDuty - 脅威検出

GuardDutyは、機械学習とAWS環境の異常検出により、悪意のある活動や不正な動作を継続的に監視するマネージドサービスです。

GuardDutyのデータソース

VPC Flow Logs

ネットワークトラフィック分析

CloudTrail Logs

API呼び出し分析

DNS Logs

DNSクエリ分析

GuardDutyの検出例

  • 既知の悪意のあるIPアドレスへの通信
  • 暗号通貨マイニング活動
  • 異常なAPIコール（地理的位置、頻度、時間帯）
  • IAM認証情報の不正使用
  • EC2インスタンスの異常な動作

Amazon Inspector - 脆弱性スキャン

Inspectorは、EC2インスタンス、コンテナイメージ、Lambda関数の脆弱性を自動的にスキャンするサービスです。

EC2スキャン

  • • OSパッケージの脆弱性（CVE）
  • • ネットワーク到達可能性の問題
  • • Systems Managerエージェント経由

コンテナイメージスキャン

  • • ECRにプッシュされたイメージをスキャン
  • • OSとプログラミング言語パッケージの脆弱性
  • • 継続的な再スキャン

Amazon Macie - 機密データ検出

Macieは、機械学習とパターンマッチングを使用して、S3内の機密データ（PII、認証情報など）を自動的に検出・保護するサービスです。

Macieが検出できる機密データ

個人情報（PII）

氏名、住所、電話番号、メールアドレス

金融情報

クレジットカード番号、銀行口座番号

認証情報

AWSアクセスキー、秘密鍵

カスタム識別子

正規表現で定義した独自パターン

13.6 AWS KMSと暗号化管理

AWS Key Management Service（KMS）は、暗号化キーの作成と管理を行うマネージドサービスです。 FIPS 140-2検証済みのハードウェアセキュリティモジュール（HSM）で保護されています。

KMSキーの種類

キータイプ 管理者 用途 料金
AWS管理キー AWS AWSサービスとの統合（S3、RDS等） 無料
カスタマー管理キー ユーザー 細かいアクセス制御、ローテーション管理 月額 $1/キー + API呼び出し
AWS所有キー AWS AWSサービス内部で使用（ユーザーからは見えない） 無料

エンベロープ暗号化

大容量データの暗号化では、エンベロープ暗号化を使用します。データはデータキーで暗号化し、そのデータキー自体をKMSのカスタマーマスターキー（CMK）で暗号化します。

sequenceDiagram participant App as アプリケーション participant KMS as AWS KMS participant S3 as Amazon S3 App->>KMS: GenerateDataKey(CMK_ID) KMS-->>App: プレーンテキストデータキー + 暗号化データキー App->>App: データをプレーンテキストデータキーで暗号化 App->>App: プレーンテキストデータキーをメモリから削除 App->>S3: 暗号化データ + 暗号化データキーを保存 Note over App,S3: 復号化時 App->>S3: 暗号化データ + 暗号化データキーを取得 App->>KMS: Decrypt(暗号化データキー) KMS-->>App: プレーンテキストデータキー App->>App: データをプレーンテキストデータキーで復号化

AWS Secrets Manager

Secrets Managerは、データベース認証情報、APIキー、OAuth トークンなどのシークレットを安全に保管・管理するサービスです。 自動ローテーションとKMSによる暗号化をサポートします。

例: Pythonでシークレットを取得 
import boto3
import json

# Secrets Managerクライアント作成
client = boto3.client('secretsmanager')

# シークレット取得
response = client.get_secret_value(
    SecretId='prod/db/password'
)

# シークレットをパース
secret = json.loads(response['SecretString'])
username = secret['username']
password = secret['password']

Secrets Manager vs Systems Manager Parameter Store

  • • Secrets Manager: シークレット専用、自動ローテーション機能、RDS/Redshift統合
  • • Parameter Store: 汎用パラメータストア、無料枠あり、手動ローテーション

13.7 AWS CloudTrailとガバナンス

CloudTrailは、AWSアカウント内のすべてのAPI呼び出しを記録するガバナンス、コンプライアンス、運用監査サービスです。

CloudTrailの証跡タイプ

管理イベント

AWSリソースに対する管理操作

  • • EC2インスタンスの起動
  • • IAMユーザーの作成
  • • S3バケットの作成

データイベント

リソース内のデータ操作

  • • S3オブジェクトのGET/PUT
  • • Lambda関数の実行
  • • DynamoDBテーブル操作

Insightsイベント

異常なAPI活動の検出

  • • 通常と異なるAPI呼び出し量
  • • 異常なエラー率
  • • 機械学習による検出

Organization証跡（組織全体のログ集約）

AWS Organizationsと統合することで、組織内のすべてのアカウントのCloudTrailログを一元管理できます。

ログファイルの整合性検証

CloudTrailは、ログファイルが改ざんされていないことを検証するためのデジタル署名を提供します。 コンプライアンス要件がある場合に有効です。

CloudTrailのベストプラクティス

  • • 組織全体で単一の証跡を有効化（Organization証跡）
  • • ログファイルの整合性検証を有効化
  • • ログをS3に保存し、ライフサイクルポリシーで長期アーカイブ
  • • CloudWatch Logsと統合し、リアルタイムアラートを設定
  • • S3バケットへのアクセスを厳格に制限（MFA Delete有効化）

実習: AWS Security Hubの有効化とセキュリティアラート確認

Security Hubを有効化し、統合されたセキュリティアラートを確認します。

手順1: Security Hubの有効化

  1. AWSマネジメントコンソールで「Security Hub」を検索
  2. 「Security Hubに移動」をクリック
  3. 「Security Hubを有効化」をクリック
  4. セキュリティ標準を選択（AWS Foundational Security Best Practicesを推奨）
  5. 有効化完了を確認

手順2: 検出結果の確認

  1. 左メニュー「検出結果」をクリック
  2. 重大度でフィルタリング（CRITICAL、HIGH）
  3. 個別の検出結果をクリックして詳細を確認
  4. 推奨される修復アクションを確認

手順3: GuardDutyの統合確認

  1. 「統合」タブをクリック
  2. GuardDutyが「有効」になっていることを確認
  3. 他のサービス（Config、Macieなど）も有効化可能

ポイント

Security Hubは、最初の30日間は無料です。実習後は、継続利用するかどうかを検討してください。

理解度確認クイズ

  1. Q1. SCP（サービスコントロールポリシー）の効果範囲は何ですか？

    回答を表示

    SCPは、適用されたOUまたはアカウント内のすべてのプリンシパル（IAMユーザー、ロール、ルートユーザー含む）の最大権限を制限します。 ただし、Management Account自体には適用されません。

  2. Q2. AWS Control Towerにおける「予防的ガードレール」と「検出的ガードレール」の違いは何ですか？

    回答を表示

    予防的ガードレールはSCPを使用して特定の操作を事前に拒否します。検出的ガードレールはConfig Rulesを使用してコンプライアンス違反を検出し、事後的に報告します。

  3. Q3. GuardDuty、Inspector、Macieはそれぞれ何を検出しますか？

    回答を表示

    ・GuardDuty: 脅威検出（悪意のある活動、異常なAPI呼び出し、不正アクセス）
    ・Inspector: 脆弱性スキャン（EC2、コンテナ、Lambda関数のCVE脆弱性）
    ・Macie: 機密データ検出（S3内のPII、認証情報、金融情報など）

  4. Q4. エンベロープ暗号化を使用する理由は何ですか？

    回答を表示

    大容量データをKMSで直接暗号化すると、データをネットワーク経由でKMSに送信する必要があり、パフォーマンスが低下します。 エンベロープ暗号化では、データキーをローカルで生成してデータを暗号化し、そのデータキーだけをKMSで暗号化するため、高速で効率的です。
