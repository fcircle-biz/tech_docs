セッションとクッキー - 状態管理の仕組み

HTTPのステートレス性、セッションフレームワークの使い方、クッキーの操作、ユーザーごとのデータ保存を学びます。

この章で学ぶこと

  • HTTPのステートレス性と状態管理の必要性
  • セッションとクッキーの基本概念と違い
  • Djangoのセッションフレームワークの使い方
  • セッションデータの保存・取得・削除とセキュリティ設定

11.1 HTTPのステートレス性と状態管理

Webアプリケーションの状態管理を理解する前に、まずHTTPプロトコルの基本的な性質を理解しましょう。

HTTPのステートレス性とは？

HTTPプロトコルはステートレス（状態を持たない）という性質を持っています。これは、サーバーが各リクエストを独立したものとして扱い、過去のリクエストの情報を保持しないことを意味します。

例えば、ECサイトで商品をカートに追加した後、別のページに移動すると、サーバーはあなたが誰なのか、何をカートに入れたのかを覚えていません。

HTTPリクエストの独立性

sequenceDiagram autonumber participant User as ユーザー participant Server as Webサーバー User->>Server: リクエスト1: ログイン Server-->>User: レスポンス1: ログイン成功 Note over Server: 接続終了 User->>Server: リクエスト2: 商品ページ Server-->>User: レスポンス2: 商品情報 Note over Server: 誰がアクセスしたか不明
（ステートレス）

このステートレス性により、サーバーはシンプルで効率的に動作できますが、ユーザーごとの情報を管理することができません。そこで、セッションとクッキーという仕組みを使って、状態を管理します。

11.2 セッションとクッキーの基本概念

セッションとクッキーは、どちらもユーザーの状態を管理するための仕組みですが、保存場所と用途が異なります。

クッキー（Cookie）

保存場所：ブラウザ（クライアント側）

  • 容量制限あり（4KB程度）
  • ユーザーが見たり削除できる
  • 有効期限を設定可能
  • 自動的にサーバーに送信される

セッション（Session）

保存場所：サーバー側

  • 容量制限が緩やか
  • ユーザーから直接見えない
  • セキュアな情報を保存可能
  • セッションIDでクッキー連携

セッションの仕組み

重要ポイント

セッションは、サーバー側にデータを保存し、ブラウザには「セッションID」というキー（鍵）だけを持たせます。このセッションIDを使って、サーバーは「このユーザーのデータはこれだ」と識別できます。

11.3 Djangoのセッションフレームワーク

Djangoにはセッション管理のための強力なフレームワークが組み込まれています。デフォルトで有効になっているため、すぐに使い始めることができます。

セッションの設定確認

settings.pyを確認すると、以下のようにセッション関連のミドルウェアとアプリがデフォルトで有効になっています。

config/settings.py 
# セッションアプリ（INSTALLされているアプリ）
INSTALLED_APPS = [
    # ...
    'django.contrib.sessions',  # セッション管理
    # ...
]

# セッションミドルウェア
MIDDLEWARE = [
    # ...
    'django.contrib.sessions.middleware.SessionMiddleware',
    # ...
]

セッションの保存先

Djangoでは、セッションデータの保存先をいくつか選択できます。デフォルトはデータベースです。

セッション保存先の種類

データベース（デフォルト）

django_session テーブルに保存。永続的で安全

キャッシュ

RedisやMemcachedを使用。高速だが揮発性

ファイル

ファイルシステムに保存。デバッグ用

クッキー

署名付きクッキーに直接保存。軽量だがサイズ制限あり

11.4 セッションデータの操作

Djangoでは、ビュー内でrequest.sessionを使ってセッションデータを操作できます。辞書（dict）のように扱えます。

セッションデータの保存

myapp/views.py 
from django.shortcuts import render
from django.http import HttpResponse

def save_to_session(request):
    # セッションにデータを保存（辞書のように扱う）
    request.session['username'] = 'tanaka'
    request.session['visit_count'] = 1
    request.session['cart'] = ['商品A', '商品B']

    return HttpResponse('セッションにデータを保存しました')

セッションデータの取得

myapp/views.py 
def get_from_session(request):
    # セッションからデータを取得
    username = request.session.get('username', 'ゲスト')
    visit_count = request.session.get('visit_count', 0)

    # デフォルト値を指定せずに取得（キーがない場合はKeyError）
    # cart = request.session['cart']

    return HttpResponse(f'ようこそ {username} さん（訪問回数: {visit_count}）')

ヒント

get()メソッドを使うと、キーが存在しない場合にデフォルト値を返すことができ、エラーを防げます。

セッションデータの更新

myapp/views.py 
def update_session(request):
    # 訪問カウントを増やす
    visit_count = request.session.get('visit_count', 0)
    request.session['visit_count'] = visit_count + 1

    # リストに要素を追加する場合は注意が必要
    cart = request.session.get('cart', [])
    cart.append('商品C')
    request.session['cart'] = cart  # 再代入が必要

    # または modified=True を設定
    request.session.modified = True

    return HttpResponse(f'訪問回数: {request.session["visit_count"]}')

注意

リストや辞書などのミュータブルオブジェクトを変更した場合、request.session.modified = Trueを設定するか、再代入しないと変更が保存されません。

セッションデータの削除

myapp/views.py 
def delete_from_session(request):
    # 特定のキーを削除
    if 'cart' in request.session:
        del request.session['cart']

    # pop()メソッドで削除（デフォルト値も指定可能）
    username = request.session.pop('username', None)

    # セッション全体をクリア
    request.session.flush()  # セッションIDも削除される

    # またはclear()でデータのみクリア
    request.session.clear()  # セッションIDは維持される

    return HttpResponse('セッションをクリアしました')

11.5 クッキーの操作

セッションとは別に、クッキーを直接操作することもできます。軽量なデータや長期保存したいデータに適しています。

クッキーの設定

myapp/views.py 
from django.http import HttpResponse

def set_cookie(request):
    response = HttpResponse('クッキーを設定しました')

    # 基本的なクッキーの設定
    response.set_cookie('theme', 'dark')

    # オプション付きでクッキーを設定
    response.set_cookie(
        key='language',
        value='ja',
        max_age=60*60*24*7,  # 7日間有効
        secure=True,  # HTTPS通信のみ
        httponly=True,  # JavaScriptからアクセス不可
    )

    return response

クッキーの取得

myapp/views.py 
def get_cookie(request):
    # クッキーから値を取得
    theme = request.COOKIES.get('theme', 'light')
    language = request.COOKIES.get('language', 'en')

    return HttpResponse(f'テーマ: {theme}, 言語: {language}')

クッキーの削除

myapp/views.py 
def delete_cookie(request):
    response = HttpResponse('クッキーを削除しました')

    # クッキーを削除
    response.delete_cookie('theme')
    response.delete_cookie('language')

    return response

11.6 実践例：ショッピングカート機能

セッションを使って、シンプルなショッピングカート機能を実装してみましょう。

myapp/views.py 
from django.shortcuts import render, redirect
from django.http import JsonResponse

def add_to_cart(request, product_id):
    """カートに商品を追加"""
    # カートを取得（なければ空の辞書を作成）
    cart = request.session.get('cart', {})

    # 商品IDをキーにして数量を管理
    product_id = str(product_id)
    if product_id in cart:
        cart[product_id] += 1  # 既存商品は数量を増やす
    else:
        cart[product_id] = 1  # 新規商品は数量1

    request.session['cart'] = cart

    return JsonResponse({'cart': cart})
myapp/views.py（続き） 
def view_cart(request):
    """カートの内容を表示"""
    cart = request.session.get('cart', {})

    # 合計個数を計算
    total_items = sum(cart.values())

    context = {
        'cart': cart,
        'total_items': total_items,
    }
    return render(request, 'cart.html', context)

def clear_cart(request):
    """カートをクリア"""
    if 'cart' in request.session:
        del request.session['cart']
    return redirect('view_cart')

11.7 セッションの設定とセキュリティ

セッションを安全に使うために、settings.pyで適切な設定を行いましょう。

config/settings.py 
# セッションの有効期限（秒）デフォルト: 2週間
SESSION_COOKIE_AGE = 60 * 60 * 24 * 14  # 2週間

# ブラウザを閉じたらセッションを削除
SESSION_EXPIRE_AT_BROWSER_CLOSE = False

# セキュリティ設定（本番環境では必須）
SESSION_COOKIE_SECURE = True  # HTTPS通信のみ
SESSION_COOKIE_HTTPONLY = True  # JavaScriptからアクセス不可
SESSION_COOKIE_SAMESITE = 'Lax'  # CSRF対策

# セッションの保存先をキャッシュに変更する場合
# SESSION_ENGINE = 'django.contrib.sessions.backends.cache'

セキュリティ設定の重要性

  • SESSION_COOKIE_SECURE: HTTPSでのみクッキーを送信。本番環境では必ずTrueに設定
  • SESSION_COOKIE_HTTPONLY: JavaScriptからのアクセスを防ぎ、XSS攻撃を防御
  • SESSION_COOKIE_SAMESITE: CSRF攻撃を防ぐため、他サイトからのリクエストでクッキーを送信しない

実習 11-1: 訪問カウンターの実装

セッションを使って、ユーザーごとの訪問回数をカウントする機能を作成しましょう。

手順

  1. 1 ビュー関数で、セッションから訪問回数を取得（初回は0）
  2. 2 訪問回数を1増やしてセッションに保存
  3. 3 テンプレートで訪問回数を表示
  4. 4 ブラウザで動作確認（リロードするたびに増えることを確認）

理解度確認クイズ

  1. Q1. HTTPのステートレス性とは何ですか？

    回答を表示

    HTTPプロトコルは各リクエストを独立したものとして扱い、過去のリクエストの情報を保持しないという性質です。サーバーは前回のやり取りを覚えていないため、ユーザーの状態を管理するにはセッションやクッキーといった仕組みが必要になります。

  2. Q2. セッションとクッキーの主な違いは何ですか？

    回答を表示

    保存場所が主な違いです。クッキーはブラウザ（クライアント側）に保存され、容量制限があり、ユーザーが見たり削除できます。一方、セッションはサーバー側に保存され、セキュアな情報を扱えます。ブラウザにはセッションIDだけが保存されます。

  3. Q3. セッション内のリストや辞書を更新した際に、必要な処理は何ですか？

    回答を表示

    リストや辞書などのミュータブルオブジェクトを変更した場合、request.session.modified = Trueを設定するか、変更後のオブジェクトをセッションに再代入する必要があります。そうしないと、変更がデータベースに保存されません。

第11章のまとめ

  • HTTPはステートレスなため、ユーザーの状態を管理するにはセッションとクッキーが必要です。
  • セッションはサーバー側にデータを保存し、セキュアな情報の管理に適しています。
  • request.sessionを辞書のように扱ってデータの保存・取得・削除ができます。
  • セキュリティ設定（SECURE、HTTPONLY、SAMESITE）は本番環境で必須です。
