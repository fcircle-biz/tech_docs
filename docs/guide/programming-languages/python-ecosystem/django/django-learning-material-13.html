テストの書き方 - 品質を保証する

Djangoテストフレームワークの使い方、単体テストと統合テストの実装、TestCaseクラスの活用、テストデータベースの扱いを学び、堅牢なアプリケーション開発を実現します。

この章で学ぶこと

  • Djangoテストフレームワークの基礎と利点
  • 単体テストと統合テストの違いと使い分け
  • TestCaseクラスを使ったテストコードの書き方
  • テストデータベースの仕組みとフィクスチャの活用

13.1 テストの重要性とDjangoテストフレームワーク

テストは、アプリケーションが期待通りに動作することを確認するための重要なプロセスです。テストコードを書くことで、コードの変更時にバグを早期に発見でき、リファクタリングも安心して行えます。

Djangoには強力なテストフレームワークが標準で組み込まれており、Pythonの標準ライブラリunittestを拡張した機能を提供します。データベースの自動セットアップ、テストクライアント、アサーションメソッドなど、Web開発に特化した便利な機能が揃っています。

テストを書くメリット

  • バグの早期発見: コード変更時に既存機能が壊れていないか自動でチェック
  • リファクタリングの安心感: テストが通れば、内部実装の変更が安全に行える
  • ドキュメントとしての役割: テストコードが仕様書として機能する
  • 品質の向上: エッジケースや異常系の動作を明確にできる

単体テスト（Unit Test）

モデルのメソッド、ビュー関数、フォームバリデーションなど、個々の小さな単位をテストします。

統合テスト（Integration Test）

複数のコンポーネントを組み合わせた動作をテストします。URLアクセス、フォーム送信、レスポンス確認など。

13.2 最初のテストを書く

それでは、実際にテストコードを書いてみましょう。各アプリケーションディレクトリには自動的にtests.pyファイルが作成されます。このファイルにテストコードを記述します。

テスト対象のモデル

まず、テスト対象となるシンプルなブログ記事モデルを作成します。

blog/models.py 
from django.db import models
from django.utils import timezone

class Post(models.Model):
    """ブログ記事モデル"""
    title = models.CharField(max_length=200)
    content = models.TextField()
    published_date = models.DateTimeField(null=True, blank=True)
    created_at = models.DateTimeField(auto_now_add=True)

    def publish(self):
        """記事を公開する"""
        self.published_date = timezone.now()
        self.save()

    def is_published(self):
        """公開済みかどうかを判定"""
        return self.published_date is not None

モデルの単体テスト

Postモデルのメソッドが正しく動作するか確認するテストを書きます。

blog/tests.py 
from django.test import TestCase
from django.utils import timezone
from .models import Post

class PostModelTestCase(TestCase):
    """Postモデルのテストケース"""

    def setUp(self):
        """各テストメソッド実行前に呼ばれる初期化処理"""
        self.post = Post.objects.create(
            title='テスト記事',
            content='これはテスト記事の本文です。'
        )

    def test_post_creation(self):
        """記事が正しく作成されるかテスト"""
        self.assertEqual(self.post.title, 'テスト記事')
        self.assertIsNotNone(self.post.created_at)
        self.assertIsNone(self.post.published_date)

    def test_publish_method(self):
        """publish()メソッドのテスト"""
        self.post.publish()
        self.assertIsNotNone(self.post.published_date)

    def test_is_published_method(self):
        """is_published()メソッドのテスト"""
        # 公開前
        self.assertFalse(self.post.is_published())
        # 公開後
        self.post.publish()
        self.assertTrue(self.post.is_published())

テストコードの重要ポイント

  • TestCaseを継承してテストクラスを定義
  • setUp()でテストデータを準備（各テストメソッド実行前に呼ばれる）
  • テストメソッド名はtest_で始める（必須）
  • アサーションメソッド（assertEqual、assertTrue等）で結果を検証

13.3 テストの実行方法

テストコードを書いたら、python manage.py testコマンドで実行します。Djangoが自動的にテスト用データベースを作成し、テストを実行後に削除します。

基本的なテスト実行

ターミナル 
# すべてのテストを実行
python manage.py test

# 特定のアプリのテストのみ実行
python manage.py test blog

# 特定のテストクラスのみ実行
python manage.py test blog.tests.PostModelTestCase

# 特定のテストメソッドのみ実行
python manage.py test blog.tests.PostModelTestCase.test_publish_method

13.4 ビューの統合テスト

ビューのテストでは、URLアクセスからレスポンスまでの一連の動作を確認します。DjangoのClientクラスを使うと、ブラウザのようにHTTPリクエストを送信できます。

テスト対象のビューとURL

blog/views.py 
from django.shortcuts import render, get_object_or_404
from .models import Post

def post_list(request):
    """公開済み記事一覧を表示"""
    posts = Post.objects.filter(
        published_date__isnull=False
    ).order_by('-published_date')
    return render(request, 'blog/post_list.html', {'posts': posts})

def post_detail(request, pk):
    """記事詳細を表示"""
    post = get_object_or_404(Post, pk=pk)
    return render(request, 'blog/post_detail.html', {'post': post})

ビューの統合テスト

blog/tests.py 
from django.test import TestCase, Client
from django.urls import reverse
from .models import Post

class PostViewTestCase(TestCase):
    """ビューのテストケース"""

    def setUp(self):
        """テストデータを準備"""
        self.client = Client()
        # 公開済み記事を作成
        self.published_post = Post.objects.create(
            title='公開済み記事',
            content='公開されています'
        )
        self.published_post.publish()
        # 未公開記事を作成
        self.draft_post = Post.objects.create(
            title='下書き記事',
            content='まだ公開されていません'
        )

    def test_post_list_view(self):
        """記事一覧ビューのテスト"""
        response = self.client.get(reverse('post_list'))
        # ステータスコード200が返る
        self.assertEqual(response.status_code, 200)
        # 公開済み記事が含まれる
        self.assertContains(response, '公開済み記事')
        # 下書き記事は含まれない
        self.assertNotContains(response, '下書き記事')

よく使うアサーションメソッド

メソッド 説明
assertEqual(a, b) a と b が等しいことを確認
assertTrue(x) x が True であることを確認
assertContains(response, text) レスポンスに指定のテキストが含まれることを確認
assertRedirects(response, url) 指定のURLにリダイレクトされることを確認

13.5 テストデータベースとフィクスチャ

Djangoのテストは、専用のテストデータベースで実行されます。テスト実行時に自動的に作成され、終了時に削除されるため、本番データに影響を与えません。

テストデータベースのライフサイクル

フィクスチャでテストデータを管理

フィクスチャ（fixture）は、テストデータをJSON形式で保存したファイルです。複雑なテストデータを再利用する際に便利です。

ターミナル（フィクスチャ作成） 
# 現在のデータからフィクスチャを生成
python manage.py dumpdata blog.Post --indent=2 > blog/fixtures/posts.json
blog/tests.py（フィクスチャ読み込み） 
class PostViewWithFixturesTestCase(TestCase):
    """フィクスチャを使用したテストケース"""
    fixtures = ['posts.json']  # 読み込むフィクスチャを指定

    def test_post_count(self):
        """フィクスチャからデータが読み込まれているかテスト"""
        count = Post.objects.count()
        self.assertGreater(count, 0)

テストのベストプラクティス

  • 独立性を保つ: 各テストは他のテストに依存せず、単独で実行可能に
  • 明確な命名: テストメソッド名で何をテストしているか分かるように
  • AAA パターン: Arrange（準備）、Act（実行）、Assert（検証）の順で記述
  • 境界値テスト: 正常系だけでなく、異常系やエッジケースもテスト

実習: フォームのテストを書こう

記事作成フォームのバリデーションをテストするコードを書いてみましょう。

課題1: フォームクラスを作成

  • タイトル（最大200文字、必須）
  • 本文（必須）のフィールドを持つフォーム

課題2: テストケースを作成

  • 正しいデータで送信した場合、バリデーションが通ることを確認
  • タイトルが空の場合、バリデーションエラーになることを確認
  • タイトルが200文字を超える場合、バリデーションエラーになることを確認

理解度確認クイズ

  1. Q1. Djangoのテストメソッド名は必ずどのように始める必要がありますか？

    回答を表示

    test_で始める必要があります。これにより、Djangoのテストランナーがそのメソッドをテストとして認識し、自動的に実行します。

  2. Q2. setUp()メソッドはいつ実行されますか？

    回答を表示

    各テストメソッドの実行前に毎回呼ばれます。これにより、各テストが独立した状態でクリーンなテストデータを使用できます。テストメソッドが3つあれば、setUp()も3回実行されます。

  3. Q3. テストデータベースは本番データベースに影響を与えますか？

    回答を表示

    いいえ、影響を与えません。Djangoはテスト実行時に専用のテストデータベースを自動的に作成し、テスト終了後に削除します。そのため、本番データやsettings.pyで指定されたデータベースには一切影響がありません。

第13章のまとめ

  • Djangoテストフレームワークを使うことで、モデル、ビュー、フォームなどの動作を自動的に検証できます。
  • 単体テストは個々のコンポーネント、統合テストは複数のコンポーネントの組み合わせをテストします。
  • TestCaseクラスを継承し、test_で始まるメソッドでテストを記述します。
  • テストデータベースは自動的に作成・削除されるため、本番データに影響を与えずにテストできます。
