デバッグとエラー処理

Django Debug Toolbarの活用、エラーページのカスタマイズ、ロギングの設定、例外処理のベストプラクティスを学びます。

14.1 デバッグとは - 問題発見と解決の技術

デバッグは、プログラム内の誤り（バグ）を発見し、修正するプロセスです。Webアプリケーション開発では、コードのエラー、パフォーマンスの問題、予期しない動作など、様々な問題に直面します。効率的なデバッグ技術を身につけることで、開発スピードと品質を大幅に向上させることができます。

開発時間の大半はデバッグ

研究によると、プログラマーは開発時間の50%以上をデバッグに費やしているとされています。適切なツールと技術を使うことで、この時間を大幅に短縮できます。

14.2 Django Debug Toolbar - 開発者の必須ツール

Django Debug Toolbarは、開発時に非常に有用なデバッグ情報をWebページ上に表示するサードパーティパッケージです。SQLクエリの実行回数、処理時間、テンプレートの情報など、パフォーマンス改善に必要な情報を視覚的に確認できます。

インストールと設定

まず、Django Debug Toolbarをインストールします。

ターミナル 
# Django Debug Toolbarをインストール
pip install django-debug-toolbar

次に、settings.pyに設定を追加します。

config/settings.py 
# INSTALLED_APPSに追加
INSTALLED_APPS = [
    # ... 既存のアプリ
    'debug_toolbar',  # Debug Toolbarを追加
]

# MIDDLEWAREに追加（できるだけ先頭に近い位置）
MIDDLEWARE = [
    'debug_toolbar.middleware.DebugToolbarMiddleware',  # 追加
    # ... 既存のミドルウェア
]

# 内部IPアドレスを設定（開発サーバーで表示されるように）
INTERNAL_IPS = [
    '127.0.0.1',
]

最後に、urls.pyにDebug ToolbarのURLを追加します。

config/urls.py 
from django.contrib import admin
from django.urls import path, include
from django.conf import settings

urlpatterns = [
    path('admin/', admin.site.urls),
    # ... 既存のURL
]

# DEBUGモードの場合のみDebug Toolbarを有効化
if settings.DEBUG:
    import debug_toolbar
    urlpatterns = [
        path('__debug__/', include(debug_toolbar.urls)),
    ] + urlpatterns

Debug Toolbarの使い方

開発サーバーを起動してWebページにアクセスすると、画面右側にDebug Toolbarのパネルが表示されます。このパネルから様々なデバッグ情報を確認できます。

SQLパネル

実行されたSQLクエリの一覧と実行時間を表示。N+1問題などのパフォーマンス問題を発見できます。

Templatesパネル

レンダリングされたテンプレートの一覧とコンテキスト変数を表示。テンプレートの問題を特定できます。

Requestパネル

HTTPリクエストの詳細（ヘッダー、GET/POSTデータ、セッション情報など）を表示します。

Timeパネル

リクエスト処理にかかった時間の内訳を表示。どの処理が遅いかを特定できます。

本番環境では無効化必須

Django Debug Toolbarは開発時のみ使用してください。本番環境では必ず無効化する必要があります。settings.pyのDEBUG = Falseにすることで自動的に無効化されます。

14.3 カスタムエラーページの作成

Djangoはデフォルトで404（ページが見つからない）、500（サーバーエラー）などのエラーページを提供しますが、ユーザーフレンドリーなカスタムエラーページを作成することで、より良いユーザー体験を提供できます。

主なHTTPエラーページ

404

Page Not Found（ページが見つからない）

存在しないURLにアクセスしたときに表示されます。

500

Internal Server Error（サーバーエラー）

プログラムのエラーや予期しない問題が発生したときに表示されます。

403

Forbidden（アクセス拒否）

アクセス権限がないページにアクセスしたときに表示されます。

カスタム404ページの作成手順

404エラーページをカスタマイズする方法を見ていきましょう。

ステップ1: テンプレートの作成

プロジェクトのルートテンプレートディレクトリに404.htmlを作成します。

templates/404.html 
<!DOCTYPE html>
<html lang="ja">
<head>
    <meta charset="UTF-8">
    <title>ページが見つかりません - 404エラー</title>
</head>
<body>
    <h1>ページが見つかりません</h1>
    <p>お探しのページは存在しないか、移動した可能性があります。</p>
    <a href="/">トップページに戻る</a>
</body>
</html>

ステップ2: DEBUGモードをオフにする

カスタムエラーページは、DEBUG = Falseの場合のみ表示されます。settings.pyで設定を変更します。

config/settings.py 
# 本番環境用の設定
DEBUG = False

# ALLOWED_HOSTSを設定（必須）
ALLOWED_HOSTS = ['localhost', '127.0.0.1']

500エラーページも同様に作成可能

500.html、403.htmlなども同じ方法で作成できます。テンプレート名をHTTPステータスコードにするだけです。

14.4 ロギング - 問題追跡の基盤

ロギングは、アプリケーションの動作状況やエラー情報をファイルやコンソールに記録する仕組みです。本番環境ではDEBUG = Falseのため詳細なエラー情報が表示されません。ログを適切に記録することで、問題の原因を特定しやすくなります。

ロギングレベルの理解

ロギングの基本設定

settings.pyにロギング設定を追加します。

config/settings.py 
import os

# ログの設定
LOGGING = {
    'version': 1,
    'disable_existing_loggers': False,
    'formatters': {
        'verbose': {
            'format': '{levelname} {asctime} {module} {message}',
            'style': '{',
        },
    },
    'handlers': {
        'file': {
            'level': 'INFO',
            'class': 'logging.FileHandler',
            'filename': os.path.join(BASE_DIR, 'logs/django.log'),
            'formatter': 'verbose',
        },
    },
    'loggers': {
        'django': {
            'handlers': ['file'],
            'level': 'INFO',
            'propagate': True,
        },
    },
}

ログファイルを保存するディレクトリを作成します。

ターミナル 
# プロジェクトルートにlogsディレクトリを作成
mkdir logs

ビューでのログ記録

ビュー関数内でログを記録する方法を見ていきましょう。

myapp/views.py 
import logging
from django.shortcuts import render

# ロガーを取得
logger = logging.getLogger(__name__)

def my_view(request):
    # 情報ログを記録
    logger.info(f'ユーザー {request.user.username} がページにアクセスしました')

    try:
        # 何らかの処理
        result = some_function()
        logger.debug(f'処理結果: {result}')

    except Exception as e:
        # エラーログを記録
        logger.error(f'エラーが発生しました: {str(e)}', exc_info=True)
        return render(request, 'error.html')

    return render(request, 'success.html')

ロギングのベストプラクティス

  • 適切なログレベルを使用する（DEBUGは開発時のみ）
  • 個人情報（パスワード、クレジットカード番号など）をログに記録しない
  • ログファイルを定期的にローテーションする（サイズ管理）
  • エラー時はexc_info=Trueでスタックトレースを記録

14.5 例外処理のベストプラクティス

適切な例外処理は、アプリケーションの安定性とユーザー体験を大きく向上させます。予期しないエラーに対して適切に対処し、ユーザーに分かりやすいメッセージを表示することが重要です。

Django固有の例外処理

Djangoには便利な例外クラスが用意されており、適切に使用することで読みやすいコードを書けます。

myapp/views.py 
from django.shortcuts import render, get_object_or_404
from django.core.exceptions import PermissionDenied, ValidationError
from django.http import Http404
from .models import Article

def article_detail(request, pk):
    # get_object_or_404を使用（存在しない場合は404を返す）
    article = get_object_or_404(Article, pk=pk)

    # 公開されていない記事は表示しない
    if not article.is_published:
        raise Http404("この記事は公開されていません")

    # 閲覧権限のチェック
    if not request.user.has_perm('myapp.view_article'):
        raise PermissionDenied("この記事を閲覧する権限がありません")

    return render(request, 'article_detail.html', {'article': article})

データベース例外の処理

データベース操作時の例外を適切に処理します。

myapp/views.py 
from django.db import IntegrityError, transaction
from django.contrib import messages

def create_article(request):
    if request.method == 'POST':
        try:
            # トランザクション内で処理
            with transaction.atomic():
                article = Article(
                    title=request.POST['title'],
                    content=request.POST['content']
                )
                article.save()
                messages.success(request, '記事を作成しました')

        except IntegrityError:
            messages.error(request, '同じタイトルの記事が既に存在します')

        except KeyError as e:
            messages.error(request, f'必須項目が入力されていません: {e}')

    return render(request, 'create_article.html')

カスタム例外の作成

アプリケーション固有のエラーには、カスタム例外を定義すると分かりやすくなります。

myapp/exceptions.py 
# カスタム例外を定義
class ArticleNotPublishedException(Exception):
    """記事が公開されていない場合の例外"""
    pass

class InsufficientCreditsException(Exception):
    """クレジット不足の例外"""
    pass
myapp/views.py 
from .exceptions import ArticleNotPublishedException, InsufficientCreditsException

def purchase_article(request, pk):
    try:
        article = Article.objects.get(pk=pk)

        if not article.is_published:
            raise ArticleNotPublishedException("この記事は公開されていません")

        if request.user.credits < article.price:
            raise InsufficientCreditsException("クレジットが不足しています")

        # 購入処理
        request.user.credits -= article.price
        request.user.save()

    except ArticleNotPublishedException as e:
        messages.error(request, str(e))
    except InsufficientCreditsException as e:
        messages.error(request, str(e))

    return redirect('article_list')

実習: Debug Toolbarを使ってN+1問題を発見する

  1. 1. Django Debug Toolbarをインストール
    pip install django-debug-toolbarを実行し、settings.pyとurls.pyに設定を追加します。
  2. 2. N+1問題が発生するビューを作成
    記事一覧を表示するビューで、各記事の作成者名を表示する際にN+1問題が発生する状態を作ります。
  3. 3. Debug ToolbarのSQLパネルで確認
    ページにアクセスして、SQLパネルで実行されたクエリ数を確認します。記事数+1回のクエリが実行されていることを確認します。
  4. 4. select_relatedで最適化
    select_related('author')を使ってクエリを最適化し、1回のクエリで済むことを確認します。

よくあるトラブルと解決方法

Debug Toolbarが表示されない

• DEBUG = Trueになっているか確認
• INTERNAL_IPSに'127.0.0.1'が含まれているか確認
• テンプレートに<body>タグが存在するか確認

カスタムエラーページが表示されない

• DEBUG = Falseになっているか確認
• ALLOWED_HOSTSが正しく設定されているか確認
• テンプレートディレクトリの設定が正しいか確認

ログファイルが作成されない

• logsディレクトリが存在し、書き込み権限があるか確認
• LOGGING設定のファイルパスが正しいか確認
• ログレベルが適切に設定されているか確認

理解度確認クイズ

  1. Q1. Django Debug Toolbarで確認できる情報として正しくないものはどれですか？

    回答を表示

    本番環境のエラーログです。Debug Toolbarは開発環境（DEBUG=True）でのみ動作し、SQLクエリ、テンプレート情報、リクエスト情報、処理時間などを確認できますが、本番環境では使用できません。

  2. Q2. カスタムエラーページ（404.htmlなど）が表示される条件は何ですか？

    回答を表示

    DEBUG = Falseに設定されていることが必要です。開発環境（DEBUG=True）では、Djangoがデフォルトの詳細なエラーページを表示します。また、ALLOWED_HOSTSも適切に設定されている必要があります。

  3. Q3. Pythonロギングの5つのレベルを、重要度の低い順に並べてください。

    回答を表示

    DEBUG → INFO → WARNING → ERROR → CRITICALの順です。DEBUGが最も詳細な情報で、CRITICALが最も深刻なエラーを表します。

第14章のまとめ

  • Django Debug Toolbarは、SQLクエリやテンプレート情報など、開発時のデバッグに必要な情報を視覚的に表示する強力なツールです。
  • カスタムエラーページ（404.html、500.htmlなど）を作成することで、ユーザーフレンドリーなエラー表示を実現できます。
  • ロギングを適切に設定することで、本番環境での問題追跡が容易になります。ログレベルを理解し、適切に使い分けましょう。
  • 例外処理では、Django固有の例外クラスを活用し、カスタム例外を定義することで、読みやすく保守しやすいコードを書けます。
