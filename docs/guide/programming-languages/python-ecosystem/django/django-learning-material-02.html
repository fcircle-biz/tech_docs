はじめてのDjangoプロジェクト

プロジェクトとアプリケーションの違いを理解し、実際にアプリケーションを作成します。また、設定ファイルの基礎とmanage.pyコマンドの活用方法を学びます。

この章で学ぶこと

  • プロジェクトとアプリケーションの概念の違い
  • プロジェクト構造とそれぞれのファイルの役割
  • アプリケーションを作成して登録する手順
  • settings.pyの基本設定項目とmanage.pyコマンドの使い方

2.1 プロジェクトとアプリケーションの違い

Djangoでは「プロジェクト」と「アプリケーション」という2つの概念を使い分けます。これはDjango開発の基礎となる重要な考え方です。

プロジェクトとアプリケーションの関係

全体の設定と統括] B[ブログアプリ
記事管理機能] C[ユーザーアプリ
認証・プロフィール] D[コメントアプリ

プロジェクト（Project）

Webサイト全体の設定や構成を管理する「入れ物」です。データベース設定、URLルーティング、セキュリティ設定など、サイト全体に関わる設定を一元管理します。

例: ECサイトプロジェクト、ブログサイトプロジェクト、社内システムプロジェクトなど

アプリケーション（Application）

特定の機能を担当する「部品」です。1つの明確な目的を持ち、再利用可能なように設計されます。プロジェクト内に複数のアプリケーションを作成できます。

例: ブログアプリ（記事管理）、ユーザーアプリ（認証）、コメントアプリ（コメント機能）など

設計思想

アプリケーションは「1つのことをうまくやる」設計にします。例えば、ブログ機能とユーザー管理機能は別々のアプリケーションに分けることで、コードの保守性が高まり、他のプロジェクトでも再利用しやすくなります。

2.2 プロジェクト構造の詳細理解

第1章で作成したプロジェクトの構造を詳しく見ていきましょう。それぞれのファイルがどんな役割を持っているかを理解することが、Djangoを使いこなす第一歩です。

プロジェクト構成（復習） 

myproject/
├── venv/              # 仮想環境
├── config/            # プロジェクト設定ディレクトリ
│   ├── __init__.py    # Pythonパッケージ識別ファイル
│   ├── settings.py    # プロジェクト全体の設定
│   ├── urls.py        # URLルーティング設定
│   ├── asgi.py        # ASGIサーバー設定
│   └── wsgi.py        # WSGIサーバー設定
└── manage.py          # Django管理コマンド

各ファイルの役割

manage.py

Djangoプロジェクトを管理するためのコマンドラインツールです。開発サーバーの起動、データベースのマイグレーション、アプリケーションの作成など、様々な操作に使用します。

python manage.py runserver - 開発サーバー起動
python manage.py startapp アプリ名 - アプリ作成

settings.py

プロジェクト全体の設定ファイルです。データベース接続情報、インストール済みアプリケーション、タイムゾーン、言語設定など、プロジェクトの動作を制御する重要な設定が記述されています。

主な設定項目: DEBUG, ALLOWED_HOSTS, INSTALLED_APPS, DATABASES, LANGUAGE_CODE, TIME_ZONE

urls.py

URLルーティングを定義するファイルです。どのURLにアクセスされたら、どのビュー（処理）を実行するかのマッピングを記述します。Webアプリケーションの「地図」のような役割を果たします。

例: /blog/ → ブログ一覧表示、/blog/1/ → 記事詳細表示

wsgi.py / asgi.py

本番環境でWebサーバーとDjangoアプリケーションを接続するための設定ファイルです。wsgi.pyは従来の同期型、asgi.pyは非同期処理に対応した新しい規格です。開発段階ではほとんど触れる必要はありません。

__init__.py

このディレクトリがPythonパッケージであることを示す空ファイルです。Pythonがこのディレクトリをモジュールとして認識するために必要ですが、通常は中身を編集することはありません。

2.3 アプリケーションの作成

実際にアプリケーションを作成して、プロジェクトに登録する手順を学びましょう。ここでは例として「blog」という名前のアプリケーションを作成します。

実習 2-1: blogアプリケーションの作成

ブログ機能を担当するアプリケーションを作成し、プロジェクトに登録します。

手順

  1. 1 ターミナルでプロジェクトのルートディレクトリ（manage.pyがある場所）にいることを確認します。
  2. 2 以下のコマンドを実行してblogアプリケーションを作成します。
ターミナル 
# blogアプリケーションを作成
python manage.py startapp blog

作成されるアプリケーション構造 

myproject/
├── config/
├── blog/              # 新しく作成されたアプリケーション
│   ├── __init__.py    # Pythonパッケージ識別ファイル
│   ├── admin.py       # 管理画面の設定
│   ├── apps.py        # アプリケーションの設定
│   ├── models.py      # データモデル定義
│   ├── tests.py       # テストコード
│   ├── views.py       # ビュー（処理）定義
│   └── migrations/    # データベースマイグレーション
│       └── __init__.py
└── manage.py

アプリケーション内の各ファイル

models.py

データベースのテーブル構造を定義します。MVTの「Model」に該当します。

views.py

リクエストを処理する関数やクラスを定義します。MVTの「View」に該当します。

admin.py

Django管理画面でこのアプリのデータを管理できるように設定します。

tests.py

アプリケーションの動作確認を自動化するテストコードを記述します。

アプリケーションをプロジェクトに登録

作成したアプリケーションは、settings.pyのINSTALLED_APPSに登録する必要があります。これにより、Djangoがこのアプリケーションを認識します。

config/settings.py 
# インストール済みアプリケーション
INSTALLED_APPS = [
    'django.contrib.admin',
    'django.contrib.auth',
    'django.contrib.contenttypes',
    'django.contrib.sessions',
    'django.contrib.messages',
    'django.contrib.staticfiles',
    'blog',  # 追加: 作成したblogアプリ
]

アプリ名の指定方法

アプリ名は文字列として追加します。より明示的にblog.apps.BlogConfigという形式で登録することもできます（apps.pyで定義されたクラス名を指定）。

2.4 settings.pyの基礎知識

settings.pyはDjangoプロジェクトの心臓部です。ここでは初心者が最初に知っておくべき重要な設定項目を紹介します。

DEBUG

デバッグモードのオン/オフを制御します。開発中はTrue、本番環境では必ずFalseに設定します。

DEBUG = True - 開発環境（詳細なエラー情報を表示）
DEBUG = False - 本番環境（エラー情報を隠す）

ALLOWED_HOSTS

アクセスを許可するホスト名のリストです。本番環境では実際のドメイン名を指定します。

ALLOWED_HOSTS = [] - 開発環境（localhostのみ）
ALLOWED_HOSTS = ['example.com'] - 本番環境

INSTALLED_APPS

プロジェクトで使用するアプリケーションのリストです。Djangoの標準アプリと自作アプリの両方を登録します。

デフォルトで含まれるアプリ: admin（管理画面）, auth（認証）, contenttypes, sessions, messages, staticfiles

DATABASES

データベースの接続設定です。デフォルトではSQLiteが設定されており、すぐに使い始められます。

デフォルト: SQLite（ファイルベース、開発に最適）
本番環境: PostgreSQL、MySQLなどに変更可能

LANGUAGE_CODE / TIME_ZONE

サイトの言語とタイムゾーンの設定です。日本向けサイトの場合は以下のように設定します。

LANGUAGE_CODE = 'ja' - 日本語
TIME_ZONE = 'Asia/Tokyo' - 日本時間
config/settings.py（日本語化の例） 
# 国際化設定
LANGUAGE_CODE = 'ja'  # 日本語に変更
TIME_ZONE = 'Asia/Tokyo'  # 日本時間に変更

USE_I18N = True  # 国際化を有効化
USE_TZ = True    # タイムゾーン対応を有効化

2.5 manage.pyコマンドの活用

manage.pyは開発作業を効率化する様々なコマンドを提供しています。よく使うコマンドを紹介します。

よく使うmanage.pyコマンド

python manage.py runserver

開発サーバーを起動します。デフォルトでhttp://127.0.0.1:8000/で起動します。

python manage.py startapp アプリ名

新しいアプリケーションを作成します。

python manage.py makemigrations

モデルの変更をマイグレーションファイルとして記録します。

python manage.py migrate

マイグレーションをデータベースに適用します。

python manage.py createsuperuser

管理画面にログインするための管理者ユーザーを作成します。

python manage.py shell

Djangoの環境が読み込まれたPythonシェルを起動します。

コマンドのヘルプ

python manage.py help ですべてのコマンド一覧が表示されます。
python manage.py help コマンド名 で特定のコマンドの詳細が確認できます。

理解度確認クイズ

  1. Q1. プロジェクトとアプリケーションの違いを簡潔に説明してください。

    回答を表示

    プロジェクトは、Webサイト全体の設定や構成を管理する「入れ物」です。一方、アプリケーションは特定の機能を担当する「部品」で、再利用可能に設計されます。1つのプロジェクトに複数のアプリケーションを含めることができます。

  2. Q2. 新しく作成したアプリケーションをDjangoに認識させるために、どのファイルのどの設定項目を編集する必要がありますか？

    回答を表示

    settings.pyファイルのINSTALLED_APPSリストに、作成したアプリケーション名を追加する必要があります。例: INSTALLED_APPS = [..., 'blog']

  3. Q3. settings.pyのDEBUG設定について、開発環境と本番環境でどう設定すべきですか？

    回答を表示

    開発環境ではDEBUG = Trueに設定して詳細なエラー情報を表示させます。しかし本番環境では必ずDEBUG = Falseに設定し、セキュリティ情報が外部に漏れないようにする必要があります。

第2章のまとめ

  • プロジェクトはWebサイト全体の設定を管理する「入れ物」、アプリケーションは特定機能を担当する「部品」です。
  • settings.pyでプロジェクト全体の設定（DEBUG、DATABASES、言語、タイムゾーンなど）を管理します。
  • python manage.py startappでアプリケーションを作成し、INSTALLED_APPSに登録することで利用可能になります。
  • manage.pyコマンドを活用することで、開発作業を効率的に進められます。
