データベースクエリの最適化

QuerySetの詳細、select_relatedとprefetch_related、N+1問題の解決、データベースインデックスの活用を習得します。

この章で学ぶこと

  • QuerySetの遅延評価の仕組みと評価タイミング
  • select_relatedによる1対1、多対1リレーションの最適化
  • prefetch_relatedによる1対多、多対多リレーションの最適化
  • N+1問題の原因と解決方法、データベースインデックスの効果的な活用

12.1 QuerySetの遅延評価

DjangoのQuerySetは遅延評価（Lazy Evaluation）という特徴を持っています。これは、QuerySetを作成しただけではデータベースクエリが実行されず、実際にデータが必要になったタイミングで初めてクエリが実行される仕組みです。

QuerySet評価の流れ

flowchart LR A[QuerySetを作成] -->|評価されない| B[フィルタを追加] B -->|評価されない| C[並び替えを指定] C -->|評価される| D[データにアクセス] D -->|SQL実行| E[(データベース)] E -->|結果を返す| F[Pythonオブジェクト] style A fill:#dbeafe,stroke:#3b82f6 style B fill:#dbeafe,stroke:#3b82f6 style C fill:#dbeafe,stroke:#3b82f6 style D fill:#fef3c7,stroke:#f59e0b style E fill:#dcfce7,stroke:#10b981 style F fill:#e0f2fe,stroke:#0ea5e9
views.py - 遅延評価の動作例 
# QuerySetを作成（この時点ではクエリ未実行）
articles = Article.objects.all()

# フィルタを追加（この時点でもクエリ未実行）
published_articles = articles.filter(status='published')

# 並び替えを指定（この時点でもクエリ未実行）
sorted_articles = published_articles.order_by('-created_at')

# データにアクセス（ここで初めてクエリ実行）
for article in sorted_articles[:10]:
    print(article.title)

遅延評価のメリット

複数のフィルタ条件を組み合わせても、最終的に1つのSQL文にまとめられるため、データベースへのアクセス回数を最小限に抑えられます。必要なデータだけを効率的に取得できるのが特徴です。

12.2 QuerySetが評価されるタイミング

QuerySetは以下のような操作を行った時に評価（データベースクエリの実行）が行われます。主な評価タイミングを理解しておくことが重要です。

イテレーション

forループでQuerySetを反復処理する時

スライシング

articles[:5]のようにスライス記法でアクセスする時

len()関数

len(articles)で件数を取得する時

list()関数

list(articles)でリストに変換する時

評価タイミングの例 
# クエリ未実行
articles = Article.objects.all()

# イテレーション - クエリ実行
for article in articles:
    print(article.title)

# スライシング - クエリ実行（LIMIT句付き）
first_five = articles[:5]

# 件数取得 - クエリ実行
count = articles.count()  # count()を使う方が効率的

12.3 N+1問題とは

N+1問題は、リレーションを持つデータを取得する際に発生する典型的なパフォーマンス問題です。1回のクエリでN件のデータを取得し、その後、各データに関連する情報を取得するためにN回の追加クエリが発生してしまう問題を指します。

N+1問題が発生する例

10件の記事を取得し、各記事の著者名を表示する場合、合計11回（1回＋10回）のクエリが発生してしまいます。

models.py - モデル定義 
from django.db import models

class Author(models.Model):
    name = models.CharField(max_length=100)
    email = models.EmailField()

class Article(models.Model):
    title = models.CharField(max_length=200)
    content = models.TextField()
    author = models.ForeignKey(Author, on_delete=models.CASCADE)
    created_at = models.DateTimeField(auto_now_add=True)
❌ 悪い例 - N+1問題が発生 
# 記事を10件取得（1回のクエリ）
articles = Article.objects.all()[:10]

# 各記事の著者名を表示（10回のクエリが追加発生）
for article in articles:
    print(f"{article.title} by {article.author.name}")

N+1問題のクエリ実行パターン

sequenceDiagram autonumber participant V as View participant DB as Database Note over V,DB: 最初のクエリ V->>DB: SELECT * FROM article LIMIT 10 DB-->>V: 10件の記事 Note over V,DB: N回の追加クエリ loop 記事ごと V->>DB: SELECT * FROM author WHERE id=1 DB-->>V: 著者情報 end Note over V,DB: 合計11回のクエリが発生

12.4 select_related - JOIN による最適化

select_related()は、ForeignKeyやOneToOneFieldなどの単一のリレーション（多対1、1対1）を最適化するメソッドです。SQL JOINを使用して、関連データを1回のクエリで取得します。

✅ 良い例 - select_relatedで最適化 
# 記事と著者を1回のクエリで取得（JOIN使用）
articles = Article.objects.select_related('author')[:10]

# 追加のクエリは発生しない（キャッシュから取得）
for article in articles:
    print(f"{article.title} by {article.author.name}")

select_relatedのクエリ実行パターン

sequenceDiagram autonumber participant V as View participant DB as Database Note over V,DB: 1回のJOINクエリで完結 V->>DB: SELECT * FROM article
LEFT JOIN author ON article.author_id = author.id
LIMIT 10 DB-->>V: 10件の記事 + 著者情報 Note over V,DB: 追加クエリなし（合計1回）

select_relatedの効果

N+1問題を解決し、クエリ回数を大幅に削減できます。特にForeignKeyやOneToOneFieldを参照する場合は必ず使用しましょう。

複数のリレーションを指定

select_related()には複数のフィールドを指定できます。また、ネストしたリレーション（リレーション先のリレーション）も取得可能です。

複数リレーションの例 
# 複数のForeignKeyを一度に取得
articles = Article.objects.select_related(
    'author',      # 著者
    'category'     # カテゴリ
)

# ネストしたリレーション（著者の所属組織）
articles = Article.objects.select_related(
    'author__organization'  # 著者の組織まで取得
)

12.5 prefetch_related - 複数レコードの最適化

prefetch_related()は、逆参照やManyToManyFieldなど、複数のレコードを返すリレーション（1対多、多対多）を最適化するメソッドです。別途クエリを発行して関連データを取得し、Python側でまとめます。

models.py - コメントモデルの追加 
class Comment(models.Model):
    article = models.ForeignKey(
        Article,
        on_delete=models.CASCADE,
        related_name='comments'  # 逆参照名
    )
    content = models.TextField()
    created_at = models.DateTimeField(auto_now_add=True)
❌ 悪い例 - N+1問題が発生 
# 記事を10件取得
articles = Article.objects.all()[:10]

# 各記事のコメント件数を表示（10回のクエリが追加発生）
for article in articles:
    comment_count = article.comments.count()
    print(f"{article.title}: {comment_count}件のコメント")
✅ 良い例 - prefetch_relatedで最適化 
# 記事とコメントを効率的に取得（2回のクエリ）
articles = Article.objects.prefetch_related('comments')[:10]

# 追加のクエリは発生しない
for article in articles:
    comment_count = article.comments.count()
    print(f"{article.title}: {comment_count}件のコメント")

prefetch_relatedのクエリ実行パターン

sequenceDiagram autonumber participant V as View participant DB as Database Note over V,DB: 1回目: メインのクエリ V->>DB: SELECT * FROM article LIMIT 10 DB-->>V: 10件の記事 Note over V,DB: 2回目: 関連データをまとめて取得 V->>DB: SELECT * FROM comment
WHERE article_id IN (1,2,3,...,10) DB-->>V: 全コメント Note over V,DB: Python側でマージ（合計2回）

select_relatedとの使い分け

select_related: ForeignKey、OneToOneField（多対1、1対1）
prefetch_related: 逆参照、ManyToManyField（1対多、多対多）

12.6 組み合わせて使う

実際のアプリケーションでは、select_related()とprefetch_related()を組み合わせて使用することで、複雑なリレーションも効率的に取得できます。

両方を組み合わせた例 
# 記事、著者、コメントを効率的に取得
articles = Article.objects.select_related(
    'author',           # ForeignKey: JOIN
    'category'          # ForeignKey: JOIN
).prefetch_related(
    'comments',         # 逆参照: 別クエリ
    'tags'              # ManyToMany: 別クエリ
)[:10]

# すべてのデータがキャッシュされている
for article in articles:
    print(f"タイトル: {article.title}")
    print(f"著者: {article.author.name}")
    print(f"カテゴリ: {article.category.name}")
    print(f"コメント数: {article.comments.count()}")
    print(f"タグ: {', '.join(tag.name for tag in article.tags.all())}")

12.7 データベースインデックス

インデックスは、データベースの検索を高速化するための仕組みです。本の索引のように、特定のカラムの値から目的のデータを素早く見つけられるようにします。

インデックスの追加方法

models.py - インデックスの定義 
class Article(models.Model):
    title = models.CharField(max_length=200)
    slug = models.SlugField(unique=True, db_index=True)  # 単一カラムインデックス
    author = models.ForeignKey(Author, on_delete=models.CASCADE)
    status = models.CharField(max_length=20)
    created_at = models.DateTimeField(auto_now_add=True, db_index=True)

    class Meta:
        # 複合インデックス（複数カラム）
        indexes = [
            models.Index(fields=['status', 'created_at']),
            models.Index(fields=['author', 'created_at']),
        ]

インデックスを追加すべき場合

  • 頻繁にWHERE句で検索するカラム
  • ORDER BY句で並び替えに使うカラム
  • JOIN時に使用するForeignKey
  • uniqueを指定したカラム（自動的にインデックスが作成される）

インデックスの注意点

インデックスは検索を高速化しますが、データ挿入・更新時のオーバーヘッドが増加します。すべてのカラムにインデックスを張るのではなく、実際に検索に使われるカラムに限定しましょう。

12.8 クエリのデバッグと確認

実際に発行されているSQLクエリを確認することで、パフォーマンス問題を発見し、最適化できます。

方法1: query属性でSQL確認

発行されるSQLを確認 
# QuerySetのSQLを確認
queryset = Article.objects.select_related('author').filter(status='published')
print(queryset.query)

# 出力例:
# SELECT "article"."id", "article"."title", "author"."id", "author"."name"
# FROM "article"
# LEFT JOIN "author" ON ("article"."author_id" = "author"."id")
# WHERE "article"."status" = 'published'

方法2: Django Debug Toolbar

Django Debug Toolbarを使うと、各ページで実行されたクエリの回数、実行時間、重複クエリなどを視覚的に確認できます。

Django Debug Toolbarのインストール

実習 12-1: クエリ最適化の実践

ブログアプリでN+1問題を解決し、クエリを最適化してみましょう。

手順

  1. 1 Article、Author、Commentモデルを作成し、マイグレーションを実行
  2. 2 Django管理画面から、複数の記事・著者・コメントデータを作成
  3. 3 最適化前のビューを作成し、querysetのqueryを出力して確認
  4. 4 select_related()とprefetch_related()を使って最適化
  5. 5 最適化前後でクエリ回数を比較し、効果を確認

理解度確認クイズ

  1. Q1. N+1問題とは何ですか？また、どのような場合に発生しますか？

    回答を表示

    N+1問題は、1回のクエリでN件のデータを取得した後、各データの関連情報を取得するためにN回の追加クエリが発生してしまう問題です。例えば、10件の記事を取得し、各記事の著者名を表示する際に、合計11回（1+10）のクエリが発生します。リレーションを持つデータを扱う際に最適化を行わないと発生します。

  2. Q2. select_related()とprefetch_related()の違いは何ですか？

    回答を表示

    select_related()は、ForeignKeyやOneToOneFieldなどの単一のリレーション（多対1、1対1）に使用し、SQL JOINを使って1回のクエリで関連データを取得します。prefetch_related()は、逆参照やManyToManyFieldなど、複数のレコードを返すリレーション（1対多、多対多）に使用し、別のクエリで関連データを取得してPython側で結合します。

  3. Q3. データベースインデックスはどのような場合に追加すべきですか？

    回答を表示

    インデックスは、頻繁にWHERE句で検索するカラム、ORDER BY句で並び替えに使うカラム、JOIN時に使用するForeignKeyなどに追加すべきです。ただし、インデックスはデータの挿入・更新時のオーバーヘッドを増加させるため、実際に検索に使われるカラムに限定し、すべてのカラムに張るのは避けましょう。

第12章のまとめ

  • QuerySetは遅延評価され、データに実際にアクセスするまでクエリは実行されません。複数のフィルタ条件を効率的に1つのSQLにまとめられます。
  • N+1問題は、リレーションデータを取得する際に多数のクエリが発生する問題です。select_related()やprefetch_related()で解決できます。
  • select_related()は多対1、1対1のリレーションにJOINを使用し、prefetch_related()は1対多、多対多のリレーションに別クエリを使用します。
  • データベースインデックスは検索を高速化しますが、挿入・更新のコストも考慮して適切に設定する必要があります。
