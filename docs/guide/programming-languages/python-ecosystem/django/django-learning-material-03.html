URLルーティング - リクエストの道筋を設計する

URLconfの仕組み、URLパターンの定義方法、パラメータの受け渡し、名前付きURLとリバースURLの活用方法を理解します。

この章で学ぶこと

  • URLconfの役割とルーティングの基本概念
  • path()関数を使ったURLパターンの定義方法
  • パスコンバータを使ったパラメータの受け渡し
  • 名前付きURLとリバースURL解決の活用方法

3.1 URLルーティングとは

URLルーティングは、ユーザーがアクセスしたURL（例: /articles/123/）を、適切なビュー関数に振り分ける仕組みです。DjangoではURLconfという設定ファイル（urls.py）でこのルーティングを定義します。

URLルーティングは、Webアプリケーションの「入口」や「案内板」にあたります。適切なルーティング設計により、わかりやすいURLを提供し、ユーザビリティとSEOの向上に貢献します。

URLルーティングの主な役割

  • URLパターンとビュー関数の紐付け
  • URLからパラメータを抽出してビューに渡す
  • わかりやすく美しいURL構造の実現
  • 柔軟なURL設計と変更への対応

3.2 基本的なURLパターンの定義

DjangoのURLパターンはpath()関数を使って定義します。基本的な構文を見ていきましょう。

path()関数の基本構文

urls.py（基本構文） 
from django.urls import path
from . import views

urlpatterns = [
    # path(URLパターン, ビュー関数, name=名前)
    path('', views.index, name='index'),
    path('about/', views.about, name='about'),
    path('contact/', views.contact, name='contact'),
]

path()関数の引数

第1引数: URLパターン（文字列）

マッチさせるURLのパターン。末尾のスラッシュ（/）に注意

第2引数: ビュー関数

URLにマッチした時に呼び出される関数

name引数: URL名（オプション）

テンプレートやビューから逆引きする際に使用する名前

実践例：ブログアプリのURL設計

blog/urls.py 
from django.urls import path
from . import views

app_name = 'blog'  # アプリケーション名前空間

urlpatterns = [
    # トップページ（記事一覧）
    path('', views.post_list, name='post_list'),
    # 記事詳細
    path('post/<int:pk>/', views.post_detail, name='post_detail'),
    # カテゴリ別記事一覧
    path('category/<slug:category_slug>/', views.category_posts, name='category_posts'),
]

app_nameの重要性

app_nameを設定することで、異なるアプリ間で同じURL名を使っても衝突を避けられます。テンプレートではblog:post_listのように名前空間付きで参照します。

3.3 パスコンバータ - URLからパラメータを取得

パスコンバータは、URLパターン内で<型:変数名>の形式で記述し、URLから値を抽出してビュー関数に渡します。

主なパスコンバータ

str（文字列）

スラッシュ以外の任意の文字列にマッチ

<str:username>
例: /users/john/

int（整数）

0または正の整数にマッチ

<int:pk>
例: /articles/123/

slug（スラッグ）

英数字、ハイフン、アンダースコア

<slug:slug>
例: /blog/hello-world/

uuid（UUID）

UUID形式の文字列にマッチ

<uuid:id>
例: /item/075194d3.../

ビュー関数でのパラメータ受け取り

views.py 
from django.shortcuts import render, get_object_or_404
from .models import Post

# 記事詳細ビュー - URLから記事IDを受け取る
def post_detail(request, pk):
    # pkパラメータを使ってデータベースから記事を取得
    post = get_object_or_404(Post, pk=pk)
    return render(request, 'blog/post_detail.html', {'post': post})

# カテゴリ別記事一覧 - スラッグを受け取る
def category_posts(request, category_slug):
    posts = Post.objects.filter(category__slug=category_slug)
    return render(request, 'blog/category_posts.html', {'posts': posts})

パラメータ名の一致

URLパターンの変数名（例: pk）とビュー関数の引数名は一致させる必要があります。Djangoは変数名をもとに値をビュー関数に渡します。

3.4 名前付きURLとリバースURL解決

URLパターンに名前を付けることで、テンプレートやビュー内から名前でURLを参照できます。これにより、実際のURLパスを変更しても、コードの修正箇所を最小限に抑えられます。

リバースURL解決のメリット

  • URLの変更に強い設計（DRY原則）
  • ハードコーディングの排除
  • 保守性の向上
  • タイプミスの防止

テンプレートでのurl タグ

template.html 
<!-- パラメータなしのURL -->
<a href="{% url 'blog:post_list' %}">記事一覧</a>

<!-- パラメータ付きのURL -->
<a href="{% url 'blog:post_detail' pk=post.id %}">記事詳細</a>

<!-- 複数パラメータ -->
<a href="{% url 'blog:post_edit' pk=post.id slug=post.slug %}">編集</a>

Pythonコードでのreverse()関数

views.py 
from django.shortcuts import redirect
from django.urls import reverse

def create_post(request):
    # 記事作成処理...
    post = Post.objects.create(title='新記事')

    # URLを名前から生成してリダイレクト
    url = reverse('blog:post_detail', kwargs={'pk': post.pk})
    return redirect(url)

    # またはショートカット
    return redirect('blog:post_detail', pk=post.pk)

ハードコーディング vs リバースURL

悪い例（ハードコーディング）

<a href="/post/{{ post.id }}/">

URLが変わると全箇所修正が必要

良い例（リバースURL）

{% url 'blog:post_detail' pk=post.id %}

URL名が同じなら修正不要

3.5 include()によるURL設定の分割

大規模なプロジェクトでは、アプリケーションごとにURL設定を分割して管理します。include()関数を使うことで、各アプリのurls.pyを読み込めます。

プロジェクトのurls.py

config/urls.py（プロジェクト全体） 
from django.contrib import admin
from django.urls import path, include

urlpatterns = [
    # 管理画面
    path('admin/', admin.site.urls),

    # ブログアプリのURL
    path('blog/', include('blog.urls')),

    # ショップアプリのURL
    path('shop/', include('shop.urls')),

    # ユーザー認証のURL
    path('accounts/', include('accounts.urls')),
]

各アプリのurls.py

blog/urls.py（アプリ固有） 
from django.urls import path
from . import views

app_name = 'blog'

urlpatterns = [
    path('', views.post_list, name='post_list'),
    path('post/<int:pk>/', views.post_detail, name='post_detail'),
    path('category/<slug:slug>/', views.category_posts, name='category'),
]

プレフィックスの自動付与

path('blog/', include('blog.urls'))と設定すると、blog/urls.py内のすべてのパターンに自動的にblog/が前置されます。

実習 3-1: 基本的なURLルーティングを設定する

簡単なブログアプリのURLルーティングを実装し、パラメータの受け渡しを確認しましょう。

手順

  1. 1 既存のDjangoプロジェクトにblogアプリを作成
  2. 2 blog/views.pyに、post_list、post_detail、category_postsの3つのビュー関数を作成（まずはダミーのHTTPResponseを返すだけでOK）
  3. 3 blog/urls.pyを作成し、3つのURLパターンを定義
  4. 4 プロジェクトのurls.pyでinclude()を使ってblog.urlsを読み込む
  5. 5 ブラウザで各URLにアクセスし、パラメータが正しく渡っているか確認

トラブルシューティング

404エラー「Page not found」が表示される

原因: URLパターンがリクエストURLにマッチしていない

解決方法:

  • URLパターンの末尾スラッシュ（/）の有無を確認
  • include()でアプリURLを読み込んでいるか確認
  • DEBUG=Trueの場合、エラーページに定義済みURLの一覧が表示されるので確認
NoReverseMatch エラーが発生する

原因: 指定したURL名が見つからない、またはパラメータが不足している

解決方法:

  • URL名のスペルミスを確認
  • app_name名前空間を正しく指定しているか確認
  • 必要なパラメータをすべて渡しているか確認
ビュー関数にパラメータが渡ってこない

原因: URLパターンの変数名とビュー関数の引数名が一致していない

解決方法:

  • URLパターンの<変数名>とビュー関数の引数名を一致させる
  • パスコンバータの型（int、str等）が適切か確認

理解度確認クイズ

  1. Q1. URLパターン path('post/<int:pk>/', views.post_detail) において、URLが /post/42/ だった場合、ビュー関数に渡されるpkの値と型は何ですか？

    回答を表示

    値は42、型はint（整数型）です。パスコンバータintが自動的に文字列を整数に変換してビュー関数に渡します。

  2. Q2. テンプレートで名前付きURLを使う場合と、ハードコーディングでURLを書く場合の違いは何ですか？

    回答を表示

    名前付きURL（{% url 'post_detail' pk=1 %}）を使うと、URLパスを変更してもテンプレートの修正が不要です。ハードコーディング（/post/1/）の場合、URLを変更すると全ての箇所を修正する必要があります。これはDRY原則に基づく設計です。

  3. Q3. include()関数を使う主な目的は何ですか？

    回答を表示

    アプリケーションごとにURL設定を分割して管理し、保守性を高めることが目的です。各アプリが独自のurls.pyを持つことで、プロジェクト全体のURL設定がシンプルになり、アプリの再利用性も向上します。

第3章のまとめ

  • URLルーティングはURLとビュー関数を紐付ける仕組みで、urls.pyのpath()関数で定義します。
  • パスコンバータ（int、str、slug、uuid等）により、URLから型付きのパラメータを抽出できます。
  • 名前付きURLとリバースURL解決により、URLの変更に強い保守性の高いコードを書けます。
  • include()を使ってURL設定を分割することで、アプリケーションの独立性と再利用性が向上します。
