第7章

FastAPIで簡単なWeb APIを作る

SQLModelで管理したデータをWeb API経由で提供する方法を学びます。ブラウザから確認できる簡単なAPIを作成し、データベースとWebの連携を体験します。

この章で学ぶこと

  • Web APIとは何か、REST APIの基本概念を理解する
  • FastAPIの特徴とSQLModelとの親和性を理解する
  • FastAPIとSQLModelを統合してCRUD APIを実装できるようになる
  • インタラクティブドキュメント（Swagger UI）を活用してAPIをテストできるようになる

7.1 Web APIとは何か

Web API（Application Programming Interface）は、Webを通じてプログラム同士がデータをやり取りするための仕組みです。 スマートフォンアプリやWebサイトが、サーバーと通信してデータを取得したり更新したりする際に使われます。

sequenceDiagram autonumber participant U as スマホアプリ participant A as Web API participant D as データベース U->>A: ユーザー一覧を取得リクエスト A->>D: SELECT * FROM users D-->>A: ユーザーデータ A-->>U: JSONで返却

REST APIの基本

REST APIは、Web APIの代表的な設計スタイルです。 HTTPメソッド（GET、POST、PUT、DELETE）を使って、データの取得・作成・更新・削除を行います。

HTTPメソッド 操作 例
GET データの取得 GET /users → ユーザー一覧を取得
POST データの作成 POST /users → 新しいユーザーを作成
PUT データの更新 PUT /users/1 → ID=1のユーザーを更新
DELETE データの削除 DELETE /users/1 → ID=1のユーザーを削除

エンドポイントとは

エンドポイントは、APIにアクセスするためのURLのことです。 例えば「/users」「/products」などが該当します。各エンドポイントが特定の機能を提供します。

7.2 FastAPIとは

FastAPIは、Pythonで高速なWeb APIを作成するための最新のWebフレームワークです。 SQLModelと同じ作者（Sebastián Ramírez氏）が開発しており、SQLModelとの相性が非常に良いのが特徴です。

FastAPIの主な特徴

高速

非同期処理（async/await）に対応しており、NodeJSやGoに匹敵する高速性を持ちます。

型ヒント活用

Pythonの型ヒントを活用して、コードの自動補完やバリデーションを実現します。

自動ドキュメント生成

Swagger UIとReDocによるインタラクティブなAPIドキュメントを自動生成します。

SQLModelとの統合

SQLModelのモデル定義をそのままAPIの入出力スキーマとして使用できます。

FastAPI + SQLModelの強み

両方とも型ヒントを活用した設計なので、1つのモデル定義でデータベース構造とAPI仕様の両方を定義できます。 コードの重複を減らし、保守性を高められます。

7.3 FastAPIのインストールと基本設定

FastAPIを使い始めるには、FastAPIと、サーバーを起動するためのUvicornをインストールします。

インストール

ターミナル 
# FastAPIとUvicornをインストール
pip install fastapi uvicorn[standard]

最小限のFastAPIアプリケーション

まずは、シンプルなAPIを作成してFastAPIの動作を確認しましょう。

main.py 
from fastapi import FastAPI

# FastAPIアプリケーションを作成
app = FastAPI()

# ルートエンドポイント
@app.get("/")
def read_root():
    return {"message": "Hello, FastAPI!"}

# ユーザー情報を返すエンドポイント
@app.get("/users/{user_id}")
def read_user(user_id: int):
    return {"user_id": user_id, "name": "田中太郎"}

サーバーの起動

ターミナル 
# サーバーを起動（リロード機能付き）
uvicorn main:app --reload

動作確認

サーバー起動後、ブラウザで以下のURLにアクセスしてみましょう。

  • API: http://localhost:8000/
  • Swagger UI: http://localhost:8000/docs
  • ReDoc: http://localhost:8000/redoc

7.4 SQLModelとFastAPIの統合

SQLModelのモデル定義を、FastAPIのAPIエンドポイントで活用する方法を学びます。 同じモデル定義がデータベース操作とAPI入出力の両方に使えるのが大きなメリットです。

ユーザーモデルの定義

models.py 
from sqlmodel import SQLModel, Field
from typing import Optional

class User(SQLModel, table=True):
    id: Optional[int] = Field(default=None, primary_key=True)
    name: str = Field(max_length=50)
    email: str = Field(max_length=100)
    age: int

FastAPIアプリケーション

main.py 
from fastapi import FastAPI, Depends
from sqlmodel import Session, create_engine, select
from models import User

# データベース接続設定
DATABASE_URL = "sqlite:///./users.db"
engine = create_engine(DATABASE_URL, echo=True)

app = FastAPI()

# セッション依存性
def get_session():
    with Session(engine) as session:
        yield session

Depends（依存性注入）とは

FastAPIのDependsは、関数に必要なリソース（データベースセッションなど）を自動的に提供してくれる仕組みです。 エンドポイント関数内で毎回セッションを作成する手間が省けます。

7.5 CRUDエンドポイントの実装

SQLModelとFastAPIを組み合わせて、Create（作成）、Read（読み取り）、Update（更新）、Delete（削除）の基本的なAPI操作を実装します。

1. ユーザー作成（POST）

main.py 
@app.post("/users/", response_model=User)
def create_user(user: User, session: Session = Depends(get_session)):
    # データベースに保存
    session.add(user)
    session.commit()
    session.refresh(user)
    return user

2. ユーザー一覧取得（GET）

main.py 
@app.get("/users/", response_model=list[User])
def read_users(session: Session = Depends(get_session)):
    # 全ユーザーを取得
    users = session.exec(select(User)).all()
    return users

3. 特定ユーザー取得（GET）

main.py 
from fastapi import HTTPException

@app.get("/users/{user_id}", response_model=User)
def read_user(user_id: int, session: Session = Depends(get_session)):
    user = session.get(User, user_id)
    if not user:
        raise HTTPException(status_code=404, detail="ユーザーが見つかりません")
    return user

4. ユーザー更新（PUT）

main.py 
@app.put("/users/{user_id}", response_model=User)
def update_user(user_id: int, updated: User, session: Session = Depends(get_session)):
    user = session.get(User, user_id)
    if not user:
        raise HTTPException(status_code=404, detail="ユーザーが見つかりません")
    user.name = updated.name
    user.email = updated.email
    user.age = updated.age
    session.commit()
    return user

5. ユーザー削除（DELETE）

main.py 
@app.delete("/users/{user_id}")
def delete_user(user_id: int, session: Session = Depends(get_session)):
    user = session.get(User, user_id)
    if not user:
        raise HTTPException(status_code=404, detail="ユーザーが見つかりません")
    session.delete(user)
    session.commit()
    return {"message": "ユーザーを削除しました"}

7.6 インタラクティブドキュメント（Swagger UI）の活用

FastAPIは、Swagger UIという対話型のAPIドキュメントを自動生成します。 ブラウザ上で直接APIを試せるので、開発中のテストやデバッグに非常に便利です。

Swagger UIへのアクセス

サーバー起動後、http://localhost:8000/docsにアクセスすると、 すべてのエンドポイントが一覧表示され、その場でテストできます。

Swagger UIでできること

  • 各エンドポイントの機能説明を確認
  • リクエストパラメータとレスポンス形式を確認
  • 「Try it out」ボタンで実際にAPIリクエストを送信してテスト
  • レスポンス結果をその場で確認

開発効率の向上

Swagger UIを使えば、curlコマンドやPostmanなどのツールを使わなくても、 ブラウザだけでAPIのテストができます。開発とテストのサイクルが格段に速くなります。

実習 7-1: TODO API作成

これまでに学んだ知識を活用して、簡単なTODO管理APIを作成しましょう。

タスクモデル

class Task(SQLModel, table=True):
    id: Optional[int] = Field(default=None, primary_key=True)
    title: str = Field(max_length=100)
    completed: bool = Field(default=False)

実装するエンドポイント

  1. 1 POST /tasks/ - 新しいタスクを作成
  2. 2 GET /tasks/ - タスク一覧を取得
  3. 3 PUT /tasks/{task_id} - タスクを完了済みに更新
  4. 4 Swagger UIで各エンドポイントをテスト

理解度確認クイズ

  1. Q1. REST APIにおいて、データの取得、作成、更新、削除にそれぞれどのHTTPメソッドを使いますか？

    回答を表示

    取得: GET、作成: POST、更新: PUT、削除: DELETEです。 これらはHTTPの標準メソッドで、REST APIの基本的な操作を表します。

  2. Q2. FastAPIとSQLModelを組み合わせる最大のメリットは何ですか？

    回答を表示

    1つのモデル定義でデータベース構造とAPI仕様の両方を定義できることです。 型ヒントを活用した設計により、コードの重複を減らし、保守性を高められます。

  3. Q3. FastAPIのDepends（依存性注入）は何のために使いますか？

    回答を表示

    エンドポイント関数に必要なリソース（データベースセッションなど）を自動的に提供するために使います。 毎回セッションを手動で作成する手間が省け、コードがシンプルになります。

  4. Q4. Swagger UIでできることを2つ挙げてください。

    回答を表示

    1. 各エンドポイントの機能説明やパラメータ形式を確認できます。
    2. 「Try it out」ボタンで実際にAPIリクエストを送信してテストできます。
    他にも、レスポンス結果の確認や、APIドキュメントの自動生成などができます。
