パフォーマンスの基礎知識

データ量が増えた時の対処法、効率的なクエリの書き方、インデックスの基本概念を学びます。よくあるパフォーマンス問題と解決方法を実例を通じて理解します。

この章で学ぶこと

  • データベースのパフォーマンスが遅くなる主な原因を理解する
  • インデックスの基本概念と、適切な使い方を学ぶ
  • N+1問題とその解決方法を理解する
  • 効率的なクエリの書き方とベストプラクティスを習得する

10.1 データベースのパフォーマンスとは

アプリケーションの開発中は少量のデータで動作確認を行うため、パフォーマンス問題に気づきにくいことがあります。 しかし、実際の運用環境では数千、数万件のデータを扱うため、パフォーマンスを考慮した設計が重要になります。

パフォーマンスの指標

データベースのパフォーマンスは主に「クエリの実行時間」と「データベースへのアクセス回数」で評価されます。 レスポンスタイムが0.1秒以内であれば、ユーザーにとって快適な体験になると言われています。

よくあるパフォーマンス問題

N+1問題

リレーションを持つデータを取得する際、メインのクエリ1回+関連データごとにN回のクエリが発行され、大量のアクセスが発生する問題です。

全件スキャン

インデックスがない列で検索を行うと、全てのレコードを1件ずつチェックする必要があり、データ量が増えると著しく遅くなります。

過剰なクエリ実行

ループ内でクエリを実行すると、同じデータを何度も取得することになり、パフォーマンスが低下します。

大量データの一括取得

必要以上に多くのデータを一度に取得すると、メモリ使用量が増え、アプリケーション全体が遅くなります。

10.2 インデックスとは何か

インデックスは、データベースの検索を高速化するための仕組みです。 本の「索引」と同じで、特定のデータを素早く見つけるための目印となります。

高速検索] B -- なし --> D[全件スキャン

インデックスの仕組み

例えば、10,000人のユーザーの中から「email」で特定のユーザーを検索する場合を考えてみましょう。

検索方法 処理内容 処理時間の目安
インデックスなし 10,000件すべてを順番にチェック 約10秒
インデックスあり 索引から目的のデータへ直接アクセス 約0.01秒

インデックスを作成すべき列

WHERE句でよく使う列：ユーザーID、メールアドレスなど検索条件として頻繁に使われる列
外部キー：他のテーブルとの関連付けに使用される列
ORDER BYで使う列：並び替えに使用される列

SQLModelでインデックスを作成する

models.py 
from sqlmodel import SQLModel, Field

class User(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    email: str = Field(index=True)  # インデックスを作成
    name: str
    age: int = Field(index=True)  # 年齢でもよく検索するのでインデックス作成

インデックスの注意点

インデックスは検索を高速化しますが、データ追加・更新時にインデックスも更新する必要があるため、書き込み処理が遅くなります。 すべての列にインデックスを作成すると逆効果になることがあります。

10.3 N+1問題とその解決方法

N+1問題は、ORMを使う際に最も頻繁に遭遇するパフォーマンス問題です。 関連データを取得する際、気づかないうちに大量のクエリが発行されてしまいます。

N+1問題が起きる例

悪い例（N+1問題あり） 
# ユーザー一覧を取得（1回目のクエリ）
users = session.exec(select(User)).all()

# 各ユーザーの投稿を取得（ユーザー数分のクエリが発行される！）
for user in users:
    posts = session.exec(
        select(Post).where(Post.user_id == user.id)
    ).all()
    print(f"{user.name}: {len(posts)}件の投稿")

何が問題なのか？

100人のユーザーがいる場合、最初のクエリで1回、各ユーザーの投稿取得で100回の合計101回のクエリが発行されます。 これが「1 + N = N+1」の由来です。

sequenceDiagram autonumber participant App as アプリケーション participant DB as データベース App->>DB: SELECT * FROM users (1回目) DB-->>App: 100件のユーザー loop 100回繰り返し App->>DB: SELECT * FROM posts WHERE user_id = X DB-->>App: 該当する投稿 end Note over App,DB: 合計101回のクエリ！

解決方法：Joinを使う

良い例（Joinで一度に取得） 
from sqlmodel import select

# Joinを使って一度のクエリで取得
statement = (
    select(User, Post)
    .join(Post)
    .where(User.id == Post.user_id)
)

results = session.exec(statement).all()

# グループ化して表示
user_posts = {}
for user, post in results:
    if user.id not in user_posts:
        user_posts[user.id] = {"name": user.name, "posts": []}
    user_posts[user.id]["posts"].append(post)

for data in user_posts.values():
    print(f"{data['name']}: {len(data['posts'])}件の投稿")

改善効果

クエリの実行回数が101回から1回に削減されました。 データ量が増えるほど、この違いは大きくなります。

10.4 効率的なクエリの書き方

1. 必要な列だけを取得する

悪い例 
# すべての列を取得（大きなテキストや画像データも含む）
users = session.exec(select(User)).all()
良い例 
# 必要な列だけを指定して取得
statement = select(User.id, User.name, User.email)
users = session.exec(statement).all()

2. ページネーションを使う

大量のデータを一度に取得せず、必要な分だけを取得します。

ページネーション実装 
page = 1  # 現在のページ
per_page = 20  # 1ページあたりの件数

statement = (
    select(User)
    .offset((page - 1) * per_page)  # スキップする件数
    .limit(per_page)  # 取得する件数
)

users = session.exec(statement).all()
print(f"ページ{page}: {len(users)}件のユーザー")

3. 集計はデータベース側で行う

悪い例（Pythonで集計） 
# 全データを取得してからPythonで集計
users = session.exec(select(User)).all()
adult_count = len([u for u in users if u.age >= 20])
良い例（データベースで集計） 
from sqlmodel import func

# データベース側で集計（高速）
statement = select(func.count(User.id)).where(User.age >= 20)
adult_count = session.exec(statement).one()
print(f"成人ユーザー数: {adult_count}件")

10.5 パフォーマンスの測定方法

パフォーマンス改善を行う際は、まず現状を測定し、改善後の効果を確認することが重要です。

実行時間の測定

performance_test.py 
import time
from sqlmodel import select

# 測定開始
start_time = time.time()

# 処理を実行
users = session.exec(select(User).limit(100)).all()

# 測定終了
end_time = time.time()
elapsed_time = end_time - start_time

print(f"実行時間: {elapsed_time:.4f}秒")

SQLログの確認

SQLログを有効化 
from sqlmodel import create_engine

    "sqlite:///database.db",
    echo=True  # 実行されるSQLをコンソールに出力

開発中のヒント

開発中はecho=Trueを有効にして、どんなSQLが実行されているか確認しましょう。 N+1問題やその他のパフォーマンス問題を早期に発見できます。

実習 10-1: パフォーマンス改善を体験する

実際にN+1問題を体験し、改善前後のパフォーマンスを比較してみましょう。

手順

  1. 1 モデルを定義（User、Post）し、リレーションを設定します
  2. 2 100人のユーザーと、各ユーザーごとに10件の投稿を作成します
  3. 3 N+1問題が発生するコードを書き、実行時間を測定します
  4. 4 Joinを使った改善版を作成し、実行時間を測定します
  5. 5 改善前後の実行時間の差を確認します

トラブルシューティング

インデックスを作成したのに速くならない

原因: WHERE句で関数を使うとインデックスが利用されません。

悪い例: WHERE LOWER(email) = 'test@example.com'

良い例: WHERE email = 'test@example.com'

データ取得は速いが表示が遅い

原因: 大量のデータを一度にメモリに読み込んでいる可能性があります。

解決方法: ページネーションを使って少しずつ取得しましょう。

Joinを使うと重複したデータが返ってくる

原因: 1対多の関連で、多側のレコード分だけ主側が重複します。

解決方法: Pythonのコードでグループ化するか、distinct()を使用してください。

理解度確認クイズ

  1. Q1. インデックスの主な目的は何ですか？

    回答を表示

    データベースの検索を高速化することです。 インデックスは本の索引のように、特定のデータを素早く見つけるための仕組みで、 全件スキャンを避けて目的のレコードに直接アクセスできるようにします。

  2. Q2. N+1問題とは何ですか？

    回答を表示

    関連データを取得する際に、メインのクエリ1回+関連データごとにN回のクエリが発行されてしまう問題です。 例えば、100人のユーザーとその投稿を取得する際、ユーザー取得で1回、各ユーザーの投稿取得で100回、 合計101回のクエリが実行されてしまいます。

  3. Q3. インデックスを作成する際の注意点は何ですか？

    回答を表示

    書き込み処理が遅くなるという点です。 インデックスは検索を高速化しますが、データの追加・更新時にインデックスも更新する必要があるため、 すべての列にインデックスを作成すると逆効果になることがあります。

  4. Q4. N+1問題を解決するにはどうすればよいですか？

    回答を表示

    Joinを使って一度のクエリで関連データを取得することで解決できます。 これにより、クエリの実行回数を大幅に削減でき、パフォーマンスが向上します。

この章のまとめ

  • インデックスは検索を高速化する仕組みで、検索条件によく使う列に作成すると効果的です
  • N+1問題は関連データ取得時によく発生し、Joinを使って解決できます
  • 必要な列だけを取得し、ページネーションを使うことでパフォーマンスが向上します
  • 集計はデータベース側で行う方が効率的です
  • 開発中はecho=TrueでSQLログを確認し、パフォーマンス問題を早期に発見しましょう
