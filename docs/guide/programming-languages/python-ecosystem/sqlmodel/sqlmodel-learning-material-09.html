データの妥当性をチェックする方法、エラーが発生した時の適切な処理方法を学びます。メールアドレスの形式チェック、必須項目の確認など、実用的なバリデーションを実装します。

9.1 バリデーションの必要性

バリデーション（validation）とは、データが正しい形式や条件を満たしているかをチェックすることです。 アプリケーションにデータを保存する前に、そのデータが適切かどうかを確認する重要な処理です。

なぜバリデーションが必要なのか

ユーザーが入力したデータをそのまま信用してしまうと、様々な問題が発生します。

データの不整合

メールアドレスに「abc」だけ、年齢にマイナスの値など、意味のないデータが保存されてしまいます。

セキュリティリスク

悪意のあるデータ（SQLインジェクションなど）を防ぐことができず、システムが攻撃される可能性があります。

アプリケーションエラー

不正なデータが原因でプログラムがクラッシュしたり、予期しない動作をしたりします。

ユーザー体験の悪化

エラーメッセージが分かりにくかったり、保存後にエラーになったりすると、ユーザーが混乱します。

9.2 SQLModelの基本的なバリデーション

SQLModelは、Pydanticをベースにしているため、強力なバリデーション機能を持っています。 まずは、型ヒントによる基本的なバリデーションを見ていきましょう。

型による自動バリデーション

# 基本的な型ヒントによるバリデーション
from sqlmodel import Field, SQLModel

class User(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    name: str  # 文字列型を強制
    age: int  # 整数型を強制
    email: str

# 正しい例
user1 = User(name="田中太郎", age=25, email="tanaka@example.com")

# エラー例: 年齢に文字列を渡そうとする
# user2 = User(name="佐藤花子", age="twenty", email="sato@example.com")
# → ValidationError: age は整数でなければなりません

Pythonの型ヒント

str、int、float、boolなどの型を指定すると、 SQLModelが自動的にその型に変換できるかチェックしてくれます。

Fieldによる詳細なバリデーション

Field()を使うと、より細かい制約を設定できます。 

# Fieldを使った詳細なバリデーション
from sqlmodel import Field, SQLModel

class Product(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    # 必須項目（Noneを許可しない）
    name: str = Field(min_length=1, max_length=100)
    # 最小値・最大値の制約
    price: int = Field(gt=0, le=1000000)  # 0より大きく、100万以下
    # 在庫数は0以上
    stock: int = Field(ge=0)
    # オプション項目（Noneを許可）
    description: str | None = Field(default=None, max_length=500)

# 正しい例
product = Product(
    name="ノートPC",
    price=89800,
    stock=5,
    description="高性能ノートパソコン"
)
バリデーション 説明 例
min_length 文字列の最小長 min_length=1
max_length 文字列の最大長 max_length=100
gt より大きい（greater than） gt=0
ge 以上（greater or equal） ge=0
lt より小さい（less than） lt=100
le 以下（less or equal） le=100

9.3 実用的なバリデーション例

実際のアプリケーションでよく使われるバリデーションパターンを見ていきましょう。

メールアドレスのバリデーション

# メールアドレスの形式チェック
from pydantic import EmailStr, field_validator
from sqlmodel import Field, SQLModel

class User(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    name: str = Field(min_length=1, max_length=50)
    # EmailStr型で自動的にメール形式をチェック
    email: EmailStr
    age: int = Field(ge=0, le=150)

# 正しい例
user1 = User(name="田中太郎", email="tanaka@example.com", age=25)

# エラー例: 不正なメールアドレス
# user2 = User(name="佐藤花子", email="invalid-email", age=30)
# → ValidationError: emailの形式が正しくありません

EmailStr使用時の注意

EmailStr型を使うには、email-validatorパッケージが必要です。
インストールコマンド: pip install email-validator

カスタムバリデータの作成

独自のバリデーションルールを作成することもできます。 

# カスタムバリデータの例
from pydantic import field_validator
from sqlmodel import Field, SQLModel

class User(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    username: str = Field(min_length=3, max_length=20)
    password: str

    # パスワードの強度をチェックするバリデータ
    @field_validator('password')
    @classmethod
    def validate_password(cls, v: str) -> str:
        if len(v) < 8:
            raise ValueError('パスワードは8文字以上である必要があります')
        if not any(c.isupper() for c in v):
            raise ValueError('パスワードには大文字を含める必要があります')
        if not any(c.isdigit() for c in v):
            raise ValueError('パスワードには数字を含める必要があります')
        return v

# 正しい例
user = User(username="tanaka", password="SecurePass123")

# エラー例
# user = User(username="sato", password="weak")
# → ValueError: パスワードは8文字以上である必要があります

複数フィールドを組み合わせたバリデーション

# 複数のフィールドを使ったバリデーション
from datetime import date
from pydantic import model_validator
from sqlmodel import Field, SQLModel

class Event(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    name: str = Field(min_length=1, max_length=100)
    start_date: date
    end_date: date

    # 終了日が開始日より後であることをチェック
    @model_validator(mode='after')
    def check_dates(self):
        if self.end_date < self.start_date:
            raise ValueError('終了日は開始日より後でなければなりません')
        return self

# 正しい例
event = Event(
    name="技術カンファレンス",
    start_date=date(2024, 6, 1),
    end_date=date(2024, 6, 3)
)

9.4 エラーハンドリングの基本

バリデーションエラーが発生した時に、適切にエラーを処理することが重要です。

基本的なエラー処理

# try-exceptを使ったエラーハンドリング
from pydantic import ValidationError
from sqlmodel import Field, SQLModel

class User(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    name: str = Field(min_length=1, max_length=50)
    age: int = Field(ge=0, le=150)

try:
    # 不正なデータで作成を試みる
    user = User(name="", age=-5)
except ValidationError as e:
    # エラーメッセージを表示
    print("データが不正です:")
    for error in e.errors():
        field = error['loc'][0]  # エラーが発生したフィールド名
        message = error['msg']  # エラーメッセージ
        print(f"  {field}: {message}")

データベース操作時のエラー処理

# データベース保存時のエラーハンドリング
from sqlmodel import Session, create_engine, SQLModel, Field
from sqlalchemy.exc import IntegrityError

class User(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    email: str = Field(unique=True)  # 重複不可
    name: str

engine = create_engine("sqlite:///database.db")
SQLModel.metadata.create_all(engine)

def create_user(email: str, name: str):
    try:
        with Session(engine) as session:
            user = User(email=email, name=name)
            session.add(user)
            session.commit()
            print(f"ユーザー作成成功: {name}")
    except IntegrityError:
        print(f"エラー: メールアドレス {email} は既に使用されています")

実習: 会員登録フォームのバリデーション

会員登録システムを想定して、適切なバリデーションを実装してみましょう。 

# 実習: 会員登録フォームのバリデーション
from datetime import date
from pydantic import EmailStr, field_validator
from sqlmodel import Field, SQLModel, Session, create_engine

class Member(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    username: str = Field(min_length=3, max_length=20, unique=True)
    email: EmailStr = Field(unique=True)
    password: str
    birth_date: date
    phone: str | None = Field(default=None, max_length=15)

    @field_validator('username')
    @classmethod
    def validate_username(cls, v: str) -> str:
        # 英数字とアンダースコアのみ許可
        if not v.replace('_', '').isalnum():
            raise ValueError('ユーザー名は英数字とアンダースコアのみ使用できます')
        return v

    @field_validator('password')
    @classmethod
    def validate_password(cls, v: str) -> str:
        if len(v) < 8:
            raise ValueError('パスワードは8文字以上必要です')
        return v

# データベース作成
engine = create_engine("sqlite:///members.db")
SQLModel.metadata.create_all(engine)

# 会員登録関数
def register_member(username, email, password, birth_date):
    try:
        member = Member(
            username=username,
            email=email,
            password=password,
            birth_date=birth_date
        )
        with Session(engine) as session:
            session.add(member)
            session.commit()
            print(f"✓ 会員登録成功: {username}")
    except Exception as e:
        print(f"✗ 登録失敗: {str(e)}")

9.5 バリデーションのベストプラクティス

1. 早期バリデーション

データベースに保存する前に、できるだけ早い段階でバリデーションを行いましょう。

2. 分かりやすいエラーメッセージ

ユーザーが何を修正すべきか明確に伝えるエラーメッセージを提供しましょう。

3. セキュリティを意識

特にユーザー入力は信用せず、必ず適切なバリデーションとサニタイゼーションを行いましょう。

4. テストを書く

バリデーションロジックが正しく動作するか、単体テストで確認しましょう。

重要ポイント

クライアント側とサーバー側の両方でバリデーションを実施することが理想的です。 クライアント側（JavaScript等）では即座にフィードバックを提供し、 サーバー側（SQLModel）ではセキュリティを確保します。

理解度確認クイズ

  1. Q1. バリデーションが必要な理由を3つ挙げてください。

    回答を表示

    1. データの整合性を保つ：意味のないデータが保存されるのを防ぎます。
    2. セキュリティリスクを防ぐ：悪意のあるデータからシステムを守ります。
    3. アプリケーションの安定性を確保：不正なデータによるエラーを防ぎます。

  2. Q2. Field(gt=0, le=100) という制約の意味を説明してください。

    回答を表示

    0より大きく、100以下の値のみを許可する制約です。
    gtは「greater than（より大きい）」、leは「less or equal（以下）」を意味します。 つまり、1〜100の範囲の値が有効です。

  3. Q3. カスタムバリデータを作る時に使うデコレータは何ですか？

    回答を表示

    @field_validatorデコレータを使用します。 単一のフィールドを検証する場合は@field_validator('field_name')、 複数のフィールドを組み合わせて検証する場合は@model_validatorを使います。

  4. Q4. ValidationErrorをキャッチする理由は何ですか？

    回答を表示

    ユーザーに分かりやすいエラーメッセージを提供するためです。 ValidationErrorをキャッチすることで、エラー内容を解析し、 どのフィールドがどのようなルールに違反したのかをユーザーに伝えることができます。

  • バリデーションは、データの整合性、セキュリティ、アプリケーションの安定性を確保するために不可欠です
  • SQLModelは型ヒントとField制約による強力な自動バリデーション機能を提供します
  • カスタムバリデータを使うことで、独自のビジネスルールを実装できます
  • 適切なエラーハンドリングにより、ユーザーに分かりやすいフィードバックを提供できます
