複数のテーブルを関連付ける（リレーション入門）

「ユーザーが投稿した記事」「商品とカテゴリー」のような関連性のあるデータの扱い方を学びます。外部キーの概念を図解を交えて理解します。

この章で学ぶこと

  • リレーションシップ（関連性）とは何かを理解する
  • 外部キーの概念と役割を理解する

  • 関連するデータの作成・取得方法を習得する

6.1 リレーションシップ（関連性）とは

現実世界のデータは、孤立して存在することは稀です。例えば、「ユーザーが投稿したブログ記事」や 「商品が属するカテゴリー」など、データとデータの間には関連性があります。

このようなデータ間の関連性を、データベースの世界ではリレーションシップと呼びます。 リレーションシップを正しく定義することで、データを効率的に管理し、矛盾のない状態を保つことができます。

リレーションシップの身近な例

• 顧客と注文（1人の顧客が複数の注文をする）
• 著者と書籍（1人の著者が複数の本を書く）
• 学生と履修科目（1人の学生が複数の科目を履修する）

リレーションシップの種類

主なリレーションシップには以下の3種類があります。この章では、最も基本的な1対多の関係に焦点を当てます。

種類 説明 例
1対1 1つのレコードが別のテーブルの1つのレコードと対応 ユーザーとプロフィール詳細
1対多 1つのレコードが複数のレコードと対応 ユーザーと投稿記事
多対多 複数のレコードが複数のレコードと対応 学生と授業

6.2 外部キー（Foreign Key）の基礎

テーブル間の関連性を実現するために使われるのが外部キーです。 外部キーとは、他のテーブルの主キー（Primary Key）を参照する列のことです。

例えば、「ユーザー」テーブルと「投稿」テーブルがあるとします。 投稿テーブルにuser_idという外部キーを持たせることで、 「この投稿はどのユーザーが書いたか」という関連性を表現できます。

erDiagram USER ||--o{ POST : "投稿する" USER { int id PK string name string email } POST { int id PK string title string content int user_id FK }

外部キーの役割

• データの整合性を保証：存在しないユーザーIDを参照できないようにする
• 関連データの取得を可能にする：ユーザーから投稿一覧を簡単に取得できる
• 削除時の動作を制御：ユーザーを削除したときに関連する投稿をどう扱うか定義できる

6.3 SQLModelで1対多の関係を定義する

それでは、実際にSQLModelを使って1対多の関係を定義してみましょう。 「ユーザー」と「投稿」の関係を例に、コードを見ていきます。

モデルの定義

models.py 
from typing import Optional, List
from sqlmodel import SQLModel, Field, Relationship

# ユーザーモデル
class User(SQLModel, table=True):
    id: Optional[int] = Field(default=None, primary_key=True)
    name: str
    email: str

    # リレーションシップ定義（このユーザーの投稿一覧）
    posts: List["Post"] = Relationship(back_populates="author")

# 投稿モデル
class Post(SQLModel, table=True):
    id: Optional[int] = Field(default=None, primary_key=True)
    title: str
    content: str

    # 外部キー（ユーザーIDを参照）
    user_id: int = Field(foreign_key="user.id")

    # リレーションシップ定義（この投稿の著者）
    author: Optional[User] = Relationship(back_populates="posts")

コードの解説

• Field(foreign_key="user.id")：user_idがUserテーブルのidを参照することを宣言
• Relationship(back_populates="posts")：双方向の関連を定義
• List["Post"]：1人のユーザーが複数の投稿を持つことを表現
• Optional[User]：1つの投稿は1人のユーザーに属することを表現

6.4 関連データの作成

リレーションシップを定義したら、実際にデータを作成してみましょう。 まずユーザーを作成し、そのユーザーに関連する投稿を作成します。

create_related_data.py 
from sqlmodel import Session, create_engine
from models import User, Post

# データベース接続
engine = create_engine("sqlite:///blog.db")

with Session(engine) as session:
    # 1. ユーザーを作成
    user = User(name="田中太郎", email="tanaka@example.com")
    session.add(user)
    session.commit()
    session.refresh(user)  # IDを取得

    # 2. ユーザーに関連する投稿を作成
    post1 = Post(
        title="SQLModelの基礎",
        content="SQLModelの使い方を学びました",
        user_id=user.id  # 外部キーにユーザーIDを設定
    )
    post2 = Post(
        title="リレーション入門",
        content="1対多の関係を理解しました",
        user_id=user.id
    )

    session.add(post1)
    session.add(post2)
    session.commit()

    print(f"ユーザー '{user.name}' と2件の投稿を作成しました")

6.5 関連データの取得

リレーションシップを定義すると、関連するデータを簡単に取得できるようになります。 ここでは、2つの方向から関連データを取得する方法を学びます。

パターン1：ユーザーから投稿一覧を取得

get_user_posts.py 
from sqlmodel import Session, select
from models import User

with Session(engine) as session:
    # ユーザーを取得
    user = session.exec(
        select(User).where(User.name == "田中太郎")
    ).first()

    # そのユーザーの投稿一覧を表示
    print(f"{user.name}さんの投稿:")
    for post in user.posts:
        print(f"  - {post.title}")

パターン2：投稿から著者情報を取得

get_post_author.py 
from sqlmodel import Session, select
from models import Post

with Session(engine) as session:
    # 投稿を取得
    post = session.exec(
        select(Post).where(Post.title == "SQLModelの基礎")
    ).first()

    # その投稿の著者情報を表示
    print(f"投稿: {post.title}")
    print(f"著者: {post.author.name}")
    print(f"メール: {post.author.email}")

ポイント

リレーションシップを定義すると、user.postsやpost.authorのように、 Pythonの属性として直感的に関連データにアクセスできます。

6.6 削除時の動作制御（CASCADE）

リレーションシップを持つデータを削除する際、関連するデータをどう扱うかを制御できます。 これをCASCADE（カスケード）と呼びます。

CASCADE設定 動作
CASCADE 親を削除すると、子も自動的に削除される
RESTRICT 子が存在する場合、親の削除を拒否する（デフォルト）
SET NULL 親を削除すると、子の外部キーをNULLに設定
cascade_example.py 
from sqlmodel import SQLModel, Field, Relationship
from typing import Optional, List

class User(SQLModel, table=True):
    id: Optional[int] = Field(default=None, primary_key=True)
    name: str

    # CASCADE設定：ユーザー削除時に投稿も削除
    posts: List["Post"] = Relationship(
        back_populates="author",
        cascade_delete=True  # これを追加
    )

class Post(SQLModel, table=True):
    id: Optional[int] = Field(default=None, primary_key=True)
    title: str
    user_id: int = Field(foreign_key="user.id")

    author: Optional[User] = Relationship(back_populates="posts")

注意

CASCADE削除は便利ですが、意図しないデータ損失を招く可能性があります。 本番環境では慎重に設定しましょう。

理解度確認クイズ

  1. Q1. 外部キーの主な役割を2つ挙げてください。

    回答を表示

    1. データの整合性を保証する：存在しないIDを参照できないようにする
    2. 関連データの取得を可能にする：テーブル間の関連性を明示的に定義できる

  2. Q2. SQLModelでリレーションシップを定義する際に使う関数は何ですか？

    回答を表示

    Relationship()関数を使います。back_populatesパラメータで双方向の関連を定義します。

  3. Q3. CASCADE削除とは何ですか？どのような場合に使うべきですか？

    回答を表示

    CASCADE削除は、親レコードを削除した際に、関連する子レコードも自動的に削除される機能です。 親と子が強く結びついており、親なしでは子が意味を持たない場合（例：ユーザーと投稿）に使います。 ただし、意図しないデータ損失を避けるため、慎重に使用すべきです。
