パラメータ化クエリ - SQLインジェクション対策の基本

プレースホルダ（%s）を使ったパラメータ化クエリの重要性と書き方を学びます。SQLインジェクション攻撃の危険性を理解し、安全なSQL実行方法を習得しましょう。

この章で学ぶこと

  • SQLインジェクション攻撃とは何か、その危険性を理解する
  • プレースホルダ（%s）を使ったパラメータ化クエリの書き方
  • 文字列連結による危険なSQL実行と安全な方法の違い

6.1 SQLインジェクション攻撃とは

SQLインジェクションとは、悪意のあるユーザーがSQL文を不正に操作して、意図しないデータベース操作を行う攻撃手法です。Webアプリケーションにおける最も深刻なセキュリティ脆弱性の一つとされています。

SQLインジェクションの危険性

  • データの流出: 全ユーザーの個人情報やパスワードが盗まれる
  • データの改ざん: データベース内のデータを不正に書き換えられる
  • データの削除: テーブル全体を削除される（DROP TABLE）
  • 認証回避: ログイン認証を回避して管理者権限を取得される

危険な例: 文字列連結による脆弱なコード

以下は絶対にやってはいけない、ユーザー入力を直接SQL文に埋め込むコード例です。

vulnerable.py 危険 
import psycopg2

# ユーザーから入力を受け取る
user_id = input("ユーザーIDを入力してください: ")

conn = psycopg2.connect(
    host="localhost",
    database="mydb",
    user="postgres",
    password="password"
)
cursor = conn.cursor()

# 危険! 文字列連結でSQL文を作成
# ユーザー入力を直接埋め込んでいる
query = "SELECT * FROM users WHERE user_id = " + user_id
cursor.execute(query)

result = cursor.fetchone()
print(result)

cursor.close()
conn.close()

攻撃シナリオ

攻撃者が以下のような入力を行った場合、どうなるでしょうか？

攻撃者の入力:

1 OR 1=1

この入力により、実際に実行されるSQL文は以下のようになります。

実行されるSQL:

SELECT * FROM users WHERE user_id = 1 OR 1=1

1=1は常に真（True）なので、このクエリは全てのユーザー情報を返してしまいます。攻撃者は意図しない全ユーザーのデータを取得できてしまうのです。

SQLインジェクション攻撃の流れ

sequenceDiagram participant 攻撃者 participant アプリケーション participant データベース 攻撃者->>アプリケーション: 悪意のある入力
全文字列"1 OR 1=1"全文字串 アプリケーション->>アプリケーション: 文字列連結でSQL作成
全文字列"SELECT * FROM users WHERE user_id = 1 OR 1=1"全文字串 アプリケーション->>データベース: 不正なSQL実行 データベース->>アプリケーション: 全ユーザーのデータを返す アプリケーション->>攻撃者: 全ユーザー情報が漏洩 Note over 攻撃者,データベース: WHERE条件が常に真になり、
全データが取得されてしまう

文字列連結によるSQL構築は、悪意のある入力を許してしまいます。

6.2 パラメータ化クエリとは

パラメータ化クエリ（プリペアドステートメント）とは、SQL文の中で変動する部分をプレースホルダで表し、値を別途渡す方法です。psycopg2では%sというプレースホルダを使います。

パラメータ化クエリの仕組み

  • SQL文の構造と値を分離して管理する
  • 値は自動的にエスケープされ、SQLとして解釈されない
  • データベース側でクエリが事前コンパイルされ、パフォーマンスも向上

安全な例: パラメータ化クエリ

以下が推奨される正しい書き方です。

safe.py 安全 
import psycopg2

# ユーザーから入力を受け取る
user_id = input("ユーザーIDを入力してください: ")

conn = psycopg2.connect(
    host="localhost",
    database="mydb",
    user="postgres",
    password="password"
)
cursor = conn.cursor()

# 安全! プレースホルダ %s を使用
# 値はタプルで別途渡す
query = "SELECT * FROM users WHERE user_id = %s"
cursor.execute(query, (user_id,))  # タプルで渡す（カンマ必須）

result = cursor.fetchone()
print(result)

cursor.close()
conn.close()

重要: プレースホルダは常に %s を使う

psycopg2では、データ型に関わらず常に %s を使います。Python標準の文字列フォーマット（%d, %fなど）とは異なりますので注意してください。

  • 整数でも %s
  • 文字列でも %s
  • 浮動小数でも %s

攻撃者が同じように 1 OR 1=1 を入力しても、パラメータ化クエリでは以下のように解釈されます。

実行されるSQL:

SELECT * FROM users WHERE user_id = '1 OR 1=1'

入力値全体が一つの文字列として扱われるため、user_idが「1 OR 1=1」という値のユーザーを検索する（該当なし）という動作になり、攻撃は失敗します。

6.3 複数のパラメータを渡す

複数の値を渡す場合は、プレースホルダを複数使い、対応する値をタプルで渡します。

multiple_params.py 
import psycopg2

conn = psycopg2.connect(
    host="localhost",
    database="mydb",
    user="postgres",
    password="password"
)
cursor = conn.cursor()

# 複数のプレースホルダを使用
query = """
SELECT * FROM products
WHERE category = %s AND price <= %s
ORDER BY price DESC
"""

# 対応する値をタプルで渡す
cursor.execute(query, ("電化製品", 50000))

results = cursor.fetchall()
for row in results:
    print(f"商品名: {row[1]}, 価格: {row[2]}円")

cursor.close()
conn.close()

INSERT文でのパラメータ化クエリ

データ挿入時も必ずパラメータ化クエリを使用します。

insert_param.py 
import psycopg2

conn = psycopg2.connect(
    host="localhost",
    database="mydb",
    user="postgres",
    password="password"
)
cursor = conn.cursor()

# ユーザーからの入力
name = input("商品名: ")
price = int(input("価格: "))
category = input("カテゴリ: ")

# パラメータ化クエリでINSERT
query = """
INSERT INTO products (name, price, category)
VALUES (%s, %s, %s)
"""
cursor.execute(query, (name, price, category))

conn.commit()  # 変更を確定
print(f"{cursor.rowcount}件のデータを追加しました")

cursor.close()
conn.close()

6.4 辞書型でのパラメータ指定

psycopg2では、名前付きプレースホルダを使って、辞書形式でパラメータを渡すこともできます。可読性が高く、パラメータが多い場合に便利です。

dict_params.py 
import psycopg2

conn = psycopg2.connect(
    host="localhost",
    database="mydb",
    user="postgres",
    password="password"
)
cursor = conn.cursor()

# 名前付きプレースホルダ %(name)s を使用
query = """
INSERT INTO employees (name, department, salary)
VALUES (%(name)s, %(dept)s, %(salary)s)
"""

# 辞書で値を渡す
params = {
    "name": "田中太郎",
    "dept": "営業部",
    "salary": 350000
}

cursor.execute(query, params)
conn.commit()

print(f"{cursor.rowcount}件のデータを追加しました")

cursor.close()
conn.close()

名前付きプレースホルダの利点

  • パラメータが多い場合でも順序を気にしなくて良い
  • コードの可読性が向上する
  • 同じパラメータを複数箇所で使いやすい

実習: パラメータ化クエリを使ってみよう

課題1: ユーザー検索プログラム

ユーザーから名前の一部を入力してもらい、LIKE句を使ってあいまい検索するプログラムを作成してください。パラメータ化クエリを使用すること。

課題2: 商品登録プログラム

ユーザーから商品名、価格、カテゴリを入力してもらい、productsテーブルに登録するプログラムを作成してください。辞書型パラメータを使用すること。

課題3: 危険なコードを修正する

文字列連結でSQL文を作成している既存コードを、パラメータ化クエリに書き換えてください。

6.5 注意点とベストプラクティス

テーブル名・カラム名にはプレースホルダを使えない

プレースホルダは値にのみ使用可能で、テーブル名やカラム名には使用できません。

❌ 間違い:

SELECT * FROM %s WHERE %s = %s

タプルで渡す際のカンマに注意

パラメータが1つの場合、タプルであることを示すため、末尾にカンマが必要です。

✅ 正しい:

cursor.execute(query, (value,))

❌ 間違い:

cursor.execute(query, (value))

クォートは自動で付与される

psycopg2が自動的に適切なエスケープとクォート処理を行うため、手動でクォートを追加する必要はありません。

✅ 正しい:

"SELECT * FROM users WHERE name = %s"

❌ 間違い:

"SELECT * FROM users WHERE name = '%s'"

よくあるエラーと対処法

TypeError: not all arguments converted during string formatting

原因: プレースホルダの数と渡した値の数が一致していない
対処: %sの数とタプル内の要素数を確認する

IndexError: tuple index out of range

原因: パラメータが1つの場合、タプルの末尾にカンマがない
対処: (value,) のようにカンマを付ける

第6章のまとめ

  • SQLインジェクションは、悪意のあるSQL文を注入してデータベースを不正操作する深刻な攻撃手法である
  • パラメータ化クエリ（プレースホルダ %s）を使うことで、SQLインジェクションを防止できる
  • 文字列連結による動的なSQL生成は絶対に行わないこと
  • 辞書型を使った名前付きプレースホルダも利用でき、可読性が向上する
  • パラメータ化クエリはセキュリティとパフォーマンス両方の観点から必須のテクニックである

  2. Q2. 以下のコードの問題点を指摘し、修正してください。

    username = input("ユーザー名: ")
    query = "SELECT * FROM users WHERE name = '" + username + "'"
    cursor.execute(query)
    回答を表示

    問題点: 文字列連結でSQL文を作成しており、SQLインジェクションの脆弱性がある

    username = input("ユーザー名: ")
