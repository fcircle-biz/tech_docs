接続プール - 効率的なコネクション管理

psycopg2.poolを使った接続プールの作成と活用方法を学習します。なぜ接続プールが必要なのか、SimpleConnectionPool・ThreadedConnectionPoolの使い分け、Webアプリケーションでの活用について理解しましょう。

この章で学ぶこと

  • 接続プールとは何か、なぜ必要なのか
  • SimpleConnectionPoolとThreadedConnectionPoolの違いと使い分け
  • Webアプリケーションでの接続プール活用方法

11.1 接続プールとは何か

接続プール（Connection Pool）とは、データベース接続を事前に作成して再利用する仕組みです。毎回新しい接続を作成・破棄するのではなく、プールから接続を借りて使い終わったら返却します。

データベース接続の確立には、ネットワーク通信や認証処理などのコストがかかります。特にWebアプリケーションのように頻繁にデータベースアクセスが発生する場合、毎回接続を作り直すのは非効率です。

通常接続 vs 接続プール

sequenceDiagram participant App as アプリケーション participant Pool as 接続プール participant DB as PostgreSQL Note over App,DB: 通常接続（毎回作成） App->>DB: 接続作成（遅い） App->>DB: SQL実行 App->>DB: 接続を閉じる App->>DB: また接続作成（遅い） App->>DB: SQL実行 App->>DB: 接続を閉じる Note over App,DB: 接続プール（再利用） App->>Pool: プールから接続を取得（速い） App->>DB: SQL実行 App->>Pool: プールに返却 App->>Pool: プールから接続を取得（速い） App->>DB: SQL実行 App->>Pool: プールに返却

接続プールを使うと、接続の再利用により大幅な高速化が実現できます。

接続プールのメリット

  • パフォーマンス向上（接続作成コストを削減）
  • 接続数の上限を制御可能（リソース管理）
  • スレッドセーフな接続管理

注意すべき点

  • 接続を借りたら必ず返却する必要がある
  • プールサイズの調整が必要（多すぎ・少なすぎに注意）
  • 長時間接続を保持すると他の処理がブロックされる

11.2 SimpleConnectionPoolの使い方

SimpleConnectionPoolは、psycopg2.poolモジュールが提供する基本的な接続プールです。シングルスレッドアプリケーションや、スレッドセーフ性が不要な場面で使用します。

simple_pool_basic.py 
import psycopg2
from psycopg2 import pool

# 接続プールを作成（最小1、最大5接続）
connection_pool = pool.SimpleConnectionPool(
    minconn=1,  # 最小接続数
    maxconn=5,  # 最大接続数
    host="localhost",
    database="mydb",
    user="postgres",
    password="password"
)

# プールから接続を取得
conn = connection_pool.getconn()

# 接続を使って処理を実行
cursor = conn.cursor()
cursor.execute("SELECT * FROM users;")
users = cursor.fetchall()
print(users)
cursor.close()

# 使い終わったらプールに返却
connection_pool.putconn(conn)

# プールを閉じる（アプリケーション終了時）
connection_pool.closeall()

SimpleConnectionPoolの主なメソッド

  • getconn() - プールから接続を取得
  • putconn(conn) - 接続をプールに返却
  • closeall() - プール内の全接続を閉じる

11.3 ThreadedConnectionPoolの使い方

ThreadedConnectionPoolは、マルチスレッド環境で使用する接続プールです。複数のスレッドが同時にプールから接続を取得しても安全に動作します。

Webアプリケーション（Flask、FastAPIなど）では、リクエストごとに異なるスレッドが処理を行うため、ThreadedConnectionPoolを使用するのが一般的です。

threaded_pool.py 
import psycopg2
from psycopg2 import pool
import threading

# スレッドセーフな接続プールを作成
connection_pool = pool.ThreadedConnectionPool(
    minconn=2,  # 最小接続数
    maxconn=10,  # 最大接続数
    host="localhost",
    database="mydb",
    user="postgres",
    password="password"
)

def fetch_users(thread_id):
    """スレッドごとに実行される処理"""
    conn = connection_pool.getconn()
    try:
        cursor = conn.cursor()
        cursor.execute("SELECT * FROM users;")
        users = cursor.fetchall()
        print(f"Thread {thread_id}: {len(users)} users")
        cursor.close()
    finally:
        connection_pool.putconn(conn)  # 必ず返却

# 5つのスレッドを起動
threads = []
for i in range(5):
    t = threading.Thread(target=fetch_users, args=(i,))
    threads.append(t)
    t.start()

# 全スレッドの終了を待つ
for t in threads:
    t.join()

connection_pool.closeall()

重要ポイント

try-finallyブロックを使って、例外が発生しても必ずputconn()が実行されるようにしましょう。接続を返却し忘れると、プールが枯渇して他の処理がブロックされてしまいます。

11.4 接続プールのプールサイズ設定

minconn（最小接続数）とmaxconn（最大接続数）の設定は、アプリケーションの特性に応じて調整する必要があります。

パラメータ 説明 推奨値の例
minconn 起動時に作成される接続数。常に確保される最低限の接続数。 1〜5（小規模アプリ）
maxconn プールが保持できる最大接続数。この数を超えると待機が発生する。 10〜50（中規模Webアプリ）

プールサイズ設定の注意点

  • 多すぎる接続数: データベースサーバーに負荷がかかり、メモリを圧迫
  • 少なすぎる接続数: 接続待ちが頻発し、アプリケーションが遅くなる
  • PostgreSQL側の設定も確認: postgresql.confのmax_connectionsより大きくしない

11.5 Webアプリケーションでの接続プール活用

Flask、FastAPIなどのWebフレームワークでは、アプリケーション起動時に接続プールを作成し、各リクエスト処理で接続を取得・返却するパターンが一般的です。

flask_with_pool.py 
from flask import Flask, jsonify
import psycopg2
from psycopg2 import pool

app = Flask(__name__)

# アプリ起動時に接続プールを作成
connection_pool = pool.ThreadedConnectionPool(
    minconn=2,
    maxconn=10,
    host="localhost",
    database="mydb",
    user="postgres",
    password="password"
)

@app.route('/users')
def get_users():
    conn = connection_pool.getconn()
    try:
        cursor = conn.cursor()
        cursor.execute("SELECT id, name FROM users;")
        users = cursor.fetchall()
        cursor.close()
        return jsonify(users)
    finally:
        connection_pool.putconn(conn)

if __name__ == '__main__':
    app.run(debug=True)

リクエストごとに接続を取得し、処理が終わったら即座に返却することで、効率的にデータベース接続を管理できます。

11.6 コンテキストマネージャーと接続プールの併用

接続プールをより安全に使うため、カスタムコンテキストマネージャーを実装すると便利です。

pool_context_manager.py 
from contextlib import contextmanager
import psycopg2
from psycopg2 import pool

connection_pool = pool.ThreadedConnectionPool(
    minconn=2, maxconn=10,
    host="localhost", database="mydb",
    user="postgres", password="password"
)

@contextmanager
def get_db_connection():
    """接続を取得・返却するコンテキストマネージャー"""
    conn = connection_pool.getconn()
    try:
        yield conn  # 接続を提供
    finally:
        connection_pool.putconn(conn)  # 確実に返却

# 使用例
with get_db_connection() as conn:
    cursor = conn.cursor()
    cursor.execute("SELECT * FROM users;")
    users = cursor.fetchall()
    cursor.close()
# withブロックを抜けると自動で返却される

コンテキストマネージャーのメリット

withステートメントを使うことで、例外が発生しても確実に接続が返却されます。コードも簡潔になり、返却忘れのリスクがなくなります。

実習: 接続プールを使ったデータ取得

ThreadedConnectionPoolを作成し、複数のクエリで接続を再利用してみましょう。

課題:

  1. ThreadedConnectionPoolを作成する（minconn=1, maxconn=3）
  2. 接続を取得してusersテーブルから全件取得
  3. 同じ接続を使ってproductsテーブルから全件取得
  4. 接続をプールに返却し、もう一度取得して別のクエリを実行
  5. 最後にプールを閉じる

第11章のまとめ

  • 接続プールは、データベース接続を再利用することでパフォーマンスを向上させる仕組み
  • SimpleConnectionPoolはシングルスレッド用、ThreadedConnectionPoolはマルチスレッド用
  • プールサイズ（minconn/maxconn）は、アプリケーションの特性に応じて適切に設定する
  • 接続を取得したら必ず返却すること（try-finallyやコンテキストマネージャーを活用）
  • WebアプリケーションではThreadedConnectionPoolを使い、リクエストごとに接続を取得・返却する

理解度確認クイズ

  1. Q1. 接続プールを使うことで得られる主なメリットは何ですか？

    回答を表示

    パフォーマンスの向上です。データベース接続の作成・破棄には時間がかかるため、接続を再利用することで大幅な高速化が実現できます。また、接続数の上限を制御することで、リソースの管理もしやすくなります。

  2. Q2. SimpleConnectionPoolとThreadedConnectionPoolの違いは何ですか？

    回答を表示

    SimpleConnectionPoolはシングルスレッド環境用で、スレッドセーフ性を保証しません。ThreadedConnectionPoolはマルチスレッド環境でも安全に使えるスレッドセーフなプールです。Webアプリケーションでは通常ThreadedConnectionPoolを使用します。

  3. Q3. 接続をプールから取得した後、必ず行うべき処理は何ですか？

    回答を表示

    putconn()で接続をプールに返却することが必須です。返却しないと、プールの接続が枯渇して他の処理がブロックされてしまいます。try-finallyブロックやコンテキストマネージャーを使って、例外が発生しても確実に返却されるようにしましょう。
