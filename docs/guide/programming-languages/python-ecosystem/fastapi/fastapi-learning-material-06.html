認証とセキュリティ

JWTトークン認証の実装、OAuth2の基本、パスワードハッシングなど、FastAPIでセキュアなAPIを構築するための認証・認可の仕組みとセキュリティベストプラクティスを学習します。

この章で学ぶこと

  • OAuth2とJWTトークンを使った認証の基本概念
  • パスワードのハッシュ化とセキュアな保存方法
  • FastAPIでの認証・認可システムの実装方法
  • セキュリティベストプラクティスと脆弱性対策

6.1 認証と認可の基本概念

セキュアなAPIを構築するには、認証（Authentication）と認可（Authorization）の違いを理解することが重要です。

認証と認可のフロー

flowchart TD A[クライアント] --> B{ログイン要求} B --> C[認証: ユーザー確認] C -->|成功| D[トークン発行] C -->|失敗| E[エラー: 401 Unauthorized] D --> F[保護されたリソースへのアクセス] F --> G{認可: 権限チェック} G -->|権限あり| H[リソース提供] G -->|権限なし| I[エラー: 403 Forbidden] style C fill:#dbeafe,stroke:#3b82f6,stroke-width:2px style G fill:#fef3c7,stroke:#f59e0b,stroke-width:2px

認証（Authentication）

「あなたは誰ですか？」という問い。ユーザーが本人であることを確認するプロセスです。

  • ユーザー名とパスワード
  • トークン（JWT、APIキー等）
  • 生体認証、二要素認証

認可（Authorization）

「何ができますか？」という問い。認証されたユーザーが特定のリソースにアクセスする権限があるかを確認するプロセスです。

  • ロールベースアクセス制御（RBAC）
  • 権限ベースアクセス制御
  • スコープとパーミッション

HTTPステータスコードの使い分け

401 Unauthorized: 認証が必要（ログインしていない）
403 Forbidden: 認証済みだが権限不足（アクセス拒否）

6.2 OAuth2の基本

OAuth2は、APIの認証・認可を行うための標準的なプロトコルです。FastAPIはOAuth2を組み込みでサポートしています。

OAuth2 Password Flowの流れ

sequenceDiagram autonumber participant C as クライアント participant A as 認証エンドポイント participant R as 保護リソース participant DB as データベース C->>A: POST /token
username & password A->>DB: ユーザー検証 DB-->>A: ユーザー情報 A-->>C: アクセストークン（JWT） C->>R: GET /protected
Authorization: Bearer {token} R->>R: トークン検証 R-->>C: 保護されたデータ

OAuth2の主要コンポーネント

リソースオーナー（Resource Owner）

保護されたリソースを所有するユーザー。ログインしてアクセストークンを取得します。

クライアント（Client）

ユーザーに代わってリソースにアクセスするアプリケーション（フロントエンド、モバイルアプリなど）。

認可サーバー（Authorization Server）

ユーザーを認証し、トークンを発行するサーバー。FastAPIで実装する部分です。

リソースサーバー（Resource Server）

保護されたリソースを提供するサーバー。トークンを検証してアクセスを許可します。

6.3 パスワードのハッシュ化

パスワードを平文で保存することは極めて危険です。必ずハッシュ化してデータベースに保存します。

絶対にやってはいけないこと

パスワードを平文でデータベースに保存することは、深刻なセキュリティリスクです。データベースが漏洩した場合、全ユーザーのパスワードが露呈します。

passlib を使ったハッシュ化

ターミナル 
# passlibとpython-joseのインストール
pip install "passlib[bcrypt]" python-jose[cryptography]
auth.py 
from passlib.context import CryptContext

# bcryptを使ったハッシュ化コンテキスト
pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

def hash_password(password: str) -> str:
    """パスワードをハッシュ化"""
    return pwd_context.hash(password)

def verify_password(plain_password: str, hashed_password: str) -> bool:
    """パスワードの検証"""
    return pwd_context.verify(plain_password, hashed_password)

# 使用例
hashed = hash_password("mysecretpassword")
print(hashed)  # $2b$12$...（ハッシュ値）

# 検証
is_valid = verify_password("mysecretpassword", hashed)
print(is_valid)  # True

bcryptの利点

bcryptはソルト（salt）を自動生成し、計算コストを調整できる強力なハッシュアルゴリズムです。同じパスワードでも毎回異なるハッシュ値が生成されるため、レインボーテーブル攻撃に強いです。

6.4 JWT認証の実装

JWT（JSON Web Token）は、クレーム（ユーザー情報など）を安全に伝送するためのトークン形式です。署名により改ざんを検出できます。

JWTトークンの生成と検証

jwt_auth.py 
from datetime import datetime, timedelta
from jose import JWTError, jwt
from typing import Optional

# 秘密鍵（本番環境では環境変数から取得）
SECRET_KEY = "your-secret-key-keep-it-secret"
ALGORITHM = "HS256"
ACCESS_TOKEN_EXPIRE_MINUTES = 30

def create_access_token(data: dict, expires_delta: Optional[timedelta] = None):
    """アクセストークンの生成"""
    to_encode = data.copy()
    if expires_delta:
        expire = datetime.utcnow() + expires_delta
    else:
        expire = datetime.utcnow() + timedelta(minutes=15)

    to_encode.update({"exp": expire})
    encoded_jwt = jwt.encode(to_encode, SECRET_KEY, algorithm=ALGORITHM)
    return encoded_jwt

秘密鍵の管理

SECRET_KEYは絶対にコードに直接書かないでください。環境変数や設定ファイル（.envなど）で管理し、本番環境では強力なランダム文字列を使用します。

ログインエンドポイントの実装

main.py 
from fastapi import FastAPI, Depends, HTTPException, status
from fastapi.security import OAuth2PasswordBearer, OAuth2PasswordRequestForm
from pydantic import BaseModel

app = FastAPI()

# OAuth2スキーム（トークン取得URLを指定）
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="token")

# ユーザーモデル（簡易版）
class User(BaseModel):
    username: str
    email: str

@app.post("/token")
async def login(form_data: OAuth2PasswordRequestForm = Depends()):
    """ログインしてトークンを取得"""
    # ユーザー検証（実際にはDBから取得）
    if form_data.username != "testuser" or form_data.password != "testpass":
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Incorrect username or password"
        )

    # トークン生成
    access_token = create_access_token(data={"sub": form_data.username})
    return {"access_token": access_token, "token_type": "bearer"}

6.5 認証が必要なエンドポイントの保護

依存関係（Depends）を使って、特定のエンドポイントに認証を要求できます。

main.py 
async def get_current_user(token: str = Depends(oauth2_scheme)):
    """トークンから現在のユーザーを取得"""
    credentials_exception = HTTPException(
        status_code=status.HTTP_401_UNAUTHORIZED,
        detail="Could not validate credentials",
        headers={"WWW-Authenticate": "Bearer"},
    )

    try:
        # トークンをデコード
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
        username: str = payload.get("sub")
        if username is None:
            raise credentials_exception
    except JWTError:
        raise credentials_exception

    # 実際にはDBからユーザーを取得
    return User(username=username, email=f"{username}@example.com")

@app.get("/users/me")
async def read_users_me(current_user: User = Depends(get_current_user)):
    """認証済みユーザー情報の取得"""
    return current_user

Dependsによる認証チェーン

Depends(get_current_user) を使うことで、エンドポイントにアクセスする前に自動的に認証チェックが実行されます。認証に失敗すると、エンドポイント関数は実行されず、401エラーが返されます。

6.6 ロールベースのアクセス制御（RBAC）

ユーザーにロール（役割）を付与し、ロールに基づいてアクセスを制御します。

rbac.py 
from enum import Enum
from fastapi import HTTPException, status

class Role(str, Enum):
    ADMIN = "admin"
    USER = "user"
    GUEST = "guest"

class UserWithRole(BaseModel):
    username: str
    role: Role

def require_role(required_role: Role):
    """指定されたロールを要求する依存関係"""
    async def role_checker(current_user: UserWithRole = Depends(get_current_user)):
        if current_user.role != required_role:
            raise HTTPException(
                status_code=status.HTTP_403_FORBIDDEN,
                detail="Insufficient permissions"
            )
        return current_user
    return role_checker

# 管理者のみアクセス可能なエンドポイント
@app.delete("/users/{user_id}")
async def delete_user(
    user_id: int,
    current_user: UserWithRole = Depends(require_role(Role.ADMIN))
):
    """ユーザー削除（管理者のみ）"""
    return {"message": f"User {user_id} deleted"}

6.7 セキュリティベストプラクティス

1. 強力な秘密鍵を使用

SECRET_KEYは最低32文字以上のランダムな文字列を使用し、環境変数で管理します。定期的にローテーションすることも推奨されます。

2. トークン有効期限を設定

JWTトークンには必ず有効期限（exp）を設定します。短い有効期限とリフレッシュトークンの組み合わせが理想的です。

3. HTTPS通信を強制

本番環境では必ずHTTPSを使用します。HTTP通信ではトークンが平文で送信されるため、中間者攻撃のリスクがあります。

4. 最小権限の原則

ユーザーには必要最小限の権限のみを付与します。デフォルトで権限を制限し、必要に応じて追加します。

よくあるセキュリティ脆弱性

  • パスワードの平文保存
  • 秘密鍵のハードコーディング
  • トークンに有効期限を設定しない
  • エラーメッセージに機密情報を含める
  • SQLインジェクション対策の欠如

理解度確認クイズ

  1. Q1. 認証（Authentication）と認可（Authorization）の違いを説明してください。また、それぞれに対応するHTTPステータスコードは何ですか？

    回答を表示

    認証: ユーザーが本人であることを確認するプロセス（「あなたは誰？」）。対応するステータスコードは 401 Unauthorized（認証が必要）。
    認可: 認証されたユーザーが特定のリソースにアクセスする権限があるかを確認するプロセス（「何ができる？」）。対応するステータスコードは 403 Forbidden（権限不足）。

  2. Q2. パスワードを平文で保存してはいけない理由と、bcryptを使うメリットを説明してください。

    回答を表示

    平文保存の危険性: データベースが漏洩した場合、全ユーザーのパスワードが露呈し、深刻なセキュリティ事故につながります。
    bcryptのメリット: ソルトを自動生成し、同じパスワードでも毎回異なるハッシュ値を生成します。計算コストを調整できるため、将来のハードウェア性能向上にも対応でき、レインボーテーブル攻撃やブルートフォース攻撃に強いです。

  3. Q3. JWTトークンに必ず含めるべき情報と、含めてはいけない情報は何ですか？

    回答を表示

    含めるべき情報: ユーザー識別子（sub）、発行時刻（iat）、有効期限（exp）、必要に応じてロールやスコープ。
    含めてはいけない情報: パスワード、クレジットカード番号、個人情報などの機密データ。JWTは署名されていますが暗号化されていないため、Base64デコードすれば内容を読み取れます。

  4. Q4. ロールベースアクセス制御（RBAC）において、「最小権限の原則」とは何を意味しますか？

    回答を表示

    ユーザーには業務遂行に必要な最小限の権限のみを付与するという原則です。デフォルトで権限を制限し、必要に応じて段階的に追加することで、セキュリティリスクを最小化します。例えば、一般ユーザーは閲覧のみ、管理者のみが削除権限を持つといった設計です。

第6章のまとめ

認証はユーザー確認、認可は権限チェックであり、それぞれHTTPステータスコード401と403で区別されます

パスワードは必ずbcryptなどでハッシュ化し、平文でデータベースに保存してはいけません

JWTトークンを使ったOAuth2認証により、ステートレスでスケーラブルな認証システムを構築できます

ロールベースアクセス制御により、ユーザーの役割に応じた細やかな権限管理が可能になります

秘密鍵の環境変数管理、HTTPS通信、トークン有効期限設定などのベストプラクティスを実践することが重要です
