リクエストとレスポンスモデル（Pydantic）

Pydanticモデルの定義と活用、型ヒントの効果的な使用、データバリデーション、レスポンスモデルの設計方法を学習します。

この章で学ぶこと

  • Pydanticの基本概念と型安全性の重要性
  • リクエストボディとレスポンスのためのモデル定義方法
  • データバリデーションの自動実行と検証ルールの設定
  • ネストしたモデルの作成と実践的な活用方法

3.1 Pydanticとは

Pydanticは、Pythonの型ヒントを使ってデータ検証とパースを行うライブラリです。FastAPIはPydanticと深く統合されており、リクエストとレスポンスのデータを安全かつ効率的に扱うことができます。

型安全性とは

型安全性とは、データの型が正しいことをコンパイル時や実行時に保証する仕組みです。Pydanticは実行時に型を検証し、誤ったデータを早期に検出します。

3.2 Pydanticモデルの基本

Pydanticモデルは、BaseModelを継承したクラスとして定義します。各フィールドには型ヒントを付けることで、自動的にバリデーションが行われます。

models.py 
# Pydanticからベースモデルをインポート
from pydantic import BaseModel

# ユーザー情報を表すモデル
class User(BaseModel):
    """
    ユーザーモデル
    """
    id: int              # 整数型のID
    name: str            # 文字列型の名前
    email: str           # メールアドレス
    is_active: bool      # アクティブ状態（True/False）

# モデルのインスタンス化
user = User(
    id=1,
    name="山田太郎",
    email="yamada@example.com",
    is_active=True
)

Pydanticの主な機能

自動型変換

文字列の"123"を整数123に自動変換するなど、可能な範囲で型変換を試みます。変換不可能な場合はエラーを返します。

データ検証

型だけでなく、最小値・最大値、文字列パターン、カスタムルールなど、詳細な検証ルールを設定できます。

シリアライズ

モデルのインスタンスを辞書やJSONに簡単に変換できます。APIレスポンスの生成に最適です。

エディタサポート

型ヒントにより、IDEやエディタの補完機能が充実します。開発効率が大幅に向上します。

3.3 リクエストボディの定義

POSTやPUTリクエストでクライアントから送信されるデータを、Pydanticモデルで定義します。FastAPIは自動的にリクエストボディをパースし、検証します。

main.py 
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

# アイテム作成用のリクエストモデル
class ItemCreate(BaseModel):
    """
    アイテム作成リクエスト
    """
    name: str           # アイテム名（必須）
    description: str    # 説明文（必須）
    price: float        # 価格（必須）
    tax: float = 0.0    # 税額（オプション、デフォルト0.0）

# アイテム作成エンドポイント
@app.post("/items/")
def create_item(item: ItemCreate):
    """
    新しいアイテムを作成する
    リクエストボディをItemCreateモデルで受け取る
    """
    # 税込み価格を計算
    total = item.price + item.tax
    return {
        "name": item.name,
        "total_price": total
    }

デフォルト値の設定

tax: float = 0.0 のように値を指定すると、そのフィールドは省略可能になります。省略された場合はデフォルト値が使用されます。

リクエスト例

request.json 
{
  "name": "ノートPC",
  "description": "高性能な開発用ノートパソコン",
  "price": 150000,
  "tax": 15000
}

3.4 レスポンスモデルの設計

エンドポイントが返すデータの構造も、Pydanticモデルで定義することができます。response_modelパラメータを使用することで、レスポンスの型を保証できます。

main.py 
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

# レスポンスモデル
class ItemResponse(BaseModel):
    """
    アイテム情報レスポンス
    """
    id: int
    name: str
    price: float

# response_modelを指定
@app.get("/items/{item_id}", response_model=ItemResponse)
def get_item(item_id: int):
    """
    アイテムを取得
    戻り値はItemResponseモデルの形式に自動変換される
    """
    # 実際はDBから取得する想定
    return {
        "id": item_id,
        "name": "サンプル商品",
        "price": 1000.0
    }

response_modelの利点

  • 自動ドキュメントにレスポンス形式が明記される
  • 余分なフィールドを自動的に除外できる
  • 型チェックにより誤ったレスポンスを防止
  • 一貫性のあるAPI設計が可能になる

3.5 データバリデーション

Pydanticでは、Field関数を使用して、詳細なバリデーションルールを設定できます。

models.py 
from pydantic import BaseModel, Field, EmailStr

class UserCreate(BaseModel):
    """
    ユーザー作成リクエスト（バリデーション付き）
    """
    # 名前: 最小1文字、最大50文字
    name: str = Field(..., min_length=1, max_length=50)

    # メールアドレス: 形式を自動検証
    email: EmailStr

    # 年齢: 0以上120以下
    age: int = Field(..., ge=0, le=120)

    # パスワード: 最小8文字
    password: str = Field(..., min_length=8)

よく使う検証ルール

パラメータ 説明 例
min_length 文字列の最小長 min_length=3
max_length 文字列の最大長 max_length=100
ge 以上（Greater than or Equal） ge=0
le 以下（Less than or Equal） le=100
regex 正規表現パターン regex="^[A-Z]"

ネストしたモデル

モデルの中に別のモデルを含めることで、複雑なデータ構造を表現できます。

models.py 
from pydantic import BaseModel
from typing import List

# 住所モデル
class Address(BaseModel):
    """
    住所情報
    """
    postal_code: str
    prefecture: str
    city: str
    street: str

# ユーザーモデル（住所を含む）
class User(BaseModel):
    """
    ユーザー情報（住所付き）
    """
    name: str
    email: str
    address: Address           # Addressモデルをネスト
    tags: List[str] = []       # 文字列のリスト

実習 3-1: 商品登録APIの作成

Pydanticモデルを使用して、商品情報を登録するAPIエンドポイントを作成しましょう。

手順

  1. 1 商品情報を表すPydanticモデルを定義します（名前、価格、在庫数）
  2. 2 価格は0以上、在庫数は0以上のバリデーションを追加します
  3. 3 POSTエンドポイント /products/ を作成します
  4. 4 Swagger UIでAPIをテストします

実装例

from fastapi import FastAPI
from pydantic import BaseModel, Field

app = FastAPI()

class Product(BaseModel):
    """
    商品情報モデル
    """
    name: str = Field(..., min_length=1)
    price: float = Field(..., ge=0)
    stock: int = Field(..., ge=0)

@app.post("/products/")
def create_product(product: Product):
    """
    商品を登録
    """
    return {
        "message": "商品を登録しました",
        "product": product
    }

理解度確認クイズ

  1. Q1. Pydanticモデルを使用することで得られる利点を3つ挙げてください。

    回答を表示

    1. 自動データ検証: 型やバリデーションルールに合わないデータを自動的に拒否
    2. 型安全性: 型ヒントにより、コードの誤りを早期に発見できる
    3. 自動ドキュメント生成: リクエスト・レスポンスの形式がAPIドキュメントに反映される

  2. Q2. Field(..., ge=0, le=100) のバリデーションルールの意味を説明してください。

    回答を表示

    このフィールドは必須（...）で、値は0以上（ge=0）かつ100以下（le=100）でなければなりません。範囲外の値が送信された場合、自動的にエラーが返されます。

  3. Q3. response_modelを指定する利点は何ですか？

    回答を表示

    1. 自動フィルタリング: モデルに定義されていないフィールドは自動的に除外されます（機密情報の漏洩防止）
    2. ドキュメント化: レスポンスの構造が自動ドキュメントに明記されます
    3. 型チェック: 誤ったレスポンスを返すコードを早期に発見できます

  4. Q4. ネストしたモデルを使用する場面の例を挙げてください。

    回答を表示

    例えば、注文情報（Order）の中に商品情報（Product）や配送先住所（Address）を含める場合です。これにより、複雑なデータ構造を階層的に表現でき、各層で適切なバリデーションを行うことができます。

第3章のまとめ

PydanticはFastAPIと統合され、リクエストとレスポンスの型安全性を保証します

BaseModelを継承したクラスで、型ヒント付きのフィールドを定義するだけで自動検証が行われます

Field関数を使用することで、詳細なバリデーションルール（最小値、最大値、文字列長など）を設定できます

response_modelを指定することで、レスポンスの型チェックと自動ドキュメント化が実現します

ネストしたモデルを使用することで、複雑なデータ構造を階層的に表現できます
