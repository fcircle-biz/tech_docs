10.1 本番環境用サーバー設定

開発時は uvicorn main:app --reload で十分ですが、本番環境では高可用性と安定性を確保するため、適切なサーバー設定が必要です。

本番環境のサーバー構成

Uvicorn単体での本番設定

小規模なアプリケーションでは、Uvicornを直接本番環境で実行できます。

本番環境での起動コマンド 
# 本番環境用のUvicorn起動
uvicorn main:app \
  --host 0.0.0.0 \
  --port 8000 \
  --workers 4 \
  --no-access-log \
  --log-config logging.yaml

Gunicorn + Uvicorn構成

より高い可用性が必要な場合は、Gunicornをプロセスマネージャーとして使用し、Uvicornワーカーを管理します。

ターミナル 
# Gunicornのインストール
pip install gunicorn

# Gunicorn + Uvicornワーカーで起動
gunicorn main:app \
  --workers 4 \
  --worker-class uvicorn.workers.UvicornWorker \
  --bind 0.0.0.0:8000 \
  --timeout 60 \
  --keep-alive 5

ワーカー数の決定

推奨されるワーカー数は (2 × CPUコア数) + 1 です。4コアの場合は9ワーカーが目安ですが、メモリ使用量も考慮して調整してください。

環境変数の管理

本番環境では、データベース接続文字列やAPIキーなどの機密情報を環境変数で管理します。

config.py 
# 環境変数ベースの設定管理
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    """アプリケーション設定"""
    app_name: str = "FastAPI App"
    database_url: str  # 必須
    secret_key: str    # 必須
    debug: bool = False

    class Config:
        # .envファイルから環境変数を読み込む
        env_file = ".env"

# グローバル設定インスタンス
settings = Settings()
.env.example 
# .envファイルのサンプル（本番環境では実際の値を設定）
DATABASE_URL=postgresql://user:password@localhost:5432/dbname
SECRET_KEY=your-secret-key-here
DEBUG=false

重要: .envファイルの管理

.envファイルは機密情報を含むため、必ず.gitignoreに追加してバージョン管理から除外してください。リポジトリには.env.exampleのみをコミットします。

10.2 Dockerコンテナ化

Dockerを使用することで、環境の差異を吸収し、どこでも同じように動作するアプリケーションを構築できます。

マルチステージビルドのDockerfile

マルチステージビルドを使用することで、最終的なイメージサイズを削減し、セキュリティも向上します。

Dockerfile 
# ビルドステージ: 依存関係のインストール
FROM python:3.11-slim as builder

WORKDIR /app
COPY requirements.txt .

# 依存関係をビルド
RUN pip install --user --no-cache-dir -r requirements.txt

# 実行ステージ: 最小限の環境
FROM python:3.11-slim

WORKDIR /app

# ビルドステージから依存関係をコピー
COPY --from=builder /root/.local /root/.local
COPY . .

# PATHに.local/binを追加
ENV PATH=/root/.local/bin:$PATH

# 非rootユーザーで実行（セキュリティ向上）
RUN useradd -m appuser && chown -R appuser /app
USER appuser

# ポート公開
EXPOSE 8000

# Gunicorn + Uvicornで起動
CMD ["gunicorn", "main:app", \
     "--workers", "4", \
     "--worker-class", "uvicorn.workers.UvicornWorker", \
     "--bind", "0.0.0.0:8000"]

docker-compose.ymlの作成

複数のサービス（アプリケーション、データベース、Redis等）を連携させる場合は、docker-composeを使用します。

docker-compose.yml 
version: '3.8'

services:
  # FastAPIアプリケーション
  app:
    build: .
    ports:
      - "8000:8000"
    environment:
      - DATABASE_URL=postgresql://user:password@db:5432/mydb
      - SECRET_KEY=${SECRET_KEY}
    depends_on:
      - db
      - redis

  # PostgreSQLデータベース
  db:
    image: postgres:15
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: password
      POSTGRES_DB: mydb
    volumes:
      - postgres_data:/var/lib/postgresql/data

  # Redisキャッシュ
  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"

volumes:
  postgres_data:
ターミナル 
# Dockerコンテナのビルドと起動
docker-compose up -d --build

# ログの確認
docker-compose logs -f app

# コンテナの停止
docker-compose down

10.4 ログ管理とモニタリング

本番環境では、適切なログ管理とモニタリングが障害の早期発見と迅速な対応に不可欠です。

構造化ログの導入

JSON形式の構造化ログを使用することで、ログの検索・分析が容易になります。

logger.py 
# 構造化ログの設定
import logging
import json
from datetime import datetime

class JSONFormatter(logging.Formatter):
    """JSON形式のログフォーマッター"""
    def format(self, record):
        log_data = {
            "timestamp": datetime.utcnow().isoformat(),
            "level": record.levelname,
            "message": record.getMessage(),
            "module": record.module,
            "function": record.funcName,
        }
        # エラーの場合はスタックトレースも含める
        if record.exc_info:
            log_data["exception"] = self.formatException(record.exc_info)
        return json.dumps(log_data)

# ロガーの設定
def setup_logger():
    logger = logging.getLogger("fastapi_app")
    handler = logging.StreamHandler()
    handler.setFormatter(JSONFormatter())
    logger.addHandler(handler)
    logger.setLevel(logging.INFO)
    return logger

リクエストログのミドルウェア

各リクエストの処理時間やステータスコードをログに記録します。

middleware.py 
# リクエストログミドルウェア
import time
from fastapi import Request
from starlette.middleware.base import BaseHTTPMiddleware

class RequestLoggingMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        start_time = time.time()

        # リクエスト情報をログに記録
        logger.info({
            "event": "request_started",
            "method": request.method,
            "path": request.url.path,
            "client_ip": request.client.host,
        })

        response = await call_next(request)

        # レスポンス情報と処理時間をログに記録
        process_time = time.time() - start_time
        logger.info({
            "event": "request_completed",
            "method": request.method,
            "path": request.url.path,
            "status_code": response.status_code,
            "duration_ms": round(process_time * 1000, 2),
        })

        return response

# main.pyでミドルウェアを追加
app.add_middleware(RequestLoggingMiddleware)

10.5 パフォーマンスチューニング

FastAPIアプリケーションのパフォーマンスを最適化するためのベストプラクティスを紹介します。

1. データベース接続プールの最適化

データベース接続プールのサイズを適切に設定することで、接続の再利用を促進し、レスポンス時間を改善します。

database.py 
# 接続プールの最適化
from sqlalchemy import create_engine

engine = create_engine(
    DATABASE_URL,
    pool_size=20,          # 通常の接続プールサイズ
    max_overflow=10,       # 追加可能な接続数
    pool_pre_ping=True,    # 接続の健全性チェック
    pool_recycle=3600,     # 1時間で接続をリサイクル
)

2. レスポンスキャッシング

頻繁にアクセスされるエンドポイントのレスポンスをキャッシュすることで、データベースへの負荷を軽減します。

cache.py 
# Redisキャッシュの実装
import redis
import json
from functools import wraps

redis_client = redis.Redis(host='localhost', port=6379, decode_responses=True)

def cache_response(expire_seconds=60):
    """レスポンスキャッシュデコレータ"""
    def decorator(func):
        @wraps(func)
        async def wrapper(*args, **kwargs):
            # キャッシュキーを生成
            cache_key = f"{func.__name__}:{str(args)}:{str(kwargs)}"

            # キャッシュから取得を試みる
            cached = redis_client.get(cache_key)
            if cached:
                return json.loads(cached)

            # キャッシュミスの場合は実行
            result = await func(*args, **kwargs)

            # 結果をキャッシュに保存
            redis_client.setex(
                cache_key,
                expire_seconds,
                json.dumps(result)
            )
            return result
        return wrapper
    return decorator

4. 非同期データベース操作

非同期データベースドライバを使用することで、I/O待機時間を最小限に抑えられます。

async_database.py 
# 非同期SQLAlchemyの使用
from sqlalchemy.ext.asyncio import create_async_engine, AsyncSession

# 非同期エンジンの作成
async_engine = create_async_engine(
    "postgresql+asyncpg://user:password@localhost/db",
    echo=False,
    pool_size=20,
)

# 非同期セッションの取得
async def get_async_session():
    async with AsyncSession(async_engine) as session:
        yield session

# 非同期エンドポイント
@app.get("/users/{user_id}")
async def get_user(user_id: int, db: AsyncSession = Depends(get_async_session)):
    result = await db.execute(
        select(User).where(User.id == user_id)
    )
    return result.scalar_one_or_none()

実習 10-1: Dockerコンテナ化とローカルデプロイ

これまで作成したFastAPIアプリケーションをDockerコンテナ化し、docker-composeで起動してみましょう。

手順

  1. 1 Dockerfileを作成し、マルチステージビルドを実装します
  2. 2 docker-compose.ymlを作成し、FastAPIとPostgreSQLを定義します
  3. 3 docker-compose upでコンテナを起動し、APIの動作を確認します
  4. 4 ログを確認し、ヘルスチェックエンドポイントをテストします

確認ポイント

  • コンテナが正常に起動し、APIにアクセスできること
  • データベース接続が成功し、データの永続化ができること
  • /healthエンドポイントが200を返すこと

理解度確認クイズ

  1. Q1. Gunicornをプロセスマネージャーとして使用する理由と、推奨されるワーカー数の計算方法を説明してください。

    回答を表示

    Gunicornは複数のワーカープロセスを管理し、1つのワーカーがクラッシュしても他のワーカーで処理を継続できるため、高可用性を実現できます。推奨ワーカー数は (2 × CPUコア数) + 1 です。例えば4コアの場合は9ワーカーが目安ですが、メモリ使用量も考慮して調整します。

  2. Q2. Dockerのマルチステージビルドを使用する利点を3つ挙げてください。

    回答を表示

    1. イメージサイズの削減: ビルドツールやキャッシュを最終イメージに含めないため、デプロイ時のイメージが小さくなります
    2. セキュリティ向上: ソースコードや開発用ツールを本番イメージから除外できます
    3. ビルド時間の短縮: レイヤーキャッシュを効果的に活用でき、変更がない部分は再利用されます

  3. Q3. 構造化ログ（JSON形式）を使用する利点と、通常のテキストログとの違いを説明してください。

    回答を表示

    構造化ログはJSON形式でログを出力するため、各フィールドが明確に区別され、ログ分析ツール（CloudWatch、Datadog等）で簡単に検索・集計できます。通常のテキストログは人間が読みやすい反面、機械的な処理が困難です。構造化ログでは、タイムスタンプ、レベル、メッセージ、メタデータを構造化して保存でき、特定の条件（例: エラーレベルで特定のユーザーIDに関連するログ）での検索が容易になります。

  4. Q4. ヘルスチェックエンドポイント（/health）とレディネスチェックエンドポイント（/readiness）の違いは何ですか？

    回答を表示

    ヘルスチェック（/health）: アプリケーションプロセスが起動しているかを確認する単純なチェックです。外部依存関係は確認せず、即座に応答します。
    レディネスチェック（/readiness）: データベースやRedisなど外部サービスへの接続も含めて、リクエストを受け付ける準備ができているかを確認します。この確認には時間がかかる場合がありますが、より包括的な健全性チェックを提供します。

  5. Q5. パフォーマンスチューニングの観点から、データベース接続プールのサイズを決定する際に考慮すべき要素を挙げてください。

    回答を表示

    1. アプリケーションのワーカー数: 各ワーカーが同時に使用する接続数を考慮
    2. データベースサーバーの最大接続数: PostgreSQLのmax_connections設定を超えないように
    3. 同時リクエスト数: ピーク時の予想同時リクエスト数
    4. メモリ使用量: 各接続がメモリを消費するため、サーバーのメモリ容量を考慮
    一般的には、ワーカー数の2〜3倍のプールサイズが推奨されます。

トラブルシューティング

Dockerコンテナがビルドエラーで失敗する

原因: requirements.txtに記載されたパッケージのバージョン不整合や、ベースイメージとの互換性問題

解決方法: docker build --no-cache でキャッシュを使わずにビルドしてみてください。また、requirements.txtのバージョンを固定（pip freezeを使用）することで再現性が向上します。

本番環境でメモリ使用量が急増する

原因: ワーカー数が多すぎる、またはデータベース接続プールが大きすぎる

解決方法: ワーカー数を減らし（--workers 2から試す）、接続プールサイズも調整してください。また、Uvicornの--limit-concurrencyオプションで同時接続数を制限することも有効です。

Cloud Runにデプロイ後、コールドスタート時にタイムアウトする

原因: アプリケーション起動時にデータベースマイグレーションなど時間のかかる処理を実行している

解決方法: マイグレーションは別のジョブとして実行し、アプリケーション起動時には実行しないようにします。また、Cloud Runの最小インスタンス数を1に設定することで、コールドスタートを回避できます。

ログが本番環境で出力されない

原因: ログレベルがWARNINGやERRORに設定されており、INFO以下のログが出力されていない

解決方法: 環境変数 LOG_LEVEL=INFO を設定するか、logging設定で明示的にレベルを指定してください。また、Uvicornの--log-level infoオプションも確認してください。
