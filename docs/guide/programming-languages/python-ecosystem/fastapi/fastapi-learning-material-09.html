9.1 pytestによるテスト

pytestは、Pythonで最も人気のあるテストフレームワークです。シンプルな記法と豊富な機能により、FastAPIアプリケーションのテストに最適です。

pytestのインストール

ターミナル 
# pytestと必要なパッケージのインストール
pip install pytest pytest-cov httpx

# インストール確認
pytest --version

必要なパッケージ

pytest-cov: テストカバレッジを測定するプラグイン
httpx: FastAPIのTestClientが内部で使用する非同期HTTPクライアント

プロジェクト構成

テストコードは、プロジェクトルートに tests/ ディレクトリを作成して管理します。

テストディレクトリ構成 

myproject/
├── app/
│   ├── __init__.py
│   ├── main.py          # FastAPIアプリケーション
│   ├── models.py        # データモデル
│   └── routers/
│       └── items.py     # ルーター
├── tests/               ← テストディレクトリ
│   ├── __init__.py
│   ├── conftest.py      # 共通フィクスチャ
│   ├── test_main.py     # メイン機能のテスト
│   └── test_items.py    # アイテムAPIのテスト
├── requirements.txt
└── pytest.ini           # pytest設定ファイル

9.2 FastAPIのTestClient

FastAPIは、Starletteの TestClient を提供しており、実際のHTTPリクエストを送信せずにAPIをテストできます。

基本的な使い方

app/main.py 
from fastapi import FastAPI

app = FastAPI()

@app.get("/")
def read_root():
    """ルートエンドポイント"""
    return {"message": "Hello World"}

@app.get("/items/{item_id}")
def read_item(item_id: int):
    """アイテム取得エンドポイント"""
    return {"item_id": item_id, "name": f"Item {item_id}"}
tests/test_main.py 
from fastapi.testclient import TestClient
from app.main import app

# TestClientインスタンスを作成
client = TestClient(app)

def test_read_root():
    """ルートエンドポイントのテスト"""
    response = client.get("/")
    assert response.status_code == 200
    assert response.json() == {"message": "Hello World"}

def test_read_item():
    """アイテム取得エンドポイントのテスト"""
    response = client.get("/items/42")
    assert response.status_code == 200
    assert response.json() == {"item_id": 42, "name": "Item 42"}

テストの実行

ターミナル 
# すべてのテストを実行
pytest

# 詳細表示で実行
pytest -v

# 特定のファイルのみ実行
pytest tests/test_main.py

# 特定のテスト関数のみ実行
pytest tests/test_main.py::test_read_root

TestClientの利点

TestClientは実際のHTTPサーバーを起動せずにテストできるため、高速で効率的です。また、非同期エンドポイントも同期的にテストできます。

9.3 単体テストの実装

単体テストは、個々の関数やメソッドが正しく動作するかを検証します。外部依存（データベース、外部API等）をモック化して、独立してテストします。

Pydanticモデルのテスト

app/models.py 
from pydantic import BaseModel, field_validator

class Item(BaseModel):
    """アイテムモデル"""
    name: str
    price: float
    is_active: bool = True

    @field_validator('price')
    def price_must_be_positive(cls, v):
        """価格は正の数でなければならない"""
        if v <= 0:
            raise ValueError('価格は0より大きい必要があります')
        return v
tests/test_models.py 
import pytest
from pydantic import ValidationError
from app.models import Item

def test_item_creation():
    """正常なアイテム作成のテスト"""
    item = Item(name="Test Item", price=100.0)
    assert item.name == "Test Item"
    assert item.price == 100.0
    assert item.is_active is True

def test_item_invalid_price():
    """不正な価格でバリデーションエラーが発生することを確認"""
    with pytest.raises(ValidationError) as exc_info:
        Item(name="Test Item", price=-10.0)
    assert "価格は0より大きい必要があります" in str(exc_info.value)

フィクスチャの活用

pytestのフィクスチャを使うと、テストで繰り返し使うデータやオブジェクトを効率的に管理できます。

tests/conftest.py 
import pytest
from fastapi.testclient import TestClient
from app.main import app

@pytest.fixture
def client():
    """TestClientフィクスチャ（全テストで共有）"""
    return TestClient(app)

@pytest.fixture
def sample_item_data():
    """サンプルアイテムデータフィクスチャ"""
    return {
        "name": "Test Item",
        "price": 99.99,
        "is_active": True
    }
tests/test_items.py 
def test_create_item(client, sample_item_data):
    """フィクスチャを使ったアイテム作成テスト"""
    response = client.post("/items/", json=sample_item_data)
    assert response.status_code == 200
    data = response.json()
    assert data["name"] == sample_item_data["name"]
    assert data["price"] == sample_item_data["price"]

フィクスチャのスコープ

@pytest.fixture(scope="session") を使うと、テストセッション全体で1回だけ実行されます。データベース接続など、初期化コストが高いリソースに有効です。

9.4 統合テストの実装

統合テストは、複数のコンポーネント（エンドポイント、データベース、認証等）が連携して正しく動作するかを検証します。

データベースを使った統合テスト

tests/conftest.py 
import pytest
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker
from app.database import Base, get_db
from app.main import app

# テスト用データベースURL（インメモリSQLite）
SQLALCHEMY_DATABASE_URL = "sqlite:///./test.db"

@pytest.fixture(scope="function")
def db_session():
    """テスト用データベースセッション"""
    engine = create_engine(SQLALCHEMY_DATABASE_URL)
    Base.metadata.create_all(bind=engine)
    TestingSessionLocal = sessionmaker(bind=engine)
    db = TestingSessionLocal()
    try:
        yield db
    finally:
        db.close()
        Base.metadata.drop_all(bind=engine)
tests/test_integration.py 
from fastapi.testclient import TestClient
from app.main import app, get_db

def test_create_and_get_item(db_session):
    """アイテム作成と取得の統合テスト"""
    # データベースの依存性をオーバーライド
    app.dependency_overrides[get_db] = lambda: db_session
    client = TestClient(app)

    # アイテムを作成
    create_response = client.post(
        "/items/",
        json={"name": "Integration Test Item", "price": 150.0}
    )
    assert create_response.status_code == 200
    item_id = create_response.json()["id"]

    # 作成したアイテムを取得
    get_response = client.get(f"/items/{item_id}")
    assert get_response.status_code == 200
    assert get_response.json()["name"] == "Integration Test Item"

認証が必要なエンドポイントのテスト

tests/test_auth.py 
def test_protected_endpoint_without_auth(client):
    """認証なしでアクセスすると401が返ることを確認"""
    response = client.get("/protected/")
    assert response.status_code == 401

def test_protected_endpoint_with_auth(client):
    """認証トークン付きでアクセスできることを確認"""
    # ログインしてトークン取得
    login_response = client.post(
        "/token",
        data={"username": "testuser", "password": "testpass"}
    )
    token = login_response.json()["access_token"]

    # トークンを使って保護されたエンドポイントにアクセス
    response = client.get(
        "/protected/",
        headers={"Authorization": f"Bearer {token}"}
    )
    assert response.status_code == 200

9.5 テストカバレッジの計測

テストカバレッジは、コードのどの部分がテストされているかを示す指標です。pytest-covプラグインを使って計測できます。

カバレッジの計測

ターミナル 
# カバレッジレポート付きでテスト実行
pytest --cov=app tests/

# HTMLレポートを生成
pytest --cov=app --cov-report=html tests/

# カバレッジ率の下限を設定（80%未満でエラー）
pytest --cov=app --cov-fail-under=80 tests/

pytest.ini設定ファイル

pytest.ini 
[pytest]
# テストディレクトリ
testpaths = tests

# 詳細表示
addopts = -v --strict-markers --cov=app --cov-report=term-missing

# カバレッジの最低ライン
cov-fail-under = 80

# Pythonパス設定
pythonpath = .

カバレッジの目標

一般的に70〜80%のカバレッジが推奨されます。100%を目指すのではなく、重要なビジネスロジックとエラーハンドリングに焦点を当てましょう。

9.6 APIドキュメントのカスタマイズ

FastAPIの自動生成ドキュメントは、タイトルや説明、バージョン情報などをカスタマイズできます。

基本的なカスタマイズ

app/main.py 
from fastapi import FastAPI

app = FastAPI(
    title="My API",
    description="This is my awesome API with auto docs",
    version="1.0.0",
    terms_of_service="http://example.com/terms/",
    contact={
        "name": "API Support",
        "email": "support@example.com",
    },
    license_info={
        "name": "MIT License",
        "url": "https://opensource.org/licenses/MIT",
    },
)

エンドポイントのドキュメント強化

app/routers/items.py 
from fastapi import APIRouter, HTTPException
from pydantic import BaseModel

router = APIRouter()

class Item(BaseModel):
    name: str
    price: float

@router.post(
    "/items/",
    response_model=Item,
    status_code=201,
    summary="新しいアイテムを作成",
    description="アイテム名と価格を指定して新しいアイテムを作成します",
    response_description="作成されたアイテムの情報",
    tags=["items"],
)
def create_item(item: Item):
    """
    アイテムを作成します。

    - **name**: アイテムの名前
    - **price**: アイテムの価格（正の数）
    """
    return item

タグによるグルーピング

app/main.py

# タグのメタデータを定義

ドキュメントのベストプラクティス

  • エンドポイントごとに明確な説明を記載する
  • レスポンス例とエラーケースを文書化する
  • パラメータの制約や期待される形式を明示する
  • 関連するエンドポイントをタグでグルーピングする

9.7 OpenAPI仕様の理解

FastAPIは、OpenAPI（旧Swagger）仕様に準拠したJSON/YAMLスキーマを自動生成します。このスキーマを活用することで、クライアントコードの自動生成やAPI検証が可能になります。

OpenAPIスキーマの取得

FastAPIアプリケーションのOpenAPIスキーマは、/openapi.json エンドポイントから取得できます。

OpenAPIスキーマURL:

http://127.0.0.1:8000/openapi.json

OpenAPIエコシステム

スキーマ生成] B --> C[Swagger UI] B --> D[ReDoc] B --> E[Postman
インポート] B --> F[クライアントコード
自動生成] B --> G[API バリデーション
ツール] F --> F1[TypeScript SDK] F --> F2[Python クライアント] F --> F3[Java SDK] style B fill:#fef3c7,stroke:#f59e0b,stroke-width:2px style C fill:#dbeafe,stroke:#3b82f6 style D fill:#e9d5ff,stroke:#a855f7

OpenAPIスキーマのカスタマイズ

app/main.py 
from fastapi import FastAPI
from fastapi.openapi.utils import get_openapi

app = FastAPI()

def custom_openapi():
    """OpenAPIスキーマをカスタマイズ"""
    if app.openapi_schema:
        return app.openapi_schema

    openapi_schema = get_openapi(
        title="Custom API",
        version="2.0.0",
        description="This is a custom OpenAPI schema",
        routes=app.routes,
    )

    # カスタムフィールドを追加
    openapi_schema["info"]["x-logo"] = {
        "url": "https://example.com/logo.png"
    }

    app.openapi_schema = openapi_schema
    return app.openapi_schema

# カスタムスキーマを設定
app.openapi = custom_openapi

クライアントコードの自動生成

OpenAPI Generatorなどのツールを使うと、OpenAPIスキーマから各言語のクライアントSDKを自動生成できます。

ターミナル 
# OpenAPI Generatorをインストール
npm install -g @openapitools/openapi-generator-cli

# TypeScriptクライアントを生成
openapi-generator-cli generate \
  -i http://localhost:8000/openapi.json \
  -g typescript-axios \
  -o ./client/typescript

# Pythonクライアントを生成
openapi-generator-cli generate \
  -i http://localhost:8000/openapi.json \
  -g python \
  -o ./client/python

実習 9-1: テストスイートの作成

これまで作成したFastAPIアプリケーションに対して、包括的なテストスイートを作成しましょう。

手順

  1. 1 プロジェクトルートに tests/ ディレクトリを作成
  2. 2 pytest、pytest-cov、httpxをインストール
  3. 3 各エンドポイントに対する単体テストを作成
  4. 4 認証フローを含む統合テストを実装
  5. 5 テストカバレッジを計測し、80%以上を目指す

期待される成果

  • すべてのテストがpassすること
  • カバレッジレポートが80%以上であること
  • 正常系とエラー系の両方をテストしていること

理解度確認クイズ

  1. Q1. FastAPIのTestClientを使用する主な利点を3つ挙げてください。

    回答を表示

    1. 高速: 実際のHTTPサーバーを起動せずテストできるため高速
    2. シンプル: Requestsライクな直感的なAPIでテストを記述できる
    3. 非同期対応: 非同期エンドポイントも同期的にテストできる

  2. Q2. 単体テストと統合テストの違いを説明してください。

    回答を表示

    単体テスト: 個々の関数やメソッドを独立してテストし、外部依存をモック化します。高速で実行でき、バグの原因を特定しやすい利点があります。

    統合テスト: 複数のコンポーネント（API、DB、認証等）が連携して動作することを検証します。実際の動作環境に近い状態でテストできる利点があります。

  3. Q3. テストカバレッジ100%を目指すべきですか？その理由も説明してください。

    回答を表示

    一般的には100%を目指す必要はありません。70〜80%が適切な目標です。

    理由: カバレッジが高くても品質が保証されるわけではなく、単純なgetter/setterや設定コードまでテストするのは非効率です。重要なビジネスロジック、エラーハンドリング、セキュリティ関連のコードに焦点を当てることが重要です。

  4. Q4. OpenAPIスキーマを活用できる具体的な場面を3つ挙げてください。

    回答を表示

    1. クライアントSDKの自動生成: TypeScript、Python、Javaなどのクライアントコードを自動生成
    2. API仕様の共有: PostmanやInsomnia等のツールにインポートしてチーム間で共有
    3. APIバリデーション: 仕様違反やセキュリティ問題を自動検出するツールで検証

第9章のまとめ

pytestとTestClientを使うことで、FastAPIアプリケーションを効率的にテストできます

単体テストで個別機能を検証し、統合テストで全体の動作を確認する階層的なテスト戦略が重要です

pytest-covでテストカバレッジを計測し、重要なコードが適切にテストされているか確認できます

FastAPIの自動生成ドキュメントはカスタマイズ可能で、OpenAPIスキーマを活用することで様々なツールと連携できます
