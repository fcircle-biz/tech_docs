基本的なルーティングとHTTPメソッド

パスオペレーションの定義方法、HTTPメソッドの使い分け、ステータスコードの設定、レスポンスヘッダーのカスタマイズについて学習します。

この章で学ぶこと

  • パスオペレーションの定義方法と複数エンドポイントの管理

  • ステータスコードの意味と適切な返却方法
  • レスポンスヘッダーのカスタマイズとCookie設定

2.1 パスオペレーションとは

パスオペレーション（Path Operation）は、FastAPIにおいてURLパスとHTTPメソッドを組み合わせてAPIエンドポイントを定義する仕組みです。RESTful APIの基礎となる概念で、リソースの操作を明確に表現します。

パスオペレーションの構造

GET/POST/PUT/DELETE] A --> C[URLパス
/items/123] A --> D[関数
実行する処理] style A fill:#dbeafe,stroke:#3b82f6,stroke-width:3px

RESTful APIの基本原則

REST（Representational State Transfer）では、リソースをURLで表現し、HTTPメソッドでリソースに対する操作を表現します。例えば、GET /users/1 は「ユーザーID=1のデータを取得」を意味します。

複数のパスオペレーションの定義

複数のエンドポイントをまとめて定義することで、APIの全体構造を構築します。

main.py 
from fastapi import FastAPI

app = FastAPI()

# 全ユーザー一覧を取得
@app.get("/users")
def get_users():
    """全ユーザーのリストを返す"""
    return {"users": ["Alice", "Bob", "Charlie"]}

# 特定ユーザーの詳細を取得
@app.get("/users/{user_id}")
def get_user(user_id: int):
    """指定されたIDのユーザー情報を返す"""
    return {"user_id": user_id, "name": "Alice"}

# 新規ユーザーを作成
@app.post("/users")
def create_user():
    """新しいユーザーを作成"""
    return {"message": "User created successfully"}

パスの評価順序

FastAPIは定義された順番にパスを評価します。より具体的なパス（/users/me）を先に定義し、パラメータを含む汎用的なパス（/users/{user_id}）を後に定義する必要があります。

2.2 HTTPメソッドの使い分け

HTTPメソッドは、リソースに対する操作の種類を表現します。FastAPIでは主要な5つのHTTPメソッドを使用します。

GET - 取得

データの取得に使用。サーバーの状態を変更しません（べき等性あり）。

@app.get("/items")

POST - 作成

新しいリソースの作成に使用。複数回実行すると異なる結果になります。

@app.post("/items")

PUT - 全体更新

リソース全体を置き換え。指定したフィールド以外はクリアされます（べき等性あり）。

@app.put("/items/{id}")

PATCH - 部分更新

リソースの一部だけを更新。他のフィールドはそのまま保持されます。

@app.patch("/items/{id}")

DELETE - 削除

リソースの削除に使用。複数回実行しても結果は同じ（べき等性あり）。

@app.delete("/items/{id}")

実装例: ブログ記事API

blog_api.py 
from fastapi import FastAPI

app = FastAPI()

# 記事一覧の取得
@app.get("/posts")
def list_posts():
    """全記事のリストを返す"""
    return {"posts": [{"id": 1, "title": "はじめての投稿"}]}

# 記事詳細の取得
@app.get("/posts/{post_id}")
def get_post(post_id: int):
    """特定の記事を取得"""
    return {"id": post_id, "title": "記事タイトル", "content": "本文"}

# 新規記事の作成
@app.post("/posts")
def create_post():
    """新しい記事を作成"""
    return {"id": 2, "message": "記事が作成されました"}

# 記事の全体更新
@app.put("/posts/{post_id}")
def update_post(post_id: int):
    """記事全体を置き換え"""
    return {"id": post_id, "message": "記事を更新しました"}

# 記事の部分更新
@app.patch("/posts/{post_id}")
def partial_update_post(post_id: int):
    """記事の一部を更新"""
    return {"id": post_id, "message": "タイトルを更新しました"}

# 記事の削除
@app.delete("/posts/{post_id}")
def delete_post(post_id: int):
    """記事を削除"""
    return {"message": f"記事ID {post_id} を削除しました"}

2.3 ステータスコードの設定

HTTPステータスコードは、リクエストの処理結果をクライアントに伝えるための3桁の数字です。FastAPIでは、デコレータのstatus_codeパラメータで設定できます。

主要なステータスコード

2xx 成功レスポンス

  • 200 OK - リクエスト成功
  • 201 Created - リソース作成成功
  • 204 No Content - 成功したがレスポンスボディなし（削除時など）

4xx クライアントエラー

  • 400 Bad Request - リクエストが不正
  • 404 Not Found - リソースが見つからない
  • 422 Unprocessable Entity - バリデーションエラー（FastAPIのデフォルト）

5xx サーバーエラー

  • 500 Internal Server Error - サーバー内部エラー
  • 503 Service Unavailable - サービス一時停止中

ステータスコードの実装

status_example.py 
from fastapi import FastAPI, status

app = FastAPI()

# 200 OK（デフォルト）
@app.get("/items")
def get_items():
    """ステータスコードを明示的に指定しない場合は200"""
    return {"items": []}

# 201 Created - リソース作成時
@app.post("/items", status_code=status.HTTP_201_CREATED)
def create_item():
    """新規作成時は201を返す"""
    return {"id": 1, "name": "新しいアイテム"}

# 204 No Content - 削除成功時
@app.delete("/items/{item_id}", status_code=status.HTTP_204_NO_CONTENT)
def delete_item(item_id: int):
    """削除成功時はレスポンスボディなしで204を返す"""
    # return文なし、または return None
    pass

statusモジュールの活用

FastAPIのstatusモジュールを使用すると、数字の代わりにstatus.HTTP_201_CREATEDのような定数を使えます。コードの可読性が向上し、タイプミスも防げます。

2.4 レスポンスヘッダーのカスタマイズ

レスポンスヘッダーをカスタマイズすることで、キャッシュ制御、Cookie設定、CORSの制御など、高度なAPI機能を実装できます。

Responseオブジェクトの使用

headers_example.py 
from fastapi import FastAPI, Response

app = FastAPI()

@app.get("/items")
def get_items(response: Response):
    """
    レスポンスヘッダーをカスタマイズ
    """
    # カスタムヘッダーの追加
    response.headers["X-Custom-Header"] = "Custom Value"
    response.headers["Cache-Control"] = "max-age=3600"

    return {"items": ["item1", "item2"]}

Cookieの設定

cookie_example.py 
from fastapi import FastAPI, Response

app = FastAPI()

@app.post("/login")
def login(response: Response):
    """
    ログイン処理とセッションCookieの設定
    """
    # Cookieの設定
    response.set_cookie(
        key="session_id",
        value="abc123",
        max_age=3600,  # 有効期限（秒）
        httponly=True,  # JavaScriptからのアクセスを防ぐ
        secure=True,    # HTTPS接続時のみ送信
        samesite="lax"  # CSRF対策
    )

    return {"message": "ログイン成功"}

Cookieのセキュリティ設定

httponly=TrueでXSS攻撃を防ぎ、secure=Trueで通信の安全性を確保します。samesite属性でCSRF攻撃を防げます。

理解度確認クイズ

  1. Q1. PUTとPATCHの違いを説明してください。

    回答を表示

    PUT: リソース全体を置き換えます。指定しなかったフィールドはクリアされます。例: ユーザー情報の全項目を送信する必要がある。
    PATCH: リソースの一部だけを更新します。指定したフィールドのみが変更され、他のフィールドは保持されます。例: ユーザー名だけを変更する。

  2. Q2. ステータスコード201と200の使い分けを説明してください。

    回答を表示

    200 OK: リクエストが成功した場合の汎用的なステータスコード。データの取得や更新に使用します。
    201 Created: 新しいリソースが正常に作成された場合に使用します。POSTリクエストでリソースを作成した際に返すのが適切です。

  3. Q3. FastAPIで複数のパスオペレーションを定義する際、定義順序が重要なのはなぜですか？

    回答を表示

    FastAPIは定義された順番にパスを評価します。例えば、/users/{user_id}を先に定義すると、/users/meへのリクエストが{user_id}にマッチしてしまいます。より具体的なパスを先に定義することで、意図した動作を保証できます。

  4. Q4. Cookieにhttponly=Trueを設定する理由は何ですか？

    回答を表示

    httponly=Trueを設定すると、JavaScriptからCookieにアクセスできなくなります。これにより、XSS（クロスサイトスクリプティング）攻撃でセッション情報が盗まれるリスクを軽減できます。セキュリティ上、認証情報を含むCookieには必須の設定です。

第2章のまとめ

パスオペレーションは、URLパスとHTTPメソッドを組み合わせてエンドポイントを定義する仕組みです

HTTPメソッド（GET、POST、PUT、PATCH、DELETE）は、リソースに対する操作を明確に表現します

ステータスコードは処理結果を伝えるための重要な手段で、適切に設定することが重要です

レスポンスヘッダーとCookieをカスタマイズすることで、高度なAPI機能を実装できます
