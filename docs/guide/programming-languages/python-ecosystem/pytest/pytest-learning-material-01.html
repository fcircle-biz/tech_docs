テストとは何か - なぜテストを書くのか

ソフトウェアテストの基本概念と重要性を理解します。手動テストと自動テストの違い、テストを書くことで得られるメリット、そしてPythonにおけるテストフレームワークの全体像を把握します。

この章で学ぶこと

  • ソフトウェアテストの目的と種類（単体・結合・システムテスト）を説明できる
  • 手動テストと自動テストの違いとそれぞれの特徴を理解する
  • テストを書くことで得られる3つのメリット（早期バグ発見・リファクタリングの安心感・仕様のドキュメント化）を説明できる
  • Pythonのテストフレームワーク（pytest・unittest）の概要を把握する

1.1 ソフトウェアテストとは何か

プログラムを書いたとき、「ちゃんと動いているかな？」と確認するために実際に動かしてみた経験はありませんか？それがテストの出発点です。

ソフトウェアテストとは、プログラムが期待どおりに動作することを確認する行為です。単に「動く」だけでなく、「正しく動く」ことを証明するために行います。

電卓アプリの「2 + 3 = 5」になるかを確認する作業がテストです。入力（2と3の足し算）を与えて、出力（5）が期待どおりかを確認しています。

1.2 テストの種類 - どこを対象にするか

テストには対象の範囲によっていくつかの種類があります。「小さな部品」から「システム全体」まで、段階的にテストを積み重ねることで品質を確保します。

単体テスト

1つの関数やクラスを単独でテストします。最も小さい単位のテストで、問題の原因を特定しやすいのが特徴です。

Unit Test

結合テスト

複数の部品を組み合わせたテストです。それぞれが正しく動いていても、組み合わせると問題が起きることがあります。

Integration Test

システムテスト

システム全体を実際のユーザー視点でテストします。「画面からボタンを押して〇〇できること」のような確認です。

System Test / E2E Test

底辺（単体テスト）を多く積み上げることが、効率的かつ安定したテスト戦略の基本です。pytestはこの単体テスト・結合テストを書くためのツールです。

1.3 手動テスト vs 自動テスト

テストには「人間が手作業で行う手動テスト」と「プログラムが自動的に行う自動テスト」の2種類があります。それぞれの特徴を理解することが、なぜ自動テストを書くのかの理解につながります。

観点 手動テスト 自動テスト
実行速度 遅い（人が操作する時間が必要） 速い（数百件を数秒で実行可能）
繰り返し実行 コストが高い（毎回人手が必要） コストが低い（何度でも実行可能）
見つけやすいバグ UI・使い勝手の問題（人間の目線） ロジックのバグ・回帰バグ
初期コスト 低い（すぐ始められる） 高い（テストコードを書く必要がある）
メンテナンス 仕様変更への追従が容易 コード変更時にテストも修正が必要

重要ポイント：自動テストの真価は「繰り返し」にある

1回だけのテストなら手動でも十分かもしれません。しかし、開発中は何十回・何百回とテストを実行します。コードを変更するたびに「壊れていないか」を確認する作業が自動化されることで、開発の安心感が大きく向上します。

1.4 テストを書くことの3つのメリット

「テストコードを書く時間があれば、本体のコードを書いたほうがいい」と思う方もいるかもしれません。しかし、テストを書くことには長期的に見て大きなメリットがあります。

1

バグの早期発見

バグは発見が遅れるほど、修正コストが指数関数的に増大します。開発中に発見できれば数分で直せるバグが、リリース後に発見されると数日かかることもあります。

バグ発見コストのイメージ

開発中: 1 テスト中: 10 リリース後: 100
2

リファクタリングの安心感

リファクタリングとは、動作を変えずにコードの構造を改善することです。テストがあれば「変更後も正しく動くか」を即座に確認でき、積極的に改善できます。

テストのない状態でのリファクタリングは、セーフティネットなしでの綱渡りのようなものです。テストというセーフティネットがあれば、安心して改善できます。

3

仕様のドキュメント化

テストコードは「このコードがどのように動くべきか」を記述した、実行可能なドキュメントでもあります。Wordで書かれたドキュメントと異なり、実際のコードと常に同期されています。

test_calculator.py（テストが仕様を示す例） 
# テストコードを読むだけで「仕様」がわかる

def test_割り算は商を返す():
    assert divide(10, 2) == 5

def test_ゼロ除算は例外を発生させる():
    with pytest.raises(ZeroDivisionError):
        divide(10, 0)

def test_少数の割り算も正確に計算する():
    assert divide(1, 3) == pytest.approx(0.333, rel=1e-3)

1.5 Pythonのテストフレームワーク概要

Pythonには複数のテストフレームワークがあります。このガイドでは主にpytestを使いますが、まず全体像を把握しましょう。

unittest

Python標準ライブラリ
  • 追加インストール不要・Pythonに標準で付属
  • Java の JUnit に似たクラスベースの書き方
  • 第3章で詳しく学習します

pytest

このガイドのメイン
  • シンプルな assert 文でテストを書ける
  • 詳細なエラーレポートで原因を特定しやすい
  • 豊富なプラグインで機能を拡張できる
  • 業界で最も広く使われているテストフレームワーク

同じテストをpytestとunittestで書いた場合の比較

unittest の書き方（第3章で詳しく学習）

unittest版 
import unittest

class TestCalculator(unittest.TestCase):
    def test_add(self):
        self.assertEqual(2 + 3, 5)

    def test_divide_by_zero(self):
        with self.assertRaises(ZeroDivisionError):
            1 / 0

if __name__ == '__main__':
    unittest.main()

pytest の書き方（このガイドで学ぶ）

pytest版 
import pytest

def test_add():
    # シンプルな assert 文でOK！
    assert 2 + 3 == 5

def test_divide_by_zero():
    with pytest.raises(ZeroDivisionError):
        1 / 0

pytestのほうがコード量が少なく、クラスを継承する必要もありません。assert文だけでテストを書けるシンプルさが、pytestが広く採用されている理由の一つです。

1.6 テストを書くタイミング

テストを書くタイミングは、開発手法によって異なります。代表的なアプローチを理解しておきましょう。

テストファースト / TDD（テスト駆動開発）

コードを書く前にテストを書く手法です。「まずテストを書いて失敗させ、次にテストが通るコードを書き、最後にリファクタリングする」というサイクルを繰り返します。

Red（失敗） → Green（成功） → Refactor（改善）

コードと並行して書く

関数やクラスを実装した直後にテストを書く手法です。最も一般的なアプローチで、実装の記憶が新鮮なうちにテストを書けるメリットがあります。

後からテストを追加する

既存のコードに対して後からテストを追加する手法です。レガシーコードへのテスト追加や、バグ修正時に「このバグが再発しないことを確認するテスト」を書く場合に行います。

このガイドでの進め方

このガイドでは「コードと並行して書く」アプローチを基本とします。まずテスト対象の関数を実装し、次にそれをテストするコードを書く流れで進めます。慣れてきたらTDDにも挑戦してみましょう。

実習 1-1: テストが必要なシーンを考えてみよう

テストを自動化すると特に便利なシーンを身の回りから考え、言語化する練習をします。コードを書く必要はありません。

課題

  1. 1 あなたが作ったことがある（または作りたい）プログラムを一つ思い浮かべてください。（例：計算機、買い物リスト、天気予報表示アプリなど）
  2. 2 そのプログラムでテストすべき「重要な動作」を3つ挙げてください。（例：マイナスの数値を入力したときに正しく処理できるか）
  3. 3 それぞれのテストについて「どんな入力を与えて、何が出力されるべきか」を整理してみましょう。

考え方のヒント

  • • 正常なケース: 通常の使い方で正しい結果が出るか
  • • 境界値: ゼロ、負の数、最大値・最小値はどうか
  • • 異常なケース: 文字列を数値として渡したらどうなるか

第2章: pytestの導入と最初のテスト

理解度確認クイズ

  1. Q1. 「1つの関数やクラスを単独でテストする」テストを何と呼びますか？

    回答を表示

    単体テスト（ユニットテスト / Unit Test）です。最も小さい単位のテストで、問題の原因を特定しやすく、テストピラミッドの底辺を支える重要なテストです。

  2. Q2. 自動テストが手動テストより優れている点を1つ挙げてください。

    回答を表示

    代表的な回答例: 繰り返し実行のコストが低い（一度書けば何度でも実行できる）、実行速度が速い（数百件のテストを数秒で実行できる）、などです。

  3. Q3. テストを書くことの3つのメリットを答えてください。

    回答を表示

    1. バグの早期発見（開発中に見つけるほど修正コストが低い）、2. リファクタリングの安心感（変更後も動作を確認できる）、3. 仕様のドキュメント化（テストコードが実行可能な仕様書になる）の3つです。

  4. Q4. pytestとunittestの大きな違いを一言で表すと？

    回答を表示

    pytestはシンプルな assert 文だけでテストを書ける（クラス継承不要）のに対し、unittestはクラスを継承して専用のassertメソッドを使う必要があります。pytestはより少ないコードで同じテストを書けます。
