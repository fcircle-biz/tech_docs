モックとパッチ - 外部依存をテストから分離する

unittest.mockモジュールとpytestのmonkeypatchフィクスチャを使い、外部APIコール、データベースアクセス、ファイルI/Oなどの外部依存を置き換えてテストする方法を学びます。Mock、MagicMock、patchデコレータの使い方を理解します。

この章で学ぶこと

  • モック（Mock）とは何か、なぜ外部依存を置き換える必要があるかを説明できる
  • unittest.mockのMock・MagicMockを使って戻り値や例外をシミュレートできる
  • patchデコレータ・コンテキストマネージャを使い、特定の関数やクラスを置き換えられる
  • pytestのmonkeypatchフィクスチャで属性・環境変数・関数を安全に置き換えられる
  • モックが呼ばれたかどうか・どのような引数で呼ばれたかを検証できる

10.1 モックとは何か - なぜ外部依存を置き換えるのか

プログラムは多くの場合、外部のシステムに依存しています。たとえば天気予報APIに問い合わせたり、データベースに接続してデータを取得したりします。これらの「外部依存」をそのままテストしようとすると、さまざまな問題が生じます。

ネットワーク依存

外部APIを呼び出すテストは、ネットワークが切れていると失敗します。CI環境や社内ネットワーク制限で動かないことも。

副作用のリスク

テストでDBを操作すると本番データが変わるリスクがあります。メール送信APIを実際に呼ぶとメールが飛んでしまいます。

テスト速度

外部APIの応答待ちやDBの処理に時間がかかり、テスト全体が遅くなります。数百のテストを高速に実行できなくなります。

モックとスタブの違い（参考）

テストダブルには複数の種類があります。スタブは決まった値を返すだけの偽オブジェクト、モックは「呼ばれたか」「何回呼ばれたか」も検証できる偽オブジェクトです。Pythonではunittest.mockのMockクラスが両方の機能を持ちます。

10.2 unittest.mock の基本 - Mock と MagicMock

Python標準ライブラリのunittest.mockモジュールには、モックオブジェクトを作成するためのクラスが用意されています。最も基本的なMockと、マジックメソッド（__len__や__str__等）もモックできるMagicMockを理解しましょう。

Mock オブジェクトの基本的な使い方

test_mock_basic.py 
from unittest.mock import Mock, MagicMock

# Mock オブジェクトを作成
mock_obj = Mock()

# 戻り値を設定できる
mock_obj.return_value = 42

# 呼び出すと設定した値が返る
result = mock_obj()
assert result == 42  # 通過

# 属性にアクセスすると自動的に Mock が作られる
mock_obj.some_attribute.return_value = "hello"
assert mock_obj.some_attribute() == "hello"

# 呼び出し回数・引数を検証できる
mock_obj.reset_mock()
mock_obj(1, 2, key="value")
mock_obj.assert_called_once_with(1, 2, key="value")  # OK
print(mock_obj.call_count)  # 1

MagicMock - マジックメソッドも扱える

test_magic_mock.py 
from unittest.mock import MagicMock

# MagicMock はマジックメソッドをサポート
magic = MagicMock()

# __len__ をモック（len() で使える）
magic.__len__.return_value = 5
assert len(magic) == 5

# __str__ をモック（str() で使える）
magic.__str__.return_value = "偽のオブジェクト"
assert str(magic) == "偽のオブジェクト"

# コンテキストマネージャとしても使える
with MagicMock() as mock_file:
    mock_file.read.return_value = "ファイルの内容"
    content = mock_file.read()
    assert content == "ファイルの内容"
メソッド 説明
assert_called() 少なくとも1回呼ばれたことを確認
assert_called_once() ちょうど1回だけ呼ばれたことを確認
assert_called_with(*args, **kwargs) 最後の呼び出しが指定した引数であることを確認
assert_called_once_with(*args, **kwargs) 1回だけ・指定した引数で呼ばれたことを確認
assert_not_called() 1回も呼ばれていないことを確認
call_count 呼ばれた回数（属性）
call_args_list 全呼び出しの引数リスト（属性）

重要ポイント：MagicMock が推奨される理由

実際のコードを書くときはほとんどの場合MagicMockを使います。Mockと機能的に上位互換であり、ファイルオブジェクトやイテレータなどマジックメソッドを持つオブジェクトも自然にモックできるからです。

10.3 patch デコレータ - 実際のオブジェクトを置き換える

モックの核心は「実際に使われているオブジェクトを偽物に差し替える」ことです。unittest.mock.patchを使うと、テスト実行中だけ特定のモジュールや関数を置き換えることができます。

patch の3つの使い方

1

デコレータとして使う

テスト関数またはクラス全体に適用する最もよく使われる方法です。

test_patch_decorator.py 
import requests
from unittest.mock import patch, MagicMock

def get_weather(city):
    """天気APIを呼び出す関数（テスト対象）"""
    response = requests.get(f"https://api.weather.com/{city}")
    return response.json()["temperature"]

# @patch で requests.get を置き換える
@patch("requests.get")
def test_get_weather(mock_get):
    # 偽の応答オブジェクトを設定
    mock_response = MagicMock()
    mock_response.json.return_value = {"temperature": 25}
    mock_get.return_value = mock_response

    # 実際のAPIを呼ばずにテストできる
    result = get_weather("tokyo")
    assert result == 25

    # requests.get が正しいURLで呼ばれたか確認
    mock_get.assert_called_once_with("https://api.weather.com/tokyo")
2

コンテキストマネージャとして使う（with 文）

テスト関数の中の一部だけ置き換えたいときに便利です。

test_patch_context.py 
from unittest.mock import patch, MagicMock

def test_get_weather_context():
    # with 文の中だけ requests.get が置き換わる
    with patch("requests.get") as mock_get:
        mock_response = MagicMock()
        mock_response.json.return_value = {"temperature": 20}
        mock_get.return_value = mock_response

        result = get_weather("osaka")
        assert result == 20

    # with 文を抜けると元の requests.get に戻る
3

side_effect で例外や複数の戻り値を設定

エラーケースのテストにはside_effectを使います。

test_patch_side_effect.py 
import pytest
import requests
from unittest.mock import patch

@patch("requests.get")
def test_get_weather_network_error(mock_get):
    # side_effect に例外クラスを設定すると、呼び出し時に例外が発生
    mock_get.side_effect = requests.exceptions.ConnectionError("接続エラー")

    with pytest.raises(requests.exceptions.ConnectionError):
        get_weather("tokyo")

@patch("requests.get")
def test_retry_logic(mock_get):
    # リストを渡すと呼び出しごとに順番に返す
    mock_get.side_effect = [
        requests.exceptions.Timeout,  # 1回目: タイムアウト
        MagicMock(json=lambda: {"temperature": 25}),  # 2回目: 成功
    ]

patch のパス指定：「使われている場所」を指定する

patch のパスは「定義されている場所」ではなく「使われている場所」を指定します。
例: weather.py で import requests している場合は patch("weather.requests.get") ではなく patch("requests.get") を使います（インポートの仕方に応じて変わります）。

10.4 実践例 - 外部API・DB・ファイルI/Oのモック

実務でよく遭遇する3つのパターン（外部API・データベース・ファイルI/O）のモック方法を見ていきましょう。

パターン1: 外部APIのモック

test_api_mock.py（モジュール構成を含む） 
# === テスト対象コード（user_service.py）===
import requests

def fetch_user(user_id):
    """外部の User API からユーザー情報を取得"""
    response = requests.get(f"https://api.example.com/users/{user_id}")
    if response.status_code == 200:
        return response.json()
    return None

# === テストコード ===
from unittest.mock import patch, MagicMock
import pytest

@patch("requests.get")
def test_fetch_user_success(mock_get):
    """正常系: ユーザーが見つかる場合"""
    mock_response = MagicMock()
    mock_response.status_code = 200
    mock_response.json.return_value = {"id": 1, "name": "田中太郎"}
    mock_get.return_value = mock_response

    user = fetch_user(1)
    assert user["name"] == "田中太郎"

@patch("requests.get")
def test_fetch_user_not_found(mock_get):
    """異常系: ユーザーが存在しない場合"""
    mock_response = MagicMock()
    mock_response.status_code = 404
    mock_get.return_value = mock_response

    user = fetch_user(999)
    assert user is None

パターン2: データベースアクセスのモック

test_db_mock.py 
from unittest.mock import patch, MagicMock

class UserRepository:
    """DBアクセスを担うリポジトリクラス"""
    def __init__(self, db_connection):
        self.conn = db_connection

    def find_by_id(self, user_id):
        cursor = self.conn.cursor()
        cursor.execute("SELECT * FROM users WHERE id = ?", (user_id,))
        return cursor.fetchone()

def test_find_user_by_id():
    # DBコネクションをモック
    mock_conn = MagicMock()
    mock_cursor = MagicMock()
    mock_conn.cursor.return_value = mock_cursor
    mock_cursor.fetchone.return_value = {"id": 1, "name": "佐藤花子"}

    repo = UserRepository(mock_conn)
    user = repo.find_by_id(1)

    # 正しいSQLが実行されたか検証
    mock_cursor.execute.assert_called_once_with(
        "SELECT * FROM users WHERE id = ?", (1,)
    )
    assert user["name"] == "佐藤花子"

パターン3: ファイルI/Oのモック

test_file_mock.py 
from unittest.mock import patch, mock_open

def read_config(filepath):
    """設定ファイルを読み込む関数"""
    with open(filepath, "r") as f:
        return f.read()

def test_read_config():
    # mock_open で open() をモック
    fake_content = "debug=true\nport=8080"
    with patch("builtins.open", mock_open(read_data=fake_content)):
        result = read_config("config.ini")
        # 実際のファイルを読まずにテストできる
        assert result == "debug=true\nport=8080"

10.5 monkeypatch - pytestのフィクスチャでモックする

pytestにはmonkeypatchという組み込みフィクスチャが用意されています。unittest.mock.patchより直感的な書き方ができ、テスト終了後に自動的に元の状態に戻してくれます。

メソッド 用途
monkeypatch.setattr(obj, name, value) オブジェクトの属性や関数を置き換える
monkeypatch.delattr(obj, name) オブジェクトの属性を削除する
monkeypatch.setenv(name, value) 環境変数を設定する
monkeypatch.delenv(name) 環境変数を削除する
monkeypatch.syspath_prepend(path) sys.path にパスを追加する
monkeypatch.chdir(path) カレントディレクトリを変更する
test_monkeypatch.py 
import os
import requests

# テスト対象: 環境変数からAPIキーを読み込む
def get_api_key():
    key = os.environ.get("API_KEY")
    if not key:
        raise ValueError("API_KEY 環境変数が設定されていません")
    return key

# テスト対象: 外部サービスを呼び出す
def send_notification(message):
    response = requests.post("https://api.notify.example.com/send",
                             json={"message": message})
    return response.status_code == 200

# monkeypatch で環境変数をモック
def test_get_api_key(monkeypatch):
    monkeypatch.setenv("API_KEY", "test-key-12345")
    key = get_api_key()
    assert key == "test-key-12345"

# monkeypatch で関数をモック
def test_send_notification(monkeypatch):
    def fake_post(url, **kwargs):
        # 偽の応答オブジェクトを返す
        class FakeResponse:
            status_code = 200
        return FakeResponse()

    # requests.post を偽の関数に置き換え
    monkeypatch.setattr(requests, "post", fake_post)
    result = send_notification("テスト通知")
    assert result is True

monkeypatch と unittest.mock.patch の使い分け

観点 monkeypatch unittest.mock.patch
書き方 pytest らしいシンプルな書き方 デコレータ・コンテキストマネージャ
呼び出し検証 別途 Mock を使う必要あり Mock の assert メソッドで直接検証
環境変数操作 setenv/delenv で簡単 patch.dict(os.environ, ...) が必要
自動リセット テスト終了後に自動復元 デコレータ・with 文範囲で自動復元

10.6 モックのアンチパターンと注意点

モックは強力な道具ですが、使いすぎるとテストの価値が下がります。モックを使うべき場面と使いすぎに注意すべき場面を理解しましょう。

モックを使うべき場面

  • 外部APIやネットワーク通信
  • データベースの読み書き（単体テスト時）
  • メール送信・SMS送信などの副作用
  • 現在時刻・ランダム値などの非決定的な値
  • テスト環境では用意できないハードウェア

モックの使いすぎに注意

  • テスト対象コード自体をモックにしてしまう（何もテストしていない）
  • 内部実装の詳細に依存しすぎる（リファクタリングでテストが壊れやすい）
  • 単純な関数まで全部モックにする（過剰テスト）
  • モックと実際の動作が乖離して、本番バグを見逃す

実践例: 現在時刻のモック

「今日の日付」に依存する関数はテストが難しい典型例です。モックを使って日付を固定しましょう。

test_datetime_mock.py 
from datetime import date
from unittest.mock import patch

def is_weekend():
    """今日が週末かどうかを返す関数"""
    today = date.today()
    return today.weekday() >= 5  # 5=土曜, 6=日曜

def test_is_weekend_saturday():
    """土曜日のテスト"""
    # date.today() が 2024/1/6（土曜）を返すようにモック
    with patch("__main__.date") as mock_date:
        mock_date.today.return_value = date(2024, 1, 6)
        assert is_weekend() is True

def test_is_not_weekend_monday():
    """月曜日のテスト"""
    with patch("__main__.date") as mock_date:
        mock_date.today.return_value = date(2024, 1, 8)  # 月曜
        assert is_weekend() is False

実習 10-1: 天気通知サービスのテストを書く

外部APIと通知メールの送信を含む「天気通知サービス」にモックを使ったテストを実装します。実際のAPI・メール送信を行わずにロジックを検証します。

手順

  1. 1 以下のコードを weather_notifier.py として保存してください。
  2. 2 test_weather_notifier.py を作成し、@patch を使って各メソッドをモックするテストを書いてください。
  3. 3 「気温が30度以上の場合だけ通知する」というロジックを、正常系・境界値（ちょうど30度）・異常系（API失敗）の3パターンでテストしてください。
weather_notifier.py（実習用コード） 
import requests
import smtplib

NOTIFY_THRESHOLD = 30  # 通知する気温のしきい値

def get_temperature(city):
    """天気APIから気温を取得"""
    response = requests.get(f"https://api.weather.example.com/{city}")
    data = response.json()
    return data["temperature"]

def send_alert_email(city, temperature):
    """高温アラートメールを送信"""
    with smtplib.SMTP("smtp.example.com") as server:
        server.sendmail(
            "alert@example.com",
            "user@example.com",
            f"Subject: 高温アラート\n{city}の気温が{temperature}度です！"
        )

def notify_if_hot(city):
    """気温が閾値以上なら通知を送る"""
    temperature = get_temperature(city)
    if temperature >= NOTIFY_THRESHOLD:
        send_alert_email(city, temperature)
        return True
    return False

ヒント

  • • @patch("weather_notifier.get_temperature") のようにモジュール名を前置します
  • • send_alert_email が呼ばれたかどうかを assert_called_once() で検証しましょう
  • • 複数の @patch をスタックするときは、引数の順序が下から上になることに注意！

第10章のまとめ

学んだこと

  • モックとは外部依存を偽物に置き換え、テストを安定・高速化する技術
  • Mock・MagicMockで戻り値や例外をシミュレートし、呼び出しも検証できる
  • patchでテスト中だけ特定の関数・クラスを差し替えられる
  • side_effectで例外の発生や複数の戻り値のシミュレーションができる

  • モックは適切に使えば強力だが、使いすぎると保守性が下がる

理解度確認クイズ

  1. Q1. テストで外部APIを実際に呼び出すことの問題点を2つ挙げてください。

    回答を表示

    代表的な問題点: ネットワーク障害でテストが失敗する（不安定）、テスト実行が遅くなる、副作用が発生する（本番データを変更するリスク）、APIの使用制限に達してしまうなどが挙げられます。

  2. Q2. MockとMagicMockの違いは何ですか？

    回答を表示

    MagicMock は Mock のサブクラスで、__len__、__str__、__enter__/__exit__などのマジックメソッドもサポートしています。ファイルオブジェクトやコンテキストマネージャをモックする場合は MagicMock を使います。実務では MagicMock を使うことが多いです。

  3. Q3. @patch("module.function") のパス指定で正しいのはどちらですか？「関数が定義されている場所」か「関数が使われている場所」か答えてください。

    回答を表示

    「関数が使われている場所（インポートされている場所）」を指定します。たとえば weather.py が import requests して requests.get を呼んでいる場合、@patch("requests.get") を使います。

  4. Q4. side_effect を使うのはどのような場合ですか？

    回答を表示

    主に2つの場合です。1. 例外を発生させたいとき（mock.side_effect = ValueError("エラー")）と、2. 呼び出しごとに異なる値を返したいとき（mock.side_effect = [値1, 値2, 値3]）です。リトライロジックや段階的な処理のテストに役立ちます。
