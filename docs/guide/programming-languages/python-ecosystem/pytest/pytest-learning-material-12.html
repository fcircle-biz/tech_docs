テストカバレッジ - テストの網羅率を可視化する

pytest-covを使ったコードカバレッジの計測方法を学びます。ステートメントカバレッジとブランチカバレッジの違い、カバレッジレポートの読み方、カバレッジ目標の設定方法、そしてカバレッジが高いだけでは十分でない理由を理解します。

この章で学ぶこと

  • テストカバレッジ（コードカバレッジ）とは何かを説明できる
  • pytest-covをインストールし、カバレッジを計測・レポート出力できる
  • ステートメントカバレッジとブランチカバレッジの違いを理解する
  • カバレッジレポート（ターミナル・HTML）の読み方を身につける
  • 「カバレッジが高い = テストが良い」ではない理由を説明できる

12.1 テストカバレッジとは何か

テストをたくさん書いても、「コードのどこがテストされていて、どこがされていないのか」を把握することは難しいです。テストカバレッジは、テストがコードのどの割合を実行したかを数値で示す指標です。

たとえば「カバレッジ80%」は、プロダクションコード全体のうち80%の行がテスト実行中に少なくとも1回実行されたことを意味します。残りの20%は一度も実行されていない「未テスト領域」です。

カバレッジを計測する意義

カバレッジを計測することで「テストが通過しているから大丈夫」という思い込みを排除できます。一度も実行されていないコードは、バグが潜んでいても気づけない危険な領域です。

12.2 pytest-covのインストールと基本的な使い方

pytestでカバレッジを計測するには、pytest-covプラグインを使います。これはPythonの標準カバレッジツールであるcoverage.pyをpytestに統合したプラグインです。

インストール

ターミナル 
# pytest-covをインストール
pip install pytest-cov

# インストール確認
pip show pytest-cov

基本的なカバレッジ計測コマンド

ターミナル 
# テスト実行とカバレッジ計測（myapp フォルダが対象）
pytest --cov=myapp

# ターミナルに詳細を表示（テストされていない行番号を表示）
pytest --cov=myapp --cov-report=term-missing

# HTMLレポートを生成（htmlcov/index.html が作成される）
pytest --cov=myapp --cov-report=html

# ターミナル表示とHTML生成を同時に行う
pytest --cov=myapp --cov-report=term-missing --cov-report=html

--cov オプションで対象フォルダを指定する

--cov に何も指定しないと、テストコード自体もカバレッジ対象に含まれて数値が水増しされます。必ずプロダクションコードのフォルダ名（例: myapp、src）を指定しましょう。

12.3 カバレッジレポートの読み方

--cov-report=term-missingを使うと、ターミナルにカバレッジの詳細が表示されます。このレポートの読み方を理解しましょう。

ターミナルレポートの例

ターミナル出力例 
---------- coverage: platform linux, python 3.11 -----------
Name                 Stmts   Miss  Cover   Missing
----------------------------------------------------
myapp/__init__.py        0      0   100%
myapp/calculator.py     20      4    80%   15, 22-24
myapp/utils.py          10      2    80%   8, 12
----------------------------------------------------
TOTAL                   30      6    80%
列名 意味
Name 計測対象のファイル名
Stmts ファイル内のステートメント（実行可能な文）の総数
Miss テストで一度も実行されなかったステートメントの数
Cover カバレッジ率（(Stmts - Miss) / Stmts × 100）
Missing テストが通過していない行番号（ここを優先的にテスト追加）

上の例では myapp/calculator.py の15行目と22〜24行目がテストされていません。この行番号をもとに、エディタでソースコードを開いてどのロジックがカバーされていないか確認します。

HTMLレポートの活用

--cov-report=htmlを指定すると、htmlcov/index.htmlにビジュアルなレポートが生成されます。ブラウザで開くと、コードが色分けされて表示されます。

緑色の行

テストで実行された行（カバー済み）

赤色の行

一度も実行されなかった行（未カバー）

黄色の行

部分的にカバーされた行（ブランチの片方のみ）

12.4 ステートメントカバレッジとブランチカバレッジ

カバレッジには計測の粒度（レベル）があります。デフォルトはステートメントカバレッジですが、より厳密なブランチカバレッジもあります。

ステートメントカバレッジ（行カバレッジ）

各「行（ステートメント）」が少なくとも1回実行されたかどうかを計測します。最も基本的なカバレッジです。

calculator.py 
def classify_score(score):
    """スコアを分類する関数"""
    if score >= 80:
        return "合格"        # この行だけテストされると...
    else:
        return "不合格"      # この行はテストされない

score=90でテストすると、return "合格"の行が実行され、if score >= 80:も実行されます。ステートメントカバレッジでは「if文の行は実行済み」と判定されますが、elseブランチは未テストです。

ブランチカバレッジ

分岐（if/else、while、try/exceptなど）のすべての経路が実行されたかを計測します。ステートメントカバレッジよりも厳密で、見落としを減らせます。

ブランチカバレッジが検出する未テスト経路

flowchart TD A["classify_score(score)"] --> B{"score >= 80?"} B -->|"True（テスト済み）"| C["return 合格"] B -->|"False（未テスト！）"| D["return 不合格"] style C fill:#d1fae5,stroke:#10b981,color:#065f46 style D fill:#fee2e2,stroke:#ef4444,color:#7f1d1d
ターミナル（ブランチカバレッジを有効化） 
# --cov-branch オプションでブランチカバレッジを計測
pytest --cov=myapp --cov-branch --cov-report=term-missing
種類 計測対象 厳密さ 推奨用途
ステートメント 各行が実行されたか 低め 最初の導入・目安として
ブランチ すべての分岐経路が実行されたか 高め ロジックが複雑なコード・本番システム

重要ポイント：ブランチカバレッジを習慣にする

実務では--cov-branchを常用することを推奨します。if/elseやtry/exceptのある処理で、テストが片方の経路しか通過していないケースを自動で検出できます。

12.5 カバレッジ目標の設定 - --cov-fail-under

--cov-fail-underオプションを使うと、カバレッジが指定した値を下回った場合にpytestをエラー終了させることができます。CI/CDパイプラインに組み込むことで、カバレッジの低下を自動的に検出できます。

ターミナル 
# カバレッジが80%を下回った場合にエラー終了（exit code 2）
pytest --cov=myapp --cov-report=term-missing --cov-fail-under=80
ターミナル出力（カバレッジ不足の場合） 
FAIL Required test coverage of 80% not reached. Total coverage: 75.00%

設定ファイルでの目標値管理

毎回コマンドにオプションを付けるのは手間がかかります。.coveragercファイルかpyproject.tomlに設定を書くと、コマンドを簡略化できます。

pyproject.toml 
[tool.pytest.ini_options]
# pytestのデフォルトオプション
addopts = "--cov=myapp --cov-report=term-missing --cov-branch --cov-fail-under=80"

[tool.coverage.run]
# カバレッジ計測の設定
branch = true          # ブランチカバレッジを有効化
source = ["myapp"]     # 対象フォルダ

[tool.coverage.report]
# 除外するファイルのパターン
omit = [
    "myapp/__init__.py",   # __init__.py は除外
    "tests/*",             # テストフォルダ自体は除外
]

設定ファイルを置いておけば、pytestを実行するだけで自動的にカバレッジが計測されます。

12.6 カバレッジが高くても安心できない理由

「カバレッジ100%を達成すれば、すべてのバグを発見できる」と思いたくなりますが、これは誤りです。カバレッジは「コードが実行されたか」を示すだけで、「コードが正しく動くか」は別の問題です。

問題のある例：アサーションのないテスト

test_bad_coverage.py（問題のあるテスト） 
def test_calculate_total():
    """カバレッジ100%でも全く意味がないテスト"""
    result = calculate_total([10, 20, 30])
    # assert を書き忘れている！
    # コードは実行される（カバレッジに貢献）が、
    # 正しいかどうかを全く確認していない

カバレッジは「品質の保証」ではなく「未テスト領域の発見ツール」

アサーションのないテストはカバレッジを100%にできますが、バグを一切検出しません。カバレッジはあくまで「どこがテストされていないか」を見つける道具であり、「テストが正しいか」は別途レビューが必要です。

カバレッジの限界

カバレッジが検出できないもの

  • アサーションが間違っている（assert 1 == 1 のような意味のないテスト）
  • テストケースの漏れ（境界値・エッジケースの未確認）
  • テスト対象のビジネスロジック自体が間違っている場合

カバレッジが得意なこと

  • 「一度も実行されていないコード」の可視化
  • デッドコード（到達不能なコード）の発見
  • テストが不足しているモジュールの特定

カバレッジへの健全な向き合い方

カバレッジは「80%以上を維持する」など最低ラインを設ける道具として使い、「100%を目指す」ことに執着しないことが大切です。質の高いアサーションを持つテストを、重要なロジックに集中して書くことが本質です。

12.7 特定コードをカバレッジ対象から除外する

デバッグ用コードや、テストが現実的でない箇所（メインの起動コードなど）はカバレッジから除外できます。# pragma: no coverコメントを使います。

main.py 
def run_app():
    """アプリケーション起動"""
    print("Starting application...")

if __name__ == "__main__":  # pragma: no cover
    # スクリプト直接実行時のみ動くコードはテストが難しい
    run_app()

# pragma: no coverを付けた行は、カバレッジの計算から除外されます。ただし、乱用するとカバレッジが実態を反映しなくなるため、本当にテストが困難な箇所だけに限定しましょう。

除外が適切なケースの例

if __name__ == "__main__":ブロック、TYPE_CHECKINGブロック、コマンドライン引数のパーサー初期化など、テストより実行環境依存のコードが対象です。

実習 12-1: カバレッジを計測してみよう

簡単な関数を作成し、pytest-covでカバレッジを計測してみましょう。意図的にテストが不足している状態からスタートし、カバレッジレポートを見ながらテストを追加していきます。

手順

  1. 1 以下の grader.py を作成し、myapp/ フォルダに保存します
  2. 2 最初は合格ケースだけのテストを書き、pytest --cov=myapp --cov-report=term-missing を実行してカバレッジを確認します
  3. 3 Missing に表示された行番号を確認し、不足しているテストを追加して100%を目指します
  4. 4 --cov-branch を追加して、ブランチカバレッジの違いも確認してみましょう
myapp/grader.py（テスト対象コード） 
def grade(score):
    """スコアを成績に変換する"""
    if score < 0 or score > 100:
        raise ValueError("スコアは0〜100の範囲で指定してください")
    if score >= 90:
        return "A"
    elif score >= 70:
        return "B"
    elif score >= 50:
        return "C"
    else:
        return "D"

考え方のヒント

  • • 最初は grade(90) のみのテストからスタートし、カバレッジを確認してみましょう
  • • "Missing" 列に表示された行番号のコードが何をしているか確認し、テストを追加します
  • • 最終的には A/B/C/D すべての評価と、範囲外エラーのテストが必要です

動作確認

以下のコマンドを実行し、カバレッジレポートが正しく出力されることを確認してください。

ターミナル 
pytest --cov=myapp --cov-branch --cov-report=term-missing --cov-report=html

期待される結果 

---------- coverage: platform linux, python 3.11 -----------
Name                Stmts   Miss Branch BrPart  Cover   Missing
----------------------------------------------------------------
myapp/grader.py        10      0      8      0   100%
----------------------------------------------------------------
TOTAL                  10      0      8      0   100%

HTMLレポートは htmlcov/index.html に生成されます。ブラウザで開いて行ごとのカバレッジ状況を確認してみましょう。

  • ブランチカバレッジはステートメントカバレッジより厳密で、分岐の全経路を確認できる

理解度確認クイズ

  1. Q1. pytest-covでターミナルにカバレッジを表示し、テストされていない行番号も確認するコマンドはどれですか？

    回答を表示

    pytest --cov=myapp --cov-report=term-missing です。term-missingを指定することで、テストされていない行番号が "Missing" 列に表示されます。

  2. Q2. ステートメントカバレッジとブランチカバレッジの違いを説明してください。

    回答を表示

    ステートメントカバレッジは各行が少なくとも1回実行されたかを計測します。ブランチカバレッジはif/elseなど分岐のすべての経路（True側・False側の両方）が実行されたかを計測します。ブランチカバレッジの方がより厳密で、片方の分岐しかテストしていないケースを検出できます。

  3. Q3. カバレッジが90%あれば十分な品質が保証されていると言えますか？理由も答えてください。

    回答を表示

    いいえ、品質の保証にはなりません。カバレッジはコードが「実行された」かを示すだけで、「正しく動くか」は別の問題です。アサーションのないテストや、意味のないアサーション（assert 1 == 1 など）でもカバレッジを上げることができます。カバレッジは未テスト領域を発見するためのツールであり、テスト内容の質は別途確認が必要です。

  4. Q4. カバレッジ計測から特定の行を除外するには何と書きますか？

    回答を表示

    行末に # pragma: no cover コメントを付けます。この行以降（ブロック全体）がカバレッジの計算から除外されます。if __name__ == "__main__": のようなテストしにくいコードに使用します。
