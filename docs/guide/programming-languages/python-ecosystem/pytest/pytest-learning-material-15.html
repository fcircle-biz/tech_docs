CI/CDとの連携 - テスト自動化の仕組みを作る

GitHub ActionsやGitLab CIを例に、pytestをCI/CDパイプラインに組み込む方法を学びます。プッシュ時の自動テスト実行、カバレッジレポートの自動生成、テスト失敗時の通知設定など、チーム開発でテストを継続的に活用する仕組みを構築します。

この章で学ぶこと

  • CI/CDの基本概念と、テスト自動化をパイプラインに組み込む意義を説明できる
  • GitHub Actionsでpytestを自動実行するワークフローファイルを作成できる
  • GitLab CI/CDの基本的なパイプライン設定を理解する
  • カバレッジレポートの自動生成とPRへのコメント通知を設定できる
  • 複数Pythonバージョンでのマトリックステストを実現できる

15.1 CI/CDとは何か

第14章までで、pytestを使ってテストを書くスキルを身につけました。しかし、せっかく書いたテストも「誰かが忘れずにコマンドを実行する」依存では、チーム開発で機能しません。この章では、テストを自動的に・継続的に実行する仕組み「CI/CD」を学びます。

CI（継続的インテグレーション）

Continuous Integrationの略。コードをリポジトリにプッシュするたびに、自動的にテストやビルドを実行する仕組みです。

  • コードの統合問題を早期に発見
  • テスト忘れを防止
  • チームの品質基準を自動で担保

CD（継続的デリバリー/デプロイ）

Continuous Delivery / Deploymentの略。テストを通過したコードを、自動的に本番環境などへデプロイする仕組みです。

  • リリース作業の自動化
  • 人的ミスの排除
  • 頻繁な小さなリリースを実現

なぜCIにテストを組み込むのか

ローカルでテストを書いても「実行を忘れる」「自分の環境では動くが他の環境で動かない」という問題が発生します。CIに組み込むことで、全てのコード変更に対して、統一された環境でテストが自動実行されます。これにより「テスト忘れ」と「環境依存の問題」を同時に解決できます。

15.2 GitHub Actionsの基本概念

GitHub Actionsは、GitHubに組み込まれたCI/CDプラットフォームです。GitHubのリポジトリを使っている場合、追加料金なし（一定制限内）で使用できます。設定はYAMLファイルで行い、リポジトリの.github/workflows/ディレクトリに置きます。

W

ワークフロー（Workflow）

自動化のまとまりの単位。YAMLファイル1つがワークフロー1つに対応します。「どんなイベントが起きたとき」「どんな処理を実行するか」を定義します。

J

ジョブ（Job）

ワークフロー内の処理の塊。独立した仮想マシン上で実行されます。複数のジョブを並列または順番に実行できます。

S

ステップ（Step）

ジョブ内の1つ1つの処理。コマンドを実行したり、公開されているアクション（再利用可能な処理の単位）を使ったりします。

E

イベント（Event）

ワークフローを実行するきっかけ。「プッシュ時」「プルリクエスト時」「手動実行時」「毎日決まった時刻に」など様々なトリガーを設定できます。

15.3 基本的なGitHub Actionsワークフローの作成

実際にpytestを自動実行するワークフローファイルを作成します。プロジェクトのルートに.github/workflows/ディレクトリを作り、YAMLファイルを配置します。

プロジェクト構成 

myproject/
├── .github/
│   └── workflows/
│       └── test.yml      ← ここに作成
├── src/
│   └── calculator.py
├── tests/
│   └── test_calculator.py
├── requirements.txt
└── pyproject.toml

最小限のワークフロー設定

.github/workflows/test.yml（基本版） 
# ワークフローの名前
name: Python Tests

# 実行タイミング（いつ動かすか）
on:
  push:               # プッシュ時
    branches: [ main ]
  pull_request:       # PR作成・更新時
    branches: [ main ]

# ジョブの定義
jobs:
  test:
    # 実行環境（GitHub提供のUbuntu）
    runs-on: ubuntu-latest

    steps:
      # ステップ1: コードを取得
      - name: リポジトリをチェックアウト
        uses: actions/checkout@v4

      # ステップ2: Pythonをセットアップ
      - name: Pythonをセットアップ
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      # ステップ3: 依存関係をインストール
      - name: 依存関係をインストール
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt
          pip install pytest pytest-cov

      # ステップ4: テストを実行
      - name: pytestでテストを実行
        run: pytest tests/ -v

YAMLの読み方のコツ

YAMLはインデント（字下げ）が重要です。同じ階層のものは同じ字下げにします。uses:は公開されているアクションを使う指定、run:はシェルコマンドを直接実行する指定です。

requirements.txtの準備

CIが依存関係をインストールできるよう、requirements.txtを用意します。

requirements.txt 
# 本番用の依存関係
requests==2.31.0

# テスト用の依存関係
pytest==7.4.0
pytest-cov==4.1.0

依存関係の分離のベストプラクティス

本番環境にはテスト用ライブラリ（pytest等）を入れないようにするため、requirements-dev.txtやpyproject.tomlの[tool.pytest.ini_options]で分離管理するのが一般的です。

15.4 カバレッジレポートの自動生成

CI上でカバレッジを計測し、結果をPull Requestのコメントとして自動投稿する方法を学びます。これにより、コードレビュー時にカバレッジの変化を即座に確認できます。

.github/workflows/test.yml（カバレッジ付き） 
name: Python Tests with Coverage

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Pythonをセットアップ
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: 依存関係をインストール
        run: |
          pip install pytest pytest-cov

      # カバレッジ付きでテストを実行しXMLレポートを生成
      - name: テストとカバレッジを実行
        run: |
          pytest tests/ -v \
            --cov=src \
            --cov-report=xml \
            --cov-report=term-missing

      # カバレッジレポートをPRコメントとして投稿
      - name: カバレッジをPRにコメント
        uses: codecov/codecov-action@v4
        with:
          token: ${{ secrets.CODECOV_TOKEN }}
          files: ./coverage.xml
          fail_ci_if_error: false

重要ポイント: ${{ secrets.CODECOV_TOKEN }}について

secrets.CODECOV_TOKENは、GitHubのリポジトリ設定のSecretsに登録した秘密情報です。APIキーやパスワードはコードに直接書かず、必ずSecretsに登録して参照します。Codecovのサイトでアカウント登録後にトークンを取得できます。

カバレッジ閾値でCIを失敗させる

カバレッジが一定以下の場合にCIを失敗させて、マージをブロックすることもできます。

カバレッジ閾値の設定（pyproject.toml） 
[tool.pytest.ini_options]
addopts = "--cov=src --cov-fail-under=80"
# カバレッジが80%未満の場合、テストが失敗扱いになる

15.5 マトリックステスト - 複数バージョンで同時に確認

Pythonのバージョン間で動作が異なる場合があります。GitHub Actionsのマトリックス機能を使うと、複数のPythonバージョンでテストを並列実行できます。ライブラリを公開する場合や、様々な環境で使われるアプリケーションには特に有効です。

.github/workflows/test.yml（マトリックス版） 
name: Python Tests Matrix

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest

    # マトリックス: 複数バージョンで並列実行
    strategy:
      matrix:
        python-version: ['3.9', '3.10', '3.11', '3.12']
      # 1つが失敗しても他を継続実行
      fail-fast: false

    steps:
      - uses: actions/checkout@v4

      # matrix.python-versionで各バージョンを参照
      - name: Python ${{ matrix.python-version }} をセットアップ
        uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}

      - name: 依存関係をインストール
        run: pip install pytest

      - name: テストを実行
        run: pytest tests/ -v

15.6 GitLab CI/CDへの適用

GitHubの代わりにGitLabを使っているプロジェクトでは、GitLab CI/CDを使います。設定ファイルはリポジトリのルートに.gitlab-ci.ymlという名前で配置します。

.gitlab-ci.yml（基本版） 
# 使用するDockerイメージ
image: python:3.11

# ステージの順番を定義
stages:
  - test
  - coverage

# テストジョブ
pytest:
  stage: test
  before_script:
    - pip install pytest pytest-cov
  script:
    - pytest tests/ -v --cov=src --cov-report=xml
  # テスト結果をGitLabのUIで確認可能にする
  artifacts:
    reports:
      coverage_report:
        coverage_format: cobertura
        path: coverage.xml
    # 失敗しても成果物を保持
    when: always
  # MRとpushの両方で実行
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
    - if: '$CI_COMMIT_BRANCH == "main"'

処理のグループ化 jobs → steps stages → jobs
マーケットプレイス GitHub Marketplace（大量のアクション） テンプレートは少なめ

15.7 本格的なワークフロー - 実務で使える設定

実務では、テスト実行の効率化のためにキャッシュを活用したり、テスト結果をより詳細に管理したりします。実務に近い形の設定例を見てみましょう。

.github/workflows/test.yml（実務に近い設定） 
name: CI - Test & Coverage

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ['3.10', '3.11']

    steps:
      - uses: actions/checkout@v4

      - name: Python ${{ matrix.python-version }} をセットアップ
        uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}

      # pip キャッシュで高速化（2回目以降のインストールが速くなる）
      - name: pipキャッシュを設定
        uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: ${{ runner.os }}-pip-${{ hashFiles('**/requirements*.txt') }}
          restore-keys: |
            ${{ runner.os }}-pip-

      - name: 依存関係をインストール
        run: |
          pip install --upgrade pip
          pip install -r requirements.txt
          pip install pytest pytest-cov pytest-html

      # 詳細なJUnit形式のテスト結果を出力
      - name: テストを実行
        run: |
          pytest tests/ -v \
            --cov=src \
            --cov-report=xml \
            --cov-report=html \
            --junitxml=test-results.xml

      # テスト結果をGitHub ActionsのUIに表示
      - name: テスト結果を公開
        uses: actions/upload-artifact@v4
        if: always()           # 失敗時も成果物を保存
        with:
          name: test-results-${{ matrix.python-version }}
          path: |
            test-results.xml
            htmlcov/

キャッシュでCI実行時間を短縮

actions/cacheを使うと、毎回のpipインストールをスキップして前回のキャッシュを再利用できます。依存関係が多いプロジェクトでは、CI実行時間を数分から数十秒に短縮できることもあります。hashFilesでrequirements.txtの変更を検知し、変更があった場合のみ再インストールします。

PRのステータスチェックとしてのCI

GitHubでは、CIの結果をPull Requestのマージ条件として設定できます。「全てのテストが通らないとマージできない」という設定により、品質を自動で担保します。

PRページでの表示イメージ

15.8 テスト実行戦略 - CIを効率よく使う

CIでテストを実行する際は、テストの速度と網羅性のバランスを考えることが重要です。テスト数が増えると実行時間も増えるため、適切な戦略を立てます。

高速チェック

PR作成時に単体テストだけを実行。数十秒で完了し、基本的なバグを即座に発見します。

pytest -m "not slow"

フルテスト

mainブランチへのマージ時に全テストを実行。結合テストや時間のかかるテストも含めた完全な検証。

pytest tests/ --cov

定期実行

毎日深夜に外部APIとの結合テストを実行。外部サービスの変更によるバグを定期的に検出。

schedule: cron

スケジュール実行の設定例

定期実行ワークフロー 
on:
  # 毎日午前2時（UTC）に実行
  schedule:
    - cron: '0 2 * * *'
  # 手動でも実行可能にする
  workflow_dispatch:

jobs:
  nightly-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install pytest
      # 外部依存を含む全テストを実行
      - run: pytest tests/ -v -m "integration"

pytestのマーカーとCIの組み合わせ

第9章で学んだ@pytest.markを使ってテストを分類することで、CI上での実行対象を柔軟に切り替えられます。@pytest.mark.slowや@pytest.mark.integrationのようなカスタムマーカーを活用しましょう。

15.9 通知とレポートの設定

テストが失敗したとき、関係者に素早く伝えることが重要です。SlackやTeamsへの通知設定を組み込むことで、失敗を即座に検知できます。

Slackへの通知設定

テスト失敗時のSlack通知 
      # テスト実行（失敗しても次のステップに進む）
      - name: テストを実行
        id: pytest
        run: pytest tests/ -v
        continue-on-error: true

      # テスト失敗時にSlackへ通知
      - name: Slack通知（失敗時のみ）
        if: steps.pytest.outcome == 'failure'
        uses: slackapi/slack-github-action@v1
        with:
          channel-id: 'C1234567890'
          slack-message: |
            テストが失敗しました!
            ブランチ: ${{ github.ref_name }}
            実行者: ${{ github.actor }}
            詳細: ${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}
        env:
          SLACK_BOT_TOKEN: ${{ secrets.SLACK_BOT_TOKEN }}

      # テスト結果に応じてジョブを成功/失敗させる
      - name: テスト結果を確定
        if: steps.pytest.outcome == 'failure'
        run: exit 1

通知の使いすぎに注意

通知が多すぎると「通知疲れ」が起きて、重要な失敗を見逃す原因になります。通知は「mainブランチのテスト失敗」や「夜間実行の失敗」など、本当に重要なものに絞ることをお勧めします。個人のPRのテスト失敗は、GitHubのUI確認で十分なことが多いです。

実習 15-1: GitHub Actionsでpytestを自動実行する

GitHubリポジトリにワークフローファイルを追加し、プッシュ時にpytestが自動実行されることを確認します。

手順

  1. 1 GitHubにサンプルPythonプロジェクト（簡単な関数とテスト）のリポジトリを用意する（または作成する）
  2. 2 .github/workflows/test.ymlファイルを作成し、基本的なワークフローを記述する
  3. 3 コミット&プッシュして、GitHubの「Actions」タブでワークフローが実行されることを確認する
  4. 4 意図的に失敗するテストを追加してプッシュし、CIが「失敗」を検知することを確認する
  5. 5 テストを修正してプッシュし、CIが再び「成功」になることを確認する

確認ポイント

  • • GitHubの「Actions」タブでワークフローの実行履歴が見えるか
  • • 各ステップの実行ログが確認できるか
  • • テスト失敗時にPRページにバツ印が表示されるか
  • • テスト成功時にPRページにチェックマークが表示されるか

動作確認 - CIのログ確認

トラブルシューティング

「ModuleNotFoundError」でテストが失敗する

原因: CIの仮想環境にインストールされていないモジュールをimportしている。requirements.txtに記載がないか、インストールステップに漏れがある。

解決方法: requirements.txtに必要なパッケージを全て記載し、インストールのステップでpip install -r requirements.txtを実行しているか確認する。

ローカルでは通るがCIでは失敗する

原因: 環境依存のコード（タイムゾーン、ファイルパスの区切り文字、環境変数）があるか、テストの順番依存がある。

解決方法: CIのログを詳細に確認し、ローカルとの環境差異を特定する。pytest-randomlyを使って実行順をランダム化し、順番依存を発見する。

CIの実行時間が長すぎる（10分以上かかる）

原因: 依存関係のインストールに毎回時間がかかっている、または外部APIを呼ぶ遅いテストが多い。

解決方法: actions/cacheでpipキャッシュを有効化する。外部依存のテストはモックに置き換えるか@pytest.mark.slowでマークして別ジョブに分ける。

YAMLの構文エラーでワークフローが動かない

原因: YAMLはインデントが厳格なため、スペースの数がずれると構文エラーになる。タブ文字の混入も原因になることがある。

解決方法: VS Codeの「YAML」拡張機能を使ってリアルタイムでエラーを確認する。GitHubのAction編集画面には構文チェッカーが内蔵されているので活用する。

  • CI（継続的インテグレーション）はコード変更のたびにテストを自動実行する仕組み
  • GitHub Actionsは.github/workflows/*.ymlで設定し、プッシュ時などに自動実行
  • マトリックス機能で複数のPythonバージョンでのテストを並列実行できる
  • カバレッジレポートを自動生成してPRにコメントする仕組みを構築できる
  • pipキャッシュでCI実行時間を短縮し、効率的なパイプラインを構築できる

第16章: 総合演習 - 実践プロジェクトでテストを書こう

理解度確認クイズ

  1. Q1. CI（継続的インテグレーション）とはどのような仕組みですか？

    回答を表示

    コードをリポジトリにプッシュするたびに、自動的にテストやビルドを実行する仕組みです。「テスト忘れ」を防ぎ、環境に依存しない統一された条件でテストを実行することで、コードの品質を継続的に担保します。

  2. Q2. GitHub Actionsのワークフローファイルはどこに置きますか？

    回答を表示

    リポジトリの.github/workflows/ディレクトリにYAMLファイル形式で配置します。例：.github/workflows/test.yml

  3. Q3. マトリックステストとは何ですか？どんな場合に有効ですか？

    回答を表示

    複数の環境（Pythonバージョン、OSなど）でテストを並列実行する機能です。strategy.matrixで設定します。ライブラリを公開する場合や、様々な環境で使われるアプリケーションの動作保証に特に有効です。

  4. Q4. CIでpipインストールを毎回行うのは時間がかかります。どうすれば高速化できますか？

    回答を表示

    actions/cacheアクションを使ってpipのキャッシュを有効化します。hashFiles('**/requirements*.txt')でキャッシュキーを生成し、requirements.txtが変わった場合のみ再インストールが行われます。変わっていない場合はキャッシュが再利用され、インストール時間が大幅に短縮されます。
