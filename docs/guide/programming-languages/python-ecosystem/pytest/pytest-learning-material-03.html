unittest入門と比較 - 標準ライブラリのテストフレームワーク

Python標準ライブラリのunittestの基本的な使い方を学び、pytestとの構文やアプローチの違いを比較します。TestCaseクラスの継承、assertメソッド群、setUp/tearDownなどの特徴を理解したうえで、pytestがなぜ広く選ばれているのかを実感します。

この章で学ぶこと

  • unittest.TestCaseクラスを継承したテストクラスの基本的な書き方を理解する
  • setUp / tearDown メソッドによるテストの前処理・後処理の仕組みを理解する
  • unittestの代表的なassertメソッド（assertEqual、assertRaises等）を使いこなせる
  • pytestとunittestの構文・設計思想の違いを具体的なコード例で比較できる
  • pytestがunittestのテストコードをそのまま実行できることを知る

3.1 unittestとは何か

unittestはPythonに標準で組み込まれているテストフレームワークです。追加のインストールは不要で、Pythonをインストールした瞬間から使えます。

unittestはJavaのテストフレームワーク「JUnit」を参考に設計されており、クラスベースのアプローチを採用しています。テストを書くにはunittest.TestCaseというクラスを継承したテストクラスを作る必要があります。

pytestはunittestで書かれたテストコードをそのまま実行できます。既存のunittestプロジェクトをpytestに移行する際も安心です。

なぜunittestを学ぶのか

実務では既存プロジェクトがunittestで書かれていることがあります。また、Djangoなどのフレームワークはデフォルトでunittestベースのテストを提供しています。pytestメインで学習しながらも、unittestの読み書きができると現場で役立ちます。

3.2 unittestの基本構造

unittestでテストを書くには、決まった「型」に従う必要があります。この型を理解することがunittestの第一歩です。

unittestテストの3つの必須要素

1

import unittest

標準ライブラリからunittestモジュールをインポートします。

2

class TestXxx(unittest.TestCase):

TestCaseクラスを継承したテストクラスを作ります。クラス名はTestで始める慣習があります。

3

def test_xxx(self):

テストメソッドはtest_で始める必要があります。引数にselfが必要です。

以下が最もシンプルなunittestの例です。テスト対象の関数（add関数）と、それをテストするコードを合わせて確認しましょう。

test_calculator_unittest.py 
import unittest

# テスト対象の関数（本来は別ファイルからimport）
def add(a, b):
    return a + b

def divide(a, b):
    if b == 0:
        raise ZeroDivisionError("0で割ることはできません")
    return a / b

# --- テストクラス ---
class TestCalculator(unittest.TestCase):

    def test_add_正の数(self):
        # assertEqual で値が等しいことを確認
        self.assertEqual(add(2, 3), 5)

    def test_add_負の数(self):
        self.assertEqual(add(-1, -2), -3)

    def test_divide_正常系(self):
        self.assertEqual(divide(10, 2), 5)

    def test_divide_ゼロ除算(self):
        # assertRaises で例外が発生することを確認
        with self.assertRaises(ZeroDivisionError):
            divide(10, 0)

# スクリプトとして直接実行する場合
if __name__ == '__main__':
    unittest.main()

実行方法

unittestのテストは python -m unittest test_calculator_unittest.py で実行します。pytestがインストールされていれば pytest test_calculator_unittest.py でも実行できます。

3.3 unittestのassertメソッド一覧

unittestでは、検証に専用のassertメソッドを使います。assert文を直接書くのではなく、状況に応じたメソッドを選ぶのがunittestの流儀です。

メソッド 確認する内容 使用例
assertEqual(a, b) a == b（最頻出） self.assertEqual(add(2,3), 5)
assertNotEqual(a, b) a != b self.assertNotEqual(add(1,2), 5)
assertTrue(x) x が True self.assertTrue(is_even(4))
assertFalse(x) x が False self.assertFalse(is_even(3))
assertIsNone(x) x is None self.assertIsNone(find_user(999))
assertIsNotNone(x) x is not None self.assertIsNotNone(find_user(1))
assertIn(a, b) a in b（リスト・文字列等） self.assertIn("hello", greet())
assertRaises(exc) 指定した例外が発生する with self.assertRaises(ValueError):
assertAlmostEqual(a, b) a ≒ b（浮動小数点比較） self.assertAlmostEqual(0.1+0.2, 0.3, places=7)
assertメソッドの使用例 
import unittest

class TestAssertMethods(unittest.TestCase):

    def test_assertEqual(self):
        # 最もよく使うassertメソッド
        self.assertEqual(1 + 1, 2)
        self.assertEqual("hello".upper(), "HELLO")

    def test_assertTrue_assertFalse(self):
        # 真偽値のチェック
        self.assertTrue(isinstance(42, int))
        self.assertFalse(isinstance("text", int))

    def test_assertIn(self):
        # リストや文字列に含まれているかチェック
        fruits = ["apple", "banana", "cherry"]
        self.assertIn("banana", fruits)
        self.assertNotIn("grape", fruits)

    def test_assertRaises(self):
        # 例外が発生することをチェック（コンテキストマネージャ形式）
        with self.assertRaises(ValueError):
            int("not_a_number")  # ValueError が発生するはず

    def test_assertAlmostEqual(self):
        # 浮動小数点の誤差を許容した比較
        self.assertAlmostEqual(0.1 + 0.2, 0.3, places=7)

3.4 setUp / tearDown - テストの前準備と後片付け

複数のテストで共通の「準備」が必要な場合、毎回同じコードを書くのは非効率です。unittestにはsetUpとtearDownという特別なメソッドが用意されています。

各テストメソッドの実行フロー

flowchart LR A(["テスト開始"]) --> B["setUp()実行
（前処理）"] B --> C["test_xxx()実行
（テスト本体）"] C --> D["tearDown()実行
（後処理）"] D --> E{次のテスト\nがある?} E -->|"はい"| B E -->|"いいえ"| F(["全テスト完了"]) style B fill:#dbeafe,stroke:#3b82f6 style C fill:#d1fae5,stroke:#10b981 style D fill:#fef3c7,stroke:#f59e0b

setUp と tearDown は各テストメソッドの前後に必ず実行されます。テストが失敗した場合もtearDownは実行されます。

setUp/tearDownの使用例 
import unittest

class ShoppingCart:
    """テスト対象のショッピングカートクラス"""
    def __init__(self):
        self.items = []

    def add_item(self, name, price):
        self.items.append({"name": name, "price": price})

    def total(self):
        return sum(item["price"] for item in self.items)

    def count(self):
        return len(self.items)


class TestShoppingCart(unittest.TestCase):

    def setUp(self):
        """各テストの前に実行: カートを初期化してアイテムを追加"""
        self.cart = ShoppingCart()       # 毎回新しいカートを作成
        self.cart.add_item("りんご", 150)  # テスト用のデータを準備

    def tearDown(self):
        """各テストの後に実行: クリーンアップ処理"""
        # 今回は特に不要だが、DBのロールバック等をここで行う
        pass

    def test_合計金額の計算(self):
        # setUpで追加した「りんご150円」が存在する状態でテスト
        self.assertEqual(self.cart.total(), 150)

    def test_アイテム追加後の合計(self):
        # このテストでもsetUpが実行されるため、カートはリセットされている
        self.cart.add_item("みかん", 80)
        self.assertEqual(self.cart.total(), 230)  # 150 + 80

    def test_アイテム数(self):
        self.assertEqual(self.cart.count(), 1)  # setUpで1つ追加済み

重要ポイント: setUpはテストの独立性を保証する

各テストメソッドの前にsetUpが実行されるため、あるテストが別のテストの状態を引き継ぐことがありません。これにより「テストの実行順序に依存しない」独立したテストを書けます。

3.5 pytestとunittestの徹底比較

同じテストをpytestとunittestで書き比べることで、両者の違いを明確に把握しましょう。

1 シンプルなアサーション

unittest
unittest版 
import unittest

class TestAdd(unittest.TestCase):
    def test_add(self):
        # 専用メソッドを使う
        self.assertEqual(1 + 1, 2)
        self.assertTrue(2 > 1)
        self.assertIn(3, [1, 2, 3])

if __name__ == '__main__':
    unittest.main()
pytest
pytest版 
import pytest

def test_add():
    # 普通の assert 文でOK！
    assert 1 + 1 == 2
    assert 2 > 1
    assert 3 in [1, 2, 3]

# if __name__ == ... も不要

2 前処理・後処理

unittest setUp / tearDown
unittest版 
class TestCart(unittest.TestCase):

    def setUp(self):
        # 各テスト前に実行
        self.cart = ShoppingCart()

    def tearDown(self):
        # 各テスト後に実行
        pass

    def test_empty_cart(self):
        self.assertEqual(self.cart.total(), 0)
pytest フィクスチャ（第6章で詳しく学習）
pytest版 
import pytest

@pytest.fixture
def cart():
    # フィクスチャで前準備
    return ShoppingCart()

def test_empty_cart(cart):
    # 引数でフィクスチャを受け取る
    assert cart.total() == 0

3 例外テスト

unittest
unittest版 
class TestDivide(unittest.TestCase):

    def test_ゼロ除算(self):
        with self.assertRaises(ZeroDivisionError):
            divide(10, 0)
pytest
pytest版 
import pytest

def test_ゼロ除算():
    with pytest.raises(ZeroDivisionError):
        divide(10, 0)

例外テストの構文は非常によく似ています。self.assertRaises vs pytest.raises の違いだけです。

観点 unittest pytest
インストール 不要（標準ライブラリ） pip install pytest が必要
テストの書き方 クラスを継承して self.assertXxx を使う 関数で assert 文を使うだけ
コード量 多い（クラス・メソッド・self が必要） 少ない（最小限のコード）
エラーレポート シンプル（失敗のみ表示） 詳細（期待値と実際の値を並べて表示）
前処理・後処理 setUp / tearDown メソッド フィクスチャ（より柔軟で再利用しやすい）
プラグイン 限定的 豊富なエコシステム
業界採用 Django等フレームワーク標準 現在最も広く使われている

3.6 pytestはunittestを実行できる

pytestの重要な特徴のひとつは、unittestで書かれたテストをそのまま実行できることです。既存のunittestプロジェクトに対して、いきなりpytestを使い始めることができます。

unittestで書いたテストをpytestで実行 
# unittestで書いたファイルをpytestで実行できる
pytest test_calculator_unittest.py

# 出力例（pytest形式で表示される）
# collected 4 items
#
# test_calculator_unittest.py ....    [ 100%]
#
# 4 passed in 0.12s

unittestからpytestへの段階的移行

flowchart LR A["既存の\nunittest\nテスト"] -->|"pytestで実行"| B{"テスト\n通過?"} B -->|"はい"| C["新しいテストを\npytestスタイルで追加"] B -->|"いいえ"| D["テストコードの\nバグを修正"] C --> E["徐々に既存テストも\npytestに書き換え"] D --> B style A fill:#f3f4f6,stroke:#9ca3af style C fill:#d1fae5,stroke:#10b981 style E fill:#dbeafe,stroke:#3b82f6

互換性があるため、一度にすべてを書き換える必要はありません。新しいテストだけpytestスタイルで書き、既存のものは徐々に移行できます。

setUpClass / tearDownClass - クラス全体の前後処理

setUp/tearDownが各テストメソッドごとに実行されるのに対し、setUpClass/tearDownClassはクラス内の全テストの最初と最後に1回だけ実行されます。データベース接続など、コストの高い初期化に使います。@classmethodデコレータが必要です。

3.7 どちらを選ぶべきか

「unittestとpytestのどちらを使うべきか」は、状況によって異なります。判断の指針を整理します。

unittestが向いているケース

  • 追加ライブラリを一切インストールしたくない環境
  • Djangoのdjango.test.TestCaseを使う場合
  • 既存チームがunittestで統一している場合
  • Java/JUnitの経験者がPythonを書く場合

pytestが向いているケース

  • 新規プロジェクトでゼロから始める場合（現在の主流）
  • テストコードをシンプルに保ちたい場合
  • カバレッジ計測、並列実行などのプラグインを使いたい場合
  • パラメータ化テストやフィクスチャを活用したい場合

現場での現実

現在の多くのPythonプロジェクトではpytestが使われています。一方、DjangoプロジェクトではDjango独自のテスト機能（unittestベース）を使うことも多いです。どちらも読み書きできると、より多くの現場に対応できます。

実習 3-1: unittestでテストクラスを書いてみよう

以下のBankAccountクラスに対して、unittestを使ったテストクラスを作成します。setUp/tearDownの使い方と、複数のassertメソッドを練習します。

テスト対象のコード

bank_account.py 
class BankAccount:
    """銀行口座クラス（テスト対象）"""

    def __init__(self, owner: str, balance: float = 0):
        self.owner = owner
        self.balance = balance

    def deposit(self, amount: float):
        """入金する"""
        if amount <= 0:
            raise ValueError("入金額は0より大きい必要があります")
        self.balance += amount

    def withdraw(self, amount: float):
        """出金する"""
        if amount <= 0:
            raise ValueError("出金額は0より大きい必要があります")
        if amount > self.balance:
            raise ValueError("残高が不足しています")
        self.balance -= amount

    def get_balance(self) -> float:
        """残高を取得する"""
        return self.balance

課題

  1. 1 unittest.TestCaseを継承したTestBankAccountクラスを作成してください
  2. 2 setUpで「田中太郎」という名前、残高1000円の口座を作成してください
  3. 3 入金テスト・出金テスト・残高不足エラーテスト・マイナス金額エラーテストの4つのテストメソッドを書いてください
  4. 4 同じテストをpytestスタイルでも書いて、コード量の違いを比較してみてください

実行方法

  • • unittestで実行: python -m unittest test_bank_account.py
  • • pytestで実行: pytest test_bank_account.py -v（両方対応）
  • • -vオプションで各テストの名前も表示されます

第4章: アサーションの詳細 - テスト結果を正確に検証する

理解度確認クイズ

  1. Q1. unittestでテストクラスを定義するには何が必要ですか？

    回答を表示

    unittest.TestCaseクラスを継承する必要があります。例: class TestMyFeature(unittest.TestCase):。クラス名は慣習としてTestで始め、テストメソッドはtest_で始める必要があります。

  2. Q2. unittestの setUp メソッドはいつ実行されますか？

    回答を表示

    各テストメソッドの実行前に毎回実行されます。3つのテストメソッドがあれば、setUpは3回実行されます。これにより、テスト間で状態が共有されることなく、各テストが独立した環境で実行されます。

  3. Q3. unittestで「例外が発生すること」をテストするには何を使いますか？

    回答を表示

    assertRaisesメソッドをコンテキストマネージャとして使います。with self.assertRaises(ValueError):のように書き、withブロック内で例外が発生することを確認します。pytestのpytest.raisesに相当します。

  4. Q4. pytestはunittestで書かれたテストを実行できますか？

    回答を表示

    はい、実行できます。pytestはunittestのテストクラスを自動的に検出し、そのまま実行します。既存のunittestプロジェクトに対してpytestを導入する際、テストコードを書き換える必要がないため、段階的な移行が可能です。

  5. Q5. pytestとunittestを比較したとき、コード量が少ない・シンプルに書けるのはどちらですか？その理由も答えてください。

    回答を表示

    pytestの方がシンプルです。理由は、クラスの継承が不要で、普通の関数としてテストを書けること、専用のassertメソッドの代わりに標準のassert文が使えること、self引数が不要なこと、などです。
