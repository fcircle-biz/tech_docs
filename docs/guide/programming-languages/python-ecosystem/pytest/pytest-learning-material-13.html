設定ファイルとカスタマイズ - pytest.iniとpyproject.toml

pytest.ini、pyproject.toml、setup.cfgを使ったpytestの設定方法を学びます。テストディレクトリの指定、デフォルトオプションの設定、マーカーの登録、テスト収集のカスタマイズなど、プロジェクトに合わせた設定の最適化を行います。

この章で学ぶこと

  • pytest.ini、pyproject.toml、setup.cfgの役割と使い分けを説明できる
  • テストディレクトリの指定やデフォルトオプションを設定ファイルで管理できる
  • カスタムマーカーを設定ファイルに登録して警告を解消できる
  • テスト収集のパターンをカスタマイズし、プロジェクトに合わせた設定を行える

13.1 設定ファイルが必要な理由

pytestを使い始めると、コマンドラインでいつも同じオプションを入力したり、マーカーを使うたびに警告が出たりすることに気づきます。こうした課題を解決するのが設定ファイルです。

設定ファイルを使うことで、「チーム全員が同じ設定でテストを実行できる」環境を作れます。個人の環境差による「自分のPCでは動くのに」という問題を防ぎ、一貫したテスト体験を提供します。

設定ファイルの役割

pytest.ini

pytest専用の設定ファイルです。シンプルで直感的な書き方ができます。pytestの設定のみを管理するため、他のツールとの競合がありません。

pytest専用・シンプル

pyproject.toml

Python標準の設定ファイルです。pytest以外のツール（black、mypy、isortなど）の設定も1つのファイルにまとめられます。現在の推奨形式です。

現代の標準・統合管理

setup.cfg

setuptoolsの設定ファイルで、pytestも対応しています。古いプロジェクトでよく使われますが、新規プロジェクトではpyproject.tomlを推奨します。

レガシー・既存PJで使用

どのファイルを選ぶべきか

新しいプロジェクトではpyproject.tomlを使いましょう。既存プロジェクトにpytest.iniがある場合はそのままでも問題ありません。どれか1つのファイルに設定を集約することが大切です（複数ファイルに分散させない）。

13.2 pytest.iniの書き方

pytest.iniはプロジェクトのルートディレクトリに置くINI形式のファイルです。[pytest]セクションにpytestの設定を記述します。

pytest.ini（基本的な設定例） 
[pytest]
# テストファイルの検索ディレクトリ
testpaths = tests

# テストファイルの命名パターン
python_files = test_*.py *_test.py

# テストクラスの命名パターン
python_classes = Test*

# テスト関数の命名パターン
python_functions = test_*

# デフォルトで付与するコマンドラインオプション
addopts = -v --tb=short

# カスタムマーカーの登録（警告解消のため）
markers =
    slow: 実行に時間がかかるテスト
    integration: 外部サービスと連携するテスト
    unit: 単体テスト

主要な設定項目の解説

testpaths

pytestがテストファイルを探すディレクトリを指定します。指定しない場合はカレントディレクトリから再帰的に検索します。testsやsrc/testsのように指定することで、検索範囲を限定してテストの収集を高速化できます。

testpaths = tests src/tests
addopts

pytestを実行するたびに自動的に付与されるコマンドラインオプションです。毎回手動で入力していたオプションをここに書いておくことで、コマンドを短くできます。

addopts = -v --tb=short --strict-markers

--strict-markers: 未登録のマーカーをエラーとして扱う（マーカーの管理を徹底したい場合に有効）

markers

カスタムマーカーを登録します。第9章で学んだ@pytest.mark.slowなどを使うと「PytestUnknownMarkWarning」という警告が出ます。ここに登録することで警告が消え、マーカーの意味もドキュメント化されます。

markers =
    slow: 処理が遅いテスト（5秒以上）
    network: ネットワーク接続が必要なテスト

警告のフィルタリング設定

テスト実行時の警告表示を制御するfilterwarningsも便利な設定項目です。

pytest.ini（filterwarnings設定例） 
[pytest]
filterwarnings =
    # 特定の警告を無視する（例: 廃止予定のAPIを使うライブラリ）
    ignore::DeprecationWarning:some_old_library
    # すべての DeprecationWarning をエラーとして扱う
    error::DeprecationWarning

13.3 pyproject.tomlによる設定

pyproject.tomlはTOML形式のファイルで、Pythonプロジェクト全体の設定を1ファイルで管理できます。pytestの設定は[tool.pytest.ini_options]セクションに記述します。

pyproject.toml（pytest設定例） 
[tool.pytest.ini_options]
# テストファイルの検索ディレクトリ
testpaths = ["tests"]

# デフォルトオプション（TOML形式は文字列リスト）
addopts = [
    "-v",
    "--tb=short",
    "--strict-markers",
]

# カスタムマーカーの登録
markers = [
    "slow: 実行に時間がかかるテスト",
    "integration: 外部サービスと連携するテスト",
    "unit: 単体テスト",
]

# テストファイルの命名パターン
python_files = ["test_*.py", "*_test.py"]

# 最小pytestバージョンの指定
minversion = "7.0"

pytest.ini と pyproject.toml の記法比較

設定項目 pytest.ini pyproject.toml
testpaths testpaths = tests testpaths = ["tests"]
addopts addopts = -v --tb=short addopts = ["-v", "--tb=short"]
markers markers =
  slow: 説明		 markers = ["slow: 説明"]
セクション名 [pytest] [tool.pytest.ini_options]

pyproject.tomlで複数ツールを統合管理する例

pyproject.tomlの最大のメリットは、複数のツールの設定を1ファイルにまとめられることです。

pyproject.toml（複数ツール統合例） 
[tool.pytest.ini_options]
# pytestの設定
testpaths = ["tests"]
addopts = ["-v", "--tb=short"]
markers = ["slow: 遅いテスト"]

[tool.black]
# コードフォーマッターblackの設定
line-length = 88
target-version = ["py311"]

[tool.mypy]
# 型チェッカーmypyの設定
python_version = "3.11"
strict = true

[tool.coverage.run]
# カバレッジ設定
source = ["src"]
omit = ["tests/*"]

1つのファイルで全ツールの設定を管理できるため、プロジェクトのルートがすっきりします。

13.4 setup.cfgによる設定

setup.cfgはpytest.iniとよく似た書き方ですが、[tool:pytest]セクションを使います（コロンに注意）。

setup.cfg（pytest設定例） 
[tool:pytest]
testpaths = tests
addopts = -v --tb=short
markers =
    slow: 実行に時間がかかるテスト
    integration: 外部サービスと連携するテスト

setup.cfgは新規プロジェクトでは非推奨

setup.cfgはsetuptoolsの設定ファイルですが、Pythonコミュニティではpyproject.tomlへの移行が進んでいます。既存プロジェクトではそのまま使えますが、新規プロジェクトではpyproject.tomlを選択してください。

13.5 テスト収集のカスタマイズ

pytestはデフォルトでtest_*.pyや*_test.pyファイルを収集します。プロジェクトの規約に合わせてこのパターンを変更できます。

典型的なプロジェクト構成とテストディレクトリ 

myproject/
├── pyproject.toml       ← 設定ファイル（ここに pytest 設定も書く）
├── src/
│   └── myapp/
│       ├── __init__.py
│       ├── calculator.py
│       └── utils.py
└── tests/               ← testpaths = ["tests"] で指定
    ├── __init__.py
    ├── conftest.py      ← 共通フィクスチャ
    ├── unit/            ← 単体テスト
    │   ├── test_calculator.py
    │   └── test_utils.py
    └── integration/     ← 結合テスト
        └── test_api.py

テスト収集の高度な設定

pytest.ini（収集カスタマイズ例） 
[pytest]
# 複数のディレクトリを指定できる
testpaths = tests src/tests

# 特定のディレクトリを収集対象から除外する
norecursedirs = .git .tox dist build __pycache__ node_modules

# ファイル命名パターンのカスタマイズ
python_files = test_*.py check_*.py

# クラス命名パターンのカスタマイズ
python_classes = Test* Check*

# 関数命名パターンのカスタマイズ
python_functions = test_* check_*

設定の確認コマンド: --collect-only

設定ファイルを作成したら、pytest --collect-onlyコマンドで実際に収集されるテストの一覧を確認できます。テストを実行せずに「どのテストが認識されているか」を把握できる便利なオプションです。

--collect-only の出力例 
$ pytest --collect-only

rootdir: /path/to/myproject, configfile: pyproject.toml

  <Function test_add>
  <Function test_subtract>
<Module tests/unit/test_utils.py>
  <Function test_format_string>
<Module tests/integration/test_api.py>
  <Function test_get_users>
  <Function test_create_user>

13.6 よく使う設定パターン集

実務でよく使われる設定パターンをまとめます。プロジェクトの状況に合わせて参考にしてください。

1

最小構成（入門者向け）

まず始めるための最小限の設定です。テストディレクトリと詳細出力だけを設定します。

pyproject.toml（最小構成） 
[tool.pytest.ini_options]
testpaths = ["tests"]
addopts = ["-v"]
2

チーム開発向け構成

マーカーの厳格管理と詳細なトレースバックを設定します。

pyproject.toml（チーム開発向け） 
[tool.pytest.ini_options]
testpaths = ["tests"]
addopts = [
    "-v",
    "--tb=short",
    "--strict-markers",
    "--strict-config",
]
markers = [
    "slow: 実行時間が5秒以上かかるテスト",
    "integration: 外部サービスとの結合テスト",
    "unit: 単体テスト",
    "smoke: スモークテスト（最重要の基本動作確認）",
]
minversion = "7.0"
3

CI/CD対応構成

CI環境での実行を考慮した設定です。カバレッジ計測とXMLレポート出力を含みます。

pyproject.toml（CI/CD対応） 
[tool.pytest.ini_options]
testpaths = ["tests"]
# CI環境ではシンプルな出力が見やすい
addopts = [
    "--tb=short",
    "--strict-markers",
    # カバレッジ（pytest-cov が必要）
    "--cov=src",
    "--cov-report=xml",
    "--cov-report=term-missing",
]
markers = [
    "slow: 遅いテスト（CIでスキップ可能）",
    "integration: 外部サービス連携テスト",
]

13.7 設定ファイルの優先順位と検索ルール

pytestは起動時に設定ファイルを自動的に探します。どのファイルが使われるかのルールを理解しておきましょう。

設定ファイルの探索順序

rootdirの概念

pytestは設定ファイルを見つけた場所を「rootdir」として認識します。設定ファイルのパスの指定（testpaths等）はすべてrootdirからの相対パスとして解釈されます。pytest --versionを実行すると現在のrootdirを確認できます。

設定内容の確認方法

設定確認コマンドと出力例 
# 設定ファイルの内容と rootdir を確認
$ pytest --version
pytest 7.4.0

# テスト収集対象の確認（テストは実行しない）
$ pytest --collect-only

# 現在有効な設定を確認（verboseモードでrootdirが表示される）
$ pytest -v --co
============================= test session starts ==============================
platform darwin -- Python 3.11.0, pytest-7.4.0
rootdir: /Users/myname/myproject, configfile: pyproject.toml
...

実習 13-1: pyproject.tomlを作成してpytestを設定しよう

既存のプロジェクト（または新規作成したダミープロジェクト）にpyproject.tomlを追加し、pytestの設定を行います。

手順

  1. 1 プロジェクトのルートにpyproject.tomlファイルを作成してください。
  2. 2 [tool.pytest.ini_options]セクションにtestpathsとaddoptsを設定してください。
  3. 3 カスタムマーカー（slowなど）をmarkersに登録してください。
  4. 4 pytest --collect-onlyを実行して、設定が正しく反映されているか確認してください。
  5. 5 pytestを実行して、addoptsに設定したオプションが自動で適用されることを確認してください。

確認ポイント

  • • テスト実行時にrootdirとconfigfileが正しく表示されるか
  • • @pytest.mark.slowを使ったテストで警告が出なくなったか
  • • オプションを指定せずにpytestだけで詳細出力されるか

動作確認: 設定ファイルが認識されているかチェック

設定ファイルを作成したら以下のコマンドで確認してください。

確認コマンド 
# 設定ファイルを確認しながらテストを実行
pytest -v

期待される結果 

============================= test session starts ==============================
platform darwin -- Python 3.11.0, pytest-7.4.0
rootdir: /path/to/myproject, configfile: pyproject.toml
...

configfile: pyproject.tomlと表示されれば設定ファイルが正しく認識されています。

トラブルシューティング

設定ファイルを作ったのに反映されない

原因: 設定ファイルの場所やセクション名が間違っている可能性があります。

解決方法: ファイルがプロジェクトのルートに置かれているか確認してください。pyproject.tomlの場合は[tool.pytest.ini_options]（ドット区切り）、setup.cfgの場合は[tool:pytest]（コロン区切り）が正しい書き方です。

PytestUnknownMarkWarning が消えない

原因: 設定ファイルのmarkersにマーカーが登録されていないか、スペルが一致していません。

解決方法: コード中の@pytest.mark.xxxのマーカー名と設定ファイルのmarkersの名前が完全に一致しているか確認してください。

addopts のオプションが二重に適用される

原因: addoptsに設定したオプションとコマンドラインで指定したオプションが重複しています。

解決方法: addoptsに-vを設定している場合、コマンドラインでは-vを省略できます。-p no:addoptsでaddoptsを一時的に無効にして確認することもできます。

テストが収集されない（0 tests collected）

原因: testpathsに指定したディレクトリが存在しない、またはテストファイルが命名規則に合っていません。

解決方法: pytest --collect-onlyを実行してrootdirとconfigfileを確認してください。testpathsのディレクトリが実際に存在するか、ファイル名がtest_*.pyの形式になっているかチェックしてください。

第14章: 実践パターン - 実務で使えるテスト設計

理解度確認クイズ

  1. Q1. 新規Pythonプロジェクトでpytestの設定ファイルを1つだけ選ぶとしたら、どのファイルが推奨されますか？

    回答を表示

    pyproject.tomlが推奨されます。Pythonの標準設定ファイルとして採用されており、pytest以外のツール（black、mypy、coverageなど）の設定も1ファイルにまとめて管理できるため、現代のPythonプロジェクトの標準的な選択肢です。

  2. Q2. pyproject.tomlにpytestの設定を書くとき、セクション名は何になりますか？

    回答を表示

    [tool.pytest.ini_options]です。pytest.iniの[pytest]やsetup.cfgの[tool:pytest]と混同しないように注意してください。

  3. Q3. pytest実行時に毎回「PytestUnknownMarkWarning」が表示されています。設定ファイルでどう解消できますか？

    回答を表示

    設定ファイルのmarkersにカスタムマーカーを登録することで解消できます。例えばmarkers = ["slow: 遅いテスト"]と記述することで、そのマーカーが公式に認識され警告が消えます。さらに--strict-markersをaddoptsに追加すると、未登録のマーカーはエラーとして扱われ、マーカーの管理を強制できます。

  4. Q4. テスト収集対象を確認するコマンド（テストは実行しない）は何ですか？

    回答を表示

    pytest --collect-onlyです。このコマンドはテストを実行せずに「どのテストファイル・テスト関数が収集されるか」の一覧を表示します。設定ファイルを変更した際の動作確認に非常に役立ちます。
