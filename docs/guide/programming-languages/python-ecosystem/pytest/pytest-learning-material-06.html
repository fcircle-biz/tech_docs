フィクスチャ基礎 - テストの前準備と後片付け

pytestのフィクスチャ（fixture）の基本概念と使い方を学びます。@pytest.fixtureデコレータによるテストデータの準備、yield文を使った後処理（teardown）、フィクスチャのスコープを理解します。

この章で学ぶこと

  • フィクスチャの概念と、テストコードの重複を減らす仕組みを説明できる
  • @pytest.fixtureデコレータを使ってテストデータを準備するフィクスチャを定義できる
  • yield文を使って前処理（setup）と後処理（teardown）を1つの関数で書ける
  • フィクスチャのスコープ（function・class・module・session）の違いを理解し使い分けられる

6.1 フィクスチャとは何か

テストを書いていると、同じような「準備作業」を何度も繰り返す場面に出くわします。たとえば、複数のテスト関数でデータベース接続を作ったり、同じテストデータを用意したりする場面です。

フィクスチャ（fixture）とは、テストの前準備（setup）と後片付け（teardown）を担う仕組みです。複数のテストで共通して使う準備処理をフィクスチャとして定義しておくことで、コードの重複をなくし、テストをすっきり書けます。

身近な例で考えてみよう

料理のレシピ本で「下ごしらえ：玉ねぎをみじん切りにする」が複数のレシピで共通して使われるようなイメージです。毎回同じ手順を書くのではなく、「下ごしらえA参照」とまとめられます。フィクスチャはテストにおける「共通の下ごしらえ」です。

6.2 フィクスチャの基本的な書き方

フィクスチャは@pytest.fixtureデコレータを付けた関数として定義します。テスト関数の引数に同名のフィクスチャを書くだけで、pytestが自動的に準備して渡してくれます。

フィクスチャを使わない場合（重複が多い）

test_cart_without_fixture.py 
def test_カートに商品を追加できる():
    # 毎回同じ準備コードが必要
    cart = ShoppingCart()
    cart.add_item("apple", price=100, qty=2)
    assert cart.total() == 200

def test_カートから商品を削除できる():
    # まったく同じ準備コードを繰り返す
    cart = ShoppingCart()
    cart.add_item("apple", price=100, qty=2)
    cart.remove_item("apple")
    assert cart.is_empty()

def test_カートの合計金額を計算できる():
    # また同じ準備コードを繰り返す
    cart = ShoppingCart()
    cart.add_item("apple", price=100, qty=2)
    assert len(cart.items) == 1

フィクスチャを使った場合（スッキリ！）

test_cart_with_fixture.py 
import pytest

# フィクスチャの定義：テストデータを準備して返す
@pytest.fixture
def cart_with_apple():
    """リンゴが2個入ったカートを準備する"""
    cart = ShoppingCart()
    cart.add_item("apple", price=100, qty=2)
    return cart  # テスト関数に渡される

# 引数名でフィクスチャを受け取る
def test_カートに商品を追加できる(cart_with_apple):
    assert cart_with_apple.total() == 200

def test_カートから商品を削除できる(cart_with_apple):
    cart_with_apple.remove_item("apple")
    assert cart_with_apple.is_empty()

def test_カートの合計金額を計算できる(cart_with_apple):
    assert len(cart_with_apple.items) == 1

フィクスチャの動作の仕組み

pytestはテスト関数の引数名を見て、同名のフィクスチャを自動的に探し出し、実行してから渡します。この「依存関係の自動解決」がpytestフィクスチャの核心です。テスト関数は「準備済みのデータをもらって検証するだけ」に専念できます。

フィクスチャが呼ばれる順序

sequenceDiagram participant P as pytest participant F as fixture関数 participant T as test関数 P->>F: フィクスチャを呼び出す F->>F: 準備処理（setup）を実行 F-->>P: 準備したオブジェクトを返す P->>T: テスト関数を呼び出す（引数にフィクスチャを渡す） T->>T: テスト実行（assert） T-->>P: テスト完了

6.3 yield を使った後処理（teardown）

テストが終わった後に「後片付け」が必要なケースがあります。ファイルを開いたら閉じる、データベース接続を作ったら切断する、といった後処理です。pytestではyield文を使って、前処理と後処理を1つのフィクスチャにまとめて書けます。

yieldフィクスチャの基本構造

yield_fixture_pattern.py 
import pytest

@pytest.fixture
def database_connection():
    # ===== 前処理（setup）=====
    print("データベース接続を作成")
    conn = create_db_connection()

    yield conn  # ← ここでテスト関数に conn を渡す
                # テスト実行中はここで「一時停止」

    # ===== 後処理（teardown）=====
    # テスト完了後（成功・失敗に関わらず）ここが実行される
    print("データベース接続を切断")
    conn.close()

実行の流れ

1. 前処理（接続作成） 2. yield（テスト実行） 3. 後処理（切断）

具体的な使用例：一時ファイルの作成と削除

test_file_operations.py 
import pytest
import os
import tempfile

@pytest.fixture
def temp_file():
    """テスト用の一時ファイルを作成・削除するフィクスチャ"""
    # 前処理：一時ファイルを作成
    fd, filepath = tempfile.mkstemp(suffix=".txt")
    os.close(fd)
    print(f"\n一時ファイル作成: {filepath}")

    yield filepath  # テスト関数にファイルパスを渡す

    # 後処理：一時ファイルを削除
    if os.path.exists(filepath):
        os.remove(filepath)
        print(f"一時ファイル削除: {filepath}")

# フィクスチャを使うテスト
def test_ファイルに書き込める(temp_file):
    with open(temp_file, "w") as f:
        f.write("Hello, pytest!")
    with open(temp_file) as f:
        assert f.read() == "Hello, pytest!"
    # テスト後、自動でファイルが削除される

重要：後処理はテスト失敗時にも実行される

yield後の後処理コードは、テストが成功しても失敗しても必ず実行されます。これにより「テストが失敗してもリソースが確実に解放される」ことが保証されます。try/finallyと同じ効果を、よりシンプルに書けます。

6.4 フィクスチャのスコープ

フィクスチャは「どの範囲で使い回すか」をスコープで制御できます。スコープが広いほど、フィクスチャの初期化コストを節約できます。一方で、テスト間でデータが汚染されるリスクも高まります。

function（デフォルト）

最も狭い

テスト関数ごとに毎回フィクスチャが作られ、テスト後に破棄されます。テスト間の独立性が保たれます。

@pytest.fixture(scope="function")

class

クラス単位

同一テストクラス内のすべてのテストで共有されます。クラスをまたぐと新たに作られます。

@pytest.fixture(scope="class")

module

ファイル単位

同一テストファイル（モジュール）内のすべてのテストで共有されます。ファイルをまたぐと新たに作られます。

@pytest.fixture(scope="module")

session

最も広い

テスト実行全体で1回だけ作られます。コストの高い初期化（DBセットアップ等）をまとめて1回だけ行いたいときに使います。

@pytest.fixture(scope="session")

スコープの範囲イメージ

スコープの指定方法

conftest.py（スコープの例） 
import pytest

# function スコープ（デフォルト）：テストごとに新しいインスタンス
@pytest.fixture
def user_data():
    return {"name": "Alice", "age": 30}

# module スコープ：同じファイル内で使い回し
@pytest.fixture(scope="module")
def db_connection_module():
    conn = create_connection()
    yield conn
    conn.close()

# session スコープ：テスト実行全体で1回だけ
@pytest.fixture(scope="session")
def app_config():
    """重い設定の読み込みを1度だけ行う"""
    return load_app_config("test_settings.yaml")
スコープ 使いどころ 注意点
function テスト間で状態が変わるデータ（辞書、リストなど） 毎回初期化されるためコスト注意
class 関連するテストをクラスでまとめた場合の共有データ クラス内テスト順序に注意
module ファイル全体で共通の読み取り専用設定・テストデータ 状態を変えるテストに使うと干渉する
session コストの高い初期化（DBスキーマ構築、設定ファイル読み込み） テスト間での状態共有に最も注意が必要

6.5 フィクスチャの組み合わせ

フィクスチャは別のフィクスチャを引数に取ることができます。これにより、フィクスチャ同士を組み合わせて複雑な準備処理を段階的に構築できます。

test_with_combined_fixtures.py 
import pytest

@pytest.fixture
def user():
    """基本のユーザーデータ"""
    return {"id": 1, "name": "Alice", "email": "alice@example.com"}

@pytest.fixture
def order(user):
    """user フィクスチャを使って注文データを作る"""
    return {
        "order_id": 101,
        "user_id": user["id"],  # user フィクスチャの値を活用
        "items": ["apple", "banana"],
        "total": 300
    }

# test 関数は order だけ受け取れば OK
def test_注文にユーザーIDが設定される(order):
    assert order["user_id"] == 1

def test_注文の合計金額が正しい(order):
    assert order["total"] == 300

複数のフィクスチャを受け取ることも可能

テスト関数が複数のフィクスチャを必要とする場合は、引数を複数書くだけで受け取れます。例：def test_something(user, order, db_connection):

6.6 フィクスチャの自動使用（autouse）

autouse=Trueを指定すると、テスト関数の引数に書かなくてもフィクスチャが自動的に適用されます。「すべてのテストで必ず実行したい前処理・後処理」に使います。

test_with_autouse.py 
import pytest

@pytest.fixture(autouse=True)
def テスト環境のリセット():
    """全テストの前後にDBをクリーンな状態に保つ"""
    # 前処理：テスト用DBをリセット
    reset_test_database()
    yield
    # 後処理：テストデータをクリーンアップ
    cleanup_test_data()

# autouse なので引数不要。毎テストで自動的に実行される
def test_ユーザー作成():
    user = create_user("Bob")
    assert user.name == "Bob"

def test_商品追加():
    product = add_product("Widget")
    assert product.name == "Widget"

autouse の使い過ぎに注意

autouse=Trueはすべてのテストに影響するため、意図しないスロー化や副作用が生じることがあります。本当に全テストで必要な処理にだけ使いましょう。多くの場合は、必要なテストの引数に明示的に書くほうが意図が伝わりやすくなります。

6.7 フィクスチャ定義場所のルール

フィクスチャはどこに書いてもよいわけではありません。pytestが認識できる場所のルールがあります。

フィクスチャを書ける場所 

myproject/
├── tests/
│   ├── conftest.py          ← 複数ファイルで共有するフィクスチャ（推奨）
│   ├── test_users.py        ← このファイル内だけのフィクスチャも書ける
│   └── test_orders.py
└── pytest.ini

テストファイル内に書く

そのファイル内のテストだけで使うフィクスチャ。ファイルをまたいでは使えません。

適用範囲：同一ファイルのみ

conftest.py に書く

同じフォルダ以下のすべてのテストで使えるフィクスチャ。チームで共有するフィクスチャはここに書くのが定番です。

適用範囲：配置フォルダ以下全体

conftest.py は自動的に読み込まれる

conftest.pyはimportなしで、pytestが自動的に読み込みます。複数のテストファイルで使いたいフィクスチャはconftest.pyにまとめましょう（第7章で詳しく解説します）。

課題

  1. 1 以下のコードをそのままtest_stack.pyとして保存してください。
stack.py（テスト対象のクラス） 
class Stack:
    """シンプルなスタック実装"""
    def __init__(self):
        self._items = []

    def push(self, item):
        self._items.append(item)

    def pop(self):
        if not self._items:
            raise IndexError("スタックが空です")
        return self._items.pop()

    def peek(self):
        if not self._items:
            raise IndexError("スタックが空です")
        return self._items[-1]

    def is_empty(self):
        return len(self._items) == 0

    def size(self):
        return len(self._items)

def test_sizeは要素数を返す():

    assert stack.size() == 2

第7章: フィクスチャ応用 - 再利用とパラメータ化

理解度確認クイズ

  1. Q1. フィクスチャをテスト関数で使うにはどうすればよいですか？

    回答を表示

    テスト関数の引数にフィクスチャ名と同じ名前を書くだけです。例：def test_something(my_fixture):。pytestが自動的に対応するフィクスチャを見つけて実行し、引数として渡してくれます。

  2. Q2. フィクスチャに後処理（teardown）を書きたい場合、何を使いますか？

    回答を表示

    yield文を使います。yieldの前が前処理（setup）、yieldの後が後処理（teardown）になります。後処理はテストの成功・失敗に関わらず実行されます。

  3. Q3. スコープをsessionにすると、フィクスチャは何回実行されますか？

    回答を表示

    テスト実行全体で1回だけ実行されます。何百個のテストがあっても、そのフィクスチャは最初に1回だけ初期化され、最後に1回だけ後処理が実行されます。コストの高い初期化処理を節約したいときに使います。

  4. Q4. 引数に書かなくてもすべてのテストに自動適用されるフィクスチャを作るにはどうしますか？

    回答を表示

    @pytest.fixture(autouse=True)を指定します。このフィクスチャは、スコープ範囲内のすべてのテスト関数に対して自動的に適用されます。

  5. Q5. 複数のテストファイルで共通フィクスチャを共有したい場合、どこに書けばよいですか？

    回答を表示

    conftest.pyファイルに書きます。pytestは自動的にconftest.pyを読み込むため、importなしで同フォルダ以下のすべてのテストから利用できます。
