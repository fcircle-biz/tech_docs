第11章

エラー処理とデバッグ

エラーの種類と対処法、try-catch文による例外処理、デバッグテクニック、ログの活用方法を学びます。

この章で学ぶこと

  • エラーとは何か、なぜエラーが発生するのかを理解する
  • PHPのエラーの種類と特徴を知る
  • エラーメッセージを読んで原因を特定する方法を習得する
  • try-catch文を使った例外処理を実装できる
  • 実践的なデバッグテクニックを身につける
  • ログを活用してエラーを追跡する方法を学ぶ

11.1 エラーとは何か

プログラミングにおけるエラーとは、プログラムが正常に動作しない状態のことです。 エラーは大きく分けて「文法エラー」と「実行時エラー」の2種類があります。

エラーは「学習のチャンス」

初心者の方は「エラーが出た = 失敗した」と感じがちですが、実はエラーは「ここを直してください」というヒントです。 エラーメッセージを読み解く力をつけることで、プログラミングスキルは大きく向上します。

エラーが発生する主な原因

  • 文法ミス: セミコロン忘れ、カッコの閉じ忘れなど
  • タイプミス: 変数名や関数名のスペルミス
  • 論理エラー: プログラムの考え方やロジックのミス
  • 環境の問題: ファイルパスの間違い、権限不足など

エラーの分類

flowchart TD A[エラー] --> B[文法エラー] A --> C[実行時エラー] B --> D["構文エラー（Parse Error）"] B --> E["致命的エラー（Fatal Error）"] C --> F["警告（Warning）"] C --> G["通知（Notice）"] C --> H["例外（Exception）"]

11.2 PHPのエラーの種類

PHPには主に4種類のエラーレベルがあります。それぞれの特徴と対処法を理解しましょう。

Parse Error

構文エラー。文法が間違っているためプログラムが実行されません。

例: セミコロン忘れ、カッコの閉じ忘れ

Fatal Error

致命的エラー。実行中に回復不可能なエラーが発生し、プログラムが停止します。

例: 存在しない関数の呼び出し、必須ファイルが見つからない

Warning

警告。エラーが発生していますが、プログラムは継続して実行されます。

例: includeファイルが見つからない、0で除算

Notice

通知。軽微な問題を通知しますが、プログラムは実行されます。

例: 未定義の変数を使用、配列の存在しないキーにアクセス

エラーレベル 深刻度 プログラムの挙動
Parse Error 最高 実行されない
Fatal Error 高 即座に停止
Warning 中 継続実行
Notice 低 継続実行

11.3 エラーメッセージの読み方

エラーメッセージには問題解決のヒントが含まれています。 正しく読み解くことで、素早くエラーを修正できます。

エラーメッセージの例

Parse error: syntax error, unexpected '}' in /var/www/html/test.php on line 12

エラーメッセージの構成要素

1

エラータイプ

Parse error、Fatal error、Warningなど、エラーの種類を示します。

2

エラー内容

何が問題なのかを説明します（syntax error、unexpected '}' など）。

3

発生場所

ファイル名と行番号を示します（in /path/to/file.php on line 12）。

実習 11-1: エラーメッセージを読む

意図的にエラーを発生させて、エラーメッセージの読み方を練習しましょう。

手順 1: セミコロン忘れのエラー

<?php
// セミコロンを忘れる（意図的なエラー）
$message = "Hello, PHP"  // ← セミコロンがない

echo $message;
?>

エラーメッセージ例:
Parse error: syntax error, unexpected 'echo' (T_ECHO) in ... on line 5

→ 4行目のセミコロン忘れが原因ですが、エラーは次の行（5行目）で検出されます。 エラー行の前の行も確認することが重要です。

手順 2: 未定義変数の使用

<?php
// 定義していない変数を使用
echo $undefined_variable;
?>

エラーメッセージ例:
Notice: Undefined variable: undefined_variable in ... on line 3

→ Noticeなのでプログラムは動きますが、バグの原因になるため修正すべきです。

エラー解決のコツ

  • • エラーメッセージは最後まで読む（行番号が重要）
  • • エラーが表示された行だけでなく、その前後も確認する
  • • エラーメッセージを検索エンジンで調べる（英語のまま検索）
  • • 一度に複数の変更を加えず、1つずつ修正して動作確認する

11.4 例外処理（try-catch文）

例外処理は、エラーが発生する可能性がある処理を安全に実行するための仕組みです。 try-catch文を使うことで、エラーが発生してもプログラムが停止せず、適切に対処できます。

try-catch文の動作フロー

flowchart TD A([処理開始]) --> B[try ブロック実行] B --> C{エラー発生?} C -- いいえ --> D[正常終了] C -- はい --> E[catch ブロック実行] E --> F[エラー処理] F --> G[プログラム継続] D --> G G --> H([処理終了])

基本的な構文

<?php
try {
    // エラーが発生する可能性がある処理
    $result = 10 / 0;  // 0で除算

} catch (Exception $e) {
    // エラーが発生した時の処理
    echo "エラーが発生しました: " . $e->getMessage();
}
?>

ファイル読み込みの例外処理

<?php
try {
    // ファイルが存在しない場合は例外を投げる
    $filename = "data.txt";

    if (!file_exists($filename)) {
        throw new Exception("ファイルが見つかりません: " . $filename);
    }

    $content = file_get_contents($filename);
    echo "ファイルの内容: " . $content;

} catch (Exception $e) {
    // エラーメッセージを表示
    echo "エラー: " . $e->getMessage();

    // ログファイルに記録
    error_log($e->getMessage());
}
?>

複数の例外をキャッチする

<?php
try {
    // データベース接続
    $pdo = new PDO("mysql:host=localhost;dbname=test", "user", "pass");

} catch (PDOException $e) {
    // データベース関連のエラー
    echo "DB接続エラー: " . $e->getMessage();

} catch (Exception $e) {
    // その他のエラー
    echo "一般エラー: " . $e->getMessage();
}
?>

例外処理を使うべき場面

  • • ファイル操作（ファイルが存在しない可能性）
  • • データベース接続（接続失敗の可能性）
  • • 外部APIとの通信（ネットワークエラーの可能性）
  • • ユーザー入力の処理（想定外の値が入る可能性）

11.5 デバッグテクニック

デバッグとは、プログラムの問題（バグ）を見つけて修正する作業のことです。 PHPには便利なデバッグ用の関数がいくつか用意されています。

var_dump()

変数の型と値を詳細に表示します。最も使用頻度が高いデバッグ関数です。 

var_dump($変数名);

print_r()

変数の内容を見やすく表示します。配列の構造確認に便利です。 

print_r($変数名);

die() / exit()

メッセージを表示してプログラムを停止します。どこまで実行されたか確認できます。 

die("ここまで実行");

error_reporting()

表示するエラーレベルを設定します。開発時は全エラーを表示するのが推奨です。 

error_reporting(E_ALL);

実習 11-2: デバッグ関数を使う

デバッグ関数を使って変数の中身を確認しましょう。 

<?php
// 全てのエラーを表示
error_reporting(E_ALL);
ini_set('display_errors', 1);

// デバッグ対象のデータ
$user = [
    'id' => 1,
    'name' => '山田太郎',
    'email' => 'yamada@example.com',
    'age' => 25
];

// var_dump で詳細表示
echo "<h3>var_dump の結果</h3>";
var_dump($user);

echo "<hr>";

// print_r で見やすく表示
echo "<h3>print_r の結果</h3>";
echo "<pre>";
print_r($user);
echo "</pre>";

echo "<hr>";

// 特定の値だけ確認
echo "<h3>個別の値確認</h3>";
echo "名前: " . $user['name'] . "<br>";
echo "年齢: " . $user['age'] . "<br>";

// ここで処理を止めたい場合
// die("デバッグ終了");
?>

実践的なデバッグ手順

  1. 1. エラーメッセージを確認する（表示されていない場合はerror_reportingを有効化）
  2. 2. エラーが発生している行の前後にvar_dumpを挿入して変数の状態を確認
  3. 3. 条件分岐やループの中に「通過確認」のechoを入れる
  4. 4. 問題が解決したらデバッグ用のコードを削除

11.6 ログの活用

ログとは、プログラムの動作履歴や発生したエラーを記録したファイルのことです。 本番環境ではエラーを画面に表示せず、ログファイルに記録することが一般的です。

error_log関数でログを記録する

<?php
// 基本的なログ記録
error_log("ログメッセージ");

// ファイルに記録
error_log("エラー発生: " . date('Y-m-d H:i:s'), 3, "/var/log/myapp.log");

// 変数の内容をログに記録
$data = ['user_id' => 123, 'action' => 'login'];
error_log("ユーザーアクション: " . print_r($data, true));
?>

カスタムログ関数を作成する

<?php
/**
 * ログを記録する関数
 * @param string $level ログレベル（INFO, WARNING, ERROR）
 * @param string $message ログメッセージ
 */
function writeLog($level, $message) {
    $logFile = __DIR__ . "/app.log";
    $timestamp = date('Y-m-d H:i:s');
    $logMessage = "[{$timestamp}] [{$level}] {$message}\n";

    // ファイルに追記
    file_put_contents($logFile, $logMessage, FILE_APPEND);
}

// 使用例
writeLog("INFO", "アプリケーション起動");
writeLog("WARNING", "データベース接続に時間がかかっています");
writeLog("ERROR", "ファイルが見つかりません: data.txt");
?>

ログの活用例

<?php
try {
    // データベース接続を試みる
    $pdo = new PDO("mysql:host=localhost;dbname=mydb", "user", "password");
    writeLog("INFO", "データベース接続成功");

    // クエリ実行
    $stmt = $pdo->query("SELECT * FROM users");
    $users = $stmt->fetchAll();
    writeLog("INFO", "ユーザー情報取得: " . count($users) . "件");

} catch (PDOException $e) {
    // エラーをログに記録
    writeLog("ERROR", "データベースエラー: " . $e->getMessage());

    // ユーザーには一般的なメッセージを表示
    echo "システムエラーが発生しました。管理者に連絡してください。";
}
?>

本番環境でのエラー表示設定

本番環境では、エラーを画面に表示せず、ログに記録する設定にしましょう：

// 本番環境の設定
error_reporting(E_ALL);
ini_set('display_errors', 0);  // 画面表示OFF
ini_set('log_errors', 1);      // ログ記録ON
環境 エラー表示 ログ記録 理由
開発環境 ON ON すぐにエラーを確認したい
本番環境 OFF ON ユーザーにエラー詳細を見せない

11.7 よくあるエラーと解決方法

PHPプログラミングでよく遭遇するエラーとその解決方法を紹介します。

Parse error: syntax error, unexpected '}'

原因: カッコの数が合っていない、セミコロンが抜けている

解決方法: エディタの括弧マッチング機能を使って、開きカッコと閉じカッコの数が合っているか確認する。エラー行の前の行も確認する。

Fatal error: Call to undefined function

原因: 存在しない関数を呼び出している、関数名のスペルミス

解決方法: 関数名のスペルを確認する。関数が定義されているファイルがinclude/requireされているか確認する。

Notice: Undefined variable

原因: 変数が定義される前に使用している

解決方法: 変数を使う前に初期化する。isset()関数で存在確認してから使用する。

Notice: Undefined index

原因: 配列に存在しないキーにアクセスしている

解決方法: isset()やarray_key_exists()で存在確認してからアクセスする。または ?? 演算子を使う。

$value = $_POST['name'] ?? 'デフォルト値';

Fatal error: Maximum execution time exceeded

原因: 無限ループ、処理に時間がかかりすぎている

解決方法: ループの終了条件を確認する。set_time_limit()で制限時間を延長する（一時的な対処）。処理を分割する。

// エラーを含むコード（練習用）

理解度確認クイズ

  1. Q1. PHPのエラーレベルを深刻度が高い順に並べてください。

    回答を表示

    1. Parse Error（構文エラー）
    2. Fatal Error（致命的エラー）
    3. Warning（警告）
    4. Notice（通知）

    Parse ErrorとFatal Errorはプログラムを停止させますが、WarningとNoticeは継続実行されます。

  2. Q2. try-catch文を使う目的は何ですか？

    回答を表示

    エラーが発生する可能性がある処理を安全に実行するためです。 tryブロック内でエラーが発生した場合、catchブロックでエラーを捕捉して適切に処理することで、プログラムが突然停止するのを防ぎます。 また、エラー情報をログに記録したり、ユーザーにわかりやすいメッセージを表示したりできます。

  3. Q3. 本番環境でエラーを画面に表示しない理由は何ですか？

    回答を表示

    主な理由は以下の通りです：
    1. セキュリティ: ファイルパスやデータベース情報など、システムの内部構造が漏洩する可能性がある
    2. ユーザー体験: 技術的なエラーメッセージはユーザーを混乱させる
    3. プロフェッショナルな印象: エラー詳細を見せるのは未完成に見える

    そのため、本番環境ではエラーをログファイルに記録し、ユーザーには一般的なメッセージを表示します。

  4. Q4. var_dump()とprint_r()の違いは何ですか？

    回答を表示

    var_dump():
    • 変数の型と値の両方を詳細に表示
    • データ型の確認が必要な時に使う
    • 出力例: string(5) "hello"

    print_r():
    • 変数の値を見やすく表示（型情報は少ない）
    • 配列の構造を確認する時に便利
    • 出力がシンプルで読みやすい

    どちらもデバッグに便利ですが、目的に応じて使い分けましょう。

第11章のまとめ

  • エラーはプログラミングで避けられないものであり、学習のチャンスと捉えることが大切です。
  • PHPのエラーにはParse Error、Fatal Error、Warning、Noticeの4種類があり、それぞれ深刻度が異なります。
  • エラーメッセージはエラータイプ、内容、発生場所の3要素から成り、これを読み解くことで原因を特定できます。
  • try-catch文を使うことで、エラーが発生してもプログラムが停止せず、適切に処理できます。
  • var_dump()、print_r()、die()などのデバッグ関数を使うことで、プログラムの状態を確認できます。
  • ログを活用することで、本番環境でも安全にエラーを追跡・記録できます。
  • 本番環境ではエラーを画面に表示せず、ログに記録する設定にすることがセキュリティとユーザー体験の観点から重要です。
