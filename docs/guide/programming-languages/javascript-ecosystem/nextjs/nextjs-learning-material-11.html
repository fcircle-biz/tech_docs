第11章

パフォーマンス最適化とSEO

Core Web Vitalsの改善からSEO最適化まで、ユーザー体験と検索エンジン可視性を同時に高めるための実践的手法を学びます。

この章で学ぶこと

  • Core Web Vitalsの3つの指標とその改善方法
  • バンドルサイズの最適化とコード分割のベストプラクティス
  • メタデータAPIによるSEO最適化と構造化データの実装
  • Lighthouse、WebPageTestなどの計測ツールの効果的な活用方法

11.1 Core Web Vitalsの理解

Core Web Vitalsは、Googleが定義するユーザー体験を測る3つの重要な指標です。これらは検索ランキングにも影響するため、SEOとパフォーマンス両方の観点で重要です。

flowchart LR A[Core Web Vitals] --> B["LCP
Largest Contentful Paint"] A --> C["FID/INP
First Input Delay / Interaction to Next Paint"] A --> D["CLS
Cumulative Layout Shift"] B --> B1["読み込みパフォーマンス
2.5秒以内"] C --> C1["インタラクティブ性
100ms/200ms以内"] D --> D1["視覚的安定性
0.1未満"]

LCP

Largest Contentful Paint: ページの最大コンテンツ要素（画像やテキストブロック）が表示されるまでの時間。

目標: 2.5秒以内

FID/INP

First Input Delay / Interaction to Next Paint: ユーザーの最初の操作からブラウザが応答するまでの時間。

目標: 100ms/200ms以内

CLS

Cumulative Layout Shift: ページ読み込み中に要素が予期せず移動する視覚的不安定性の累積量。

目標: 0.1未満

Next.jsでCore Web Vitalsを改善する方法

  • LCP改善: 画像最適化（next/image）、フォントの最適読み込み（next/font）、サーバーコンポーネントでの初期HTML生成
  • FID/INP改善: コード分割、不要なJavaScriptの削除、Server Actionsの活用
  • CLS改善: 画像・動画のサイズ指定、フォントのfallback設定、dynamic importのSuspenseラッピング

11.2 バンドルサイズの最適化

JavaScriptバンドルのサイズが大きいと、ダウンロード時間とパース時間が増加し、パフォーマンスが低下します。Next.jsは自動的にコード分割を行いますが、さらなる最適化も可能です。

実習 11-1: バンドルアナライザーでサイズを分析する

@next/bundle-analyzerを使用して、どのライブラリがバンドルサイズを圧迫しているかを可視化します。

手順

  1. 1 バンドルアナライザーをインストール
  2. 2 next.config.jsにアナライザーを追加
  3. 3 ビルドを実行してレポートを確認

実装例

terminal 
# バンドルアナライザーのインストール
npm install @next/bundle-analyzer
next.config.js 
// バンドルアナライザーの設定
const withBundleAnalyzer = require('@next/bundle-analyzer')({
  enabled: process.env.ANALYZE === 'true',
})

/** @type {import('next').NextConfig} */
const nextConfig = {
  // その他の設定
}

module.exports = withBundleAnalyzer(nextConfig)
実行コマンド 
# アナライザー付きでビルド
ANALYZE=true npm run build

バンドルサイズ削減のベストプラクティス

  • Tree Shaking対応のライブラリを選択: lodashではなくlodash-es、moment.jsではなくdate-fnsを使用
  • Named Importを使用: import { Button } from 'library'
  • サーバーコンポーネントで処理: クライアントに送信するJavaScriptを削減
  • 外部スクリプトはnext/scriptで最適化: 遅延読み込みやworkerオプションを活用

11.3 コード分割と動的インポート

Next.jsはページごとに自動的にコード分割を行いますが、next/dynamicを使用すると、コンポーネントレベルでさらに細かく制御できます。

flowchart TB A["初回ページロード"] --> B["必須コードのみ読み込み
app.js, page.js"] B --> C{ユーザー操作} C --> D["モーダル表示
modal.js 動的読み込み"] C --> E["チャート表示
chart.js 動的読み込み"] C --> F["地図表示
map.js 動的読み込み"] D --> G["必要なときだけダウンロード"] E --> G F --> G
app/components/HeavyChart.tsx (クライアントコンポーネント) 
'use client'
import dynamic from 'next/dynamic'
import { Suspense } from 'react'

// 重いチャートライブラリを動的インポート
const Chart = dynamic(() => import('react-chartjs-2'), {
  ssr: false, // SSRを無効化（ブラウザでのみ実行）
  loading: () => <div>チャート読み込み中...</div>
})

export default function HeavyChart() {
  return (
    <div>
      <Chart data={chartData} />
    </div>
  )
}

動的インポートのメリット

  • ✅ 初回ページロードが高速化
  • ✅ 必要なコンポーネントのみダウンロード
  • ✅ JavaScriptパース時間の削減
  • ✅ メモリ消費の最適化

注意が必要なケース

  • ⚠️ Above the Foldコンテンツには使わない
  • ⚠️ Suspense境界を適切に設定
  • ⚠️ ローディング状態のUXに配慮
  • ⚠️ SSR必須なコンポーネントはssr: true

11.4 プリフェッチング戦略

Next.jsの<Link>コンポーネントは、ビューポートに入ったリンクを自動的にプリフェッチします。これにより、ユーザーがリンクをクリックしたときに即座にページ遷移できます。

プリフェッチの動作

本番環境

ビューポートに入った<Link>のページデータを自動的にバックグラウンドでプリフェッチ

開発環境

ホバー時にプリフェッチ（開発サーバーの負荷軽減のため）

静的ページ

完全なページデータをプリフェッチ

動的ページ

共有レイアウトとローディング状態のみをプリフェッチ

プリフェッチの制御 
import Link from 'next/link'

export default function Navigation() {
  return (
    <>
      {/* デフォルト: プリフェッチ有効 */}
      <Link href="/dashboard">
        ダッシュボード
      </Link>

      {/* プリフェッチ無効（ログアウトリンクなど） */}
      <Link href="/logout" prefetch={false}>
        ログアウト
      </Link>

      {/* 明示的にプリフェッチ有効 */}
      <Link href="/important-page" prefetch={true}>
        重要ページ
      </Link>
    </>
  )
}

プリフェッチの最適化Tips

  • 優先度の高いページ: prefetch={true}で明示的に指定
  • 認証が必要なページ: ログイン前はプリフェッチを無効化
  • 大量のリンク: Intersection Observerで可視範囲のみプリフェッチ
  • 外部リンク: rel="preconnect"でドメイン接続を事前確立

11.5 メタデータAPIによるSEO最適化

Next.js 13以降のApp Routerでは、Metadata APIを使用して、型安全で自動的に最適化されたメタタグを生成できます。

app/layout.tsx（静的メタデータ） 
import type { Metadata } from 'next'

export const metadata: Metadata = {
  title: {
    template: '%s | MyApp', // 各ページで上書き可能
    default: 'MyApp - 最高のサービス', // デフォルトタイトル
  },
  description: 'Next.jsで構築された最高のアプリケーション',
  keywords: ['Next.js', 'React', 'TypeScript', 'フルスタック'],
  authors: [{ name: 'Your Name' }],
  openGraph: {
    title: 'MyApp',
    description: 'Next.jsで構築された最高のアプリケーション',
    images: ['/og-image.jpg'],
  },
  twitter: {
    card: 'summary_large_image',
    title: 'MyApp',
    description: 'Next.jsで構築された最高のアプリケーション',
    images: ['/og-image.jpg'],
  },
}

export default function RootLayout({ children }) {
  return <html><body>{children}</body></html>
}
app/blog/[slug]/page.tsx（動的メタデータ） 
import type { Metadata } from 'next'

// 動的にメタデータを生成
export async function generateMetadata(
  { params }: { params: { slug: string } }
): Promise<Metadata> {
  // 記事データを取得
  const post = await fetch(`/api/posts/${params.slug}`).then(r => r.json())

  return {
    title: post.title,
    description: post.excerpt,
    openGraph: {
      title: post.title,
      description: post.excerpt,
      images: [post.ogImage],
    },
  }
}

export default function BlogPost({ params }) {
  return <article>...</article>
}

SEOで重要なメタデータ項目

項目 説明 重要度
title ページタイトル（50-60文字推奨） 🔴 必須
description ページ概要（150-160文字推奨） 🔴 必須
openGraph SNSシェア時の表示内容 🟡 推奨
robots クローラー制御（index, follow等） 🟡 推奨
canonical 正規URLの指定（重複コンテンツ回避） 🟡 推奨
alternates 多言語ページのリンク関係 🟢 任意

11.6 構造化データの実装

構造化データ（JSON-LD）は、検索エンジンにページの内容をより正確に伝え、リッチリザルト（レビュー星、FAQ、パンくずリストなど）の表示を可能にします。

app/blog/[slug]/page.tsx（記事の構造化データ） 
export default function BlogPost({ post }) {
  const jsonLd = {
    '@context': 'https://schema.org',
    '@type': 'BlogPosting',
    headline: post.title,
    description: post.excerpt,
    image: post.ogImage,
    datePublished: post.publishedAt,
    dateModified: post.updatedAt,
    author: {
      '@type': 'Person',
      name: post.author.name,
    },
  }

  return (
    <article>
      {/* JSON-LDをscriptタグで埋め込み */}
      <script
        type="application/ld+json"
        dangerouslySetInnerHTML={{ __html: JSON.stringify(jsonLd) }}
      />
      <h1>{post.title}</h1>
      <div>{post.content}</div>
    </article>
  )
}

主要な構造化データタイプ

  • Article/BlogPosting: 記事・ブログ投稿
  • Product: 商品情報（価格、在庫、レビュー）
  • FAQ: よくある質問
  • BreadcrumbList: パンくずリスト
  • Organization: 組織情報
  • Recipe: レシピ（料理サイト）

構造化データのメリット

  • ✅ 検索結果のリッチスニペット表示
  • ✅ CTR（クリック率）の向上
  • ✅ Googleアシスタント等の音声検索対応
  • ✅ ナレッジグラフへの掲載可能性
  • ✅ より正確なコンテンツ理解

検証ツール

Googleのリッチリザルトテストで構造化データの実装が正しいか確認できます。

11.7 サイトマップとrobots.txt

Next.jsでは、サイトマップとrobots.txtを簡単に生成できます。これらはクローラーがサイトを効率的に巡回するために重要です。

app/sitemap.ts（動的サイトマップ） 
import { MetadataRoute } from 'next'

export default async function sitemap(): Promise<MetadataRoute.Sitemap> {
  // 静的ページ
  const staticPages = [
    {
      url: 'https://example.com',
      lastModified: new Date(),
      changeFrequency: 'yearly' as const,
      priority: 1,
    },
    {
      url: 'https://example.com/about',
      lastModified: new Date(),
      changeFrequency: 'monthly' as const,
      priority: 0.8,
    },
  ]

  // 動的ページ（記事一覧を取得）
  const posts = await fetch('https://example.com/api/posts').then(r => r.json())
  const postPages = posts.map((post) => ({
    url: `https://example.com/blog/${post.slug}`,
    lastModified: new Date(post.updatedAt),
    changeFrequency: 'weekly' as const,
    priority: 0.6,
  }))

  return [...staticPages, ...postPages]
}
app/robots.ts 
import { MetadataRoute } from 'next'

export default function robots(): MetadataRoute.Robots {
  return {
    rules: [
      {
        userAgent: '*',
        allow: '/',
        disallow: ['/admin/', '/api/'], // クロール禁止パス
      },
      {
        userAgent: 'Googlebot',
        allow: '/',
        crawlDelay: 1, // クロール間隔（秒）
      },
    ],
    sitemap: 'https://example.com/sitemap.xml',
  }
}

ベストプラクティス

  • サイトマップは50,000 URL以内: 超える場合は分割してサイトマップインデックスを作成
  • 優先度（priority）は相対的: 最重要ページを1.0、その他を相対的に設定
  • changeFrequencyは参考値: Googleは実際の更新頻度を自動検出
  • robots.txtで管理画面を保護: disallowで非公開パスを指定

11.8 パフォーマンス計測ツールの活用

パフォーマンス最適化は計測なしには改善できません。主要なツールを活用して継続的にモニタリングしましょう。

Lighthouse

Chrome DevToolsに統合されたGoogle公式の総合診断ツール。パフォーマンス、アクセシビリティ、SEO、PWAを100点満点で評価。

使い方: Chrome DevTools → Lighthouse → Generate report

WebPageTest

実機環境でのテストが可能。世界中のロケーション・デバイス・回線速度を選択してリアルな条件で計測できます。

URL: webpagetest.org

Google Search Console

Core Web Vitalsレポートで実ユーザーのパフォーマンスデータを確認。インデックス状況やSEO問題も診断できます。

URL: search.google.com/search-console

Next.js Analytics

Vercelが提供する実ユーザーモニタリング（RUM）。Core Web Vitalsをリアルタイムで追跡し、ページごとのパフォーマンスを可視化。

有効化: Vercelダッシュボード → Analytics

パフォーマンス監査チェックリスト

  • Lighthouse スコア 90点以上（特にPerformance）
  • LCP 2.5秒以内、FID/INP 100ms/200ms以内、CLS 0.1未満
  • JavaScript バンドルサイズ 200KB以内（圧縮後）
  • 画像を next/image で最適化、WebP/AVIF形式を使用
  • フォントを next/font で最適化、font-display: swap設定
  • サードパーティスクリプトを next/script で遅延読み込み
  • サイトマップとrobots.txtが適切に設定されている
  • 構造化データが実装され、リッチリザルトテストに合格

理解度確認クイズ

  1. Q1. Core Web Vitalsの3つの指標とその目標値を説明してください。

    回答を表示

    ①LCP（Largest Contentful Paint）: 最大コンテンツの表示時間、目標2.5秒以内、②FID/INP（First Input Delay / Interaction to Next Paint）: 初回入力遅延/次のペイントまでのインタラクション時間、目標100ms/200ms以内、③CLS（Cumulative Layout Shift）: レイアウトシフトの累積量、目標0.1未満。これらはユーザー体験の質を測る指標で、検索ランキングにも影響します。

  2. Q2. バンドルサイズを削減する具体的な手法を3つ挙げてください。

    回答を表示

    ①Tree Shaking対応のライブラリを選択し、Named Importを使用する、②next/dynamicで重いコンポーネントを動的インポートする、③サーバーコンポーネントで処理を行い、クライアントに送信するJavaScriptを削減する。また、@next/bundle-analyzerでバンドル構成を可視化し、大きなライブラリを特定して置き換えることも効果的です。

  3. Q3. 構造化データ（JSON-LD）を実装するメリットは何ですか？

    回答を表示

    構造化データを実装すると、検索エンジンがページの内容をより正確に理解でき、検索結果にリッチスニペット（レビュー星、FAQ、パンくずリストなど）が表示される可能性が高まります。これによりCTR（クリック率）が向上し、Googleアシスタント等の音声検索にも対応しやすくなります。また、ナレッジグラフへの掲載可能性も高まります。

この章のまとめ

  • Core Web Vitals（LCP、FID/INP、CLS）はユーザー体験と検索ランキングの両方に影響する重要指標
  • バンドルアナライザーと動的インポートを活用してJavaScriptサイズを最適化できる
  • Metadata APIによる型安全なSEO最適化と構造化データでリッチスニペット表示を実現
  • Lighthouse、WebPageTest、Google Search Consoleで継続的にパフォーマンスを計測・改善
