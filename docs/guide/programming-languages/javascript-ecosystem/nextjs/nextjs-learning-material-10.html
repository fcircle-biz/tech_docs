国際化（i18n）対応

Next.jsアプリケーションを多言語対応にする方法を学びます。ルーティングの国際化、翻訳ファイルの管理、動的な言語切り替え、SEO対応など、グローバル展開に必要な技術を習得します。

この章で学ぶこと

  • Next.jsにおける国際化（i18n）の基本概念とルーティング設定
  • 翻訳ファイルの管理とnext-intlライブラリの実装
  • 動的な言語切り替え機能と日付・数値のフォーマット
  • SEOのための多言語対応とhreflangタグの設定

10.1 国際化（i18n）の基礎

国際化（Internationalization、i18n）は、アプリケーションを複数の言語や地域に対応させることを指します。Next.jsでは、App Routerでの柔軟な国際化対応が可能です。

i18nとl10nの違い

  • i18n（国際化）：アプリケーションを多言語対応可能な構造にすること
  • l10n（ローカライゼーション）：特定の言語・地域向けに翻訳や調整を行うこと
flowchart LR A[ユーザーアクセス] --> B{言語検出} B -->|ja| C["日本語版
ja/"] B -->|en| D["英語版
en/"] B -->|zh| E["中国語版
zh/"] C --> F[翻訳ファイル
ja.json] D --> G[翻訳ファイル
en.json] E --> H[翻訳ファイル
zh.json] F --> I[レンダリング] G --> I H --> I

重要ポイント

Next.js App Routerでは、URLベース（/en/about、/ja/aboutなど）の国際化が標準的なアプローチです。これによりSEOにも有利になります。

10.2 ルーティングの国際化設定

App Routerでは、Dynamic Segmentsを使用して言語ごとのルーティングを実装します。

国際化対応のプロジェクト構成 

app/
├── [locale]/              # 動的言語セグメント
│   ├── layout.tsx         # 言語ごとのレイアウト
│   ├── page.tsx           # トップページ
│   ├── about/
│   │   └── page.tsx       # /[locale]/about
│   └── blog/
│       ├── page.tsx       # /[locale]/blog
│       └── [slug]/
│           └── page.tsx   # /[locale]/blog/[slug]
├── middleware.ts          # 言語リダイレクト処理
└── i18n/
    ├── request.ts         # リクエストごとの翻訳設定
    └── routing.ts         # ルーティング設定

ルーティング設定の実装

i18n/routing.ts 
import { defineRouting } from 'next-intl/routing';
import { createNavigation } from 'next-intl/navigation';

// サポートするロケールを定義
export const routing = defineRouting({
  locales: ['ja', 'en', 'zh'], // 日本語、英語、中国語
  defaultLocale: 'ja',         // デフォルト言語
  localePrefix: 'always'       // URLに常にロケールを含める
});

// 型安全なナビゲーションヘルパーを生成
export const { Link, redirect, usePathname, useRouter } =
  createNavigation(routing);

Middlewareでの言語検出

middleware.ts 
import createMiddleware from 'next-intl/middleware';
import { routing } from './i18n/routing';

// next-intlのミドルウェアを作成
export default createMiddleware(routing);

export const config = {
  // ロケール検出を適用するパスを指定
  matcher: ['/', '/(ja|en|zh)/:path*']
};

ヒント

Middlewareは、ブラウザのAccept-Languageヘッダーやクッキーから自動的にユーザーの言語を検出し、適切なロケールにリダイレクトします。

10.3 翻訳ファイルの管理

翻訳テキストはJSON形式で管理し、ネームスペースごとに分割することで保守性を高めます。

翻訳ファイルの構成 

messages/
├── ja/
│   ├── common.json        # 共通翻訳
│   ├── home.json          # トップページ翻訳
│   ├── about.json         # Aboutページ翻訳
│   └── blog.json          # ブログページ翻訳
├── en/
│   ├── common.json
│   ├── home.json
│   ├── about.json
│   └── blog.json
└── zh/
    ├── common.json
    ├── home.json
    ├── about.json
    └── blog.json

next-intlの設定

i18n/request.ts 
import { getRequestConfig } from 'next-intl/server';
import { routing } from './routing';

export default getRequestConfig(async ({ locale }) => {
  // ロケールの検証
  if (!routing.locales.includes(locale as any)) {
    locale = routing.defaultLocale;
  }

  return {
    messages: (await import(`../messages/${locale}/common.json`)).default
  };
});

10.4 コンポーネントでの翻訳使用

サーバーコンポーネントとクライアントコンポーネントで異なる方法で翻訳を取得します。

サーバーコンポーネントでの使用

app/[locale]/page.tsx 
import { useTranslations } from 'next-intl';

export default function HomePage() {
  // 翻訳関数を取得
  const t = useTranslations('common');

  return (
    <div>
      <h1>{t('nav.home')}</h1>
      <nav>
        <a href="/about">{t('nav.about')}</a>
        <a href="/blog">{t('nav.blog')}</a>
      </nav>
    </div>
  );
}

クライアントコンポーネントでの使用

components/LanguageSwitcher.tsx 
'use client';

import { useLocale } from 'next-intl';
import { useRouter, usePathname } from '@/i18n/routing';

export default function LanguageSwitcher() {
  const locale = useLocale();
  const router = useRouter();
  const pathname = usePathname();

  const changeLanguage = (newLocale: string) => {
    // 現在のパスで言語を切り替え
    router.replace(pathname, { locale: newLocale });
  };

  return (
    <select value={locale} onChange={(e) => changeLanguage(e.target.value)}>
      <option value="ja">日本語</option>
      <option value="en">English</option>
      <option value="zh">中文</option>
    </select>
  );
}

10.5 日付・数値・通貨のフォーマット

next-intlは、Intl APIを使用した日付・数値・通貨のローカライズをサポートしています。

components/FormattedContent.tsx 
import { useLocale, useFormatter } from 'next-intl';

export default function FormattedContent() {
  const locale = useLocale();
  const format = useFormatter();

  const date = new Date('2025-12-08');
  const price = 12345.67;
  const quantity = 1234567;

  return (
    <div>
      {/* 日付フォーマット */}
      <p>日付: {format.dateTime(date, {
        year: 'numeric',
        month: 'long',
        day: 'numeric'
      })}</p>
      {/* ja: 2025年12月8日, en: December 8, 2025 */}

      {/* 数値フォーマット */}
      <p>数値: {format.number(quantity)}</p>
      {/* ja: 1,234,567, en: 1,234,567 */}

      {/* 通貨フォーマット */}
      <p>価格: {format.number(price, {
        style: 'currency',
        currency: locale === 'ja' ? 'JPY' : 'USD'
      })}</p>
      {/* ja: ¥12,346, en: $12,345.67 */}
    </div>
  );
}

10.6 SEOと多言語対応

検索エンジンが各言語版を正しく認識できるよう、hreflangタグとメタデータを設定します。

メタデータの国際化

app/[locale]/layout.tsx 
import { Metadata } from 'next';
import { routing } from '@/i18n/routing';

type Props = {
  params: { locale: string };
};

// 各ロケールのメタデータを定義
const metaByLocale: Record<string, Metadata> = {
  ja: {
    title: 'F-Circle - ビジネスソリューション',
    description: '最高のビジネスソリューションを提供します'
  },
  en: {
    title: 'F-Circle - Business Solutions',
    description: 'Providing the best business solutions'
  }
};

export function generateMetadata({ params }: Props): Metadata {
  const locale = params.locale;
  const meta = metaByLocale[locale] || metaByLocale.ja;

  return {
    ...meta,
    // hreflangタグの設定
    alternates: {
      canonical: `https://example.com/${locale}`,
      languages: {
        ja: 'https://example.com/ja',
        en: 'https://example.com/en',
        zh: 'https://example.com/zh'
      }
    }
  };
}

generateStaticParamsでの言語別ページ生成

app/[locale]/blog/[slug]/page.tsx 
import { routing } from '@/i18n/routing';

// 全言語×全記事のパスを生成
export async function generateStaticParams() {
  const posts = await getPosts(); // 記事一覧取得

  return routing.locales.flatMap((locale) =>
    posts.map((post) => ({
      locale,
      slug: post.slug
    }))
  );
}

10.7 パフォーマンスとベストプラクティス

推奨事項

  • URLベースのロケール管理（/ja/、/en/）
  • 翻訳ファイルをネームスペースで分割
  • サーバーコンポーネントで翻訳を取得
  • generateStaticParamsで事前生成
  • hreflangタグを必ず設定

避けるべき事項

  • クライアント側のみでの言語管理
  • 全翻訳を1つのJSONファイルに集約
  • 翻訳キーのハードコード
  • デフォルト言語の未設定
  • 言語切り替え時のフルリロード

トラブルシューティング

翻訳が表示されない

原因: 翻訳ファイルのパスが間違っているか、キーが存在しない

解決方法: messages/[locale]/common.jsonが正しく配置されているか確認し、翻訳キーのスペルをチェック

言語切り替えが動作しない

原因: Middlewareが正しく設定されていない

解決方法: middleware.tsのmatcherパターンが正しいか確認し、開発サーバーを再起動

理解度確認クイズ

  1. Q1. Next.jsのApp Routerで推奨される国際化のアプローチは？

    回答を表示

    URLベースの国際化（/ja/about、/en/aboutなど）が推奨されます。これにより、SEOに有利で、各言語版を独立したURLとして認識できます。

  2. Q2. hreflangタグの役割は何ですか？

    回答を表示

    hreflangタグは、検索エンジンに対して同じコンテンツの異なる言語版が存在することを伝えます。これにより、ユーザーの言語や地域に応じて適切なページが検索結果に表示されます。

  3. Q3. 翻訳ファイルを管理する際のベストプラクティスは？

    回答を表示

    翻訳ファイルをネームスペース（common.json、home.json、blog.jsonなど）ごとに分割し、保守性を高めます。また、翻訳キーは階層構造（nav.home、footer.copyrightなど）で管理すると、関連性が明確になります。

この章のまとめ

  • Next.js App RouterではURLベース（/[locale]/）の国際化が標準的で、SEOにも有利
  • next-intlライブラリを使用すると、翻訳管理とルーティングが簡単に実装できる
  • 日付・数値・通貨のフォーマットは、useFormatterフックで自動的にローカライズされる
  • hreflangタグとメタデータの設定により、検索エンジンが各言語版を正しく認識できる
