認証とセッション管理

NextAuth.js（Auth.js）を使用した認証の実装、OAuthプロバイダーの設定、JWTとセッションの管理、ミドルウェアによるルート保護、そしてセキュアなアプリケーション構築の手法を学びます。

この章で学ぶこと

  • NextAuth.js（Auth.js）のセットアップと基本設定
  • Google、GitHub等のOAuthプロバイダーとの連携
  • JWTとセッションの違いと適切な使い分け
  • ミドルウェアによるルート保護とロールベースアクセス制御

9.1 認証の基礎知識

Webアプリケーションにおける認証（Authentication）とは、ユーザーが本人であることを確認するプロセスです。一方、認可（Authorization）は、認証されたユーザーが特定のリソースにアクセスする権限を持っているかを判断します。

flowchart LR A[ユーザー] --> B{認証
Authentication} B -->|本人確認成功| C[セッション/JWT発行] B -->|失敗| D[ログインページへ] C --> E{認可
Authorization} E -->|権限あり| F[リソースへアクセス] E -->|権限なし| G[403 Forbidden]

認証方式の種類

  • パスワードベース認証：メールアドレス/パスワードによる伝統的な認証
  • OAuth 2.0：Google、GitHub等の外部プロバイダーによる認証
  • 多要素認証（MFA）：パスワード + SMS/認証アプリによる2段階認証
  • マジックリンク：メールで送信されたリンクをクリックして認証
  • 生体認証：指紋、顔認識等（WebAuthn API使用）

セッション vs JWT

項目 セッションベース JWTベース
保存場所 サーバー側のDB/メモリ クライアント側（Cookie/LocalStorage）
スケーラビリティ ⚠️ サーバー負荷増 ✅ ステートレスで水平スケール可能
即時無効化 ✅ サーバー側で即座に削除可能 ⚠️ 有効期限まで無効化困難
セキュリティ ✅ セッションIDのみをCookieに保存 ⚠️ トークン盗難リスク（XSS対策必須）
推奨用途 管理画面、高セキュリティ要求 API、モバイルアプリ、マイクロサービス

9.2 NextAuth.jsのセットアップ

NextAuth.js（現在はAuth.jsにブランド名変更）は、Next.js専用の認証ライブラリです。OAuth、パスワード認証、マジックリンクなど、多様な認証方式に対応し、セッション管理も自動化されています。

実習 9-1: NextAuth.jsのインストールと基本設定

NextAuth.jsをインストールし、Google OAuthプロバイダーを使用した認証を実装します。

手順

  1. 1 NextAuth.jsをインストール
  2. 2 環境変数の設定
  3. 3 Route Handlerの作成

実行例

terminal 
# NextAuth.jsのインストール
npm install next-auth
.env.local 
# NextAuth.js認証シークレット
NEXTAUTH_SECRET=your-secret-key-here
NEXTAUTH_URL=http://localhost:3000

# Google OAuthクレデンシャル
GOOGLE_CLIENT_ID=your-google-client-id
GOOGLE_CLIENT_SECRET=your-google-client-secret
app/api/auth/[...nextauth]/route.ts 
import NextAuth from 'next-auth';
import GoogleProvider from 'next-auth/providers/google';

// NextAuth設定
const handler = NextAuth({
  providers: [
    GoogleProvider({
      clientId: process.env.GOOGLE_CLIENT_ID!,
      clientSecret: process.env.GOOGLE_CLIENT_SECRET!,
    }),
  ],
});

export { handler as GET, handler as POST };

Google OAuth認証情報の取得

Google Cloud Consoleで新しいプロジェクトを作成し、OAuth 2.0クライアントIDを取得してください。承認済みのリダイレクトURIには http://localhost:3000/api/auth/callback/google を設定します。

9.3 サインイン/サインアウトの実装

NextAuth.jsが提供する signIn() と signOut() 関数を使用して、簡単に認証フローを実装できます。

app/components/LoginButton.tsx 
'use client';

import { signIn, signOut, useSession } from 'next-auth/react';

export default function LoginButton() {
  const { data: session } = useSession();

  // ログインしていない場合
  if (!session) {
    return (
      <button onClick={() => signIn('google')}>
        Googleでログイン
      </button>
    );
  }

  // ログイン済みの場合
  return (
    <div>
      <p>ようこそ、{session.user?.name}さん</p>
      <button onClick={() => signOut()}>ログアウト</button>
    </div>
  );
}

9.4 ミドルウェアによるルート保護

Next.jsのMiddlewareを使用すると、ページにアクセスする前に認証状態を確認し、未認証ユーザーをログインページにリダイレクトできます。

middleware.ts 
import { withAuth } from 'next-auth/middleware';

// ミドルウェアで認証をチェック
export default withAuth({
  callbacks: {
    authorized: ({ token }) => !!token, // トークンがあればアクセス許可
  },
});

// 保護対象のルートを指定
export const config = {
  matcher: ['/dashboard/:path*', '/admin/:path*'],
};

ミドルウェアの動作

/dashboard や /admin 配下のページにアクセスしようとすると、ミドルウェアが認証状態を確認します。未認証の場合は自動的に /api/auth/signin にリダイレクトされます。

9.5 ロールベースアクセス制御（RBAC）

ユーザーに役割（Role）を割り当て、役割に応じてアクセス権限を制御する仕組みです。例えば、管理者（admin）のみが特定のページにアクセスできるよう制限できます。

app/api/auth/[...nextauth]/route.ts 
import NextAuth from 'next-auth';
import GoogleProvider from 'next-auth/providers/google';

const handler = NextAuth({
  providers: [GoogleProvider({ /* ... */ })],
  callbacks: {
    // JWTトークンにロール情報を追加
    async jwt({ token, user }) {
      if (user) {
        token.role = user.role || 'user'; // デフォルトはuser
      }
      return token;
    },
    // セッションにロール情報を含める
    async session({ session, token }) {
      if (session.user) {
        session.user.role = token.role;
      }
      return session;
    },
  },
});

export { handler as GET, handler as POST };
middleware.ts（ロールチェック） 
import { withAuth } from 'next-auth/middleware';

export default withAuth({
  callbacks: {
    authorized: ({ token, req }) => {
      // /adminパスは管理者のみアクセス可能
      if (req.nextUrl.pathname.startsWith('/admin')) {
        return token?.role === 'admin';
      }
      // その他は認証済みであればOK
      return !!token;
    },
  },
});

export const config = {
  matcher: ['/dashboard/:path*', '/admin/:path*'],
};

9.6 CSRFトークンとセキュリティ対策

NextAuth.jsはCSRF（Cross-Site Request Forgery）攻撃を防ぐため、デフォルトでCSRFトークンを自動生成します。また、セキュアなCookie設定により、セッションハイジャックを防止します。

セキュリティのベストプラクティス

  • 環境変数を安全に管理： .env.local をGit管理から除外
  • HTTPSを必須に：本番環境では必ずHTTPS通信を使用
  • セッション有効期限を設定：長期間有効なセッションは避ける
  • JWTシークレットは強力に：最低32文字のランダムな文字列を使用
セキュアなNextAuth設定例 
const handler = NextAuth({
  providers: [/* ... */],
  session: {
    strategy: 'jwt', // JWTベースのセッション
    maxAge: 30 * 24 * 60 * 60, // 30日間有効
  },
  cookies: {
    sessionToken: {
      name: '__Secure-next-auth.session-token',
      options: {
        httpOnly: true, // JavaScriptからアクセス不可
        sameSite: 'lax', // CSRF対策
        path: '/',
        secure: true, // HTTPS必須
      },
    },
  },
});

トラブルシューティング

ログイン後にリダイレクトされない

原因: NextAuth.jsのコールバックURLが正しく設定されていない可能性があります。

解決方法: Google Cloud Consoleで承認済みのリダイレクトURIに http://localhost:3000/api/auth/callback/google が登録されているか確認してください。

セッション情報が取得できない

原因: SessionProvider でラップされていない可能性があります。

解決方法: app/layout.tsx で <SessionProvider> を使用し、全体をラップしてください。

理解度確認クイズ

  1. Q1. セッションベース認証とJWTベース認証の主な違いは何ですか？

    回答を表示

    セッションベース認証はサーバー側にセッション情報を保存し、クライアントにはセッションIDのみを渡します。一方、JWTベース認証はクライアント側にトークンを保存し、ステートレスな認証が可能です。セッションベースは即座に無効化できますが、JWTは有効期限まで無効化が困難です。

  2. Q2. ロールベースアクセス制御（RBAC）とは何ですか？

    回答を表示

    ユーザーに役割（Role）を割り当て、その役割に応じてアクセス権限を制御する仕組みです。例えば、管理者（admin）のみが管理画面にアクセスでき、一般ユーザー（user）は閲覧のみ可能といった制御が実現できます。

  3. Q3. CSRF攻撃を防ぐためにNextAuth.jsが行っている対策を説明してください。

    回答を表示

    NextAuth.jsはデフォルトでCSRFトークンを自動生成し、リクエストごとに検証します。また、Cookieの sameSite 属性を lax に設定することで、クロスサイトリクエストを制限しています。
