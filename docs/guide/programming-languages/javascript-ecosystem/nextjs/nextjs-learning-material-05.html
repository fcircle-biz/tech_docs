API Routesでバックエンド開発

Next.jsのRoute Handlersを使って、同じリポジトリ内でフロントエンドとバックエンドAPIを構築する方法を学びます。RESTful API、認証、データベース連携、エラーハンドリングなど、実践的なバックエンド開発の基礎を習得します。

この章で学ぶこと

  • Route Handlersの作成とHTTPメソッド（GET、POST、PUT、DELETE）の処理
  • リクエストの受け取り方（パラメータ、クエリ、ボディ）とレスポンスの返し方
  • ミドルウェアの実装とCORS設定による外部からのアクセス制御
  • 認証・認可の実装、データベース接続、外部API連携、エラーハンドリング

5.1 Route Handlersとは

Route Handlersは、Next.jsのApp Routerで提供されるバックエンドAPI構築の仕組みです。従来のPages RouterではAPI Routesと呼ばれていました。

Route Handlersを使うことで、フロントエンドとバックエンドを同じリポジトリ内で管理でき、フルスタック開発が可能になります。

Route Handlersの特徴

  • ファイルベースルーティング：app/api/フォルダ内の構造がそのままAPIエンドポイントになる
  • HTTPメソッド対応：GET、POST、PUT、PATCH、DELETE、OPTIONS、HEADをサポート
  • Web標準API：RequestとResponseオブジェクトを使用
  • サーバーサイド実行：データベース接続やシークレット情報の安全な扱いが可能
  • Edge/Node.js Runtime：実行環境を選択可能

Route Handler] B -->|データ取得| C[データベース] B -->|API呼び出し| D[外部API] B -->|認証確認| E[認証サービス] B -->|JSONレスポンス| A

5.2 Route Handlersの作成

Route Handlersはapp/api/フォルダ内にroute.tsという名前のファイルを作成します。

API Routesのファイル構造 

app/
├── api/
│   ├── hello/
│   │   └── route.ts           → /api/hello
│   ├── users/
│   │   ├── route.ts           → /api/users
│   │   └── [id]/
│   │       └── route.ts       → /api/users/:id
│   └── products/
│       ├── route.ts           → /api/products
│       └── [slug]/
│           └── route.ts       → /api/products/:slug

実習 5-1: シンプルなGET APIを作成する

「Hello, API!」を返すシンプルなGET APIを作成します。

app/api/hello/route.ts

TypeScript 
// NextRequestとNextResponseをインポート
import { NextResponse } from 'next/server';

// GETメソッドのハンドラー
export async function GET() {
  return NextResponse.json({
    message: 'Hello, API!',
    timestamp: new Date().toISOString(),
  });
}

動作確認

ブラウザで http://localhost:3000/api/hello にアクセスすると、JSON形式のレスポンスが表示されます。

5.3 HTTPメソッドの処理

Route Handlersでは、HTTPメソッドごとに名前付きエクスポート関数を定義します。

サポートされているHTTPメソッド

メソッド 用途 例
GET データの取得 ユーザー情報取得、記事一覧取得
POST データの作成 新規ユーザー登録、記事投稿
PUT データの完全更新 ユーザー情報全体の更新
PATCH データの部分更新 ユーザー名のみ変更
DELETE データの削除 記事の削除、アカウント削除

実習 5-2: CRUD操作を実装する

ユーザーデータのCRUD（作成、読み取り、更新、削除）操作を実装します。

app/api/users/route.ts

TypeScript 
import { NextRequest, NextResponse } from 'next/server';

// インメモリデータ（本番環境ではデータベースを使用）
let users = [
  { id: 1, name: '山田太郎', email: 'yamada@example.com' },
  { id: 2, name: '佐藤花子', email: 'sato@example.com' },
];

// GET: 全ユーザー取得
export async function GET() {
  return NextResponse.json(users);
}

// POST: 新規ユーザー作成
export async function POST(request: NextRequest) {
  const body = await request.json();
  const newUser = {
    id: users.length + 1,
    name: body.name,
    email: body.email,
  };
  users.push(newUser);
  return NextResponse.json(newUser, { status: 201 });
}

クエリパラメータ

  request: NextRequest

    .searchParams.get('page');
  // ...
}

リクエストボディ 

// POST /api/users
export async function POST(
  request: NextRequest
) {
  const body = await
    request.json();
  // ...
}

レスポンスの種類

  • NextResponse.json(data) - JSON形式で返す
  • NextResponse.redirect(url) - リダイレクト
  • new Response(text) - プレーンテキストで返す
  • NextResponse.json(error, {`{status: 500}`}) - エラーレスポンス

5.5 エラーハンドリング

APIでは適切なエラーハンドリングが重要です。HTTPステータスコードとエラーメッセージを適切に返すことで、クライアント側でのデバッグが容易になります。

実習 5-3: エラーハンドリングを実装する

app/api/users/[id]/route.ts

TypeScript 
import { NextRequest, NextResponse } from 'next/server';

// GET: 特定ユーザー取得
export async function GET(
  request: NextRequest,
  { params }: { params: { id: string } }
) {
  try {
    const id = parseInt(params.id);

    // IDのバリデーション
    if (isNaN(id)) {
      return NextResponse.json(
        { error: 'Invalid user ID' },
        { status: 400 } // Bad Request
      );
    }

    // ユーザー検索（仮のデータ）
    const user = users.find(u => u.id === id);

    if (!user) {
      return NextResponse.json(
        { error: 'User not found' },
        { status: 404 } // Not Found
      );
    }

    return NextResponse.json(user);
  } catch (error) {
    return NextResponse.json(
      { error: 'Internal server error' },
      { status: 500 }
    );
  }
}

主要なHTTPステータスコード

コード 意味 使用例
200 OK リクエスト成功
201 Created リソース作成成功
400 Bad Request 不正なリクエスト
401 Unauthorized 認証が必要
404 Not Found リソースが見つからない
500 Internal Server Error サーバー内部エラー

5.6 CORS設定

CORS（Cross-Origin Resource Sharing）は、異なるドメインからのAPIアクセスを制御する仕組みです。外部のフロントエンドアプリからNext.js APIにアクセスする場合に必要です。

CORSエラーとは

異なるドメイン（例: http://example.com から http://localhost:3000）へのリクエストは、デフォルトでブロックされます。これはブラウザのセキュリティ機能です。

実習 5-4: CORS設定を追加する

app/api/public-data/route.ts

TypeScript 
import { NextRequest, NextResponse } from 'next/server';

export async function GET(request: NextRequest) {
  // データ取得
  const data = { message: 'This is public data' };

  // CORSヘッダー付きレスポンス
  return NextResponse.json(data, {
    headers: {
      'Access-Control-Allow-Origin': '*', // 全てのドメインを許可
      'Access-Control-Allow-Methods': 'GET, POST, PUT, DELETE',
      'Access-Control-Allow-Headers': 'Content-Type',
    },
  });
}

// OPTIONSメソッド（プリフライトリクエスト）
export async function OPTIONS() {
  return new Response(null, {
    headers: {
      'Access-Control-Allow-Origin': '*',
      'Access-Control-Allow-Methods': 'GET, POST, PUT, DELETE',
      'Access-Control-Allow-Headers': 'Content-Type',
    },
  });
}

5.7 認証・認可の実装

APIエンドポイントを保護するために、認証（Authentication）と認可（Authorization）を実装します。一般的にはJWT（JSON Web Token）を使用します。

sequenceDiagram participant Client as クライアント participant API as Route Handler participant Auth as 認証システム Client->>API: リクエスト + トークン API->>API: トークン検証 alt トークン有効 API->>Auth: ユーザー情報取得 Auth-->>API: ユーザー情報 API-->>Client: データ返却 else トークン無効 API-->>Client: 401 Unauthorized end

実習 5-5: 認証ミドルウェアを実装する

app/api/protected/route.ts

TypeScript 
import { NextRequest, NextResponse } from 'next/server';

// 認証トークンを検証する関数（簡易版）
function verifyToken(token: string): boolean {
  // 実際はJWTライブラリを使用して検証
  return token === 'valid-token-12345';
}

export async function GET(request: NextRequest) {
  // Authorizationヘッダーからトークン取得
  const authHeader = request.headers.get('authorization');

  if (!authHeader || !authHeader.startsWith('Bearer ')) {
    return NextResponse.json(
      { error: 'Unauthorized: No token provided' },
      { status: 401 }
    );
  }

  const token = authHeader.split(' ')[1];

  // トークン検証
  if (!verifyToken(token)) {
    return NextResponse.json(
      { error: 'Unauthorized: Invalid token' },
      { status: 401 }
    );
  }

  // 認証成功 - 保護されたデータを返す
  return NextResponse.json({
    message: 'This is protected data',
    userId: 123,
  });
}

5.8 データベース接続

Route Handlersから直接データベースに接続できます。ORM（Object-Relational Mapping）として、Prismaの使用が推奨されます。

Prismaとは

Prismaは、Node.js/TypeScript向けのモダンなORM（Object-Relational Mapping）です。型安全なデータベース操作が可能で、PostgreSQL、MySQL、SQLiteなどに対応しています。

実習 5-6: Prismaでデータベースからデータ取得

app/api/posts/route.ts

TypeScript 
import { NextResponse } from 'next/server';
import { PrismaClient } from '@prisma/client';

const prisma = new PrismaClient();

// GET: 全記事取得
export async function GET() {
  try {
    const posts = await prisma.post.findMany({
      include: {
        author: true, // リレーション含む
      },
      orderBy: {
        createdAt: 'desc', // 作成日時で降順ソート
      },
    });

    return NextResponse.json(posts);
  } catch (error) {
    return NextResponse.json(
      { error: 'Failed to fetch posts' },
      { status: 500 }
    );
  }
}

5.9 外部APIとの連携

Route Handlersから外部APIを呼び出すことで、天気情報、決済サービス、SNS連携など、様々な機能を統合できます。

実習 5-7: 外部APIから天気情報を取得する

app/api/weather/route.ts

TypeScript 
import { NextRequest, NextResponse } from 'next/server';

export async function GET(request: NextRequest) {
  // クエリパラメータから都市名を取得
  const city = request.nextUrl.searchParams.get('city') || 'Tokyo';

  try {
    // 外部API呼び出し（OpenWeatherMap API）
    const apiKey = process.env.WEATHER_API_KEY;
    const response = await fetch(
      `https://api.openweathermap.org/data/2.5/weather?q=${city}&appid=${apiKey}&units=metric`
    );

    if (!response.ok) {
      return NextResponse.json(
        { error: 'Failed to fetch weather data' },
        { status: response.status }
      );
    }

    const data = await response.json();

    // レスポンスを整形して返す
    return NextResponse.json({
      city: data.name,
      temperature: data.main.temp,
      description: data.weather[0].description,
      humidity: data.main.humidity,
    });
  } catch (error) {
    return NextResponse.json(
      { error: 'Internal server error' },
      { status: 500 }
    );
  }
}

環境変数の重要性

APIキーやシークレット情報は必ず環境変数に格納し、.env.localファイルで管理してください。コードに直接書くと、GitHubに公開された際に悪用されるリスクがあります。

理解度確認クイズ

  1. Q1. Route HandlersとPages Routerの違いは何ですか？

    回答を表示

    Route HandlersはApp Router（Next.js 13+）で使用されるバックエンドAPI構築の仕組みで、app/api/フォルダ内にroute.tsファイルを作成します。Pages Routerではpages/api/にAPI Routesを作成します。両者とも機能は似ていますが、ファイル名と配置場所が異なります。

  2. Q2. CORSとは何ですか？なぜ必要ですか？

    回答を表示

    CORS（Cross-Origin Resource Sharing）は、異なるドメインからのAPIアクセスを制御する仕組みです。ブラウザのセキュリティ機能により、デフォルトでは異なるドメイン間のリクエストがブロックされます。外部のフロントエンドアプリからNext.js APIにアクセスする場合は、CORSヘッダーを設定してアクセスを許可する必要があります。

  3. Q3. APIキーを安全に管理する方法を説明してください。

    回答を表示

    APIキーやシークレット情報は環境変数に格納し、.env.localファイルで管理します。このファイルは.gitignoreに追加してGit管理対象外にすることで、GitHubなどに公開されないようにします。本番環境では、Vercelなどのホスティングサービスの環境変数設定機能を使用します。
