Server Actionsでフォーム処理

Server Actionsを活用したフォーム送信・データ更新処理、バリデーション、楽観的更新、そしてuseFormStateとuseFormStatusによる状態管理について実践的に学びます。

この章で学ぶこと

  • Server Actionsの基本概念とAPI Routesとの違い
  • フォーム送信の処理とバリデーションの実装
  • 楽観的更新とプログレッシブエンハンスメントの実現
  • useFormStateとuseFormStatusによる状態管理とUI反映

7.1 Server Actionsとは

Server Actionsは、Next.js 13以降で導入された機能で、サーバーサイドで実行される非同期関数をクライアントから直接呼び出すことができます。これにより、API Routesを作成せずにフォーム送信やデータ更新が可能になります。

関数実行] D[データベース

キャッシュ更新] end A --> C B --> C C --> D D --> E E --> F[自動的に
UI更新]

Server Actionsの特徴

  • API Routesが不要：別途エンドポイントを作成せずにサーバー処理を実行
  • 型安全性：TypeScriptで引数と戻り値の型が保証される
  • プログレッシブエンハンスメント：JavaScriptなしでもフォームが動作
  • 自動シリアライゼーション：FormDataが自動的にサーバーに送信される
  • 組み込みセキュリティ：CSRF保護が自動的に適用される

従来のAPI Routes

  • 別ファイルでエンドポイント作成
  • URLパスを指定してfetch
  • 手動でリクエスト/レスポンス処理
  • CSRF対策を手動実装

Server Actions（推奨）

  • 同じファイルに関数定義
  • action属性で直接指定
  • FormDataが自動送信
  • CSRF保護が自動適用

7.2 Server Actionsの基本実装

Server Actionsを使用してシンプルなフォーム送信を実装してみましょう。'use server'ディレクティブで関数をサーバー側で実行することを宣言します。

実習 7-1: Server Actionでお問い合わせフォームを作成

Server Actionsを使って、お問い合わせフォームのデータをサーバー側で処理します。

実装例

app/contact/page.tsx 
// Server Actionの定義
async function submitContact(formData: FormData) {
  'use server'

  // FormDataから値を取得
  const name = formData.get('name') as string
  const email = formData.get('email') as string
  const message = formData.get('message') as string

  // サーバー側でデータ処理（例: DBに保存）
  console.log({ name, email, message })

  // 成功時は何もreturnしないか、リダイレクト
}

export default function ContactPage() {
  return (
    <form action={submitContact}>
      <input type="text" name="name" required />
      <input type="email" name="email" required />
      <textarea name="message" required></textarea>
      <button type="submit">送信</button>
    </form>
  )
}

Server Actionの定義方法

'use server'はファイルの先頭に書くか、関数の内部に書きます。関数内部に書く場合、その関数のみがサーバー側で実行されます。ファイル先頭に書くと、そのファイルのすべての関数がServer Actionsになります。

7.3 バリデーションの実装

Server Actionsでは、Zodなどのバリデーションライブラリを使ってサーバー側で入力値を検証できます。エラー時にはフィードバックをクライアントに返却します。

app/actions/contact.ts 
'use server'

import { z } from 'zod'
import { redirect } from 'next/navigation'

// バリデーションスキーマ
const contactSchema = z.object({
  name: z.string().min(2, '名前は2文字以上'),
  email: z.string().email('有効なメールアドレス'),
  message: z.string().min(10, 'メッセージは10文字以上'),
})

// Server Action
export async function submitContact(
  prevState: any,
  formData: FormData
) {
  // バリデーション
  const result = contactSchema.safeParse({
    name: formData.get('name'),
    email: formData.get('email'),
    message: formData.get('message'),
  })

  if (!result.success) {
    return { errors: result.error.flatten().fieldErrors }
  }

  // データベースに保存など
  // await db.contacts.create({ data: result.data })

  redirect('/contact/success')
}

重要ポイント

Server Actionsはサーバー側で実行されるため、クライアント側バリデーションをバイパスされる可能性があります。必ずサーバー側でバリデーションを実装してください。

7.4 useFormStateとuseFormStatus

useFormStateはServer Actionの結果を状態管理し、useFormStatusはフォームの送信状態を取得するReact Hooksです。

sequenceDiagram participant U as User participant F as Form participant A as Server Action participant S as useFormState U->>F: フォーム送信 F->>A: データ送信 Note over F: useFormStatus: pending=true A->>A: バリデーション A->>A: データ処理 A-->>S: 結果を返却 Note over F: useFormStatus: pending=false S-->>F: エラーまたは成功状態 F-->>U: UI更新
app/contact/form.tsx 
'use client'

import { useFormState, useFormStatus } from 'react-dom'
import { submitContact } from './actions/contact'

function SubmitButton() {
  const { pending } = useFormStatus()

  return (
    <button type="submit" disabled={pending}>
      {pending ? '送信中...' : '送信'}
    </button>
  )
}

export default function ContactForm() {
  const [state, formAction] = useFormState(submitContact, null)

  return (
    <form action={formAction}>
      <div>
        <input type="text" name="name" />
        {state?.errors?.name && (
          <p className="error">{state.errors.name[0]}</p>
        )}
      </div>
      <SubmitButton />
    </form>
  )
}

useFormStateとuseFormStatusの違い

Hook 用途 返却値
useFormState Server Actionの結果を管理 [state, formAction]
useFormStatus フォームの送信状態を取得 { pending, data, method, action }

7.5 楽観的更新（Optimistic Update）

楽観的更新は、サーバーの応答を待たずにUIを即座に更新する手法です。useOptimisticフックを使用して実装できます。

app/todos/page.tsx 
'use client'

import { useOptimistic } from 'react'
import { addTodo } from './actions'

export default function TodoList({ todos }) {
  const [optimisticTodos, addOptimisticTodo] = useOptimistic(
    todos,
    (state, newTodo) => [...state, newTodo]
  )

  async function handleSubmit(formData: FormData) {
    const title = formData.get('title')

    // 即座にUIを更新（楽観的更新）
    addOptimisticTodo({ id: Date.now(), title, status: 'pending' })

    // サーバー処理を実行
    await addTodo(formData)
  }

  return (
    <>
      <form action={handleSubmit}>
        <input type="text" name="title" />
        <button type="submit">追加</button>
      </form>

      <ul>
        {optimisticTodos.map(todo => (
          <li key={todo.id}>{todo.title}</li>
        ))}
      </ul>
    </>
  )
}

楽観的更新のメリット

ユーザーはサーバー応答を待たずに即座にフィードバックを得られるため、UXが向上します。ただし、エラー時のロールバック処理も考慮する必要があります。

7.6 revalidatePathとrevalidateTag

Server Actionsでデータを更新した後、Next.jsのキャッシュを無効化してUIを最新状態に更新する必要があります。revalidatePathとrevalidateTagを使用します。

revalidatePath

特定のパス配下のキャッシュを無効化

import { revalidatePath } from 'next/cache'

async function createPost(formData: FormData) {
  'use server'
  // データ作成処理
  revalidatePath('/posts')
}

revalidateTag

タグでマークされたキャッシュを無効化

import { revalidateTag } from 'next/cache'

async function updatePost(formData: FormData) {
  'use server'
  // データ更新処理
  revalidateTag('posts')
}

注意

revalidateを呼び出さないと、データ更新後も古いキャッシュが表示され続けます。Server Actionsでデータを変更した場合は、必ず適切なrevalidateを実行してください。

7.7 ファイルアップロードの実装

Server ActionsはFormDataを通じてファイルアップロードも処理できます。アップロードされたファイルはサーバー側で検証・保存します。

app/upload/actions.ts 
'use server'

import { writeFile } from 'fs/promises'
import { join } from 'path'

export async function uploadFile(formData: FormData) {
  const file = formData.get('file') as File

  if (!file) {
    return { error: 'ファイルが選択されていません' }
  }

  // ファイルサイズ制限（例: 5MB）
  if (file.size > 5 * 1024 * 1024) {
    return { error: 'ファイルサイズは5MB以下にしてください' }
  }

  // ファイル形式チェック
  const allowedTypes = ['image/jpeg', 'image/png', 'image/webp']
  if (!allowedTypes.includes(file.type)) {
    return { error: '画像ファイルのみアップロード可能です' }
  }

  // ファイル保存
  const bytes = await file.arrayBuffer()
  const buffer = Buffer.from(bytes)
  const path = join(process.cwd(), 'public/uploads', file.name)
  await writeFile(path, buffer)

  return { success: true, path: `/uploads/${file.name}` }
}

つまずきポイント

ファイルアップロードでは必ずサーバー側でファイルサイズ・形式・拡張子を検証してください。クライアント側の検証だけでは不正なファイルがアップロードされる可能性があります。

7.8 プログレッシブエンハンスメント

Server Actionsの大きな利点は、JavaScriptが無効でもフォームが動作するプログレッシブエンハンスメントを実現できることです。

flowchart TD A[ユーザーがフォーム送信] --> B{JavaScriptが有効?} B -->|有効| C[非同期でServer Action実行] C --> D[ページ遷移なしでUIを更新] B -->|無効| E[通常のHTMLフォーム送信] E --> F[サーバーでServer Action実行] F --> G[結果ページにリダイレクト] D --> H[どちらでも動作する] G --> H

重要ポイント

Server Actionsは基本的なHTMLフォームとして機能するため、JavaScriptが無効な環境やブラウザでも動作します。これによりアクセシビリティとSEOが向上します。

トラブルシューティング

Server Actionが実行されない

原因: 'use server'ディレクティブが欠けている、または関数がクライアントコンポーネントで定義されている。

解決方法: Server Actionの先頭に'use server'を追加し、別ファイルに定義するか、サーバーコンポーネント内で定義してください。

データ更新後にUIが古いまま

原因: revalidatePathまたはrevalidateTagを呼び出していない。

解決方法: Server Action内で適切なrevalidate関数を呼び出してキャッシュを無効化してください。

useFormStateがundefinedエラー

原因: React 19またはNext.js 14以降が必要ですが、古いバージョンを使用している。

解決方法: npm install next@latest react@latest react-dom@latestで最新版にアップデートしてください。

理解度確認クイズ

  1. Q1. Server ActionsとAPI Routesの主な違いを説明してください。

    回答を表示

    Server Actionsは別途エンドポイントを作成せずにサーバー処理を実行でき、フォームのaction属性で直接指定できます。TypeScriptの型安全性が保証され、CSRF保護が自動適用されます。一方、API Routesは専用のエンドポイントファイルを作成し、URLを指定してfetchする必要があります。

  2. Q2. useFormStateとuseFormStatusの違いは何ですか？

    回答を表示

    useFormStateはServer Actionの結果（エラーメッセージや成功状態）を管理するHookで、useFormStatusはフォームの送信状態（pending、data、methodなど）を取得するHookです。useFormStateは結果の表示に、useFormStatusはローディング状態の表示に使用します。

  3. Q3. データ更新後にUIが自動更新されない場合、どうすればよいですか？

    回答を表示

    Server Action内でrevalidatePath('/path')またはrevalidateTag('tag-name')を呼び出してキャッシュを無効化する必要があります。これによりNext.jsのキャッシュが更新され、UIに最新データが反映されます。

この章のまとめ

  • Server Actionsは別途API Routesを作成せずにサーバー処理を実行でき、型安全性とセキュリティが保証される
  • Zodなどのバリデーションライブラリでサーバー側の入力検証を実装し、エラー時はuseFormStateで表示する
  • useOptimisticで楽観的更新を実装し、ユーザー体験を向上させることができる
  • データ更新後はrevalidatePathまたはrevalidateTagでキャッシュを無効化し、UIを最新状態に保つ
