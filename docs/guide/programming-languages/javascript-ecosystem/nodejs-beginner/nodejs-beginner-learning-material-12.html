12.1 作成するAPIの仕様

タスク管理APIは、タスクの作成・取得・更新・削除（CRUD操作）を提供するシンプルなRESTful APIです。データはJSONファイルに保存し、サーバーを再起動してもデータが保持されるようにします。

APIエンドポイント一覧

メソッド パス 説明
GET /api/tasks 全タスクを取得
GET /api/tasks/:id 指定IDのタスクを取得
POST /api/tasks 新規タスクを作成
PUT /api/tasks/:id 指定IDのタスクを更新
DELETE /api/tasks/:id 指定IDのタスクを削除

タスクのデータ構造

各タスクは以下の構造を持ちます：

task.json 
{
  "id": 1,
  "title": "Node.jsの復習",
  "description": "非同期処理とExpressを復習する",
  "completed": false,
  "createdAt": "2025-12-31T12:00:00.000Z"
}

12.3 データモデルの作成

まず、タスクデータを管理するモデル層を作成します。ファイル操作を抽象化し、データの読み書きを一元管理します。

実習 12-2: データモデルの実装

タスクデータを管理するモデルファイルを作成します。

models/tasksModel.js 
// ファイルシステムモジュール（非同期版）
const fs = require('fs').promises;
const path = require('path');

// データファイルのパス
const dataPath = path.join(__dirname, '../data/tasks.json');

// 全タスクを取得
async function getAllTasks() {
  const data = await fs.readFile(dataPath, 'utf8');
  return JSON.parse(data);
}

// 単一タスクを取得
async function getTaskById(id) {
  const tasks = await getAllTasks();
  return tasks.find(task => task.id === parseInt(id));
}

// 新規タスクを作成
async function createTask(taskData) {
  const tasks = await getAllTasks();
  const newTask = {
    id: tasks.length > 0 ? Math.max(...tasks.map(t => t.id)) + 1 : 1,
    ...taskData,
    completed: false,
    createdAt: new Date().toISOString()
  };
  tasks.push(newTask);
  await saveTasks(tasks);
  return newTask;
}

// タスクを更新
async function updateTask(id, updates) {
  const tasks = await getAllTasks();
  const index = tasks.findIndex(task => task.id === parseInt(id));
  if (index === -1) return null;

  tasks[index] = { ...tasks[index], ...updates };
  await saveTasks(tasks);
  return tasks[index];
}

// タスクを削除
async function deleteTask(id) {
  const tasks = await getAllTasks();
  const filtered = tasks.filter(task => task.id !== parseInt(id));
  if (filtered.length === tasks.length) return false;

  await saveTasks(filtered);
  return true;
}

// タスクデータをファイルに保存
async function saveTasks(tasks) {
  await fs.writeFile(dataPath, JSON.stringify(tasks, null, 2));
}

module.exports = {
  getAllTasks,
  getTaskById,
  createTask,
  updateTask,
  deleteTask
};

重要ポイント

すべての関数を async/await で実装しています。ファイルI/Oは非同期処理が基本です。また、新規ID生成には既存タスクの最大IDに1を加える方式を使っています。

12.5 Expressサーバーの実装

モデルを使ってExpressルートを実装します。CRUD操作のすべてを網羅します。

実習 12-3: Expressサーバー実装

RESTful APIのルートを実装します。

app.js 
const express = require('express');
const tasksModel = require('./models/tasksModel');

const app = express();
const PORT = 3000;

// JSONボディパーサー
app.use(express.json());

// 1. 全タスク取得
app.get('/api/tasks', async (req, res) => {
  try {
    const tasks = await tasksModel.getAllTasks();
    res.json(tasks);
  } catch (error) {
    res.status(500).json({ error: 'サーバーエラー' });
  }
});

// 2. 単一タスク取得
app.get('/api/tasks/:id', async (req, res) => {
  try {
    const task = await tasksModel.getTaskById(req.params.id);
    if (!task) {
      return res.status(404).json({ error: 'タスクが見つかりません' });
    }
    res.json(task);
  } catch (error) {
    res.status(500).json({ error: 'サーバーエラー' });
  }
});

// 3. タスク作成
app.post('/api/tasks', async (req, res) => {
  try {
    // バリデーション
    const { title, description } = req.body;
    if (!title) {
      return res.status(400).json({ error: 'タイトルは必須です' });
    }

    const newTask = await tasksModel.createTask({ title, description });
    res.status(201).json(newTask);
  } catch (error) {
    res.status(500).json({ error: 'サーバーエラー' });
  }
});

// 4. タスク更新
app.put('/api/tasks/:id', async (req, res) => {
  try {
    const updatedTask = await tasksModel.updateTask(req.params.id, req.body);
    if (!updatedTask) {
      return res.status(404).json({ error: 'タスクが見つかりません' });
    }
    res.json(updatedTask);
  } catch (error) {
    res.status(500).json({ error: 'サーバーエラー' });
  }
});

// 5. タスク削除
app.delete('/api/tasks/:id', async (req, res) => {
  try {
    const deleted = await tasksModel.deleteTask(req.params.id);
    if (!deleted) {
      return res.status(404).json({ error: 'タスクが見つかりません' });
    }
    res.status(204).send();
  } catch (error) {
    res.status(500).json({ error: 'サーバーエラー' });
  }
});

// サーバー起動
app.listen(PORT, () => {
  console.log(`タスク管理APIサーバー起動: http://localhost:${PORT}`);
});

重要ポイント

各ルートで適切なHTTPステータスコードを返しています。成功時は200/201/204、クライアントエラーは400/404、サーバーエラーは500です。これはREST APIの重要な設計原則です。

トラブルシューティング

「ENOENT: no such file or directory」エラー

原因: データファイルが存在しない、またはパスが間違っている

解決方法: data ディレクトリと tasks.json が正しく配置されているか確認してください。

JSONパースエラー

原因: tasks.jsonの形式が正しくない

解決方法: JSONファイルが正しい配列形式になっているか、カンマや括弧が正しいか確認してください。

タスク作成時に400エラー

原因: リクエストボディにタイトルが含まれていない

解決方法: POSTリクエストで {"title":"..."} を含めているか確認してください。

理解度確認クイズ

  1. Q1. この実装でデータをJSONファイルに永続化する利点と欠点を説明してください。

    回答を表示

    利点:

    • データベース不要でシンプル
    • データが人間に読みやすい形式
    • サーバー再起動してもデータが残る

    欠点:

    • 同時書き込みに弱い（競合状態が発生しやすい）
    • 大量データには向かない
    • 検索やフィルタリングが非効率

  2. Q2. このAPIでHTTPステータスコード201、204、404、500をそれぞれどのような場面で返していますか？

    回答を表示
    • 201 Created: タスク作成成功時（POSTリクエスト）
    • 204 No Content: タスク削除成功時（DELETEリクエスト）
    • 404 Not Found: 指定IDのタスクが見つからない場合
    • 500 Internal Server Error: サーバー側でエラーが発生した場合（ファイル読み書き失敗など）

  3. Q3. モデル層（tasksModel.js）とルート層（app.js）を分離する利点は何ですか？

    回答を表示

    関心の分離（Separation of Concerns）により、コードが整理されます。ルート層はHTTPリクエスト処理に集中し、モデル層はデータ操作に集中します。これにより、テストがしやすくなり、将来データベースに切り替える際もモデル層だけを書き換えればよくなります。
