npmとパッケージ管理

npm（Node Package Manager）を使ったパッケージの管理方法を学びます。package.jsonの作成、パッケージのインストール・更新の方法を習得します。

この章で学ぶこと

  • npmとは何か、なぜ必要なのかを理解する
  • package.jsonの役割と作成方法を習得する
  • パッケージのインストール・更新・削除ができるようになる
  • npmスクリプトを使ってタスクを自動化する方法を学ぶ

6.1 npmとは

npm（Node Package Manager）は、Node.jsのパッケージ管理ツールです。パッケージとは、特定の機能を持つコードのまとまりのことで、他の開発者が作成したライブラリやツールを簡単にプロジェクトに組み込むことができます。

わかりやすい例え

npmは「アプリストア」のようなものです。スマートフォンでアプリをダウンロードするように、npm経由で必要な機能（パッケージ）をダウンロードして、自分のプロジェクトで使うことができます。車輪の再発明をせず、既存のパッケージを活用することで開発効率が大幅に向上します。

npmの3つの機能

リポジトリ

世界最大のパッケージレジストリ（100万以上のパッケージ）

CLIツール

コマンドライン経由でパッケージをインストール・管理

依存関係管理

パッケージの依存関係を自動的に解決

6.2 package.jsonの役割と作成

package.jsonは、プロジェクトの設定ファイルです。プロジェクト名、バージョン、使用するパッケージ（依存関係）などの情報を記録します。

package.jsonの作成

npm init コマンドを使って、対話形式でpackage.jsonを作成できます。

実習 6-1: package.jsonを作成してみよう

新しいプロジェクトを作成し、package.jsonファイルを生成します。

手順

  1. 1 新しいフォルダを作成してターミナルで開く
  2. 2
    以下のコマンドを実行 
    # 対話形式で作成
npm init

# すべてデフォルトで作成（スキップ）
npm init -y
  3. 3 生成されたpackage.jsonファイルをエディタで開いて確認
package.json 
{
  "name": "my-project",
  "version": "1.0.0",
  "description": "私の最初のNode.jsプロジェクト",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1"
  },
  "keywords": [],
  "author": "",
  "license": "ISC"
}

主要なフィールド

name プロジェクト名（小文字、ハイフン区切り）
version バージョン番号（セマンティックバージョニング形式）
description プロジェクトの説明
main エントリーポイント（メインファイル）
scripts npmスクリプト（コマンドショートカット）
dependencies 本番環境で必要なパッケージ
devDependencies 開発時のみ必要なパッケージ

6.3 パッケージのインストール

ローカルインストール（プロジェクト内）

最も一般的な方法です。パッケージはプロジェクトの node_modules フォルダにインストールされます。

ターミナル 
# 本番環境用のパッケージをインストール
npm install lodash

# 短縮形（同じ意味）
npm i lodash

# 開発環境専用のパッケージをインストール
npm install --save-dev nodemon

# 短縮形
npm i -D nodemon
flowchart TD A["npm install lodash"] --> B["npmレジストリから

dependenciesに記録"]

重要ポイント

--save-dev または -D オプションを使うと、開発時のみ必要なパッケージ（テストツール、ビルドツールなど）を devDependencies に記録できます。

グローバルインストール（システム全体）

コマンドラインツールとして使いたいパッケージは、グローバルにインストールします。

ターミナル 
# グローバルインストール
npm install -g nodemon

# インストール済みグローバルパッケージの確認
npm list -g --depth=0

注意

グローバルインストールは、CLIツールのみに使用してください。プロジェクトで使うライブラリは必ずローカルインストールしましょう。これにより、プロジェクトごとに異なるバージョンを使うことができます。

実習 6-2: パッケージをインストールしてみよう

人気のあるライブラリ「lodash」をインストールして、実際に使ってみます。

手順

  1. 1
    lodashをインストール 
    npm install lodash
  2. 2
    app.jsを作成 
    // lodashをインポート
const _ = require('lodash');

// 配列からユニークな値を取得
const numbers = [1, 2, 2, 3, 4, 4, 5];
const unique = _.uniq(numbers);
console.log(unique); // [1, 2, 3, 4, 5]

// オブジェクトの深いコピー
const original = { a: 1, b: { c: 2 } };
const copy = _.cloneDeep(original);
copy.b.c = 999;
console.log(original.b.c); // 2（影響なし）
console.log(copy.b.c);     // 999
  3. 3
    実行して動作確認 
    node app.js

6.4 パッケージの更新と削除

パッケージの更新

ターミナル 
# 古いパッケージのチェック
npm outdated

# 特定のパッケージを最新版に更新
npm update lodash

# すべてのパッケージを更新
npm update

# メジャーバージョンアップも含めて更新（手動確認推奨）
npm install lodash@latest

パッケージの削除

ターミナル 
# パッケージをアンインストール
npm uninstall lodash

# 短縮形
npm un lodash

セマンティックバージョニング（SemVer）

npmパッケージのバージョン番号は「メジャー.マイナー.パッチ」形式です。

  • メジャー（例: 2.0.0）: 互換性のない変更
  • マイナー（例: 1.1.0）: 後方互換性のある機能追加
  • パッチ（例: 1.0.1）: 後方互換性のあるバグ修正

バージョン指定記号

  • ^1.2.3: 1.x.x の最新（1.2.3以上2.0.0未満）
  • ~1.2.3: 1.2.x の最新（1.2.3以上1.3.0未満）
  • 1.2.3: 完全一致（そのバージョンのみ）

6.5 package-lock.jsonの役割

package-lock.json は、パッケージのインストール時に自動生成されるファイルで、インストールされたパッケージの正確なバージョン情報を記録します。

package-lock.jsonの役割

バージョンの固定

誰がいつインストールしても同じバージョンを使える

高速化

依存関係の解決が不要になり、インストールが高速化

依存関係ツリーの記録

パッケージが依存する他のパッケージも含めて完全に記録

重要ポイント

package-lock.json は必ずGitにコミットしてください。これにより、チーム全員が同じ環境で開発できます。手動で編集する必要はありません。

6.6 npmスクリプトで作業を自動化

package.json の scripts フィールドに、よく使うコマンドを登録できます。

package.json 
{
  "name": "my-project",
  "version": "1.0.0",
  "scripts": {
    "start": "node app.js",
    "dev": "nodemon app.js",
    "test": "jest",
    "lint": "eslint ."
  },
  "dependencies": {
    "express": "^4.18.0"
  },
  "devDependencies": {
    "nodemon": "^2.0.20",
    "jest": "^29.0.0",
    "eslint": "^8.30.0"
  }
}
スクリプトの実行 
# scriptsに登録したコマンドを実行
npm run start

# start, test, stop はrunを省略可能
npm start
npm test

# 開発モードで起動（ファイル変更時に自動再起動）
npm run dev

ヒント

npmスクリプトを使うことで、チームメンバー全員が同じコマンドで作業できます。「どうやって起動するんだっけ？」という疑問がなくなります。

実習 6-3: npmスクリプトを作成してみよう

開発を効率化するnpmスクリプトを設定します。

手順

  1. 1
    nodemonをインストール（開発時にファイル監視して自動再起動） 
    npm install --save-dev nodemon
  2. 2 package.jsonのscriptsセクションに以下を追加 
"scripts": {
  "start": "node app.js",
  "dev": "nodemon app.js"
}
  1. 3
    開発モードで起動してファイルを編集し、自動再起動を確認 
    npm run dev

6.7 node_modulesフォルダの取り扱い

パッケージをインストールすると、node_modules フォルダに大量のファイルが保存されます。このフォルダは非常に大きくなるため、Gitには含めません。

プロジェクト構成 

my-project/
├── node_modules/     ← Gitに含めない（.gitignoreに追加）
├── package.json      ← Gitにコミット
├── package-lock.json ← Gitにコミット
└── app.js
.gitignore 
# 依存関係
node_modules/

# ログ
*.log

# 環境設定
.env

プロジェクトの共有方法

Gitリポジトリをクローンしたら、以下のコマンドで依存パッケージをインストールします。 

# package.jsonに記録された全パッケージをインストール
npm install

# 短縮形
npm i

これにより、package-lock.json に記録された正確なバージョンがインストールされます。

トラブルシューティング

npmコマンドが見つからない

原因: Node.jsが正しくインストールされていない、またはPATHが通っていない

解決方法:

  • node -v と npm -v を実行してインストール確認
  • Node.jsを公式サイトから再インストール
  • ターミナルを再起動
パッケージのインストールが失敗する

原因: ネットワークエラー、キャッシュの問題、権限の問題

解決方法:

# キャッシュをクリア
npm cache clean --force

# node_modulesとpackage-lock.jsonを削除して再インストール
rm -rf node_modules package-lock.json
npm install
バージョンの競合エラー

原因: 依存パッケージのバージョン要求が矛盾している

解決方法:

  • エラーメッセージを読んで、どのパッケージが競合しているか確認
  • npm ls [パッケージ名] で依存関係ツリーを確認
  • package.jsonのバージョン指定を調整（^や~の範囲を見直す）

理解度確認クイズ

  1. Q1. package.jsonとpackage-lock.jsonの違いは何ですか？

    回答を表示

    package.jsonは、プロジェクトの設定ファイルで、依存パッケージを「範囲」で指定します（例: ^1.2.3）。一方、package-lock.jsonは、実際にインストールされたパッケージの「正確なバージョン」を記録し、誰がインストールしても同じバージョンになるようにします。package-lock.jsonは自動生成されるため、手動で編集する必要はありません。

  2. Q2. npm install --save-dev と npm install の違いは何ですか？

    回答を表示

    npm install パッケージ名 は、本番環境でも必要なパッケージを dependencies に追加します。npm install --save-dev パッケージ名（または -D）は、開発時のみ必要なパッケージ（テストツール、ビルドツールなど）を devDependencies に追加します。これにより、本番環境で不要なパッケージをインストールしないようにできます。

  3. Q3. node_modulesフォルダをGitにコミットすべきでない理由を説明してください。

    回答を表示

    node_modulesフォルダには数千〜数万のファイルが含まれ、容量が非常に大きくなります。Gitリポジトリのサイズが肥大化し、クローンやプッシュが遅くなります。また、package.jsonとpackage-lock.jsonがあれば、npm install コマンドで同じ環境を再現できるため、node_modulesをコミットする必要はありません。

この章のまとめ

  • npmはNode.jsのパッケージ管理ツールで、100万以上のパッケージを利用できる
  • package.jsonはプロジェクトの設定ファイルで、依存関係やスクリプトを記録する
  • ローカルインストールはプロジェクト内、グローバルインストールはシステム全体に適用される
  • package-lock.jsonは正確なバージョンを記録し、環境の再現性を保証する
  • npmスクリプトでよく使うコマンドを登録し、作業を効率化できる
  • node_modulesフォルダはGitに含めず、.gitignoreに追加する
