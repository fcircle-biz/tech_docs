REST APIサーバーを構築しよう

Expressを使ってREST APIサーバーを構築します。CRUD操作、JSONデータの送受信、エラーハンドリングを実装します。

この章で学ぶこと

  • REST APIの基本概念とRESTful設計の原則を理解する
  • HTTPメソッド（GET、POST、PUT、DELETE）を使い分けられるようになる
  • CRUD操作を実装し、JSONデータの送受信を実践する
  • 適切なHTTPステータスコードを使い、エラーハンドリングを実装する

11.1 REST APIとは何か

APIの基本概念

API（Application Programming Interface）とは、アプリケーション間で情報をやり取りするための窓口です。レストランで例えると、お客さん（クライアント）が直接キッチン（サーバー）に入って料理を作ることはできません。代わりに、ウェイター（API）に注文を伝えて、料理を受け取ります。

sequenceDiagram participant C as クライアント全文Webアプリ、モバイルアプリ全文 participant API as REST API participant S as サーバー全文データベース全文 C->>API: HTTPリクエスト全文GET /books全文 API->>S: データ取得処理 S-->>API: データ API-->>C: HTTPレスポンス全文JSON全文

RESTとは

REST（REpresentational State Transfer）は、Webサービスを設計するためのアーキテクチャスタイルです。RESTに従って設計されたAPIを「RESTful API」と呼びます。

RESTの核心的な考え方

すべてのデータを「リソース（資源）」として扱い、それぞれに一意のURL（エンドポイント）を割り当てます。HTTPメソッド（GET、POST、PUT、DELETE）を使って、そのリソースに対する操作を表現します。

RESTの6つの制約

  1. 1

    クライアント・サーバー

    UI（クライアント）とデータ処理（サーバー）を分離する

  2. 2

    ステートレス

    各リクエストは独立していて、サーバーはクライアントの状態を保持しない

  3. 3

    キャッシュ可能

    レスポンスはキャッシュ可能かどうか明示する

  4. 4

    統一インターフェース

    すべてのリソースに対して統一的な方法でアクセスできる

  5. 5

    階層化システム

    クライアントとサーバーの間に複数の層を配置できる

  6. 6

    コードオンデマンド（任意）

    必要に応じてサーバーからクライアントにコードを送信できる

11.2 RESTful設計の原則

リソース指向設計

RESTful APIでは、すべてのデータを「リソース」として扱います。リソースは名詞で表現し、URLで識別します。

リソースの例

リソース URL（エンドポイント）
書籍のコレクション /books
特定の書籍 /books/123
ユーザーのコレクション /users
特定のユーザー /users/456

HTTPメソッドの使い分け

HTTPメソッドを使って、リソースに対する操作（CRUD）を表現します。

CRUD操作とHTTPメソッドの対応

操作 HTTPメソッド エンドポイント例 説明
Create POST POST /books 新しい書籍を作成
Read GET GET /books
GET /books/123		 書籍一覧または特定の書籍を取得
Update PUT PUT /books/123 特定の書籍を更新
Delete DELETE DELETE /books/123 特定の書籍を削除

重要ポイント

URLには動詞（create、delete等）を含めません。HTTPメソッドが操作を表すため、URLはリソース（名詞）のみで構成します。例: POST /books ○、POST /createBook ×

エンドポイント設計のベストプラクティス

  • 複数形を使用: /books（○）、/book（△）
  • 階層構造を表現: /users/123/orders（ユーザー123の注文一覧）
  • 小文字とハイフンを使用: /order-items（○）、/OrderItems（×）
  • クエリパラメータでフィルタリング: /books?category=fiction&sort=title

11.3 HTTPステータスコードの理解

HTTPステータスコードは、リクエストの結果を数値で表します。適切なステータスコードを返すことで、クライアントが結果を正しく処理できるようになります。

flowchart TD A[HTTPリクエスト] --> B{処理結果} B -->|成功| C[2xx Success] B -->|クライアントエラー| D[4xx Client Error] B -->|サーバーエラー| E[5xx Server Error] C --> C1["200 OK"] C --> C2["201 Created"] C --> C3["204 No Content"] D --> D1["400 Bad Request"] D --> D2["404 Not Found"] D --> D3["401 Unauthorized"] E --> E1["500 Internal Server Error"] E --> E2["503 Service Unavailable"]

よく使うステータスコード一覧

2xx 成功

  • 200

    OK

    リクエスト成功（GET、PUT、DELETEで使用）

  • 201

    Created

    新しいリソースが作成された（POSTで使用）

  • 204

    No Content

    成功したがレスポンスボディなし（DELETEで使用）

4xx クライアントエラー

  • 400

    Bad Request

    リクエストが不正（バリデーションエラー等）

  • 401

    Unauthorized

    認証が必要、または認証に失敗

  • 404

    Not Found

    リソースが見つからない

5xx サーバーエラー

  • 500

    Internal Server Error

    サーバー内部でエラーが発生

  • 503

    Service Unavailable

    サーバーが一時的に利用不可

11.4 書籍管理APIの実装（CRUD操作）

実際に書籍管理APIを構築して、CRUD操作を実装しましょう。まずはメモリ上にデータを保持する簡単な実装から始めます。

プロジェクトのセットアップ

ターミナル 
# プロジェクトフォルダを作成
mkdir book-api
cd book-api

# package.jsonを作成
npm init -y

# Expressをインストール
npm install express

基本的なサーバーの構築

app.js 
const express = require('express');
const app = express();

// JSONデータを扱えるようにする
app.use(express.json());

// メモリ上にデータを保持（本来はデータベースを使用）
let books = [
  { id: 1, title: 'JavaScript入門', author: '山田太郎', year: 2023 },
  { id: 2, title: 'Node.js実践', author: '佐藤花子', year: 2024 }
];

// IDの自動採番用
let nextId = 3;

const PORT = 3000;
app.listen(PORT, () => {
  console.log(`サーバーがポート${PORT}で起動しました`);
});

READ: 書籍一覧の取得（GET /books）

app.js（続き） 
// すべての書籍を取得
app.get('/books', (req, res) => {
  // 200 OKとともに書籍一覧を返す
  res.status(200).json({
    success: true,
    data: books
  });
});

READ: 特定の書籍を取得（GET /books/:id）

app.js（続き） 
// 特定の書籍を取得
app.get('/books/:id', (req, res) => {
  // URLパラメータからIDを取得（文字列なので数値に変換）
  const id = parseInt(req.params.id);

  // 該当する書籍を検索
  const book = books.find(b => b.id === id);

  if (!book) {
    // 見つからない場合は404を返す
    return res.status(404).json({
      success: false,
      message: '書籍が見つかりません'
    });
  }

  // 見つかった場合は200とデータを返す
  res.status(200).json({
    success: true,
    data: book
  });
});

CREATE: 新しい書籍を作成（POST /books）

app.js（続き） 
// 新しい書籍を作成
app.post('/books', (req, res) => {
  // リクエストボディからデータを取得
  const { title, author, year } = req.body;

  // バリデーション（必須項目チェック）
  if (!title || !author) {
    return res.status(400).json({
      success: false,
      message: 'タイトルと著者は必須です'
    });
  }

  // 新しい書籍オブジェクトを作成
  const newBook = {
    id: nextId++,
    title,
    author,
    year: year || new Date().getFullYear()
  };

  // 配列に追加
  books.push(newBook);

  // 201 Createdとともに作成されたデータを返す
  res.status(201).json({
    success: true,
    data: newBook
  });
});

UPDATE: 書籍を更新（PUT /books/:id）

app.js（続き） 
// 書籍を更新
app.put('/books/:id', (req, res) => {
  const id = parseInt(req.params.id);
  const { title, author, year } = req.body;

  // 該当する書籍を検索
  const bookIndex = books.findIndex(b => b.id === id);

  if (bookIndex === -1) {
    return res.status(404).json({
      success: false,
      message: '書籍が見つかりません'
    });
  }

  // データを更新
  books[bookIndex] = {
    id,
    title: title || books[bookIndex].title,
    author: author || books[bookIndex].author,
    year: year || books[bookIndex].year
  };

  res.status(200).json({
    success: true,
    data: books[bookIndex]
  });
});

DELETE: 書籍を削除（DELETE /books/:id）

app.js（続き） 
// 書籍を削除
app.delete('/books/:id', (req, res) => {
  const id = parseInt(req.params.id);

  // 該当する書籍を検索
  const bookIndex = books.findIndex(b => b.id === id);

  if (bookIndex === -1) {
    return res.status(404).json({
      success: false,
      message: '書籍が見つかりません'
    });
  }

  // 配列から削除
  books.splice(bookIndex, 1);

  // 204 No Content（削除成功、レスポンスボディなし）
  res.status(204).send();
});

11.6 エラーハンドリングの実装

より堅牢なAPIにするため、エラーハンドリングを追加しましょう。

バリデーションミドルウェア

app.js（改良版） 
// バリデーション関数
function validateBook(req, res, next) {
  const { title, author } = req.body;

  if (!title || !author) {
    return res.status(400).json({
      success: false,
      error: {
        code: 'VALIDATION_ERROR',
        message: 'タイトルと著者は必須です',
        details: {
          title: !title ? '必須項目です' : undefined,
          author: !author ? '必須項目です' : undefined
        }
      }
    });
  }

  next();
}

// POSTリクエストでバリデーションを使用
app.post('/books', validateBook, (req, res) => {
  // バリデーション通過後の処理
  // ...
});

エラーハンドリングミドルウェア

app.js（続き） 
// 404エラーハンドリング（すべてのルートの後に配置）
app.use((req, res) => {
  res.status(404).json({
    success: false,
    error: {
      code: 'NOT_FOUND',
      message: 'エンドポイントが見つかりません'
    }
  });
});

// サーバーエラーハンドリング（最後に配置）
app.use((err, req, res, next) => {
  console.error(err.stack);
  res.status(500).json({
    success: false,
    error: {
      code: 'INTERNAL_SERVER_ERROR',
      message: 'サーバー内部でエラーが発生しました'
    }
  });
});

重要ポイント

エラーレスポンスの形式を統一することで、クライアント側でエラー処理がしやすくなります。successフラグとerrorオブジェクトを含めるのがベストプラクティスです。

理解度確認クイズ

  1. Q1. REST APIにおいて、新しいリソースを作成する際に使用すべきHTTPメソッドとステータスコードは何ですか?

    回答を表示

    HTTPメソッド: POST、ステータスコード: 201 Created。POSTメソッドで新しいリソースを作成し、成功時には201ステータスコードと作成されたリソースのデータを返すのがRESTfulな設計です。

  2. Q2. URLに動詞を含めない理由は何ですか？良い例と悪い例を挙げてください。

    回答を表示

    HTTPメソッド自体が操作（動詞）を表すため、URLはリソース（名詞）のみで構成すべきです。
    良い例: POST /books（書籍を作成）、DELETE /books/123（書籍を削除）
    悪い例: POST /createBook、GET /getBooks

  3. Q3. ステータスコード 404、400、500 の違いを説明してください。

    回答を表示
    • 404 Not Found: リクエストされたリソースが存在しない（クライアントエラー）
    • 400 Bad Request: リクエストの内容が不正（バリデーションエラー等、クライアントエラー）
    • 500 Internal Server Error: サーバー側で予期しないエラーが発生（サーバーエラー）

この章のまとめ

  • REST APIはリソース指向設計で、HTTPメソッドを使ってCRUD操作を表現する
  • エンドポイントURLは名詞（リソース）のみで構成し、動詞を含めない
  • HTTPステータスコードを適切に使い分けることで、処理結果を明確に伝える
  • express.json()ミドルウェアでJSONデータの送受信が可能になる
  • バリデーションとエラーハンドリングでAPIの堅牢性を高める
