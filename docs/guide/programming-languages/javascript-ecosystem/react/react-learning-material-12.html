12.1 テストの重要性とテスト戦略

プロダクション品質のReactアプリケーションを構築するには、適切なテスト戦略が不可欠です。テストピラミッドの概念を理解し、各レベルでのテストの役割を把握しましょう。

テストピラミッド

各テストレベルの特徴

ユニットテスト

  • ・実行速度が速い
  • ・テスト数が多い
  • ・関数やフックを検証
  • ・バグの早期発見

統合テスト

  • ・コンポーネント連携
  • ・API連携を検証
  • ・中程度の実行速度
  • ・現実的なシナリオ

E2Eテスト

  • ・実行速度が遅い
  • ・テスト数は少なめ
  • ・実際のUX検証
  • ・重要シナリオ

テストのバランス

ユニットテストを多く、統合テストを中程度、E2Eテストを少なくすることで、テスト実行速度とカバレッジのバランスを取ります。一般的な比率は、ユニット:統合:E2E = 70:20:10 程度です。

12.2 React Testing Libraryでのコンポーネントテスト

React Testing Library（RTL）は、ユーザーの視点からコンポーネントをテストするためのライブラリです。実装の詳細ではなく、ユーザーの操作と結果に焦点を当てます。

React Testing Libraryの哲学

ユーザー中心のテスト

「ソフトウェアが使われる方法に近い形でテストすれば、より自信を持てる」という哲学に基づき、内部状態やpropsを直接テストするのではなく、ユーザーが見る・操作する内容に焦点を当てます。

基本的なテストの書き方

Button.test.tsx 
import { render, screen, fireEvent } from '@testing-library/react';
import { Button } from './Button';

describe('Button コンポーネント', () => {
  test('ボタンがレンダリングされる', () => {
    render(<Button>クリック</Button>);
    // getByRole: アクセシビリティロールで要素を取得
    const button = screen.getByRole('button', { name: 'クリック' });
    expect(button).toBeInTheDocument();
  });

  test('クリックイベントが発火する', () => {
    const handleClick = jest.fn(); // モック関数
    render(<Button onClick={handleClick}>クリック</Button>);

    const button = screen.getByRole('button');
    fireEvent.click(button); // クリックイベント発火

    expect(handleClick).toHaveBeenCalledTimes(1);
  });
});

主要なクエリメソッド

メソッド 用途 例
getByRole アクセシビリティロールで検索 getByRole('button')
getByLabelText ラベルテキストで検索 getByLabelText('名前')
getByText テキスト内容で検索 getByText('送信')
findBy... 非同期要素の検索 await findByText('完了')
queryBy... 存在しない要素の検索 queryByText('エラー')

フォームコンポーネントのテスト

LoginForm.test.tsx 
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { LoginForm } from './LoginForm';

test('ログインフォームの入力と送信', async () => {
  const handleSubmit = jest.fn();
  const user = userEvent.setup(); // ユーザー操作のシミュレーション

  render(<LoginForm onSubmit={handleSubmit} />);

  // 入力フィールドを取得
  const emailInput = screen.getByLabelText('メールアドレス');
  const passwordInput = screen.getByLabelText('パスワード');
  const submitButton = screen.getByRole('button', { name: 'ログイン' });

  // ユーザー操作をシミュレート
  await user.type(emailInput, 'user@example.com');
  await user.type(passwordInput, 'password123');
  await user.click(submitButton);

  // 送信ハンドラーが正しいデータで呼ばれたか検証
  expect(handleSubmit).toHaveBeenCalledWith({
    email: 'user@example.com',
    password: 'password123'
  });
});

12.3 Jestでのユニットテスト

Jestは、Facebookが開発したJavaScriptテストフレームワークです。React Testing Libraryと組み合わせることで、強力なテスト環境を構築できます。

Jestのセットアップ（Viteプロジェクト）

インストールと設定 
# 必要なパッケージをインストール
npm install -D vitest @testing-library/react @testing-library/jest-dom @testing-library/user-event jsdom

# package.jsonにテストスクリプトを追加
# "scripts": {
#   "test": "vitest",
#   "test:ui": "vitest --ui"
# }
vitest.config.ts 
import { defineConfig } from 'vitest/config';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],
  test: {
    environment: 'jsdom', // DOMをシミュレート
    globals: true,
    setupFiles: './src/test/setup.ts',
  },
});

モック関数の活用

API呼び出しのモック 
import { vi } from 'vitest';
import { fetchUserData } from './api';

// APIモジュール全体をモック
vi.mock('./api');

test('ユーザーデータを取得して表示', async () => {
  // モック実装を定義
  vi.mocked(fetchUserData).mockResolvedValue({
    id: 1,
    name: 'テストユーザー',
    email: 'test@example.com'
  });

  // テストコード...
  expect(fetchUserData).toHaveBeenCalled();
});

スナップショットテスト

UIの変更検知 
test('Card コンポーネントのスナップショット', () => {
  const { container } = render(
    <Card title="タイトル" description="説明文" />
  );

  // 初回実行時にスナップショットが保存される
  expect(container).toMatchSnapshot();
});

スナップショットテストの注意点

スナップショットテストは便利ですが、意図しない変更も全て検知するため、頻繁に更新が必要になることがあります。UIの大きな構造変更の検知に限定して使用することを推奨します。

12.4 カスタムフックのテスト

カスタムフックは、コンポーネントとは独立してテストすることが推奨されます。@testing-library/reactのrenderHookを使用します。

useCounter.test.ts 
import { renderHook, act } from '@testing-library/react';
import { useCounter } from './useCounter';

describe('useCounter フック', () => {
  test('初期値が正しく設定される', () => {
    const { result } = renderHook(() => useCounter(10));
    expect(result.current.count).toBe(10);
  });

  test('incrementで値が増える', () => {
    const { result } = renderHook(() => useCounter(0));

    act(() => {
      result.current.increment();
    });

    expect(result.current.count).toBe(1);
  });

  test('resetで初期値に戻る', () => {
    const { result } = renderHook(() => useCounter(5));

    act(() => {
      result.current.increment();
      result.current.increment();
    });
    expect(result.current.count).toBe(7);

    act(() => {
      result.current.reset();
    });
    expect(result.current.count).toBe(5);
  });
});

非同期フックのテスト

useFetch.test.ts 
import { renderHook, waitFor } from '@testing-library/react';
import { vi } from 'vitest';
import { useFetch } from './useFetch';

test('データフェッチが成功する', async () => {
  global.fetch = vi.fn(() =>
    Promise.resolve({
      ok: true,
      json: () => Promise.resolve({ data: 'テストデータ' }),
    } as Response)
  );

  const { result } = renderHook(() => useFetch('/api/data'));

  // 最初はローディング状態
  expect(result.current.loading).toBe(true);

  // データ取得完了を待つ
  await waitFor(() => {
    expect(result.current.loading).toBe(false);
  });

  expect(result.current.data).toEqual({ data: 'テストデータ' });
  expect(result.current.error).toBeNull();
});

12.5 E2Eテスト（Playwright）

Playwrightは、Microsoft製のE2Eテストフレームワークです。Chromium、Firefox、WebKitの3つのブラウザエンジンでテストを実行できます。

Playwrightのセットアップ

インストール 
# Playwrightをインストール
npm init playwright@latest

# ブラウザをインストール
npx playwright install

基本的なE2Eテスト

tests/login.spec.ts 
import { test, expect } from '@playwright/test';

test('ログインフローのテスト', async ({ page }) => {
  // ログインページに移動
  await page.goto('http://localhost:5173/login');

  // フォームに入力
  await page.fill('input[name="email"]', 'user@example.com');
  await page.fill('input[name="password"]', 'password123');

  // ログインボタンをクリック
  await page.click('button[type="submit"]');

  // ダッシュボードに遷移したことを確認
  await expect(page).toHaveURL('http://localhost:5173/dashboard');
  await expect(page.locator('h1')).toHaveText('ダッシュボード');
});

ページオブジェクトパターン

pages/LoginPage.ts 
import { Page } from '@playwright/test';

export class LoginPage {
  constructor(private page: Page) {}

  async goto() {
    await this.page.goto('/login');
  }

  async login(email: string, password: string) {
    await this.page.fill('input[name="email"]', email);
    await this.page.fill('input[name="password"]', password);
    await this.page.click('button[type="submit"]');
  }
}

// テストでの使用
test('ログイン', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.goto();
  await loginPage.login('user@example.com', 'password123');
});

12.6 CI/CDパイプライン構築

GitHub Actionsを使用して、コードのプッシュやプルリクエスト作成時に自動的にテストを実行し、デプロイまで行うパイプラインを構築します。

flowchart LR Push["git push"] --> Action["GitHub Actions"] Action --> Lint["リント検証"] Lint --> Test["テスト実行"] Test --> Build["ビルド"] Build --> Deploy["デプロイ"] Deploy --> Prod["本番環境"]
.github/workflows/ci.yml 
name: CI/CD Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v3

      - name: Node.jsセットアップ
        uses: actions/setup-node@v3

          node-version: '18'
          cache: 'npm'

      - name: 依存関係インストール
        run: npm ci

      - name: リント実行
        run: npm run lint

      - name: テスト実行
        run: npm run test

      - name: ビルド
        run: npm run build

12.7 Vercel/Netlifyへのデプロイ

VercelとNetlifyは、Reactアプリケーションを簡単にホスティングできるプラットフォームです。GitHubと連携することで、自動デプロイが可能です。

Vercelへのデプロイ

Vercel CLIでのデプロイ 
# Vercel CLIをインストール
npm install -g vercel

# ログイン
vercel login

# デプロイ
vercel

# 本番環境へデプロイ
vercel --prod

環境変数の管理

.env.example 
# API設定
VITE_API_URL=https://api.example.com
VITE_API_KEY=your_api_key_here

# 機能フラグ
VITE_ENABLE_ANALYTICS=true
環境変数の使用 
// Viteでは VITE_ プレフィックスが必要
const apiUrl = import.meta.env.VITE_API_URL;
const apiKey = import.meta.env.VITE_API_KEY;

// 型定義
interface ImportMetaEnv {
  readonly VITE_API_URL: string;
  readonly VITE_API_KEY: string;
  readonly VITE_ENABLE_ANALYTICS: string;
}

セキュリティ注意

VITE_プレフィックスの環境変数はクライアント側のコードにバンドルされます。APIキーなどの機密情報は絶対に含めないでください。バックエンドAPIで認証を行い、トークンベースの認証を使用しましょう。

12.8 パフォーマンス監視

Web Vitalsは、Googleが提唱するユーザー体験の品質を測定する指標です。Reactアプリケーションでこれらの指標を監視しましょう。

主要なWeb Vitals指標

LCP

Largest Contentful Paint

最大コンテンツの描画時間。2.5秒以内が目標。

FID

First Input Delay

初回入力遅延。100ms以内が目標。

CLS

Cumulative Layout Shift

累積レイアウトシフト。0.1以下が目標。

Web Vitals測定 
import { getCLS, getFID, getLCP } from 'web-vitals';

function sendToAnalytics(metric: any) {
  // 分析サービスに送信
  console.log(metric);
}

getCLS(sendToAnalytics);
getFID(sendToAnalytics);
getLCP(sendToAnalytics);

12.9 エラートラッキング

Sentryは、本番環境でのエラーを自動的に収集・監視するサービスです。Reactアプリケーションと簡単に統合できます。

Sentryセットアップ 
# Sentryパッケージインストール
npm install @sentry/react
main.tsx 
import * as Sentry from '@sentry/react';
import ReactDOM from 'react-dom/client';
import App from './App';

// Sentry初期化
Sentry.init({
  dsn: 'YOUR_SENTRY_DSN',
  environment: import.meta.env.MODE,
  tracesSampleRate: 1.0, // パフォーマンス監視のサンプリングレート
});

ReactDOM.createRoot(document.getElementById('root')!).render(
  <Sentry.ErrorBoundary fallback={<ErrorFallback />}>
    <App />
  </Sentry.ErrorBoundary>
);

カスタムエラーバウンダリ

ErrorBoundary.tsx 
import { Component, ReactNode } from 'react';
import * as Sentry from '@sentry/react';

interface Props {
  children: ReactNode;
}

interface State {
  hasError: boolean;
}

class ErrorBoundary extends Component<Props, State> {
  state = { hasError: false };

  static getDerivedStateFromError() {
    return { hasError: true };
  }

  componentDidCatch(error: Error, errorInfo: any) {
    // Sentryにエラーを送信
    Sentry.captureException(error, { contexts: { react: errorInfo } });
  }

  render() {
    if (this.state.hasError) {
      return (
        <div className="error-fallback">
          <h1>エラーが発生しました</h1>
          <button onClick={() => window.location.reload()}>
            再読み込み
          </button>
        </div>
      );
    }
    return this.props.children;
  }
}

理解度確認クイズ

  1. Q1. テストピラミッドにおいて、最も多く書くべきテストはどれですか？

    回答を表示

    回答: ユニットテストです。実行速度が速く、バグを早期に発見できるため、テストピラミッドの底辺として最も多く書くべきです。一般的な比率は、ユニット:統合:E2E = 70:20:10 程度です。

  2. Q2. React Testing Libraryの哲学として正しいものはどれですか？

    回答を表示

    回答: 「ソフトウェアが使われる方法に近い形でテストすれば、より自信を持てる」という哲学に基づき、内部状態ではなくユーザーの視点からテストします。アクセシビリティロールやテキスト内容で要素を検索することを推奨します。

  3. Q3. Viteで環境変数を定義する際、クライアントコードで使用できるようにするために必要なプレフィックスは何ですか？

    回答を表示

    回答: VITE_ プレフィックスが必要です（例: VITE_API_URL）。このプレフィックスがない環境変数はクライアントコードにバンドルされません。ただし、APIキーなどの機密情報は絶対に含めないでください。

  4. Q4. Web VitalsのLCPとは何を測定する指標ですか？

    回答を表示

    回答: LCP（Largest Contentful Paint）は、ページの最大コンテンツが描画されるまでの時間を測定します。2.5秒以内が目標とされ、ユーザーがコンテンツを見られるまでの体感速度を示す重要な指標です。
