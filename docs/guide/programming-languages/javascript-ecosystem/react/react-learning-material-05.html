5.1 useEffectとは何か - 副作用の概念

useEffectは、Reactコンポーネント内で副作用（side effects）を実行するためのフックです。副作用とは、コンポーネントのレンダリング以外の処理、例えばデータフェッチング、DOM操作、購読の設定、タイマーの設定などを指します。

副作用とは

純粋な関数と副作用

Reactコンポーネントは基本的に純粋な関数であるべきです（同じ入力に対して常に同じ出力を返す）。しかし、実際のアプリケーションでは、外部システムとの通信、DOMの操作、非同期処理などが必要です。これらが「副作用」です。

useEffectの基本構文

useEffectの基本構文 
import { useEffect } from 'react';

function MyComponent() {
  useEffect(() => {
    // 副作用の処理をここに記述
    console.log('コンポーネントがマウント、または更新されました');

    // クリーンアップ関数（オプション）
    return () => {
      console.log('クリーンアップ処理');
    };
  }, []); // 依存配列

  return <div>My Component</div>;
}

重要なポイント

useEffectは、レンダリングが完了してブラウザに描画された後に実行されます。これにより、副作用の処理がレンダリングをブロックせず、パフォーマンスが向上します。

5.2 依存配列の完全理解

依存配列は、useEffectがいつ再実行されるかを制御する重要な要素です。正しく理解しないと、無限ループや期待しない動作の原因となります。

依存配列の3つのパターン

依存配列なし 

useEffect(() => {
  // 処理
}); // 配列なし

実行タイミング: 毎回のレンダリング後に実行されます。通常は推奨されません。

空の依存配列 

useEffect(() => {
  // 処理
}, []); // 空配列

実行タイミング: マウント時に1回だけ実行されます。初期化処理に使用します。

依存配列あり 

useEffect(() => {
  // 処理
}, [count]); // 依存値

実行タイミング: マウント時と、指定した値が変更された時に実行されます。

依存配列の正しい設定方法

正しい依存配列の例 
function UserProfile({ userId }: { userId: string }) {
  const [user, setUser] = useState<User | null>(null);

  useEffect(() => {
    // userIdが変更されるたびに実行
    fetchUser(userId).then(data => setUser(data));
  }, [userId]); // userIdを依存配列に含める

  return <div>{user?.name}</div>;
}

よくある間違い

エフェクト内で使用している変数やpropsを依存配列に含めないと、古い値を参照してしまいます。ESLintのルール react-hooks/exhaustive-deps を有効にすると、警告が表示されます。

5.3 クリーンアップ関数の実装

クリーンアップ関数は、コンポーネントがアンマウントされる前、または次のエフェクトが実行される前に実行される関数です。購読の解除、タイマーのクリア、イベントリスナーの削除などに使用します。

クリーンアップが必要な例

タイマーのクリーンアップ 
function Timer() {
  const [seconds, setSeconds] = useState(0);

  useEffect(() => {
    // タイマーを設定
    const intervalId = setInterval(() => {
      setSeconds(prev => prev + 1);
    }, 1000);

    // クリーンアップ: タイマーを停止
    return () => {
      clearInterval(intervalId);
    };
  }, []); // マウント時のみ実行

  return <div>経過時間: {seconds}秒</div>;
}
イベントリスナーのクリーンアップ 
function WindowSize() {
  const [size, setSize] = useState({ width: 0, height: 0 });

  useEffect(() => {
    // リサイズイベントハンドラー
    const handleResize = () => {
      setSize({ width: window.innerWidth, height: window.innerHeight });
    };

    // イベントリスナーを登録
    window.addEventListener('resize', handleResize);
    handleResize(); // 初期値を設定

    // クリーンアップ: イベントリスナーを削除
    return () => {
      window.removeEventListener('resize', handleResize);
    };
  }, []);

  return <div>画面サイズ: {size.width} x {size.height}</div>;
}

5.4 データフェッチングのベストプラクティス

useEffectを使ったデータフェッチングは最も一般的なユースケースの1つです。しかし、正しく実装しないと競合状態やメモリリークの原因となります。

基本的なデータフェッチング

データフェッチングの基本実装 
interface User {
  id: string;
  name: string;
  email: string;
}

function UserDetail({ userId }: { userId: string }) {
  const [user, setUser] = useState<User | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    // ローディング開始
    setLoading(true);
    setError(null);

    // データ取得
    fetch(`/api/users/${userId}`)
      .then(res => res.json())
      .then(data => {
        setUser(data);
        setLoading(false);
      })
      .catch(err => {
        setError(err);
        setLoading(false);
      });
  }, [userId]);

  if (loading) return <div>読み込み中...</div>;
  if (error) return <div>エラー: {error.message}</div>;
  if (!user) return <div>ユーザーが見つかりません</div>;

  return <div>{user.name}</div>;
}

競合状態の回避とAbortController

競合状態とは

ユーザーが素早く異なるIDを選択した場合、複数のリクエストが同時に実行され、古いリクエストの結果が後から到着して、最新の状態を上書きしてしまう問題です。

AbortControllerを使った改善版 
function UserDetail({ userId }: { userId: string }) {
  const [user, setUser] = useState<User | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    // AbortControllerを作成
    const controller = new AbortController();
    const signal = controller.signal;

    setLoading(true);
    setError(null);

    fetch(`/api/users/${userId}`, { signal })
      .then(res => res.json())
      .then(data => {
        if (!signal.aborted) {
          setUser(data);
          setLoading(false);
        }
      })
      .catch(err => {
        if (err.name !== 'AbortError' && !signal.aborted) {
          setError(err);
          setLoading(false);
        }
      });

    // クリーンアップ: リクエストをキャンセル
    return () => {
      controller.abort();
    };
  }, [userId]);

  // レンダリング処理は同じ
}

5.5 useLayoutEffectとの使い分け

useLayoutEffectは、useEffectとほぼ同じAPIですが、実行タイミングが異なります。DOM更新後、ブラウザが描画する前に同期的に実行されます。

実行タイミングの違い

項目 useEffect useLayoutEffect
実行タイミング 描画後（非同期） 描画前（同期）
ブロッキング しない する
ユースケース データフェッチング、購読 DOM測定、スクロール位置
推奨度 デフォルト 必要な時のみ
useLayoutEffectの使用例 
import { useLayoutEffect, useRef, useState } from 'react';

function Tooltip({ children }: { children: React.ReactNode }) {
  const ref = useRef<HTMLDivElement>(null);
  const [position, setPosition] = useState({ x: 0, y: 0 });

  useLayoutEffect(() => {
    // DOM測定は描画前に実行する必要がある
    if (ref.current) {
      const rect = ref.current.getBoundingClientRect();
      setPosition({
        x: rect.left + rect.width / 2,
        y: rect.top - 10
      });
    }
  }, [children]);

  return (
    <div ref={ref} style={{ position: 'absolute', left: position.x, top: position.y }}>
      {children}
    </div>
  );
}

使い分けのガイドライン

基本的にはuseEffectを使用し、DOM測定やちらつき防止が必要な場合のみuseLayoutEffectを使用します。

5.6 React18のStrictModeでの二重実行

React 18以降、開発モードのStrictModeでは、useEffectが意図的に2回実行されます。これは、将来のReactの機能（コンポーネントの再マウント）に備えた挙動です。

二重実行の流れ

flowchart LR A["コンポーネント
マウント"] --> B["エフェクト実行
1回目"] B --> C["クリーンアップ
実行"] C --> D["エフェクト実行
2回目"] D --> E["通常の動作へ"]

なぜ二重実行するのか

将来のReactでは、ユーザーが画面を離れてから戻ってきた際に、コンポーネントの状態を保持したまま再マウントする機能が追加される予定です。この機能に対応するため、クリーンアップ関数が正しく実装されているかを確認できるよう、開発モードで意図的に二重実行しています。

二重実行に対応したコード

二重実行に対応したデータフェッチング 
function UserList() {
  const [users, setUsers] = useState<User[]>([]);

  useEffect(() => {
    let ignore = false; // クリーンアップ用フラグ

    fetch('/api/users')
      .then(res => res.json())
      .then(data => {
        // クリーンアップされていなければ状態を更新
        if (!ignore) {
          setUsers(data);
        }
      });

    // クリーンアップ: フラグを立てる
    return () => {
      ignore = true;
    };
  }, []);

  return <ul>{users.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
}

実習 5-1: リアルタイム検索機能の実装

useEffectとAbortControllerを使って、入力に応じてリアルタイムに検索結果を表示する機能を実装してみましょう。

手順

  1. 1 新しいコンポーネント SearchBox.tsx を作成
  2. 2 以下のコードを実装
  3. 3 App.tsxで使用して動作を確認

実装コード

import { useState, useEffect } from 'react';

interface SearchResult {
  id: string;
  title: string;
}

function SearchBox() {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState<SearchResult[]>([]);
  const [loading, setLoading] = useState(false);

  useEffect(() => {
    // 空文字の場合は何もしない
    if (!query.trim()) {
      setResults([]);
      return;
    }

    const controller = new AbortController();
    setLoading(true);

    // API呼び出し（遅延を追加してリアルタイム感を出す）
    const timeoutId = setTimeout(() => {
      fetch(`/api/search?q=${encodeURIComponent(query)}`, {
        signal: controller.signal
      })
        .then(res => res.json())
        .then(data => {
          if (!controller.signal.aborted) {
            setResults(data);
            setLoading(false);
          }
        })
        .catch(err => {
          if (err.name !== 'AbortError') {
            console.error(err);
            setLoading(false);
          }
        });
    }, 300); // デバウンス: 300ms待つ

    return () => {
      clearTimeout(timeoutId);
      controller.abort();
    };
  }, [query]);

  return (
    <div>
      <input
        type="text"
        value={query}
        onChange={e => setQuery(e.target.value)}
        placeholder="検索..."
        className="border p-2 rounded w-full"
      />
      {loading && <p>検索中...</p>}
      <ul>
        {results.map(result => (
          <li key={result.id}>{result.title}</li>
        ))}
      </ul>
    </div>
  );
}

期待される動作

入力フィールドに文字を入力すると、300ms後に検索APIが呼ばれ、結果が表示されます。入力が変更されると前のリクエストはキャンセルされます。

理解度確認クイズ

  1. Q1. useEffectが実行されるタイミングはいつですか？

    回答を表示

    回答: useEffectは、コンポーネントのレンダリングが完了し、ブラウザに描画された後に実行されます。これにより、副作用の処理がレンダリングをブロックしません。

  2. Q2. 空の依存配列 [] を指定した場合、useEffectはいつ実行されますか？

    回答を表示

    回答: コンポーネントがマウントされた時に1回だけ実行されます。初期化処理やデータの初回取得に使用します。

  3. Q3. クリーンアップ関数が必要なのはどのような場合ですか？

    回答を表示

    回答: タイマーの停止、イベントリスナーの削除、WebSocketの切断、リクエストのキャンセルなど、後片付けが必要な副作用を実行した場合に必要です。クリーンアップしないとメモリリークの原因となります。

  4. Q4. AbortControllerを使う目的は何ですか？

    回答を表示

    回答: コンポーネントがアンマウントされた時や、新しいリクエストが開始された時に、前のリクエストをキャンセルするために使用します。これにより競合状態を回避できます。

第5章のまとめ

  • useEffectは副作用を実行するためのフックで、レンダリング後に実行される
  • 依存配列により、エフェクトの実行タイミングを制御できる
  • クリーンアップ関数で、タイマーやイベントリスナーの後片付けを行う
  • AbortControllerで競合状態を回避し、安全なデータフェッチングを実現
  • useLayoutEffectは描画前に実行され、DOM測定などに使用
  • React18のStrictModeでは二重実行され、クリーンアップの重要性が増した
