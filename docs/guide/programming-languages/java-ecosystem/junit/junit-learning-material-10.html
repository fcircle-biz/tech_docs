10.1 前章の復習

前章では、電卓アプリの基本的な機能（足し算、引き算、掛け算、割り算）のテストを作成しました。今回は、異常系のテストを追加し、テスト全体の品質を評価していきます。

前章で作成したCalculatorクラス

Calculator.java 
// 簡単な電卓クラス
public class Calculator {
    // 足し算
    public int add(int a, int b) {
        return a + b;
    }

    // 引き算
    public int subtract(int a, int b) {
        return a - b;
    }

    // 掛け算
    public int multiply(int a, int b) {
        return a * b;
    }

    // 割り算
    public double divide(int a, int b) {
        if (b == 0) {
            throw new IllegalArgumentException("0で割ることはできません");
        }
        return (double) a / b;
    }
}

今回やること

前章では正常系のテストを書きました。今回は異常系（エラーケース）のテストを追加し、テストの網羅性を確認します。

10.2 異常系テストの追加

正常に動くケースだけでなく、エラーが発生するケースもテストする必要があります。電卓アプリでは、特に0で割るケースが重要です。

オーバーフローのテスト

整数の計算では、結果が大きくなりすぎるとオーバーフロー（桁あふれ）が発生することがあります。

CalculatorTest.java（オーバーフローテスト） 
@Test
@DisplayName("足し算でオーバーフローしても正しく計算される")
void testAddOverflow() {
    Calculator calc = new Calculator();

    // Integer.MAX_VALUEに近い値で計算
    // 注: Javaの整数はオーバーフローすると負の値になる
    int result = calc.add(Integer.MAX_VALUE, 1);

    // オーバーフローが発生することを確認
    assertTrue(result < 0, "オーバーフロー発生");
}

境界値のテスト

境界値（きょうかいち）とは、0や負の数、最大値など、特別な意味を持つ値のことです。これらの値でテストすることが重要です。

CalculatorTest.java（境界値テスト） 
@Nested
@DisplayName("境界値テスト")
class BoundaryValueTests {

    @Test
    @DisplayName("0と0の足し算")
    void testAddZeros() {
        Calculator calc = new Calculator();
        assertEquals(0, calc.add(0, 0));
    }

    @Test
    @DisplayName("負の数同士の足し算")
    void testAddNegatives() {
        Calculator calc = new Calculator();
        assertEquals(-8, calc.add(-3, -5));
    }

    @Test
    @DisplayName("負の数同士の掛け算")
    void testMultiplyNegatives() {
        Calculator calc = new Calculator();
        // 負の数×負の数＝正の数
        assertEquals(15, calc.multiply(-3, -5));
    }
}

境界値テストのコツ

  • 0（ゼロ）- 特別な値
  • 負の数 - 正の数とは違う動作をすることがある
  • 最大値・最小値 - オーバーフローのリスク
  • 1つ前・1つ後の値 - 境界の前後をテスト

10.3 テストカバレッジとは？

テストカバレッジ（Test Coverage）とは、「プログラムのどれくらいがテストされているか」を表す指標です。

たとえ話：地図の塗りつぶし

日本地図に、訪れたことがある都道府県を塗りつぶしていくとします。 47都道府県のうち20都道府県を訪れたなら、カバレッジは約43%です。 プログラムのテストも同じで、全体のうちどれくらいがテストで実行されたかを測ります。

カバレッジの種類

行カバレッジ（Line Coverage）

プログラムの各行が1回以上実行されたかを測ります。最も基本的な指標です。

分岐カバレッジ（Branch Coverage）

if文の「true」「false」両方のパターンが実行されたかを測ります。

メソッドカバレッジ（Method Coverage）

各メソッドが1回以上呼び出されたかを測ります。

flowchart TB A[プログラムコード] --> B[行カバレッジ] A --> C[分岐カバレッジ] A --> D[メソッドカバレッジ] B --> E["実行された行 / 全体の行
例: 80行 / 100行 = 80%"] C --> F["実行された分岐 / 全ての分岐
例: 6分岐 / 10分岐 = 60%"] D --> G["実行されたメソッド / 全メソッド
例: 8メソッド / 10メソッド = 80%"]

10.4 JaCoCoでカバレッジを測定する

JaCoCo（Java Code Coverage）は、Javaプログラムのテストカバレッジを測定するツールです。

JaCoCoの設定（Maven）

pom.xmlにJaCoCoプラグインを追加します。

pom.xml 
<build>
    <plugins>
        <plugin>
            <groupId>org.jacoco</groupId>
            <artifactId>jacoco-maven-plugin</artifactId>
            <version>0.8.11</version>
            <executions>
                <execution>
                    <goals>
                        <goal>prepare-agent</goal>
                    </goals>
                </execution>
                <execution>
                    <id>report</id>
                    <phase>test</phase>
                    <goals>
                        <goal>report</goal>
                    </goals>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>

カバレッジレポートの生成

以下のコマンドでテストを実行し、カバレッジレポートを生成します。

コマンド 
mvn clean test

レポートはtarget/site/jacoco/index.htmlに生成されます。

実習: カバレッジレポートを見てみよう

  1. 1. pom.xmlにJaCoCoプラグインを追加する
  2. 2. mvn clean testコマンドを実行する
  3. 3. target/site/jacoco/index.htmlをブラウザで開く
  4. 4. Calculatorクラスのカバレッジを確認する

カバレッジレポートの見方

レポートの例

Calculator クラス
┌────────────────┬─────────┐
│ 項目           │カバレッジ│
├────────────────┼─────────┤
│ 行カバレッジ    │ 85%     │
│ 分岐カバレッジ  │ 75%     │
│ メソッドカバレッジ│ 100%  │
└────────────────┴─────────┘

緑色の行はテストで実行された部分、黄色は一部だけ実行、赤色は未実行の部分を表します。

10.5 どのくらいのカバレッジが必要？

「テストカバレッジは何%にすべきか？」という質問がよくあります。

注意：100%を目指す必要はない

カバレッジ100%を目指すのは非効率です。テストが難しい部分や、テストの価値が低い部分もあります。 重要なのは「意味のあるテストを書くこと」です。

一般的な目安

60%未満

テストが不足している可能性があります。

70〜80%

良好なカバレッジです。多くのプロジェクトの目標値です。

80%以上

優秀なカバレッジです。ただし品質も重要です。

カバレッジよりも大切なこと

重要な機能を優先的にテストする

エラーが起きると困る機能や、よく使われる機能を重点的にテストしましょう。

意味のあるテストを書く

数字を上げるだけの形式的なテストではなく、バグを見つけられるテストを書きましょう。

境界値や異常系をテストする

通常のケースだけでなく、エラーケースもしっかりテストしましょう。

目安"] C --> G["バグを見つける
テスト"] D --> H["0、負の数
最大値など"] E --> I["エラーが

10.6 VSCodeでカバレッジを確認する

VSCodeには、テストカバレッジを視覚的に表示してくれる拡張機能があります。

Coverage Gutters拡張機能

拡張機能のインストール

  1. VSCodeの拡張機能タブを開く（Ctrl+Shift+X）
  2. 「Coverage Gutters」を検索してインストール
  3. コマンドパレット（Ctrl+Shift+P）を開く
  4. 「Coverage Gutters: Display Coverage」を実行

この拡張機能を使うと、コードエディタの左側に緑色（実行された）、赤色（未実行）のマークが表示されます。

メリット

どの行がテストされていないかが一目で分かるので、テストの追加が簡単になります。

10.7 総合演習：カバレッジを向上させよう

電卓アプリのテストカバレッジを測定し、80%以上を目指してテストを追加してみましょう。

実習手順

  1. 1.
    現在のカバレッジを確認

    mvn clean testを実行し、カバレッジレポートを見る

  2. 2.
    不足しているテストを見つける

    赤色や黄色で表示されている部分を確認する

  3. 3.
    境界値テストを追加

    0、負の数、最大値などのテストを追加する

  4. 4.
    再度カバレッジを確認

    カバレッジが向上したか確認する

動作確認のチェックポイント

  • 全てのテストが成功する（緑色）
  • 行カバレッジが80%以上になっている
  • 正常系と異常系の両方がテストされている
  • 境界値（0、負の数など）がテストされている

この章のまとめ

  • 異常系テストは、エラーが正しく発生することを確認する重要なテスト
  • 境界値テストで、0や負の数、最大値などの特別な値をテストする
  • テストカバレッジは、プログラムのどれくらいがテストされているかを表す指標
  • JaCoCoを使うと、カバレッジを自動的に測定してレポートを生成できる
  • カバレッジの目安は70〜80%だが、100%を目指す必要はない
  • 重要なのはカバレッジの数値ではなく、意味のあるテストを書くこと

理解度確認クイズ

  1. Q1. テストカバレッジとは何ですか？

    回答を表示
    テストカバレッジとは、プログラムのどれくらいの部分がテストで実行されたかを表す指標です。 行カバレッジ、分岐カバレッジ、メソッドカバレッジなどの種類があります。

  2. Q2. 境界値テストとは何ですか？どんな値をテストしますか？

    回答を表示

    境界値テストは、特別な意味を持つ値でテストすることです。

    テストする値の例：

    • 0（ゼロ）
    • 負の数
    • 最大値・最小値
    • 境界の前後の値

  3. Q3. JaCoCoとは何ですか？

    回答を表示
    JaCoCo（Java Code Coverage）は、Javaプログラムのテストカバレッジを測定し、 レポートを生成してくれるツールです。どの部分がテストされているか、されていないかを視覚的に確認できます。

  4. Q4. テストカバレッジは100%を目指すべきですか？理由も説明してください。

    回答を表示

    いいえ、100%を目指す必要はありません。

    理由：

    • テストが難しい部分や、テストの価値が低い部分もある
    • 100%を目指すと、形式的なテストが増えて非効率
    • 重要なのは数値ではなく、意味のあるテストを書くこと
    • 70〜80%が一般的な目標値として適切
