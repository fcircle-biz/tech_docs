様々な状況を確かめよう - パラメータ化テスト入門

同じテストロジックを異なるデータで繰り返し実行する「パラメータ化テスト」について学びます。九九の計算を一つ一つ手で確認するのではなく、一度に複数のパターンをテストする効率的な方法を解説します。

この章で学ぶこと

  • パラメータ化テストとは何かを理解する
  • @ParameterizedTestアノテーションの使い方を学ぶ
  • @ValueSource、@CsvSource、@MethodSourceを使い分ける
  • 複数のテストケースを効率的に実行する方法を習得する

6.1 パラメータ化テストとは？

これまで書いてきたテストは、1つのテストメソッドで1つのケースしか確認できませんでした。 しかし、実際には同じロジックを複数のデータでテストしたい場面が多くあります。

九九で例えると

「2 × 3 = 6」が正しいか確認するテストを書いた後、「2 × 4 = 8」「2 × 5 = 10」...と同じようなテストを何個も書くのは大変ですよね。 パラメータ化テストを使えば、「掛け算のテスト」を1つ書いて、複数の数字のパターンを一気にテストできます。

従来の方法の問題点

例えば、数値が正の数かどうかを判定するisPositiveメソッドをテストする場合：

従来の方法（非効率） 
@Test
void testIsPositive_1() {
    assertTrue(isPositive(1)); // 1は正の数
}

@Test
void testIsPositive_5() {
    assertTrue(isPositive(5)); // 5は正の数
}

@Test
void testIsPositive_100() {
    assertTrue(isPositive(100)); // 100は正の数
}

// 同じようなテストがたくさん...

この方法の問題点

  • 同じようなテストを何度も書く必要がある（コードが冗長）
  • テストメソッドがたくさんできて管理が大変
  • 新しいパターンを追加するたびに新しいメソッドを作る必要がある

パラメータ化テストの利点

パラメータ化テストを使うと、テストロジックは1つだけ書いて、複数の入力データでそのテストを実行できます。

パラメータ化テストのメリット

  • テストコードが短く、読みやすくなる
  • 新しいテストパターンを簡単に追加できる
  • テストロジックが一箇所にまとまるので保守しやすい
  • どのデータでテストが失敗したかがすぐに分かる
flowchart LR A[テストロジック] --> B[データ1: 1] A --> C[データ2: 5] A --> D[データ3: 100] A --> E[データ4: 999] B --> F[実行結果] C --> F D --> F E --> F

6.2 @ParameterizedTestの基本

パラメータ化テストを書くには、@Testの代わりに@ParameterizedTestアノテーションを使います。

必要な準備

パラメータ化テストを使うには、JUnit 5の追加モジュールjunit-jupiter-paramsが必要です。

Maven の場合、pom.xmlに以下の依存関係を追加します：

pom.xml（依存関係の追加） 
<dependency>
    <groupId>org.junit.jupiter</groupId>
    <artifactId>junit-jupiter-params</artifactId>
    <version>5.10.0</version>
    <scope>test</scope>
</dependency>

基本的な書き方

では、実際にパラメータ化テストを書いてみましょう。

NumberUtils.java（テスト対象クラス） 
public class NumberUtils {

    // 正の数かどうかを判定
    public static boolean isPositive(int number) {
        return number > 0;
    }
}
パラメータ化テストの例 
import org.junit.jupiter.params.ParameterizedTest;
import org.junit.jupiter.params.provider.ValueSource;
import static org.junit.jupiter.api.Assertions.*;

class NumberUtilsTest {

    @ParameterizedTest
    @ValueSource(ints = {1, 5, 10, 100, 999})
    void testIsPositive(int number) {
        // 各数値に対してテストを実行
        assertTrue(NumberUtils.isPositive(number));
    }
}

何が起きているか

  • @ParameterizedTest：このテストがパラメータ化テストであることを示す
  • @ValueSource(ints = {1, 5, 10, 100, 999})：テストするデータを指定
  • JUnitが自動的に5回テストを実行（データごとに1回）
  • 各実行で異なる数値がnumberパラメータに渡される

実行結果

このテストを実行すると、5つのテストケースが個別に実行されます： 

✓ testIsPositive(int) [1] - passed
✓ testIsPositive(int) [2] - passed
✓ testIsPositive(int) [3] - passed
✓ testIsPositive(int) [4] - passed
✓ testIsPositive(int) [5] - passed

Tests run: 5, Failures: 0

6.3 @ValueSourceの使い方

@ValueSourceは、単一の値を複数提供するときに使います。 int型だけでなく、様々な型のデータを扱えます。

使える型

整数型（ints） 
@ParameterizedTest
@ValueSource(ints = {1, 2, 3, 10, 100})
void testWithInts(int value) {
    assertTrue(value > 0);
}
文字列型（strings） 
@ParameterizedTest
@ValueSource(strings = {"apple", "banana", "cherry"})
void testWithStrings(String fruit) {
    // 各文字列が5文字以下かテスト
    assertTrue(fruit.length() <= 6);
}
真偽値型（booleans） 
@ParameterizedTest
@ValueSource(booleans = {true, false})
void testWithBooleans(boolean value) {
    // true と false 両方でテスト
    assertNotNull(value);
}
小数型（doubles） 
@ParameterizedTest
@ValueSource(doubles = {1.5, 2.0, 3.14, 10.5})
void testWithDoubles(double value) {
    assertTrue(value > 0.0);
}

その他の型

他にもlongs（long型）、shorts（short型）、bytes（byte型）、floats（float型）、 chars（char型）、classes（Class型）が使えます。

6.4 @CsvSourceで複数の値を渡す

@ValueSourceは単一の値しか渡せませんが、@CsvSourceを使うと複数の値をセットで渡せます。

CSV とは

CSV（Comma-Separated Values）は「カンマ区切りの値」のことです。 例えば「2, 3, 6」は「2と3と6」という3つの値を表します。 Excelでよく使われる形式です。

基本的な使い方

例えば、足し算のテストで「入力1、入力2、期待結果」をセットで渡す場合：

Calculator.java（テスト対象） 
public class Calculator {

    public int add(int a, int b) {
        return a + b;
    }
}
@CsvSourceを使ったテスト 
import org.junit.jupiter.params.ParameterizedTest;
import org.junit.jupiter.params.provider.CsvSource;
import static org.junit.jupiter.api.Assertions.*;

class CalculatorTest {

    @ParameterizedTest
    @CsvSource({
        "1, 2, 3",     // 1 + 2 = 3
        "5, 5, 10",    // 5 + 5 = 10
        "10, 20, 30",  // 10 + 20 = 30
        "0, 0, 0"      // 0 + 0 = 0
    })
    void testAdd(int a, int b, int expected) {
        Calculator calc = new Calculator();
        assertEquals(expected, calc.add(a, b));
    }
}

何が起きているか

  • 各行が1つのテストケース
  • カンマで区切られた値が順番にパラメータに渡される
  • 「1, 2, 3」→ a=1, b=2, expected=3
  • テストが4回実行される（データセットが4つ）

文字列データのテスト

文字列を扱う場合も同じです。文字列にスペースやカンマが含まれる場合は、シングルクォートで囲みます。

文字列の長さをテスト 
@ParameterizedTest
@CsvSource({
    "apple, 5",
    "banana, 6",
    "cherry, 6",
    "'hello world', 11"  // スペースを含む場合はシングルクォート
})
void testStringLength(String text, int length) {
    assertEquals(length, text.length());
}

null や空文字列のテスト

特殊な値のテスト 
@ParameterizedTest
@CsvSource({
    "apple, false",
    "'', true",      // 空文字列
    ", true"         // null（カンマの後に何もない）
})
void testIsEmpty(String text, boolean expected) {
    boolean result = (text == null || text.isEmpty());
    assertEquals(expected, result);
}

6.5 @MethodSourceでより複雑なデータを提供

より複雑なテストデータを用意したい場合、@MethodSourceを使うと、 別のメソッドでテストデータを生成して提供できます。

基本的な使い方

@MethodSourceの例 
import org.junit.jupiter.params.ParameterizedTest;
import org.junit.jupiter.params.provider.MethodSource;
import java.util.stream.Stream;

class CalculatorTest {

    @ParameterizedTest
    @MethodSource("provideNumbersForAdd")
    void testAdd(int a, int b, int expected) {
        Calculator calc = new Calculator();
        assertEquals(expected, calc.add(a, b));
    }

    // テストデータを生成するメソッド
    static Stream provideNumbersForAdd() {
        return Stream.of(
            Arguments.of(1, 2, 3),
            Arguments.of(5, 5, 10),
            Arguments.of(10, 20, 30)
        );
    }
}

ポイント

  • • データ提供メソッドはstaticで、Stream<Arguments>を返す
  • • Arguments.of()で複数の値をセットで渡す
  • • メソッド名を@MethodSourceで指定する

単一の値を提供する場合

パラメータが1つだけの場合は、Streamで直接値を返せます：

単一パラメータの例 
@ParameterizedTest
@MethodSource("providePositiveNumbers")
void testIsPositive(int number) {
    assertTrue(NumberUtils.isPositive(number));
}

static Stream providePositiveNumbers() {
    return Stream.of(1, 5, 10, 100, 999);
}

複雑なオブジェクトをテストする

@MethodSourceの真価は、複雑なオブジェクトをテストデータとして使えることです。

User.java（テスト対象） 
public class User {
    private String name;
    private int age;

    public User(String name, int age) {
        this.name = name;
        this.age = age;
    }

    public boolean isAdult() {
        return age >= 18;
    }
}
Userオブジェクトのテスト 
@ParameterizedTest
@MethodSource("provideUsers")
void testIsAdult(User user, boolean expected) {
    assertEquals(expected, user.isAdult());
}

static Stream provideUsers() {
    return Stream.of(
        Arguments.of(new User("Alice", 20), true),
        Arguments.of(new User("Bob", 17), false),
        Arguments.of(new User("Charlie", 18), true)
    );
}

@MethodSourceのメリット

  • 複雑なオブジェクトをテストデータとして使える
  • ループや条件分岐を使って動的にデータを生成できる
  • 外部ファイルからデータを読み込むこともできる
  • 再利用可能なテストデータを定義できる

実習：パラメータ化テストを書いてみよう

以下のメソッドに対して、パラメータ化テストを作成してください。

StringUtils.java 
public class StringUtils {

    // 文字列を反転
    public static String reverse(String str) {
        return new StringBuilder(str).reverse().toString();
    }

    // 回文かどうかを判定
    public static boolean isPalindrome(String str) {
        String reversed = reverse(str);
        return str.equals(reversed);
    }
}

課題1：@ValueSourceを使う

reverseメソッドをテストしてください。複数の文字列で確認しましょう。

解答例を表示 
@ParameterizedTest
@ValueSource(strings = {"abc", "hello", "12345"})
void testReverse(String input) {
    String result = StringUtils.reverse(input);
    // 2回反転すると元に戻る
    assertEquals(input, StringUtils.reverse(result));
}

課題2：@CsvSourceを使う

isPalindromeメソッドをテストしてください。文字列と期待結果のペアを用意しましょう。

解答例を表示 
@ParameterizedTest
@CsvSource({
    "radar, true",    // 回文
    "level, true",    // 回文
    "hello, false",   // 回文ではない
    "noon, true"      // 回文
})
void testIsPalindrome(String str, boolean expected) {
    assertEquals(expected, StringUtils.isPalindrome(str));
}

この章のまとめ

  • パラメータ化テストを使うと、同じロジックを複数のデータで効率的にテストできる
  • @ParameterizedTestアノテーションで、パラメータ化テストであることを宣言する
  • @ValueSourceは、単一の値を複数提供する（ints, strings, doublesなど）
  • @CsvSourceは、複数の値をセットで提供する（カンマ区切り）
  • @MethodSourceは、複雑なオブジェクトや動的に生成したデータを提供できる

理解度確認クイズ

  1. Q1. パラメータ化テストの主なメリットを3つ挙げてください。

    回答を表示
    1. テストコードが短く、読みやすくなる
    2. 新しいテストパターンを簡単に追加できる
    3. テストロジックが一箇所にまとまるので保守しやすい

  2. Q2. @ValueSource、@CsvSource、@MethodSourceの違いを説明してください。

    回答を表示

    @ValueSource：単一の値を複数提供する（例：複数の数値や文字列）

    @CsvSource：複数の値をセットで提供する（例：入力値と期待結果のペア）

    @MethodSource：メソッドでテストデータを生成し、複雑なオブジェクトや動的なデータを提供する

  3. Q3. 次のコードを@CsvSourceを使ったパラメータ化テストに書き換えてください。

    @Test
void testMax_2_5() {
    assertEquals(5, Math.max(2, 5));
}

@Test
void testMax_10_3() {
    assertEquals(10, Math.max(10, 3));
}
    回答を表示 
    @ParameterizedTest
@CsvSource({
    "2, 5, 5",
    "10, 3, 10"
})
void testMax(int a, int b, int expected) {
    assertEquals(expected, Math.max(a, b));
}
