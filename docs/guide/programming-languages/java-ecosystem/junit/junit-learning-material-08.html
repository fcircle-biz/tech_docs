テストをもっと読みやすく - DisplayNameとネストクラス

テストの実行結果を分かりやすくする@DisplayNameアノテーションと、テストをグループ化する@Nestedアノテーションについて学びます。テストレポートを読む人のことを考えた、分かりやすいテストコードの書き方を解説します。

この章で学ぶこと

  • @DisplayNameアノテーションの使い方を理解する
  • テストレポートを分かりやすくする方法を学ぶ
  • @Nestedアノテーションでテストをグループ化する方法を習得する
  • 読みやすいテスト構造の作り方を身につける

8.1 なぜテストを読みやすくするのか？

これまでの章で、テストメソッドにはtestAdd()やtestDivideByZero()のような英語の名前を付けてきました。 でも、テストが失敗したとき、こんなレポートが表示されたらどうでしょう？

テスト失敗レポート（英語のメソッド名）

✗ testAdd()
✗ testDivideByZero()
✓ testSubtract()

これでも分かりますが、日本語で「何をテストしているのか」が分かる方が便利ですよね。

本棚のラベルで例えると

本棚に「Book1」「Book2」とラベルを貼るより、「料理本」「旅行ガイド」と書いた方が分かりやすいですよね。 テストも同じで、「何をテストしているか」が一目で分かる名前を付けると、チームで開発するときに便利です。

JUnit 5では、@DisplayNameアノテーションを使うことで、テスト結果に日本語の説明を表示できます。

8.2 @DisplayNameの基本

@DisplayNameアノテーションは、テストの表示名を日本語で指定できるアノテーションです。

基本的な使い方

CalculatorTest.java 
import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

@DisplayName("電卓のテスト")  // クラス全体の表示名
class CalculatorTest {

    @Test
    @DisplayName("2つの正の数を足し算できる")  // このテストの表示名
    void testAdd() {
        Calculator calc = new Calculator();
        int result = calc.add(2, 3);
        assertEquals(5, result);
    }

    @Test
    @DisplayName("0で割り算するとArithmeticExceptionが発生する")
    void testDivideByZero() {
        Calculator calc = new Calculator();
        assertThrows(ArithmeticException.class, () -> {
            calc.divide(10, 0);
        });
    }
}

このテストを実行すると、テスト結果が日本語で表示されます。

テスト結果（日本語表示）

✓ 電卓のテスト
  ✓ 2つの正の数を足し算できる
  ✓ 0で割り算するとArithmeticExceptionが発生する

ポイント

  • クラスにも@DisplayNameを付けられる
  • 各テストメソッドにも@DisplayNameを付けられる
  • 日本語で「何をテストしているか」を明確に書く

8.3 良い表示名の書き方

@DisplayNameで表示名を付けるときは、以下のポイントを意識しましょう。

1. 具体的に書く

「足し算のテスト」より「2つの正の数を足し算できる」の方が何をテストしているか明確です。

❌ 悪い例: "足し算のテスト"

✅ 良い例: "2つの正の数を足し算できる"

2. 期待する結果を書く

「何を確認するか」だけでなく、「どうなるべきか」も書きましょう。

❌ 悪い例: "0で割り算する"

✅ 良い例: "0で割り算するとArithmeticExceptionが発生する"

3. テストの状況を書く

「いつ」「どんなとき」をテストするかを明記すると、より分かりやすくなります。

✅ 例: "空のリストに要素を追加すると、サイズが1になる"

✅ 例: "残高が0円のとき、出金するとエラーになる"

flowchart LR A[テスト名] --> B{具体的?} B -->|はい| C{結果を書いた?} B -->|いいえ| D[もっと詳しく] C -->|はい| E[良い表示名] C -->|いいえ| F[結果を追加] D --> C F --> E

8.4 @Nestedでテストをグループ化しよう

テストが増えてくると、「足し算のテスト」「引き算のテスト」「エラーのテスト」など、 関連するテストをグループにまとめたいときがあります。

そんなときに使うのが@Nestedアノテーションです。

フォルダで例えると

ファイルをフォルダに分けて整理するように、テストも関連するものをグループにまとめると分かりやすくなります。 「足し算フォルダ」「引き算フォルダ」のようにテストを整理するイメージです。

@Nestedの使い方

CalculatorTest.java 
import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.Nested;
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

@DisplayName("電卓のテスト")
class CalculatorTest {

    // 足し算のテストグループ
    @Nested
    @DisplayName("足し算のテスト")
    class AddTests {
        @Test
        @DisplayName("2つの正の数を足せる")
        void positiveNumbers() {
            Calculator calc = new Calculator();
            assertEquals(5, calc.add(2, 3));
        }

        @Test
        @DisplayName("負の数も足せる")
        void negativeNumbers() {
            Calculator calc = new Calculator();
            assertEquals(-5, calc.add(-2, -3));
        }
    }

    // 割り算のテストグループ
    @Nested
    @DisplayName("割り算のテスト")
    class DivideTests {
        @Test
        @DisplayName("正しく割り算できる")
        void normalDivide() {
            Calculator calc = new Calculator();
            assertEquals(5, calc.divide(10, 2));
        }

        @Test
        @DisplayName("0で割るとエラーになる")
        void divideByZero() {
            Calculator calc = new Calculator();
            assertThrows(ArithmeticException.class,
                () -> calc.divide(10, 0));
        }
    }
}

このテストを実行すると、テストが階層的に表示されます。

テスト結果（階層表示）

✓ 電卓のテスト
  ✓ 足し算のテスト
    ✓ 2つの正の数を足せる
    ✓ 負の数も足せる
  ✓ 割り算のテスト
    ✓ 正しく割り算できる
    ✓ 0で割るとエラーになる

@Nestedのメリット

  • 関連するテストをグループにまとめられる
  • テストレポートが階層的で分かりやすい
  • テストの構造が一目で分かる
  • 同じ準備処理を共有できる（@BeforeEachを各グループで使える）

8.5 実践例：銀行口座クラスのテスト

実際のテストで、@DisplayNameと@Nestedを組み合わせて使ってみましょう。

BankAccountTest.java 
import org.junit.jupiter.api.*;
import static org.junit.jupiter.api.Assertions.*;

@DisplayName("銀行口座のテスト")
class BankAccountTest {

    // 入金のテストグループ
    @Nested
    @DisplayName("入金機能")
    class DepositTests {
        private BankAccount account;

        @BeforeEach
        void setUp() {
            account = new BankAccount(1000);  // 残高1000円
        }

        @Test
        @DisplayName("正の金額を入金できる")
        void depositPositiveAmount() {
            account.deposit(500);
            assertEquals(1500, account.getBalance());
        }

        @Test
        @DisplayName("0円は入金できない")
        void depositZero() {
            assertThrows(IllegalArgumentException.class,
                () -> account.deposit(0));
        }

        @Test
        @DisplayName("負の金額は入金できない")
        void depositNegative() {
            assertThrows(IllegalArgumentException.class,
                () -> account.deposit(-100));
        }
    }

    // 出金のテストグループ
    @Nested
    @DisplayName("出金機能")
    class WithdrawTests {
        private BankAccount account;

        @BeforeEach
        void setUp() {
            account = new BankAccount(1000);
        }

        @Test
        @DisplayName("残高以内の金額を出金できる")
        void withdrawWithinBalance() {
            account.withdraw(500);
            assertEquals(500, account.getBalance());
        }

        @Test
        @DisplayName("残高を超える金額は出金できない")
        void withdrawExceedsBalance() {
            assertThrows(IllegalArgumentException.class,
                () -> account.withdraw(1500));
        }

        @Test
        @DisplayName("残高ちょうどの金額を出金できる")
        void withdrawExactBalance() {
            account.withdraw(1000);
            assertEquals(0, account.getBalance());
        }
    }

    // 残高照会のテストグループ
    @Nested
    @DisplayName("残高照会")
    class BalanceTests {
        @Test
        @DisplayName("初期残高が正しく設定される")
        void initialBalance() {
            BankAccount account = new BankAccount(5000);
            assertEquals(5000, account.getBalance());
        }

        @Test
        @DisplayName("初期残高が負の値の場合エラーになる")
        void negativeInitialBalance() {
            assertThrows(IllegalArgumentException.class,
                () -> new BankAccount(-100));
        }
    }
}

このテストを実行すると、以下のように分かりやすく表示されます。

テスト結果

✓ 銀行口座のテスト
  ✓ 入金機能
    ✓ 正の金額を入金できる
    ✓ 0円は入金できない
    ✓ 負の金額は入金できない
  ✓ 出金機能
    ✓ 残高以内の金額を出金できる
    ✓ 残高を超える金額は出金できない
    ✓ 残高ちょうどの金額を出金できる
  ✓ 残高照会
    ✓ 初期残高が正しく設定される
    ✓ 初期残高が負の値の場合エラーになる
flowchart TB A[銀行口座のテスト] --> B[入金機能] A --> C[出金機能] A --> D[残高照会] B --> B1[正の金額を入金できる] B --> B2[0円は入金できない] B --> B3[負の金額は入金できない] C --> C1[残高以内を出金できる] C --> C2[残高を超えると出金できない] C --> C3[残高ちょうど出金できる] D --> D1[初期残高が正しい] D --> D2[負の初期残高はエラー]

8.6 @Nestedの階層を深くする

@Nestedはさらにネスト（入れ子）にすることもできます。

深い階層の例 
@DisplayName("ショッピングカートのテスト")
class ShoppingCartTest {

    @Nested
    @DisplayName("商品の追加")
    class AddItemTests {

        @Nested
        @DisplayName("通常の商品")
        class NormalItemTests {
            @Test
            @DisplayName("在庫がある商品を追加できる")
            void addAvailableItem() {
                // テストコード
            }
        }

        @Nested
        @DisplayName("特別な商品")
        class SpecialItemTests {
            @Test
            @DisplayName("予約商品を追加できる")
            void addPreOrderItem() {
                // テストコード
            }
        }
    }
}

注意：階層を深くしすぎない

階層が深すぎると逆に分かりにくくなります。通常は2〜3階層までにしましょう。

実習：読みやすいテストを書こう

以下のテストクラスに@DisplayNameと@Nestedを追加して、読みやすくしてみましょう。

課題：

  1. StringUtilsTestクラスに適切な@DisplayNameを付ける
  2. isEmpty関連のテストと、reverse関連のテストを@Nestedでグループ化する
  3. 各テストメソッドに日本語の@DisplayNameを付ける
// 修正前
class StringUtilsTest {
    @Test
    void testIsEmpty1() {
        assertTrue(StringUtils.isEmpty(""));
    }

    @Test
    void testIsEmpty2() {
        assertTrue(StringUtils.isEmpty(null));
    }

    @Test
    void testReverse1() {
        assertEquals("cba", StringUtils.reverse("abc"));
    }

    @Test
    void testReverse2() {
        assertEquals("", StringUtils.reverse(""));
    }
}

この章のまとめ

  • @DisplayNameを使うと、テスト結果を日本語で分かりやすく表示できる
  • 良い表示名は具体的で、結果が明確で、状況が分かるもの
  • @Nestedを使うと、関連するテストをグループにまとめられる
  • テストの構造が階層的になり、レポートが読みやすくなる
  • @Nestedの各グループで@BeforeEachなどを個別に使える
  • 階層が深すぎると逆に分かりにくくなるので、2〜3階層までが目安

理解度確認クイズ

  1. Q1. @DisplayNameアノテーションの役割は何ですか？

    回答を表示
    @DisplayNameは、テスト結果に表示される名前を日本語などの分かりやすい文字列で指定できるアノテーションです。 テストレポートが読みやすくなり、何をテストしているかが一目で分かります。

  2. Q2. 良い表示名を書くときの3つのポイントを挙げてください。

    回答を表示
    1. 具体的に書く（何をテストしているか明確に）
    2. 期待する結果を書く（どうなるべきかを明記）
    3. テストの状況を書く（いつ・どんなときをテストするか）

  3. Q3. @Nestedアノテーションを使うメリットを3つ挙げてください。

    回答を表示
    1. 関連するテストをグループにまとめられる
    2. テストレポートが階層的で分かりやすくなる
    3. 各グループで@BeforeEachなどの準備処理を個別に使える

  4. Q4. @Nestedの階層を深くしすぎない方がいい理由は何ですか？

    回答を表示
    階層が深すぎると、逆にテスト構造が複雑になり分かりにくくなるためです。 通常は2〜3階層までにしておくと、シンプルで読みやすいテスト構造を保てます。
