テスト駆動開発（TDD）入門 - テストから始めるプログラミング

「先にテストを書いてから実装する」というテスト駆動開発（TDD）の考え方を学びます。Red-Green-Refactorサイクルを体験し、テストファーストのメリットを実感します。

この章で学ぶこと

  • テスト駆動開発（TDD）とは何かを理解する
  • Red-Green-Refactorサイクルを体験する
  • TDDのメリットと考え方を理解する
  • 簡単な機能をTDDで実装する

11.1 テスト駆動開発（TDD）とは？

テスト駆動開発（Test-Driven Development、略してTDD）は、 「実装コードを書く前に、まずテストコードを書く」という開発手法です。

通常のプログラミングでは、「コードを書く→テストする」という順番ですが、 TDDでは「テストを書く→コードを書く」という逆の順番で進めます。

料理で例えると

通常は「料理を作ってから味見をする」ですが、TDDは「先に『こういう味にしたい』という基準を決めてから料理を作る」ようなものです。 目標がはっきりしているので、迷わず料理できますね。

flowchart TB subgraph traditional["通常の開発"] T1[実装コードを書く] --> T2[テストコードを書く] T2 --> T3[テスト実行] T3 --> T4{合格?} T4 -->|いいえ| T5[コード修正] T5 --> T3 T4 -->|はい| T6[完成] end subgraph tdd["TDD（テスト駆動開発）"] D1[テストコードを書く] --> D2[実装コードを書く] D2 --> D3[テスト実行] D3 --> D4{合格?} D4 -->|いいえ| D2 D4 -->|はい| D5[リファクタリング] D5 --> D6[完成] end

11.2 Red-Green-Refactorサイクル

TDDの開発は、Red→Green→Refactorという3つのステップを繰り返すサイクルで進みます。

1. Red（赤）- 失敗するテストを書く

まだ実装されていない機能のテストを書きます。当然、テストは失敗（赤色）します。 これが「目標」になります。

2. Green（緑）- テストを通す最小限のコードを書く

テストが合格（緑色）するように、最小限の実装コードを書きます。 綺麗なコードである必要はありません。まずは「動く」ことが優先です。

3. Refactor（リファクタリング）- コードを綺麗にする

テストが通っている状態で、コードを整理・改善します。 テストがあるので、安心してリファクタリングできます。

flowchart LR Red["🔴 Red
失敗するテストを書く"] --> Green["🟢 Green
テストを通す"] Green --> Refactor["🔵 Refactor
コードを整理"] Refactor --> Red

重要ポイント

TDDでは、必ず「失敗するテスト」から始めます。最初から成功するテストを書いても意味がありません。 「赤→緑」という変化が、正しく機能を実装できた証拠になるのです。

11.3 TDDの実践：FizzBuzz問題

TDDを実際に体験してみましょう。有名なFizzBuzz問題をTDDで解いてみます。

FizzBuzz問題とは

数字を受け取り、以下のルールで文字列を返す関数を作ります。

  • 3の倍数なら「Fizz」
  • 5の倍数なら「Buzz」
  • 3と5の両方の倍数なら「FizzBuzz」
  • それ以外はその数字を文字列として返す

ステップ1：Red（失敗するテストを書く）

まず、一番簡単なケースのテストを書きます。「3でも5でも割り切れない数字は、そのまま文字列として返す」というテストです。

FizzBuzzTest.java（最初のテスト） 
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

class FizzBuzzTest {
    @Test
    void testNormalNumber() {
        // まだFizzBuzzクラスは存在しないので、これはコンパイルエラー
        FizzBuzz fizzBuzz = new FizzBuzz();
        assertEquals("1", fizzBuzz.convert(1));
    }
}

実行結果：テスト失敗（Red）

FizzBuzzクラスが存在しないので、コンパイルエラーになります。これが「Red（赤）」の状態です。

ステップ2：Green（テストを通す最小限のコード）

テストが通るように、最小限のコードを書きます。

FizzBuzz.java（最小実装） 
public class FizzBuzz {
    // テストを通すだけの最小限の実装
    public String convert(int number) {
        return "1"; // とりあえず固定値を返す
    }
}

実行結果：テスト成功（Green）

テストが通りました！まだ完璧ではありませんが、とりあえず「Green（緑）」になりました。

ステップ3：次のテストを追加（再びRed）

次のケースのテストを追加します。「別の数字でも正しく動くか」を確認します。

FizzBuzzTest.java（テスト追加） 
@Test
void testAnotherNumber() {
    FizzBuzz fizzBuzz = new FizzBuzz();
    assertEquals("2", fizzBuzz.convert(2)); // これは失敗する
}

このテストを実行すると、「1」が返ってくるので失敗します（再びRed）。実装を改善しましょう。

FizzBuzz.java（改善版） 
public class FizzBuzz {
    public String convert(int number) {
        // 数字を文字列に変換して返す
        return String.valueOf(number);
    }
}

ステップ4：Fizzのテストを追加

次は「3の倍数なら『Fizz』を返す」というテストを追加します。

FizzBuzzTest.java（Fizzテスト） 
@Test
void testFizz() {
    FizzBuzz fizzBuzz = new FizzBuzz();
    assertEquals("Fizz", fizzBuzz.convert(3)); // 3の倍数
    assertEquals("Fizz", fizzBuzz.convert(6)); // 3の倍数
}

テストが失敗するので（Red）、実装を追加します。

FizzBuzz.java（Fizz対応） 
public class FizzBuzz {
    public String convert(int number) {
        // 3の倍数なら「Fizz」を返す
        if (number % 3 == 0) {
            return "Fizz";
        }
        return String.valueOf(number);
    }
}

ステップ5：BuzzとFizzBuzzも同様に追加

同じ要領で「5の倍数でBuzz」「15の倍数でFizzBuzz」のテストと実装を追加していきます。

FizzBuzz.java（完成版） 
public class FizzBuzz {
    public String convert(int number) {
        // 15の倍数（3と5の両方の倍数）を先にチェック
        if (number % 15 == 0) {
            return "FizzBuzz";
        }
        // 3の倍数
        if (number % 3 == 0) {
            return "Fizz";
        }
        // 5の倍数
        if (number % 5 == 0) {
            return "Buzz";
        }
        // それ以外はそのまま文字列化
        return String.valueOf(number);
    }
}

TDDのポイント

  • 一度に全部作らず、少しずつ機能を追加
  • テストが失敗（Red）→成功（Green）のサイクルを繰り返す
  • テストがあるので、安心して実装を進められる

11.4 TDDのメリット

TDDで開発すると、以下のようなメリットがあります。

1. 設計が明確になる

先にテストを書くことで、「この機能をどう使うか」を考えるので、設計が明確になります。

2. バグが少なくなる

小さなステップで進めるので、バグが入りにくく、入っても早く発見できます。

3. 自信を持ってリファクタリングできる

テストがあるので、コードを整理しても「壊れていないか」をすぐ確認できます。

4. テストを書き忘れない

「後でテストを書こう」と思っても忘れがちですが、TDDなら必ずテストが書かれます。

5. ドキュメント代わりになる

テストを読めば「この機能はどう使うのか」がわかります。

11.5 TDDの注意点と使い分け

TDDは素晴らしい手法ですが、万能ではありません。以下の点に注意しましょう。

TDDに向いていないケース

  • UIのデザイン試行錯誤（見た目の調整は実際に見ながらやる方が効率的）
  • 新しい技術の学習・実験（まず動かしてみてから理解する段階）
  • 要件が不明確な段階（何をテストすべきかわからない）

TDDに向いているケース

  • ビジネスロジック（計算、判定など）
  • 複雑なアルゴリズム
  • 要件が明確な機能
  • バグが許されない重要な機能

TDDは「必ず使わなければいけない」ものではありません。 プロジェクトの状況や、自分のスタイルに合わせて使い分けることが大切です。

実習 11-1: TDDで偶数判定メソッドを作ろう

TDDのサイクルを体験するため、「数字が偶数かどうかを判定するメソッド」を実装してみましょう。

手順

  1. 1 まず、テストケースを考えます：2は偶数、3は奇数
  2. 2 テストを書きます（まだNumberUtilsクラスは存在しない）
  3. 3 NumberUtilsクラスとisEvenメソッドを作成してテストを通す
  4. 4 必要に応じてリファクタリング
ヒント：テストコード 
@Test
void testEvenNumber() {
    assertTrue(NumberUtils.isEven(2)); // 2は偶数
}

@Test
void testOddNumber() {
    assertFalse(NumberUtils.isEven(3)); // 3は奇数
}

理解度確認クイズ

  1. Q1. TDD（テスト駆動開発）とは何ですか？通常の開発との違いは何ですか？

    回答を表示
    TDDは「実装コードを書く前に、まずテストコードを書く」開発手法です。 通常の開発は「コードを書く→テストする」ですが、TDDは「テストを書く→コードを書く」という逆の順番で進めます。

  2. Q2. Red-Green-Refactorサイクルの3つのステップを説明してください。

    回答を表示

    Red（赤）：失敗するテストを書く（まだ実装されていない機能のテスト）

    Green（緑）：テストを通す最小限のコードを書く（まずは動けばOK）

    Refactor（リファクタリング）：テストが通っている状態でコードを綺麗にする

  3. Q3. TDDのメリットを3つ挙げてください。

    回答を表示
    1. 設計が明確になる（先にテストを書くことで使い方を考える）
    2. バグが少なくなる（小さなステップで進めるので早く発見できる）
    3. 自信を持ってリファクタリングできる（テストがあるので安心）

  4. Q4. TDDに向いていないケースはどのような場合ですか？

    回答を表示
    • UIのデザイン試行錯誤（見た目の調整は実際に見ながらやる方が効率的）
    • 新しい技術の学習・実験段階（まず動かしてから理解する）
    • 要件が不明確な段階（何をテストすべきかわからない）
