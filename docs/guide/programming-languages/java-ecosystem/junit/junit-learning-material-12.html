良いテストコードの特徴と、避けるべきアンチパターンを学びます。F.I.R.S.T原則を理解し、保守しやすいテストを書けるようになります。

12.1 F.I.R.S.T原則 - 良いテストの5つの特徴

良いテストコードには共通する特徴があります。それを「F.I.R.S.T原則」と呼びます。

F.I.R.S.Tは、5つの英単語の頭文字を取ったものです。この原則を守ることで、保守しやすい高品質なテストを書けるようになります。

mindmap root((F.I.R.S.T原則)) Fast
高速 瞬時に実行 待ち時間なし Independent
独立 他に依存しない 実行順序自由 Repeatable
再現可能 どこでも実行可能 環境に依存しない Self-validating
自己検証 自動で合否判定 手動確認不要 Timely
適時 実装前または直後 後回しにしない

F - Fast（高速）

テストは高速に実行されるべきです。待ち時間があると、開発者がテストを実行しなくなってしまいます。

良い例

  • 単体テストは1つあたり数ミリ秒で完了
  • 重い処理（DB接続、ファイルI/O）はモックを使う
  • テストスイート全体が数秒で完了

I - Independent（独立）

各テストは独立していて、他のテストに依存してはいけません。どのテストから実行しても、同じ結果になるべきです。

悪い例

test1でデータを準備し、test2でそのデータを使う。 test2単独では実行できません。

良い例

各テストで@BeforeEachを使って必要なデータを準備する。 どのテストも単独で実行可能。

R - Repeatable（再現可能）

テストはどんな環境でも同じ結果を返すべきです。実行するたびに結果が変わるテストは信頼できません。

悪い例

  • 現在時刻に依存するテスト（実行タイミングで結果が変わる）
  • ランダムな値を使うテスト
  • 外部APIに依存するテスト（ネットワーク状態で結果が変わる）

S - Self-validating（自己検証）

テストは自動で合格/不合格を判定すべきです。人間が結果を見て判断する必要があってはいけません。

良い例

assertEqualsやassertTrueなどのアサーションを使って、明確に検証する。 緑色のチェックマークで成功がわかる。

T - Timely（適時）

テストは適切なタイミングで書くべきです。理想はTDD（テスト駆動開発）のように実装前、または実装と同時です。

なぜ早く書くべきか

  • 実装後に書くと、テストしにくい設計になりがち
  • 後回しにすると、結局書かなくなる
  • バグを早く発見できる

12.2 わかりやすいテスト名の付け方

テストメソッド名は、「何をテストしているか」が一目でわかる名前にしましょう。

命名パターン

パターン1: Given-When-Then

givenX_whenY_thenZ

例：givenEmptyList_whenAddElement_thenSizeIsOne

パターン2: Should-When

shouldX_whenY

例：shouldThrowException_whenDivideByZero

わかりやすい命名の例 
// 良い例：テストの意図が明確
@Test
@DisplayName("空のリストに要素を追加すると、サイズが1になる")
void shouldReturnSizeOne_whenAddElementToEmptyList() {
    // テスト内容
}

@Test
@DisplayName("0で割り算すると、ArithmeticExceptionが発生する")
void shouldThrowArithmeticException_whenDivideByZero() {
    // テスト内容
}

// 悪い例：何をテストしているかわからない
@Test
void test1() {
    // これでは内容がわからない
}

@Test
void testCalculator() {
    // 計算機の何をテストしているのか不明
}

12.3 Arrange-Act-Assertパターン

テストコードは「準備→実行→検証」の3つの部分に分けると読みやすくなります。これをArrange-Act-Assertパターン（AAAパターン）と呼びます。

flowchart LR A[Arrange
準備] --> B[Act
実行] B --> C[Assert
検証] style A fill:#fef3c7 style B fill:#dbeafe style C fill:#dcfce7
1

Arrange（準備）

テストに必要なオブジェクトやデータを準備します。

2

Act（実行）

テスト対象のメソッドを実行します。

3

Assert（検証）

結果が期待通りかを検証します。

AAAパターンの実例 
@Test
@DisplayName("2つの数値を足すと、正しい合計が返される")
void shouldReturnCorrectSum_whenAddTwoNumbers() {
    // Arrange（準備）
    Calculator calculator = new Calculator();
    int num1 = 5;
    int num2 = 3;

    // Act（実行）
    int result = calculator.add(num1, num2);

    // Assert（検証）
    assertEquals(8, result, "5 + 3 は 8 になるべき");
}

コメントで明示するのもおすすめ

コメントで「// Arrange」「// Act」「// Assert」と書くと、テストの構造が一目でわかります。 特に複雑なテストでは効果的です。

12.4 避けるべきアンチパターン

よくある「悪いテストの書き方」を知って、同じ間違いを避けましょう。

1. 複数のことをテストする

悪い例

1つのテストで足し算、引き算、掛け算を全部テストする。 失敗したとき、どの機能が壊れたのかわかりません。

良い例

1つのテストは1つのことだけをテストする。 足し算のテスト、引き算のテスト、掛け算のテスト…と分ける。

2. テストの中で複雑なロジックを書く

悪い例

テストの中でif文やループを使って複雑な処理を書く。 テストコード自体にバグが入る可能性があります。

良い例

テストはシンプルに保つ。準備→実行→検証の流れを直線的に書く。

3. マジックナンバーを使う

マジックナンバーの例 
// 悪い例：100という数字の意味がわからない
@Test
void test() {
    calculator.add(50, 50);
    assertEquals(100, calculator.getResult());
}

// 良い例：変数名で意図が明確
@Test
void shouldReturn100_whenAdd50And50() {
    int firstNumber = 50;
    int secondNumber = 50;
    int expected = 100;

    calculator.add(firstNumber, secondNumber);

    assertEquals(expected, calculator.getResult());
}

4. テストに共有状態を持たせる

悪い例

static変数を使って複数のテスト間でデータを共有する。 テストの実行順序で結果が変わってしまいます。

5. 過度なテスト

悪い例

  • getterやsetterのような単純なメソッドまで全てテストする
  • フレームワークやライブラリの機能をテストする
  • 同じことを何度もテストする

バランスが大切

重要なビジネスロジックはしっかりテストし、単純な処理は省略するなど、 テストのコストと効果のバランスを考えましょう。

12.5 テストコードの保守性を高める

テストコードも「コード」です。本体コードと同じように、読みやすく保守しやすく書くことが重要です。

ヘルパーメソッドを使う

複数のテストで同じ準備処理を書いている場合、ヘルパーメソッドにまとめましょう。

ヘルパーメソッドの活用 
public class UserServiceTest {

    // ヘルパーメソッド：テスト用ユーザーを作成
    private User createTestUser(String name, int age) {
        User user = new User();
        user.setName(name);
        user.setAge(age);
        return user;
    }

    @Test
    void shouldReturnTrue_whenUserIsAdult() {
        // ヘルパーメソッドを使ってシンプルに
        User user = createTestUser("太郎", 20);

        assertTrue(userService.isAdult(user));
    }
}

テストデータビルダーパターン

複雑なオブジェクトを作る場合、ビルダーパターンを使うと可読性が上がります。

テストデータビルダー 
// テスト用のビルダークラス
public class UserTestBuilder {
    private String name = "デフォルト太郎";
    private int age = 20;

    public UserTestBuilder withName(String name) {
        this.name = name;
        return this;
    }

    public UserTestBuilder withAge(int age) {
        this.age = age;
        return this;
    }

    public User build() {
        User user = new User();
        user.setName(name);
        user.setAge(age);
        return user;
    }
}

// テストでの使用
@Test
void test() {
    User user = new UserTestBuilder()
        .withName("次郎")
        .withAge(18)
        .build();

    // テスト実行
}

実習：既存のテストをリファクタリング

これまで書いたテストコードを見直して、F.I.R.S.T原則やAAAパターンに沿っているか確認しましょう。

チェックポイント：

  • テストメソッド名は内容を表していますか？
  • 1つのテストで1つのことだけをテストしていますか？
  • 他のテストに依存していませんか？
  • Arrange-Act-Assertの構造が明確ですか？
  • マジックナンバーを使わず、意図が明確ですか？

この章のまとめ

  • F.I.R.S.T原則：Fast（高速）、Independent（独立）、Repeatable（再現可能）、Self-validating（自己検証）、Timely（適時）
  • テスト名は、何をテストしているかが一目でわかるように命名する
  • AAAパターン：Arrange（準備）、Act（実行）、Assert（検証）で構造化する
  • 1つのテストで1つのことだけをテストする
  • テストコードも本体コードと同じように、読みやすく保守しやすく書く

理解度確認クイズ

  1. Q1. F.I.R.S.T原則の5つの要素をすべて挙げてください。

    回答を表示
    1. Fast（高速）- テストは素早く実行されるべき
    2. Independent（独立）- テストは他のテストに依存しない
    3. Repeatable（再現可能）- どんな環境でも同じ結果になる
    4. Self-validating（自己検証）- 自動で合否判定する
    5. Timely（適時）- 適切なタイミングで書く

  2. Q2. AAAパターンの3つのフェーズを説明してください。

    回答を表示
    1. Arrange（準備）：テストに必要なオブジェクトやデータを準備する
    2. Act（実行）：テスト対象のメソッドを実行する
    3. Assert（検証）：結果が期待通りかを検証する

  3. Q3. 避けるべきアンチパターンを3つ挙げてください。

    回答を表示
    • 1つのテストで複数のことをテストする
    • テストの中で複雑なロジックを書く
    • マジックナンバーを使う
    • テストに共有状態を持たせる
    • 過度なテスト（単純なgetterまでテストする等）

    （上記から3つ以上答えられればOK）

  4. Q4. なぜテストの独立性（Independent）が重要なのですか？

    回答を表示
    テストが他のテストに依存していると、実行順序によって結果が変わったり、1つのテストの失敗が他のテストに影響を与えたりします。 独立していれば、どのテストも単独で実行でき、失敗の原因も特定しやすくなります。
