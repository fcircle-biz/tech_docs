セッション管理とログイン機能の実装

HTTPセッションを活用したユーザー認証とログイン状態の管理方法を理解しましょう。

この章で学ぶこと

  • ActionFormのスコープ（request/session）を使い分けられる
  • ログイン認証機能を実装できる
  • セッションにユーザー情報を保存・取得できる
  • ログインチェック処理を実装できる

9.1 HTTPセッションの基礎

HTTPセッションとは、サーバー側でユーザーごとの状態を保持する仕組みです。 HTTPプロトコルは本来ステートレス（状態を持たない）ですが、セッションを使うことでログイン状態や買い物カゴの内容などをユーザー単位で管理できます。

HTTPセッションの仕組み

sequenceDiagram autonumber participant User as ユーザー
ブラウザ participant Server as Webサーバー participant Session as セッションストア User->>Server: ログインリクエスト Server->>Session: セッション作成 Session-->>Server: セッションID発行 Server-->>User: レスポンス + Cookie
セッションID Note over User: 以降のリクエストで
セッションIDを送信 User->>Server: 2回目のリクエスト
セッションID付き Server->>Session: セッションID照合 Session-->>Server: ユーザー情報取得 Server-->>User: 認証済みとして処理

セッションIDとCookie

セッションIDはCookieに保存され、ブラウザが自動的に毎回送信します。サーバーはこのIDを使ってセッションストアから ユーザー固有のデータを取り出します。JavaではHttpSessionオブジェクトとして扱います。

9.2 ActionFormのスコープ設定（request vs session）

ActionFormにはスコープ（保存範囲）を指定できます。 リクエストスコープ（request）とセッションスコープ（session）の違いを理解し、適切に使い分けることが重要です。

スコープの比較

項目 requestスコープ sessionスコープ
保持期間 1回のリクエスト～レスポンスまで セッション終了まで（通常30分程度）
用途 1画面完結の入力フォーム 複数画面にまたがるウィザード形式
メモリ使用量 少ない（すぐ解放される） 多め（セッション終了まで保持）
設定例 scope="request" scope="session"
struts-config.xml 
<!-- リクエストスコープの例: 検索フォーム -->
<form-bean name="searchForm"
           type="com.example.form.SearchForm"/>

<action path="/search"
        type="com.example.action.SearchAction"
        name="searchForm"
        scope="request">
    <forward name="success" path="/result.jsp"/>
</action>

<!-- セッションスコープの例: ウィザード形式 -->
<form-bean name="wizardForm"
           type="com.example.form.WizardForm"/>

<action path="/wizard/step1"
        type="com.example.action.WizardStep1Action"
        name="wizardForm"
        scope="session">
    <forward name="success" path="/wizard_step2.jsp"/>
</action>

使い分けの基準

  • 迷ったらrequestスコープを選ぶ（メモリ節約）
  • ログインフォームはrequestでOK（1回で完結）
  • ユーザー情報はsessionに保存（ログイン中ずっと使う）

9.3 ログイン機能の実装

ログイン機能を実装するには、ログインフォーム（LoginForm）とログイン処理を行うAction（LoginAction）が必要です。 認証成功時にセッションへユーザー情報を保存します。

実習 9-1: ログインフォームとActionの作成

IDとパスワードで認証するログイン機能を実装します。

手順

  1. 1 LoginFormクラスを作成（username, passwordプロパティ）
  2. 2 LoginActionクラスを作成し、認証ロジックを実装
  3. 3 struts-config.xmlに設定を追加
  4. 4 login.jspを作成してフォームを実装
LoginForm.java 
package com.example.form;

import org.apache.struts.action.ActionForm;

public class LoginForm extends ActionForm {
    // ユーザー名
    private String username;
    // パスワード
    private String password;

    // Getter/Setter
    public String getUsername() {
        return username;
    }

    public void setUsername(String username) {
        this.username = username;
    }

    public String getPassword() {
        return password;
    }

    public void setPassword(String password) {
        this.password = password;
    }
}
LoginAction.java 
package com.example.action;

import javax.servlet.http.HttpServletRequest;
import javax.servlet.http.HttpServletResponse;
import javax.servlet.http.HttpSession;
import org.apache.struts.action.*;
import com.example.form.LoginForm;

public class LoginAction extends Action {
    @Override
    public ActionForward execute(
        ActionMapping mapping,
        ActionForm form,
        HttpServletRequest request,
        HttpServletResponse response) throws Exception {

        LoginForm loginForm = (LoginForm) form;
        String username = loginForm.getUsername();
        String password = loginForm.getPassword();

        // 簡易認証（実際はDBで照合）
        if ("admin".equals(username) && "pass123".equals(password)) {
            // セッション取得
            HttpSession session = request.getSession();
            // ユーザー情報をセッションに保存
            session.setAttribute("loginUser", username);
            // ホーム画面へ
            return mapping.findForward("success");
        } else {
            // エラーメッセージ設定
            ActionErrors errors = new ActionErrors();
            errors.add(ActionErrors.GLOBAL_MESSAGE,
                new ActionMessage("error.login.failed"));
            saveErrors(request, errors);
            // ログイン画面に戻る
            return mapping.findForward("failure");
        }
    }
}

セキュリティの注意

上記のコードは学習用の簡易版です。本番環境ではパスワードのハッシュ化（bcrypt等）、 SQLインジェクション対策、HTTPS通信の使用が必須です。

9.4 セッションへのユーザー情報の保存と取得

ログイン成功後、ユーザー情報をセッションに保存することで、以降のリクエストで「誰がログインしているか」を判別できます。

セッションへの保存と取得 
// セッションオブジェクトを取得
HttpSession session = request.getSession();

// ユーザー情報を保存（キーは任意の名前）
session.setAttribute("loginUser", username);
session.setAttribute("userId", userId);
session.setAttribute("userRole", role);

// セッションから取得（別のActionで）
String loginUser = (String) session.getAttribute("loginUser");
if (loginUser != null) {
    // ログイン済み
    System.out.println("ログイン中: " + loginUser);
} else {
    // 未ログイン
    System.out.println("ログインしていません");
}

// セッションから削除（ログアウト時）
session.removeAttribute("loginUser");
// またはセッション全体を破棄
session.invalidate();

より実践的な方法: UserBeanの活用

複数の情報をまとめて管理したい場合は、UserBeanなどのクラスを作成してセッションに保存すると便利です。

UserBean.java 
package com.example.bean;

import java.io.Serializable;

// セッションに保存するためSerializableを実装
public class UserBean implements Serializable {
    private Integer userId;
    private String username;
    private String email;
    private String role;

    // コンストラクタ
    public UserBean(Integer userId, String username,
                    String email, String role) {
        this.userId = userId;
        this.username = username;
        this.email = email;
        this.role = role;
    }

    // Getter/Setter (省略)
    // ...
}
UserBeanをセッションに保存 
// ログイン成功時
UserBean user = new UserBean(1, "admin", "admin@example.com", "ADMIN");
session.setAttribute("loginUser", user);

// 別のActionで取得
UserBean loginUser = (UserBean) session.getAttribute("loginUser");
if (loginUser != null) {
    String role = loginUser.getRole();
    // 権限チェックなどに使用
}

9.5 ログインチェック処理の実装

ログインが必要なページでは、セッションをチェックして未ログインユーザーをログイン画面にリダイレクトする必要があります。 実装方法は主に2つあります。

方法1: BaseActionクラス

全Actionで継承する共通クラスを作成し、executeメソッド実行前にログインチェックを行う。

方法2: Filterクラス

Servlet Filterを作成し、リクエスト到達前に認証チェックを行う（より汎用的）。

方法1: BaseActionでのログインチェック

BaseAction.java 
package com.example.action;

import javax.servlet.http.HttpServletRequest;
import javax.servlet.http.HttpServletResponse;
import javax.servlet.http.HttpSession;
import org.apache.struts.action.*;

public abstract class BaseAction extends Action {
    @Override
    public final ActionForward execute(
        ActionMapping mapping, ActionForm form,
        HttpServletRequest request, HttpServletResponse response)
        throws Exception {

        // ログインチェック
        HttpSession session = request.getSession();
        Object loginUser = session.getAttribute("loginUser");

        if (loginUser == null) {
            // 未ログイン → ログイン画面へ
            return mapping.findForward("login");
        }

        // ログイン済み → 各Actionの処理を実行
        return doExecute(mapping, form, request, response);
    }

    // 継承先で実装するメソッド
    protected abstract ActionForward doExecute(
        ActionMapping mapping, ActionForm form,
        HttpServletRequest request, HttpServletResponse response)
        throws Exception;
}
HomeAction.java（BaseActionを継承） 
package com.example.action;

import javax.servlet.http.HttpServletRequest;
import javax.servlet.http.HttpServletResponse;
import org.apache.struts.action.*;

// BaseActionを継承
public class HomeAction extends BaseAction {
    @Override
    protected ActionForward doExecute(
        ActionMapping mapping, ActionForm form,
        HttpServletRequest request, HttpServletResponse response)
        throws Exception {

        // ログイン済みのユーザーのみこの処理が実行される
        return mapping.findForward("success");
    }
}

方法2: Filterでのログインチェック

LoginCheckFilter.java 
package com.example.filter;

import java.io.IOException;
import javax.servlet.*;
import javax.servlet.http.*;

public class LoginCheckFilter implements Filter {
    @Override
    public void doFilter(ServletRequest req, ServletResponse res,
                         FilterChain chain)
        throws IOException, ServletException {

        HttpServletRequest request = (HttpServletRequest) req;
        HttpServletResponse response = (HttpServletResponse) res;
        HttpSession session = request.getSession();

        String path = request.getServletPath();

        // ログイン画面へのアクセスはチェック不要
        if (path.startsWith("/login")) {
            chain.doFilter(req, res);
            return;
        }

        // セッションチェック
        if (session.getAttribute("loginUser") == null) {
            // 未ログイン → ログイン画面へ
            response.sendRedirect(
                request.getContextPath() + "/login.do");
            return;
        }

        // ログイン済み → 次の処理へ
        chain.doFilter(req, res);
    }

    @Override
    public void init(FilterConfig config) {}

    @Override
    public void destroy() {}
}
web.xml（Filter設定） 
<filter>
    <filter-name>LoginCheckFilter</filter-name>
    <filter-class>com.example.filter.LoginCheckFilter</filter-class>
</filter>
<filter-mapping>
    <filter-name>LoginCheckFilter</filter-name>
    <url-pattern>*.do</url-pattern>
</filter-mapping>

推奨はFilterパターン

FilterはAction以外のリソース（JSP直アクセスなど）も保護できるため、より強固なセキュリティを実現できます。

9.6 ログアウト機能の実装

ログアウト機能は、セッションを破棄してログイン画面に戻す処理です。

LogoutAction.java 
package com.example.action;

import javax.servlet.http.HttpServletRequest;
import javax.servlet.http.HttpServletResponse;
import javax.servlet.http.HttpSession;
import org.apache.struts.action.*;

public class LogoutAction extends Action {
    @Override
    public ActionForward execute(
        ActionMapping mapping, ActionForm form,
        HttpServletRequest request, HttpServletResponse response)
        throws Exception {

        HttpSession session = request.getSession(false);
        if (session != null) {
            // セッション全体を破棄
            session.invalidate();
        }

        // ログイン画面へリダイレクト
        return mapping.findForward("login");
    }
}

動作確認

以下の流れで動作確認を行います。

  1. ログイン画面にアクセスして、正しい認証情報を入力
  2. ログイン成功後、ホーム画面に遷移することを確認
  3. ブラウザのアドレスバーから直接ホーム画面URLにアクセス（セッション継続確認）
  4. ログアウトボタンをクリック
  5. 再度ホーム画面URLにアクセス → ログイン画面にリダイレクトされることを確認

トラブルシューティング

ログイン後もログイン画面に戻されてしまう

原因: セッションへの保存がうまくいっていない可能性があります。

解決方法: LoginActionでsession.setAttribute()が正しく実行されているか確認。 デバッグログでsession.getAttribute("loginUser")の値を出力してチェックしましょう。

セッションがすぐに切れてしまう

原因: セッションタイムアウトの設定が短すぎる可能性があります。

解決方法: web.xmlで<session-timeout>30</session-timeout>（分単位）を設定します。

Filterが動作しない

原因: web.xmlのurl-patternが正しくない、またはFilter定義の順序の問題。

解決方法: url-patternを*.doや/*に修正。 また、Filterは定義順に実行されるため、LoginCheckFilterを適切な位置に配置しましょう。

理解度確認クイズ

  1. Q1. requestスコープとsessionスコープの主な違いは何ですか？

    回答を表示

    requestスコープは1回のリクエスト～レスポンスまでの間だけデータを保持します。 一方、sessionスコープはセッション終了（ログアウトやタイムアウト）までデータを保持します。 メモリ使用量を考えると、通常はrequestスコープを優先し、必要な場合のみsessionを使います。

  2. Q2. セッションにユーザー情報を保存するメソッドと、取得するメソッドの名前を答えてください。

    回答を表示

    保存: session.setAttribute("キー", オブジェクト)
    取得: Object obj = session.getAttribute("キー")
    削除: session.removeAttribute("キー")
    セッション破棄: session.invalidate()

  3. Q3. ログインチェックの実装方法として、BaseActionとFilterのどちらがより推奨されますか？その理由も答えてください。

    回答を表示

    Filterの方が推奨されます。理由は、FilterはAction以外のリソース（JSP直アクセス、静的ファイル等）も 保護できるため、より包括的なセキュリティを実現できるからです。BaseActionはAction経由のリクエストしか チェックできないため、JSPへの直接アクセスを許してしまう可能性があります。

第9章のまとめ

  • HTTPセッションはユーザーごとの状態を保持する仕組みで、セッションIDをCookieで管理する
  • ActionFormのスコープはrequestとsessionがあり、データの保持期間とメモリ使用量が異なる
  • ログイン機能はLoginFormとLoginActionで実装し、認証成功時にセッションへユーザー情報を保存する
  • ログインチェックはBaseActionまたはFilterで実装し、未ログインユーザーをログイン画面にリダイレクトする
  • ログアウトはsession.invalidate()でセッションを破棄して実現する
