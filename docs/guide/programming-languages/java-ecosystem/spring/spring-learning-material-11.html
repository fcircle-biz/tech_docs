エラーハンドリングと例外処理

Springアプリケーションでの適切なエラーハンドリングを学びます。カスタム例外クラスの作成、@ControllerAdviceと@ExceptionHandlerによる共通エラー処理、エラーページのカスタマイズ、ログ出力の設定（SLF4J/Logback）を実装します。

この章で学ぶこと

  • Springアプリケーションにおける例外の種類と処理の流れを理解する
  • カスタム例外クラスを作成し、ビジネスロジックのエラーを表現する
  • @ControllerAdviceと@ExceptionHandlerで共通エラー処理を実装する
  • エラーページをカスタマイズし、ユーザーフレンドリーな画面を提供する
  • SLF4J/Logbackでログ出力を設定し、デバッグと運用を支援する

11.1 エラーハンドリングの重要性

本番環境で稼働するアプリケーションでは、予期しないエラーが発生することを前提とした設計が必要です。適切なエラーハンドリングは、以下の3つの観点で重要です。

ユーザー体験

エラー内容をわかりやすく伝え、次に取るべき行動を示す

デバッグ効率

詳細なログとスタックトレースで問題の特定を迅速化

セキュリティ

内部情報を隠し、攻撃者にヒントを与えない

悪いエラーハンドリングの例

スタックトレースをそのままユーザーに表示する、エラーログを出力しない、すべての例外をcatchして無視する、といった実装は、デバッグを困難にし、セキュリティリスクを高めます。

11.2 Springの例外処理の流れ

Spring MVCでは、コントローラーで発生した例外がDispatcherServletによって処理されます。この仕組みを理解することで、効果的なエラーハンドリングを実装できます。

例外処理の流れ

flowchart TD A([HTTPリクエスト]) --> B[DispatcherServlet] B --> C[Controller] C --> D{例外発生?} D -- なし --> E[正常レスポンス] D -- あり --> F{@ExceptionHandler
定義あり?} F -- あり --> G[@ExceptionHandlerで処理] F -- なし --> H{@ControllerAdvice
定義あり?} H -- あり --> I[共通エラー処理] H -- なし --> J[デフォルトエラーページ] G --> K([エラーレスポンス]) I --> K J --> K

例外処理の優先順位

1. コントローラー内の@ExceptionHandler → 2. @ControllerAdviceの@ExceptionHandler → 3. Springのデフォルトエラーページ の順に処理されます。

11.3 カスタム例外クラスの作成

ビジネスロジック特有のエラーを表現するため、独自の例外クラスを定義します。RuntimeExceptionを継承することで、チェック例外ではなく非チェック例外として扱えます。

exception/ResourceNotFoundException.java 
package com.example.demo.exception;

// リソースが見つからない場合の例外
public class ResourceNotFoundException extends RuntimeException {

    // エラーメッセージのみを受け取るコンストラクタ
    public ResourceNotFoundException(String message) {
        super(message);
    }

    // エラーメッセージと原因例外を受け取るコンストラクタ
    public ResourceNotFoundException(String message, Throwable cause) {
        super(message, cause);
    }
}
exception/ValidationException.java 
package com.example.demo.exception;

// バリデーションエラーの例外
public class ValidationException extends RuntimeException {

    public ValidationException(String message) {
        super(message);
    }
}

カスタム例外の使用例

service/UserService.java 
@Service
public class UserService {
    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public User findById(Long id) {
        // IDでユーザーを検索、見つからなければ例外をスロー
        return userRepository.findById(id)
            .orElseThrow(() -> new ResourceNotFoundException(
                "ユーザーID " + id + " が見つかりません"));
    }
}

11.4 @ExceptionHandlerによる例外処理

コントローラー内で特定の例外をキャッチし、カスタムレスポンスを返すには@ExceptionHandlerアノテーションを使用します。

controller/UserController.java 
@Controller
@RequestMapping("/users")
public class UserController {
    private final UserService userService;

    public UserController(UserService userService) {
        this.userService = userService;
    }

    @GetMapping("/{id}")
    public String showUser(@PathVariable Long id, Model model) {
        User user = userService.findById(id); // 例外が発生する可能性
        model.addAttribute("user", user);
        return "user/detail";
    }

    // このコントローラー内で発生したResourceNotFoundExceptionをキャッチ
    @ExceptionHandler(ResourceNotFoundException.class)
    public String handleNotFound(ResourceNotFoundException ex, Model model) {
        model.addAttribute("errorMessage", ex.getMessage());
        return "error/notfound"; // エラーページを表示
    }
}

重要ポイント

@ExceptionHandlerはそのコントローラー内でのみ有効です。複数のコントローラーで共通のエラー処理を行うには、次のセクションで学ぶ@ControllerAdviceを使用します。

11.5 @ControllerAdviceによる共通エラー処理

@ControllerAdviceアノテーションを使うと、アプリケーション全体で共通のエラー処理を一箇所に集約できます。これにより、各コントローラーでエラー処理を重複して実装する必要がなくなります。

exception/GlobalExceptionHandler.java 
package com.example.demo.exception;

import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.ui.Model;
import org.springframework.web.bind.annotation.ControllerAdvice;
import org.springframework.web.bind.annotation.ExceptionHandler;

// すべてのコントローラーに適用されるエラーハンドラー
@ControllerAdvice
public class GlobalExceptionHandler {

    private static final Logger logger = LoggerFactory.getLogger(GlobalExceptionHandler.class);

    // ResourceNotFoundExceptionの処理
    @ExceptionHandler(ResourceNotFoundException.class)
    public String handleResourceNotFound(ResourceNotFoundException ex, Model model) {
        logger.error("リソースが見つかりません: {}", ex.getMessage());
        model.addAttribute("errorMessage", ex.getMessage());
        return "error/404";
    }

    // ValidationExceptionの処理
    @ExceptionHandler(ValidationException.class)
    public String handleValidation(ValidationException ex, Model model) {
        logger.warn("バリデーションエラー: {}", ex.getMessage());
        model.addAttribute("errorMessage", ex.getMessage());
        return "error/validation";
    }

    // 予期しない例外の処理
    @ExceptionHandler(Exception.class)
    public String handleGeneral(Exception ex, Model model) {
        logger.error("予期しないエラーが発生しました", ex);
        model.addAttribute("errorMessage", "システムエラーが発生しました");
        return "error/500";
    }
}

11.6 エラーページのカスタマイズ

Springのデフォルトエラーページではなく、独自のエラーページを表示することで、ユーザー体験を向上させます。

templates/error/404.html 
<!DOCTYPE html>
<html xmlns:th="http://www.thymeleaf.org">
<head>
    <meta charset="UTF-8">
    <title>ページが見つかりません</title>
</head>
<body>
    <h1>404 - ページが見つかりません</h1>
    <p th:text="${errorMessage}">エラーメッセージ</p>
    <a href="/">トップページに戻る</a>
</body>
</html>

HTTPステータスコードごとのエラーページ

Spring Bootでは、templates/errorフォルダにHTTPステータスコード.htmlというファイルを配置すると、自動的に対応するエラーページが表示されます。

ファイル名 HTTPステータス 用途
404.html 404 Not Found リソースが見つからない
500.html 500 Internal Server Error サーバー内部エラー
403.html 403 Forbidden アクセス拒否

11.7 ログ出力の設定（SLF4J/Logback）

ログは、エラーの原因を特定するための重要な情報源です。Spring Bootは、デフォルトでSLF4J（Simple Logging Facade for Java）とLogbackを使用します。

SLF4JとLogbackの関係

SLF4Jはログ出力のためのAPI（インターフェース）で、Logbackはその実装です。SLF4Jを使うことで、将来的にロギングライブラリを変更する際もコードの変更が不要になります。

基本的なログ出力

service/UserService.java（ログ出力例） 
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.stereotype.Service;

@Service
public class UserService {
    // ロガーの初期化
    private static final Logger logger = LoggerFactory.getLogger(UserService.class);

    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public User findById(Long id) {
        logger.debug("ユーザー検索開始: ID={}", id); // デバッグログ

        User user = userRepository.findById(id)
            .orElseThrow(() -> {
                logger.error("ユーザーが見つかりません: ID={}", id); // エラーログ
                return new ResourceNotFoundException("ユーザーID " + id + " が見つかりません");
            });

        logger.info("ユーザー検索成功: {}", user.getName()); // 情報ログ
        return user;
    }
}

ログレベル

レベル メソッド 用途
ERROR logger.error() エラーが発生した場合
WARN logger.warn() 警告（潜在的な問題）
INFO logger.info() 重要な処理の開始・完了
DEBUG logger.debug() デバッグ情報
TRACE logger.trace() 非常に詳細なトレース情報

application.ymlでのログレベル設定

src/main/resources/application.yml 
logging:
  level:
    # ルートロガー（全体のデフォルト）
    root: INFO

    # 自分のアプリケーションパッケージはDEBUGレベル
    com.example.demo: DEBUG

    # SpringのSQLログを表示
    org.springframework.jdbc.core: DEBUG

  # ログファイルへの出力
  file:
    name: logs/application.log
    max-size: 10MB
    max-history: 30

重要ポイント

本番環境ではINFO以上、開発環境ではDEBUG以上のログを出力するように設定します。デバッグログには機密情報（パスワード、クレジットカード番号など）を含めないよう注意してください。

実習 11-2: ログ出力の実装

サービス層にログ出力を追加し、application.ymlでログレベルを設定します。

手順

  1. 1. UserServiceにLoggerを追加

    LoggerFactory.getLogger()でロガーを初期化します。

  2. 2. 各メソッドにログ出力を追加

    処理開始時にlogger.debug()、エラー時にlogger.error()を呼び出します。

  3. 3. application.ymlでログレベルを設定

    com.example.demoパッケージのログレベルをDEBUGに設定します。

  4. 4. アプリケーションを実行してログを確認

    コンソールにDEBUGログが出力されることを確認します。

動作確認

  1. 1. 存在しないIDでアクセス

    http://localhost:8080/users/999 にアクセスし、カスタム404エラーページが表示されることを確認します。

  2. 2. ログファイルの確認

    logs/application.log にエラーログが出力されていることを確認します。

  3. 3. 正常系のログ確認

    存在するIDでアクセスし、INFO/DEBUGログが出力されることを確認します。

トラブルシューティング

カスタムエラーページが表示されない

templates/errorフォルダの配置を確認してください。ファイル名は404.htmlのようにステータスコードのみで、拡張子は.htmlです。

ログが出力されない

application.ymlのlogging.level設定を確認してください。パッケージ名が正確に一致している必要があります。

@ControllerAdviceが動作しない

@ControllerAdviceクラスがコンポーネントスキャン対象のパッケージに配置されていることを確認してください。通常はメインクラスと同じパッケージか、そのサブパッケージに配置します。

理解度確認クイズ

  1. Q1. @ControllerAdviceと@ExceptionHandlerの違いを説明してください。

    回答を表示

    @ExceptionHandlerは、そのコントローラー内でのみ例外処理を行います。@ControllerAdviceは、アプリケーション全体のコントローラーに適用される共通例外処理を定義します。複数のコントローラーで同じエラーハンドリングが必要な場合は、@ControllerAdviceを使用することでコードの重複を避けられます。

  2. Q2. ログレベルERROR、WARN、INFO、DEBUGのうち、本番環境でデフォルトで出力すべきレベルはどれですか？その理由も答えてください。

    回答を表示

    INFO以上（INFO、WARN、ERROR）が適切です。理由は、DEBUGログは大量に出力されるため、パフォーマンスに影響を与える可能性があること、また本番環境では詳細なデバッグ情報よりも重要な処理の記録とエラーの検出が優先されるためです。問題発生時に必要に応じてDEBUGレベルに切り替えることもできます。

  3. Q3. カスタム例外クラスがRuntimeExceptionを継承する理由を説明してください。

    回答を表示

    RuntimeExceptionを継承すると非チェック例外になり、メソッドシグネチャにthrows宣言を書く必要がなくなります。これにより、Springのトランザクション管理（@Transactional）でデフォルトでロールバック対象になるほか、コードがシンプルになります。ビジネスロジック上の例外は、通常非チェック例外として扱うことが推奨されています。

第11章のまとめ

  • 適切なエラーハンドリングは、ユーザー体験、デバッグ効率、セキュリティの3つの観点で重要です。
  • カスタム例外クラスを作成することで、ビジネスロジック特有のエラーを明示的に表現できます。
  • @ControllerAdviceと@ExceptionHandlerを使うことで、アプリケーション全体のエラー処理を一箇所に集約できます。
  • templates/errorフォルダにHTMLファイルを配置することで、HTTPステータスコードごとのカスタムエラーページを提供できます。
  • SLF4J/Logbackを使ったログ出力により、エラーの原因特定と運用監視が可能になります。ログレベルは環境に応じて適切に設定します。
