第14章

テストの基礎

Spring Bootアプリケーションのテスト方法を学びます。JUnit 5の基礎、@SpringBootTestによる統合テスト、MockMvcを使ったコントローラーテスト、Mockitoによるモック化、サービス層の単体テスト、テスト用データベース（H2）の設定を実装します。

この章で学ぶこと

  • JUnit 5の基本的な使い方とアサーション、アノテーションを理解する
  • @SpringBootTestを使った統合テストの実装方法を学ぶ
  • MockMvcによるコントローラー層のテスト方法をマスターする
  • Mockitoを使ったモックオブジェクトの作成とサービス層の単体テストを実装する
  • テスト用インメモリデータベース（H2）の設定と活用方法を理解する

14.1 なぜテストが重要なのか

テストコードは、アプリケーションの品質を保証し、リファクタリングや機能追加時の安全性を確保する重要な役割を果たします。Spring Bootは、テストを容易にするための強力なサポート機能を提供しています。

テストの種類 目的 対象範囲
単体テスト 個別クラス・メソッドの動作確認 Service、Repository単体
統合テスト 複数コンポーネントの連携確認 Controller + Service + Repository
E2Eテスト システム全体の動作確認 ブラウザ操作を含む全体

重要ポイント

本章では、Spring Bootアプリケーションで最も頻繁に使用される「単体テスト」と「統合テスト」に焦点を当てます。JUnit 5とSpring Boot Testの組み合わせにより、効率的なテスト実装が可能です。

14.2 JUnit 5の基礎

JUnit 5は、Javaの標準的なテストフレームワークです。Spring Bootのプロジェクトには、spring-boot-starter-testに含まれており、追加設定なしで利用できます。

基本的なテストクラスの作成

CalculatorTest.java 
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

// テストクラス（publicは不要）
class CalculatorTest {

    @Test // テストメソッドを示すアノテーション
    void 足し算のテスト() {
        // 準備（Arrange）
        Calculator calculator = new Calculator();

        // 実行（Act）
        int result = calculator.add(2, 3);

        // 検証（Assert）
        assertEquals(5, result);
    }
}

主要なアサーションメソッド

AssertionExamples.java 
@Test
void アサーション例() {
    // 等価性チェック
    assertEquals(expected, actual);

    // 真偽値チェック
    assertTrue(condition);
    assertFalse(condition);

    // null チェック
    assertNull(object);
    assertNotNull(object);

    // 例外チェック
    assertThrows(Exception.class, () -> {
        // 例外が発生するコード
    });
}

テストメソッド名の付け方

JUnit 5では、テストメソッド名に日本語を使用できます。「〇〇のテスト」「〇〇の場合_〇〇を返す」など、わかりやすい名前を付けると、テスト失敗時の原因特定が容易になります。

14.3 @SpringBootTestによる統合テスト

@SpringBootTestは、Spring Bootアプリケーション全体のコンテキストをロードし、実際の環境に近い状態でテストを実行できます。

UserServiceIntegrationTest.java 
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.beans.factory.annotation.Autowired;

@SpringBootTest // Springコンテキストをロード
class UserServiceIntegrationTest {

    @Autowired // 実際のBeanを注入
    private UserService userService;

    @Test
    void ユーザー登録のテスト() {
        User user = new User("test@example.com", "password");
        User saved = userService.register(user);

        assertNotNull(saved.getId());
        assertEquals("test@example.com", saved.getEmail());
    }
}

統合テストの注意点

@SpringBootTestは全てのBeanをロードするため、起動に時間がかかります。テストの実行速度が問題になる場合は、後述のモックを活用した単体テストと組み合わせましょう。

14.4 MockMvcによるコントローラーテスト

MockMvcは、Webサーバーを起動せずにコントローラーのHTTPリクエスト/レスポンスをテストできる機能です。@WebMvcTestアノテーションと組み合わせて使用します。

UserControllerTest.java 
import org.springframework.boot.test.autoconfigure.web.servlet.WebMvcTest;
import org.springframework.test.web.servlet.MockMvc;
import static org.springframework.test.web.servlet.request.MockMvcRequestBuilders.*;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.*;

@WebMvcTest(UserController.class) // コントローラー層のみテスト
class UserControllerTest {

    @Autowired
    private MockMvc mockMvc; // HTTPリクエストをシミュレート

    @Test
    void ユーザー一覧表示のテスト() throws Exception {
        mockMvc.perform(get("/users"))
               .andExpect(status().isOk())
               .andExpect(view().name("users/list"));
    }
}

POSTリクエストのテスト

POST処理のテスト 
@Test
void ユーザー登録のPOSTテスト() throws Exception {
    mockMvc.perform(post("/users/create")
               .param("email", "test@example.com")
               .param("password", "password123"))
           .andExpect(status().is3xxRedirection())
           .andExpect(redirectedUrl("/users"));
}

14.5 Mockitoによるモック化とサービス層テスト

Mockitoは、依存オブジェクトの動作を模擬（モック）し、テスト対象のクラスを独立してテストできるライブラリです。Spring Boot Testに標準で含まれています。

モック化の仕組み

graph LR Test[テストクラス] --> Service[UserService
テスト対象] Service -.モック化.-> MockRepo[UserRepository
モックオブジェクト] style Service fill:#dbeafe,stroke:#3b82f6 style MockRepo fill:#fef3c7,stroke:#f59e0b
UserServiceTest.java 
import org.mockito.InjectMocks;
import org.mockito.Mock;
import org.mockito.junit.jupiter.MockitoExtension;
import static org.mockito.Mockito.*;

@ExtendWith(MockitoExtension.class) // Mockito有効化
class UserServiceTest {

    @Mock // モックオブジェクトを作成
    private UserRepository userRepository;

    @InjectMocks // モックを注入
    private UserService userService;

    @Test
    void ユーザー検索のテスト() {
        // モックの振る舞いを定義
        User mockUser = new User(1L, "test@example.com");
        when(userRepository.findById(1L))
            .thenReturn(Optional.of(mockUser));

        // テスト実行
        User result = userService.findById(1L);

        // 検証
        assertEquals("test@example.com", result.getEmail());
        verify(userRepository).findById(1L); // メソッド呼び出しを確認
    }
}

when()とverify()の使い分け

when(): モックメソッドが呼ばれたときの戻り値を定義
verify(): モックメソッドが実際に呼ばれたかを検証

14.6 テスト用データベース（H2）の設定

H2は、インメモリで動作する軽量データベースで、テスト実行のたびに初期化されるため、テストの独立性を保つのに最適です。

依存関係の追加

pom.xml 
<dependency>
    <groupId>com.h2database</groupId>
    <artifactId>h2</artifactId>
    <scope>test</scope> <!-- テスト時のみ使用 -->
</dependency>

テスト用設定ファイル

src/test/resources/application.yml 
spring:
  datasource:
    url: jdbc:h2:mem:testdb
    driver-class-name: org.h2.Driver
    username: sa
    password:
  sql:
    init:
      mode: always # テスト実行時にSQLを実行

重要ポイント

テスト用設定ファイルは、src/test/resourcesに配置します。本番用（src/main/resources）と分離することで、テスト環境と本番環境を明確に区別できます。

実習: ユーザーサービスのテスト実装

以下のUserServiceクラスに対して、JUnit 5とMockitoを使ったテストを作成してみましょう。

UserService.java（テスト対象） 
@Service
public class UserService {
    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public List<User> findAll() {
        return userRepository.findAll();
    }

    public User findById(Long id) {
        return userRepository.findById(id)
            .orElseThrow(() -> new UserNotFoundException(id));
    }
}
  1. 1. UserServiceTestクラスを作成し、@ExtendWith(MockitoExtension.class)を付与
  2. 2. UserRepositoryをモック化し、UserServiceに注入
  3. 3. findAll()のテストケースを実装（when()で空リストを返すように設定）
  4. 4. findById()のテストケースを実装（存在する場合と存在しない場合の2パターン）

14.7 テストのベストプラクティス

1. テストは独立させる

各テストは他のテストに依存せず、どの順序で実行しても同じ結果になるように設計します。

2. 高速なテストを心がける

可能な限りモックを活用し、データベースアクセスを最小限に抑えることで、テストの実行速度を改善します。

3. わかりやすいテスト名

テストメソッド名から、何をテストしているのかが一目でわかるように命名します。

4. AAA パターン

Arrange（準備）、Act（実行）、Assert（検証）の3段階でテストを構成し、可読性を高めます。

理解度確認クイズ

  1. Q1. Spring Bootアプリケーション全体のコンテキストをロードして統合テストを実行するために使用するアノテーションは何ですか？

    回答を表示

    @SpringBootTest
    このアノテーションは、実際のアプリケーションコンテキストを起動し、全てのBeanを利用可能にします。実環境に近い状態でテストできる反面、起動に時間がかかるため、必要な場合にのみ使用します。

  2. Q2. コントローラー層のHTTPリクエスト/レスポンスをWebサーバーなしでテストするために使用するクラスは何ですか？

    回答を表示

    MockMvc
    MockMvcは、@WebMvcTestと組み合わせて使用し、perform()メソッドでHTTPリクエストをシミュレートします。andExpect()でステータスコード、ビュー名、レスポンス内容を検証できます。

  3. Q3. Mockitoで、モックオブジェクトのメソッドが呼ばれたときの戻り値を定義するメソッドと、実際に呼ばれたかを検証するメソッドの名前をそれぞれ答えてください。

    回答を表示

    戻り値の定義: when()
    呼び出しの検証: verify()
    when()でモックの振る舞いを定義し、verify()でメソッドが期待通りに呼ばれたかを確認します。この2つを組み合わせることで、依存オブジェクトに依存しない単体テストが実現できます。

第14章のまとめ

  • JUnit 5は、@Testアノテーションとアサーションメソッドで簡潔にテストを記述できます。
  • @SpringBootTestは統合テストに、@WebMvcTestとMockMvcはコントローラーテストに使用します。
  • Mockitoの@Mockと@InjectMocksにより、依存オブジェクトをモック化してサービス層を単体テストできます。
  • H2データベースをテスト用に設定することで、テストの独立性と高速化を実現できます。
  • 独立性、高速性、わかりやすい命名、AAAパターンがテストのベストプラクティスです。
