第10章

サービス層の設計

適切なレイヤードアーキテクチャの設計方法を学びます。Controller、Service、Repositoryの責務分離、@Serviceアノテーションの役割、ビジネスロジックのサービス層への集約、例外処理の設計、DTOパターンによるレイヤー間のデータ受け渡しを実装します。

この章で学ぶこと

  • レイヤードアーキテクチャにおけるController、Service、Repositoryの責務と役割を理解する
  • @Serviceアノテーションの役割とサービス層の実装方法を習得する
  • ビジネスロジックをサービス層に集約し、Controllerをシンプルに保つ設計を実践する
  • DTOパターンを用いたレイヤー間のデータ受け渡しとカスタム例外の設計ができる

10.1 レイヤードアーキテクチャの概念

レイヤードアーキテクチャ（階層化アーキテクチャ）とは、アプリケーションを複数の層（Layer）に分割し、それぞれの層が明確な責務を持つ設計パターンです。Spring MVCでは、一般的にPresentation層（Controller）、Business層（Service）、Data Access層（Repository）の3層構造を採用します。

レイヤー分離のメリット

各層が独立しているため、変更の影響範囲を限定でき、テストも各層ごとに実施できます。例えば、データベースの変更はRepository層のみに影響し、Controller層やService層は変更不要です。

各層の責務

Controller層

  • • HTTPリクエスト受付
  • • パラメータ検証
  • • Serviceメソッド呼び出し
  • • ビュー選択とレスポンス

Service層

  • • ビジネスロジック実装
  • • トランザクション制御
  • • 複数Repositoryの統合
  • • 例外処理

Repository層

  • • CRUD操作
  • • カスタムクエリ実行
  • • Entityマッピング
  • • DB例外の変換

10.2 @Serviceアノテーションとサービス層の実装

@Serviceアノテーションは、@Componentの特殊化版で、ビジネスロジックを担当するクラスであることを示します。Spring Bootはこのアノテーションが付いたクラスを自動的にDIコンテナに登録し、他のクラスから@Autowiredで注入できるようにします。

UserService.java（サービス層の基本構造） 
package com.example.demo.service;

import com.example.demo.entity.User;
import com.example.demo.repository.UserRepository;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

import java.util.List;

@Service // ビジネスロジック層であることを示す
public class UserService {

    private final UserRepository userRepository;

    // コンストラクタインジェクション
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    // ユーザー一覧取得
    public List<User> findAll() {
        return (List<User>) userRepository.findAll();
    }

    // ユーザー詳細取得
    public User findById(Long id) {
        return userRepository.findById(id)
            .orElseThrow(() -> new UserNotFoundException("ユーザーが見つかりません: ID=" + id));
    }

    // ユーザー登録（トランザクション管理）
    @Transactional
    public User createUser(User user) {
        // ビジネスルール: メールアドレス重複チェック
        if (userRepository.existsByEmail(user.getEmail())) {
            throw new DuplicateEmailException("メールアドレスが既に登録されています: " + user.getEmail());
        }
        return userRepository.save(user);
    }
}

重要ポイント

@Serviceは@Componentと機能的には同じですが、「このクラスはビジネスロジックを担当する」という意図を明示するために使用します。可読性と保守性の向上につながります。

10.3 ビジネスロジックのサービス層への集約

ビジネスロジック（業務ルール、計算処理、データ整合性チェック等）はController層ではなく、Service層に集約します。これにより、Controllerはリクエスト処理とレスポンス生成に専念でき、ビジネスロジックの再利用とテストが容易になります。

悪い例：Controllerにビジネスロジックを記述

UserController.java（アンチパターン） 
@Controller
public class UserController {

    @Autowired
    private UserRepository userRepository;

    @PostMapping("/users")
    public String createUser(@ModelAttribute User user, Model model) {
        // ❌ Controllerにビジネスロジックを記述（アンチパターン）
        if (userRepository.existsByEmail(user.getEmail())) {
            model.addAttribute("error", "メールアドレスが重複しています");
            return "user-form";
        }

        // パスワードのハッシュ化もControllerで実施（良くない）
        String hashedPassword = BCrypt.hashpw(user.getPassword(), BCrypt.gensalt());
        user.setPassword(hashedPassword);

        userRepository.save(user);
        return "redirect:/users";
    }
}

なぜ問題なのか

1. Controllerが肥大化し、可読性が低下します。
2. 同じビジネスロジックを他のControllerで使いたい場合、コードの重複が発生します。
3. ビジネスロジックのテストにHTTPリクエストが必要になり、単体テストが困難です。

良い例：Service層にビジネスロジックを集約

UserService.java（ベストプラクティス） 
@Service
public class UserService {

    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder;

    public UserService(UserRepository userRepository, PasswordEncoder passwordEncoder) {
        this.userRepository = userRepository;
        this.passwordEncoder = passwordEncoder;
    }

    @Transactional
    public User createUser(User user) {
        // ビジネスルール1: メールアドレス重複チェック
        if (userRepository.existsByEmail(user.getEmail())) {
            throw new DuplicateEmailException("メールアドレスが既に使用されています");
        }

        // ビジネスルール2: パスワードのハッシュ化
        String hashedPassword = passwordEncoder.encode(user.getPassword());
        user.setPassword(hashedPassword);

        return userRepository.save(user);
    }
}
UserController.java（シンプルなController） 
@Controller
public class UserController {

    private final UserService userService;

    public UserController(UserService userService) {
        this.userService = userService;
    }

    @PostMapping("/users")
    public String createUser(@ModelAttribute User user, Model model) {
        // ✅ ビジネスロジックはServiceに委譲
        try {
            userService.createUser(user);
            return "redirect:/users";
        } catch (DuplicateEmailException e) {
            model.addAttribute("error", e.getMessage());
            return "user-form";
        }
    }
}

10.4 DTOパターンによるレイヤー間のデータ受け渡し

DTO（Data Transfer Object）は、レイヤー間でデータをやり取りするための専用オブジェクトです。Entityクラスをそのままビュー層に渡すのではなく、DTOに変換することで、データベースのスキーマ変更の影響を局所化し、必要な情報だけを公開できます。

Entity vs DTO の使い分け

graph LR Controller[Controller] Service[Service] Repository[Repository] DB[(Database)] Controller -->|UserDTO| Service Service -->|UserDTO| Controller Service <-->|User Entity| Repository Repository <-->|User Entity| DB style Controller fill:#dbeafe style Service fill:#d1fae5 style Repository fill:#fae8ff style DB fill:#f1f5f9
UserDTO.java（DTO定義） 
package com.example.demo.dto;

public class UserDTO {
    private Long id;
    private String name;
    private String email;
    // パスワードは含めない（セキュリティ）

    // コンストラクタ
    public UserDTO(Long id, String name, String email) {
        this.id = id;
        this.name = name;
        this.email = email;
    }

    // Getter/Setter
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    public String getEmail() { return email; }
    public void setEmail(String email) { this.email = email; }
}
UserService.java（Entity→DTO変換） 
@Service
public class UserService {

    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public List<UserDTO> findAll() {
        List<User> users = (List<User>) userRepository.findAll();
        // Entity → DTO変換
        return users.stream()
            .map(this::convertToDTO)
            .collect(Collectors.toList());
    }

    private UserDTO convertToDTO(User user) {
        return new UserDTO(user.getId(), user.getName(), user.getEmail());
    }
}

DTOのメリット

1. セキュリティ: パスワードなど機密情報をビュー層に渡さない
2. 変更の局所化: Entityのフィールド変更がControllerに影響しない
3. 最適化: 必要な情報だけを含めることでネットワーク転送量を削減

10.5 カスタム例外の設計

ビジネスロジックでエラーが発生した場合、独自の例外クラスを定義することで、エラーの種類を明確にし、Controller層で適切なエラーハンドリングができるようになります。

DuplicateEmailException.java（カスタム例外） 
package com.example.demo.exception;

// RuntimeExceptionを継承（非チェック例外）
public class DuplicateEmailException extends RuntimeException {

    public DuplicateEmailException(String message) {
        super(message);
    }
}
UserNotFoundException.java（カスタム例外） 
package com.example.demo.exception;

public class UserNotFoundException extends RuntimeException {

    public UserNotFoundException(String message) {
        super(message);
    }
}
UserService.java（例外のスロー） 
@Service
public class UserService {

    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public UserDTO findById(Long id) {
        User user = userRepository.findById(id)
            .orElseThrow(() -> new UserNotFoundException("ユーザーが見つかりません: ID=" + id));
        return convertToDTO(user);
    }

    @Transactional
    public UserDTO createUser(User user) {
        if (userRepository.existsByEmail(user.getEmail())) {
            throw new DuplicateEmailException("メールアドレスが既に使用されています");
        }
        User savedUser = userRepository.save(user);
        return convertToDTO(savedUser);
    }
}

ハンズオン: 商品管理システムのサービス層設計

商品管理システムのサービス層を実装してみましょう。

要件:

  • • ProductServiceクラスを作成し、@Serviceアノテーションを付与
  • • ProductRepositoryをコンストラクタインジェクションで注入
  • • findAll()メソッドで全商品をProductDTOのリストとして取得
  • • createProduct()メソッドで商品登録時に価格が0円以下の場合はInvalidPriceException例外をスロー
  • • ProductDTO、InvalidPriceExceptionクラスも作成
解答例を表示
ProductDTO.java 
package com.example.demo.dto;

public class ProductDTO {
    private Long id;
    private String name;
    private Integer price;

    public ProductDTO(Long id, String name, Integer price) {
        this.id = id;
        this.name = name;
        this.price = price;
    }

    // Getter/Setter
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    public Integer getPrice() { return price; }
    public void setPrice(Integer price) { this.price = price; }
}
InvalidPriceException.java 
package com.example.demo.exception;

public class InvalidPriceException extends RuntimeException {
    public InvalidPriceException(String message) {
        super(message);
    }
}
ProductService.java 
package com.example.demo.service;

import com.example.demo.dto.ProductDTO;
import com.example.demo.entity.Product;
import com.example.demo.exception.InvalidPriceException;
import com.example.demo.repository.ProductRepository;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

import java.util.List;
import java.util.stream.Collectors;

@Service
public class ProductService {

    private final ProductRepository productRepository;

    public ProductService(ProductRepository productRepository) {
        this.productRepository = productRepository;
    }

    public List<ProductDTO> findAll() {
        List<Product> products = (List<Product>) productRepository.findAll();
        return products.stream()
            .map(this::convertToDTO)
            .collect(Collectors.toList());
    }

    @Transactional
    public ProductDTO createProduct(Product product) {
        // ビジネスルール: 価格は1円以上
        if (product.getPrice() <= 0) {
            throw new InvalidPriceException("価格は1円以上で設定してください");
        }
        Product savedProduct = productRepository.save(product);
        return convertToDTO(savedProduct);
    }

    private ProductDTO convertToDTO(Product product) {
        return new ProductDTO(product.getId(), product.getName(), product.getPrice());
    }
}

理解度確認クイズ

  1. Q1. レイヤードアーキテクチャにおける各層の責務として正しくないものはどれですか？

    回答を表示

    正解: 「Controller層がSQLクエリを直接実行する」
    SQLクエリの実行はRepository層の責務です。Controller層はHTTPリクエスト受付とレスポンス生成に専念し、データアクセスはService経由でRepositoryに委譲します。

  2. Q2. DTOパターンを使う主な理由を3つ挙げてください。

    回答を表示

    1. セキュリティ: パスワードなど機密情報をビュー層に渡さない
    2. 変更の局所化: Entityのフィールド変更がControllerに影響しない
    3. 最適化: 必要な情報だけを含めることでデータ転送量を削減

  3. Q3. カスタム例外クラスを作成する際、RuntimeExceptionとExceptionどちらを継承すべきですか？その理由も説明してください。

    回答を表示

    正解: RuntimeExceptionを継承
    理由: RuntimeException（非チェック例外）を継承することで、メソッドシグネチャにthrows宣言が不要になり、コードがシンプルになります。ビジネスロジックの例外は予期しないエラーではなく、通常の処理フローの一部として扱われるため、非チェック例外が適しています。

第10章のまとめ

  • レイヤードアーキテクチャでは、Controller（プレゼンテーション）、Service（ビジネス）、Repository（データアクセス）の3層に責務を分離します。
  • @Serviceアノテーションを使い、ビジネスロジックをService層に集約することで、Controllerはシンプルになり、ロジックの再利用とテストが容易になります。
  • DTOパターンを用いることで、Entityをそのまま公開せず、必要な情報だけをビュー層に渡し、セキュリティと保守性を高めます。
  • カスタム例外クラス（RuntimeException継承）を定義することで、ビジネスルール違反を明確に表現し、適切なエラーハンドリングができます。
