デプロイと運用準備

アプリケーションの本番環境へのデプロイ準備を学びます。実行可能JARの作成、環境変数による設定の外部化、ログ設定の本番対応、Actuatorによるヘルスチェック、本番環境でのセキュリティ考慮事項を理解します。

この章で学ぶこと

  • Spring Bootアプリケーションを実行可能JARとしてビルドする方法
  • 環境変数と外部設定ファイルによる本番環境設定の管理方法
  • 本番環境のログ設定とログレベルの適切な管理方法
  • Spring Boot Actuatorを使ったヘルスチェックとメトリクス監視の実装
  • 本番環境で考慮すべきセキュリティ設定とベストプラクティス

18.1 実行可能JARの作成

Spring Bootアプリケーションは、Tomcatなどのアプリケーションサーバーを別途インストールすることなく、単一のJARファイルとして実行できます。これにより、デプロイが簡素化され、環境間での移植性が向上します。

JARファイルのビルド

Mavenを使ってアプリケーションをビルドし、実行可能JARファイルを作成します。

コマンドライン 
# Maven を使ったビルド
mvn clean package

# テストをスキップしてビルド（本番推奨外）
mvn clean package -DskipTests

ビルドが成功すると、target/ディレクトリにアプリ名-0.0.1-SNAPSHOT.jarというファイルが生成されます。

JARファイルの実行

アプリケーション実行 
# JARファイルの実行
java -jar target/taskmanager-0.0.1-SNAPSHOT.jar

# プロファイルを指定して実行
java -jar -Dspring.profiles.active=prod target/taskmanager-0.0.1-SNAPSHOT.jar

実行可能JARの仕組み

Spring Bootの実行可能JARは、アプリケーションクラス、依存ライブラリ、組込みTomcatを全て含んだ「Fat JAR」形式です。MANIFEST.MFファイルで起動クラスが指定されており、java -jarで直接実行できます。

18.2 環境変数と外部設定

本番環境では、データベースパスワードやAPIキーなどの機密情報をソースコードに含めず、環境変数や外部設定ファイルで管理することが重要です。

環境変数による設定の上書き

Spring Bootは、環境変数を自動的に認識し、application.ymlの設定を上書きします。

環境変数設定例（Linux/macOS） 
# 環境変数の設定
export SPRING_DATASOURCE_URL=jdbc:postgresql://prod-db:5432/taskdb
export SPRING_DATASOURCE_USERNAME=prod_user
export SPRING_DATASOURCE_PASSWORD=SecureP@ssw0rd

# アプリケーション実行
java -jar taskmanager.jar

環境変数名のルール

Spring Bootの設定キー（spring.datasource.url）は、環境変数では大文字＋アンダースコア形式（SPRING_DATASOURCE_URL）に変換されます。ドット（.）はアンダースコア（_）に置き換えられます。

外部設定ファイルの読み込み

JARファイルと同じディレクトリにapplication-prod.ymlを配置すると、それが優先的に読み込まれます。

application-prod.yml（本番用設定） 
spring:
  datasource:
    url: jdbc:postgresql://prod-db:5432/taskdb
    username: ${DB_USERNAME}  # 環境変数から読み込み
    password: ${DB_PASSWORD}  # 環境変数から読み込み
  sql:
    init:
      mode: never  # 本番環境では自動初期化を無効化

logging:
  level:
    root: WARN
    com.example: INFO

設定の優先順位

最高優先度] --> B[コマンドライン引数
-Dspring.xxx] B --> C[外部設定ファイル
./application-prod.yml] C --> D[JAR内設定ファイル

18.3 ログ設定の本番対応

本番環境では、適切なログレベル設定、ログローテーション、ログ出力先の管理が重要です。Spring BootではLogbackがデフォルトのログフレームワークとして使用されます。

ログレベルの設定

application-prod.yml（ログ設定） 
logging:
  level:
    root: WARN  # デフォルトはWARNレベル
    com.example.taskmanager: INFO  # 自アプリはINFO
    org.springframework.web: WARN
    org.springframework.security: INFO
    org.hibernate.SQL: ERROR  # SQLログは本番では無効化
  file:
    name: /var/log/taskmanager/application.log  # ログファイルパス
    max-size: 10MB  # ファイルサイズ上限
    max-history: 30  # 保持日数
  pattern:
    console: "%d{yyyy-MM-dd HH:mm:ss} - %msg%n"
    file: "%d{yyyy-MM-dd HH:mm:ss} [%thread] %-5level %logger{36} - %msg%n"

ログレベルの使い分け

ERROR: エラー発生時のみ記録
WARN: 潜在的な問題を記録
INFO: 重要なイベント（起動、停止、主要処理）
DEBUG: 詳細なデバッグ情報（開発環境のみ）
TRACE: 最も詳細な情報（通常使用しない）

カスタムLogback設定

より詳細な制御が必要な場合は、logback-spring.xmlを使用します。

src/main/resources/logback-spring.xml 
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
    <!-- コンソール出力 -->
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>%d{yyyy-MM-dd HH:mm:ss} [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>

    <!-- ファイル出力（ローテーション付き） -->
    <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>/var/log/taskmanager/app.log</file>
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>/var/log/taskmanager/app.%d{yyyy-MM-dd}.log</fileNamePattern>
            <maxHistory>30</maxHistory>
        </rollingPolicy>
        <encoder>
            <pattern>%d{yyyy-MM-dd HH:mm:ss} [%thread] %-5level - %msg%n</pattern>
        </encoder>
    </appender>

    <!-- ルートロガー -->
    <root level="INFO">
        <appender-ref ref="CONSOLE" />
        <appender-ref ref="FILE" />
    </root>
</configuration>

18.4 Spring Boot Actuatorによる監視

Spring Boot Actuatorは、アプリケーションの稼働状態を監視するためのエンドポイントを提供します。ヘルスチェック、メトリクス、環境情報などを取得できます。

Actuatorの依存関係追加

pom.xml 
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>

Actuatorエンドポイントの設定

application-prod.yml 
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,prometheus  # 公開エンドポイント
      base-path: /actuator  # ベースパス
  endpoint:
    health:
      show-details: when-authorized  # 認証ユーザーのみ詳細表示
  health:
    db:
      enabled: true  # データベースヘルスチェック
    diskspace:
      enabled: true  # ディスク容量チェック

主要なActuatorエンドポイント

エンドポイント 説明 用途
/actuator/health アプリケーションの稼働状態 ロードバランサーのヘルスチェック
/actuator/info アプリケーション情報 バージョン情報の確認
/actuator/metrics メトリクス情報 CPU、メモリ、リクエスト数
/actuator/prometheus Prometheus形式のメトリクス 監視システムとの連携

動作確認

アプリケーション起動後、以下のURLにアクセスしてヘルスチェックを確認します。

http://localhost:8080/actuator/health

期待されるレスポンス:
{
  "status": "UP",
  "components": {
    "db": { "status": "UP" },
    "diskSpace": { "status": "UP" }
  }
}

セキュリティ上の注意

本番環境では、Actuatorエンドポイントを公開する際は必ずSpring Securityで保護してください。機密情報が含まれる可能性があるため、未認証アクセスを許可しないよう設定します。

18.5 本番環境のセキュリティ考慮事項

本番環境では、開発環境とは異なるセキュリティ設定が必要です。以下のチェックリストを確認し、適切に設定しましょう。

セキュリティチェックリスト

  • HTTPSを有効化し、全通信を暗号化する
  • 機密情報（パスワード、APIキー）を環境変数で管理する
  • CSRFトークン検証を有効にする（Spring Securityデフォルト）
  • CORS設定を適切に制限する
  • エラーメッセージで内部情報を露出しない
  • デバッグモードを無効化する（spring.devtools無効）

HTTPS設定（SSL/TLS）

application-prod.yml（HTTPS設定） 
server:
  port: 8443  # HTTPSポート
  ssl:
    enabled: true
    key-store: classpath:keystore.p12  # キーストアファイル
    key-store-password: ${KEYSTORE_PASSWORD}  # 環境変数から読み込み
    key-store-type: PKCS12
    key-alias: taskmanager

エラーページのカスタマイズ

本番環境では、エラー発生時にスタックトレースなどの内部情報を表示しないようにします。

application-prod.yml 
server:
  error:
    include-message: never  # エラーメッセージを非表示
    include-stacktrace: never  # スタックトレースを非表示
    include-exception: false  # 例外クラス名を非表示

重要ポイント

本番環境では、セキュリティとパフォーマンスのバランスを考慮した設定が重要です。定期的なセキュリティアップデート、ログ監視、バックアップ体制の整備も忘れずに実施しましょう。

実習 18-1: 本番環境用デプロイパッケージの作成

第17章で作成したタスク管理アプリケーションを本番環境用にビルドし、実行してみましょう。

手順

  1. 1 application-prod.ymlを作成し、本番環境用の設定を記述する
  2. 2 mvn clean packageコマンドでJARファイルをビルドする
  3. 3 環境変数を設定し、java -jar -Dspring.profiles.active=prodでアプリケーションを起動する
  4. 4 /actuator/healthエンドポイントにアクセスし、正常に起動していることを確認する

理解度確認クイズ

  1. Q1. Spring Bootアプリケーションを実行可能JARとしてビルドするMavenコマンドは何ですか？

    回答を表示

    mvn clean package
    このコマンドにより、target/ディレクトリに実行可能JARファイルが生成されます。JARファイルには、アプリケーションクラス、依存ライブラリ、組込みTomcatが全て含まれています。

  2. Q2. application.ymlの設定キー「spring.datasource.password」を環境変数で上書きする場合、環境変数名はどうなりますか？

    回答を表示

    SPRING_DATASOURCE_PASSWORD
    Spring Bootでは、設定キーのドット（.）をアンダースコア（_）に、小文字を大文字に変換した環境変数名で設定を上書きできます。

  3. Q3. Spring Boot Actuatorの/actuator/healthエンドポイントの主な用途は何ですか？

    回答を表示

    アプリケーションの稼働状態を監視するためのヘルスチェック
    ロードバランサーや監視システムがこのエンドポイントを定期的に呼び出し、アプリケーションが正常に動作しているかを確認します。データベース接続やディスク容量などのコンポーネント単位の状態も取得できます。

第18章のまとめ

  • Spring Bootアプリケーションは、mvn clean packageで実行可能JARとしてビルドでき、java -jarで起動できます。
  • 環境変数や外部設定ファイル（application-prod.yml）により、環境ごとの設定を管理します。
  • 本番環境では適切なログレベル（WARN/INFO）を設定し、ログローテーションを有効化します。
  • Spring Boot Actuatorにより、ヘルスチェック、メトリクス監視、運用情報の取得が可能になります。
  • 本番環境ではHTTPS、CSRF保護、エラーメッセージの非表示など、セキュリティ設定を強化します。
