第9章

Spring Data JDBCの実践

Spring Data JDBCをより実践的に活用する方法を学びます。カスタムクエリ（@Query）の作成、ページングとソート機能、1対多リレーションの実装（集約ルート）、トランザクション管理（@Transactional）、従来のJDBCTemplateとの使い分けを理解します。

この章で学ぶこと

  • @Queryアノテーションを使ったカスタムクエリの作成方法
  • ページング（Pageable）とソート（Sort）機能の実装
  • 1対多リレーションの実装と集約ルート（Aggregate Root）の概念
  • @Transactionalによるトランザクション管理の実装
  • Spring Data JDBCとJDBCTemplateの使い分け方

9.1 カスタムクエリ（@Query）

Spring Data JDBCは、メソッド名から自動的にクエリを生成しますが、複雑な検索条件や最適化されたSQLを使いたい場合は、@Queryアノテーションでカスタムクエリを定義できます。

@Queryアノテーションとは

リポジトリインターフェースのメソッドに直接SQLクエリを記述できるアノテーションです。複雑なJOINや集計処理、パフォーマンスが重要な箇所で活用します。

基本的な@Queryの使い方

UserRepository.java（カスタムクエリ例） 
// リポジトリインターフェース
public interface UserRepository extends CrudRepository<User, Long> {

    // メソッド名ベース（自動生成）
    List<User> findByAge(int age);

    // カスタムクエリ（@Query使用）
    @Query("SELECT * FROM users WHERE age >= :minAge AND age <= :maxAge")
    List<User> findByAgeRange(@Param("minAge") int min, @Param("maxAge") int max);

    // 更新系クエリ
    @Modifying
    @Query("UPDATE users SET status = :status WHERE age < :age")
    int updateStatusByAge(@Param("status") String status, @Param("age") int age);
}

重要ポイント

更新・削除クエリには@Modifyingアノテーションも必要です。また、Named Parameter（:paramName）を使うことで、可読性とメンテナンス性が向上します。

JOIN を使った複雑なクエリ

OrderRepository.java（JOIN例） 
// 注文情報にユーザー名を含めたクエリ
@Query("SELECT o.*, u.name as user_name " +
       "FROM orders o " +
       "INNER JOIN users u ON o.user_id = u.id " +
       "WHERE o.status = :status " +
       "ORDER BY o.created_at DESC")
List<OrderWithUserName> findOrdersWithUserName(@Param("status") String status);

9.2 ページングとソート機能

大量のデータを扱う場合、すべてのレコードを一度に取得するのは非効率です。Spring Data JDBCは、PageableとSortインターフェースでページング・ソート機能を提供します。

PagingAndSortingRepository の使用

ProductRepository.java（ページング対応） 
// PagingAndSortingRepositoryを継承
public interface ProductRepository
    extends PagingAndSortingRepository<Product, Long>, CrudRepository<Product, Long> {

    // ページングとソートに対応したメソッド
    Page<Product> findByCategory(String category, Pageable pageable);

    // ソートのみ
    List<Product> findByPriceGreaterThan(int price, Sort sort);
}

サービス層での使用例

ProductService.java（ページング利用） 
@Service
public class ProductService {
    private final ProductRepository productRepository;

    public ProductService(ProductRepository productRepository) {
        this.productRepository = productRepository;
    }

    // 1ページ目、1ページ20件、価格降順でソート
    public Page<Product> getProductList(int page) {
        Pageable pageable = PageRequest.of(page, 20, Sort.by("price").descending());
        return productRepository.findAll(pageable);
    }
}
インターフェース 戻り値 説明
Pageable Page<T> ページ情報（総件数、総ページ数）付き
Sort List<T> ソートのみ（ページ情報なし）

9.3 1対多リレーションと集約ルート

Spring Data JDBCでは、1対多のリレーションを集約ルート（Aggregate Root）という概念で管理します。これは、親エンティティが子エンティティのライフサイクルを管理する設計パターンです。

注文（Order）と注文明細（OrderItem）の関係

erDiagram ORDER ||--o{ ORDER_ITEM : contains ORDER { bigint id PK bigint user_id timestamp created_at } ORDER_ITEM { bigint id PK bigint order_id FK bigint product_id int quantity int price }

エンティティクラスの実装

Order.java（集約ルート） 
@Table("orders")
public class Order {
    @Id
    private Long id;

    private Long userId;
    private LocalDateTime createdAt;

    // 1対多リレーション（OrderItemのSetを保持）
    @MappedCollection(idColumn = "order_id")
    private Set<OrderItem> items = new HashSet<>();

    // ゲッター・セッター省略
}
OrderItem.java（子エンティティ） 
@Table("order_items")
public class OrderItem {
    @Id
    private Long id;

    private Long productId;
    private Integer quantity;
    private Integer price;

    // ゲッター・セッター省略
}

集約ルートの重要ポイント

集約ルート（Order）を保存すると、関連する子エンティティ（OrderItem）も自動的に保存・更新・削除されます。これにより、一貫性のあるデータ操作が可能になります。

9.4 トランザクション管理（@Transactional）

複数のデータベース操作をまとめて実行し、すべて成功するか、すべて失敗するかを保証する仕組みがトランザクションです。Spring Data JDBCでは、@Transactionalアノテーションで簡単に実装できます。

トランザクションの概念

sequenceDiagram participant C as Controller participant S as Service participant R1 as OrderRepository participant R2 as StockRepository participant DB as Database C->>S: createOrder() activate S Note over S: @Transactional開始 S->>R1: save(order) R1->>DB: INSERT orders S->>R2: decreaseStock() R2->>DB: UPDATE stocks alt 成功 Note over S: COMMIT S-->>C: 注文完了 else エラー発生 Note over S: ROLLBACK S-->>C: エラー end deactivate S

@Transactional の使用例

OrderService.java（トランザクション管理） 
@Service
public class OrderService {
    private final OrderRepository orderRepository;
    private final StockRepository stockRepository;

    // コンストラクタ省略

    @Transactional // トランザクション境界
    public Order createOrder(OrderRequest request) {
        // 1. 在庫チェック
        Stock stock = stockRepository.findByProductId(request.getProductId())
            .orElseThrow(() -> new RuntimeException("商品が見つかりません"));

        if (stock.getQuantity() < request.getQuantity()) {
            throw new RuntimeException("在庫不足");
        }

        // 2. 注文作成
        Order order = new Order();
        order.setUserId(request.getUserId());
        order = orderRepository.save(order);

        // 3. 在庫減少（エラーが起きると自動的にロールバック）
        stock.setQuantity(stock.getQuantity() - request.getQuantity());
        stockRepository.save(stock);

        return order;
    }
}

注意事項

@Transactionalはpublicメソッドにのみ有効です。また、同じクラス内のメソッド呼び出しでは動作しないため、サービス層に配置するのが一般的です。

トランザクション属性の設定

トランザクション属性の例 
// 読み取り専用トランザクション（SELECT のみ）
@Transactional(readOnly = true)
public List<Order> findAll() {
    return orderRepository.findAll();
}

// タイムアウト設定（5秒）
@Transactional(timeout = 5)
public void longOperation() {
    // 処理
}

9.5 Spring Data JDBC と JDBCTemplate の使い分け

Spring Data JDBCは便利ですが、すべてのケースに適しているわけではありません。JDBCTemplateを併用することで、柔軟性の高い実装が可能になります。

ケース 推奨技術 理由
基本的なCRUD操作 Spring Data JDBC コード量が少なく、可読性が高い
複雑な集計クエリ JDBCTemplate SQLを直接制御でき、最適化しやすい
バッチ処理 JDBCTemplate batchUpdateでパフォーマンス向上
動的SQL生成 JDBCTemplate 条件分岐でSQLを組み立てられる
リレーション管理 Spring Data JDBC 集約ルートで一貫性を保ちやすい

JDBCTemplate の併用例

OrderRepository.java（カスタム実装） 
@Repository
public class OrderRepositoryCustom {
    private final JdbcTemplate jdbcTemplate;

    public OrderRepositoryCustom(JdbcTemplate jdbcTemplate) {
        this.jdbcTemplate = jdbcTemplate;
    }

    // 複雑な集計クエリ
    public List<OrderSummary> findMonthlySummary(int year, int month) {
        String sql = "SELECT DATE(created_at) as date, COUNT(*) as count, " +
                     "SUM(total_price) as total " +
                     "FROM orders " +
                     "WHERE YEAR(created_at) = ? AND MONTH(created_at) = ? " +
                     "GROUP BY DATE(created_at)";

        return jdbcTemplate.query(sql,
            new Object[]{year, month},
            (rs, rowNum) -> new OrderSummary(
                rs.getDate("date"),
                rs.getInt("count"),
                rs.getBigDecimal("total")
            )
        );
    }
}

理解度確認クイズ

  1. Q1. @Queryアノテーションで更新・削除クエリを実行する場合、併せて必要なアノテーションは何ですか？

    回答を表示

    @Modifying
    @Modifyingアノテーションを付けることで、Spring Data JDBCに「このクエリは更新・削除処理である」ことを伝えます。付けないとエラーになります。

  2. Q2. PageRequest.of(0, 20, Sort.by("price").descending()) の意味を説明してください。

    回答を表示

    最初のページ（0ページ目）を、1ページ20件で取得し、priceカラムで降順（高い順）にソートする設定です。ページ番号は0から始まることに注意してください。

  3. Q3. 集約ルート（Aggregate Root）とは何ですか？なぜ重要ですか？

    回答を表示

    集約ルートは、関連する複数のエンティティをまとめて管理する親エンティティです。例えば、OrderがOrderItemを管理します。重要な理由は、データの一貫性を保ちやすく、保存・更新・削除が親を通じて一括管理できるためです。

第9章のまとめ

  • @Queryアノテーションで複雑なクエリを記述でき、更新系には@Modifyingも必要です。
  • PageableとSortを使うことで、大量データのページング・ソート処理が簡単に実装できます。
  • 1対多リレーションは集約ルートパターンで管理し、親エンティティが子のライフサイクルを制御します。
  • @Transactionalでトランザクション境界を定義し、複数のDB操作の一貫性を保証できます。
  • Spring Data JDBCとJDBCTemplateを使い分けることで、保守性とパフォーマンスを両立できます。
