第15章

REST API開発入門

RESTful APIの開発方法を学びます。@RestControllerの使い方、@RequestBodyと@ResponseBodyによるJSON処理、HTTPステータスコードの適切な返却、REST APIのテスト方法、Postmanを使った動作確認を実装します。

この章で学ぶこと

  • RESTful APIの基本概念とHTTPメソッドの使い分けを理解する
  • @RestControllerと@RequestBodyによるJSON形式のデータ処理を実装する
  • ResponseEntityを使った適切なHTTPステータスコードとヘッダーの返却を学ぶ
  • MockMvcとPostmanを使ったREST APIのテスト手法を習得する

15.1 RESTful APIとは

REST（Representational State Transfer）は、Webシステムにおけるアーキテクチャスタイルの一つです。HTTPプロトコルの仕組みを活用し、リソース（データ）に対する操作をHTTPメソッド（GET、POST、PUT、DELETE）で表現します。

RESTful APIとは

RESTの原則に従って設計されたAPIをRESTful APIと呼びます。従来のHTMLを返すWebアプリケーションとは異なり、JSON形式のデータをやり取りし、フロントエンド（React、Vue.jsなど）やモバイルアプリとの連携に適しています。

HTTPメソッドとCRUD操作の対応

graph LR GET[GET
読み取り] --> Read[Read
リソース取得] POST[POST
作成] --> Create[Create
リソース作成] PUT[PUT
更新] --> Update[Update
リソース全体更新] DELETE[DELETE
削除] --> Delete[Delete
リソース削除]

RESTful APIの基本原則

  • リソース指向：URLでリソースを表現（例：/api/users/1）
  • HTTPメソッド使用：操作の意図をメソッドで表現
  • ステートレス：各リクエストは独立し、サーバーはセッション状態を保持しない
  • JSONフォーマット：データはJSON形式でやり取り

15.2 @RestControllerの基本

Spring MVCで通常のWebアプリケーションを開発する際は@Controllerを使いますが、REST APIでは@RestControllerを使用します。@RestControllerは、すべてのメソッドに@ResponseBodyが自動的に付与され、戻り値が自動的にJSON形式に変換されます。

UserRestController.java（基本的なREST API） 
@RestController
@RequestMapping("/api/users")
public class UserRestController {

    private final UserService userService;

    public UserRestController(UserService userService) {
        this.userService = userService;
    }

    // GET: 全ユーザー取得
    @GetMapping
    public List<User> getAllUsers() {
        return userService.findAll();
    }

    // GET: 特定ユーザー取得
    @GetMapping("/{id}")
    public User getUserById(@PathVariable Long id) {
        return userService.findById(id);
    }
}

@ControllerとRestControllerの違い

@ControllerはHTMLビューを返すのに対し、@RestControllerはJSONデータを返します。@RestController = @Controller + @ResponseBodyの関係です。

15.3 @RequestBodyによるJSON受け取り

クライアントから送信されたJSON形式のデータを受け取るには、@RequestBodyアノテーションを使用します。SpringはJacksonライブラリを使って、JSONを自動的にJavaオブジェクトに変換します。

UserRestController.java（POST、PUT実装） 
@RestController
@RequestMapping("/api/users")
public class UserRestController {

    private final UserService userService;

    public UserRestController(UserService userService) {
        this.userService = userService;
    }

    // POST: 新規ユーザー作成
    @PostMapping
    public User createUser(@RequestBody User user) {
        return userService.save(user);
    }

    // PUT: ユーザー情報更新
    @PutMapping("/{id}")
    public User updateUser(
        @PathVariable Long id,
        @RequestBody User user
    ) {
        return userService.update(id, user);
    }

    // DELETE: ユーザー削除
    @DeleteMapping("/{id}")
    public void deleteUser(@PathVariable Long id) {
        userService.deleteById(id);
    }
}

POSTリクエストのJSON例

POST /api/users リクエストボディ 
{
  "username": "yamada",
  "email": "yamada@example.com",
  "password": "Password123!"
}

15.4 ResponseEntityによるHTTPステータス制御

ResponseEntityを使用すると、レスポンスボディだけでなく、HTTPステータスコードやヘッダーも柔軟に設定できます。REST APIでは、操作の成功・失敗を適切なステータスコードで表現することが重要です。

主なHTTPステータスコード

コード 意味 用途
200 OK 成功 GET、PUT成功時
201 Created 作成成功 POSTでリソース作成成功時
204 No Content 成功（本文なし） DELETE成功時
400 Bad Request 不正なリクエスト バリデーションエラー時
404 Not Found リソースが存在しない 指定IDのデータが見つからない時
500 Internal Server Error サーバーエラー 予期しないエラー発生時
UserRestController.java（ResponseEntity使用） 
@RestController
@RequestMapping("/api/users")
public class UserRestController {

    private final UserService userService;

    public UserRestController(UserService userService) {
        this.userService = userService;
    }

    // GET: 特定ユーザー取得（404対応）
    @GetMapping("/{id}")
    public ResponseEntity<User> getUserById(@PathVariable Long id) {
        User user = userService.findById(id);
        if (user == null) {
            return ResponseEntity.notFound().build();
        }
        return ResponseEntity.ok(user);
    }

    // POST: 新規作成（201 Created）
    @PostMapping
    public ResponseEntity<User> createUser(@RequestBody User user) {
        User created = userService.save(user);
        return ResponseEntity.status(HttpStatus.CREATED).body(created);
    }

    // DELETE: 削除（204 No Content）
    @DeleteMapping("/{id}")
    public ResponseEntity<Void> deleteUser(@PathVariable Long id) {
        userService.deleteById(id);
        return ResponseEntity.noContent().build();
    }
}

15.5 REST APIのエラーハンドリング

REST APIでは、エラー発生時にクライアントが理解できる形式でエラー情報を返す必要があります。@ControllerAdviceを使って、共通のエラーハンドリングを実装しましょう。

ErrorResponse.java（エラーレスポンス用DTO） 
public class ErrorResponse {
    private int status;
    private String message;
    private LocalDateTime timestamp;

    public ErrorResponse(int status, String message) {
        this.status = status;
        this.message = message;
        this.timestamp = LocalDateTime.now();
    }

    // getter/setterは省略
}
RestExceptionHandler.java（共通エラーハンドラ） 
@RestControllerAdvice
public class RestExceptionHandler {

    // リソースが見つからない場合
    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(
        ResourceNotFoundException ex
    ) {
        ErrorResponse error = new ErrorResponse(404, ex.getMessage());
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(error);
    }

    // バリデーションエラー
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidation(
        MethodArgumentNotValidException ex
    ) {
        String message = ex.getBindingResult()
            .getFieldErrors()
            .stream()
            .map(FieldError::getDefaultMessage)
            .collect(Collectors.joining(", "));

        ErrorResponse error = new ErrorResponse(400, message);
        return ResponseEntity.badRequest().body(error);
    }
}

実習: シンプルなタスクREST APIの実装

タスク管理のREST APIを実装し、基本的なCRUD操作をマスターしましょう。

1. エンティティクラスの作成

Task.java 
@Table("tasks")
public class Task {
    @Id
    private Long id;
    private String title;
    private String description;
    private boolean completed;

    // getter/setter省略
}

2. リポジトリの作成

TaskRepository.java 
public interface TaskRepository extends CrudRepository<Task, Long> {
    List<Task> findByCompleted(boolean completed);
}

3. サービス層の実装

TaskService.java 
@Service
public class TaskService {
    private final TaskRepository taskRepository;

    public TaskService(TaskRepository taskRepository) {
        this.taskRepository = taskRepository;
    }

    public List<Task> findAll() {
        return (List<Task>) taskRepository.findAll();
    }

    public Task findById(Long id) {
        return taskRepository.findById(id).orElse(null);
    }

    public Task save(Task task) {
        return taskRepository.save(task);
    }

    public void deleteById(Long id) {
        taskRepository.deleteById(id);
    }
}

4. REST Controllerの実装

TaskRestController.java 
@RestController
@RequestMapping("/api/tasks")
public class TaskRestController {
    private final TaskService taskService;

    public TaskRestController(TaskService taskService) {
        this.taskService = taskService;
    }

    @GetMapping
    public ResponseEntity<List<Task>> getAllTasks() {
        return ResponseEntity.ok(taskService.findAll());
    }

    @GetMapping("/{id}")
    public ResponseEntity<Task> getTaskById(@PathVariable Long id) {
        Task task = taskService.findById(id);
        if (task == null) {
            return ResponseEntity.notFound().build();
        }
        return ResponseEntity.ok(task);
    }

    @PostMapping
    public ResponseEntity<Task> createTask(@RequestBody Task task) {
        Task created = taskService.save(task);
        return ResponseEntity.status(HttpStatus.CREATED).body(created);
    }

    @PutMapping("/{id}")
    public ResponseEntity<Task> updateTask(
        @PathVariable Long id,
        @RequestBody Task task
    ) {
        Task existing = taskService.findById(id);
        if (existing == null) {
            return ResponseEntity.notFound().build();
        }
        task.setId(id);
        Task updated = taskService.save(task);
        return ResponseEntity.ok(updated);
    }

    @DeleteMapping("/{id}")
    public ResponseEntity<Void> deleteTask(@PathVariable Long id) {
        taskService.deleteById(id);
        return ResponseEntity.noContent().build();
    }
}

15.6 REST APIのテスト

REST APIのテストには、MockMvcを使った統合テストとPostmanを使った手動テストの2つのアプローチがあります。

MockMvcによる統合テスト

TaskRestControllerTest.java 
@SpringBootTest
@AutoConfigureMockMvc
class TaskRestControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @Autowired
    private ObjectMapper objectMapper;

    @Test
    void getAllTasksを呼ぶと全タスクが返る() throws Exception {
        mockMvc.perform(get("/api/tasks"))
            .andExpect(status().isOk())
            .andExpect(content().contentType(MediaType.APPLICATION_JSON));
    }

    @Test
    void createTaskでPOSTするとタスクが作成される() throws Exception {
        Task task = new Task();
        task.setTitle("テストタスク");
        task.setDescription("テスト用");

        mockMvc.perform(post("/api/tasks")
                .contentType(MediaType.APPLICATION_JSON)
                .content(objectMapper.writeValueAsString(task)))
            .andExpect(status().isCreated())
            .andExpect(jsonPath("$.title").value("テストタスク"));
    }
}

Postmanによる手動テスト

Postmanは、REST APIを手動でテストするための便利なツールです。以下の手順でテストできます。

  1. 1. Postmanをダウンロード・インストール
  2. 2. Spring Bootアプリケーションを起動
  3. 3. Postmanで新規リクエストを作成し、メソッド（GET、POST等）とURL（http://localhost:8080/api/tasks）を設定
  4. 4. POSTの場合、Bodyタブで「raw」「JSON」を選択し、JSON形式のデータを入力
  5. 5. 「Send」ボタンでリクエストを送信し、レスポンスを確認

開発時のテスト戦略

開発初期はPostmanで手動テストし、動作を確認します。その後、MockMvcによる自動テストコードを作成し、リグレッションテスト（既存機能が壊れていないか確認）を自動化します。

動作確認: PostmanでREST APIを呼び出す

1. アプリケーション起動

Spring Bootアプリケーションを起動してください。

mvn spring-boot:run

2. GET /api/tasksでタスク一覧取得

  • • メソッド: GET
  • • URL: http://localhost:8080/api/tasks

3. POST /api/tasksで新規タスク作成

  • • メソッド: POST
  • • URL: http://localhost:8080/api/tasks
  • • Body (JSON):
{
  "title": "Spring学習",
  "description": "REST APIの章を終える",
  "completed": false
}

4. レスポンスの確認

POSTリクエストが成功すると、ステータスコード201 Createdが返り、作成されたタスクのJSONが返されることを確認してください。

トラブルシューティング

Q1. JSONが返ってこず、HTMLエラーページが表示される

@RestControllerではなく@Controllerを使っていませんか？REST APIでは必ず@RestControllerを使用してください。

Q2. POSTリクエストで415 Unsupported Media Typeエラーが出る

PostmanのリクエストヘッダーにContent-Type: application/jsonが設定されているか確認してください。Body形式は「raw」「JSON」を選択します。

Q3. 400 Bad Requestが返るが原因がわからない

JSONの形式が間違っている可能性があります。送信したJSONがJavaクラスのフィールド名と一致しているか、カンマやブレースの位置が正しいか確認してください。

Q4. @RequestBodyで受け取った値がnullになる

Javaクラスにgetter/setterが定義されているか確認してください。Jacksonはgetter/setterを使ってJSONをマッピングします。

理解度確認クイズ

  1. Q1. @ControllerではなくREST APIで使用すべきアノテーションは何ですか？その理由も説明してください。

    回答を表示

    @RestController
    @RestControllerを使用すると、すべてのメソッドに自動的に@ResponseBodyが適用され、戻り値がJSON形式に変換されます。これにより、ビューテンプレートではなくデータを返すREST APIに適しています。

  2. Q2. HTTPメソッドGET、POST、PUT、DELETEはそれぞれどのCRUD操作に対応しますか？

    回答を表示

    GET → Read（読み取り）
    POST → Create（作成）
    PUT → Update（更新）
    DELETE → Delete（削除）

  3. Q3. POST /api/usersで新規ユーザーが作成された時、返すべき適切なHTTPステータスコードは何ですか？

    回答を表示

    201 Created
    リソース作成が成功した場合は201 Createdを返します。200 OKでも動作しますが、RESTful APIの設計原則では201 Createdがより適切です。

第15章のまとめ

  • RESTful APIは、HTTPメソッドとURLでリソース操作を表現し、JSON形式でデータをやり取りします。
  • @RestControllerを使うと、戻り値が自動的にJSONに変換され、REST API開発が簡潔になります。
  • @RequestBodyでJSON形式のリクエストボディを受け取り、ResponseEntityで適切なHTTPステータスコードを返却します。
  • MockMvcによる自動テストとPostmanによる手動テストを組み合わせて、REST APIの品質を保証します。
