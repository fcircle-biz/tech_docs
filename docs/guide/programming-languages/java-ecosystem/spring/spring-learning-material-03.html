DI（依存性注入）とIoCコンテナ

Spring Coreの中核概念であるDI（Dependency Injection）とIoC（Inversion of Control）コンテナを学びます。なぜDIが必要なのか、従来のnewによるオブジェクト生成との違い、@Component、@Autowiredアノテーションの使い方、コンストラクタインジェクションの実装方法を理解します。

この章で学ぶこと

  • DI（依存性注入）とIoC（制御の反転）の概念を理解する
  • 従来のnewによるオブジェクト生成との違いと、DIの利点を説明できる
  • @Component、@Autowiredアノテーションの役割と使い方を習得する
  • コンストラクタインジェクションを実装し、疎結合な設計を実現する

3.1 DI（依存性注入）とは

DI（Dependency Injection：依存性注入）は、オブジェクトが必要とする依存オブジェクトを、外部から注入する設計パターンです。Spring Frameworkの中核となる概念であり、疎結合で保守性の高いアプリケーションを構築するための基盤となります。

依存とは

あるクラスが別のクラスの機能を利用している状態を「依存している」と言います。例えば、UserServiceクラスがUserRepositoryクラスを使ってデータベースアクセスを行う場合、UserServiceはUserRepositoryに依存しています。

従来のnewによるオブジェクト生成の問題点

まず、従来のJava開発でのオブジェクト生成方法を見てみましょう。

UserService.java（従来の方法） 
public class UserService {
    // new演算子で依存オブジェクトを直接生成
    private UserRepository userRepository = new UserRepository();

    public User findUserById(Long id) {
        return userRepository.findById(id);
    }
}

このコードには以下の問題があります：

1. 密結合

UserServiceクラス内でUserRepositoryを直接newしているため、両クラスが密に結合し、変更に弱くなります。

2. テスト困難

単体テスト時にモックオブジェクトを差し替えられず、実際のデータベース接続が必要になります。

3. 再利用性低下

UserRepositoryの実装を変更したい場合、UserServiceのコードも修正する必要があります。

4. オブジェクト管理困難

複数箇所でnewするとインスタンスが乱立し、ライフサイクル管理が複雑になります。

DIによる解決

DIを使うと、依存オブジェクトを外部から注入するため、上記の問題が解決されます。

UserService.java（DI使用） 
@Service
public class UserService {
    private final UserRepository userRepository;

    // コンストラクタで依存オブジェクトを受け取る
    @Autowired
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public User findUserById(Long id) {
        return userRepository.findById(id);
    }
}

重要ポイント

UserServiceはUserRepositoryのインスタンス化方法を知らず、コンストラクタで受け取るだけです。この「知らない」状態が疎結合の鍵であり、変更に強い設計を実現します。

3.2 IoC（制御の反転）コンテナ

IoC（Inversion of Control：制御の反転）は、オブジェクトのライフサイクル管理をアプリケーションコードからフレームワークに委ねる考え方です。SpringのIoCコンテナは、オブジェクトの生成、依存関係の解決、破棄までを管理します。

IoCコンテナの動作フロー

graph LR A[アプリケーション起動] --> B[IoCコンテナ起動] B --> C[@Componentなどが
付いたクラスをスキャン] C --> D[Bean登録] D --> E[依存関係解決
@Autowired処理] E --> F[Beanインスタンス生成] F --> G[アプリケーション実行]

Beanとは

SpringのIoCコンテナによって管理されるオブジェクトのことを「Bean」と呼びます。@Component、@Service、@Repositoryなどのアノテーションを付けたクラスがBean登録の対象となります。

制御の反転とは

従来の開発では、開発者がnewでオブジェクトを生成し、そのライフサイクルを管理していました。IoCでは、この「制御」がフレームワーク側に反転します。

観点 従来の方法 IoC（Spring）
オブジェクト生成 開発者がnewで生成 IoCコンテナが生成
依存関係解決 開発者が手動で設定 IoCコンテナが自動注入
ライフサイクル管理 開発者が明示的に破棄 IoCコンテナが自動管理
制御の主体 アプリケーションコード フレームワーク（Spring）

3.3 @Componentと@Autowiredの使い方

SpringでDIを実現するための基本的なアノテーションである@Componentと@Autowiredの使い方を学びます。

@Component - Bean登録

@Componentアノテーションを付けたクラスは、SpringのIoCコンテナによってBeanとして登録されます。

MessageService.java 
// @ComponentでBean登録
@Component
public class MessageService {
    public String getMessage() {
        return "Hello, Spring DI!";
    }
}

@Autowired - 依存注入

@Autowiredアノテーションを使うと、IoCコンテナが自動的に適切なBeanを注入してくれます。

MessageController.java 
@Component
public class MessageController {
    private final MessageService messageService;

    // コンストラクタに@Autowiredを付けて依存注入
    @Autowired
    public MessageController(MessageService messageService) {
        this.messageService = messageService;
    }

    public void printMessage() {
        System.out.println(messageService.getMessage());
    }
}

@Autowiredの省略

Spring 4.3以降、コンストラクタが1つしかない場合、@Autowiredは省略可能です。しかし、明示的に記述することでコードの意図が明確になるため、付けることを推奨します。

3.4 コンストラクタインジェクション

Springでは、コンストラクタ、セッター、フィールドの3種類のインジェクション方法がありますが、コンストラクタインジェクションが最も推奨されます。

3種類のインジェクション方法

1. コンストラクタインジェクション（推奨） 
@Service
public class UserService {
    private final UserRepository userRepository;

    // コンストラクタで注入
    @Autowired
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}
2. セッターインジェクション 
@Service
public class UserService {
    private UserRepository userRepository;

    // セッターで注入
    @Autowired
    public void setUserRepository(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}
3. フィールドインジェクション 
@Service
public class UserService {
    // フィールドに直接注入
    @Autowired
    private UserRepository userRepository;
}

なぜコンストラクタインジェクションが推奨されるのか

1. 不変性の保証

finalフィールドとして宣言でき、一度設定した依存オブジェクトが変更されないことを保証します。

2. 必須依存の明示

コンストラクタ引数として必須の依存関係が明確になり、依存オブジェクトなしでインスタンス化できません。

3. テスト容易性

コンストラクタでモックオブジェクトを渡せるため、Springコンテナなしで単体テストが可能です。

4. 循環依存の検出

クラスAとクラスBが互いに依存する循環依存を、アプリケーション起動時に検出できます。

実習: DIを体験しよう

簡単なDIのサンプルを作成し、動作を確認しましょう。

ステップ1: GreetingServiceクラスの作成

GreetingService.java 
package com.example.demo.service;

import org.springframework.stereotype.Component;

@Component
public class GreetingService {
    public String greet(String name) {
        return "こんにちは、" + name + "さん！";
    }
}

ステップ2: GreetingControllerクラスの作成

GreetingController.java 
package com.example.demo.controller;

import com.example.demo.service.GreetingService;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Component;

@Component
public class GreetingController {
    private final GreetingService greetingService;

    @Autowired
    public GreetingController(GreetingService greetingService) {
        this.greetingService = greetingService;
    }

    public void showGreeting() {
        String message = greetingService.greet("太郎");
        System.out.println(message);
    }
}

ステップ3: メインクラスで実行

DemoApplication.java 
package com.example.demo;

import com.example.demo.controller.GreetingController;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.context.ApplicationContext;

@SpringBootApplication
public class DemoApplication {
    public static void main(String[] args) {
        // Springコンテナ起動
        ApplicationContext context =
            SpringApplication.run(DemoApplication.class, args);

        // Beanを取得
        GreetingController controller =
            context.getBean(GreetingController.class);

        // メソッド実行
        controller.showGreeting();
    }
}

理解度確認クイズ

  1. Q1. DI（依存性注入）を使うことで解決される問題を3つ挙げてください。

    回答を表示

    1. 密結合の解消：クラス間の依存関係が疎結合になり、変更に強くなる
    2. テスト容易性の向上：モックオブジェクトを注入でき、単体テストが簡単
    3. 再利用性の向上：実装を差し替えやすく、コードの再利用性が高まる

  2. Q2. IoC（制御の反転）とは何か、従来の開発方法と比較して説明してください。

    回答を表示

    従来の方法：開発者がnewでオブジェクトを生成し、ライフサイクルを管理
    IoC：オブジェクトの生成・管理の制御がフレームワーク（Spring）に反転し、IoCコンテナが自動的に行う

    この反転により、開発者はビジネスロジックに集中でき、保守性の高いコードを実現できます。

  3. Q3. なぜコンストラクタインジェクションが推奨されるのか、その理由を説明してください。

    回答を表示

    1. 不変性の保証：finalフィールドとして宣言でき、依存オブジェクトが変更されない
    2. 必須依存の明示：コンストラクタ引数で必須の依存関係が明確になる
    3. テスト容易性：コンストラクタでモックを注入でき、Springなしでテスト可能
    4. 循環依存の検出：起動時に循環依存を検出し、設計ミスに早く気づける

第3章のまとめ

  • DI（依存性注入）は、オブジェクトの依存関係を外部から注入する設計パターンで、疎結合で保守性の高いコードを実現します。
  • IoC（制御の反転）により、オブジェクトのライフサイクル管理がSpringのIoCコンテナに委譲され、開発者はビジネスロジックに集中できます。
  • @ComponentでBeanを登録し、@Autowiredで依存オブジェクトを自動注入できます。
  • コンストラクタインジェクションは、不変性の保証、必須依存の明示、テスト容易性の観点から最も推奨される注入方法です。
