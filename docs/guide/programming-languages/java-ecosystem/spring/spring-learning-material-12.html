Spring Securityによる認証・認可

Spring Securityを使ったセキュリティ実装の基礎を学びます。認証と認可の概念、SecurityFilterChainの設定、フォームベース認証の実装、パスワードのハッシュ化（BCrypt）、URLベースのアクセス制御、ログイン・ログアウト機能の実装を行います。

この章で学ぶこと

  • 認証（Authentication）と認可（Authorization）の概念を理解する
  • Spring Securityの依存関係を追加し、基本設定を行う
  • SecurityFilterChainを設定し、URLベースのアクセス制御を実装する
  • フォームベース認証とログイン・ログアウト機能を実装する
  • BCryptPasswordEncoderを使ったパスワードのハッシュ化を理解する

12.1 認証と認可の基礎

Webアプリケーションのセキュリティを実装する際、認証（Authentication）と認可（Authorization）という2つの重要な概念を理解する必要があります。

認証と認可の違い

認証（Authentication）：ユーザーが誰であるかを確認すること（例: ログイン）
認可（Authorization）：認証されたユーザーが何をできるかを制御すること（例: 管理者のみアクセス可能）

認証・認可のフロー

Spring Securityとは

Spring Securityは、Springアプリケーションに包括的なセキュリティ機能を提供するフレームワークです。認証、認可、CSRF対策、セッション管理など、Webアプリケーションに必要なセキュリティ機能を簡単に実装できます。

Spring Securityの主な機能

フォーム認証、BASIC認証、OAuth 2.0、パスワード暗号化、CSRF保護、セッション管理、メソッドレベルのセキュリティ、URL単位のアクセス制御など、エンタープライズグレードのセキュリティ機能を提供します。

12.2 Spring Security依存関係の追加

Spring SecurityをSpring Bootプロジェクトに追加するには、pom.xmlに依存関係を追加します。

pom.xml 
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-security</artifactId>
</dependency>

依存関係追加後の動作変化

Spring Securityを追加すると、デフォルトですべてのURLが保護され、起動時にランダムなパスワードがコンソールに表示されます。ユーザー名は「user」です。この設定は後ほど変更します。

12.3 SecurityFilterChainの設定

Spring Security 5.7以降では、SecurityFilterChainをBean定義することでセキュリティ設定を行います。従来のWebSecurityConfigurerAdapterは非推奨となりました。

SecurityConfig.java（基本設定） 
package com.example.demo.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.web.SecurityFilterChain;

@Configuration
public class SecurityConfig {

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(authorize -> authorize
                .requestMatchers("/public/**").permitAll()  // 公開エンドポイント
                .anyRequest().authenticated()                // それ以外は認証必須
            )
            .formLogin(form -> form
                .defaultSuccessUrl("/home", true)            // ログイン成功後のURL
            )
            .logout(logout -> logout
                .logoutSuccessUrl("/login?logout")           // ログアウト後のURL
            );
        return http.build();
    }
}

設定項目の解説

メソッド 説明
authorizeHttpRequests() URL単位のアクセス制御を設定
requestMatchers() 特定のURLパターンを指定
permitAll() すべてのユーザーにアクセスを許可
authenticated() 認証されたユーザーのみアクセス可能
formLogin() フォームベース認証を有効化
logout() ログアウト機能を設定

12.4 パスワードのハッシュ化（BCrypt）

パスワードを平文でデータベースに保存することは重大なセキュリティリスクです。Spring Securityでは、BCryptPasswordEncoderを使ってパスワードをハッシュ化します。

平文パスワードの危険性

パスワードを平文で保存すると、データベースが漏洩した際にすべてのユーザーパスワードが流出します。BCryptなどのハッシュアルゴリズムを使用し、元のパスワードを復元できない形式で保存することが必須です。

SecurityConfig.java（PasswordEncoder追加） 
@Configuration
public class SecurityConfig {

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();  // BCryptでハッシュ化
    }

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        // 省略（前セクションと同じ）
        return http.build();
    }
}

BCryptの特徴

  • ソルト（Salt）の自動生成: 同じパスワードでも毎回異なるハッシュ値が生成される
  • ストレッチング（Stretching）: 計算コストを高めてブルートフォース攻撃を困難にする
  • 一方向性: ハッシュ値から元のパスワードを復元できない
パスワードハッシュ化の例 
@Autowired
private PasswordEncoder passwordEncoder;

public void createUser(String username, String rawPassword) {
    // 平文パスワードをハッシュ化
    String hashedPassword = passwordEncoder.encode(rawPassword);

    // DBに保存（ハッシュ化済み）
    User user = new User();
    user.setUsername(username);
    user.setPassword(hashedPassword);  // ハッシュ化済みパスワードを保存
    userRepository.save(user);
}

12.5 インメモリユーザーでの動作確認

データベース連携の前に、まずインメモリ（メモリ上）でユーザーを定義して、Spring Securityの動作を確認しましょう。

SecurityConfig.java（インメモリユーザー設定） 
@Configuration
public class SecurityConfig {

    @Bean
    public UserDetailsService userDetailsService(PasswordEncoder passwordEncoder) {
        // インメモリでユーザーを定義（テスト用）
        UserDetails user = User.builder()
            .username("user")
            .password(passwordEncoder.encode("password"))  // BCryptでハッシュ化
            .roles("USER")  // ロール: ROLE_USER
            .build();

        UserDetails admin = User.builder()
            .username("admin")
            .password(passwordEncoder.encode("admin123"))
            .roles("ADMIN")  // ロール: ROLE_ADMIN
            .build();

        return new InMemoryUserDetailsManager(user, admin);
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        // 省略（前セクションと同じ）
        return http.build();
    }
}

ロール（Role）とは

ロールは、ユーザーの権限グループを表します。例えば「USER」「ADMIN」などを定義し、ロールに基づいてアクセス制御を行います。Spring Securityでは、ロール名に「ROLE_」プレフィックスが自動付与されます。

実習 12-1: Spring Securityの基本設定

インメモリユーザーを設定し、フォームログインを実装しましょう。

手順

  1. 1 pom.xmlにspring-boot-starter-securityを追加し、Mavenプロジェクトを更新
  2. 2 SecurityConfig.javaを作成し、PasswordEncoderとUserDetailsServiceをBean定義
  3. 3 SecurityFilterChainで、/publicは認証不要、それ以外は認証必須と設定
  4. 4 アプリケーションを起動し、http://localhost:8080/にアクセスしてログイン画面が表示されることを確認
  5. 5 username: user, password: passwordでログインし、/homeにリダイレクトされることを確認

12.6 カスタムログインページの作成

Spring Securityのデフォルトログイン画面ではなく、独自のログインページを作成しましょう。

login.html（Thymeleaf） 
<!DOCTYPE html>
<html xmlns:th="http://www.thymeleaf.org">
<head>
    <title>ログイン</title>
</head>
<body>
    <h1>ログイン</h1>
    <form th:action="@{/login}" method="post">
        <div>
            <label>ユーザー名:</label>
            <input type="text" name="username" />
        </div>
        <div>
            <label>パスワード:</label>
            <input type="password" name="password" />
        </div>
        <button type="submit">ログイン</button>
    </form>
</body>
</html>
SecurityConfig.java（カスタムログインページ設定） 
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(authorize -> authorize
            .requestMatchers("/login", "/css/**", "/js/**").permitAll()  // ログインページとリソースは公開
            .anyRequest().authenticated()
        )
        .formLogin(form -> form
            .loginPage("/login")                         // カスタムログインページのURL
            .defaultSuccessUrl("/home", true)
            .permitAll()
        )
        .logout(logout -> logout
            .logoutSuccessUrl("/login?logout")
            .permitAll()
        );
    return http.build();
}

理解度確認クイズ

  1. Q1. 認証（Authentication）と認可（Authorization）の違いを説明してください。

    回答を表示

    認証（Authentication）：ユーザーが誰であるかを確認するプロセス。ログイン時にユーザー名とパスワードを検証すること。
    認可（Authorization）：認証されたユーザーが特定のリソースやアクションにアクセスできるかを制御するプロセス。例えば、管理者ロールのみが管理画面にアクセスできるといった制御。

  2. Q2. BCryptPasswordEncoderを使う理由を3つ挙げてください。

    回答を表示

    1. ソルトの自動生成：同じパスワードでも毎回異なるハッシュ値が生成され、レインボーテーブル攻撃を防ぐ
    2. ストレッチング：計算コストを高め、ブルートフォース攻撃を困難にする
    3. 一方向性：ハッシュ値から元のパスワードを復元できないため、データベース漏洩時の被害を最小化

  3. Q3. SecurityFilterChainのrequestMatchers("/public/**").permitAll()は何を意味しますか？

    回答を表示

    /public/以下のすべてのURLは、認証なしでアクセス可能という設定です。例えば、/public/index.html、/public/images/logo.pngなどは、ログインしていないユーザーでもアクセスできます。静的リソースやランディングページなどで使用します。
