例外処理とデバッグ

プログラムのエラーに対処するための例外処理について学びます。try-catch-finally構文、例外の種類、カスタム例外の作成、そして効果的なデバッグ手法を理解します。

この章で学ぶこと

  • 例外とは何か、なぜ例外処理が必要なのかを理解する
  • try-catch-finally構文を使った例外処理の基本
  • 検査例外と非検査例外の違いと使い分け
  • カスタム例外の作成とthrows宣言
  • 効果的なデバッグ手法とログ出力

12.1 例外とは何か

プログラムの実行中に、予期しない状況や問題が発生することがあります。このような異常な状態を例外（Exception）と呼びます。例外が発生すると、通常のプログラムの流れが中断され、適切に処理されなければプログラムが異常終了してしまいます。

例外が発生する典型的な状況としては、以下のようなものがあります。

  • 存在しないファイルへのアクセス: 開こうとしたファイルが見つからない
  • ゼロ除算: 数値を0で割ろうとする
  • 配列の範囲外アクセス: 存在しないインデックスの要素を参照しようとする
  • nullオブジェクトへのアクセス: nullのオブジェクトのメソッドを呼び出そうとする
  • ネットワーク接続エラー: 通信先のサーバーに接続できない

発生?} B -->|正常| C[処理継続] B -->|異常| D[例外発生] D --> E{例外処理
あり?} E -->|Yes| F[例外を補足

異常終了] F --> H[エラー回復処理] H --> I[処理継続または
適切な終了] C --> J[正常終了]

例外処理の重要性

例外処理を適切に実装することで、プログラムがエラー発生時にも異常終了せず、エラー情報をユーザーに分かりやすく伝えたり、自動で回復したりできるようになります。これは堅牢なアプリケーションを作るために必須のスキルです。

12.2 try-catch-finally基本構文

Javaではtry-catch-finally構文を使って例外を処理します。これは「例外が発生する可能性のあるコードを試してみて（try）、例外が発生したら捕まえて（catch）処理し、最後に必ず実行する（finally）処理を行う」という仕組みです。

基本構文

try-catch構文 
try {
    // 例外が発生する可能性のあるコード
    int result = 10 / 0;  // ゼロ除算例外が発生
} catch (ArithmeticException e) {
    // 例外が発生した時の処理
    System.out.println("エラー: ゼロで割ることはできません");
} finally {
    // 例外の有無に関わらず必ず実行される処理
    System.out.println("処理を終了します");
}

実際の例: ファイル読み込み

ファイル読み込みでは、ファイルが存在しない場合に例外が発生します。

FileReadExample.java 
import java.io.*;

public class FileReadExample {
    public static void main(String[] args) {
        BufferedReader reader = null;
        try {
            // ファイルを開く（存在しない場合は例外発生）
            reader = new BufferedReader(new FileReader("data.txt"));
            String line = reader.readLine();
            System.out.println("内容: " + line);
        } catch (FileNotFoundException e) {
            System.out.println("エラー: ファイルが見つかりません");
        } catch (IOException e) {
            System.out.println("エラー: ファイル読み込みに失敗");
        } finally {
            // リソース解放は必ずfinally内で行う
            if (reader != null) {
                try {
                    reader.close();
                } catch (IOException e) {
                    // クローズ時のエラーも処理
                }
            }
        }
    }
}

try-with-resources（推奨）

Java 7以降では、try-with-resources構文を使うことで、finallyを使わずに自動的にリソースを解放できます。より安全で読みやすいコードになります。

try-with-resources構文 
// リソースを自動的にクローズ
try (BufferedReader reader = new BufferedReader(
        new FileReader("data.txt"))) {
    String line = reader.readLine();
    System.out.println("内容: " + line);
} catch (IOException e) {
    System.out.println("エラー: " + e.getMessage());
}

12.3 例外の種類と階層構造

Javaの例外は検査例外（Checked Exception）と非検査例外（Unchecked Exception）の2種類に分類されます。

graph TB A[Throwable] --> B[Error] A --> C[Exception] C --> D[RuntimeException
非検査例外] C --> E[IOException など
検査例外] D --> D1[NullPointerException] D --> D2[ArithmeticException] D --> D3[ArrayIndexOutOfBoundsException] E --> E1[FileNotFoundException] E --> E2[SQLException] style D fill:#ffdddd style E fill:#ddffdd

検査例外（Checked Exception）

コンパイル時にチェックされる例外で、必ずtry-catchで処理するか、throwsで宣言する必要があります。主にプログラム外部の要因で発生する例外です。

  • IOException

    ファイル入出力やネットワーク通信の失敗

  • SQLException

    データベース操作の失敗

  • FileNotFoundException

    指定されたファイルが見つからない

非検査例外（Unchecked Exception）

RuntimeExceptionを継承した例外で、try-catchでの処理は任意です。主にプログラムのバグやロジックエラーで発生します。

  • NullPointerException

    nullオブジェクトのメソッドやフィールドにアクセスしようとした

  • ArithmeticException

    ゼロ除算などの算術エラー

  • ArrayIndexOutOfBoundsException

    配列の範囲外インデックスにアクセスしようとした

  • IllegalArgumentException

    不正な引数がメソッドに渡された

12.4 カスタム例外の作成

独自の例外クラスを作成することで、アプリケーション固有のエラーをより明確に表現できます。

カスタム例外の定義

InsufficientBalanceException.java 
// 検査例外として定義（Exceptionを継承）
public class InsufficientBalanceException extends Exception {
    private double balance;     // 現在残高
    private double amount;       // 引き出そうとした金額

    public InsufficientBalanceException(double balance, double amount) {
        super("残高不足: 残高=" + balance + ", 必要=" + amount);
        this.balance = balance;
        this.amount = amount;
    }

    public double getBalance() { return balance; }
    public double getAmount() { return amount; }
}

カスタム例外の使用

BankAccount.java 
public class BankAccount {
    private double balance;

    // throwsで例外をスローすることを宣言
    public void withdraw(double amount)
            throws InsufficientBalanceException {
        if (amount > balance) {
            // カスタム例外をスロー
            throw new InsufficientBalanceException(balance, amount);
        }
        balance -= amount;
    }

    public static void main(String[] args) {
        BankAccount account = new BankAccount();
        account.balance = 1000;

        try {
            account.withdraw(1500);  // 残高不足
        } catch (InsufficientBalanceException e) {
            System.out.println(e.getMessage());
            System.out.println("不足額: " +
                (e.getAmount() - e.getBalance()));
        }
    }
}

throws宣言

メソッドが検査例外をスローする可能性がある場合、メソッド宣言にthrows 例外クラス名を付けて、呼び出し側にその例外を処理する責任を伝えます。

12.5 効果的なデバッグ手法

プログラムのバグを見つけて修正するデバッグは、開発において非常に重要なスキルです。

1. ログ出力によるデバッグ

プログラムの実行状況を確認するために、戦略的にログを出力します。

デバッグ用ログ出力 
public void processData(String data) {
    System.out.println("[DEBUG] 処理開始: data=" + data);

    try {
        // 処理
        int value = Integer.parseInt(data);
        System.out.println("[DEBUG] 変換成功: value=" + value);
    } catch (NumberFormatException e) {
        // スタックトレースを出力（詳細なエラー情報）
        e.printStackTrace();
        System.err.println("[ERROR] 数値変換失敗: " + data);
    }

    System.out.println("[DEBUG] 処理終了");
}

2. IDEのデバッガーを使う

IntelliJ IDEAやEclipseなどのIDEには強力なデバッガーが搭載されています。

  • ブレークポイント: 指定した行でプログラムを一時停止
  • ステップ実行: 1行ずつプログラムを実行して変数の値を確認
  • 変数ウォッチ: 変数の値をリアルタイムで監視
  • 式の評価: デバッグ中に任意の式を評価して値を確認

3. スタックトレースの読み方

例外が発生すると、スタックトレース（エラーが発生した場所の情報）が出力されます。 

Exception in thread "main" java.lang.NullPointerException
    at MyApp.processData(MyApp.java:25)  ← 例外が発生した場所
    at MyApp.main(MyApp.java:15)         ← mainから呼ばれた

スタックトレースの読み方のコツ

スタックトレースは上から順に読みます。一番上の行が実際に例外が発生した場所で、下に行くほど呼び出し元を遡っていきます。自分のコードのファイル名（MyApp.javaなど）を探し、その行番号を確認しましょう。

実習 12-1: 入力値チェックプログラムを作ろう

ユーザーが入力した年齢を検証し、不正な値に対して適切な例外を投げるプログラムを作成しましょう。

課題

年齢（0〜150）を入力として受け取り、範囲外の値に対してカスタム例外InvalidAgeExceptionをスローするメソッドを実装してください。

手順

  1. 1 InvalidAgeExceptionクラスを作成（Exceptionを継承）
  2. 2 validateAge(int age)メソッドを実装し、年齢が0〜150の範囲外ならInvalidAgeExceptionをスロー
  3. 3 mainメソッドでtry-catchを使って例外を処理

実装例

// カスタム例外
class InvalidAgeException extends Exception {
    public InvalidAgeException(int age) {
        super("不正な年齢: " + age + "（0〜150の範囲で入力してください）");
    }
}

public class AgeValidator {
    public static void validateAge(int age) throws InvalidAgeException {
        if (age < 0 || age > 150) {
            throw new InvalidAgeException(age);
        }
        System.out.println("年齢は正常です: " + age);
    }

    public static void main(String[] args) {
        try {
            validateAge(25);   // 正常
            validateAge(-5);   // 例外発生
        } catch (InvalidAgeException e) {
            System.err.println(e.getMessage());
        }
    }
}

チャレンジ: 年齢が負の場合と150超の場合で異なるメッセージを表示するように改良してみましょう。

トラブルシューティング

unreported exception エラーが出る

原因: 検査例外（Checked Exception）を処理していない

解決方法: メソッドにthrows宣言を追加するか、try-catchで例外を補足してください。

NullPointerExceptionが頻発する

原因: オブジェクトがnullの状態でメソッドやフィールドにアクセスしている

解決方法: オブジェクトを使用する前にif (obj != null)でチェックするか、Optional<T>を使用してnullを明示的に扱いましょう。

catchブロックで例外を握りつぶしてしまった

原因: catchブロック内で何も処理せず、例外を無視している

解決方法: 最低限e.printStackTrace()を呼ぶか、ログに記録しましょう。空のcatchブロックはバグの原因になります。

理解度確認クイズ

  1. Q1. try-catch-finally構文で、必ず実行されるブロックはどれですか？

    回答を表示

    finallyブロックです。finallyブロックは例外の有無に関わらず必ず実行されます。リソースの解放（ファイルクローズなど）に使われます。

  2. Q2. 検査例外（Checked Exception）と非検査例外（Unchecked Exception）の違いは何ですか？

    回答を表示

    検査例外: コンパイル時にチェックされ、try-catchまたはthrowsで処理が必須（例: IOException）
    非検査例外: RuntimeExceptionを継承し、処理は任意（例: NullPointerException）

  3. Q3. カスタム例外を作成する時、どのクラスを継承しますか？

    回答を表示

    検査例外にする場合はExceptionクラス、非検査例外にする場合はRuntimeExceptionクラスを継承します。業務ロジックのエラーは通常Exceptionを継承します。

  4. Q4. メソッドが例外をスローする可能性があることを宣言するキーワードは何ですか？

    回答を表示

    throwsキーワードです。
    例: public void readFile() throws IOException { ... }

この章のまとめ

  • 例外はプログラム実行中の異常状態で、適切に処理しないとプログラムが異常終了する
  • try-catch-finally構文で例外を補足・処理し、finallyでリソースを確実に解放する
  • 検査例外（Checked）は処理が必須、非検査例外（Unchecked）は処理が任意
  • ExceptionまたはRuntimeExceptionを継承してカスタム例外を作成できる
  • デバッグにはログ出力、IDEデバッガー、スタックトレース分析を活用する
