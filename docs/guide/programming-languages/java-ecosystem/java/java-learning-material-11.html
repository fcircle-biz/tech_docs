同じ名前で違う動き - ポリモーフィズム

同じメソッド名でも呼び出すオブジェクトによって動きが変わる「ポリモーフィズム」について学びます。抽象クラスとインターフェースも解説します。

この章で学ぶこと

  • ポリモーフィズムの概念と、同じメソッド名で異なる動作をする仕組みを理解する
  • 抽象クラスの役割と、サブクラスに実装を強制する方法を学ぶ
  • インターフェースを使った柔軟な設計と、複数インターフェースの実装を理解する

11.1 ポリモーフィズムって何だろう？

前章で「継承」について学びましたね。親クラスの機能を子クラスが引き継ぐ仕組みでした。 今回学ぶポリモーフィズム（polymorphism：多態性）は、 「同じ名前のメソッドでも、呼び出すオブジェクトによって違う動作をする」という仕組みです。

ポリモーフィズムとは？

「poly（多い）+ morph（形）」で「多態性」「多様性」を意味します。 プログラミングでは、同じメソッドを呼んでも、実際のオブジェクトの型によって動作が変わることを指します。

身近な例で理解しよう

たとえ話: 「鳴る」という動作

スマートフォン

「鳴る」→ 電子音が鳴る

黒電話

「鳴る」→ ベルがリンリン鳴る

呼び鈴

「鳴る」→ ピンポーンと鳴る

どれも「鳴る」という同じ動作名ですが、実際の音や鳴り方は違いますよね。 これがポリモーフィズムの考え方です。

Javaでは、親クラス型の変数に子クラスのオブジェクトを代入できます。 そして、その変数経由でメソッドを呼ぶと、実際に入っているオブジェクトのメソッドが実行されます。 これがポリモーフィズムです。

11.2 ポリモーフィズムを使ってみよう

動物クラスとその子クラス（犬、猫、鳥）を使って、ポリモーフィズムを実際に動かしてみましょう。

Animal.java 
// 親クラス: 動物
public class Animal {
    protected String name; // 名前

    public Animal(String name) {
        this.name = name;
    }

    // 鳴く（親クラスでは一般的な鳴き声）
    public void makeSound() {
        System.out.println(name + "が鳴きました");
    }
}
Dog.java 
// 子クラス: 犬
public class Dog extends Animal {
    public Dog(String name) {
        super(name);
    }

    @Override // オーバーライドしている印
    public void makeSound() {
        System.out.println(name + "がワンワン！と鳴きました");
    }
}
Cat.java 
// 子クラス: 猫
public class Cat extends Animal {
    public Cat(String name) {
        super(name);
    }

    @Override
    public void makeSound() {
        System.out.println(name + "がニャーニャー！と鳴きました");
    }
}
PolymorphismDemo.java 
// ポリモーフィズムの実演
public class PolymorphismDemo {
    public static void main(String[] args) {
        // 親クラス型の配列に、異なる子クラスのオブジェクトを入れる
        Animal[] animals = new Animal[3];
        animals[0] = new Dog("ポチ");
        animals[1] = new Cat("タマ");
        animals[2] = new Animal("謎の生物");

        // 同じmakeSoundメソッドを呼んでも、動作が違う！
        for (Animal animal : animals) {
            animal.makeSound();
        }
    }
}

実行結果

ポチがワンワン！と鳴きました
タマがニャーニャー！と鳴きました
謎の生物が鳴きました

ポイント

  • ・Animal[] という親クラス型の配列に、Dog、Cat、Animalすべて入れられる
  • ・同じ makeSound() を呼んでも、実際のオブジェクトに応じた動作をする
  • ・@Override アノテーションで、親のメソッドを上書きしていることを明示

11.3 抽象クラス - 実装を強制する仕組み

「動物クラス」のように、実体のない概念的なクラスを作ることがあります。 このようなクラスは抽象クラス（abstract class）として定義します。

抽象クラスとは？

abstract キーワードを付けたクラスです。 抽象クラスは直接インスタンス化できません（new できない）。 必ず子クラスを作って使います。

抽象メソッド

抽象クラスの中には抽象メソッド（abstract method）を定義できます。 抽象メソッドは「こういうメソッドを作ってね」という約束だけを書いたもので、 中身（実装）がありません。子クラスで必ず実装する必要があります。

Shape.java（抽象クラス） 
// 抽象クラス: 図形
public abstract class Shape {
    protected String name; // 図形の名前

    public Shape(String name) {
        this.name = name;
    }

    // 抽象メソッド: 面積を計算（中身なし、子クラスで実装）
    public abstract double getArea();

    // 通常のメソッド: 名前を表示
    public void displayInfo() {
        System.out.println("図形: " + name);
        System.out.println("面積: " + getArea());
    }
}
Circle.java 
// 子クラス: 円
public class Circle extends Shape {
    private double radius; // 半径

    public Circle(double radius) {
        super("円");
        this.radius = radius;
    }

    @Override
    public double getArea() {
        // 円の面積 = π × 半径²
        return Math.PI * radius * radius;
    }
}
Rectangle.java 
// 子クラス: 長方形
public class Rectangle extends Shape {
    private double width;  // 幅
    private double height; // 高さ

    public Rectangle(double width, double height) {
        super("長方形");
        this.width = width;
        this.height = height;
    }

    @Override
    public double getArea() {
        // 長方形の面積 = 幅 × 高さ
        return width * height;
    }
}

重要なルール

  • ・抽象クラスは new Shape() のように直接インスタンス化できない
  • ・抽象メソッドを持つクラスは、必ず抽象クラスにする必要がある
  • ・子クラスは抽象メソッドを必ず実装しなければならない

11.4 インターフェース - より柔軟な約束

抽象クラスよりもさらに柔軟な仕組みがインターフェース（interface）です。 インターフェースは「こういうメソッドを持ってくださいね」という約束だけを書いたものです。

インターフェースとは？

クラスが「こういう機能を持っています」と宣言するための仕組みです。 Javaでは1つのクラスは1つの親クラスしか継承できませんが、 複数のインターフェースを実装できます。

抽象クラスとインターフェースの違い

項目 抽象クラス インターフェース
キーワード abstract class interface
フィールド 通常のフィールドを持てる 定数のみ（暗黙的に public static final）
メソッド 通常のメソッドと抽象メソッド 抽象メソッド（Java 8以降はデフォルトメソッドも可）
継承/実装 1つのクラスのみ継承可能 複数のインターフェースを実装可能
使い分け 共通の基底クラスを作りたいとき 複数の「できること」を組み合わせたいとき

インターフェースの例

Flyable.java 
// インターフェース: 飛べる
public interface Flyable {
    void fly(); // 飛ぶメソッド（自動的に public abstract）
}
Swimmable.java 
// インターフェース: 泳げる
public interface Swimmable {
    void swim(); // 泳ぐメソッド
}
Bird.java 
// 鳥クラス: Flyableインターフェースを実装
public class Bird implements Flyable {
    private String name;

    public Bird(String name) {
        this.name = name;
    }

    @Override
    public void fly() {
        System.out.println(name + "が空を飛んでいます");
    }
}
Duck.java 
// アヒルクラス: 複数のインターフェースを実装
public class Duck implements Flyable, Swimmable {
    private String name;

    public Duck(String name) {
        this.name = name;
    }

    @Override
    public void fly() {
        System.out.println(name + "が低空飛行しています");
    }

    @Override
    public void swim() {
        System.out.println(name + "が水面を泳いでいます");
    }
}

ポイント

  • ・implements キーワードでインターフェースを実装する
  • ・複数のインターフェースをカンマ区切りで実装できる（例: implements Flyable, Swimmable）
  • ・インターフェースのメソッドは必ずすべて実装する必要がある

実習 11-1: 支払い方法インターフェースを作ろう

異なる支払い方法（クレジットカード、電子マネー、現金）を統一的に扱えるインターフェースを作成します。

手順

  1. 1 Payable.java というインターフェースを作成し、pay(int amount) メソッドを定義
  2. 2 CreditCard.java、ElectronicMoney.java、Cash.java の3つのクラスを作成し、それぞれPayableを実装
  3. 3 各クラスで pay メソッドを実装し、それぞれの支払い方法に応じたメッセージを表示
  4. 4 メインクラスで配列を使い、異なる支払い方法を統一的に処理

ヒント: Payable.java

Payable.java 
public interface Payable {
    void pay(int amount); // 支払い処理
}

11.5 ポリモーフィズムの利点

ポリモーフィズムを使うと、どんな良いことがあるのでしょうか？

コードの再利用性が高まる

同じコードで異なるオブジェクトを扱えるため、 処理を一般化でき、重複コードが減ります。

拡張性が高まる

新しいクラスを追加しても、既存のコードを変更せずに 新しい動作を組み込めます。

柔軟な設計ができる

インターフェースを使うことで、実装の詳細を隠し、 クラス間の結合度を下げられます。

チーム開発がしやすい

インターフェースで「約束」を決めておけば、 各自が独立して実装を進められます。

ポリモーフィズムによる設計の柔軟性

理解度確認クイズ

  1. Q1. ポリモーフィズムとは何ですか？

    回答を表示

    同じメソッド名でも、呼び出すオブジェクトの実際の型によって動作が変わる仕組みです。 親クラス型の変数に子クラスのオブジェクトを代入し、 実際のオブジェクトに応じたメソッドが実行されます。

  2. Q2. 抽象クラスと通常のクラスの違いは何ですか？

    回答を表示

    抽象クラスは abstract キーワードで定義され、 直接インスタンス化できません（new できない）。 抽象メソッド（実装のないメソッド）を持つことができ、 子クラスで必ず実装する必要があります。

  3. Q3. インターフェースを使う最大のメリットは何ですか？

    回答を表示

    複数のインターフェースを実装できることです。 Javaでは1つのクラスしか継承できませんが、 インターフェースは複数実装できるため、 様々な「できること」を組み合わせた柔軟な設計が可能になります。

  4. Q4. @Override アノテーションの役割は何ですか？

    回答を表示

    親クラスやインターフェースのメソッドを上書き（オーバーライド）していることを明示します。 もし名前や引数を間違えていた場合、コンパイル時にエラーが出るため、 バグの早期発見につながります。

11.6 この章のまとめ

  • ポリモーフィズムは、同じメソッド名でオブジェクトによって動作が変わる仕組み。
  • 抽象クラスは abstract キーワードで定義し、直接インスタンス化できない。
  • 抽象メソッドは実装がなく、子クラスで必ず実装しなければならない。
  • インターフェースは interface キーワードで定義し、メソッドの約束を書く。
  • 1つのクラスは複数のインターフェースを実装できる。
  • ポリモーフィズムにより、コードの再利用性・拡張性・柔軟性が向上する。
