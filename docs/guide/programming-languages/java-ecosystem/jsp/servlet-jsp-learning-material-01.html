Webアプリケーション開発の世界へ

Webアプリケーションの仕組みを理解し、開発環境をセットアップして最初のプログラムを動かすところまでを学びます。

この章で学ぶこと

  • Webアプリケーションの基本的な仕組みを理解する
  • HTTPプロトコルの基礎を理解する
  • Apache Tomcatをインストールし設定できる
  • IDEでWebアプリケーションプロジェクトを作成できる

1.1 Webアプリケーションとは

クライアント・サーバーモデルの理解

Webアプリケーションは、クライアント（ブラウザ）とサーバー（Webサーバー）が協調して動作するシステムです。クライアントがリクエストを送信し、サーバーが処理を行って結果を返すという基本的な流れを理解しましょう。

Tomcat]
    B -->|2. Servlet呼び出し| C[Servlet/JSP
アプリケーション]
    C -->|3. HTMLレスポンス| B
    B -->|4. HTTPレスポンス| A

静的Webサイトと動的Webアプリケーションの違い

静的なWebサイトはHTMLファイルをそのまま表示するだけですが、動的なWebアプリケーションはリクエストに応じてHTMLを生成し、ユーザーごとに異なる内容を表示できます。

動的Webアプリケーションの例

ショッピングサイトのカート機能、SNSのタイムライン表示、ネットバンキングなど、ユーザーの操作やデータに応じて内容が変わるものは全て動的Webアプリケーションです。

ServletとJSPの役割と関係性

Servlet（サーブレット）はJavaで書かれたサーバー側プログラムで、リクエスト処理やビジネスロジックを担当します。JSP（JavaServer Pages）はHTMLの中にJavaコードを埋め込める技術で、主に画面表示を担当します。

1.2 HTTPプロトコルの基礎

HTTPリクエストとレスポンスの構造

HTTP（HyperText Transfer Protocol）は、WebブラウザとWebサーバーが通信するためのプロトコルです。リクエストとレスポンスは、それぞれヘッダーとボディで構成されます。

HTTPリクエストの例 
GET /hello HTTP/1.1
Host: localhost:8080
User-Agent: Mozilla/5.0
Accept: text/html
Accept-Language: ja

HTTPメソッド

HTTPメソッドは、サーバーに対してどのような操作を行いたいかを示します。主なメソッドは以下の通りです。

  • GET: データの取得（検索、表示）
  • POST: データの送信（登録、更新）
  • PUT: データの更新
  • DELETE: データの削除

ステータスコード

レスポンスには必ずステータスコードが含まれ、処理の結果を示します。

  • 200 OK: 正常に処理完了
  • 404 Not Found: ページが見つからない
  • 500 Internal Server Error: サーバー内部エラー

1.3 開発環境のセットアップ

必要なソフトウェア

Servlet・JSP開発には以下のソフトウェアが必要です。

  • JDK 17以降: Javaの開発キット
  • Apache Tomcat 10.x: サーブレットコンテナ（Webサーバー）
  • IDE: Eclipse または IntelliJ IDEA

Apache Tomcatのインストール

実習 1-1: Tomcatのインストールと起動

Apache TomcatをインストールしてWebサーバーを起動します。

手順

  1. 1 Apache Tomcat公式サイト（https://tomcat.apache.org/）からTomcat 10.xをダウンロード
  2. 2 ダウンロードしたZIPファイルを適当な場所に解凍（例: C:\tomcat10）
  3. 3 環境変数JAVA_HOMEがJDKのパスに設定されていることを確認
  4. 4 bin/startup.bat（Windows）またはbin/startup.sh（Mac/Linux）を実行
  5. 5 ブラウザでhttp://localhost:8080にアクセスして、Tomcatの起動ページが表示されることを確認

期待される結果

ブラウザにTomcatのウェルカムページ（猫のロゴ）が表示されれば成功です。

つまずきポイント

Tomcatが起動しない場合は、以下を確認してください：
- ポート8080が他のプログラムで使用されていないか
- JAVA_HOME環境変数が正しく設定されているか
- JDKのバージョンが17以降か

1.4 Webアプリケーションの構造

ディレクトリ構成

ServletとJSPで作成したWebアプリケーションは、決められたディレクトリ構造に従って配置する必要があります。

Webアプリケーションのディレクトリ構成 

myapp/                  ← アプリケーションルート
├── index.html          ← 静的ファイル（HTML、CSS、JS、画像など）
├── hello.jsp           ← JSPファイル
├── WEB-INF/            ← 外部から直接アクセスできない領域
│   ├── web.xml         ← デプロイメント記述子（設定ファイル）
│   ├── classes/        ← Servletのclassファイル配置場所
│   │   └── com/
│   │       └── example/
│   │           └── HelloServlet.class
│   └── lib/            ← 使用するJARファイル（ライブラリ）
│       └── xxx.jar

重要ポイント

WEB-INF配下のファイルは、ブラウザから直接アクセスできません。これにより、Javaのソースコードや設定ファイルを保護できます。

web.xmlの役割

web.xmlはWebアプリケーションの設定ファイルで、Servletのマッピングやセッションのタイムアウトなどを定義します。最近はアノテーションで代替することも多いですが、基本構造を理解しておきましょう。

web.xml（基本構造） 
<?xml version="1.0" encoding="UTF-8"?>
<web-app xmlns="https://jakarta.ee/xml/ns/jakartaee"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="https://jakarta.ee/xml/ns/jakartaee
         https://jakarta.ee/xml/ns/jakartaee/web-app_6_0.xsd"
         version="6.0">

    <!-- ウェルカムファイルの設定 -->
    <welcome-file-list>
        <welcome-file>index.html</welcome-file>
        <welcome-file>index.jsp</welcome-file>
    </welcome-file-list>

</web-app>

理解度確認クイズ

  1. Q1. HTTPのGETメソッドとPOSTメソッドの主な違いは何ですか？

    回答を表示

    GETはデータの取得に使い、パラメータがURLに表示されます。POSTはデータの送信に使い、パラメータはリクエストボディに含まれるため、大量のデータや機密情報の送信に適しています。

  2. Q2. WEB-INFディレクトリの特徴は何ですか？

    回答を表示

    WEB-INFディレクトリとその配下のファイルは、ブラウザから直接アクセスできません。これにより、Servletのclassファイル、web.xml、ライブラリなどを外部から保護できます。

  3. Q3. TomcatでHTTPステータスコード404が表示される主な原因は何ですか？

    回答を表示

    404エラーは「ページが見つからない」ことを示します。主な原因は、URLのスペルミス、ファイルの配置場所が間違っている、Servletのマッピング設定が正しくない、などです。

この章のまとめ

  • Webアプリケーションはクライアント・サーバーモデルで動作し、HTTPプロトコルで通信します
  • Servletはビジネスロジック、JSPは画面表示を担当し、両者を組み合わせて開発します
  • Apache Tomcatはサーブレットコンテナとして、ServletとJSPを実行します
  • Webアプリケーションは決められたディレクトリ構造に従って配置する必要があります
