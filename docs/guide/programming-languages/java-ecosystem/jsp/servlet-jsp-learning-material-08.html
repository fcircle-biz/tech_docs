ServletとJSPの連携 - MVCパターン入門

ServletとJSPを適切に連携させ、保守性の高いWebアプリケーションを構築する方法を学びます。

この章で学ぶこと

  • フォワードとリダイレクトを使い分けられる
  • MVCパターンの概念を理解する
  • Servlet（Controller）とJSP（View）の役割分担ができる
  • RequestDispatcherを使いこなせる

8.1 フォワードとリダイレクト

フォワード - サーバー内での転送

フォワード（Forward）は、サーバー内部でServletからJSPへ処理を引き継ぐ仕組みです。クライアント（ブラウザ）は転送を認識せず、URLは変化しません。リクエスト属性に設定したデータを引き継げるため、ServletでビジネスロジックとJSPでの表示を分離する際に使用します。 

sequenceDiagram
    participant B as ブラウザ
    participant S as Servlet
    participant J as JSP

    B->>S: HTTPリクエスト
    Note over S: リクエスト処理
データ準備
    S->>J: フォワード
(サーバー内部)
    Note over J: 表示生成
    J->>B: HTTPレスポンス
    Note over B: URLは変わらない
フォワードの実装例 
// Servletでの処理
protected void doGet(HttpServletRequest request,
                     HttpServletResponse response)
        throws ServletException, IOException {
    // ビジネスロジック実行
    String result = "処理結果";

    // リクエスト属性にデータを設定
    request.setAttribute("message", result);

    // JSPにフォワード
    RequestDispatcher dispatcher =
        request.getRequestDispatcher("/result.jsp");
    dispatcher.forward(request, response);
}

リダイレクト - クライアントへの再リクエスト指示

リダイレクト（Redirect）は、サーバーがクライアントに対して別のURLへ再度リクエストするよう指示する仕組みです。ブラウザのURLが変化し、新しいリクエストとなるためリクエスト属性は引き継がれません。フォーム送信後の二重送信防止（PRGパターン）などで使用します。 

sequenceDiagram
    participant B as ブラウザ
    participant S as Servlet
    participant D as 別ページ

    B->>S: HTTPリクエスト(POST)
    Note over S: データ保存処理
    S->>B: 302リダイレクト
Location: /complete
    Note over B: URLが変わる
    B->>D: 新しいGETリクエスト
    D->>B: HTTPレスポンス
リダイレクトの実装例 
// Servletでの処理
protected void doPost(HttpServletRequest request,
                      HttpServletResponse response)
        throws ServletException, IOException {
    // データ保存処理
    saveData(request);

    // 完了ページへリダイレクト
    response.sendRedirect("complete.jsp");
}

使い分けの基準

項目 フォワード リダイレクト
URLの変化 変化しない 変化する
リクエスト属性 引き継がれる 引き継がれない
リクエスト回数 1回 2回（再リクエスト）
主な用途 MVC連携 POST後の遷移（PRG）

PRGパターン（Post-Redirect-Get）

フォーム送信（POST）後に直接結果ページを表示すると、ブラウザのリロード時にフォームが再送信されてしまいます。これを防ぐため、POST処理後にリダイレクトでGETリクエストに切り替える「PRGパターン」を使用します。

重要ポイント

POST処理後は必ずリダイレクトを使用することで、二重送信を防止できます。これは実務でも必須のパターンです。

8.2 RequestDispatcher

getRequestDispatcher()の使い方

RequestDispatcherは、サーバー内部でリクエストを転送するためのインターフェースです。ServletからJSPへのフォワードや、別のServletへのインクルードを行う際に使用します。

RequestDispatcherの基本的な使い方 
// 1. RequestDispatcherの取得
RequestDispatcher dispatcher =
    request.getRequestDispatcher("/view/result.jsp");

// 2. フォワード実行
dispatcher.forward(request, response);

// または include()で部分的に組み込み
// dispatcher.include(request, response);

forward()メソッド

forward()メソッドは、現在のServletでの処理を終了し、指定したJSPやServletに処理を完全に引き渡します。forward()実行後、元のServletのコードは実行されません。

forward()の使用例 
protected void doGet(HttpServletRequest request,
                     HttpServletResponse response)
        throws ServletException, IOException {
    // データの準備
    List<Product> products = productService.findAll();
    request.setAttribute("productList", products);

    // JSPにフォワード
    RequestDispatcher dispatcher =
        request.getRequestDispatcher("/WEB-INF/views/list.jsp");
    dispatcher.forward(request, response);

    // この行は実行されない
    System.out.println("Never executed");
}

パスの指定方法

RequestDispatcherでのパス指定には、絶対パス（コンテキストルートからのパス）と相対パスの2種類があります。

  • 絶対パス: "/WEB-INF/views/result.jsp" のようにスラッシュから始まる
  • 相対パス: "result.jsp" のように現在の場所からの相対位置を指定

ヒント

WEB-INF配下にJSPを配置すると、ブラウザから直接アクセスできなくなります。必ずServletを経由させることで、セキュリティとアーキテクチャの統一を図れます。

8.3 MVCパターンの基礎

MVCとは

MVC（Model-View-Controller）パターンは、アプリケーションを3つの役割に分離する設計パターンです。各層の責務を明確にすることで、保守性とテスト容易性を向上させます。

  • Model（モデル）: ビジネスロジックとデータを担当。JavaBeansやDAOクラスなど
  • View（ビュー）: 表示を担当。JSPでHTMLを生成
  • Controller（コントローラー）: リクエスト制御を担当。Servletでリクエストを受け、ModelとViewを連携 
flowchart TD
    B[ブラウザ] -->|1. HTTPリクエスト| C[Controller
Servlet]
    C -->|2. ビジネスロジック呼び出し| M[Model
JavaBeans/DAO]
    M -->|3. 処理結果| C
    C -->|4. データ設定してフォワード| V[View
JSP]
    V -->|5. HTMLレスポンス| B
    M -.->|データアクセス| DB[(Database)]

なぜMVCが重要か

MVCパターンを使うことで得られるメリット：

  • 保守性向上: 各層の役割が明確で、変更の影響範囲が限定される
  • 再利用性: Modelは他の画面からも使える
  • テスト容易性: ビジネスロジックを単体でテストできる
  • チーム開発: 役割分担がしやすい（デザイナーはJSP、プログラマーはServlet）

重要ポイント

JSPにビジネスロジックを書かない、ServletにHTMLを書かないという原則を守ることが、MVCパターン成功の鍵です。

実装例：商品検索機能

実習 8-1: MVCパターンで商品検索機能を実装

ServletとJSPを適切に役割分担させて、商品検索機能を作成します。

1. Model（Product.java）

Product.java 
// Model: 商品データを表すJavaBean
public class Product {
    private int id;
    private String name;
    private int price;

    // コンストラクタ
    public Product(int id, String name, int price) {
        this.id = id;
        this.name = name;
        this.price = price;
    }

    // getter/setter（省略）
}

2. Controller（SearchServlet.java）

SearchServlet.java 
@WebServlet("/search")
public class SearchServlet extends HttpServlet {
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {
        // 1. リクエストパラメータ取得
        String keyword = request.getParameter("keyword");

        // 2. Modelでビジネスロジック実行（検索処理）
        List<Product> products = searchProducts(keyword);

        // 3. 結果をリクエスト属性に設定
        request.setAttribute("products", products);
        request.setAttribute("keyword", keyword);

        // 4. Viewにフォワード
        RequestDispatcher dispatcher =
            request.getRequestDispatcher("/WEB-INF/result.jsp");
        dispatcher.forward(request, response);
    }

    // ビジネスロジック（本来はDAOクラスに分離）
    private List<Product> searchProducts(String keyword) {
        // 検索処理（簡略化）
        List<Product> list = new ArrayList<>();
        list.add(new Product(1, "ノートPC", 89800));
        list.add(new Product(2, "マウス", 1980));
        return list;
    }
}

3. View（result.jsp）

result.jsp 
<%@ page contentType="text/html;charset=UTF-8" %>
<%@ taglib prefix="c" uri="jakarta.tags.core" %>
<html>
<head>
    <title>検索結果</title>
</head>
<body>
    <h1>検索結果: ${keyword}</h1>

    <table border="1">
        <tr>
            <th>商品ID</th>
            <th>商品名</th>
            <th>価格</th>
        </tr>
        <c:forEach var="product" items="${products}">
        <tr>
            <td>${product.id}</td>
            <td>${product.name}</td>
            <td>${product.price}円</td>
        </tr>
        </c:forEach>
    </table>
</body>
</html>

期待される結果

/search?keyword=PC にアクセスすると、検索結果が表形式で表示されます。URLは変化せず、リクエスト属性を通じてデータが引き渡されます。

つまずきポイント

フォワード後にデータが表示されない場合は、リクエスト属性への設定漏れを確認してください。また、forward()後にresponse.getWriter()などでレスポンスを出力しようとするとIllegalStateExceptionが発生します。

理解度確認クイズ

  1. Q1. フォワードとリダイレクトの使い分けで正しいものはどれですか？

    回答を表示

    フォワードはMVC連携（ServletからJSPへのデータ引き渡し）に使用し、リダイレクトはPOST処理後の画面遷移（PRGパターン）に使用します。フォワードはURLが変わらずリクエスト属性を引き継ぎますが、リダイレクトはURLが変わり新しいリクエストになります。

  2. Q2. MVCパターンにおける各層の役割を説明してください。

    回答を表示

    Model（モデル）はビジネスロジックとデータを担当、View（ビュー）は表示を担当、Controller（コントローラー）はリクエスト制御とModelとViewの連携を担当します。ServletとJSPでは、ServletがController、JSPがView、JavaBeansがModelとなります。

  3. Q3. RequestDispatcherのforward()メソッド実行後に発生する可能性のあるエラーは何ですか？

    回答を表示

    forward()実行後にresponse.getWriter()などでレスポンスを出力しようとするとIllegalStateExceptionが発生します。forward()は処理を完全に転送するため、その後のレスポンス操作はできません。

この章のまとめ

  • フォワードはサーバー内部での転送でURLが変わらず、リダイレクトはクライアントへの再リクエスト指示でURLが変わります
  • POST処理後はリダイレクトを使用するPRGパターンで二重送信を防止します
  • MVCパターンでModel、View、Controllerの役割を分離し、保守性を向上させます
  • Servletでビジネスロジックを処理し、リクエスト属性でJSPにデータを引き渡して表示します
