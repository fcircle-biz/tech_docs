第7章

セッション管理 - ユーザー状態の保持

HTTPのステートレス性を補うセッション管理を学び、ログイン機能などを実装できるようになります。

この章で学ぶこと

  • HTTPのステートレス性とセッションの必要性を理解する
  • HttpSessionを使ってデータを管理できる
  • Cookieの仕組みを理解する
  • ログイン・ログアウト機能を実装できる

7.1 セッションとは

HTTPのステートレス性

HTTPプロトコルは「ステートレス（状態を持たない）」という特徴があります。これは、各リクエストが独立しており、前のリクエストの情報を保持しないことを意味します。例えば、ログインした後に別のページに移動すると、サーバーは「このユーザーがログインした」という情報を覚えていません。 

sequenceDiagram
    participant ブラウザ
    participant サーバー
    ブラウザ->>サーバー: リクエスト1（ログイン）
    サーバー->>ブラウザ: レスポンス1（ログイン成功）
    Note over サーバー: 状態を保持しない
（次のリクエストで忘れる）
    ブラウザ->>サーバー: リクエスト2（商品一覧）
    サーバー->>ブラウザ: 誰がログインしたか不明...

セッションの仕組み

セッションは、複数のリクエストにわたってユーザーの状態を保持する仕組みです。サーバーは各ユーザーに一意の「セッションID」を発行し、このIDを使ってユーザーを識別します。セッションIDは通常Cookieを使ってブラウザとサーバー間でやり取りされます。 

sequenceDiagram
    participant ブラウザ
    participant サーバー
    participant セッション保存領域
    ブラウザ->>サーバー: リクエスト1（ログイン）
    サーバー->>セッション保存領域: セッションID=ABC123を作成
ユーザー情報を保存
    サーバー->>ブラウザ: レスポンス1（Set-Cookie: JSESSIONID=ABC123）
    ブラウザ->>サーバー: リクエスト2（Cookie: JSESSIONID=ABC123）
    サーバー->>セッション保存領域: ABC123のセッション情報を取得
    セッション保存領域->>サーバー: ユーザー情報を返す
    サーバー->>ブラウザ: レスポンス2（ログイン済みユーザーとして処理）

セッションが必要な場面

  • ログイン認証: ユーザーがログイン状態を保持
  • ショッピングカート: 複数ページにわたってカート内容を保持
  • ウィザード形式: 複数画面の入力を段階的に保持
  • ユーザー設定: 表示言語やテーマなどの設定を保持

セキュリティ考慮事項

セッションIDが盗まれると、他人がなりすましてアクセスできてしまいます。HTTPSを使用し、セッションIDの漏洩を防ぐことが重要です。また、セッションタイムアウトを適切に設定しましょう。

7.2 HttpSessionの使い方

セッションの取得・作成

Servletでセッションを利用するには、HttpServletRequestのgetSession()メソッドを使用します。セッションが存在しない場合は新しく作成され、既に存在する場合は既存のセッションが返されます。

セッションの取得 
// セッションを取得（存在しない場合は新規作成）
HttpSession session = request.getSession();

// セッションが既に存在する場合のみ取得（新規作成しない）
HttpSession session = request.getSession(false);
if (session == null) {
    // セッションが存在しない場合の処理
}

属性の設定と取得

セッションにデータを保存するにはsetAttribute()、取得するにはgetAttribute()を使用します。セッションには任意のJavaオブジェクトを格納できます。

セッション属性の操作 
// セッションにデータを保存
session.setAttribute("username", "yamada");
session.setAttribute("loginTime", new Date());

// セッションからデータを取得（Object型で返されるので型変換が必要）
String username = (String) session.getAttribute("username");
Date loginTime = (Date) session.getAttribute("loginTime");

// 存在しない属性を取得するとnullが返される
String email = (String) session.getAttribute("email");  // null

属性の削除

セッション属性の削除 
// 特定の属性を削除
session.removeAttribute("username");

// セッション全体を無効化（すべての属性が削除される）
session.invalidate();

セッションタイムアウトの設定

セッションは一定時間操作がないと自動的に破棄されます。タイムアウト時間を設定することで、セキュリティを強化できます。

タイムアウトの設定 
// タイムアウトを30分（1800秒）に設定
session.setMaxInactiveInterval(1800);

// タイムアウトを取得（秒単位）
int timeout = session.getMaxInactiveInterval();

getAttributeの注意点

getAttribute()はObject型を返すため、必ず型変換（キャスト）が必要です。存在しない属性を取得するとnullが返されるため、NullPointerExceptionに注意してください。

7.3 Cookieの基礎

Cookieの仕組みと用途

Cookieは、サーバーがブラウザに保存させる小さなデータです。セッションIDの保存以外にも、ユーザーの設定や訪問履歴などを記憶させるために使われます。セッションと違い、ブラウザを閉じても保持できます（有効期限を設定した場合）。 

sequenceDiagram
    participant ブラウザ
    participant サーバー
    ブラウザ->>サーバー: リクエスト
    Note over サーバー: Cookieを作成
    サーバー->>ブラウザ: レスポンス
Set-Cookie: name=value
    Note over ブラウザ: Cookieを保存
    ブラウザ->>サーバー: 次のリクエスト
Cookie: name=value
    Note over サーバー: Cookieを読み取り

Cookieの作成と送信

Cookieの作成 
// Cookieを作成（名前と値を指定）
Cookie cookie = new Cookie("username", "yamada");

// 有効期限を設定（秒単位、7日間 = 7 * 24 * 60 * 60）
cookie.setMaxAge(7 * 24 * 60 * 60);

// Cookieの適用範囲を設定（パス）
cookie.setPath("/");

// ブラウザにCookieを送信
response.addCookie(cookie);

Cookieの読み取り

Cookieの読み取り 
// リクエストからすべてのCookieを取得
Cookie[] cookies = request.getCookies();

// 特定のCookieを探す
String username = null;
if (cookies != null) {
    for (Cookie cookie : cookies) {
        if ("username".equals(cookie.getName())) {
            username = cookie.getValue();
            break;
        }
    }
}

Cookieの削除

Cookieを削除するには、同じ名前のCookieを作成し、有効期限を0に設定して送信します。

Cookieの削除 
// 削除したいCookieと同じ名前のCookieを作成
Cookie cookie = new Cookie("username", "");
cookie.setMaxAge(0);  // 有効期限を0に設定
cookie.setPath("/");  // 元のCookieと同じパスを指定
response.addCookie(cookie);

セッションとCookieの使い分け

  • セッション: サーバー側で管理。機密性の高いデータに適している。ブラウザを閉じると消える。
  • Cookie: クライアント側で管理。設定や訪問履歴など、機密性の低いデータに適している。ブラウザを閉じても保持できる。

7.4 ログイン機能の実装

ログイン機能の流れ

graph TD
    A[ログインフォーム] -->|ID/PW送信| B[ログインServlet]
    B -->|認証処理| C{認証成功?}
    C -->|成功| D[セッションに
ユーザー情報保存]
    C -->|失敗| E[エラーメッセージ表示]
    D --> F[メインページへ遷移]
    E --> A

ログインフォーム（JSP）

login.jsp 
<%@ page contentType="text/html; charset=UTF-8" %>
<!DOCTYPE html>
<html>
<head><title>ログイン</title></head>
<body>
    <h2>ログイン</h2>
    <form action="LoginServlet" method="post">
        ユーザーID: <input type="text" name="userId"><br>
        パスワード: <input type="password" name="password"><br>
        <input type="submit" value="ログイン">
    </form>
</body>
</html>

ログイン処理（Servlet）

LoginServlet.java 
@WebServlet("/LoginServlet")
public class LoginServlet extends HttpServlet {
    protected void doPost(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {

        request.setCharacterEncoding("UTF-8");
        String userId = request.getParameter("userId");
        String password = request.getParameter("password");

        // 簡易的な認証（実際はデータベースと照合）
        if ("admin".equals(userId) && "pass123".equals(password)) {
            // 認証成功 - セッションにユーザー情報を保存
            HttpSession session = request.getSession();
            session.setAttribute("loginUser", userId);
            response.sendRedirect("main.jsp");
        } else {
            // 認証失敗
            request.setAttribute("errorMsg", "IDまたはパスワードが正しくありません");
            request.getRequestDispatcher("login.jsp").forward(request, response);
        }
    }
}

ログインチェック処理

保護されたページでは、セッションにログイン情報があるかチェックします。

main.jsp（保護されたページ） 
<%@ page contentType="text/html; charset=UTF-8" %>
<%
    // ログインチェック
    String loginUser = (String) session.getAttribute("loginUser");
    if (loginUser == null) {
        // ログインしていない場合はログインページへ
        response.sendRedirect("login.jsp");
        return;
    }
%>
<!DOCTYPE html>
<html>
<head><title>メインページ</title></head>
<body>
    <h2>ようこそ、<%= loginUser %>さん</h2>
    <a href="LogoutServlet">ログアウト</a>
</body>
</html>

ログアウト処理

LogoutServlet.java 
@WebServlet("/LogoutServlet")
public class LogoutServlet extends HttpServlet {
    protected void doGet(HttpServletRequest request,
                        HttpServletResponse response)
            throws ServletException, IOException {

        // セッションを無効化（すべての属性が削除される）
        HttpSession session = request.getSession(false);
        if (session != null) {
            session.invalidate();
        }

        // ログインページへリダイレクト
        response.sendRedirect("login.jsp");
    }
}

実習 7-1: 簡易ログイン機能の実装

セッションを使った基本的なログイン・ログアウト機能を実装します。

要件

  • • ログインフォーム（login.jsp）を作成
  • • ログイン処理Servlet（LoginServlet）を作成し、認証成功時にセッションにユーザー名を保存
  • • メインページ（main.jsp）でログイン状態をチェックし、未ログインならログインページへリダイレクト
  • • ログアウト処理Servlet（LogoutServlet）を作成し、セッションを無効化

期待される動作

  1. 1. 正しいID/PWでログインすると、セッションが作成されメインページが表示される
  2. 2. メインページにユーザー名が表示される
  3. 3. ログアウトボタンでセッションが破棄され、ログインページに戻る
  4. 4. ログアウト後にブラウザの「戻る」ボタンでメインページにアクセスしようとすると、ログインページにリダイレクトされる

動作確認

  1. 1 login.jspにアクセスし、正しいID/PWでログイン
  2. 2 main.jspに遷移し、ユーザー名が表示されることを確認
  3. 3 ブラウザの開発者ツールでCookieを確認し、JSESSIONIDが設定されていることを確認
  4. 4 ログアウトリンクをクリックし、login.jspに戻ることを確認

トラブルシューティング

セッションが維持されない

  • • ブラウザでCookieが無効になっていないか確認
  • • セッションタイムアウトの設定を確認
  • • 開発者ツールでCookie（JSESSIONID）が送信されているか確認

ClassCastExceptionが発生する

  • • getAttribute()で取得した値を正しい型にキャストしているか確認
  • • セッションに保存したオブジェクトの型を確認

ログアウト後もページにアクセスできる

  • • ブラウザのキャッシュが表示されている可能性（Ctrl+F5で強制リロード）
  • • 各保護ページでログインチェックを実装しているか確認

理解度確認クイズ

  1. Q1. HTTPのステートレス性とは何ですか？また、セッションはどのようにこの問題を解決しますか？

    回答を表示

    HTTPのステートレス性とは、各リクエストが独立しており、前のリクエストの情報を保持しないことです。セッションは、サーバーが各ユーザーに一意のセッションIDを発行し、このIDをCookie等で保持することで、複数のリクエストにわたってユーザーの状態を管理できるようにします。

  2. Q2. session.getAttribute()がnullを返す場合、どのような原因が考えられますか？

    回答を表示

    主な原因は以下の通りです：1) 指定した名前の属性がセッションに設定されていない、2) セッションがタイムアウトして無効化された、3) session.invalidate()によってセッションが破棄された、4) 属性名のスペルミス。

  3. Q3. セッションとCookieの違いを説明してください。

    回答を表示

    セッションはサーバー側でデータを管理し、機密性の高い情報の保存に適しています。通常はブラウザを閉じると失われます。Cookieはクライアント側（ブラウザ）でデータを管理し、ユーザーが改変できるため機密性の低いデータに適しています。有効期限を設定すればブラウザを閉じても保持されます。

この章のまとめ

  • HTTPはステートレスなプロトコルで、セッションによってユーザー状態を保持できます
  • HttpSessionを使ってサーバー側でデータを管理し、setAttribute/getAttributeで操作します
  • Cookieはクライアント側でデータを保持する仕組みで、セッションIDの保存などに使われます
  • ログイン機能はセッションを使って実装し、適切なセキュリティ対策が必要です
