はじめてのServlet - Javaでリクエストを処理する

Servletの基本を学び、Javaプログラムでブラウザからのリクエストを処理できるようになります。

この章で学ぶこと

  • Servletの概念と役割を理解する
  • 簡単なServletを作成できる
  • Servletのライフサイクルを理解する
  • アノテーションとweb.xmlでのマッピングができる

2.1 Servletとは何か

Servletの定義と特徴

Servlet（サーブレット）は、Webサーバー上で動作するJavaプログラムです。ブラウザからのHTTPリクエストを受け取り、処理を行い、HTMLなどのレスポンスを返す役割を担います。

Servletの主な特徴

  • Java標準技術: Jakarta EE（旧Java EE）の一部
  • サーバーサイド実行: Tomcatなどのサーブレットコンテナで動作
  • 高性能: マルチスレッドで複数リクエストを効率的に処理
  • プラットフォーム独立: JVMが動けばどこでも実行可能

Servletコンテナ（Tomcat）の役割

Servletコンテナは、Servletのライフサイクルを管理し、HTTPリクエストをServletに渡す役割を持ちます。Apache Tomcatは最も広く使われているServletコンテナです。 

graph LR
    A[ブラウザ] -->|HTTPリクエスト| B[Tomcat
Servletコンテナ]
    B -->|リクエスト振り分け| C[HelloServlet]
    C -->|レスポンス生成| B
    B -->|HTTPレスポンス| A

HTTPリクエストとレスポンスの構造

ブラウザとサーバー間の通信は、HTTPリクエストとHTTPレスポンスという2つのメッセージで構成されます。それぞれにヘッダー部とボディ部があり、メソッド、URL、ステータスコードなどの重要な情報が含まれています。

CGIとの違い、Servletの利点

従来のCGI（Common Gateway Interface）では、リクエストごとに新しいプロセスが起動されるため、パフォーマンスが悪いという問題がありました。Servletは一度読み込まれるとメモリに常駐し、複数のリクエストをマルチスレッドで処理するため、高速に動作します。

重要ポイント

Servletはインスタンスが一度作成されると、複数のリクエストで再利用されます。そのため、初期化処理は最初の一度だけ実行されます。

2.2 最初のServlet作成

HttpServletクラスの継承

Servletを作成するには、jakarta.servlet.http.HttpServletクラスを継承します。このクラスがHTTPプロトコルに対応したServletの基本機能を提供します。

実習 2-1: 最初のServletを作成する

「Hello, Servlet!」と表示する最初のServletを作成しましょう。

手順

  1. 1 IDEで新しいJavaクラス「HelloServlet」を作成
  2. 2 HttpServletクラスを継承し、doGet()メソッドをオーバーライド
  3. 3 @WebServletアノテーションでURLマッピングを設定
  4. 4 Tomcatを起動し、ブラウザで http://localhost:8080/myapp/hello にアクセス
HelloServlet.java 
package com.example;

import jakarta.servlet.ServletException;
import jakarta.servlet.annotation.WebServlet;
import jakarta.servlet.http.HttpServlet;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import java.io.IOException;
import java.io.PrintWriter;

// URLマッピング: /hello というパスでアクセス可能
@WebServlet("/hello")
public class HelloServlet extends HttpServlet {

    // GETリクエストを処理するメソッド
    @Override
    protected void doGet(HttpServletRequest request,
                        HttpServletResponse response)
            throws ServletException, IOException {

        // レスポンスの文字コードとコンテンツタイプを設定
        response.setContentType("text/html; charset=UTF-8");

        // レスポンスに書き込むためのWriterを取得
        PrintWriter out = response.getWriter();

        // HTMLを出力
        out.println("<!DOCTYPE html>");
        out.println("<html>");
        out.println("<head><title>Hello</title></head>");
        out.println("<body>");
        out.println("<h1>Hello, Servlet!</h1>");
        out.println("</body>");
        out.println("</html>");
    }
}

コードの解説

各部分の役割

  • @WebServlet("/hello"): このServletが「/hello」というURLで呼ばれることを指定
  • doGet()メソッド: GETリクエストを受け取ったときに実行される処理
  • HttpServletRequest: リクエスト情報を持つオブジェクト
  • HttpServletResponse: レスポンスを返すためのオブジェクト
  • PrintWriter: HTMLなどのテキストを出力するためのクラス

2.2.5 GETメソッドとPOSTメソッドの違い

HTTPリクエストにはいくつかのメソッドがありますが、最もよく使われるのがGETとPOSTです。この2つは目的と使い方が大きく異なります。

GETとPOSTの比較

項目 GET POST
用途 データの取得・閲覧 データの送信・更新
パラメータ位置 URLに含まれる（クエリ文字列） リクエストボディ内
ブラウザ表示 URLにパラメータが見える URLにパラメータが見えない
データサイズ 制限あり（通常2KB程度） 大きなデータも送信可能
セキュリティ URLに表示されるため注意 URLに表示されない

使い分けの基準

  • GETを使う場合: 検索、一覧表示、詳細表示など、データを変更しない操作
  • POSTを使う場合: ログイン、登録、更新、削除など、データを変更する操作

2.3 Servletのマッピング

@WebServletアノテーションの使い方

@WebServletアノテーションを使うと、web.xmlに設定を書かなくてもURLマッピングを簡単に設定できます。これは最もモダンで推奨される方法です。

アノテーションの例 
// 単一URL
@WebServlet("/users")

// 複数URL
@WebServlet(urlPatterns = {"/user", "/users"})

// ワイルドカード
@WebServlet("/admin/*")

// 名前付き
@WebServlet(name = "UserServlet", urlPatterns = "/users")

web.xmlでの設定方法（従来方式）

アノテーションが使えない環境や、設定を外部化したい場合はweb.xmlで設定します。

web.xml 
<servlet>
    <servlet-name>HelloServlet</servlet-name>
    <servlet-class>com.example.HelloServlet</servlet-class>
</servlet>

<servlet-mapping>
    <servlet-name>HelloServlet</servlet-name>
    <url-pattern>/hello</url-pattern>
</servlet-mapping>

2.4 Servletのライフサイクル

ライフサイクルの流れ

Servletには明確なライフサイクルがあります。コンテナが自動的に各段階のメソッドを呼び出します。 

stateDiagram-v2
    [*] --> ロード: Tomcat起動時または
初回アクセス時
    ロード --> インスタンス生成
    インスタンス生成 --> init: 初期化処理
1回だけ実行
    init --> サービス中: 準備完了

    サービス中 --> service: リクエスト到着
    service --> doGet: GETリクエスト
    service --> doPost: POSTリクエスト
    doGet --> サービス中: レスポンス返却
    doPost --> サービス中: レスポンス返却

    サービス中 --> destroy: Tomcat停止時
    destroy --> [*]

主なライフサイクルメソッド

  • init(): Servletインスタンス作成後に1回だけ呼ばれる初期化処理
  • service(): リクエストを受け取り、doGet()やdoPost()に振り分ける
  • doGet(): GETリクエストを処理
  • doPost(): POSTリクエストを処理
  • destroy(): Servletが破棄される前に1回だけ呼ばれる終了処理
ライフサイクルメソッドの実装例 
@WebServlet("/lifecycle")
public class LifecycleServlet extends HttpServlet {

    // 初期化処理（最初に1回だけ実行）
    @Override
    public void init() throws ServletException {
        System.out.println("init() が呼ばれました");
    }

    // GETリクエスト処理
    @Override
    protected void doGet(HttpServletRequest request,
                        HttpServletResponse response)
            throws ServletException, IOException {
        System.out.println("doGet() が呼ばれました");
        response.setContentType("text/html; charset=UTF-8");
        PrintWriter out = response.getWriter();
        out.println("<h1>リクエスト処理中</h1>");
    }

    // 終了処理（破棄される前に1回だけ実行）
    @Override
    public void destroy() {
        System.out.println("destroy() が呼ばれました");
    }
}

つまずきポイント

Servletはマルチスレッドで動作するため、複数のリクエストが同時にdoGet()/doPost()を実行することがあります。インスタンス変数を使うと競合が起きる可能性があるため、できるだけローカル変数を使いましょう。

2.5 HTTPステータスコードの理解

ステータスコードとは

HTTPレスポンスには、リクエストの処理結果を示す3桁の数字（ステータスコード）が含まれます。これにより、処理が成功したか、エラーが発生したか、リダイレクトが必要かなどを判断できます。

ステータスコードの分類

  • 2xx（成功）: リクエストが正常に処理された（例: 200 OK, 201 Created）
  • 3xx（リダイレクト）: 別のURLに移動する必要がある（例: 301 Moved Permanently, 302 Found）
  • 4xx（クライアントエラー）: クライアント側に問題がある（例: 400 Bad Request, 404 Not Found）
  • 5xx（サーバーエラー）: サーバー側で問題が発生した（例: 500 Internal Server Error, 503 Service Unavailable）

Servletでのステータスコード設定

Servletでは、HttpServletResponseオブジェクトを使ってステータスコードを設定できます。

ステータスコードの設定例 
// 正常処理（200 OK） - デフォルトなので通常は設定不要
response.setStatus(HttpServletResponse.SC_OK);

// リソースが見つからない（404 Not Found）
response.setStatus(HttpServletResponse.SC_NOT_FOUND);

// サーバー内部エラー（500 Internal Server Error）
response.setStatus(HttpServletResponse.SC_INTERNAL_SERVER_ERROR);

// エラーページにリダイレクト
response.sendError(404, "リソースが見つかりません");

実務での活用

適切なステータスコードを返すことで、ブラウザやAPIクライアントがエラーハンドリングを正しく行えるようになります。特にRESTful APIを作る際には、ステータスコードの正確な使用が重要です。

理解度確認クイズ

  1. Q1. Servletのinit()メソッドは何回呼ばれますか？

    回答を表示

    init()メソッドは、Servletインスタンスが作成された後に1回だけ呼ばれます。その後、複数のリクエストが来てもinit()は再度呼ばれることはありません。

  2. Q2. @WebServlet("/hello")と設定したServletにアクセスするURLは？（コンテキストパスが/myappの場合）

    回答を表示

    http://localhost:8080/myapp/hello となります。URLパターンの前にコンテキストパスが付きます。

  3. Q3. CGIと比べたServletの主な利点は何ですか？

    回答を表示

    Servletは一度読み込まれるとメモリに常駐し、マルチスレッドで複数のリクエストを処理するため、リクエストごとにプロセスを起動するCGIと比べて高速に動作します。

この章のまとめ

  • ServletはHttpServletクラスを継承したJavaクラスで、HTTPリクエストを処理します
  • @WebServletアノテーションでURLマッピングを簡単に設定できます
  • Servletのライフサイクルは init() → service()/doGet()/doPost() → destroy() という流れです
  • Servletはマルチスレッドで動作し、CGIより高速に複数のリクエストを処理できます
