第5章

PreparedStatementの活用

パラメータ化されたクエリを使って、安全で効率的なデータベースアクセスを実現する方法を学びます。

この章で学ぶこと

  • PreparedStatementとStatementの違いとメリット
  • SQLインジェクション攻撃とその対策方法
  • パラメータの設定方法と様々なデータ型への対応
  • PreparedStatementを使った安全なCRUD操作の実装

5.1 PreparedStatementとは

PreparedStatementは、パラメータ化されたSQL文を実行するためのインターフェースです。これまで学んだStatementと比較して、以下のような重要な特徴があります。

StatementとPreparedStatementの違い

Statement: SQL文を文字列として直接組み立てて実行

PreparedStatement: SQL文のテンプレートを事前に準備し、パラメータを後から設定して実行

PreparedStatementの3つのメリット

セキュリティ

SQLインジェクション攻撃を防止

パフォーマンス

SQL文の事前コンパイルで実行速度向上

可読性

コードが読みやすく保守しやすい

StatementとPreparedStatementの処理の違い

flowchart LR
    subgraph Statement処理
        A1[SQL文字列を
動的に組み立て] --> A2[データベースに送信]
        A2 --> A3[SQLパース]
        A3 --> A4[実行計画作成]
        A4 --> A5[実行]
    end

    subgraph PreparedStatement処理

    end
StatementとPreparedStatementの比較 
// ❌ Statementの場合（非推奨）
String name = "田中太郎";
String sql = "SELECT * FROM users WHERE name = '" + name + "'";
Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery(sql);

// ✅ PreparedStatementの場合（推奨）
String sql = "SELECT * FROM users WHERE name = ?";
PreparedStatement pstmt = conn.prepareStatement(sql);
pstmt.setString(1, "田中太郎");  // パラメータを安全に設定
ResultSet rs = pstmt.executeQuery();

5.2 SQLインジェクション攻撃とは

SQLインジェクションは、悪意のあるユーザーが入力値を通じてSQL文を改ざんし、データベースに不正アクセスする攻撃手法です。PreparedStatementを使用することで、この脅威から保護できます。

危険な例：Statementの脆弱性

ユーザー入力を直接SQL文に組み込むと、以下のような攻撃を受けます。

VulnerableExample.java 
// ❌ 危険なコード（SQLインジェクションの脆弱性）
String userId = request.getParameter("userId");  // ユーザー入力
String sql = "SELECT * FROM users WHERE id = " + userId;
Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery(sql);

// 攻撃者が userId に "1 OR 1=1" を入力すると...
// 実行されるSQL: SELECT * FROM users WHERE id = 1 OR 1=1
// → 全てのユーザー情報が取得されてしまう！

SQLインジェクション攻撃の仕組み

sequenceDiagram
    participant U as 攻撃者
    participant W as Webアプリ
    participant DB as データベース

    U->>W: userId = "1 OR 1=1"
(悪意のある入力)
    W->>W: SQL組み立て
"SELECT * FROM users WHERE id = " + userId
    W->>DB: SELECT * FROM users WHERE id = 1 OR 1=1
    Note over DB: OR 1=1 が常に真のため
全レコードを返却
    DB-->>W: 全ユーザーのデータ
    W-->>U: 機密情報が漏洩

安全な実装：PreparedStatementによる対策

PreparedStatementではパラメータが文字列リテラルとして扱われるため、SQL文の構造が変更されません。

SecureExample.java 
// ✅ 安全なコード（SQLインジェクション対策済み）
String userId = request.getParameter("userId");
String sql = "SELECT * FROM users WHERE id = ?";
PreparedStatement pstmt = conn.prepareStatement(sql);
pstmt.setString(1, userId);  // パラメータとして安全に設定
ResultSet rs = pstmt.executeQuery();

// 攻撃者が userId に "1 OR 1=1" を入力しても...
// データベースは "1 OR 1=1" という文字列として検索
// → SQLインジェクションは発生しない！

5.3 PreparedStatementの基本的な使い方

PreparedStatementの基本的な使用手順は、①SQL文テンプレート作成、②PreparedStatementオブジェクト生成、③パラメータ設定、④実行の4ステップです。

基本的な使用手順

flowchart TD
    A[1. SQL文テンプレート作成
プレースホルダ ? を使用] --> B[2. PreparedStatementを生成
conn.prepareStatement]
    B --> C[3. パラメータを設定
setXxx メソッド]
    C --> D{クエリ種別}
    D -->|SELECT| E[executeQuery
→ ResultSet]
    D -->|INSERT/UPDATE/DELETE| F[executeUpdate
→ 影響行数]
    E --> G[5. 結果の処理]
    F --> G
    G --> H[6. リソース解放]
PreparedStatementBasic.java 
// 1. SQL文テンプレートを作成（? がプレースホルダ）
String sql = "SELECT * FROM users WHERE age >= ? AND city = ?";

// 2. PreparedStatementオブジェクトを生成
PreparedStatement pstmt = conn.prepareStatement(sql);

// 3. パラメータを設定（番号は1から始まる）
pstmt.setInt(1, 20);        // 第1パラメータ（age）に整数値を設定
pstmt.setString(2, "東京");  // 第2パラメータ（city）に文字列を設定

// 4. クエリを実行
ResultSet rs = pstmt.executeQuery();

// 5. 結果を処理
while (rs.next()) {
    System.out.println(rs.getString("name"));
}

主要なsetXxxメソッド一覧

メソッド データ型 使用例
setString() 文字列（VARCHAR, CHAR） pstmt.setString(1, "田中")
setInt() 整数（INT） pstmt.setInt(1, 25)
setLong() 長整数（BIGINT） pstmt.setLong(1, 1000000L)
setDouble() 浮動小数点（DOUBLE） pstmt.setDouble(1, 98.5)
setBoolean() 真偽値（BOOLEAN） pstmt.setBoolean(1, true)
setDate() 日付（DATE） pstmt.setDate(1, sqlDate)
setTimestamp() 日時（TIMESTAMP） pstmt.setTimestamp(1, ts)

5.4 PreparedStatementを使ったCRUD操作

ここでは、PreparedStatementを使った4つの基本操作（Create, Read, Update, Delete）の実装例を見ていきましょう。

INSERT例 - データ登録 
// 新規ユーザーを登録
String sql = "INSERT INTO users (name, email, age, city) VALUES (?, ?, ?, ?)";
PreparedStatement pstmt = conn.prepareStatement(sql);

pstmt.setString(1, "佐藤花子");
pstmt.setString(2, "hanako@example.com");
pstmt.setInt(3, 28);
pstmt.setString(4, "大阪");

int rowsInserted = pstmt.executeUpdate();
System.out.println(rowsInserted + "件のデータを登録しました。");
SELECT例 - データ検索 
// 特定の都市に住む20歳以上のユーザーを検索
String sql = "SELECT * FROM users WHERE city = ? AND age >= ?";
PreparedStatement pstmt = conn.prepareStatement(sql);

pstmt.setString(1, "東京");
pstmt.setInt(2, 20);

ResultSet rs = pstmt.executeQuery();
while (rs.next()) {
    System.out.println("名前: " + rs.getString("name"));
    System.out.println("年齢: " + rs.getInt("age"));
}
UPDATE例 - データ更新 
// 特定ユーザーの都市を更新
String sql = "UPDATE users SET city = ? WHERE email = ?";
PreparedStatement pstmt = conn.prepareStatement(sql);

pstmt.setString(1, "横浜");
pstmt.setString(2, "hanako@example.com");

int rowsUpdated = pstmt.executeUpdate();
System.out.println(rowsUpdated + "件のデータを更新しました。");
DELETE例 - データ削除 
// 特定年齢未満のユーザーを削除
String sql = "DELETE FROM users WHERE age < ?";
PreparedStatement pstmt = conn.prepareStatement(sql);

pstmt.setInt(1, 18);

int rowsDeleted = pstmt.executeUpdate();
System.out.println(rowsDeleted + "件のデータを削除しました。");

実習 5-1: PreparedStatementでユーザー検索機能を作成

PreparedStatementを使用して、名前の一部で検索できる機能を実装しましょう。LIKE演算子とワイルドカードを組み合わせて使います。

手順

  1. 1 LIKE演算子を使ったSQL文テンプレートを作成（"SELECT * FROM users WHERE name LIKE ?"）
  2. 2 PreparedStatementオブジェクトを生成
  3. 3 検索キーワードの前後に%を付けてパラメータに設定（部分一致検索）
  4. 4 executeQuery()を実行し、結果を表示

実装例

String keyword = "田中";  // ユーザー入力
String sql = "SELECT * FROM users WHERE name LIKE ?";
PreparedStatement pstmt = conn.prepareStatement(sql);

// ワイルドカードを付けて部分一致検索
pstmt.setString(1, "%" + keyword + "%");

ResultSet rs = pstmt.executeQuery();
while (rs.next()) {
    System.out.println("ID: " + rs.getInt("id"));
    System.out.println("名前: " + rs.getString("name"));
    System.out.println("---");
}

期待される結果

ID: 1
名前: 田中太郎
---
ID: 5
名前: 田中花子
---

ポイント: LIKE演算子を使う際、ワイルドカード（%）はJavaコード側で付与します。SQL文には含めません。

5.5 PreparedStatementのベストプラクティス

PreparedStatementを効果的に使用するためのベストプラクティスを理解しましょう。

推奨される使い方

  • ✅ ユーザー入力を含むSQL文は必ずPreparedStatementを使用
  • ✅ 同じSQL文を繰り返し実行する場合は再利用
  • ✅ try-with-resources文でリソースを適切に管理
  • ✅ パラメータインデックスは1から始まることを意識

避けるべき使い方

  • ❌ テーブル名やカラム名をパラメータで指定
  • ❌ パラメータ設定後にSQL文を再構築
  • ❌ 閉じ忘れによるリソースリーク
  • ❌ 不要な部分でStatementを使い続ける
ベストプラクティス例 
// ✅ try-with-resourcesで自動的にリソース解放
String sql = "SELECT * FROM users WHERE city = ? AND age >= ?";

try (PreparedStatement pstmt = conn.prepareStatement(sql)) {
    // パラメータを設定
    pstmt.setString(1, "東京");
    pstmt.setInt(2, 20);

    // クエリを実行
    try (ResultSet rs = pstmt.executeQuery()) {
        while (rs.next()) {
            System.out.println(rs.getString("name"));
        }
    }
} catch (SQLException e) {
    e.printStackTrace();
}
// pstmt と rs は自動的に close() される

注意：PreparedStatementでできないこと

PreparedStatementのパラメータ（?）は値のみを指定できます。テーブル名やカラム名、SQL構文の一部は指定できません。 

// ❌ これはできない（テーブル名をパラメータにできない）
String sql = "SELECT * FROM ? WHERE id = ?";
pstmt.setString(1, "users");  // エラー！

// ✅ テーブル名を動的にする場合は文字列連結（ただし入力検証が必須）
String tableName = "users";  // ホワイトリストで検証済みの値
String sql = "SELECT * FROM " + tableName + " WHERE id = ?";
pstmt.setInt(1, userId);

理解度確認クイズ

  1. Q1. PreparedStatementを使用する主なメリットを3つ挙げてください。

    回答を表示

    3つのメリット:

    1. セキュリティ: SQLインジェクション攻撃を防止できる
    2. パフォーマンス: SQL文が事前にコンパイルされるため、繰り返し実行時の性能が向上
    3. 可読性: SQL文とパラメータが分離され、コードが読みやすくメンテナンスしやすい

  2. Q2. 次のコードにはどんな問題がありますか？また、どのように修正すべきでしょうか？

    String userId = request.getParameter("id");
String sql = "DELETE FROM users WHERE id = " + userId;
Statement stmt = conn.createStatement();
stmt.executeUpdate(sql);
    回答を表示

    問題点:

    • ユーザー入力を直接SQL文に連結しているため、SQLインジェクションの脆弱性がある
    • 攻撃者が "1 OR 1=1" などを入力すると、全てのユーザーが削除される可能性がある

    修正方法:

    String userId = request.getParameter("id");
String sql = "DELETE FROM users WHERE id = ?";
PreparedStatement pstmt = conn.prepareStatement(sql);
pstmt.setString(1, userId);
pstmt.executeUpdate();

  3. Q3. PreparedStatementのパラメータインデックスは何番から始まりますか？また、間違えやすい点は何ですか？

    回答を表示

    インデックスの開始番号: 1

    PreparedStatementのパラメータインデックスは1から始まります（0ではない）。

    間違えやすい点:

    • Javaの配列は0から始まるため、0から始めてしまうミス
    • パラメータの順序を間違える（SQL文の?の順番とsetXxxメソッドの順番を一致させる必要がある）
    • パラメータの個数とsetXxxメソッドの呼び出し回数が一致していない

  4. Q4. PreparedStatementで部分一致検索（LIKE演算子）を実装する場合、ワイルドカード（%）はどこで付けるべきですか？

    回答を表示

    正解: Javaコード側で付ける

    SQL文テンプレート内ではなく、setStringメソッドで値を設定する際にワイルドカードを含めます。

    // ✅ 正しい方法
String sql = "SELECT * FROM users WHERE name LIKE ?";
pstmt.setString(1, "%" + keyword + "%");

// ❌ 間違った方法
String sql = "SELECT * FROM users WHERE name LIKE '%?%'";
pstmt.setString(1, keyword);  // これでは動作しない

    SQL文内に%を含めると、?がプレースホルダとして認識されなくなります。

第5章のまとめ

  • PreparedStatementは、パラメータ化されたSQL文を実行するための安全で効率的な方法
  • SQLインジェクションは深刻なセキュリティ脅威であり、PreparedStatementで対策可能
  • setXxxメソッドで様々なデータ型をパラメータとして安全に設定できる
  • パラメータインデックスは1から始まり、SQL文の?の順序と対応させる
  • try-with-resourcesを使用してリソースを適切に管理する
