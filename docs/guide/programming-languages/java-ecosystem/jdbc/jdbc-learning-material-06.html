トランザクション処理と例外処理

データの整合性を保つためのトランザクション制御（commit、rollback）、自動コミットの設定、try-catch-finallyを使った適切な例外処理とリソース管理を学びます。

この章で学ぶこと

  • トランザクションとは何か、なぜ必要なのかを理解する
  • commit（確定）とrollback（取り消し）の使い方を学ぶ
  • 自動コミットの設定とその影響を理解する
  • try-catch-finallyを使った適切な例外処理とリソース管理を習得する

6.1 トランザクションとは？

トランザクションとは、複数のデータベース操作を「1つのまとまった処理」として扱う仕組みです。すべての操作が成功したときだけデータベースに反映し、途中で1つでも失敗したら全体を取り消します。

例えば、銀行の送金処理を考えてみましょう。「Aさんの口座から1万円引き出す」「Bさんの口座に1万円入金する」という2つの操作があります。もし片方だけ成功してしまったら、お金が消えたり増えたりして大変ですよね。トランザクションを使えば、両方成功したときだけ確定し、片方が失敗したら両方とも取り消すことができます。

トランザクションの動作イメージ

トランザクションの4つの性質（ACID）

  • 原子性（Atomicity）: すべて成功か、すべて失敗か。中途半端な状態にならない
  • 一貫性（Consistency）: データベースの整合性が保たれる
  • 独立性（Isolation）: 複数のトランザクションが干渉しない
  • 永続性（Durability）: 確定した内容は失われない

6.2 自動コミットの設定

JDBCのConnectionは、デフォルトで自動コミット（Auto-Commit）モードが有効になっています。自動コミットがONの場合、SQL文を1つ実行するたびに自動的にcommitされます。

複数の操作をまとめて1つのトランザクションとして扱いたい場合は、自動コミットをOFFにする必要があります。

自動コミットの設定 
import java.sql.Connection;
import java.sql.DriverManager;

public class TransactionExample {
    public static void main(String[] args) {
        Connection conn = null;

        try {
            // データベースに接続
            conn = DriverManager.getConnection(
                "jdbc:h2:~/test", "sa", ""
            );

            // 自動コミットをOFFにする（トランザクション開始）
            conn.setAutoCommit(false);

            // ここで複数のSQL文を実行...

        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}

初心者向けヒント

setAutoCommit(false)にすると、明示的にcommit()を呼ぶまでデータベースに反映されません。エラーが発生したらrollback()で取り消すことができます。

6.3 commitとrollbackの使い方

トランザクション内のすべての操作が成功したらcommit()を呼んでデータベースに確定します。途中でエラーが発生したらrollback()を呼んですべての操作を取り消します。

トランザクション制御の実装例 
import java.sql.*;

public class BankTransferExample {
    public static void main(String[] args) {
        Connection conn = null;
        PreparedStatement pstmt1 = null;
        PreparedStatement pstmt2 = null;

        try {
            // データベース接続
            conn = DriverManager.getConnection("jdbc:h2:~/test", "sa", "");

            // 自動コミットOFF（トランザクション開始）
            conn.setAutoCommit(false);

            // 操作1: Aさんの口座から1万円引き出し
            String sql1 = "UPDATE accounts SET balance = balance - ? WHERE name = ?";
            pstmt1 = conn.prepareStatement(sql1);
            pstmt1.setInt(1, 10000);  // 引き出し額
            pstmt1.setString(2, "Aさん");
            pstmt1.executeUpdate();

            // 操作2: Bさんの口座に1万円入金
            String sql2 = "UPDATE accounts SET balance = balance + ? WHERE name = ?";
            pstmt2 = conn.prepareStatement(sql2);
            pstmt2.setInt(1, 10000);  // 入金額
            pstmt2.setString(2, "Bさん");
            pstmt2.executeUpdate();

            // すべて成功したのでコミット（確定）
            conn.commit();
            System.out.println("送金が完了しました");

        } catch (SQLException e) {
            // エラー発生時はロールバック（取り消し）
            if (conn != null) {
                try {
                    conn.rollback();
                    System.out.println("エラーが発生したため送金を取り消しました");
                } catch (SQLException ex) {
                    ex.printStackTrace();
                }
            }
            e.printStackTrace();
        }
    }
}

重要な注意点

commit()を呼ぶ前にConnectionをcloseしてしまうと、未確定の変更は自動的にrollbackされてしまいます。必ずcommit()を先に呼んでからcloseしましょう。

6.4 try-catch-finallyを使った例外処理

JDBCを使う際は、必ず例外処理を行う必要があります。データベース接続エラー、SQL構文エラー、ネットワークエラーなど、さまざまなエラーが発生する可能性があるためです。

try-catch-finally文を使って、適切にエラー処理とリソース解放を行いましょう。

完全な例外処理とリソース管理の例 
import java.sql.*;

public class ProperExceptionHandling {
    public static void main(String[] args) {
        Connection conn = null;
        PreparedStatement pstmt = null;

        try {
            // データベース接続
            conn = DriverManager.getConnection("jdbc:h2:~/test", "sa", "");
            conn.setAutoCommit(false);  // トランザクション開始

            // SQL実行
            String sql = "UPDATE products SET stock = stock - ? WHERE id = ?";
            pstmt = conn.prepareStatement(sql);
            pstmt.setInt(1, 5);
            pstmt.setInt(2, 1);
            pstmt.executeUpdate();

            // 成功したのでコミット
            conn.commit();

        } catch (SQLException e) {
            // エラー発生時はロールバック
            if (conn != null) {
                try {
                    conn.rollback();
                } catch (SQLException ex) {
                    ex.printStackTrace();
                }
            }
            e.printStackTrace();

        } finally {
            // リソースを必ず解放（finallyブロック）
            if (pstmt != null) {
                try {
                    pstmt.close();
                } catch (SQLException e) {
                    e.printStackTrace();
                }
            }
            if (conn != null) {
                try {
                    conn.close();
                } catch (SQLException e) {
                    e.printStackTrace();
                }
            }
        }
    }
}

finallyブロックの重要性

finallyブロックは、tryブロックで例外が発生してもしなくても必ず実行されます。データベース接続やStatementなどのリソースは、使い終わったら必ず解放する必要があるため、finallyブロックで確実にclose処理を行います。

6.5 try-with-resourcesを使った簡潔な書き方

Java 7以降では、try-with-resources文を使うことで、リソースの解放を自動的に行うことができます。これにより、finallyブロックを書かなくてもConnectionやStatementが自動的にcloseされます。

try-with-resourcesを使った実装例 
import java.sql.*;

public class TryWithResourcesExample {
    public static void main(String[] args) {
        // try-with-resources: リソースを自動的にcloseする
        try (
            Connection conn = DriverManager.getConnection("jdbc:h2:~/test", "sa", "");
            PreparedStatement pstmt = conn.prepareStatement(
                "UPDATE products SET stock = stock - ? WHERE id = ?"
            )
        ) {
            // 自動コミットOFF
            conn.setAutoCommit(false);

            // SQL実行
            pstmt.setInt(1, 5);
            pstmt.setInt(2, 1);
            pstmt.executeUpdate();

            // コミット
            conn.commit();

        } catch (SQLException e) {
            // エラー処理
            e.printStackTrace();
        }
        // ConnectionとPreparedStatementは自動的にcloseされる
    }
}

try-with-resourcesのメリット

  • コードが短く、読みやすくなる
  • close忘れを防げる
  • 例外が発生しても確実にリソースが解放される

実習：トランザクションを使った商品購入処理

トランザクションを使って、商品の在庫を減らし、注文履歴に記録する処理を実装してみましょう。

実装する処理：

  1. 商品テーブル（products）の在庫数を1減らす
  2. 注文テーブル（orders）に注文履歴を追加する
  3. 両方成功したらcommit、失敗したらrollback

ヒント：

  • setAutoCommit(false)でトランザクション開始
  • 2つのUPDATE/INSERT文を実行
  • 成功したらcommit()、エラー時はrollback()
  • try-with-resourcesを使ってリソース管理

理解度確認クイズ

  1. Q1. トランザクションとは何ですか？なぜ必要なのでしょうか？

    回答を表示

    トランザクションは、複数のデータベース操作を「1つのまとまった処理」として扱う仕組みです。すべての操作が成功したときだけデータベースに反映し、途中で1つでも失敗したら全体を取り消すことで、データの整合性を保つために必要です。

  2. Q2. commit()とrollback()の違いを説明してください。

    回答を表示

    commit(): トランザクション内の変更をデータベースに確定する（反映する）
    rollback(): トランザクション内の変更をすべて取り消す（元に戻す）

  3. Q3. setAutoCommit(false)の役割は何ですか？

    回答を表示

    自動コミットを無効にして、複数のSQL文を1つのトランザクションとして扱えるようにします。これにより、明示的にcommit()を呼ぶまでデータベースに反映されず、エラー時にはrollback()で取り消すことができます。

  4. Q4. try-with-resources文を使うメリットを3つ挙げてください。

    回答を表示

    1. コードが短く、読みやすくなる
    2. close忘れを防げる
    3. 例外が発生しても確実にリソースが解放される

この章のまとめ

  • トランザクションは、複数の操作をまとめて扱い、すべて成功したときだけ確定する仕組み
  • setAutoCommit(false)で自動コミットを無効にし、トランザクションを開始
  • commit()で確定、rollback()で取り消し
  • try-catch-finallyで例外処理とリソース解放を確実に行う
  • try-with-resourcesを使えば、リソースの自動解放が可能
