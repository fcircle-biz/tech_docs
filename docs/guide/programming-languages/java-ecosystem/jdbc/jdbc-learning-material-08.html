DAOパターン入門

データアクセス処理を分離するDAOパターンの基本を学び、保守性と再利用性の高いコード設計の考え方を習得します。

この章で学ぶこと

  • DAOパターンの目的と利点を理解する
  • データアクセス処理とビジネスロジックの分離方法を学ぶ
  • 基本的なDAOクラスの設計と実装ができるようになる
  • 保守性と再利用性の高いコード設計の考え方を理解する

8.1 DAOパターンとは

DAO（Data Access Object）パターンは、データベースへのアクセス処理を専用のクラスに分離するデザインパターンです。これまで学んだJDBCのコードをビジネスロジックから分離することで、保守性と再利用性を高めることができます。

DAOパターンの主な利点

  • 関心の分離: データアクセス処理とビジネスロジックを明確に分離できる
  • 保守性の向上: データベース操作の変更が他のコードに影響しにくい
  • 再利用性: 同じデータアクセス処理を複数の場所で利用できる
  • テスト容易性: DAOをモック化することで単体テストが容易になる

DAOパターンでは、エンティティクラス（データを表現するクラス）、DAOインターフェース（操作を定義）、DAO実装クラス（実際のデータベース操作）の3つが基本的な構成要素となります。

8.2 エンティティクラスの作成

まず、データベースのテーブル構造に対応するエンティティクラスを作成します。エンティティクラスは、データベースの1行のデータを表現するJavaオブジェクトです。

Userテーブルの定義

create_user_table.sql 
-- ユーザーテーブルの作成
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

Userエンティティクラス

User.java 
package com.example.model;

import java.time.LocalDateTime;

/**
 * ユーザーエンティティクラス
 * データベースのusersテーブルに対応
 */
public class User {
    private int id;              // ユーザーID
    private String name;         // ユーザー名
    private String email;        // メールアドレス
    private LocalDateTime createdAt;  // 作成日時

    // デフォルトコンストラクタ
    public User() {
    }

    // 全フィールドを設定するコンストラクタ
    public User(int id, String name, String email, LocalDateTime createdAt) {
        this.id = id;
        this.name = name;
        this.email = email;
        this.createdAt = createdAt;
    }

    // 新規ユーザー作成用コンストラクタ（IDなし）
    public User(String name, String email) {
        this.name = name;
        this.email = email;
    }

    // Getter/Setterメソッド
    public int getId() { return id; }
    public void setId(int id) { this.id = id; }

    public String getName() { return name; }
    public void setName(String name) { this.name = name; }

    public String getEmail() { return email; }
    public void setEmail(String email) { this.email = email; }

    public LocalDateTime getCreatedAt() { return createdAt; }
    public void setCreatedAt(LocalDateTime createdAt) {
        this.createdAt = createdAt;
    }

    @Override
    public String toString() {
        return String.format("User[id=%d, name=%s, email=%s, createdAt=%s]",
                           id, name, email, createdAt);
    }
}

エンティティクラスの設計ポイント

  • • テーブルの各カラムに対応するフィールドを定義する
  • • すべてのフィールドはprivateにし、Getter/Setterでアクセスする
  • • デフォルトコンストラクタを用意する（フレームワークで必要な場合がある）
  • • toString()メソッドをオーバーライドしてデバッグを容易にする

8.3 DAOインターフェースの定義

次に、データアクセス操作を定義するDAOインターフェースを作成します。インターフェースを使用することで、実装の詳細を隠蔽し、異なる実装（例：テスト用のモック）に簡単に切り替えられます。

UserDAO.java 
package com.example.dao;

import com.example.model.User;
import java.util.List;

/**
 * ユーザーデータアクセスインターフェース
 * データベースのusersテーブルへの操作を定義
 */
public interface UserDAO {

    /**
     * 指定されたIDのユーザーを取得
     * @param id ユーザーID
     * @return ユーザーオブジェクト（存在しない場合はnull）
     */
    User findById(int id);

    /**
     * すべてのユーザーを取得
     * @return ユーザーリスト
     */
    List findAll();

    /**
     * メールアドレスでユーザーを検索
     * @param email メールアドレス
     * @return ユーザーオブジェクト（存在しない場合はnull）
     */
    User findByEmail(String email);

    /**
     * 新しいユーザーを登録
     * @param user 登録するユーザーオブジェクト
     * @return 登録されたユーザーのID
     */
    int insert(User user);

    /**
     * ユーザー情報を更新
     * @param user 更新するユーザーオブジェクト
     * @return 更新された行数
     */
    int update(User user);

    /**
     * 指定されたIDのユーザーを削除
     * @param id 削除するユーザーID
     * @return 削除された行数
     */
    int delete(int id);
}

CRUD操作の命名規則

DAOインターフェースのメソッド名は、一般的に以下の規則に従います：

  • • find〜: データの取得（Read）
  • • insert: データの登録（Create）
  • • update: データの更新（Update）
  • • delete: データの削除（Delete）

8.4 DAO実装クラスの作成

DAOインターフェースを実装する実装クラスを作成します。ここに実際のJDBCコードを記述します。

UserDAOImpl.java 
package com.example.dao;

import com.example.model.User;
import java.sql.*;
import java.util.ArrayList;
import java.util.List;

/**
 * UserDAOインターフェースの実装クラス
 */
public class UserDAOImpl implements UserDAO {

    private Connection connection;

    // コンストラクタでデータベース接続を受け取る
    public UserDAOImpl(Connection connection) {
        this.connection = connection;
    }

    @Override
    public User findById(int id) {
        String sql = "SELECT * FROM users WHERE id = ?";

        try (PreparedStatement pstmt = connection.prepareStatement(sql)) {
            pstmt.setInt(1, id);

            try (ResultSet rs = pstmt.executeQuery()) {
                if (rs.next()) {
                    return mapResultSetToUser(rs);
                }
            }
        } catch (SQLException e) {
            e.printStackTrace();
        }
        return null;
    }

    @Override
    public List findAll() {
        List users = new ArrayList<>();
        String sql = "SELECT * FROM users ORDER BY id";

        try (Statement stmt = connection.createStatement();
             ResultSet rs = stmt.executeQuery(sql)) {

            while (rs.next()) {
                users.add(mapResultSetToUser(rs));
            }
        } catch (SQLException e) {
            e.printStackTrace();
        }
        return users;
    }

    @Override
    public User findByEmail(String email) {
        String sql = "SELECT * FROM users WHERE email = ?";

        try (PreparedStatement pstmt = connection.prepareStatement(sql)) {
            pstmt.setString(1, email);

            try (ResultSet rs = pstmt.executeQuery()) {
                if (rs.next()) {
                    return mapResultSetToUser(rs);
                }
            }
        } catch (SQLException e) {
            e.printStackTrace();
        }
        return null;
    }

    @Override
    public int insert(User user) {
        String sql = "INSERT INTO users (name, email) VALUES (?, ?)";

        try (PreparedStatement pstmt = connection.prepareStatement(
                sql, Statement.RETURN_GENERATED_KEYS)) {

            pstmt.setString(1, user.getName());
            pstmt.setString(2, user.getEmail());

            int affectedRows = pstmt.executeUpdate();

            // 自動生成されたIDを取得
            if (affectedRows > 0) {
                try (ResultSet generatedKeys = pstmt.getGeneratedKeys()) {
                    if (generatedKeys.next()) {
                        return generatedKeys.getInt(1);
                    }
                }
            }
        } catch (SQLException e) {
            e.printStackTrace();
        }
        return -1;
    }

    @Override
    public int update(User user) {
        String sql = "UPDATE users SET name = ?, email = ? WHERE id = ?";

        try (PreparedStatement pstmt = connection.prepareStatement(sql)) {
            pstmt.setString(1, user.getName());
            pstmt.setString(2, user.getEmail());
            pstmt.setInt(3, user.getId());

            return pstmt.executeUpdate();
        } catch (SQLException e) {
            e.printStackTrace();
        }
        return 0;
    }

    @Override
    public int delete(int id) {
        String sql = "DELETE FROM users WHERE id = ?";

        try (PreparedStatement pstmt = connection.prepareStatement(sql)) {
            pstmt.setInt(1, id);
            return pstmt.executeUpdate();
        } catch (SQLException e) {
            e.printStackTrace();
        }
        return 0;
    }

    /**
     * ResultSetからUserオブジェクトへのマッピング
     * @param rs ResultSet
     * @return Userオブジェクト
     */
    private User mapResultSetToUser(ResultSet rs) throws SQLException {
        User user = new User();
        user.setId(rs.getInt("id"));
        user.setName(rs.getString("name"));
        user.setEmail(rs.getString("email"));
        user.setCreatedAt(rs.getTimestamp("created_at").toLocalDateTime());
        return user;
    }
}

実装のポイント

  • • mapResultSetToUser(): ResultSetをUserオブジェクトに変換する共通メソッドを作成することでコードの重複を避ける
  • • 例外処理: SQLException を適切にハンドリングし、エラー情報をログに記録する
  • • リソース管理: try-with-resources構文でPreparedStatementとResultSetを自動的にクローズする

8.5 DAOの使用例

作成したDAOを実際に使用するサンプルコードを見ていきましょう。DAOを使用することで、ビジネスロジックからJDBCの詳細を隠蔽できます。

UserService.java 
package com.example.service;

import com.example.dao.UserDAO;
import com.example.dao.UserDAOImpl;
import com.example.model.User;
import java.sql.Connection;
import java.sql.DriverManager;
import java.util.List;

/**
 * ユーザー管理サービスクラス
 * ビジネスロジックを実装
 */
public class UserService {

    private static final String DB_URL = "jdbc:postgresql://localhost:5432/mydb";
    private static final String DB_USER = "myuser";
    private static final String DB_PASSWORD = "mypassword";

    public static void main(String[] args) {
        // データベース接続とDAOの初期化
        try (Connection conn = DriverManager.getConnection(
                DB_URL, DB_USER, DB_PASSWORD)) {

            UserDAO userDAO = new UserDAOImpl(conn);

            // 1. 新しいユーザーを登録
            System.out.println("=== ユーザー登録 ===");
            User newUser = new User("山田太郎", "yamada@example.com");
            int userId = userDAO.insert(newUser);
            System.out.println("登録されたユーザーID: " + userId);

            // 2. IDでユーザーを取得
            System.out.println("\n=== ユーザー取得 ===");
            User user = userDAO.findById(userId);
            System.out.println(user);

            // 3. すべてのユーザーを取得
            System.out.println("\n=== 全ユーザー一覧 ===");
            List users = userDAO.findAll();
            for (User u : users) {
                System.out.println(u);
            }

            // 4. ユーザー情報を更新
            System.out.println("\n=== ユーザー更新 ===");
            user.setName("山田花子");
            user.setEmail("hanako@example.com");
            int updated = userDAO.update(user);
            System.out.println("更新された行数: " + updated);

            // 5. メールアドレスで検索
            System.out.println("\n=== メールアドレスで検索 ===");
            User foundUser = userDAO.findByEmail("hanako@example.com");
            System.out.println(foundUser);

            // 6. ユーザーを削除
            System.out.println("\n=== ユーザー削除 ===");
            int deleted = userDAO.delete(userId);
            System.out.println("削除された行数: " + deleted);

        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}

8.6 DAOパターンの応用

接続管理の改善

実際のアプリケーションでは、接続管理を専用のクラスで行うことが一般的です。

DatabaseConnection.java 
package com.example.util;

import java.sql.Connection;
import java.sql.DriverManager;
import java.sql.SQLException;

/**
 * データベース接続管理クラス
 */
public class DatabaseConnection {

    private static final String DB_URL = "jdbc:postgresql://localhost:5432/mydb";
    private static final String DB_USER = "myuser";
    private static final String DB_PASSWORD = "mypassword";

    /**
     * データベース接続を取得
     * @return データベース接続
     */
    public static Connection getConnection() throws SQLException {
        return DriverManager.getConnection(DB_URL, DB_USER, DB_PASSWORD);
    }
}

DAOファクトリーパターン

DAOインスタンスの生成を一元管理するファクトリークラスを作成すると、より柔軟な設計になります。

DAOFactory.java 
package com.example.factory;

import com.example.dao.UserDAO;
import com.example.dao.UserDAOImpl;
import com.example.util.DatabaseConnection;
import java.sql.Connection;
import java.sql.SQLException;

/**
 * DAOファクトリークラス
 * DAOインスタンスの生成を一元管理
 */
public class DAOFactory {

    /**
     * UserDAOのインスタンスを取得
     * @return UserDAOインスタンス
     */
    public static UserDAO getUserDAO() throws SQLException {
        Connection connection = DatabaseConnection.getConnection();
        return new UserDAOImpl(connection);
    }
}

発展的な設計パターン

実際のプロジェクトでは、以下のような発展的なパターンも使用されます：

  • • コネクションプーリング: 接続を再利用してパフォーマンスを向上
  • • トランザクション管理: 複数のDAO操作を一つのトランザクションで管理
  • • 例外のカスタマイズ: DAOException独自の例外クラスを定義
  • • ジェネリクスの活用: 共通のDAO基底クラスを作成

理解度確認クイズ

  1. Q1. DAOパターンを使用する主な利点を3つ挙げてください。

    回答を表示

    1. 関心の分離: データアクセス処理とビジネスロジックを明確に分離できる
    2. 保守性の向上: データベース操作の変更が他のコードに影響しにくくなる
    3. 再利用性: 同じデータアクセス処理を複数の場所で利用できる
    その他、テスト容易性の向上なども利点として挙げられます。

  2. Q2. DAOパターンにおけるエンティティクラスの役割は何ですか？

    回答を表示

    エンティティクラスは、データベースのテーブル構造に対応し、1行のデータをJavaオブジェクトとして表現します。各カラムに対応するフィールドを持ち、Getter/Setterメソッドでアクセスできるようにします。データの運搬役（DTO: Data Transfer Object）として機能します。

  3. Q3. なぜDAOをインターフェースと実装クラスに分けるのが推奨されるのですか？

    回答を表示

    インターフェースと実装を分離することで、実装の詳細を隠蔽できます。これにより、テスト時にモックオブジェクトに置き換えたり、異なるデータベースに対応した実装に切り替えることが容易になります。また、依存性の注入（DI）などの設計パターンとの相性も良くなります。

  4. Q4. mapResultSetToUser()のような共通メソッドを作成する利点は何ですか？

    回答を表示

    ResultSetをエンティティオブジェクトに変換する処理をメソッドとして共通化することで、コードの重複を避け、保守性が向上します。カラム名の変更やデータ型の変更があった場合も、1箇所を修正するだけで対応できます。また、コードの可読性も向上します。
