8.1 try-with-resources文でリソースを自動解放

これまでの章では、finallyブロックでConnectionやStatementをclose()していましたが、try-with-resources文を使うと、より簡潔で安全にリソースを解放できます。

try-with-resources文とは？

Java 7から追加された機能で、tryブロックが終了すると自動的にリソースをclose()してくれます。AutoCloseableインターフェースを実装したクラス（Connection、Statement、ResultSetなど）で使用できます。

従来の書き方（finallyブロック）

OldStyleConnection.java 
// 従来の書き方：長くて複雑
Connection conn = null;
PreparedStatement pstmt = null;
try {
    conn = DriverManager.getConnection(url, user, password);
    pstmt = conn.prepareStatement("SELECT * FROM users WHERE id = ?");
    pstmt.setInt(1, 1);
    // ...処理...
} catch (SQLException e) {
    e.printStackTrace();
} finally {
    // 手動でclose（nullチェックも必要）
    if (pstmt != null) { try { pstmt.close(); } catch (SQLException e) {} }
    if (conn != null) { try { conn.close(); } catch (SQLException e) {} }
}

推奨: try-with-resources文

ModernStyleConnection.java 
// try-with-resources文：簡潔で安全
try (Connection conn = DriverManager.getConnection(url, user, password);
     PreparedStatement pstmt = conn.prepareStatement("SELECT * FROM users WHERE id = ?")) {

    pstmt.setInt(1, 1);
    ResultSet rs = pstmt.executeQuery();
    // ...処理...

} catch (SQLException e) {
    e.printStackTrace();
}
// tryブロックを抜けると自動的にclose()される！

メリット

  • • コードが短く読みやすい: finallyブロックが不要
  • • close忘れ防止: 自動的に解放される
  • • 例外処理が安全: close()中の例外もキャッチされる

8.2 コネクションプーリングの導入

毎回DriverManager.getConnection()でデータベース接続を作成すると、パフォーマンスが低下します。コネクションプーリングを使えば、接続を再利用して高速化できます。

コネクションプーリングとは？

データベース接続をあらかじめ複数用意しておき、必要なときに貸し出し、使い終わったら返却する仕組みです。接続の作成・破棄コストを削減できます。

コネクションプーリングの仕組み

代表的なコネクションプールライブラリ

HikariCP（推奨）

高速で軽量、Springでも標準採用されている人気のライブラリです。

Apache DBCP

Apache Commonsプロジェクトの一部で、古くから使われている安定したライブラリです。

C3P0

機能が豊富で設定項目が多い、柔軟性の高いライブラリです。

Tomcat JDBC Pool

Apache Tomcatに組み込まれているコネクションプールです。

HikariCPの使用例

HikariCPExample.java 
import com.zaxxer.hikari.HikariConfig;
import com.zaxxer.hikari.HikariDataSource;

// コネクションプールの設定
HikariConfig config = new HikariConfig();
config.setJdbcUrl("jdbc:mysql://localhost:3306/mydb");
config.setUsername("root");
config.setPassword("password");
config.setMaximumPoolSize(10); // 最大接続数

// データソース作成（アプリ起動時に1回だけ）
HikariDataSource dataSource = new HikariDataSource(config);

// 接続を取得（プールから貸出）
try (Connection conn = dataSource.getConnection()) {
    // ...SQL実行...
} // 自動的にプールに返却される

8.3 よくあるエラーと対処法

JDBC開発でよく遭遇するエラーと、その解決方法をまとめました。

トラブルシューティング

ClassNotFoundException: com.mysql.cj.jdbc.Driver

原因: JDBCドライバのJARファイルがクラスパスに含まれていません。

解決方法: JDBCドライバのJARファイルをプロジェクトに追加してください（Eclipseなら「ビルド・パス」→「外部JARの追加」）。

SQLException: Access denied for user 'root'@'localhost'

原因: ユーザー名またはパスワードが間違っています。

解決方法: データベースの認証情報を確認してください。MySQLの場合、rootユーザーのパスワード設定を確認しましょう。

SQLException: No suitable driver found for jdbc:mysql://...

原因: JDBC URLの書式が間違っているか、ドライバが読み込まれていません。

解決方法: JDBC URLの書式を確認（例: jdbc:mysql://localhost:3306/dbname）し、Class.forName()でドライバを明示的に読み込んでください。

SQLException: Communications link failure

原因: データベースサーバーに接続できません（サーバー停止、ネットワーク問題など）。

解決方法: データベースサーバーが起動しているか確認し、ホスト名・ポート番号が正しいかチェックしてください。

SQLException: ResultSet is closed

原因: close()済みのResultSetにアクセスしようとしています。

解決方法: ResultSetはStatementやConnectionがclose()されると自動的にclose()されます。データの取得はtry-with-resourcesブロック内で完了させましょう。

SQLException: Parameter index out of range (1 > number of parameters, which is 0)

原因: PreparedStatementのプレースホルダ（?）が存在しないのに、パラメータを設定しようとしています。

解決方法: SQL文に「?」があるか確認し、setXxx()の引数インデックスが正しいか（1から始まる）チェックしてください。

8.4 パフォーマンス向上のヒント

JDBCプログラムのパフォーマンスを向上させるための実践的なテクニックを紹介します。

1. PreparedStatementを使う

Statementより高速で、SQLインジェクション対策にもなります。特に同じSQLを繰り返し実行する場合は、プリコンパイルされるため大幅に高速化します。

// ループで同じSQLを実行する場合
PreparedStatement pstmt = conn.prepareStatement(
    "INSERT INTO users (name, email) VALUES (?, ?)"
);
for (User user : userList) {
    pstmt.setString(1, user.getName());
    pstmt.setString(2, user.getEmail());
    pstmt.executeUpdate(); // 高速
}

2. バッチ更新を活用する

複数のINSERT/UPDATEをまとめて実行することで、データベースへの通信回数を減らせます。

pstmt = conn.prepareStatement(
    "INSERT INTO users (name, email) VALUES (?, ?)"
);
for (User user : userList) {
    pstmt.setString(1, user.getName());
    pstmt.setString(2, user.getEmail());
    pstmt.addBatch(); // バッチに追加
}
int[] results = pstmt.executeBatch(); // 一括実行！

3. 必要なカラムのみ取得する

SELECT * ではなく、必要なカラムだけを指定することで、データ転送量を削減できます。

// 悪い例
SELECT * FROM users WHERE id = 1;

// 良い例（必要なカラムのみ）
SELECT id, name, email FROM users WHERE id = 1;

4. トランザクションをまとめる

autoCommitをfalseにして、複数の操作を1つのトランザクションにまとめることで、コミット回数を減らせます。

conn.setAutoCommit(false); // 自動コミット無効化
try {
    // 複数のSQL実行
    pstmt1.executeUpdate();
    pstmt2.executeUpdate();
    pstmt3.executeUpdate();
    conn.commit(); // まとめてコミット
} catch (SQLException e) {
    conn.rollback();
}

5. コネクションプーリングを使う

前述のとおり、HikariCPなどのコネクションプールを導入することで、接続の作成・破棄コストを大幅に削減できます。

6. ResultSetのフェッチサイズを調整

大量のデータを取得する場合、setFetchSize()でフェッチサイズを調整すると、メモリ使用量と速度のバランスを最適化できます。

Statement stmt = conn.createStatement();
stmt.setFetchSize(100); // 100件ずつ取得
ResultSet rs = stmt.executeQuery("SELECT * FROM large_table");

8.5 セキュリティのベストプラクティス

安全なJDBCプログラムを書くために、覚えておくべきセキュリティのポイントをまとめました。

必ずPreparedStatementを使う

ユーザー入力をSQL文に直接埋め込むと、SQLインジェクション攻撃を受けます。必ずPreparedStatementのプレースホルダを使いましょう。

データベース認証情報を安全に管理

パスワードをソースコードに直接書かず、環境変数や設定ファイル（.envなど）で管理しましょう。設定ファイルはGitにコミットしないよう注意してください。

最小権限の原則

アプリケーションで使うデータベースユーザーには、必要最低限の権限のみを付与しましょう（例: 読み取り専用アプリならSELECT権限のみ）。

SSL/TLS接続を使用

本番環境では、データベース接続にSSL/TLSを使って通信を暗号化しましょう（JDBC URLに「useSSL=true」を追加）。

理解度確認クイズ

  1. Q1. try-with-resources文を使うメリットを3つ挙げてください。

    回答を表示

    1. コードが短く読みやすい: finallyブロックが不要
    2. close忘れ防止: 自動的にリソースが解放される
    3. 例外処理が安全: close()中の例外も適切にキャッチされる

  2. Q2. コネクションプーリングとは何ですか？どのようなメリットがありますか？

    回答を表示

    コネクションプーリングは、データベース接続をあらかじめ複数用意しておき、必要なときに貸し出し、使い終わったら返却する仕組みです。メリットは、接続の作成・破棄コストを削減し、パフォーマンスを大幅に向上できることです。

  3. Q3. バッチ更新とは何ですか？どのような場合に有効ですか？

    回答を表示

    バッチ更新は、複数のINSERT/UPDATE文をまとめて実行する機能です（addBatch()とexecuteBatch()を使用）。大量のデータを一括登録・更新する場合に有効で、データベースへの通信回数を減らすことで大幅に高速化できます。

  4. Q4. SQLインジェクション対策として最も重要なことは何ですか？

    回答を表示

    PreparedStatementを使い、プレースホルダ（?）でパラメータをバインドすることです。ユーザー入力を文字列連結でSQL文に直接埋め込むと、SQLインジェクション攻撃を受ける危険があります。
