第7章: リリース・運用・保守フェーズ - 本番稼働と継続的改善

7.1 リリース計画とカットオーバー戦略

リリースフェーズでは、開発したシステムを本番環境に移行し、ユーザーが実際に使える状態にします。リスクを最小化するため、綿密な計画が必要です。

カットオーバー方式の種類

方式特徴メリットデメリット
一斉切替 全機能を一度に本番化 移行期間が短い リスクが高い
段階的切替 機能や部門ごとに順次移行 リスク分散、問題の早期発見 並行運用期間が長い
並行稼働 旧システムと新システムを併用 戻せる安心感 運用コストが高い

7.2 デプロイメント手順

本番環境へのデプロイは、手順書に従って慎重に実施します。

デプロイメント手順例
  1. 事前準備
    • 本番環境のバックアップ取得
    • デプロイパッケージの準備
    • ロールバック手順の確認
  2. メンテナンス告知
    • ユーザーへの事前通知
    • サービス停止時間の周知
  3. デプロイ実施
    • アプリケーション停止
    • DBスキーマ変更（必要に応じて）
    • 新バージョンデプロイ
    • 設定ファイル更新
  4. 動作確認
    • スモークテスト実施
    • 主要機能の動作確認
    • ログ確認
  5. サービス再開
    • アプリケーション起動
    • 監視開始
    • ユーザーへの完了通知

7.3 データ移行

既存システムから新システムへデータを移行する場合、データの整合性確保が重要です。

データ移行の流れ

flowchart TD A[旧システムからデータ抽出] --> B[データクレンジング] B --> C[データ変換] C --> D[新システムへロード] D --> E[整合性チェック] E -->|OK| F[移行完了] E -->|NG| G[原因調査・修正] G --> C

7.4 ユーザートレーニングとマニュアル

ユーザーが新システムをスムーズに使えるよう、トレーニングとマニュアルを提供します。

操作マニュアルの種類

  • クイックスタートガイド：最小限の操作で始められる簡易マニュアル
  • ユーザーマニュアル：全機能の詳細な操作手順
  • FAQドキュメント：よくある質問と回答
  • トラブルシューティングガイド：問題発生時の対処法

7.5 運用フェーズと障害対応

システムが本番稼働した後は、安定運用と迅速な障害対応が重要です。

障害対応フロー

flowchart TD A[障害検知] --> B[インシデント登録] B --> C{重要度判定} C -->|Critical| D[緊急対応チーム招集] C -->|High/Medium| E[担当者アサイン] D --> F[原因調査] E --> F F --> G[暫定対応] G --> H[本対応] H --> I[動作確認] I --> J[クローズ] J --> K[再発防止策検討]

インシデント管理の重要度レベル

レベル定義対応時間例
P1（Critical）サービス全停止30分以内システムダウン
P2（High）主要機能停止2時間以内経費申請不可
P3（Medium）一部機能に影響1営業日以内レポート表示異常
P4（Low）軽微な問題次回保守時表示の乱れ

7.6 システム監視

システムの稼働状況を常に監視し、問題を早期発見します。

監視項目

  • 死活監視：サーバー、アプリケーションが稼働しているか
  • 性能監視：CPU、メモリ、ディスク使用率
  • アプリケーション監視：応答時間、エラー率
  • ログ監視：エラーログ、警告ログ
  • セキュリティ監視：不正アクセス、脆弱性攻撃
実習 7-1: アラート設定

以下の状況でアラートを設定してください：

監視項目閾値アラートレベル
CPU使用率80%以上が5分継続Warning
メモリ使用率90%以上Critical
ディスク空き容量10%未満Warning
応答時間5秒以上Warning
エラー率1%以上Critical

7.7 保守開発とバージョン管理

システムリリース後も、機能追加やバグ修正などの保守開発が継続します。

保守の種類

  • 是正保守：バグ修正
  • 予防保守：潜在的な問題の修正
  • 適応保守：環境変化への対応（OS更新等）
  • 完全化保守：機能追加、性能改善

7.8 SLA（サービスレベル契約）

SLAでは、システムの品質基準を明確に定義します。

SLA例
  • 稼働率：平日9:00-18:00の稼働率99.9%以上
  • 応答時間：画面表示3秒以内（95パーセンタイル）
  • 障害対応：P1は30分以内に初動対応
  • データバックアップ：日次バックアップ、30日間保持
  • 復旧時間目標（RTO）：4時間以内
  • 復旧時点目標（RPO）：1時間以内（データ損失許容範囲）
理解度確認クイズ
  1. カットオーバー方式の3種類を挙げてください。
  2. データ移行の主な手順を順番に説明してください。
  3. インシデント管理で最も重要度が高いレベルは何ですか？また、どのような状況ですか？
  4. システム監視で監視すべき項目を5つ挙げてください。
  5. 保守の4種類を挙げてください。
  6. SLAにおけるRTOとRPOの違いを説明してください。
解答例
  1. 一斉切替、段階的切替、並行稼働
  2. データ抽出 → クレンジング → 変換 → ロード → 整合性チェック
  3. P1（Critical）です。サービス全停止など、ビジネスに重大な影響がある状況です。
  4. 例：死活監視、性能監視、アプリケーション監視、ログ監視、セキュリティ監視
  5. 是正保守（バグ修正）、予防保守、適応保守（環境変化対応）、完全化保守（機能追加）
  6. RTO（復旧時間目標）は障害からシステムを復旧させるまでの目標時間。RPO（復旧時点目標）はデータ損失を許容できる時間範囲。
第7章のまとめ
  • リリースは一斉切替、段階的切替、並行稼働などの方式から選択
  • デプロイは手順書に従い、バックアップとロールバック準備を徹底
  • データ移行は抽出→クレンジング→変換→ロードの流れで実施
  • ユーザートレーニングとマニュアルで利用を支援
  • 障害は重要度に応じて迅速に対応
  • システム監視で問題を早期発見
  • 保守開発でシステムを継続的に改善
  • SLAで品質基準を明確化
