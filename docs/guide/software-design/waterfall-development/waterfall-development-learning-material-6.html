6.1 テストの種類と目的

ソフトウェアテストは、システムが要件を満たし、品質基準に達していることを確認するプロセスです。ウォーターフォール開発では、段階的にテストを実施します。

Unit Test"] --> B["結合テスト
Integration Test"] B --> C["システムテスト
System Test"] C --> D["受け入れテスト
Acceptance Test"] A1["個々のモジュール

各テストフェーズの詳細

テスト種類目的実施者テスト観点
単体テスト 個々のモジュールの正常動作を確認 開発者 内部ロジック、分岐条件
結合テスト モジュール間のインターフェースを確認 開発者・テスター データの受け渡し、連携
システムテスト システム全体が要件を満たすか確認 テスター 機能要件、非機能要件
受け入れテスト ユーザーの業務で使えるか確認 ユーザー 業務適合性、使いやすさ

6.2 テストケース設計技法

効率的で網羅的なテストケースを作成するため、様々な設計技法があります。

1. 同値分割（Equivalence Partitioning）

入力値を「同じ挙動をする範囲」でグループ化し、各グループから代表値を選んでテストします。

実習 6-1: 同値分割の例

仕様：経費金額は1円以上1,000,000円以下である必要があります。

グループ範囲代表値期待結果
有効同値クラス1 ≤ 金額 ≤ 1,000,0005,000正常登録
無効同値クラス1金額 < 10エラー
無効同値クラス2金額 > 1,000,0001,000,001エラー

2. 境界値分析（Boundary Value Analysis）

範囲の境界値と、その直前・直後の値をテストします。

境界値テストケース例
テストケース入力値期待結果
TC-0010円（下限-1）エラー
TC-0021円（下限）正常
TC-0032円（下限+1）正常
TC-004999,999円（上限-1）正常
TC-0051,000,000円（上限）正常
TC-0061,000,001円（上限+1）エラー

3. デシジョンテーブル（Decision Table）

複数の条件の組み合わせを表形式で整理し、網羅的にテストします。

デシジョンテーブル例：経費承認ルール
条件ケース1ケース2ケース3ケース4
金額 < 10,000円YesYesNoNo
上長承認済みYesNoYesNo
結果承認却下承認部門長承認へ

6.3 テスト計画書の作成

テスト計画書では、テスト全体の方針、範囲、スケジュール、体制を定義します。

テスト計画書の構成例
  1. テストの目的：品質目標の設定
  2. テスト対象：機能一覧、テスト範囲・対象外
  3. テスト戦略：テストレベル、テスト技法
  4. テスト環境：ハードウェア、ソフトウェア構成
  5. テストスケジュール：各テストフェーズの期間
  6. テスト体制：役割分担、責任者
  7. 合格基準：バグ収束基準、テストカバレッジ目標
  8. リスクと対策：遅延リスク、品質リスクへの対応

6.4 不具合管理

発見した不具合は、管理システムで追跡し、確実に修正します。

不具合管理の流れ

flowchart LR A[不具合発見] --> B[不具合登録] B --> C[優先度設定] C --> D[開発者へアサイン] D --> E[修正] E --> F[再テスト] F -->|OK| G[クローズ] F -->|NG| D

不具合の重要度分類

重要度定義例
Criticalシステムが停止するアプリケーションクラッシュ
High主要機能が使えない経費申請ができない
Medium一部機能に問題レポート出力の一部が誤り
Low使用に支障なし表示の乱れ（データは正しい）

6.5 テストカバレッジ

テストカバレッジは、コードのどれだけがテストされたかを測定する指標です。

カバレッジの種類

  • ステートメントカバレッジ（行カバレッジ）：全コード行のうち実行された割合
  • ブランチカバレッジ：if文などの分岐条件の網羅率
  • 条件カバレッジ：条件式の真/偽の組み合わせの網羅率
カバレッジの注意点

カバレッジ100%でもバグがないとは限りません。カバレッジは品質の一指標であり、テストケースの質も重要です。目標値は一般的に70-80%程度が現実的です。

6.6 テスト自動化

回帰テストやリリース前の確認テストを自動化することで、効率化と品質向上を図ります。

自動化の対象

  • 単体テスト：JUnit、pytest等のフレームワーク
  • API テスト：Postman、RestAssured
  • UIテスト：Selenium、Cypress
  • 性能テスト：JMeter、Gatling
理解度確認クイズ
  1. ウォーターフォール開発における4つのテストレベルを順番に挙げてください。
  2. 同値分割とは何ですか？簡潔に説明してください。
  3. 境界値分析でテストすべき値を3つのタイプで説明してください。
  4. 不具合の重要度はどのように分類されますか？
  5. テストカバレッジ100%でもバグが残る可能性がある理由を説明してください。
解答例
  1. 単体テスト → 結合テスト → システムテスト → 受け入れテスト
  2. 入力値を「同じ挙動をする範囲」でグループ化し、各グループから代表値を選んでテストする技法です。
  3. 境界値（範囲の端の値）、境界値-1（直前の値）、境界値+1（直後の値）をテストします。
  4. 一般的に、Critical（システム停止）、High（主要機能不可）、Medium（一部機能に問題）、Low（使用に支障なし）の4段階で分類します。
  5. カバレッジは「コードが実行されたか」を示すだけで、「正しく動作したか」は保証しません。テストケースの質（観点や入力値）も重要です。
第6章のまとめ
  • テストは単体→結合→システム→受け入れの順で段階的に実施
  • 同値分割、境界値分析、デシジョンテーブルで効率的にテストケース設計
  • テスト計画書で方針、範囲、スケジュール、体制を定義
  • 不具合は管理システムで追跡し、確実に修正
  • テストカバレッジで網羅率を測定するが、100%が目標ではない
  • 回帰テストは自動化で効率化
