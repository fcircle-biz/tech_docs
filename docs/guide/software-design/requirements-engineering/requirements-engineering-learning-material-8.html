8.1 検証と妥当性確認の基本概念

要件の品質を確保するために、検証（Verification）と妥当性確認（Validation）は異なる観点から要件を評価します。両者の違いを理解し、適切に使い分けることが重要です。

検証 vs 妥当性確認
検証（Verification）
  • 質問：「正しく作っているか？」
  • 目的：要件の内部品質を確認
  • 評価観点：完全性、一貫性、実現可能性
  • 手法：レビュー、検査、分析
  • 実施者：開発チーム、品質管理者
妥当性確認（Validation）
  • 質問：「正しいものを作っているか？」
  • 目的：ビジネス価値との適合性を確認
  • 評価観点：適切性、有用性、受容性
  • 手法：プロトタイプ、シミュレーション、ユーザーテスト
  • 実施者：ステークホルダー、エンドユーザー

8.2 要件レビューの実施方法

要件レビューは、複数の関係者が協力して要件の品質を系統的に評価するプロセスです。効果的なレビューには、適切な準備・実施・フォローアップが不可欠です。

レビューの種類と適用場面

1. ウォークスルー（Walkthrough）
  • 目的：要件の共有・理解促進
  • 参加者：要件作成者が主導、関係者が参加
  • 適用場面：初期段階の要件確認、新規参入者への説明
  • メリット：理解促進、知識共有、関係構築
2. インスペクション（Inspection）
  • 目的：欠陥の検出と除去
  • 参加者：専門的知識を持つレビューア
  • 適用場面：クリティカルな要件、最終確認段階
  • メリット：高い欠陥検出率、品質向上
実習 8-1: 要件レビューの計画と実施

オンライン学習システムの認証機能要件をレビューしてみましょう。

レビュー対象要件
【機能要件】
・ユーザーはメールアドレスとパスワードでログインできる
・パスワードは8文字以上である
・ログイン失敗時は適切なエラーメッセージを表示する
・ユーザーはパスワードリセットを要求できる

【非機能要件】
・レスポンス時間は2秒以内
・同時ログイン数は1000ユーザーまで対応
・個人情報は暗号化して保存する
レビューチェックポイント
【完全性の確認】
・必要な機能がすべて含まれているか？
・例外処理やエラーケースは網羅されているか？
・セキュリティ要件は十分か？

【一貫性の確認】
・用語の使い方は統一されているか？
・機能要件と非機能要件に矛盾はないか？
・他の要件との整合性は取れているか？

【実現可能性の確認】
・技術的に実装可能か？
・性能要件は現実的か？
・予算・スケジュール内で実現できるか？
発見された問題点と改善案
【問題点】
1. パスワード強度の要件が不十分
2. 「適切なエラーメッセージ」が曖昧
3. パスワードリセットの詳細手順が不明
4. 暗号化方式が未指定

【改善案】
1. パスワードは8文字以上、英数字・記号を含む
2. エラーメッセージの具体例を明記
3. パスワードリセットのフローを詳細化
4. AES-256による暗号化を明記
期待される結果

欠陥の早期発見と修正により、要件の品質が向上すること

8.3 チェックリストによる品質検証

チェックリストは、要件の品質を体系的に評価するためのツールです。過去の経験や業界のベストプラクティスを基に作成し、継続的に改善していきます。

品質特性別チェックリスト

完全性（Completeness）チェック
  • すべての機能要件が明確に記述されているか？
  • 非機能要件（性能、セキュリティ等）は網羅されているか？
  • インターフェース要件は定義されているか？
  • 制約条件や前提条件は明記されているか？
  • 例外処理やエラーケースは考慮されているか？
正確性（Correctness）チェック
  • 要件は技術的に正確に記述されているか？
  • 数値や計算式に誤りはないか？
  • 参照している標準や規格は最新版か？
  • 業務ルールは正確に反映されているか？
検証可能性（Verifiability）チェック
  • 要件は測定可能な形で記述されているか？
  • 受け入れ基準は明確に定義されているか？
  • テスト方法は実現可能か？
  • 成功・失敗の判定基準は明確か？

8.4 プロトタイプによる妥当性確認

プロトタイプは、要件を早期に可視化し、ステークホルダーからのフィードバックを得るための効果的な手法です。様々な種類のプロトタイプを適切に使い分けることが重要です。

プロトタイプの種類と特徴

graph TD A["プロトタイプの種類"] --> B["水平プロトタイプ
（Horizontal）"] A --> C["垂直プロトタイプ
（Vertical）"] B --> D["UI/UX全体を浅く実装"] B --> E["画面遷移の確認"] B --> F["ユーザビリティ検証"] C --> G["特定機能を深く実装"] C --> H["技術的実現性確認"] C --> I["性能特性の評価"]
プロトタイプ活用の成功要因
  • 目的の明確化：何を検証したいかを事前に定義
  • 適切なレベル設定：目的に応じた詳細度の調整
  • 早期フィードバック：短いサイクルでの確認・改善
  • 廃棄前提：本格実装とは別物として扱う
  • 関係者の参加：実際のユーザーによる評価
実習 8-2: ペーパープロトタイプによるUI検証

モバイルアプリの注文画面をペーパープロトタイプで検証してみましょう。

検証対象の要件
  • 商品選択から注文完了まで3画面以内
  • 片手操作で完了可能
  • 高齢者でも直感的に操作可能
  • 注文内容の確認が容易
プロトタイプ評価項目
【使いやすさ評価】
・操作手順は直感的か？
・必要な情報は適切に配置されているか？
・エラーを起こしにくい設計か？
・画面遷移は分かりやすいか？

【技術的実現性】
・既存システムとの連携は可能か？
・レスポンス時間の要求は満たせるか？
・異なるデバイスサイズに対応できるか？

【ビジネス価値】
・ユーザーの目標達成を支援しているか？
・競合他社との差別化ポイントはあるか？
・売上向上に寄与する要素はあるか？
期待される結果

要件の妥当性が確認され、必要に応じて要件の修正・追加が行われること

8.5 要件の一貫性と整合性の検証

複数の要件間の矛盾や不整合は、開発後期に発覚すると大きなコストとなります。体系的なチェック手法により、早期に問題を発見・解決します。

一貫性チェックの観点

1. 用語の一貫性
  • 同じ概念に対して一貫した用語を使用
  • 用語集の整備と参照の徹底
  • 略語・同義語の統一的な扱い
チェック例
【問題のある記述】
要件A: ユーザーがシステムにログインする
要件B: 利用者認証機能を実装する
要件C: 会員がサインインできる

【改善後】
すべて「ユーザー」「ログイン」で統一
2. 制約条件の整合性
  • 性能要件間の矛盾チェック
  • 技術制約とビジネス要件の適合性
  • 法的要件との整合性確認
チェック例
【矛盾の例】
要件A: レスポンス時間1秒以内
要件B: すべての通信はSSL暗号化
要件C: 同時ユーザー数10,000人対応

→ 暗号化処理により1秒以内の応答が困難
→ 性能とセキュリティのバランス調整が必要

8.6 トレーサビリティ分析

要件トレーサビリティは、要件がシステム設計・実装・テストに正しく反映されていることを確認する手法です。双方向のトレースにより網羅性を保証します。

（Forward Traceability）"] A --> G F --> H["後方トレース
（Backward Traceability）"] H --> A
実習 8-3: トレーサビリティマトリクスの作成

ECサイトの検索機能について、要件から実装までのトレーサビリティを確認してみましょう。

トレーサビリティ分析結果
【前方トレース分析】
・UR-001 → SR-001, SR-002 → DS-001, DS-002 → IM-001, IM-002 → TC-001, TC-002
・すべての要件が下流工程に反映されている ✓

【後方トレース分析】  
・TC-001, TC-002 ← IM-001, IM-002 ← DS-001, DS-002 ← SR-001, SR-002 ← UR-001
・すべての実装・テストが要件に根拠を持つ ✓

【欠陥・リスクの特定】
・性能要件（応答時間）のテストケースが不足
・エラーハンドリングの設計仕様が未完成
期待される結果

要件の漏れや余剰実装が特定され、品質とコストの最適化が図られること

理解度確認クイズ
  1. 検証（Verification）と妥当性確認（Validation）の違いを、具体例を用いて説明してください。
    ヒント：「正しく作る」と「正しいものを作る」の観点で考えてみましょう
  2. 要件レビューでウォークスルーとインスペクションを使い分ける基準を3つ挙げてください。
    ヒント：目的、参加者、実施タイミングの違いを考えてみましょう
  3. 以下の要件から一貫性の問題を見つけ、改善案を提示してください：
    「利用者は商品を注文できる」「ユーザーは決済を実行する」「顧客の購入履歴を管理する」
    ヒント：用語の統一性に注目してください
  4. プロトタイプによる妥当性確認を成功させるための重要なポイントを5つ挙げてください。
    ヒント：目的設定、実装レベル、フィードバック、関係者参加の観点で考えてみましょう
  5. 要件トレーサビリティマトリクスで「孤児要件」と「幽霊要件」が発見された場合の対処法をそれぞれ説明してください。
    ヒント：前方トレースと後方トレースで発見される問題の性質を考えてみましょう
