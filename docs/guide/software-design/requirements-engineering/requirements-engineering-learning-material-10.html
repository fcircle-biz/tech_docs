実践的な要件定義書作成演習

この章で学ぶこと
  • 実践的なケーススタディを通じた要件工学プロセスの統合的理解
  • ECサイト・在庫管理・予約システムの業務分析と要件抽出
  • 業界標準テンプレートに基づく要件定義書の構造化作成
  • ステークホルダー分析から要件検証までの一貫したプロセス実践

10.1 総合演習の概要

この章では、これまで学習した要件工学の技法を統合し、実際のシステム開発を想定した包括的な演習を行います。3つの異なる業務領域のケーススタディを通じて、理論を実践に応用する能力を養います。

演習で習得するスキル
  • ビジネス理解力：業務プロセスの分析と課題特定
  • 要件獲得力：ステークホルダーから効果的な要件引き出し
  • 要件分析力：優先度付けと実現可能性の判断
  • 文書化能力：明確で検証可能な要件記述
  • 品質管理力：レビューと改善のサイクル運用
  • 変更対応力：変更要求への適切な対処

10.2 ケーススタディ1: ECサイトリニューアルプロジェクト

実習 10-1: ECサイトの要件定義書作成

中堅アパレル企業のECサイトリニューアルプロジェクトの要件定義書を作成してみましょう。

プロジェクト背景
【企業情報】
・業種: アパレル（レディース・メンズ）
・従業員: 120名（本社50名、店舗70名）
・年商: 15億円（EC売上30%、実店舗70%）

【現状の課題】
・既存ECサイトの売上伸び悩み（前年比-5%）
・モバイル対応不十分（モバイル流入60%だが離脱率50%）
・在庫連携の問題（実店舗との在庫データ不一致）
・顧客管理の分散（ECと店舗の顧客情報が別管理）

【プロジェクト目標】
・EC売上を年商の50%に向上（5億円→7.5億円）
・モバイル離脱率を30%以下に改善
・オムニチャネル化の実現（店舗・EC・SNS連携）
・運用コストの20%削減
ステークホルダー分析
ステークホルダー 役割・関心事 影響力 期待・要求
代表取締役 経営戦略・ROI 高 売上向上、コスト削減
マーケティング部 集客・ブランディング 高 SEO向上、SNS連携、分析機能
店舗スタッフ 接客・在庫管理 中 在庫確認、顧客情報参照
EC運用担当 日常運用・顧客対応 中 管理画面の使いやすさ、効率化
顧客 購買体験・満足度 高 使いやすさ、豊富な商品情報
主要機能要件
【FR-001】商品カタログ機能
・商品の階層的カテゴリー分類（大分類→中分類→小分類）
・多角度商品画像表示（最大10枚）
・商品詳細情報（素材、サイズ展開、ケア方法等）
・在庫状況のリアルタイム表示
・関連商品・コーディネート提案機能

【FR-002】検索・絞り込み機能
・キーワード検索（商品名、ブランド、カテゴリー）
・ファセット検索（価格帯、色、サイズ、ブランド等）
・検索結果の並び替え（人気順、価格順、新着順等）
・検索履歴とおすすめ機能

【FR-003】ショッピングカート・決済機能
・商品のカート追加・編集・削除
・複数決済手段対応（クレジット、代引き、電子マネー等）
・配送オプション選択（日時指定、店舗受取等）
・注文確認・キャンセル機能

【FR-004】会員機能
・会員登録・ログイン（SNSアカウント連携対応）
・マイページ（注文履歴、お気に入り、配送先管理）
・ポイントシステム
・メール配信管理
非機能要件
【NFR-001】性能要件
・ページ表示速度: 3秒以内（95%のページで）
・同時接続数: 500ユーザー（通常時）、2000ユーザー（セール時）
・サーバー応答時間: 1秒以内（95%のリクエストで）

【NFR-002】ユーザビリティ要件
・モバイルファーストデザイン
・アクセシビリティ: WCAG 2.1 AA準拠
・直感的な操作（3クリック以内で目的達成）
・多言語対応（日本語・英語）

【NFR-003】セキュリティ要件
・SSL/TLS 1.3による通信暗号化
・個人情報の暗号化保存
・PCI DSS準拠（カード情報処理）
・定期的なセキュリティ診断実施

【NFR-004】可用性・信頼性要件
・システム稼働率: 99.9%以上
・データバックアップ: 日次フルバックアップ
・障害復旧時間: 4時間以内
・データ損失: 1時間分以内

10.3 ケーススタディ2: 在庫管理システム

実習 10-2: 製造業向け在庫管理システム要件定義

中小製造業向けの在庫管理システムの要件を定義してみましょう。

システム概要
【対象業界】電子部品製造業
【導入規模】従業員80名（製造50名、管理30名）
【管理対象】原材料500品目、製品200品目、仕掛品300品目

【業務フロー】
1. 受注管理 → 生産計画立案
2. 資材所要量計算 → 発注処理
3. 入庫検査 → 在庫登録
4. 生産指示 → 材料払出
5. 製品完成 → 製品在庫登録
6. 出荷指示 → 在庫更新
主要要件（EARS記法での記述例）
【在庫管理要件】
REQ-001: When 材料を入庫する時, the system shall 入庫数量を在庫テーブルに加算する
REQ-002: When 在庫数量が安全在庫を下回る時, the system shall 発注推奨アラートを表示する
REQ-003: If 製品在庫が欠品状態の場合, the system shall 受注時に納期遅延警告を出力する

【トレーサビリティ要件】
REQ-004: When ロット番号で検索する時, the system shall 使用した原材料ロットを表示する
REQ-005: When 不良品が発生した場合, the system shall 同一ロット製品リストを出力する

【分析・レポート要件】
REQ-006: The system shall 月次在庫回転率レポートを自動生成する
REQ-007: When 棚卸を実施する時, the system shall 理論在庫と実地在庫の差異リストを出力する

10.4 ケーススタディ3: 医療予約システム

実習 10-3: クリニック向けオンライン予約システム

地域医療クリニックのオンライン予約システムを設計してみましょう。

ユーザーストーリー例
【患者向け機能】
US-001: As a 患者, I want 診療科から医師を選択したい, So that 専門性の高い診療を受けられる
US-002: As a 初診患者, I want Web問診票を事前入力したい, So that 来院時の待ち時間を短縮できる
US-003: As a リピート患者, I want 前回と同じ医師を予約したい, So that 継続的な診療を受けられる

【医師向け機能】  
US-004: As a 医師, I want 自分のスケジュールを調整したい, So that プライベート時間を確保できる
US-005: As a 医師, I want 患者の過去診療記録を確認したい, So that 適切な診断・治療ができる

【受付スタッフ向け機能】
US-006: As a 受付スタッフ, I want 予約変更・キャンセルを代行したい, So that 電話対応を効率化できる
US-007: As a 受付スタッフ, I want 当日受付と予約の調整をしたい, So that 待ち時間を最小化できる
ユースケース記述例
【ユースケース名】診療予約を取る
【主アクター】患者
【副アクター】予約管理システム、カレンダーシステム
【事前条件】
・患者がシステムにログインしている
・希望する診療科が決定している
【事後条件】
・予約が正常に登録される
・患者に確認メールが送信される
・医師のスケジュールに反映される

【基本フロー】
1. 患者が診療科を選択する
2. システムが対応医師リストを表示する
3. 患者が医師を選択する
4. システムが医師の空き時間を表示する
5. 患者が希望日時を選択する
6. 患者が予約内容を確認する
7. 患者が予約を確定する
8. システムが予約確認画面を表示する

【代替フロー】
A1: 希望日時が満席の場合
  5a. システムが「満席」メッセージを表示
  6a. 代替候補日時を提示
  7a. フロー3に戻る

【例外フロー】
E1: システム障害が発生した場合
  E1a. エラーメッセージ表示
  E1b. 電話予約への誘導メッセージ表示

10.5 業界標準テンプレートの活用

実際のプロジェクトでは、IEEE 830やISO/IEC/IEEE 29148などの国際標準に基づいたテンプレートを使用します。標準的な構造を理解し、プロジェクトに適合させることが重要です。

要件定義書の標準構造

IEEE 830に基づく要件定義書構成
  1. 序論（Introduction）
    • 目的（Purpose）
    • 範囲（Scope）
    • 定義・略語・省略形（Definitions, acronyms, and abbreviations）
    • 参考文献（References）
    • 概要（Overview）
  2. 全体的な記述（Overall description）
    • 製品の展望（Product perspective）
    • 製品機能（Product functions）
    • ユーザー特性（User characteristics）
    • 制約（Constraints）
    • 前提条件と依存関係（Assumptions and dependencies）
  3. 特定要件（Specific requirements）
    • 機能要件（Functional requirements）
    • 非機能要件（Non-functional requirements）
    • インターフェース要件（Interface requirements）
  4. 付録（Appendices）

10.6 要件品質の評価とレビュー

作成した要件定義書の品質を客観的に評価し、継続的な改善を図ります。

実習 10-4: 要件品質評価チェックリスト

作成した要件定義書を以下のチェックリストで評価してみましょう。

品質評価チェックリスト
評価項目 チェック内容 評価 改善点
完全性 必要な要件がすべて含まれているか ○/△/×
正確性 要件の記述に技術的誤りがないか ○/△/×
一貫性 要件間に矛盾や不整合がないか ○/△/×
明確性 要件が曖昧さなく記述されているか ○/△/×
検証可能性 要件の実現を確認できる基準があるか ○/△/×
追跡可能性 要件の出所と実装の対応が明確か ○/△/×
総合評価基準
  • 優秀（90%以上○）：本格実装に移行可能
  • 良好（70-89%○）：軽微な修正後に実装移行
  • 要改善（50-69%○）：重要な問題の解決が必要
  • 要再作成（50%未満○）：大幅な見直しが必要

10.7 実践的なドキュメント作成技術

読みやすく、保守しやすい要件定義書を作成するための実践的なテクニックを身につけます。

効果的な文書作成のコツ
1. 構造化された記述
  • 階層的な章立て（1. 1.1 1.1.1）
  • 統一された番号体系（REQ-001, UC-001, NFR-001）
  • 相互参照の活用（「REQ-001を参照」）
2. 視覚的な表現
  • 図表の効果的活用（ユースケース図、画面遷移図）
  • 表による整理（機能一覧、要件マトリクス）
  • 色分けによる重要度表示
3. メンテナンス性の確保
  • 変更履歴の記録
  • 承認者・承認日の明記
  • 次回レビュー予定日の設定

    ヒント：抽象度と具体性のレベルの違いを考えてください

    ヒント：優先度付けの手法と合意形成のプロセスを考えてみましょう
