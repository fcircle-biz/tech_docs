第7章: 非機能要件の定義と管理

非機能要件の定義と管理

この章で学ぶこと
  • 非機能要件の重要性とシステム品質への影響
  • ISO/IEC 25010品質モデルによる体系的品質分類
  • 性能要件（レスポンスタイム・スループット・リソース使用率）の定義
  • セキュリティ要件の体系的整理と測定可能な指標設定
  • 可用性・信頼性要件の定量的表現方法
  • 使いやすさ（ユーザビリティ）要件の客観的評価指標
  • 保守性・移植性要件の実践的定義方法
  • 非機能要件のトレードオフ分析と最適化

7.1 非機能要件の基本概念と重要性

非機能要件は、システムが「何をするか」ではなく「どのように動作するか」を定義する要件です。システムの品質特性を決定し、ユーザー体験とビジネス成果に直接的な影響を与えます。

非機能要件が軽視される理由と問題
  • 目に見えにくい：機能要件と違い、動作している様子が分かりにくい
  • 測定が困難：「使いやすい」「高速」などの曖昧な表現が多用される
  • 後回しにされがち：機能実装を優先し、品質は後で改善しようとする
  • コスト増加要因：品質向上には追加的な設計・実装・テストが必要

結果として発生する問題：

  • システム稼働後のパフォーマンス問題
  • セキュリティ脆弱性の発覚
  • ユーザビリティの低さによる利用率低下
  • 運用・保守コストの予想外の増大

非機能要件の経済的インパクト

品質の経済的価値
  • 性能向上：1秒のレスポンス改善で7%のコンバージョン向上（Amazon調査）
  • 可用性向上：99.9%から99.99%への改善で大幅な収益機会損失防止
  • セキュリティ：データ漏洩1件あたり平均4.45百万ドルの損失（IBM調査）
  • ユーザビリティ：優れたUI/UXにより売上200%向上の事例多数

7.2 ISO/IEC 25010品質モデル

ISO/IEC 25010は、ソフトウェア品質を体系的に分類・評価するための国際標準です。8つの主特性と31の副特性で品質を包括的にカバーします。

Usability"] A --> F["信頼性
Reliability"] A --> G["セキュリティ

Maintainability"] A --> I["移植性
Portability"]
8つの品質特性の詳細
1. 機能適合性（Functional Suitability）
  • 機能完全性
  • 機能正確性
  • 機能適切性
2. 性能効率性（Performance Efficiency）
  • 時間効率性
  • 資源効率性
  • 容量満足性
3. 互換性（Compatibility）
  • 共存性
  • 相互運用性
4. 使いやすさ（Usability）
  • 認識性
  • 習得性
  • 運用性
  • 利用者エラー防止性
  • 利用者インターフェース美観性
  • アクセシビリティ
5. 信頼性（Reliability）
  • 成熟性
  • 可用性
  • 障害許容性
  • 回復性
6. セキュリティ（Security）
  • 機密性
  • 完全性
  • 否認防止性
  • 責任追跡性
  • 真正性
7. 保守性（Maintainability）
  • モジュール性
  • 再利用性
  • 解析性
  • 修正性
  • 試験性
8. 移植性（Portability）
  • 適応性
  • 設置性
  • 置換性

7.3 性能要件の定義と測定

性能要件は、システムの応答性・処理能力・資源使用効率を定量的に定義します。ビジネス目標と技術制約の両方を考慮した現実的な目標設定が重要です。

性能要件の3つの側面

1. レスポンスタイム（Response Time）
  • 定義：ユーザーの操作から結果表示までの時間
  • 測定指標：平均レスポンス時間、95パーセンタイル、最大レスポンス時間
  • 業界基準：Webページ読み込み3秒以内、API応答200ms以内
  • 記述例：「商品検索結果の表示は、検索ボタン押下から2秒以内に完了すること」
2. スループット（Throughput）
  • 定義：単位時間あたりの処理量
  • 測定指標：TPS（Transactions Per Second）、RPS（Requests Per Second）
  • 考慮要素：同時ユーザー数、ピーク時の負荷
  • 記述例：「同時ユーザー数1,000人でも1秒間に100件の注文処理が可能であること」
3. リソース使用率（Resource Utilization）
  • 定義：CPU・メモリ・ディスク・ネットワークの使用効率
  • 測定指標：平均使用率、ピーク使用率、使用量の推移
  • 最適化観点：コスト効率性、拡張性、安定性
  • 記述例：「通常負荷時のCPU使用率は70%以下、メモリ使用率は80%以下であること」
実習 7-1: ECサイトの性能要件定義

月間100万ユーザーのECサイトの性能要件を定義してみましょう。

ビジネス要求
  • ピーク時（セール期間）は通常の5倍のアクセス
  • カート離脱率の軽減が最優先
  • 検索・閲覧の快適性が売上に直結
  • インフラコストは現在の1.5倍まで許容
性能要件の定義例
【レスポンスタイム要件】
・トップページ表示: 1.5秒以内（95%のリクエストで）
・商品検索結果表示: 2秒以内（平均）
・商品詳細ページ表示: 1秒以内（平均）
・カート追加処理: 0.5秒以内（95%のリクエストで）
・決済処理完了: 5秒以内（最大）

【スループット要件】  
・通常時: 500TPS（1日平均）
・ピーク時: 2,500TPS（セール開始時）
・同時接続ユーザー: 10,000人まで対応

【リソース使用率要件】
・CPU使用率: 平均60%以下、ピーク時80%以下
・メモリ使用率: 平均70%以下、ピーク時85%以下  
・データベース応答時間: 平均50ms以下
・CDN経由での静的コンテンツ配信率: 90%以上
期待される結果

測定可能で実現可能な性能目標が設定され、ビジネス価値と技術制約のバランスが取れていること

7.4 セキュリティ要件の体系的整理

セキュリティ要件は、情報資産を脅威から保護するための包括的な対策を定義します。CIA（機密性・完全性・可用性）の3要素を中心に体系的に整理します。

セキュリティ要件の分類と具体例

graph TD A["セキュリティ要件"] --> B["機密性
Confidentiality"] A --> C["完全性
Integrity"] A --> D["可用性
Availability"] A --> E["認証
Authentication"] A --> F["認可
Authorization"] A --> G["否認防止
Non-repudiation"] B --> B1["暗号化"] B --> B2["アクセス制御"] C --> C1["データ整合性"] C --> C2["改ざん検知"] D --> D1["システム可用性"] D --> D2["DDoS対策"]
1. 認証（Authentication）要件
・パスワードは8文字以上、英数字・記号を含むこと
・多要素認証（MFA）をすべての管理者アカウントに適用すること
・ログイン試行失敗は5回でアカウントロックすること
・パスワード有効期限は90日とし、過去12回と同一は禁止すること
・生体認証（指紋・顔認証）をモバイルアプリで利用可能にすること
2. 認可（Authorization）要件
・ロールベースアクセス制御（RBAC）を実装すること
・最小権限の原則に従い、必要最小限の権限のみを付与すること  
・管理者権限の操作はすべて承認ワークフローを経ること
・個人情報へのアクセスは業務上必要な場合に限定すること
・API呼び出しはOAuth 2.0による認可を必須とすること
3. データ保護要件
・個人情報はAES-256で暗号化して保存すること
・データベース通信はSSL/TLS 1.3で暗号化すること
・クレジットカード情報は保存せず、PCI DSS準拠の外部サービスを利用すること
・機密データのバックアップも同等レベルで暗号化すること
・データの国外転送は暗号化と法的要件遵守を前提とすること

7.5 可用性・信頼性要件

可用性と信頼性は、システムが継続的に期待される機能を提供できる能力を表します。ビジネス継続性と直結するため、定量的な目標設定が不可欠です。

可用性の測定指標

SLA（Service Level Agreement）の設定
可用性レベル 年間ダウンタイム 月間ダウンタイム 適用システム例
99.9%（Three 9s） 8時間46分 43分 社内システム
99.95% 4時間23分 22分 ECサイト
99.99%（Four 9s） 52分 4分 金融システム
99.999%（Five 9s） 5分 26秒 ミッションクリティカル

信頼性要件の定義方法

実習 7-2: オンライン銀行システムの信頼性要件

オンライン銀行システムの信頼性要件を定義してみましょう。

ビジネス要求
  • 金融庁の規制要件を満たす
  • 顧客の信頼維持が最重要
  • システム停止は重大な損失
  • 24時間365日の運用が前提
信頼性要件の定義例
【可用性要件】
・システム全体の可用性: 99.99%以上（年間52分以下のダウンタイム）
・計画停止: 月1回、深夜2-4時の2時間以内
・緊急メンテナンス: 年間4時間以内

【MTBF（平均故障間隔）要件】
・ハードウェア故障: 8,760時間以上（1年以上）
・ソフトウェア障害: 720時間以上（1ヶ月以上）

【MTTR（平均復旧時間）要件】  
・システム全停止: 30分以内に復旧
・部分機能停止: 15分以内に復旧
・データ不整合検知時: 5分以内にサービス停止

【データ整合性要件】
・金融取引データの不整合率: 0.0001%以下
・バックアップからの完全復旧: 4時間以内
・データ損失: 完全にゼロ（RPO = 0）
期待される結果

業界標準と規制要件を満たし、ビジネス価値を保護する明確な信頼性目標

7.6 使いやすさ（ユーザビリティ）要件

ユーザビリティ要件は、システムの使いやすさを客観的・定量的に評価できる指標で定義します。主観的な表現を避け、測定可能な基準を設定することが重要です。

ユーザビリティの5つの構成要素

1. 学習しやすさ（Learnability）
  • 測定方法：新規ユーザーが主要機能を習得するまでの時間
  • 例：「初回利用者が商品注文を完了するまで5分以内」
2. 効率性（Efficiency）
  • 測定方法：熟練ユーザーのタスク完了時間
  • 例：「既存顧客の再注文は3クリック以内で完了可能」
3. 記憶しやすさ（Memorability）
  • 測定方法：一定期間後の操作成功率
  • 例：「1週間後でも90%のユーザーが補助なしで操作可能」
4. エラー防止・回復（Error Prevention & Recovery）
  • 測定方法：エラー発生率とエラーからの回復時間
  • 例：「入力エラーの90%は事前検証で防止、残り10%は1クリックで修正可能」
5. 満足度（Satisfaction）
  • 測定方法：ユーザビリティテスト、NPS（Net Promoter Score）
  • 例：「ユーザビリティスコア80点以上、NPS50以上を達成」

7.7 保守性・移植性要件

保守性と移植性は、システムの長期的な価値を決定する重要な品質特性です。技術の進歩や事業環境の変化に柔軟に対応できるシステム設計を支援します。

保守性要件の具体例

コードの保守性
・コードカバレッジ: 90%以上を維持すること
・サイクロマティック複雑度: 関数あたり10以下とすること  
・コード重複率: 5%以下を維持すること
・新機能追加時の既存テスト成功率: 95%以上とすること
・バグ修正の平均所要時間: 4時間以内とすること
移植性要件
・対応OS: Windows 10/11, macOS 12+, Ubuntu 20.04+
・ブラウザ対応: Chrome, Firefox, Safari, Edge（各々最新版含む過去2バージョン）
・モバイル対応: iOS 14+, Android 10+
・クラウド移行時間: 既存環境から新環境へ24時間以内
・データ移行時の互換性: 100%保証（データ損失ゼロ）

7.8 非機能要件のトレードオフ分析

異なる品質特性間にはトレードオフの関係があります。すべての品質を最高レベルで実現することは現実的ではないため、ビジネス価値に基づく優先順位付けと最適化が必要です。

トレードオフ最適化の実践例

実習 7-3: ソーシャルメディアアプリの品質バランス

ソーシャルメディアアプリの非機能要件のトレードオフを分析し、最適なバランスを見つけてみましょう。

トレードオフの例
【性能 vs セキュリティ】
問題: 暗号化処理がレスポンス時間を0.3秒増加
解決: 
・重要データは完全暗号化（決済情報）
・一般データは軽量暗号化（投稿内容）
・CDN活用でネットワーク遅延を0.5秒短縮

【使いやすさ vs セキュリティ】  
問題: 多要素認証がユーザー体験を阻害
解決:
・デバイス認証で認証頻度を削減
・生体認証で認証時間を短縮  
・リスクベース認証で必要時のみMFA

【可用性 vs コスト】
問題: 99.99%可用性は現在の3倍のコスト
解決:
・営業時間（9-21時）は99.99%
・深夜時間（21-9時）は99.9%  
・自動スケーリングで需要に応じたリソース調整
期待される結果

ビジネス価値を最大化する現実的な品質バランスと、その根拠となる定量的分析

理解度確認クイズ
  1. ISO/IEC 25010品質モデルの8つの主特性をすべて挙げ、ECサイトにとって最も重要な3つを選んで理由を説明してください。
    ヒント：ECサイトのビジネス特性（売上・信頼・利便性）を考慮してください
  2. 「Webサイトは高速でなければならない」という曖昧な要件を、測定可能な性能要件に変換してください。レスポンスタイム、スループット、リソース使用率の観点を含めて記述してください。
    ヒント：具体的な数値、測定条件、測定指標を明確に設定しましょう
  3. オンライン医療予約システムのセキュリティ要件を、CIA（機密性・完全性・可用性）の観点から具体的に定義してください。
    ヒント：医療情報の特殊性と法的要件（個人情報保護法、医師法等）を考慮してください
  4. 99.9%と99.99%の可用性の違いが、ビジネスに与える影響を具体的な数値とともに説明してください。
    ヒント：年間ダウンタイム、機会損失、顧客満足度への影響を考えましょう
  5. 性能とセキュリティのトレードオフが発生する具体的な場面を3つ挙げ、それぞれの最適化案を提案してください。
    ヒント：暗号化、認証、データ検証などの処理がパフォーマンスに与える影響を考えてください
