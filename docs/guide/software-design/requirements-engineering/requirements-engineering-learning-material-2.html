2.1 要件獲得（Requirements Elicitation）とは

要件獲得は、ステークホルダーが持つニーズ、期待、制約を体系的に発見・理解するプロセスです。単純な「聞き取り」ではなく、明示されていない暗黙の要求や、ステークホルダー自身が気づいていない潜在的なニーズを引き出すための戦略的アプローチです。

要件獲得の3つの難しさ

1. 知識の境界問題

ステークホルダーは業務の専門家ですが、ITシステムの可能性や制約を理解していません。一方、開発者はIT技術に詳しいが、業務の細かな要求や文脈を理解していません。

2. 暗黙知の言語化困難

長年の経験で身についた業務ノウハウは「当たり前すぎて」言葉にされず、重要な要件が見落とされがちです。

3. 変化する要求

要件獲得プロセス自体がステークホルダーの理解を深め、新たな要求を生み出します。初期の要求が最終的な要求とは限りません。

効果的な要件獲得のポイント

  • 複数の技法の組み合わせ：一つの技法だけでは限界がある
  • 反復的アプローチ：段階的に理解を深める
  • 多様な視点の確保：異なる立場のステークホルダーから収集
  • 文脈の理解：要求の背景や理由まで把握する

2.2 インタビュー技法

インタビューは最も基本的で重要な要件獲得技法です。一対一または少人数でのコミュニケーションにより、詳細で具体的な要件を収集します。

インタビューの種類と特徴
1. 構造化インタビュー
  • 特徴：事前に準備した質問リストに沿って進行
  • 適用場面：同一情報を複数人から収集する場合
  • メリット：効率的、比較しやすい
  • デメリット：予想外の情報を得にくい
2. 半構造化インタビュー
  • 特徴：基本質問は準備するが、柔軟に深掘りする
  • 適用場面：要件獲得の標準的手法
  • メリット：効率性と柔軟性のバランス
  • デメリット：インタビュアーのスキルに依存
3. 非構造化インタビュー
  • 特徴：自由な対話形式で進行
  • 適用場面：探索的調査、アイデア創出
  • メリット：予想外の重要な情報を発見
  • デメリット：時間がかかる、まとめが困難

効果的なインタビューの進め方

質問技法の実践
オープン質問（情報を広く収集）
  • 「現在の業務で困っていることは何ですか？」
  • 「理想的なシステムはどのようなものですか？」
  • 「一日の業務の流れを教えてください」
クローズド質問（具体的確認）
  • 「処理時間は何秒以内である必要がありますか？」
  • 「同時利用者数は最大何人ですか？」
  • 「この機能は必須ですか、それとも希望ですか？」
深掘り質問（背景・理由の理解）
  • 「なぜそれが重要なのですか？」
  • 「それができないとどんな問題が起こりますか？」
  • 「具体的にはどんな場面で使いますか？」

2.3 ワークショップファシリテーション

ワークショップは複数のステークホルダーが参加する協働的な要件獲得手法です。異なる立場の人々が一堂に会し、議論と合意形成を通じて要件を明確化します。

ワークショップの設計要素
参加者構成
  • エンドユーザー：実際にシステムを使用する人
  • ビジネスオーナー：業務の責任者・意思決定者
  • 技術者：実現可能性を判断できる人
  • ファシリテーター：議論を導く中立的な立場
ワークショップの基本流れ

効果的なファシリテーション技法

  • アイデア発散と収束の管理：ブレインストーミング → 分類・統合 → 優先度付け
  • 全員参加の促進：発言機会の均等化、異なる意見の尊重
  • 視覚化の活用：ホワイトボード、付箋、図解による理解促進
  • 時間管理：明確なタイムボックス、進行状況の見える化
  • 合意形成のサポート：対立点の整理、妥協案の探索
実習 2-1: インタビュー質問設計

オンライン予約システムの要件獲得のため、レストランオーナーにインタビューを行います。効果的な質問を設計してください。

前提条件
  • 対象：個人経営の小規模レストラン（席数30席）
  • 現状：電話予約のみ、手書き台帳で管理
  • 課題：予約の重複、電話応対の負担
設計課題
  1. 現状把握のためのオープン質問を3つ作成
  2. 機能要件を明確にするクローズド質問を3つ作成
  3. 非機能要件（性能、使いやすさ等）に関する質問を2つ作成
質問設計のポイント
オープン質問の例:
「現在の予約管理で一番困っていることは何ですか？」
「理想的な予約システムがあるとしたら、どんな機能があるといいですか？」

クローズド質問の例:
「一日の予約数は平均何件ぐらいですか？」
「予約の変更・キャンセルはどの程度発生しますか？」

非機能要件の例:
「スタッフがシステムを使えるようになるまで、どの程度の時間なら許容できますか？」
「予約の確認はどの程度の速さで表示される必要がありますか？」

2.4 観察法とプロトタイピング

観察法は、ステークホルダーの実際の業務を直接観察することで、言葉では表現されない暗黙的な要件を発見する手法です。

観察法の種類と特徴

1. 直接観察
  • 手法：現場で実際の業務を観察
  • 発見できること：無意識の行動、業務の流れ、例外処理
  • 注意点：観察される側の行動変化（ホーソン効果）
2. 参加観察
  • 手法：実際に業務に参加しながら観察
  • 発見できること：業務の詳細、判断基準、コツ
  • 注意点：客観性の維持、業務への影響
3. ワークフロー分析
  • 手法：業務プロセスを段階的に記録・分析
  • 発見できること：ボトルネック、改善機会、依存関係
  • 成果物：業務フロー図、作業時間分析

プロトタイピングによる要件獲得

プロトタイプを作成・提示することで、ステークホルダーから具体的なフィードバックを得る手法です。

プロトタイプの種類
  • ペーパープロトタイプ：手描きの画面レイアウト、迅速で安価
  • デジタルモックアップ：ツールで作成した静的画面、視覚的理解促進
  • インタラクティブプロトタイプ：クリック操作可能、リアルな体験提供
  • 実働プロトタイプ：基本機能が動作、詳細な検証可能

2.5 その他の重要な要件獲得技法

ブレインストーミング

創造的な発想によって新しいアイデアや要件を発見する集団的技法です。

ブレインストーミングの4原則
  1. 判断・批評の禁止：アイデア出しの段階では評価しない
  2. 自由奔放：どんな突飛なアイデアも歓迎
  3. 質より量：多くのアイデアを出すことを重視
  4. 結合改善：他者のアイデアを発展させる
効果的な進行方法
  • 明確なテーマ設定（「○○の課題を解決するアイデア」）
  • 時間制限の設定（集中力の維持）
  • すべてのアイデアの記録（付箋、ホワイトボード）
  • 発散後の収束作業（分類、統合、評価）

質問票調査

大多数のステークホルダーから効率的に情報を収集する手法です。

  • 定量調査：選択式質問による統計的分析
  • 定性調査：自由記述による詳細な意見収集
  • 適用場面：多数の利用者がいるシステム、初期調査
  • 注意点：質問の設計、回答率の確保、バイアスの除去

既存文書分析

組織内の既存資料から要件に関する情報を抽出する手法です。

分析対象となる文書例
  • 業務マニュアル：業務プロセス、規則
  • システム仕様書：既存システムの機能、制約
  • 法的文書：法律、規制、契約
  • 組織文書：組織図、職務記述書
  • 過去の問題報告：改善すべき課題

ユーザーストーリーマッピング

ユーザーの行動を時系列で整理し、システム要件を可視化する手法です。

実習 2-2: 技法選択の判断

以下の状況に最適な要件獲得技法を選択し、その理由を説明してください。

状況設定

プロジェクト：大学の履修登録システム更新

ステークホルダー：

  • 学生：約1万人
  • 教授：約500人
  • 事務職員：約50人
  • システム管理者：5人

制約：

  • プロジェクト期間：8ヶ月
  • 要件獲得期間：2ヶ月
  • 既存システムあり（10年前に導入）
検討課題
  1. 学生からの要件獲得に適した技法は？
  2. 教授からの詳細要件を得る技法は？
  3. 事務職員の業務フローを理解する技法は？
  4. ステークホルダー間の合意形成に適した技法は？
技法選択の考慮要素
考慮要素：
- ステークホルダーの数と分散度
- 情報の種類（定量・定性）
- 時間とコストの制約
- 参加者の技術知識レベル
- 既存情報の活用可能性

2.6 要件獲得技法の効果的な組み合わせ

実際のプロジェクトでは、複数の技法を段階的に組み合わせて使用します。各技法の特徴を理解し、プロジェクトの状況に応じて最適な組み合わせを選択することが重要です。

段階的要件獲得戦略

第1段階：全体把握（広く浅く）
  • 目的：プロジェクトの全体像とステークホルダーの把握
  • 主要技法：既存文書分析、質問票調査、初期インタビュー
  • 成果物：ステークホルダー一覧、要件概要、課題リスト
第2段階：詳細探索（深く詳しく）
  • 目的：具体的な要件の詳細化と暗黙知の発見
  • 主要技法：詳細インタビュー、観察法、ブレインストーミング
  • 成果物：詳細要件リスト、業務フロー、アイデア集
第3段階：統合・合意（調整・統一）
  • 目的：多様な要件の統合と優先度合意
  • 主要技法：ワークショップ、ユーザーストーリーマッピング
  • 成果物：統合要件仕様、優先度付きバックログ
第4段階：検証・確認（正確性確保）
  • 目的：要件の正確性と実現可能性の確認
  • 主要技法：プロトタイプ検証、確認インタビュー
  • 成果物：検証済み要件、承認された仕様書
理解度確認クイズ
  1. 要件獲得が「単純な聞き取り」と異なる理由を3つ説明してください。
  2. 構造化インタビューと非構造化インタビューの違いと、それぞれの適用場面を述べてください。
  3. ワークショップに参加すべき4種類のステークホルダーと、それぞれの役割を説明してください。
  4. 観察法で発見できる「暗黙的要件」の具体例を3つ挙げてください。
  5. 大学生1万人を対象とした学習支援システムの要件獲得で、最初に選択すべき技法とその理由を述べてください。
