6.1 要件優先度付けの重要性

限られた予算・時間・リソースの中で最大のビジネス価値を生み出すために、要件の優先度付けは不可欠です。適切な優先度付けにより、プロジェクトの成功確率を大幅に向上させることができます。

優先度付けが必要な理由
  • リソースの最適配分：限られた開発リソースを最も価値の高い機能に集中
  • 早期価値提供：重要な機能から優先的にリリースして早期にROIを実現
  • リスク軽減：高リスク要件を早期に実装して不確実性を低減
  • ステークホルダー合意：優先度の透明化により利害関係者間の合意形成
  • 変更対応力：明確な優先度によりスコープ変更時の意思決定を迅速化

優先度付けの失敗がもたらす影響

よくある失敗パターン
  • 「すべてが最優先」症候群：実質的な優先順位が存在しない状況
  • 技術者視点の優先度：ビジネス価値よりも技術的興味を優先
  • 声の大きいステークホルダー優先：政治的な力関係で優先度が決定
  • 依存関係の軽視：実装順序の制約を考慮しない優先度設定

6.2 MoSCoW法による優先度分類

MoSCoW法は、要件を4つのカテゴリに分類する手法です。シンプルで理解しやすく、ステークホルダーとの合意形成に効果的です。

graph TD A[MoSCoW法] --> B["Must have
（必須要件）"] A --> C["Should have
（重要要件）"] A --> D["Could have
（希望要件）"] A --> E["Won't have
（対象外要件）"] B --> F["プロダクト成功に不可欠"] C --> G["重要だが一時的に延期可能"] D --> H["あれば嬉しい機能"] E --> I["今回は対象外"]

各カテゴリの詳細定義と判断基準

Must have（必須要件）- 約60%
  • 特徴：この要件なしでは製品が成り立たない
  • 判断基準：法的要件、基本的なビジネス機能、安全要件
  • 例：ログイン機能、決済機能、データ保存機能
  • 質問：「この機能がなければプロダクトは使い物になりませんか？」
Should have（重要要件）- 約20%
  • 特徴：重要だが、短期間であれば代替手段で対応可能
  • 判断基準：競合優位性、ユーザビリティ向上、業務効率化
  • 例：高速検索、推奨機能、詳細レポート
  • 質問：「この機能がないと大きな不便が生じますか？」
Could have（希望要件）- 約20%
  • 特徴：付加価値を提供するが、なくても問題ない
  • 判断基準：ユーザー体験の向上、将来への投資
  • 例：ダークモード、SNS連携、ゲーミフィケーション
  • 質問：「この機能があると嬉しいですが、今回なくても大丈夫ですか？」
実習 6-1: MoSCoW法による優先度分類演習

オンライン学習プラットフォームの機能要件をMoSCoW法で分類してみましょう。

対象機能リスト
  • ユーザー登録・ログイン
  • コース検索・閲覧
  • 動画再生機能
  • 進捗管理
  • 修了証発行
  • SNS共有機能
  • オフライン視聴
  • 多言語対応
  • AI推奨機能
  • バーチャル教室
分類結果例
【Must have（必須要件）】
・ユーザー登録・ログイン: セキュリティとアクセス制御の基盤
・コース検索・閲覧: プラットフォームの主要機能
・動画再生機能: 学習コンテンツの提供手段
・進捗管理: 学習効果測定と継続動機の基盤

【Should have（重要要件）】
・修了証発行: 学習成果の証明と動機付け
・多言語対応: グローバル展開に必要

【Could have（希望要件）】
・SNS共有機能: ユーザー獲得とエンゲージメント向上
・オフライン視聴: ユーザビリティ向上
・AI推奨機能: 個人化された学習体験

【Won't have（対象外）】
・バーチャル教室: 技術的複雑性が高く、MVP外
期待される結果

ステークホルダー間で合意された明確な優先順位と、その根拠が説明できること

6.3 カノモデルによる顧客満足度分析

カノモデルは、機能と顧客満足度の関係を分析し、顧客の潜在的ニーズを発見する手法です。品質を3つのカテゴリに分類します。

カノモデルの3つの品質カテゴリ

graph TD A["カノモデル"] --> B["当たり前品質
（Must-be）"] A --> C["一元的品質
（Performance）"] A --> D["魅力的品質
（Attractive）"] B --> E["ないと不満だが
あっても感動はない"] C --> F["充足度に比例して
満足度が向上"] D --> G["なくても問題ないが
あると大きな感動"]
1. 当たり前品質（Must-be Quality）
  • 特徴：充足されて当然と思われる品質
  • 満足度への影響：ないと不満、あっても感動なし
  • 例：Webサイトが正常に動作する、セキュリティが確保されている
  • 戦略：最低限のコストで確実に実装
2. 一元的品質（Performance Quality）
  • 特徴：充足度に比例して満足度が向上
  • 満足度への影響：多いほど満足、少ないほど不満
  • 例：検索速度、バッテリー持続時間、画面の大きさ
  • 戦略：競合との差別化ポイントとして投資
3. 魅力的品質（Attractive Quality）
  • 特徴：予期しない価値を提供
  • 満足度への影響：なくても問題ないが、あると大満足
  • 例：AIアシスタント、ダークモード、ゲーミフィケーション
  • 戦略：イノベーションと競合優位性の源泉

6.4 ペアワイズ比較法

ペアワイズ比較法は、要件を2つずつ比較することで相対的な重要度を算出する手法です。主観的な判断を数値化し、客観的な優先順位を導出できます。

実習 6-2: ペアワイズ比較による優先度算出

ECサイトの5つの機能要件について、ペアワイズ比較で優先順位を決定してみましょう。

対象機能
  • A: 商品検索機能
  • B: 決済機能
  • C: レビュー機能
  • D: 推奨機能
  • E: 在庫表示機能
比較マトリクス（1=同程度、2=やや重要、3=重要、4=かなり重要、5=非常に重要）
ABCDE合計優先度
A-1/345313.333位
B3-554171位
C1/41/5-213.455位
D1/51/51/2-1/21.44位
E1/31/412-3.582位
結論
優先順位：
1位: 決済機能（17.0） - ビジネスの根幹
2位: 在庫表示機能（3.58） - 購入判断に必須
3位: 商品検索機能（13.33） - 商品発見の基盤
4位: 推奨機能（1.4） - 売上向上の付加価値
5位: レビュー機能（3.45） - ユーザー体験向上

6.5 100ドルテスト（Buy a Feature）

100ドルテストは、ステークホルダーに仮想の予算を配分してもらい、各機能への投資意欲を測定する手法です。民主的で直感的な優先度付けが可能です。

実施手順と注意点

100ドルテストの実施方法
  1. 機能の価格設定：開発コストに基づいて各機能に価格を設定
  2. 予算配布：各参加者に100ドルの仮想予算を配布
  3. 投資決定：参加者が予算を機能に自由に配分
  4. 結果集計：全参加者の投資額を合計し、優先度を算出
  5. 議論・調整：結果を基に詳細な議論を実施
実習 6-3: 100ドルテストによる機能優先度付け

プロジェクト管理ツールの機能について、100ドルテストを実施してみましょう。

6.6 リスク-価値マトリクス

リスク-価値マトリクスは、各要件のビジネス価値と実装リスクを2軸で評価し、戦略的な優先順位を決定する手法です。

graph TB subgraph "リスク-価値マトリクス" A["高価値
低リスク
（優先実装）"] B["高価値
高リスク
（慎重検討）"] C["低価値
低リスク
（後回し）"] D["低価値
高リスク
（実装不要）"] end A --> E["Quick Wins"] B --> F["Major Projects"] C --> G["Fill-ins"] D --> H["Thankless Tasks"]
4つの象限の戦略
  • 高価値・低リスク（Quick Wins）：最優先で実装
  • 高価値・高リスク（Major Projects）：リスク軽減策を検討してから実装
  • 低価値・低リスク（Fill-ins）：リソースに余裕があれば実装
  • 低価値・高リスク（Thankless Tasks）：実装を避ける

6.7 依存関係分析

要件間の依存関係を分析することで、実装順序の制約を把握し、現実的な開発計画を立案できます。

依存関係の種類

技術的依存関係
  • 例：ユーザー認証 → 個人設定管理
  • 特徴：システムアーキテクチャに起因する必須の順序
ビジネス的依存関係
  • 例：基本機能 → 高度機能
  • 特徴：ユーザー価値やビジネス戦略に起因する論理的順序
graph TD A["ユーザー登録"] --> B["ログイン機能"] B --> C["プロフィール管理"] B --> D["商品閲覧"] D --> E["カート機能"] E --> F["決済機能"] C --> G["注文履歴"] F --> G F --> H["在庫管理"]

6.8 実現可能性評価

技術的・経済的・運用的な観点から各要件の実現可能性を評価し、リスクを事前に特定します。

実現可能性の評価観点

技術的実現可能性（Technical Feasibility）
  • 既存技術で実現可能か
  • 性能要件を満たせるか
  • セキュリティ要件をクリアできるか
  • 既存システムとの統合は可能か
経済的実現可能性（Economic Feasibility）
  • 開発コストは妥当な範囲か
  • 運用コストは継続可能か
  • ROI（投資収益率）は見込めるか
  • 予算制約内で実現可能か
運用的実現可能性（Operational Feasibility）
  • 組織の運用体制で対応可能か
  • ユーザーの受け入れは期待できるか
  • 法的・規制的制約はクリアできるか
  • スケジュール的に実現可能か
理解度確認クイズ
  1. MoSCoW法で「Must have」に分類すべき要件の判断基準を3つ挙げ、それぞれ具体例を示してください。
    ヒント：法的要件、基本的なビジネス機能、安全要件の観点で考えてみましょう
  2. カノモデルにおける「魅力的品質」が時間の経過とともに「当たり前品質」に変化する例を挙げ、この現象が要件管理に与える影響を説明してください。
    ヒント：スマートフォンの機能の変遷を考えてみましょう
  3. ペアワイズ比較法の利点と欠点を挙げ、どのような状況で最も効果的に使用できるか説明してください。
    ヒント：要件数の制約と主観的判断の定量化について考えてみましょう
  4. リスク-価値マトリクスで「高価値・高リスク」の象限に分類された要件に対する3つの対策を提案してください。
    ヒント：リスク軽減、段階的実装、代替案検討の観点で考えてみましょう
  5. 要件間の依存関係分析を行う際に、見落としがちな依存関係の種類を2つ挙げ、それぞれの対策を説明してください。
    ヒント：データ依存関係、ユーザビリティ依存関係、運用依存関係を考えてみましょう
