第9章

データの関係を理解する - リレーションシップ

複数のエンティティ間の関係（1対多、多対多）について学び、「顧客と注文」「商品とカテゴリ」のような関連付けの方法を理解しましょう。

この章で学ぶこと

  • リレーションシップ（データ間の関係）とは何かを理解する
  • 1対多の関係（顧客と注文など）の考え方を学ぶ
  • 多対多の関係（商品とカテゴリなど）の実装方法を理解する
  • OutSystemsでリレーションシップを設定して、関連データを扱う

9.1 リレーションシップって何だろう？

これまでの章で、エンティティ（データの入れ物）を学びました。 しかし、実際のアプリケーションでは、複数のエンティティが互いに関連していることがほとんどです。

現実世界の関係性

例えば、オンラインショッピングサイトを考えてみましょう。以下のような「関係性」があります：

顧客と注文

1人の顧客は複数の注文を出すことができます。 逆に、1つの注文は必ず1人の顧客に属します。

注文と商品

1つの注文には複数の商品が含まれます。 また、1つの商品は複数の注文に登場することがあります。

商品とカテゴリ

1つの商品は複数のカテゴリに属することがあります。 また、1つのカテゴリには複数の商品が含まれます。

会社と社員

1つの会社には複数の社員が所属します。 逆に、1人の社員は必ず1つの会社に所属します。

リレーションシップとは

リレーションシップ（Relationship）は、 データベース上でエンティティ同士の「関係性」を定義する仕組みです。 現実世界の関係をそのままデータベースに反映させることができます。

9.2 1対多の関係 - 顧客と注文

1対多（One-to-Many）の関係は、 最も一般的なリレーションシップの形です。片方のエンティティの1つのレコードが、もう片方のエンティティの複数のレコードと関連する関係です。

具体例: 顧客と注文

1対多の関係図

erDiagram CUSTOMER ||--o{ ORDER : "1人の顧客が複数の注文" CUSTOMER { int CustomerId PK string Name string Email string Phone } ORDER { int OrderId PK int CustomerId FK date OrderDate decimal TotalAmount }

Customer（顧客）エンティティ

CustomerId Name
1 山田太郎
2 佐藤花子

Order（注文）エンティティ

OrderId CustomerId Amount
101 1 ¥5,000
102 1 ¥8,000
103 2 ¥3,500

ポイント

注文エンティティのCustomerIdが、 顧客エンティティの主キー（Primary Key）を参照しています。 これを外部キー（Foreign Key）と呼びます。 外部キーによって、2つのエンティティが「関連付け」られます。

OutSystemsでの実装

1対多リレーションシップの設定手順

  1. 1 まず、CustomerエンティティとOrderエンティティを作成します
  2. 2 Orderエンティティに、Customer Identifier型のアトリビュートを追加します
  3. 3 アトリビュートのデータ型で「Customer Identifier」を選択すると、自動的にリレーションシップが設定されます
  4. 4 これで、注文データから顧客情報にアクセスできるようになります

Identifier型とは

OutSystemsでは、他のエンティティを参照するアトリビュートに「エンティティ名 Identifier」という型を使います。 例えば、Customerエンティティを参照する場合は「Customer Identifier」型を使用します。

9.3 多対多の関係 - 商品とカテゴリ

多対多（Many-to-Many）の関係は、 両方のエンティティで、1つのレコードが相手の複数のレコードと関連する関係です。

具体例: 商品とカテゴリ

商品 → カテゴリ

「ノートパソコン」という商品は、 「電子機器」「パソコン」「オフィス用品」など複数のカテゴリに属することができます。

カテゴリ → 商品

「電子機器」というカテゴリには、 「ノートパソコン」「スマートフォン」「タブレット」など複数の商品が含まれます。

中間テーブルの必要性

多対多の関係を実装するには、中間テーブル（Junction Table）が必要です。 中間テーブルは、2つのエンティティを繋ぐ「橋渡し役」となります。

多対多の関係図（中間テーブル使用）

erDiagram PRODUCT ||--o{ PRODUCT_CATEGORY : "商品は複数カテゴリ" CATEGORY ||--o{ PRODUCT_CATEGORY : "カテゴリは複数商品" PRODUCT { int ProductId PK string ProductName decimal Price } CATEGORY { int CategoryId PK string CategoryName } PRODUCT_CATEGORY { int ProductCategoryId PK int ProductId FK int CategoryId FK }

この例では、「ノートPC（Id=1）」が「電子機器（Id=10）」と「パソコン（Id=11）」の2つのカテゴリに属していることが分かります。

なぜ中間テーブルが必要？

1対多の関係では、「多」側のテーブルに外部キーを1つ持たせるだけで済みました。 しかし、多対多では両方が「多」なので、どちらに外部キーを置いても表現できません。 そこで、中間テーブルを作って両方の外部キーを保持させます。

9.4 実習: リレーションシップを使った注文管理システム

それでは、実際にOutSystemsでリレーションシップを設定して、顧客と注文の管理システムを作成してみましょう。

実習 9-1: 顧客と注文の1対多リレーションシップ

顧客と注文の関係を設定し、顧客ごとの注文履歴を表示するアプリを作りましょう。

手順

  1. 1

    Customerエンティティを作成

    Service Studioで新しいエンティティ「Customer」を作成します。 以下のアトリビュートを追加してください：

    • • Name（Text型）- 顧客名
    • • Email（Email型）- メールアドレス
    • • Phone（Phone Number型）- 電話番号
  2. 2

    Orderエンティティを作成

    新しいエンティティ「Order」を作成し、以下のアトリビュートを追加します：

    • • CustomerId（Customer Identifier型） - 顧客への参照
    • • OrderDate（Date型）- 注文日
    • • TotalAmount（Currency型）- 合計金額
    • • Status（Text型）- ステータス
  3. 3

    サンプルデータを登録

    Customerエンティティに2〜3人の顧客を登録し、 各顧客に対して複数の注文データを作成します。

  4. 4

    顧客一覧画面を作成

    「CustomerList」という画面を作成し、 すべての顧客を一覧表示します。各顧客の名前をクリックすると注文履歴画面に遷移するようにします。

  5. 5

    注文履歴画面を作成

    「OrderHistory」という画面を作成し、 特定の顧客の注文履歴を表示します。アグリゲートで「Order.CustomerId = CustomerId（パラメータ）」というフィルタ条件を設定します。

  6. 6

    公開して動作確認

    アプリを公開して、顧客ごとに異なる注文履歴が表示されることを確認します。

これで、1対多のリレーションシップを使って、顧客と注文のデータを関連付けることができました！

動作確認

以下の点を確認してください：

  • 顧客一覧画面で、すべての顧客が表示される
  • 顧客名をクリックすると、その顧客の注文履歴画面に遷移する
  • 注文履歴画面では、選択した顧客の注文だけが表示される
  • 各注文の詳細（注文日、金額、ステータス）が正しく表示される

9.5 よくあるトラブルと解決方法

トラブルシューティング

リレーションシップが設定できない

原因:

  • • アトリビュートのデータ型が間違っている
  • • 参照先エンティティが存在しない

解決策:

アトリビュートのデータ型で「[エンティティ名] Identifier」を選択してください。 例えば、Customerエンティティを参照する場合は「Customer Identifier」を選びます。

関連データが取得できない

原因:

  • • アグリゲートでJOINが設定されていない
  • • フィルタ条件が間違っている

解決策:

アグリゲートで関連エンティティをJOINし、適切なフィルタ条件を設定してください。 例：「Order.CustomerId = CustomerId」

多対多の関係がうまく動かない

原因:

  • • 中間テーブルを作成していない
  • • 中間テーブルに両方の外部キーが設定されていない

解決策:

多対多の関係では、必ず中間テーブルを作成し、 両方のエンティティへの外部キー（Identifier型）を設定してください。

理解度確認クイズ

  1. Q1. リレーションシップを使うメリットを3つ挙げてください。

    回答を表示

    1. データの重複を防ぐ - 同じ情報を複数の場所に保存する必要がない
    2. データの整合性を保つ - 関連するデータが常に正しい状態を維持できる
    3. メンテナンスが簡単 - データの更新が1箇所で済む

  2. Q2. 1対多の関係で、外部キーはどちらのエンティティに設定しますか？

    回答を表示

    「多」側のエンティティに外部キーを設定します。
    例えば、「顧客と注文」の関係では、注文（多）側に顧客への外部キー（CustomerId）を設定します。

  3. Q3. 多対多の関係を実装するために必要なものは何ですか？

    回答を表示

    中間テーブル（Junction Table）が必要です。
    中間テーブルには、両方のエンティティへの外部キーを設定します。 例えば、「商品とカテゴリ」の関係では、ProductCategoryという中間テーブルを作成し、 ProductIdとCategoryIdの両方を持たせます。

  4. Q4. OutSystemsで他のエンティティを参照するアトリビュートには、どんなデータ型を使いますか？

    回答を表示

    「[エンティティ名] Identifier」型を使います。
    例えば、Customerエンティティを参照する場合は「Customer Identifier」、 Productエンティティを参照する場合は「Product Identifier」を選択します。

第9章のまとめ

  • リレーションシップは、複数のエンティティ間の関係を定義する仕組みで、現実世界のデータの関連性を表現できます
  • 1対多の関係では、「多」側のエンティティに外部キー（Identifier型）を設定して関連付けます
  • 多対多の関係では、中間テーブルを作成し、両方のエンティティへの外部キーを持たせます
  • OutSystemsでは、Identifier型を使うことで、簡単にリレーションシップを設定できます
  • リレーションシップを活用することで、データの重複を防ぎ、整合性を保ちながら効率的にデータを管理できます
