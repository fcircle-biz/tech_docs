セキュリティを確保する - ユーザー管理とアクセス制御

アプリケーションを安全に保つ方法を学び、ユーザーのログイン機能、ロールによる権限管理、データのアクセス制御を実装しましょう。

この章で学ぶこと

  • アプリケーションセキュリティの重要性と基本的な脅威を理解する
  • ユーザー認証（ログイン機能）の仕組みと実装方法を学ぶ
  • ロールベースのアクセス制御（RBAC）で権限を管理する
  • データレベルでのアクセス制御とセキュリティのベストプラクティスを習得する

12.1 なぜセキュリティが重要なのか

アプリケーションを開発するとき、機能や見た目だけでなくセキュリティも非常に重要です。 セキュリティが不十分だと、個人情報の流出や不正アクセス、データの改ざんなどの重大な問題が発生する可能性があります。

アプリケーションが直面する脅威

不正アクセス

権限のないユーザーが他人のアカウントを使って不正にログインし、機密情報にアクセスする攻撃です。

データ漏洩

適切なアクセス制御がないと、個人情報や業務データが外部に流出してしまいます。

データの改ざん

権限のないユーザーがデータを不正に変更・削除してしまう攻撃です。

権限昇格

一般ユーザーが管理者権限を不正に取得し、システム全体を操作できてしまう攻撃です。

セキュリティの3要素（CIA）

セキュリティには、機密性（Confidentiality）: 許可されたユーザーだけがアクセスできる、 完全性（Integrity）: データが改ざんされない、 可用性（Availability）: 必要な時にアクセスできる、という3つの基本要素があります。

12.2 ユーザー認証とログイン機能

認証（Authentication）とは、 「あなたは誰ですか？」を確認することです。一般的にはユーザー名とパスワードでログインすることで、 そのユーザーが本人であることを証明します。

OutSystemsの認証機能

OutSystemsには、Userエンティティという組み込みのユーザー管理機能が用意されています。 これを使うことで、簡単にログイン機能を実装できます。

項目 説明
User エンティティ ユーザー情報（Username、Email、名前など）を保存
Login アクション ユーザー名とパスワードでログイン処理を実行
Logout アクション 現在のユーザーをログアウトさせる
GetUserId 関数 現在ログイン中のユーザーIDを取得

実習 12-1: ログイン画面を作成する

ユーザー認証機能を実装して、ログインが必要なアプリケーションを作りましょう。

手順

  1. 1 新しいアプリケーション「SecureApp」を作成します
  2. 2 「Common」→「User」→「Login」という組み込み画面テンプレートを追加します
  3. 3 保護したい画面のプロパティで「Roles」を設定します（例: Registered）
  4. 4 Publishして、ブラウザで確認します。ログインしていないと保護された画面にアクセスできません

OutSystemsでは、画面に「Roles」プロパティを設定するだけで、 自動的にログインチェックが行われ、未ログインユーザーはログイン画面にリダイレクトされます。

パスワードのセキュリティ

ハッシュ化

OutSystemsでは、パスワードはハッシュ化（一方向暗号化）されてデータベースに保存されます。 これにより、万一データベースが漏洩しても、元のパスワードを復元できません。

パスワードポリシー

パスワードの最小文字数、大文字・小文字・数字・記号の組み合わせなど、 パスワードの強度ルールを設定できます。User Applicationの設定で変更可能です。

12.3 ロールベースのアクセス制御（RBAC）

認可（Authorization）とは、 「あなたは何ができますか？」を管理することです。認証でユーザーを特定した後、 そのユーザーにどんな操作を許可するかをロール（役割）で管理します。

ロールとは

ロールは、ユーザーの役割や権限のグループです。 例えば「管理者」「編集者」「閲覧者」のようにユーザーを分類し、それぞれに異なる権限を与えます。

OutSystemsでのロール設定

実習 12-2: ロールを作成して権限を管理する

「管理者」と「一般ユーザー」で異なる権限を設定しましょう。

手順

  1. 1

    ロールの作成

    「Logic」→「Roles」を右クリック→「Add Role」で「Administrator」ロールを作成します

  2. 2

    管理画面にロールを設定

    管理者専用の画面（例: AdminScreen）のプロパティで「Roles」に「Administrator」を設定します

  3. 3

    条件付きUI表示

    「削除」ボタンなど特定の操作を、Check Administrator Role 関数で管理者のみに表示するようにします

  4. 4

    ユーザーにロールを割り当て

    Service CenterまたはUsersアプリで、ユーザーに「Administrator」ロールを付与します

コード例: ロールチェック

ボタンの「Visible」プロパティに以下を設定：

Check AdministratorRole()

これで管理者ロールを持つユーザーにのみボタンが表示されます。

関数 説明
Check [RoleName] Role() 現在のユーザーが指定ロールを持っているか確認
GrantRole ユーザーにロールを付与する
RevokeRole ユーザーからロールを剥奪する

12.4 データのアクセス制御

画面レベルでの権限管理だけでなく、データレベルでのアクセス制御も重要です。 例えば、「自分が作成したデータのみ編集可能」「所属部門のデータのみ閲覧可能」といった細かい制御が必要になります。

データの所有者フィルター

データの所有者をUserIdで記録し、アグリゲートでGetUserId()を使ってフィルタリングすることで、 ユーザーが自分のデータだけにアクセスできるようにします。

実習 12-3: 自分のデータだけを表示する

ログインユーザーが作成したデータのみを表示するように制御しましょう。

手順

  1. 1

    エンティティに所有者フィールドを追加

    例えば「Task」エンティティに「CreatedBy」アトリビュート（User Identifier型）を追加します

  2. 2

    作成時に所有者を記録

    データ作成時、CreatedBy に GetUserId() の値を設定します

  3. 3

    アグリゲートでフィルター

    Task を取得するアグリゲートに、フィルター条件「Task.CreatedBy = GetUserId()」を追加します

  4. 4

    更新・削除時にもチェック

    データ更新/削除アクションで、所有者が一致するか確認してから処理を実行します

これにより、ユーザーAが作成したタスクは、ユーザーBからは見えなくなり、安全にデータを保護できます。

組織階層に基づくアクセス制御

企業のアプリケーションでは、「自分の部門のデータだけ見える」「上司は部下のデータも見える」といった 組織階層に基づくアクセス制御が必要になることがあります。

ベストプラクティス

データアクセス制御のロジックは、Server Actionにまとめて実装し、 複数の画面で共通利用できるようにしましょう。これにより、セキュリティの一貫性が保たれます。

12.5 セキュリティのベストプラクティス

セキュリティを確保するために、以下のベストプラクティスを守りましょう。

1. 最小権限の原則

ユーザーには、必要最小限の権限のみを付与します。 「全員に管理者権限」のような設定は避け、本当に必要なユーザーにのみ強い権限を与えましょう。

2. サーバーサイドでの検証

クライアント側（画面上）での制御だけでなく、サーバーサイド（Server Action）でも権限チェックを実装します。 クライアント側の制御は簡単に回避できるため、必ずサーバー側でも検証しましょう。

3. セッション管理

OutSystemsは自動的にセッションを管理してくれますが、 セッションタイムアウトの設定を適切に行い、長期間ログインしたままにならないようにしましょう。

4. ログと監査証跡

重要な操作は必ずログに記録します。 「誰が」「いつ」「何を」したかを追跡できるようにすることで、問題発生時の調査やコンプライアンス対応が可能になります。

5. HTTPSの使用

OutSystemsはデフォルトでHTTPSを使用します。 通信が暗号化されるため、ネットワーク上での盗聴からデータを守ることができます。

6. SQLインジェクション対策

OutSystemsのアグリゲートやエンティティアクションを使う限り、SQLインジェクションは自動的に防止されます。 SQLクエリを直接書く場合は、必ずパラメータ化（プレースホルダー）を使いましょう。

7. XSS（クロスサイトスクリプティング）対策

OutSystemsは自動的にHTMLエスケープを行うため、通常の使い方ではXSSは発生しません。 ただし、「Escape Content = No」に設定する場合は、ユーザー入力を表示しないよう注意が必要です。

よくあるセキュリティの問題と対処法

問題: ログアウト後もデータが見える

原因: 画面にRolesが設定されていない
対処: 保護が必要な画面のプロパティで、Roles に「Registered」を設定する

問題: 他のユーザーのデータが編集できてしまう

原因: データの所有者チェックがない
対処: 更新/削除アクションで、データの CreatedBy = GetUserId() を確認する

問題: 一般ユーザーが管理機能にアクセスできる

原因: 管理画面にロール制限がない
対処: 管理画面のRolesプロパティに「Administrator」を設定し、Server Actionでも権限チェックを追加する

理解度確認クイズ

  1. Q1. 認証（Authentication）と認可（Authorization）の違いを説明してください。

    回答を表示

    認証は「あなたは誰ですか？」を確認すること（ログイン）で、 認可は「あなたは何ができますか？」を管理すること（権限チェック）です。 認証でユーザーを特定してから、認可でそのユーザーの操作を制限します。

  2. Q2. OutSystemsで画面をログインユーザーのみにアクセス制限するには、どのプロパティを設定しますか？

    回答を表示

    画面のプロパティで「Roles」に「Registered」を設定します。 これにより、未ログインユーザーは自動的にログイン画面にリダイレクトされます。

  3. Q3. ユーザーが自分のデータだけを見られるようにするには、どのような実装が必要ですか？

    回答を表示

    1. エンティティにCreatedBy（User Identifier型）アトリビュートを追加
    2. データ作成時にGetUserId()で所有者を記録
    3. アグリゲートでCreatedBy = GetUserId()のフィルターを設定
    これにより、ログインユーザーが作成したデータのみが取得されます。

  4. Q4. セキュリティのベストプラクティスとして「最小権限の原則」とは何ですか？

    回答を表示

    ユーザーには業務に必要な最小限の権限だけを与えるという原則です。 必要以上に強い権限を与えると、誤操作や不正利用のリスクが高まるため、 「全員に管理者権限」のような設定は避け、本当に必要なユーザーにのみ強い権限を付与します。

第12章のまとめ

  • セキュリティは、不正アクセス、データ漏洩、改ざんなどの脅威からアプリケーションを守るために不可欠です
  • 認証（Authentication）でユーザーを特定し、認可（Authorization）で操作権限を管理します
  • OutSystemsではRolesを使ったロールベースアクセス制御（RBAC）で、画面やアクションへのアクセスを制限できます
  • データレベルのアクセス制御では、GetUserId()を使ってユーザーが自分のデータだけにアクセスできるようにします
  • 最小権限の原則、サーバーサイド検証、ログ記録などのベストプラクティスを守ることで、安全なアプリケーションを構築できます
