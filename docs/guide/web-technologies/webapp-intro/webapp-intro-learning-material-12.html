第12章

CORSとセキュリティ基礎

同一オリジンポリシー、CORSエラーの原因と対処法、HTTPSについて学びます。Webアプリケーションのセキュリティの基礎を理解します。

この章で学ぶこと

  • 同一オリジンポリシーとは何か理解する
  • CORSエラーの原因と対処法を学ぶ
  • HTTPSの役割とセキュリティの基礎を把握する

12.1 同一オリジンポリシー（Same-Origin Policy）

同一オリジンポリシー（Same-Origin Policy）は、Webブラウザに組み込まれているセキュリティの仕組みです。 異なるオリジン（ドメイン）からのリソースへのアクセスを制限することで、悪意のあるWebサイトからユーザーを守ります。

ポイント: オリジンとは？

オリジンは「スキーム（プロトコル）」「ホスト（ドメイン）」「ポート番号」の3つの組み合わせで決まります。

https://example.com:443
スキーム: https / ホスト: example.com / ポート: 443

同一オリジンの判定例

基準URL 比較URL 結果
https://example.com https://example.com/page ✓ 同一
https://example.com http://example.com ✗ 異なる（スキーム違い）
https://example.com https://api.example.com ✗ 異なる（ホスト違い）
https://example.com:443 https://example.com:8080 ✗ 異なる（ポート違い）

同一オリジンポリシーにより、例えば https://mybank.com のページから https://attacker.com のAPIにアクセスすることは、ブラウザによってブロックされます。

12.2 CORS（Cross-Origin Resource Sharing）

CORS（Cross-Origin Resource Sharing、オリジン間リソース共有）は、 異なるオリジン間でのリソースへのアクセスを安全に許可するための仕組みです。

身近な例で考えてみよう

CORSは「国境の税関」のようなものです。 通常は外国からの荷物を受け取ることはできませんが、税関が「この国からの荷物は安全」と許可してくれれば受け取れます。 CORSヘッダーが、この「許可証」の役割を果たします。

CORSの動作フロー

sequenceDiagram participant B as ブラウザ (https://myapp.com) participant API as APIサーバー (https://api.example.com) B->>API: リクエスト (Origin: https://myapp.com) Note over API: CORSヘッダーをチェック alt CORS許可あり API->>B: レスポンス + Access-Control-Allow-Origin B->>B: データ取得成功 else CORS許可なし API->>B: レスポンス (CORSヘッダーなし) B->>B: CORSエラー発生 end

サーバーがCORSヘッダーを返さないと、ブラウザがアクセスをブロックする

12.3 CORSエラーの原因と対処法

開発中によく遭遇するCORSエラーについて、原因と対処法を理解しましょう。

典型的なCORSエラーメッセージ

Access to fetch at 'https://api.example.com/data'
from origin 'http://localhost:3000' has been blocked by CORS policy

CORSエラーの対処法

フロントエンド側でできること

1. プロキシを使う（開発環境のみ）

開発用サーバー（Vite、Webpack Dev Serverなど）のプロキシ機能を使い、同一オリジンからのリクエストに見せかける方法です。

2. バックエンドに連絡する

フロントエンド単独では解決できないため、バックエンド開発者にCORSヘッダーの設定を依頼します。

バックエンド側の対処法（サーバー側設定）

解決方法: CORSヘッダーを返す

サーバーのレスポンスに以下のヘッダーを含めることで、ブラウザがCORSを許可します。

HTTPレスポンスヘッダーの例 
Access-Control-Allow-Origin: https://myapp.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
ワイルドカード使用時の注意

開発環境: Access-Control-Allow-Origin: *

すべてのオリジンからのアクセスを許可（テスト用のみ）

本番環境: Access-Control-Allow-Origin: https://myapp.com

特定のオリジンのみ許可（セキュリティのため必須）

12.4 HTTPS（HTTP over SSL/TLS）

HTTPSは、HTTPに暗号化の仕組みを追加したプロトコルです。 通信内容を暗号化することで、第三者による盗聴や改ざんを防ぎます。

HTTP（暗号化なし）

  • 通信内容が平文
  • 盗聴・改ざんのリスクあり
  • パスワードなどが漏れる可能性
  • ブラウザに「安全ではない」と表示

HTTPS（暗号化あり）

  • 通信内容が暗号化される
  • 盗聴・改ざんを防ぐ
  • パスワードなどが守られる
  • ブラウザに鍵マークが表示

HTTPSの暗号化イメージ

sequenceDiagram participant B as ブラウザ participant S as Webサーバー (HTTPS) Note over B,S: 暗号化通信の確立 B->>S: SSL/TLS ハンドシェイク S->>B: サーバー証明書 B->>B: 証明書の検証 Note over B,S: 暗号化通信開始 B->>S: 暗号化されたリクエスト S->>B: 暗号化されたレスポンス Note over B: 第三者は内容を読めない

HTTPSでは通信内容が暗号化され、第三者による盗聴が不可能

SSL証明書とは

SSL証明書は、Webサイトの身元を証明する「デジタル身分証」のようなものです。 認証局（CA）という信頼できる第三者機関が発行します。 ブラウザはこの証明書を確認し、接続先のサーバーが本物かどうかをチェックします。

12.5 Mixed Content（混在コンテンツ）

Mixed Contentとは、HTTPSページの中にHTTP（暗号化なし）のリソースが混在している状態です。 これはセキュリティリスクとなるため、ブラウザによってブロックされます。

Mixed Contentの例

✗ 悪い例（ブロックされる）

<!-- HTTPSページ内で -->
<img src="http://example.com/image.jpg">
<script src="http://example.com/script.js"></script>

✓ 良い例（すべてHTTPS）

<!-- HTTPSページ内で -->
<img src="https://example.com/image.jpg">
<script src="https://example.com/script.js"></script>

注意

HTTPSページでは、画像、CSS、JavaScript、APIリクエストなど、すべてのリソースをHTTPSで読み込む必要があります。 HTTPリソースはブラウザによってブロックされ、正常に動作しません。

12.6 Webセキュリティのベストプラクティス

Webアプリケーション開発において、セキュリティを意識することは非常に重要です。 初心者が押さえておくべき基本的なポイントを紹介します。

1

HTTPSを使用する

本番環境では必ずHTTPSを使い、通信を暗号化します。無料のSSL証明書（Let's Encrypt）も利用可能です。

2

CORSを適切に設定する

ワイルドカード（*）は開発環境のみで使用し、本番環境では特定のオリジンのみを許可します。

3

機密情報をURLに含めない

パスワードやAPIキーなどの機密情報は、クエリパラメータではなくリクエストボディやヘッダーで送信します。

4

入力値の検証を行う

ユーザーからの入力は必ず検証し、不正なデータが処理されないようにします（フロントエンド・バックエンド両方で実施）。

5

依存ライブラリを最新に保つ

使用しているライブラリやフレームワークは、セキュリティパッチが適用された最新版を使用します。

12.7 この章のまとめ

1

同一オリジンポリシーは、異なるオリジンからのアクセスを制限する仕組み

2

CORSは、異なるオリジン間でのリソース共有を安全に行う仕組み

3

CORSエラーは、サーバー側でCORSヘッダーを設定することで解決する

4

HTTPSは通信を暗号化し、盗聴や改ざんを防ぐ

5

Mixed Contentを避け、すべてのリソースをHTTPSで読み込む

理解度確認クイズ

  1. Q1. オリジンの判定に使われる3つの要素を挙げてください。

    回答を表示

    (1) スキーム（プロトコル: http、httpsなど）
    (2) ホスト（ドメイン名: example.comなど）
    (3) ポート番号（:80、:443、:8080など）

  2. Q2. CORSエラーが発生した場合、誰が（どこで）対処する必要がありますか？

    回答を表示

    主にバックエンド（APIサーバー側）で対処します。 サーバーのレスポンスに「Access-Control-Allow-Origin」などのCORSヘッダーを追加する必要があります。 フロントエンド側では、開発環境でプロキシを使って一時的に回避することも可能ですが、根本的な解決にはなりません。

  3. Q3. HTTPSがHTTPより優れている点を2つ挙げてください。

    回答を表示

    (1) 通信内容が暗号化されるため、第三者による盗聴を防ぐことができる
    (2) データの改ざんを検知できるため、通信途中でデータが書き換えられても気づくことができる
    （他にも、サーバーの身元を証明できる、ブラウザに「安全」と表示される、なども正解）
