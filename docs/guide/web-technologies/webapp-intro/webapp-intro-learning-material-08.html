第8章

REST APIの基本

APIとは何か、RESTの設計原則について学びます。エンドポイントの設計方法を理解し、現代的なWebサービスの仕組みを把握します。

この章で学ぶこと

  • APIとは何かを説明できるようになる
  • RESTの設計原則を理解する
  • RESTful APIのエンドポイント設計方法を把握する

8.1 APIとは何か

API（Application Programming Interface）は、ソフトウェア同士がやり取りするための「窓口」や「約束事」です。 日本語では「アプリケーション・プログラミング・インターフェース」と呼ばれます。

身近な例で考えてみよう

レストランの注文方法がAPIと似ています。

  • お客さん = あなたのアプリ（クライアント）
  • メニュー = API仕様書（何が注文できるか）
  • 厨房 = サーバー（実際に処理する場所）

お客さんは厨房の中身を知らなくても、メニューを見て注文すれば料理が出てきます。 同じように、APIがあればサーバー内部の仕組みを知らなくても、決められた方法でリクエストすればデータが返ってきます。

Webアプリケーションの世界では、APIは主に「データをやり取りするための窓口」として使われます。 例えば、天気予報アプリが気象データを取得したり、SNSアプリが投稿を取得したりするのもAPIを通じて行われています。

日常で使われているAPIの例

天気予報API

現在地の天気や週間天気予報を取得

地図API

住所から緯度経度を取得、ルート検索

決済API

クレジットカード決済処理を実行

SNS API

投稿の取得、いいね、コメントの追加

8.2 RESTとは何か

REST（Representational State Transfer）は、Webアプリケーションでよく使われるAPI設計の考え方です。 RESTの原則に従って設計されたAPIをRESTful APIと呼びます。

RESTの4つの重要な考え方

1. リソース指向

URLで「リソース」（データやモノ）を表現する。例: /users/123

2. HTTPメソッドの活用

GET（取得）、POST（作成）、PUT（更新）、DELETE（削除）を正しく使い分ける

3. ステートレス

サーバーは前回のリクエストを覚えていない。毎回必要な情報をすべて送る

4. 標準的なフォーマット

データは主にJSON形式でやり取りする

RESTの考え方を採用することで、APIが分かりやすく、使いやすくなります。 世界中の多くのWebサービスがRESTful APIを採用しています。

8.3 エンドポイントとは

エンドポイントは、APIにアクセスするための「URL」のことです。 レストランのメニューで言えば、各料理を注文するための「メニュー番号」のようなものです。

ブログAPIのエンドポイント例

HTTPメソッド エンドポイント 説明
GET /posts 全ての記事を取得
GET /posts/123 ID=123の記事を取得
POST /posts 新しい記事を作成
PUT /posts/123 ID=123の記事を更新
DELETE /posts/123 ID=123の記事を削除

ポイント: URLの命名規則

良い例: /users, /posts, /comments

  • 名詞の複数形を使う
  • 小文字で統一
  • 階層構造を使う（例: /posts/123/comments）

悪い例: /getUser, /createPost

  • 動詞を入れない（HTTPメソッドで表現）
  • 大文字小文字が混在

8.4 APIリクエストとレスポンスの流れ

APIを使うときの流れを見てみましょう。クライアント（アプリ）がサーバーにリクエストを送り、サーバーがレスポンスを返します。

記事取得APIの流れ

sequenceDiagram participant C as クライアント（アプリ） participant S as サーバー（API） participant DB as データベース C->>S: GET /posts/123 Note over S: リクエストを受け取る S->>DB: ID=123の記事を検索 DB->>S: 記事データを返す Note over S: JSONに変換 S->>C: JSON形式で記事データを返す Note over C: 画面に表示

APIを通じてデータを取得する一連の流れ

具体的なリクエストとレスポンスの例

リクエスト
HTTPリクエスト

Host: api.example.com
Accept: application/json
レスポンス

このように、クライアントはエンドポイントにリクエストを送り、サーバーはJSON形式でデータを返します。 この仕組みがあることで、様々なアプリケーションが同じAPIを利用できます。

8.5 RESTful APIの設計パターン

実際のWebサービスでよく使われるAPIの設計パターンをいくつか見てみましょう。

パターン1: コレクションの取得

GET /users

全てのユーザー一覧を取得

使用例: ユーザー一覧画面、検索結果の表示

パターン2: 個別リソースの取得

GET /users/456

ID=456のユーザー情報を取得

使用例: ユーザー詳細画面、プロフィール表示

パターン3: ネストしたリソース

GET /users/456/posts

ID=456のユーザーが書いた記事一覧を取得

使用例: ユーザーの投稿一覧、特定カテゴリの商品一覧

パターン4: クエリパラメータでのフィルタリング

GET /posts?category=tech&limit=10

カテゴリ=techの記事を10件まで取得

使用例: 検索機能、ページネーション、ソート

設計のコツ

  • URLは「名詞」で表現し、「動詞」は入れない
  • 複数形を使う（users, posts, comments）
  • 階層構造を適切に使う（深すぎない、浅すぎない）
  • HTTPメソッドで操作内容を表現する

8.6 APIとHTTPステータスコード

RESTful APIでは、HTTPステータスコードを使って処理結果を伝えます。 適切なステータスコードを返すことで、クライアントは処理が成功したか失敗したかを判断できます。

よく使われるステータスコード

200 OK

リクエストが成功（GET、PUTなど）

201 Created

リソースの作成に成功（POST）

204 No Content

リクエストは成功したが返すデータなし（DELETE）

400 Bad Request

リクエストの形式が間違っている

401 Unauthorized

認証が必要

404 Not Found

リソースが見つからない

500 Internal Server Error

サーバー内部エラー

理解度確認クイズ

  1. Q1. APIとは何ですか？身近な例を使って説明してください。

    回答を表示

    APIはソフトウェア同士がやり取りするための窓口です。レストランで例えると、メニュー（API仕様）を見て注文（リクエスト）すれば、厨房の仕組みを知らなくても料理（レスポンス）が出てくるのと似ています。

  2. Q2. RESTful APIでブログ記事を取得・作成・更新・削除する場合、それぞれどのHTTPメソッドとエンドポイントを使いますか？

    回答を表示
    • • 取得: GET /posts/123
    • • 作成: POST /posts
    • • 更新: PUT /posts/123
    • • 削除: DELETE /posts/123

  3. Q3. なぜRESTful APIではURLに動詞を入れずに名詞を使うのですか？

    回答を表示

    操作の種類（取得、作成、更新、削除）はHTTPメソッド（GET、POST、PUT、DELETE）で表現するため、URLには対象となるリソース（名詞）だけを記述します。これにより、URLがシンプルになり、同じURLでもHTTPメソッドを変えることで異なる操作ができます。
