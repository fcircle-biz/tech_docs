Webアプリの3層構造

フロントエンド、バックエンド、データベースの3層構造について学びます。それぞれの役割と連携方法を理解し、Webアプリケーション全体の構成を把握します。

この章で学ぶこと

  • Webアプリケーションの3層構造を理解する
  • フロントエンド、バックエンド、データベースの役割を説明できる
  • 各層の連携方法と全体的なデータの流れを把握する

10.1 Webアプリケーションの全体像

現代のWebアプリケーションは、複数の層（レイヤー）に分かれて構成されています。 それぞれの層が明確な役割を持ち、互いに連携することで、一つのWebアプリケーションとして機能します。

身近な例で考えてみよう

レストランを例に考えてみましょう。
ホールスタッフ（フロントエンド）が注文を受け、キッチンスタッフ（バックエンド）が料理を作り、冷蔵庫（データベース）に食材が保管されています。 それぞれが役割分担をすることで、スムーズに料理が提供されます。

拡張性: 一つの層だけを強化することができる

10.2 第1層: フロントエンド（Presentation Layer）

フロントエンドは、ユーザーが直接目にし、操作する部分です。 Webブラウザ上で動作し、ユーザーインターフェース（UI）を提供します。

フロントエンドの主な役割

1
画面表示とデザイン

HTML/CSSを使って、ユーザーが見やすく、使いやすい画面を構築します。

2
ユーザー操作の処理

JavaScriptを使って、ボタンクリック、フォーム入力などのユーザー操作に反応します。

3
バックエンドとの通信

APIを通じてバックエンドにデータを要求し、受け取ったデータを画面に表示します。

HTML

Webページの構造と内容を定義

CSS

見た目（色、サイズ、レイアウト）を定義

JavaScript

ユーザー操作への反応や動的な処理

代表的なフロントエンドフレームワーク

  • React: Facebookが開発した人気のフレームワーク
  • Vue.js: 学習しやすいことで評価が高い
  • Angular: Googleが開発した大規模向けフレームワーク
  • Next.js: Reactをベースにした高機能フレームワーク

10.3 第2層: バックエンド（Application Layer）

バックエンドは、ユーザーから見えないサーバー側の処理を担当します。 ビジネスロジック（アプリケーションの中核となる処理）を実行し、データベースとやり取りします。

バックエンドの主な役割

1
リクエストの受け取りとレスポンス

フロントエンドからのリクエストを受け取り、適切な処理を実行して結果を返します。

2
ビジネスロジックの実行

データの加工、計算、検証など、アプリケーションの核となる処理を実行します。

3
データベースとのやり取り

データベースからデータを取得したり、新しいデータを保存したりします。

4
セキュリティとアクセス制御

ユーザー認証や権限チェックなど、セキュリティに関わる処理を担当します。

バックエンド処理の例（Python/FastAPI） 
# ユーザー登録のエンドポイント
@app.post("/api/users")
def create_user(user_data: dict):
    # 1. データの検証
    if not user_data.get("email"):
        raise HTTPException(400, "メールアドレスが必要です")

    # 2. ビジネスロジック（パスワードのハッシュ化）
    hashed_password = hash_password(user_data["password"])

    # 3. データベースへの保存
    db.save_user(user_data["email"], hashed_password)

    # 4. レスポンスを返す
    return {"message": "ユーザーを登録しました"}

10.4 第3層: データベース（Data Layer）

データベースは、アプリケーションが扱うデータを永続的に保存する層です。 ユーザー情報、投稿内容、商品データなど、様々な情報がここに蓄積されます。

データベースの主な役割

1
データの永続化

サーバーが再起動してもデータが失われないように、安全に保存します。

2
効率的なデータ検索

大量のデータの中から必要な情報を素早く見つけ出すことができます。

3
データの整合性保持

複数のユーザーが同時にデータを操作しても、矛盾が生じないように管理します。

リレーショナルDB

表形式でデータを管理。構造化されたデータに適している。

例: MySQL、PostgreSQL、SQLite

NoSQL

柔軟な形式でデータを管理。非構造化データに適している。

例: MongoDB、Redis、DynamoDB

10.5 3層の連携とデータの流れ

実際のWebアプリケーションでは、3つの層が連携して動作します。 ユーザーの操作がどのように処理され、結果が返ってくるのかを見てみましょう。

ユーザー登録時のデータフロー

sequenceDiagram participant User as ユーザー participant FE as フロントエンド participant BE as バックエンド participant DB as データベース User->>FE: フォームに情報入力
送信ボタンをクリック FE->>FE: 入力内容を検証 FE->>BE: POST /api/users
（JSONデータ送信） BE->>BE: データ検証
パスワードをハッシュ化 BE->>DB: INSERT INTO users... DB->>BE: 保存完了 BE->>FE: {"success": true} FE->>User: "登録が完了しました"

各層が連携してユーザー登録処理を実行

具体例: ECサイトで商品を閲覧する

  1. フロントエンド: ユーザーが商品ページを開く
  2. バックエンド: 商品IDを受け取り、データベースに問い合わせ
  3. データベース: 商品情報を返す（商品名、価格、在庫など）
  4. バックエンド: JSONに変換してフロントエンドに返す
  5. フロントエンド: データを受け取り、きれいに整形して画面表示

なぜ3層に分けるのか？

開発の分業がしやすい

フロントエンド担当、バックエンド担当、データベース担当など、役割を分けて効率的に開発できます。

変更の影響範囲が限定される

例えば、デザイン変更はフロントエンドだけで対応でき、バックエンドやデータベースに影響しません。

異なる技術を組み合わせられる

フロントエンドはReact、バックエンドはPython、データベースはPostgreSQLなど、最適な技術を選べます。

10.6 この章のまとめ

1

Webアプリはフロントエンド、バックエンド、データベースの3層構造で構成される

2

3
