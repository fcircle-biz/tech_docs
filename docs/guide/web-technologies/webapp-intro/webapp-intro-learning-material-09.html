データベースの役割

なぜWebアプリにデータベースが必要なのか、データの永続化とは何かを学びます。リレーショナルデータベース（RDB）とNoSQLの違いについても解説します。

この章で学ぶこと

  • データベースの必要性と役割を理解する
  • データの永続化とは何かを説明できるようになる
  • リレーショナルデータベース（RDB）とNoSQLの違いを理解する

9.1 データベースとは何か

データベース（Database）は、大量のデータを効率的に保存・管理・検索するための仕組みです。 Webアプリケーションでは、ユーザー情報、商品情報、投稿内容など、様々なデータをデータベースに保存します。

身近な例で考えてみよう

データベースは「図書館」のようなものです。 図書館には大量の本が整理されて保管されており、検索システムを使えば欲しい本をすぐに見つけることができます。 データベースも同じように、大量のデータを整理して保管し、必要なデータをすぐに取り出せるようにします。

例えば、ECサイト（ネットショップ）では、以下のようなデータがデータベースに保存されています：

ユーザー情報

名前、メールアドレス、住所、電話番号など

商品情報

商品名、価格、在庫数、説明文、画像URLなど

注文情報

注文日時、購入者、購入商品、金額など

レビュー情報

レビュー本文、評価、投稿日時など

9.2 データの永続化とは

データの永続化とは、データを長期的に保存し、プログラムが終了しても消えないようにすることです。

ポイント: なぜ永続化が必要？

サーバーのメモリ上に保存したデータは、サーバーを再起動すると消えてしまいます。

ユーザー登録情報や注文履歴など、長期的に保存したいデータはデータベースに保存する必要があります。

メモリとデータベースの違い

一時的なデータはメモリ、長期保存が必要なデータはデータベースへ

永続化が必要なデータ・不要なデータ

永続化が必要（データベース）
  • • ユーザーのアカウント情報
  • • 投稿した記事やコメント
  • • 購入履歴
  • • 商品マスタデータ
永続化不要（メモリ）
  • • セッション情報（ログイン状態）
  • • 一時的なキャッシュデータ
  • • 計算途中の値
  • • リクエスト処理中のデータ

9.3 リレーショナルデータベース（RDB）

リレーショナルデータベース（RDB: Relational Database）は、データを「表（テーブル）」の形で管理するデータベースです。 最も広く使われているデータベースの種類で、SQL（Structured Query Language）という言語でデータを操作します。

RDBの特徴

1
表形式でデータを管理

Excelのように、行（レコード）と列（カラム）でデータを整理します

2
テーブル同士を関連付けられる

「ユーザーテーブル」と「注文テーブル」を関連付けて、誰がどの商品を買ったか管理できます

3
データの整合性を保つ

存在しないユーザーの注文データを作成できないなど、データの矛盾を防ぎます

テーブルの例: ユーザーテーブル

ID 名前 メールアドレス 登録日
1 山田太郎 yamada@example.com 2024-01-15
2 佐藤花子 sato@example.com 2024-01-20
3 鈴木一郎 suzuki@example.com 2024-02-01

1行が1ユーザーのデータ、各列が属性（ID、名前、メールアドレスなど）

PostgreSQL

高機能でオープンソース、企業でも広く使われる

MySQL

世界中で人気、Webアプリ開発で定番

SQLite

軽量で組み込み可能、学習・開発に便利

9.4 NoSQLデータベース

NoSQL（Not Only SQL）は、RDBとは異なる方式でデータを管理するデータベースです。 表形式ではなく、JSONのようなドキュメント形式や、キー・バリュー形式でデータを保存します。

NoSQLの特徴

1
柔軟なデータ構造

表形式にこだわらず、自由な形でデータを保存できます

2
スケーラビリティ（拡張性）

大量のデータや高いアクセス数に対応しやすい設計

3
高速な読み書き

シンプルな操作に特化し、高速処理が可能

ドキュメント型

JSON形式でデータを保存

例: MongoDB, Firestore

キー・バリュー型

キーに対して値を1つ保存

例: Redis, DynamoDB

MongoDBのドキュメント例（JSON形式） 
{
  "_id": "507f1f77bcf86cd799439011",
  "name": "山田太郎",
  "email": "yamada@example.com",
  "orders": [
    { "product": "ノートPC", "price": 80000 },
    { "product": "マウス", "price": 2000 }
  ]
}

9.5 RDBとNoSQLの使い分け

RDBとNoSQLには、それぞれ得意な用途があります。どちらが優れているというわけではなく、システムの要件に応じて使い分けます。

RDB vs NoSQL 比較

比較項目 RDB NoSQL
データ構造 表形式（テーブル） JSON、キー・バリューなど
スキーマ 固定（事前定義が必要） 柔軟（動的に変更可能）
データ整合性 強い（ACID保証） 緩い場合が多い
スケーラビリティ 垂直拡張が中心 水平拡張が得意
得意な用途 複雑なクエリ、トランザクション 大量データ、高速読み書き

用語解説

ACID保証: データの信頼性を保つ仕組み（原子性、一貫性、独立性、永続性）

垂直拡張: サーバーの性能を上げて対応（CPU、メモリを増やす）

水平拡張: サーバーの台数を増やして対応

RDBが向いている場合
  • • 銀行の口座管理システム
  • • ECサイトの注文管理
  • • 社員情報管理システム
  • • 複雑な検索条件がある場合
NoSQLが向いている場合
  • • SNSの投稿データ
  • • リアルタイムチャット
  • • アクセスログ分析
  • • 高速な読み書きが必要な場合

理解度確認クイズ

  1. Q1. データの永続化とは何ですか？なぜ必要ですか？

    回答を表示

    データの永続化とは、データを長期的に保存し、プログラム終了後も消えないようにすることです。サーバーのメモリに保存したデータは再起動すると消えてしまうため、ユーザー情報や注文履歴など長期保存が必要なデータはデータベースに保存する必要があります。

  2. Q2. RDBとNoSQLの主な違いは何ですか？

    回答を表示

    RDBは表形式でデータを管理し、データ整合性が強く、複雑なクエリに向いています。NoSQLはJSON形式やキー・バリュー形式など柔軟なデータ構造を持ち、大規模データの高速処理やスケーラビリティに優れています。

  3. Q3. 銀行の口座管理システムにはRDBとNoSQLのどちらが向いていますか？その理由も説明してください。

    回答を表示

    RDBが向いています。銀行の口座管理では、残高の整合性やトランザクションの正確性が極めて重要です。RDBはACID保証により、データの整合性を強力に保つことができるため、金融システムには必須の特性です。
