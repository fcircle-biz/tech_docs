APIとWebhookの違い 〜プル型とプッシュ型〜

通常のAPI（ポーリング）とWebhookの違いを「プル型」と「プッシュ型」という視点から比較します。 それぞれの長所・短所を理解し、どのような場面でWebhookが適しているのかを学びます。

この章で学ぶこと

  • 通常のAPI（プル型）とWebhook（プッシュ型）の根本的な違いを説明できる
  • ポーリングの仕組みと問題点を理解できる
  • それぞれの長所・短所を整理し、使い分けの判断基準を持てる
  • 「Webhookが適している場面」と「そうでない場面」を具体例で挙げられる

2.1 「プル型」と「プッシュ型」とは何か

データ通信の世界では、情報の受け取り方に大きく2つのパターンがあります。 これを「プル型（Pull）」と「プッシュ型（Push）」と呼びます。

この2つの違いを、まずは日常生活のイメージで理解しましょう。

プル型（Pull）

自分から取りに行く

「情報が必要になったら、自分から問い合わせる」方式。 川の水を自分でくみに行くイメージです。

日常の例

  • ・メールを確認したいときにアプリを開く
  • ・天気予報サイトを自分で開いて確認する
  • ・株価を調べたいときにサイトを見に行く

プッシュ型（Push）

向こうから送ってくれる

「何かが起きたとき、相手が自動的に送ってくれる」方式。 川の水が自動的にパイプで届くイメージです。

日常の例

  • ・LINEでメッセージが届くと通知が来る
  • ・大雨警報がスマホに自動で届く
  • ・株価が急変したときにアラートが届く

重要ポイント

通常のAPIはプル型（自分でリクエストを送り、データを取得する）、 Webhookはプッシュ型（サービス側からデータを送ってもらう）です。 どちらも「データをやり取りする仕組み」ですが、主導権が異なります。

2.2 通常のAPI（プル型）の仕組み

通常のAPIは「クライアント（あなたのアプリ）がリクエストを送り、サーバーが返答する」という仕組みです。 これは「質問と回答」の形式で、必ず自分から動き出す必要があります。

通常のAPIの通信フロー

sequenceDiagram participant C as クライアント participant S as サーバー（APIサービス） C->>S: リクエスト送信（データをください） S-->>C: レスポンス（はい、データです） Note over C: しばらく後... C->>S: リクエスト送信（新しいデータありますか？） S-->>C: レスポンス（変更はありません） Note over C: また少し後... C->>S: リクエスト送信（データをください） S-->>C: レスポンス（新しいデータが2件あります）

この「何度も確認しに行く」動作をポーリング（Polling）と呼びます。 ポーリングは英語の「poll（調査する・確認する）」から来た言葉で、 定期的に状態を問い合わせる動作を指します。

ポーリングの3つの問題点

1. 無駄なリクエストが多い

「何も変化がない」ときでもリクエストを送り続けるため、サーバーに余分な負荷がかかります。 100回確認して1回しかデータが来なかった場合、99回は無駄です。

2. リアルタイム性に限界がある

「1分ごとに確認」していると、最大1分間のタイムラグが生じます。 注文が来た瞬間に処理したい場合には不向きです。

3. 実装・運用コストがかかる

定期的に確認するための仕組み（タイマー処理）を自分で作る必要があります。 確認間隔を短くするほど、コストが増えます。

2.3 Webhook（プッシュ型）の仕組み

Webhookはポーリングの問題点を解決するために生まれた仕組みです。 「イベントが発生したときだけ通知する」という考え方で、 無駄な通信をなくすことができます。

Webhookの通信フロー

sequenceDiagram participant C as あなたのアプリ participant S as 外部サービス Note over C,S: 事前設定（1回だけ） C->>S: URLを登録（このURLに通知してください） S-->>C: 了解しました Note over C,S: イベント発生まで待機... Note over S: イベント発生！ S->>C: Webhook通知（イベントが起きました！） C-->>S: 受け取りました（200 OK）

ポーリングと比べると、やり取りの回数が劇的に減っていることがわかります。 Webhookでは「URLを登録する」という1回の設定をするだけで、 あとはイベントが起きたときだけ通信が発生します。

即時通知

イベント発生と同時に通知が届くため、タイムラグがほぼありません。

効率的な通信

必要なときだけ通信するので、サーバーへの負荷が最小限で済みます。

シンプルな設定

URLを1つ登録するだけで連携が始まります。複雑なタイマー処理が不要です。

2.4 ポーリングとWebhookを徹底比較

2つの方式の違いを、具体的な数字や特徴で比較してみましょう。

ポーリング vs Webhook 比較表

比較項目 ポーリング（プル型） Webhook（プッシュ型）
主導権 クライアント側（自分） サーバー側（相手）
通信タイミング 定期的（常に確認） イベント発生時のみ
リアルタイム性 低い（タイムラグあり） 高い（即時通知）
サーバー負荷 高い（無駄なリクエスト多） 低い（必要時のみ）
実装のしやすさ 比較的簡単（呼び出すだけ） 受信エンドポイントの準備が必要
向いている場面 定期的なデータ取得・シンプルな要件 リアルタイム通知・イベント駆動処理
典型的な用途 天気予報の取得・ニュース一覧の更新 決済完了通知・メッセージ受信

通信量のイメージ比較

flowchart LR subgraph Polling["ポーリング（プル型）"] direction TB P1["問い合わせ①: 変化なし"] --> P2["問い合わせ②: 変化なし"] P2 --> P3["問い合わせ③: 変化なし"] P3 --> P4["問い合わせ④: 変化なし"] P4 --> P5["問い合わせ⑤: イベント発生！"] end subgraph Webhook["Webhook（プッシュ型）"] direction TB W1["URL登録（1回のみ）"] --> W2["待機..."] W2 --> W3["イベント発生！通知が届く"] end

具体的なイメージ

ポーリングは「郵便受けに何度も確認しに行く」、Webhookは「宅配業者がドアベルを鳴らしてくれる」イメージです。 後者のほうが効率的なのは明らかですね。

2.5 どちらを使うべき？ 使い分けの考え方

Webhookが優れているからといって、常にWebhookを使えばよいわけではありません。 状況によっては通常のAPIのほうが適している場合もあります。

どちらを選ぶかの判断フロー

flowchart TD A[データを取得したい] --> B{イベント発生時にすぐ処理したい？} B -->|はい| C{相手サービスがWebhookを提供している？} B -->|いいえ| D["通常のAPI
（定期バッチや手動取得）"] C -->|はい| E["Webhookを使う"] C -->|いいえ| F["ポーリングで代替
（または短いポーリング間隔）"]

シナリオ別の使い分け例

Webhookが適している場面

  • 決済完了の通知: 支払いが確定した瞬間に在庫更新や確認メール送信を行いたい
  • チャットボット: ユーザーがメッセージを送ったときに即座に応答したい
  • CI/CDパイプライン: コードがプッシュされたら自動でテストを実行したい
  • 在庫アラート: 在庫が変化したときにチームに通知したい

通常のAPIが適している場面

  • 商品一覧の取得: ユーザーが画面を開いたタイミングで最新データを表示したい
  • 天気予報の取得: 1時間ごとに最新の天気情報を更新したい
  • 相手がWebhookに未対応: 連携したいサービスがWebhookを提供していない
  • データの検索・取得: ユーザーの操作に応じてデータを取得する場合

注意

Webhookを使う場合、自分のアプリ側にも「受信できる環境」が必要です。 インターネットからアクセス可能なURL（エンドポイント）を用意する必要があります。 ローカル（自分のパソコン）で開発中の場合は追加の設定が必要になります。

2.6 WebhookとAPIは対立するものではない

「WebhookかAPIか」は、どちらかを選んで他方を捨てるという話ではありません。 実際の現場では、両方を組み合わせて使うことがよくあります。

ECサイトでのAPIとWebhookの組み合わせ例

flowchart TD A[ユーザー] -->|"商品を検索する（API呼び出し）"| B["ECサイト"] B -->|"商品一覧を返す（APIレスポンス）"| A A -->|"購入ボタンを押す（API呼び出し）"| C["決済サービス"] C -->|"支払い完了（Webhook通知）"| B B -->|"注文確定メールを送る"| D[メールサービス]

この例では、「商品を検索する」「購入ボタンを押す」というユーザーのアクションにはAPIを使い、 「支払い完了の通知」という外部サービスからのイベントにはWebhookを使っています。

まとめの重要ポイント

APIは「自分から情報を取りに行く道具」、Webhookは「相手が情報を送ってくれる仕組み」です。 どちらが優れているという話ではなく、状況に応じて使い分けること、 そして両方を組み合わせることが実際の開発では一般的です。

理解度確認クイズ

  1. Q1. 「ポーリング」とはどのような通信方式ですか？ (A) サーバーが変化を検知してクライアントに通知する方式 (B) クライアントが定期的にサーバーへ問い合わせ続ける方式 (C) クライアントとサーバーが常時接続を維持する方式

    回答を表示

    答えは (B) です。ポーリングはクライアントが「何か変化がありましたか？」と繰り返し問い合わせる方式です。 (A) はWebhook（プッシュ型）の説明、(C) はWebSocketなどの「常時接続」の説明です。

  2. Q2. ポーリングの問題点として正しくないものはどれですか？ (A) 無駄なリクエストが多くサーバーに負荷がかかる (B) イベント発生のタイミングと通知のタイミングにズレが生じる (C) URLを登録する設定作業が難しい

    回答を表示

    正しくないのは (C) です。「URLを登録する設定作業」はWebhookの特徴であり、ポーリングには関係ありません。 ポーリングの実際の問題点は (A) 無駄なリクエストと (B) タイムラグです。

  3. Q3. 次のうち、Webhookよりも通常のAPI（ポーリング）が向いている場面はどれですか？ (A) クレジットカードの支払いが完了したときに、すぐ在庫を更新したい (B) ユーザーが「検索」ボタンを押したときに、商品一覧を表示したい (C) GitHubにコードが追加されたとき、自動でテストを実行したい

    回答を表示

    答えは (B) です。「ユーザーのアクションに応じてデータを取得する」場合は、通常のAPI呼び出しが適しています。 (A) と (C) はイベント発生時の即時処理が必要なため、Webhookが向いています。
