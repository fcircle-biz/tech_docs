身近なWebhook活用例（1）〜チャットツール連携〜

SlackやMicrosoft Teamsなどのチャットツールで使われるWebhook（Incoming Webhook）の仕組みを学びます。「特定のイベントが起きたらチャットに自動通知する」という身近なユースケースを理解しましょう。

この章で学ぶこと

  • Incoming Webhook（着信Webhook）とは何か、通常のWebhookとの違いを説明できる
  • SlackのIncoming Webhookの仕組みと、メッセージ送信の流れを理解できる
  • Microsoft TeamsのWebhookコネクタの使い方の概要を把握できる
  • チャットツール連携の典型的なユースケース（監視アラート・自動通知など）を挙げることができる

7.1 チャットツール連携とは

現代のビジネス現場では、SlackやMicrosoft Teamsなどのチャットツールが情報共有の中心になっています。 ここにWebhookを組み合わせることで、「何か重要なことが起きたら自動でチャットに通知する」という仕組みが作れます。

たとえば、以下のような「自動通知」が実現できます。

サーバー障害アラート

監視ツールがサーバーの異常を検知したとき、Slackの「障害対応」チャンネルに即座にアラートを送信する

注文通知

ECサイトに新しい注文が入ったとき、「営業」チャンネルに「注文が入りました！」と自動で知らせる

コードレビュー依頼

GitHubにプルリクエストが作成されたとき、「開発」チャンネルにレビュー依頼の通知を送る

定期レポート自動投稿

毎朝9時に、前日の売上集計をWebhookでTeamsに自動投稿する

重要ポイント

チャットツール連携でのWebhookは「外部システム → チャットツール」という方向が一般的です。 これを特に Incoming Webhook（着信Webhook） と呼びます。 チャットツールが「受け取る側」になるのが特徴です。

7.2 Incoming Webhookとは

チャットツールにはメッセージを受け取るための専用URL（Incoming Webhook URL）が発行できます。 このURLに向けてHTTPリクエストを送るだけで、チャンネルにメッセージを投稿できます。

Incoming Webhook vs 通常のWebhook

項目 Incoming Webhook 通常のWebhook
データの方向 外部→チャットツール（受け取る） 外部→あなたのアプリ（受け取る）
URLの用意 チャットツールが発行してくれる 自分でサーバーを用意する必要がある
目的 チャットへのメッセージ投稿 任意の処理（メール送信・DB保存など）
難易度 簡単（URL取得＋リクエスト送信のみ） やや複雑（受信サーバーの実装が必要）

豆知識

Incoming Webhookは「誰でも簡単にチャットへ通知を送れる」ように設計されています。 プログラムから送ることもできますし、ノーコードツール（Zapierなど）からも使えます。 初心者がWebhookを体験するのにも最適な入口です。

7.3 SlackのIncoming Webhook

Slackは世界で広く使われているビジネスチャットツールです。 SlackのIncoming Webhookを使うと、外部システムから簡単にSlackチャンネルへメッセージを投稿できます。

Slack Incoming Webhook 設定の流れ

  1. 1 Slackワークスペースの「Apps」から「Incoming Webhooks」アプリを追加する
  2. 2 メッセージを投稿したいチャンネルを選択して「許可する」をクリック
  3. 3 発行されたWebhook URL（https://hooks.slack.com/services/...）をコピーする
  4. 4 このURLにJSONデータをPOSTするだけで、指定チャンネルにメッセージが届く

SlackへのメッセージデータはJSON形式で送る

Slackに送るメッセージは、JSONというデータ形式で書きます（JSONの詳細は第5章を参照）。 最もシンプルな形は以下のとおりです。

slack-message.json（最もシンプルなメッセージ） 
{
  "text": "サーバーの監視アラート: CPU使用率が90%を超えました！"
}

また、メッセージを見やすく装飾する「Block Kit」という仕組みも使えます。 ボタンや絵文字、色付きのサイドバーなどを追加したリッチなメッセージを作れます。

slack-rich-message.json（装飾付きメッセージの例） 
{
  "text": "注文通知",
  "attachments": [
    {
      "color": "#36a64f",
      "title": "新しい注文が入りました！",
      "fields": [
        {
          "title": "注文番号",
          "value": "#12345",
          "short": true
        },
        {
          "title": "金額",
          "value": "¥3,500",
          "short": true
        }
      ]
    }
  ]
}

Slackに届くメッセージのイメージ

監視Bot 今日 9:05

新しい注文が入りました！

注文番号

#12345

金額

¥3,500

注意

Slack Incoming Webhook URLは「シークレット」として扱ってください。 このURLを知っている人は誰でもそのチャンネルにメッセージを投稿できてしまいます。 GitHubのコードや公開ドキュメントにURLをそのまま書かないようにしましょう。

7.4 Microsoft TeamsのWebhook連携

Microsoft Teams（通称「Teams」）は、Microsoft 365の一部として多くの企業で使われているチャット・コラボレーションツールです。 TeamsにもSlack同様、外部からメッセージを受け取る「Incoming Webhook」の仕組みがあります。

Slack Incoming Webhook

  • Slackアプリから設定
  • シンプルなJSONを送るだけでOK
  • Block Kitで柔軟なメッセージ装飾が可能

Teams Incoming Webhook

  • チャンネルの「コネクタ」から設定
  • Office 365 Connector Cardという形式でメッセージを送る
  • Adaptive Cardsでリッチなメッセージも可能

TeamsへのメッセージJSON例

teams-message.json（Teams向けシンプルメッセージ） 
{
  "@type": "MessageCard",
  "@context": "https://schema.org/extensions",
  "summary": "障害アラート",
  "themeColor": "FF0000",
  "title": "サーバー障害を検知しました",
  "text": "本番サーバー(web-01)がダウンしています。すぐに確認してください。"
}

ヒント

SlackとTeams、どちらを選べばよいかは「あなたの会社・チームが何を使っているか」で決まります。 両方を使い分けたい場合は、同じイベントから複数のWebhook URLにリクエストを送ることも技術的には可能です。

7.5 Webhookの送受信の全体フロー

チャットツール連携におけるWebhookの動きを、シーケンス図（時系列の流れ図）で確認しましょう。 ここでは「監視ツールがサーバー障害を検知してSlackに通知する」例で説明します。

監視ツール → Slack通知のシーケンス

sequenceDiagram autonumber participant M as 監視ツール participant S as 外部サービス participant SL as Slack API M->>S: サーバーの死活監視（定期チェック） S-->>M: 応答なし（タイムアウト） Note over M: 障害を検知！ M->>SL: POSTリクエスト送信
Webhook URLへメッセージデータを送る SL-->>M: 200 OK（受信成功） Note over SL: 指定チャンネルに
メッセージ投稿

このフローで重要なのは、Slackへのメッセージ送信は監視ツール側から能動的に行うという点です。 Slackは「誰かが送ってきたら受け取る」という受け身の立場です。

成功時（200 OK）

Slackは「ok」というレスポンスを返します。メッセージが正常に投稿されたことを意味します。

HTTP 200 OK
Body: "ok"

失敗時（4xx エラー）

URLが無効・JSON形式が間違い・権限不足などの場合、エラーメッセージが返ります。

HTTP 400 Bad Request
Body: "invalid_payload"

7.6 代表的な活用パターン

チャットツールへのWebhook通知は、実際の現場でさまざまな場面で活用されています。代表的なパターンを整理してみましょう。

パターン1: システム監視・アラート通知

トリガーとなるイベント

  • ・サーバーがダウン
  • ・CPU/メモリが閾値超過
  • ・エラーが大量発生

使われるツール

  • ・Datadog / Zabbix
  • ・PagerDuty
  • ・CloudWatch (AWS)

通知先チャンネル例

  • ・#障害対応
  • ・#インフラ監視
  • ・@エンジニアチーム

パターン2: 開発ワークフロー通知

トリガーとなるイベント

  • ・プルリクエスト作成
  • ・コードレビュー依頼
  • ・自動テスト完了・失敗

使われるツール

  • ・GitHub / GitLab
  • ・CircleCI / Jenkins
  • ・Jira / Trello

通知先チャンネル例

  • ・#開発
  • ・#デプロイ通知
  • ・#コードレビュー

パターン3: ビジネス業務の自動通知

トリガーとなるイベント

  • ・新規注文・問い合わせ
  • ・支払い完了・失敗
  • ・フォーム送信

使われるツール

  • ・Stripe（決済）
  • ・HubSpot（CRM）
  • ・Salesforce

通知先チャンネル例

  • ・#営業
  • ・#決済通知
  • ・#カスタマーサクセス

7.7 Webhook URLを安全に管理する

チャットツールのWebhook URLは「誰でもその場所にメッセージを投稿できるURL」です。 管理が甘いとスパムメッセージの被害に遭ったり、機密情報が外部に漏れたりする可能性があります。

Webhook URLの安全な管理方法

1

コードにURLを直接書かない

URLを SLACK_WEBHOOK_URL などの環境変数に保存し、プログラムから参照するようにします。GitHubにコードをアップロードしてもURLが漏れません。

2

必要最低限のチャンネルにだけ設定する

1つのWebhook URLは特定のチャンネルだけに投稿できます。重要なチャンネルへのURLは厳重に管理し、用途ごとに別々のURLを発行しましょう。

3

不要になったURLは無効化する

使わなくなったWebhook URLは、Slackの設定から削除（無効化）しておきましょう。古いURLが悪用されるリスクを減らせます。

環境変数でWebhook URLを管理する例（Python） 
import os
import requests

# 環境変数からWebhook URLを取得（コードに直書きしない）
webhook_url = os.environ.get('SLACK_WEBHOOK_URL')

# Slackに送るメッセージ
message = {
    "text": "注文が入りました！注文番号: #12345"
}

# POSTリクエストを送信
response = requests.post(webhook_url, json=message)

理解度確認クイズ

  1. Q1. Incoming Webhookを使うとき、あなたが最初に行うことは何ですか？ (A) 自分のサーバーにWebhookを受け取るプログラムを書く (B) チャットツール側でWebhook URLを発行してもらう (C) データベースにチャンネル情報を登録する

    回答を表示

    答えは (B) です。Incoming Webhookの特徴は「チャットツール側がURLを発行してくれる」点です。 自分でサーバーを用意する必要がなく、発行されたURLに向けてリクエストを送るだけで使えます。

  2. Q2. SlackのIncoming WebhookにメッセージをPOSTしたとき、成功した場合のHTTPステータスコードはどれですか？ (A) 201 Created (B) 200 OK (C) 204 No Content

    回答を表示

    答えは (B) 200 OK です。SlackはメッセージをWebhookで受け取り、正常に処理できた場合「HTTP 200 OK」と「ok」というレスポンスを返します。

  3. Q3. Slack Incoming Webhook URLを安全に管理する方法として、正しいものはどれですか？ (A) GitHubのREADMEにURLをコピーして誰でも見られるようにする (B) 環境変数に保存し、コードには直接書かない (C) 1つのURLをすべての用途で使い回す

    回答を表示

    答えは (B) です。Webhook URLは「秘密情報」として扱い、環境変数や安全な設定ファイルに保存します。 コードに直接書くとGitHubにアップロードしたときに公開されてしまいます。 (A) は絶対にやってはいけません。(C) もURLが漏れた場合の被害を最小限にするために避けましょう。
