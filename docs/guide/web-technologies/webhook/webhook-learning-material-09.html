身近なWebhook活用例（3）〜GitHub・開発ツール連携〜

GitHubのWebhookを例に、開発ツールでの活用方法を学びます。「コードがプッシュされたら自動でテストを実行する」「プルリクエストが作られたら通知する」といったCI/CD連携の基礎概念を理解しましょう。

この章で学ぶこと

  • GitHubのWebhook機能の概要と、設定できるイベントの種類を理解できる
  • プッシュ・プルリクエスト・マージなど主要なGitHubイベントがどう使われるか説明できる
  • CI/CD（継続的インテグレーション・継続的デリバリー）の基礎概念とWebhookの関係を理解できる
  • GitHubと他の開発ツール（Slack、Jiraなど）の連携パターンを具体例で挙げることができる

9.1 GitHubとWebhookの関係

GitHubはプログラマーが「ソースコード」を管理・共有するためのサービスです。 「GitHub」という名前を聞いたことがある方も多いかもしれません。 GitHub上でコードの変更やレビューなど様々な作業が行われており、 それらの「出来事（イベント）」をWebhookで外部サービスに通知できる仕組みが整っています。

開発現場では、GitHubのWebhookを使って「チームへの自動通知」「テストの自動実行」「本番環境への自動展開」 といった作業を自動化することが一般的です。これにより、人手でやっていた繰り返し作業が省け、 ミスも大幅に減ります。

GitHubはプログラマーが使うツールですが、この章ではWebhookの仕組みを理解することが目的です。 「コードの変更があったとき、それを他のサービスに自動で知らせる」というイメージで読み進めてください。

9.2 GitHubで使える主なイベント

GitHubでは非常に多くの「イベント」が定義されており、それぞれに対してWebhookを設定できます。 代表的なイベントを見ていきましょう。

GitHubの主なWebhookイベント

イベント名 いつ発生する？ よくある活用例
push コードが更新されたとき 自動テスト、Slackへの通知
pull_request レビュー依頼が作られたとき チームへの通知、自動チェック
issues 課題・バグ報告が作られたとき Jiraチケット自動作成
release 新バージョンが公開されたとき リリースノート通知、自動デプロイ
create ブランチやタグが作られたとき 開発環境の自動構築

重要ポイント

GitHubのWebhookはイベントの種類を細かく選択できます。 「全てのイベント」を受け取るのではなく、「pushイベントだけ」「pull_requestだけ」と絞ることで、 不要な通知を減らし、システムを効率的に動かすことができます。

9.3 コードプッシュからSlack通知までの流れ

最も身近な活用例として「GitHubにコードが追加されたとき、SlackにWebhookで通知を送る」という流れを 詳しく見てみましょう。プログラマーが普段どのようにこの仕組みを使っているかが分かります。

1

プログラマーがコードを書いてGitHubに送る（プッシュ）

プログラマーが自分のパソコンでコードを修正し、GitHubの「リポジトリ（コード置き場）」に送信します。 この操作を「プッシュ」と呼びます。

2

GitHubがWebhookを送信する

プッシュを受け取ったGitHubは、あらかじめ設定されたURL（エンドポイント）に対して Webhookリクエストを自動で送信します。 このリクエストには「誰が」「何を」「いつ」変更したかの情報が含まれています。

3

受信先がWebhookデータを受け取る

Slackとの連携ツール（またはZapierなどの自動化ツール）がGitHubからのWebhookを受け取ります。 JSONデータの中から「コミットメッセージ」「変更者名」などの情報を取り出します。

4

SlackにメッセージがBOT投稿される

取得した情報をもとにSlackのチャンネルにメッセージを投稿します。 チームメンバーは「誰が何を変更したか」をSlack上で即座に確認できます。

GitHubプッシュからSlack通知までのシーケンス

9.4 CI/CDとWebhook 〜自動テスト・自動デプロイの仕組み〜

開発の現場でよく聞く「CI/CD」という言葉があります。難しそうに聞こえますが、 Webhookを活用した「自動化の仕組み」のことです。一つずつ見ていきましょう。

CI（継続的インテグレーション）

コードが追加されるたびに自動でテストを実行して、 「このコードは正しく動くか？」を自動で確認する仕組みです。

Webhookとの関係

GitHubのpushイベントをWebhookで受け取り、 テスト用サーバーに「テストを開始して」と伝える

CD（継続的デリバリー）

テストが通ったコードを自動的に本番環境（ユーザーが使うサーバー）に反映する仕組みです。 「デプロイ」とも呼ばれます。

Webhookとの関係

テスト成功のイベントをWebhookで受け取り、 本番サーバーに「最新コードに更新して」と指示を出す

注意

実際のCI/CDには「GitHub Actions」「Jenkins」「CircleCI」など様々なツールが使われます。 これらはどれもWebhookを活用してGitHubと連携しています。 詳細な設定は各ツールの学習で扱いますが、「Webhookがベースになっている」ことを覚えておきましょう。

9.5 プルリクエストと自動通知

「プルリクエスト（Pull Request / PR）」とは、プログラマーが「このコードを取り込んでください」と チームに依頼する機能です。コードレビュー（確認・チェック）の出発点になります。

プルリクエストが作られたタイミングで、Webhookを使って関係するメンバーに自動で通知する仕組みが 多くの開発チームで使われています。

プルリクエストのライフサイクルとWebhook通知

開

プルリクエストを作成（opened）

→ Slackの #review チャンネルに自動投稿

審

コードレビューのコメントが追加（review_requested）

→ 担当者にメールまたはSlackでDM通知

承

プルリクエストが承認・マージされる（closed）

→ 自動デプロイ処理を開始、Slackへの完了通知

ヒント

GitHubの pull_request イベントは 「作成」「更新」「クローズ」「マージ」などの細かいアクションを区別できます。 action フィールドの値（opened, closed, merged等） を見て、どのタイミングで通知するかを制御します。

9.6 GitHubと他の開発ツールとの連携例

GitHubのWebhookは、Slackだけでなく様々な開発ツールと連携することができます。 代表的なケースを紹介します。

GitHub × Slack

コードのプッシュ、プルリクエストの作成、イシュー（バグ報告）の投稿などを Slackのチャンネルに自動通知します。 チームが GitHub を開かなくても、Slack上でリポジトリの動きをリアルタイムで把握できます。

GitHub × Jira（プロジェクト管理ツール）

GitHubのコミットメッセージに「PROJ-123を修正」と記載しておくと、 Webhookを通じてJiraのチケット（タスク管理カード）のステータスを自動更新できます。 コードの変更とタスクの進捗が自動で紐づきます。

GitHub × Vercel / Netlify（ホスティングサービス）

VercelやNetlifyはGitHubのWebhookを使って、mainブランチへのプッシュを検知し、 Webサイトを自動でビルド・公開します。 コードを更新するだけで本番サイトが自動更新される仕組みです。

GitHub Actions（GitHub内蔵の自動化ツール）

GitHubが提供するCI/CDツールで、Webhookイベントをトリガーにして様々な処理を自動実行します。 テスト・ビルド・デプロイなどをすべてGitHub上で設定でき、 外部サービスとの連携も容易です。

9.7 GitHubのWebhook設定のイメージ

実際にGitHubのWebhookを設定する画面はどのような構成になっているか、 概念的なイメージを確認しておきましょう。実際の操作は第12章で体験できます。

GitHub > Settings > Webhooks > Add webhook
https://example.com/webhook/github

Webhookデータを受け取るURL（エンドポイント）を入力します

application/json

データをJSON形式で送信します（推奨）

Just the push event（プッシュイベントのみ）
Send me everything（全イベント）
Let me select individual events（イベントを個別選択）

重要ポイント

GitHubのWebhook設定は「受け取るURL」「送るデータ形式」「どのイベントで送るか」の3点を設定するだけです。 第6章で学んだ「エンドポイント登録」の考え方がそのまま当てはまります。 基本的な仕組みはどのサービスでも同じです。

9.8 開発ツール連携での注意点

GitHubなどの開発ツールとWebhookを連携する際には、いくつかの注意点があります。

秘密鍵（Secret）の設定を忘れずに

GitHubのWebhook設定には「Secret（シークレット）」という秘密のパスワードを設定できます。 これを設定しておくと、受け取ったWebhookが本当にGitHubから送られたものかどうかを確認できます。 悪意のある第三者がWebhookを偽装して送ってくる攻撃（なりすまし）を防ぐために重要です。

大量のpushに注意

開発が活発なリポジトリでは、1日に何十回もpushが発生することがあります。 すべてのpushでWebhookが送られると、受け取り側のサーバーに大きな負荷がかかる場合があります。 イベントフィルタリングを活用して、必要なイベントだけを受け取るよう設計しましょう。

受信先URLは公開されている必要がある

GitHubからWebhookを受け取るためには、受信先のURLがインターネットから アクセスできる状態（公開URL）でなければなりません。 自分のパソコン上でテストする場合は、ngrokなどのトンネリングツールを使って 一時的な公開URLを作成する方法があります。

理解度確認クイズ

  1. Q1. GitHubでコードが更新されたときに発生するWebhookイベントの名前は何ですか？

    回答を表示

    「push」イベントです。プログラマーがコードをGitHubに送信（プッシュ）したときに発生します。 このイベントを受け取って自動テストを実行したり、Slackに通知を送ったりします。

  2. Q2. CI（継続的インテグレーション）とは何ですか？WebhookとどのようにつながっていますかQ2. CI（継続的インテグレーション）とは何ですか？Webhookとの関係も説明してください。

    回答を表示

    CIとは、コードが追加されるたびに自動でテストを実行し、正しく動くかを確認する仕組みです。 GitHubのpushイベントをWebhookで受け取り、CIサーバーに「テストを開始して」と伝えることで実現します。 Webhookがなければ、人が手動でテストを実行する必要があります。

  3. Q3. GitHubのWebhook設定でSecretを設定する目的は何ですか？

    回答を表示

    受け取ったWebhookが本当にGitHubから送られたものかどうかを検証するためです。 Secretがないと、悪意のある第三者がGitHubのふりをして偽のWebhookを送りつけ、 不正な処理を実行させる「なりすまし攻撃」のリスクがあります。 セキュリティ対策として重要な設定です。
