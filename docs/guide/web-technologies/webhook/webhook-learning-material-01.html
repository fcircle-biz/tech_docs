Webhookとは何か？ 〜身近な例で理解する〜

「Webhookとは何か」を、スマホの通知や宅配便の配達通知といった身近な例を使って分かりやすく解説します。「自分から確認しに行く」のと「相手から教えてもらう」の違いを理解し、Webhookの基本的な考え方を掴みましょう。

この章で学ぶこと

  • Webhookとは何かを、日常生活の「通知」に例えて理解できる
  • 「自分から確認しに行く」と「教えてもらう」の違いを説明できる
  • Webhookが活躍する場面を身近なサービスの例で挙げることができる
  • Webhookの別名「逆のAPI」「HTTPコールバック」の意味を理解できる

1.1 まず「通知」について考えてみよう

Webhookを理解するには、まず私たちの日常生活にある「通知」の仕組みを考えるのが一番の近道です。

あなたは友人からのメッセージを確認するとき、どうしていますか？

方法A: 自分で確認しに行く

「そろそろメッセージ来てないかな？」と思って、5分おきにアプリを開いて確認する。 来ていなければまた後で確認…を繰り返す。

これがポーリング（プル型）

自分から積極的に「何か来てる？」と聞きに行く方式

方法B: 来たら通知してもらう

スマートフォンの通知設定をONにして、メッセージが届いたときに「ピン！」と知らせてもらう。 来るまで待っているだけでよい。

これがWebhook（プッシュ型）

相手が「届いたよ！」と教えてくれる方式

重要ポイント

Webhookとは「何かが起きたとき、相手が自動的に教えてくれる仕組み」です。 毎回自分で確認しに行く必要がなく、イベントが発生したときだけ通知を受け取れます。

1.2 宅配便で理解するWebhook

もう一つ、よりイメージしやすい例で考えてみましょう。宅配便の荷物追跡です。

宅配便の追跡方法の比較

自分で確認 1時間ごとに追跡サイトを開いてステータスを確認 手間がかかる、見逃す可能性がある

Webhookは、この「配達員が自動でメールを送ってくれる仕組み」に相当します。 コンピューターの世界では、「何かのイベントが起きたとき、指定したURLにデータを送る」という形で動作します。

1.3 Webhookの正式な定義

ここまでの例を踏まえて、Webhookを技術的に定義してみましょう。

Webhookの定義

Webhookとは、Webアプリケーション間でリアルタイムに情報を通知するための仕組みです。 特定のイベント（出来事）が発生したとき、あらかじめ登録しておいたURL（エンドポイント）に対して 自動的にHTTPリクエストを送信します。

キーワードを整理しよう

イベント

「何かが起きた」という出来事のこと。例：「注文が入った」「支払いが完了した」「コードが更新された」など。

URL（エンドポイント）

通知の送り先アドレス。Webhookを受け取るための「住所」のようなもの。 例：https://example.com/webhook

HTTPリクエスト

インターネット上でデータを送る方法の一つ。Webhookでは主にPOSTメソッドが使われます。 「郵便」のようなものだと思ってください。

1.4 Webhookの別名を知ろう

Webhookにはいくつかの別名・愛称があります。それぞれの意味を理解すると、Webhookの本質がより深く分かります。

逆のAPI（Reverse API）

通常のAPIは「あなたがサービスにデータを取りに行く」もの。 Webhookは「サービスがあなたにデータを送ってくる」もの。 方向が逆なので「逆のAPI」と呼ばれます。

HTTPコールバック

「コールバック」とは「後で連絡する」という意味。 「用事ができたら、このURLに連絡してね」と登録しておくのがWebhookです。 HTTPの仕組みを使って「折り返し連絡」するイメージです。

豆知識

「Webhook」という名前は「Web」＋「Hook（フック・引っかける）」から来ています。 イベントが発生したときに「引っかかって（フックして）」処理が実行されるイメージです。

1.5 身近なWebhookの活用例

実は、私たちが日常的に使っているサービスの多くでWebhookが活用されています。

ECサイト・オンライン決済

StripeやPayPalなどの決済サービスは、支払いが完了したときにWebhookを送信します。 これを受け取ったECサイトは「注文確定メールを送る」「在庫を減らす」などの処理を自動で行います。

SlackやMicrosoft Teams

GitHubで新しいコードが追加されたとき、SlackにWebhookで通知を送ることができます。 チームメンバーが「誰かがコードを更新した」ことを即座に把握できます。

GitHub・開発ツール

GitHubではコードのプッシュ、プルリクエストの作成など様々なイベントでWebhookを送信できます。 これを利用して「テストを自動で実行する」「本番環境へのデプロイ」などが自動化されます。

LINE・SNSサービス

LINEのMessaging APIでは、ユーザーがメッセージを送ったときにWebhookでサーバーに通知が来ます。 チャットボットはこの仕組みを使って「ユーザーの発言に反応する」処理を行います。

1.6 Webhookを使うメリット

Webhookがなければ、どのような問題が起きるでしょうか？メリットを整理してみましょう。

Webhookなし vs Webhookあり

sequenceDiagram participant A as あなたのアプリ participant B as 外部サービス Note over A,B: Webhookなしの場合（ポーリング） A->>B: 注文来てますか？ B-->>A: まだないですよ A->>B: 注文来てますか？ B-->>A: まだないですよ A->>B: 注文来てますか？ B-->>A: 来ました！（注文データ） Note over A,B: Webhookありの場合 B->>A: 注文が入りました！（自動通知） A-->>B: 受け取りました（200 OK）

リアルタイム性

イベント発生と同時に通知が届くので、素早い対応が可能です。ポーリングのような「タイムラグ」がありません。

サーバー負荷軽減

「常に確認しに行く」のをやめられるので、サーバーへの無駄なリクエストが減り、効率的なシステムになります。

シンプルな実装

URLを登録するだけで連携が始まるため、複雑なポーリング処理を自分で書く必要がありません。

理解度確認クイズ

  1. Q1. Webhookを日常生活に例えると、次のどちらに近いですか？ (A) 5分おきに天気予報サイトを開いて確認する (B) 雨が降ったらスマホが通知してくれる

    回答を表示

    答えは (B) です。Webhookは「イベントが起きたときに相手から通知してくれる」プッシュ型の仕組みです。 (A) はポーリング（プル型）と呼ばれ、自分から確認しに行く方式です。

  2. Q2. Webhookの別名として正しいものを2つ選んでください。 (A) 逆のAPI (B) クラウドAPI (C) HTTPコールバック (D) RESTful API

    回答を表示

    答えは (A) 逆のAPI と (C) HTTPコールバック です。 「逆のAPI」はデータを取りに行くのではなく送ってもらう方向が逆だから。 「HTTPコールバック」はHTTPで「後で連絡」する仕組みだからです。

  3. Q3. Webhookを使うメリットとして正しいものはどれですか？ (A) 常に通信し続けるのでリアルタイム性が高い (B) イベント発生時だけ通信するのでサーバーへの負荷が少ない (C) プログラムを書かなくても自動で動く

    回答を表示

    答えは (B) です。Webhookはイベントが起きたときだけ通信するため、 ポーリング（常に確認）と比べてサーバーへの無駄なリクエストが減り、効率的です。 (A) は「常に通信」ではなくイベント時のみです。(C) は受信側の処理を書く必要があります。
