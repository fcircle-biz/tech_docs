身近なWebhook活用例（2）〜決済・ECサービス〜

Stripeなどの決済サービスやECプラットフォームで使われるWebhookを紹介します。「支払い完了時に自動で処理を実行する」など、ビジネスの現場でのWebhook活用を学びます。

この章で学ぶこと

  • 決済サービス（Stripe）がWebhookをどのように使っているかを理解できる
  • 「支払い完了」「サブスクリプション更新」などのイベントとWebhookの関係を説明できる
  • ECサイトでWebhookが果たす役割（在庫管理・出荷通知・返金処理）を理解できる
  • 決済Webhookのペイロード（送られてくるデータ）の基本的な構造を読み解くことができる

8.1 オンライン決済とWebhookの深い関係

インターネットでの買い物や、月額サービスの支払いには、Stripeや PayPalといった「決済サービス」が使われています。 これらのサービスとWebhookは切っても切れない関係にあります。

なぜなら、決済は「非同期」な処理だからです。ユーザーがカード番号を入力してから、銀行での認証・決済会社との通信など複数のステップを経て、 最終的に「支払い成功」か「失敗」かが決まります。この結果をリアルタイムにECサイト側へ通知するのがWebhookの仕事です。

オンライン決済の全体フロー

sequenceDiagram autonumber participant U as ユーザー participant EC as ECサイト participant SP as 決済サービス participant BK as 銀行/カード会社 U->>EC: 「購入する」ボタンをクリック EC->>SP: 決済リクエストを送る SP->>BK: カード認証・決済処理 BK-->>SP: 認証結果を返す SP-->>EC: Webhookで結果を通知 EC-->>U: 注文完了画面を表示 EC-->>U: 注文確認メールを送信

重要ポイント

決済サービスがWebhookを使う理由は「決済の結果を確実にECサイトに伝えるため」です。 もしWebhookがなければ、ECサイトは何度も決済サービスに「支払いが完了しましたか？」と確認しなければなりません。 Webhookによって、完了した瞬間に自動で通知が届きます。

8.2 Stripeのwebhookイベント一覧

世界中のECサイトで使われている決済サービス「Stripe」は、様々なタイミングでWebhookを送信します。 どんな「イベント」が用意されているか見てみましょう。

Stripeの主なWebhookイベント

イベント名 発生タイミング ECサイトの対応例
payment_intent.succeeded 支払いが成功したとき 注文確定メール送信・在庫を減らす
payment_intent.payment_failed 支払いが失敗したとき エラーメール送信・カート内容を保持
customer.subscription.created サブスク登録されたとき プレミアム機能を有効化する
customer.subscription.deleted サブスクが解約されたとき プレミアム機能を無効化する
charge.refunded 返金が行われたとき 返金完了メール送信・在庫を戻す
invoice.payment_succeeded 請求書の支払いが成功したとき 領収書の自動発行

イベント名の読み方

Stripeのイベント名は「オブジェクト名.アクション名」という形式になっています。 例えば payment_intent.succeeded は 「payment_intent（決済処理）がsucceeded（成功した）」という意味です。 この命名規則を理解すると、見慣れないイベント名でも何が起きているか推測できます。

8.3 決済Webhookのペイロードを読んでみよう

Stripeから送られてくるWebhookのデータ（ペイロード）を見てみましょう。 前章（第5章）でJSONの読み方を学んだので、実際のデータを読み解くことができます。

Stripeから送られるWebhookペイロードの例 
{
  "id": "evt_1234567890",
  "type": "payment_intent.succeeded",
  "created": 1700000000,
  "data": {
    "object": {
      "id": "pi_9876543210",
      "amount": 9800,
      "currency": "jpy",
      "status": "succeeded",
      "metadata": {
        "order_id": "ORD-2024-001",
        "customer_email": "user@example.com"
      }
    }
  }
}

各フィールドの意味を読み解こう

id

このWebhookイベント固有のID。同じイベントが重複して届いた場合の識別に使います（べき等性）。

type

何のイベントかを示す名前。payment_intent.succeeded は「決済成功」を意味します。 受信したプログラムはこの値を見て処理を分岐させます。

amount

決済金額（単位は円）。9800 は9,800円を意味します。 Stripeは日本円の場合、最小単位の「円」で金額を送ります。

metadata

ECサイト側が決済リクエスト時に自由に付加できる追加情報。 注文IDやメールアドレスを入れておくことで、「どの注文の支払いが完了したか」をWebhook受信時に特定できます。

8.4 ECサイトにおけるWebhookの活用シーン

決済以外にも、ECサイト全体の運営においてWebhookは様々な場面で活躍しています。 代表的な活用シーンを見ていきましょう。

在庫管理の自動化

注文が入るたびに手動で在庫を減らすのは大変です。 注文完了のWebhookを受け取ったら、自動的に在庫データベースの数量を更新できます。 在庫がなくなったら「売り切れ」表示に切り替えることも自動化できます。

トリガー：order.created（注文作成）

→ 在庫数を-1する処理を自動実行

出荷・配送通知

倉庫管理システムで荷物が出荷されたとき、WebhookでECサイトに通知が来ます。 ECサイトはそれを受けて、ユーザーに「発送しました」メールと追跡番号を自動送信できます。

トリガー：shipment.shipped（出荷完了）

→ 追跡番号付き発送メールを自動送信

返金・キャンセル処理

Stripeで返金処理が行われると、charge.refunded というWebhookが届きます。 これを受けてECサイトは在庫を元に戻したり、返金完了メールを送ったりすることができます。

トリガー：charge.refunded（返金完了）

→ 注文ステータスを「返金済み」に変更・在庫を戻す

サブスクリプション管理

月額・年額サービスでは、毎月の自動更新のたびにWebhookが届きます。 支払いが成功したらサービスを継続し、失敗したらアクセスを制限するという処理が自動化できます。

トリガー：invoice.payment_failed（請求失敗）

→ 支払い失敗メール送信・サービス一時停止の準備

8.5 Shopify・ECプラットフォームのWebhook

決済サービス以外にも、ECプラットフォーム自体もWebhookを提供しています。 代表的なShopifyを例に見てみましょう。

Shopifyの注文関連イベント

  • orders/create：注文が作成された
  • orders/fulfilled：注文が出荷された
  • orders/cancelled：注文がキャンセルされた
  • orders/paid：支払いが完了した

Shopifyの顧客・商品イベント

  • customers/create：新規顧客が登録した
  • products/create：商品が追加された
  • inventory_levels/update：在庫数が変わった
  • checkouts/create：カートが作成された

ShopifyのようなECプラットフォームでは、外部のシステム（倉庫管理システム、CRM、マーケティングツールなど）と 連携するためにWebhookが活用されます。新規顧客が登録されたら外部のメールマーケティングツールに追加する、 在庫が変わったら倉庫システムに反映するといった連携が自動化できます。

8.6 決済Webhookを安全に受け取るための工夫

決済に関わるWebhookは特に重要度が高いため、安全に受け取るための仕組みが用意されています。 第10章でセキュリティを詳しく学びますが、ここでは決済Webhookに特有のポイントを紹介します。

Stripeの署名検証の仕組み

（送信元）"] -->|"1. Webhookデータ
＋署名を付けて送信"| B["あなたの
Webサーバー"] B -->|"2. 署名を検証
（本物か確認）"| C{"検証結果"} C -->|"OK: 本物"| D["処理を実行
（注文確定など）"] C -->|"NG: 偽物"| E["リクエストを

なぜ署名検証が重要なのか

もし悪意のある第三者があなたのWebhookエンドポイントに「支払い成功」という偽のデータを送った場合、 その内容を信じてしまうと商品を無料で発送してしまう危険性があります。 Stripeの署名検証を使うことで、「本当にStripeから送られてきたデータか」を確認できます。

同じWebhookが複数回届く問題（重複処理の防止）

ネットワークの問題などで、同じWebhookが複数回送られてくることがあります。 「支払い成功」の通知が2回届いた場合、注文確定メールが2通送られたり、在庫が2回減ってしまったりする問題が起きます。

問題のある処理

Webhookを受け取るたびに処理を実行する。同じイベントが2回届くと処理も2回実行されてしまう。

正しい処理（べき等性）

イベントIDを記録しておき、「このIDはすでに処理済み」なら処理をスキップする。何度届いても結果が同じになる。

8.7 決済Webhookを使った自動化の全体像

ここまで学んだ内容を踏まえて、決済Webhookを活用した注文処理の自動化フローを整理しましょう。

ECサイトの注文処理自動化フロー

カード情報を入力"] B --> C["Stripeで決済処理"] C --> D{"決済結果"} D -->|"成功"| E["payment_intent.succeeded
WebhookをECサイトに送信"] D -->|"失敗"| F["payment_intent.payment_failed

再試行を促すメール送信"]

Webhookがなければどうなるか？

もしWebhookがなければ、ECサイトは「支払いが完了したか」を常にStripeに確認し続けなければなりません。 1000人が同時に購入すると、1000×n回のAPIリクエストが発生します。 WebhookはStripeからの「完了通知」を受け取るだけで済むため、無駄な通信がなくシステムが効率的に動きます。

理解度確認クイズ

  1. Q1. Stripeのイベント payment_intent.succeeded はどのタイミングで発生しますか？ (A) ユーザーが購入ボタンを押したとき (B) 支払いが正常に完了したとき (C) カートに商品が追加されたとき

    回答を表示

    答えは (B) です。payment_intent.succeeded の「succeeded」は「成功した」という意味なので、 支払いが正常に完了したときに発生します。(A) の購入ボタンを押したタイミングや (C) のカート操作では発生しません。

  2. Q2. 決済Webhookで「署名検証」を行う主な理由はどれですか？ (A) 処理の速度を上げるため (B) データが本当に決済サービスから送られてきたものか確認するため (C) Webhookのデータを圧縮するため

    回答を表示

    答えは (B) です。署名検証は「このデータが本当にStripeから送られてきたものか」を確認するためのセキュリティ対策です。 悪意のある第三者が偽の「支払い成功」データを送ってきても、署名が一致しないため無視できます。

  3. Q3. Webhookの「べき等性（idempotency）」とはどういう意味ですか？ (A) データの処理速度が常に一定であること (B) 同じ処理を何度実行しても結果が変わらないこと (C) Webhookが常に成功すること

    回答を表示

    答えは (B) です。べき等性とは「何度実行しても同じ結果になる」性質のことです。 Webhookが重複して届いた場合でも、処理済みのイベントIDを記録しておくことで二重処理を防ぎ、 同じメールが2通送られたり在庫が2回減ったりする問題を避けられます。
