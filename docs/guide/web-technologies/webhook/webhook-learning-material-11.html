Webhookのエラーとリトライ 〜失敗した時の対処〜

Webhookの配信が失敗する原因と、その対処方法について学びます。リトライ（再送）の仕組み、べき等性（同じデータが複数回届いても問題ない設計）の考え方を理解します。

この章で学ぶこと

  • Webhookの配信が失敗する主な原因を説明できる
  • リトライ（再送）の仕組みと「指数バックオフ」の意味を理解できる
  • べき等性とは何か、なぜ重要なのかを自分の言葉で説明できる
  • デッドレターキューなどの失敗時の対処パターンを知っている

11.1 Webhookの配信が失敗するとはどういうこと？

Webhookは「イベントが起きたら指定URLにHTTPリクエストを送る」仕組みですが、このリクエストが相手に届かないことがあります。 これを 「Webhook配信の失敗」 と呼びます。

宅配便の例で考えると、「配達員が荷物を運んだけれど、受取人が不在だった」状況に似ています。 荷物（データ）は送り出したが、受け取ってもらえなかったというイメージです。

Webhookが「失敗した」と判断される条件

接続できない

受信サーバーがダウンしていたり、ネットワーク障害でリクエストが届かない場合

タイムアウト

受信サーバーが一定時間内にレスポンスを返さない場合（一般的に5〜30秒以内）

エラーステータスコード

受信サーバーが 4xx や 5xx のHTTPステータスコードを返した場合

重要ポイント

Webhookの送信側は、受信側から 200〜299番台のHTTPステータスコード が返ってきた場合のみ 「正常に受け取ってもらえた」と判断します。それ以外はすべて「失敗」として扱います。

11.2 失敗の主な原因を知ろう

Webhook配信が失敗する原因はいくつかあります。原因を知ることで、対処法が見えてきます。

受信側（自分のサーバー）の問題

  • サーバーがメンテナンス中・停止中
  • アプリケーションのバグでエラーが発生
  • 処理が重くタイムアウトしてしまう
  • URLが変更されエンドポイントが存在しない

ネットワーク・その他の問題

  • インターネット回線の一時的な障害
  • ファイアウォールでブロックされている
  • SSL証明書の期限切れ（HTTPS通信エラー）
  • リクエストの急増によるサーバー過負荷

Webhook配信失敗の流れ

sequenceDiagram participant S as 送信元サービス participant R as 受信サーバー S->>R: Webhookリクエスト送信 note over R: サーバーダウン or 処理エラー R-->>S: 接続エラー or 5xx エラー note over S: 配信失敗を検知 S->>S: 失敗ログ記録

豆知識

受信側が「処理中に時間がかかる」場合は、まず素早く 200 OK を返してから処理を非同期で行う 「即時レスポンス」 という設計パターンがよく使われます。 タイムアウトを防ぐための重要なテクニックです。

11.3 リトライ（再送）の仕組み

Webhook配信が失敗した場合、ほとんどのサービスは自動的に再送信（リトライ）を行います。 「宅配便が不在だったので、翌日また配達に来る」イメージです。

リトライとは

Webhookの配信が失敗した場合に、一定時間をおいて再度同じリクエストを送信する仕組みです。 一時的なサーバー障害やネットワーク問題の場合に、時間をおけば成功することが多いため、 多くのWebhookサービスがこの機能を提供しています。

指数バックオフとは？

リトライを行う際、毎回すぐに再送してしまうと、障害中のサーバーにさらに負荷をかけてしまいます。 そこで多くのシステムでは 「指数バックオフ（Exponential Backoff）」 という方式を使います。

「指数バックオフ」とは、リトライのたびに待ち時間を指数関数的に増やしていく方式です。 失敗が続くほど間隔を長くすることで、障害中のサーバーへの過負荷を防ぎます。

指数バックオフの例（Stripe等のサービス参考）

リトライ回数 待ち時間の目安 イメージ
1回目 5秒後 すぐもう一度
2回目 30秒後 少し待ってから
3回目 5分後 しばらく待ってから
4回目 30分後 かなり待ってから
5回目以降 数時間後〜最終的に諦め 長期間待ってから / 最終的に失敗確定

※ 実際の間隔はサービスによって異なります。上記はイメージを掴むための参考値です。

リトライの流れ

注意事項

リトライによって同じWebhookが複数回届く可能性があります。 「支払い完了」通知が2回届いたのに2回分の処理をしてしまうと大変です。 次のセクションで学ぶ「べき等性」が重要になります。

11.4 べき等性とは何か？

リトライによって同じWebhookが複数回届く可能性があります。 このとき「何回受け取っても結果が同じになる」ように設計することを 「べき等性（Idempotency）を保つ」と言います。

べき等性（Idempotency）の定義

「同じ操作を何回繰り返しても、1回実行したときと同じ結果になること」をべき等性と言います。 Webhookにおいては、同じ通知が複数回届いても、処理が重複して実行されないことを意味します。

べき等性なし（危険）

同じ「支払い完了」通知が2回届くと、 「注文確定メール」が2通送られ、 「在庫を1個減らす処理」が2回実行されてしまう。

問題

2通のメール送信、在庫が2個分減少する

べき等性あり（安全）

同じ通知IDを持つWebhookが2回届いたとき、 「もう処理済みのIDだ」と判断し、 2回目の処理をスキップする。

結果

1通のメール送信、在庫は1個分のみ減少

べき等性を実現するための基本的なアプローチ

方法1

一意のIDで重複チェック

Webhookには通常、一意の識別子（ID）が含まれています。 受信したIDを記録しておき、同じIDが来たら処理をスキップします。 最もシンプルで確実な方法です。

方法2

ステータスを確認してから処理

処理を実行する前に「この注文はすでに処理済みか？」を確認します。 例：注文のステータスが「完了」になっていれば処理を行わない。

方法3

Idempotency Keyの活用

サービスによっては「Idempotency-Key」というHTTPヘッダーを使って、 同じリクエストの重複処理を防ぐ仕組みを提供しています。 Stripeなどの決済APIで使われています。

11.5 Webhook IDを使ったべき等性の実装イメージ

実際にどのようにべき等性を実現するか、Webhook IDを使った基本的なイメージを見てみましょう。 プログラムを書かない場合でも「何をしているか」のイメージは大切です。

WebhookのJSONデータ例（Webhook IDが含まれる） 

{
  "id": "evt_1234567890abcdef",
  "type": "payment.completed",
  "created": 1700000000,
  "data": {
    "order_id": "order_001",
    "amount": 5000,
    "currency": "jpy"
  }
}

id フィールドがWebhookの一意識別子です。 同じ id が届いたら「リトライ」と判断できます。

Webhook IDによる重複チェックの流れ

flowchart TD A([Webhookを受信]) --> B["Webhook IDを取得
（例: evt_1234567890）"] B --> C{このIDは処理済み？} C -- はい --> D["処理をスキップ
（200 OK を返す）"] C -- いいえ --> E[本来の処理を実行] E --> F[IDを処理済みとして記録] F --> G["200 OK を返す"] style D fill:#f0fdfa,stroke:#14b8a6 style G fill:#ecfdf5,stroke:#10b981

重要ポイント

スキップした場合でも 200 OK を返すことが重要です。 エラーコードを返してしまうと、送信元は「失敗した」と判断してさらにリトライを繰り返してしまいます。

11.6 最終的な失敗への対処 〜デッドレターキューとアラート〜

リトライをすべて試みても配信に失敗した場合、そのWebhookはどこへ行くのでしょうか？ 本節では「最終的な失敗」にどう備えるかを学びます。

デッドレターキュー（DLQ）

全リトライが失敗したWebhookを蓄積しておく「隔離場所」です。 「Dead Letter Queue（死んだ手紙の箱）」という名前通り、配達できなかったデータを一時保管します。 後で手動で確認・再処理することができます。

アラート・通知

配信失敗が続く場合に、管理者にメールやSlackで通知を送ります。 問題を早期発見し、手動での対処や調査のきっかけになります。 多くのWebhookサービスがこの機能を提供しています。

配信履歴の確認

Stripeや GitHubなど多くのサービスでは、Webhookの配信履歴を管理画面で確認できます。 「いつ、どのイベントが、何回試みて失敗したか」を確認し、 手動で「再配信」ボタンを押すことも可能です。

リトライと最終失敗対処の全体像

flowchart TD A([イベント発生]) --> B[Webhookを送信] B --> C{配信成功？} C -- 成功 --> D([完了]) C -- 失敗 --> E{リトライ残り？} E -- あり --> F["待機（指数バックオフ）"] --> B E -- なし --> G[デッドレターキューへ保存] G --> H[管理者にアラート送信] H --> I{原因調査・解決} I -- 解決済み --> J[手動で再配信] --> B style D fill:#ecfdf5,stroke:#10b981 style G fill:#fef2f2,stroke:#ef4444 style J fill:#f0fdfa,stroke:#14b8a6

よくある問題と対処法

問題: 同じWebhookが何度も届いて二重処理が発生する

対処: べき等性の実装が必要です。Webhook IDを使った重複チェック処理を追加しましょう。 データベースに「処理済みIDテーブル」を作るのが基本的なアプローチです。

問題: タイムアウトエラーが頻発する

対処: Webhookを受け取ったら即座に 200 OK を返し、重い処理は「非同期」で行うよう設計を変更します。 キューを使った非同期処理が有効です。

問題: 配信失敗が多すぎてデータが失われそう

対処: サービスのダッシュボードで配信履歴を確認し、失敗の原因を特定します。 サーバーが安定したら「手動再配信」機能を使ってデータを再送してもらいましょう。

第11章のまとめ

  • 1

    Webhook配信の失敗は「接続エラー」「タイムアウト」「エラーステータスコード」が主な原因です

  • 2

    多くのサービスは失敗時に「リトライ（再送）」を行います。「指数バックオフ」で間隔を広げながら再試行します

  • 3

    リトライにより同じWebhookが複数回届く可能性があるため、「べき等性（同じ処理が重複実行されない設計）」が重要です

  • 4

    全リトライ失敗後は「デッドレターキュー」「アラート通知」「配信履歴の手動確認・再配信」で対処します

理解度確認クイズ

  1. Q1. Webhook送信側が「配信成功」と判断するのは、受信側がどのレスポンスを返した場合ですか？ (A) 100番台のステータスコード (B) 200〜299番台のステータスコード (C) 500番台のステータスコード

    回答を表示

    答えは (B) です。200〜299番台のHTTPステータスコードは「正常に処理された」ことを意味します。 それ以外のコード（4xx、5xxなど）やタイムアウトは「失敗」と判断されます。

  2. Q2. 「指数バックオフ」とはどのような方式ですか？ (A) リトライのたびに毎回同じ間隔（例: 1秒）で再送する (B) リトライのたびに間隔を指数的に増やす（例: 5秒→30秒→5分） (C) リトライは1回のみ行い、失敗したら即座に諦める

    回答を表示

    答えは (B) です。指数バックオフは失敗するたびに待ち時間を指数的に増やす方式です。 障害中のサーバーへの過負荷を防ぎつつ、一時的な障害からの回復を待つことができます。

  3. Q3. べき等性を保つ実装として最も一般的な方法はどれですか？ (A) Webhookを受け取るたびに必ず処理を実行する (B) Webhookの受信をランダムに半分だけ処理する (C) WebhookのIDを記録して、同じIDが来たらスキップする

    回答を表示

    答えは (C) です。WebhookのIDを「処理済み」として記録しておき、 同じIDが再度届いた場合はスキップすることで、リトライによる二重処理を防ぎます。 スキップの場合も 200 OK を返すことを忘れずに。
