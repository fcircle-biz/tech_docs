Webhookの仕組み 〜データが届くまで〜

Webhookでデータが送信元から受信先に届くまでの流れを、図解を用いて段階的に学びます。「イベント発生」「HTTPリクエスト送信」「レスポンス返却」という一連の流れを理解しましょう。

この章で学ぶこと

  • Webhookの通信が始まる「イベント発生」の仕組みを説明できる
  • HTTPリクエストがどのように送信されるかの流れを理解できる
  • 受信側がレスポンスを返す理由と重要性を説明できる
  • 送信元・受信先・エンドポイントそれぞれの役割を整理できる

3.1 Webhookの登場人物を整理しよう

Webhookの仕組みを理解するために、まず「誰が・何をしているか」を整理しましょう。 Webhookの通信には、必ず3つの要素が登場します。

送信元サービス

イベントが発生したときにWebhookを送り出す側。

具体例

  • ・GitHubのリポジトリ
  • ・Stripeの決済システム
  • ・SlackのBot

エンドポイントURL

Webhookが届く「住所」となるURL。受信先があらかじめ公開しておく。

具体例

https://myapp.com/webhook

受信先アプリケーション

Webhookを受け取り、データを処理する側。

具体例

  • ・自社のWebサーバー
  • ・自動化ツール（Zapier等）
  • ・チャットBot

郵便に例えると

「送信元サービス」は手紙を書いて投函する人、「エンドポイントURL」は受け取る住所、 「受信先アプリケーション」は手紙を読んで行動する人に例えられます。 宅配便では荷物が届いたら「受け取った！」とサインするように、 Webhookでも受け取ったことを返信します。

3.2 データが届くまでの全体の流れ

Webhookのデータがどのように「送信元」から「受信先」まで届くのか、 全体の流れを4つのステップで見ていきましょう。

Webhookデータ配信の全体フロー

sequenceDiagram autonumber participant U as ユーザー participant S as 送信元サービス participant E as エンドポイントURL participant A as 受信先アプリ U->>S: イベントを起こす（例: 購入ボタンを押す） S->>S: イベントを検知する S->>E: HTTPリクエストを送信（POSTメソッド＋データ） E->>A: リクエストを受け付ける A->>A: データを処理する（例: メール送信） A-->>S: レスポンスを返す（200 OK）

この流れを一つひとつのステップに分解して詳しく見ていきましょう。

1 STEP 1: イベントが発生する

Webhookの通信は、必ず「イベント」の発生から始まります。 イベントとは、システム内で起きた「出来事」のことです。

イベントの例

商品が購入された
支払いが完了した
コードがプッシュされた
新規ユーザーが登録した
コメントが投稿された
エラーが発生した

2 STEP 2: HTTPリクエストが送信される

イベントを検知した送信元サービスは、あらかじめ登録されたエンドポイントURLに向けて HTTPリクエストを送信します。このリクエストには、イベントの詳細情報（データ）が含まれています。

HTTPリクエストに含まれるもの

  • メソッド: ほとんどの場合「POST」が使われる
  • URL: 受信先のエンドポイントアドレス
  • ヘッダー: コンテンツタイプや認証情報など
  • ボディ（本文）: イベントの詳細データ（JSON形式が一般的）
Webhookリクエストのイメージ（GitHub プッシュイベント） 
POST /webhook HTTP/1.1
Host: myapp.example.com
Content-Type: application/json
X-GitHub-Event: push

{
  "ref": "refs/heads/main",
  "repository": {
    "name": "my-project"
  },
  "pusher": {
    "name": "taro"
  }
}

3 STEP 3: 受信先がデータを処理する

エンドポイントURLでリクエストを受け取った受信先アプリケーションは、 ボディ（本文）のデータを読み取り、必要な処理を実行します。

受信後に行われる処理の例

  • 購入完了メールをユーザーに自動送信する
  • データベースに注文情報を保存する
  • Slackチャンネルに通知メッセージを投稿する
  • 自動テストを実行してビルド結果を確認する

処理は素早く行うことが大切

Webhookの受信側は、リクエストを受け取ったらできるだけ素早く（通常5〜30秒以内に） レスポンスを返す必要があります。時間がかかりすぎると、送信元が「失敗した」と判断してしまうことがあります。

4 STEP 4: レスポンスを返す

処理が完了したら、受信先アプリケーションは送信元サービスに「受け取りました」という レスポンスを返します。これが宅配便の「受領サイン」にあたります。

代表的なレスポンスコード

200 OK 正常に受け取り、処理が完了した
202 受け取ったが、処理は後で行う（非同期）
4xx クライアント側のエラー（認証失敗など）
5xx サーバー側のエラー（再送が発生することがある）

3.3 具体的なシナリオで流れを追う

「ECサイトで商品が購入された」という具体的なシナリオを使って、 Webhookのデータ配信の流れをより細かく見てみましょう。

重要ポイント

1つのWebhookでも「どのイベントか」によって処理を分岐させることができます。 送信元サービスは異なるイベント（決済成功・失敗・返金など）を同じエンドポイントに送ることが多く、 受信側はイベントの種別を確認してから適切な処理を行います。

3.4 同期処理と非同期処理の考え方

Webhookを受け取った後の処理には「同期処理」と「非同期処理」の2つのパターンがあります。 初心者の方は「どちらが向いているか」を理解することが大切です。

同期処理

Webhookを受け取ってから処理が完了するまで待ち、 完了後に「200 OK」を返す方式。

向いているケース

  • ・処理時間が短い場合（数秒以内）
  • ・シンプルなデータ保存処理
  • ・即座に結果を確認したい場合

非同期処理

Webhookを受け取ったらすぐに「202 Accepted（受け付けた）」を返し、 実際の処理はバックグラウンドで行う方式。

向いているケース

  • ・処理に時間がかかる場合
  • ・メール送信など外部サービスを呼ぶ場合
  • ・タイムアウトを避けたい場合

タイムアウトに注意

多くの送信元サービスは、Webhookのレスポンスが一定時間（例: 10〜30秒）内に返ってこないと 「配信失敗」と判断します。処理が長くなりそうな場合は、先に「受け取った」と返信してから バックグラウンドで処理を続けましょう。

3.5 Webhookの「登録」と「準備」の仕組み

Webhookが機能するためには、通信が始まる前に「事前の準備」が必要です。 この準備のことを「Webhookの登録」と呼びます。

Webhook利用前の準備手順

  1. 1

    受信側: エンドポイントURLを用意する

    Webhookを受け取るためのURLを作成します。これはインターネットからアクセスできるサーバー上の URLである必要があります（例: https://myapp.com/webhook/github）。

  2. 2

    送信側: エンドポイントURLを登録する

    GitHubやStripeなど送信元サービスの設定画面で、手順1のURLを登録します。 同時に「どのイベントで通知するか」も選択します。

  3. 3

    疎通確認（テスト送信）を行う

    多くのサービスでは「テスト送信」機能が用意されています。 登録直後に疎通確認を行い、正常に受け取れるか確認しましょう。

  4. 実際のイベントを待つ

    準備が完了すると、指定したイベントが発生するたびに自動的にWebhookが届くようになります。

補足: ローカル開発環境でのWebhookテスト

自分のパソコン（ローカル環境）で開発中の場合、インターネットからアクセスできないため Webhookのテストが難しくなります。そのような場合は「ngrok（エングロック）」などのツールを使って、 ローカルのサーバーを一時的にインターネットに公開することができます。 第12章のWebhook体験演習でも紹介します。

用語 意味

第3章のまとめ

  • 1

    Webhookには「送信元サービス」「エンドポイントURL」「受信先アプリケーション」の3つの登場人物がいる

  • 2

    通信の流れは「イベント発生 → HTTPリクエスト送信 → データ処理 → レスポンス返却」の4ステップ

  • 3

    受信側は素早くレスポンスを返すことが重要。長い処理は「非同期」で行うのが基本

  • 4

    Webhookを使うには事前の「エンドポイントURL登録」が必要。1回設定すれば以降は自動で動く

理解度確認クイズ

  1. Q1. Webhookの通信フローを順番に並べてください。 (ア) レスポンスを返す (イ) HTTPリクエストを送信する (ウ) イベントが発生する (エ) データを処理する

    回答を表示

    正しい順番は「(ウ) イベントが発生する → (イ) HTTPリクエストを送信する → (エ) データを処理する → (ア) レスポンスを返す」です。 イベントで通信が始まり、リクエストでデータが届き、処理後にレスポンスで完了を知らせます。

  2. Q2. Webhookの受信側が最初に返すべきHTTPステータスコードとして、最も適切なものはどれですか？ (A) 404 (B) 200 OK (C) 500 (D) 301

    回答を表示

    答えは (B) 200 OK です。Webhookを正常に受け取って処理が完了したことを示すには「200 OK」を返します。 404は「見つからない」、500は「サーバーエラー」、301は「リダイレクト」を意味します。

  3. Q3. Webhookの処理に30秒以上かかりそうな場合、どのように対応するのが適切ですか？

    回答を表示

    「非同期処理」を使うのが適切です。Webhookを受け取ったらすぐに「202 Accepted（受け付けた）」を返し、 実際の処理はバックグラウンド（別のプロセスやキューなど）で行います。 こうすることで、送信元がタイムアウトと判断して再送するのを防げます。
