Webhookの登録と設定 〜エンドポイントとは〜

Webhookを利用するために必要な設定手順を学びます。「エンドポイントURL」「イベントの種類の選択」「登録手順」など、サービス側でWebhookを設定する一般的な流れを理解します。

この章で学ぶこと

  • エンドポイントURLとは何かを、「住所」の例えで理解できる
  • Webhookの登録手順（URL入力 → イベント選択 → 保存）の流れを説明できる
  • イベントの種類を選択することの意味と重要性を理解できる
  • Webhookの受信先（エンドポイント）が満たすべき条件を説明できる

6.1 エンドポイントとは？

Webhookを設定するとき、必ず求められるのが「エンドポイントURL」の入力です。 でも、「エンドポイント」という言葉を聞いただけでは、何のことか分かりづらいですよね。

エンドポイントを日常の言葉に置き換えると、「通知の届け先住所」です。 郵便物を送るとき、宛先の住所が必要なのと同じように、 Webhookを受け取るには「どこに送ればよいか」をURL（Web上の住所）で指定します。

郵便の「住所」

郵便物を届けるには、配達先の住所（例: 東京都渋谷区〇〇町1-2-3）が必要。 この住所宛に郵便物が届きます。

Webhookの「エンドポイントURL」

Webhookデータを届けるには、受け取り先のURL（例: https://example.com/webhook）が必要。 このURL宛にWebhookデータが届きます。

エンドポイントURLの例

// よくあるエンドポイントURLの形式
https://myapp.example.com/webhook
https://api.example.com/webhooks/github
https://service.example.com/notifications/stripe
https:// = 安全な通信を示すプロトコル
ドメイン = サーバーの名前（住所の町名）
パス = サーバー内の場所（住所の番地）

重要ポイント

エンドポイントとは「Webhookデータを受け取るためのURL（Web上の住所）」のことです。 Webhookを設定するときは、このURLをサービス側に登録することで、 イベントが発生したときに自動でデータが届くようになります。

6.2 エンドポイントが満たすべき条件

Webhookのエンドポイントとして登録するURLには、いくつかの条件があります。 普通のWebページのURLと少し違う点がありますので、確認しておきましょう。

1

公開されている（インターネットからアクセスできる）

送信元サービスがデータを届けるために、URLがインターネット上からアクセスできる状態でなければなりません。 自分のパソコンの中だけで動いているURL（localhost）は通常使えません。

2

HTTPSで始まる（安全な通信）

多くのサービスはHTTPS（暗号化された安全な通信）のURLしか受け付けません。 HTTPSを使うことで、データが途中で盗み見られるリスクを減らせます。

3

POSTリクエストを受け取れる

WebhookはHTTPのPOSTメソッドでデータを送ってきます。 エンドポイントはこのPOSTリクエストを処理できるプログラムである必要があります。 （普通のWebページはGETメソッドで表示されますが、それとは別です）

4

レスポンスを素早く返す（200 OK）

Webhookを受け取ったら、数秒以内に「受け取りました」という意味の「200 OK」を返す必要があります。 応答が遅いと、送信元サービスが「失敗した」と判断して再送する場合があります。

注意

開発中は自分のパソコン（localhost）でWebhookをテストしたいことがあります。 その場合は「ngrok」「webhook.site」などのツールを使うと、 インターネットからアクセスできる一時的なURLを発行できます。第12章で詳しく学びます。

6.3 イベントの種類を選択する

Webhookを登録するとき、URLの入力に加えて「どのイベントのときに送るか」を指定します。 これを「イベントの種類の選択」と呼びます。

なぜイベントを選択する必要があるのか？

サービスによっては、発生するイベントが非常に多くあります。 たとえばGitHubでは「コードのプッシュ」「プルリクエストの作成」「Issue（課題）の登録」など、 数十種類のイベントがあります。

「すべてのイベントを送る」設定にすると、必要のないイベントのデータまで大量に届いてしまいます。 自分が必要なイベントだけを選ぶことで、効率的に処理できます。

すべてのイベントを受け取る（非効率）

  • ・コードのプッシュ → 受け取る
  • ・プルリクエスト → 受け取る
  • ・Issue登録 → 受け取る（不要）
  • ・コメント追加 → 受け取る（不要）
  • ・スター追加 → 受け取る（不要）

必要なイベントだけ選択（効率的）

  • ・コードのプッシュ → 受け取る ✓
  • ・プルリクエスト → 受け取る ✓
  • ・Issue登録 → 受け取らない
  • ・コメント追加 → 受け取らない
  • ・スター追加 → 受け取らない

各サービスのイベント例

サービス 代表的なイベント 利用場面
GitHub push, pull_request, issues CI/CD自動化
Stripe payment_intent.succeeded, invoice.paid 決済完了処理
Slack message.posted, member.joined チャット連携
LINE message, follow, unfollow チャットボット

6.4 Webhook登録の一般的な手順

Webhookの登録方法はサービスによって多少異なりますが、基本的な流れはどのサービスでもほぼ同じです。 ここでは一般的な手順を学びましょう。

Webhook登録の一般的な流れ

flowchart TD A([開始: Webhookを設定したい]) --> B["サービスの設定画面を開く
（管理画面 → 設定 → Webhook）"] B --> C["エンドポイントURLを入力
（https://example.com/webhook）"] C --> D["受け取るイベントの種類を選択
（例: 支払い完了、コードのプッシュ）"] D --> E["シークレットキーの設定
（セキュリティのため）"] E --> F["設定を保存する"] F --> G{テスト送信を実行} G -->|成功| H(["設定完了！"]) G -->|失敗| I["エラーを確認して修正"] I --> C
1

設定画面を探す

サービスの管理画面やダッシュボードを開き、「設定（Settings）」「開発者（Developer）」「API」 などのメニューを探します。そこに「Webhook」または「Webhooks」という項目があります。

GitHubなら「リポジトリ → Settings → Webhooks」、 Stripeなら「ダッシュボード → 開発者 → Webhook」から設定できます。

2

エンドポイントURLを入力する

「Payload URL（ペイロードURL）」や「エンドポイントURL」という入力欄に、 Webhookデータを受け取るURLを入力します。
入力例 
https://myapp.example.com/webhook/github
3

イベントの種類を選択する

「どのイベントが発生したときにWebhookを送るか」を選択します。 「すべてのイベント」を選ぶか、「特定のイベントのみ」を選ぶか指定できます。 必要なイベントだけを選ぶことで、無駄な通信を減らせます。
4

シークレット（秘密鍵）を設定する

多くのサービスでは「シークレット」と呼ばれる秘密の文字列を設定できます。 これを設定しておくと、Webhookが本当に正しいサービスから送られてきたものかを確認できます。 セキュリティの観点から設定することを推奨します（詳しくは第10章で学びます）。
5

保存してテスト送信を確認する

設定を保存したら、多くのサービスが「テスト送信（Test delivery）」機能を提供しています。 これを使うと、実際のイベントを待たずにWebhookの動作確認ができます。 エンドポイントが正しく200 OKを返すことを確認しましょう。

6.5 登録時によく設定する項目まとめ

サービスによって画面や名称は異なりますが、Webhook登録でよく見かける設定項目を整理しておきましょう。

Webhook設定項目一覧

設定項目 必須/任意 説明
Payload URL 必須 Webhookデータを受け取るURLを指定します
Content Type 推奨 データの形式（通常はapplication/json）を指定します
Secret 推奨 署名検証に使う秘密鍵。セキュリティのために設定します
イベント選択 必須 どのイベント発生時に送信するかを選びます
有効/無効 任意 Webhookを一時的にOFF（無効）にできます

ヒント

「Content Type」は「application/json」を選んでおくと、 受け取ったデータがJSONフォーマットで届き、プログラムから扱いやすくなります。 JSON形式については第5章で学んでいます。

6.6 登録後に起きること

Webhookを登録すると、どのような流れでデータが届くのかを確認しましょう。

登録後にイベントが発生したときの流れ

sequenceDiagram participant S as 送信元サービス participant E as エンドポイント
（あなたのURL） Note over S,E: Webhookが登録済みの状態 S->>S: イベント発生
（例: 注文が入った） S->>E: POSTリクエスト送信
（JSONデータ付き） E-->>S: 200 OK（受け取りました） E->>E: データを処理
（メール送信など）

登録後の重要ポイント

  • 登録が完了すると、指定したイベントが発生するたびに自動でWebhookが届きます
  • エンドポイントが200以外のHTTPステータスコードを返すと、送信元は「失敗」とみなしてリトライします
  • 設定画面から「配信ログ（Delivery log）」を確認でき、成功・失敗の履歴が見られます
  • 不要になったWebhookは設定画面から削除または無効化できます

理解度確認クイズ

  1. Q1. 「エンドポイント」を日常的な言葉に置き換えると次のどれが最も近いですか？ (A) 送信するデータの内容 (B) 通知の届け先住所 (C) イベントの発生時刻

    回答を表示

    答えは (B) 通知の届け先住所 です。エンドポイントはWebhookデータを受け取るためのURLで、 郵便の宛先住所のように「どこに届けるか」を示すものです。

  2. Q2. Webhookのエンドポイントとして使えないURLはどれですか？ (A) https://myapp.example.com/webhook (B) http://localhost:3000/webhook (C) https://api.myservice.com/notifications

    回答を表示

    答えは (B) http://localhost:3000/webhook です。 localhostは自分のパソコン内だけでしかアクセスできないURLなので、 インターネット上の送信元サービスからはアクセスできません。 また多くのサービスはHTTPS（http:// ではなく https://）を必須としています。

  3. Q3. Webhook登録時にイベントの種類を選択する主な目的はどれですか？ (A) セキュリティを高めるため (B) 必要なイベントの通知だけを受け取り、無駄な通信を減らすため (C) 送信先URLを複数設定するため

    回答を表示

    答えは (B) 必要なイベントの通知だけを受け取り、無駄な通信を減らすため です。 サービスには多くのイベントがあるため、不要なものまで受け取ると処理が増えます。 必要なイベントだけ選ぶことで効率的なシステムが作れます。
