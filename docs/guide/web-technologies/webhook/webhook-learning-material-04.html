Webhookで使われるHTTPの基礎

Webhookの通信で使われるHTTPの基本要素（POSTメソッド、ヘッダー、ボディ、ステータスコード）を学びます。プログラミング未経験でも理解できるよう、各要素を具体例とともに解説します。

この章で学ぶこと

  • HTTPとは何かを「手紙のやり取り」に例えて理解できる
  • WebhookでPOSTメソッドが使われる理由を説明できる
  • HTTPヘッダーとボディの役割の違いを理解できる
  • Webhookでよく使われるステータスコード（200・4xx・5xx）の意味を説明できる

4.1 HTTPとは何か？

Webhookを理解するには、まず「HTTP」という通信の仕組みを知る必要があります。 HTTPは難しく聞こえますが、私たちが毎日使っている「手紙のやり取り」にそっくりです。

手紙のやり取りとHTTPの比較

手紙のやり取り HTTPの世界
手紙を送る人（差出人） クライアント（送信元）
手紙を受け取る人（受取人） サーバー（受信先）
封筒の宛名・差出人・切手 HTTPヘッダー（付加情報）
手紙の本文 HTTPボディ（データ本体）
「受け取りました」という返信 HTTPレスポンス（応答）

重要ポイント

HTTPとは、インターネット上でデータをやり取りするための「共通のルール（プロトコル）」です。 Webhookもこのルールに従ってデータを送受信します。 「HyperText Transfer Protocol」の略称ですが、名前よりも「手紙のルール」だと覚えておきましょう。

HTTPリクエストとレスポンスの基本的な流れ

sequenceDiagram participant C as 送信元
（Webhookを送る側） participant S as 受信先
（あなたのサーバー） C->>S: HTTPリクエスト送信
（POSTメソッド＋データ） Note right of S: ヘッダーとボディを読んで処理 S-->>C: HTTPレスポンス返却
（ステータスコード: 200 OK） Note left of C: 正常受信を確認

4.2 HTTPメソッドとは？ 〜GETとPOSTの違い〜

HTTPには「メソッド」と呼ばれる「何をしたいか」を指定する仕組みがあります。 Webhookでは主にPOSTメソッドが使われます。なぜでしょうか？

GETメソッド

「情報を取ってくる（取得する）」ためのメソッドです。 Webブラウザでページを開くときに使われます。

例え

図書館で「この本を見せてください」とお願いする行為

Webhookでは使わない理由

GETはデータを「送る」のに不向き。URLにデータをつけるため、大量データや秘密情報を送れない

POSTメソッド（Webhookで使用）

「データを送る（投稿する）」ためのメソッドです。 フォームの送信やWebhookで活用されます。

例え

図書館に「この本を返します（＋返却書を添付）」と届ける行為

Webhookで使われる理由

ボディにJSONデータを含めて送れる。大量データも安全に送信できる

主なHTTPメソッド一覧

GET

データを取得する

Webページを開く、APIからデータを読む

POST

データを送信する ← Webhookで主に使用

フォーム送信、Webhookの通知送信

PUT

データを更新する（全体）

既存のリソースを丸ごと置き換える

DELETE

データを削除する

リソースの削除操作

4.3 HTTPヘッダー 〜封筒の表書きに相当する情報〜

HTTPヘッダーは、データ本体（ボディ）と一緒に送られる「付加情報」です。 手紙の封筒に書かれた宛先・差出人・切手の種類のようなものです。

Webhookでは、ヘッダーを使って「どんな種類のデータを送るか」「本物のWebhookか確認するための署名」などの情報を伝えます。

Webhookでよく使われるHTTPヘッダー

ヘッダー名 → 値の例
Content-Type

application/json

「ボディのデータはJSON形式です」と伝える。Webhookでは必ずこれが指定される

User-Agent

GitHub-Hookshot/abc123

「このWebhookを送ったのはGitHubです」という送信元の識別情報

X-Webhook-Signature

sha256=abc123...

署名（サイン）。「このデータは本物の送信元から来ています」という証明。第10章で詳しく学びます

X-GitHub-Event

push / pull_request

GitHubの例。「どんなイベントが起きたか」を伝えるカスタムヘッダー

豆知識: X-プレフィックスとは？

ヘッダー名が「X-」で始まるものは「カスタムヘッダー」と呼ばれます。 標準のHTTPヘッダーにない独自の情報を追加するときに使います。 GitHubの「X-GitHub-Event」やStripeの「Stripe-Signature」もカスタムヘッダーです。

4.4 HTTPボディ 〜手紙の本文に相当するデータ〜

HTTPボディとは、リクエストに含まれる「データ本体」のことです。 Webhookでは、イベントの詳細情報がボディに入って送られてきます。

ボディの中身はJSON形式（第5章で詳しく学びます）で書かれることがほとんどです。 ここでは「こんなデータが届く」というイメージをつかんでください。

GitHubからWebhookが届いたときのボディ例

「プッシュがありました」というイベントで届くデータのイメージ：

{
  "event": "push",
  "repository": {
    "name": "my-project",
    "full_name": "user/my-project"
  },
  "pusher": {
    "name": "tanaka",
    "email": "tanaka@example.com"
  },
  "commits": [
    {
      "message": "バグを修正しました",
      "timestamp": "2026-03-25T10:00:00Z"
    }
  ]
}

何が起きたか

"event": "push"

どのリポジトリか

"name": "my-project"

誰がやったか

"name": "tanaka"

HTTPリクエスト全体の構造（Webhookの場合）

リクエストライン

POST https://example.com/webhook HTTP/1.1

→ POSTメソッドで、このURLに送ります

ヘッダー（付加情報）

Content-Type: application/json X-GitHub-Event: push X-Hub-Signature-256: sha256=abc...

→ データの形式・イベント種類・署名を伝えます

ボディ（データ本体）

{'{'} "event": "push", "repository": {'{'}"name": "my-project"{'}'} {'}'}

→ イベントの詳細情報が入っています

4.5 HTTPステータスコード 〜返事の種類を数字で表す〜

Webhookを受け取ったサーバーは、送信元に「結果」を返します。 この結果を数字で表したのがHTTPステータスコードです。

宅配便の「配達完了」「不在のため持ち帰り」「住所不明で返送」のような状態を、3桁の数字で表現しています。

2xx

成功（Success）

リクエストが正常に処理されました。Webhookでは必ずこの応答を返す必要があります。

200 OK

最も一般的な成功応答。「問題なく受け取りました」

204 No Content

成功したが返すデータなし。Webhookでよく使われる

4xx

クライアントエラー（送信側の問題）

送信したリクエストに問題がありました。送信元の設定や認証を確認する必要があります。

400 Bad Request

リクエストの形式が間違っている。JSONの書き方が不正など

401 Unauthorized

認証が必要。正しい署名やトークンがない

403 Forbidden

アクセス権がない。IPアドレス制限に引っかかった場合など

404 Not Found

URLが存在しない。エンドポイントのパスが間違っている

5xx

サーバーエラー（受信側の問題）

受信するサーバー側で問題が発生しました。Webhookを受け取ったサーバーのプログラムが壊れているときなどに発生します。

500 Internal Server Error

サーバー内部でエラーが発生。プログラムのバグなど

503 Service Unavailable

サーバーが利用不可。メンテナンス中や過負荷など

重要: Webhookでステータスコードが大切な理由

Webhookを受け取ったら、必ず 2xx のステータスコードを返すことが重要です。 5xx などのエラーが返ると、送信元サービスは「届かなかった」と判断して再送（リトライ）を行います。 同じWebhookが何度も届いてしまう原因になります。（第11章で詳しく学びます）

4.6 HTTPSとは？ 〜安全な通信のための暗号化〜

Webhookに使われるURLはほぼ必ず「HTTPS」で始まります。 HTTPとHTTPSはどう違うのでしょうか？

HTTP（暗号化なし）

データが「素のまま」インターネットを流れます。 途中で第三者に盗み見られる可能性があります。

例え: ハガキで手紙を送る（内容が丸見え）

⚠ Webhookでは使用禁止

HTTPS（暗号化あり）

データが暗号化されてインターネットを流れます。 途中で盗み見られても意味のある情報は読めません。

例え: 封筒に入れて暗号で書いた手紙を送る

✓ Webhookで必ず使用する

重要ポイント

WebhookのエンドポイントURLは必ずhttps://で始まるものを使いましょう。 多くのWebhook送信サービス（GitHub、Stripe等）は、HTTPSでないURLへの送信を拒否または警告します。 個人情報や認証情報が含まれるWebhookデータの盗聴を防ぐために必須です。

4.7 Webhookのリクエストを読み解いてみよう

ここまで学んだHTTPの知識を使って、実際のWebhookリクエストを読み解いてみましょう。 以下はStripe（決済サービス）から届くWebhookの例です。

Stripe決済完了Webhookのリクエスト例

メソッド ＋ URL

POST https://your-server.com/webhook/stripe

ヘッダー

Content-Type: application/json
Stripe-Signature: t=1679999999,v1=abc123def456...
User-Agent: Stripe/1.0 (+https://stripe.com/docs/webhooks)

ボディ（JSONデータ）

{
  "type": "payment_intent.succeeded",
  "data": {
    "object": {
      "amount": 5000,
      "currency": "jpy",
      "status": "succeeded",
      "customer": "cus_abc123"
    }
  }
}

このリクエストから読み取れること：

POSTメソッドで、あなたのサーバーにデータを「送っている」

Content-TypeがJSONなので、ボディはJSON形式のデータ

Stripe-Signatureで本物のStripeからの通知か確認できる

ボディから「決済が完了した」「5000円」「顧客ID」が分かる

  • 2

    WebhookではPOSTメソッドが使われる。GETは「取得」用で、POSTは「送信」用。データをボディに含めて送れるため

  • 4

    ステータスコードで処理結果を数字で返す。Webhookを受け取ったら必ず200などの2xxを返すことが重要

  • 5

    Webhookのエンドポイントは必ずHTTPSを使う。暗号化によりデータの盗聴を防げる

理解度確認クイズ

  1. Q1. WebhookではなぜGETではなくPOSTメソッドが使われるのですか？ (A) GETより速いから (B) データをボディに含めて送れるから (C) GETはWebhookに対応していないから

    回答を表示

    答えは (B) です。POSTメソッドはリクエストのボディにデータを含めて送ることができます。 GETはURLにデータをつけるため大量のデータや秘密情報（署名など）を安全に送ることができません。 Webhookはイベントの詳細データをボディのJSONとして送るため、POSTが適しています。

  2. Q2. Webhookを受け取ったサーバーが「500 Internal Server Error」を返すと、どうなりますか？ (A) 何も起きない (B) 送信元サービスが「届かなかった」と判断して再送する可能性がある (C) 送信元サービスがWebhookを停止する

    回答を表示

    答えは (B) です。5xxのステータスコードを受け取った送信元サービスは「受信側でエラーが起きた」と判断し、 しばらく後に同じWebhookを再送（リトライ）します。 これにより同じデータが複数回届くことがあります。Webhookを受け取ったら必ず200を返しましょう。

  3. Q3. HTTPヘッダーとHTTPボディの違いを正しく説明しているのはどれですか？ (A) ヘッダーはデータ本体、ボディは付加情報 (B) ヘッダーは付加情報（メタデータ）、ボディはデータ本体 (C) ヘッダーとボディは同じもので、呼び方が違うだけ

    回答を表示

    答えは (B) です。HTTPヘッダーは「Content-Type」「User-Agent」「署名」などの付加情報で、 手紙の封筒の表書きに相当します。HTTPボディはJSON形式のイベントデータなど「データ本体」で、 手紙の本文に相当します。両者は別々の情報です。
