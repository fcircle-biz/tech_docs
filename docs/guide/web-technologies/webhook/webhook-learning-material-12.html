Webhookを体験しよう 〜Webhook.siteで実際に受け取る〜

Webhook.siteなどの無料ツールを使って、実際にWebhookのデータを受け取る体験をします。プログラミングなしでWebhookの動作を確認し、リクエストの中身を観察する方法を学びます。

この章で学ぶこと

  • Webhook.siteとは何か、どのように使うかを理解できる
  • プログラミングなしでWebhookの送受信を体験できる
  • 受信したHTTPリクエストのヘッダーとボディの内容を読み解ける
  • 実際のサービス（GitHub等）からWebhookを受け取る手順を理解できる

12.1 Webhook.siteとは何か

これまでの章では、Webhookの仕組みや使われ方を理論として学んできました。この章では、実際に手を動かして「本物のWebhookデータを受け取る」体験をしましょう。

しかし、Webhookを受け取るには通常「サーバー」が必要です。サーバーとは、インターネット上で24時間稼働してリクエストを待ち受けるコンピューターのことです。これを自分で用意するのは、プログラミング初心者には難しく感じるかもしれません。

そこで登場するのが Webhook.site です。

Webhook.site とは

Webhook.site は、無料で利用できるWebhookテスト用のオンラインサービスです。 アクセスするだけで、あなただけの「Webhook受け取り専用URL」が自動生成されます。 そのURLにHTTPリクエストを送ると、画面上でリクエストの内容をリアルタイムで確認できます。

URL: https://webhook.site

無料で使用可能（アカウント登録不要）

通常のWebhook受け取り

自分でサーバーを用意してプログラムを書き、インターネットに公開する必要がある。 準備に時間がかかり、プログラミングの知識も必要。

Webhook.siteを使う場合

ブラウザでアクセスするだけで即座にURLが発行される。 プログラミング不要で、届いたデータをリアルタイムで確認できる。

学習ポイント

Webhook.siteは「仮の宅配ボックス」のようなものです。本番サービスで使うためのものではありませんが、 Webhookの動作を学習・確認するための便利なツールです。

12.2 Webhook.siteの使い方 〜URLを発行する〜

実際にWebhook.siteを使ってみましょう。手順はとてもシンプルです。

体験 12-1: Webhook受け取りURLを発行する

Webhook.siteにアクセスして、自分専用のURLを発行します。

手順

  1. 1 ブラウザで https://webhook.site にアクセスします
  2. 2 画面上部に「Your unique URL」として長いURLが表示されます（例: https://webhook.site/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx）
  3. 3 このURLをコピーしておきます（後で使います）
  4. 4 画面下部には「Requests」という一覧が表示されます（最初は空です）。ここにWebhookが届くと表示されます

Webhook.site の画面構成

表示エリア 内容
Your unique URL あなた専用のWebhook受け取りURL。ここにリクエストを送ります
Requests 一覧 届いたリクエストが時系列で並びます。クリックで詳細を確認できます
詳細パネル 選択したリクエストのヘッダー、ボディ、メソッドなどの詳細情報

注意

Webhook.siteで発行されたURLは一時的なものです。ブラウザを閉じたり、長時間経過すると無効になります。 また、届いたデータは誰でも見られる可能性があるため、本番の機密情報は送らないようにしましょう。

12.3 実際にWebhookを送ってみよう 〜curlコマンドを使う〜

URLが発行できたら、実際にそこにデータを送ってみましょう。コマンドラインツール「curl」を使ってHTTPリクエストを送信します。

curlとは

curl（カール）は、コマンドラインからHTTPリクエストを送ることができるツールです。 macOSやLinuxには標準で搭載されており、Windowsでも最新版には含まれています。 ターミナル（WindowsはコマンドプロンプトまたはPowerShell）で使用します。

以下のコマンドを実行すると、Webhook.siteのURLにJSONデータを送信できます。 YOUR_WEBHOOK_URL の部分を、実際に発行されたURLに置き換えてください。

ターミナル / コマンドプロンプト 
# Webhook.siteにJSONデータをPOSTで送信する
curl -X POST https://webhook.site/YOUR_WEBHOOK_URL \
  -H "Content-Type: application/json" \
  -d '{"event": "test", "message": "はじめてのWebhook体験！"}'

curlコマンドのオプション解説

-X POST

HTTPメソッドをPOSTに指定します。Webhookでは通常POSTが使われます

-H "Content-Type: application/json"

ヘッダーを指定します。「送るデータはJSON形式ですよ」という宣言です

-d '{"event": ...}'

ボディ（本文）を指定します。ここに送りたいJSONデータを記述します

コマンドを実行したら、Webhook.siteのブラウザ画面を確認してみましょう。 「Requests」一覧に新しいエントリが追加されているはずです！

動作確認

curlコマンド実行後にWebhook.siteで確認すること

期待される結果 

- Requests一覧に新しいエントリが追加される
- メソッド: POST
- 「Body」タブに送ったJSONが表示される:
  {"event": "test", "message": "はじめてのWebhook体験！"}
- 「Headers」タブに Content-Type: application/json が表示される

12.4 届いたリクエストの中身を読み解こう

Webhook.siteにリクエストが届いたら、その内容を詳しく確認してみましょう。 実際のWebhookデータがどのような構造になっているか、画面で確認できます。

リクエスト詳細パネルの見方

Method HTTPメソッド

POST と表示されるはずです。Webhookでは主にPOSTが使われます。 送信元サービスが「新しいデータを送りますよ」という意味でPOSTを使います。

Headers ヘッダー情報

リクエストのメタ情報が含まれます。よく見るヘッダーの例：

ヘッダーの例 
Content-Type: application/json
User-Agent: curl/7.68.0
X-Hub-Signature-256: sha256=abc123...
Body ボディ（本文）

実際に送られたデータが表示されます。先ほど送ったコマンドの場合：

ボディの例（JSON） 
{
  "event": "test",
  "message": "はじめてのWebhook体験！"
}

12.5 実際のサービスからWebhookを受け取る 〜GitHubを例に〜

curlでの手動送信の次は、実際のサービス（ここではGitHub）から本物のWebhookを受け取る体験をしてみましょう。 GitHubはWebhookの設定がわかりやすく、無料アカウントでも利用できるため、学習に最適です。

体験 12-2: GitHubリポジトリにWebhookを設定する

GitHubのリポジトリに、Webhook.siteのURLを設定します。（GitHubアカウントが必要です）

手順

  1. 1 GitHubで任意のリポジトリを開き、「Settings（設定）」タブをクリックします
  2. 2 左メニューの「Webhooks」をクリックし、「Add webhook」ボタンを押します
  3. 3 「Payload URL」にWebhook.siteで発行したURLを貼り付けます
  4. 4 「Content type」を application/json に変更します
  5. 5 「Which events would you like to trigger this webhook?」で「Just the push event」を選択して「Add webhook」をクリックします
  6. 6 その後、リポジトリに何らかのファイルを追加・編集してコミット＆プッシュすると、Webhook.siteにリクエストが届きます

GitHubからのWebhookペイロードは非常に詳細な情報を含んでいます。以下は実際のペイロードの一部です。

GitHub push イベントのWebhookペイロード（一部抜粋） 
{
  "ref": "refs/heads/main",
  "before": "abc123...",
  "after": "def456...",
  "repository": {
    "name": "my-project",
    "full_name": "username/my-project"
  },
  "pusher": {
    "name": "username",
    "email": "user@example.com"
  },
  "commits": [
    {
      "id": "def456...",
      "message": "ファイルを追加しました",
      "author": { "name": "username" }
    }
  ]
}

重要ポイント

GitHubのWebhookペイロードには「誰が」「何を」「どのブランチに」プッシュしたか、 「どのファイルが変更されたか」など、イベントに関するあらゆる情報が含まれています。 これが第5章で学んだ「Webhookのペイロード」です。

12.6 Webhook.site以外の代替ツール

Webhook.siteと同様に、Webhookのテストに使えるツールが他にもあります。用途や好みに合わせて使い分けましょう。

RequestBin（Pipedream）

https://pipedream.com/requestbin で利用できるWebhookテストツールです。 ワークフロー自動化と組み合わせた高度な利用も可能です。

無料プランあり

ngrok

自分のコンピューター上で動かしているサーバーを一時的にインターネットに公開できるツールです。 「ローカルのサーバーでWebhookを受け取りたい」場合に使います。プログラミングの学習が進んだら試してみましょう。

無料プランあり・要インストール

Postman

APIやWebhookの開発・テストに広く使われるGUIツールです。 curlコマンドの代わりに、グラフィカルな画面でHTTPリクエストを送ることができます。 「Mock Server」機能でWebhook受け取りのテストも可能です。

無料プランあり・要インストール

12.7 体験で気づくこと 〜Webhookの実感〜

実際にWebhookを受け取ってみると、理論だけでは分からなかったことが体感できます。 ここでは体験を通じて得られる気づきを整理します。

気づき1: リアルタイムの速さ

curlコマンドを実行した瞬間、Webhook.siteの画面にリクエストが表示されます。 「送ったら即届く」というリアルタイム性を実感できます。 これがポーリングとの大きな違いです。

気づき2: データの構造

ヘッダーとボディが明確に分かれていることがわかります。 第4章で学んだHTTPの基礎（ヘッダー・ボディ・ステータスコード）が、 実際の画面で確認できます。

気づき3: 情報量の多さ

GitHubのWebhookペイロードを見ると、予想以上に多くの情報が含まれていることがわかります。 「どの情報が必要か」を選択して使うのが実際の開発です。

気づき4: セキュリティの大切さ

Webhook.siteのURLを誰かに教えると、その人もリクエストを送れてしまいます。 第10章で学んだ「署名検証」が必要な理由を、体験を通じて実感できます。

よくある疑問と回答

curlコマンドが動かない

原因: curlがインストールされていないか、パスが通っていない可能性があります

解決方法: Windowsの場合は「Windows PowerShell」を使うか、Postmanなどのツールでリクエストを送信してください。macOS/Linuxでは標準でcurlが使えます

Webhook.siteに何も届かない

原因: URLが正しくない、またはセッションが切れている可能性があります

解決方法: Webhook.siteを再読み込みして新しいURLを発行し直してください。curlコマンドのURLが最新のものか確認しましょう

GitHubのWebhook設定でエラーになる

原因: GitHubはWebhook設定後にテスト送信を行い、URLが有効かチェックします。Webhook.siteのURLが無効になっていると失敗します

解決方法: まずWebhook.siteでURLを発行し直し、すぐにGitHubに設定してください。設定完了後すみやかにテストしましょう

理解度確認クイズ

  1. Q1. Webhook.siteの主な用途として最も適切なものはどれですか？ (A) Webhookを本番環境で運用するためのサーバー (B) Webhookのリクエストをテスト・確認するためのツール (C) Webhookのセキュリティを強化するサービス

    回答を表示

    答えは (B) です。Webhook.siteは学習・テスト目的のツールです。本番運用には使わず、 本番サービスでは適切なサーバーやクラウド環境を使います。

  2. Q2. curlコマンドで -X POST オプションは何を指定していますか？ (A) 送信先のURL (B) HTTPメソッド (C) 認証情報

    回答を表示

    答えは (B) HTTPメソッド です。-X POSTはHTTPメソッドとして「POST」を使うことを指定しています。 Webhookでは一般的にPOSTメソッドが使われます。

  3. Q3. GitHubのWebhookペイロードで確認できる情報として正しいものはどれですか？ (A) リポジトリの中身のコード全文 (B) 誰がどのブランチにプッシュしたか、コミットメッセージなどのイベント情報 (C) GitHubのパスワードやAPIキー

    回答を表示

    答えは (B) です。WebhookペイロードにはイベントのメタデータとしてPushしたユーザー名、ブランチ名、コミットメッセージなどが含まれます。 コード全文や機密情報は含まれません。
