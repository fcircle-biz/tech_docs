Webhookのセキュリティ 〜安全に使うための基礎〜

Webhookを安全に利用するために必要なセキュリティの考え方を学びます。署名検証（シークレットキー）、HTTPS通信、IPアドレス制限など、基本的なセキュリティ対策を初心者向けに解説します。

この章で学ぶこと

  • Webhookがなぜセキュリティリスクをもつのかを理解できる
  • 署名検証（シークレットキー）の仕組みと目的を説明できる
  • HTTPS通信がWebhookでなぜ必須なのかを理解できる
  • IPアドレス制限など基本的なセキュリティ対策を列挙できる
  • Webhookのセキュリティチェックリストを活用できる

10.1 Webhookのセキュリティリスクとは

Webhookはとても便利な仕組みですが、使い方を間違えると深刻なセキュリティ問題につながることがあります。 まず「どのようなリスクがあるのか」を理解しましょう。

Webhookの主なセキュリティリスク

  • なりすまし攻撃：悪意のある第三者が、正規のサービスのふりをして偽のWebhookを送りつける
  • データの盗聴：HTTP（暗号化なし）で通信すると、送受信のデータが第三者に読まれる可能性がある
  • データの改ざん：通信途中でデータが書き換えられても、気づかずに処理してしまう
  • リプレイ攻撃：一度キャプチャした正規のWebhookをコピーして繰り返し送りつける

具体的なイメージで考えてみましょう。あなたが運営するオンラインショップが 「支払い完了のWebhookを受け取ったら商品を発送する」という仕組みを作ったとします。

セキュリティ対策がない場合の攻撃シナリオ

sequenceDiagram participant A as 悪意のある第三者 participant B as あなたのサーバー participant C as 倉庫・発送システム Note over A,C: セキュリティ対策なしの場合 A->>B: 偽の「支払い完了」Webhookを送信 Note over B: 誰でも送れる！検証なし B->>C: 商品を発送してください C-->>B: 発送完了 Note over A,C: お金を払わずに商品をだまし取れてしまう

重要ポイント

Webhookのエンドポイント（受け取るURL）は、インターネット上で公開されています。 「誰でもアクセスできる場所」に置かれているため、適切なセキュリティ対策がなければ 悪意のあるリクエストを受け入れてしまいます。

10.2 HTTPS通信 〜データを守る暗号化〜

Webhookで最初に意識すべきセキュリティ対策が HTTPS通信 です。 まずHTTPとHTTPSの違いを理解しましょう。

HTTP（暗号化なし）

データがそのままインターネット上を流れます。途中で第三者が盗み見したり、 書き換えることができる「はがき」のようなものです。

Webhookでの使用は危険。機密データが漏洩する可能性あり

HTTPS（SSL/TLS暗号化）

データが暗号化されてやり取りされます。途中で盗み見られても解読できない 「封筒に入った手紙」のようなものです。

Webhookでは必ずHTTPSを使用すること

注意

多くのWebhookサービス（GitHub、Stripe等）は、HTTPSのエンドポイントしか受け付けません。 HTTPのURLを登録しようとするとエラーになります。これは意図的なセキュリティ設計です。

HTTPS通信でのWebhookの流れ

flowchart LR A["送信元サービス"] -->|"暗号化されたデータ"| B{{"インターネット"}} B -->|"暗号化されたデータ"| C["あなたのサーバー"] style A fill:#f0fdfa,stroke:#14b8a6 style B fill:#fef3c7,stroke:#f59e0b style C fill:#f0fdfa,stroke:#14b8a6

通信経路全体が暗号化されるため、途中での盗聴・改ざんを防げます

10.3 署名検証 〜送信元が本物かを確認する〜

Webhookセキュリティの核心が 署名検証（シグネチャ検証）です。 これは「届いたWebhookが本当に正規のサービスから送られてきたのか」を確認する仕組みです。

署名検証の仕組み

送信元サービスとあなたが シークレットキー（秘密の合言葉）を共有します。 送信元はWebhookデータをシークレットキーで「署名」し、受信側はその署名が正しいかを検証します。

合言葉を知らない第三者は、正しい署名を作れません。 これにより「本物の送信元からのWebhookだ」と確認できます。

署名検証の全体的な流れ

sequenceDiagram participant S as 送信元サービス participant N as あなたのサーバー Note over S,N: 事前準備（一度だけ） S->>N: シークレットキーを発行・共有 Note over S,N: Webhook送信時 S->>S: シークレットキーでデータに署名を作成 S->>N: Webhookデータ + 署名をHTTPヘッダーに添付して送信 N->>N: 受け取ったデータで署名を再計算 N->>N: 添付された署名と比較して一致を確認 alt 署名が一致 N-->>S: 処理実行 + 200 OK else 署名が不一致 N-->>S: 拒否 + 401 Unauthorized end

実際のサービスでの署名の例

GitHubの署名検証

GitHubは X-Hub-Signature-256 というHTTPヘッダーに 署名を含めて送信します。HMAC-SHA256というアルゴリズムで計算されます。

X-Hub-Signature-256: sha256=abc123def456...（長い文字列）

Stripeの署名検証

Stripeは Stripe-Signature ヘッダーを使います。 タイムスタンプも含まれており、リプレイ攻撃（古いデータの再送）への対策も組み込まれています。

Stripe-Signature: t=1702000000,v1=abc123...

HMAC署名のしくみ（概念）

材料 内容 例
Webhookデータ 送信するJSONデータ（ボディ） {"event":"payment.succeeded",...}
シークレットキー 送受信双方が知っている秘密の文字列 whsec_abcdefgh123...
署名（結果） 2つを組み合わせて計算された固有の文字列 sha256=3e2f1a...

シークレットキーを知らない人は同じデータから同じ署名を作れないため、なりすましを防げます。

豆知識：シークレットキーの扱い方

シークレットキーは絶対に公開しないでください。GitHubに誤ってコードと一緒に アップロードしてしまうと、第三者に署名を偽造されてしまいます。 環境変数（.envファイル等）に保存し、コードに直接書かないようにしましょう。

10.4 IPアドレス制限 〜許可された送信元だけ受け付ける〜

署名検証と組み合わせて使われる対策が IPアドレス制限 です。 これは「特定のIPアドレスからのリクエストしか受け付けない」という設定です。

IPアドレス制限のイメージ

（IPアドレス: 192.30.252.x）"] -->|"Webhookリクエスト"| D{{"あなたのサーバー\nIPフィルタリング"}} B["不明なサーバー
（IPアドレス: 攻撃者）"] -->|"偽のWebhookリクエスト"| D D -->|"GitHubのIPから？YES"| E["処理実行"] D -->|"不明なIPから？NO"| F["アクセス拒否 403"] style A fill:#f0fdfa,stroke:#14b8a6 style B fill:#fef2f2,stroke:#ef4444 style D fill:#fef3c7,stroke:#f59e0b style E fill:#ecfdf5,stroke:#10b981 style F fill:#fef2f2,stroke:#ef4444

IPアドレス制限のメリット

  • 署名検証の前段階でブロックできる
  • サーバーへの不要な負荷を減らせる
  • 多層防御の一層として機能する

注意点

  • サービスがIPを変更した場合に動かなくなる
  • IPの偽装（スプーフィング）は完全に防げない
  • 署名検証と組み合わせて使うのが基本

主要サービスのIPアドレス帯域の確認方法

GitHub

api.github.com/meta のエンドポイントでhooksのIPリストを確認できます

Stripe

Stripeのドキュメントページで送信元IPの範囲を確認できます

Slack

Slackのヘルプセンターで送信元IPの情報を確認できます

10.5 リプレイ攻撃への対策 〜タイムスタンプの活用〜

署名検証だけではまだ不十分なケースがあります。それが リプレイ攻撃 への対策です。

リプレイ攻撃とは？

sequenceDiagram participant A as 正規の送信元 participant H as 攻撃者 participant B as あなたのサーバー A->>B: 正規のWebhook（署名付き）を送信 H->>H: 通信をキャプチャして保存 Note over H,B: 時間が経った後（例：1日後） H->>B: 保存した同じWebhookをコピーして再送 Note over B: 署名は本物と同じなので通過してしまう Note over B: 同じ処理が二重に実行される！

タイムスタンプを使ったリプレイ攻撃対策

手順1

送信元がWebhookデータに 現在の時刻（タイムスタンプ）を含めて署名する

手順2

受信側はWebhookを受け取ったとき、タイムスタンプと現在時刻の 差が許容範囲内（例：5分以内）か確認する

手順3

時刻の差が許容範囲外（例：5分以上）なら、古いデータとして 拒否する

重要ポイント

StripeなどのWebhookサービスでは、タイムスタンプを署名に含める仕組みが標準で組み込まれています。 SDKやライブラリを使うと、こうした検証処理を自動で行ってくれます。

10.6 その他のセキュリティ対策

HTTPSと署名検証の他にも、Webhookを安全に使うためのベストプラクティスがいくつかあります。

すぐにレスポンスを返す（DoS攻撃対策）

Webhookを受け取ったらすぐに「200 OK」を返し、重い処理は後で別途実行しましょう。 レスポンスが遅いと、送信元がタイムアウトして再送を繰り返し、サーバーに大きな負荷がかかります。 目安として 5秒以内にレスポンスを返すのが一般的です。

べき等性の確保（重複処理の防止）

Webhookは何らかの理由で同じデータが複数回届くことがあります（第11章で詳しく解説）。 「同じWebhookが2回来ても、処理結果は1回と同じ」になるよう設計することを べき等性といいます。注文IDなどのユニークなIDで重複チェックを行いましょう。

受信ログの記録（監査と調査用）

受け取ったWebhookのログ（記録）を保存しておきましょう。 不審なアクセスを検知したり、問題が起きたときの原因調査に役立ちます。 特に日時、IPアドレス、ペイロードのハッシュ値は記録しておくと便利です。

シークレットキーの定期的な更新

シークレットキーは定期的に変更することが推奨されます。 万が一キーが漏洩してしまった場合は、すぐに再発行して古いキーを無効にしましょう。 多くのサービスで管理画面からキーの再発行が簡単にできます。

10.7 Webhookセキュリティの全体像

これまで学んだセキュリティ対策を一つの図で整理してみましょう。 「多層防御」という考え方で、複数の対策を組み合わせるのが基本です。

Webhookセキュリティの多層防御

1

HTTPS通信（必須）

データの暗号化・盗聴防止

2

IPアドレス制限（推奨）

許可された送信元のみ受け付ける

3

署名検証（必須）

送信元の正当性確認・改ざん検知

4

タイムスタンプ検証（推奨）

リプレイ攻撃の防止

5

べき等性・ログ記録（推奨）

重複処理の防止・監査証跡

理解度確認クイズ

  1. Q1. Webhookのエンドポイントには必ずHTTPSを使う理由として正しいものはどれですか？ (A) HTTPSの方が処理速度が速いから (B) データを暗号化して盗聴や改ざんを防ぐため (C) HTTPSでないとWebhookが届かないから

    回答を表示

    答えは (B) です。HTTPSはSSL/TLS暗号化によりデータの盗聴・改ざんを防ぎます。 (A) 速度は関係ありません。(C) HTTPも技術的には届きますが、多くのサービスがセキュリティ上の理由でHTTPSのみ受け付けます。

  2. Q2. 署名検証の目的は何ですか？ (A) Webhookの送信速度を向上させるため (B) 送信元が本物であることを確認し、データの改ざんを検知するため (C) Webhookのデータを圧縮して転送量を削減するため

    回答を表示

    答えは (B) です。署名検証はシークレットキーを使い、届いたWebhookが正規の送信元からのものかを確認し、 データが途中で改ざんされていないかを検知します。

  3. Q3. シークレットキーの管理として正しい方法はどれですか？ (A) ソースコードに直接書いてGitHubで管理する (B) メモ帳に書いてデスクトップに置く (C) 環境変数（.envファイル等）に保存し、Gitの管理対象外にする

    回答を表示

    答えは (C) です。シークレットキーをソースコードに直接書いてGitHubにアップすると、 世界中に公開されてしまいます。環境変数に保存し、.gitignoreで管理対象外にするのが基本です。
