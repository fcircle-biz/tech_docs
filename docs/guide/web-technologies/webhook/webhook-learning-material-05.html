JSONデータの読み方 〜Webhookで届くデータ〜

Webhookで送受信されるデータの形式であるJSONについて学びます。JSONの構造（キーと値、配列、ネスト）を理解し、実際のWebhookペイロードを読み解く力を身につけましょう。

この章で学ぶこと

  • JSONとは何かを説明でき、なぜWebhookで使われるのかを理解できる
  • JSONの基本構造（キーと値、オブジェクト、配列、ネスト）を読み解ける
  • 実際のWebhookペイロード（GitHub、Slack、Stripe）の例を読むことができる
  • JSONで使えるデータ型（文字列、数値、真偽値、null、配列、オブジェクト）を覚えられる

5.1 JSONとは？ 〜データを整理する「箱」〜

第4章でWebhookのHTTPリクエストについて学びました。リクエストの「ボディ（本文）」に含まれるのがペイロード（payload）と呼ばれるデータです。このペイロードは、ほとんどのWebhookではJSON（ジェイソン）という形式で書かれています。

JSONとは「JavaScript Object Notation（ジャバスクリプト オブジェクト ノーテーション）」の略です。難しそうに見えますが、要は「コンピューター同士がデータをやり取りするための共通の書き方のルール」です。

JSON以前の問題

「注文者名: 山田太郎、商品: りんご、数量: 3」という情報を送りたいとき、どうフォーマットするか決まっていなければ受け取る側が困ります。

フォーマットがバラバラ

システムごとに独自の書き方 → 連携が難しい

JSONによる解決

JSONという共通のルールで書けば、どのプログラミング言語でも、どのシステムでも同じように読み書きができます。

共通フォーマット

世界中のシステムが同じルールで通信 → 連携が簡単

重要ポイント

JSONは「軽くて、人間にも読みやすく、コンピューターにも処理しやすい」データ形式です。 WebhookだけでなくAPIのレスポンスにも広く使われており、Web開発の世界の標準フォーマットになっています。

5.2 JSONの基本構造 〜キーと値のペア〜

JSONの最も基本的な形は「キー（key）」と「値（value）」のペアです。辞書や住所録のようなイメージで考えてみましょう。

キーと値の考え方

日常の例（住所録）

名前 山田太郎
年齢 30
メール yamada@...

JSONで表すと

{
  "name": "山田太郎",
  "age": 30,
  "email": "yamada@example.com"
}

ルール確認： キーは必ず"（ダブルクォーテーション）で囲みます。 キーと値は:（コロン）で区切り、 複数のペアは,（カンマ）で区切ります。 全体は{ }（波かっこ）で囲みます。

波かっこ { } で囲まれたキーと値のセットを オブジェクト（object） と呼びます。一つのJSONデータにはオブジェクトがいくつも入れ子になって入っていることがよくあります。

JSONの基本形 
{
  "event": "order.completed",
  "order_id": 12345,
  "customer_name": "山田太郎",
  "total_price": 3200,
  "paid": true,
  "note": null
}

5.3 JSONで使えるデータ型

JSONの「値」の部分には、いくつかの種類（データ型）が使えます。それぞれの特徴を覚えましょう。

JSONのデータ型一覧

データ型 説明 例
文字列（string） テキスト。必ず " で囲む "山田太郎"
数値（number） 整数や小数。クォートは不要 42 3.14
真偽値（boolean） 「はい/いいえ」。true または false true false
null 「値なし」「空」を表す null
配列（array） リスト。[ ] で囲む ["りんご", "みかん"]
オブジェクト（object） 入れ子になったデータ。{ } で囲む {"name": "太郎"}
全データ型を含むJSONの例 
{
  "event_name": "order.completed",
  "order_id": 12345,
  "is_paid": true,
  "shipping_address": null,
  "items": ["りんご", "みかん", "ぶどう"],
  "customer": {
    "name": "山田太郎",
    "email": "yamada@example.com"
  }
}

ヒント：文字列と数値の見分け方

"30"（クォートあり）は文字列の「30」、 30（クォートなし）は数値の30です。 計算に使う値は数値型、そうでないものは文字列型で書くのが一般的です。

5.4 配列とネスト 〜データを入れ子にする〜

実際のWebhookで送られてくるJSONは、複数の商品リストや住所情報など、より複雑な構造を持っています。「配列」と「ネスト（入れ子）」を使うことで、こうした複雑なデータも表現できます。

配列（array）- 複数のデータをリストにする

配列は [ ]（角かっこ）で囲まれた、複数の値のリストです。スーパーのかごに商品を入れていくイメージです。

  • 複数の注文商品リスト
  • 変更されたファイルの一覧
  • 担当者のメールアドレス一覧 
{
  "items": [
    "りんご",
    "みかん",
    "ぶどう"
  ],
  "tags": [
    "urgent",
    "VIP"
  ]
}

ネスト（入れ子）- オブジェクトの中にオブジェクトを入れる

オブジェクトの値として、さらにオブジェクトを書くことができます。「本棚の中に引き出し、引き出しの中に小物」のような構造です。

ネスト構造のJSON例 
{
  "order_id": 12345,
  "customer": {
    "name": "山田太郎",
    "email": "yamada@example.com",
    "address": {
      "prefecture": "東京都",
      "city": "渋谷区",
      "zip": "150-0001"
    }
  }
}

JSONのネスト構造イメージ

5.5 実際のWebhookペイロードを読んでみよう

ここからは、実際によく使われるサービスのWebhookペイロードを見てみましょう。最初は複雑に見えますが、「キーと値」「配列」「ネスト」の組み合わせだと分かれば、必ず読み解けます。

GitHub Pushイベントのペイロード（簡略版）

GitHubでコードがプッシュされたときに送られるWebhookの例です。「誰が」「どのブランチに」「何のコミットを」プッシュしたかが分かります。

GitHub push ペイロード例 
{
  "ref": "refs/heads/main",
  "pusher": {
    "name": "yamada-taro",
    "email": "yamada@example.com"
  },
  "commits": [
    {
      "id": "abc1234",
      "message": "バグ修正: ログイン処理を改善",
      "added": ["src/login.js"],
      "modified": ["README.md"]
    }
  ],
  "repository": {
    "name": "my-project",
    "full_name": "yamada-taro/my-project"
  }
}

読み解きポイント

ref プッシュされたブランチのパス
pusher プッシュした人の情報（オブジェクト）
commits コミット情報のリスト（配列）
added 追加されたファイルのリスト（配列）

Stripe 決済完了イベントのペイロード（簡略版）

Stripeで支払いが完了したときに送られるWebhookの例です。「何円の支払いが」「誰から」「完了したか」が含まれます。

Stripe payment_intent.succeeded ペイロード例 
{
  "type": "payment_intent.succeeded",
  "data": {
    "object": {
      "id": "pi_abc123",
      "amount": 3200,
      "currency": "jpy",
      "status": "succeeded",
      "metadata": {
        "order_id": "order_789",
        "customer_name": "山田太郎"
      }
    }
  },
  "created": 1700000000
}

Slack メッセージ受信イベントのペイロード（簡略版）

SlackでBotにメッセージが送られたときのWebhookペイロード例です。チャットボット開発でよく使われます。

Slack message イベントペイロード例 
{
  "type": "message",
  "channel": "C1234567890",
  "user": "U0987654321",
  "text": "こんにちは！",
  "ts": "1700000000.000100",
  "team": "T0987654321"
}

注意

ここで示したペイロードはすべて「簡略版」です。実際のWebhookではより多くのフィールドが含まれます。 最初はすべてを理解しようとせず、「自分が必要な情報はどのキーにあるか」を探す感覚でOKです。

5.6 JSONを読むときのコツ

実際のWebhookペイロードを受け取ったとき、慌てずに読み解くためのコツを覚えておきましょう。

1

まず全体の「型」を確認する

最初に { } で始まるか [ ] で始まるかを見ます。 { } ならオブジェクト、[ ] なら配列です。Webhookの多くはオブジェクトです。

2

「event」や「type」キーを探す

多くのWebhookには "event" や "type" というキーがあります。 ここを見れば「何が起きたのか」が分かります（例: "payment.completed"）。

3

インデント（字下げ）で構造を把握する

JSONはインデント（行頭のスペース）で階層の深さを表します。同じ深さにあるキーは同じレベルの情報です。 インデントを手がかりに「どのオブジェクトに属する情報か」を追いかけましょう。

4

オンラインツールで整形・可視化する

「JSON Formatter」などの無料ツールをブラウザで検索すると、JSONを色分けして見やすく表示してくれます。 第12章で使うWebhook.siteでも受信したJSONを自動で整形表示してくれます。

便利なツール: JSON Formatter

受け取ったJSONが読みにくい場合は、オンラインの整形ツールが役立ちます。 jsonformatter.curiousconcept.com などのサイトにJSONをペーストすると、色分けされた読みやすい形式に変換してくれます。

第12章「Webhookを体験しよう」では実際にWebhook.siteを使ってペイロードを受信・確認します。

理解度確認クイズ

  1. Q1. JSONのキーはどのように書く必要がありますか？ (A) シングルクォート（'）で囲む (B) ダブルクォート（"）で囲む (C) クォートは不要

    回答を表示

    答えは (B) です。JSONのキーは必ずダブルクォート（"）で囲む必要があります。 シングルクォートはJSONの仕様では認められておらず、クォートなしもエラーになります。

  2. Q2. 次のJSONで "commits" の値の型は何ですか？ {"commits": ["abc123", "def456"]}

    回答を表示

    答えは「配列（array）」です。角かっこ [ ] で囲まれているので配列です。 この配列の中には文字列型の値が2つ入っています。

  3. Q3. Webhookのペイロードを受け取ったとき、まず何のキーを確認するのが効率的ですか？

    回答を表示

    "event" や "type" キーを確認するのが効率的です。ここには「何が起きたか」が書かれているため、 「payment.succeeded」「push」「message」などのイベント種別を把握してから、 そのイベントに関連するフィールドを探す手順が効率的です。
