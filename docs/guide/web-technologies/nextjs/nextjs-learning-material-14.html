デプロイメントとCI/CD

Vercelへのデプロイ、環境変数の管理、GitHub Actionsによる自動化、Docker化、エッジ関数、監視とアラート、ロールバック戦略など、本番運用に必要な知識を総合的に学びます。

この章で学ぶこと

  • Vercelへのデプロイ手順と本番環境の最適化設定
  • 環境変数の管理とセキュリティベストプラクティス
  • GitHub ActionsによるCI/CDパイプラインの構築
  • Docker化とコンテナデプロイの実装方法
  • 監視・アラート設定とパフォーマンス分析

14.1 デプロイメントの選択肢

Next.jsアプリケーションは、さまざまなプラットフォームにデプロイできます。それぞれの特徴を理解し、プロジェクトに最適な選択をしましょう。

この章での学習方針

初めての方にはVercelを推奨します。Next.jsの公式プラットフォームとして、最も簡単で強力な機能を提供します。その後、GitHub ActionsによるCI/CDとDocker化についても学習します。

14.2 Vercelへのデプロイ

Vercelは、Next.jsの開発元であるVercel社が提供するホスティングプラットフォームです。GitHubとの連携により、コミットするだけで自動デプロイされます。

実習 14-1: Vercelにデプロイする

Next.jsプロジェクトをVercelにデプロイして、本番公開してみましょう。

手順

  1. 1 Vercel公式サイト（vercel.com）でアカウント作成（GitHub連携推奨）
  2. 2 プロジェクトをGitHubにプッシュ
  3. 3 Vercelダッシュボードで「New Project」→ GitHubリポジトリを選択
  4. 4 「Deploy」ボタンをクリック（設定は自動検出されます）

デプロイ後の確認

  • ✅ 本番URL（例: https://your-project.vercel.app）が発行される
  • ✅ 各コミットごとにプレビューURLが自動生成される
  • ✅ ビルドログで問題がないか確認
  • ✅ カスタムドメインの設定も可能

Vercelの自動最適化機能

  • エッジネットワーク：世界中のCDNエッジから高速配信
  • 自動HTTPSレスポンス：SSL証明書の自動発行と更新
  • プレビューデプロイ：プルリクエストごとに専用URLを生成
  • Analytics：リアルタイムのアクセス分析とCore Web Vitals計測

14.3 環境変数の管理

本番環境では、APIキーやデータベース接続情報などの機密情報を安全に管理する必要があります。環境変数の適切な扱い方を学びましょう。

ローカル環境（.env.local） 

# .env.local（ローカル開発用）
DATABASE_URL=postgres://localhost:5432/dev
NEXT_PUBLIC_API_URL=http://localhost:3000
SECRET_KEY=dev-secret-key-123

ローカル開発では.env.localに記述。Gitにはコミットしません。

本番環境（Vercelダッシュボード）

Vercelでの設定手順：

  1. プロジェクト → Settings → Environment Variables
  2. 変数名・値を入力し、環境を選択（Production/Preview/Development）
  3. 「Save」で保存

本番環境の機密情報はダッシュボードで管理し、コードには含めません。

環境変数のプレフィックス規則

NEXT_PUBLIC_*

クライアントサイドで使用可能：ブラウザにも公開される環境変数。APIのエンドポイントURL等、公開情報のみに使用します。 

// ブラウザでもアクセス可能
const apiUrl = process.env.NEXT_PUBLIC_API_URL;
その他（プレフィックスなし）

サーバーサイド専用：APIキーやデータベース認証情報など、機密情報に使用します。クライアントからはアクセスできません。 

// サーバーコンポーネントまたはAPI Routesでのみアクセス可能
const dbUrl = process.env.DATABASE_URL;

セキュリティ上の注意

  • ❌ .env.localをGitにコミットしない
  • ❌ APIキーやシークレットをNEXT_PUBLIC_*で公開しない
  • ✅ .env.exampleで変数名のテンプレートを共有
  • ✅ 本番とプレビュー環境で異なる値を使用

14.4 GitHub ActionsによるCI/CD

GitHub Actionsを使用すると、コミット時の自動テスト、リント、ビルド確認など、CI/CDパイプラインを構築できます。

flowchart LR A[コミット & プッシュ] --> B[GitHub Actions起動] B --> C{CI/CDパイプライン} C --> D[Lint実行] C --> E[テスト実行] C --> F[ビルド確認] D --> G{全て成功?} E --> G F --> G G -->|Yes| H[自動デプロイ] G -->|No| I[失敗通知]

実習 14-2: GitHub Actions CI/CDを設定する

プルリクエスト時に自動でテストとビルドを実行するワークフローを作成します。

ワークフローファイルの作成

.github/workflows/ci.yml 
name: CI

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '18'
      - run: npm ci        # 依存関係インストール
      - run: npm run lint  # Lint実行
      - run: npm run build # ビルド確認

動作確認

このファイルをコミットしてプッシュすると、GitHub Actionsが自動実行されます。リポジトリの「Actions」タブでログを確認できます。

CI/CDの活用ポイント

  • テスト自動化：全てのプルリクエストでテストを実行し、品質を保証
  • ビルド確認：本番ビルドが成功するか事前に確認
  • 自動デプロイ：mainブランチへのマージで自動デプロイ
  • 通知連携：Slack/Discordへの結果通知で素早い対応

14.5 Docker化とコンテナデプロイ

Next.jsアプリケーションをDockerコンテナ化することで、どこでも同じ環境で動作するポータブルなアプリケーションを作成できます。

Next.js用Dockerfile

Dockerfile 
# ベースイメージ
FROM node:18-alpine AS base

# 依存関係インストール
FROM base AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci

# ビルドステージ
FROM base AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN npm run build

# 本番実行ステージ
FROM base AS runner
WORKDIR /app
ENV NODE_ENV production
COPY --from=builder /app/public ./public
COPY --from=builder /app/.next ./.next
COPY --from=builder /app/node_modules ./node_modules
COPY package.json ./
EXPOSE 3000
CMD ["npm", "start"]

Dockerコマンド

terminal 
# イメージをビルド
docker build -t my-nextjs-app .

# コンテナを起動
docker run -p 3000:3000 my-nextjs-app

# docker-composeで起動
docker-compose up

Docker化のメリット

  • 環境の一貫性：開発、ステージング、本番で同じ環境を保証
  • 移植性：AWS、GCP、Azure、オンプレミスなどどこでも動作
  • スケーラビリティ：Kubernetesと組み合わせて自動スケーリング
  • CI/CD統合：Docker Registryを使った自動デプロイ

14.6 監視とパフォーマンス分析

本番環境では、アプリケーションの稼働状況を継続的に監視し、問題を早期に発見することが重要です。

Real User Monitoring] D --> D1[エラー検知

Vercel Analytics

  • ✅ Core Web Vitals：LCP、FID、CLSの計測
  • ✅ リアルユーザー監視：実際のユーザー体験を計測
  • ✅ ページパフォーマンス：各ページの読み込み速度
  • ✅ 地域別分析：地域ごとのパフォーマンス比較

Sentry（エラートラッキング）

  • ✅ エラー検知：リアルタイムでエラーを検知
  • ✅ スタックトレース：詳細なエラー情報を記録
  • ✅ 通知連携：Slack/メールで即座に通知
  • ✅ リリース追跡：デプロイ後のエラー発生を追跡

実習 14-3: Sentryでエラートラッキングを設定する

Sentryを統合して、本番環境のエラーを自動で収集・通知します。

セットアップ手順

terminal 
# Sentry SDKをインストール
npx @sentry/wizard@latest -i nextjs

このコマンドを実行すると、対話形式でSentryの設定が完了し、sentry.client.config.jsなどのファイルが自動生成されます。

環境変数の設定

# .env.local
NEXT_PUBLIC_SENTRY_DSN=your-dsn-here

監視のベストプラクティス

  • アラート設定：エラー率が閾値を超えたら即座に通知
  • 定期レビュー：週次でパフォーマンスレポートを確認
  • ユーザー体験重視：Core Web Vitalsを継続的に改善
  • ロールバック準備：問題発生時の即座のロールバック体制

理解度確認クイズ

  1. Q1. Vercelでデプロイする際、プルリクエストごとに自動生成される機能の名前は何ですか？

    回答を表示

    プレビューデプロイメントです。各プルリクエストに専用のURLが自動生成され、本番環境に影響を与えずに変更内容を確認できます。

  2. Q2. Next.jsの環境変数で、クライアントサイドからもアクセス可能にするプレフィックスは何ですか？

    回答を表示

    NEXT_PUBLIC_プレフィックスを付けた環境変数は、ブラウザからもアクセス可能になります。ただし、APIキーなどの機密情報には使用しないでください。

  3. Q3. GitHub Actionsを使ったCI/CDの主なメリットを2つ挙げてください。

    回答を表示

    ①コミット時に自動でテスト・リント・ビルドを実行し、品質を保証できる。②本番デプロイ前に問題を検出できるため、バグの早期発見と修正が可能になる。

トラブルシューティング

Vercelでビルドが失敗する

原因：環境変数の未設定、依存関係のバージョン不一致、TypeScriptエラーなど。

解決策：Vercelのビルドログを確認し、エラーメッセージを特定します。環境変数が必要な場合はVercelダッシュボードで設定してください。ローカルでnpm run buildを実行し、同じエラーが再現するか確認します。

環境変数が反映されない

原因：環境変数の設定後に再デプロイしていない、または誤った環境（Production/Preview/Development）に設定している。

解決策：Vercelダッシュボードで正しい環境に変数が設定されているか確認し、変更後は必ず再デプロイ（Deployments → Redeploy）を実行してください。

Dockerコンテナが起動しない

原因：Dockerfileの設定ミス、ポートの競合、メモリ不足など。

解決策：docker logs [container-id]でログを確認します。ポート3000が既に使用されている場合はdocker run -p 3001:3000のように別ポートを指定してください。

この章のまとめ

  • Vercelは公式プラットフォームとして、最も簡単で強力なデプロイメント環境を提供する
  • 環境変数は機密情報を安全に管理するために不可欠で、NEXT_PUBLIC_プレフィックスの使い分けが重要
  • GitHub ActionsでCI/CDパイプラインを構築すると、品質保証と自動デプロイが実現できる
  • Docker化により、環境の一貫性とポータビリティが向上し、複雑なインフラ構成にも対応可能
  • SentryやVercel Analyticsによる監視で、本番環境の問題を早期発見・対応できる
