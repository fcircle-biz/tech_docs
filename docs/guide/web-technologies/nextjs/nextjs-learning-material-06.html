第6章

静的生成（SSG）と増分静的再生成（ISR）

ビルド時にHTMLを事前生成するSSGと、定期的に再生成するISRの仕組みを学び、高速でスケーラブルなWebサイトを構築する方法を習得します。

この章で学ぶこと

  • 静的生成（SSG）の仕組みとメリット、適用場面
  • generateStaticParamsによる動的ルートの事前生成
  • 増分静的再生成（ISR）による定期的なコンテンツ更新
  • On-Demand RevalidationとCDN連携によるパフォーマンス最適化

6.1 静的生成（SSG）とは

静的生成（Static Site Generation: SSG）は、ビルド時に全てのページをHTMLとして事前生成する手法です。生成されたHTMLはCDNから直接配信されるため、極めて高速なレスポンスを実現できます。

flowchart LR A[ビルド時] --> B[全ページを
HTML生成] B --> C[静的ファイル
として保存] C --> D[CDNに
デプロイ] D --> E[ユーザーリクエスト] E --> F[CDNから
高速配信]

SSGのメリット

  • 最速のパフォーマンス：事前生成されたHTMLをCDNから直接配信するため、サーバー処理が不要
  • 優れたSEO：完全なHTMLがクローラーに即座に提供される
  • スケーラビリティ：静的ファイルなので大量トラフィックにも耐えられる
  • 低コスト：サーバー処理が不要なため、インフラコストを削減
  • 高セキュリティ：動的処理が少ないため攻撃面が小さい

SSGが適しているサイト

ブログ、ドキュメントサイト、企業サイト、ランディングページなど、コンテンツの更新頻度が低く、全ユーザーに同じ内容を表示するサイトに最適です。

6.2 App RouterでのSSG実装

App Routerでは、デフォルトでSSGが適用されます。動的関数（cookies、headersなど）を使わない限り、自動的にビルド時に生成されます。

実習 6-1: 基本的なSSGページの作成

シンプルなブログ記事一覧ページを静的生成します。

app/blog/page.tsx

TypeScript 
// ブログ記事一覧ページ（SSG自動適用）
async function getPosts() {
  const res = await fetch('https://api.example.com/posts');
  return res.json();
}

export default async function BlogPage() {
  const posts = await getPosts();

  return (
    <div>
      <h1>ブログ記事一覧</h1>
      {posts.map((post: any) => (
        <article key={post.id}>
          <h2>{post.title}</h2>
        </article>
      ))}
    </div>
  );
}

このページはビルド時に一度だけfetch()が実行され、HTMLが生成されます。

6.3 generateStaticParamsによる動的ルート生成

generateStaticParamsを使うと、動的ルート（例：/blog/[id]）のページを事前に全て生成できます。

flowchart TD A[generateStaticParams実行] --> B[生成するパスのリスト
を返す] B --> C{各パスについて} C --> D[ページコンポーネント
を実行] D --> E[HTML生成] E --> F[静的ファイル
として保存] C --> D

実習 6-2: 動的ルートのSSG

全ての記事詳細ページを事前生成します。

app/blog/[id]/page.tsx

TypeScript 
// 動的ルートのパラメータを事前生成
export async function generateStaticParams() {
  const posts = await fetch('https://api.example.com/posts')
    .then(res => res.json());

  // 全記事IDのリストを返す
  return posts.map((post: any) => ({
    id: post.id.toString(),
  }));
}

// 記事詳細ページ
export default async function PostPage({
  params,
}: {
  params: { id: string };
}) {
  const post = await fetch(
    `https://api.example.com/posts/${params.id}`
  ).then(res => res.json());

  return <article><h1>{post.title}</h1></article>;
}

ビルド時に全ての記事IDについてHTMLが生成されます。

注意点

記事数が膨大な場合（数万件以上）、全てを事前生成するとビルド時間が非常に長くなります。このような場合はISR（次セクション）やSSRの使用を検討してください。

6.4 増分静的再生成（ISR）

増分静的再生成（Incremental Static Regeneration: ISR）は、SSGの利点を保ちつつ、定期的にページを再生成する仕組みです。全体を再ビルドせずに、個別ページだけを更新できます。

sequenceDiagram participant U as ユーザー participant C as CDN participant S as Next.jsサーバー U->>C: ページリクエスト C->>U: キャッシュ済みHTML配信 Note over C,S: revalidate時間経過後 U->>C: 次のリクエスト C->>S: 再生成をトリガー S->>C: 新しいHTML返却 C->>U: 古いHTML配信（stale） Note over C: バックグラウンドで更新 U->>C: さらに次のリクエスト C->>U: 新しいHTML配信

ISRの動作原理（Stale-While-Revalidate）

  1. 1回目のリクエスト：キャッシュされたHTMLを即座に返す（古い可能性あり）
  2. バックグラウンド処理：revalidate期間経過後、サーバーで新しいHTMLを生成
  3. 2回目以降のリクエスト：更新されたHTMLが配信される

実習 6-3: ISRの実装

60秒ごとに記事一覧を自動更新するページを作成します。

app/blog/page.tsx（ISR対応）

TypeScript 
// 60秒ごとに再生成
async function getPosts() {
  const res = await fetch(
    'https://api.example.com/posts',
    {
      next: { revalidate: 60 } // 60秒でキャッシュ再検証
    }
  );
  return res.json();
}

export default async function BlogPage() {
  const posts = await getPosts();

  return (
    <div>
      <h1>ブログ記事一覧</h1>
      <p>更新日時: {new Date().toLocaleString()}</p>
      {posts.map((post: any) => (
        <article key={post.id}>
          <h2>{post.title}</h2>
        </article>
      ))}
    </div>
  );
}

revalidate: 60により、60秒ごとにバックグラウンドで再生成されます。

6.5 On-Demand Revalidation

On-Demand Revalidationを使うと、時間ベースではなく、イベント駆動で特定のページだけを即座に再生成できます。例えば、記事を編集した直後に該当ページだけを更新できます。

On-Demand Revalidationの使い所

  • CMSで記事を公開・編集したとき
  • 商品情報を更新したとき
  • Webhookを受け取ったとき
  • 管理画面から手動で再生成したいとき

実習 6-4: On-Demand Revalidationの実装

APIエンドポイントを作成し、特定のパスを即座に再生成します。

app/api/revalidate/route.ts

TypeScript 
import { revalidatePath } from 'next/cache';
import { NextRequest, NextResponse } from 'next/server';

export async function POST(request: NextRequest) {
  const secret = request.nextUrl.searchParams.get('secret');

  // セキュリティ: シークレットトークンで認証
  if (secret !== process.env.REVALIDATE_SECRET) {
    return NextResponse.json({ message: '無効なトークン' }, { status: 401 });
  }

  const path = request.nextUrl.searchParams.get('path');

  // 特定のパスを再生成
  revalidatePath(path || '/blog');

  return NextResponse.json({ revalidated: true, now: Date.now() });
}

呼び出し例

bash 
# 特定のパスを再生成
curl -X POST \
  "https://your-site.com/api/revalidate?secret=YOUR_SECRET&path=/blog/123"

セキュリティ: 環境変数でシークレットトークンを設定し、不正なリクエストを防ぎましょう。

6.6 fallback戦略とdynamicParams

動的ルートでgenerateStaticParamsを使う場合、全てのパスを事前生成できないこともあります。そのような場合の挙動をdynamicParamsで制御できます。

dynamicParamsの設定

設定値 動作 使い所
true（デフォルト） 未生成パスをオンデマンドで生成 記事が頻繁に追加されるブログ
false 未生成パスは404を返す 固定された商品カタログ
TypeScript 
// app/blog/[id]/page.tsx

// 未生成パスの挙動を制御
export const dynamicParams = true; // デフォルト

export async function generateStaticParams() {
  // 人気記事の上位100件だけ事前生成
  const posts = await fetch('https://api.example.com/posts?limit=100')
    .then(res => res.json());

  return posts.map((post: any) => ({ id: post.id.toString() }));
}

ベストプラクティス

記事数が多い場合は、人気記事や最新記事だけを事前生成し、その他はオンデマンド生成（dynamicParams: true）にすることでビルド時間を短縮できます。

6.7 CDN連携とパフォーマンス最適化

SSG/ISRで生成された静的ファイルは、CDN（Content Delivery Network）から配信することで、世界中のユーザーに高速にコンテンツを届けられます。

CDNのメリット

  • 低レイテンシ：ユーザーに最も近いエッジサーバーから配信
  • 高可用性：複数のエッジサーバーで冗長化
  • DDoS対策：大量リクエストを分散処理
  • 帯域削減：オリジンサーバーの負荷を軽減

SSG/ISRのパフォーマンス最適化チェックリスト

  • 変更頻度が低いページはSSGで事前生成
  • 定期的に更新されるコンテンツはISRを使用（適切なrevalidate値を設定）
  • 動的ルートは人気コンテンツだけ事前生成し、残りはオンデマンド生成
  • 画像はnext/imageで自動最適化
  • 重要でないリソースはdynamic importで遅延ロード
  • Vercel等のCDN対応プラットフォームでデプロイ

動作確認: SSG/ISRの確認方法

  1. 1 npm run buildでビルドし、出力されるログを確認
  2. 2 ○印（SSG）または λ印（動的）でレンダリング方法が表示される
  3. 3 npm startで本番モードで起動し、ページの更新タイミングを確認
  4. 4 ブラウザの開発者ツールでレスポンスヘッダーを確認（x-vercel-cache: HITなど）

理解度確認クイズ

  1. Q1. SSGとISRの違いは何ですか？

    回答を表示

    SSGはビルド時に一度だけHTMLを生成し、コンテンツ更新には再ビルドが必要です。一方、ISRはビルド時の生成に加えて、指定した間隔（revalidate）でバックグラウンドで再生成し、再ビルドなしで個別ページを更新できます。

  2. Q2. generateStaticParamsは何のために使いますか？

    回答を表示

    動的ルート（例：/blog/[id]）のページをビルド時に事前生成するために使います。関数は生成するパスのパラメータリストを返し、Next.jsが各パスについてHTMLを生成します。

  3. Q3. On-Demand Revalidationはどのような場面で有用ですか？

    回答を表示

    CMSで記事を編集した直後、商品情報を更新した直後など、特定のイベントが発生したタイミングで即座に該当ページだけを再生成したい場合に有用です。時間ベースのISRと異なり、必要なときだけ更新できます。

この章のまとめ

  • 静的生成（SSG）は最速のパフォーマンスと優れたSEOを提供し、更新頻度が低いコンテンツに最適
  • generateStaticParamsで動的ルートを事前生成し、全ページを静的配信できる
  • ISRは定期的にコンテンツを更新しつつ、静的配信の高速性を保つ最適なバランス
  • On-Demand Revalidationでイベント駆動の即座な更新が可能になり、柔軟なコンテンツ管理を実現
  • CDN連携により、世界中のユーザーに低レイテンシで静的コンテンツを配信できる
