第4章

データフェッチングとキャッシュ戦略

Next.jsの拡張されたfetch API、サーバーコンポーネントでのデータ取得パターン、キャッシュメカニズム、そしてローディング・ストリーミングUIの実装方法を学びます。

この章で学ぶこと

  • Next.jsが提供するfetch APIの拡張機能とキャッシュオプション
  • サーバーコンポーネントでの並列・順次データフェッチングパターン
  • Data CacheとFull Route Cacheの仕組みとrevalidate戦略
  • 動的関数（cookies、headers）の使用とローディング・ストリーミングUI

4.1 Next.jsのfetch拡張機能

Next.jsは標準のfetch APIを拡張し、強力なキャッシュ制御とデータ再検証の仕組みを提供しています。サーバーコンポーネント内で使用すると、自動的にリクエストがキャッシュされ、パフォーマンスが向上します。

fetch拡張のキャッシュオプション

  • cache: 'force-cache'（デフォルト）：リクエストを永続的にキャッシュ
  • cache: 'no-store'：常に新しいデータを取得（キャッシュなし）
  • next.revalidate：指定秒数後にキャッシュを再検証
  • next.tags：キャッシュをタグ付けして選択的に無効化
app/products/page.tsx 
// デフォルト: 永続キャッシュ
async function getProducts() {
  const res = await fetch('https://api.example.com/products');
  return res.json();
}

// キャッシュなし（常に最新データ）
async function getDynamicData() {
  const res = await fetch('https://api.example.com/live', {
    cache: 'no-store'
  });
  return res.json();
}

// 60秒ごとに再検証
async function getRevalidatedData() {
  const res = await fetch('https://api.example.com/posts', {
    next: { revalidate: 60 }
  });
  return res.json();
}

4.2 サーバーコンポーネントでのデータ取得

App Routerでは、サーバーコンポーネントがデフォルトです。サーバーコンポーネント内で直接データフェッチを行うことで、クライアントサイドでの状態管理が不要になり、コードがシンプルになります。

flowchart LR A[ユーザーリクエスト] --> B[サーバー
コンポーネント] B --> C{データフェッチ} C --> D[API呼び出し] C --> E[DB クエリ] C --> F[ファイル読み込み] D --> G[データをHTMLに埋め込み] E --> G F --> G G --> H[クライアントに
完成したHTMLを送信]
app/blog/page.tsx 
// サーバーコンポーネント内で直接データ取得
export default async function BlogPage() {
  // async関数としてコンポーネントを定義
  const posts = await fetch('https://api.example.com/posts', {
    next: { revalidate: 3600 } // 1時間ごとに再検証
  }).then(res => res.json());

  return (
    <div>
      <h1>ブログ記事一覧</h1>
      <ul>
        {posts.map((post: any) => (
          <li key={post.id}>{post.title}</li>
        ))}
      </ul>
    </div>
  );
}

サーバーコンポーネントでのデータ取得のメリット

  • useEffectやuseStateが不要でコードがシンプル
  • APIキーなどの機密情報をクライアントに公開しない
  • データベースへの直接接続が可能
  • 初回ロード時に完全なHTMLが送られるためSEO効果が高い

4.3 並列・順次データフェッチング

複数のデータを取得する際、並列（Parallel）と順次（Sequential）のどちらで実行するかによってパフォーマンスが大きく変わります。

順次フェッチ（遅い）

// 順番に待機（遅い）
const user = await fetchUser();
const posts = await fetchPosts();
// 合計: 2秒 + 3秒 = 5秒

各リクエストが前のリクエスト完了を待つため、合計時間が長くなります。

並列フェッチ（速い）

// 同時実行（速い）
const [user, posts] = await Promise.all([
  fetchUser(),
  fetchPosts()
]);
// 合計: max(2秒, 3秒) = 3秒

Promise.allを使うことで並列実行され、最も遅いリクエストの時間のみで完了します。

実習 4-1: 並列データフェッチの実装

複数のAPIエンドポイントから並列でデータを取得し、パフォーマンスを最適化します。

実装例

app/dashboard/page.tsx 
async function getUser() {
  const res = await fetch('https://api.example.com/user');
  return res.json();
}

async function getPosts() {
  const res = await fetch('https://api.example.com/posts');
  return res.json();
}

export default async function DashboardPage() {
  // 並列実行で高速化
  const [user, posts] = await Promise.all([
    getUser(),
    getPosts()
  ]);

  return (
    <div>
      <h1>{user.name}のダッシュボード</h1>
      <p>投稿数: {posts.length}</p>
    </div>
  );
}

4.4 キャッシュの仕組み（Data Cache & Full Route Cache）

Next.jsは2層のキャッシュシステムを提供し、パフォーマンスを最大化しています。

flowchart TB A[ユーザーリクエスト] --> B{Full Route Cache} B -->|キャッシュHIT| C[HTMLを即座に返却] B -->|キャッシュMISS| D[サーバーで処理] D --> E{Data Cache} E -->|キャッシュHIT| F[キャッシュデータを使用] E -->|キャッシュMISS| G[API/DBから取得] G --> H[Data Cacheに保存] F --> I[HTMLをレンダリング] H --> I I --> J[Full Route Cacheに保存] J --> K[クライアントに返却]

2つのキャッシュレイヤー

キャッシュ種類 対象 保存場所 無効化方法
Data Cache fetchリクエストの結果 サーバー（永続的） revalidate、revalidateTag
Full Route Cache レンダリング済みHTML サーバー（永続的） revalidatePath、revalidateTag
タグベースキャッシュ無効化 
// データフェッチ時にタグを付与
async function getPosts() {
  const res = await fetch('https://api.example.com/posts', {
    next: { tags: ['posts'] } // タグ付け
  });
  return res.json();
}

// Server Action内でタグを指定してキャッシュを無効化
import { revalidateTag } from 'next/cache';

export async function createPost(data: FormData) {
  // 投稿を作成
  await savePost(data);

  // 'posts'タグのキャッシュを無効化
  revalidateTag('posts');
}

4.5 revalidateによる再検証

revalidateを使うことで、キャッシュの有効期限を設定し、定期的にデータを更新できます。これにより、ISR（Incremental Static Regeneration）のような動作を実現できます。

fetchレベルでのrevalidate 
// 個別のfetchリクエストに対して設定
const res = await fetch('https://api.example.com/data', {
  next: { revalidate: 60 } // 60秒後に再検証
});
ルートセグメントレベルでのrevalidate 
// page.tsxやlayout.tsxでエクスポート
export const revalidate = 3600; // 1時間

export default async function Page() {
  // このページ内のすべてのfetchが1時間キャッシュされる
  const data = await fetch('https://api.example.com/data');
  // ...
}
On-Demand Revalidation（オンデマンド再検証） 
import { revalidatePath, revalidateTag } from 'next/cache';

// 特定のパスを再検証
revalidatePath('/blog/posts');

// 特定のタグを再検証
revalidateTag('posts');

4.6 動的関数（cookies、headers）の使用

cookies()やheaders()などの動的関数を使用すると、そのページは自動的に動的レンダリング（SSR）にオプトインされます。

注意：動的関数を使うとキャッシュが無効化される

動的関数を使用すると、Full Route Cacheが無効化され、リクエストごとにサーバーで処理されます。本当に必要な場合のみ使用してください。

動的関数の使用例 
import { cookies, headers } from 'next/headers';

export default async function UserPage() {
  // クッキーを取得（動的レンダリングにオプトイン）
  const cookieStore = cookies();
  const token = cookieStore.get('auth_token');

  // ヘッダーを取得
  const headersList = headers();
  const userAgent = headersList.get('user-agent');

  // トークンを使ってユーザー情報を取得
  const user = await fetch('https://api.example.com/me', {
    headers: { Authorization: `Bearer ${token?.value}` }
  }).then(res => res.json());

  return (
    <div>
      <h1>{user.name}のプロフィール</h1>
      <p>User Agent: {userAgent}</p>
    </div>
  );
}

4.7 ローディングとストリーミングUI

App Routerでは、loading.tsxとReactのSuspenseを使って、データ読み込み中のローディング状態を簡単に実装できます。

flowchart LR A[ユーザーアクセス] --> B[loading.tsx表示] B --> C[データフェッチ中] C --> D[page.tsx
コンテンツ表示] D --> E[完了]

実習 4-2: loading.tsxとSuspenseでローディングUI実装

ローディング状態を表示し、ユーザー体験を向上させます。

app/blog/loading.tsx

export default function Loading() {
  return (
    <div className="animate-pulse space-y-4">
      <div className="h-8 bg-gray-200 rounded w-1/2"></div>
      <div className="h-4 bg-gray-200 rounded w-3/4"></div>
      <div className="h-4 bg-gray-200 rounded w-5/6"></div>
    </div>
  );
}

app/blog/page.tsx（Suspenseを使用）

import { Suspense } from 'react';

async function Posts() {
  const posts = await fetch('https://api.example.com/posts')
    .then(res => res.json());
  return <ul>{/* 記事リスト */}</ul>;
}

export default function BlogPage() {
  return (
    <div>
      <h1>ブログ</h1>
      <Suspense fallback={<p>読み込み中...</p>}>
        <Posts />
      </Suspense>
    </div>
  );
}

ストリーミングSSRの利点

Suspenseを使うと、ページ全体を待つことなく、準備できた部分から順次クライアントに送信（ストリーミング）できます。これにより、体感速度が向上します。

理解度確認クイズ

  1. Q1. Next.jsのfetch APIで `cache: 'no-store'` を指定するとどうなりますか？

    回答を表示

    リクエストがキャッシュされず、常に最新のデータを取得します。これは動的なデータ（ユーザー固有の情報や頻繁に更新されるコンテンツ）に適しています。

  2. Q2. 並列データフェッチングを実現するために使用するJavaScriptの機能は何ですか？

    回答を表示

    `Promise.all()`を使用します。複数の非同期処理を配列で渡すことで、並列実行され、すべてが完了するまで待機します。これにより、順次実行よりも大幅に高速化できます。

  3. Q3. Data CacheとFull Route Cacheの違いを説明してください。

    回答を表示

    Data Cacheは個別のfetchリクエストの結果をキャッシュし、Full Route Cacheはレンダリング済みのHTML全体をキャッシュします。Data Cacheはデータレベル、Full Route Cacheはページレベルのキャッシュと考えることができます。
