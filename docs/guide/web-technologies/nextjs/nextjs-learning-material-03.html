サーバーコンポーネントとクライアントコンポーネント

React Server Components（RSC）の仕組み、サーバーとクライアントコンポーネントの使い分け、コンポーネントの境界設計、そしてパフォーマンス最適化の実践方法を学びます。

この章で学ぶこと

  • React Server Components（RSC）の仕組みと従来のReactとの違い
  • 'use client'ディレクティブによるコンポーネント境界の定義
  • サーバーとクライアントコンポーネントのデータ受け渡しパターン
  • ストリーミングとSuspenseによるパフォーマンス最適化

3.1 React Server Componentsとは？

React Server Components（RSC）は、Reactの新しいパラダイムで、コンポーネントをサーバーでのみ実行することを可能にします。これにより、データベースへの直接アクセスや、クライアントへのJavaScriptバンドルサイズの削減が実現します。

従来のReactとの違い

  • 従来のReact：全てのコンポーネントがクライアントで実行され、JavaScriptバンドルに含まれる
  • RSC：サーバーコンポーネントはサーバーでのみ実行され、HTMLとして配信される
  • メリット：バンドルサイズの削減、初回ロードの高速化、サーバー専用リソースへの直接アクセス

3.2 サーバーコンポーネントとクライアントコンポーネントの使い分け

Next.jsのApp Routerでは、デフォルトで全てのコンポーネントがサーバーコンポーネントです。クライアントコンポーネントが必要な場合のみ、ファイルの先頭に'use client'を記述します。

サーバーコンポーネント

  • データベースへの直接クエリ
  • 機密情報（APIキー等）の安全な使用
  • 外部APIからのデータ取得
  • 大きなライブラリの使用（バンドルサイズに影響しない）
  • async/awaitによる非同期処理

クライアントコンポーネント

  • インタラクティブな機能（クリック、入力等）
  • useState、useEffectなどのReact Hooks
  • ブラウザAPI（localStorage、geolocation等）
  • カスタムHooksの使用
  • リアルタイム更新（WebSocket等）

3.3 'use client'ディレクティブの理解

'use client'ディレクティブは、コンポーネントがクライアントで実行されることを宣言します。このディレクティブはファイルの最上部に記述する必要があります。

実習 3-1: サーバーとクライアントコンポーネントを作成する

シンプルなカウンターコンポーネントを例に、サーバーとクライアントコンポーネントの違いを体験します。

サーバーコンポーネント（デフォルト）

app/page.tsx 
// サーバーコンポーネント（デフォルト）
// 'use client'がないので、サーバーでのみ実行される
async function ServerComponent() {
  // サーバーでのみ実行可能な処理
  const data = await fetch('https://api.example.com/data');
  const posts = await data.json();

  return (
    <div>
      <h1>投稿一覧</h1>
      <ul>
        {posts.map(post => (
          <li key={post.id}>{post.title}</li>
        ))}
      </ul>
    </div>
  );
}

クライアントコンポーネント

app/components/Counter.tsx 
'use client'; // クライアントコンポーネントの宣言

import { useState } from 'react';

export default function Counter() {
  // React Hooksはクライアントコンポーネントでのみ使用可能
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>カウント: {count}</p>
      <button onClick={() => setCount(count + 1)}>
        増やす
      </button>
    </div>
  );
}

重要なルール

  • 'use client'はファイルの最上部（importより前）に記述
  • サーバーコンポーネントはクライアントコンポーネントをインポート可能
  • クライアントコンポーネントはサーバーコンポーネントを直接インポートできない（propsで渡すことは可能）

3.4 コンポーネントの境界設計

効率的なアプリケーションを構築するには、クライアントコンポーネントを必要最小限に保つことが重要です。インタラクティブな部分だけをクライアントコンポーネントに切り出し、残りはサーバーコンポーネントにします。

実習 3-2: コンポーネント境界を最適化する

ブログ投稿ページを例に、適切なコンポーネント境界を設計します。

悪い例：全体をクライアントコンポーネントにする

❌ app/post/[id]/page.tsx 
'use client'; // ❌ 全体がクライアントになる

import { useState, useEffect } from 'react';

export default function PostPage({ params }) {
  const [post, setPost] = useState(null);

  // クライアントでデータ取得（非効率）
  useEffect(() => {
    fetch(`/api/posts/${params.id}`)
      .then(res => res.json())
      .then(setPost);
  }, [params.id]);

  return <div>{post?.title}</div>;
}

良い例：インタラクティブ部分のみクライアントコンポーネントに

✅ app/post/[id]/page.tsx 
// サーバーコンポーネント（デフォルト）
import LikeButton from './LikeButton'; // クライアントコンポーネント

async function getPost(id: string) {
  const res = await fetch(`https://api.example.com/posts/${id}`);
  return res.json();
}

export default async function PostPage({ params }) {
  // サーバーでデータ取得
  const post = await getPost(params.id);

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
      {/* インタラクティブ部分のみクライアントコンポーネント */}
      <LikeButton postId={post.id} initialLikes={post.likes} />
    </article>
  );
}

3.5 データの受け渡しパターン

サーバーとクライアントコンポーネント間でデータを受け渡す際には、いくつかのルールとパターンがあります。

データ受け渡しのルール

✅ 可能な受け渡し

  • サーバー → クライアント：propsでシリアライズ可能なデータを渡す
  • クライアント → サーバー：Server Actionsを通じてデータを送信

❌ 不可能な受け渡し

  • クライアント → サーバー：サーバーコンポーネントを直接インポートできない
  • 関数、Dateオブジェクト、undefinedなどシリアライズできないデータの直接受け渡し

実習 3-3: サーバーコンポーネントをクライアントコンポーネントに渡す

childrenパターンを使って、サーバーコンポーネントをクライアントコンポーネント内で使用します。

app/components/ClientWrapper.tsx 
'use client';

import { ReactNode } from 'react';

export default function ClientWrapper({ children }: { children: ReactNode }) {
  // クライアントコンポーネントだが、
  // childrenとしてサーバーコンポーネントを受け取れる
  return (
    <div className="client-wrapper">
      {children}
    </div>
  );
}

// 使用例（親コンポーネント）
// <ClientWrapper>
//   <ServerComponent /> {/* サーバーコンポーネント */}
// </ClientWrapper>

3.6 ストリーミングとSuspenseによるパフォーマンス最適化

Next.jsのApp RouterはストリーミングSSRに対応しており、ReactのSuspenseを使うことで、ページの一部を段階的にレンダリングできます。

ストリーミングSSRの仕組み

sequenceDiagram autonumber participant Browser as ブラウザ participant Server as Next.jsサーバー participant DB as データベース Browser->>Server: ページリクエスト Server->>Browser: 即座にヘッダー+スケルトンUIを送信 Server->>DB: データ取得開始 DB-->>Server: データ返却 Server->>Browser: コンテンツをストリーミング送信 Browser->>Browser: 段階的にレンダリング

実習 3-4: Suspenseで段階的ローディングを実装する

遅いデータ取得を持つコンポーネントをSuspenseでラップし、ローディング体験を改善します。

app/page.tsx 
import { Suspense } from 'react';

// 遅いデータ取得を行うコンポーネント
async function SlowComponent() {
  // 3秒かかるデータ取得をシミュレート
  await new Promise(resolve => setTimeout(resolve, 3000));
  return <div>遅いコンテンツが読み込まれました</div>;
}

export default function Page() {
  return (
    <div>
      <h1>即座に表示されるヘッダー</h1>

      {/* Suspenseで囲むと、fallbackが即座に表示され、
          コンポーネントの準備ができ次第置き換わる */}
      <Suspense fallback={<div>読み込み中...</div>}>
        <SlowComponent />
      </Suspense>

      <footer>フッターも即座に表示</footer>
    </div>
  );
}

Suspenseのメリット

  • 遅いデータ取得がページ全体のレンダリングをブロックしない
  • ユーザーは重要なコンテンツを即座に見ることができる
  • 並列データ取得により、全体の待ち時間が短縮される

理解度確認クイズ

  1. Q1. サーバーコンポーネントとクライアントコンポーネントの主な違いは何ですか？

    回答を表示

    サーバーコンポーネントはサーバーでのみ実行され、データベースへの直接アクセスや機密情報の使用が可能です。クライアントコンポーネントはブラウザで実行され、useState、useEffectなどのReact Hooksやインタラクティブな機能が使用できます。デフォルトではサーバーコンポーネントで、'use client'を記述するとクライアントコンポーネントになります。

  2. Q2. いつクライアントコンポーネントを使用すべきですか？

    回答を表示

    クリックやフォーム入力などのユーザーインタラクションが必要な場合、useState/useEffectなどのReact Hooksを使用する場合、ブラウザAPI（localStorage、geolocation等）を使用する場合にクライアントコンポーネントを使用します。それ以外はサーバーコンポーネントを使用することで、バンドルサイズを削減できます。

  3. Q3. Suspenseを使用するメリットは何ですか？

    回答を表示

    Suspenseを使用すると、遅いデータ取得を持つコンポーネントが準備できるまでの間、fallbackコンテンツ（ローディングUI）を表示できます。これによりページ全体のレンダリングがブロックされず、重要なコンテンツを即座に表示できるため、ユーザー体験が向上します。また、複数のコンポーネントが並列にデータを取得できるため、全体の待ち時間が短縮されます。

この章のまとめ

  • React Server Componentsはサーバーでのみ実行され、バンドルサイズの削減と初回ロード高速化を実現する
  • App Routerではデフォルトでサーバーコンポーネントが使用され、'use client'で明示的にクライアントコンポーネントを宣言する
  • インタラクティブな部分のみをクライアントコンポーネントに切り出すことで、パフォーマンスを最適化できる
  • SuspenseとストリーミングSSRを活用することで、段階的なページレンダリングとローディング体験の向上が可能
