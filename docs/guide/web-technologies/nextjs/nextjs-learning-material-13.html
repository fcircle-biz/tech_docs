第13章

データベース統合とORM

Prismaを使用したデータベース統合、スキーマ設計、マイグレーション管理、CRUD操作の実装、そしてNext.jsとのシームレスな連携方法を学びます。

この章で学ぶこと

  • Prismaのセットアップと基本的な使い方
  • データベーススキーマの設計とマイグレーション管理
  • サーバーコンポーネントとAPI RouteでのCRUD操作
  • リレーション、トランザクション、接続プールの実践的な活用

13.1 なぜORMが必要なのか

ORM（Object-Relational Mapping）は、データベースのテーブルとアプリケーションのオブジェクトをマッピングし、SQLを直接書かずにデータ操作を行える仕組みです。Next.jsアプリケーションでは、Prismaが最も人気のあるORMです。

生SQL直接実行の課題

  • SQLインジェクションのリスク
  • 型安全性がなく、実行時エラーが発生しやすい
  • スキーマ変更時の追跡が困難
  • データベース依存のコードになりやすい

Prisma ORMのメリット

  • TypeScript完全対応で型安全なクエリ
  • マイグレーション管理で変更履歴を追跡
  • 自動補完とIntelliSenseのサポート
  • 複数データベースに対応（PostgreSQL、MySQL、SQLite等）

13.2 Prismaのセットアップ

まずはNext.jsプロジェクトにPrismaをインストールし、基本設定を行います。

実習 13-1: Prismaのインストールと初期化

Next.jsプロジェクトにPrismaをセットアップし、データベース接続を確立します。

手順

  1. 1 Prismaをインストールして初期化する
  2. 2 環境変数ファイル（.env）にデータベース接続URLを設定
  3. 3 Prisma Clientのシングルトンインスタンスを作成

実行例

terminal 
# Prismaをインストール
npm install prisma @prisma/client

# Prismaを初期化（prismaフォルダとschema.prismaが生成される）
npx prisma init
.env 
# PostgreSQLの例（Vercel Postgresやローカル環境）
DATABASE_URL="postgresql://user:password@localhost:5432/mydb?schema=public"

# SQLiteの例（開発環境での簡易利用）
# DATABASE_URL="file:./dev.db"
lib/prisma.ts 
// Prisma Clientのシングルトンインスタンス
import { PrismaClient } from '@prisma/client'

const globalForPrisma = global as unknown as { prisma: PrismaClient }

export const prisma = globalForPrisma.prisma || new PrismaClient()

// 開発環境ではホットリロード時に新しいインスタンスが作られないようにする
if (process.env.NODE_ENV !== 'production') globalForPrisma.prisma = prisma

シングルトンパターンの重要性

Next.jsの開発環境ではホットリロードが頻繁に発生するため、Prisma Clientを毎回新規作成するとコネクションが枯渇します。上記のシングルトンパターンでインスタンスを再利用しましょう。

13.3 スキーマ設計とモデル定義

Prismaではschema.prismaファイルでデータベーススキーマを定義します。このファイルから型安全なPrisma Clientが自動生成されます。

prisma/schema.prisma 
// データソース設定
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

// Prisma Client生成設定
generator client {
  provider = "prisma-client-js"
}

// Userモデル（ユーザー情報）
model User {
  id        String   @id @default(cuid())
  email     String   @unique
  name      String?
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  posts     Post[]   // リレーション: 1対多
}

// Postモデル（ブログ記事）
model Post {
  id        String   @id @default(cuid())
  title     String
  content   String?
  published Boolean  @default(false)
  authorId  String
  author    User     @relation(fields: [authorId], references: [id])
  createdAt DateTime @default(now())
}

主要なフィールド属性

属性 説明
@id 主キー（Primary Key）
@unique 一意制約（例: メールアドレス）
@default() デフォルト値を設定
@updatedAt 更新時に自動で現在時刻を設定
@relation リレーションの外部キーを定義
erDiagram USER ||--o{ POST : writes USER { string id PK string email string name datetime createdAt datetime updatedAt } POST { string id PK string title string content boolean published string authorId FK datetime createdAt }

13.4 マイグレーション管理

スキーマの変更履歴を管理し、データベースを安全に更新するためにマイグレーション機能を使います。

terminal 
# マイグレーションを作成（スキーマ変更を検出）
npx prisma migrate dev --name init

# Prisma Clientを再生成（型定義を更新）
npx prisma generate

# 本番環境でマイグレーション適用
npx prisma migrate deploy

重要ポイント

prisma migrate devは開発環境専用です。本番環境ではprisma migrate deployを使い、CI/CDパイプラインで自動実行するのがベストプラクティスです。

13.5 CRUD操作の実装

Prisma Clientを使って、サーバーコンポーネントとAPI RouteでCRUD操作を実装します。

app/users/page.tsx（サーバーコンポーネント） 
import { prisma } from '@/lib/prisma'

// サーバーコンポーネントでデータ取得
export default async function UsersPage() {
  // 全ユーザーを取得（関連する投稿も含む）
  const users = await prisma.user.findMany({
    include: {
      posts: true, // リレーションを含める
    },
  })

  return (
    <div>
      {users.map(user => (
        <div key={user.id}>
          <h2>{user.name}</h2>
          <p>{user.email}</p>
          <p>投稿数: {user.posts.length}</p>
        </div>
      ))}
    </div>
  )
}
app/api/users/route.ts（API Route） 
import { prisma } from '@/lib/prisma'
import { NextResponse } from 'next/server'

// GET: ユーザー一覧取得
export async function GET() {
  const users = await prisma.user.findMany()
  return NextResponse.json(users)
}

// POST: ユーザー作成
export async function POST(request: Request) {
  const body = await request.json()
  const user = await prisma.user.create({
    data: {
      email: body.email,
      name: body.name,
    },
  })
  return NextResponse.json(user, { status: 201 })
}

主要なPrismaメソッド

メソッド 操作 例
findMany() 複数レコード取得 prisma.user.findMany()
findUnique() 単一レコード取得 prisma.user.findUnique({where:{id}})
create() 新規作成 prisma.user.create({data})
update() 更新 prisma.user.update({where,data})
delete() 削除 prisma.user.delete({where})

13.6 リレーションとトランザクション

複雑なデータ操作では、リレーションを使った参照やトランザクションによる複数操作の原子性保証が重要です。

リレーションを含むクエリ 
// ユーザーと関連する投稿を取得
const userWithPosts = await prisma.user.findUnique({
  where: { id: userId },
  include: {
    posts: {
      where: { published: true }, // 公開済み投稿のみ
      orderBy: { createdAt: 'desc' },
    },
  },
})

// ネストした作成（ユーザーと投稿を同時作成）
const user = await prisma.user.create({
  data: {
    email: 'user@example.com',
    name: 'John Doe',
    posts: {
      create: [
        { title: '初めての投稿', content: '...' },
      ],
    },
  },
})
トランザクション処理 
// 複数の操作を1つのトランザクションで実行
const result = await prisma.$transaction([
  prisma.user.create({ data: { email: 'user1@example.com', name: 'User1' } }),
  prisma.user.create({ data: { email: 'user2@example.com', name: 'User2' } }),
])

// インタラクティブトランザクション（より複雑な処理）
await prisma.$transaction(async (tx) => {
  const user = await tx.user.create({
    data: { email: 'author@example.com', name: 'Author' },
  })

  await tx.post.create({
    data: {
      title: '新しい投稿',
      authorId: user.id,
    },
  })
})

トランザクションのベストプラクティス

トランザクション内では、すべての操作が成功するか、すべてロールバックされます。金銭処理や在庫管理など、データの整合性が重要な場面で活用しましょう。

13.7 接続プールとEdge Runtime

本番環境では、データベース接続の効率的な管理が重要です。また、Edge Runtimeでは従来のPrisma Clientが動作しないため、特別な対応が必要です。

接続プールの設定

schema.prisma 
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
  // 接続プールの設定
  // connection_limit: 最大接続数（デフォルト: 無制限）
  // pool_timeout: 接続待機タイムアウト（秒）
}

推奨設定: Vercelなどサーバーレス環境では、接続数制限に注意してください。PgBouncerなどのコネクションプーリングツールの利用も検討しましょう。

Edge RuntimeでのPrisma利用

Edge RuntimeではPrisma Clientが動作しないため、以下の方法を検討してください：

  • Prisma Accelerate: Prismaの公式コネクションプーリングサービス
  • Vercel Postgres SDK: Edge Runtime対応のデータベースクライアント
  • PlanetScale: Edge対応のMySQLデータベースプラットフォーム

13.8 Vercel PostgresやPlanetScaleとの統合

Next.jsアプリを本番環境にデプロイする際、Vercel PostgresやPlanetScaleといったマネージドデータベースサービスと統合するのが一般的です。

Vercel Postgres

  • ✅ Vercelと完全統合
  • ✅ PostgreSQL互換
  • ✅ 自動スケーリング
  • ✅ 環境変数の自動設定
# Vercel CLIでPostgres作成
vercel postgres create

PlanetScale

  • ✅ MySQL互換
  • ✅ ブランチベースのワークフロー
  • ✅ スキーマレスマイグレーション
  • ✅ 無料枠が充実
// schema.prisma
datasource db {
  provider = "mysql"
  url = env("DATABASE_URL")
  relationMode = "prisma"
}

理解度確認クイズ

  1. Q1. Prismaを使う主なメリットを3つ挙げてください。

    回答を表示

    ①TypeScript完全対応で型安全なクエリが書ける、②マイグレーション機能でスキーマ変更を追跡できる、③自動補完やIntelliSenseにより開発体験が向上する。

  2. Q2. Prisma Clientのシングルトンパターンが必要な理由は？

    回答を表示

    Next.jsの開発環境ではホットリロードが頻繁に発生し、毎回新しいPrisma Clientインスタンスを作成するとデータベース接続が枯渇します。シングルトンパターンでインスタンスを再利用することで接続数を抑制できます。

  3. Q3. Edge RuntimeでPrismaを使う際の課題と解決策を説明してください。

    回答を表示

    Edge Runtimeでは従来のPrisma Clientが動作しないため、Prisma Accelerate（公式コネクションプーリング）、Vercel Postgres SDK、またはPlanetScaleのようなEdge対応データベースクライアントを利用する必要があります。
