第9章: パフォーマンス最適化とベストプラクティス

パフォーマンス最適化とベストプラクティス

この章で学ぶこと
  • 本番環境向けの最適化手法を習得する
  • 未使用CSSの削除とファイルサイズ最小化を学ぶ
  • JITモードの活用方法を理解する
  • メンテナブルなコードの書き方を学ぶ
  • 開発ワークフローのベストプラクティスを理解する

9.1 本番環境ビルドの最適化

開発環境と本番環境では、Tailwind CSSの出力を最適化する必要があります。

NODE_ENV設定

// package.json
{
  "scripts": {
    "dev": "tailwindcss -i ./src/input.css -o ./dist/output.css --watch",
    "build": "NODE_ENV=production tailwindcss -i ./src/input.css -o ./dist/output.css --minify"
  }
}

最適化の仕組み

flowchart TD A["ソースファイル"] --> B["Tailwind CSSビルド"] B --> C["未使用クラスの削除"] C --> D["CSS最小化（minify）"] D --> E["最適化されたCSS"]

9.2 Content設定の最適化

content配列を正確に設定することで、未使用のスタイルを確実に削除できます。

// tailwind.config.js
module.exports = {
  content: [
    './src/**/*.{html,js,jsx,ts,tsx,vue}',  // すべてのファイルタイプを含む
    './public/index.html',                   // 特定のファイルも含む
  ],
  // ...
}

除外パターン

module.exports = {
  content: {
    files: [
      './src/**/*.{html,js}',
    ],
    // node_modulesなど不要なディレクトリを除外
    exclude: ['./node_modules/**', './dist/**'],
  },
}

9.3 JITモード（Just-In-Time）

Tailwind CSS v3以降では、JITモードがデフォルトで有効です。JITモードは必要なスタイルのみを動的に生成し、開発体験とパフォーマンスを向上させます。

JITモードの利点

  • 高速ビルド - 初回ビルドが劇的に速い
  • 小さいファイルサイズ - 使用しているクラスのみ生成
  • 任意の値 - カスタム値を直接クラス名に使用可能

任意の値の使用例

<!-- JITモードでは任意の値を直接指定可能 -->
<div class="w-[137px]">幅137px</div>
<div class="top-[117px]">上から117px</div>
<div class="bg-[#1da1f2]">カスタムカラー</div>
<div class="text-[0.925rem]">カスタムフォントサイズ</div>

9.4 ベストプラクティス

1. コンポーネント抽出の判断基準

すべてのスタイルを@applyで抽出するのではなく、以下の基準で判断します：

  • 同じスタイルの組み合わせを10回以上使用する場合のみ抽出
  • コンポーネントライブラリとして共有する場合
  • プロジェクト全体で統一すべき基本スタイル

2. クラス名の順序

一貫した順序でクラスを記述することで、可読性が向上します。

<!-- 推奨される順序：
1. レイアウト（display、position）
2. サイズ（width、height）
3. スペーシング（margin、padding）
4. タイポグラフィ
5. 色
6. その他の装飾
-->
<div class="flex w-full px-4 py-2 text-lg text-white bg-blue-600 rounded-lg shadow-md">
  順序の良い例
</div>

実習 9-1: ビルドサイズの確認と最適化
手順
  1. 開発ビルドのサイズ確認
npm run dev
# 生成されたCSSファイルのサイズを確認
ls -lh dist/output.css
  1. 本番ビルドの実行
npm run build
# 最適化されたCSSファイルのサイズを確認
ls -lh dist/output.css
期待される結果

本番ビルドでは、ファイルサイズが大幅に削減されます（通常90%以上の削減）。

9.5 メンテナビリティの向上

1. プロジェクト構造の整理

project/
├── src/
│   ├── css/
│   │   ├── base.css        # ベーススタイル
│   │   ├── components.css  # コンポーネント
│   │   └── utilities.css   # カスタムユーティリティ
│   ├── js/
│   └── index.html
├── tailwind.config.js
└── package.json

2. カスタムクラスの命名規則

/* src/css/components.css */
@layer components {
  /* プロジェクト固有のプレフィックスを使用 */
  .app-card {
    @apply bg-white rounded-lg shadow-md p-6;
  }

  .app-btn-primary {
    @apply bg-blue-600 hover:bg-blue-700 text-white px-6 py-2 rounded-lg;
  }
}

3. ドキュメント化

<!-- コンポーネントの用途をコメントで記載 -->
<!-- カードコンポーネント: 商品情報の表示に使用 -->
<div class="bg-white rounded-lg shadow-md p-6">
  ...
</div>
理解度確認クイズ
  1. JITモードの主な利点を3つ挙げてください。

    解答：1) 高速なビルド時間、2) 必要なスタイルのみ生成されるため小さいファイルサイズ、3) 任意の値を直接クラス名に使用できる柔軟性

  2. content配列を正確に設定する重要性は何ですか？

    解答：未使用のスタイルを確実に削除し、最終的なCSSファイルサイズを最小化するためです。設定が不正確だと、必要なスタイルが削除されたり、不要なスタイルが残ったりします。

  3. @applyを使用してスタイルを抽出すべきタイミングは？

    解答：同じスタイルの組み合わせを頻繁に使用する場合（10回以上が目安）、コンポーネントライブラリとして共有する場合、プロジェクト全体で統一すべき基本スタイルの場合です。

9.6 パフォーマンス測定

実際のパフォーマンス改善を確認するためのツール：

  • Lighthouse - Chrome DevToolsのパフォーマンス測定
  • Bundle Analyzer - CSSファイルサイズの分析
  • webpack Bundle Analyzer - ビルド全体の分析

9.7 まとめ

  • 本番環境では必ずビルド最適化を実施する
  • content配列を正確に設定して未使用スタイルを削除
  • JITモードで開発体験とパフォーマンスを向上
  • 一貫したコーディング規約とプロジェクト構造を維持
  • パフォーマンス測定ツールで効果を確認
