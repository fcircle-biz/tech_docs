処理の流れを制御しよう - アクションとロジック

ボタンを押したときの処理をより詳しく設定します。条件によって処理を変える方法を学びます。

このステップで実装すること

  • クライアントアクションとサーバーアクションの違いを理解する
  • フローチャート形式でロジックを作成する
  • 条件分岐（If）を使って処理を変える
  • メッセージ（成功、エラー、警告）を表示する
  • 画面の再読み込み（Refresh）を実装する

前提条件

  • Step 7までの内容が完了していること
  • タスクの登録、編集、削除、完了切り替えの各機能が動作していること

8.1 アクションとは？

OutSystemsでは、ボタンをクリックしたときやデータを保存するときなど、「何か処理を実行する」ことをアクションと呼びます。 アクションはフローチャートのような形で視覚的に作成でき、「この後にこれをする」という処理の流れを簡単に設計できます。

アクションの基本

アクションは「何をする」という処理のかたまりです。たとえば、「タスクを保存する」「エラーメッセージを表示する」「画面を更新する」など、一つひとつの動作を組み合わせて、複雑な処理を作り上げます。

8.2 クライアントアクションとサーバーアクション

OutSystemsには2種類のアクションがあります。それぞれ実行される場所が異なります。

種類 実行場所 使いどころ
クライアントアクション ブラウザ（ユーザーのPC） 画面の表示切り替え、入力チェック、メッセージ表示など
サーバーアクション サーバー（OutSystemsのクラウド） データベースへの保存、複雑な計算、外部APIとの連携など
flowchart LR subgraph ブラウザ A[クライアントアクション] A1[画面操作] A2[入力チェック] A3[メッセージ表示] A --> A1 A --> A2 A --> A3 end subgraph サーバー B[サーバーアクション] B1[データベース操作] B2[複雑な計算] B3[外部API連携] B --> B1 B --> B2 B --> B3 end A -.通信.-> B style A fill:#dbeafe,stroke:#3b82f6 style B fill:#fce7f3,stroke:#ec4899

どちらを使うべき？

基本的にはクライアントアクションを使うことをお勧めします。画面の操作が速く、ユーザー体験が良くなります。 データベースへの保存が必要な場合は、自動的にサーバー側の処理も呼び出されます。

8.3 条件分岐（If）を使おう

アクションの中で「もし○○なら△△する、そうでなければ××する」という条件による処理の切り替えができます。 これを条件分岐（If）と呼びます。

実装 8-1: タスク名が空のときエラーを表示する

アクションフローの確認が中心です。手順4（Mandatory設定）と手順5（日本語メッセージ設定）のみ作業が必要です。

タスク名が未入力のまま保存しようとした場合のエラー表示機能を確認し、必須設定を行います。

1

タスク登録画面を開く

Service Studioで、タスク登録画面（TaskDetailスクリーン）を開きます。

2

保存ボタンのアクションを確認

SaveButtonOnClickをダブルクリックして、アクションフローを開きます。 Formにエンティティをドラッグした際に、以下が自動生成されています：

  • Form1.Valid?: フォームの入力チェック（必須項目など）
  • CreateOrUpdateTask: データベースへの保存
  • Message: 成功メッセージの表示
  • Destination: Home画面への遷移
3

Form1.Valid?の動作を理解する

Form1.Valid?は、フォーム内の入力ウィジェットのバリデーションを自動的にチェックします。 Trueの場合のみ保存処理が実行され、Falseの場合はEndに進みます。

4

タイトル入力欄を必須に設定

TaskDetail画面に戻り、タイトル入力欄（Input）を選択します。プロパティパネルで以下を設定：

  • Mandatory: True

これで、タイトルが空のときにForm1.Valid?がFalseを返し、保存処理がスキップされます。

5

エラーメッセージを日本語に設定

画面右側のWidget Treeで「MyFirstApp」（モジュール名）を選択します。プロパティパネルの「Validation Messages」セクションで以下を設定：

  • Mandatory Input: 「この項目は必須です」

この設定はモジュール全体に適用され、すべての必須項目で日本語メッセージが表示されます。

6

動作を確認する

1-Click Publishでアプリを公開し、タスク名を空のまま保存ボタンをクリックしてみましょう。

  • 入力欄の下に「この項目は必須です」と表示されます
  • 保存処理は実行されません

8.4 メッセージの種類

Feedback Messageには3つの種類があります。使い分けることで、ユーザーにわかりやすい情報を伝えることができます。

種類 表示色 使いどころ 例
Success 緑 処理が成功したとき "タスクを保存しました"
Error 赤 エラーが発生したとき "タスク名を入力してください"
Warning 黄 注意が必要なとき "この操作は取り消せません"

実装 8-2: 保存成功メッセージを確認する

このセクションは内容確認のみです。Messageノードは既に自動生成されているため、新たな作業は不要です。

SaveButtonOnClickに自動生成されているMessageノードの設定を確認しましょう。

1

Messageノードを選択

SaveButtonOnClickのアクションフローで、CreateOrUpdateTaskの後にある「Message」ノードを選択します。

2

プロパティを確認

プロパティパネルで以下の設定を確認します（必要に応じて変更できます）：

  • Message: 任意のメッセージ（例: "タスクを保存しました"）
  • Message Type: Success（緑色で表示）

8.5 画面の再読み込み（Refresh）

タスクの完了状態を切り替えたときなど、データベースのデータが変わった後に画面を最新の状態に更新する必要があります。 これをRefreshと呼びます。

実装 8-3: 完了切り替え後に一覧を更新する

タスクの完了/未完了を切り替えた後、一覧を自動的に再読み込みします。

1

一覧画面を開く

Service Studioで、タスク一覧画面（Homeスクリーン）を開きます。

2

チェックボックスのアクションを編集

完了/未完了を切り替えるチェックボックスの「On Change」アクションを開きます。

3

Refresh Dataを追加

ツールボックスから「Data」→「Refresh Data」をドラッグして、CreateOrUpdateの後に配置します。

4

更新対象を選択

Refresh Dataノードを選択し、プロパティの「Data Source」でタスク一覧のアグリゲート（GetTasksなど）を選択します。

これで、チェックボックスを操作するたびに一覧が自動的に更新されます。

8.6 処理の順序を考えよう

アクションフローでは、処理は上から下、左から右へと順番に実行されます。 処理の順序を間違えると、期待した通りに動かないことがあります。

よくある間違い

❌ 間違い: データを保存する前にメッセージを表示してしまう

✅ 正しい: データを保存した後にメッセージを表示する

処理の順序が重要です。「何をしてから何をするか」を常に意識しましょう。

間違った順序

flowchart TD A[開始] --> B[成功メッセージ表示] B --> C[CreateOrUpdate実行] C --> D[終了] style A fill:#fee2e2,stroke:#ef4444 style B fill:#fee2e2,stroke:#ef4444 style C fill:#fee2e2,stroke:#ef4444 style D fill:#fee2e2,stroke:#ef4444

保存前にメッセージを表示してしまっています。

正しい順序

flowchart TD A[開始] --> B[CreateOrUpdate実行] B --> C[成功メッセージ表示] C --> D[終了] style A fill:#d1fae5,stroke:#10b981 style B fill:#d1fae5,stroke:#10b981 style C fill:#d1fae5,stroke:#10b981 style D fill:#d1fae5,stroke:#10b981

保存してからメッセージを表示しています。

8.7 動作確認

アクションの動作確認

1-Click Publishでアプリを公開し、以下の項目を確認しましょう。

確認すべきポイント

  • タスク名を空のまま保存しようとすると、赤いエラーメッセージが表示される
  • タスクを正しく保存すると、緑の成功メッセージが表示される
  • チェックボックスで完了/未完了を切り替えると、すぐに一覧が更新される
  • メッセージが適切なタイミングで表示される（保存後など）

トラブルシューティング

条件式でエラーが出る

原因: 式の書き方が間違っている可能性があります。

解決方法:

  • プロパティ名を正しく入力しているか確認してください（大文字小文字も区別されます）
  • 空文字列のチェックは = "" を使います（= null ではありません）
  • 式エディタの赤い波線を確認して、エラー箇所を特定してください
メッセージが表示されない

原因: Feedback Messageがアクションフローに接続されていない、または処理の順序が間違っている可能性があります。

解決方法:

  • Feedback Messageノードがアクションフローのどこかにつながっているか確認してください
  • メッセージは、関連する処理（保存など）の後に配置されているか確認してください
  • 1-Click Publishで公開してから確認してください
Refreshしても一覧が更新されない

原因: Refresh Dataで正しいアグリゲートを指定していない可能性があります。

解決方法:

  • Refresh Dataの「Data Source」プロパティで、タスク一覧のアグリゲートを選択しているか確認してください
  • Refresh Dataが、データ更新処理（CreateOrUpdateなど）の後に配置されているか確認してください
  • 1-Click Publishで公開してから確認してください
