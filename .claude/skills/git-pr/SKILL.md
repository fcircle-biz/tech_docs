# Git PR

完了した変更を安全に確認し、コミット → プッシュ → プルリクエスト作成までを日本語で進めるスキル。Git操作では既存のユーザー変更を尊重し、意図しないファイルを含めない。

## 使用方法

```
/git-pr
```

例:
- `/git-pr` — 現在ブランチの変更をコミット・push し、main 宛の日本語PRを作成
- 「PRを作って」「プルリク出して」などの依頼でも起動する

## 前提・既定値

- リモート: `origin`（`git@github.com:fcircle-biz/tech_docs.git`）
- PRのベースブランチ: `main`（既定）
- GitHub操作には `gh` CLI を用いる（利用不可ならPR本文と手動作成手順を提示する）

## 運用ルール（このリポジトリ）

- ブランチ命名: `<type>/<英語スラッグ>`（例: `feature/skill-multiagent-refactor`、`fix/dark-mode-contrast`）。関連Issueがあれば `<type>/<issue#>-<英語スラッグ>` も可。
- PRタイトル: `<type>: <日本語要約>`（Conventional Commits。git-commit スキルと同じ type 体系）。
- 関連Issueがある場合はPR本文に `Closes #<番号>` を入れる（Issueが無い chore/docs 等は省略可）。
- `main` ブランチへ直接コミットしない。作業はブランチを切って行い、PRでマージする。

## 実行手順

1. `git status --short --branch` と差分を確認し、変更内容と現在ブランチを把握する。
2. ユーザーが含めたい変更範囲を推定する。曖昧で危険な場合だけ確認する。
3. 現在 `main` ブランチにいる場合は、作業ブランチを作成してから進める。
4. 関連する変更をステージし、git-commit スキルの規約に沿った日本語コミットメッセージでコミットする。
5. `origin` の対応ブランチへ push する（`git push -u origin <branch>`）。
6. PRのベースブランチ（既定 `main`）を確認する。
7. 「PR本文テンプレート」に沿って日本語PR本文を作成する。
8. `gh pr create` でPRを作成する。作成できない場合はPR本文と手動作成手順を提示する。
9. 完了時にコミットSHA・ブランチ・PR URL・実行した検証を報告する。

## 安全ルール

- 作業前に必ず `git status --short --branch` を確認する。
- 未追跡ファイルや無関係な変更は、ユーザー指示がない限りステージしない。
- 既存のユーザー変更を戻さない。破壊的操作（`git reset --hard` 等）はユーザーが明確に依頼した場合だけ行う。
- push前にリモート名・ブランチ名・ターゲットリポジトリを確認する。
- PR作成前にベースブランチを確認する。

## コミットメッセージ方針

- git-commit スキルと同じ規約を使い、件名は必ず `<type>: <日本語要約>` に統一する。
- `type` は `feat` / `fix` / `docs` / `test` / `refactor` / `chore` / `style` / `ci` から選ぶ。
- 件名だけで意図が伝わりにくい場合だけ本文を付ける。

## PR本文テンプレート

```markdown
## 概要
- [変更内容の要約]

## 変更理由
- [背景・目的・関連Issue]

Closes #<番号>

## テスト / 確認
- [実行した確認手順（例: ブラウザでの表示確認、リンク確認、テンプレート規約準拠）]

## 注意事項
- [レビュー時の注意、未検証事項、移行手順など]

🤖 Generated with [Claude Code](https://claude.com/claude-code)
```

- 関連Issueが無い場合は `Closes #<番号>` 行を省略し、タイトルだけで意図が分かるようにする。
- AI（Claude）が作成したPRは、本文末尾に `🤖 Generated with [Claude Code](https://claude.com/claude-code)` を含める。
- このリポジトリは静的HTML/Markdown教材が中心のため、「テスト / 確認」にはビルドではなくブラウザ表示確認・リンク確認・テンプレート規約準拠などを記載する。

## エラー時の扱い

- push が失敗した場合は、認証・権限・リモート・ブランチ保護・コンフリクトのどれが原因か切り分け、次の手順を示す。
- PR作成が失敗した場合は、失敗理由・push済みブランチ・手動作成に使えるPR本文を提示する。
- コミット対象がない場合は、作業ツリーがクリーンであることを報告する。
- すべての説明を自然な日本語で行う。

## 品質基準

- PRタイトルが `<type>: <日本語要約>` 形式である。
- PR本文に概要・変更理由・確認内容が含まれている。
- 関連Issueがあれば `Closes #<番号>` がある。
- AI作成PRには Claude Code トレーラが付いている。
- 完了報告にPR URL・ブランチ・コミットSHAが含まれている。
