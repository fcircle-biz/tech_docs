# 共通ルール（shared-rules）

docs-practice-creator スキルの全フェーズ・全サブエージェントが従う共通ルールを集約したファイル。重複を避けるため、他の references はルールをここに集約し参照する（DRY）。

---

## 1. 言語・トーン

- 応答・成果物・コメントはすべて**日本語**で出力する。
- トーン: プロフェッショナル・簡潔・分かりやすく。

---

## 2. エージェント実行ルール（CLAUDE.md準拠・厳守）

処理中は以下を厳守する。

1. **提案しない**: 処理中に最適化や代替アプローチを提案しない。
2. **中断しない**: ユーザー確認のために処理を止めない。
3. **完遂する**: 手順の最後まで処理を完了する。
4. **並列は必ず並列**: ワークフローで並列指定された処理は必ず並列で実行する。
5. **「これから生成します」で終了しない**: 実際にファイルが生成されるまで処理を継続する。「開始しました」「これから作成します」等と報告して終了することは禁止。
6. **ファイル未生成で終了しない**: 必ず実ファイルを Write してから報告する。

**禁止例:**
- 「時間がかかるので、どの方法がよいですか？」→ 禁止
- 「先にプレースホルダー版を素早く生成することを提案します」→ 禁止
- 処理中の確認ダイアログ → 禁止

---

## 3. 回答形式（最重要）

- 回答は **`<details>`/`<summary>` クリック展開式のみ** を使用する。「回答を表示」をクリックすると展開される形式。
- **textarea 入力欄は使用禁止。**
- **「実行して確認」ボタン等のインタラクティブ実行UIは使用禁止。**
- 問題スニペットは `question-components.md` に集約。必ずそれを使用する。

---

## 4. 解説の書き方

- **正解だけでなく「なぜそうなるか」を説明する。**
- **よくある間違いとその理由も記載する。**
- 関連する概念へのリンクを含める。
- コード例は実行可能な完全なコードを記載する。

---

## 5. Mermaid記法（CLAUDE.md方式に統一）

旧 step に `&#124;` 等のHTMLエンティティ方式が残っていても、必ず以下のCLAUDE.md方式へ寄せる。詳細パターンは `templates/v3/reference/mermaid-patterns.md` を参照。

1. **HTMLエンティティ禁止**: `&#40;` `&#41;` `&#38;` `&#124;` 等はMermaidで正しく解釈されない。使用しない。
2. **括弧の扱い**: 半角括弧 `()` は特殊文字。**全角括弧** を使うか、ノードテキスト全体を**ダブルクォートで囲む**（例: `A["テキスト（括弧つき）"]`）。
3. **アンパサンド**: `&` ではなく**全角 `＆`** を使う。
4. **改行**: ノード内で `<br/>` を使う場合はノードをダブルクォートで囲む（例: `A["1行目<br/>2行目"]`）。
5. **darkテーマ禁止**: Mermaid の dark テーマは使用しない。

**正しい例:**
```
flowchart TD
    A["第1章: 概要（現在）"] --> B{OS}
    B -->|Windows| C["Windows環境構築"]
```

**誤った例（構文エラー）:**
```
flowchart TD
    A[第1章: 概要<br/>&#40;現在&#41;] --> B{OS}
    B -->|Windows| C[Windows環境<br/>構築]
```

---

## 6. テキスト視認性

カード内テキストの可読性を確保するため以下を守る。

1. **半透明背景禁止**: `bg-white/70`、`bg-white/60`、`bg-white/50` 等は使用しない。
2. **不透明な同系色背景を使う**: カード内に内側ボックスを置く場合、同系色の `-100` シェードを使う。

| 親カード色 | 内側ボックス背景 | テキスト色 |
|------------|------------------|------------|
| Purple | `bg-purple-100` | `text-purple-900` |
| Blue | `bg-blue-100` | `text-blue-900` |
| Green | `bg-emerald-100` | `text-emerald-900` |
| Orange | `bg-orange-100` | `text-orange-900` |

---

## 7. ヘッダー構造（禁止事項）

⚠️ 以下の変更は絶対に行わない。

1. **ダークモードボタンをHTMLに手書きしない**: ダークモードボタンは `main.js` が動的に生成する。
2. **ヘッダー右側のボタンエリア構造を変更しない。**
3. **ヘッダーのクラス名・構造を変更しない。**

---

## 8. スクリプト読込順（厳守）

HTML下部のスクリプト読込は以下の順序を守る。

```
styles.css （<head> 内） → sidebar-content.js → main.js → drawing-tool.js
```

最後に `<script>hljs.highlightAll();</script>` を置く。

---

## 9. カラープレースホルダー置換（必須）

`styles.css` の以下のプレースホルダーは、コピー後に**必ず実カラー値へ置換**する（未置換はCSS破損の原因）。

- `{{PRIMARY_300}}`
- `{{PRIMARY_400}}`
- `{{PRIMARY_500}}`
- `{{PRIMARY_600}}`
- `{{PRIMARY_700}}`
- `{{PRIMARY_RGB}}`

技術別のカラー値は `templates-and-paths.md` の抜粋表、または `templates/v3/reference/color-themes.md` を参照。

---

## 10. バージョン管理

- 出力先フォルダに既存ファイルがある場合、`v1/`/`v2/`/`v3/` ... の順に既存をバージョンフォルダへ退避してから新規作成する。
- 詳細手順は `templates-and-paths.md` を参照。
- 注意: ガイドディレクトリ内の `v1/`/`v2/` 等は過去バージョンのバックアップであり、アクティブなコンテンツではない。Glob検索時は旧バージョンフォルダを除外する。

---

## 11. パス・拡張子のバグ是正（厳守）

- **絶対パスのハードコード禁止**: `/home/ichimaru/...`、`/Users/s-ichimaru/...` 等の旧パスは使わない。リポジトリルート相対で記述する。
- 画像拡張子はPNGに統一する（illustration系を扱う場合）。
- 旧バージョンフォルダ（`v1/`、`v2/` 等）はGlob検索から除外する。

---

## 12. 検証チェックリスト（検証フェーズ）

生成物を以下で検証する（docs-reviewer スキルの手順があればそちらも併用）。

- [ ] 全回HTMLが命名規則 `[技術名]-practice-[NN].html` で生成されている
- [ ] 回答はすべて `<details>` 展開式。textarea・実行ボタンが存在しない
- [ ] 解説に「なぜそうなるか」＋よくある間違いが含まれる
- [ ] Mermaid図がCLAUDE.md方式（エンティティ不使用・dark不使用）
- [ ] 半透明背景がない。内側ボックスは `-100`/`-900`
- [ ] ヘッダーにダークモードボタンの手書きがない／ヘッダー構造が不変
- [ ] スクリプト読込順が styles.css → sidebar-content.js → main.js → drawing-tool.js
- [ ] styles.css の `{{PRIMARY_*}}` がすべて実値へ置換済み
- [ ] sidebar-content.js は全回定義済みで `createSidebar()`/`insertSidebar()` が不変
- [ ] 共通部品（JS/CSS）が上書きされていない
- [ ] 各回HTMLが第1回HTMLの構造から逸脱していない
