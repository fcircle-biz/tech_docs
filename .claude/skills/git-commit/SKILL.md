# Git Commit

変更内容を確認し、意図しない差分を避けながら、統一された日本語コミットメッセージで安全にコミットするスキル。Git操作では既存のユーザー変更を尊重し、含める変更範囲が曖昧な場合だけ確認する。

## 使用方法

```
/git-commit
```

例:
- `/git-commit` — ステージ済み/作業ツリーの変更を確認し、規約に沿ってコミット
- 「コミットして」「コミットメッセージを作って」などの依頼でも起動する

## 実行手順

1. `git status --short --branch` と差分（`git diff` / `git diff --cached`）を確認し、変更内容と現在ブランチを把握する。
2. コミット対象に含めるファイルを特定し、無関係な差分や未追跡ファイルを除外する。
3. 変更内容を要約し、コミット種別(type)・対象・目的を整理する。
4. 「コミットメッセージ規約」に従って日本語コミットメッセージを作成する。
5. 対象ファイルだけをステージし、コミットを実行する。
6. 完了時にコミットメッセージ、コミットSHA、含めた変更範囲、未コミットの差分有無を報告する。

## 安全ルール

- 作業前に必ず `git status --short --branch` を確認する。
- 未追跡ファイルや無関係な変更は、ユーザー指示がない限りステージしない。
- 既存のユーザー変更を戻さない。
- `git reset --hard` や `git checkout --` など破壊的操作は、ユーザーが明確に依頼した場合だけ行う。
- コミット対象が曖昧で、意図しない変更を巻き込む可能性がある場合だけ確認する。
- コミット対象が存在しない場合は、コミットせずに作業ツリーの状態を報告する。
- デフォルトブランチ（main）で作業している場合は、コミット前にブランチ作成を検討する。

## コミットメッセージ規約

件名は必ず次の形式に統一する。

```text
<type>: <変更内容の要約>
```

- `type` は `feat` / `fix` / `docs` / `test` / `refactor` / `chore` / `style` / `ci` から選ぶ。
- 件名は日本語で簡潔に書く。先頭は小文字の Conventional Commits 形式に統一する。
- 要約は「何を変えたか」が一読で分かる表現にする。
- 句点は付けない。可能であれば 30〜50 文字程度に収める。

必要な場合だけ、本文を以下の形式で追記する。

```text
<type>: <変更内容の要約>

背景:
- [必要性や問題]

対応:
- [主な変更点]

確認:
- [実行したテストや確認内容]
```

- 本文は、件名だけでは意図や影響範囲が伝わりにくい場合に付ける。

### AI が作成するコミットのトレーラ（tech_docs 規約）

Claude（AI）が作成したコミットは、メッセージ末尾に共同作成者トレーラを付与する。

```text
Co-Authored-By: Claude <noreply@anthropic.com>
```

> 実行中のモデル名を含む形式（例: `Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>`）でもよい。ハーネス／CLAUDE.md の規約に従う。

## type の使い分け

- `feat`: ユーザー向け機能追加・仕様追加（このリポジトリでは新規教材・新機能ページ等）。
- `fix`: バグ修正・不具合是正（表示崩れ・リンク切れ・内容誤り等）。
- `docs`: ドキュメント追加・更新・整理（README、CLAUDE.md、教材コンテンツ）。
- `test`: テスト追加・修正・安定化。
- `refactor`: 振る舞いを変えない内部改善（テンプレート/スキル整理等）。
- `chore`: 雑務・設定・依存関係・運用補助。
- `style`: フォーマットや命名など非機能の見た目調整。
- `ci`: CI設定・ワークフロー・ビルド自動化変更。

> このリポジトリは静的HTML/Markdown教材が中心のため、`docs` と `chore` の使用頻度が高い。

## 例

```text
docs: Docker入門ガイドに第5章を追加
fix: Pythonガイドのダークモードで配色の視認性を修正
chore: 練習問題テンプレートのスクリプト読込順を統一
refactor: サブエージェントをスキル化
```

## 判断ルール

- 複数の変更が混在する場合は、コミットの主目的に最も近い `type` を選ぶ。
- 大きく別の目的が混ざる場合は、コミットを分割できないか検討する。
- 変更内容が複数領域をまたぐ場合でも、主目的に合わせて1つの `type` を選ぶ。
- メッセージに迷う場合は、ユーザーに見せたい変更の中心を件名にする。

## 品質基準

- 件名の形式が `<type>: <日本語要約>` に統一されている。
- 件名だけで変更の中心が分かる。
- `type` が変更の主目的に合っている。
- コミット対象に無関係な差分が含まれていない。
- AI 作成コミットには Co-Authored-By トレーラが付いている。
- 報告内容にコミットSHAと残差分の有無が含まれている。
