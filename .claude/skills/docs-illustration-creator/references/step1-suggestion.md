# 提案フェーズ（HTML分析 → 図解提案レポート）

旧 `illustration-creator-step1` の移植。HTML形式の技術ガイドを分析し、図解（イラスト、ダイアグラム、フローチャート等）を追加することで学習効果が向上する箇所を特定し、提案レポート（`chapter-XX.md`）を生成する。

共通ルールは `common-rules.md` を参照。

## 入力パラメータ

- **TargetDirectory**: 分析対象のHTMLガイドが含まれるディレクトリパス（例: `docs/guide/programming-languages/java-ecosystem/jsp`）。
- **ChapterNumber**（オプション）: 処理対象の章番号（例: `3`）。指定しない場合は全章。並列実行時は特定章のみを処理。

## 出力形式

```
[TargetDirectory]/illustration_suggestions/
├── README.md              # 全体サマリー（全章処理モードのみ）
├── chapter-01.md          # 第1章の提案
├── chapter-02.md          # 第2章の提案
└── ...
```

## 実行モード

### 全章処理モード（ChapterNumber なし）

- すべてのHTMLファイルを順番に処理（メモリ効率のため1ファイルずつ。前のファイル内容は保持不要）。
- 最後に README.md（全体サマリー）を生成。

### 単一章処理モード（ChapterNumber 指定あり）

- 指定章のHTMLファイルのみを処理。
- `chapter-[XX].md` のみ生成（README.md は生成しない）。
- 並列実行時はこのモードを使用。

## 実行手順

### 1. HTMLファイルの検索

Glob で `TargetDirectory` 内の `*-learning-material-*.html` を検索する。

- **除外対象**: `v1/` `v2/` 等の過去バージョンディレクトリ（`common-rules.md`）。
- **単一章モード時**: `*-learning-material-[XX].html`（XX=指定章番号・2桁ゼロパディング）のみ対象。

### 2. フォルダ準備

`[TargetDirectory]/illustration_suggestions/` フォルダを作成（存在しない場合のみ）。

### 3. HTML分析と提案レポート生成

各HTMLファイルについて:

1. ファイルを読み込む。
2. 図解が有効な箇所を特定（後述の「図解が有効なコンテンツタイプ」参照）。
3. `chapter-[XX].md` として保存。
4. 次のファイルへ。

### 4. README.md 生成（全章処理モードのみ）

> 単一章処理モードではこのステップをスキップ。

すべての章の処理完了後、全体サマリーファイルを生成する。

## リファレンス

### 図解が有効なコンテンツタイプ

| カテゴリ | 具体例 |
| :--- | :--- |
| 抽象的な概念 | アーキテクチャ概念、デザインパターン、理論モデル |
| フロー・プロセス | リクエスト/レスポンス、データ処理、ライフサイクル、状態遷移 |
| コンポーネント関係 | システム構成、クラス依存関係、モジュール連携 |
| 階層・分類 | ディレクトリ構造、クラス階層、カテゴリ分類 |
| 比較・対比 | 技術比較、メリット/デメリット、選択肢の違い |
| 時系列・シーケンス | 処理順序、イベント発生順、通信シーケンス |

### 図解タイプ一覧

| タイプ | 用途 |
| :--- | :--- |
| フローチャート | プロセス、処理フロー、条件分岐 |
| シーケンス図 | コンポーネント間通信、API呼び出し順序 |
| アーキテクチャ図 | システム構成、レイヤー構造 |
| クラス図 | クラス関係、継承構造 |
| 状態遷移図 | オブジェクトの状態変化 |
| 概念図 | 抽象的概念の視覚化 |
| 比較表 | 複数選択肢や技術の比較 |
| タイムライン | 時系列での変化や歴史 |

## 注意事項

- 必ず日本語で出力する。
- **各章の提案数は1〜2件に厳選する**（最も学習効果が高い箇所のみを選定）。
- 既存のMermaidダイアグラムがある箇所は、追加提案の必要性を評価する。十分なら「既存のMermaid図で十分」と明記し、後続フェーズでスキップさせる。
- コードサンプルが十分に説明的な場合は、無理に図解を提案しない。
- 各提案には具体的で実行可能な内容を含める。
- 技術分類・命名規則の参照が必要な場合は `tech-knowledge-map.md` を参照する（丸写ししない）。
